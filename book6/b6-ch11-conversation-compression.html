Chapter 11: Conversation Summarization and Compression in the Database

Every production chatbot eventually runs into the same wall: context windows are finite, tokens are expensive, and users have long memories. A customer who started a support conversation three weeks ago expects your AI to remember the resolution — but storing every message verbatim and replaying the full transcript on each request is neither scalable nor economical. Conversation summarization solves this by compressing older turns into dense representations while keeping recent turns verbatim, giving the model enough context without burning your token budget. This chapter shows how to design, implement, and operate a conversation compression pipeline entirely within PostgreSQL, with async summarization jobs, rolling summaries, embedding-based retrieval for relevant history, and production-grade monitoring. PostgreSQL is the right place to build this — it gives you transactional consistency between raw messages and their summaries, trigger-driven automation, JSONB for metadata, and pgvector for semantic recall, all in one system rather than five.

---

11.1 The Context Window Problem and Why Compression Belongs in the Database

A language model doesn't read your database. It reads a prompt. Everything your application knows about a user's conversation history has to be squeezed into that prompt before the model ever sees it. This creates a hard engineering constraint: the prompt must be short enough to fit the context window, recent enough to feel coherent, and complete enough to be useful. Most teams handle this badly — they either truncate blindly (losing critical information), replay everything (burning tokens), or move to an external vector store (losing transactional consistency).

The database-centric approach treats conversation memory as a first-class data problem. Messages are rows. Summaries are rows. Embeddings are columns. The rules for when to summarize, what to include, and how to retrieve relevant history are queries. Every operation is transactional, auditable, and indexable.

The strategy has three tiers:

Tier 1 — Verbatim recent history: the last N turns, always included in full. N is typically 4–10 depending on average turn length.

Tier 2 — Rolling summary: a periodically regenerated compressed narrative of older turns, included as a paragraph-length summary.

Tier 3 — Semantic recall: embeddings of older turns, queried at retrieval time to pull back turns that are semantically similar to the current user message, even if they happened long ago.

-- Complete schema for a three-tier conversation memory system

CREATE EXTENSION IF NOT EXISTS pgvector;
CREATE EXTENSION IF NOT EXISTS pg_trgm;  -- for fuzzy search on text content

-- Core conversation and session tracking
CREATE TABLE conversations (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id             BIGINT NOT NULL,
    channel             TEXT NOT NULL DEFAULT 'web',   -- web, slack, api, mobile
    title               TEXT,                           -- auto-generated or user-set
    status              TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'archived', 'deleted')),
    metadata            JSONB NOT NULL DEFAULT '{}',
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    last_active_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    message_count       INT NOT NULL DEFAULT 0,
    total_tokens_used   BIGINT NOT NULL DEFAULT 0
);

CREATE INDEX idx_conversations_user_active
    ON conversations (user_id, last_active_at DESC)
    WHERE status = 'active';

-- Raw message store — the source of truth
CREATE TABLE messages (
    id              BIGSERIAL PRIMARY KEY,
    conversation_id UUID NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
    turn_number     INT NOT NULL,           -- sequential within conversation
    role            TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system', 'tool')),
    content         TEXT NOT NULL,
    content_tokens  INT,                    -- populated by application layer
    model_used      TEXT,                   -- which model generated this (for assistant turns)
    metadata        JSONB NOT NULL DEFAULT '{}',  -- tool calls, citations, latency, etc.
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    summarized      BOOLEAN NOT NULL DEFAULT FALSE,  -- has this turn been rolled into a summary?
    UNIQUE (conversation_id, turn_number)
);

CREATE INDEX idx_messages_conversation_turn
    ON messages (conversation_id, turn_number DESC);

CREATE INDEX idx_messages_unsummarized
    ON messages (conversation_id, turn_number)
    WHERE summarized = FALSE;

-- Rolling summaries — one or more per conversation
CREATE TABLE conversation_summaries (
    id                  BIGSERIAL PRIMARY KEY,
    conversation_id     UUID NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
    summary_version     INT NOT NULL DEFAULT 1,  -- increments on each regeneration
    turn_range_start    INT NOT NULL,            -- first turn number covered
    turn_range_end      INT NOT NULL,            -- last turn number covered
    content             TEXT NOT NULL,           -- the summary text
    content_tokens      INT,
    model_used          TEXT NOT NULL,
    generation_latency_ms INT,
    is_current          BOOLEAN NOT NULL DEFAULT TRUE,
    embedding           VECTOR(1536),            -- embed the summary for semantic retrieval
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (conversation_id, summary_version)
);

CREATE INDEX idx_summaries_conversation_current
    ON conversation_summaries (conversation_id, summary_version DESC)
    WHERE is_current = TRUE;

CREATE INDEX idx_summaries_embedding
    ON conversation_summaries USING ivfflat (embedding vector_cosine_ops)
    WITH (lists = 100);

-- Per-message embeddings for semantic recall (Tier 3)
CREATE TABLE message_embeddings (
    message_id      BIGINT PRIMARY KEY REFERENCES messages(id) ON DELETE CASCADE,
    embedding       VECTOR(1536) NOT NULL,
    model_used      TEXT NOT NULL DEFAULT 'text-embedding-3-small',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_message_embeddings_vec
    ON message_embeddings USING ivfflat (embedding vector_cosine_ops)
    WITH (lists = 100);

-- Compression job queue — tracks summarization work
CREATE TABLE summarization_jobs (
    id                  BIGSERIAL PRIMARY KEY,
    conversation_id     UUID NOT NULL REFERENCES conversations(id),
    status              TEXT NOT NULL DEFAULT 'pending'
                        CHECK (status IN ('pending', 'running', 'completed', 'failed')),
    trigger_reason      TEXT NOT NULL,  -- 'turn_count', 'token_budget', 'manual', 'scheduled'
    turns_to_summarize  INT[],          -- array of turn numbers to compress
    attempt_count       INT NOT NULL DEFAULT 0,
    last_error          TEXT,
    scheduled_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    started_at          TIMESTAMPTZ,
    completed_at        TIMESTAMPTZ,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_summarization_jobs_pending
    ON summarization_jobs (scheduled_at)
    WHERE status = 'pending';

This schema is intentionally normalized. Summaries are rows, not JSONB fields buried inside a conversation record. That gives you version history, allows multiple granularities of summary (chapter-level, session-level), and makes it straightforward to query which turns have and haven't been compressed.

11.2 Trigger-Driven Compression Scheduling

The best compression pipelines are invisible to application code. When a message is inserted, a database trigger evaluates whether compression is needed and enqueues a job if so. The application doesn't make that decision — the database does, consistently, every time.

-- Function to count unsummarized tokens and decide whether to enqueue compression
CREATE OR REPLACE FUNCTION estimate_message_tokens(content TEXT)
RETURNS INT
LANGUAGE sql
IMMUTABLE
AS $$
    -- Rough approximation: 1 token ≈ 4 characters for English
    -- Replace with a more precise calculation if needed
    SELECT greatest(1, length(content) / 4);
$$;

CREATE OR REPLACE FUNCTION update_conversation_stats()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
BEGIN
    UPDATE conversations
    SET
        message_count      = message_count + 1,
        last_active_at     = now(),
        total_tokens_used  = total_tokens_used + COALESCE(NEW.content_tokens, estimate_message_tokens(NEW.content))
    WHERE id = NEW.conversation_id;

    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_update_conversation_stats
    AFTER INSERT ON messages
    FOR EACH ROW
    EXECUTE FUNCTION update_conversation_stats();

-- Core compression scheduler: enqueues a job when unsummarized turns exceed threshold
CREATE OR REPLACE FUNCTION maybe_schedule_summarization()
RETURNS TRIGGER
LANGUAGE plpgsql
AS $$
DECLARE
    v_unsummarized_count    INT;
    v_unsummarized_tokens   BIGINT;
    v_turn_threshold        INT     := 15;   -- summarize after 15 unsummarized turns
    v_token_threshold       INT     := 6000; -- or after ~6k unsummarized tokens
    v_keep_verbatim         INT     := 6;    -- always keep last 6 turns verbatim
    v_turns_to_summarize    INT[];
    v_pending_exists        BOOLEAN;
BEGIN
    -- Don't schedule if there's already a pending job for this conversation
    SELECT EXISTS (
        SELECT 1 FROM summarization_jobs
        WHERE conversation_id = NEW.conversation_id
          AND status IN ('pending', 'running')
    ) INTO v_pending_exists;

    IF v_pending_exists THEN
        RETURN NEW;
    END IF;

    -- Count unsummarized turns, excluding the most recent ones we want to keep verbatim
    SELECT
        COUNT(*),
        COALESCE(SUM(COALESCE(content_tokens, estimate_message_tokens(content))), 0)
    INTO v_unsummarized_count, v_unsummarized_tokens
    FROM messages m
    WHERE m.conversation_id = NEW.conversation_id
      AND m.summarized = FALSE
      AND m.turn_number < (
          SELECT turn_number
          FROM messages
          WHERE conversation_id = NEW.conversation_id
          ORDER BY turn_number DESC
          LIMIT 1 OFFSET (v_keep_verbatim - 1)
      );

    -- Check if we've crossed either threshold
    IF v_unsummarized_count >= v_turn_threshold OR v_unsummarized_tokens >= v_token_threshold THEN

        -- Collect the turn numbers that need summarizing
        SELECT array_agg(turn_number ORDER BY turn_number)
        INTO v_turns_to_summarize
        FROM messages m
        WHERE m.conversation_id = NEW.conversation_id
          AND m.summarized = FALSE
          AND m.turn_number < (
              SELECT turn_number
              FROM messages
              WHERE conversation_id = NEW.conversation_id
              ORDER BY turn_number DESC
              LIMIT 1 OFFSET (v_keep_verbatim - 1)
          );

        INSERT INTO summarization_jobs (
            conversation_id,
            trigger_reason,
            turns_to_summarize
        ) VALUES (
            NEW.conversation_id,
            CASE
                WHEN v_unsummarized_tokens >= v_token_threshold THEN 'token_budget'
                ELSE 'turn_count'
            END,
            v_turns_to_summarize
        );

        -- Notify the async worker via pg_notify
        PERFORM pg_notify(
            'summarization_needed',
            json_build_object(
                'conversation_id', NEW.conversation_id,
                'turn_count',      v_unsummarized_count,
                'token_estimate',  v_unsummarized_tokens
            )::text
        );
    END IF;

    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_maybe_schedule_summarization
    AFTER INSERT ON messages
    FOR EACH ROW
    WHEN (NEW.role IN ('assistant', 'user'))  -- don't trigger on system/tool messages
    EXECUTE FUNCTION maybe_schedule_summarization();

-- Helper view: what context should be assembled for a given conversation?
CREATE OR REPLACE VIEW conversation_context AS
SELECT
    c.id AS conversation_id,
    c.user_id,
    -- Current summary (if any)
    cs.content           AS summary_text,
    cs.turn_range_start  AS summary_covers_from,
    cs.turn_range_end    AS summary_covers_to,
    -- Recent verbatim turns
    (
        SELECT json_agg(
            json_build_object(
                'role',       m.role,
                'content',    m.content,
                'turn',       m.turn_number,
                'created_at', m.created_at
            )
            ORDER BY m.turn_number
        )
        FROM messages m
        WHERE m.conversation_id = c.id
          AND m.summarized = FALSE
    ) AS recent_turns,
    c.message_count,
    c.last_active_at
FROM conversations c
LEFT JOIN conversation_summaries cs
    ON cs.conversation_id = c.id
    AND cs.is_current = TRUE;

The pg_notify call at the end of the trigger is what connects the database event to your async Python worker. The trigger does the decision-making; the worker does the expensive LLM call. This separation means your INSERT latency is unaffected by summarization time.

11.3 The Summarization Worker

The async worker listens on the PostgreSQL notification channel and processes jobs when they appear. It uses Celery for retry logic and Claude or GPT-4o for the actual summarization, then writes the result back and marks messages as summarized — atomically.
