Chapter 16: Integrations

Databases rarely live in isolation. In modern production environments, PostgreSQL and SQL Server must speak to application servers, message queues, ETL pipelines, cloud platforms, reporting tools, and sometimes each other. How well your database integrates with the surrounding ecosystem determines how useful and resilient your data infrastructure actually is. This chapter covers the most common integration patterns DBAs encounter — from application connectivity and connection pooling, to foreign data wrappers, linked servers, ETL pipelines, and event-driven architectures. By the end, you will understand not just how to wire things together, but how to do it in a way that holds up under production load and scales without drama.

---

Application Connectivity: Drivers, Connection Strings, and What Goes Wrong

Before any integration can happen, an application needs a connection. Both PostgreSQL and SQL Server have mature driver ecosystems, but the choices you make at the driver level ripple through everything above it — query performance, connection handling, error recovery, and security.

For PostgreSQL, the most common drivers are libpq (the native C library that most other drivers wrap), psycopg2 and psycopg3 for Python, pgx for Go, node-postgres for Node.js, and JDBC with the official PostgreSQL JDBC driver for JVM languages. For SQL Server, the primary options are the Microsoft JDBC Driver, pyodbc or pymssql for Python, the SqlClient library for .NET, and Go's database/sql with the mssql driver.

Connection strings carry more information than most developers realize. The hostname, port, database name, and credentials are obvious, but production connection strings often also encode SSL requirements, connection timeouts, application names, and target session attributes (for PostgreSQL read replicas). The application name parameter deserves special attention — it shows up in pg_stat_activity and SQL Server's sys.dm_exec_sessions, making it far easier to identify which service is responsible for a problematic query.

PostgreSQL:

-- Check active connections and their application names
SELECT pid,
       usename,
       application_name,
       client_addr,
       state,
       wait_event_type,
       wait_event,
       now() - query_start AS query_duration,
       left(query, 80) AS query_preview
FROM   pg_stat_activity
WHERE  state <> 'idle'
ORDER  BY query_duration DESC;

SQL Server:

-- Check active sessions and their application names
SELECT s.session_id,
       s.login_name,
       s.program_name,
       s.client_interface_name,
       s.host_name,
       r.status,
       r.wait_type,
       r.total_elapsed_time / 1000.0 AS elapsed_seconds,
       LEFT(t.text, 80)              AS query_preview
FROM   sys.dm_exec_sessions s
JOIN   sys.dm_exec_requests r
       ON s.session_id = r.session_id
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
WHERE  s.is_user_process = 1
ORDER  BY r.total_elapsed_time DESC;

SSL/TLS at the connection level is another area where integration decisions matter. PostgreSQL supports several SSL modes — disable, allow, prefer, require, verify-ca, and verify-full — and the choice affects both security and performance. In production, anything less than verify-full can expose you to man-in-the-middle risks, but many teams ship with require and skip certificate validation. SQL Server uses encrypted connections controlled at both the server and driver level, and the TrustServerCertificate flag in the connection string is a common shortcut that bypasses certificate validation in ways that should make any security-conscious DBA uncomfortable.

A common connectivity failure pattern is applications that open connections during startup, hold them for the lifetime of the process, and never reconnect cleanly after a network interruption, failover, or maintenance restart. Both databases close idle connections after a timeout (configurable via tcp_keepalives_idle in PostgreSQL and connection timeout settings in SQL Server), and applications that lack reconnection logic will begin throwing errors in ways that look like database problems but are actually driver-level issues. Understanding this failure mode saves significant debugging time.

Connection Pooling: Why Every Production System Needs It

PostgreSQL's process-per-connection model means that every new connection spawns an operating system process. At low concurrency this is fine. Under sustained load with hundreds or thousands of clients, the cost of process creation, memory overhead per connection, and context-switching becomes significant. SQL Server uses a thread-based model that handles concurrency somewhat better natively, but even there, uncontrolled connection counts create resource contention and can exhaust server-side connection limits.

Connection pooling sits between the application and the database, maintaining a pool of pre-established connections that applications borrow and return. Rather than each application thread opening its own connection for every request, threads check out a connection from the pool, use it, and return it — typically in milliseconds. From the database's perspective, it sees a small, stable number of connections instead of a constantly churning storm of connects and disconnects.

PgBouncer is the de facto connection pooler for PostgreSQL in production. It operates in three modes: session pooling (one server connection per client session, barely better than no pooling), transaction pooling (server connection returned to pool after each transaction, the most common production choice), and statement pooling (server connection returned after each statement, rarely used due to constraints it imposes). Transaction pooling is the right default for almost every OLTP workload.

PgBouncer's configuration file controls pool sizes, authentication methods, and per-database limits:

[databases]
myapp = host=127.0.0.1 port=5432 dbname=myapp

[pgbouncer]
listen_port = 6432
listen_addr = 0.0.0.0
auth_type = md5
auth_file = /etc/pgbouncer/userlist.txt
pool_mode = transaction
max_client_conn = 1000
default_pool_size = 25
reserve_pool_size = 5
reserve_pool_timeout = 3
server_idle_timeout = 600
log_connections = 0
log_disconnections = 0

With this configuration, up to 1,000 client connections share 25 server connections to the database. The math is the point: your application scales to hundreds of concurrent requests, but the database handles a small, predictable number of connections. The reserve_pool_size adds a small buffer for traffic spikes before clients start experiencing connection wait timeouts.

For SQL Server, connection pooling is typically handled at the driver/ADO.NET layer rather than through a separate process. The .NET SqlClient driver manages connection pools automatically, keying pools by connection string — identical connection strings share a pool, while any difference in connection string (including a trailing space) creates a separate pool. You can observe pool activity through SQL Server's DMVs:

PostgreSQL:

-- Monitor PgBouncer-style pool health via pg_stat_activity
SELECT datname,
       usename,
       state,
       COUNT(*) AS connection_count
FROM   pg_stat_activity
WHERE  pid <> pg_backend_pid()
GROUP  BY datname, usename, state
ORDER  BY connection_count DESC;

SQL Server:

-- Monitor connection pool usage by program and login
SELECT s.program_name,
       s.login_name,
       s.status,
       COUNT(*)             AS session_count,
       SUM(s.memory_usage) AS total_memory_pages
FROM   sys.dm_exec_sessions s
WHERE  s.is_user_process = 1
GROUP  BY s.program_name, s.login_name, s.status
ORDER  BY session_count DESC;

One important architectural note: when using PgBouncer in transaction mode, features that depend on session-level state become unreliable. Prepared statements, advisory locks, SET LOCAL configuration, and temporary tables all live at the session level. If your application relies on these across transaction boundaries, you either need session pooling, or you need to refactor the application to avoid session-level state. This is one of the more consequential tradeoffs in the PostgreSQL integration world.

Pgpool-II is an alternative that offers connection pooling alongside load balancing and query routing. It can automatically send read queries to replicas and writes to the primary, which makes it a meaningful architectural component rather than just a pooler. The tradeoff is operational complexity — Pgpool-II has more moving parts and is harder to operate correctly. For most teams, PgBouncer with an application-level read/write split is a simpler and more transparent approach.

Foreign Data Wrappers and Linked Servers: Querying Across Databases

There are legitimate scenarios where data that logically belongs together lives in separate databases — legacy systems that cannot be migrated, vendor-owned databases, archival stores, or analytical databases that need to be queried alongside operational data. Both PostgreSQL and SQL Server have mechanisms for querying remote data sources without moving the data first.

PostgreSQL's Foreign Data Wrapper (FDW) framework is a formal extension API that allows PostgreSQL to treat external data sources as if they were local tables. postgres_fdw ships with PostgreSQL and allows cross-instance querying between PostgreSQL servers. mysql_fdw, tds_fdw (for SQL Server), oracle_fdw, file_fdw, and dozens of community wrappers exist for other data sources.

Setting up postgres_fdw involves three steps: creating the foreign server object, creating a user mapping, and then creating foreign tables that map to remote tables.

PostgreSQL:

-- Step 1: Create the foreign server pointing to the remote PostgreSQL instance
CREATE SERVER remote_analytics
    FOREIGN DATA WRAPPER postgres_fdw
    OPTIONS (host 'analytics-db.internal', port '5432', dbname 'analytics');

-- Step 2: Map the local user to a remote credential
CREATE USER MAPPING FOR reporting_user
    SERVER remote_analytics
    OPTIONS (user 'fdw_reader', password 'secure_password_here');

-- Step 3: Import the remote schema (auto-creates foreign tables)
IMPORT FOREIGN SCHEMA public
    FROM SERVER remote_analytics
    INTO fdw_schema;

-- Now query across databases as if they were local tables
SELECT o.order_id,
       o.customer_id,
       a.customer_lifetime_value
FROM   orders o
JOIN   fdw_schema.customer_analytics a
       ON o.customer_id = a.customer_id
WHERE  o.order_date >= CURRENT_DATE - INTERVAL '30 days';

The query planner is aware of FDW tables and will push down WHERE clauses, JOINs, and aggregations to the remote server when the FDW implementation supports it (postgres_fdw does). This pushdown behavior is significant — without it, PostgreSQL would fetch entire remote tables over the network and filter locally, which is catastrophically slow for large tables. You can verify what gets pushed down using EXPLAIN VERBOSE, which will show Remote SQL clauses in the plan for FDW nodes.

SQL Server's equivalent is Linked Servers, which allow T-SQL queries to reference objects on remote SQL Server instances, Oracle databases, or any OLE DB or ODBC data source. A linked server is created using sp_addlinkedserver and queried using four-part naming (server.database.schema.table).

SQL Server:

-- Create a linked server to a remote SQL Server instance
EXEC sp_addlinkedserver
    @server     = N'REMOTE_ANALYTICS',
    @srvproduct = N'SQL Server';

-- Set up login mapping
EXEC sp_addlinkedsrvlogin
    @rmtsrvname  = N'REMOTE_ANALYTICS',
    @useself     = N'FALSE',
    @locallogin  = NULL,
    @rmtuser     = N'fdw_reader',
    @rmtpassword = N'secure_password_here';

-- Query across linked servers using four-part naming
SELECT o.order_id,
       o.customer_id,
       a.customer_lifetime_value
FROM   dbo.orders o
JOIN   REMOTE_ANALYTICS.analytics.dbo.customer_analytics a
       ON o.customer_id = a.customer_id
WHERE  o.order_date >= DATEADD(DAY, -30, GETDATE());

Both mechanisms carry performance caveats that must be understood before they are used in production. FDW queries and linked server queries introduce network latency on every execution. If the query planner does not push down filtering effectively, you risk large data transfers. Linked server queries in SQL Server can sometimes be hard to predict in terms of what the optimizer chooses to execute remotely versus locally, and OPENQUERY (which sends a literal SQL string to the remote server) can be used to force specific remote execution when the optimizer makes poor choices.

Neither FDW nor linked servers should be used as a substitute for proper data architecture. They are useful for operational queries and migrations, but building application logic that depends on cross-server queries at high frequency is a reliability and performance risk. Network partitions, remote server maintenance, and latency variability will all manifest as errors or slowdowns in your application.

ETL and Data Pipeline Integration

Extract, Transform, Load (ETL) and its inverse ELT (Extract, Load, Transform) represent the most common pattern for moving data between databases and into analytical systems. As a DBA, you are not usually writing the pipeline code, but you are responsible for what the pipeline does to your database — the load it places on the server, the locking behavior it exhibits, and how it handles failures and restarts.

Bulk loading is the most efficient way to get large volumes of data into either database. PostgreSQL's COPY command and SQL Server's BULK INSERT / bcp utility bypass row-by-row processing and write data in bulk, with significantly lower overhead per row than individual INSERT statements.

PostgreSQL:

-- Bulk load from a CSV file using COPY
COPY staging.raw_events (event_id, user_id, event_type, occurred_at)
FROM '/data/pipeline/events_20240115.csv'
WITH (FORMAT CSV,
      HEADER TRUE,
      DELIMITER ',',
      NULL '\\N',
      ENCODING 'UTF8');

-- Alternatively, load from a program output (useful for piped decompression)
COPY staging.raw_events
FROM PROGRAM 'gunzip -c /data/pipeline/events_20240115.csv.gz'
WITH (FORMAT CSV, HEADER TRUE);

SQL Server:

-- Bulk load from a CSV file using BULK INSERT
BULK INSERT staging.raw_events
FROM 'D:\pipeline\events_20240115.csv'
WITH (FIELDTERMINATOR  = ',',
      ROWTERMINATOR    = '\n',
      FIRSTROW         = 2,
      TABLOCK,
      MAXERRORS        = 0,
      CODEPAGE         = '65001');

-- Check how many rows were loaded
SELECT COUNT(*) AS rows_loaded
FROM   staging.raw_events;

Staging tables are the right pattern for almost all pipeline loads. Rather than inserting directly into production tables, data arrives in a staging table first, where it can be validated, deduplicated, and transformed before being merged into the production schema. This approach also decouples pipeline failures from production data integrity — a partially loaded staging table has no impact on users.

The MERGE statement (in SQL Server, called MERGE; in PostgreSQL, called INSERT ... ON CONFLICT or the MERGE statement available since PostgreSQL 15) handles the common upsert pattern: insert new rows, update existing rows.

PostgreSQL:

-- Upsert from staging to production using INSERT ... ON CONFLICT
INSERT INTO public.customers (customer_id, email, full_name, updated_at)
SELECT customer_id,
       email,
       full_name,
       updated_at
FROM   staging.customer_updates
ON CONFLICT (customer_id)
DO UPDATE SET
    email      = EXCLUDED.email,
    full_name  = EXCLUDED.full_name,
    updated_at = EXCLUDED.updated_at
WHERE  public.customers.updated_at < EXCLUDED.updated_at;

SQL Server:

-- Upsert from staging to production using MERGE
MERGE dbo.customers AS target
USING staging.customer_updates AS source
    ON target.customer_id = source.customer_id
WHEN MATCHED AND target.updated_at < source.updated_at THEN
    UPDATE SET
        target.email      = source.email,
        target.full_name  = source.full_name,
        target.updated_at = source.updated_at
WHEN NOT MATCHED BY TARGET THEN
    INSERT (customer_id, email, full_name, updated_at)
    VALUES (source.customer_id, source.email, source.full_name, source.updated_at);

CDC (Change Data Capture) has become a foundational technology for modern data pipelines. Rather than running scheduled batch loads that must determine what changed since the last run, CDC reads the database's transaction log and emits a stream of row-level changes in near-real-time. This is both more efficient and more accurate than timestamp-based polling, which requires updated_at columns on every table and cannot capture deletes.

SQL Server has built-in CDC support that can be enabled per table. PostgreSQL relies on logical replication — either via the built-in pgoutput plugin or tools like Debezium, which reads PostgreSQL's logical replication stream and publishes changes to Kafka. Debezium has become one of the most widely deployed integration tools in the PostgreSQL ecosystem, enabling streaming data pipelines from PostgreSQL into data warehouses, Elasticsearch, or other databases.

Enabling logical replication on PostgreSQL requires setting wal_level = logical in postgresql.conf and ensuring the tables you want to capture have primary keys (or replica identity set appropriately). The DBA's role in a CDC pipeline is to ensure that wal_level is configured correctly, that replication slots are monitored (an unconsumed replication slot will cause WAL accumulation that can fill your disk), and that the pipeline consumer is healthy enough to keep up with the change stream.

Integrating with Cloud Platforms, Analytics, and External Services

As databases have moved into cloud and hybrid environments, DBAs must understand how their PostgreSQL or SQL Server instances connect to managed services, object storage, analytics platforms, and event systems. The integration surface area has expanded substantially beyond traditional client-server connections.

Cloud-managed databases — Amazon RDS, Aurora, Azure SQL Database, Google Cloud SQL, Azure Database for PostgreSQL — expose the same SQL interfaces as on-premises instances, but some integration mechanisms change. Direct filesystem access for COPY FROM is unavailable on managed instances, replaced by cloud-native alternatives. On Amazon RDS for PostgreSQL, aws_s3 extension handles S3-based COPY operations. On Azure SQL Database, BULK INSERT can reference Azure Blob Storage. These are not just convenience features — in a serverless or containerized pipeline architecture, there is no shared filesystem, and cloud-native bulk load paths become necessary.

PostgreSQL:

-- Load from S3 on Amazon RDS for PostgreSQL (requires aws_s3 extension)
SELECT aws_s3.table_import_from_s3(
    'staging.raw_events',
    'event_id, user_id, event_type, occurred_at',
    '(FORMAT CSV, HEADER true)',
    'my-data-bucket',
    'pipeline/events_20240115.csv',
    'us-east-1'
);

SQL Server:

-- Load from Azure Blob Storage using BULK INSERT with external data source
CREATE EXTERNAL DATA SOURCE azure_pipeline_storage
WITH (TYPE = BLOB_STORAGE,
      LOCATION = 'https://mystorageaccount.blob.core.windows.net/pipeline');

BULK INSERT staging.raw_events
FROM 'events_20240115.csv'
WITH (DATA_SOURCE     = 'azure_pipeline_storage',
      FIELDTERMINATOR = ',',
      ROWTERMINATOR   = '\n',
      FIRSTROW        = 2);

NOTIFY/LISTEN in PostgreSQL is a lightweight publish-subscribe mechanism built directly into the database. Applications can LISTEN on a named channel and receive notifications sent by NOTIFY — either from application code or from triggers. This is useful for event-driven architectures where downstream consumers need to react to database changes without polling. A common pattern is a trigger that fires a NOTIFY when a new job row is inserted into a work queue table, waking up a worker process immediately rather than waiting for the next polling interval.

The mechanism is not designed for high-throughput streaming — it does not queue missed notifications if no client is listening, and the payload size is limited. For high-volume event streams, Debezium-to-Kafka or a proper message queue is the right architecture. But for low-volume coordination signals, NOTIFY/LISTEN integrates directly with the database transaction and requires no external infrastructure.

SQL Server's equivalent for application event notification is Service Broker, an asynchronous messaging system built into the engine. Service Broker supports reliable message queuing, conversation-based messaging, and activation (stored procedures that fire automatically when messages arrive on a queue). It is more powerful than PostgreSQL's NOTIFY/LISTEN but also substantially more complex to set up and operate. In practice, many teams bypass Service Broker in favor of external message queues (RabbitMQ, Azure Service Bus, Amazon SQS) that integrate with both their database and non-database components.

ODBC and JDBC as integration bridges remain relevant for tools that need a generic database connection: reporting tools like Tableau, Power BI, and Looker; ETL platforms like Apache Spark, dbt, and Talend; and data science environments like R and Python's pandas. Both PostgreSQL and SQL Server have ODBC and JDBC drivers that work reliably with this ecosystem. The DBA's role is to ensure that the credentials used by these tools have the minimum permissions necessary, that read-heavy reporting tools are pointed at replicas rather than primaries, and that long-running analytical queries are managed with statement timeouts so they cannot monopolize resources.

pg_stat_statements in PostgreSQL and Query Store in SQL Server are your primary tools for identifying which integration endpoints are responsible for expensive queries. When a reporting tool generates an unexpected full table scan at 2 AM, these tools tell you the exact query and when it ran.

Handling Schema Changes Across Integration Boundaries

Schema changes — adding columns, renaming tables, changing data types — are unavoidable in evolving systems, and they create one of the most fragile points in database integration. A pipeline that selects SELECT * from a table will silently return different data after a column is added. An ETL job that inserts by column position will fail immediately after a column is added in the middle. Downstream systems that depend on a specific column name will break when that column is renamed. Managing schema evolution across integration boundaries is one of the more underappreciated responsibilities in database operations.

Explicit column lists in all integration queries are non-negotiable in production. Every SELECT, INSERT, COPY, and MERGE statement in an integration should name columns explicitly. This makes queries robust to column additions and exposes column removals as explicit errors rather than silent data corruption.

Version-controlled schema migrations — using tools like Flyway, Liquibase, Alembic, or sqitch — create a documented history of schema changes and allow changes to be replayed on multiple environments. For PostgreSQL, these tools typically connect via standard JDBC/psycopg2 and execute migration scripts in transaction blocks, with a metadata table tracking which migrations have been applied. For SQL Server, the same tools work via JDBC or ODBC.

The operational challenge with schema migrations in integrated systems is coordination. If a column rename requires updating the ETL pipeline, the reporting tool, the application code, and the database simultaneously, deployment becomes a synchronized multi-system operation. The more practical pattern is an expand/contract approach: first add the new column alongside the old one and write to both (expand), update downstream consumers to use the new column, then drop the old column once no consumer references it (contract). This decouples the database migration from downstream deployment timelines.

Tracking which external systems reference which database objects is a harder problem. Most organizations solve this imperfectly with documentation or by centralizing access through views — views act as a stable public interface, and the underlying tables can be restructured without breaking consumers as long as the view's output schema remains compatible. This is a design pattern worth establishing early in any system with multiple integration points.

Key Takeaways

  • → Connection poolers like PgBouncer are essential for PostgreSQL in production; always use transaction pooling mode for OLTP workloads, but understand the constraints it places on session-level features like prepared statements and advisory locks.
  • → Foreign data wrappers (PostgreSQL) and linked servers (SQL Server) enable cross-database querying, but they are operational tools — building application logic that depends on them at high frequency creates reliability and performance risks that are difficult to manage.
