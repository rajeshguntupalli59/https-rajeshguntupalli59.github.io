Chapter 26: Data Sovereignty and Residency Compliance

Data sovereignty — the principle that data is subject to the laws and governance structures of the nation where it physically resides — has transformed from a legal abstraction into an operational constraint that every cloud DBA must architect around. GDPR in the European Union, the Personal Data Protection Bill in India, LGPD in Brazil, PIPL in China, and sector-specific mandates like HIPAA and FedRAMP in the United States all impose requirements that translate directly into database placement decisions, replication topology restrictions, encryption key custody, and audit logging configurations. If you've spent your career managing on-premises databases, you controlled data residency by controlling which rack the server sat in. In the cloud, that control is exercised through region selection, availability zone pinning, replication constraints, and — critically — understanding what the managed service does automatically on your behalf that you may not have realized. This chapter maps the compliance landscape to the actual configuration knobs available in AWS RDS, Azure SQL, and Google Cloud SQL, and shows you how to enforce residency guarantees that will survive a regulator's audit.

---

26.1 Understanding the Regulatory Landscape for Cloud DBAs

Before configuring anything, you need to understand what regulators actually care about, because the mandates are more specific — and more operationally demanding — than most DBAs expect until they're sitting across from an auditor.

What regulators want to know:

  • → Where does the data physically reside at rest?
  • → Where does the data travel during replication, backup, and recovery operations?
  • → Who holds the encryption keys, and where are those keys stored?
  • → Who can access the data — including cloud vendor employees?
  • → Can you prove all of the above with durable audit logs?

    • GDPR Article 44 prohibits transfers of personal data to third countries unless specific adequacy decisions or contractual protections are in place. This applies to your database backups. AWS automated backups for RDS are stored in S3 within the same region by default, but cross-region automated backup copies, Read Replicas in other regions, and cross-region snapshot copies are all "data transfers" under a strict GDPR reading. The fact that the transfer is automated and invisible doesn't make it compliant by default — it makes it a trap.

    The sovereignty matrix for cloud DBAs:

    | Requirement | On-Prem Equivalent | Cloud Mechanism |

    |---|---|---|

    | Data at rest in country X | Server physically in country X | Region selection + no cross-region replication |

    | Encryption key custody | HSM in your data center | Customer-managed keys (CMK) with regional KMS |

    | Access logging | Audit plugin, pgaudit | CloudTrail, Azure Audit Logs, Cloud Audit Logs |

    | Vendor employee access restriction | Physical security, OS ACLs | Nitro enclaves, Confidential Computing, access controls |

    | Backup residency | Tape library location | Backup region configuration, cross-region copy policies |

    The managed service assumption problem:

    When you self-managed PostgreSQL, you decided explicitly where every file went. RDS, Azure SQL, and Cloud SQL all make decisions on your behalf — automated backups, transaction log backups, minor version patching, and parameter group replication all involve data movement and access that happens without a specific operator action. Your job as a cloud DBA is to audit every one of those automatic behaviors against your compliance requirements before you're in production.

    A common shock: AWS RDS Multi-AZ with synchronous replication stays within a single region, but the two AZs may be in different cities. For most regulations, this is fine — they're within the same legal jurisdiction. But some financial regulators in specific countries have interpreted "data residency" down to the metropolitan area. Know your specific regulatory language before assuming Multi-AZ is compliant.

    26.2 AWS RDS and Aurora: Region, Replication, and Key Custody

    AWS gives you fine-grained control over data residency, but you must exercise that control explicitly. The defaults are permissive, not restrictive.

    Region selection and subnet architecture:

    Every RDS or Aurora instance is deployed into a VPC subnet, which lives in a specific Availability Zone within a specific region. The region is the top-level residency boundary. For EU data subject to GDPR, eu-west-1 (Ireland), eu-central-1 (Frankfurt), eu-south-1 (Milan), and eu-north-1 (Stockholm) are all within the EU, but they are different legal jurisdictions for some regulatory interpretations. German financial regulators and French data authorities have historically preferred data to remain within their national borders — not just the EU broadly.

    # Create an RDS PostgreSQL instance explicitly pinned to eu-central-1 (Frankfurt)
# and document the region choice in tags for audit purposes

aws rds create-db-instance \
  --db-instance-identifier prod-gdpr-postgres \
  --db-instance-class db.r6g.2xlarge \
  --engine postgres \
  --engine-version 15.4 \
  --master-username dbadmin \
  --master-user-password "$(aws secretsmanager get-secret-value \
    --secret-id prod/postgres/masterpassword \
    --region eu-central-1 \
    --query SecretString \
    --output text)" \
  --db-subnet-group-name prod-eu-central-private-subnet-group \
  --vpc-security-group-ids sg-0a1b2c3d4e5f67890 \
  --storage-encrypted \
  --kms-key-id arn:aws:kms:eu-central-1:123456789012:key/mrk-abc123 \
  --backup-retention-period 7 \
  --preferred-backup-window "02:00-03:00" \
  --no-publicly-accessible \
  --region eu-central-1 \
  --tags \
    Key=DataResidency,Value=EU-Germany \
    Key=ComplianceFramework,Value=GDPR \
    Key=DataClassification,Value=PersonalData \
    Key=RegionLocked,Value=true

    Customer-managed keys and key residency:

    This is where most DBAs underestimate the complexity. AWS KMS regional keys are region-specific — a key created in eu-central-1 cannot be used to encrypt data in us-east-1. This is good for residency, because it forces you to use a region-local key for your data. But AWS KMS Multi-Region Keys (MRKs) can replicate key material across regions, which is a compliance problem if your regulation requires key material to remain in-country.

    For the strictest sovereignty requirements, use AWS CloudHSM deployed in your target region, then configure KMS to use CloudHSM as its custom key store. Key material never leaves the HSM, and the HSM is physically in a specific AWS data center within your region.

    # Verify that an existing RDS instance is using a CMK (not AWS-managed key)
# and confirm the key is regional, not a multi-region key

aws rds describe-db-instances \
  --db-instance-identifier prod-gdpr-postgres \
  --region eu-central-1 \
  --query 'DBInstances[0].{
    KmsKeyId: KmsKeyId,
    StorageEncrypted: StorageEncrypted,
    DBInstanceArn: DBInstanceArn
  }'

# Check if the KMS key is a multi-region key (MultiRegion: true would be a compliance flag)
aws kms describe-key \
  --key-id arn:aws:kms:eu-central-1:123456789012:key/mrk-abc123 \
  --region eu-central-1 \
  --query 'KeyMetadata.{
    KeyId: KeyId,
    MultiRegion: MultiRegion,
    KeyState: KeyState,
    KeyManager: KeyManager,
    Origin: Origin
  }'
# Origin: AWS_KMS = key material in KMS
# Origin: EXTERNAL = you imported key material (highest control)
# Origin: AWS_CLOUDHSM = key material in CloudHSM (strongest for compliance)

    Preventing cross-region replication and backup copies:

    The most dangerous residency violation is an automated process that no one remembers setting up. Use AWS Service Control Policies (SCPs) at the AWS Organizations level to deny cross-region RDS actions for accounts handling sovereign data.

    # SCP to prevent cross-region RDS snapshot copies OUT of eu-central-1
# Apply this to the OU containing GDPR-scoped accounts
# Save as gdpr-rds-residency-scp.json and attach via Organizations

cat > gdpr-rds-residency-scp.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyRDSCrossRegionSnapshotCopy",
      "Effect": "Deny",
      "Action": [
        "rds:CopyDBSnapshot",
        "rds:CopyDBClusterSnapshot"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": "eu-central-1"
        }
      }
    },
    {
      "Sid": "DenyRDSCrossRegionReadReplica",
      "Effect": "Deny",
      "Action": [
        "rds:CreateDBInstanceReadReplica"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": "eu-central-1"
        }
      }
    },
    {
      "Sid": "DenyRDSGlobalCluster",
      "Effect": "Deny",
      "Action": [
        "rds:CreateGlobalCluster",
        "rds:AddMemberToGlobalCluster"
      ],
      "Resource": "*"
    }
  ]
}
EOF

# Attach to the target organizational unit
aws organizations create-policy \
  --name "GDPR-RDS-Residency-Enforcement" \
  --description "Prevents RDS data from leaving eu-central-1" \
  --content file://gdpr-rds-residency-scp.json \
  --type SERVICE_CONTROL_POLICY

aws organizations attach-policy \
  --policy-id p-examplepolicyid \
  --target-id ou-exampleouid

    Aurora Global Database and sovereignty:

    Aurora Global Database provides sub-second replication across regions for disaster recovery — which is architecturally attractive but potentially legally problematic. If you operate a Global Database with a primary in eu-central-1 and a secondary in us-east-1, you have EU personal data replicating to US servers. This may violate GDPR without appropriate data transfer mechanisms (Standard Contractual Clauses, Binding Corporate Rules) in place. The SCP above prevents this at the organizational level. If you need cross-region DR within compliant boundaries (e.g., eu-central-1 to eu-west-1), that's permissible for most EU regulations — but document it explicitly with your Data Protection Officer.

    26.3 Azure SQL: Geo-Replication, Paired Regions, and Data Boundary

    Microsoft has invested heavily in its Azure Data Boundary commitment — a contractual and technical guarantee that customer data stored and processed in the EU stays within the EU. For cloud DBAs, this is useful, but it doesn't eliminate the need for explicit configuration.

    Azure SQL geo-redundant backup storage:

    When you create an Azure SQL Database, the default backup storage redundancy is Geo — meaning your backups are replicated to the Azure paired region automatically. For West Europe (Netherlands), the paired region is North Europe (Ireland). Both are in the EU, so GDPR compliance is maintained for EU data subjects. But if you're in Brazil South, the paired region is Brazil Southeast. Both are in Brazil, satisfying LGPD. Know your paired regions before accepting the default.

    For regulations that prohibit any cross-location replication (some government cloud mandates), set backup storage redundancy to Zone or Local:

    # Create Azure SQL Database with zone-redundant backup storage
# (no cross-region backup replication)

az sql db create \
  --resource-group rg-prod-eu \
  --server sql-prod-westeurope \
  --name gdpr-production-db \
  --edition GeneralPurpose \
  --compute-model Provisioned \
  --family Gen5 \
  --capacity 8 \
  --zone-redundant true \
  --backup-storage-redundancy Zone \
  --location westeurope

# Verify the backup storage redundancy setting
az sql db show \
  --resource-group rg-prod-eu \
  --server sql-prod-westeurope \
  --name gdpr-production-db \
  --query '{
    name: name,
    location: location,
    backupStorageRedundancy: currentBackupStorageRedundancy,
    zoneRedundant: zoneRedundant
  }'

    Azure SQL Always Encrypted for column-level sovereignty:

    Always Encrypted is the Azure SQL equivalent of a technical guarantee that the cloud provider — including Microsoft employees — cannot read sensitive column values. Encryption and decryption happen exclusively in the client application. The database engine stores and retrieves ciphertext and never sees plaintext values. For regulated industries where vendor access is the residency concern (not geographic placement), this is a powerful tool.

    Always Encrypted uses a two-tier key hierarchy: Column Master Keys (CMK) stored in Azure Key Vault, and Column Encryption Keys (CEK) encrypted with the CMK. Your Key Vault can be configured with HSM-backed keys for the strongest custody guarantee.

    -- Configure Always Encrypted on Azure SQL for a PII table
-- This runs in SSMS or Azure Data Studio with the Always Encrypted wizard
-- or can be scripted via PowerShell/az CLI

-- Step 1: Create the Column Master Key reference in the database
-- (actual key material is in Azure Key Vault, not the database)
CREATE COLUMN MASTER KEY CMK_AzureKeyVault
WITH (
    KEY_STORE_PROVIDER_NAME = 'AZURE_KEY_VAULT',
    KEY_PATH = 'https://prod-keyvault-eu.vault.azure.net/keys/AlwaysEncryptedCMK/abc123def456'
);

-- Step 2: Create the Column Encryption Key, encrypted with the CMK
CREATE COLUMN ENCRYPTION KEY CEK_PersonalData
WITH VALUES (
    COLUMN_MASTER_KEY = CMK_AzureKeyVault,
    ALGORITHM = 'RSA_OAEP',
    ENCRYPTED_VALUE = 0x016E...  -- Generated by client tooling with access to Key Vault
);

-- Step 3: Create the table with encrypted columns
-- Deterministic encryption: allows equality comparisons (for WHERE clauses)
-- Randomized encryption: stronger, but no WHERE/ORDER BY/GROUP BY on the column
CREATE TABLE dbo.CustomerPersonalData (
    CustomerID      INT IDENTITY(1,1) PRIMARY KEY,
    FullName        NVARCHAR(200) COLLATE Latin1_General_BIN2
                    ENCRYPTED WITH (
                        COLUMN_ENCRYPTION_KEY = CEK_PersonalData,
                        ENCRYPTION_TYPE = Deterministic,
                        ALGORITHM = 'AEAD_AES_256_CBC_HMAC_SHA_256'
                    ) NOT NULL,
    NationalID      NVARCHAR(20) COLLATE Latin1_General_BIN2
                    ENCRYPTED WITH (
                        COLUMN_ENCRYPTION_KEY = CEK_PersonalData,
                        ENCRYPTION_TYPE = Randomized,
                        ALGORITHM = 'AEAD_AES_256_CBC_HMAC_SHA_256'
                    ) NULL,
    DateOfBirth     DATE
                    ENCRYPTED WITH (
                        COLUMN_ENCRYPTION_KEY = CEK_PersonalData,
                        ENCRYPTION_TYPE = Randomized,
                        ALGORITHM = 'AEAD_AES_256_CBC_HMAC_SHA_256'
                    ) NULL,
    CreatedAt       DATETIME2 NOT NULL DEFAULT SYSUTCDATETIME()
);

-- The database engine sees only ciphertext in FullName, NationalID, DateOfBirth
-- Azure support engineers with direct server access cannot read these values
-- SELECT * FROM dbo.CustomerPersonalData will return encrypted bytes to non-AE clients

    Azure Policy for residency enforcement:

    Like AWS SCPs, Azure Policy can enforce residency at the management group or subscription level, preventing resources from being created in non-compliant regions.

    # Create and assign an Azure Policy that restricts SQL resources to West Europe only
# Built-in policy: "Allowed locations" - policy definition ID varies by tenant

# First, get the built-in policy definition for "Allowed locations"
POLICY_DEF_ID=$(az policy definition list \
  --query "[?displayName=='Allowed locations'].id" \
  --output tsv)

# Assign the policy to a resource group, restricting to westeurope only
az policy assignment create \
  --name "sql-residency-westeurope" \
  --display-name "SQL Resources Must Stay in West Europe" \
  --scope "/subscriptions/sub-id-here/resourceGroups/rg-prod-eu" \
  --policy "$POLICY_DEF_ID" \
  --params '{
    "listOfAllowedLocations": {
      "value": ["westeurope"]
    }
  }' \
  --enforcement-mode Default

# Verify the assignment
az policy assignment show \
  --name "sql-residency-westeurope" \
  --scope "/subscriptions/sub-id-here/resourceGroups/rg-prod-eu" \
  --query '{
    name: name,
    enforcementMode: enforcementMode,
    parameters: parameters
  }'

    Active Geo-Replication and legal review:

    Azure SQL Active Geo-Replication lets you create up to four readable secondary databases in any region. Unlike AWS Global Clusters, this is configured per-database rather than per-cluster, giving you finer control. But the sovereignty risk is identical: verify that all secondary regions are within legally compliant jurisdictions before enabling it. For financial institutions in the EU operating under DORA (Digital Operational Resilience Act), cross-region replication may be required for resilience but must stay within approved geographic boundaries. DORA explicitly permits DR replicas in other EU member states.

    26.4 Google Cloud SQL, AlloyDB, and Spanner: Residency Controls

    Google Cloud's approach to data residency spans three products with significantly different residency capabilities.

    Cloud SQL: regional by design, with explicit controls:

    Cloud SQL instances are regional. Unlike Cloud Spanner, they don't span regions by default. Cross-region read replicas are opt-in, and automated backups stay in the same region as the instance. For most sovereignty requirements, Cloud SQL's defaults are residency-safe, but verify them explicitly.

    # Create a Cloud SQL PostgreSQL instance with explicit region pinning
# and verify residency configuration

gcloud sql instances create prod-gdpr-postgres \
  --database-version=POSTGRES_15 \
  --tier=db-custom-8-32768 \
  --region=europe-west3 \
  --availability-type=REGIONAL \
  --storage-type=SSD \
  --storage-size=500GB \
  --storage-auto-increase \
  --backup-start-time=02:00 \
  --enable-bin-log \
  --retained-backups-count=7 \
  --deletion-protection \
  --no-assign-ip \
  --network=projects/my-project/global/networks/prod-vpc \
  --labels=data-residency=eu-germany,compliance=gdpr,data-classification=personal

# Verify the instance location and backup configuration
gcloud sql instances describe prod-gdpr-postgres \
  --format="yaml(name,region,settings.backupConfiguration,settings.locationPreference)"

# List all replicas for this instance to verify none are cross-region
gcloud sql instances list \
  --filter="masterInstanceName:prod-gdpr-postgres" \
  --format="table(name,region,databaseVersion,state)"

    Assured Workloads for regulatory compliance:

    Google Cloud Assured Workloads is the platform-level mechanism for enforcing data residency and access controls for regulated workloads. It restricts which Google Cloud services and regions can be used within a folder, enforces organization policies, and — for sovereign cloud configurations — restricts Google personnel access to a subset of employees in specific jurisdictions.

    For EU sovereign requirements, the "EU Regions and Support" configuration restricts your folder to EU regions and ensures that Google support staff who can access your data are located in the EU. This addresses the vendor employee access concern that regulators increasingly scrutinize.

    # Enable Assured Workloads folder for EU sovereign configuration
# This requires Organization Admin permissions

gcloud assured workloads create \
  --organization=123456789 \
  --location=europe-west3 \
  --display-name="GDPR-Production-Workload" \
  --compliance-regime=EU_REGIONS_AND_SUPPORT \
  --billing-account=billingAccounts/ABCDEF-123456-GHIJKL \
  --resource-settings='[
    {
      "resourceType": "CONSUMER_PROJECT",
      "displayName": "gdpr-prod-project"
    }
  ]'

# After creation, verify the organization policies applied to the folder
gcloud resource-manager org-policies list \
  --folder=FOLDER_ID_FROM_ABOVE \
  --format="table(constraint,spec.rules[0].enforce)"
# Look for:
# constraints/gcp.resourceLocations - restricts to EU regions
# constraints/iam.allowedPolicyMemberDomains - restricts IAM membership
# constraints/gcp.restrictServiceUsage - restricts to approved services

    Cloud Spanner and the multi-region trap:

    Spanner's multi-region configurations are architecturally elegant for global applications but are a sovereignty minefield. A nam6 (North America multi-region) configuration spans multiple US states. A eur3 configuration spans Netherlands and Belgium — both EU member states, so GDPR-compliant. But global configurations span multiple continents, and asia1 spans Tokyo and Osaka.

    The key residency risk with Spanner is the witness replica. In a multi-region configuration, Spanner places leader replicas in two regions and witness replicas in a third. The witness replica location is fixed per configuration and may not be in your desired jurisdiction. The eur3 configuration places witnesses in Belgium with leaders in Netherlands — acceptable for EU rules. Before choosing any multi-region Spanner configuration, verify every replica placement against your regulatory requirements, not just the primary region.

    If you need single-region Spanner for strict sovereignty, use a regional instance — you sacrifice global reads and multi-region writes but gain unambiguous data residency.

    # Create a single-region Spanner instance for strict residency
# (regional, not multi-region - no data leaves europe-west3)

gcloud spanner instances create prod-sovereign-spanner \
  --config=regional-europe-west3 \
  --description="GDPR-compliant single-region Spanner instance" \
  --nodes=3 \
  --project=gdpr-production-project

# Verify the instance configuration - inspect which regions are involved
gcloud spanner instance-configs describe regional-europe-west3 \
  --format="yaml(displayName,replicas)"

# List all databases on this instance
gcloud spanner databases list \
  --instance=prod-sovereign-spanner \
  --format="table(name,state,encryptionInfo.encryptionType)"

# Create a database with customer-managed encryption
gcloud spanner databases create customer-data \
  --instance=prod-sovereign-spanner \
  --kms-key=projects/gdpr-production-project/locations/europe-west3/keyRings/spanner-keyring/cryptoKeys/spanner-key

    26.5 Audit Logging for Residency Compliance

    Regulatory frameworks don't just require that you implement residency controls — they require that you prove those controls are in effect and working. Audit logging for sovereignty compliance is different from performance monitoring or application debugging. It must capture specific events: data access, schema changes, privilege escalations, failed access attempts, and — for the highest-sensitivity requirements — every query that touches regulated columns.
