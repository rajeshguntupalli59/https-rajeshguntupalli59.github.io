Chapter 31: IAM-Based Database Authentication

Cloud-managed databases fundamentally change how you think about authentication. The pg_hba.conf entries you've maintained for years, the SQL Server logins tied to Windows accounts, the service accounts rotated manually every quarter — all of that shifts toward a unified identity model built on the cloud provider's IAM plane. IAM-based database authentication means the database validates identity through short-lived tokens issued by the cloud provider rather than long-lived passwords stored in the database itself. This chapter walks through how AWS RDS, Azure SQL, and Google Cloud SQL implement IAM authentication, how to configure it for production workloads, and where the sharp edges are. The goal is not to replace your understanding of database security — it is to map that knowledge onto the identity primitives each cloud provider offers.

---

31.1 Why IAM Authentication Changes the Security Model

Traditional database authentication operates on a shared-secret model. A password is created, stored in the database, distributed to the application, and periodically rotated. Each step introduces risk: passwords end up in configuration files, rotation breaks connections, and auditing who authenticated when requires database-level logging that many teams never configure properly.

IAM authentication flips this. Instead of a password, a client requests a short-lived token from the IAM service — typically valid for 15 minutes — and presents that token as the password to the database. The database validates the token against the IAM service in real time. No static secret leaves the IAM boundary. When the token expires, it cannot be reused. Rotation happens automatically because tokens are never stored.

The analogy to your existing knowledge: if pg_hba.conf is the gatekeeper that says "this IP range may connect with md5 authentication," then IAM roles are the gatekeeper that says "this EC2 instance profile or this Kubernetes service account may connect with a token it obtained from the IAM service." The database still has user accounts — a PostgreSQL role or a SQL Server login — but the credential used to prove identity is ephemeral and cloud-managed.

The operational implications are significant. You no longer rotate passwords; you manage which IAM principals have permission to generate tokens. Audit trails move from database logs into CloudTrail, Azure Activity Log, or Cloud Audit Logs — platforms with retention, alerting, and SIEM integration that most teams already operate. The attack surface of stolen credentials shrinks because a token captured in transit is useless after 15 minutes.

The tradeoffs are real. Applications that open connections slowly or use connection pools with long-lived connections need to handle token renewal. Not every database engine on every cloud platform supports IAM authentication. Some configurations require specific parameter groups or engine versions. You need to understand these constraints before committing to IAM auth in production.

31.2 IAM Authentication on AWS RDS and Aurora

AWS supports IAM database authentication for RDS MySQL, RDS PostgreSQL, Aurora MySQL, and Aurora PostgreSQL. It is not available for RDS SQL Server, RDS Oracle, or RDS MariaDB. The mechanism is the same across supported engines: the application calls generate-db-auth-token to get a signed token, then passes that token as the password when connecting. The token is valid for 15 minutes.

Enabling IAM Authentication

IAM authentication is an instance-level setting, not a database-level setting. You enable it when creating or modifying an instance.

# Enable IAM authentication on an existing RDS PostgreSQL instance
aws rds modify-db-instance \
  --db-instance-identifier prod-postgres-01 \
  --enable-iam-database-authentication \
  --apply-immediately

# Verify the setting
aws rds describe-db-instances \
  --db-instance-identifier prod-postgres-01 \
  --query 'DBInstances[0].IAMDatabaseAuthenticationEnabled'

Enabling IAM auth does not disable password authentication. Existing users can still connect with passwords. You enable IAM auth at the instance level, then selectively grant the rds_iam role to specific database users. This lets you migrate gradually.

Creating the Database User

On PostgreSQL, you create a regular role and grant it the rds_iam group. The role must not have a password — or more precisely, the password is irrelevant because it will never be used for IAM-authenticated connections.

-- On RDS PostgreSQL or Aurora PostgreSQL
-- Create an application user that authenticates via IAM
CREATE USER app_service WITH LOGIN;
GRANT rds_iam TO app_service;

-- Grant necessary schema privileges
GRANT CONNECT ON DATABASE production TO app_service;
GRANT USAGE ON SCHEMA public TO app_service;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_service;

-- For future tables created in the schema
ALTER DEFAULT PRIVILEGES IN SCHEMA public 
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_service;

On RDS MySQL and Aurora MySQL, the syntax is slightly different:

-- On RDS MySQL or Aurora MySQL
CREATE USER 'app_service'@'%' IDENTIFIED WITH AWSAuthenticationPlugin AS 'RDS';
GRANT SELECT, INSERT, UPDATE, DELETE ON production.* TO 'app_service'@'%';

IAM Policy for Token Generation

The IAM principal that needs database access — an EC2 instance role, an ECS task role, a Lambda execution role, a Kubernetes service account with IRSA — needs an IAM policy that allows the rds-db:connect action. The resource ARN encodes the database instance, region, account, and database username.

# Get the resource ID needed for the IAM policy (it's NOT the instance identifier)
aws rds describe-db-instances \
  --db-instance-identifier prod-postgres-01 \
  --query 'DBInstances[0].DbiResourceId'
# Returns something like: db-ABCDEFGHIJKLMNOPQRSTUVWXYZ

The IAM policy:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "rds-db:connect",
      "Resource": "arn:aws:rds-db:us-east-1:123456789012:dbuser:db-ABCDEFGHIJKLMNOPQRSTUVWXYZ/app_service"
    }
  ]
}

The resource format is: arn:aws:rds-db:::dbuser:/. Using the DbiResourceId rather than the instance identifier is a common gotcha — the IAM policy silently allows the action but the token will not work if you use the wrong resource identifier.

Generating and Using the Auth Token

import boto3
import psycopg2
import ssl

def get_connection():
    rds_client = boto3.client('rds', region_name='us-east-1')
    
    # Token is valid for 15 minutes — generate it close to connection time
    token = rds_client.generate_db_auth_token(
        DBHostname='prod-postgres-01.abc123.us-east-1.rds.amazonaws.com',
        Port=5432,
        DBUsername='app_service',
        Region='us-east-1'
    )
    
    # SSL is required for IAM authentication — the token is useless without it
    # Download the RDS CA bundle: https://truststore.pki.rds.amazonaws.com/
    ssl_context = ssl.create_default_context()
    ssl_context.load_verify_locations('us-east-1-bundle.pem')
    
    conn = psycopg2.connect(
        host='prod-postgres-01.abc123.us-east-1.rds.amazonaws.com',
        port=5432,
        database='production',
        user='app_service',
        password=token,
        sslmode='verify-full',
        sslrootcert='us-east-1-bundle.pem'
    )
    
    return conn

SSL is mandatory, not optional. AWS validates the token only over encrypted connections. If your application currently connects with sslmode=disable or sslmode=allow, IAM authentication will not work regardless of how correctly you've configured everything else.

IAM Auth with RDS Proxy

RDS Proxy adds an important capability: it handles token renewal on behalf of the application. When using RDS Proxy with IAM authentication, the application authenticates to the Proxy using a token, but the Proxy manages the connection pool to the underlying RDS instance using whatever credential model you've configured. This is the recommended pattern for Lambda and ECS applications that open many short-lived connections.

# Create an RDS Proxy with IAM authentication enabled
aws rds create-db-proxy \
  --db-proxy-name prod-postgres-proxy \
  --engine-family POSTGRESQL \
  --auth '[{
    "AuthScheme": "SECRETS",
    "IAMAuth": "REQUIRED",
    "SecretArn": "arn:aws:secretsmanager:us-east-1:123456789012:secret:rds-proxy-secret"
  }]' \
  --role-arn arn:aws:iam::123456789012:role/rds-proxy-role \
  --vpc-subnet-ids subnet-abc123 subnet-def456 \
  --vpc-security-group-ids sg-789xyz \
  --require-tls

With IAMAuth: REQUIRED on the proxy, every connection to the proxy endpoint must use a valid IAM token. The proxy handles the actual database connection internally.

31.3 Azure SQL and Entra ID (Azure Active Directory) Authentication

Azure SQL calls its IAM authentication "Microsoft Entra ID authentication" (formerly Azure Active Directory authentication). The architecture differs from AWS: rather than generating a token manually, the application obtains an OAuth 2.0 access token from the Entra ID endpoint and uses it as the password for SQL connections. The token is bearer-based, valid for approximately one hour.

Azure SQL supports Entra ID authentication for Azure SQL Database, Azure SQL Managed Instance, and Azure Synapse Analytics. Unlike AWS, where password and IAM auth coexist on the same user, Azure SQL can enforce Entra-only authentication at the server level, blocking all SQL authentication logins.

Configuring Entra ID Authentication

# Set an Entra ID admin for the Azure SQL server
# This is a prerequisite — Entra ID auth requires an Entra admin be set
az sql server ad-admin create \
  --resource-group prod-rg \
  --server prod-sql-server \
  --display-name "DBA-Admin-Group" \
  --object-id "a1b2c3d4-e5f6-7890-abcd-ef1234567890"

# Optionally enforce Entra-only authentication (disables SQL logins entirely)
az sql server ad-only-auth enable \
  --resource-group prod-rg \
  --name prod-sql-server

# Verify the setting
az sql server ad-only-auth get \
  --resource-group prod-rg \
  --name prod-sql-server

The ad-only-auth enable command is a one-way operation in terms of impact — it immediately breaks any application using SQL authentication. Test this in staging with realistic connection patterns before applying to production. The master admin password continues to work even with Entra-only auth enabled because Azure maintains an emergency bypass for server administration, but you cannot create new SQL logins or reset passwords for existing ones.

Creating Entra ID Users in the Database

-- Connect to the Azure SQL database as the Entra ID admin
-- Create a contained database user mapped to an Entra ID service principal
-- (No password is specified — authentication happens at the Entra level)

-- For an Entra ID group
CREATE USER [platform-data-app-group] FROM EXTERNAL PROVIDER;
ALTER ROLE db_datareader ADD MEMBER [platform-data-app-group];
ALTER ROLE db_datawriter ADD MEMBER [platform-data-app-group];

-- For a specific managed identity (application service principal)
CREATE USER [prod-app-service-identity] FROM EXTERNAL PROVIDER;
ALTER ROLE db_datareader ADD MEMBER [prod-app-service-identity];
ALTER ROLE db_datawriter ADD MEMBER [prod-app-service-identity];

-- Check what external users exist in the database
SELECT name, type_desc, authentication_type_desc
FROM sys.database_principals
WHERE authentication_type_desc = 'EXTERNAL';

The FROM EXTERNAL PROVIDER clause is the Azure SQL equivalent of creating a Windows-authenticated login in SQL Server on-premises. If you've managed SQL Server in a domain environment, this will feel familiar — except you're mapping to Entra ID identities instead of Active Directory accounts.

Acquiring Tokens and Connecting from Applications

from azure.identity import DefaultAzureCredential, ManagedIdentityCredential
import pyodbc
import struct

def get_azure_sql_connection():
    # DefaultAzureCredential tries: environment variables, managed identity,
    # Azure CLI credentials, Visual Studio Code — in that order.
    # In production on Azure (App Service, AKS, VM), this resolves to managed identity.
    credential = DefaultAzureCredential()
    
    # Request a token for Azure SQL Database
    token = credential.get_token("https://database.windows.net/.default")
    
    # pyodbc requires the token in a specific binary format
    # This is a known gotcha — the raw token string does not work directly
    token_bytes = token.token.encode("UTF-16-LE")
    token_struct = struct.pack(f'<I{len(token_bytes)}s', len(token_bytes), token_bytes)
    
    SQL_COPT_SS_ACCESS_TOKEN = 1256  # defined in msodbcsql.h
    
    conn = pyodbc.connect(
        "Driver={ODBC Driver 18 for SQL Server};"
        "Server=prod-sql-server.database.windows.net,1433;"
        "Database=production;"
        "Encrypt=yes;"
        "TrustServerCertificate=no;",
        attrs_before={SQL_COPT_SS_ACCESS_TOKEN: token_struct}
    )
    
    return conn

The struct.pack step is the most common source of errors when first implementing this. The ODBC driver expects the token encoded as UTF-16-LE with a 4-byte little-endian length prefix. Passing the token as a plain string in the connection password field does not work with pyodbc; it works only with certain JDBC drivers that handle token injection differently.

System-Assigned vs. User-Assigned Managed Identity

For Azure resources connecting to Azure SQL, managed identities are the standard pattern. A system-assigned managed identity is created with the resource and deleted when the resource is deleted. A user-assigned managed identity is a standalone Azure resource you can assign to multiple services — useful when you want multiple App Service instances to share a single database identity without maintaining separate database users for each.

# Create a user-assigned managed identity
az identity create \
  --name prod-app-db-identity \
  --resource-group prod-rg \
  --location eastus

# Assign it to an App Service
az webapp identity assign \
  --resource-group prod-rg \
  --name prod-web-app \
  --identities /subscriptions/sub-id/resourceGroups/prod-rg/providers/Microsoft.ManagedIdentity/userAssignedIdentities/prod-app-db-identity

# Get the client ID — needed when explicitly requesting a token for a specific identity
az identity show \
  --name prod-app-db-identity \
  --resource-group prod-rg \
  --query clientId

When a resource has multiple managed identities assigned, DefaultAzureCredential may not select the right one. In that case, use ManagedIdentityCredential(client_id="") explicitly to target the correct identity.

31.4 Google Cloud SQL and AlloyDB IAM Authentication

Google Cloud SQL supports IAM database authentication for PostgreSQL instances and MySQL instances. AlloyDB for PostgreSQL supports IAM authentication as well. The implementation on Cloud SQL uses the Cloud SQL Auth Proxy or the direct IAM authentication feature added in later versions; both ultimately validate against the Google Cloud IAM service.

Enabling IAM Authentication on Cloud SQL

# Create a Cloud SQL PostgreSQL instance with IAM authentication enabled
gcloud sql instances create prod-postgres-01 \
  --database-version=POSTGRES_15 \
  --tier=db-custom-8-32768 \
  --region=us-central1 \
  --availability-type=REGIONAL \
  --storage-type=SSD \
  --storage-size=500GB \
  --storage-auto-increase \
  --database-flags=cloudsql.iam_authentication=on

# Enable IAM authentication on an existing instance
gcloud sql instances patch prod-postgres-01 \
  --database-flags=cloudsql.iam_authentication=on

# Verify
gcloud sql instances describe prod-postgres-01 \
  --format="json(settings.databaseFlags)"

Adding IAM Users

On Cloud SQL, IAM users are added at the instance level through the Cloud SQL API, which automatically creates the corresponding PostgreSQL role. You do not create the role manually in psql.

# Add a service account as an IAM database user
# The username in PostgreSQL will be the service account email truncated at the @ symbol
gcloud sql users create prod-app-sa@project-id.iam.gserviceaccount.com \
  --instance=prod-postgres-01 \
  --type=CLOUD_IAM_SERVICE_ACCOUNT

# Add a human user (Google account)
gcloud sql users create rajesh@company.com \
  --instance=prod-postgres-01 \
  --type=CLOUD_IAM_USER

# List IAM users on the instance
gcloud sql users list --instance=prod-postgres-01

After adding the IAM user at the instance level, you still need to grant database-level privileges inside PostgreSQL. The service account email becomes a PostgreSQL role, but the PostgreSQL role name drops the domain suffix:

-- Connect to the database as the superuser
-- Grant privileges to the IAM service account role
-- Note: the role name is truncated — just the part before the @
GRANT CONNECT ON DATABASE production TO "prod-app-sa";
GRANT USAGE ON SCHEMA public TO "prod-app-sa";
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO "prod-app-sa";

-- The full email format is used for CLOUD_IAM_USER types
GRANT CONNECT ON DATABASE production TO "rajesh@company.com";

-- Verify the roles exist
SELECT rolname, rolcanlogin FROM pg_roles 
WHERE rolname IN ('prod-app-sa', 'rajesh@company.com');

Using the Cloud SQL Auth Proxy

The Cloud SQL Auth Proxy handles IAM authentication and SSL termination transparently. It runs as a sidecar process alongside your application and listens on a local port. Your application connects to 127.0.0.1:5432 as if connecting to a local PostgreSQL instance; the proxy handles authentication to Cloud SQL.

# Download and run the Cloud SQL Auth Proxy
# In Kubernetes, this runs as a sidecar container
curl -o cloud-sql-proxy \
  https://storage.googleapis.com/cloud-sql-connectors/cloud-sql-proxy/v2.7.0/cloud-sql-proxy.linux.amd64
chmod +x cloud-sql-proxy

# Run the proxy with IAM authentication
./cloud-sql-proxy \
  --auto-iam-authn \
  --port=5432 \
  project-id:us-central1:prod-postgres-01

# The application then connects to localhost:5432 with the IAM username
# No password needed — the proxy provides the credential
psql "host=127.0.0.1 port=5432 dbname=production user=prod-app-sa sslmode=disable"

The --auto-iam-authn flag is what enables IAM-based authentication through the proxy. Without it, the proxy provides SSL termination and authorized network access but still requires password authentication. When connecting through the proxy with IAM auth, you do not provide a password — the proxy inserts the token automatically.

Kubernetes Deployment Pattern

In GKE, the Cloud SQL Auth Proxy runs as a sidecar container within the application pod. The pod uses a Kubernetes service account annotated with Workload Identity to assume a Google service account.

# Create a Google service account for the application
gcloud iam service-accounts create prod-app-db-sa \
  --display-name="Production App Database SA"

# Grant it the Cloud SQL Client role
gcloud projects add-iam-policy-binding project-id \
  --member="serviceAccount:prod-app-db-sa@project-id.iam.gserviceaccount.com" \
  --role="roles/cloudsql.client"

# Allow the Kubernetes service account to impersonate the Google service account
gcloud iam service-accounts add-iam-policy-binding prod-app-db-sa@project-id.iam.gserviceaccount.com \
  --role="roles/iam.workloadIdentityUser" \
  --member="serviceAccount:project-id.svc.id.goog[prod-namespace/prod-app-ksa]"

# Annotate the Kubernetes service account
kubectl annotate serviceaccount prod-app-ksa \
  --namespace prod-namespace \
  iam.gke.io/gcp-service-account=prod-app-db-sa@project-id.iam.gserviceaccount.com

31.5 Connection Pooling and Token Renewal in Production

The 15-minute token lifetime on AWS and the approximately one-hour token lifetime on Azure and GCP create a challenge for connection pools. A connection pool that opens connections at startup and reuses them indefinitely will present expired tokens when it tries to open new connections after the token window has passed. The existing connections continue to work — IAM authentication validates the token at connection establishment, not on every query — but new connections will fail.

This distinction matters: IAM auth does not re-validate the token on every query. Once a connection is established, it stays open until closed or until the database server terminates it. The problem occurs when the pool needs to create a new connection, typically because the application load increases or an idle connection times out.

AWS with PgBouncer

PgBouncer in transaction or session pooling mode works with IAM authentication if you implement token rotation in the configuration. PgBouncer reads the password from its userlist.txt file; you need an external process to refresh that file with a new token before expiration.
