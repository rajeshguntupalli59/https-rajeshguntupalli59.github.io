Chapter 15: Azure SQL Hyperscale: When You Need It

Azure SQL Hyperscale is Microsoft's answer to a problem that on-prem DBAs know well: what happens when your database outgrows your storage architecture? Traditional SQL Server installations hit ceiling after ceiling — SAN I/O limits, filegroup management headaches, backup windows measured in hours, and the constant anxiety of a database that has grown beyond what your infrastructure was originally sized to handle. Azure SQL Hyperscale reimagines the storage layer entirely, decoupling compute from storage and introducing a distributed log service and page server architecture that lets you scale to 100 TB without a weekend maintenance window. This chapter covers when Hyperscale is the right choice, how it differs architecturally from standard Azure SQL Database tiers, how to provision and configure it correctly, what the production gotchas look like, and how to migrate into — and out of — it with your eyes open.

---

What Hyperscale Actually Is (And What It Isn't)

Before you provision a Hyperscale database, you need to understand that it is not simply "a bigger Azure SQL Database." It is a fundamentally different storage architecture that happens to present a familiar SQL Server surface. If you treat it like a standard Azure SQL tier with a higher storage cap, you will be surprised by behaviors that don't match your mental model.

The Architecture

In a traditional SQL Server instance — on-prem or in an Azure SQL General Purpose or Business Critical tier — your compute node owns the data files. Pages live on local or attached storage that the database engine reads and writes directly. Backups are full snapshots of that storage. Replica setup involves copying or synchronizing those files. Everything is tightly coupled.

Hyperscale decouples these layers. The architecture has four distinct components:

1. Compute replicas — These are your SQL Server engine instances. They hold the buffer pool (in-memory pages) and execute queries, but they do not own the data files.

2. Log service — A distributed, append-only transaction log service that sits between compute and storage. When your primary compute node commits a transaction, it writes to the log service first. This is where the acknowledged commit happens. Page servers and secondary replicas consume from this log asynchronously.

3. Page servers — Multiple independent page server processes that maintain portions of the database's page space and serve pages to compute nodes on demand. When a compute node needs a page not in its buffer pool, it fetches it from a page server over the network rather than from local disk. Each page server handles a shard of the database's page space, and they scale out as the database grows.

4. Azure Storage — The durable backing store underneath the page servers. This is Azure Premium Storage, providing the actual durability guarantee.

The practical consequence: your compute node's buffer pool is a cache in front of a distributed page store. A cold query that misses the buffer pool on a Hyperscale instance will be slower than the equivalent query on a Business Critical instance with local SSD, because it pays a network round-trip to a page server. On hot, well-cached workloads, Hyperscale performs comparably or better than other tiers for reads because you can add named replicas that each have their own independent buffer pool.

What It Is Designed For

Hyperscale makes sense when one or more of the following conditions apply:

  • → Your database exceeds 4 TB (the limit for General Purpose and Business Critical tiers)
  • → You need near-instant database creation and restore times regardless of database size
  • → Your backup and restore windows are a compliance or operational problem — a 50 TB Hyperscale database restores in roughly 5–10 minutes because Azure only needs to instantiate compute and point it at existing page server snapshots
  • → You have a mixed workload where you want to offload read-heavy reporting or analytics to a separate compute replica with its own memory and CPU, without the overhead of maintaining a full replica
  • → You have bursty storage growth patterns and don't want to over-provision

    • What It Is Not Designed For

    Hyperscale is not a general upgrade from Business Critical. Business Critical uses local SSD with in-memory OLTP support and has lower read latency for cold pages because storage is local. If your workload is primarily OLTP with a well-tuned buffer pool hit ratio above 98%, Business Critical will often outperform Hyperscale for raw query latency. Hyperscale also has restrictions that on-prem DBAs find surprising: as of the current GA release, you cannot change a Hyperscale database back to another service tier. That decision is one-directional. You migrate in, and if you need to migrate out, you do it through export/import or backup/restore — not through a service tier change.

    The no-downgrade restriction is not a technical limitation that might be lifted later. It reflects the fact that Hyperscale storage format is fundamentally different from the other tiers. Plan accordingly before you migrate production.

    Provisioning Hyperscale: Configuration That Matters

    When you create a Hyperscale database, the decisions that matter most are compute generation, vCore count, read replica topology, and backup redundancy. Storage is automatic and does not require sizing decisions the way you were accustomed to on-prem.

    Creating a Hyperscale Database with Azure CLI

    # Create a resource group and logical server first if not already present
az group create \
  --name rg-hyperscale-prod \
  --location eastus2

az sql server create \
  --name sql-hs-prod-eastus2 \
  --resource-group rg-hyperscale-prod \
  --location eastus2 \
  --admin-user sqladmin \
  --admin-password "$(az keyvault secret show \
    --vault-name kv-prod \
    --name sql-admin-password \
    --query value -o tsv)"

# Create the Hyperscale database
# Edition: Hyperscale
# Service objective: HS_Gen5_8 (Gen5, 8 vCores)
# High-availability replicas: 1 (adds a secondary compute replica for HA)
# Backup storage redundancy: Zone (zone-redundant backup storage)
az sql db create \
  --resource-group rg-hyperscale-prod \
  --server sql-hs-prod-eastus2 \
  --name OrdersDB \
  --edition Hyperscale \
  --service-objective HS_Gen5_8 \
  --ha-replicas 1 \
  --backup-storage-redundancy Zone \
  --read-scale Enabled

# Verify the database properties
az sql db show \
  --resource-group rg-hyperscale-prod \
  --server sql-hs-prod-eastus2 \
  --name OrdersDB \
  --query "{tier:sku.tier, size:sku.name, maxSize:maxSizeBytes, haReplicas:highAvailabilityReplicaCount}" \
  --output table

    Compute Tiers and Sizing

    Hyperscale compute uses the same vCore-based sizing as other Azure SQL tiers, but the service objective naming prefix is HS_. Common choices:

    | Service Objective | vCores | Memory | Notes |

    |---|---|---|---|

    | HS_Gen5_4 | 4 | 20.4 GB | Minimum recommended for production |

    | HS_Gen5_8 | 8 | 40.8 GB | Good starting point for moderate OLTP |

    | HS_Gen5_16 | 16 | 83 GB | High-concurrency workloads |

    | HS_Gen5_32 | 32 | 166 GB | Large in-memory working sets |

    | HS_Gen5_80 | 80 | 408 GB | Very large working sets, high throughput |

    There are also Fsv2 (compute-optimized) and M-series (memory-optimized) hardware generations available in select regions. The M-series configurations go up to 128 vCores and 3.7 TB of memory, which is territory that previously required dedicated hardware and significant capital expenditure.

    You can scale compute up or down online. Scaling to a different vCore count within the same hardware generation takes approximately 2–3 minutes with minimal disruption. Changing hardware generation (Gen5 to Fsv2, for example) takes longer and involves a reconfiguration event.

    Backup Storage Redundancy — Read This Before You Deploy

    The --backup-storage-redundancy flag matters more than it appears. The options are:

  • → `Local` — Backups stored in a single datacenter. Lowest cost, lowest durability. Not appropriate for most production workloads.
  • → `Zone` — Zone-redundant backup storage. Appropriate for single-region deployments that need AZ tolerance.
  • → `Geo` — Geo-redundant backup storage. Replicates backups to a paired region. Default for most Azure SQL tiers historically.
  • → `GeoZone` — Zone-redundant in primary region plus geo-replication. Highest durability.

    • The critical gotcha: you can only set backup storage redundancy when you create the Hyperscale database. You cannot change it afterward without creating a new database. If you provision with Local and later realize you need geo-redundant backups for your DR policy, you are rebuilding the database.

    Named Replicas: Read Scale-Out Without the Traditional Overhead

    One of the most operationally useful Hyperscale features is named replicas. A named replica is a Hyperscale compute node pointed at the same page servers as the primary. It sees a consistent, near-real-time view of the data (typically within 1–5 seconds of the primary, sometimes less). Critically, it is a fully independent Azure SQL Database resource with its own service tier, its own connection string, its own buffer pool, and its own independent security configuration.

    On-prem, you might have handled this with Always On readable secondaries. The fundamental difference is that a named replica in Hyperscale shares storage infrastructure with the primary — it is not maintaining a full copy of the data files. Adding a named replica takes minutes regardless of database size, because Azure is not copying 50 TB of data. It is instantiating a new compute node that can already read from the existing page servers.

    When Named Replicas Make Sense

    Named replicas are the right choice when:

  • → You have a large reporting or analytics workload that competes with OLTP on the primary
  • → You need to give a third-party team or application its own connection endpoint with no risk of their workload affecting production OLTP
  • → You need read scale-out during peak periods and want to spin replicas up and down dynamically

    • Named replicas are billed as separate Hyperscale databases with their own vCore cost. If you provision a named replica as HS_Gen5_16, you pay for those 16 vCores separately. For pure read offloading, you can often provision named replicas at a lower vCore count than the primary.

    # Create a named replica of an existing Hyperscale database
# The named replica lives in the same logical server
# It uses a different service objective than the primary if needed

az sql db replica create \
  --resource-group rg-hyperscale-prod \
  --server sql-hs-prod-eastus2 \
  --name OrdersDB \
  --partner-database OrdersDB_ReportingReplica \
  --partner-server sql-hs-prod-eastus2 \
  --service-objective HS_Gen5_4 \
  --partner-resource-group rg-hyperscale-prod

# Verify replication link
az sql db replica list-links \
  --resource-group rg-hyperscale-prod \
  --server sql-hs-prod-eastus2 \
  --name OrdersDB \
  --output table

# Connect to the named replica using its own connection string
# Named replica endpoint: sql-hs-prod-eastus2.database.windows.net
# Database: OrdersDB_ReportingReplica
# Application read intent:
# Server=sql-hs-prod-eastus2.database.windows.net;
# Database=OrdersDB_ReportingReplica;
# ApplicationIntent=ReadOnly;

    Replication Lag Monitoring

    Named replicas and HA secondary replicas consume the log service asynchronously. You need to monitor replication lag in production, particularly for workloads where reporting queries need data that is close to real-time.

    -- Query replication lag on the primary (connect to primary, not replica)
-- sys.dm_database_replica_states is your tool here
SELECT
    r.replica_id,
    r.role_desc,
    r.secondary_lag_seconds,
    r.log_send_queue_size,
    r.log_send_rate,
    r.redo_queue_size,
    r.redo_rate,
    r.synchronization_health_desc,
    r.last_hardened_lsn,
    r.last_received_lsn
FROM sys.dm_hadr_database_replica_states r
WHERE r.is_local = 0;  -- Non-local = secondary replicas

-- For Hyperscale-specific log service metrics
-- Check log generation rate and log service throughput
SELECT
    database_id,
    log_bytes_used,
    log_bytes_used_percent,
    log_committed_bytes,
    log_committed_percent,
    log_recovery_size_mb,
    log_since_last_log_backup_mb,
    log_since_last_checkpoint_mb
FROM sys.dm_db_log_stats(DB_ID());

-- Monitor page server health and latency from compute node perspective
-- This DMV shows remote page fetches (page server hits vs buffer pool hits)
SELECT
    page_server_reads,
    page_server_reads_in_progress,
    page_server_read_in_progress_ms,
    page_server_read_elapsed_ms,
    checkpoint_phase_desc,
    checkpoint_start_time
FROM sys.dm_exec_requests
WHERE page_server_reads > 0
ORDER BY page_server_reads DESC;

    A replication lag above 30 seconds warrants investigation. Common causes are high log generation rate on the primary (large bulk operations, high-frequency small transactions), temporary page server pressure, or network congestion within the Azure fabric. You cannot directly tune the log service, but you can reduce peak log generation rate by batching large DML operations.

    Migrating Into Hyperscale: Patterns That Work in Production

    Migrating to Hyperscale from an on-prem SQL Server or from another Azure SQL tier requires more planning than a tier change, because Hyperscale is one-directional and the migration itself requires database-level operations rather than just a service tier slider.

    Option 1: Backup and Restore via Azure Blob Storage

    For on-prem databases below roughly 1 TB, this is the fastest path. You take a COPY_ONLY backup, upload to Azure Blob Storage, and restore into a Hyperscale database using RESTORE DATABASE ... FROM URL.

    -- On-premises SQL Server: Create credential and backup to Azure Blob Storage
-- Step 1: Create SAS credential on source SQL Server
CREATE CREDENTIAL [https://storprodmigration.blob.core.windows.net/backups]
WITH IDENTITY = 'SHARED ACCESS SIGNATURE',
SECRET = 'sv=2023-01-03&ss=b&srt=sco&sp=rwdlacupiytfx&se=2024-12-31T00:00:00Z&st=2024-01-01T00:00:00Z&spr=https&sig=<your-sas-token>';
GO

-- Step 2: Full backup with COPY_ONLY (non-disruptive to existing backup chain)
BACKUP DATABASE [OrdersDB]
TO URL = 'https://storprodmigration.blob.core.windows.net/backups/OrdersDB_full_20240115.bak'
WITH 
    COPY_ONLY,
    COMPRESSION,
    STATS = 10,
    BLOCKSIZE = 65536,
    MAXTRANSFERSIZE = 4194304;
GO

-- For large databases, stripe the backup across multiple files
BACKUP DATABASE [OrdersDB]
TO 
    URL = 'https://storprodmigration.blob.core.windows.net/backups/OrdersDB_stripe1.bak',
    URL = 'https://storprodmigration.blob.core.windows.net/backups/OrdersDB_stripe2.bak',
    URL = 'https://storprodmigration.blob.core.windows.net/backups/OrdersDB_stripe3.bak',
    URL = 'https://storprodmigration.blob.core.windows.net/backups/OrdersDB_stripe4.bak'
WITH 
    COPY_ONLY,
    COMPRESSION,
    STATS = 5;
GO

-- Step 3: On the Hyperscale target (via SSMS or Azure Cloud Shell with sqlcmd)
-- Create credential on target server
CREATE CREDENTIAL [https://storprodmigration.blob.core.windows.net/backups]
WITH IDENTITY = 'SHARED ACCESS SIGNATURE',
SECRET = 'sv=2023-01-03&ss=b&srt=sco&sp=rwdlacupiytfx&se=2024-12-31T00:00:00Z&st=2024-01-01T00:00:00Z&spr=https&sig=<your-sas-token>';
GO

-- Step 4: Restore into Hyperscale
-- Note: You must restore to a NEW database name, not overwrite an existing one
RESTORE DATABASE [OrdersDB]
FROM 
    URL = 'https://storprodmigration.blob.core.windows.net/backups/OrdersDB_stripe1.bak',
    URL = 'https://storprodmigration.blob.core.windows.net/backups/OrdersDB_stripe2.bak',
    URL = 'https://storprodmigration.blob.core.windows.net/backups/OrdersDB_stripe3.bak',
    URL = 'https://storprodmigration.blob.core.windows.net/backups/OrdersDB_stripe4.bak'
WITH 
    RECOVERY,
    STATS = 5;
GO

    For minimal-downtime migrations, you restore the full backup, then apply differential and log backups during a short cutover window. This is functionally the same as the SQL Server DBA's traditional migration playbook — the cloud just adds the credential layer.

    Option 2: Azure Database Migration Service

    For larger databases or when you want a more managed migration process, Azure DMS handles the orchestration. You configure it to perform an initial load followed by continuous replication of changes until you cut over. It is comparable in concept to setting up log shipping or replication to a new instance, with the monitoring and retry logic handled for you.

    # Create an Azure DMS instance (Premium tier for online migrations)
az dms create \
  --resource-group rg-hyperscale-prod \
  --name dms-migration-prod \
  --location eastus2 \
  --sku-name Premium_4vCores \
  --subnet /subscriptions/<sub-id>/resourceGroups/rg-network/providers/Microsoft.Network/virtualNetworks/vnet-prod/subnets/snet-dms \
  --storage-account /subscriptions/<sub-id>/resourceGroups/rg-hyperscale-prod/providers/Microsoft.Storage/storageAccounts/storprodmigration

# Create a migration project
az dms project create \
  --resource-group rg-hyperscale-prod \
  --service-name dms-migration-prod \
  --name proj-ordersdb-migration \
  --source-platform SQL \
  --target-platform AzureSqlDb \
  --location eastus2

# Create a migration task (online migration with continuous sync)
az dms project task create \
  --resource-group rg-hyperscale-prod \
  --service-name dms-migration-prod \
  --project-name proj-ordersdb-migration \
  --name task-ordersdb-migrate \
  --task-type OnlineMigration \
  --source-connection-json @source_connection.json \
  --target-connection-json @target_connection.json \
  --database-options-json @database_migration_options.json

# Monitor migration status
az dms project task show \
  --resource-group rg-hyperscale-prod \
  --service-name dms-migration-prod \
  --project-name proj-ordersdb-migration \
  --name task-ordersdb-migrate \
  --expand output \
  --query "output[0].migrationState"

    Pre-Migration Compatibility Check

    Hyperscale supports most SQL Server features, but there are exclusions. Run the Database Experimentation Assistant (DEA) or use the assessment feature in Azure Migrate before committing to a migration timeline. Common compatibility gaps that surface for on-prem databases:

  • → Linked servers — not supported in Azure SQL Database including Hyperscale. Replace with OPENDATASOURCE, external tables, or application-level joins.
  • → SQL Server Agent — not available. Use Azure Elastic Jobs or Azure Automation.
  • → Filestream / Filetable — not supported.
  • → Cross-database queries within the same instance — Azure SQL is database-scoped. If your application uses three-part names to query across databases, that needs rearchitecting.
  • → BULK INSERT from local file paths — requires Azure Blob Storage as the source.
