Chapter 29: Backup and Restore in Cloud: Automated Snapshots vs Manual

Cloud-managed databases don't eliminate the need for backup strategy — they redistribute the responsibility. The platforms handle the mechanics of snapshot creation and storage, but the decisions about retention windows, cross-region durability, point-in-time recovery granularity, and restore validation remain squarely in your hands. This chapter walks through how AWS RDS, Aurora, Azure SQL, and Google Cloud SQL implement backup systems, where automated snapshots fall short of what on-premises DBAs are accustomed to controlling, and how to build a production-grade backup posture that accounts for the gaps. You will also see the restore side in detail, because backup systems that have never been tested are guesses, not strategies.

---

29.1 How Automated Snapshots Work Under the Hood

When you ran backups on-premises, you controlled everything: backup agent, target storage, compression, parallelism, verification jobs. In cloud-managed RDS, the platform takes daily automated snapshots during a configurable maintenance window and retains transaction logs for point-in-time recovery (PITR). Understanding what actually happens during these operations — not just what the console tells you — changes how you design around them.

RDS Automated Snapshots

RDS snapshots are storage-volume snapshots backed by EBS. For single-AZ instances, the snapshot process briefly suspends I/O — typically for a few seconds on modern instances, but measurable. For Multi-AZ deployments, the snapshot is taken from the standby replica, which eliminates the I/O pause on the primary. This is the same reason you should always run production workloads as Multi-AZ, not just for failover, but for backup behavior.

The retention window accepts values from 0 to 35 days. Setting it to 0 disables automated snapshots entirely — and also disables PITR, because transaction log uploads to S3 depend on automated snapshots being enabled. A common misconfiguration is setting retention to 1 day to save costs on a "non-critical" system, then discovering that PITR only reaches back 24 hours when a logic error surfaces three days later.

Transaction logs are uploaded to S3 every five minutes. This gives you a recovery point objective (RPO) of approximately five minutes for PITR restores. You cannot restore to an arbitrary second — you restore to a specific UTC timestamp and RDS selects the nearest log position.

Aurora Automated Backups

Aurora's backup model differs architecturally. Because Aurora stores data in a distributed storage layer (six copies across three AZs), snapshots are not EBS volume captures — they are metadata pointers into the shared storage. This means Aurora snapshots are nearly instantaneous and impose no I/O impact regardless of whether you're using a Multi-AZ writer or not. Snapshot creation time is measured in seconds even for multi-terabyte clusters.

Aurora's PITR window covers the same 1–35 day range, but the continuous backup into Aurora's storage layer means the effective RPO is closer to one second for supported restore points, versus the five-minute log upload interval on standard RDS.

Azure SQL Automated Backups

Azure SQL Database takes full backups weekly, differential backups every 12 hours, and transaction log backups every 5–12 minutes. These are stored in geo-redundant storage (GRS) by default, giving you cross-region durability without additional configuration. The retention window for Point-in-Time Restore (PITR) ranges from 1 to 35 days on General Purpose and Business Critical tiers; the Hyperscale tier handles backup differently and warrants its own treatment below.

One difference from RDS that catches on-premises DBAs: in Azure SQL, you cannot directly access backup files. You restore to a new database — you cannot restore over an existing database in-place. This forces a rename-and-swap procedure that has implications for connection strings and dependent jobs.

Cloud SQL Automated Backups

Cloud SQL stores automated backups in the same region as the instance by default. Cross-region backup storage requires explicit configuration via backup export or cross-region replicas. Cloud SQL's PITR uses binary log retention for MySQL-compatible instances and WAL archiving for PostgreSQL-compatible instances, with a configurable retention window from 1 to 7 days — shorter than AWS or Azure. If you need more than 7 days of PITR coverage on Cloud SQL PostgreSQL, you are responsible for shipping WAL files to Cloud Storage manually using pg_basebackup exports or the pg_receivewal utility.

29.2 Configuring Automated Backup Windows and Retention

The defaults are rarely appropriate for production. Every platform gives you the ability to adjust backup windows, retention periods, and storage locations, but the configuration paths differ.

RDS Backup Configuration via AWS CLI

# Create an RDS PostgreSQL instance with backup configured for production
aws rds create-db-instance \
  --db-instance-identifier prod-postgres-01 \
  --db-instance-class db.r6g.2xlarge \
  --engine postgres \
  --engine-version 15.4 \
  --master-username dbadmin \
  --master-user-password "${DB_MASTER_PASS}" \
  --allocated-storage 500 \
  --storage-type gp3 \
  --multi-az \
  --backup-retention-period 14 \
  --preferred-backup-window "02:00-03:00" \
  --preferred-maintenance-window "sun:04:00-sun:05:00" \
  --deletion-protection \
  --no-publicly-accessible \
  --db-subnet-group-name prod-db-subnet-group \
  --vpc-security-group-ids sg-0abc123def456789 \
  --tags Key=Environment,Value=Production Key=Owner,Value=DBA-Team

# Modify an existing instance's backup retention (takes effect immediately)
aws rds modify-db-instance \
  --db-instance-identifier prod-postgres-01 \
  --backup-retention-period 21 \
  --apply-immediately

The --preferred-backup-window uses UTC. Set it outside your peak transaction period. If you don't specify it, AWS assigns a random window — and randomly assigned maintenance windows have a habit of landing during business hours in some time zones.

RDS Cross-Region Backup Replication

Automated snapshots do not replicate cross-region by default. You must enable cross-region automated backup replication separately:

# Enable cross-region automated backup replication
# Source: us-east-1, Destination: us-west-2
aws rds start-export-task \
  --export-task-identifier backup-export-$(date +%Y%m%d) \
  --source-arn arn:aws:rds:us-east-1:123456789012:snapshot:rds:prod-postgres-01-2024-01-15 \
  --s3-bucket-name prod-db-backups-dr \
  --s3-prefix postgres-exports/ \
  --iam-role-arn arn:aws:iam::123456789012:role/RDSExportRole \
  --kms-key-id arn:aws:kms:us-west-2:123456789012:key/mrk-abcdef1234567890

# Enable cross-region backup copy for all future automated snapshots
aws rds enable-cross-region-automated-backups \
  --source-db-instance-identifier prod-postgres-01 \
  --backup-retention-period 7 \
  --s3-export-path s3://prod-db-backups-dr/cross-region/ \
  --destination-region us-west-2

Note: cross-region backup replication for RDS is a separate feature from cross-region read replicas. The replicated backups are usable independently of the source instance — if your primary region fails, you can restore from the replicated backups in the DR region without access to the source.

Terraform: Aurora Cluster with Backup Configuration

resource "aws_rds_cluster" "aurora_prod" {
  cluster_identifier      = "aurora-prod-cluster"

  engine_version          = "15.4"
  database_name           = "appdb"

  preferred_backup_window   = "03:00-04:00"
  preferred_maintenance_window = "sun:05:00-sun:06:00"
  
  # Cross-region backup replication
  enable_global_write_forwarding = false

  deletion_protection       = true
  skip_final_snapshot       = false

    Environment = "Production"

  }
}

Azure SQL Backup Configuration

# Check current backup policy on an Azure SQL Database
az sql db show \
  --resource-group prod-rg \
  --server prod-sql-server \
  --name proddb \
  --query "{name:name, shortTermRetention:requestedServiceObjectiveName}" \
  --output table

# Configure short-term backup retention (PITR window)
az sql db str-policy set \
  --resource-group prod-rg \
  --server prod-sql-server \
  --name proddb \
  --retention-days 21

# Configure long-term retention (LTR) policy
# Weekly: keep 12 weeks, Monthly: keep 12 months, Yearly: keep 5 years
az sql db ltr-policy set \
  --resource-group prod-rg \
  --server prod-sql-server \
  --name proddb \
  --weekly-retention P12W \
  --monthly-retention P12M \
  --yearly-retention P5Y \
  --week-of-year 1

# List available LTR backups
az sql db ltr-backup list \
  --location eastus \
  --server prod-sql-server \
  --database proddb \
  --output table

The LTR (Long-Term Retention) policy in Azure SQL maps to what on-premises DBAs implemented with backup-to-tape or offsite archive policies. Azure stores LTR backups in geo-redundant storage and they survive even if you delete the database or server — a meaningful difference from RDS where manual snapshots are deleted if you delete the instance unless you explicitly retain them.

Cloud SQL Backup Configuration

# Create a Cloud SQL PostgreSQL instance with backup configured
gcloud sql instances create prod-postgres-01 \
  --database-version=POSTGRES_15 \
  --tier=db-custom-8-32768 \
  --region=us-central1 \
  --availability-type=REGIONAL \
  --backup-start-time=03:00 \
  --enable-bin-log \
  --retained-backups-count=14 \
  --retained-transaction-log-days=7 \
  --storage-type=SSD \
  --storage-size=500GB \
  --storage-auto-increase \
  --deletion-protection \
  --no-assign-ip \
  --network=projects/myproject/global/networks/prod-vpc

# Verify backup configuration
gcloud sql instances describe prod-postgres-01 \
  --format="json(settings.backupConfiguration)"

The --retained-transaction-log-days cap of 7 on Cloud SQL is the most significant constraint compared to RDS or Azure SQL. If your organization requires 14 or 30 days of PITR, Cloud SQL PostgreSQL cannot satisfy it natively, and you must supplement with exported logical backups.

29.3 Manual Snapshots and On-Demand Backups

Automated snapshots cover the daily cadence. Manual snapshots address the scenarios automated backups were not designed for: pre-migration checkpoints, pre-schema-change captures, compliance archiving, and cross-account copies for DR testing.

When Manual Snapshots Matter

On-premises DBAs took pre-change backups before every major schema migration. In cloud managed databases, that discipline translates directly to manual snapshots. Unlike automated snapshots — which are deleted when their retention window expires or when the instance is deleted (unless explicitly protected) — manual snapshots persist until you explicitly delete them. This makes them appropriate for compliance retention.

# Create a manual RDS snapshot before a schema migration
aws rds create-db-snapshot \
  --db-instance-identifier prod-postgres-01 \
  --db-snapshot-identifier prod-postgres-01-pre-migration-v2-$(date +%Y%m%d%H%M) \
  --tags Key=Purpose,Value=pre-migration Key=Schema,Value=v2 Key=Retention,Value=90days

# Check snapshot status
aws rds describe-db-snapshots \
  --db-snapshot-identifier prod-postgres-01-pre-migration-v2-20240115-1430 \
  --query "DBSnapshots[0].{Status:Status,Progress:PercentProgress,Size:AllocatedStorage}" \
  --output table

# Copy a manual snapshot to another region for DR validation
aws rds copy-db-snapshot \
  --source-db-snapshot-identifier arn:aws:rds:us-east-1:123456789012:snapshot:prod-postgres-01-pre-migration-v2-20240115-1430 \
  --target-db-snapshot-identifier prod-postgres-01-pre-migration-v2-dr-copy \
  --kms-key-id arn:aws:kms:us-west-2:123456789012:key/mrk-abcdef1234567890 \
  --source-region us-east-1 \
  --region us-west-2

# Share a snapshot with another AWS account (for DR account access)
aws rds modify-db-snapshot-attribute \
  --db-snapshot-identifier prod-postgres-01-pre-migration-v2-20240115-1430 \
  --attribute-name restore \
  --values-to-add 987654321098  # DR account ID

Cloud SQL On-Demand Backups

# Create an on-demand backup before a maintenance operation
gcloud sql backups create \
  --instance=prod-postgres-01 \
  --description="Pre-major-version-upgrade backup $(date +%Y%m%d)"

# List all backups (automated and on-demand)
gcloud sql backups list \
  --instance=prod-postgres-01 \
  --filter="type=ON_DEMAND" \
  --format="table(id, windowStartTime, status, diskEncryptionConfiguration.kmsKeyName)"

# Export a logical backup to Cloud Storage (supplements snapshot-based backup)
gcloud sql export sql prod-postgres-01 \
  gs://prod-db-exports/postgres/prod-postgres-01-$(date +%Y%m%d).sql.gz \
  --database=appdb \
  --offload

The --offload flag on Cloud SQL exports runs the export from a serverless export process rather than the instance itself, reducing impact on production. Without it, the export operation uses instance CPU and creates transient I/O load.

Aurora Manual Snapshots and Export to S3

Aurora adds a capability that standard RDS does not have in the same form: you can export a snapshot to S3 as Parquet files organized by table, making the data queryable by Athena without restoring the full database. This is useful for compliance audits and forensic queries on point-in-time data.

# Export Aurora snapshot to S3 as Parquet (for Athena querying)
aws rds start-export-task \
  --export-task-identifier audit-export-20240115 \
  --source-arn arn:aws:rds:us-east-1:123456789012:cluster-snapshot:aurora-prod-cluster-2024-01-15 \
  --s3-bucket-name prod-audit-exports \
  --s3-prefix aurora-exports/2024-01-15/ \
  --iam-role-arn arn:aws:iam::123456789012:role/RDSExportRole \
  --kms-key-id arn:aws:kms:us-east-1:123456789012:key/mrk-export-key \
  --export-only "appdb.orders" "appdb.customers"  # Export only specific tables

# Monitor export progress
aws rds describe-export-tasks \
  --export-task-identifier audit-export-20240115 \
  --query "ExportTasks[0].{Status:Status,PercentProgress:PercentProgress,TotalExtractedDataInGB:TotalExtractedDataInGB}"

29.4 Point-in-Time Recovery: Configuration and Execution

PITR is where backup strategy meets recovery strategy. The mechanics differ substantially across platforms, and the gotchas matter more here than anywhere else.

RDS PITR Restore

PITR on RDS always creates a new DB instance — it does not restore in place. This is a fundamental difference from on-premises PostgreSQL where you recover into the existing data directory. The new instance inherits the parameter group, security group, and subnet group from the original, but you must manually update application connection strings or your DNS CNAME to point to the new instance.

# Restore RDS to a specific point in time
# Note: this creates a NEW instance, not an in-place restore
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier prod-postgres-01 \
  --target-db-instance-identifier prod-postgres-01-restored-20240115T0230 \
  --restore-time 2024-01-15T02:30:00Z \
  --db-instance-class db.r6g.2xlarge \
  --multi-az \
  --no-publicly-accessible \
  --db-subnet-group-name prod-db-subnet-group \
  --vpc-security-group-ids sg-0abc123def456789 \
  --tags Key=Purpose,Value=PITR-restore Key=RestoreTime,Value=2024-01-15T02:30:00Z

# Check restore status
aws rds describe-db-instances \
  --db-instance-identifier prod-postgres-01-restored-20240115T0230 \
  --query "DBInstances[0].{Status:DBInstanceStatus,Endpoint:Endpoint.Address,Port:Endpoint.Port}"

# After validation, modify DNS to point to restored instance
# (or promote to replace the original — must be done outside AWS)

Determining the Correct Restore Timestamp

This is where the DBA skill matters. Identifying the exact timestamp before a problematic event requires querying application logs, CloudWatch Logs Insights, or PostgreSQL log files. A script to correlate an error event with the appropriate restore target:

import boto3
from datetime import datetime, timedelta, timezone

def find_pitr_restore_window(instance_id: str, event_timestamp: datetime) -> dict:
    """
    Given an incident timestamp, identify the earliest and latest
    usable PITR restore points around that event.
    """
    rds = boto3.client('rds', region_name='us-east-1')
    
    # Get instance details to check backup retention and earliest restore time
    response = rds.describe_db_instances(DBInstanceIdentifier=instance_id)
    instance = response['DBInstances'][0]
    
    earliest_restore = instance.get('LatestRestorableTime')
    # Note: AWS SDK returns LatestRestorableTime but you need both ends
    
    # Check automated snapshots to find earliest available point
    snapshots = rds.describe_db_snapshots(
        DBInstanceIdentifier=instance_id,
        SnapshotType='automated',
        Filters=[
            {'Name': 'snapshot-create-time',
             'Values': ['2024-01-01T00:00:00Z']}  # Adjust range
        ]
    )
    
    snapshot_times = sorted([
        s['SnapshotCreateTime'] 
        for s in snapshots['DBSnapshots']
        if s['Status'] == 'available'
    ])
    
    # Recommended restore point: 5 minutes before the event
    recommended_restore = event_timestamp - timedelta(minutes=5)
    
    result = {
        'incident_time': event_timestamp.isoformat(),
        'recommended_restore_target': recommended_restore.isoformat(),
        'earliest_available_snapshot': snapshot_times[0].isoformat() if snapshot_times else None,
        'latest_restorable_time': earliest_restore.isoformat() if earliest_restore else None,
        'pitr_feasible': (
            snapshot_times[0] <= recommended_restore <= earliest_restore
            if snapshot_times and earliest_restore else False
        )
    }
    
    return result


# Example usage
incident_time = datetime(2024, 1, 15, 2, 47, 0, tzinfo=timezone.utc)
window = find_pitr_restore_window('prod-postgres-01', incident_time)

print(f"Incident occurred at: {window['incident_time']}")
print(f"Recommended restore to: {window['recommended_restore_target']}")
print(f"PITR feasible: {window['pitr_feasible']}")

Azure SQL PITR

# Restore Azure SQL Database to a point in time
# Creates a new database — cannot restore in place
az sql db restore \
  --resource-group prod-rg \
  --server prod-sql-server \
  --name proddb \
  --dest-name proddb-restored-20240115 \
  --time "2024-01-15T02:30:00" \
  --edition GeneralPurpose \
  --service-objective GP_Gen5_8 \
  --backup-storage-redundancy Geo

# Monitor restore status
az sql db show \
  --resource-group prod-rg \
  --server prod-sql-server \
  --name proddb-restored-20240115 \
  --query "{status:status, createTime:createTime}" \
  --output table

# After validation: rename original (if replacing), then rename restored
# Azure SQL supports database rename via ALTER DATABASE
# Connect to master and run:
# ALTER DATABASE [proddb] MODIFY NAME = [proddb-old-20240115];
# ALTER DATABASE [proddb-restored-20240115] MODIFY NAME = [proddb];

The rename approach in Azure SQL is your equivalent of the on-premises "restore over the original" workflow. It requires brief downtime during the rename operation, and active connections must be terminated. Azure SQL does not allow renaming a database with active connections — you must set it to single-user mode first via the portal or SSMS, or terminate connections via sys.dm_exec_sessions.
