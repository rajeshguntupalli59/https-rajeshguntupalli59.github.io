Chapter 34: Access Control on Vector Data

Vector databases introduce a category of access control problems that traditional RBAC models weren't designed to solve. A row in a relational table is discrete and bounded — you can grant SELECT on a specific table, filter rows by tenant ID, and call it done. A vector embedding is a compressed semantic fingerprint of content that may span topics, contain implicit references to sensitive material, or be retrievable through queries that have no obvious surface-level connection to the underlying data. When a similarity search returns the top-k nearest neighbors, you cannot predict at query time which documents will surface. This chapter covers how to apply row-level security, namespace isolation, API key scoping, and query-time filtering to vector data across pgvector, Pinecone, Weaviate, and Qdrant — and how to build a unified authorization layer that treats vector search as a first-class security boundary.

---

Why Vector Access Control Is Different

In a traditional database, access control answers a simple question: does this user have permission to read this row? The data model is explicit. You can enumerate rows, apply predicates, and reason about what any given query will return before it executes.

Vector search inverts this relationship. You start with a query embedding and find what's similar. The result set is determined by geometric proximity in high-dimensional space, not by a filter predicate written at design time. A user querying for "quarterly financial projections" might pull a document fragment that was never explicitly tagged as sensitive but happens to be semantically close to their query vector. Conversely, strict keyword-based filtering might block a legitimate user from finding a document because the metadata doesn't match even though the content is relevant.

Three properties of vector data create novel access control challenges:

Semantic bleed. Embeddings from different security domains can cluster near each other if the underlying content is topically adjacent. A public FAQ about drug interactions and a restricted clinical trial protocol might embed within similar vector space because they share vocabulary. Namespace separation prevents this — you don't filter after the fact, you never allow the query to reach restricted vectors.

Inference risk. Even if a user cannot read a document directly, the fact that a similarity search returns zero results from a namespace they have permission to query can reveal that certain content exists. This is a vector analogue of a timing side-channel. Designing around it requires either returning null results uniformly regardless of permission state or ensuring the user never knows a restricted namespace exists.

Metadata exposure. Most vector databases store metadata payloads alongside embeddings. The embedding itself may be meaningless to an attacker, but the metadata often contains document titles, author IDs, timestamps, and excerpts. Access control that secures the embedding but exposes the payload is incomplete.

The practical architecture for addressing these challenges combines three layers: storage-level isolation (namespaces and collections), query-time filtering (metadata predicates applied before returning results), and application-level enforcement (authorization checks before the vector query is even dispatched).

---

Row-Level Security in pgvector

PostgreSQL's Row-Level Security (RLS) is the most mature access control mechanism available in the vector database ecosystem because pgvector is a PostgreSQL extension. Every security primitive PostgreSQL exposes — roles, policies, column-level grants, views — applies directly to vector data.

The baseline pattern stores a tenant_id or user_id alongside the embedding vector and attaches RLS policies that automatically filter queries to rows the current user owns.

-- PostgreSQL / pgvector

-- Create the documents table with tenant isolation
CREATE TABLE document_embeddings (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id   TEXT NOT NULL,
    content     TEXT NOT NULL,
    embedding   vector(1536),
    metadata    JSONB DEFAULT '{}',
    created_at  TIMESTAMPTZ DEFAULT now()
);

-- Index for ANN search
CREATE INDEX ON document_embeddings 
USING ivfflat (embedding vector_cosine_ops) 
WITH (lists = 100);

-- Index for tenant filtering (critical for performance)
CREATE INDEX ON document_embeddings (tenant_id);

-- Enable RLS
ALTER TABLE document_embeddings ENABLE ROW LEVEL SECURITY;

-- Policy: tenants can only read their own rows
CREATE POLICY tenant_isolation ON document_embeddings
    FOR ALL
    USING (tenant_id = current_setting('app.current_tenant', true));

-- Policy: service role bypasses RLS for admin operations
CREATE POLICY service_bypass ON document_embeddings
    FOR ALL
    TO service_role
    USING (true);

-- Grant table access to application role (RLS handles row filtering)
GRANT SELECT, INSERT ON document_embeddings TO app_role;

At query time, the application sets the tenant context before issuing the vector search:

-- PostgreSQL / pgvector

-- Set tenant context for the current session
SET LOCAL app.current_tenant = 'tenant_acme_corp';

-- Similarity search — RLS automatically scopes to tenant_acme_corp rows
SELECT 
    id,
    content,
    metadata,
    1 - (embedding <=> $1::vector) AS similarity_score
FROM document_embeddings
WHERE 1 - (embedding <=> $1::vector) > 0.75
ORDER BY embedding <=> $1::vector
LIMIT 10;

-- Verify the plan respects RLS (check for Filter on tenant_id)
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, content, 1 - (embedding <=> $1::vector) AS score
FROM document_embeddings
ORDER BY embedding <=> $1::vector
LIMIT 10;

A frequent performance pitfall: when RLS adds a WHERE tenant_id = X predicate to an IVFFlat or HNSW scan, PostgreSQL must evaluate the policy after the ANN index returns candidates, not before. If your lists = 100 index returns 200 candidates but only 3 belong to the current tenant, your effective recall is severely degraded and you've wasted I/O on irrelevant rows.

The solution is to use a composite index that includes the tenant ID as a partition key, or to maintain separate IVFFlat indexes per tenant using table partitioning:

-- PostgreSQL / pgvector

-- Partition by tenant for performance isolation
CREATE TABLE document_embeddings_partitioned (
    id          UUID NOT NULL DEFAULT gen_random_uuid(),
    tenant_id   TEXT NOT NULL,
    content     TEXT NOT NULL,
    embedding   vector(1536),
    metadata    JSONB DEFAULT '{}'
) PARTITION BY LIST (tenant_id);

-- Create per-tenant partitions
CREATE TABLE doc_embeddings_acme 
    PARTITION OF document_embeddings_partitioned
    FOR VALUES IN ('tenant_acme_corp');

CREATE TABLE doc_embeddings_globex
    PARTITION OF document_embeddings_partitioned
    FOR VALUES IN ('tenant_globex');

-- Per-partition IVFFlat indexes
CREATE INDEX ON doc_embeddings_acme 
USING ivfflat (embedding vector_cosine_ops) WITH (lists = 50);

CREATE INDEX ON doc_embeddings_globex
USING ivfflat (embedding vector_cosine_ops) WITH (lists = 50);

-- RLS still applies, but the planner now prunes to a single partition
ALTER TABLE document_embeddings_partitioned ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation_partitioned ON document_embeddings_partitioned
    FOR ALL
    USING (tenant_id = current_setting('app.current_tenant', true));

For Supabase deployments, which run managed PostgreSQL with pgvector, this RLS pattern works identically. Supabase's auth system automatically populates auth.uid() which you can use as the tenant identifier directly in policies:

-- PostgreSQL / pgvector (Supabase)

-- Supabase-specific policy using auth context
CREATE POLICY user_owns_documents ON document_embeddings
    FOR SELECT
    USING (user_id = auth.uid());

CREATE POLICY user_inserts_own ON document_embeddings
    FOR INSERT
    WITH CHECK (user_id = auth.uid());

Namespace Isolation in Pinecone and Weaviate

Managed vector databases handle multi-tenancy through namespace and collection isolation rather than SQL-style policies. The security model is coarser — you're working at the API key and namespace level rather than per-row policies — but the operational simplicity trades off well for most SaaS applications.

Pinecone namespaces partition vectors within an index so that operations on one namespace never touch vectors in another. All three operations — upsert, query, and delete — accept a namespace parameter. Vectors without a namespace land in the default namespace, which is a common source of tenant data leakage during development.

# Python
import os
from pinecone import Pinecone, ServerlessSpec
from openai import OpenAI

pc = Pinecone(api_key=os.environ["PINECONE_API_KEY"])
openai_client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

# Create index if it doesn't exist
index_name = "document-store"
if index_name not in pc.list_indexes().names():
    pc.create_index(
        name=index_name,
        dimension=1536,
        metric="cosine",
        spec=ServerlessSpec(cloud="aws", region="us-east-1")
    )

index = pc.Index(index_name)

def embed_text(text: str) -> list[float]:
    response = openai_client.embeddings.create(
        input=text,
        model="text-embedding-3-small"
    )
    return response.data[0].embedding

def upsert_for_tenant(tenant_id: str, documents: list[dict]) -> None:
    """
    All writes are scoped to the tenant's namespace.
    Cross-namespace writes require separate API key issuance at the
    infrastructure level — namespaces themselves are not credential-bound.
    """
    vectors = []
    for doc in documents:
        embedding = embed_text(doc["content"])
        vectors.append({
            "id": doc["id"],
            "values": embedding,
            "metadata": {
                "content": doc["content"],
                "source": doc.get("source", ""),
                "tenant_id": tenant_id,   # redundant but useful for audit
                "classification": doc.get("classification", "internal")
            }
        })
    
    # namespace= is the hard isolation boundary
    index.upsert(vectors=vectors, namespace=tenant_id)

def query_for_tenant(
    tenant_id: str, 
    query_text: str, 
    top_k: int = 10,
    classification_filter: str | None = None
) -> list[dict]:
    """
    Query is restricted to tenant namespace.
    Optional metadata filter further restricts by classification.
    """
    query_embedding = embed_text(query_text)
    
    # Build metadata filter if classification scope is provided
    filter_dict = None
    if classification_filter:
        filter_dict = {"classification": {"$eq": classification_filter}}
    
    results = index.query(
        vector=query_embedding,
        top_k=top_k,
        namespace=tenant_id,          # hard namespace isolation
        filter=filter_dict,            # optional metadata predicate
        include_metadata=True
    )
    
    return [
        {
            "id": match.id,
            "score": match.score,
            "content": match.metadata.get("content", ""),
            "classification": match.metadata.get("classification", "")
        }
        for match in results.matches
    ]

# Usage
upsert_for_tenant("acme_corp", [
    {"id": "doc-001", "content": "Q3 revenue targets", "classification": "confidential"},
    {"id": "doc-002", "content": "Employee handbook", "classification": "internal"}
])

# This query cannot reach globex_corp documents — namespace enforces that
results = query_for_tenant("acme_corp", "financial targets", classification_filter="confidential")

Pinecone's namespace model has a significant architectural constraint: there is no native per-namespace API key. A single API key has access to all namespaces in the index. This means namespace isolation is enforced entirely by your application layer — if your query function is called with the wrong tenant ID, you'll get cross-tenant data. The architecture that addresses this is a server-side authorization proxy that validates the caller's identity before injecting the namespace parameter:

# Python
# Authorization proxy pattern — never trust client-supplied tenant_id
import jwt
from functools import wraps
from fastapi import FastAPI, HTTPException, Depends
from fastapi.security import HTTPBearer, HTTPAuthorizationCredentials

app = FastAPI()
security = HTTPBearer()

SECRET_KEY = os.environ["JWT_SECRET"]

def get_verified_tenant(
    credentials: HTTPAuthorizationCredentials = Depends(security)
) -> str:
    """
    Validate JWT and extract tenant claim.
    The tenant_id used in Pinecone namespace queries comes ONLY
    from the verified token, never from the request body.
    """
    try:
        payload = jwt.decode(
            credentials.credentials, 
            SECRET_KEY, 
            algorithms=["HS256"]
        )
        tenant_id = payload.get("tenant_id")
        if not tenant_id:
            raise HTTPException(status_code=403, detail="No tenant claim in token")
        return tenant_id
    except jwt.ExpiredSignatureError:
        raise HTTPException(status_code=401, detail="Token expired")
    except jwt.InvalidTokenError:
        raise HTTPException(status_code=401, detail="Invalid token")

@app.post("/search")
async def search_documents(
    query: str,
    top_k: int = 10,
    tenant_id: str = Depends(get_verified_tenant)  # extracted from JWT, not request
):
    results = query_for_tenant(tenant_id, query, top_k=top_k)
    return {"results": results, "tenant": tenant_id}

Weaviate organizes data into classes (schema types) and supports multi-tenancy as a first-class feature. In Weaviate v1.17+, tenant isolation can be enabled at the class level, after which each tenant's data is stored in separate segments and queries must specify a tenant key:

# Python
import weaviate
from weaviate.classes.config import Configure, Property, DataType
from weaviate.classes.tenants import Tenant

client = weaviate.connect_to_cloud(
    cluster_url=os.environ["WEAVIATE_URL"],
    auth_credentials=weaviate.auth.AuthApiKey(os.environ["WEAVIATE_API_KEY"])
)

# Create a multi-tenant collection
documents = client.collections.create(
    name="Document",
    multi_tenancy_config=Configure.multi_tenancy(enabled=True),
    properties=[
        Property(name="content", data_type=DataType.TEXT),
        Property(name="source", data_type=DataType.TEXT),
        Property(name="classification", data_type=DataType.TEXT),
    ],
    vectorizer_config=Configure.Vectorizer.text2vec_openai(
        model="text-embedding-3-small"
    )
)

# Register tenants explicitly
documents.tenants.create([
    Tenant(name="acme_corp"),
    Tenant(name="globex"),
])

# Insert objects scoped to a tenant
acme_collection = client.collections.get("Document").with_tenant("acme_corp")
acme_collection.data.insert({
    "content": "Annual security audit findings for ACME infrastructure",
    "source": "security-team",
    "classification": "restricted"
})

# Query scoped to a tenant — physically isolated from other tenants
def search_weaviate(tenant_id: str, query: str, top_k: int = 5) -> list[dict]:
    collection = client.collections.get("Document").with_tenant(tenant_id)
    
    results = collection.query.near_text(
        query=query,
        limit=top_k,
        filters=weaviate.classes.query.Filter.by_property("classification").equal("restricted"),
        return_properties=["content", "source", "classification"]
    )
    
    return [
        {
            "content": obj.properties["content"],
            "classification": obj.properties["classification"],
            "distance": obj.metadata.distance if obj.metadata else None
        }
        for obj in results.objects
    ]

results = search_weaviate("acme_corp", "security vulnerabilities")
client.close()

Weaviate's multi-tenancy implementation physically separates tenant data into different segments on disk, which means a misconfigured query that omits the tenant key raises an error rather than silently returning cross-tenant data. This fail-closed behavior is architecturally superior to Pinecone's namespace model for high-assurance environments.

Metadata Filtering as a Security Predicate

Namespace and collection isolation enforces coarse-grained boundaries between tenants. Within a tenant's namespace, you often need finer-grained access control: a user in acme_corp should only see documents classified at or below their clearance level, or only documents in projects they're assigned to.

Metadata filters serve this purpose, but they carry an important caveat: metadata filters are not a substitute for namespace isolation. They operate after the ANN index has already identified candidate vectors. If a metadata filter is the only barrier between a user and restricted data, a misconfigured filter or a library bug that drops the filter predicate is a security incident.

The correct architecture stacks both controls:

1. Namespace/collection isolation prevents cross-tenant data access unconditionally.

2. Metadata filters enforce within-tenant access control.

# Python
# Qdrant with payload filtering for within-tenant access control
from qdrant_client import QdrantClient
from qdrant_client.models import (
    Distance, VectorParams, PointStruct,
    Filter, FieldCondition, MatchValue, MatchAny,
    models
)

client = QdrantClient(
    url=os.environ["QDRANT_URL"],
    api_key=os.environ["QDRANT_API_KEY"]
)

# Create collection
client.recreate_collection(
    collection_name="documents_acme_corp",  # collection per tenant
    vectors_config=VectorParams(size=1536, distance=Distance.COSINE)
)

def build_access_filter(user_roles: list[str], clearance_level: int) -> Filter:
    """
    Build a Qdrant filter from user's authorization attributes.
    This filter is constructed server-side from the verified identity —
    the calling user never constructs this directly.
    """
    # Map clearance levels: 1=public, 2=internal, 3=confidential, 4=restricted
    allowed_classifications = []
    level_map = {1: "public", 2: "internal", 3: "confidential", 4: "restricted"}
    for level in range(1, clearance_level + 1):
        if level in level_map:
            allowed_classifications.append(level_map[level])
    
    conditions = [
        FieldCondition(
            key="classification",
            match=MatchAny(any=allowed_classifications)
        )
    ]
    
    # Role-based project access
    if "engineering" in user_roles:
        conditions.append(
            FieldCondition(key="department", match=MatchAny(any=["engineering", "all"]))
        )
    elif "finance" in user_roles:
        conditions.append(
            FieldCondition(key="department", match=MatchAny(any=["finance", "all"]))
        )
    
    return Filter(must=conditions)


def authorized_search(
    collection_name: str,
    query_vector: list[float],
    user_roles: list[str],
    clearance_level: int,
    top_k: int = 10
) -> list[dict]:
    access_filter = build_access_filter(user_roles, clearance_level)
    
    results = client.search(
        collection_name=collection_name,
        query_vector=query_vector,
        query_filter=access_filter,
        limit=top_k,
        with_payload=True
    )
    
    # Secondary enforcement: verify returned results match expected criteria
    # Defense in depth — catch any filter bypass at application layer
    verified_results = []
    for result in results:
        payload = result.payload or {}
        doc_classification = payload.get("classification", "restricted")
        
        if doc_classification not in [
            "public", "internal", "confidential", "restricted"
        ][:clearance_level]:
            # Log this as a potential security anomaly and skip
            print(f"SECURITY: Unexpected classification {doc_classification} for user clearance {clearance_level}")
            continue
        
        verified_results.append({
            "id": result.id,
            "score": result.score,
            "content": payload.get("content", ""),
            "classification": doc_classification
        })
    
    return verified_results

The double-verification pattern — filter at the database level AND check again at the application layer — is defense in depth. It adds a small amount of application-side overhead but ensures that a misconfigured index, a library version change that silently drops filter syntax, or an API contract change doesn't result in unauthorized data exposure.

API Key Management and Least-Privilege Design

Every managed vector database issues API keys that carry full read/write/delete privileges by default. For production systems serving multiple users or integration points, this is architecturally equivalent to sharing a database superuser password. The correct approach is to issue the minimum credential required for each integration point.

Pinecone distinguishes between Admin keys (can create/delete indexes) and project-scoped keys (read/write to indexes within a project). Weaviate supports API key authentication with role assignments. Qdrant Cloud has JWT-based API keys with optional scope restrictions.

For systems where the vector database doesn't offer per-namespace API key scoping, the standard pattern is a service mesh or API gateway that:

1. Authenticates the calling service or user with your identity provider

2. Maps the identity to permitted namespaces and operations

3. Injects the appropriate vector DB API key into the upstream request

4. Logs the operation for audit
