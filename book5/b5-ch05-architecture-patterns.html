Chapter 5: Cloud Database Architecture Patterns

Cloud databases don't just move your workload to someone else's hardware — they introduce architectural patterns that simply don't exist in on-premises environments. Multi-AZ failover that completes in 20 seconds, read replicas that spin up in minutes, serverless capacity that scales from zero to 128 ACUs without a DBA touching a single configuration file: these capabilities change how you design systems from the ground up. This chapter covers the core architectural patterns you'll implement on AWS RDS, Azure SQL, and Google Cloud SQL, maps them to what you already know from on-premises work, and shows you exactly where managed services expand your options and where they constrain them.

---

5.1 High Availability and Failover Architecture

#### From Manual Standbys to Managed HA

On premises, you built HA with Pacemaker/Corosync clusters, SQL Server Always On Availability Groups with manual witness configuration, or PostgreSQL streaming replication with Patroni. You managed VIPs, fencing, and split-brain scenarios yourself. In the cloud, the managed HA layer abstracts most of this — but understanding what's happening underneath matters when things go wrong at 2 AM.

AWS RDS Multi-AZ deploys a synchronous standby replica in a separate Availability Zone. The standby is not a readable replica — it exists solely for failover. RDS uses a DNS endpoint; when failover occurs, the CNAME flips from primary to standby. Client applications must handle the brief DNS TTL-based reconnect window, typically 20–40 seconds. The standby is on different physical hardware, a different power domain, and in some cases a different facility within the same AWS region. Your application connection string doesn't change; only the IP behind the DNS name does.

Aurora goes further. Aurora Multi-AZ isn't traditional synchronous replication — the storage layer itself is shared across six copies in three AZs. The "replica" Aurora creates is a hot standby that already has the data; failover is a matter of promoting the reader endpoint to writer, not copying data. Aurora failover typically completes in under 30 seconds with no data loss because Aurora replicas share the storage layer with the primary. This is meaningfully different from RDS Multi-AZ, which is block-level replication of the EBS volume.

Azure SQL offers multiple HA tiers. The General Purpose tier uses remote storage with a hot standby; failover takes 20–30 seconds. Business Critical tier maintains three synchronous replicas locally (similar in concept to Always On AG with three synchronous partners) and achieves sub-10-second failover. Hyperscale uses a page server architecture with compute and storage fully separated, similar in philosophy to Aurora, and supports rapid replica addition.

Cloud SQL (GCP) uses regional instances with a standby in a second zone within the same region. Failover is automatic and typically completes in 60–120 seconds — meaningfully slower than Aurora or Azure Business Critical. AlloyDB for PostgreSQL, Google's premium offering, uses a Paxos-based distributed storage layer similar to Aurora, with failover under 60 seconds and readable replicas that share the underlying storage.

#### Configuring Multi-AZ on RDS

# Create a Multi-AZ RDS PostgreSQL instance
aws rds create-db-instance \
  --db-instance-identifier prod-postgres-primary \
  --db-instance-class db.r6g.4xlarge \
  --engine postgres \
  --engine-version 15.4 \
  --master-username dbadmin \
  --master-user-password "${DB_PASSWORD}" \
  --allocated-storage 500 \
  --storage-type gp3 \
  --iops 12000 \
  --storage-throughput 500 \
  --multi-az \
  --vpc-security-group-ids sg-0a1b2c3d4e5f \
  --db-subnet-group-name prod-db-subnet-group \
  --backup-retention-period 7 \
  --preferred-backup-window "03:00-04:00" \
  --preferred-maintenance-window "sun:05:00-sun:06:00" \
  --deletion-protection \
  --enable-performance-insights \
  --performance-insights-retention-period 7 \
  --no-publicly-accessible

# Verify Multi-AZ status and get endpoint
aws rds describe-db-instances \
  --db-instance-identifier prod-postgres-primary \
  --query 'DBInstances[0].{MultiAZ:MultiAZ,Endpoint:Endpoint.Address,SecondaryAZ:SecondaryAvailabilityZone,Status:DBInstanceStatus}'

#### Configuring Aurora Cluster HA with Terraform

# Aurora PostgreSQL cluster with Multi-AZ writer and readers
resource "aws_rds_cluster" "prod_aurora" {
  cluster_identifier     = "prod-aurora-postgres"
  engine                 = "aurora-postgresql"
  engine_version         = "15.4"
  database_name          = "proddb"
  master_username        = "dbadmin"
  master_password        = var.db_password
  
  # Storage is automatically distributed across 3 AZs
  availability_zones     = ["us-east-1a", "us-east-1b", "us-east-1c"]
  
  backup_retention_period   = 7
  preferred_backup_window   = "03:00-04:00"
  preferred_maintenance_window = "sun:05:00-sun:06:00"
  
  vpc_security_group_ids = [aws_security_group.aurora_sg.id]
  db_subnet_group_name   = aws_db_subnet_group.prod.name
  
  deletion_protection    = true
  skip_final_snapshot    = false
  final_snapshot_identifier = "prod-aurora-final-${formatdate("YYYY-MM-DD", timestamp())}"
  
  enabled_cloudwatch_logs_exports = ["postgresql"]
  
  db_cluster_parameter_group_name = aws_rds_cluster_parameter_group.prod.name
  
  tags = {
    Environment = "production"
    ManagedBy   = "terraform"
  }
}

# Writer instance — placed in us-east-1a
resource "aws_rds_cluster_instance" "writer" {
  identifier         = "prod-aurora-writer"
  cluster_identifier = aws_rds_cluster.prod_aurora.id
  instance_class     = "db.r6g.4xlarge"
  engine             = aws_rds_cluster.prod_aurora.engine
  engine_version     = aws_rds_cluster.prod_aurora.engine_version
  
  availability_zone           = "us-east-1a"
  performance_insights_enabled = true
  performance_insights_retention_period = 7
  
  auto_minor_version_upgrade = false
}

# Reader instances — one each in 1b and 1c for AZ redundancy
resource "aws_rds_cluster_instance" "readers" {
  count              = 2
  identifier         = "prod-aurora-reader-${count.index + 1}"
  cluster_identifier = aws_rds_cluster.prod_aurora.id
  instance_class     = "db.r6g.2xlarge"
  engine             = aws_rds_cluster.prod_aurora.engine
  engine_version     = aws_rds_cluster.prod_aurora.engine_version
  
  availability_zone  = count.index == 0 ? "us-east-1b" : "us-east-1c"
  
  performance_insights_enabled = true
  performance_insights_retention_period = 7
}

#### What You Can't Control

On a self-managed PostgreSQL cluster, you control the exact failover timing: synchronous_commit, synchronous_standby_names, and the Patroni TTL settings. In RDS Multi-AZ, you don't choose the replication protocol — it's block-level synchronous replication of EBS volumes managed by AWS. You can't query the replication lag on the standby because it isn't a readable PostgreSQL replica; it's a shadow volume. You can't apply patches independently to the standby before promoting it. You accept these constraints in exchange for not managing the HA stack yourself.

The one genuine gotcha: RDS Multi-AZ failover requires DNS TTL propagation. If your application has hardcoded IP addresses or DNS caches with long TTLs, failover won't be transparent. Always configure your JDBC, psycopg2, or .NET connection strings to use the RDS DNS endpoint and set connect_timeout values that allow for reconnection. In Java, set socketTimeout=30 and implement retry logic — don't assume the managed service makes reconnection unnecessary.

5.2 Read Scaling and Replica Patterns

#### Shifting the Mental Model

On premises, adding a read replica meant provisioning a server, configuring streaming replication, managing replication slots, and worrying about max_replication_slots and max_wal_senders. In the cloud, replicas are a billing line item — you add them with a CLI call. The architectural question shifts from "can we afford the operational overhead?" to "how do we route traffic, manage replica lag, and avoid read-your-own-writes consistency bugs?"

RDS Read Replicas are asynchronous replicas using native PostgreSQL streaming replication or SQL Server log shipping. They're readable, each with their own endpoint, and can be promoted to standalone instances. You can have up to five read replicas per RDS instance. Replication lag is the same concern you had on-premises: under normal conditions you'll see sub-second lag, but long-running transactions on the primary that generate heavy WAL, or large bulk operations, can push lag to minutes. Monitor ReplicaLag in CloudWatch and alert at 30 seconds; investigate at 60 seconds.

Aurora Replicas work differently. Because all Aurora instances share the same storage layer, a reader instance doesn't need to ship or apply WAL — it reads the same storage pages the writer uses, with a lag that reflects only page cache propagation. Aurora replica lag is typically under 20ms for reads. You can have up to 15 Aurora replicas. This is a genuinely different operational experience from PostgreSQL streaming replication, and it means Aurora replicas are appropriate for latency-sensitive reporting workloads that RDS read replicas are not.

Azure SQL Hyperscale supports named replicas (up to 30) that also share the underlying page server storage. This is Microsoft's architectural answer to Aurora's storage disaggregation. The General Purpose and Business Critical tiers support geo-replicas for disaster recovery, but they have higher replication lag than Hyperscale named replicas.

#### RDS Proxy for Connection Pooling

One on-premises pattern that doesn't translate cleanly to RDS is PgBouncer or pgpool-II sitting in front of your replica pool. RDS Proxy is the managed alternative. It maintains a connection pool to your RDS or Aurora instances and exposes a single endpoint to applications. When you have Lambda functions or containerized microservices opening thousands of short-lived connections — a workload that would exhaust max_connections on PostgreSQL — RDS Proxy pins connections to backend instances and handles multiplexing.

# Create RDS Proxy in front of Aurora cluster
aws rds create-db-proxy \
  --db-proxy-name prod-aurora-proxy \
  --engine-family POSTGRESQL \
  --auth '[
    {
      "AuthScheme": "SECRETS",
      "SecretArn": "arn:aws:secretsmanager:us-east-1:123456789:secret:prod/aurora/dbadmin",
      "IAMAuth": "REQUIRED"
    }
  ]' \
  --role-arn arn:aws:iam::123456789:role/rds-proxy-role \
  --vpc-subnet-ids subnet-0a1b2c3d subnet-0e5f6g7h subnet-0i9j0k1l \
  --vpc-security-group-ids sg-0proxy1234 \
  --require-tls

# Register the Aurora cluster as the proxy target
aws rds register-db-proxy-targets \
  --db-proxy-name prod-aurora-proxy \
  --db-cluster-identifiers prod-aurora-postgres

# Create a separate proxy endpoint for read replicas
aws rds create-db-proxy-endpoint \
  --db-proxy-name prod-aurora-proxy \
  --db-proxy-endpoint-name prod-aurora-proxy-reader \
  --vpc-subnet-ids subnet-0a1b2c3d subnet-0e5f6g7h subnet-0i9j0k1l \
  --vpc-security-group-ids sg-0proxy1234 \
  --target-role READ_ONLY

RDS Proxy has a real limitation that catches DBAs by surprise: it doesn't support all PostgreSQL client features. COPY commands, LISTEN/NOTIFY, pg_terminate_backend(), and SET LOCAL within certain transaction modes behave unexpectedly through the proxy. If your application relies on advisory locks or NOTIFY-based event queuing, test against the proxy endpoint explicitly before deploying to production. The proxy also pins connections (holds a dedicated backend connection for a client) when it detects certain session-state changes, which reduces the multiplexing benefit. Keep transactions short and avoid session-level SET commands that force connection pinning.

#### Aurora Global Database for Cross-Region Reads

For applications serving multiple geographic regions, Aurora Global Database adds cross-region replication with typical lag under one second. This is different from RDS cross-region read replicas, which use the native PostgreSQL replication stream and typically show 2–10 seconds of lag depending on bandwidth.

# boto3: describe Aurora Global Cluster replication lag
import boto3

rds = boto3.client('rds', region_name='us-east-1')

response = rds.describe_global_clusters(
    GlobalClusterIdentifier='prod-global-cluster'
)

for cluster in response['GlobalClusters']:
    print(f"Global Cluster: {cluster['GlobalClusterIdentifier']}")
    print(f"Status: {cluster['Status']}")
    print(f"Engine: {cluster['EngineVersion']}")
    
    for member in cluster['GlobalClusterMembers']:
        print(f"  Member: {member['DBClusterArn']}")
        print(f"  Is Writer: {member['IsWriter']}")
        
        if not member['IsWriter']:
            # Fetch replication lag from the secondary region's CloudWatch
            secondary_region = member['DBClusterArn'].split(':')[3]
            cw = boto3.client('cloudwatch', region_name=secondary_region)
            
            lag_response = cw.get_metric_statistics(
                Namespace='AWS/RDS',
                MetricName='AuroraGlobalDBReplicationLag',
                Dimensions=[{'Name': 'DBClusterIdentifier', 
                             'Value': member['DBClusterArn'].split(':')[-1]}],
                StartTime='2024-01-15T00:00:00Z',
                EndTime='2024-01-15T01:00:00Z',
                Period=60,
                Statistics=['Average', 'Maximum']
            )
            
            for point in lag_response['Datapoints'][-5:]:
                print(f"  Lag: avg={point['Average']:.1f}ms max={point['Maximum']:.1f}ms")

5.3 Storage Architecture and Auto-Scaling

#### Cloud Storage Is Not a SAN

On premises, your DBA team provisioned LUNs from a SAN, set RAID levels, tuned stripe sizes, and called the storage team when you needed more space — a process that took days. In AWS RDS, storage is gp3 or io2 backed by networked EBS volumes. The performance characteristics are fundamentally different from local NVMe, and understanding those differences prevents over-provisioning and unexpected I/O bottlenecks.

RDS gp3 provides a baseline of 3,000 IOPS and 125 MB/s throughput for any volume, regardless of size. Unlike gp2 (which was 3 IOPS/GB with a burst bucket), gp3 decouples IOPS from capacity. You can have a 100GB volume with 12,000 IOPS if you need it. Maximum gp3 performance is 16,000 IOPS and 1,000 MB/s. For PostgreSQL workloads with heavy random I/O — OLTP with small row updates, index-heavy queries — gp3 at 12,000–16,000 IOPS covers the majority of production workloads on instances up to db.r6g.8xlarge.

RDS io2 is the choice for I/O-intensive OLTP — think 50,000–256,000 IOPS, 4,000 MB/s throughput. At these levels you're typically also running db.r6g.16xlarge or db.r6i.32xlarge, where the network bandwidth to the EBS volume (25–50 Gbps) becomes the binding constraint, not the storage tier itself.

Aurora storage auto-scales in 10GB increments up to 128TB and you pay only for what you use. There is no pre-provisioning step; Aurora handles storage growth automatically. However, Aurora storage doesn't shrink automatically — once expanded to 50TB, the billing floor is 50TB even if you delete data. Run VACUUM and monitor aurora_storage_utilization to understand actual versus allocated space.

Cloud SQL storage auto-increases by default: when utilization exceeds 90%, Cloud SQL automatically adds 25% more capacity (minimum 5GB increment) up to the configured maximum. You can disable this and manage storage manually, but for production workloads the auto-increase behavior is generally preferable.

-- Query Aurora storage utilization (run from psql connected to Aurora)
SELECT 
    current_setting('server_version') AS pg_version,
    pg_size_pretty(sum(pg_database_size(datname))) AS total_logical_size,
    -- Aurora-specific storage stats via aurora_stat_utils extension
    pg_size_pretty(
        (SELECT sum(allocated_storage) 
         FROM aurora_storage_object_stats() 
         WHERE object_type = 'data')
    ) AS aurora_allocated_storage;

-- Monitor bloat that inflates Aurora storage costs
SELECT 
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS total_size,
    pg_size_pretty(
        pg_total_relation_size(schemaname||'.'||tablename) - 
        pg_relation_size(schemaname||'.'||tablename)
    ) AS bloat_estimate,
    n_dead_tup,
    last_autovacuum
FROM pg_stat_user_tables
WHERE n_dead_tup > 100000
ORDER BY n_dead_tup DESC
LIMIT 20;

#### Storage Scaling Gotchas

RDS storage scaling is one-directional and time-bounded. You can increase allocated storage, but you cannot decrease it. Once you scale from 500GB to 1TB, your minimum billing is 1TB from that point on. More critically, during an RDS storage scale operation on Multi-AZ instances, the process scales the standby first, then performs a failover, then scales the former primary (now standby). The entire operation can take 30–60 minutes for large volumes, during which a maintenance window effectively exists for any workload that can't tolerate the failover window.

For Azure SQL, storage scales online within the same service tier with no downtime for Business Critical and General Purpose. The Hyperscale tier is unique: Hyperscale storage auto-scales to 100TB with no size pre-configuration required, and this is genuinely transparent to the application. The Hyperscale architecture separates log service, page servers, and compute nodes so that storage scaling doesn't touch the compute layer at all.

5.4 Network Architecture: VPC, Private Link, and Connectivity

#### The pg_hba.conf Replacement

On premises, pg_hba.conf controlled which hosts could connect to PostgreSQL and with what authentication method. In the cloud, that control is split across multiple layers, and understanding which layer does what determines whether you have a security gap.

VPC Security Groups (AWS) act as stateful host-based firewalls at the instance level. You configure inbound rules that specify source CIDR ranges, specific security group IDs, or individual IPs. For a production RDS instance, you should never open port 5432 to 0.0.0.0/0. The pattern is:

1. RDS instance lives in a private subnet with no internet gateway route

2. Application servers have a security group (e.g., sg-app-tier)

3. RDS security group allows inbound 5432 only from sg-app-tier

4. Any DBA access uses a bastion host, VPN, or AWS Systems Manager Session Manager tunneling

# Create an RDS security group that allows only from the app tier security group
aws ec2 create-security-group \
  --group-name prod-rds-sg \
  --description "Production RDS - allow from app tier only" \
  --vpc-id vpc-0a1b2c3d4e5f6789

export RDS_SG_ID=$(aws ec2 describe-security-groups \
  --filters Name=group-name,Values=prod-rds-sg \
  --query 'SecurityGroups[0].GroupId' \
  --output text)

# Allow PostgreSQL port from app security group only
aws ec2 authorize-security-group-ingress \
  --group-id ${RDS_SG_ID} \
  --protocol tcp \
  --port 5432 \
  --source-group sg-0app12345678

# Verify no public access rule exists
aws ec2 describe-security-groups \
  --group-ids ${RDS_SG_ID} \
  --query 'SecurityGroups[0].IpPermissions'

# Create DB subnet group using private subnets only
aws rds create-db-subnet-group \
  --db-subnet-group-name prod-private-subnets \
  --db-subnet-group-description "Private subnets for production RDS" \
  --subnet-ids subnet-private-1a subnet-private-1b subnet-private-1c

AWS PrivateLink solves a specific architectural problem: you need to access an RDS instance from a different VPC or from an on-premises network via Direct Connect without exposing the database to the public internet or setting up full VPC peering. PrivateLink creates an interface VPC endpoint with a private IP in your VPC that proxies to the RDS endpoint. This is the right pattern for:

  • → SaaS applications that need to query a customer's database
  • → Analytics platforms in a centralized VPC querying databases in spoke VPCs
  • → On-premises applications connecting through Direct Connect to RDS

    • Azure SQL network isolation uses service endpoints (simple, lower complexity) or Private Endpoint (production recommended). With a Private Endpoint, Azure SQL gets a private IP in your VNet. The Azure equivalent of VPC security groups is Network Security Groups (NSGs), and Azure SQL also supports a firewall rule layer at the server level — you configure allowed IP ranges and VNet rules. Unlike AWS, Azure SQL has a server-level "Allow Azure services and resources to access this server" toggle that is enabled by default and should be disabled in production environments where all connectivity is via Private Endpoint.
