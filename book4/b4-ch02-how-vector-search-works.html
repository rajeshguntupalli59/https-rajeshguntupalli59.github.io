Chapter 2: How Vector Search Works: ANN vs Exact Search

Vector search is the engine underneath every modern AI application — the mechanism that turns a user's question into a ranked list of relevant documents, products, or records. If you have spent your career building indexes on B-trees and tuning sequential scan thresholds, vector search will feel both familiar and disorienting. The data structures are different, the query semantics are different, and the performance tradeoffs are inverted in ways that will surprise you. This chapter walks through exactly how vector search works at the algorithmic level, why approximate nearest neighbor search dominates production systems, and how to make informed decisions about which algorithm and which tool to deploy for a given workload.

---

The Geometry of Similarity: What Vector Search Is Actually Doing

Before touching any index structure, it helps to understand what the database is computing when it runs a vector query. A vector is simply an array of floating-point numbers — typically 384 to 3072 dimensions depending on the embedding model. Each number represents some learned feature of the original content. You do not interpret the individual numbers the way you would a column value. The meaning lives in the geometric relationships between vectors.

When a user asks "what are the symptoms of iron deficiency?", an embedding model converts that string into a vector. The database then finds vectors in storage that are geometrically close to that query vector. Closeness is measured by a distance or similarity function. Three are in common use:

Cosine similarity measures the angle between two vectors, ignoring magnitude. It is the default choice for text embeddings because the same concept expressed in a long document versus a short sentence should still score as similar.

Euclidean distance (L2) measures straight-line distance in the high-dimensional space. It works well for image embeddings and some multimodal models, but is sensitive to vector magnitude.

Dot product is the raw inner product without normalization. When vectors are unit-normalized, dot product and cosine similarity produce identical rankings. Many approximate index structures optimize for dot product because it is cheaper to compute at scale.

The choice of distance function is not just a configuration option — it changes your results. A system tuned with cosine similarity and then switched to L2 will return a different ranked list for the same query. This matters when you are integrating vector search with an existing scoring pipeline.

# Python example — computing distances manually to build intuition
import numpy as np

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

def euclidean_distance(a, b):
    return np.linalg.norm(np.array(a) - np.array(b))

def dot_product(a, b):
    return np.dot(a, b)

# Simulating two embedding vectors (normally 384–3072 dims; using 8 here for clarity)
query_vec   = np.array([0.12, 0.87, 0.34, 0.56, 0.91, 0.23, 0.67, 0.45])
doc_vec_a   = np.array([0.11, 0.85, 0.36, 0.54, 0.89, 0.25, 0.65, 0.43])
doc_vec_b   = np.array([0.90, 0.10, 0.80, 0.05, 0.12, 0.95, 0.07, 0.88])

print("=== Cosine Similarity ===")
print(f"Query vs Doc A: {cosine_similarity(query_vec, doc_vec_a):.4f}")  # ~0.9998
print(f"Query vs Doc B: {cosine_similarity(query_vec, doc_vec_b):.4f}")  # ~0.45

print("\n=== Euclidean Distance (lower = closer) ===")
print(f"Query vs Doc A: {euclidean_distance(query_vec, doc_vec_a):.4f}")  # ~0.057
print(f"Query vs Doc B: {euclidean_distance(query_vec, doc_vec_b):.4f}")  # ~1.38

print("\n=== Dot Product ===")
print(f"Query vs Doc A: {dot_product(query_vec, doc_vec_a):.4f}")
print(f"Query vs Doc B: {dot_product(query_vec, doc_vec_b):.4f}")

One concept that trips up database engineers is the curse of dimensionality. In low-dimensional spaces (2D, 3D), nearest neighbor search is intuitive and B-tree-style partitioning works well. As dimensions grow past roughly 20–30, the distance between any two random points converges toward the same value. By the time you reach 768 or 1536 dimensions — standard for OpenAI's text-embedding-3-small and text-embedding-3-large — traditional spatial indexes like R-trees become nearly useless. Every point looks roughly equidistant from every other point when you measure raw L2, which is why embedding models are designed so that semantically similar content clusters in specific regions of the space despite the high dimensionality.

Exact Search (KNN): When Brute Force Is the Right Answer

Exact k-nearest neighbor search is conceptually simple: compute the distance from the query vector to every vector in the dataset and return the k smallest distances. This guarantees 100% recall — you will never miss a closer neighbor. In database terms, it is a full table scan with a computed distance column, sorted and limited.

pgvector supports exact search natively. If you create a table with a vector column and do not create an HNSW or IVFFlat index, every query runs exact KNN against the full table.

-- PostgreSQL / pgvector
-- Exact KNN search — no index, guaranteed 100% recall
-- Assumes: table documents(id, content, embedding vector(1536))

SELECT
    id,
    content,
    1 - (embedding <=> query.vec) AS cosine_similarity
FROM
    documents,
    (SELECT '[0.12, 0.87, 0.34, ...]'::vector(1536) AS vec) AS query
ORDER BY
    embedding <=> query.vec   -- <=> is cosine distance in pgvector
LIMIT 10;

-- For L2 distance use <-> operator
-- SELECT id, embedding <-> '[...]'::vector AS l2_dist FROM documents ORDER BY l2_dist LIMIT 10;

-- For inner product use <#> operator (returns negative dot product; negate for similarity)
-- SELECT id, -(embedding <#> '[...]'::vector) AS dot_sim FROM documents ORDER BY dot_sim DESC LIMIT 10;

When is exact search the right answer? More often than you might think:

  • → **Small datasets**: Under 100,000 vectors with 1536 dimensions, exact search on modern NVMe storage with pgvector typically completes in under 100ms. You do not need an ANN index.
  • → **High-recall requirements**: Compliance, medical, and legal workloads sometimes cannot accept the recall tradeoff that ANN introduces. Exact search is the only way to guarantee you found the actual nearest neighbors.
  • → **Development and evaluation**: When benchmarking your embedding model or tuning your chunking strategy, exact search gives you ground truth. Use it to measure recall of your ANN configuration.
  • → **Rarely queried, frequently updated datasets**: ANN indexes must be rebuilt or updated as data changes. If your dataset changes constantly and is queried infrequently, maintaining an HNSW index costs more than the query savings justify.

    • The performance ceiling of exact search is real. A table of 1 million 1536-dimensional vectors requires computing approximately 1.536 billion floating-point operations per query. On a modern CPU with AVX-512 SIMD instructions (which pgvector uses when available), that runs in roughly 3–8 seconds for a cold query on a single node — too slow for a user-facing application. At 10 million vectors, exact search becomes impractical regardless of hardware. This is where approximate search earns its place.

    Approximate Nearest Neighbor Search: The Algorithmic Landscape

    Approximate nearest neighbor (ANN) search accepts a controlled recall loss in exchange for query latency that is orders of magnitude lower than exact search. The term "approximate" means the index might miss some true nearest neighbors — it returns results that are close, but not guaranteed to be the absolute closest. In practice, a well-tuned ANN index returns 95–99% of the true nearest neighbors at 5–50ms latency on datasets of tens of millions of vectors.

    Three families of algorithms dominate production systems. Understanding how each works lets you read vendor documentation and benchmark results critically rather than accepting default configurations.

    IVFFlat (Inverted File Index with Flat Quantization)

    IVFFlat divides the vector space into nlist Voronoi cells (clusters) using k-means clustering during index build. Each vector is assigned to the nearest cluster centroid. At query time, the algorithm identifies the nprobe closest centroids to the query vector and performs exact search only within those cells.

    Think of it as a hash partition scheme, but the partitioning function is "geometric proximity" instead of a hash function. If your data has 1 million vectors and you build with nlist=1000, each cell contains roughly 1000 vectors. Setting nprobe=10 means the query searches approximately 10,000 vectors instead of 1,000,000 — a 100x reduction in distance computations.

    The tradeoff: k-means clustering happens at index build time. If your data distribution shifts significantly after the index is built (new product categories, new languages, domain shift in embeddings), recall degrades. IVFFlat requires periodic index rebuilds to stay healthy.

    HNSW (Hierarchical Navigable Small World)

    HNSW is the algorithm behind most production vector databases today, including pgvector (since version 0.5.0), Weaviate, Qdrant, and Milvus. It constructs a layered graph where each vector is a node connected to its approximate nearest neighbors. The graph is hierarchical — upper layers have fewer nodes with longer-range connections, lower layers are dense with short-range connections. Queries start at the top layer and greedily descend toward the query vector.

    For database engineers, HNSW is analogous to a B-tree in one important way: it supports efficient incremental inserts without full rebuilds. Unlike IVFFlat, you can add vectors to an HNSW index and they are integrated into the graph structure at insert time. The ef_construction parameter controls how aggressively the graph is built (higher values = better recall but slower inserts), and m controls the number of bidirectional links per node (higher values = better recall and faster queries, but larger index size).

    Product Quantization (PQ) and Scalar Quantization (SQ)

    Quantization is a compression layer applied on top of IVFFlat or HNSW. Rather than storing full 32-bit floats for each dimension, vectors are compressed to smaller representations — sometimes 8x to 32x smaller. This dramatically reduces memory requirements and can improve query throughput because more vectors fit in CPU cache.

    The tradeoff is recall loss from quantization error plus additional complexity in tuning. Pinecone and Weaviate apply quantization transparently. In pgvector 0.7+, halfvec support (16-bit floats) gives you a 2x storage reduction with minimal recall impact.

    The table below summarizes the practical characteristics:

    | Property | Exact KNN | IVFFlat | HNSW |

    |---|---|---|---|

    | Recall | 100% | 85–98% (tunable) | 90–99% (tunable) |

    | Query latency (1M vecs) | 3–8s | 5–50ms | 1–20ms |

    | Build time | None | Fast (k-means) | Slow (graph construction) |

    | Incremental inserts | Instant | Requires rebuild | Supported natively |

    | Memory overhead | Low | Moderate | High (graph links) |

    | Best for | <100K vectors | Batch workloads, stable datasets | Production real-time search |

    HNSW in pgvector and Weaviate: Configuration That Actually Matters

    HNSW is where most production teams spend their tuning time. The two parameters that dominate behavior are m and ef_construction at build time, and ef (also called ef_search or hnsw.ef_search) at query time.

    -- PostgreSQL / pgvector
-- Creating an HNSW index with production-appropriate parameters

-- First, set memory for index construction (run before CREATE INDEX)
SET maintenance_work_mem = '4GB';

-- Create the HNSW index
-- m: number of connections per node (default 16; range 4–64)
-- ef_construction: search depth during build (default 64; range 4–1000)
CREATE INDEX ON documents
USING hnsw (embedding vector_cosine_ops)
WITH (m = 24, ef_construction = 128);

-- At query time, control recall vs speed tradeoff
-- Higher ef_search = better recall, slower queries (default 40)
SET hnsw.ef_search = 100;

-- Now run the approximate nearest neighbor query
SELECT
    d.id,
    d.content,
    d.metadata->>'source' AS source,
    1 - (d.embedding <=> $1::vector) AS similarity
FROM documents d
WHERE similarity > 0.75        -- pre-filter: only high-similarity results
ORDER BY d.embedding <=> $1::vector
LIMIT 5;

-- Check if your index is being used
EXPLAIN (ANALYZE, BUFFERS)
SELECT id FROM documents
ORDER BY embedding <=> '[0.1, 0.2, ...]'::vector
LIMIT 5;
-- Look for "Index Scan using documents_embedding_idx" in output
-- If you see "Seq Scan", increase enable_seqscan=off or check vector dimensions match

    The interaction between m, ef_construction, and ef_search is the core tuning loop. A reasonable starting point for a production RAG application with 1–5 million vectors:

  • → `m = 16` (default): acceptable for most workloads
  • → `m = 24–32`: better recall, ~20% larger index, measurably faster queries on high-traffic systems because the graph navigation converges faster
  • → `ef_construction = 128`: good balance of build time and recall; doubling to 256 costs roughly 2x build time for marginal recall improvement beyond 99%
  • → `ef_search = 64–200` at query time: tune this per query type based on latency SLA

    • Weaviate exposes the same HNSW parameters through its schema configuration:

    # Python example — configuring HNSW in Weaviate with the v4 Python client

import weaviate
import weaviate.classes as wvc

client = weaviate.connect_to_local()  # or connect_to_weaviate_cloud(cluster_url=...)

# Define a collection with explicit HNSW configuration
client.collections.create(
    name="DocumentChunks",
    vectorizer_config=wvc.config.Configure.Vectorizer.text2vec_openai(
        model="text-embedding-3-small",
        dimensions=1536
    ),
    vector_index_config=wvc.config.Configure.VectorIndex.hnsw(
        distance_metric=wvc.config.VectorDistances.COSINE,
        ef_construction=128,       # build-time recall vs speed
        max_connections=24,        # equivalent to pgvector's m parameter
        ef=-1,                     # -1 = dynamic: Weaviate auto-tunes ef at query time
        dynamic_ef_min=100,
        dynamic_ef_max=500,
        dynamic_ef_factor=8,
    ),
    properties=[
        wvc.config.Property(name="content", data_type=wvc.config.DataType.TEXT),
        wvc.config.Property(name="source", data_type=wvc.config.DataType.TEXT),
        wvc.config.Property(name="chunk_index", data_type=wvc.config.DataType.INT),
    ]
)

print("Collection created with HNSW index")
client.close()

    One production reality that surprises engineers coming from relational databases: HNSW index memory requirements are not optional. pgvector loads the HNSW graph into shared_buffers or the OS page cache during queries. For a 1-million-vector, 1536-dimension dataset with m=16, expect approximately 3–6GB of index size on disk and similar working memory requirements for sustained query performance. On systems where the index does not fit in memory, query latency degrades sharply as the OS pages in graph nodes from disk. This is why managed vector database services like Pinecone and Qdrant Cloud advertise memory-optimized infrastructure — they are solving a real problem, not just upselling.

    IVFFlat for Batch Workloads and the Recall-Speed Tradeoff in Practice

    IVFFlat gets less attention than HNSW in the AI application ecosystem, but it has a legitimate place in workloads where you control data loading patterns and can tolerate periodic index rebuilds. The build process is faster than HNSW, and for very large datasets (50M+ vectors) the memory footprint is more predictable.

    The pgvector IVFFlat configuration follows a well-established rule of thumb: set nlist to sqrt(N) where N is the number of vectors, and set nprobe to nlist / 10 as a starting point for tuning.

    -- PostgreSQL / pgvector
-- IVFFlat index configuration and recall tuning

-- For a dataset of 1 million vectors:
-- sqrt(1,000,000) = 1000 → nlist = 1000
-- Starting nprobe = 100 (10% of nlist)

-- IMPORTANT: Index should be created AFTER data is loaded
-- IVFFlat builds centroids from existing data — an empty-table index is useless
CREATE INDEX ON documents
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 1000);

-- At query time, tune nprobe for your latency vs recall SLA
-- More nprobe = higher recall, more latency
SET ivfflat.probes = 100;  -- searches 10% of clusters

-- Benchmark different nprobe values against exact search results
-- Run exact search first to capture ground truth
WITH exact_results AS (
    SELECT id, embedding <=> query_vec AS dist
    FROM documents, (SELECT embedding FROM documents WHERE id = 12345) AS q(query_vec)
    ORDER BY dist LIMIT 10
),
ann_results AS (
    SET LOCAL ivfflat.probes = 50;  -- test with probes=50
    SELECT id FROM documents
    ORDER BY embedding <=> (SELECT embedding FROM documents WHERE id = 12345)
    LIMIT 10
)
-- Count overlap to compute recall@10
SELECT
    COUNT(*) AS matching,
    COUNT(*) / 10.0 AS recall_at_10
FROM exact_results e
JOIN ann_results a USING (id);

-- Typical results:
-- probes=10  → recall@10 ~0.82, latency ~5ms
-- probes=50  → recall@10 ~0.94, latency ~18ms
-- probes=100 → recall@10 ~0.97, latency ~35ms
-- probes=200 → recall@10 ~0.99, latency ~65ms

    The choice between IVFFlat and HNSW in pgvector comes down to update patterns and memory budget. If you are loading a fixed dataset of embeddings (say, pre-computed embeddings for a product catalog that refreshes weekly), IVFFlat is a reasonable choice because it is built once and queries are efficient. If your dataset grows continuously — new documents indexed in real time, new user content, streaming embeddings — HNSW's incremental insert support makes it operationally far simpler.

    When you move beyond pgvector to dedicated vector databases, the algorithm choice is often abstracted away. Pinecone uses HNSW-family algorithms internally. Qdrant defaults to HNSW but exposes quantization configuration through its collection API:

    # Python example — Qdrant collection creation with HNSW and scalar quantization

from qdrant_client import QdrantClient
from qdrant_client.models import (
    Distance, VectorParams, HnswConfigDiff,
    ScalarQuantizationConfig, ScalarType, QuantizationConfig
)

client = QdrantClient(url="http://localhost:6333")

client.create_collection(
    collection_name="knowledge_base",
    vectors_config=VectorParams(
        size=1536,                     # dimensions must match embedding model output
        distance=Distance.COSINE,
        hnsw_config=HnswConfigDiff(
            m=16,
            ef_construct=128,
            full_scan_threshold=10_000,  # vectors below this count use exact search
            on_disk=False,               # keep graph in RAM for latency-sensitive queries
        )
    ),
    quantization_config=QuantizationConfig(
        scalar=ScalarQuantizationConfig(
            type=ScalarType.INT8,       # compress float32 to int8: 4x size reduction
            quantile=0.99,             # 99th percentile used for range calibration
            always_ram=True,           # keep quantized vectors in memory
        )
    )
)

print("Qdrant collection created with HNSW + INT8 quantization")
print("Expected memory reduction: ~4x vs raw float32 vectors")
print("Expected recall impact: ~1-3% depending on data distribution")

    The quantization numbers matter for cost estimation. A collection of 10 million 1536-dimensional float32 vectors occupies roughly 61GB of raw storage (10M × 1536 × 4 bytes). With INT8 scalar quantization, that drops to approximately 15GB. On managed cloud services billing by memory allocation, quantization is not a theoretical optimization — it is a budget decision.

    Recall, Latency, and Throughput: Benchmarking Your Search Configuration

    Reading vendor benchmarks is necessary but insufficient. The ANN Benchmarks project (ann-benchmarks.com) provides standardized recall-vs-throughput curves across algorithms and datasets, but the results only hold for the specific dimensionality and data distribution in the benchmark. Your production embeddings, generated by a specific model on a specific domain, will have a different recall curve.

    Every production vector search deployment should include a recall evaluation pipeline. The process is straightforward: run your production queries against both exact search and your ANN index, compute recall@k (what fraction of the true top-k results appear in the ANN results), and set ef_search or nprobe to meet your target.

    For most RAG applications, recall@10 above 0.95 is sufficient because the LLM synthesizes answers across retrieved chunks — missing one of ten relevant chunks rarely degrades the final answer quality. For semantic deduplication or recommendation systems where the exact ranking matters more, you may need recall@10 above 0.99, which pushes toward higher ef_search values and larger m configurations.

    Throughput is the metric that catches teams off guard at scale. A single pgvector query returning results in 15ms is easy to achieve. Sustaining 500 queries per second across concurrent connections is a different problem. At that scale, you are contending for CPU across parallel graph traversals, competing for shared buffer cache with simultaneous page faults, and hitting connection pool limits. This is the threshold where the architecture question shifts from "is HNSW fast enough?" to "should I be running Pinecone or Qdrant with horizontal sharding instead of pgvector on a single RDS instance?"

    The rough production thresholds based on real-world deployments:

  • → **pgvector on a single node** handles approximately 100–300 queries per second at <50ms latency for a 1–5 million vector HNSW index, depending on CPU cores, memory, and `ef_search`. Supabase and Neon offer managed pgvector and can vertically scale to handle moderate traffic.
  • → **Pinecone** in pod-based deployments is designed for thousands of queries per second across distributed indexes. The serverless tier optimizes for cost over raw throughput.
  • → **Qdrant** with multiple nodes and sharding has been benchmarked at 10,000+ queries per second at p99 latency under 20ms on appropriately provisioned hardware.
  • → **Weaviate** with its built-in HNSW implementation and optional product
