Chapter 9: Semantic Memory: Finding Relevant Past Conversations

Every conversation an AI application handles is potential context for future ones. A user who asked about deploying containers three weeks ago and is now asking about resource limits is carrying implicit context that a stateless system will never see. Semantic memory — the ability to surface relevant past exchanges based on meaning rather than exact keywords — transforms a chatbot that starts from zero on every request into one that accumulates understanding over time. This chapter covers how to build that system end-to-end using PostgreSQL as the memory store: schema design for conversation embeddings, vector similarity retrieval, hybrid ranking that combines semantic and recency signals, memory consolidation to prevent context overflow, and the production concerns that separate a demo from a deployed system.

---

9.1 What Semantic Memory Actually Means for LLM Applications

Before writing any SQL, it's worth being precise about what problem we're solving. There are three distinct memory problems in LLM applications, and confusing them leads to over-engineered solutions.

Short-term context is the conversation window — the messages you pass directly to the model in the current API call. This is limited by the model's context window and managed by your application logic. Nothing stored in PostgreSQL here.

Episodic memory is the record of past conversations — what the user said, what the assistant said, when, and in what context. This is fully storable in relational tables. You already have this if you're logging conversations.

Semantic memory is the ability to retrieve episodic records based on meaning. A user asking "how do I make my API faster?" should surface past conversations about caching, database query optimization, or load balancing — not just conversations that contain the literal phrase "make my API faster." This requires embeddings.

The unique advantage of PostgreSQL here is that semantic similarity search doesn't have to exist in isolation. You can combine it with user ownership checks, time decay, conversation metadata filtering, and the full power of relational joins — all in a single query. A pure vector database forces you to pre-filter or post-filter in application code, losing performance and atomicity. PostgreSQL gives you one coherent query that does everything.

Here's the foundation schema:

-- Foundation schema for semantic memory
-- Run this once per application; extend as needed

CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS pg_trgm;  -- for hybrid text search later

-- Users table (minimal; integrate with your auth system)
CREATE TABLE users (
    user_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    external_id    TEXT UNIQUE NOT NULL,  -- your auth provider's ID
    created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
    metadata       JSONB NOT NULL DEFAULT '{}'
);

-- Conversations: one row per session/thread
CREATE TABLE conversations (
    conversation_id   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id           UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    title             TEXT,                    -- generated or user-provided
    started_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    last_active_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    metadata          JSONB NOT NULL DEFAULT '{}',
    is_archived       BOOLEAN NOT NULL DEFAULT false
);

-- Individual messages within conversations
CREATE TABLE messages (
    message_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    conversation_id   UUID NOT NULL REFERENCES conversations(conversation_id) ON DELETE CASCADE,
    user_id           UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    role              TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system', 'tool')),
    content           TEXT NOT NULL,
    token_count       INTEGER,
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
    metadata          JSONB NOT NULL DEFAULT '{}'   -- model used, latency, cost, etc.
);

-- Embeddings for semantic retrieval
-- One embedding per conversation turn (user+assistant pair) or per message
CREATE TABLE message_embeddings (
    embedding_id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    message_id        UUID NOT NULL REFERENCES messages(message_id) ON DELETE CASCADE,
    conversation_id   UUID NOT NULL REFERENCES conversations(conversation_id) ON DELETE CASCADE,
    user_id           UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    embedding         vector(1536) NOT NULL,   -- OpenAI text-embedding-3-small dimensions
    embedded_text     TEXT NOT NULL,           -- what was actually embedded (may differ from raw message)
    model_name        TEXT NOT NULL DEFAULT 'text-embedding-3-small',
    created_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Index for fast ANN search
-- HNSW is preferred for production; IVFFlat works for smaller datasets
CREATE INDEX idx_message_embeddings_vector
    ON message_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- Supporting indexes
CREATE INDEX idx_message_embeddings_user
    ON message_embeddings (user_id, created_at DESC);

CREATE INDEX idx_messages_conversation
    ON messages (conversation_id, created_at ASC);

CREATE INDEX idx_conversations_user_active
    ON conversations (user_id, last_active_at DESC)
    WHERE is_archived = false;

This schema makes user_id available directly on message_embeddings — a denormalization that matters. Every ANN query for a specific user can filter on user_id before or alongside the vector search, dramatically narrowing the index scan. Without it, you'd retrieve globally similar vectors and then join/filter in application code.

9.2 Embedding Strategy: What to Encode and When

The quality of semantic retrieval depends entirely on what you embed. Embedding raw message content verbatim is the baseline, but it has real problems: very short messages ("yes", "thanks", "can you retry?") produce noisy embeddings, and assistant responses are often long and topically diffuse.

A more robust strategy is to embed *conversation turns* — a user message paired with its assistant response — with a lightweight summarization step for long exchanges. This gives the embedding model enough context to capture intent and resolution, not just surface text.

Here's a Python service that handles embedding on message completion:

# memory_embedder.py
# Embeds conversation turns after each assistant response
# Run as a Celery task or directly from your message handler

import asyncio
import asyncpg
import openai
import anthropic
from dataclasses import dataclass
from typing import Optional
import hashlib
import json

@dataclass
class ConversationTurn:
    message_id: str          # the assistant's message_id (we embed the full turn)
    conversation_id: str
    user_id: str
    user_content: str
    assistant_content: str
    model_used: str


async def build_embeddable_text(turn: ConversationTurn) -> str:
    """
    Construct the text we'll actually embed.
    For short turns, use verbatim content.
    For long turns, summarize the assistant response first.
    """
    MAX_VERBATIM_CHARS = 2000
    
    user_part = f"User: {turn.user_content.strip()}"
    
    if len(turn.assistant_content) <= MAX_VERBATIM_CHARS:
        assistant_part = f"Assistant: {turn.assistant_content.strip()}"
    else:
        # Summarize long assistant responses using Claude
        # This keeps the embedding focused on the key information
        client = anthropic.Anthropic()
        summary_response = client.messages.create(
            model="claude-opus-4-5",
            max_tokens=200,
            messages=[{
                "role": "user",
                "content": (
                    f"Summarize this assistant response in 2-3 sentences, "
                    f"preserving the main topic and key points:\n\n{turn.assistant_content}"
                )
            }]
        )
        summary = summary_response.content[0].text
        assistant_part = f"Assistant (summary): {summary.strip()}"
    
    return f"{user_part}\n{assistant_part}"


async def embed_conversation_turn(
    pool: asyncpg.Pool,
    openai_client: openai.AsyncOpenAI,
    turn: ConversationTurn
) -> Optional[str]:
    """
    Embed a conversation turn and store in message_embeddings.
    Returns embedding_id on success, None if already embedded.
    """
    embeddable_text = await build_embeddable_text(turn)
    
    # Generate embedding
    response = await openai_client.embeddings.create(
        model="text-embedding-3-small",
        input=embeddable_text,
        encoding_format="float"
    )
    embedding_vector = response.data[0].embedding
    
    # Store in PostgreSQL
    # Using ON CONFLICT to make this idempotent — safe to retry
    async with pool.acquire() as conn:
        result = await conn.fetchrow("""
            INSERT INTO message_embeddings (
                message_id,
                conversation_id,
                user_id,
                embedding,
                embedded_text,
                model_name
            )
            VALUES ($1, $2, $3, $4::vector, $5, $6)
            ON CONFLICT (message_id) DO NOTHING
            RETURNING embedding_id
        """,
            turn.message_id,
            turn.conversation_id,
            turn.user_id,
            str(embedding_vector),  # asyncpg accepts vector as string repr
            embeddable_text,
            "text-embedding-3-small"
        )
        
        if result:
            # Update conversation last_active_at
            await conn.execute("""
                UPDATE conversations
                SET last_active_at = now()
                WHERE conversation_id = $1
            """, turn.conversation_id)
            
            return str(result['embedding_id'])
        
        return None  # Already embedded (idempotent)


# Celery task version for async processing
# celery_tasks.py

from celery import Celery
import asyncio
import asyncpg
import openai

app = Celery('memory_tasks', broker='redis://localhost:6379/0')

@app.task(bind=True, max_retries=3, default_retry_delay=5)
def embed_turn_async(self, turn_data: dict, db_url: str, openai_api_key: str):
    """
    Celery task: embed a conversation turn without blocking the response path.
    Called after the assistant response is stored but before returning to the user.
    """
    async def _run():
        pool = await asyncpg.create_pool(db_url, min_size=1, max_size=3)
        client = openai.AsyncOpenAI(api_key=openai_api_key)
        turn = ConversationTurn(**turn_data)
        
        try:
            await embed_conversation_turn(pool, client, turn)
        finally:
            await pool.close()
    
    try:
        asyncio.run(_run())
    except Exception as exc:
        raise self.retry(exc=exc)

Note the ON CONFLICT (message_id) DO NOTHING — you need a unique constraint on message_id in message_embeddings for this to work:

-- Add after initial schema creation
ALTER TABLE message_embeddings
    ADD CONSTRAINT uq_message_embeddings_message_id UNIQUE (message_id);

9.3 Retrieval: Hybrid Semantic + Recency Ranking

Raw cosine similarity ranking has a well-known failure mode in memory systems: a conversation from two years ago that's highly similar to the current query will outrank a moderately similar conversation from yesterday that's actually more relevant given user evolution. Real systems need a hybrid score.

The formula we'll use combines vector similarity with time decay:

score = (α × cosine_similarity) + (β × recency_decay)

Where recency_decay = exp(-λ × days_since_conversation) and λ controls how fast old memories lose relevance. A λ of 0.05 means a conversation from 14 days ago has roughly 50% of its recency score; 0.02 makes memory decay more slowly.

-- Hybrid semantic + recency retrieval function
-- Returns the most relevant past conversation turns for a given query embedding

CREATE OR REPLACE FUNCTION find_relevant_memories(
    p_user_id        UUID,
    p_query_embedding vector(1536),
    p_limit          INTEGER DEFAULT 5,
    p_max_age_days   INTEGER DEFAULT 90,
    p_alpha          FLOAT DEFAULT 0.7,   -- weight for semantic similarity
    p_lambda         FLOAT DEFAULT 0.03   -- recency decay rate
)
RETURNS TABLE (
    message_id          UUID,
    conversation_id     UUID,
    embedded_text       TEXT,
    cosine_similarity   FLOAT,
    recency_score       FLOAT,
    hybrid_score        FLOAT,
    days_ago            FLOAT,
    conversation_title  TEXT
)
LANGUAGE sql STABLE PARALLEL SAFE AS $$
    SELECT
        me.message_id,
        me.conversation_id,
        me.embedded_text,
        (1 - (me.embedding <=> p_query_embedding))        AS cosine_similarity,
        EXP(-p_lambda * EXTRACT(EPOCH FROM (now() - me.created_at)) / 86400.0) AS recency_score,
        (
            p_alpha * (1 - (me.embedding <=> p_query_embedding))
            +
            (1 - p_alpha) * EXP(-p_lambda * EXTRACT(EPOCH FROM (now() - me.created_at)) / 86400.0)
        )                                                  AS hybrid_score,
        EXTRACT(EPOCH FROM (now() - me.created_at)) / 86400.0 AS days_ago,
        c.title                                            AS conversation_title
    FROM message_embeddings me
    JOIN conversations c ON c.conversation_id = me.conversation_id
    WHERE
        me.user_id = p_user_id
        AND me.created_at >= now() - (p_max_age_days || ' days')::interval
        AND NOT c.is_archived
    ORDER BY hybrid_score DESC
    LIMIT p_limit;
$$;

-- Usage example — finding relevant memories for a query about API rate limiting:
-- (Replace the zero vector with an actual query embedding from your application)

SELECT
    conversation_title,
    days_ago::INTEGER AS days_ago,
    ROUND(cosine_similarity::NUMERIC, 3) AS semantic,
    ROUND(recency_score::NUMERIC, 3) AS recency,
    ROUND(hybrid_score::NUMERIC, 3) AS score,
    LEFT(embedded_text, 120) AS preview
FROM find_relevant_memories(
    '00000000-0000-0000-0000-000000000001'::UUID,  -- user_id
    '[0.1, 0.2, ...]'::vector(1536),               -- query embedding
    5,    -- return top 5
    90,   -- look back 90 days
    0.7,  -- 70% semantic, 30% recency
    0.03  -- moderate decay
)
ORDER BY score DESC;

For cases where users want to find memories by keyword in addition to semantic similarity — useful when the query is very specific like a product name or error code — add a BM25-style text rank:

-- Hybrid retrieval with optional keyword boost
-- Uses pg_trgm similarity as a lightweight keyword signal

CREATE OR REPLACE FUNCTION find_relevant_memories_hybrid(
    p_user_id        UUID,
    p_query_embedding vector(1536),
    p_query_text     TEXT,            -- the raw query text for keyword matching
    p_limit          INTEGER DEFAULT 5,
    p_max_age_days   INTEGER DEFAULT 90
)
RETURNS TABLE (
    message_id        UUID,
    conversation_id   UUID,
    embedded_text     TEXT,
    final_score       FLOAT,
    match_type        TEXT   -- 'semantic', 'keyword', 'hybrid'
)
LANGUAGE sql STABLE AS $$
    WITH ranked AS (
        SELECT
            me.message_id,
            me.conversation_id,
            me.embedded_text,
            (1 - (me.embedding <=> p_query_embedding))               AS sem_score,
            similarity(me.embedded_text, p_query_text)                AS kw_score,
            EXP(-0.03 * EXTRACT(EPOCH FROM (now() - me.created_at)) / 86400.0) AS rec_score
        FROM message_embeddings me
        JOIN conversations c ON c.conversation_id = me.conversation_id
        WHERE
            me.user_id = p_user_id
            AND me.created_at >= now() - (p_max_age_days || ' days')::interval
            AND NOT c.is_archived
        ORDER BY me.embedding <=> p_query_embedding
        LIMIT 50  -- retrieve top-50 by vector, then re-rank
    )
    SELECT
        message_id,
        conversation_id,
        embedded_text,
        (0.6 * sem_score + 0.2 * kw_score + 0.2 * rec_score) AS final_score,
        CASE
            WHEN sem_score > 0.75 AND kw_score > 0.3 THEN 'hybrid'
            WHEN kw_score > 0.3                       THEN 'keyword'
            ELSE                                           'semantic'
        END AS match_type
    FROM ranked
    ORDER BY final_score DESC
    LIMIT p_limit;
$$;

The inner ORDER BY me.embedding <=> p_query_embedding LIMIT 50 is key: it uses the HNSW index efficiently to retrieve a candidate set, then the outer query re-ranks that candidate set using the full formula. Applying all three signals directly in a WHERE clause or ORDER BY would prevent index usage.
