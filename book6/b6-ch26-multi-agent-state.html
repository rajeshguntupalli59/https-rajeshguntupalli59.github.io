Chapter 26: Multi-Agent Coordination via Shared PostgreSQL State

When you move beyond single-agent architectures into systems where multiple AI agents run concurrently — each with its own specialty, each handling a slice of a larger workflow — you immediately run into the coordination problem. How does a research agent know that a summarization agent has already processed a document? How does a planner agent hand off a subtask to a worker agent without race conditions? How do you debug what happened when five agents were all writing state at once? The answer that scales, audits cleanly, and integrates naturally with the rest of your application stack is PostgreSQL as the shared state backbone. This chapter shows you how to design that backbone, wire real agents to it, and run it in production without losing your mind.

---

The Coordination Problem in Multi-Agent Systems

Most multi-agent frameworks hide coordination behind in-memory queues or proprietary state stores. That works in demos. In production, it creates a second system you have to operate, monitor, back up, and reason about — separate from your application database. More importantly, it separates your agent state from your business data, which means you lose the ability to join them, audit them together, or apply the same access control policies to both.

The core coordination challenges that PostgreSQL solves well:

Task assignment and work stealing. Agents need to claim work atomically. Without atomic claiming, two agents pick up the same task. PostgreSQL's SELECT ... FOR UPDATE SKIP LOCKED is the correct primitive for this — it was built for exactly this pattern and has been production-hardened for years.

State visibility across agents. An orchestrator agent needs to see the current state of all worker agents: what they're doing, what they've produced, where they've failed. If that state lives in five different in-memory structures across five processes, you have no coherent view.

Event-driven wake-up. When agent A finishes its work, agent B should learn about it immediately without polling. PostgreSQL's LISTEN/NOTIFY mechanism (backed by pg_notify) gives you sub-millisecond event delivery within the database connection pool.

Audit and replay. When a multi-agent workflow produces wrong output, you need to reconstruct exactly what each agent did, in what order, with what inputs. An append-only event log in PostgreSQL gives you this for free.

Why not Redis? Redis excels at ephemeral queuing and caching, and we'll use it for exactly that — caching embedding lookups and rate-limit counters. But Redis is not your source of truth. You can't do a transactional join between your Redis agent state and your PostgreSQL user table. You can't apply row-level security to Redis keys. You can't write a migration that reshapes your coordination schema and run it through your normal deployment pipeline. PostgreSQL handles the parts that need to be true.

---

Schema Design for Multi-Agent Coordination

The schema has to carry three concerns simultaneously: the workflow definition (what work needs to happen), the execution state (what's happening right now), and the event log (what happened and when). These map to three table groups.

-- Enable required extensions
CREATE EXTENSION IF NOT EXISTS pgvector;
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- ============================================================
-- WORKFLOW DEFINITIONS
-- ============================================================

CREATE TABLE agent_types (
    agent_type_id   UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    type_name       TEXT NOT NULL UNIQUE,         -- 'researcher', 'summarizer', 'critic', 'planner'
    description     TEXT,
    capabilities    JSONB NOT NULL DEFAULT '[]',  -- list of capability strings
    model_config    JSONB NOT NULL DEFAULT '{}',  -- default model, temperature, max_tokens
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE workflows (
    workflow_id     UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    workflow_name   TEXT NOT NULL,
    description     TEXT,
    definition      JSONB NOT NULL,               -- DAG of tasks with dependencies
    status          TEXT NOT NULL DEFAULT 'draft'
                        CHECK (status IN ('draft', 'active', 'archived')),
    created_by      TEXT NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- ============================================================
-- EXECUTION STATE
-- ============================================================

CREATE TABLE workflow_runs (
    run_id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    workflow_id     UUID NOT NULL REFERENCES workflows(workflow_id),
    triggered_by    TEXT NOT NULL,               -- user_id or 'system' or upstream run_id
    input_payload   JSONB NOT NULL DEFAULT '{}',
    status          TEXT NOT NULL DEFAULT 'pending'
                        CHECK (status IN ('pending', 'running', 'completed', 'failed', 'cancelled')),
    started_at      TIMESTAMPTZ,
    completed_at    TIMESTAMPTZ,
    error_message   TEXT,
    metadata        JSONB NOT NULL DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_workflow_runs_status ON workflow_runs(status)
    WHERE status IN ('pending', 'running');
CREATE INDEX idx_workflow_runs_workflow ON workflow_runs(workflow_id, created_at DESC);

CREATE TABLE agent_tasks (
    task_id         UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    run_id          UUID NOT NULL REFERENCES workflow_runs(run_id) ON DELETE CASCADE,
    parent_task_id  UUID REFERENCES agent_tasks(task_id),  -- for subtask hierarchies
    task_type       TEXT NOT NULL,               -- maps to agent_types.type_name
    task_name       TEXT NOT NULL,
    priority        INTEGER NOT NULL DEFAULT 5,  -- 1 = highest, 10 = lowest
    status          TEXT NOT NULL DEFAULT 'pending'
                        CHECK (status IN ('pending', 'claimed', 'running', 'completed', 'failed', 'cancelled', 'retrying')),
    retry_count     INTEGER NOT NULL DEFAULT 0,
    max_retries     INTEGER NOT NULL DEFAULT 3,
    input_data      JSONB NOT NULL DEFAULT '{}',
    output_data     JSONB,
    error_detail    JSONB,
    assigned_to     TEXT,                        -- agent instance identifier
    claimed_at      TIMESTAMPTZ,
    started_at      TIMESTAMPTZ,
    completed_at    TIMESTAMPTZ,
    deadline        TIMESTAMPTZ,
    depends_on      UUID[] NOT NULL DEFAULT '{}', -- task_ids that must complete first
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Critical index for the work-stealing queue pattern
CREATE INDEX idx_tasks_claimable ON agent_tasks(task_type, priority, created_at)
    WHERE status = 'pending';

CREATE INDEX idx_tasks_run ON agent_tasks(run_id, status);
CREATE INDEX idx_tasks_assigned ON agent_tasks(assigned_to)
    WHERE status IN ('claimed', 'running');

-- ============================================================
-- SHARED AGENT MEMORY / BLACKBOARD
-- ============================================================

-- The blackboard is the shared memory space all agents can read and write.
-- Named after the AI blackboard architecture pattern from the 1980s —
-- still the right abstraction for multi-agent coordination.

CREATE TABLE blackboard (
    entry_id        UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    run_id          UUID NOT NULL REFERENCES workflow_runs(run_id) ON DELETE CASCADE,
    namespace       TEXT NOT NULL,               -- 'facts', 'hypotheses', 'artifacts', 'context'
    key             TEXT NOT NULL,
    value           JSONB NOT NULL,
    embedding       vector(1536),               -- for semantic search across blackboard entries
    written_by      TEXT NOT NULL,              -- agent instance that wrote this
    written_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    expires_at      TIMESTAMPTZ,
    version         INTEGER NOT NULL DEFAULT 1,
    is_current      BOOLEAN NOT NULL DEFAULT TRUE,
    UNIQUE (run_id, namespace, key, version)
);

CREATE INDEX idx_blackboard_lookup ON blackboard(run_id, namespace, key)
    WHERE is_current = TRUE;
CREATE INDEX idx_blackboard_embedding ON blackboard
    USING ivfflat (embedding vector_cosine_ops)
    WITH (lists = 100)
    WHERE embedding IS NOT NULL AND is_current = TRUE;

-- ============================================================
-- EVENT LOG (append-only)
-- ============================================================

CREATE TABLE agent_events (
    event_id        UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    run_id          UUID REFERENCES workflow_runs(run_id),
    task_id         UUID REFERENCES agent_tasks(task_id),
    agent_instance  TEXT NOT NULL,
    event_type      TEXT NOT NULL,               -- 'task_claimed', 'llm_call_start', 'llm_call_end',
                                                 -- 'tool_invoked', 'blackboard_write', 'task_complete', etc.
    event_data      JSONB NOT NULL DEFAULT '{}',
    token_usage     JSONB,                       -- {prompt_tokens, completion_tokens, model}
    duration_ms     INTEGER,
    occurred_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Partition by month for manageable table sizes in production
-- (shown without partitioning here for clarity; add it before going live)
CREATE INDEX idx_events_run ON agent_events(run_id, occurred_at DESC);
CREATE INDEX idx_events_task ON agent_events(task_id, occurred_at DESC);
CREATE INDEX idx_events_type ON agent_events(event_type, occurred_at DESC);

-- ============================================================
-- AGENT HEARTBEATS (detect stalled agents)
-- ============================================================

CREATE TABLE agent_heartbeats (
    agent_instance  TEXT PRIMARY KEY,
    agent_type      TEXT NOT NULL,
    current_task_id UUID REFERENCES agent_tasks(task_id),
    process_id      INTEGER,
    host            TEXT,
    last_seen       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    metadata        JSONB NOT NULL DEFAULT '{}'
);

-- ============================================================
-- Helper function: notify on task state changes
-- ============================================================

CREATE OR REPLACE FUNCTION notify_task_change()
RETURNS TRIGGER AS $$
DECLARE
    payload JSONB;
BEGIN
    payload := jsonb_build_object(
        'task_id',   NEW.task_id,
        'run_id',    NEW.run_id,
        'task_type', NEW.task_type,
        'status',    NEW.status,
        'priority',  NEW.priority
    );
    PERFORM pg_notify('agent_tasks_' || NEW.task_type, payload::TEXT);
    PERFORM pg_notify('agent_tasks_all', payload::TEXT);
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_task_notify
    AFTER INSERT OR UPDATE OF status ON agent_tasks
    FOR EACH ROW EXECUTE FUNCTION notify_task_change();

-- ============================================================
-- Helper function: update updated_at automatically
-- ============================================================

CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_tasks_updated_at
    BEFORE UPDATE ON agent_tasks
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();

This schema gives every agent a consistent interface. The blackboard table is the key innovation here — it's a versioned, semantically searchable shared memory space. An agent can write a discovery, another agent can search that discovery space with a vector query ("find all facts related to this topic"), and the whole thing is transactionally consistent with the task state.

---

Implementing the Core Agent Coordination Patterns

With the schema in place, three patterns make everything work: atomic task claiming, blackboard read/write with semantic search, and event-driven agent wake-up via pg_notify.
