Chapter 9: Self-Healing Database Operations

Self-healing is the operational core of any autonomous database system. While previous chapters addressed detection and decision-making in isolation, this chapter synthesizes those capabilities into end-to-end remediation loops — systems that not only observe failure conditions but act on them without waiting for a human to read a PagerDuty alert at 2 AM. The patterns here draw from production architectures running PostgreSQL under Patroni and pg_auto_failover, SQL Server Always On Availability Groups managed by Kubernetes operators, and hybrid orchestration layers that treat remediation as a deterministic state machine rather than a collection of runbooks. The goal is not to automate existing manual procedures. It is to design systems whose default state is recovery — systems where the question is never "will it heal?" but "how fast, and at what cost?"

---

Anatomy of a Remediation Loop: Detect, Classify, Act, Verify

A self-healing system is architecturally identical to a control system in industrial engineering. There is a sensor layer, a comparator that measures deviation from desired state, an actuator that drives corrective action, and a feedback loop that confirms the system returned to setpoint. Every remediation loop in a production database platform must implement all four stages, or it is not self-healing — it is just automated alerting with extra steps.

Detection at the ADE level is not threshold crossing. Threshold crossing is the 2015 version of monitoring. Real detection uses composite signals: a connection pool saturation event that correlates with a blocker query age exceeding 30 seconds and a transaction rate drop of more than 15% within the same 90-second window means something different than any of those signals alone. The detection layer must produce classified events, not raw metrics.

Classification determines which remediation playbook applies. A primary node that fails a health check because its WAL receiver is 500MB behind the primary but the replication slot is still active and network latency is normal is in a different failure class than a primary that fails a health check because the OS ran out of file descriptors. One requires a controlled switchover. The other requires a replica promotion and an investigation of the host before the old primary is ever allowed back.

Action must be implemented as a state machine. The actuator doesn't execute a single command — it drives the system through a sequence of states with defined transitions, rollback conditions, and timeouts at each stage.

Verification closes the loop. After remediation, the system must re-measure the originating signal and confirm it has returned to a healthy range before marking the incident closed. Without this, you get cascading automation: a loop that fires the same remediation action repeatedly because it never confirmed the first execution succeeded.

The following Python class implements a minimal remediation loop skeleton that production systems can extend:

import time
import logging
from enum import Enum, auto
from dataclasses import dataclass, field
from typing import Callable, Optional

logger = logging.getLogger("remediation_loop")

class RemediationState(Enum):
    IDLE = auto()
    DETECTED = auto()
    CLASSIFIED = auto()
    ACTING = auto()
    VERIFYING = auto()
    RESOLVED = auto()
    FAILED = auto()
    ESCALATED = auto()

@dataclass
class RemediationContext:
    incident_id: str
    signal: dict
    classification: Optional[str] = None
    playbook: Optional[str] = None
    action_result: Optional[dict] = None
    attempt_count: int = 0
    max_attempts: int = 3
    state: RemediationState = RemediationState.IDLE
    history: list = field(default_factory=list)

class RemediationLoop:
    def __init__(
        self,
        classifier: Callable,
        playbook_registry: dict,
        verifier: Callable,
        escalation_hook: Callable,
    ):
        self.classifier = classifier
        self.playbook_registry = playbook_registry
        self.verifier = verifier
        self.escalation_hook = escalation_hook

    def run(self, ctx: RemediationContext) -> RemediationState:
        ctx.state = RemediationState.DETECTED
        ctx.history.append(("DETECTED", time.time()))

        # Classify
        ctx.classification = self.classifier(ctx.signal)
        if ctx.classification not in self.playbook_registry:
            logger.error(f"No playbook for classification: {ctx.classification}")
            ctx.state = RemediationState.ESCALATED
            self.escalation_hook(ctx)
            return ctx.state

        ctx.state = RemediationState.CLASSIFIED
        ctx.playbook = self.playbook_registry[ctx.classification]
        logger.info(f"[{ctx.incident_id}] Classified as {ctx.classification}, "
                    f"playbook: {ctx.playbook.__name__}")

        # Act with retry
        while ctx.attempt_count < ctx.max_attempts:
            ctx.state = RemediationState.ACTING
            ctx.attempt_count += 1
            try:
                ctx.action_result = ctx.playbook(ctx)
                break
            except Exception as e:
                logger.warning(f"[{ctx.incident_id}] Attempt {ctx.attempt_count} failed: {e}")
                if ctx.attempt_count >= ctx.max_attempts:
                    ctx.state = RemediationState.ESCALATED
                    self.escalation_hook(ctx)
                    return ctx.state
                time.sleep(2 ** ctx.attempt_count)  # exponential backoff

        # Verify
        ctx.state = RemediationState.VERIFYING
        for attempt in range(5):
            time.sleep(10)
            if self.verifier(ctx):
                ctx.state = RemediationState.RESOLVED
                ctx.history.append(("RESOLVED", time.time()))
                logger.info(f"[{ctx.incident_id}] Resolved after "
                            f"{ctx.attempt_count} action attempt(s)")
                return ctx.state
            logger.debug(f"[{ctx.incident_id}] Verification attempt {attempt+1}/5 pending")

        # Verification failed
        ctx.state = RemediationState.FAILED
        self.escalation_hook(ctx)
        return ctx.state

This structure enforces the invariant that no action is taken without a classification, no incident is closed without verification, and repeated failures escalate rather than loop indefinitely. Every production self-healing system must enforce these three invariants. Systems that don't eventually create automation storms — a well-documented failure mode where remediation logic runs in a tight loop and makes the outage worse.

Connection and Session Healing: Killing Blockers, Recycling Pools, and Recovering from Lock Storms

Lock storms and connection exhaustion are the two most common causes of database brownouts that are not hardware failures. They are also the easiest class of problem to self-heal because the corrective actions are low-risk, reversible, and fast. A self-healing system should handle both classes autonomously, with human notification rather than human intervention.

Detecting and killing blockers requires more than identifying the longest-running query. The system must identify the root blocker in a blocking chain — the session holding the lock that all others are waiting on — and evaluate whether terminating it is safe given the current workload context.

PostgreSQL:

-- Identify root blockers with full chain context
WITH RECURSIVE blocking_chain AS (
    -- Anchor: sessions blocked by nothing (root blockers)
    SELECT
        pid,
        usename,
        application_name,
        state,
        wait_event_type,
        wait_event,
        query_start,
        state_change,
        query,
        0 AS depth,
        ARRAY[pid] AS chain
    FROM pg_stat_activity
    WHERE cardinality(pg_blocking_pids(pid)) = 0
      AND pid IN (
          SELECT UNNEST(pg_blocking_pids(pid))
          FROM pg_stat_activity
          WHERE wait_event_type = 'Lock'
      )

    UNION ALL

    -- Recursive: sessions blocked by a member of the chain
    SELECT
        sa.pid,
        sa.usename,
        sa.application_name,
        sa.state,
        sa.wait_event_type,
        sa.wait_event,
        sa.query_start,
        sa.state_change,
        sa.query,
        bc.depth + 1,
        bc.chain || sa.pid
    FROM pg_stat_activity sa
    JOIN blocking_chain bc
        ON bc.pid = ANY(pg_blocking_pids(sa.pid))
    WHERE NOT sa.pid = ANY(bc.chain)  -- prevent cycles
)
SELECT
    pid,
    depth,
    usename,
    application_name,
    state,
    EXTRACT(EPOCH FROM (now() - query_start))::int AS blocked_seconds,
    LEFT(query, 120) AS query_snippet,
    chain
FROM blocking_chain
ORDER BY depth, blocked_seconds DESC;

SQL Server:

-- Identify root blockers with wait chain and resource details
WITH BlockingChain AS (
    SELECT
        r.session_id,
        r.blocking_session_id,
        r.wait_type,
        r.wait_time / 1000.0 AS wait_seconds,
        r.status,
        DB_NAME(r.database_id) AS database_name,
        s.login_name,
        s.program_name,
        s.host_name,
        st.text AS current_sql,
        0 AS depth
    FROM sys.dm_exec_requests r
    JOIN sys.dm_exec_sessions s ON r.session_id = s.session_id
    CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) st
    WHERE r.blocking_session_id = 0
      AND r.session_id IN (
          SELECT DISTINCT blocking_session_id
          FROM sys.dm_exec_requests
          WHERE blocking_session_id > 0
      )

    UNION ALL

    SELECT
        r.session_id,
        r.blocking_session_id,
        r.wait_type,
        r.wait_time / 1000.0,
        r.status,
        DB_NAME(r.database_id),
        s.login_name,
        s.program_name,
        s.host_name,
        st.text,
        bc.depth + 1
    FROM sys.dm_exec_requests r
    JOIN sys.dm_exec_sessions s ON r.session_id = s.session_id
    CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) st
    JOIN BlockingChain bc ON r.blocking_session_id = bc.session_id
    WHERE bc.depth < 10  -- cap recursion depth
)
SELECT
    session_id,
    blocking_session_id,
    depth,
    wait_type,
    wait_seconds,
    database_name,
    login_name,
    program_name,
    LEFT(current_sql, 200) AS sql_snippet
FROM BlockingChain
ORDER BY depth, wait_seconds DESC;

Once a root blocker is identified, the system applies a kill decision policy. The policy must evaluate: is this session running a known scheduled job that should not be interrupted? Is the blocking duration below the configured tolerance (e.g., 60 seconds for OLTP, 300 seconds for batch windows)? Is the application name on a protected list? Only after all three conditions pass does the system issue the termination.

Connection pool exhaustion — pgBouncer pools full, SQL Server connection counts at max — requires a different healing path. The system first checks whether the exhaustion is caused by a blocker (which is handled above) or by a genuine connection leak. A leak manifests as a large number of idle connections with increasing age and no active transaction. The healing action is to terminate idle connections beyond a configurable age, then recycle the pool.

PostgreSQL:

-- Terminate idle connections exceeding age threshold (safe to automate)
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE state = 'idle'
  AND query_start < now() - interval '10 minutes'
  AND pid <> pg_backend_pid()
  AND usename NOT IN ('replication', 'pgbouncer', 'monitoring')
  AND application_name NOT LIKE 'pgAgent%';

SQL Server:

-- Kill idle sessions exceeding age threshold
DECLARE @session_id INT;
DECLARE @kill_sql NVARCHAR(50);

DECLARE idle_cursor CURSOR FOR
    SELECT session_id
    FROM sys.dm_exec_sessions
    WHERE status = 'sleeping'
      AND login_time < DATEADD(MINUTE, -10, GETDATE())
      AND is_user_process = 1
      AND session_id > 50  -- exclude system sessions
      AND program_name NOT LIKE 'SQL Agent%'
      AND login_name NOT IN ('sa', 'monitoring_user');

OPEN idle_cursor;
FETCH NEXT FROM idle_cursor INTO @session_id;

WHILE @@FETCH_STATUS = 0
BEGIN
    SET @kill_sql = N'KILL ' + CAST(@session_id AS NVARCHAR(10));
    EXEC sp_executesql @kill_sql;
    FETCH NEXT FROM idle_cursor INTO @session_id;
END

CLOSE idle_cursor;
DEALLOCATE idle_cursor;

After terminations execute, the verification step must confirm that pool availability has recovered: pgBouncer's SHOW POOLS should show free server connections, and SQL Server's connection count should drop below 80% of the configured maximum. If neither condition is met within 60 seconds, the incident escalates — there is an application-side leak that cannot be healed at the database layer.

Automated Failover Architecture: Patroni, pg_auto_failover, and SQL Server AG Under Kubernetes

Failover is the highest-stakes self-healing action a database system takes. The cost of an incorrect failover decision — promoting a stale replica, failing over when the primary is actually healthy, or triggering split-brain — is frequently worse than the original failure. The architecture of an autonomous failover system must therefore be conservative by design: it should require convergent evidence from multiple independent observers before acting, and it must enforce a fencing protocol that guarantees the old primary cannot accept writes after the new primary is promoted.

Patroni manages this through Distributed Configuration Store (DCS) consensus — typically etcd. The primary holds a leader lock with a TTL. If the primary fails to renew the lock before TTL expiry, a standby with the most advanced WAL position acquires it and promotes. The self-healing capability is inherent to the design, but production deployments require careful tuning of three parameters:

  • → `ttl`: the duration in seconds before a lock is considered expired. Too short causes false failovers under load; too long delays legitimate failovers. A value of 30 seconds with a `loop_wait` of 10 seconds is a common production starting point for low-latency OLTP.
  • → `maximum_lag_on_failover`: the maximum bytes of WAL lag a replica can have and still be eligible for promotion. Setting this to 0 means only the most current replica can be promoted, at the cost of potentially having no eligible candidates if all replicas are lagged.
  • → `retry_timeout`: how long Patroni retries a DCS operation before assuming the DCS is unreachable and ceasing automatic failover (the safe-mode behavior).

    • pg_auto_failover uses a separate monitor node that tracks the state of each node in the formation. The monitor implements a state machine with explicit transitions — primary, wait_primary, secondary, catchingup, draining, demoted — and will only transition nodes when the complete set of pre-conditions for that transition is satisfied. This makes it safer than Patroni for environments where the DBA team is less experienced with distributed systems, at the cost of the monitor node becoming a single point of failure itself unless it is made highly available.

    For SQL Server Always On running on Kubernetes, the SQL Server Operator (either the Microsoft SQL Server Operator or community operators) manages AG health through Kubernetes health probes and custom resource definitions. The operator pattern here is significant: the operator reconciliation loop is itself a self-healing construct. Any deviation between the desired state declared in the SqlServer CRD and the observed state triggers the operator's reconcile function, which drives corrective action.

    A production Kubernetes-based AG self-healing architecture uses three components:

    1. Liveness probe at the SQL Server pod level — if the pod fails the liveness probe, Kubernetes restarts it. The probe must be scoped to verify AG listener responsiveness, not just TCP port availability.

    2. Operator reconciler — detects AG replica role divergence (e.g., a secondary that has been in RESOLVING state for more than 60 seconds) and triggers a manual failover via T-SQL.

    3. External DNS or load balancer controller — updates the listener IP binding after a successful failover so application connection strings remain stable.

    The reconcile function for a simplified AG failover:

    import subprocess
import pyodbc
import logging
from kubernetes import client, config

logger = logging.getLogger("ag_reconciler")

def get_ag_status(conn_str: str, ag_name: str) -> dict:
    """Query AG replica status from the current primary."""
    conn = pyodbc.connect(conn_str, autocommit=True)
    cursor = conn.cursor()
    cursor.execute("""
        SELECT
            ar.replica_server_name,
            ars.role_desc,
            ars.synchronization_health_desc,
            ars.connected_state_desc,
            ars.last_commit_time
        FROM sys.dm_hadr_availability_replica_states ars
        JOIN sys.availability_replicas ar
            ON ars.replica_id = ar.replica_id
        JOIN sys.availability_groups ag
            ON ar.group_id = ag.group_id
        WHERE ag.name = ?
    """, ag_name)
    rows = cursor.fetchall()
    conn.close()
    return {
        row.replica_server_name: {
            "role": row.role_desc,
            "sync_health": row.synchronization_health_desc,
            "connected": row.connected_state_desc,
            "last_commit": row.last_commit_time,
        }
        for row in rows
    }

def trigger_failover(conn_str: str, ag_name: str, target_replica: str):
    """Execute controlled failover to target replica."""
    # Connect to the TARGET replica, not the primary
    conn = pyodbc.connect(conn_str, autocommit=True)
    cursor = conn.cursor()
    logger.warning(f"Initiating failover of {ag_name} to {target_replica}")
    cursor.execute(f"ALTER AVAILABILITY GROUP [{ag_name}] FAILOVER;")
    conn.close()
    logger.info(f"Failover command issued to {target_replica}")

def reconcile_ag(ag_name: str, primary_conn: str, replica_conns: dict):
    status = get_ag_status(primary_conn, ag_name)

    primary_count = sum(1 for r in status.values() if r["role"] == "PRIMARY")
    healthy_secondary = [
        name for name, r in status.items()
        if r["role"] == "SECONDARY"
        and r["sync_health"] == "HEALTHY"
        and r["connected"] == "CONNECTED"
    ]

    if primary_count == 0:
        logger.error(f"No primary detected for AG {ag_name}. "
                     f"Attempting failover to best available replica.")
        if healthy_secondary:
            target = healthy_secondary[0]
            target_conn = replica_conns.get(target)
            if target_conn:
                trigger_failover(target_conn, ag_name, target)
            else:
                logger.error(f"No connection string for target replica {target}. Escalating.")
        else:
            logger.critical(f"No healthy secondary available for {ag_name}. "
                            f"Manual intervention required.")
    elif primary_count > 1:
        logger.critical(f"Split-brain detected for {ag_name}: "
                        f"{primary_count} nodes reporting PRIMARY role. "
                        f"Escalating immediately — no automated action.")
    else:
        logger.debug(f"AG {ag_name} healthy. Primary confirmed, "
                     f"{len(healthy_secondary)} healthy secondaries.")

    The critical design choice in this reconciler is the primary_count > 1 branch: the system deliberately refuses to take automated action on split-brain and escalates immediately. This is the correct behavior. Automated systems must know which failure classes require human judgment, and split-brain is categorically one of them.

    Autonomous Space Management: Bloat Remediation, Autovacuum Tuning, and Filegroup Rebalancing

    Space management failures — table bloat causing query plan degradation, transaction ID wraparound approaching, or a SQL Server filegroup filling to capacity — are slow-moving incidents that are ideal for autonomous remediation. They are predictable, the corrective actions are well-understood, and the window for action is measured in hours or days rather than seconds.

    PostgreSQL bloat and autovacuum are intimately connected. When autovacuum falls behind — because a table's dead tuple accumulation rate exceeds what autovacuum can process with current settings — the table bloats, index scans slow down, and eventually the system approaches transaction ID wraparound, which is an emergency. A self-healing system monitors this trajectory and intervenes before the emergency materializes.

    The detection query identifies tables where autovacuum is losing the race:

    PostgreSQL:

    -- Tables where bloat is growing faster than autovacuum can process
SELECT
    schemaname,
    relname,
    n_dead_tup,
    n_live_tup,
    ROUND(n_dead_tup::numeric / NULLIF(n_live_tup + n_dead_tup, 0) * 100, 2)
        AS dead_tuple_pct,
    last_autovacuum,
    last_autoanalyze,
    autovacuum_count,
    EXTRACT(EPOCH FROM (now() - last_autovacuum))::int / 60 AS minutes_since_vacuum,
    -- Estimated pages requiring vacuum
    CEIL(n_dead_tup / 100.0)::int AS estimated_vacuum_pages,
    pg_size_pretty(pg_total_relation_size(schemaname || '.' || relname)) AS total_size
FROM pg_stat_user_tables
WHERE n_dead_tup > 10000
  AND (
      n_dead_tup::float / NULLIF(n_live_tup + n_dead_tup, 0) > 0.10
      OR last_autovacuum < now() - interval '2 hours'
      OR last_autovacuum IS NULL
  )
ORDER BY dead_tuple_pct DESC, n_dead_tup DESC
LIMIT 20;

    When this query identifies candidates, the self-healing system applies a graduated response. If dead tuple percentage is between 10% and 25%, it increases the autovacuum aggressiveness for that specific table using ALTER TABLE ... SET (autovacuum_vacuum_scale_factor = 0.01, autovacuum_vacuum_cost_delay = 2). If it exceeds 25%, it schedules an explicit VACUUM ANALYZE during the next maintenance window. If it exceeds 40% and the table is a high-traffic table with degraded query performance, it schedules VACUUM ANALYZE immediately with a coordinated reduction in vacuum_cost_delay at the session level to allow it to run faster.

    For transaction ID wraparound, the autonomous system monitors age(datfrozenxid) against the autovacuum_freeze_max_age setting and enforces a two-stage alert: when age exceeds 80% of autovacuum_freeze_max_age, the system increases aggressive vacuum scheduling. When it exceeds 90%, it enforces an emergency VACUUM FREEZE on the oldest-aged tables regardless of maintenance windows.
