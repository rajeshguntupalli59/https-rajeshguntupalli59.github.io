Chapter 1 of 40 AI-Native Apps on PostgreSQL

Chapter 1: Why PostgreSQL is the Default AI Application Backend

Every AI application eventually needs a place to store conversations, user profiles, embeddings, audit logs, documents, permissions, and the relational glue that holds all of it together. Developers new to the AI space often reach for a vector database first — Pinecone, Weaviate, Chroma — because that's where the embedding tutorials point. Then they add Redis for sessions, a separate Postgres for user data, another store for conversation history, and find themselves managing five systems instead of one. This chapter makes the case that PostgreSQL, extended with pgvector and a handful of well-chosen extensions, can serve as the single operational backbone for production AI applications. Not because it's "good enough," but because the relational model, transactional guarantees, row-level security, and notification primitives give you capabilities that purpose-built vector stores cannot replicate. By the end of this chapter you will have a fully operational schema, a working RAG retrieval path, and a production deployment pattern standing on PostgreSQL alone.

---

1.1 The Vector-Only Trap and Why Relational Wins

The pitch for standalone vector databases sounds compelling: low latency, purpose-built for similarity search, scales horizontally. The pitch collapses the moment you ask a question that isn't purely "find me the nearest neighbors." Real AI applications ask questions like:

  • → *Find the most relevant documents for this user, but only ones they have permission to read, published in the last 90 days, not previously shown in this session.*
  • → *Store this conversation turn, update the session token count, emit a notification so the billing service can charge, and roll back everything if the embedding write fails.*
  • → *Which users have had more than five conversations about topic X in the last week?*

    • None of those questions are answerable by a vector store alone. Every single one requires joins, filters, transactions, and access control — the things PostgreSQL has been doing reliably for thirty years.

    The strategic insight is this: pgvector turns PostgreSQL into a vector database. It does not turn a vector database into PostgreSQL. You lose nothing by adding vector search to PostgreSQL; you lose a great deal by starting with a vector store and bolting relational features on top of it.

    The table below summarizes what each layer of a typical AI application needs and which system provides it natively.

    | Capability | Vector-only store | PostgreSQL + pgvector |

    |---|---|---|

    | Embedding storage and ANN search | ✓ | ✓ |

    | Exact metadata filtering | Limited | ✓ Full SQL |

    | Transactional consistency | ✗ | ✓ ACID |

    | Row-level security | ✗ | ✓ Native |

    | Conversation history | ✗ | ✓ |

    | Full-text search hybrid | ✗ | ✓ tsvector |

    | Change notifications | ✗ | ✓ pg_notify |

    | JSON document storage | ✗ | ✓ JSONB |

    | Audit logging | ✗ | ✓ |

    | Foreign keys and referential integrity | ✗ | ✓ |

    This is not a theoretical argument. Let's build the foundation and prove it with running code.

    1.2 The Core Schema for an AI Application

    Before writing a single line of Python, the schema needs to reflect the actual entities in an AI application. Every production AI system this author has built maps to roughly the same set of tables. The names change; the relationships do not.

    -- ============================================================
-- CORE AI APPLICATION SCHEMA
-- Run this in order. Requires pgvector extension.
-- ============================================================

-- Enable required extensions
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS pg_trgm;    -- trigram similarity for hybrid search
CREATE EXTENSION IF NOT EXISTS pgcrypto;   -- gen_random_uuid()

-- ----------------------------------------------------------------
-- Users and tenants
-- Every AI app is multi-tenant at some level.
-- ----------------------------------------------------------------
CREATE TABLE tenants (
    tenant_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            TEXT NOT NULL,
    plan            TEXT NOT NULL DEFAULT 'free' CHECK (plan IN ('free','pro','enterprise')),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    settings        JSONB NOT NULL DEFAULT '{}'
);

CREATE TABLE users (
    user_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id       UUID NOT NULL REFERENCES tenants(tenant_id) ON DELETE CASCADE,
    email           TEXT NOT NULL,
    display_name    TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    last_active_at  TIMESTAMPTZ,
    metadata        JSONB NOT NULL DEFAULT '{}',
    UNIQUE(tenant_id, email)
);

CREATE INDEX idx_users_tenant ON users(tenant_id);

-- ----------------------------------------------------------------
-- Documents — the corpus that gets embedded for RAG
-- ----------------------------------------------------------------
CREATE TABLE documents (
    document_id     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id       UUID NOT NULL REFERENCES tenants(tenant_id) ON DELETE CASCADE,
    title           TEXT NOT NULL,
    source_url      TEXT,
    content         TEXT NOT NULL,
    content_hash    TEXT GENERATED ALWAYS AS (encode(digest(content, 'sha256'), 'hex')) STORED,
    content_tsv     TSVECTOR GENERATED ALWAYS AS (to_tsvector('english', content)) STORED,
    metadata        JSONB NOT NULL DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    is_active       BOOLEAN NOT NULL DEFAULT true
);

CREATE INDEX idx_documents_tenant        ON documents(tenant_id);
CREATE INDEX idx_documents_content_tsv   ON documents USING GIN(content_tsv);
CREATE INDEX idx_documents_metadata      ON documents USING GIN(metadata);
CREATE INDEX idx_documents_content_hash  ON documents(content_hash);

-- ----------------------------------------------------------------
-- Document chunks — what actually gets embedded
-- A single document may produce hundreds of chunks.
-- ----------------------------------------------------------------
CREATE TABLE document_chunks (
    chunk_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id     UUID NOT NULL REFERENCES documents(document_id) ON DELETE CASCADE,
    tenant_id       UUID NOT NULL REFERENCES tenants(tenant_id) ON DELETE CASCADE,
    chunk_index     INTEGER NOT NULL,   -- position within parent document
    content         TEXT NOT NULL,
    token_count     INTEGER,
    embedding       VECTOR(1536),       -- OpenAI text-embedding-3-small dimensions
    embedding_model TEXT,               -- track which model produced this embedding
    embedded_at     TIMESTAMPTZ,
    metadata        JSONB NOT NULL DEFAULT '{}',
    UNIQUE(document_id, chunk_index)
);

-- HNSW index for fast approximate nearest-neighbor search
-- ef_construction=128 is a safe default; tune higher for better recall
CREATE INDEX idx_chunks_embedding ON document_chunks
    USING hnsw (embedding vector_cosine_ops)
    WITH (m=16, ef_construction=128);

CREATE INDEX idx_chunks_document   ON document_chunks(document_id);
CREATE INDEX idx_chunks_tenant     ON document_chunks(tenant_id);
CREATE INDEX idx_chunks_embedded   ON document_chunks(embedded_at) WHERE embedding IS NOT NULL;

-- ----------------------------------------------------------------
-- Conversations and messages — the memory layer
-- ----------------------------------------------------------------
CREATE TABLE conversations (
    conversation_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    tenant_id       UUID NOT NULL REFERENCES tenants(tenant_id) ON DELETE CASCADE,
    title           TEXT,
    model           TEXT NOT NULL DEFAULT 'claude-3-5-sonnet-20241022',
    total_tokens    INTEGER NOT NULL DEFAULT 0,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    metadata        JSONB NOT NULL DEFAULT '{}'
);

CREATE INDEX idx_conversations_user   ON conversations(user_id, created_at DESC);
CREATE INDEX idx_conversations_tenant ON conversations(tenant_id);

CREATE TABLE messages (
    message_id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    conversation_id UUID NOT NULL REFERENCES conversations(conversation_id) ON DELETE CASCADE,
    tenant_id       UUID NOT NULL REFERENCES tenants(tenant_id) ON DELETE CASCADE,
    role            TEXT NOT NULL CHECK (role IN ('user','assistant','system','tool')),
    content         TEXT NOT NULL,
    token_count     INTEGER,
    model           TEXT,
    finish_reason   TEXT,
    citations       JSONB,              -- [{chunk_id, score, text_snippet}]
    tool_calls      JSONB,              -- raw tool_calls from the LLM response
    latency_ms      INTEGER,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_messages_conversation ON messages(conversation_id, created_at ASC);
CREATE INDEX idx_messages_tenant       ON messages(tenant_id);

-- ----------------------------------------------------------------
-- Audit log — every LLM call that touches production
-- ----------------------------------------------------------------
CREATE TABLE llm_audit_log (
    log_id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id       UUID REFERENCES tenants(tenant_id),
    user_id         UUID REFERENCES users(user_id),
    conversation_id UUID REFERENCES conversations(conversation_id),
    model           TEXT NOT NULL,
    prompt_tokens   INTEGER NOT NULL DEFAULT 0,
    completion_tokens INTEGER NOT NULL DEFAULT 0,
    total_tokens    INTEGER NOT NULL DEFAULT 0,
    latency_ms      INTEGER,
    status          TEXT NOT NULL DEFAULT 'success' CHECK (status IN ('success','error','timeout')),
    error_message   TEXT,
    request_hash    TEXT,               -- SHA-256 of the prompt for deduplication
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_audit_tenant      ON llm_audit_log(tenant_id, created_at DESC);
CREATE INDEX idx_audit_user        ON llm_audit_log(user_id, created_at DESC);
CREATE INDEX idx_audit_created     ON llm_audit_log(created_at DESC);

    This schema is not a toy. It handles multi-tenancy, document versioning through the content hash, hybrid search through the generated tsvector column, embedding provenance through embedding_model, and a complete audit trail. Everything is connected by foreign keys, so you can answer questions like "which users have consumed the most tokens this month" with a single SQL query.

    1.3 Ingestion Pipeline: From Documents to Searchable Embeddings

    Getting documents into this schema requires chunking text, calling an embedding model, and writing the results transactionally. This is where Celery handles asynchronous ingestion so the API stays responsive while background workers process large corpora.

    # ingestion/pipeline.py
# Dependencies: pip install openai celery psycopg2-binary redis tiktoken langchain-text-splitters

import hashlib
import time
from datetime import datetime, timezone
from typing import Optional

import psycopg2
import tiktoken
from celery import Celery
from langchain_text_splitters import RecursiveCharacterTextSplitter
from openai import OpenAI

# ----------------------------------------------------------------
# Configuration — in production these come from environment variables
# ----------------------------------------------------------------
DATABASE_URL = "postgresql://aiapp:secret@localhost:5432/aiapp"
REDIS_URL    = "redis://localhost:6379/0"
EMBED_MODEL  = "text-embedding-3-small"
EMBED_DIM    = 1536
CHUNK_SIZE   = 512    # tokens
CHUNK_OVERLAP = 64    # tokens

openai_client = OpenAI()   # reads OPENAI_API_KEY from env
celery_app    = Celery("ingestion", broker=REDIS_URL, backend=REDIS_URL)

# ----------------------------------------------------------------
# Tokenizer — count tokens before chunking so we don't over-embed
# ----------------------------------------------------------------
tokenizer = tiktoken.encoding_for_model("text-embedding-3-small")

def count_tokens(text: str) -> int:
    return len(tokenizer.encode(text))

# ----------------------------------------------------------------
# Text splitter — recursive character splitting with token awareness
# ----------------------------------------------------------------
splitter = RecursiveCharacterTextSplitter.from_tiktoken_encoder(
    model_name=EMBED_MODEL,
    chunk_size=CHUNK_SIZE,
    chunk_overlap=CHUNK_OVERLAP,
)

# ----------------------------------------------------------------
# Embedding helper — batch up to 100 texts per API call
# ----------------------------------------------------------------
def embed_texts(texts: list[str]) -> list[list[float]]:
    """Call OpenAI embedding API and return list of embedding vectors."""
    response = openai_client.embeddings.create(
        model=EMBED_MODEL,
        input=texts,
    )
    # Preserve original order — OpenAI returns results sorted by index
    return [item.embedding for item in sorted(response.data, key=lambda x: x.index)]

# ----------------------------------------------------------------
# Celery task — processes one document asynchronously
# ----------------------------------------------------------------
@celery_app.task(bind=True, max_retries=3, default_retry_delay=30)
def ingest_document(
    self,
    document_id: str,
    tenant_id: str,
    content: str,
    title: str,
    source_url: Optional[str] = None,
    metadata: Optional[dict] = None,
) -> dict:
    """
    Full ingestion pipeline for a single document:
    1. Write document record (idempotent via content_hash)
    2. Split into chunks
    3. Embed chunks in batches
    4. Write chunks transactionally
    5. Notify downstream services via pg_notify
    """
    conn = psycopg2.connect(DATABASE_URL)
    try:
        with conn:
            with conn.cursor() as cur:
                # Step 1: Upsert document
                cur.execute("""
                    INSERT INTO documents
                        (document_id, tenant_id, title, source_url, content, metadata)
                    VALUES (%s, %s, %s, %s, %s, %s)
                    ON CONFLICT (content_hash, tenant_id) DO UPDATE
                        SET updated_at = now(),
                            is_active  = true
                    RETURNING document_id, content_hash
                """, (
                    document_id, tenant_id, title,
                    source_url, content,
                    psycopg2.extras.Json(metadata or {}),
                ))
                # Note: ON CONFLICT with a generated column requires a unique index.
                # In practice, create: CREATE UNIQUE INDEX ON documents(tenant_id, content_hash);
                row = cur.fetchone()
                actual_doc_id, content_hash = row

                # Step 2: Chunk
                chunks = splitter.split_text(content)
                if not chunks:
                    return {"document_id": actual_doc_id, "chunks": 0}

                # Step 3: Delete existing chunks for this document (re-ingestion)
                cur.execute(
                    "DELETE FROM document_chunks WHERE document_id = %s",
                    (actual_doc_id,)
                )

                # Step 4: Embed in batches of 100
                BATCH_SIZE = 100
                all_embeddings: list[list[float]] = []
                for i in range(0, len(chunks), BATCH_SIZE):
                    batch = chunks[i : i + BATCH_SIZE]
                    all_embeddings.extend(embed_texts(batch))

                # Step 5: Bulk insert chunks
                import psycopg2.extras
                chunk_rows = []
                for idx, (chunk_text, embedding) in enumerate(zip(chunks, all_embeddings)):
                    chunk_rows.append((
                        actual_doc_id,
                        tenant_id,
                        idx,
                        chunk_text,
                        count_tokens(chunk_text),
                        embedding,            # psycopg2 will call __str__ → '[0.1, 0.2, ...]'
                        EMBED_MODEL,
                        datetime.now(timezone.utc),
                    ))

                psycopg2.extras.execute_values(
                    cur,
                    """
                    INSERT INTO document_chunks
                        (document_id, tenant_id, chunk_index, content,
                         token_count, embedding, embedding_model, embedded_at)
                    VALUES %s
                    """,
                    chunk_rows,
                    template="(%s, %s, %s, %s, %s, %s::vector, %s, %s)",
                )

                # Step 6: Notify downstream consumers (e.g., search index warmup)
                cur.execute(
                    "SELECT pg_notify('document_ingested', %s)",
                    (f'{{"document_id": "{actual_doc_id}", "tenant_id": "{tenant_id}", '
                     f'"chunks": {len(chunks)}}}',)
                )

        return {
            "document_id": str(actual_doc_id),
            "chunks": len(chunks),
            "content_hash": content_hash,
        }

    except Exception as exc:
        conn.rollback()
        raise self.retry(exc=exc)
    finally:
        conn.close()

    Two design decisions here deserve attention. First, the entire document write plus all chunk writes happen inside a single transaction. If the embedding API returns 200 chunks and the database write fails on chunk 150, nothing is committed — no partial state, no phantom documents. Second, pg_notify fires after the commit, so downstream consumers only see the notification when the data is actually readable.

    1.4 Retrieval: Hybrid Search Combines Vector and Full-Text

    Pure vector search misses exact keyword matches. Pure full-text search misses semantic similarity. Production RAG systems use both, and PostgreSQL is the only single system that makes this a SQL query rather than an application-level merge.

    -- ============================================================
-- HYBRID SEARCH FUNCTION
-- Combines cosine similarity (pgvector) with BM25-style full-text
-- ranking (ts_rank_cd) using Reciprocal Rank Fusion.
-- ============================================================

CREATE OR REPLACE FUNCTION hybrid_search(
    p_tenant_id     UUID,
    p_query_text    TEXT,
    p_query_vector  VECTOR(1536),
    p_match_count   INTEGER DEFAULT 10,
    p_full_text_weight FLOAT DEFAULT 0.3,
    p_vector_weight    FLOAT DEFAULT 0.7,
    p_rrf_k            INTEGER DEFAULT 60    -- RRF smoothing constant
)
RETURNS TABLE (
    chunk_id        UUID,
    document_id     UUID,
    chunk_index     INTEGER,
    content         TEXT,
    metadata        JSONB,
    vector_rank     BIGINT,
    fulltext_rank   BIGINT,
    rrf_score       FLOAT,
    cosine_distance FLOAT
)
LANGUAGE SQL
STABLE PARALLEL SAFE
AS $$
    WITH
    -- --------------------------------------------------------
    -- Vector search: top-N by cosine distance
    -- --------------------------------------------------------
    vector_results AS (
        SELECT
            dc.chunk_id,
            dc.document_id,
            dc.chunk_index,
            dc.content,
            dc.metadata,
            dc.embedding <=> p_query_vector AS cosine_distance,
            ROW_NUMBER() OVER (ORDER BY dc.embedding <=> p_query_vector) AS rank
        FROM document_chunks dc
        WHERE dc.tenant_id = p_tenant_id
          AND dc.embedding IS NOT NULL
        ORDER BY cosine_distance
        LIMIT p_match_count * 2    -- oversample for RRF merge
    ),
    -- --------------------------------------------------------
    -- Full-text search: ranked by ts_rank_cd
    -- --------------------------------------------------------
    fulltext_results AS (
        SELECT
            dc.chunk_id,
            dc.document_id,
            dc.chunk_index,
            dc.content,
            dc.metadata,
            0.0::FLOAT AS cosine_distance,
            ROW_NUMBER() OVER (
                ORDER BY ts_rank_cd(
                    to_tsvector('english', dc.content),
                    websearch_to_tsquery('english', p_query_text)
                ) DESC
            ) AS rank
        FROM document_chunks dc
        WHERE dc.tenant_id = p_tenant_id
          AND to_tsvector('english', dc.content) @@
              websearch_to_tsquery('english', p_query_text)
        LIMIT p_match_count * 2
    ),
    -- --------------------------------------------------------
    -- Union both result sets, keeping all distinct chunk_ids
    -- --------------------------------------------------------
    all_chunks AS (
        SELECT chunk_id FROM vector_results
        UNION
        SELECT chunk_id FROM fulltext_results
    ),
    -- --------------------------------------------------------
    -- Reciprocal Rank Fusion score
    -- --------------------------------------------------------
    rrf AS (
        SELECT
            ac.chunk_id,
            COALESCE(
                p_vector_weight * (1.0 / (p_rrf_k + vr.rank)), 0
            ) + COALESCE(
                p_full_text_weight * (1.0 / (p_rrf_k + fr.rank)), 0
            ) AS rrf_score,
            COALESCE(vr.rank, 9999999)    AS vector_rank,
            COALESCE(fr.rank, 9999999)    AS fulltext_rank,
            COALESCE(vr.cosine_distance, 1.0) AS cosine_distance
        FROM all_chunks ac
        LEFT JOIN vector_results   vr ON ac.chunk_id = vr.chunk_id
        LEFT JOIN fulltext_results fr ON ac.chunk_id = fr.chunk_id
    )
    SELECT
        dc.chunk_id,
        dc.document_id,
        dc.chunk_index,
        dc.content,
        dc.metadata,
        rrf.vector_rank,
        rrf.fulltext_rank,
        rrf.rrf_score,
        rrf.cosine_distance
    FROM rrf
    JOIN document_chunks dc ON rrf.chunk_id = dc.chunk_id
    ORDER BY rrf.rrf_score DESC
    LIMIT p_match_count;
$$;

-- ============================================================
-- Row-Level Security: tenants can only search their own chunks
-- ============================================================
ALTER TABLE document_chunks ENABLE ROW LEVEL SECURITY;

CREATE POLICY rls_chunks_tenant ON document_chunks
    USING (tenant_id = current_setting('app.current_tenant_id')::UUID);

ALTER TABLE documents ENABLE ROW LEVEL SECURITY;

CREATE POLICY rls_documents_tenant ON documents
    USING (tenant_id = current_setting('app.current_tenant_id')::UUID);

    The RRF merge in SQL means there is no Python code sorting two lists from two systems. The database does the work. The RLS policies mean that even if application code forgets a WHERE tenant_id = ? clause, PostgreSQL enforces the boundary automatically — an enforcement layer that no external vector store can provide.
