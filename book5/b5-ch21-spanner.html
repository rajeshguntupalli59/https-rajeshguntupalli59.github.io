Chapter 21: Spanner: When You Need Global Consistency

Google Cloud Spanner sits in a category by itself. It is the only production-grade relational database that offers external consistency across geographically distributed nodes — not eventual consistency, not tunable consistency, but true serializable transactions that span continents without the application needing to know where the data physically lives. For experienced DBAs, this sounds like marketing copy. It is not. Spanner achieves this through TrueTime, Google's GPS and atomic clock-based timekeeping infrastructure, which bounds clock uncertainty to single-digit milliseconds and uses that bound as a synchronization primitive. If you have ever spent nights managing multi-region PostgreSQL with Patroni, or fought with SQL Server Always On availability groups across data centers, understanding what Spanner removes from your operational surface area is genuinely striking. This chapter covers when Spanner is the right choice, how its data model differs from what you know, how to design schemas that perform at scale, how to operate Spanner in production, and where the limits and costs will surprise you.

---

21.1 TrueTime and External Consistency: What It Actually Means

Before touching configuration or schema design, you need to internalize the consistency model, because it drives every subsequent architectural decision.

What you're used to. In PostgreSQL with streaming replication, you have a primary that accepts writes and standbys that replay WAL. If you read from a standby, you accept replication lag — typically milliseconds in a healthy system but unbounded in theory. If you need your read to reflect a write that just happened, you read from the primary. With Citus or logical replication across regions, the problem compounds. You either accept stale reads or you route everything through one node and sacrifice the performance benefit of having multiple regions. With SQL Server Distributed Availability Groups, the story is similar: synchronous commit to one replica, asynchronous everywhere else, and a failover that requires manual intervention or complex automation.

What Spanner provides. Spanner guarantees external consistency, which means: if transaction T1 commits before transaction T2 begins (in real-world time), then T2 will always see T1's effects. Always. Regardless of which Spanner node processes T2. Regardless of which geographic region that node is in. There is no replication lag observable to the application. A write that commits in Iowa is immediately visible to a read in Singapore.

This works because TrueTime assigns each transaction a commit timestamp from a globally synchronized clock. Spanner will actually pause a transaction before committing — for up to the clock uncertainty bound, roughly 7ms — to ensure that the committed timestamp is genuinely in the past relative to real time. No subsequent transaction can receive an earlier timestamp. The consequence is a globally consistent external view of all committed data.

What this means operationally. The consistency model that you used to enforce through careful read/write routing, synchronous replication configuration, and application-level retry logic is handled by the database engine itself. You do not configure synchronous commit. You do not choose which replica to read from. You do not manage replication lag monitoring as a first-class operational concern. Those problems simply do not exist in Spanner in the way they exist in traditional distributed systems.

The tradeoff is write latency. A write to Spanner must be replicated across a quorum of nodes before it commits. In a single-region configuration, that means three nodes within one GCP region — latency comparable to synchronous standby commits in PostgreSQL. In a multi-region configuration, quorum involves nodes across multiple regions, and your commit latency will reflect the round-trip time between those regions. A write that spans us-central1 and us-east1 will see approximately 30–50ms commit latency. A write with a quorum that includes europe-west1 will see 80–120ms. This is not tunable. You do not choose eventual consistency to reduce latency. If low write latency matters more than global consistency, Spanner is the wrong tool.

The external consistency guarantee in SQL terms. Spanner's consistency level maps directly to SERIALIZABLE isolation in PostgreSQL terms — the strictest isolation level. Spanner does not offer READ COMMITTED or REPEATABLE READ as selectable alternatives. Every read-write transaction is serializable. Read-only transactions use a snapshot that is consistent with the external consistency model. This means your application cannot accidentally be misconfigured to use weaker isolation. It also means some transaction patterns that worked under weaker isolation levels will behave differently, and you need to be aware of that during migration.

21.2 Spanner's Data Model: Interleaving, Keys, and Schema Design

Spanner is a relational database. It supports ANSI SQL (via GoogleSQL dialect), foreign keys, secondary indexes, check constraints, and views. But its physical data organization is fundamentally different from heap-based storage in PostgreSQL or the B-tree clustered index model in SQL Server, and designing schemas without understanding this will result in severe performance problems at scale.

Tables and the primary key requirement. Every Spanner table must have a primary key. There are no heap tables. There are no sequences in the PostgreSQL or SQL Server sense. This matters immediately: you cannot use SERIAL or IDENTITY columns that generate monotonically increasing integers. If you do, you will create a write hotspot — all inserts go to the same tablet (Spanner's unit of data sharding), serialized behind each other, and you will never achieve horizontal write scalability. This is the single most common mistake DBAs make when migrating to Spanner.

The correct patterns for generating primary keys in Spanner:

  • → **UUID v4**: Randomly distributed across the key space. Spanner distributes tablets based on key ranges, so random UUIDs spread writes across many splits immediately.
  • → **Bit-reversed sequences**: Spanner natively supports `GET_NEXT_SEQUENCE_VALUE` with bit-reversal, which takes an incrementing integer and reverses its bits, producing a value that distributes evenly across the key space without the monotonicity problem.
  • → **Hash prefix**: If you need a meaningful natural key, prepend a hash of the key to distribute writes.
    • -- CREATE TABLE with UUID primary key (GoogleSQL dialect)
CREATE TABLE Orders (
    OrderId    STRING(36)   NOT NULL,
    CustomerId STRING(36)   NOT NULL,
    OrderDate  TIMESTAMP    NOT NULL,
    Status     STRING(32)   NOT NULL,
    TotalAmount NUMERIC      NOT NULL,
    CreatedAt  TIMESTAMP    NOT NULL OPTIONS (allow_commit_timestamp = true),
) PRIMARY KEY (OrderId);

-- Bit-reversed sequence (avoids hotspotting on sequential inserts)
CREATE SEQUENCE OrderSequence OPTIONS (
    sequence_kind = "bit_reversed_positive"
);

CREATE TABLE OrdersSeq (
    OrderId     INT64        NOT NULL DEFAULT (GET_NEXT_SEQUENCE_VALUE(SEQUENCE OrderSequence)),
    CustomerId  STRING(36)   NOT NULL,
    OrderDate   TIMESTAMP    NOT NULL,
    Status      STRING(32)   NOT NULL,
) PRIMARY KEY (OrderId);

    Interleaving: physically co-locating parent and child rows. In PostgreSQL, if you join Orders and OrderItems frequently, you rely on indexes, buffer cache, and query planning to make that join fast. The rows for a given order's items are scattered across the heap based on insertion order. In Spanner, you can declare that OrderItems rows are physically interleaved within Orders rows — the storage engine places each OrderItems row immediately after its parent Orders row on disk. The join between Orders and OrderItems for a specific OrderId becomes a sequential scan of a contiguous storage region rather than a random access join.

    -- Interleaved table: OrderItems rows stored inside Orders rows
CREATE TABLE OrderItems (
    OrderId      STRING(36)   NOT NULL,
    ItemId       STRING(36)   NOT NULL,
    ProductId    STRING(36)   NOT NULL,
    Quantity     INT64        NOT NULL,
    UnitPrice    NUMERIC      NOT NULL,
    LineTotal    NUMERIC      NOT NULL,
) PRIMARY KEY (OrderId, ItemId),
  INTERLEAVE IN PARENT Orders ON DELETE CASCADE;

    The constraint is that the child table's primary key must begin with the parent's primary key. OrderItems(OrderId, ItemId) begins with OrderId, which is Orders' primary key. Deletion from the parent with ON DELETE CASCADE removes interleaved children atomically. Without ON DELETE CASCADE, Spanner enforces referential integrity at the storage level — you cannot delete a parent row that has children.

    Interleaving depth can go up to six levels. In practice, two or three levels covers the vast majority of real schemas. Over-interleaving creates wide rows and affects tablet splitting behavior.

    Secondary indexes. Spanner supports secondary indexes with a behavior that DBA experience on traditional RDBMS does not fully prepare you for. When you create a secondary index on a column, Spanner creates a separate interleaved table to store the index. Reads against a secondary index may need to join back to the base table to retrieve columns not stored in the index — called a back-join or index back-lookup. For large tables, this back-join crosses tablet boundaries and is expensive.

    The solution is to store additional columns in the index itself using STORING:

    -- Secondary index without STORING: back-join required for Status and TotalAmount
CREATE INDEX IX_Orders_CustomerId ON Orders(CustomerId);

-- Secondary index with STORING: no back-join needed for common query
CREATE INDEX IX_Orders_CustomerId_Storing ON Orders(CustomerId)
STORING (Status, TotalAmount, OrderDate);

    When you define a query in your application that filters by CustomerId and reads Status and TotalAmount, use the second index form. The query reads entirely from the index without touching the base table rows. At 100 million rows, the difference between a back-join and a covering index in Spanner is the difference between a 50ms query and a 5000ms query.

    Array and STRUCT columns. Spanner supports ARRAY columns natively, allowing you to store repeated values within a single row. This is useful for tags, labels, and other set-valued attributes that would require a junction table in a normalized relational schema. This is not a replacement for interleaving — use arrays for small, bounded sets of scalar values, and use interleaved tables for child entities with their own attribute columns.

    21.3 Provisioning and Configuration: Instances, Databases, and Processing Units

    Spanner's resource model does not map to anything in traditional DBA experience. There are no instance types to select, no vCPU counts to choose, no memory configurations to evaluate. Spanner's compute capacity is expressed in Processing Units (PUs).

    Processing Units and nodes. 1 node = 1000 PUs. Google's documentation expresses smaller configurations in PUs (100 PU, 500 PU) and larger ones in nodes (1 node, 2 nodes, 10 nodes). For production workloads, the minimum meaningful configuration is 1 node (1000 PU) per region. Sub-node configurations (100–500 PU) are suitable for development and testing. At 1 node, Spanner guarantees:

  • → Up to 2000 QPS of reads and 1000 QPS of writes per region (highly dependent on row size and transaction complexity)
  • → Up to 2 TB of storage per node before performance degrades (Spanner auto-splits data, but very large datasets relative to node count increase cross-split transaction overhead)

    • Provisioning with Terraform:

    # Spanner instance and database via Terraform
resource "google_spanner_instance" "production" {
  name         = "prod-spanner"
  config       = "nam6"          # North America multi-region config
  display_name = "Production Spanner"
  num_nodes    = 3               # 3 nodes across nam6 multi-region footprint

  labels = {
    environment = "production"
    team        = "platform-data"
  }
}

resource "google_spanner_database" "orders_db" {
  instance = google_spanner_instance.production.name
  name     = "orders"

  version_retention_period = "7d"   # Point-in-time recovery window: 7 days
  deletion_protection      = true

  ddl = [
    <<-EOT
      CREATE TABLE Orders (
        OrderId     STRING(36)  NOT NULL,
        CustomerId  STRING(36)  NOT NULL,
        OrderDate   TIMESTAMP   NOT NULL,
        Status      STRING(32)  NOT NULL,
        TotalAmount NUMERIC     NOT NULL,
        CreatedAt   TIMESTAMP   NOT NULL OPTIONS (allow_commit_timestamp = true)
      ) PRIMARY KEY (OrderId)
    EOT
    ,
    <<-EOT
      CREATE INDEX IX_Orders_CustomerId ON Orders(CustomerId)
      STORING (Status, TotalAmount, OrderDate)
    EOT
  ]
}

    Instance configurations: regional vs. multi-region. The config parameter in the instance definition determines the geographic footprint. This is not a setting you change after the fact without recreating the instance. Choose carefully.

    | Configuration | Regions | Typical Write Latency | Use Case |

    |---|---|---|---|

    | us-central1 | Single region, 3 zones | ~5ms | Regional apps, dev/test |

    | nam4 | Iowa + Virginia (2 regions) | ~30ms | US multi-region |

    | nam6 | 4 North American regions | ~45ms | High-availability US |

    | eur5 | 5 European regions | ~30ms | EU data residency |

    | global2 | Multiple continents | ~80–120ms | Truly global workloads |

    Multi-region configurations also affect cost significantly. A 3-node nam6 instance costs roughly 3× a 3-node us-central1 instance, because Spanner replicates across more physical nodes under the hood to maintain quorum across all participating regions.

    Autoscaler. Spanner does not automatically scale compute capacity. You must either use the open-source [Spanner Autoscaler](https://github.com/cloudspannerecosystem/autoscaler) project (a Cloud Function-based solution that monitors CPU utilization and adjusts node count) or manage scaling manually. Google recommends keeping Spanner CPU utilization below 65% for single-region and below 45% for high-priority traffic in multi-region configurations. Above these thresholds, tail latency increases significantly.

    # Check current instance configuration and node count
gcloud spanner instances describe prod-spanner \
  --format="table(name, config, nodeCount, processingUnits, state)"

# Scale up to 5 nodes for peak traffic period
gcloud spanner instances update prod-spanner \
  --nodes=5 \
  --description="Scaled for Q4 peak"

# Verify the update applied (scaling is online, no downtime)
gcloud spanner instances describe prod-spanner \
  --format="value(nodeCount)"

    Scaling is online. There is no restart, no maintenance window, no connection interruption. Spanner redistributes tablet ownership across the new capacity within minutes.

    21.4 Transactions, Mutations, and the Application Layer

    If your application was built against PostgreSQL or SQL Server, the way it issues reads, writes, and transactions will need adjustment for Spanner. Not because Spanner is semantically weaker — it is stronger — but because some patterns that are idiomatic in traditional databases are inefficient or unsupported in Spanner.

    Transaction types. Spanner offers three transaction modes that the application explicitly chooses between:

    1. Read-write transactions: Full ACID, serializable, can include both reads and writes. All reads within the transaction see a consistent snapshot. Writes are buffered and committed atomically. Any conflict causes a retry-able abort error (ABORTED status). Your application must handle retries.

    2. Read-only transactions: No locks, no aborts, no retries needed. Reads are served from a consistent snapshot at a timestamp you specify (or the current time). Extremely efficient and scalable. If you are reading data and not writing, always use a read-only transaction.

    3. Partitioned DML: For large-scale data modification (backfills, bulk deletes, archiving). Runs as a series of smaller transactions internally. Does not guarantee atomicity across the full affected row set. Suitable for operations where eventual completion matters more than all-or-nothing atomicity (clearing expired sessions, archiving old orders). The SQL analog is running a batch delete in chunks.

    The most common operational mistake from PostgreSQL-experienced DBAs is using read-write transactions for read-only operations. In PostgreSQL, BEGIN; SELECT ...; COMMIT; is harmless. In Spanner, it acquires shared locks unnecessarily and increases contention on hot rows.

    Mutations vs. DML. Spanner supports two ways to write data: DML (INSERT, UPDATE, DELETE statements) and Mutations (a batch API that specifies key-value changes directly). Mutations are more efficient for bulk operations because they bypass the SQL parsing and execution layer. For single-row writes in OLTP context, the difference is negligible. For batch loading thousands of rows, mutations are substantially faster.

    # Python client: read-write transaction with proper retry handling
from google.cloud import spanner
from google.cloud.spanner_v1 import param_types
import uuid

client = spanner.Client(project="your-gcp-project")
instance = client.instance("prod-spanner")
database = instance.database("orders")

def insert_order_with_items(order_data, items_data):
    """
    Insert an order and its items in a single read-write transaction.
    Spanner client handles ABORTED retries automatically via run_in_transaction.
    """
    def _transaction_body(transaction):
        order_id = str(uuid.uuid4())

        # DML insert for the parent row
        transaction.execute_update(
            """
            INSERT INTO Orders (OrderId, CustomerId, OrderDate, Status, TotalAmount, CreatedAt)
            VALUES (@order_id, @customer_id, @order_date, @status, @total, PENDING_COMMIT_TIMESTAMP())
            """,
            params={
                "order_id":    order_id,
                "customer_id": order_data["customer_id"],
                "order_date":  order_data["order_date"],
                "status":      "PENDING",
                "total":       order_data["total_amount"],
            },
            param_types={
                "order_id":    param_types.STRING,
                "customer_id": param_types.STRING,
                "order_date":  param_types.TIMESTAMP,
                "status":      param_types.STRING,
                "total":       param_types.NUMERIC,
            }
        )

        # Batch mutations for child rows (more efficient than per-row DML)
        item_mutations = []
        for item in items_data:
            item_mutations.append(
                spanner.param_types  # placeholder — use spanner.Mutation directly
            )

        # Using mutations for bulk child inserts
        transaction.insert(
            table="OrderItems",
            columns=("OrderId", "ItemId", "ProductId", "Quantity", "UnitPrice", "LineTotal"),
            values=[
                (order_id, str(uuid.uuid4()), item["product_id"],
                 item["quantity"], item["unit_price"], item["line_total"])
                for item in items_data
            ]
        )

        return order_id

    # run_in_transaction retries automatically on ABORTED
    return database.run_in_transaction(_transaction_body)


def read_customer_orders(customer_id, limit=100):
    """
    Read-only transaction: no locks, no retry needed, served from snapshot.
    Always use with_snapshot for read-only workloads.
    """
    with database.snapshot() as snapshot:
        results = snapshot.execute_sql(
            """
            SELECT o.OrderId, o.OrderDate, o.Status, o.TotalAmount
            FROM   Orders@{FORCE_INDEX=IX_Orders_CustomerId_Storing} o
            WHERE  o.CustomerId = @customer_id
            ORDER BY o.OrderDate DESC
            LIMIT  @limit
            """,
            params={"customer_id": customer_id, "limit": limit},
            param_types={
                "customer_id": param_types.STRING,
                "limit":       param_types.INT64,
            }
        )
        return [dict(row) for row in results]

    Note the @{FORCE_INDEX=...} hint in the SQL. Spanner's query optimizer will use an index if it determines the index is beneficial, but for queries on large tables where you know the index is correct, forcing index usage prevents surprising full-table-scan query plans after statistics updates change the optimizer's estimate.

    Stale reads. If your application can tolerate reading data that is up to N seconds old, stale reads dramatically improve throughput and reduce latency. A stale read can be served by any Spanner replica locally, without coordination across regions. For analytics queries, dashboard data, and reference data lookups, 15-second stale reads are typically acceptable and can cut read latency by 50% in multi-region configurations.

    import datetime
from google.cloud.spanner_v1 import DirectedReadOptions

# Bounded staleness read: data no older than 15 seconds
with database.snapshot(max_staleness=datetime.timedelta(seconds=15)) as snapshot:
    results = snapshot.execute_sql(
        "SELECT COUNT(*) as order_count FROM Orders WHERE Status = 'PENDING'"
    )
    for row in results:
        print(f"Pending orders: {row[0]}")

# Exact timestamp read: useful for repeatable analytics
# Read the database as it existed exactly 5 minutes ago
five_minutes_ago = datetime.datetime.utcnow() - datetime.timedelta(minutes=5)
with database.snapshot(read_timestamp=five_minutes_ago) as snapshot:
    results = snapshot.execute_sql(
        "SELECT SUM(TotalAmount) as revenue FROM Orders WHERE Status = 'COMPLETED'"
    )

    21.5 Observability, Operational Patterns, and Migration Realities

    Running Spanner in production requires a different observability mindset than traditional RDBMS operations. The metrics and signals that matter are different, and the absence of some traditional DBA tools is compensated by managed infrastructure but requires adjustment.

    Key metrics to monitor. In PostgreSQL, you track pg_stat_activity, lock wait events, replication lag, vacuum activity, and bloat. In Spanner, your primary operational signals are:

    | Metric | Alert Threshold | Consequence of Breach |

    |---|---|---|

    | CPU Utilization (single-region) | > 65% | Latency tail increase |

    | CPU Utilization (multi-region, high-priority) | > 45% | Latency tail increase |

    | Request Latency P99 (reads) | > 100ms | Application SLO risk |

    | Request Latency P99 (writes) | > 200ms (multi-region) | Application SLO risk |

    | Transaction Abort Rate | > 5% | Inefficient transaction design |

    | Locked Resource Wait Time | > 50ms | Hot key contention |

    # Export Spanner metrics to Cloud Monitoring and set alerting policy
gcloud alpha monitoring policies create \
  --policy-from-file=spanner-cpu-alert.yaml

# spanner-cpu-alert.yaml content:
cat <<'EOF' > spanner-cpu-alert.yaml
displayName: "Spanner High CPU Utilization"
conditions:
  - displayName: "CPU > 65% for 5 minutes"
    conditionThreshold:
      filter: >
        resource.type = "spanner_instance"
        AND metric.type = "spanner.googleapis.com/instance/cpu/utilization"
        AND resource.labels.instance_id = "prod-spanner"
      comparison: COMPARISON_GT
      thresholdValue: 0.65
      duration: 300s
      aggregations:
        - alignmentPeriod: 60s
          perSeriesAligner: ALIGN_MEAN
alertStrategy:
  autoClose: 604800s
notificationChannels:
  - projects/your-project/notificationChannels/your-channel-id
EOF

    Schema changes in production. Spanner handles DDL differently from PostgreSQL or SQL Server. In PostgreSQL, ALTER TABLE ADD COLUMN is typically fast (for nullable columns) but ALTER TABLE ADD COLUMN NOT NULL DEFAULT value requires a table rewrite. In SQL Server, online index operations require Enterprise edition. In Spanner, all DDL changes are non-blocking and online — you issue the DDL statement and Spanner applies it in the background while reads and writes continue.
