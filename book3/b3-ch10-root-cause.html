Chapter 10: Automated Root Cause Analysis

When a production database degrades at 2 AM, the difference between a five-minute recovery and a two-hour war room comes down to one thing: how fast the system understands *why* something broke. Automated root cause analysis (ARCA) is the discipline of building systems that move from symptom detection to causal diagnosis without waiting for a human to open a terminal. At the ADE level, this means architecting a pipeline that ingests telemetry across multiple signal layers, applies causal reasoning models, generates testable hypotheses, and either acts autonomously or presents the on-call engineer with a ranked, evidence-backed cause list before they finish reading the alert. This chapter covers the signal taxonomy, correlation engine design, causal graph construction, LLM-augmented diagnosis, and the production patterns that make this work reliably across PostgreSQL and SQL Server environments.

---

Signal Taxonomy: Building the Evidence Hierarchy for Database Failures

Root cause analysis fails when the system tries to reason from a single signal layer. A spike in avg_query_time is a symptom. The lock event underneath it is a contributing factor. The schema change that introduced a missing index is the cause. An ARCA system must model all three levels simultaneously.

The practical signal hierarchy for database environments has five layers:

Layer 1 — User-visible symptoms: Query latency (p50/p95/p99), error rates, connection refusals, transaction throughput drops. These are what alerts fire on. They are never root causes.

Layer 2 — Database execution signals: Lock waits, blocked queries, plan cache invalidations, wait event distributions, temp disk spills, autovacuum contention (PostgreSQL), PAGELATCH/PAGEIOLATCH waits (SQL Server).

Layer 3 — Resource signals: CPU saturation, IOPS ceiling, memory pressure, network bandwidth to storage, WAL write latency (PostgreSQL), VLF fragmentation or log growth (SQL Server).

Layer 4 — Configuration and state signals: Parameter changes, pg_hba modifications, statistics staleness (n_live_tup vs n_dead_tup drift), index bloat ratio, autovacuum threshold crossings, AG synchronization lag (SQL Server), Patroni leader election history.

Layer 5 — External causation signals: Deployment events, schema migrations, batch job start times, connection pool reconfiguration, upstream service traffic spikes, cloud provider maintenance windows.

Each layer has different collection mechanisms and different temporal characteristics. Resource signals move in seconds. Configuration signals move in minutes to hours. Deployment signals are discrete events. A proper ARCA schema stores all five layers in a unified timeline, keyed to wall-clock time with enough resolution to establish sequence — because *sequence is the foundation of causation*.

The collection architecture for PostgreSQL uses a combination of pg_stat_activity, pg_stat_bgwriter, pg_stat_user_tables, pg_locks, pg_stat_replication, and pg_wait_sampling (the extension, not the base view) for sub-second wait event sampling. SQL Server relies on DMVs: sys.dm_exec_requests, sys.dm_os_wait_stats, sys.dm_exec_query_stats, sys.dm_db_index_usage_stats, and Extended Events sessions for high-resolution capture.

PostgreSQL:

-- Unified Layer 2 snapshot: captures wait events, lock chains, and blocking topology
-- Run every 10 seconds by the ARCA collector agent

SELECT
    a.pid,
    a.usename,
    a.application_name,
    a.state,
    a.wait_event_type,
    a.wait_event,
    a.query_start,
    EXTRACT(EPOCH FROM (now() - a.query_start))::numeric(10,2) AS duration_sec,
    a.query,
    b.pid                  AS blocking_pid,
    b.usename              AS blocking_user,
    b.query                AS blocking_query,
    pg_blocking_pids(a.pid) AS full_blocking_chain
FROM pg_stat_activity a
LEFT JOIN pg_stat_activity b
    ON b.pid = ANY(pg_blocking_pids(a.pid))
WHERE a.state != 'idle'
  AND a.pid != pg_backend_pid()
ORDER BY duration_sec DESC;

SQL Server:

-- Equivalent Layer 2 snapshot for SQL Server: active requests with blocking chain
-- Deployed as a SQL Agent job step running every 10 seconds

SELECT
    r.session_id,
    r.blocking_session_id,
    r.wait_type,
    r.wait_time / 1000.0            AS wait_sec,
    r.total_elapsed_time / 1000.0   AS elapsed_sec,
    r.cpu_time / 1000.0             AS cpu_sec,
    r.logical_reads,
    r.writes,
    r.granted_query_memory          AS memory_grant_pages,
    DB_NAME(r.database_id)          AS database_name,
    s.login_name,
    s.program_name,
    SUBSTRING(t.text, (r.statement_start_offset/2)+1,
        ((CASE r.statement_end_offset
            WHEN -1 THEN DATALENGTH(t.text)
            ELSE r.statement_end_offset
          END - r.statement_start_offset)/2)+1)  AS current_statement,
    qp.query_plan
FROM sys.dm_exec_requests r
JOIN sys.dm_exec_sessions s  ON s.session_id = r.session_id
CROSS APPLY sys.dm_exec_sql_text(r.sql_handle) t
OUTER APPLY sys.dm_exec_query_plan(r.plan_handle) qp
WHERE r.session_id != @@SPID
ORDER BY r.wait_time DESC;

Both snapshots are pushed into a TimescaleDB or InfluxDB hypertable by the collector, tagged with db_host, db_name, snapshot_ts, and collection_layer. The collection_layer tag is what allows the correlation engine to traverse layers during causal chain reconstruction.

Causal Graph Construction: From Correlated Events to Directed Causation

Correlation is not causation — but in ARCA, correlation with correct temporal ordering and physical dependency modeling *is* strong evidence of causation. The architecture that makes this practical is a causal graph: a directed acyclic graph (DAG) where nodes represent observable events and edges represent either known physical dependencies or statistically inferred temporal precedence.

Physical dependency edges are hardcoded from database internals knowledge:

  • → Lock acquisition → lock wait → query timeout
  • → Autovacuum blocked by long transaction → table bloat → sequential scan preference → slow query
  • → WAL writer stall → replication lag → standby read query degradation
  • → Log file autogrowth event → I/O spike → all query latencies increase
  • → Plan cache eviction (SQL Server) → recompilation storm → CPU spike

    • Statistical edges are derived at runtime by analyzing whether event type A consistently precedes event type B within a configurable time window (typically 30–300 seconds, tuned per event class).

    The causal graph implementation uses Python's networkx library with a custom edge-scoring function that combines:

    1. Temporal precedence score — how reliably does A precede B in historical incidents?

    2. Physical dependency score — does database engine documentation confirm A can cause B?

    3. Recency weight — events in the last 5 minutes score higher than events 30 minutes ago

    4. Severity amplification — high-magnitude events on critical paths get edge weight boosted

    import networkx as nx
import pandas as pd
from datetime import datetime, timedelta
from typing import List, Dict, Tuple

class CausalGraphBuilder:
    """
    Builds a directed causal graph from multi-layer telemetry events.
    Physical dependency edges are seeded from the knowledge base.
    Statistical edges are derived from temporal precedence analysis.
    """

    PHYSICAL_DEPENDENCIES: List[Tuple[str, str, float]] = [
        ("autovacuum_blocked",       "table_bloat_increase",      0.95),
        ("table_bloat_increase",     "seq_scan_preference",       0.80),
        ("seq_scan_preference",      "query_latency_spike",       0.85),
        ("lock_acquisition_spike",   "lock_wait_queue",           0.99),
        ("lock_wait_queue",          "query_timeout_increase",    0.90),
        ("wal_writer_stall",         "replication_lag_increase",  0.95),
        ("replication_lag_increase", "standby_query_degradation", 0.75),
        ("log_autogrowth_event",     "io_stall_spike",            0.90),
        ("io_stall_spike",           "global_latency_increase",   0.85),
        ("stats_staleness",          "plan_regression",           0.70),
        ("plan_regression",          "cpu_spike",                 0.65),
        ("plan_regression",          "query_latency_spike",       0.80),
        ("connection_pool_exhaust",  "new_conn_refusal",          0.99),
        ("schema_change_event",      "stats_staleness",           0.60),
        ("schema_change_event",      "plan_regression",           0.55),
        ("deployment_event",         "schema_change_event",       0.85),
    ]

    def __init__(self, lookback_window_sec: int = 300):
        self.graph = nx.DiGraph()
        self.lookback_window = timedelta(seconds=lookback_window_sec)
        self._seed_physical_edges()

    def _seed_physical_edges(self):
        for source, target, weight in self.PHYSICAL_DEPENDENCIES:
            self.graph.add_edge(source, target,
                                edge_type="physical",
                                weight=weight)

    def ingest_events(self, events: List[Dict]) -> None:
        """
        events: list of dicts with keys:
          event_type, severity (0-1), timestamp (datetime), host, metadata
        """
        now = datetime.utcnow()
        relevant = [e for e in events
                    if e["timestamp"] >= now - self.lookback_window]

        for event in relevant:
            node = event["event_type"]
            if node not in self.graph:
                self.graph.add_node(node,
                                    severity=event["severity"],
                                    timestamp=event["timestamp"],
                                    host=event["host"],
                                    metadata=event.get("metadata", {}))
            else:
                # Update severity to max observed
                existing = self.graph.nodes[node].get("severity", 0)
                self.graph.nodes[node]["severity"] = max(existing,
                                                          event["severity"])

    def compute_root_causes(self, target_symptom: str) -> List[Dict]:
        """
        Walk backwards from target_symptom through the causal graph.
        Returns ranked list of root cause candidates with confidence scores.
        """
        if target_symptom not in self.graph:
            return []

        candidates = []
        # Find all ancestors of the symptom node
        ancestors = nx.ancestors(self.graph, target_symptom)

        for ancestor in ancestors:
            if not self.graph.nodes[ancestor]:
                continue

            # Compute path strength: product of edge weights along shortest path
            try:
                path = nx.shortest_path(self.graph, ancestor, target_symptom,
                                        weight=lambda u, v, d: 1 - d["weight"])
                path_edges = list(zip(path[:-1], path[1:]))
                path_confidence = 1.0
                for u, v in path_edges:
                    path_confidence *= self.graph[u][v]["weight"]

                node_data = self.graph.nodes[ancestor]
                candidates.append({
                    "event_type": ancestor,
                    "confidence": round(path_confidence, 4),
                    "causal_path": path,
                    "severity": node_data.get("severity", 0),
                    "timestamp": node_data.get("timestamp"),
                    "host": node_data.get("host"),
                    "metadata": node_data.get("metadata", {}),
                })
            except nx.NetworkXNoPath:
                continue

        # Rank by confidence × severity
        candidates.sort(
            key=lambda x: x["confidence"] * x["severity"],
            reverse=True
        )
        return candidates

    This graph is rebuilt every 60 seconds during an active incident. The compute_root_causes method walks backwards from the observed symptom — say, query_latency_spike — through every physical and statistical dependency edge, scoring each ancestor node by the cumulative product of edge weights along the causal path. The highest-scoring ancestor that has an observed event attached to it becomes the top root cause candidate.

    The key architectural decision is separating graph *topology* (which is pre-seeded from engine knowledge and updated slowly) from graph *node state* (which is populated in real time from live telemetry). This means the reasoning structure does not have to be re-derived during an incident — only the evidence layer changes.

    Wait Event Fingerprinting: Identifying Failure Modes from Execution Signatures

    Wait event distributions are one of the most reliable fingerprints for database failure modes. Rather than treating a PAGELATCH_EX wait or a Lock:relation wait as a single data point, an ARCA system learns the characteristic wait event *composition* of known failure modes and uses signature matching to accelerate diagnosis.

    Each failure mode has a signature: a probability distribution over wait event types observed during confirmed incidents of that type. The ARCA engine maintains a library of these signatures, built from historical incidents and augmented with known database engine behavior.

    For example, the "missing index causing full table scan under load" failure mode in PostgreSQL shows a characteristic distribution: Lock:relation waits decrease, IO:DataFileRead waits increase, autovacuum-related waits remain stable, and CPU becomes the dominant non-wait state. SQL Server's equivalent shows PAGEIOLATCH_SH climbing while CXPACKET (parallelism) waits either spike (if the planner attempts a parallel scan) or remain flat (if DOP is forced down by memory pressure).

    PostgreSQL:

    -- Capture wait event distribution snapshot for signature matching
-- The ARCA engine calls this every 30 seconds during elevated latency

WITH wait_snapshot AS (
    SELECT
        wait_event_type,
        wait_event,
        COUNT(*)                                        AS session_count,
        AVG(EXTRACT(EPOCH FROM (now() - query_start))) AS avg_wait_sec,
        SUM(EXTRACT(EPOCH FROM (now() - query_start))) AS total_wait_sec
    FROM pg_stat_activity
    WHERE state = 'active'
      AND wait_event IS NOT NULL
      AND pid != pg_backend_pid()
    GROUP BY wait_event_type, wait_event
),
total_sessions AS (
    SELECT COUNT(*) AS active_count
    FROM pg_stat_activity
    WHERE state = 'active'
      AND pid != pg_backend_pid()
)
SELECT
    w.wait_event_type,
    w.wait_event,
    w.session_count,
    w.avg_wait_sec,
    w.total_wait_sec,
    ROUND((w.session_count::numeric / NULLIF(t.active_count, 0)) * 100, 2)
        AS pct_of_active_sessions
FROM wait_snapshot w
CROSS JOIN total_sessions t
ORDER BY w.total_wait_sec DESC;

    SQL Server:

    -- Wait event distribution snapshot for SQL Server signature matching
-- Captures delta from baseline (cleared at analysis window start)

WITH current_waits AS (
    SELECT
        wait_type,
        waiting_tasks_count,
        wait_time_ms,
        max_wait_time_ms,
        signal_wait_time_ms,
        wait_time_ms - signal_wait_time_ms  AS resource_wait_time_ms
    FROM sys.dm_os_wait_stats
    WHERE wait_type NOT IN (
        -- Filter benign background waits
        'SLEEP_TASK','WAITFOR','BROKER_TO_FLUSH','BROKER_TASK_STOP',
        'CLR_AUTO_EVENT','DISPATCHER_QUEUE_SEMAPHORE','FT_IFTS_SCHEDULER_IDLE_WAIT',
        'HADR_WORK_QUEUE','HADR_FILESTREAM_IOMGR_IOCOMPLETION',
        'SQLTRACE_BUFFER_FLUSH','WAIT_XTP_OFFLINE_CKPT_NEW_LOG',
        'XE_DISPATCHER_WAIT','XE_TIMER_EVENT','SLEEP_DBSTARTUP',
        'SLEEP_DCOMSTARTUP','SLEEP_MASTERMDREADY','SLEEP_MASTERUPGRADED',
        'SLEEP_MSDBSTARTUP','SLEEP_SYSTEMTASK','SLEEP_TEMPDBSTARTUP',
        'SNI_HTTP_ACCEPT','SP_SERVER_DIAGNOSTICS_SLEEP'
    )
)
SELECT TOP 20
    wait_type,
    waiting_tasks_count,
    ROUND(wait_time_ms / 1000.0, 2)              AS wait_time_sec,
    ROUND(max_wait_time_ms / 1000.0, 2)          AS max_wait_sec,
    ROUND(resource_wait_time_ms / 1000.0, 2)     AS resource_wait_sec,
    ROUND(signal_wait_time_ms / 1000.0, 2)       AS signal_wait_sec,
    ROUND(
        wait_time_ms * 100.0 / NULLIF(SUM(wait_time_ms) OVER (), 0),
    2)                                            AS pct_total_wait
FROM current_waits
ORDER BY wait_time_ms DESC;

    The signature matching algorithm computes cosine similarity between the observed wait distribution vector and each stored failure mode vector. A match above 0.82 cosine similarity triggers a HIGH_CONFIDENCE classification for that failure mode; 0.65–0.82 triggers MODERATE_CONFIDENCE. Below 0.65, the system falls back to the causal graph traversal described in the previous section.

    This approach handles the common case where a failure mode is already known — it bypasses multi-layer correlation and delivers a diagnosis in one scoring pass. The causal graph is reserved for novel or compound failure modes where the signature library has no strong match.

    The signature library is self-expanding: every confirmed incident where a human DBA records the root cause is automatically converted into a new signature vector and merged into the library via a background reconciliation job. Over 6–12 months of operation, most environments reach 85–90% hit rates on known failure mode signatures, reducing median time-to-diagnosis below 90 seconds for incident types that have been seen before.

    LLM-Augmented Diagnosis: Structured Context Injection for Causal Reasoning

    The causal graph and wait event fingerprinting handle the *known* failure space well. What they cannot handle is novel compound failures — situations where three events interact in a way that has never appeared in training data. This is where large language model integration provides genuine value, not as a replacement for structured analysis but as a reasoning layer over structured evidence.

    The pattern is context injection with structured output enforcement. The ARCA pipeline constructs a detailed incident context object containing the ranked causal candidates from the graph engine, the top wait event signatures, recent configuration changes, active query plans for blocked sessions, and the relevant metric time series. This context is serialized into a structured prompt and submitted to an LLM (Claude 3.5 Sonnet and GPT-4o are both tested in production deployments). The LLM is instructed to reason over the provided evidence, propose a root cause hypothesis, and return a structured JSON response — never free text.

    The enforced output schema is:

    {
  "primary_cause": {
    "category": "lock_contention | resource_exhaustion | plan_regression | ...",
    "description": "string",
    "confidence": 0.0–1.0,
    "supporting_evidence": ["string", ...]
  },
  "contributing_factors": [...],
  "recommended_actions": [
    {
      "action_type": "terminate_query | reindex | update_stats | ...",
      "target": "string",
      "priority": "immediate | deferred",
      "estimated_impact": "string",
      "requires_human_approval": true | false
    }
  ],
  "confidence_in_diagnosis": 0.0–1.0,
  "requires_escalation": true | false,
  "escalation_reason": "string | null"
}

    The requires_human_approval flag on each recommended action is where the autonomous boundary is enforced. Actions that are statistically low-risk (killing a single idle-in-transaction session, running ANALYZE on a table, flushing plan cache for a specific query) are executed autonomously. Actions above the risk threshold (failover initiation, index creation on tables >50GB, parameter changes) are queued for human approval with a 15-minute timeout — if no response arrives, the system either escalates further or falls back to a pre-approved safe action.

    import anthropic
import json
from dataclasses import dataclass, asdict
from typing import Optional

@dataclass
class IncidentContext:
    db_type: str                    # "postgresql" | "sqlserver"
    db_host: str
    incident_start_utc: str
    symptom_description: str
    causal_candidates: list         # from CausalGraphBuilder.compute_root_causes()
    wait_event_snapshot: list       # from wait event query
    top_blocked_queries: list
    recent_config_changes: list     # last 2 hours from audit log
    deployment_events: list         # last 24 hours from CI/CD webhook
    active_query_plans: list        # explain output for top blocked queries
    metric_summary: dict            # p95 latency, error rate, tps delta

def build_diagnosis_prompt(ctx: IncidentContext) -> str:
    context_json = json.dumps(asdict(ctx), indent=2, default=str)
    return f"""You are a senior database reliability engineer performing root cause analysis.
You have been provided with structured telemetry from a {ctx.db_type} database cluster
experiencing degraded performance. Analyze the evidence and return ONLY a valid JSON object
conforming to the required schema.

Do not speculate beyond the provided evidence. If evidence is insufficient for high confidence,
set confidence_in_diagnosis below 0.6 and set requires_escalation to true.

INCIDENT CONTEXT:
{context_json}

Return ONLY valid JSON. No explanatory text before or after the JSON object."""

def get_llm_diagnosis(ctx: IncidentContext,
                      model: str = "claude-opus-4-5") -> dict:
    client = anthropic.Anthropic()
    prompt = build_diagnosis_prompt(ctx)

    message = client.messages.create(
        model=model,
        max_tokens=2048,
        messages=[{"role": "user", "content": prompt}]
    )

    raw_response = message.content[0].text.strip()

    # Strip markdown fences if present (some models add them despite instructions)
    if raw_response.startswith("```"):
        raw_response = raw_response.split("```")[1]
        if raw_response.startswith("json"):
            raw_response = raw_response[4:]

    try:
        diagnosis = json.loads(raw_response)
    except json.JSONDecodeError as e:
        # Fallback: log the raw response and return a minimum safe diagnosis
        return {
            "primary_cause": {"category": "unknown", "confidence": 0.0,
                               "description": "LLM response parse failure",
                               "supporting_evidence": []},
            "contributing_factors": [],
            "recommended_actions": [],
            "confidence_in_diagnosis": 0.0,
            "requires_escalation": True,
            "escalation_reason": f"LLM parse error: {str(e)}"
        }

    return diagnosis
