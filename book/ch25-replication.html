Chapter 25: Data Replication

Data replication is the process of copying and synchronizing data from one database instance to another, and it sits at the heart of nearly every high-availability, disaster recovery, and scale-out architecture you will encounter in production. Whether you are maintaining a warm standby that can take over within seconds of a primary failure, distributing read traffic across multiple replicas, or synchronizing data between geographically separated data centers, understanding how replication works at a deep level is what separates a reactive DBA from a proactive one. This chapter covers the replication mechanisms available in PostgreSQL and SQL Server, explains how they differ philosophically and technically, and builds toward the real-world decisions — configuration choices, monitoring strategies, failure handling, and latency management — that determine whether your replication setup holds up under production load or quietly degrades until it fails you at the worst possible moment.

---

How Replication Works: Models, Terminology, and Trade-offs

Before touching a single configuration file or wizard, it helps to understand the landscape of replication models, because both PostgreSQL and SQL Server offer multiple mechanisms, and choosing the wrong one for your workload can cost you months of rework.

Replication falls into two broad categories: physical replication and logical replication. Physical replication copies the raw byte-level changes from data files — it does not care what those bytes mean semantically, only that the replica ends up with identical disk pages as the primary. Logical replication, by contrast, copies higher-level change descriptions: which rows were inserted, updated, or deleted in which tables. Each model has a distinct set of strengths and constraints.

Physical replication is byte-for-byte faithful. The replica will match the primary exactly, including indexes, vacuum maps, and system catalog entries. This makes physical replicas fast to set up and simple to reason about — you never have to worry about schema divergence. The downside is that a physical replica must run the same major version of the database software, cannot have a different schema, and cannot be used to replicate selectively. You replicate everything or nothing.

Logical replication offers far more flexibility. You can replicate a single table, replicate to a different PostgreSQL major version, filter rows, or even replicate between heterogeneous systems. The trade-off is additional overhead and more opportunities for things to go wrong: schema changes on the publisher are not automatically propagated, sequences are not replicated, and large objects require special handling.

There is also a distinction between synchronous and asynchronous replication. In synchronous mode, the primary does not acknowledge a commit to the application until at least one replica has confirmed it received (and possibly applied) the change. This eliminates data loss on primary failure but adds commit latency proportional to network round-trip time. In asynchronous mode, the primary commits immediately and replication happens in the background. Asynchronous replication is far more common in practice because the latency impact of synchronous replication is difficult to absorb in OLTP workloads.

A third concept worth establishing early is replication lag — the difference in time or transaction volume between what has been committed on the primary and what has been applied on the replica. Understanding, measuring, and controlling replication lag is one of the recurring themes throughout this chapter.

SQL Server approaches replication with a broader menu of named technologies: Always On Availability Groups, Transactional Replication, Merge Replication, Snapshot Replication, and Log Shipping. PostgreSQL offers Streaming Replication (physical), Logical Replication, and the ecosystem of extensions like pglogical and tools like Patroni that build on top of these primitives. The core mechanisms are more alike than they appear on the surface, but the interfaces, configuration models, and failure behaviors differ significantly.

---

Streaming Replication in PostgreSQL

PostgreSQL's streaming replication is the foundation of almost every high-availability PostgreSQL deployment in production today. It works by having the primary continuously stream Write-Ahead Log (WAL) records to one or more standby servers, which replay those records to stay current. Because WAL is the authoritative record of every change made to the database — every insert, update, delete, index update, vacuum action — replaying WAL on a standby is guaranteed to produce an identical database state.

Setting up streaming replication starts with the primary's postgresql.conf and pg_hba.conf, a base backup, and a handful of parameters on the standby. On the primary, you need to set wal_level = replica (or logical if you plan to layer logical replication on top), configure max_wal_senders to the number of standby connections you want to support, and create a replication user.

-- On the primary: create a replication user
CREATE ROLE replicator WITH REPLICATION LOGIN PASSWORD 'strongpassword';

The standby is initialized from a base backup using pg_basebackup, and its behavior is controlled by a standby.signal file (PostgreSQL 12 and later) combined with primary_conninfo in postgresql.conf or postgresql.auto.conf. Once the standby starts, it connects to the primary, reports its current WAL position, and the primary begins streaming from that point forward.

You can monitor the state of all connected standbys from the primary using the pg_stat_replication view. This view is one of the most valuable tools in day-to-day replication management.

PostgreSQL:

-- Check replication status from the primary
SELECT
    client_addr,
    usename,
    application_name,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    (sent_lsn - replay_lsn) AS replication_lag_bytes,
    sync_state
FROM pg_stat_replication;

SQL Server:

-- Check Always On AG replica synchronization state
SELECT
    ar.replica_server_name,
    adc.database_name,
    drs.synchronization_state_desc,
    drs.synchronization_health_desc,
    drs.log_send_queue_size,
    drs.log_send_rate,
    drs.redo_queue_size,
    drs.redo_rate,
    drs.last_commit_time
FROM sys.dm_hadr_database_replica_states drs
JOIN sys.availability_replicas ar
    ON drs.replica_id = ar.replica_id
JOIN sys.availability_databases_cluster adc
    ON drs.group_database_id = adc.group_database_id;

The pg_stat_replication output shows four LSN (Log Sequence Number) positions for each standby: sent_lsn (how far WAL has been sent), write_lsn (how far the standby has written to its local disk), flush_lsn (how far it has been flushed to durable storage), and replay_lsn (how far it has actually been applied to the standby's data files). The gap between sent_lsn and replay_lsn in bytes gives you a raw measure of lag. For time-based lag, use the companion view pg_stat_replication_slots or calculate it from pg_wal_lsn_diff() combined with WAL generation rate.

Replication slots are a significant feature introduced in PostgreSQL 9.4. Without a replication slot, if a standby falls behind or disconnects, the primary is free to recycle WAL segments that the standby has not yet consumed. When the standby reconnects, it may find that the WAL it needs is gone, forcing a full re-sync. A replication slot causes the primary to retain WAL until the slot consumer acknowledges it. This solves the "standby missed WAL" problem but introduces a new one: if a standby with a slot goes offline indefinitely, the primary's pg_wal directory will grow without bound, potentially filling the disk. In production, always monitor slot lag with an alert, and consider whether an unmonitored slot is worth the risk.

PostgreSQL:

-- Check replication slot status and retained WAL
SELECT
    slot_name,
    slot_type,
    active,
    active_pid,
    confirmed_flush_lsn,
    pg_wal_lsn_diff(pg_current_wal_lsn(), confirmed_flush_lsn) AS lag_bytes,
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), confirmed_flush_lsn)
    ) AS lag_pretty
FROM pg_replication_slots;

SQL Server:

-- No direct equivalent to slots; check log send queue in AG DMV
SELECT
    ar.replica_server_name,
    drs.log_send_queue_size AS retained_log_kb,
    drs.log_send_rate AS send_rate_kb_per_sec,
    CASE
        WHEN drs.log_send_rate > 0
        THEN drs.log_send_queue_size / drs.log_send_rate
        ELSE NULL
    END AS estimated_catchup_seconds
FROM sys.dm_hadr_database_replica_states drs
JOIN sys.availability_replicas ar
    ON drs.replica_id = ar.replica_id
WHERE drs.is_local = 0;

Synchronous streaming replication in PostgreSQL is controlled by the synchronous_standby_names parameter. Setting it to the name of a standby forces the primary to wait for that standby's flush_lsn (or apply_lsn with synchronous_commit = remote_apply) before acknowledging commits. The FIRST N (...) and ANY N (...) syntax allows you to express quorum-based synchronous replication — for example, ANY 2 (standby1, standby2, standby3) ensures any two of three standbys must confirm before commit proceeds, adding resilience against any single standby being slow.

Always On Availability Groups in SQL Server

SQL Server's primary high-availability and disaster recovery mechanism for modern deployments is Always On Availability Groups, introduced in SQL Server 2012 and substantially improved through 2016, 2017, 2019, and 2022. An Availability Group (AG) is a container for one or more user databases that fail over together as a unit. The AG has a primary replica and one or more secondary replicas, with data synchronized via the transaction log — conceptually parallel to WAL-based streaming in PostgreSQL.

AGs require Windows Server Failover Clustering (WSFC) in traditional deployments, though SQL Server 2017 introduced support for Linux and a "clusterless" configuration using the CLUSTER_TYPE = NONE option, which is useful for read-scale scenarios where automatic failover is not required. SQL Server 2022 further improved cross-platform flexibility.

Creating an AG involves several steps: enabling the AG feature on each SQL Server instance, creating the databases on the primary, taking full and log backups, restoring them to the secondaries with NORECOVERY, and then joining them to the AG. The AG listener — a virtual network name and IP address — provides a single connection endpoint that redirects clients to the current primary.

Each secondary can be configured independently as synchronous or asynchronous and as readable or not readable. A readable secondary allows reporting queries and backups to be offloaded from the primary, which is one of the most compelling operational features of AGs. The readable secondary is an eventually-consistent read-only view of the primary's data, with snapshot isolation handling the concurrency.

-- SQL Server: Check AG health and failover readiness
SELECT
    ag.name AS ag_name,
    ar.replica_server_name,
    ar.availability_mode_desc,
    ar.failover_mode_desc,
    ars.role_desc,
    ars.operational_state_desc,
    ars.connected_state_desc,
    ars.synchronization_health_desc,
    ars.last_connect_error_description
FROM sys.availability_groups ag
JOIN sys.availability_replicas ar
    ON ag.group_id = ar.group_id
JOIN sys.dm_hadr_availability_replica_states ars
    ON ar.replica_id = ars.replica_id;
-- PostgreSQL equivalent: check streaming replication state
SELECT
    pid,
    usename,
    application_name,
    client_addr,
    state,
    sync_state,
    sent_lsn,
    replay_lsn,
    write_lag,
    flush_lag,
    replay_lag
FROM pg_stat_replication
ORDER BY application_name;

Failover behavior is a major consideration. In an AG with synchronous replicas and WSFC-based automatic failover configured, SQL Server can fail over to a synchronous secondary automatically when the primary becomes unavailable, with zero data loss guaranteed by definition of synchronous replication. Asynchronous replicas can only be used for manual failover, and because they may lag behind, a forced failover may involve data loss. SQL Server quantifies this with the recovery point objective (RPO) metric visible in the DMVs — the amount of committed transaction log that has not yet been hardened on the secondary.

One area where AG configuration frequently trips up teams is the impact of synchronous secondaries on primary throughput. Every commit on the primary must wait for acknowledgment from all synchronous secondaries. In environments with cross-datacenter replication where network latency is 5–20 ms, synchronous commit can cut OLTP throughput dramatically. A common production pattern is to have synchronous replication within a datacenter for automatic failover and asynchronous replication to a distant DR site, accepting potential data loss for the DR replica in exchange for no performance impact on the primary.

Logical Replication: Selective, Flexible, and Version-Aware

Logical replication in PostgreSQL became a built-in feature in version 10, and it addresses use cases that physical streaming replication cannot — specifically, replicating a subset of tables, replicating between different major versions during an upgrade, and building custom data pipelines. SQL Server's Transactional Replication serves a similar function in the SQL Server ecosystem.

PostgreSQL logical replication uses a publisher/subscriber model. The publisher defines a publication — a named collection of tables (or all tables in a schema) whose changes should be broadcast. The subscriber creates a subscription pointing at a publisher's publication and begins consuming those changes.

PostgreSQL:

-- On the publisher: create a publication for specific tables
CREATE PUBLICATION sales_pub
    FOR TABLE orders, order_items, customers
    WITH (publish = 'insert, update, delete');

-- On the subscriber: create a subscription
CREATE SUBSCRIPTION sales_sub
    CONNECTION 'host=primary-db port=5432 dbname=sales user=replicator password=strongpassword'
    PUBLICATION sales_pub;

SQL Server:

-- SQL Server: Create a transactional publication (via T-SQL)
USE [sales];
EXEC sp_replicationdboption
    @dbname = N'sales',
    @optname = N'publish',
    @value = N'true';

EXEC sp_addpublication
    @publication = N'sales_pub',
    @status = N'active',
    @sync_method = N'concurrent',
    @repl_freq = N'continuous';

EXEC sp_addarticle
    @publication = N'sales_pub',
    @article = N'orders',
    @source_object = N'orders',
    @type = N'logbased';

The mechanics behind PostgreSQL logical replication involve the logical decoding infrastructure. When wal_level = logical, PostgreSQL writes additional information into the WAL that makes it possible to reconstruct row-level changes rather than just page-level changes. The logical replication worker on the subscriber connects to the publisher, opens a replication slot, and receives a decoded stream of changes that it applies to local tables.

A key limitation to understand: logical replication in PostgreSQL does not replicate DDL changes. If you ALTER TABLE orders ADD COLUMN discount_pct numeric on the publisher, that change must be manually applied on the subscriber before the replication stream attempts to apply rows with that column — otherwise the subscription will error out and stop. This is a fundamental operational difference from physical streaming replication, where DDL is just more WAL and gets replicated automatically.

Sequences are another gap. Because sequences generate values independently on each instance, NEXTVAL calls on the publisher do not update the sequence on the subscriber. After a failover or migration using logical replication, sequence positions on the new primary must be manually advanced past any values that were used by the old primary.

SQL Server's Transactional Replication also reads from the transaction log but does so through the Log Reader Agent, a component that runs as a SQL Server Agent job. The Log Reader scans the publisher's transaction log for changes marked for replication and writes them to the distribution database — a staging database that holds the pending changes. The Distribution Agent then reads from the distribution database and applies changes to the subscriber. This three-tier architecture (publisher → distributor → subscriber) gives SQL Server replication more flexibility in routing and scheduling but adds more moving parts to monitor.

Merge Replication, which SQL Server offers and PostgreSQL does not have a direct equivalent of, handles bidirectional synchronization where both publisher and subscriber can make changes that need to be reconciled. This is powerful for occasionally-connected scenarios like mobile or branch office databases, but conflict resolution — what happens when the same row is updated differently on both sides — is notoriously complex to configure correctly and debug. Most modern architectures avoid Merge Replication in favor of application-level conflict handling or eventual-consistency designs.

Monitoring Replication Health and Managing Lag in Production

A replication topology that is not actively monitored is a topology that will surprise you during an incident. The operational reality of replication is that lag accumulates gradually under high load, replica connections drop due to network blips, and WAL accumulates on paused slots — all silently, unless you have built the observability layer.

The fundamental metric to watch is replication lag. In PostgreSQL, time-based lag (more meaningful than byte-based lag for most alerting purposes) became directly available in pg_stat_replication starting with PostgreSQL 10, in the write_lag, flush_lag, and replay_lag columns. These report how long ago the primary wrote data that the standby has yet to write, flush, or replay, respectively.

PostgreSQL:

-- Monitor replication lag with time-based metrics
SELECT
    application_name,
    client_addr,
    sync_state,
    state,
    pg_size_pretty(pg_wal_lsn_diff(sent_lsn, replay_lsn)) AS byte_lag,
    write_lag,
    flush_lag,
    replay_lag,
    CASE
        WHEN replay_lag > interval '30 seconds' THEN 'WARNING'
        WHEN replay_lag > interval '5 minutes' THEN 'CRITICAL'
        ELSE 'OK'
    END AS lag_status
FROM pg_stat_replication
ORDER BY replay_lag DESC NULLS LAST;

SQL Server:

-- Monitor AG lag and estimated data loss window
SELECT
    ar.replica_server_name,
    adc.database_name,
    drs.synchronization_state_desc,
    drs.log_send_queue_size AS unsent_log_kb,
    drs.redo_queue_size AS unapplied_log_kb,
    DATEDIFF(SECOND, drs.last_redone_time, GETUTCDATE()) AS seconds_since_last_redo,
    ar.availability_mode_desc,
    drs.estimated_data_loss_seconds
FROM sys.dm_hadr_database_replica_states drs
JOIN sys.availability_replicas ar
    ON drs.replica_id = ar.replica_id
JOIN sys.availability_databases_cluster adc
    ON drs.group_database_id = adc.group_database_id
WHERE drs.is_local = 0
ORDER BY drs.redo_queue_size DESC;

Lag spikes almost always have one of three causes. The first is a sudden burst of writes on the primary — a large batch job, a bulk load, or a runaway update — that temporarily outpaces the replication throughput. The second is I/O saturation on the standby, where the standby's disk cannot keep up with writing and flushing the incoming WAL. The third is network bandwidth exhaustion between primary and standby, particularly relevant for cross-datacenter replication or when WAL compression is not enabled.

wal_compression in PostgreSQL is worth enabling in most environments. It compresses full-page images in WAL (which can be verbose immediately after a checkpoint) and reduces the bandwidth consumed by replication. The CPU cost is usually negligible compared to the bandwidth savings.

In SQL Server, the equivalent concern is log send queue growth. The log_send_rate from dm_hadr_database_replica_states tells you how fast log is being shipped. If log_send_queue_size is growing faster than log_send_rate can drain it, you have a sending bottleneck — either network bandwidth, or the secondary's redo rate is causing backpressure.

For proactive alerting, a production-grade setup should alert on:

  • → Replication lag (bytes and time) exceeding thresholds — PostgreSQL alerts at >30s, page at >5min is a reasonable starting point
  • → Replication slots that are inactive or whose lag exceeds a WAL retention limit (e.g., 10 GB)
  • → Any standby that has been disconnected for more than a configurable window
  • → In SQL Server, any AG secondary whose synchronization health is not HEALTHY

    • For PostgreSQL environments, integrating with Prometheus via postgres_exporter makes all the pg_stat_replication metrics available as time-series data, which is invaluable for understanding lag trends over time. A lag spike that lasts two minutes during a nightly batch run is normal and acceptable; a spike that lasts two minutes every hour at random is a symptom that needs investigation.

    Planned maintenance on a physical standby in PostgreSQL — patching the OS, upgrading hardware — is straightforward: take it out of the pool, do the work, and bring it back. The standby will reconnect and stream forward from where it paused, as long as the WAL it needs has not been recycled (use replication slots to guarantee this). After a long maintenance window, you may need to increase wal_keep_size or rely on a slot to ensure the primary retained enough WAL.

    Failover, Promotion, and Post-Failover Operations

    Replication topology is only as valuable as your ability to act on it when the primary fails. The failover process — promoting a standby to primary — looks straightforward on paper but involves enough edge cases and post-promotion tasks that it deserves careful planning and regular testing.

    In PostgreSQL, promoting a standby to primary is accomplished either by calling pg_promote() from a session connected to the standby, or by creating a promote trigger file in the data directory (the legacy method, still functional). After promotion, the standby exits recovery mode, begins accepting writes, generates a new timeline, and starts advancing its own WAL sequence.

    The timeline mechanism is a critical safety feature. When a new primary is promoted, it starts writing WAL on a new timeline identifier. Any old standbys that were connected to the original primary will refuse to follow the new primary's WAL if they detect a timeline conflict, preventing the split-brain scenario where two instances both believe they are primary and both accept writes. The DBA must explicitly reconfigure old standbys to follow the new primary, which typically involves either running pg_rewind to resync the standby's WAL history or rebuilding the standby from a base backup.

    pg_rewind is a tool that compares the diverged standby's WAL history against the new primary and rewinds the standby's data directory to the point of divergence, then lets streaming replication fill in the gap. It is faster than a full base backup for standbys that are not far behind, but it requires that wal_log_hints is enabled on the primary, or that checksums were enabled at initdb time.

    In SQL Server Always On, failover is more formalized. A planned manual failover to a synchronous secondary involves connecting to the primary, verifying the secondary is synchronized, and executing the failover command:

    -- SQL Server: Initiate a planned manual failover to a synchronous secondary
-- (Connect to the secondary you want to promote)
ALTER AVAILABILITY GROUP [SalesAG] FAILOVER;
    -- PostgreSQL: Promote a standby to primary programmatically
-- (Connect to the standby)
SELECT pg_promote(wait := true, wait_seconds := 60);
