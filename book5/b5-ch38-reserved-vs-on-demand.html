Chapter 38: Reserved Instances vs On-Demand: The Math

Cloud compute pricing is one of the most consequential decisions a cloud DBA makes, yet it's also one of the most frequently delegated to finance teams who lack the technical context to optimize it properly. The result is organizations either overpaying for on-demand flexibility they don't need, or committing to Reserved Instances at the wrong tier and locking in costs that hurt when the architecture changes. This chapter walks through the actual pricing mechanics across AWS RDS, Azure SQL Database, and Google Cloud SQL—with the arithmetic laid out plainly—so you can make purchasing recommendations backed by data rather than guesswork. The patterns here apply regardless of which cloud you're standardizing on, and the tooling examples show how to audit your existing spend before signing anything.

---

38.1 How Cloud Database Pricing Works Under the Hood

Before running any numbers, you need a mental model of what you're actually buying. On-premises, you paid for hardware, licenses, and labor separately. In the cloud, those costs are bundled into an hourly or per-second rate that changes based on your commitment level.

AWS RDS Pricing Layers

RDS charges have four components that stack on top of each other:

1. Instance hours — the compute cost, billed per second with a one-minute minimum

2. Storage — GP3 is $0.115/GB-month, IO1 is $0.125/GB-month plus $0.10/provisioned IOPS-month

3. I/O requests — GP2 and GP3 include free I/O up to a baseline; io1/io2 charges per million requests

4. Data transfer — inbound is free; outbound to the internet starts at $0.09/GB; cross-AZ replication traffic for Multi-AZ RDS costs $0.02/GB in most regions

The instance hour rate is where Reserved Instances make their impact. A db.r6g.2xlarge running PostgreSQL on-demand in us-east-1 costs approximately $0.48/hour as of mid-2024. That's $4,205/year. The same instance on a 1-year No Upfront Reserved commitment costs $0.296/hour—$2,594/year. The 3-year All Upfront Reserved drops to roughly $0.185/hour equivalent, or $1,621/year. Those are not rounding errors. They are 38% and 61% discounts respectively.

Azure SQL Database Pricing Layers

Azure SQL Database pricing depends on the service tier you're running. The General Purpose tier bills per vCore-hour. The Business Critical tier carries a roughly 2.7x multiplier for the same vCore count due to the in-memory OLTP and local SSD backing.

  • → General Purpose, 8 vCores, on-demand: approximately $0.9208/hour ($8,066/year)
  • → General Purpose, 8 vCores, 1-year Reserved: approximately $0.5714/hour ($5,005/year)
  • → General Purpose, 8 vCores, 3-year Reserved: approximately $0.4071/hour ($3,566/year)

    • Azure's Reserved Instance discounts apply to the compute portion only. Storage is charged separately at $0.115/GB-month for General Purpose and is not discountable via reservations.

    Google Cloud SQL and AlloyDB Pricing Layers

    Cloud SQL charges per vCPU-hour and per GB RAM-hour separately, plus a separate storage charge. This is different from AWS and Azure, which bundle CPU and RAM into instance family pricing. For a db-custom-8-32768 (8 vCPUs, 32 GB RAM) running PostgreSQL:

  • → On-demand: $0.0413/vCPU-hour + $0.007/GB-hour = $0.3304/hour ($2,894/year for compute only)
  • → Committed Use Discount, 1-year: 25% off compute = $0.2478/hour ($2,171/year)
  • → Committed Use Discount, 3-year: 52% off compute = $0.1586/hour ($1,389/year)

    • Google calls these Committed Use Discounts (CUDs) rather than Reserved Instances, but the economic model is the same: pay upfront or commit to usage in exchange for a lower hourly rate.

    38.2 Running the Break-Even Analysis

    The decision to commit always comes down to one question: how long will this workload run at this scale? If the answer is uncertain, on-demand is the defensible choice regardless of the percentage discount. If the answer is "at least 12 months at this capacity," the math almost always favors reserving.

    The break-even formula is straightforward:

    Break-Even Hours = Upfront Cost / (On-Demand Hourly Rate - Reserved Hourly Rate)

    For AWS RDS db.r6g.2xlarge PostgreSQL, us-east-1, 1-year Partial Upfront:

  • → On-demand: $0.48/hour
  • → Reserved upfront payment: ~$1,100
  • → Reserved hourly rate: ~$0.176/hour
    • Break-Even Hours = $1,100 / ($0.48 - $0.176) = $1,100 / $0.304 = 3,618 hours ≈ 150 days

    If the instance runs for more than 150 days, the Partial Upfront option has already paid for itself compared to pure on-demand. After 365 days, the total savings are:

    Savings = (On-Demand Annual - Reserved Annual)
        = ($0.48 × 8760) - ($1,100 + $0.176 × 8760)
        = $4,205 - ($1,100 + $1,542)
        = $4,205 - $2,642 = $1,563

    That's 37% savings, which on a production cluster with three database instances adds up to nearly $4,700 per year from a single commitment decision.

    The following Python script automates this analysis across multiple instance types by pulling current pricing from the AWS Price List API:

    import boto3
import json
from decimal import Decimal

def get_rds_ondemand_price(region_code, instance_class, engine, deployment_option):
    """
    Retrieves on-demand hourly price for an RDS instance.
    Note: AWS Price List API is only available in us-east-1.
    """
    client = boto3.client('pricing', region_name='us-east-1')
    
    filters = [
        {'Type': 'TERM_MATCH', 'Field': 'instanceType',        'Value': instance_class},
        {'Type': 'TERM_MATCH', 'Field': 'databaseEngine',      'Value': engine},
        {'Type': 'TERM_MATCH', 'Field': 'deploymentOption',    'Value': deployment_option},
        {'Type': 'TERM_MATCH', 'Field': 'location',            'Value': region_code},
        {'Type': 'TERM_MATCH', 'Field': 'termType',            'Value': 'OnDemand'},
    ]
    
    response = client.get_products(
        ServiceCode='AmazonRDS',
        Filters=filters,
        MaxResults=1
    )
    
    if not response['PriceList']:
        return None
    
    price_item = json.loads(response['PriceList'][0])
    terms = price_item['terms']['OnDemand']
    
    for term_key in terms:
        for price_dimension_key in terms[term_key]['priceDimensions']:
            pd = terms[term_key]['priceDimensions'][price_dimension_key]
            if pd['unit'] == 'Hrs':
                return Decimal(pd['pricePerUnit']['USD'])
    
    return None

def calculate_breakeven(ondemand_hourly, reserved_hourly, upfront_cost):
    """
    Returns break-even in hours and days.
    """
    hourly_savings = ondemand_hourly - reserved_hourly
    if hourly_savings <= 0:
        return None, None
    
    breakeven_hours = float(upfront_cost) / float(hourly_savings)
    breakeven_days  = breakeven_hours / 24
    return breakeven_hours, breakeven_days

def project_annual_savings(ondemand_hourly, reserved_hourly, upfront_cost, 
                            utilization_hours=8760):
    """
    Projects total cost and savings over one year at a given utilization.
    Assumes utilization_hours = 8760 for 24x7 workloads.
    """
    ondemand_annual  = float(ondemand_hourly) * utilization_hours
    reserved_annual  = float(upfront_cost) + (float(reserved_hourly) * utilization_hours)
    savings          = ondemand_annual - reserved_annual
    savings_pct      = (savings / ondemand_annual) * 100
    
    return {
        'on_demand_annual':  round(ondemand_annual, 2),
        'reserved_annual':   round(reserved_annual, 2),
        'savings':           round(savings, 2),
        'savings_pct':       round(savings_pct, 1)
    }

# Example: db.r6g.2xlarge, PostgreSQL, Multi-AZ, us-east-1
# These rates are illustrative; pull live rates from AWS Price List or Savings Plans API
instances_to_evaluate = [
    {
        'instance_class':    'db.r6g.2xlarge',
        'engine':            'PostgreSQL',
        'deployment':        'Multi-AZ',
        'ondemand_hourly':   0.960,   # Multi-AZ doubles compute cost
        'reserved_upfront':  2200,
        'reserved_hourly':   0.352,
        'term':              '1yr Partial Upfront'
    },
    {
        'instance_class':    'db.r6g.4xlarge',
        'engine':            'PostgreSQL',
        'deployment':        'Multi-AZ',
        'ondemand_hourly':   1.920,
        'reserved_upfront':  4400,
        'reserved_hourly':   0.704,
        'term':              '1yr Partial Upfront'
    }
]

for inst in instances_to_evaluate:
    beh, bed = calculate_breakeven(
        inst['ondemand_hourly'],
        inst['reserved_hourly'],
        inst['reserved_upfront']
    )
    projection = project_annual_savings(
        inst['ondemand_hourly'],
        inst['reserved_hourly'],
        inst['reserved_upfront']
    )
    
    print(f"\n{'='*60}")
    print(f"Instance:      {inst['instance_class']} ({inst['deployment']})")
    print(f"Term:          {inst['term']}")
    print(f"Break-even:    {beh:.0f} hours ({bed:.0f} days)")
    print(f"On-Demand/yr:  ${projection['on_demand_annual']:,.2f}")
    print(f"Reserved/yr:   ${projection['reserved_annual']:,.2f}")
    print(f"Annual savings: ${projection['savings']:,.2f} ({projection['savings_pct']}%)")

    Running this against your actual instance inventory gives you a defensible business case for RI purchases without relying on vendor-provided calculators that tend to assume 100% utilization.

    38.3 Reserved Instance Mechanics: What the Contracts Actually Say

    Understanding the contract structure prevents the most common RI mistakes—buying the wrong scope, overcommitting on instance family, or missing the difference between Convertible and Standard RIs.

    AWS RI Types and Flexibility Rules

    AWS RDS offers two RI types:

  • → **Standard Reserved Instances**: Maximum discount, but instance family is fixed. You cannot modify a `db.r6g` Standard RI to cover a `db.r7g` instance.
  • → **Convertible Reserved Instances**: Roughly 10–15% less discount than Standard, but you can exchange them for different instance types, regions, or database engines at any time during the term.

    • The flexibility argument for Convertible RIs is real when you're within 18–24 months of a likely architecture change—say, a PostgreSQL 13 cluster that will migrate to Aurora PostgreSQL 16 within the RI term. Standard RIs would strand that commitment; Convertible RIs let you exchange them.

    AWS also separates RI scope into Regional and Zonal:

  • → Regional RIs apply to any instance of the matching type in any AZ within that region and get Availability Zone flexibility
  • → Zonal RIs apply to a specific AZ only, but guarantee capacity reservation in that AZ—useful if you're running in capacity-constrained regions

    • For production databases, Regional scope is almost always correct unless you have a specific capacity reservation requirement.

    Checking your current RDS RI inventory and utilization:

    # List all current RDS Reserved Instances

    Azure SQL Reserved Capacity

    Azure sells reserved capacity for SQL Database and SQL Managed Instance at the vCore level. A reservation for 8 vCores of General Purpose covers any General Purpose database in the selected region that together consume 8 vCores—whether that's one 8-vCore database or two 4-vCore databases. This pooling behavior is more flexible than AWS's per-instance model.

    # View Azure SQL reserved capacity offerings
az reservations catalog show \
  --service-name "sql" \
  --location eastus \
  --output table

# Purchase a 1-year reservation for 8 vCores General Purpose SQL Database
# Note: az reservations requires the reservations extension
az extension add --name reservations

az reservations reservation-order purchase \
  --reservation-order-id "$(uuidgen)" \
  --sku "SQL_Database_General_Purpose_Compute_8_vCores" \
  --location eastus \
  --reserved-resource-type "SqlDatabases" \
  --billing-scope "/subscriptions/your-subscription-id" \
  --term P1Y \
  --billing-plan Upfront \
  --quantity 1 \
  --display-name "prod-sqldb-gp-8vcores-1yr"

    Google Cloud Committed Use Discounts

    GCP's CUD model commits to a resource quantity (vCPUs and memory) rather than a specific instance type. This means a 3-year CUD for 8 vCPUs and 32 GB of memory in us-central1 can cover any Cloud SQL instance in that region as long as the aggregate resource consumption matches the committed amount.

    # Create a 1-year Committed Use Discount for Cloud SQL compute
# CUDs are created at the project level via the Compute Engine API
gcloud compute commitments create prod-cloudsql-commitment \
  --plan 12-month \
  --region us-central1 \
  --resources vcpu=8,memory=32GB \
  --type GENERAL_PURPOSE \
  --project your-project-id

# View existing commitments and their utilization
gcloud compute commitments list \
  --region us-central1 \
  --project your-project-id \
  --format="table(name, plan, status, startTimestamp, endTimestamp)"

    38.4 Sizing Commitments Against Real Workload Data

    The most expensive RI mistake is buying capacity based on current provisioning rather than actual utilization. If you're running a db.r6g.4xlarge at 15% average CPU utilization because the instance was provisioned for peak load three years ago, committing to that instance class locks in overspend.

    Before any RI purchase, pull at least 90 days of utilization data. For AWS RDS, that data lives in CloudWatch.

    import boto3
from datetime import datetime, timedelta
import statistics

def get_rds_utilization_stats(db_instance_id, region, days=90):
    """
    Retrieves CPU, FreeableMemory, DatabaseConnections, and ReadIOPS/WriteIOPS
    for the last N days to inform RI sizing decisions.
    """
    cloudwatch = boto3.client('cloudwatch', region_name=region)
    
    end_time   = datetime.utcnow()
    start_time = end_time - timedelta(days=days)
    
    metrics = {
        'CPUUtilization':       {'stat': 'Average', 'unit': 'Percent'},
        'FreeableMemory':       {'stat': 'Average', 'unit': 'Bytes'},
        'DatabaseConnections':  {'stat': 'Maximum', 'unit': 'Count'},
        'ReadIOPS':             {'stat': 'Average', 'unit': 'Count/Second'},
        'WriteIOPS':            {'stat': 'Average', 'unit': 'Count/Second'},
    }
    
    results = {}
    
    for metric_name, config in metrics.items():
        response = cloudwatch.get_metric_statistics(
            Namespace='AWS/RDS',
            MetricName=metric_name,
            Dimensions=[{'Name': 'DBInstanceIdentifier', 'Value': db_instance_id}],
            StartTime=start_time,
            EndTime=end_time,
            Period=3600,  # 1-hour granularity
            Statistics=[config['stat']]
        )
        
        datapoints = sorted(response['Datapoints'], key=lambda x: x['Timestamp'])
        values = [dp[config['stat']] for dp in datapoints]
        
        if values:
            results[metric_name] = {
                'mean':    round(statistics.mean(values), 2),
                'median':  round(statistics.median(values), 2),
                'p95':     round(sorted(values)[int(len(values) * 0.95)], 2),
                'p99':     round(sorted(values)[int(len(values) * 0.99)], 2),
                'max':     round(max(values), 2),
                'samples': len(values)
            }
    
    return results

def recommend_instance_class(stats, current_instance_class):
    """
    Provides a rough right-sizing recommendation based on CPU utilization.
    In practice, combine with memory pressure and connection count analysis.
    """
    cpu_p95 = stats.get('CPUUtilization', {}).get('p95', 100)
    cpu_mean = stats.get('CPUUtilization', {}).get('mean', 100)
    
    recommendation = current_instance_class
    notes = []
    
    if cpu_p95 < 20:
        notes.append(f"P95 CPU {cpu_p95}% — strong candidate for downsizing one instance class")
    elif cpu_p95 < 40:
        notes.append(f"P95 CPU {cpu_p95}% — current sizing is adequate; commit at this level")
    elif cpu_p95 < 70:
        notes.append(f"P95 CPU {cpu_p95}% — healthy utilization; commit at this level")
    else:
        notes.append(f"P95 CPU {cpu_p95}% — consider upsizing before committing")
    
    return recommendation, notes

# Example usage
db_instance_id  = 'prod-postgres-primary'
region          = 'us-east-1'
current_class   = 'db.r6g.4xlarge'

print(f"Analyzing {db_instance_id} over 90 days...")
stats = get_rds_utilization_stats(db_instance_id, region, days=90)

for metric, values in stats.items():
    print(f"\n{metric}:")
    print(f"  Mean: {values['mean']}  |  Median: {values['median']}")
    print(f"  P95:  {values['p95']}   |  P99:    {values['p99']}   |  Max: {values['max']}")

recommended_class, notes = recommend_instance_class(stats, current_class)
print(f"\nRight-sizing analysis for {current_class}:")
for note in notes:
    print(f"  → {note}")

    For Azure SQL, the equivalent utilization data comes from Azure Monitor. The sys.dm_db_resource_stats DMV provides per-database resource consumption at 15-second intervals for the last hour; sys.resource_stats in the master database covers the last 14 days at 1-minute granularity.
