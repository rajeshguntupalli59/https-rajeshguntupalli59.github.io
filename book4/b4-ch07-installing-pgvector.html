Chapter 7: Installing and Configuring pgvector

pgvector brings vector similarity search directly into PostgreSQL, letting you store embeddings as a native column type and query them with familiar SQL semantics. For database professionals already invested in PostgreSQL infrastructure, this is the path of least resistance into production AI applications — no new operational surface, no separate replication topology, no additional monitoring stack. This chapter walks through every stage of a pgvector deployment: from binary installation through index configuration, connection pooling, and the operational habits that keep vector workloads from interfering with the transactional workloads sharing the same cluster.

---

Getting pgvector Installed

pgvector is a PostgreSQL extension, so the installation path depends on how PostgreSQL itself was installed and where you're running it.

On Debian/Ubuntu with the PGDG repository:

# shell
# Ensure you have the PostgreSQL contrib packages and build tools
sudo apt-get update
sudo apt-get install -y postgresql-server-dev-16 build-essential git

# Clone and build pgvector
cd /tmp
git clone --branch v0.7.0 https://github.com/pgvector/pgvector.git
cd pgvector
make
sudo make install

# Confirm the shared library landed in the right place
ls $(pg_config --pkglibdir)/vector.so
ls $(pg_config --sharedir)/extension/vector*

On RHEL/Rocky Linux 9:

# shell
# Install via the PostgreSQL YUM repository (preferred for production)
sudo dnf install -y pgvector_16

# If building from source on RHEL:
sudo dnf install -y postgresql16-devel gcc git make
cd /tmp && git clone --branch v0.7.0 https://github.com/pgvector/pgvector.git
cd pgvector && make && sudo make install

On macOS with Homebrew (development only):

# shell
brew install pgvector

# If PostgreSQL was installed via Homebrew separately:
brew install postgresql@16
export PATH="/opt/homebrew/opt/postgresql@16/bin:$PATH"
cd /tmp && git clone --branch v0.7.0 https://github.com/pgvector/pgvector.git
cd pgvector && make && make install

Once the shared library is in place, enable the extension per database:

-- PostgreSQL / pgvector
-- Run as a superuser in the target database
CREATE EXTENSION IF NOT EXISTS vector;

-- Verify
SELECT extname, extversion FROM pg_extension WHERE extname = 'vector';
--  extname | extversion
-- ---------+------------
--  vector  | 0.7.0

No postgresql.conf changes are required to load the extension — unlike some extensions, pgvector does not appear in shared_preload_libraries. The CREATE EXTENSION call is sufficient.

Managed services: On AWS RDS for PostgreSQL 15+, Aurora PostgreSQL 15.2+, Google Cloud SQL for PostgreSQL 14+, Azure Database for PostgreSQL Flexible Server 13+, Supabase, and Neon, pgvector is available as a first-class extension requiring only the CREATE EXTENSION command. No binary installation is needed. Check the managed service's documentation for the maximum supported pgvector version, as it typically lags the upstream release by one or two minor versions.

The vector Data Type and Basic Schema Design

pgvector introduces three types: vector (exact 32-bit floats), halfvec (16-bit floats, introduced in 0.7.0), and sparsevec (sparse vectors for models like SPLADE). The vector type is what you will use for the vast majority of embedding workloads.

A vector column is defined with a fixed dimensionality:

-- PostgreSQL / pgvector
-- Standard embedding table for a document store
CREATE TABLE documents (
    id          BIGSERIAL PRIMARY KEY,
    content     TEXT NOT NULL,
    metadata    JSONB,
    embedding   vector(1536),          -- OpenAI text-embedding-3-small dimensions
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- For models with different dimensionalities:
-- text-embedding-3-large: vector(3072)
-- text-embedding-ada-002: vector(1536)
-- Cohere embed-english-v3.0: vector(1024)
-- nomic-embed-text-v1.5: vector(768)
-- all-MiniLM-L6-v2: vector(384)

-- Insert a row with an embedding (literal syntax for testing)
INSERT INTO documents (content, metadata, embedding)
VALUES (
    'PostgreSQL is a powerful open source relational database.',
    '{"source": "manual", "category": "database"}',
    '[0.021, -0.014, 0.033, ...]'::vector(1536)
);

The dimensionality is enforced at the column level, not the type level — you cannot insert a 768-dimensional vector into a vector(1536) column. This is the equivalent of a type mismatch; PostgreSQL will throw an error, which is exactly the behavior you want when different embedding models produce incompatible vector spaces.

Null handling: embedding columns can be NULL, and NULL embeddings are excluded from similarity searches automatically. This is useful when you ingest document metadata before the embedding pipeline has processed the content.

The halfvec type: For production deployments where storage is a constraint, halfvec(1536) cuts storage per vector from 6,144 bytes to 3,072 bytes at the cost of slight precision loss. For most embedding search workloads, the precision difference is negligible — retrieval quality benchmarks on typical RAG workloads show less than 0.5% change in recall@10 when switching from vector to halfvec with OpenAI embeddings.

-- PostgreSQL / pgvector
-- Comparing storage footprints
SELECT 
    pg_size_pretty(1536 * 4) AS vector_bytes_per_row,
    pg_size_pretty(1536 * 2) AS halfvec_bytes_per_row,
    pg_size_pretty(1536 * 4 * 1000000) AS vector_1m_rows_raw,
    pg_size_pretty(1536 * 2 * 1000000) AS halfvec_1m_rows_raw;
--  vector_bytes_per_row | halfvec_bytes_per_row | vector_1m_rows_raw | halfvec_1m_rows_raw
-- ----------------------+-----------------------+--------------------+--------------------
--  6144 bytes           | 3072 bytes            | 5859 MB            | 2930 MB

Distance Operators and Query Patterns

pgvector exposes similarity search through three distance operators. Choosing the right one is not a matter of preference — it depends on how the embedding model was trained and what the model's documentation specifies.

| Operator | Name | Use case |

|---|---|---|

| <-> | L2 (Euclidean) distance | Default; models trained with L2 loss |

| <=> | Cosine distance | Most sentence transformers; text embedding models |

| <#> | Negative inner product | Models normalized to unit sphere; dot product ranking |

All three operators return a *distance* (lower is more similar for <-> and <=>; more negative is more similar for <#>). They do not return a similarity score directly — convert if needed: 1 - (embedding <=> query_vec) gives cosine similarity in [−1, 1].

-- PostgreSQL / pgvector
-- Basic nearest-neighbor query using cosine distance
-- The query embedding would typically come from application code
SELECT
    id,
    content,
    metadata,
    1 - (embedding <=> '[0.021, -0.014, 0.033, ...]'::vector) AS cosine_similarity
FROM documents
ORDER BY embedding <=> '[0.021, -0.014, 0.033, ...]'::vector
LIMIT 10;

-- Pre-filtering with metadata before vector search
-- This pushes a WHERE clause before the ORDER BY/LIMIT
SELECT
    id,
    content,
    1 - (embedding <=> $1::vector) AS cosine_similarity
FROM documents
WHERE metadata->>'category' = 'database'
  AND created_at > NOW() - INTERVAL '90 days'
ORDER BY embedding <=> $1::vector
LIMIT 10;

-- Finding documents above a similarity threshold
-- Note: cosine distance threshold of 0.3 = cosine similarity >= 0.7
SELECT id, content, embedding <=> $1::vector AS distance
FROM documents
WHERE embedding <=> $1::vector < 0.3
ORDER BY embedding <=> $1::vector
LIMIT 50;

Without an index, these queries perform exact nearest-neighbor search via sequential scan — every row is compared to the query vector. For tables under approximately 100,000 rows, sequential scan is often fast enough and always produces exact results. Above that threshold, indexing becomes necessary to maintain acceptable query latency.

Building and Tuning Indexes

pgvector supports two index types: IVFFlat and HNSW. They represent different tradeoffs between build time, memory, query speed, and recall accuracy. Understanding this tradeoff in database terms: IVFFlat is roughly analogous to a clustered index that buckets rows into partitions, while HNSW builds a multi-layer navigable graph structure more comparable to a B-tree in that it maintains structure on insert rather than requiring a full rebuild.

IVFFlat:

-- PostgreSQL / pgvector
-- IVFFlat index — requires data to exist before building
-- Rule of thumb: lists = rows / 1000, capped at sqrt(rows)
-- For 1M rows: lists between 1000 and 1000 (sqrt(1M) = 1000)

CREATE INDEX ON documents
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);

-- For a 1M-row table:
CREATE INDEX documents_embedding_ivfflat_idx ON documents
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 1000);

-- At query time, probes controls the accuracy/speed tradeoff
-- More probes = higher recall, slower query
-- Default is 1 (fast, lower recall); production typically 10–50
SET ivfflat.probes = 20;

SELECT id, content
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;

IVFFlat requires the table to be populated before index creation — you cannot build the index on an empty table and expect good performance because the k-means clustering that determines list centroids needs representative data. This is a meaningful operational constraint: you must load your data before indexing, not the other way around.

HNSW:

-- PostgreSQL / pgvector
-- HNSW index — can be built incrementally, supports INSERT without rebuild
-- m: number of connections per layer (default 16, range 2–100)
-- ef_construction: size of dynamic candidate list during build (default 64)
-- Higher values = better recall, longer build time, more memory

CREATE INDEX documents_embedding_hnsw_idx ON documents
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- For higher recall requirements:
CREATE INDEX documents_embedding_hnsw_idx ON documents
USING hnsw (embedding vector_cosine_ops)
WITH (m = 32, ef_construction = 128);

-- At query time, ef controls search quality
-- Default 40; increase for higher recall at cost of latency
SET hnsw.ef_search = 100;

SELECT id, content, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;

-- Check index size and build progress
SELECT
    indexname,
    pg_size_pretty(pg_relation_size(indexname::regclass)) AS index_size
FROM pg_indexes
WHERE tablename = 'documents';

Choosing between IVFFlat and HNSW: For most production workloads starting fresh, HNSW is the better default. It supports incremental inserts without performance degradation, delivers higher recall at equivalent query times, and does not require probe tuning at query time. IVFFlat is appropriate when build time and memory during index construction are the binding constraints — HNSW index builds consume significantly more memory (roughly 10–20 bytes per dimension per row during construction).

Recall benchmarking before going to production:

# Python
# Measure recall@10 for your specific dataset and index configuration
# Compare ANN (approximate) results against exact sequential scan results

import psycopg2
import numpy as np
from typing import List, Tuple

def measure_recall(
    conn_string: str,
    query_vectors: List[np.ndarray],
    k: int = 10
) -> float:
    """
    Compare ANN index results against exact results to measure recall.
    recall@k = |ANN results ∩ exact results| / k
    """
    conn = psycopg2.connect(conn_string)
    
    recalls = []
    
    for qvec in query_vectors:
        vec_literal = "[" + ",".join(str(x) for x in qvec.tolist()) + "]"
        
        with conn.cursor() as cur:
            # Exact results via sequential scan
            cur.execute("SET enable_indexscan = OFF; SET enable_bitmapscan = OFF;")
            cur.execute(
                f"""
                SELECT id FROM documents
                ORDER BY embedding <=> %s::vector
                LIMIT %s
                """,
                (vec_literal, k)
            )
            exact_ids = {row[0] for row in cur.fetchall()}
            
            # ANN results via index
            cur.execute("SET enable_indexscan = ON; SET enable_bitmapscan = ON;")
            cur.execute("SET hnsw.ef_search = 100;")
            cur.execute(
                f"""
                SELECT id FROM documents
                ORDER BY embedding <=> %s::vector
                LIMIT %s
                """,
                (vec_literal, k)
            )
            ann_ids = {row[0] for row in cur.fetchall()}
        
        recall = len(exact_ids & ann_ids) / k
        recalls.append(recall)
    
    conn.close()
    avg_recall = np.mean(recalls)
    print(f"Recall@{k} over {len(query_vectors)} queries: {avg_recall:.4f}")
    return avg_recall

# Target: recall@10 >= 0.95 for most RAG applications
# If below target, increase ef_search (HNSW) or probes (IVFFlat)

A recall@10 of 0.95 or above is a reasonable target for RAG applications — this means that for 95% of queries, the 10 documents most relevant to the query are present in the results. Dropping below 0.90 will be noticeable in end-user answer quality.

postgresql.conf Tuning for Vector Workloads

Vector workloads stress PostgreSQL in ways that differ from OLTP workloads. HNSW index builds are memory-intensive. Embedding columns are wide, making row sizes larger than typical. Query plans involving vector ORDER BY/LIMIT patterns behave differently from standard B-tree index scans. Several configuration parameters need attention.

# shell
# /etc/postgresql/16/main/postgresql.conf
# Entries relevant to vector workloads — adjust to your hardware

# Memory
shared_buffers = 8GB                    # 25% of RAM; vector index pages benefit from buffer cache
work_mem = 256MB                        # HNSW build and sort operations use work_mem per operation
maintenance_work_mem = 4GB             # Controls memory available during CREATE INDEX
effective_cache_size = 24GB            # Optimizer hint for available OS + buffer cache

# Parallelism for index builds (pgvector 0.6.0+)
max_parallel_maintenance_workers = 4   # Parallel HNSW/IVFFlat index builds
max_parallel_workers = 8
max_parallel_workers_per_gather = 4

# WAL and checkpoints — vector inserts during embedding pipelines can be write-heavy
wal_buffers = 64MB
checkpoint_completion_target = 0.9
max_wal_size = 4GB

# Connection and timeout settings
statement_timeout = 60000              # 60s; vector index builds may need higher per-session override
lock_timeout = 10000

# For high-throughput embedding insert workloads, consider:
synchronous_commit = off               # Improves insert throughput; loses last ~wal_writer_delay ms on crash

Parallel index builds were added in pgvector 0.6.0 and make a substantial difference. Building an HNSW index on 1 million 1536-dimensional vectors drops from approximately 45 minutes single-threaded to around 12 minutes with 4 workers on a machine with 8 cores and 32GB RAM. Set max_parallel_maintenance_workers in postgresql.conf and optionally override per session:

-- PostgreSQL / pgvector
-- Override for a specific index build session
SET max_parallel_maintenance_workers = 7;

CREATE INDEX documents_embedding_hnsw_idx ON documents
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- Monitor build progress (PostgreSQL 12+)
SELECT
    phase,
    blocks_done,
    blocks_total,
    tuples_done,
    tuples_total,
    ROUND(100.0 * tuples_done / NULLIF(tuples_total, 0), 1) AS pct_complete
FROM pg_stat_progress_create_index
WHERE relid = 'documents'::regclass;

Connection pooling: PgBouncer in transaction mode works with pgvector queries but there is an important caveat. The SET ivfflat.probes and SET hnsw.ef_search commands set session-level parameters. In transaction pooling mode, these settings do not persist across transactions — they are lost when the connection is returned to the pool. The correct pattern is to include the SET in every transaction or query batch, or to use SET LOCAL within a transaction block, or to configure pgbouncer.ini with server_reset_query to reset state.

-- PostgreSQL / pgvector
-- Safe pattern for transaction-mode PgBouncer: SET LOCAL within explicit transaction
BEGIN;
SET LOCAL hnsw.ef_search = 100;
SELECT id, content, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
COMMIT;

Vacuuming and bloat: Embedding tables with high update or delete rates accumulate dead tuples just like any other table, and dead tuples inflate HNSW graph traversal costs. Ensure autovacuum is not suppressed on embedding tables and monitor pg_stat_user_tables.n_dead_tup. For workloads where embeddings are frequently replaced (re-indexing documents with a new embedding model, for example), consider a delete-and-reinsert pattern with periodic VACUUM ANALYZE rather than in-place UPDATE.
