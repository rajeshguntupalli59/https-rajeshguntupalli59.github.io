Chapter 4: The Vector Database Landscape: pgvector, Pinecone, Weaviate, Qdrant, Chroma

The vector database market matured rapidly between 2022 and 2024, producing a crowded field of purpose-built systems, managed cloud services, and extensions bolted onto existing relational databases. For database professionals, navigating this landscape requires the same analytical framework you apply to any storage engine selection: understand the data model, the indexing strategy, the operational model, and the cost curve. This chapter maps the five most production-relevant systems — pgvector, Pinecone, Weaviate, Qdrant, and Chroma — against those dimensions, with real connection code, query patterns, and the benchmarks that separate marketing claims from production reality.

---

The Core Decision Framework: Where Vector Search Lives in Your Stack

Before comparing systems, establish what problem you are actually solving. Vector databases are not a monolithic category — they span a spectrum from pure-vector stores to hybrid search engines to vector-capable relational databases. The position on that spectrum drives every downstream architectural decision.

The Spectrum

At one end sits pgvector: a PostgreSQL extension that adds vector similarity operators and indexes to the relational engine you already run. You get ACID transactions, foreign keys, joins, and pg_dump — at the cost of ANN performance that tops out around 100M vectors before you need to shard aggressively.

At the other end sit purpose-built vector databases like Pinecone: managed services with no DDL, no transactions, and no SQL, designed exclusively to return approximate nearest neighbors at sub-50ms latency on billion-scale corpora.

Between those poles sit systems like Weaviate, Qdrant, and Chroma, each making different tradeoffs between operational simplicity, hybrid search capability, and raw performance.

The Three Selection Questions

Before writing any connection code, answer these three questions:

1. Vector count and growth rate. Under 1M vectors: any system works. 1M–50M: all options remain viable with tuning. 50M–500M: purpose-built systems start pulling ahead. 500M+: Pinecone, Weaviate, or Qdrant with horizontal scaling.

2. Query pattern mix. Pure semantic search: any purpose-built system. Semantic search combined with relational filtering (user_id, tenant_id, date ranges, price bands): pgvector's join capability or Weaviate/Qdrant's metadata filters become significant.

3. Operational footprint. If your organization already runs managed PostgreSQL (RDS, Cloud SQL, Supabase, Neon), pgvector adds a line to your migration script. If you are building a new service with no existing database infrastructure, a managed vector service removes operational overhead that would otherwise require dedicated DBA time.

Latency Benchmarks as a Reality Check

Published benchmarks from the ann-benchmarks project and Qdrant's own studies as of early 2024 give rough reference points at the glot-wiki-multilingual-1M dataset (960 dimensions, 1M vectors):

| System | QPS (10 threads) | Recall@10 | Index Build Time |

|---|---|---|---|

| Qdrant (HNSW) | ~8,500 | 0.98 | ~4 min |

| Weaviate (HNSW) | ~5,200 | 0.97 | ~6 min |

| pgvector (ivfflat) | ~1,800 | 0.95 | ~8 min |

| pgvector (hnsw) | ~3,100 | 0.97 | ~12 min |

| Chroma (HNSW) | ~2,400 | 0.96 | ~5 min |

These numbers are illustrative under a specific workload; your production configuration, dimensionality, and hardware will shift them. The pattern they reveal — purpose-built systems outperform the extension by 2–5x at raw throughput — holds across most published benchmarks.

pgvector: Vector Search Inside PostgreSQL

pgvector is the most operationally conservative choice and, for many teams, the right one. It extends PostgreSQL with a vector data type, two index types (ivfflat and HNSW), and three distance operators (L2, cosine, inner product). The extension first appeared in 2021 and reached production-ready HNSW support in version 0.5.0 (late 2023).

Installation and Schema Setup

# shell
# Install pgvector on Ubuntu/Debian
sudo apt install postgresql-16-pgvector

# Or build from source
git clone https://github.com/pgvector/pgvector.git
cd pgvector
make
sudo make install
-- PostgreSQL / pgvector
-- Enable the extension
CREATE EXTENSION IF NOT EXISTS vector;

-- Documents table with embeddings (1536 dims = OpenAI text-embedding-ada-002)
CREATE TABLE documents (
    id          BIGSERIAL PRIMARY KEY,
    tenant_id   UUID NOT NULL,
    content     TEXT NOT NULL,
    metadata    JSONB,
    embedding   VECTOR(1536),
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- HNSW index — better recall and query latency than ivfflat
-- m: connections per layer (16 is default, 32 improves recall at memory cost)
-- ef_construction: search width during build (64 default, 128 for higher recall)
CREATE INDEX ON documents USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- Partial index on a specific tenant — useful for multi-tenant SaaS
CREATE INDEX ON documents USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64)
    WHERE tenant_id = '3f2a1b00-0000-0000-0000-000000000001';

Querying with Metadata Filters

pgvector's join capability is where it earns its place for relational workloads. You can combine ANN search with arbitrary SQL predicates in a single query planner pass.

-- PostgreSQL / pgvector
-- Set HNSW search parameter at session level
-- ef_search > ef_construction improves recall at query time
SET hnsw.ef_search = 100;

-- Semantic search scoped to a tenant, filtered by metadata, ranked by recency
-- The query embedding comes from your application layer as a literal
SELECT
    d.id,
    d.content,
    d.metadata->>'source_url'       AS source_url,
    d.metadata->>'doc_type'         AS doc_type,
    1 - (d.embedding <=> $1::vector) AS cosine_similarity,
    d.created_at
FROM documents d
WHERE
    d.tenant_id   = $2
    AND d.metadata->>'doc_type' IN ('manual', 'faq')
    AND d.created_at >= NOW() - INTERVAL '90 days'
ORDER BY d.embedding <=> $1::vector   -- cosine distance, ascending
LIMIT 10;

-- Operators reference:
-- <=>  cosine distance
-- <->  L2 (Euclidean) distance
-- <#>  negative inner product (use for maximum inner product search)

Python Integration with pgvector

# Python
import asyncpg
import numpy as np
from openai import OpenAI
from pgvector.asyncpg import register_vector

openai_client = OpenAI()

async def embed_text(text: str) -> list[float]:
    response = openai_client.embeddings.create(
        model="text-embedding-3-small",   # 1536 dims, cheaper than ada-002
        input=text
    )
    return response.data[0].embedding

async def upsert_document(pool: asyncpg.Pool, doc_id: int, tenant_id: str,
                          content: str, metadata: dict) -> None:
    embedding = await embed_text(content)
    async with pool.acquire() as conn:
        await register_vector(conn)
        await conn.execute(
            """
            INSERT INTO documents (id, tenant_id, content, metadata, embedding)
            VALUES ($1, $2, $3, $4, $5)
            ON CONFLICT (id) DO UPDATE
                SET content   = EXCLUDED.content,
                    metadata  = EXCLUDED.metadata,
                    embedding = EXCLUDED.embedding
            """,
            doc_id, tenant_id, content, metadata, embedding
        )

async def semantic_search(pool: asyncpg.Pool, query: str,
                          tenant_id: str, k: int = 10) -> list[dict]:
    query_embedding = await embed_text(query)
    async with pool.acquire() as conn:
        await register_vector(conn)
        # ef_search tunable per-connection
        await conn.execute("SET hnsw.ef_search = 100")
        rows = await conn.fetch(
            """
            SELECT id, content, metadata,
                   1 - (embedding <=> $1::vector) AS similarity
            FROM documents
            WHERE tenant_id = $2
            ORDER BY embedding <=> $1::vector
            LIMIT $3
            """,
            query_embedding, tenant_id, k
        )
    return [dict(r) for r in rows]

When pgvector is the Right Call

pgvector wins when: your vector count stays under 10M, you need transactional consistency between document metadata and embeddings, your team operates PostgreSQL already, or you need complex JOIN queries that would require application-side orchestration in a purpose-built system. It loses when: you need sub-20ms P99 latency at 50M+ vectors, your indexing throughput requirements exceed what a single PostgreSQL primary can sustain, or you need native sparse+dense hybrid scoring.

Pinecone: Managed Vector Search as a Service

Pinecone occupies the opposite corner of the design space. You get no SQL, no ACID, and no self-hosting option — but you also get no infrastructure management, automatic horizontal scaling, and consistently low latency at billion-scale. The operational model resembles an S3 bucket: provision an index, write vectors through an API, query through the same API.

Pinecone organizes data into indexes (equivalent to a table) containing namespaces (logical partitions, roughly equivalent to schema-level separation). Each vector record carries an ID, a dense embedding, an optional sparse vector for hybrid search, and a metadata dict of filterable key-value pairs.

Pinecone Index Setup and Upsert

# Python
from pinecone import Pinecone, ServerlessSpec
from openai import OpenAI
import time

pc = Pinecone(api_key="YOUR_PINECONE_API_KEY")
openai_client = OpenAI(api_key="YOUR_OPENAI_API_KEY")

INDEX_NAME = "product-catalog"
DIMENSION  = 1536   # text-embedding-3-small output dimension

# Create a serverless index (AWS us-east-1)
# Serverless charges per query/write RU, not per pod-hour
if INDEX_NAME not in [i.name for i in pc.list_indexes()]:
    pc.create_index(
        name      = INDEX_NAME,
        dimension = DIMENSION,
        metric    = "cosine",
        spec      = ServerlessSpec(cloud="aws", region="us-east-1")
    )
    # Wait for index to be ready
    while not pc.describe_index(INDEX_NAME).status["ready"]:
        time.sleep(1)

index = pc.Index(INDEX_NAME)

def embed_batch(texts: list[str]) -> list[list[float]]:
    """Embed a batch of texts. OpenAI allows up to 2048 inputs per call."""
    response = openai_client.embeddings.create(
        model="text-embedding-3-small",
        input=texts
    )
    return [item.embedding for item in response.data]

def upsert_products(products: list[dict], namespace: str = "default") -> None:
    """
    products: list of dicts with keys: id, name, description, category, price
    Pinecone upsert is idempotent — safe to re-run on updates.
    """
    BATCH_SIZE = 100   # Pinecone recommends 100–200 vectors per upsert call
    for i in range(0, len(products), BATCH_SIZE):
        batch = products[i : i + BATCH_SIZE]
        texts = [f"{p['name']} {p['description']}" for p in batch]
        embeddings = embed_batch(texts)
        vectors = [
            {
                "id":       str(p["id"]),
                "values":   emb,
                "metadata": {
                    "name":     p["name"],
                    "category": p["category"],
                    "price":    float(p["price"])   # metadata values: str, int, float, bool, list
                }
            }
            for p, emb in zip(batch, embeddings)
        ]
        index.upsert(vectors=vectors, namespace=namespace)
        print(f"Upserted batch {i // BATCH_SIZE + 1}, {len(vectors)} vectors")

def query_products(query_text: str, category: str = None,
                   max_price: float = None, top_k: int = 10,
                   namespace: str = "default") -> list[dict]:
    query_embedding = embed_batch([query_text])[0]

    # Metadata filter uses MongoDB-style operators
    filter_expr = {}
    if category:
        filter_expr["category"] = {"$eq": category}
    if max_price is not None:
        filter_expr["price"] = {"$lte": max_price}

    results = index.query(
        vector           = query_embedding,
        top_k            = top_k,
        filter           = filter_expr if filter_expr else None,
        include_metadata = True,
        namespace        = namespace
    )
    return [
        {
            "id":         m.id,
            "score":      m.score,
            "name":       m.metadata.get("name"),
            "category":   m.metadata.get("category"),
            "price":      m.metadata.get("price")
        }
        for m in results.matches
    ]

Pinecone's Operational Model

The serverless tier (launched 2024) removes the pod-sizing decision that made early Pinecone capacity planning painful. In the pod-based model, undersizing a p1.x1 pod meant reindexing into a larger pod with downtime. Serverless autoscales within the cloud region, billed per read unit (RU) and write unit (WU). A single query costs roughly 1 RU per 1K dimensions per result returned — budget accordingly for high-throughput applications.

Pinecone does not support cross-record joins or aggregation. If your application needs "find similar products, then look up inventory levels," the inventory lookup happens in a separate database call. This is not a limitation unique to Pinecone — it is the general tradeoff of purpose-built vector systems, and it drives many teams back toward pgvector when their query patterns are primarily relational.

Weaviate: Hybrid Search and the Knowledge Graph Model

Weaviate presents a different abstraction than either Pinecone or pgvector. It organizes data as collections of objects, where each object carries properties (typed fields), an optional vector, and cross-references to other objects. The cross-reference mechanism gives Weaviate a graph traversal capability that neither Pinecone nor pgvector offers natively.

Weaviate's distinguishing production feature is its native hybrid search: a single query path that combines dense vector similarity (via HNSW) with sparse BM25 keyword scoring, fused using Reciprocal Rank Fusion (RRF) or a weighted linear combination. For document retrieval workloads where users submit keyword queries alongside semantic intent, hybrid mode consistently outperforms pure vector search in NDCG metrics.

Weaviate Connection and Schema

# Python
import weaviate
import weaviate.classes as wvc
from weaviate.classes.config import Configure, Property, DataType, VectorDistances
from openai import OpenAI

# Weaviate Cloud (WCD) connection
client = weaviate.connect_to_weaviate_cloud(
    cluster_url = "https://your-cluster.weaviate.network",
    auth_credentials = weaviate.auth.AuthApiKey("YOUR_WEAVIATE_API_KEY"),
    headers = {"X-OpenAI-Api-Key": "YOUR_OPENAI_API_KEY"}  # for module vectorization
)

COLLECTION_NAME = "TechArticle"

# Create collection — Weaviate handles vectorization via modules
# text2vec-openai calls the OpenAI API at insert time automatically
if not client.collections.exists(COLLECTION_NAME):
    client.collections.create(
        name = COLLECTION_NAME,
        vectorizer_config = Configure.Vectorizer.text2vec_openai(
            model = "text-embedding-3-small"
        ),
        vector_index_config = Configure.VectorIndex.hnsw(
            distance_metric    = VectorDistances.COSINE,
            ef_construction    = 128,
            max_connections    = 16,
            ef                 = 64    # query-time search width
        ),
        properties = [
            Property(name="title",       data_type=DataType.TEXT),
            Property(name="body",        data_type=DataType.TEXT),
            Property(name="author",      data_type=DataType.TEXT),
            Property(name="category",    data_type=DataType.TEXT),
            Property(name="published_at",data_type=DataType.DATE),
            Property(name="view_count",  data_type=DataType.INT),
        ]
    )
    print(f"Collection '{COLLECTION_NAME}' created.")

collection = client.collections.get(COLLECTION_NAME)

# Batch insert — Weaviate auto-vectorizes via the configured module
with collection.batch.dynamic() as batch:
    for article in articles:   # articles: list[dict]
        batch.add_object(properties={
            "title":        article["title"],
            "body":         article["body"],
            "author":       article["author"],
            "category":     article["category"],
            "published_at": article["published_at"],   # ISO 8601 string
            "view_count":   article["view_count"]
        })

# Hybrid search: combines BM25 + vector, fused via RRF
results = collection.query.hybrid(
    query  = "optimizing PostgreSQL index performance",
    alpha  = 0.75,     # 0.0 = pure BM25, 1.0 = pure vector, 0.75 = vector-weighted
    limit  = 10,
    filters = wvc.query.Filter.by_property("category").equal("database") &
              wvc.query.Filter.by_property("view_count").greater_than(100),
    return_properties = ["title", "author", "category", "published_at"],
    return_metadata   = wvc.query.MetadataQuery(score=True, explain_score=True)
)

for obj in results.objects:
    print(f"Score: {obj.metadata.score:.4f} | Title: {obj.properties['title']}")
    # explain_score shows BM25 and vector sub-scores for debugging recall issues

client.close()

Weaviate's Generative Module

Weaviate ships a generate module that pipes retrieved objects directly into a configured LLM. This is RAG-in-one-call: vector retrieval and LLM generation in a single API round-trip.

# Python
# Generative search — retrieve + synthesize in one Weaviate call
response = collection.generate.hybrid(
    query            = "What are the best practices for PostgreSQL vacuuming?",
    alpha            = 0.7,
    limit            = 5,
    grouped_task     = "Summarize the key PostgreSQL vacuuming best practices "
                       "from these articles in three concise bullet points.",
    grouped_properties = ["title", "body"]
)

print(response.generated)   # LLM-synthesized answer from retrieved context

This pattern works well for internal tooling and prototypes but has important production caveats: you are coupling your retrieval latency to your LLM call latency, and a slow generation response blocks the entire query. In high-throughput RAG pipelines, the retrieval and generation steps are better decoupled into separate service boundaries.

Qdrant and Chroma: Performance-First and Developer-First

Qdrant

Qdrant (developed in Rust) is the performance-oriented choice among the open-source purpose-built systems. Its HNSW implementation with quantization support (scalar, product, and binary quantization) enables significantly higher vector density per node compared to Weaviate's pure float32 storage. The payload filter engine evaluates metadata conditions during the HNSW graph traversal rather than as a post-filter, which prevents the precision degradation that affects post-filter approaches at high selectivity ratios.
