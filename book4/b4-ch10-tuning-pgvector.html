Chapter 10: Tuning pgvector for High-Throughput Workloads

pgvector ships with sensible defaults, but production workloads expose its limits quickly. A single-table similarity search that performs well at 100K vectors starts to degrade noticeably past 1M, and write-heavy embedding ingestion pipelines can turn a well-tuned PostgreSQL instance into a bottleneck that blocks your entire AI application. This chapter covers the mechanics of pgvector's indexing algorithms, how to map familiar PostgreSQL tuning levers to vector-specific behavior, and the patterns that let you sustain thousands of queries per second without sacrificing recall accuracy.

---

Understanding How HNSW and IVFFlat Trade Recall for Speed

Before reaching for configuration knobs, you need a clear mental model of what the two index types are actually doing—because the tradeoffs are fundamentally different from B-tree tuning.

IVFFlat (Inverted File with Flat lists) works like a k-means partition. During CREATE INDEX, it assigns every vector to one of lists centroids. At query time, it probes probes lists, computes exact distances within those lists, and returns the top-k results. Think of it as a clustered index where the "key" is a cluster center, and you're controlling how many clusters you scan. More lists = better recall but slower build; more probes = better recall but slower query.

HNSW (Hierarchical Navigable Small World) builds a multi-layer graph where each vector is a node connected to its nearest neighbors. Queries traverse from a sparse top layer down to a dense bottom layer, pruning the search space at each hop. The key build parameters are m (edges per node, default 16) and ef_construction (beam width during build, default 64). The query parameter hnsw.ef_search controls how many candidates are evaluated during traversal.

The practical difference for a DBA: IVFFlat has a faster, more predictable build time and lower memory footprint, but recall degrades sharply if your data distribution shifts after the index is built. HNSW maintains higher recall across distribution changes, but build time scales superlinearly with m and ef_construction, and the index itself can consume 2–3× the memory of an equivalent IVFFlat index.

Here's a benchmark-driven comparison you can run yourself:

-- PostgreSQL / pgvector
-- Create a test table with 1M 1536-dim vectors (OpenAI ada-002 shape)
CREATE TABLE embeddings (
    id        BIGSERIAL PRIMARY KEY,
    content   TEXT,
    embedding VECTOR(1536)
);

-- Populate with random data for benchmarking
INSERT INTO embeddings (content, embedding)
SELECT
    'doc_' || g,
    (
        SELECT array_agg(random()::float4 - 0.5)::vector(1536)
        FROM generate_series(1, 1536)
    )
FROM generate_series(1, 1000000) g;

-- Baseline: sequential scan recall is 100% by definition
-- Time a brute-force query first
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, embedding <=> '[0.1, 0.2, ...]'::vector AS distance
FROM embeddings
ORDER BY distance
LIMIT 10;

-- Build IVFFlat: rule of thumb is sqrt(row_count) to 4*sqrt(row_count) for lists
-- For 1M rows: lists between 1000 and 4000
CREATE INDEX idx_ivfflat ON embeddings
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 2000);

-- Build HNSW with production-grade parameters
CREATE INDEX idx_hnsw ON embeddings
USING hnsw (embedding vector_cosine_ops)
WITH (m = 24, ef_construction = 128);

-- Set probes for IVFFlat at query time (session or per-transaction)
SET ivfflat.probes = 40;

-- Set ef_search for HNSW at query time
SET hnsw.ef_search = 100;

-- Measure approximate query with HNSW
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, embedding <=> '[0.1, 0.2, ...]'::vector AS distance
FROM embeddings
ORDER BY distance
LIMIT 10;

A working rule: for IVFFlat, probes / lists gives you a rough recall proxy. At probes = 1, recall is roughly 60–70% for typical text embeddings. At probes = 40 out of lists = 2000 (2%), recall is typically 95–97%. At probes = 200 (10%), recall approaches 99%. For HNSW, ef_search = 40 usually delivers ~95% recall; ef_search = 100 delivers ~98–99%. These numbers vary significantly with embedding model and data distribution—always measure recall on your actual data.

PostgreSQL Server Configuration for Vector Workloads

Vector workloads stress PostgreSQL differently than OLTP. The dominant costs are memory bandwidth, CPU cache pressure from high-dimensional distance calculations, and index build I/O. The standard postgresql.conf knobs matter, but the relevant thresholds shift.

Shared Buffers and Effective Cache Size

HNSW index traversal is heavily random-access across the graph structure. If the index fits in shared_buffers, traversal costs drop dramatically. For a table with 1M 1536-dim float32 vectors, the raw vector data is ~6GB. The HNSW index with m=16 adds roughly 1–1.5GB on top of that. Size your shared_buffers to hold the hot subset of both.

# shell — postgresql.conf tuning for a vector workload host
# Assume: 64GB RAM, dedicated PostgreSQL instance

# Memory
shared_buffers = 16GB              # ~25% of RAM; cover hot HNSW index + working set
effective_cache_size = 48GB        # planner hint: what's available OS + PG cache
work_mem = 256MB                   # each sort/hash node; vector ORDER BY sorts are large
maintenance_work_mem = 4GB         # critical for HNSW build performance

# Parallelism
max_parallel_workers_per_gather = 4
max_parallel_workers = 16
max_worker_processes = 24

# WAL and I/O (for write-heavy ingestion)
wal_buffers = 64MB
checkpoint_completion_target = 0.9
max_wal_size = 4GB
min_wal_size = 1GB

# For NVMe storage, reduce random_page_cost
random_page_cost = 1.1
effective_io_concurrency = 200

# Autovacuum tuning for high-write vector tables
autovacuum_vacuum_cost_delay = 2ms
autovacuum_vacuum_scale_factor = 0.01   # vacuum more aggressively on large tables
autovacuum_analyze_scale_factor = 0.005

Parallel Index Builds

pgvector 0.6.0 added parallel HNSW index builds. This is one of the highest-impact settings for tables with millions of vectors—a serial build of a 5M-vector HNSW index at m=24, ef_construction=128 can take 2–4 hours. With 8 parallel workers it typically drops to 25–40 minutes.

-- PostgreSQL / pgvector
-- Enable parallel workers for the build session
SET max_parallel_maintenance_workers = 7;  -- uses up to 7 workers + 1 leader = 8 total

-- Then build the index
CREATE INDEX idx_hnsw_parallel ON embeddings
USING hnsw (embedding vector_cosine_ops)
WITH (m = 24, ef_construction = 128);

-- Monitor build progress in a separate session
SELECT
    phase,
    blocks_done,
    blocks_total,
    tuples_done,
    tuples_total,
    round(100.0 * tuples_done / NULLIF(tuples_total, 0), 1) AS pct_complete
FROM pg_stat_progress_create_index
WHERE relid = 'embeddings'::regclass;

work_mem and Vector Sorts

When PostgreSQL executes ORDER BY embedding <=> $1 LIMIT k without an index (or when the planner falls back to seqscan), it materializes candidates in a sort node. With 1M rows at 1536 dims (float32), that's a 6GB sort. If work_mem is too low, it spills to disk and query latency spikes into seconds. Set work_mem high enough to keep that sort in memory, or ensure your index is always being used.

You can verify index usage:

-- PostgreSQL / pgvector
-- Confirm the planner is using the HNSW index, not a seqscan
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT id, content, embedding <=> '[0.1, 0.2, ...]'::vector AS score
FROM embeddings
ORDER BY score
LIMIT 20;

-- If you see "Seq Scan" instead of "Index Scan using idx_hnsw_parallel",
-- the planner may be choosing seqscan because the table is small enough
-- or because enable_indexscan is off. Force it during testing:
SET enable_seqscan = OFF;

-- Also check: is your query vector the right dimension?
-- Dimension mismatch silently falls back to seqscan in older pgvector versions
SELECT typmod FROM pg_attribute
WHERE attrelid = 'embeddings'::regclass
AND attname = 'embedding';

Scaling Ingestion: Bulk Loading Without Choking the Index

Most production AI applications have two phases: an initial bulk load of millions of historical embeddings, and an ongoing stream of new embeddings from user activity or document indexing. Both require different strategies.

Bulk Loading Strategy

The worst approach is inserting rows one at a time while the HNSW index is live. Each insert triggers an index entry insertion, which requires graph traversal to find the insertion point—O(log n) per insert, with high cache miss rates on a large graph. At 1M rows, individual inserts are ~10–15ms each through pgvector's HNSW; that's roughly 70–100 inserts/second.

The right approach: load data without the vector index, then build the index in one pass.

# Python — bulk loading embeddings using COPY for maximum throughput

import psycopg2
import psycopg2.extras
import numpy as np
from io import StringIO

def bulk_load_embeddings(conn_string: str, records: list[dict]) -> None:
    """
    Load embeddings using COPY for 10–50× throughput vs INSERT.
    records: list of {"id": int, "content": str, "embedding": np.ndarray}
    """
    conn = psycopg2.connect(conn_string)
    conn.autocommit = False
    cur = conn.cursor()

    # Step 1: Drop the vector index if it exists
    cur.execute("""
        DROP INDEX IF EXISTS idx_hnsw_parallel;
    """)

    # Step 2: Disable autovacuum on this table during bulk load
    cur.execute("""
        ALTER TABLE embeddings SET (autovacuum_enabled = false);
    """)

    # Step 3: Use COPY with a StringIO buffer
    # pgvector accepts vectors in '[f1,f2,...,fn]' format
    buffer = StringIO()
    for rec in records:
        vec_str = '[' + ','.join(f'{v:.6f}' for v in rec['embedding']) + ']'
        # Escape tab and newline in content
        content = rec['content'].replace('\t', ' ').replace('\n', ' ')
        buffer.write(f"{rec['id']}\t{content}\t{vec_str}\n")

    buffer.seek(0)
    cur.copy_from(buffer, 'embeddings', columns=('id', 'content', 'embedding'))

    conn.commit()

    # Step 4: Rebuild the index after load
    cur.execute("SET maintenance_work_mem = '4GB'")
    cur.execute("SET max_parallel_maintenance_workers = 7")
    cur.execute("""
        CREATE INDEX idx_hnsw_parallel ON embeddings
        USING hnsw (embedding vector_cosine_ops)
        WITH (m = 24, ef_construction = 128);
    """)

    # Step 5: Re-enable autovacuum and run analyze
    cur.execute("""
        ALTER TABLE embeddings SET (autovacuum_enabled = true);
    """)
    cur.execute("ANALYZE embeddings;")

    conn.commit()
    cur.close()
    conn.close()
    print(f"Loaded {len(records)} records and rebuilt index.")


# For ongoing streaming inserts, use batched inserts with execute_values
def stream_insert_embeddings(conn_string: str, batch: list[dict]) -> None:
    """
    For ongoing inserts: batch size of 500–2000 rows hits the
    sweet spot between commit overhead and lock contention.
    """
    conn = psycopg2.connect(conn_string)
    conn.autocommit = False
    cur = conn.cursor()

    records = [
        (rec['id'], rec['content'],
         '[' + ','.join(f'{v:.6f}' for v in rec['embedding']) + ']')
        for rec in batch
    ]

    psycopg2.extras.execute_values(
        cur,
        "INSERT INTO embeddings (id, content, embedding) VALUES %s "
        "ON CONFLICT (id) DO UPDATE SET embedding = EXCLUDED.embedding",
        records,
        template="(%s, %s, %s::vector)",
        page_size=500
    )

    conn.commit()
    cur.close()
    conn.close()

Filling Factor and Index Bloat

HNSW indexes grow as you insert rows, but they don't rebalance—unlike B-trees, there's no analog to a page split. However, deletes and updates leave dead tuples that inflate the index. Run VACUUM ANALYZE regularly on embedding tables, and monitor pg_stat_user_indexes for index size growth anomalies:

-- PostgreSQL / pgvector
-- Monitor index health and bloat indicators
SELECT
    i.relname                        AS index_name,
    pg_size_pretty(pg_relation_size(i.oid)) AS index_size,
    pg_size_pretty(pg_relation_size(t.oid)) AS table_size,
    s.idx_scan,
    s.idx_tup_read,
    s.idx_tup_fetch,
    round(100.0 * pg_relation_size(i.oid) /
          NULLIF(pg_relation_size(t.oid), 0), 1)  AS index_to_table_pct
FROM pg_index x
JOIN pg_class i ON i.oid = x.indexrelid
JOIN pg_class t ON t.oid = x.indrelid
JOIN pg_stat_user_indexes s ON s.indexrelid = i.oid
WHERE t.relname = 'embeddings'
ORDER BY pg_relation_size(i.oid) DESC;

-- If the HNSW index exceeds ~2× the raw vector data size,
-- consider a REINDEX CONCURRENTLY to reclaim space
REINDEX INDEX CONCURRENTLY idx_hnsw_parallel;

Hybrid Search Architecture: Combining Vector and Scalar Filters

Production RAG and semantic search systems rarely do pure vector search. The typical pattern is: filter by tenant, date range, document type, or access control, then rank by similarity. How you structure this filter interaction has a massive impact on both performance and recall.

The Pre-filter vs. Post-filter Problem

PostgreSQL's query planner applies scalar filters either before or after the vector index scan, depending on statistics and cost estimates. If it applies the filter *after* fetching the top-k from the index (post-filter), and the filter eliminates 90% of results, you get fewer than k results. If it applies the filter *before* (pre-filter via seqscan), you get correct results but scan every row.

pgvector doesn't natively support pre-filtered HNSW the way specialized vector databases do. Your architectural options:

-- PostgreSQL / pgvector

-- Option 1: Partial indexes — build one HNSW index per tenant
-- Effective when tenant count is small and per-tenant volume is large
CREATE INDEX idx_hnsw_tenant_42 ON embeddings
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64)
WHERE tenant_id = 42;

-- The planner will use this index when the query includes the filter
SELECT id, content, embedding <=> $1 AS score
FROM embeddings
WHERE tenant_id = 42
ORDER BY score
LIMIT 10;

-- Option 2: Partition the table by tenant, build one index per partition
-- Better for large tenant counts with predictable partitioning keys
CREATE TABLE embeddings_partitioned (
    id        BIGSERIAL,
    tenant_id INT NOT NULL,
    content   TEXT,
    embedding VECTOR(1536),
    created_at TIMESTAMPTZ DEFAULT now()
) PARTITION BY LIST (tenant_id);

CREATE TABLE embeddings_t1 PARTITION OF embeddings_partitioned
    FOR VALUES IN (1);
CREATE TABLE embeddings_t2 PARTITION OF embeddings_partitioned
    FOR VALUES IN (2);

-- Build per-partition HNSW indexes
CREATE INDEX ON embeddings_t1 USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);
CREATE INDEX ON embeddings_t2 USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- Option 3: Two-pass query — oversample from the index, then filter in app
-- Use this when filters are unpredictable or cardinality is unknown
-- Fetch 10× more candidates than needed, filter in application layer
SELECT id, content, metadata, embedding <=> $1 AS score
FROM embeddings
ORDER BY score
LIMIT 200;  -- fetch 200, apply filter in app, return top 20

-- Option 4: Use a composite score with WHERE clause on indexed scalar columns
-- Add a GIN or BRIN index on metadata columns alongside HNSW
CREATE INDEX idx_embeddings_created ON embeddings USING BRIN (created_at);
CREATE INDEX idx_embeddings_tenant  ON embeddings (tenant_id);

-- This query lets the planner choose between bitmap index scan on tenant_id
-- and HNSW scan; with good statistics it usually picks correctly
SELECT id, content, embedding <=> $1 AS score
FROM embeddings
WHERE tenant_id = $2
  AND created_at > now() - INTERVAL '30 days'
ORDER BY score
LIMIT 10;

Quantization and Dimensionality Reduction

For very large tables (10M+ vectors), you can reduce the memory footprint of the vector data itself. pgvector 0.7.0 introduced binary quantization and scalar quantization, which trade a small recall penalty for substantial memory savings:

-- PostgreSQL / pgvector 0.7+
-- Halfvec: store as float16 instead of float32 (2× memory reduction)
-- Useful for OpenAI text-embedding-3-small (1536 dims)
CREATE TABLE embeddings_hv (
    id        BIGSERIAL PRIMARY KEY,
    content   TEXT,
    embedding HALFVEC(1536)  -- 3KB per row instead of 6KB
);

CREATE INDEX idx_hnsw_halfvec ON embeddings_hv
USING hnsw (embedding halfvec_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- Binary quantization: extreme compression, ~5–10% recall loss for most models
-- Good for pre-filtering candidates before a reranking pass
CREATE TABLE embeddings_bq (
    id        BIGSERIAL PRIMARY KEY,
    content   TEXT,
    embedding BIT(1536)  -- 192 bytes per row instead of 6KB
);

CREATE INDEX idx_hnsw_bit ON embeddings_bq
USING hnsw (embedding bit_hamming_ops)
WITH (m = 16, ef_construction = 64);

-- Two-phase search: binary quantization for candidate retrieval,
-- exact float32 distance for reranking
WITH candidates AS (
    SELECT id
    FROM embeddings_bq
    ORDER BY embedding <~> $1::bit(1536)  -- Hamming distance, fast
    LIMIT 100
)
SELECT e.id, e.content, e.embedding <=> $2::vector AS score
FROM embeddings e
JOIN candidates c ON c.id = e.id
ORDER BY score
LIMIT 10;

Connection Pooling and Concurrency Patterns for AI Applications

AI applications have a distinct connection pattern: many short, parallel requests during inference, each needing a fast vector lookup. This matches OLTP patterns superficially but diverges in two ways—queries are CPU-bound (distance calculations) rather than I/O-bound, and session-level GUC settings like hnsw.ef_search complicate connection pooling.

PgBouncer Configuration for Vector Workloads

PgBouncer in transaction-mode pooling resets session state between transactions, which breaks session-level SET hnsw.ef_search. Use statement-mode only if your queries are single-statement, or use transaction-mode with SET LOCAL instead of SET:

# shell — pgbouncer.ini for a vector search service

[databases]
vectordb = host=127.0.0.1 port=5432 dbname=vectordb

[pgbouncer]
pool_mode = transaction          # transaction-mode for AI services
max_client_conn = 1000           # total clients (LLM app + workers)
default_pool_size = 20           # backend connections per db/user pair
reserve_pool_size = 5            # emergency reserve
reserve_pool_timeout = 3
max_db_connections = 50          # cap on total backend connections

# Keep connections alive — vector index traversal benefits from warm caches
server_idle_timeout = 600
client_idle_timeout = 0

# Authentication
auth_type = scram-sha-256
auth_file = /etc/pgbouncer/userlist.txt

# Logging — monitor slow vector queries
log_pooler_errors = 1
stats_period = 60
