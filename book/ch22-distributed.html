Chapter 22: Distributed Systems

Distributed systems represent one of the most profound shifts in how databases are architected, operated, and reasoned about. When a single server is no longer sufficient — whether due to data volume, throughput demands, fault tolerance requirements, or geographic distribution — databases must be spread across multiple nodes, and suddenly the comfortable guarantees DBAs rely on in standalone systems begin to bend. This chapter examines how PostgreSQL and SQL Server approach distributed architectures: the theory that underpins them, the replication and sharding mechanisms that make them possible, the consistency trade-offs that must be navigated, and the operational realities that separate a well-run distributed database from one that causes production incidents at 2 a.m.

---

Why Single-Node Databases Eventually Break Down

Every database starts on a single server. For a long time, that server is adequate — more RAM, faster disks, and a bigger CPU handle growth. This vertical scaling strategy works until it doesn't, and the inflection point arrives in one of a few recognizable ways.

The first is storage capacity. A single server has a ceiling. Even with petabyte-scale storage attached via SAN or NVMe arrays, managing hundreds of terabytes on a single logical instance carries enormous operational risk. A corrupt index, a botched migration, or a long-running transaction can cripple the entire dataset.

The second is read throughput. A single write-primary instance can only handle so many concurrent reads before queries start competing for buffer pool space and CPU. Read replicas alleviate this, but they are still a form of distribution — you have already left the world of a single node.

The third is write throughput. This is where distribution becomes genuinely hard. Reads can be fanned out to replicas with relatively simple routing logic. Writes, however, require coordination. If two nodes can each accept writes, you must decide how they reconcile differences, which one wins conflicts, and what happens when the network between them partitions.

The fourth is availability. A single node is a single point of failure. Distributing a database across nodes and data centers means that the failure of any one node does not take the system offline — but only if the architecture is designed correctly.

Understanding these four pressures shapes every architectural decision in distributed database design. DBAs who appreciate why distribution is necessary are far better equipped to choose the right approach than those who treat it as a purely technical puzzle.

CAP Theorem and the Consistency Spectrum

No conversation about distributed databases is complete without the CAP theorem, and no treatment of CAP is useful without acknowledging its nuance. The theorem, articulated by Eric Brewer in 2000 and formally proved by Gilbert and Lynch in 2002, states that a distributed data system can provide at most two of three properties simultaneously: Consistency, Availability, and Partition tolerance.

Consistency here means that every read receives the most recent write or an error. Availability means that every request receives a response, though not necessarily the most recent data. Partition tolerance means the system continues operating when network communication between nodes is disrupted.

The practical reality is that network partitions are not optional — they happen in any real distributed system. This means every distributed database is choosing between consistency and availability during a partition event. Most production systems actually operate on a spectrum rather than at hard extremes, and the nuanced version of this idea is captured by the PACELC model, which adds latency into the equation: even when there is no partition, distributed systems must trade off latency against consistency.

PostgreSQL's native streaming replication sits toward the CP end when synchronous commit is required: the primary waits for at least one standby to acknowledge a WAL record before confirming the transaction to the client. If the standby is unreachable, writes can be blocked. With asynchronous replication, the system leans toward availability, accepting the possibility that a failover will reveal a small data loss window.

SQL Server's Always On Availability Groups offer the same spectrum. A synchronous-commit replica guarantees no data loss at failover but introduces latency on every write. An asynchronous-commit replica allows faster writes but accepts potential data loss measured in seconds or minutes depending on replication lag.

The correct position on this spectrum is a business decision as much as a technical one. Financial transaction systems typically demand strong consistency. Content delivery systems, analytics pipelines, and social feed applications often tolerate eventual consistency in exchange for lower latency and higher availability.

The term "eventual consistency" deserves precision. It does not mean arbitrary inconsistency. It means that, given no new writes, all replicas will converge to the same state. The time to convergence and the visibility of stale reads during that window are the variables a DBA controls through replication configuration and application design.

Replication Architecture in PostgreSQL and SQL Server

Replication is the foundation of most distributed database deployments. Both PostgreSQL and SQL Server have mature replication subsystems, but they implement them differently, and choosing between replication topologies depends on the workload and failure model.

PostgreSQL Streaming Replication

PostgreSQL's built-in replication is physical streaming replication. The primary server generates Write-Ahead Log (WAL) records for every data change. Standby servers connect to the primary, receive the WAL stream, and replay it against their own copy of the data. Because WAL operates at the block level, standbys receive an exact byte-for-byte replica of the primary's data files.

This approach is reliable and straightforward, but it has constraints. A standby server must be at the same major PostgreSQL version. It inherits everything from the primary — all databases, all tables, all schemas — with no ability to replicate a subset. Read-only queries can be routed to standbys using hot_standby = on, but standbys cannot accept writes.

Replication slots ensure that standbys do not fall so far behind that the primary discards WAL they still need. However, an abandoned slot with a lagging standby can cause WAL to accumulate indefinitely, filling the disk on the primary. This is one of the most common operational mistakes in PostgreSQL replication deployments.

PostgreSQL:

-- Check replication slot lag to detect accumulation risk
SELECT slot_name,
       plugin,
       slot_type,
       active,
       pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) AS lag_bytes,
       restart_lsn,
       confirmed_flush_lsn
FROM pg_replication_slots
ORDER BY pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) DESC;

SQL Server:

-- Check replication distributor agent status and latency
SELECT publisher_db,
       publication,
       subscriber,
       subscriber_db,
       delivery_latency,
       delivered_commands,
       last_updated
FROM distribution.dbo.MSdistribution_status
ORDER BY delivery_latency DESC;

Logical replication, introduced in PostgreSQL 10, provides a different model. Rather than replicating raw WAL blocks, logical replication decodes WAL into row-level change events (INSERT, UPDATE, DELETE) and streams those to subscribers. Subscribers can be different major versions of PostgreSQL, can replicate a single table or a filtered subset of tables, and can apply changes to databases with a different schema structure. This flexibility makes logical replication valuable for zero-downtime major version upgrades, cross-region data synchronization, and feeding data pipelines.

SQL Server Replication and Availability Groups

SQL Server offers several replication types: snapshot, transactional, and merge. Transactional replication is the workhorse for low-latency data propagation. The Log Reader Agent reads committed transactions from the publication database's transaction log and stores them in the distribution database. Distribution Agents then deliver those changes to subscribers. Unlike PostgreSQL streaming replication, transactional replication can be selective — specific tables, columns, or even filtered rows can be included in a publication.

Always On Availability Groups (AG) are the modern high-availability mechanism in SQL Server, functioning more like PostgreSQL streaming replication in that they replicate at the log block level. An AG groups databases together and synchronizes them to secondary replicas. Readable secondaries allow read workloads to be offloaded. Automatic failover can be configured for synchronous replicas within the same data center, while asynchronous replicas in remote data centers serve as disaster recovery targets.

SQL Server:

-- Check Always On AG replica health and synchronization lag
SELECT ag.name AS ag_name,
       ar.replica_server_name,
       ars.role_desc,
       ars.synchronization_health_desc,
       ars.connected_state_desc,
       adb.database_name,
       adbs.synchronization_state_desc,
       adbs.log_send_queue_size / 1024.0 AS log_send_queue_mb,
       adbs.redo_queue_size / 1024.0 AS redo_queue_mb,
       adbs.log_send_rate,
       adbs.redo_rate
FROM sys.availability_groups ag
JOIN sys.availability_replicas ar ON ag.group_id = ar.group_id
JOIN sys.dm_hadr_availability_replica_states ars ON ar.replica_id = ars.replica_id
JOIN sys.availability_databases_cluster adb ON ag.group_id = adb.group_id
JOIN sys.dm_hadr_database_replica_states adbs 
    ON adb.group_id = adbs.group_id 
    AND adb.group_database_id = adbs.group_database_id
    AND ars.replica_id = adbs.replica_id
ORDER BY ag.name, ar.replica_server_name;

PostgreSQL:

-- Check streaming replication standby status from the primary
SELECT client_addr,
       usename,
       application_name,
       state,
       sent_lsn,
       write_lsn,
       flush_lsn,
       replay_lsn,
       pg_size_pretty(pg_wal_lsn_diff(sent_lsn, replay_lsn)) AS total_lag,
       sync_state,
       sync_priority
FROM pg_stat_replication
ORDER BY pg_wal_lsn_diff(sent_lsn, replay_lsn) DESC;

One important operational distinction: PostgreSQL streaming replication is built into the database engine and is free. SQL Server Always On Availability Groups require Enterprise Edition or, in limited form, Standard Edition. This cost consideration influences architecture decisions in cost-sensitive environments and should be factored into total cost of ownership calculations.

Sharding, Partitioning, and Horizontal Scale-Out

Replication solves read scale and high availability. It does not, by itself, solve write scale or storage scale. When the volume of writes or data exceeds what a single primary can handle, horizontal partitioning — sharding — becomes necessary.

Sharding means dividing data across multiple independent nodes, each owning a subset of the total dataset. A user with ID 1–1,000,000 might live on shard 1, users 1,000,001–2,000,000 on shard 2, and so on. Alternatively, sharding can be based on a hash of a key, a range of values, or geographic region. Each strategy has trade-offs.

Range-based sharding is natural and makes range queries efficient. If you query users created between two dates, a range shard key on creation date ensures most such queries hit a single shard. The risk is hotspots: if new records always have a higher key value, all new writes land on the last shard.

Hash-based sharding distributes writes evenly, eliminating hotspots. The trade-off is that range queries must scatter to all shards, which increases latency and coordination overhead.

PostgreSQL and Sharding

PostgreSQL does not have built-in sharding in the same way it has built-in replication. Declarative table partitioning, introduced in PostgreSQL 10 and significantly improved through version 14 and beyond, handles partitioning within a single instance. Postgres Foreign Data Wrappers (FDW) with postgres_fdw can make remote tables appear local, enabling a primitive form of manual sharding, but this lacks sophisticated query planning across shards.

For genuine distributed sharding, the ecosystem relies on third-party solutions: Citus (now part of the Microsoft ecosystem but still open source), which transforms PostgreSQL into a sharded distributed database; Patroni for HA coordination; and PgBouncer for connection pooling across shards.

Citus distributes tables across worker nodes using a distribution column. When a query filters on that column, Citus routes it to the correct shard. When the query touches multiple shards or must join distributed tables, Citus coordinates parallel execution across workers.

PostgreSQL (with declarative partitioning as a foundation for sharding logic):

-- Create a range-partitioned table suitable for per-shard attachment via FDW
CREATE TABLE orders (
    order_id     BIGINT NOT NULL,
    customer_id  BIGINT NOT NULL,
    order_date   DATE NOT NULL,
    total_amount NUMERIC(12, 2),
    status       TEXT
) PARTITION BY RANGE (order_date);

-- Create individual partitions (in a sharded setup, each would live on a different node via FDW)
CREATE TABLE orders_2023 PARTITION OF orders
    FOR VALUES FROM ('2023-01-01') TO ('2024-01-01');

CREATE TABLE orders_2024 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

CREATE TABLE orders_2025 PARTITION OF orders
    FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

-- Verify partition pruning is applied (only relevant partition scanned)
EXPLAIN ANALYZE
SELECT * FROM orders
WHERE order_date BETWEEN '2024-03-01' AND '2024-06-30'
  AND customer_id = 42;

SQL Server and Sharding

SQL Server also lacks native sharding, though its partitioned tables provide intra-instance range partitioning. For true horizontal scale-out, the historical approach involved Elastic Database tools in Azure SQL, which allow a database to be split across multiple Azure SQL databases with a shard map that routes queries to the correct database. On-premises SQL Server sharding typically requires application-level routing or middleware.

SQL Server partitioned tables use a partition function and scheme to place data across filegroups. This enables efficient partition elimination in queries and the ability to archive or purge partitions with low overhead.

SQL Server:

-- Create a partition function and scheme for date-based partitioning
CREATE PARTITION FUNCTION pf_OrderDate (DATE)
AS RANGE RIGHT FOR VALUES (
    '2023-01-01',
    '2024-01-01',
    '2025-01-01',
    '2026-01-01'
);

CREATE PARTITION SCHEME ps_OrderDate
AS PARTITION pf_OrderDate
TO ([PRIMARY], [FG_2023], [FG_2024], [FG_2025], [FG_2026]);

CREATE TABLE dbo.Orders (
    OrderID     BIGINT NOT NULL,
    CustomerID  BIGINT NOT NULL,
    OrderDate   DATE NOT NULL,
    TotalAmount DECIMAL(12, 2),
    Status      NVARCHAR(50)
) ON ps_OrderDate(OrderDate);

-- Check which partition a specific date maps to
SELECT $PARTITION.pf_OrderDate('2024-06-15') AS PartitionNumber;

-- Review row counts per partition
SELECT p.partition_number,
       p.rows AS row_count,
       prv.value AS boundary_value
FROM sys.partitions p
JOIN sys.partition_schemes ps ON ps.data_space_id = p.data_space_id
JOIN sys.objects o ON o.object_id = p.object_id
LEFT JOIN sys.partition_range_values prv 
    ON prv.function_id = ps.function_id 
    AND prv.boundary_id = p.partition_number - 1
WHERE o.name = 'Orders'
  AND p.index_id IN (0, 1)
ORDER BY p.partition_number;

The critical operational lesson with sharding is that cross-shard queries are expensive and must be minimized by design. An application that frequently needs to join data from multiple shards — for example, a report that aggregates orders across all customer regions — will see those queries become the performance bottleneck in any sharded architecture. The data model must be designed with the shard key in mind from the beginning. Retrofitting sharding onto an existing schema is one of the most painful database migrations a team can undertake.

Distributed Transactions, Two-Phase Commit, and Coordination Costs

When data is distributed across nodes, maintaining transactional consistency across those nodes introduces the hardest problem in distributed databases: the distributed transaction. If a business operation must update data on two separate nodes atomically, how do you ensure either both updates succeed or neither does?

The classical answer is the Two-Phase Commit (2PC) protocol. In 2PC, a coordinator node asks all participating nodes to prepare the transaction (Phase 1). Each node locks the affected rows and responds with a "yes" or "no." If all nodes say yes, the coordinator sends a commit message (Phase 2); if any node says no, the coordinator sends a rollback. All participants then execute accordingly.

2PC has well-known problems in the presence of failures. If the coordinator crashes after all participants have prepared but before the commit message is sent, participants are left in a blocking state — they hold locks but cannot release them without instruction from the coordinator. This is called the in-doubt transaction problem. It can cause deadlocks and requires manual DBA intervention to resolve.

PostgreSQL Prepared Transactions

PostgreSQL supports prepared transactions natively via PREPARE TRANSACTION and COMMIT PREPARED / ROLLBACK PREPARED. These are the building blocks for external 2PC coordinators to use.

PostgreSQL:

-- Phase 1: Prepare the transaction with a globally unique identifier
BEGIN;
UPDATE accounts SET balance = balance - 500 WHERE account_id = 101;
PREPARE TRANSACTION 'txn_transfer_20240615_101_202';

-- Phase 2a: Commit if all participants are ready
COMMIT PREPARED 'txn_transfer_20240615_101_202';

-- Phase 2b: Rollback if any participant reported failure
-- ROLLBACK PREPARED 'txn_transfer_20240615_101_202';

-- Monitor in-doubt prepared transactions that need resolution
SELECT gid,
       prepared,
       owner,
       database,
       transaction
FROM pg_prepared_xacts
ORDER BY prepared;

In-doubt transactions visible in pg_prepared_xacts that are hours or days old are a red flag. If the coordinator that created them is permanently gone, the DBA must determine the correct outcome from application logs and manually commit or roll back each one.

SQL Server Distributed Transactions

SQL Server uses the Microsoft Distributed Transaction Coordinator (MSDTC) for distributed transactions. Any operation that spans multiple SQL Server instances, linked servers, or external resource managers (like MSMQ) enlists MSDTC as the coordinator. MSDTC implements 2PC and manages in-doubt transactions in its log.

SQL Server:

-- Check for in-doubt distributed transactions
SELECT transaction_id,
       name,
       transaction_begin_time,
       dtc_state,
       dtc_status,
       dtc_isolation_level
FROM sys.dm_tran_active_transactions
WHERE dtc_state IS NOT NULL
ORDER BY transaction_begin_time;

-- Find all distributed transactions currently active
SELECT session_id,
       transaction_id,
       is_local,
       is_enlisted,
       is_bound,
       open_transaction_count
FROM sys.dm_tran_session_transactions
WHERE is_enlisted = 1;

The performance cost of distributed transactions is significant and often underestimated. 2PC adds at minimum two round trips to every participating node for every transaction. In a system where nodes are geographically separated, this directly translates to latency measured in tens to hundreds of milliseconds per transaction. A system that processes 5,000 write transactions per second on a single node may only sustain 500 per second once distributed 2PC overhead is introduced — without any change in the underlying data volume.

This is why modern distributed systems often avoid distributed transactions entirely. Rather than using 2PC, they use saga patterns, event sourcing, or compensating transactions that undo already-committed work if a later step fails. These approaches accept eventual consistency in exchange for eliminating coordination bottlenecks. DBAs working with distributed architectures must understand both the mechanics of 2PC and the alternatives so they can push back intelligently when application teams propose designs that would require distributed transactions at high throughput.

Operational Realities: Monitoring, Failover, and Split-Brain Prevention

Designing a distributed system on paper is one challenge. Keeping it healthy in production is another entirely. Distributed systems fail in ways that single-node databases do not, and the failure modes are often subtle.

Monitoring Replication Lag

In any replicated system, lag — the delay between a write on the primary and its application on a replica — is the primary operational metric. A replica that has fallen behind by seconds is usually acceptable. A replica that has fallen behind by hours is a disaster waiting for a failover event to expose it. Both PostgreSQL and SQL Server provide views to monitor this, and both should be feeding alerting systems continuously.

The pg_stat_replication view on PostgreSQL (shown earlier) exposes the LSN positions of every connected standby. The difference between the primary's current WAL position and a standby's replay LSN is the lag, expressed in bytes. Dividing this by the replication rate gives an estimate in time, though replication rate is variable.

SQL Server's sys.dm_hadr_database_replica_states exposes log_send_queue_size and redo_queue_size for AG replicas. These are the backlog of redo waiting to be sent and the backlog waiting to be applied, respectively. Both should be close to zero in a healthy deployment.

Automatic Failover and Fencing

When a primary fails, a replica must be promoted. This is straightforward in concept and treacherous in practice. The central danger is split-brain: a situation where the original primary recovers from a transient failure after a replica has been promoted, and now two nodes both believe they are the primary and accept writes. The resulting data divergence is painful to reconcile and may be irrecoverable without data loss.

Preventing split-brain requires fencing — a mechanism that guarantees the old primary cannot accept writes after a new one has been promoted. Common approaches include:

  • → **STONITH (Shoot The Other Node In The Head)**: Physically power off or network-isolate the old primary before promotion proceeds. Used in Linux Pacemaker clusters with PostgreSQL.
  • → **Lease-based fencing**: The primary holds a time-limited lease from a consensus system (like etcd or ZooKeeper). Patroni, the most widely deployed PostgreSQL HA solution, uses this approach. If the primary cannot renew its lease, it fences itself by stopping write acceptance.
  • → **Quorum-based failover**: SQL Server's Windows Server Failover Cluster (WSFC) uses quorum — a majority of cluster nodes must agree before a failover proceeds. If quorum is lost, the cluster stops operations rather than risk split-brain.

    • Patroni's architecture is worth understanding in detail because it represents production-grade PostgreSQL HA. Patroni uses etcd, Consul, or ZooKeeper as a distributed consensus store. The current primary continuously updates a key in that store. If the primary fails to update the key within a configurable TTL, standby nodes race to acquire the leader key. The winner promotes itself to primary. The loser becomes a standby of the new primary. The former primary, if it recovers, finds that it cannot acquire the leader key and fences itself by refusing connections. No human intervention is required for a clean failover, typically completing in 10–30 seconds.

    SQL Server's Always On AG automatic failover uses WSFC quorum. The replica configured for automatic failover monitors the primary via a lease mechanism. If the primary lease expires and quorum exists, failover is triggered. The listener — a virtual network name that applications connect to — is updated to point to the new primary, and application connections are rerouted automatically.

    Detecting and Recovering from Network Partitions

    A network partition is when some nodes can communicate with each other but not with all nodes. This is different from a total failure. In a three-node cluster with nodes A, B, and C, a partition might leave A able to communicate with B but neither A nor B able to reach C. If C was the primary, A and B should promote a new primary between themselves. But if C can still reach the majority of nodes via another path, split-brain is again the risk.

    Quorum resolves this: only the partition with a majority of nodes can elect a new primary. In a three-node cluster, a majority is two nodes. The partition containing two nodes can elect a new primary; the partition with one node cannot. The isolated node fences itself.
