Chapter 3: PostgreSQL Extensions That Enable AI: pgvector, pg_ai, pgml

PostgreSQL's extension architecture is one of its most underappreciated features — the ability to bolt new data types, operators, index strategies, and procedural languages directly into the engine without forking the codebase. For AI application builders, this means you don't need to bolt a separate vector database onto the side of your stack and then spend engineering cycles keeping it synchronized with your relational data. You can store embeddings next to the records they describe, run semantic search inside transactions, and call external ML models directly from SQL. This chapter covers the three extensions that matter most for production AI work: pgvector for embedding storage and similarity search, pg_ai for direct LLM integration from SQL, and pgml (PostgresML) for in-database model training and inference. By the end, you'll have working schema patterns, query strategies, and production-ready configurations for each.

---

3.1 pgvector: Embedding Storage and Similarity Search at the Relational Layer

pgvector is the extension that makes PostgreSQL a serious vector database. It adds a native vector data type, three index types (IVFFlat, HNSW, and flat/exact), and a suite of distance operators that the query planner understands natively. The key insight is that embeddings live *with* their source data, not in a separate system.

#### Installing and Configuring pgvector

# On Ubuntu/Debian — compile from source (recommended for production control)
sudo apt-get install postgresql-server-dev-16 build-essential git
git clone --branch v0.7.0 https://github.com/pgvector/pgvector.git
cd pgvector
make
sudo make install

# On macOS with Homebrew
brew install pgvector

# With Docker — use the official pgvector image
docker run -d \
  --name pgvector-dev \
  -e POSTGRES_PASSWORD=secret \
  -e POSTGRES_DB=aiapp \
  -p 5432:5432 \
  pgvector/pgvector:pg16

# Enable the extension in your database
psql -U postgres -d aiapp -c "CREATE EXTENSION IF NOT EXISTS vector;"

# Verify installation
psql -U postgres -d aiapp -c "SELECT * FROM pg_extension WHERE extname = 'vector';"

#### Schema Design: Embedding Tables That Actually Work

The worst mistake teams make is creating a single generic embeddings table with a foreign key pointing everywhere. That design leads to unbounded scans and makes it impossible to enforce referential integrity meaningfully. Instead, co-locate embeddings with the data they represent, and use separate embedding tables only when you need to store multiple embedding models for the same content.

-- Enable required extensions
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS pg_trgm;  -- for hybrid search

-- Core content table: product knowledge base
CREATE TABLE knowledge_articles (
    id              BIGSERIAL PRIMARY KEY,
    tenant_id       UUID NOT NULL,
    title           TEXT NOT NULL,
    body            TEXT NOT NULL,
    metadata        JSONB NOT NULL DEFAULT '{}',
    source_url      TEXT,
    published_at    TIMESTAMPTZ,
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    is_active       BOOLEAN NOT NULL DEFAULT true
);

-- Embedding table co-located with articles
-- Separate table allows multiple models per article
CREATE TABLE knowledge_article_embeddings (
    id              BIGSERIAL PRIMARY KEY,
    article_id      BIGINT NOT NULL REFERENCES knowledge_articles(id) ON DELETE CASCADE,
    tenant_id       UUID NOT NULL,  -- denormalized for RLS performance
    model_name      TEXT NOT NULL,  -- 'text-embedding-3-small', 'text-embedding-3-large', etc.
    model_version   TEXT NOT NULL DEFAULT 'v1',
    embedding       vector(1536),   -- OpenAI text-embedding-3-small dimension
    chunk_index     INT NOT NULL DEFAULT 0,  -- for chunked documents
    chunk_text      TEXT NOT NULL,           -- store the chunk for retrieval
    token_count     INT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (article_id, model_name, model_version, chunk_index)
);

-- HNSW index: best for high-QPS production workloads
-- Build this after initial bulk load for speed
CREATE INDEX idx_article_embeddings_hnsw
    ON knowledge_article_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- IVFFlat index: better for very large datasets (10M+ vectors)
-- Requires knowing approximate row count at index creation time
-- CREATE INDEX idx_article_embeddings_ivfflat
--     ON knowledge_article_embeddings
--     USING ivflat (embedding vector_cosine_ops)
--     WITH (lists = 100);  -- sqrt(row_count) is a good starting point

-- Supporting indexes for filtered similarity search
CREATE INDEX idx_article_embeddings_tenant
    ON knowledge_article_embeddings (tenant_id, model_name);

CREATE INDEX idx_article_embeddings_article
    ON knowledge_article_embeddings (article_id);

-- Full-text search index for hybrid search
CREATE INDEX idx_articles_fts
    ON knowledge_articles
    USING gin (to_tsvector('english', title || ' ' || body));

-- GIN index for metadata queries
CREATE INDEX idx_articles_metadata
    ON knowledge_articles USING gin (metadata);

-- Row-Level Security for multi-tenant isolation
ALTER TABLE knowledge_articles ENABLE ROW LEVEL SECURITY;
ALTER TABLE knowledge_article_embeddings ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation_articles
    ON knowledge_articles
    USING (tenant_id = current_setting('app.tenant_id')::UUID);

CREATE POLICY tenant_isolation_embeddings
    ON knowledge_article_embeddings
    USING (tenant_id = current_setting('app.tenant_id')::UUID);

#### Similarity Search Patterns

pgvector supports three distance operators: <-> (L2/Euclidean), <#> (inner product — use negative for max similarity), and <=> (cosine distance). For normalized embeddings from OpenAI and Anthropic models, cosine distance is the right default.

-- Pure vector similarity search with tenant isolation
-- This is your baseline RAG retrieval query
SELECT
    kae.chunk_text,
    ka.title,
    ka.metadata,
    ka.source_url,
    1 - (kae.embedding <=> $1::vector) AS cosine_similarity
FROM knowledge_article_embeddings kae
JOIN knowledge_articles ka ON ka.id = kae.article_id
WHERE kae.tenant_id = $2::UUID
  AND kae.model_name = 'text-embedding-3-small'
  AND ka.is_active = true
  AND 1 - (kae.embedding <=> $1::vector) > 0.75  -- similarity threshold
ORDER BY kae.embedding <=> $1::vector
LIMIT 10;

-- Hybrid search: combine vector similarity with full-text ranking
-- This dramatically improves recall for keyword-heavy queries
WITH semantic_results AS (
    SELECT
        kae.article_id,
        kae.chunk_text,
        kae.chunk_index,
        1 - (kae.embedding <=> $1::vector) AS semantic_score
    FROM knowledge_article_embeddings kae
    WHERE kae.tenant_id = $3::UUID
      AND kae.model_name = 'text-embedding-3-small'
    ORDER BY kae.embedding <=> $1::vector
    LIMIT 50
),
fts_results AS (
    SELECT
        ka.id AS article_id,
        ts_rank_cd(
            to_tsvector('english', ka.title || ' ' || ka.body),
            plainto_tsquery('english', $2)
        ) AS fts_score
    FROM knowledge_articles ka
    WHERE ka.tenant_id = $3::UUID
      AND ka.is_active = true
      AND to_tsvector('english', ka.title || ' ' || ka.body)
          @@ plainto_tsquery('english', $2)
)
SELECT
    sr.chunk_text,
    ka.title,
    ka.source_url,
    ka.metadata,
    sr.semantic_score,
    COALESCE(fr.fts_score, 0) AS fts_score,
    -- Reciprocal Rank Fusion scoring
    (0.7 * sr.semantic_score + 0.3 * COALESCE(fr.fts_score, 0)) AS combined_score
FROM semantic_results sr
JOIN knowledge_articles ka ON ka.id = sr.article_id
LEFT JOIN fts_results fr ON fr.article_id = sr.article_id
ORDER BY combined_score DESC
LIMIT 10;

-- Metadata-filtered similarity search
-- The JSONB filter runs before the ANN scan via the planner
SELECT
    kae.chunk_text,
    ka.title,
    ka.metadata->>'category' AS category,
    1 - (kae.embedding <=> $1::vector) AS similarity
FROM knowledge_article_embeddings kae
JOIN knowledge_articles ka ON ka.id = kae.article_id
WHERE kae.tenant_id = $2::UUID
  AND kae.model_name = 'text-embedding-3-small'
  AND ka.metadata->>'category' = $3
  AND ka.metadata->>'language' = $4
  AND ka.published_at >= NOW() - INTERVAL '90 days'
ORDER BY kae.embedding <=> $1::vector
LIMIT 5;

-- Check that the HNSW index is being used
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT chunk_text, 1 - (embedding <=> '[0.1, 0.2, ...]'::vector) AS similarity
FROM knowledge_article_embeddings
WHERE tenant_id = 'some-uuid'::UUID
  AND model_name = 'text-embedding-3-small'
ORDER BY embedding <=> '[0.1, 0.2, ...]'::vector
LIMIT 10;
-- Look for: "Index Scan using idx_article_embeddings_hnsw"
-- If you see Seq Scan, your WHERE clause is filtering too aggressively
-- for the planner to use the index — consider partial indexes per tenant

#### Tuning HNSW for Production

-- HNSW parameters control the accuracy/speed tradeoff
-- ef_search: higher = more accurate, slower. Default 40. Range: 10–500
SET hnsw.ef_search = 100;

-- For a session dedicated to high-accuracy retrieval:
SET hnsw.ef_search = 200;

-- For low-latency applications where approximate is fine:
SET hnsw.ef_search = 40;

-- Monitor index build progress (PostgreSQL 16+)
SELECT phase, blocks_done, blocks_total,
       round(100.0 * blocks_done / NULLIF(blocks_total, 0), 1) AS pct
FROM pg_stat_progress_create_index
WHERE relid = 'knowledge_article_embeddings'::regclass;

-- Measure actual recall quality against exact search
WITH exact AS (
    SELECT id, embedding <=> '[...]'::vector AS dist
    FROM knowledge_article_embeddings
    ORDER BY dist LIMIT 10
),
approx AS (
    SELECT id, embedding <=> '[...]'::vector AS dist
    FROM knowledge_article_embeddings
    ORDER BY dist LIMIT 10
)
SELECT
    COUNT(*) FILTER (WHERE e.id IN (SELECT id FROM approx)) AS recall_at_10
FROM exact e;

3.2 Embedding Generation: Connecting PostgreSQL to LLM APIs

The schema is ready. Now the question is how embeddings get created and stored. For production workloads, there are two patterns: synchronous generation at write time (for small volumes or interactive use cases), and async batch generation via Celery workers (for bulk ingestion and re-embedding when you change models).

# embedding_service.py
# Handles both OpenAI and Anthropic embeddings with automatic retry

import asyncio
import hashlib
import json
from typing import Optional
import asyncpg
import openai
import anthropic
import tiktoken
from tenacity import retry, stop_after_attempt, wait_exponential

# OpenAI client
oai_client = openai.AsyncOpenAI()

# Anthropic client (for claude-3-haiku embeddings via Voyage)
# Anthropic acquired Voyage AI for embeddings
import voyageai
voyage_client = voyageai.AsyncClient()

CHUNK_SIZE = 512    # tokens per chunk
CHUNK_OVERLAP = 64  # token overlap between chunks

def chunk_text(text: str, model: str = "text-embedding-3-small") -> list[str]:
    """Split text into overlapping chunks respecting token boundaries."""
    enc = tiktoken.encoding_for_model(model)
    tokens = enc.encode(text)
    
    chunks = []
    start = 0
    while start < len(tokens):
        end = min(start + CHUNK_SIZE, len(tokens))
        chunk_tokens = tokens[start:end]
        chunks.append(enc.decode(chunk_tokens))
        if end == len(tokens):
            break
        start += CHUNK_SIZE - CHUNK_OVERLAP
    
    return chunks


@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=2, max=10)
)
async def get_openai_embeddings(texts: list[str]) -> list[list[float]]:
    """Batch embed texts using OpenAI. Returns list of embedding vectors."""
    response = await oai_client.embeddings.create(
        model="text-embedding-3-small",
        input=texts,
        dimensions=1536
    )
    return [item.embedding for item in sorted(response.data, key=lambda x: x.index)]


async def embed_and_store_article(
    pool: asyncpg.Pool,
    article_id: int,
    tenant_id: str,
    title: str,
    body: str,
    model_name: str = "text-embedding-3-small"
) -> int:
    """
    Chunk an article, generate embeddings, and store them.
    Returns the number of chunks stored.
    
    This function is idempotent — running it twice replaces existing embeddings.
    """
    full_text = f"{title}\n\n{body}"
    chunks = chunk_text(full_text, model=model_name)
    
    if not chunks:
        return 0
    
    # Generate embeddings in a single API call (batch is much cheaper)
    embeddings = await get_openai_embeddings(chunks)
    
    enc = tiktoken.encoding_for_model(model_name)
    
    async with pool.acquire() as conn:
        # Use a transaction: either all chunks are stored or none
        async with conn.transaction():
            # Delete existing embeddings for this article + model
            await conn.execute(
                """
                DELETE FROM knowledge_article_embeddings
                WHERE article_id = $1 AND model_name = $2
                """,
                article_id, model_name
            )
            
            # Insert new chunks and embeddings
            await conn.executemany(
                """
                INSERT INTO knowledge_article_embeddings
                    (article_id, tenant_id, model_name, chunk_index, 
                     chunk_text, embedding, token_count)
                VALUES ($1, $2, $3, $4, $5, $6::vector, $7)
                """,
                [
                    (
                        article_id,
                        tenant_id,
                        model_name,
                        idx,
                        chunk,
                        json.dumps(embedding),  # asyncpg expects JSON for vector
                        len(enc.encode(chunk))
                    )
                    for idx, (chunk, embedding) in enumerate(zip(chunks, embeddings))
                ]
            )
    
    return len(chunks)


async def semantic_search(
    pool: asyncpg.Pool,
    query: str,
    tenant_id: str,
    limit: int = 5,
    similarity_threshold: float = 0.70,
    metadata_filter: Optional[dict] = None
) -> list[dict]:
    """
    Run semantic search over the knowledge base.
    Returns chunks with their source article metadata.
    """
    # Embed the query — single vector, not batched
    query_embedding = await get_openai_embeddings([query])
    embedding_vector = query_embedding[0]
    
    async with pool.acquire() as conn:
        # Set the tenant context for RLS
        await conn.execute(
            "SELECT set_config('app.tenant_id', $1, true)", tenant_id
        )
        
        if metadata_filter:
            rows = await conn.fetch(
                """
                SELECT
                    kae.chunk_text,
                    ka.title,
                    ka.source_url,
                    ka.metadata,
                    ka.published_at,
                    1 - (kae.embedding <=> $1::vector) AS similarity
                FROM knowledge_article_embeddings kae
                JOIN knowledge_articles ka ON ka.id = kae.article_id
                WHERE kae.tenant_id = $2::UUID
                  AND kae.model_name = 'text-embedding-3-small'
                  AND ka.is_active = true
                  AND ka.metadata @> $3::jsonb
                  AND 1 - (kae.embedding <=> $1::vector) > $4
                ORDER BY kae.embedding <=> $1::vector
                LIMIT $5
                """,
                json.dumps(embedding_vector),
                tenant_id,
                json.dumps(metadata_filter),
                similarity_threshold,
                limit
            )
        else:
            rows = await conn.fetch(
                """
                SELECT
                    kae.chunk_text,
                    ka.title,
                    ka.source_url,
                    ka.metadata,
                    ka.published_at,
                    1 - (kae.embedding <=> $1::vector) AS similarity
                FROM knowledge_article_embeddings kae
                JOIN knowledge_articles ka ON ka.id = kae.article_id
                WHERE kae.tenant_id = $2::UUID
                  AND kae.model_name = 'text-embedding-3-small'
                  AND ka.is_active = true
                  AND 1 - (kae.embedding <=> $1::vector) > $3
                ORDER BY kae.embedding <=> $1::vector
                LIMIT $4
                """,
                json.dumps(embedding_vector),
                tenant_id,
                similarity_threshold,
                limit
            )
        
        return [dict(row) for row in rows]

#### Async Batch Ingestion with Celery

# tasks/embedding_tasks.py
# Celery workers for bulk re-embedding when you upgrade models

from celery import Celery
import asyncpg
import asyncio
import logging
from embedding_service import embed_and_store_article

logger = logging.getLogger(__name__)

app = Celery(
    'embedding_tasks',
    broker='redis://localhost:6379/0',
    backend='redis://localhost:6379/1'
)

app.conf.update(
    task_serializer='json',
    result_serializer='json',
    accept_content=['json'],
    task_acks_late=True,           # Only ack after successful completion
    worker_prefetch_multiplier=1,  # One task at a time per worker (embedding is slow)
    task_reject_on_worker_lost=True,
)

DB_URL = "postgresql://aiapp:secret@localhost:5432/aiapp"


@app.task(bind=True, max_retries=3, default_retry_delay=30)
def embed_article_task(self, article_id: int, tenant_id: str):
    """Embed a single article. Called after insert/update."""
    async def _run():
        pool = await asyncpg.create_pool(DB_URL, min_size=1, max_size=3)
        try:
            # Fetch the article content
            async with pool.acquire() as conn:
                row = await conn.fetchrow(
                    "SELECT title, body FROM knowledge_articles WHERE id = $1",
                    article_id
                )
            
            if not row:
                logger.warning(f"Article {article_id} not found, skipping.")
                return
            
            chunks_stored = await embed_and_store_article(
                pool=pool,
                article_id=article_id,
                tenant_id=tenant_id,
                title=row['title'],
                body=row['body']
            )
            logger.info(f"Stored {chunks_stored} chunks for article {article_id}")
        finally:
            await pool.close()
    
    try:
        asyncio.run(_run())
    except Exception as exc:
        logger.error(f"Embedding task failed for article {article_id}: {exc}")
        raise self.retry(exc=exc)


@app.task
def bulk_reembed_tenant(tenant_id: str, model_name: str = "text-embedding-3-small"):
    """
    Re-embed all articles for a tenant — used when upgrading embedding models.
    Spawns individual tasks with rate limiting.
    """
    async def _get_article_ids():
        pool = await asyncpg.create_pool(DB_URL)
        async with pool.acquire() as conn:
            rows = await conn.fetch(
                "SELECT id FROM knowledge_articles WHERE tenant_id = $1 AND is_active = true",
                tenant_id
            )
        await pool.close()
        return [row['id'] for row in rows]
    
    article_ids = asyncio.run(_get_article_ids())
    logger.info(f"Re-embedding {len(article_ids)} articles for tenant {tenant_id}")
    
    # Dispatch individual tasks — Celery handles parallelism
    for article_id in article_ids:
        embed_article_task.apply_async(
            args=[article_id, tenant_id],
            countdown=0,
            retry=True
        )

3.3 pg_notify and Trigger-Based Embedding Pipelines

Polling for new content to embed is wasteful. PostgreSQL's LISTEN/NOTIFY mechanism gives you an event-driven approach: a trigger fires when content changes, publishes a notification, and your worker picks it up. This is how you build a real-time embedding pipeline without Kafka.
