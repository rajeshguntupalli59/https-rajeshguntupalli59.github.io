Chapter 30: Monitoring Cloud Databases: CloudWatch, Azure Monitor, Cloud Logging

Cloud-managed databases remove your access to the operating system, which means the monitoring stack you built around top, iostat, pg_stat_activity, and custom Nagios checks no longer has a foundation to stand on. The platform now owns the infrastructure layer, and you must rebuild your observability posture using the native tooling each cloud provider offers. This chapter walks through CloudWatch for AWS RDS and Aurora, Azure Monitor for Azure SQL and Elastic Pools, and Cloud Logging and Cloud Monitoring for Google Cloud SQL and AlloyDB. For each platform, you will learn how to configure metrics collection, set meaningful alert thresholds grounded in production experience, build dashboards that surface what matters, and avoid the most common instrumentation gaps that cause DBAs to miss incidents until users are already calling.

---

30.1 What You Lost and What You Gained: Monitoring Philosophy for Managed Databases

On-premises PostgreSQL and SQL Server gave you unrestricted access. You could run sar -u 1 10 to sample CPU across ten seconds, attach perf to the Postgres process, read /proc/meminfo for exact memory pressure, and correlate all of it with pg_stat_bgwriter in the same terminal session. The trade-off for managed databases is real: you cannot SSH into an RDS instance, you cannot inspect /var/log/postgresql/ directly, and you cannot run top on the underlying host.

What you gain in exchange is a pre-wired telemetry pipeline. CloudWatch, Azure Monitor, and Google Cloud Monitoring collect dozens of metrics automatically from the moment the instance starts, with no agent installation, no scrape configuration, and no storage management for the metric data itself. The platform also exposes metrics that would have been difficult or impossible to capture yourself — RDS Enhanced Monitoring samples OS-level statistics at one-second intervals using an agent that runs inside the managed environment, which is more granular than most self-managed deployments bother to configure.

The mental model shift is this: you are moving from pull-based, CLI-driven observation to push-based, subscription-driven alerting. Instead of checking the server, you define what matters and let the platform tell you when thresholds are crossed. This demands that you know your thresholds before an incident, not during one.

The four signal categories you must cover on any cloud database platform:

| Signal | Managed equivalent | What it replaces |

|--------|-------------------|-----------------|

| CPU and OS resource | CloudWatch, Azure Metrics, Cloud Monitoring | top, vmstat, sar |

| Storage and I/O | Platform metrics + Performance Insights | iostat, df, storage vendor tools |

| Query-level performance | Performance Insights, Query Store, Query Insights | pg_stat_statements, Profiler traces |

| Replication and HA state | Platform events, replica lag metrics | pg_stat_replication, AlwaysOn DMVs |

A note on metric resolution and cost. All three platforms charge for high-resolution metrics and extended retention. CloudWatch detailed monitoring costs roughly $0.01 per metric per month beyond the first ten. Azure Monitor logs ingestion costs approximately $2.76 per GB. Cloud Monitoring custom metrics cost $0.18 per metric-month after the first 150 MB. Understand your cost exposure before enabling everything at maximum resolution on a large fleet.

30.2 CloudWatch: Monitoring AWS RDS and Aurora in Production

CloudWatch is the primary observability surface for RDS and Aurora. It collects standard metrics at one-minute intervals automatically, and at ten-second intervals when Enhanced Monitoring is enabled on the instance. Performance Insights sits alongside CloudWatch and provides query-level attribution. Understanding which tool answers which question saves considerable time during incidents.

Standard RDS CloudWatch metrics and their production thresholds:

| Metric | Namespace | Healthy range (general guidance) |

|--------|-----------|----------------------------------|

| CPUUtilization | AWS/RDS | Sustained < 75%; alert at 85% for 5 min |

| FreeStorageSpace | AWS/RDS | Alert when < 10% of allocated storage |

| DatabaseConnections | AWS/RDS | Alert at 80% of max_connections |

| ReadLatency / WriteLatency | AWS/RDS | Alert > 20 ms for OLTP workloads |

| ReplicaLag | AWS/RDS | Alert > 30 seconds for read replicas |

| AuroraReplicaLag | AWS/RDS | Alert > 100 ms for Aurora replicas |

| Deadlocks | AWS/RDS | Alert on any sustained increase |

| FreeableMemory | AWS/RDS | Alert when < 256 MB on db.r6g.large |

ReplicaLag for standard RDS read replicas is a physical replication metric measured in seconds. Aurora's AuroraReplicaLag is fundamentally different — it measures how far behind an Aurora replica is in applying redo log pages to its local buffer cache from the shared storage layer, and it is usually measured in single-digit milliseconds. Do not apply the same alert threshold to both.

Enabling Enhanced Monitoring and Performance Insights via AWS CLI:

# Enable Enhanced Monitoring (1-second granularity) and Performance Insights
# on an existing RDS instance. The monitoring role must exist first.

# Step 1: Create the Enhanced Monitoring IAM role if it doesn't exist
aws iam create-role \
  --role-name rds-monitoring-role \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": { "Service": "monitoring.rds.amazonaws.com" },
      "Action": "sts:AssumeRole"
    }]
  }'

aws iam attach-role-policy \
  --role-name rds-monitoring-role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AmazonRDSEnhancedMonitoringRole

# Step 2: Modify the instance to enable both features
aws rds modify-db-instance \
  --db-instance-identifier prod-postgres-01 \
  --monitoring-interval 1 \
  --monitoring-role-arn arn:aws:iam::123456789012:role/rds-monitoring-role \
  --enable-performance-insights \
  --performance-insights-retention-period 731 \
  --performance-insights-kms-key-id alias/aws/rds \
  --apply-immediately

# Step 3: Verify the configuration
aws rds describe-db-instances \
  --db-instance-identifier prod-postgres-01 \
  --query 'DBInstances[0].{
    MonitoringInterval: MonitoringInterval,
    PIEnabled: PerformanceInsightsEnabled,
    PIRetention: PerformanceInsightsRetentionPeriod
  }'

Performance Insights retention of 731 days (two years) is the maximum. The default is 7 days free; anything beyond that costs approximately $0.02 per vCPU per hour. For a db.r6g.4xlarge (16 vCPUs), two-year retention costs roughly $5,500 per year per instance — budget for this explicitly.

Creating a CloudWatch alarm for connection saturation:

# Alert when DatabaseConnections exceeds 80% of max_connections.
# For db.r6g.large with default PostgreSQL max_connections of 87,
# the threshold is approximately 69 connections.
# Always verify max_connections from Parameter Groups before setting this.

aws cloudwatch put-metric-alarm \
  --alarm-name "rds-prod-postgres-01-connection-saturation" \
  --alarm-description "RDS connections above 80% of max_connections" \
  --metric-name DatabaseConnections \
  --namespace AWS/RDS \
  --dimensions Name=DBInstanceIdentifier,Value=prod-postgres-01 \
  --statistic Average \
  --period 60 \
  --evaluation-periods 3 \
  --threshold 69 \
  --comparison-operator GreaterThanOrEqualToThreshold \
  --alarm-actions arn:aws:sns:us-east-1:123456789012:prod-db-alerts \
  --ok-actions arn:aws:sns:us-east-1:123456789012:prod-db-alerts \
  --treat-missing-data notBreaching

# Retrieve current max_connections from the parameter group
aws rds describe-db-parameters \
  --db-parameter-group-name prod-postgres15-params \
  --query 'Parameters[?ParameterName==`max_connections`].{Name:ParameterName,Value:ParameterValue}'

One gotcha with DatabaseConnections: RDS Parameter Groups compute max_connections dynamically based on instance memory by default (LEAST({DBInstanceClassMemory/9531392}, 5000)). When you resize an instance, max_connections changes silently. If you hardcode alarm thresholds, they become wrong after a resize. Build automation to recalculate thresholds whenever an instance class changes, or use a CloudWatch metric math expression that accounts for the dynamic value.

Querying Performance Insights for top wait events using Python:

import boto3
from datetime import datetime, timedelta, timezone

def get_top_waits(db_instance_id: str, region: str = "us-east-1", minutes: int = 60):
    """
    Retrieve top wait events from Performance Insights for the last N minutes.
    Equivalent to querying pg_stat_activity for wait_event_type in PostgreSQL.
    """
    pi = boto3.client("pi", region_name=region)
    rds = boto3.client("rds", region_name=region)

    # PI requires the DbiResourceId, not the instance identifier
    response = rds.describe_db_instances(DBInstanceIdentifier=db_instance_id)
    resource_id = response["DBInstances"][0]["DbiResourceId"]

    end_time = datetime.now(timezone.utc)
    start_time = end_time - timedelta(minutes=minutes)

    response = pi.get_resource_metrics(
        ServiceType="RDS",
        Identifier=resource_id,
        StartTime=start_time,
        EndTime=end_time,
        PeriodInSeconds=60,
        MetricQueries=[
            {
                "Metric": "db.load.avg",
                "GroupBy": {
                    "Group": "db.wait_event",
                    "Dimensions": ["db.wait_event.name", "db.wait_event.type"],
                    "Limit": 10,
                },
            }
        ],
    )

    print(f"\nTop wait events for {db_instance_id} (last {minutes} minutes):\n")
    print(f"{'Wait Event':<45} {'Type':<20} {'Avg DB Load':>12}")
    print("-" * 80)

    for result in response.get("MetricList", []):
        for key_dim in result.get("DataPoints", []):
            pass  # We want the grouped keys

    # The grouped response structure differs — use Keys endpoint
    keys_response = pi.get_resource_metadata(
        ServiceType="RDS",
        Identifier=resource_id,
    )

    # Simpler approach: get dimension keys directly
    dim_response = pi.describe_dimension_keys(
        ServiceType="RDS",
        Identifier=resource_id,
        StartTime=start_time,
        EndTime=end_time,
        PeriodInSeconds=3600,
        Metric="db.load.avg",
        GroupBy={
            "Group": "db.wait_event",
            "Dimensions": ["db.wait_event.name", "db.wait_event.type"],
            "Limit": 10,
        },
    )

    for key in dim_response.get("Keys", []):
        wait_name = key["Dimensions"].get("db.wait_event.name", "unknown")
        wait_type = key["Dimensions"].get("db.wait_event.type", "unknown")
        avg_load = key.get("Total", 0)
        print(f"{wait_name:<45} {wait_type:<20} {avg_load:>12.4f}")

    return dim_response

if __name__ == "__main__":
    results = get_top_waits("prod-postgres-01", region="us-east-1", minutes=60)

The output maps directly to what you would see in pg_stat_activity.wait_event and wait_event_type. If LWLock:buffer_mapping appears at the top, your buffer pool is under pressure. If IO:DataFileRead dominates, your working set exceeds buffer pool capacity. The interpretation is identical to on-premises; only the collection mechanism differs.

Aurora-specific metrics worth explicit attention:

Aurora exposes metrics standard RDS does not. AuroraVolumeBytesLeftTotal tracks remaining capacity in the Aurora cluster volume. The cluster volume grows in 10 GB increments and supports up to 128 TB, but it does not shrink automatically. BufferCacheHitRatio for Aurora should stay above 99.9% for OLTP; if it drops below 99%, you are seeing genuine I/O latency against the distributed storage layer. CommitLatency and DDLLatency are Aurora-specific metrics that have no direct CloudWatch equivalent in standard RDS.

30.3 Azure Monitor: Metrics, Diagnostics, and Log Analytics for Azure SQL

Azure Monitor is a unified observability platform that covers Azure SQL Database, Managed Instances, and Elastic Pools through two distinct systems: Metrics (pre-aggregated, 93-day retention, low cost) and Diagnostic Logs (raw records streamed to a Log Analytics Workspace, Storage Account, or Event Hub, with cost proportional to data volume).

The equivalent of CloudWatch Enhanced Monitoring is the Intelligent Performance blade in the Azure portal, which surfaces Query Store data through the Query Performance Insight interface. The equivalent of Performance Insights is the sys.dm_exec_query_stats and sys.query_store_* views in Azure SQL, which remain accessible because Azure SQL gives you T-SQL access to DMVs — a significant advantage over RDS if you come from a SQL Server background.

Core Azure SQL metrics and alert thresholds:

| Metric name | Alert threshold (OLTP) | Notes |

|-------------|------------------------|-------|

| cpu_percent | > 80% for 5 minutes | Based on DTU or vCore allocation |

| storage_percent | > 85% | Alert well before 100% — autogrow is not automatic |

| connection_failed | > 10 per minute | Indicates authentication or firewall issues |

| deadlock | Any increase > baseline | Query Store records the deadlock graph |

| dtu_consumption_percent | > 85% | DTU model only — consider vCore if consistently high |

| workers_percent | > 80% | Maximum worker threads; hitting 100% causes errors |

| log_write_percent | > 90% | Indicates log I/O becoming a bottleneck |

Configuring Diagnostic Settings to stream metrics to Log Analytics:

# Prerequisites: Log Analytics Workspace already created
# Retrieve the workspace resource ID
WORKSPACE_ID=$(az monitor log-analytics workspace show \
  --resource-group prod-rg \
  --workspace-name prod-db-logs \
  --query id \
  --output tsv)

# Get the Azure SQL Database resource ID
DB_RESOURCE_ID=$(az sql db show \
  --resource-group prod-rg \
  --server prod-sql-server-01 \
  --name prod-appdb \
  --query id \
  --output tsv)

# Enable all diagnostic categories to Log Analytics
# SQLInsights includes query-level data from Query Store
# AutomaticTuning includes index recommendation events
# QueryStoreRuntimeStatistics is the most valuable for query troubleshooting
az monitor diagnostic-settings create \
  --name "prod-appdb-diagnostics" \
  --resource "$DB_RESOURCE_ID" \
  --workspace "$WORKSPACE_ID" \
  --logs '[
    {"category": "SQLInsights", "enabled": true, "retentionPolicy": {"enabled": false}},
    {"category": "AutomaticTuning", "enabled": true, "retentionPolicy": {"enabled": false}},
    {"category": "QueryStoreRuntimeStatistics", "enabled": true, "retentionPolicy": {"enabled": false}},
    {"category": "QueryStoreWaitStatistics", "enabled": true, "retentionPolicy": {"enabled": false}},
    {"category": "Errors", "enabled": true, "retentionPolicy": {"enabled": false}},
    {"category": "DatabaseWaitStatistics", "enabled": true, "retentionPolicy": {"enabled": false}},
    {"category": "Deadlocks", "enabled": true, "retentionPolicy": {"enabled": false}},
    {"category": "Blocks", "enabled": true, "retentionPolicy": {"enabled": false}}
  ]' \
  --metrics '[
    {"category": "Basic", "enabled": true},
    {"category": "InstanceAndAppAdvanced", "enabled": true}
  ]'

# Verify the diagnostic setting
az monitor diagnostic-settings list \
  --resource "$DB_RESOURCE_ID" \
  --query '[].{Name:name, Workspace:workspaceId}' \
  --output table

There is a meaningful delay between enabling Diagnostic Settings and data appearing in Log Analytics — typically 5 to 15 minutes. Do not assume the feature is broken if the workspace shows no data immediately after configuration. Also note that enabling diagnostic settings on an Elastic Pool is separate from enabling them on each database within the pool. The pool-level settings cover pool resource metrics (eDTUs, storage); per-database settings cover query-level telemetry.

Querying Log Analytics with KQL for wait statistics:

# Run a KQL query against the Log Analytics Workspace to find top wait types
# over the last 24 hours — equivalent to querying sys.dm_os_wait_stats
# on an on-premises SQL Server instance

az monitor log-analytics query \
  --workspace "$WORKSPACE_ID" \
  --analytics-query '
AzureDiagnostics
| where ResourceType == "DATABASES"
| where Category == "DatabaseWaitStatistics"
| where TimeGenerated > ago(24h)
| where Resource == "PROD-APPDB"
| summarize
    TotalWaitTime_ms = sum(delta_wait_time_ms_d),
    AvgWaitTime_ms = avg(delta_wait_time_ms_d),
    SampleCount = count()
    by wait_type_s
| sort by TotalWaitTime_ms desc
| take 15
| project
    WaitType = wait_type_s,
    TotalWait_Seconds = round(TotalWaitTime_ms / 1000.0, 1),
    AvgWait_ms = round(AvgWaitTime_ms, 2),
    Samples = SampleCount
' \
  --timespan P1D \
  --output table

# Find the top 10 CPU-consuming queries from Query Store via Log Analytics
az monitor log-analytics query \
  --workspace "$WORKSPACE_ID" \
  --analytics-query '
AzureDiagnostics
| where Category == "QueryStoreRuntimeStatistics"
| where TimeGenerated > ago(6h)
| where Resource == "PROD-APPDB"
| summarize
    TotalCPU_ms = sum(cpu_time_d),
    TotalDuration_ms = sum(duration_d),
    ExecutionCount = sum(count_executions_d),
    AvgLogicalReads = avg(avg_logical_io_reads_d)
    by query_hash_s, query_id_d
| sort by TotalCPU_ms desc
| take 10
| project
    QueryID = query_id_d,
    QueryHash = query_hash_s,
    TotalCPU_seconds = round(TotalCPU_ms / 1000000.0, 2),
    Executions = ExecutionCount,
    AvgLogicalReads = round(AvgLogicalReads, 0)
' \
  --output table

The KQL query against QueryStoreRuntimeStatistics maps directly to sys.query_store_runtime_stats joined with sys.query_store_plan and sys.query_store_query in on-premises SQL Server. If you have built Query Store dashboards on-premises, the transition to KQL is largely a syntax change rather than a conceptual one. The query_hash_s field lets you correlate across databases or time windows even when query IDs differ.

Creating a metric alert for Elastic Pool DTU consumption:

# Alert when Elastic Pool eDTU consumption exceeds 85% for 10 minutes
POOL_RESOURCE_ID=$(az sql elastic-pool show \
  --resource-group prod-rg \
  --server prod-sql-server-01 \
  --name prod-elastic-pool-01 \
  --query id \
  --output tsv)

ACTION_GROUP_ID=$(az monitor action-group show \
  --resource-group prod-rg \
  --name prod-db-oncall \
  --query id \
  --output tsv)

az monitor metrics alert create \
  --name "elastic-pool-dtu-saturation" \
  --resource-group prod-rg \
  --scopes "$POOL_RESOURCE_ID" \
  --condition "avg dtu_consumption_percent > 85" \
  --window-size 10m \
  --evaluation-frequency 5m \
  --severity 2 \
  --description "Elastic Pool eDTU consumption above 85% — consider scaling or workload redistribution" \
  --action "$ACTION_GROUP_ID"

One alert configuration mistake that appears frequently: setting the evaluation frequency equal to the window size. If your window is 10 minutes and you evaluate every 10 minutes, a spike that occurs in the last two minutes of a window will not fire an alert until the next evaluation cycle — potentially 18 minutes after the spike began. Use an evaluation frequency of 5 minutes or less when the window size is 10 minutes.

30.4 Google Cloud Monitoring and Cloud Logging for Cloud SQL and AlloyDB

Google Cloud Monitoring (formerly Stackdriver) collects Cloud SQL and AlloyDB metrics automatically with no configuration required. The metrics are available in the cloudsql.googleapis.com and alloydb.googleapis.com namespaces. Cloud Logging captures database flags, slow query logs, and general logs as structured log entries that can be queried with the Logs Explorer or exported to BigQuery for long-term analysis.

The model is similar to Azure Monitor: there is a metrics layer (low cost, pre-aggregated, 6-week default retention in Cloud Monitoring) and a logging layer (structured logs in Cloud Logging, charged per GB ingested beyond the free 50 GB per project per month, with 30-day default retention).

Critical Cloud SQL metrics:

| Metric | Full path | Alert threshold |

|--------|-----------|----------------|

| CPU utilization | cloudsql.googleapis.com/database/cpu/utilization | > 0.80 for 5 min |

| Disk utilization | cloudsql.googleapis.com/database/disk/utilization | > 0.85 |

| Memory utilization | cloudsql.googleapis.com/database/memory/utilization | > 0.90 |

| Active connections | cloudsql.googleapis.com/database/postgresql/num_backends | > 80% of max_connections |

| Replica lag (seconds) | cloudsql.googleapis.com/database/replication/replica_lag | > 30 seconds |

| Deadlock count | cloudsql.googleapis.com/database/postgresql/deadlock_count | Any non-zero trend |

| Transaction ID utilization | cloudsql.googleapis.com/database/postgresql/transaction_id_utilization | > 0.70 |
