Chapter 10: RDS Proxy: Connection Pooling at Scale

Connection pooling is a problem every production DBA has solved before — PgBouncer on a dedicated VM, SQL Server's built-in connection pooling through the driver, ProxySQL sitting between your application tier and MySQL. You know the problem well: thousands of application threads each wanting their own database connection, a database engine that pays real memory and CPU cost per connection, and an impedance mismatch between how application frameworks think about concurrency and how database engines actually work. In the cloud, that problem doesn't disappear — it gets worse. Serverless functions spin up hundreds of ephemeral connections in milliseconds. Auto-scaling application tiers surge from 10 to 500 instances in minutes. Lambda functions have no persistent process to maintain a connection pool. RDS Proxy is AWS's managed answer to this problem, and understanding it deeply — its architecture, its limits, its failure modes — is what separates a cloud DBA who simply enables it from one who operates it confidently in production.

---

10.1 Why Connection Pooling Is Different in the Cloud

On-premises, you controlled both ends of the connection. You knew exactly how many application servers existed, you sized PgBouncer or pgpool-II based on those numbers, and you tuned max_connections in postgresql.conf to match. The population was relatively static. In a cloud-native architecture, the application tier is ephemeral by design.

Consider what happens when you deploy an AWS Lambda function that queries RDS PostgreSQL. Lambda has a concurrency limit of up to 10,000 simultaneous executions per account per region by default. If each invocation opens a database connection — even briefly — and your RDS PostgreSQL instance is sized at db.r6g.2xlarge with 64 GB RAM, your practical max_connections is around 1,700 (PostgreSQL's default formula is LEAST(DBInstanceClassMemory/9531392, 5000) for RDS, which you can verify by querying the parameter group). A traffic spike that triggers 2,000 concurrent Lambda invocations doesn't just degrade performance — it throws FATAL: sorry, too many clients already errors and your application fails completely.

The same dynamic applies to containerized applications on ECS or EKS where horizontal pod autoscaling responds faster than your database can absorb connection growth. Each new pod starts fresh, establishes its own connections, and the old pod's connections may linger in TIME_WAIT or hold idle transactions open while the process terminates.

What RDS Proxy does architecturally is sit as a fully managed, highly available connection pooling layer between your clients and your RDS or Aurora database. It maintains a pool of long-lived connections to the database and multiplexes thousands of client connections across that pool. From your database's perspective, it sees a small, stable number of connections from the proxy. From your application's perspective, it sees a normal database endpoint.

The fundamental multiplexing mechanism is called *connection pinning* in RDS Proxy's terminology — and understanding when pinning happens is the most important operational concept in this chapter.

Connection pinning occurs when RDS Proxy cannot safely multiplex a client connection across the shared pool and must dedicate a backend database connection exclusively to that client for the duration of the session. When a connection is pinned, you lose the pooling benefit entirely for that client. Pinning is triggered by:

  • → `SET` statements that change session-level configuration (e.g., `SET search_path TO myschema`)
  • → Temporary tables (`CREATE TEMP TABLE`)
  • → Table locks (`LOCK TABLE`)
  • → Cursors opened with `HOLD`
  • → Prepared statements (in certain configurations)
  • → `SET TRANSACTION ISOLATION LEVEL` before a transaction begins

    • For PostgreSQL users who relied on SET search_path in connection strings — a common pattern — this is an immediate compatibility issue. The fix is to set search_path at the user level in the database itself (ALTER ROLE myapp SET search_path TO myschema) rather than at session level in the connection string.

    For SQL Server users, RDS Proxy does not support SQL Server. RDS Proxy supports MySQL-compatible engines (RDS MySQL, Aurora MySQL) and PostgreSQL-compatible engines (RDS PostgreSQL, Aurora PostgreSQL). SQL Server on RDS still uses driver-level connection pooling. Keep that limitation in mind throughout this chapter.

    Maximum connections math: When you create an RDS Proxy, you set a --max-connections-percent parameter (1–100, default 100). This tells the proxy what percentage of the database's max_connections it may use for the backend pool. The proxy itself can accept far more client connections than that — the proxy endpoint can handle up to 64,000 client connections simultaneously. The multiplexing ratio is what gives you scale. If your db.r6g.2xlarge has max_connections = 1700 and you leave max-connections-percent at 100, the proxy will hold up to 1,700 backend connections but can serve tens of thousands of client connections across them — provided those client connections aren't all executing transactions simultaneously and aren't pinned.

    10.2 Creating and Configuring RDS Proxy

    RDS Proxy runs inside your VPC. It creates network interfaces in your subnets and registers against your RDS endpoint. The proxy itself is managed by AWS — you have no EC2 instances to patch, no process to restart — but you do have configuration choices that matter.

    IAM authentication is mandatory for RDS Proxy. The proxy authenticates to your database using credentials stored in AWS Secrets Manager. This is non-negotiable — you cannot use a proxy without Secrets Manager. The proxy fetches credentials from Secrets Manager on your behalf, which means the IAM role attached to the proxy must have secretsmanager:GetSecretValue on the specific secret. Your application, meanwhile, can authenticate to the proxy either using the same database credentials (passed through) or using IAM authentication (where the application gets an auth token from STS and uses it as a password).

    Here is the sequence using AWS CLI to create a proxy for an Aurora PostgreSQL cluster:

    # Step 1: Create the Secrets Manager secret for the database user
aws secretsmanager create-secret \
  --name "prod/aurora-pg/appuser" \
  --description "RDS Proxy credentials for appuser" \
  --secret-string '{"username":"appuser","password":"StrongPassword123!"}' \
  --region us-east-1

# Step 2: Get the secret ARN (you'll need this)
SECRET_ARN=$(aws secretsmanager describe-secret \
  --secret-id "prod/aurora-pg/appuser" \
  --query 'ARN' \
  --output text)

# Step 3: Create an IAM role for RDS Proxy
# First, create the trust policy document
cat > /tmp/proxy-trust-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "rds.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
EOF

aws iam create-role \
  --role-name RDSProxyRole \
  --assume-role-policy-document file:///tmp/proxy-trust-policy.json

# Step 4: Create and attach the permissions policy
cat > /tmp/proxy-permissions.json << EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "secretsmanager:GetSecretValue",
      "Resource": "${SECRET_ARN}"
    },
    {
      "Effect": "Allow",
      "Action": "kms:Decrypt",
      "Resource": "arn:aws:kms:us-east-1:123456789012:key/your-kms-key-id",
      "Condition": {
        "StringEquals": {
          "kms:ViaService": "secretsmanager.us-east-1.amazonaws.com"
        }
      }
    }
  ]
}
EOF

aws iam put-role-policy \
  --role-name RDSProxyRole \
  --policy-name RDSProxySecretsAccess \
  --policy-document file:///tmp/proxy-permissions.json

ROLE_ARN=$(aws iam get-role --role-name RDSProxyRole --query 'Role.Arn' --output text)

# Step 5: Create the RDS Proxy
aws rds create-db-proxy \
  --db-proxy-name prod-aurora-pg-proxy \
  --engine-family POSTGRESQL \
  --auth '[
    {
      "Description": "App user credentials",
      "AuthScheme": "SECRETS",
      "SecretArn": "'"${SECRET_ARN}"'",
      "IAMAuth": "DISABLED"
    }
  ]' \
  --role-arn "${ROLE_ARN}" \
  --vpc-subnet-ids subnet-0a1b2c3d subnet-0e4f5a6b subnet-0c7d8e9f \
  --vpc-security-group-ids sg-0123456789abcdef0 \
  --require-tls \
  --idle-client-timeout 1800 \
  --debug-logging false \
  --region us-east-1

# Step 6: Register the Aurora cluster as the proxy target
aws rds register-db-proxy-targets \
  --db-proxy-name prod-aurora-pg-proxy \
  --db-cluster-identifiers prod-aurora-cluster \
  --region us-east-1

# Step 7: Verify proxy status (takes 3-5 minutes to become available)
aws rds describe-db-proxies \
  --db-proxy-name prod-aurora-pg-proxy \
  --query 'DBProxies[0].{Status:Status,Endpoint:Endpoint}' \
  --output table

    The --idle-client-timeout parameter (in seconds, default 1800) determines how long the proxy waits before closing a client connection that has sent no traffic. This is distinct from the connection keep-alive at the backend. Tune this to be shorter than any NAT gateway or load balancer timeout in your network path — hitting a silent TCP timeout from a middlebox is harder to debug than an explicit proxy timeout.

    Terraform configuration for the same setup, which is how you should manage this in production:

    # terraform/modules/rds_proxy/main.tf

resource "aws_secretsmanager_secret" "db_credentials" {
  name                    = "prod/aurora-pg/appuser"
  description             = "RDS Proxy credentials for appuser"
  recovery_window_in_days = 7

  tags = {
    Environment = "production"
    ManagedBy   = "terraform"
  }
}

resource "aws_secretsmanager_secret_version" "db_credentials" {
  secret_id = aws_secretsmanager_secret.db_credentials.id
  secret_string = jsonencode({
    username = "appuser"
    password = var.db_password  # sourced from Terraform variable, never hardcoded
  })
}

resource "aws_iam_role" "rds_proxy" {
  name = "rds-proxy-role-${var.environment}"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = { Service = "rds.amazonaws.com" }
      Action    = "sts:AssumeRole"
    }]
  })
}

resource "aws_iam_role_policy" "rds_proxy_secrets" {
  name = "rds-proxy-secrets-access"
  role = aws_iam_role.rds_proxy.id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect   = "Allow"
      Action   = "secretsmanager:GetSecretValue"
      Resource = aws_secretsmanager_secret.db_credentials.arn
    }]
  })
}

resource "aws_db_proxy" "main" {
  name                   = "prod-aurora-pg-proxy"
  debug_logging          = false
  engine_family          = "POSTGRESQL"
  idle_client_timeout    = 1800
  require_tls            = true
  role_arn               = aws_iam_role.rds_proxy.arn
  vpc_security_group_ids = [aws_security_group.proxy.id]
  vpc_subnet_ids         = var.private_subnet_ids

  auth {
    auth_scheme = "SECRETS"
    description = "App user credentials"
    iam_auth    = "DISABLED"
    secret_arn  = aws_secretsmanager_secret.db_credentials.arn
  }

  tags = {
    Environment = "production"
    ManagedBy   = "terraform"
  }
}

resource "aws_db_proxy_default_target_group" "main" {
  db_proxy_name = aws_db_proxy.main.name

  connection_pool_config {
    connection_borrow_timeout    = 120   # seconds to wait for a connection from pool
    max_connections_percent      = 90    # leave 10% headroom for direct connections
    max_idle_connections_percent = 50    # proxy keeps this % of max as idle connections
    session_pinning_filters      = ["EXCLUDE_VARIABLE_SETS"]  # reduces pinning for SET statements
  }
}

resource "aws_db_proxy_target" "aurora_cluster" {
  db_cluster_identifier = aws_rds_cluster.main.id
  db_proxy_name         = aws_db_proxy.main.name
  target_group_name     = aws_db_proxy_default_target_group.main.name
}

resource "aws_security_group" "proxy" {
  name        = "rds-proxy-sg"
  description = "Security group for RDS Proxy"
  vpc_id      = var.vpc_id

  ingress {
    from_port       = 5432
    to_port         = 5432
    protocol        = "tcp"
    security_groups = [var.application_sg_id]  # only app tier can reach proxy
  }

  egress {
    from_port       = 5432
    to_port         = 5432
    protocol        = "tcp"
    security_groups = [var.rds_sg_id]  # proxy can reach RDS
  }
}

output "proxy_endpoint" {
  value       = aws_db_proxy.main.endpoint
  description = "RDS Proxy endpoint for application connection strings"
}

    The session_pinning_filters = ["EXCLUDE_VARIABLE_SETS"] setting in the connection pool configuration is worth highlighting. When enabled, the proxy does not pin connections when it encounters SET statements for a specific list of safe variables — those that don't affect transaction semantics. Check the current AWS documentation for which variables qualify, as the list expands with proxy engine updates. This reduces pinning significantly for applications that use SET for read-only configuration like application_name or TimeZone.

    The connection_borrow_timeout (120 seconds by default) is the time a client waits for the proxy to make a backend connection available before returning a timeout error. If you see applications timing out at exactly 120 seconds during traffic spikes, this is why — and the fix is usually to reduce max_connections_percent slightly so the proxy maintains more idle connections available for borrowing rather than running the pool at full saturation.

    10.3 IAM Authentication Through the Proxy

    IAM database authentication through RDS Proxy adds a second authentication layer and changes how you think about credential management. Instead of application code storing database passwords, applications generate short-lived authentication tokens using AWS STS. Those tokens expire after 15 minutes and are cryptographically tied to the IAM identity that generated them.

    The flow is:

    1. Your EC2 instance, Lambda function, or ECS task has an IAM role attached.

    2. The application code calls rds.generate_db_auth_token() (boto3) to generate a token.

    3. The application connects to the proxy endpoint using that token as the password.

    4. The proxy validates the token against IAM, then presents stored credentials to the actual database.

    This is the cloud equivalent of what you used to manage with .pgpass files, SSL certificates, and Kerberos — except IAM handles the identity federation and there are no long-lived passwords in your application configuration. What you did with pg_hba.conf entries and certificate authentication, you now do with IAM policies that specify which IAM principals can generate tokens for which database users.

    # Python application code: IAM auth token generation for RDS Proxy
# This pattern works for Lambda, ECS tasks, and EC2 instances

import boto3
import psycopg2
import os
from botocore.exceptions import ClientError

def get_db_connection_iam():
    """
    Establish a database connection using IAM authentication through RDS Proxy.
    The IAM role attached to this execution environment must have
    rds-db:connect permission for the target database user.
    """
    
    rds_client = boto3.client('rds', region_name='us-east-1')
    
    proxy_endpoint = os.environ['DB_PROXY_ENDPOINT']
    # e.g., "prod-aurora-pg-proxy.proxy-abc123.us-east-1.rds.amazonaws.com"
    
    db_user = os.environ['DB_USER']  # "appuser" — must match IAM policy resource
    db_name = os.environ['DB_NAME']  # "production"
    db_port = int(os.environ.get('DB_PORT', '5432'))
    
    try:
        # Token is valid for 15 minutes; generate fresh on each connection
        # For connection pools in long-running processes, regenerate before expiry
        auth_token = rds_client.generate_db_auth_token(
            DBHostname=proxy_endpoint,
            Port=db_port,
            DBUsername=db_user,
            Region='us-east-1'
        )
    except ClientError as e:
        raise RuntimeError(f"Failed to generate IAM auth token: {e.response['Error']['Message']}")
    
    # SSL is required for IAM authentication — the proxy enforces TLS
    # Download the RDS certificate bundle from AWS:
    # https://truststore.pki.rds.amazonaws.com/us-east-1/us-east-1-bundle.pem
    ssl_cert_path = '/etc/ssl/certs/us-east-1-bundle.pem'
    
    conn = psycopg2.connect(
        host=proxy_endpoint,
        port=db_port,
        database=db_name,
        user=db_user,
        password=auth_token,
        sslmode='verify-full',
        sslrootcert=ssl_cert_path,
        connect_timeout=10,
        application_name='my-production-service'
    )
    
    return conn


def get_db_connection_pool_iam():
    """
    For long-running services (ECS, EC2), use a connection pool with token refresh.
    psycopg2's connection pool doesn't handle token expiry natively —
    this wrapper handles the 15-minute token refresh requirement.
    """
    import psycopg2.pool
    import threading
    import time
    
    class IAMConnectionPool:
        def __init__(self, min_conn, max_conn, **conn_kwargs):
            self.conn_kwargs = conn_kwargs
            self.min_conn = min_conn
            self.max_conn = max_conn
            self._lock = threading.Lock()
            self._pool = None
            self._token_expiry = 0
            self._refresh_pool()
        
        def _generate_token(self):
            rds_client = boto3.client('rds', region_name='us-east-1')
            return rds_client.generate_db_auth_token(
                DBHostname=self.conn_kwargs['host'],
                Port=self.conn_kwargs['port'],
                DBUsername=self.conn_kwargs['user'],
                Region='us-east-1'
            )
        
        def _refresh_pool(self):
            """Recreate the pool with a fresh token. Called when token approaches expiry."""
            token = self._generate_token()
            self._token_expiry = time.time() + (14 * 60)  # refresh at 14 min, expires at 15
            
            if self._pool:
                self._pool.closeall()
            
            self._pool = psycopg2.pool.ThreadedConnectionPool(
                self.min_conn,
                self.max_conn,
                password=token,
                **self.conn_kwargs
            )
        
        def getconn(self):
            with self._lock:
                if time.time() > self._token_expiry:
                    self._refresh_pool()
            return self._pool.getconn()
        
        def putconn(self, conn):
            self._pool.putconn(conn)
    
    return IAMConnectionPool(
        min_conn=2,
        max_conn=20,
        host=os.environ['DB_PROXY_ENDPOINT'],
        port=5432,
        database=os.environ['DB_NAME'],
        user=os.environ['DB_USER'],
        sslmode='verify-full',
        sslrootcert='/etc/ssl/certs/us-east-1-bundle.pem',
        connect_timeout=10
    )


# Lambda handler example — Lambda has no persistent pool,
# but RDS Proxy handles the pooling on the server side
def lambda_handler(event, context):
    conn = None
    try:
        conn = get_db_connection_iam()
        with conn.cursor() as cur:
            cur.execute("SELECT COUNT(*) FROM orders WHERE status = %s", ('pending',))
            count = cur.fetchone()[0]
        conn.commit()
        return {'pending_orders': count}
    except Exception as e:
        if conn:
            conn.rollback()
        raise
    finally:
        if conn:
            conn.close()  # Returns connection to proxy pool, not actual DB disconnection

    The IAM policy that allows this token generation must specify the database user as part of the resource ARN:

    # Create IAM policy permitting IAM DB auth for specific user through the proxy
aws iam create-policy \
  --policy-name RDSProxyIAMAuth \
  --policy-document '{
    "Version": "2012-10-17",
    "Statement": [
      {
        "Effect": "Allow",
        "Action": "rds-db:connect",
        "Resource": "arn:aws:rds-db:us-east-1:123456789012:dbuser:prx-0a1b2c3d4e5f/appuser"
      }
    ]
  }'

# Attach this policy to the Lambda execution role or ECS task role
aws iam attach-role-policy \
  --role-name MyLambdaExecutionRole \
  --policy-arn arn:aws:iam::123456789012:policy/RDSProxyIAMAuth

# The resource ARN format for proxy IAM auth is:
# arn:aws:rds-db:{region}:{account-id}:dbuser:{proxy-resource-id}/{db-username}
# Get the proxy resource ID:
aws rds describe-db-proxies \
  --db-proxy-name prod-aurora-pg-proxy \
  --query 'DBProxies[0].DBProxyArn' \
  --output text
# Extract "prx-XXXXXXXXXX" from the ARN — that's the resource ID portion

    One gotcha that bites teams regularly: the IAM resource ARN uses the proxy resource ID (the prx- identifier), not the proxy name or the cluster ARN. When the policy is wrong, the application gets a generic authentication failure with no indication that IAM authorization is the cause — the proxy rejects the token before the database is involved, and the error looks identical to a wrong password. Always test token generation separately from the connection attempt when debugging IAM auth failures.
