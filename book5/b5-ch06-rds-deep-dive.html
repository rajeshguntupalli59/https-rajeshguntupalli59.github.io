Chapter 6: RDS Deep Dive: Engines, Instance Types, and Storage

Amazon RDS is where most DBAs begin their cloud journey, and for good reason — it takes the engines you already know (PostgreSQL, MySQL, SQL Server, Oracle) and wraps them in a managed layer that handles provisioning, patching, and backups. But "managed" does not mean "transparent." The choices you make about engine version, instance family, and storage type have cascading effects on performance, cost, and operational flexibility that cannot be undone without downtime or data migration. This chapter breaks down those choices with the specificity that production systems demand: which instance families to use and why, how RDS storage behaves differently from what you know on-premises, where the hard limits are, and what you give up when AWS takes over the OS layer.

---

6.1 Engine Selection and Version Strategy

RDS supports six database engines: MySQL, PostgreSQL, MariaDB, Oracle, Microsoft SQL Server, and Amazon Aurora (which gets its own chapter, but shares the RDS console). Each engine runs on a managed EC2 instance underneath, and the distinction between "RDS for PostgreSQL" and "self-managed PostgreSQL on EC2" is not just operational convenience — it represents a fundamental trade-off in control.

What you lose versus self-managed PostgreSQL on EC2:

  • → No OS-level access. You cannot SSH into the underlying host.
  • → No `pg_hba.conf` direct editing. Authentication rules are enforced through security groups, IAM database authentication, and SSL certificate requirements set via parameter groups.
  • → No superuser. The `rds_superuser` role grants most capabilities, but it is not true superuser — you cannot load arbitrary shared libraries, and you cannot access system catalogs in ways that require true OS-level privilege.
  • → No WAL archive access. You can enable WAL archiving to S3 for logical replication, but you cannot read the raw WAL files directly.
  • → No `pg_basebackup` from the instance. Backups are managed by RDS using EBS snapshots.

    • What you gain:

  • → Automated backups with point-in-time recovery (PITR) up to 35 days.
  • → Automated minor version upgrades on a configurable maintenance window.
  • → Multi-AZ failover handled by the platform without you managing Pacemaker or Patroni.
  • → Monitoring metrics surfaced through CloudWatch without running exporters.

    • Engine version selection is not just about features — it determines your upgrade path. RDS maintains a list of supported minor versions, and AWS periodically deprecates older ones, issuing mandatory upgrades. PostgreSQL 11 reached end of life on AWS in 2023 and instances still running it received forced upgrades. You should never run more than one major version behind current in production.

    Check available versions before provisioning:

    # List all available PostgreSQL engine versions in us-east-1
aws rds describe-db-engine-versions \
  --engine postgres \
  --query 'DBEngineVersions[*].[EngineVersion,DBEngineVersionDescription]' \
  --output table \
  --region us-east-1

# Check which versions support a specific feature (e.g., Kerberos auth)
aws rds describe-db-engine-versions \
  --engine postgres \
  --query 'DBEngineVersions[?contains(SupportedFeatureNames, `kerberos`) == `true`].[EngineVersion]' \
  --output text \
  --region us-east-1

    SQL Server licensing deserves separate attention. RDS SQL Server offers two licensing models: License Included (LI) and Bring Your Own License (BYOL). License Included is available for Standard and Enterprise editions, with the license cost embedded in the hourly rate. Enterprise on a db.r6i.16xlarge runs roughly $12–15/hour just for the license component. If your organization already has SQL Server Enterprise Agreement licenses, BYOL reduces that cost significantly — but BYOL requires Software Assurance, and AWS audits eligibility.

    Oracle on RDS follows the same LI vs. BYOL model. Oracle Enterprise Edition is only available as BYOL. Standard Edition 2 is available as License Included. The moment you need features like Partitioning, Advanced Compression, or Real Application Testing, you're in Enterprise Edition territory, which means BYOL and your own Oracle licensing agreement.

    Version pinning strategy: In production, pin your minor version explicitly. Do not enable auto minor version upgrade on database instances that have application dependencies on specific behavior (pg_catalog views changed significantly between PostgreSQL 13 and 14, for example). Enable auto minor version upgrade only on development and QA instances.

    # Disable auto minor version upgrade on an existing instance
aws rds modify-db-instance \
  --db-instance-identifier prod-postgres-01 \
  --no-auto-minor-version-upgrade \
  --apply-immediately \
  --region us-east-1

# Verify the change
aws rds describe-db-instances \
  --db-instance-identifier prod-postgres-01 \
  --query 'DBInstances[0].AutoMinorVersionUpgrade' \
  --region us-east-1

    6.2 Instance Families and Sizing

    The RDS instance naming convention follows the same pattern as EC2: db.{family}{generation}.{size}. Understanding the family is more important than the size, because each family has different CPU architecture, memory ratios, and network characteristics.

    Current generation families:

    | Family | vCPU:Memory Ratio | Use Case | Example |

    |--------|-------------------|----------|---------|

    | db.t4g | Burstable, 1:2 | Dev/test, small workloads | db.t4g.medium: 2 vCPU, 4 GB |

    | db.m7g | General purpose, 1:4 | Web applications, moderate OLTP | db.m7g.4xlarge: 16 vCPU, 64 GB |

    | db.r7g | Memory optimized, 1:8 | Large buffers, analytics | db.r7g.8xlarge: 32 vCPU, 256 GB |

    | db.r7i | Memory optimized, Intel | SQL Server (no ARM support) | db.r7i.16xlarge: 64 vCPU, 512 GB |

    | db.x2idn | Memory intensive, 1:32 | SAP HANA, large in-memory | db.x2idn.16xlarge: 64 vCPU, 2048 GB |

    The g suffix means Graviton (ARM-based). Graviton instances offer 10–20% better price-performance for PostgreSQL and MySQL workloads compared to equivalent Intel or AMD instances. The catch: SQL Server and Oracle are not supported on Graviton because AWS cannot run those binaries on ARM. PostgreSQL and MySQL have native ARM builds that run well on Graviton.

    T-series instances (burstable) in production: The T-series uses CPU credits. When the instance is idle, it accumulates credits. Under sustained load, it burns through credits and throttles to its baseline CPU (which is 20–40% of nominal). This behavior is catastrophic for production OLTP workloads that have sustained query pressure. The only legitimate production use of T-series is for small read replicas with light traffic, or internal tooling databases with infrequent queries.

    You can detect credit exhaustion in CloudWatch:

    # Check CPU credit balance for a burstable instance
aws cloudwatch get-metric-statistics \
  --namespace AWS/RDS \
  --metric-name CPUCreditBalance \
  --dimensions Name=DBInstanceIdentifier,Value=dev-postgres-t4g \
  --start-time 2024-01-15T00:00:00Z \
  --end-time 2024-01-15T23:59:59Z \
  --period 3600 \
  --statistics Minimum \
  --output table

# Also check CPU credit usage rate
aws cloudwatch get-metric-statistics \
  --namespace AWS/RDS \
  --metric-name CPUCreditUsage \
  --dimensions Name=DBInstanceIdentifier,Value=dev-postgres-t4g \
  --start-time 2024-01-15T00:00:00Z \
  --end-time 2024-01-15T23:59:59Z \
  --period 3600 \
  --statistics Sum \
  --output table

    Sizing recommendations for PostgreSQL migration: When migrating a self-managed PostgreSQL instance, the most important metric from your source system is shared_buffers utilization — specifically, what fraction of your working set fits in the buffer cache. On RDS, shared_buffers defaults to 25% of instance memory for PostgreSQL. A db.r7g.4xlarge has 128 GB of RAM, giving you ~32 GB of shared_buffers. If your on-premises system has a 40 GB buffer pool with reasonable hit rates, move to db.r7g.8xlarge (256 GB RAM, ~64 GB shared_buffers) with headroom.

    The right way to size using actual database metrics:

    -- Run this on your SOURCE PostgreSQL instance before migration
-- to understand buffer cache behavior

-- Buffer hit rate (should be >95% in production)
SELECT 
    sum(heap_blks_hit) AS heap_hits,
    sum(heap_blks_read) AS heap_reads,
    round(
        sum(heap_blks_hit)::numeric / 
        nullif(sum(heap_blks_hit) + sum(heap_blks_read), 0) * 100, 2
    ) AS buffer_hit_rate_pct
FROM pg_statio_user_tables;

-- Working set estimate: tables + indexes that fit in cache
SELECT 
    schemaname,
    relname AS table_name,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||relname)) AS total_size,
    round(
        heap_blks_hit::numeric / nullif(heap_blks_hit + heap_blks_read, 0) * 100, 1
    ) AS table_hit_rate_pct
FROM pg_statio_user_tables
WHERE heap_blks_hit + heap_blks_read > 10000
ORDER BY pg_total_relation_size(schemaname||'.'||relname) DESC
LIMIT 20;

-- Identify tables causing physical reads (candidates for shared_buffers increase)
SELECT 
    schemaname,
    relname,
    heap_blks_read AS physical_reads,
    heap_blks_hit AS cache_hits
FROM pg_statio_user_tables
WHERE heap_blks_read > 1000
ORDER BY heap_blks_read DESC
LIMIT 10;

    Changing instance class: RDS instance class changes require a reboot. On Single-AZ, this means downtime of 3–10 minutes. On Multi-AZ, AWS performs the change on the standby first, then fails over — total visible downtime is typically 30–120 seconds for the failover. Plan instance resizing during low-traffic windows regardless of Multi-AZ, because the failover itself causes connection drops.

    6.3 RDS Storage: gp2, gp3, io1, io2, and Magnetic

    Storage in RDS is EBS-backed, and the storage type you choose determines IOPS characteristics, throughput ceilings, and cost. Getting this wrong is the most common cause of RDS performance problems, because storage bottlenecks in managed services are much harder to diagnose than on-premises where you can run iostat directly.

    Available storage types:

    gp2 (General Purpose SSD): The legacy default. Delivers 3 IOPS per GB of storage, with a burst pool for volumes under 1 TB. A 100 GB gp2 volume gets 300 IOPS baseline with burst to 3,000 IOPS using credits (same credit model as T-series CPU). Above 3.33 TB (10,000 IOPS at 3 IOPS/GB), you get the full 16,000 IOPS ceiling without burst dependency. Maximum throughput: 250 MB/s. Do not use gp2 for new deployments — it has been superseded entirely by gp3.

    gp3 (General Purpose SSD v2): The current recommended default. Provides a baseline of 3,000 IOPS and 125 MB/s throughput regardless of volume size. You can independently provision additional IOPS up to 16,000 and throughput up to 1,000 MB/s at additional cost. A 100 GB gp3 volume gets 3,000 IOPS — the same as a 1 TB gp2 volume, at one-tenth the storage cost. For most OLTP workloads under 16,000 IOPS, gp3 is the right choice.

    io1 (Provisioned IOPS SSD): Allows IOPS provisioning independent of storage size, up to 64,000 IOPS for MySQL and PostgreSQL (256,000 for SQL Server and Oracle on io1). Maximum throughput: 1,000 MB/s. More expensive than gp3 for the same IOPS level. io1 made sense before gp3 existed; today, it's primarily for SQL Server workloads needing over 16,000 IOPS.

    io2 (Provisioned IOPS SSD v2): Higher durability (99.999% vs 99.8% for io1), same IOPS ceiling. For SQL Server and Oracle, io2 supports up to 256,000 IOPS. For most PostgreSQL and MySQL workloads, gp3 with provisioned IOPS is sufficient and cheaper.

    Magnetic (standard): Legacy spinning disk. Do not use this for any workload. It exists only for backward compatibility.

    Storage autoscaling: RDS supports automatic storage scaling. When free storage drops below 10% of allocated storage (or below 5 GB, whichever is smaller), RDS automatically increases storage. Autoscaling only increases storage — it never shrinks it. You cannot reduce allocated storage on an RDS instance without creating a new instance and migrating data. Set your initial storage thoughtfully; you will pay for every GB permanently.

    Configure a new PostgreSQL instance with gp3 and autoscaling via Terraform:

    # terraform/rds_instance.tf
# Production PostgreSQL RDS with gp3 storage and appropriate settings

resource "aws_db_instance" "prod_postgres" {
  identifier        = "prod-postgres-primary"
  engine            = "postgres"
  engine_version    = "16.3"
  instance_class    = "db.r7g.4xlarge"

  # Storage configuration
  storage_type          = "gp3"
  allocated_storage     = 500          # GB - initial allocation
  max_allocated_storage = 2000         # GB - autoscaling ceiling
  iops                  = 12000        # Additional provisioned IOPS above gp3 baseline (3000)
  storage_throughput    = 500          # MB/s - above gp3 baseline (125 MB/s)
  storage_encrypted     = true
  kms_key_id            = aws_kms_key.rds_encryption.arn

  # Engine configuration
  db_name  = "appdb"
  username = "dbadmin"
  password = var.db_password   # Use Secrets Manager in production

  # High availability
  multi_az               = true
  availability_zone      = "us-east-1a"   # for single-AZ; ignored when multi_az=true

  # Backup and maintenance
  backup_retention_period = 14
  backup_window           = "03:00-04:00"
  maintenance_window      = "sun:04:30-sun:05:30"
  deletion_protection     = true

  # Network
  db_subnet_group_name   = aws_db_subnet_group.private.name
  vpc_security_group_ids = [aws_security_group.rds_postgres.id]
  publicly_accessible    = false

  # Parameter and option groups
  parameter_group_name = aws_db_parameter_group.postgres16_prod.name

  # Monitoring
  monitoring_interval             = 60   # Enhanced monitoring every 60 seconds
  monitoring_role_arn             = aws_iam_role.rds_enhanced_monitoring.arn
  performance_insights_enabled    = true
  performance_insights_retention_period = 731  # 2 years

  # Upgrade behavior
  auto_minor_version_upgrade = false
  apply_immediately          = false   # Apply changes during maintenance window

  tags = {
    Environment = "production"
    Team        = "platform-db"
    CostCenter  = "engineering"
  }
}

resource "aws_db_parameter_group" "postgres16_prod" {
  name   = "postgres16-prod"
  family = "postgres16"

  parameter {
    name  = "shared_buffers"
    value = "{DBInstanceClassMemory/4}"   # 25% of instance memory
  }

  parameter {
    name  = "effective_cache_size"
    value = "{DBInstanceClassMemory*3/4}"
  }

  parameter {
    name  = "max_connections"
    value = "500"
  }

  parameter {
    name  = "log_min_duration_statement"
    value = "1000"  # Log queries over 1 second
  }

  parameter {
    name  = "log_checkpoints"
    value = "1"
  }

  parameter {
    name  = "log_lock_waits"
    value = "1"
  }

  parameter {
    name  = "track_io_timing"
    value = "1"
  }

  parameter {
    name  = "pg_stat_statements.track"
    value = "all"
  }

  parameter {
    name  = "rds.log_retention_period"
    value = "10080"   # 7 days in minutes
  }
}

    IOPS math for production sizing: On gp3, the baseline 3,000 IOPS costs nothing extra. Each additional IOPS above 3,000 costs approximately $0.005/IOPS/month. Provisioning 12,000 IOPS means 9,000 additional IOPS at $0.005 = $45/month in IOPS charges, on top of the storage cost. For a 500 GB gp3 volume, storage is $0.08/GB/month = $40/month, plus $45/month for IOPS, plus $0.04/MB/s above 125 MB/s baseline for throughput. Compare this to io1: 12,000 IOPS on io1 costs $0.10/IOPS/month = $1,200/month, plus storage. gp3 wins decisively below 64,000 IOPS.

    Monitoring storage saturation in CloudWatch:

    # Check storage I/O metrics for an RDS instance
# ReadIOPS and WriteIOPS tell you current throughput
# ReadLatency and WriteLatency tell you if storage is saturating

aws cloudwatch get-metric-statistics \
  --namespace AWS/RDS \
  --metric-name ReadLatency \
  --dimensions Name=DBInstanceIdentifier,Value=prod-postgres-primary \
  --start-time 2024-01-15T09:00:00Z \
  --end-time 2024-01-15T10:00:00Z \
  --period 60 \
  --statistics Average Maximum \
  --output table

# DiskQueueDepth > 1 consistently indicates storage saturation
aws cloudwatch get-metric-statistics \
  --namespace AWS/RDS \
  --metric-name DiskQueueDepth \
  --dimensions Name=DBInstanceIdentifier,Value=prod-postgres-primary \
  --start-time 2024-01-15T09:00:00Z \
  --end-time 2024-01-15T10:00:00Z \
  --period 60 \
  --statistics Average Maximum \
  --output table

    If DiskQueueDepth averages above 1 and WriteLatency exceeds 10ms consistently, your provisioned IOPS are insufficient. Increase provisioned IOPS without downtime on gp3 — the change is applied immediately (though it takes up to 6 hours for the modification to complete depending on volume size).

    The storage full emergency: RDS will put your database in read-only mode when storage reaches 100%. At that point, you cannot write data including WAL. The sequence of events is: free storage drops → autoscaling triggers (if enabled) → scaling takes 5–15 minutes → if autoscaling is not enabled or is at ceiling, the database goes read-only. Set a CloudWatch alarm at 20% free storage, not 10%, so you have time to respond before hitting autoscaling or the ceiling.

    6.4 Parameter Groups: The Replacement for postgresql.conf

    On a self-managed PostgreSQL instance, you edit /etc/postgresql/16/main/postgresql.conf directly and reload with SELECT pg_reload_conf() or restart the service. On RDS, that file is read-only. All configuration changes go through Parameter Groups — a named collection of configuration values that you associate with one or more DB instances.

    There are two types: DB Parameter Groups (instance-level) and DB Cluster Parameter Groups (Aurora cluster-level). For non-Aurora RDS, you use DB Parameter Groups.

    A parameter group has a family (e.g., postgres16, mysql8.0, sqlserver-se-15.0) and contains every configurable parameter for that engine version. AWS ships a default.postgres16 parameter group that contains AWS-optimized defaults. You cannot modify the default group — you must create a custom one and attach it to your instance.

    Parameters have two apply types:

  • → **Static:** Requires a database restart to take effect. Examples: `shared_buffers`, `max_connections`, `wal_level`.
  • → **Dynamic:** Takes effect immediately on `apply_immediately` or after the maintenance window. Examples: `log_min_duration_statement`, `effective_cache_size`, `work_mem`.
