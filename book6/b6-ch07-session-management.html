Chapter 7: Session Management for Multi-Turn LLM Conversations

Every LLM API call is stateless. The model receives a list of messages, generates a response, and forgets everything. Building a chatbot or agent that feels coherent across turns — that remembers what the user said three messages ago, that can reference an earlier decision, that knows when a conversation ended and a new one began — requires you to manage all of that state yourself. PostgreSQL is exceptionally well-suited for this job, and this chapter shows you exactly how to do it.

You will design a conversation session schema that scales from prototype to production, implement context window management so you never send more tokens than a model can handle, build sliding-window and summary-based memory strategies, wire everything into a FastAPI service with async Python, and add pg_notify for real-time streaming. By the end, you will have a session layer that any LLM application can run on top of.

---

7.1 The Conversation State Problem

When a user types "what about the second option you mentioned?" the model needs the full conversation history to understand what "second option" means. You are responsible for loading that history from storage, trimming it to fit within a context window, injecting it into the API call, and then saving the new exchange back to storage — all within the latency budget of a user-facing request.

This is not a simple key-value cache problem. Conversation state has structure: turns have roles, timestamps, token counts, and optional metadata. Sessions have owners, creation times, expiry policies, and relationships to RAG retrieval results. You need to query "show me all sessions where the user asked about billing in the last 30 days" — something Redis alone cannot do efficiently. You need row-level security so that user A cannot read user B's conversation history. You need audit logs for compliance. These are exactly the things PostgreSQL was built for.

The fundamental data model has three entities: a session (a container that groups turns), a message (a single turn from user, assistant, system, or tool), and a session summary (a compressed representation of older context). Everything else flows from these three tables.

-- Core session management schema
-- Run this in your application database

CREATE EXTENSION IF NOT EXISTS "pgcrypto";
CREATE EXTENSION IF NOT EXISTS "vector";

-- ── Sessions ────────────────────────────────────────────────────────────────

CREATE TABLE ai_sessions (
    session_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id           UUID NOT NULL,
    title             TEXT,                          -- auto-generated or user-set
    model_name        TEXT NOT NULL DEFAULT 'gpt-4o',
    system_prompt     TEXT,                          -- session-level system prompt
    temperature       NUMERIC(3,2) DEFAULT 0.7,
    status            TEXT NOT NULL DEFAULT 'active'
                          CHECK (status IN ('active', 'archived', 'deleted')),
    token_budget      INTEGER NOT NULL DEFAULT 128000,
    tokens_used       INTEGER NOT NULL DEFAULT 0,
    metadata          JSONB NOT NULL DEFAULT '{}',   -- app-specific context
    created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_active_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    expires_at        TIMESTAMPTZ,                   -- NULL = never
    CONSTRAINT valid_temperature CHECK (temperature BETWEEN 0.0 AND 2.0)
);

CREATE INDEX idx_sessions_user_status
    ON ai_sessions (user_id, status, last_active_at DESC);

CREATE INDEX idx_sessions_metadata
    ON ai_sessions USING GIN (metadata);

-- ── Messages ─────────────────────────────────────────────────────────────────

CREATE TABLE ai_messages (
    message_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id        UUID NOT NULL REFERENCES ai_sessions (session_id) ON DELETE CASCADE,
    role              TEXT NOT NULL CHECK (role IN ('system', 'user', 'assistant', 'tool')),
    content           TEXT NOT NULL,
    tool_name         TEXT,                          -- populated when role = 'tool'
    tool_call_id      TEXT,                          -- OpenAI function calling correlation
    token_count       INTEGER,                       -- filled by application after counting
    prompt_tokens     INTEGER,                       -- filled from API response
    completion_tokens INTEGER,                       -- filled from API response
    model_name        TEXT,                          -- which model actually responded
    finish_reason     TEXT,                          -- 'stop', 'length', 'tool_calls', etc.
    embedding         VECTOR(1536),                  -- for semantic message search
    latency_ms        INTEGER,                       -- time to first token / full response
    metadata          JSONB NOT NULL DEFAULT '{}',   -- citations, sources, scores, etc.
    created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    sequence_num      INTEGER NOT NULL               -- explicit ordering within a session
);

CREATE UNIQUE INDEX idx_messages_session_sequence
    ON ai_messages (session_id, sequence_num);

CREATE INDEX idx_messages_session_created
    ON ai_messages (session_id, created_at);

CREATE INDEX idx_messages_role
    ON ai_messages (session_id, role);

CREATE INDEX idx_messages_embedding
    ON ai_messages USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- Automatically update session activity and token counter
CREATE OR REPLACE FUNCTION update_session_on_message()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
    UPDATE ai_sessions
    SET
        last_active_at = NOW(),
        tokens_used    = tokens_used + COALESCE(NEW.token_count, 0)
    WHERE session_id = NEW.session_id;
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_session_update
    AFTER INSERT ON ai_messages
    FOR EACH ROW EXECUTE FUNCTION update_session_on_message();

-- ── Session Summaries ─────────────────────────────────────────────────────────

CREATE TABLE ai_session_summaries (
    summary_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id        UUID NOT NULL REFERENCES ai_sessions (session_id) ON DELETE CASCADE,
    summary_text      TEXT NOT NULL,
    embedding         VECTOR(1536),
    covers_up_to_seq  INTEGER NOT NULL,              -- highest sequence_num summarized
    token_count       INTEGER,
    model_name        TEXT NOT NULL,
    created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_summaries_session
    ON ai_session_summaries (session_id, covers_up_to_seq DESC);

-- ── Audit Log ─────────────────────────────────────────────────────────────────

CREATE TABLE ai_session_audit (
    audit_id     BIGSERIAL PRIMARY KEY,
    session_id   UUID NOT NULL,
    user_id      UUID NOT NULL,
    event_type   TEXT NOT NULL,                     -- 'created', 'archived', 'deleted', 'exported'
    actor_id     UUID,                              -- could be a different admin user
    detail       JSONB DEFAULT '{}',
    created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_audit_session ON ai_session_audit (session_id, created_at);
CREATE INDEX idx_audit_user    ON ai_session_audit (user_id, created_at);

This schema gives you something no pure-vector store provides: you can write SQL like SELECT session_id, COUNT(*) as turns FROM ai_messages WHERE role = 'user' GROUP BY session_id HAVING COUNT(*) > 20 to find long conversations, join sessions to your users table for billing, or alert on sessions where finish_reason = 'length' because the model ran out of context. The relational layer is what makes conversations queryable assets rather than opaque blobs.

7.2 Context Window Management

Every model has a maximum context length — 128K tokens for GPT-4o, 200K for Claude 3.5 Sonnet, 1M for Gemini 1.5. Exceeding it causes the API to return an error or silently truncate your history. Managing what goes into the context window is one of the most important responsibilities of your session layer.

The naive approach — send all messages — breaks the moment a conversation grows past the limit. The two production-viable strategies are sliding window (keep the N most recent messages) and summarize-and-trim (compress old messages into a summary, keep recent ones verbatim). Both require knowing the token count of each message before you can make the decision.

-- Context window management queries

-- ── 1. Get messages for a context window (sliding window strategy) ──────────
-- Returns messages in chronological order, limited to stay under token budget.
-- Always includes system message(s) plus the most recent turns.

WITH session_info AS (
    SELECT token_budget FROM ai_sessions WHERE session_id = $1
),
system_msgs AS (
    SELECT message_id, role, content, token_count, sequence_num
    FROM   ai_messages
    WHERE  session_id = $1 AND role = 'system'
    ORDER  BY sequence_num
),
system_tokens AS (
    SELECT COALESCE(SUM(token_count), 0) AS total FROM system_msgs
),
recent_msgs AS (
    -- Walk backwards from most recent, accumulating tokens until we hit the budget
    SELECT
        m.message_id,
        m.role,
        m.content,
        m.tool_name,
        m.tool_call_id,
        m.token_count,
        m.sequence_num,
        SUM(m.token_count) OVER (
            ORDER BY m.sequence_num DESC
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) AS running_tokens
    FROM ai_messages m, session_info si, system_tokens st
    WHERE m.session_id = $1
      AND m.role != 'system'
    ORDER BY m.sequence_num DESC
)
SELECT role, content, tool_name, tool_call_id, sequence_num
FROM (
    SELECT * FROM system_msgs
    UNION ALL
    SELECT message_id, role, content, tool_name, token_count, sequence_num
    FROM recent_msgs r, session_info si, system_tokens st
    WHERE r.running_tokens <= (si.token_budget - st.total - 2000)  -- 2000 token reserve for response
) combined
ORDER BY sequence_num;

-- ── 2. Check whether summarization is needed ─────────────────────────────────

SELECT
    s.session_id,
    s.tokens_used,
    s.token_budget,
    ROUND(s.tokens_used::numeric / s.token_budget * 100, 1) AS pct_used,
    COUNT(m.message_id) FILTER (WHERE m.role != 'system') AS non_system_turns,
    MAX(COALESCE(ss.covers_up_to_seq, 0)) AS last_summarized_seq,
    MAX(m.sequence_num) AS latest_seq
FROM ai_sessions s
JOIN ai_messages m ON m.session_id = s.session_id
LEFT JOIN ai_session_summaries ss ON ss.session_id = s.session_id
WHERE s.session_id = $1
GROUP BY s.session_id, s.tokens_used, s.token_budget;

-- ── 3. Get messages that need to be summarized ────────────────────────────────
-- Returns the "old" portion of the conversation: everything before the most
-- recent summary, down to a configurable recent-window size.

SELECT role, content, sequence_num
FROM ai_messages
WHERE session_id    = $1
  AND sequence_num  > $2   -- $2 = covers_up_to_seq from latest summary (or 0)
  AND sequence_num  <= (
      SELECT MAX(sequence_num) - $3   -- $3 = number of recent turns to keep verbatim
      FROM ai_messages
      WHERE session_id = $1
  )
  AND role != 'system'
ORDER BY sequence_num;

-- ── 4. Insert a new summary record ───────────────────────────────────────────

INSERT INTO ai_session_summaries
    (session_id, summary_text, covers_up_to_seq, token_count, model_name)
VALUES
    ($1, $2, $3, $4, $5)
RETURNING summary_id;

-- ── 5. Session health dashboard ────────────────────────────────────────────────

SELECT
    s.session_id,
    s.user_id,
    s.model_name,
    s.status,
    s.tokens_used,
    s.token_budget,
    ROUND(s.tokens_used::numeric / s.token_budget * 100, 1)   AS context_pct,
    COUNT(m.message_id)                                         AS total_messages,
    COUNT(m.message_id) FILTER (WHERE m.role = 'user')         AS user_turns,
    AVG(m.latency_ms)  FILTER (WHERE m.role = 'assistant')     AS avg_latency_ms,
    SUM(m.completion_tokens) FILTER (WHERE m.role = 'assistant') AS total_output_tokens,
    s.last_active_at,
    s.created_at
FROM ai_sessions s
LEFT JOIN ai_messages m ON m.session_id = s.session_id
WHERE s.user_id = $1
  AND s.status  = 'active'
GROUP BY s.session_id
ORDER BY s.last_active_at DESC;

The token reserve in query 1 (the - 2000 part) is deliberate: you always want headroom for the model's response. If you pack the context to exactly the limit, the model may return finish_reason = 'length' and cut off mid-sentence. Build this reserve into the query rather than relying on the application layer to remember it.
