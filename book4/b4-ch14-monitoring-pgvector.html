Chapter 14: Monitoring and Diagnosing pgvector Query Performance

pgvector brings vector similarity search into PostgreSQL, which means you get the full benefit of a mature relational engine alongside your embedding workloads—but you also inherit PostgreSQL's particular approach to query planning, index maintenance, and execution statistics. Understanding what to measure, where to look, and how to interpret what you find is the difference between a vector search layer that degrades silently and one that you can tune with confidence. This chapter walks through the full monitoring stack for pgvector: from the query-level diagnostics you should run daily, through index health signals, to production observability patterns that integrate with your existing database monitoring infrastructure.

---

14.1 Understanding How pgvector Executes Similarity Queries

Before you can diagnose performance problems, you need a mental model of what pgvector is actually doing when it executes a vector similarity query. Seasoned DBAs often apply their intuition from B-tree indexes and sequential scan costs, but pgvector's behavior is different enough that those instincts can mislead.

The two execution paths

pgvector supports two index types: IVFFlat and HNSW. Both serve approximate nearest neighbor (ANN) searches but traverse entirely different data structures.

IVFFlat works by partitioning vectors into *lists* (Voronoi cells) at index build time. At query time, the probes parameter controls how many of those lists are scanned. A low probes value is fast but potentially inaccurate; a high value approaches brute-force performance while improving recall. This is directly analogous to a partition pruning parameter—except that unlike partition pruning, which is binary (either a partition is scanned or it isn't), probes is a continuous tradeoff.

HNSW builds a layered graph of approximate neighbors. At query time, ef_search controls how many candidates are explored during graph traversal. Higher ef_search means better recall and slower queries. HNSW generally outperforms IVFFlat on recall-latency tradeoffs for the same dataset, but it builds slower and consumes more memory.

When no index is present, or when the query planner decides an index scan is more expensive than a sequential scan, pgvector falls back to an exact brute-force search over the entire table. For tables under roughly 100,000 rows at 1,536 dimensions (a common OpenAI embedding size), this sequential scan is often competitive with or faster than indexed search—a fact that surprises DBAs accustomed to indexes always being beneficial.

What the planner sees

PostgreSQL's query planner treats vector index scans as any other access path, but it has limited visibility into ANN recall and the cost of graph or cluster traversal. The planner uses generic cost estimates for index scans, which can lead to suboptimal decisions—especially when filters combine vector predicates with scalar predicates (for example, WHERE tenant_id = $1 ORDER BY embedding <=> $2 LIMIT 10). This filter interaction is one of the most common sources of performance surprises in production pgvector deployments and is covered in detail in section 14.4.

14.2 Core Query Diagnostics: EXPLAIN, ANALYZE, and Execution Statistics

The standard PostgreSQL diagnostic toolkit works for pgvector queries, but you need to know what to look for in the output.

Reading EXPLAIN ANALYZE for vector queries

-- PostgreSQL / pgvector
-- Basic vector query with EXPLAIN ANALYZE
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT
    id,
    content,
    embedding <=> '[0.021, -0.045, 0.133, ...]'::vector AS distance
FROM documents
ORDER BY embedding <=> '[0.021, -0.045, 0.133, ...]'::vector
LIMIT 10;

Key fields to examine in the output:

  • → **Node type**: Look for `Index Scan using ` vs `Seq Scan`. A sequential scan on a large table is a signal that either no index exists, the index was not built on the same operator class, or the planner estimated the sequential scan as cheaper.
  • → **Actual Rows vs Plan Rows**: A large mismatch here indicates stale table statistics. Run `ANALYZE documents` and recheck.
  • → **Buffers: shared hit / shared read**: High `shared read` values indicate that the index or table data is not in `shared_buffers`, causing disk I/O. For HNSW indexes, the upper layers of the graph are accessed on nearly every query and should be cache-resident.
  • → **Actual Time**: The wall-clock time for node execution. For a 10-result ANN query on a properly indexed 1M-row table using HNSW with typical `ef_search=64`, expect actual time in the 5–50ms range depending on vector dimensions and hardware.

    • Confirming which index is being used

    -- PostgreSQL / pgvector
-- Verify index usage and operator class
SELECT
    indexname,
    indexdef
FROM pg_indexes
WHERE tablename = 'documents';

-- Check that the query is using the expected index
EXPLAIN SELECT id
FROM documents
ORDER BY embedding <=> '[0.021, -0.045, 0.133, ...]'::vector
LIMIT 10;

-- Force a sequential scan to establish baseline comparison
SET enable_indexscan = off;
EXPLAIN (ANALYZE, BUFFERS)
SELECT id
FROM documents
ORDER BY embedding <=> '[0.021, -0.045, 0.133, ...]'::vector
LIMIT 10;
RESET enable_indexscan;

    Sampling query latency from pg_stat_statements

    pg_stat_statements captures aggregate performance data across all executions of a normalized query. For vector queries, this gives you the sustained latency profile rather than a single EXPLAIN run.

    -- PostgreSQL / pgvector
-- Enable pg_stat_statements if not already active
-- Add to postgresql.conf: shared_preload_libraries = 'pg_stat_statements'

-- Find the slowest vector similarity queries
SELECT
    left(query, 120)                          AS query_preview,
    calls,
    round(mean_exec_time::numeric, 2)         AS mean_ms,
    round(stddev_exec_time::numeric, 2)       AS stddev_ms,
    round(total_exec_time::numeric, 2)        AS total_ms,
    round((100 * total_exec_time /
        sum(total_exec_time) OVER ())::numeric, 2) AS pct_total,
    rows / calls                              AS avg_rows_returned
FROM pg_stat_statements
WHERE query ILIKE '%<=>%'
   OR query ILIKE '%<#>%'
   OR query ILIKE '%<+>%'
ORDER BY mean_exec_time DESC
LIMIT 20;

    The three operator patterns cover the three distance metrics: <=> for cosine distance, <#> for negative inner product, and <+> for L1 distance. A well-tuned ANN query returning 10 results from a 1M-row indexed table should show mean_ms under 50ms; values above 200ms warrant investigation.

    14.3 Index Health and Build Quality

    An IVFFlat or HNSW index that was built under suboptimal conditions—too few rows, wrong lists count for IVFFlat, or insufficient ef_construction for HNSW—will deliver degraded recall or performance that no amount of query tuning can fix. Index health monitoring is a maintenance discipline, not a one-time setup check.

    Inspecting index metadata

    -- PostgreSQL / pgvector
-- View index build parameters and storage
SELECT
    i.relname                      AS index_name,
    t.relname                      AS table_name,
    pg_size_pretty(pg_relation_size(i.oid)) AS index_size,
    ix.indisvalid                  AS is_valid,
    ix.indisready                  AS is_ready,
    am.amname                      AS access_method,
    pg_get_indexdef(i.oid)         AS index_definition
FROM pg_index ix
JOIN pg_class i  ON i.oid = ix.indexrelid
JOIN pg_class t  ON t.oid = ix.indrelid
JOIN pg_am am    ON am.oid = i.relam
WHERE t.relname = 'documents'
  AND am.amname IN ('hnsw', 'ivfflat');

-- Check for bloat: rows in table vs rows at index build time
-- (IVFFlat does not automatically rebalance after inserts)
SELECT
    schemaname,
    relname,
    n_live_tup,
    n_dead_tup,
    last_vacuum,
    last_analyze,
    last_autoanalyze
FROM pg_stat_user_tables
WHERE relname = 'documents';

    IVFFlat: the reindex-after-growth rule

    IVFFlat partitions are fixed at index build time. If your table had 100,000 rows when you built the index with lists = 100 and now has 2,000,000 rows, the list distribution is wildly suboptimal. The recommended formula is lists = sqrt(row_count) for tables up to a few million rows, and lists = row_count / 1000 for larger tables.

    -- PostgreSQL / pgvector
-- Check current table size and compute recommended lists value
SELECT
    relname,
    n_live_tup                             AS live_rows,
    round(sqrt(n_live_tup::numeric))       AS recommended_lists_small,
    round(n_live_tup / 1000.0)             AS recommended_lists_large,
    pg_size_pretty(pg_relation_size(relid)) AS table_size
FROM pg_stat_user_tables
WHERE relname = 'documents';

-- Rebuild the IVFFlat index with corrected parameters
-- Use CONCURRENTLY to avoid locking in production
DROP INDEX CONCURRENTLY IF EXISTS documents_embedding_ivfflat_idx;

CREATE INDEX CONCURRENTLY documents_embedding_ivfflat_idx
ON documents
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 1000);  -- adjust based on query above

    HNSW: memory pressure signals

    HNSW indexes hold their upper graph layers in memory for performance. On a system where shared_buffers is too small or the index is being evicted, query latency spikes. Monitor buffer hit ratios specifically for the index relation:

    -- PostgreSQL / pgvector
-- Buffer hit ratio for the vector index specifically
SELECT
    c.relname,
    sum(b.heap_blks_hit)                                          AS heap_hits,
    sum(b.heap_blks_read)                                         AS heap_reads,
    round(100.0 * sum(b.heap_blks_hit) /
        nullif(sum(b.heap_blks_hit) + sum(b.heap_blks_read), 0), 2) AS heap_hit_pct,
    sum(b.idx_blks_hit)                                           AS idx_hits,
    sum(b.idx_blks_read)                                          AS idx_reads,
    round(100.0 * sum(b.idx_blks_hit) /
        nullif(sum(b.idx_blks_hit) + sum(b.idx_blks_read), 0), 2)  AS idx_hit_pct
FROM pg_statio_user_tables b
JOIN pg_class c ON c.relname = b.relname
WHERE b.relname = 'documents'
GROUP BY c.relname;

    Target idx_hit_pct above 95% for HNSW indexes. Values below 80% indicate the index is being repeatedly read from disk, which will push your median query latency above 100ms regardless of ef_search settings.

    14.4 Diagnosing the Scalar-Filter Interaction Problem

    The most common production performance issue in pgvector deployments is not slow ANN search—it is the interaction between vector ordering and scalar filter predicates. This problem is well-known in the dedicated vector database community but tends to catch PostgreSQL DBAs off guard because it violates normal index selectivity reasoning.

    Why the problem occurs

    Consider this query:

    SELECT id, content
FROM documents
WHERE tenant_id = 42
ORDER BY embedding <=> $1
LIMIT 10;

    The planner has two realistic strategies:

    1. Apply the WHERE tenant_id = 42 filter first (using an index on tenant_id), then do an exact brute-force vector search over the filtered rows.

    2. Use the HNSW or IVFFlat index to find approximate nearest neighbors across all rows, then discard results that don't match tenant_id = 42.

    Strategy 1 is correct but PostgreSQL's vector indexes don't support index-level filtering—the filter is applied as a post-scan recheck. Strategy 2 can return far fewer than LIMIT 10 results if most ANN candidates don't pass the scalar filter, causing PostgreSQL to repeatedly widen its search, which dramatically inflates execution time.

    Diagnosing with EXPLAIN

    -- PostgreSQL / pgvector
-- Demonstrate the filter-interaction problem
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT id, content, embedding <=> $1 AS distance
FROM documents
WHERE tenant_id = 42
  AND metadata->>'language' = 'en'
ORDER BY embedding <=> $1
LIMIT 10;

-- If you see "Rows Removed by Filter" that is large relative to
-- actual rows returned, the index is scanning far more candidates
-- than needed. Example problematic output:
--   Index Scan using documents_embedding_hnsw_idx on documents
--     (actual rows=10, loops=1)
--     Filter: ((tenant_id = 42) AND (...))
--     Rows Removed by Filter: 3840   <-- this is the signal

    A Rows Removed by Filter count that is 10x or more the LIMIT value is the key diagnostic signal. For HNSW, this means ef_search is being exhausted without satisfying the filter, and PostgreSQL is doing many graph traversal iterations.

    Mitigation patterns

    -- PostgreSQL / pgvector
-- Pattern 1: Partial index per tenant (effective for small tenant counts)
CREATE INDEX documents_tenant42_hnsw_idx
ON documents
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64)
WHERE tenant_id = 42;

-- Pattern 2: Increase ef_search for filtered queries only
-- (session-level, not persistent)
SET hnsw.ef_search = 200;  -- default is 40; increase until filter hit rate improves
SELECT id, content
FROM documents
WHERE tenant_id = 42
ORDER BY embedding <=> $1
LIMIT 10;
RESET hnsw.ef_search;

-- Pattern 3: Pre-filter with CTE, then vector-rank within result
-- Effective when the scalar filter is highly selective (< 5% of rows)
WITH tenant_docs AS (
    SELECT id, content, embedding
    FROM documents
    WHERE tenant_id = 42
      AND created_at > now() - interval '30 days'
)
SELECT id, content, embedding <=> $1 AS distance
FROM tenant_docs
ORDER BY embedding <=> $1
LIMIT 10;

-- Pattern 4: Check planner's estimated vs actual rows for the filter
-- to decide which strategy the planner chose
EXPLAIN (ANALYZE)
SELECT id
FROM documents
WHERE tenant_id = 42;
-- If estimated rows ~= actual rows, statistics are current;
-- if estimate is way off, the planner is making poor strategy decisions.
-- Run: ANALYZE documents;

    The right mitigation depends on filter selectivity and the number of distinct filter values. Partial indexes work well for a small number of high-traffic tenants. CTE pre-filtering works well when scalar filters remove more than 90% of the table. Raising ef_search is a stopgap for cases where recall is suffering due to filtered candidate exhaustion but you can't restructure data layout immediately.

    14.5 Production Observability: Metrics, Alerting, and Continuous Profiling

    Ad-hoc EXPLAIN runs tell you what a query did on a specific execution. Production observability tells you what your vector layer is doing continuously, across all query patterns, over time. For most PostgreSQL shops, this means wiring pgvector metrics into an existing Prometheus/Grafana stack or a managed monitoring service.

    Python-based latency profiling with psycopg2

    # Python
import time
import statistics
import psycopg2
import numpy as np
from typing import List, Tuple

def profile_vector_query(
    conn_string: str,
    query_vectors: List[np.ndarray],
    limit: int = 10,
    ef_search: int = 64,
    iterations: int = 100,
) -> dict:
    """
    Profile vector search latency over multiple query vectors.
    Returns p50, p95, p99 latencies and throughput.
    """
    conn = psycopg2.connect(conn_string)
    cur = conn.cursor()

    # Set HNSW search parameter for this session
    cur.execute(f"SET hnsw.ef_search = {ef_search}")

    latencies_ms: List[float] = []

    for i in range(iterations):
        vec = query_vectors[i % len(query_vectors)]
        vec_str = "[" + ",".join(str(round(float(v), 6)) for v in vec) + "]"

        sql = f"""
            SELECT id, embedding <=> %s::vector AS distance
            FROM documents
            ORDER BY embedding <=> %s::vector
            LIMIT %s
        """

        start = time.perf_counter()
        cur.execute(sql, (vec_str, vec_str, limit))
        rows = cur.fetchall()
        elapsed_ms = (time.perf_counter() - start) * 1000
        latencies_ms.append(elapsed_ms)

        # Verify we got the expected number of results
        if len(rows) < limit:
            print(f"Warning: iteration {i} returned only {len(rows)} results")

    cur.close()
    conn.close()

    sorted_latencies = sorted(latencies_ms)
    n = len(sorted_latencies)

    return {
        "ef_search": ef_search,
        "iterations": iterations,
        "p50_ms": round(sorted_latencies[int(n * 0.50)], 2),
        "p95_ms": round(sorted_latencies[int(n * 0.95)], 2),
        "p99_ms": round(sorted_latencies[int(n * 0.99)], 2),
        "mean_ms": round(statistics.mean(latencies_ms), 2),
        "max_ms": round(max(latencies_ms), 2),
        "qps": round(1000 / statistics.mean(latencies_ms), 1),
    }


# Usage: sweep ef_search values to find the latency-recall tradeoff point
if __name__ == "__main__":
    import os

    conn_str = os.environ["PGVECTOR_CONN"]

    # Generate synthetic query vectors (replace with real embedding samples)
    rng = np.random.default_rng(42)
    sample_queries = [
        rng.standard_normal(1536).astype(np.float32)
        for _ in range(20)
    ]
    # Normalize for cosine distance
    sample_queries = [
        v / np.linalg.norm(v) for v in sample_queries
    ]

    for ef in [20, 40, 64, 100, 150, 200]:
        result = profile_vector_query(
            conn_str, sample_queries, ef_search=ef, iterations=50
        )
        print(
            f"ef_search={ef:4d} | "
            f"p50={result['p50_ms']:6.1f}ms | "
            f"p95={result['p95_ms']:6.1f}ms | "
            f"p99={result['p99_ms']:6.1f}ms | "
            f"qps={result['qps']:6.1f}"
        )

    This sweep produces output like:

    ef_search=  20 | p50=  8.2ms | p95= 12.1ms | p99= 18.4ms | qps= 112.3
ef_search=  40 | p50= 14.6ms | p95= 19.8ms | p99= 28.7ms | qps=  65.4
ef_search=  64 | p50= 22.3ms | p95= 31.2ms | p99= 44.9ms | qps=  43.1
ef_search= 100 | p50= 33.7ms | p95= 47.6ms | p99= 68.2ms | qps=  28.9
ef_search= 150 | p50= 49.2ms | p95= 69.3ms | p99= 98.1ms | qps=  19.8
ef_search= 200 | p50= 64.8ms | p95= 91.4ms | p99=131.2ms | qps=  15.1

    The inflection point where latency grows faster than linearly is where you're hitting diminishing returns. For most production workloads, ef_search=64 to ef_search=100 is the reasonable operating range.

    Continuous monitoring with pg_stat_activity and custom metrics

    -- PostgreSQL / pgvector
-- Active vector queries right now (useful during incident investigation)
SELECT
    pid,
    now() - pg_stat_activity.query_start AS duration,
    state,
    wait_event_type,
    wait_event,
    left(query, 200) AS query_preview
FROM pg_stat_activity
WHERE state != 'idle'
  AND (query ILIKE '%<=>%' OR query ILIKE '%<#>%')
ORDER BY duration DESC;

-- Long-running vector queries (more than 500ms)
SELECT
    pid,
    round(extract(epoch FROM (now() - query_start)) * 1000) AS duration_ms,
    state,
    wait_event_type || ': ' || coalesce(wait_event, '') AS wait_detail,
    left(query, 300) AS query
FROM pg_stat_activity
WHERE state != 'idle'
  AND query ILIKE '%<=>%'
  AND (now() - query_start) > interval '500 milliseconds'
ORDER BY duration_ms DESC;

-- Aggregate: track query latency trend over time using a scheduled job
-- Store results in a monitoring table for trend analysis
CREATE TABLE IF NOT EXISTS vector_query_stats_snapshots (
    snapshot_time    timestamptz DEFAULT now(),
    query_fingerprint text,
    calls            bigint,
    mean_ms          numeric,
    p99_estimate_ms  numeric,
    total_ms         numeric
);

INSERT INTO vector_query_stats_snapshots
    (query_fingerprint, calls, mean_ms, total_ms)
SELECT
    md5(query)                    AS query_fingerprint,
    calls,
    round(mean_exec_time::numeric, 2),
    round(total_exec_time::numeric, 2)
FROM pg_stat_statements
WHERE query ILIKE '%<=>%'
ORDER BY total_exec_time DESC
LIMIT 50;

    Schedule that INSERT every 5 minutes via pg_cron or an external cron job to build a latency trend dataset you can plot in Grafana.
