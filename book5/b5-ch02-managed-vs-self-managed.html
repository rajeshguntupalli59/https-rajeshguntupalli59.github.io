Chapter 2: Managed vs Self-Managed: The Real Tradeoffs

Every DBA who moves to the cloud arrives with the same question: what exactly am I giving up, and what am I getting in return? The marketing answer is "you get managed infrastructure." The production answer is more nuanced. Managed database services abstract away the operating system, storage subsystem, replication mechanics, and backup scheduling — but they do not abstract away the need to understand them. This chapter maps the specific boundaries between what AWS RDS, Azure SQL, and Google Cloud SQL control versus what you still own, so you can make architecture decisions with your eyes open rather than discovering limits during an incident at 2 a.m.

---

What "Managed" Actually Means at the Infrastructure Layer

When you ran PostgreSQL on bare metal or VMware, your responsibility surface looked like this: hardware procurement, OS patching, filesystem layout, network configuration, PostgreSQL installation, parameter tuning, backup scheduling, replication setup, failover scripting, and monitoring. In a managed service, roughly the bottom half of that stack disappears — but it disappears in specific ways that matter to how you work.

AWS RDS runs your database inside an EC2 instance that AWS owns. You never SSH into it. The OS is there — Amazon Linux 2 or a variant depending on the engine — but it is not your OS. You cannot install extensions that are not on the approved list, you cannot modify pg_hba.conf directly, and you cannot change the PostgreSQL binary. What you can do is control almost everything above that layer: parameter groups, option groups, subnet groups, security groups, IAM authentication, automated backup retention up to 35 days, and Multi-AZ deployment.

Azure SQL Database is further abstracted than RDS. Microsoft runs SQL Server inside a service fabric that you have zero visibility into. There is no concept of an underlying VM in the standard tiers. You configure DTUs or vCores, storage limits, firewall rules, Active Directory authentication, and Transparent Data Encryption. The SQL Server instance itself — its memory settings, its tempdb layout, its trace flags — is managed by Microsoft according to tier.

Google Cloud SQL sits closer to the RDS model than to Azure SQL Database. It runs PostgreSQL, MySQL, or SQL Server on Compute Engine VMs that Google manages. You get database flags (the equivalent of postgresql.conf parameters), authorized networks, private IP configuration, and automated backups. Like RDS, the OS is not yours.

The practical consequence for an experienced DBA: your tuning vocabulary shifts from OS-level and filesystem-level settings toward database-level configuration objects — parameter groups, database flags, elastic pool settings — and cloud-native controls like IAM policies and VPC routing.

---

The Access Boundary: What You Can and Cannot Touch

This is where most on-prem DBAs first hit friction. The following table summarizes the access boundaries that matter most in production:

| Capability | Self-Managed PostgreSQL | AWS RDS PostgreSQL | Azure SQL | Cloud SQL PostgreSQL |

|---|---|---|---|---|

| OS access | Full root | None | None | None |

| Install arbitrary extensions | Yes | No (approved list only) | No | No (approved list) |

| Modify pg_hba.conf | Yes | No (use IAM/security groups) | N/A | No (use authorized networks) |

| Access WAL files directly | Yes | No (can stream via logical replication) | N/A | No |

| pg_basebackup to external target | Yes | No (use native backup export to S3) |N/A | No (use export to GCS) |

| Custom superuser access | Full | rds_superuser role (limited) | db_owner (limited) | cloudsqlsuperuser (limited) |

| Set max_connections freely | Yes | Yes (via parameter group) | Tier-dependent ceiling | Yes (via database flags) |

| Tablespace on custom paths | Yes | No | No | No |

The rds_superuser point deserves emphasis. On RDS PostgreSQL, you receive a user with the rds_superuser role. This gives you the ability to manage other users, replication slots, logical replication, and most administrative functions. But it does not give you pg_read_all_settings on internal parameters, and it cannot create C-language extensions. If you have built any tooling that depends on superuser-level OS interaction — for example, a custom archiving script that writes directly to the filesystem — that tooling requires redesign.

Azure SQL is a different situation entirely. The concept of a SQL Server instance owner with sysadmin rights does not exist in Azure SQL Database. The closest equivalent is db_owner on individual databases, plus server-level roles like loginmanager and dbmanager on the logical server. If your on-prem SQL Server processes rely on SQL Server Agent jobs with xp_cmdshell, those processes need to be rebuilt as Azure Functions, Azure Automation runbooks, or Elastic Jobs.

Parameter Tuning: From postgresql.conf to Parameter Groups

On a self-managed PostgreSQL instance, tuning work_mem, shared_buffers, checkpoint_completion_target, and effective_cache_size means editing a file and reloading or restarting. In RDS, that workflow translates to parameter groups.

A parameter group in RDS is a named configuration object that you attach to one or more DB instances. AWS provides default parameter groups (read-only) for each engine version. You create a custom parameter group, modify the parameters you need, and associate it with your instance. Parameter changes fall into two categories: static (require a reboot) and dynamic (apply immediately). Knowing which is which before you plan a change window matters.

# Create a custom RDS parameter group for PostgreSQL 15
aws rds create-db-parameter-group \
  --db-parameter-group-name prod-pg15-custom \
  --db-parameter-group-family postgres15 \
  --description "Production PostgreSQL 15 parameter group" \
  --region us-east-1

# Modify key parameters
aws rds modify-db-parameter-group \
  --db-parameter-group-name prod-pg15-custom \
  --parameters \
    "ParameterName=work_mem,ParameterValue=65536,ApplyMethod=immediate" \
    "ParameterName=checkpoint_completion_target,ParameterValue=0.9,ApplyMethod=immediate" \
    "ParameterName=max_connections,ParameterValue=500,ApplyMethod=pending-reboot" \
    "ParameterName=log_min_duration_statement,ParameterValue=1000,ApplyMethod=immediate" \
  --region us-east-1

# Apply the parameter group to an existing instance
aws rds modify-db-instance \
  --db-instance-identifier prod-pg-primary \
  --db-parameter-group-name prod-pg15-custom \
  --apply-immediately \
  --region us-east-1

On Google Cloud SQL, the equivalent concept is database flags. The syntax and application mechanism differ, but the intent is the same. Note that Cloud SQL does not support all PostgreSQL parameters — some low-level settings are fixed by Google.

# Set database flags on a Cloud SQL PostgreSQL instance
gcloud sql instances patch prod-pg-instance \
  --database-flags \
    max_connections=500,\
    work_mem=65536,\
    checkpoint_completion_target=0.9,\
    log_min_duration_statement=1000 \
  --project my-gcp-project

# Verify current flags
gcloud sql instances describe prod-pg-instance \
  --project my-gcp-project \
  --format="json(settings.databaseFlags)"

On Azure SQL, the tuning model is fundamentally different. You do not set max_connections because connection limits are determined by the service tier. For an 8-vCore General Purpose instance, Azure SQL allows approximately 800 concurrent connections. For Business Critical, limits are higher. You tune at the query level using Query Store, Automatic Plan Correction, and resource governor equivalents — but you do not manage server memory allocation directly. Azure SQL's Resource Governor equivalent is elastic pool resource governance, and it operates at the pool level rather than the instance level.

One real gotcha with RDS parameter groups: if you create a Multi-AZ deployment and modify a static parameter, the pending reboot applies to both primary and standby simultaneously during the next maintenance window — or immediately if you force it. This means you need to treat static parameter changes with the same care as a planned failover, because that is effectively what AWS does to apply them.

High Availability: What You Delegate and What You Still Own

In a self-managed PostgreSQL environment, high availability required you to configure streaming replication, set primary_conninfo, manage recovery.conf (or standby.signal in PostgreSQL 12+), build a failover mechanism with Patroni, Repmgr, or a custom script, and test it regularly. That is a significant operational surface.

RDS Multi-AZ reduces that surface dramatically. When you enable Multi-AZ on an RDS PostgreSQL instance, AWS provisions a synchronous standby in a different Availability Zone, manages the replication, monitors the primary, and executes automatic failover — typically completing within 60 to 120 seconds for standard RDS. You do not configure any of this. The connection string uses the RDS endpoint, which automatically resolves to whichever instance is currently primary.

What you still own: application reconnection logic. RDS failover changes the underlying IP that the DNS endpoint points to. DNS TTL for RDS endpoints is 5 seconds, but client-side DNS caching in the JVM, in certain connection poolers, and in some OS-level resolvers can hold stale entries for minutes. If your application opens a connection and holds it across the failover boundary, that connection will break and must be reestablished. This is not RDS failing to do its job — it is the boundary between the managed service and the application tier.

For Aurora, the HA story changes again. Aurora separates compute from storage. The Aurora storage layer replicates 6 copies of data across 3 Availability Zones synchronously before acknowledging a write. Failover from a primary to a read replica in Aurora typically completes in under 30 seconds — faster than standard RDS Multi-AZ — because there is no data volume to synchronize; the replicas already share the storage layer.

# Python example: RDS-aware connection handling with retry logic
# This pattern is necessary because RDS/Aurora failover breaks existing connections

import boto3
import psycopg2
import time
import logging
from psycopg2 import OperationalError

logger = logging.getLogger(__name__)

def get_rds_connection(host, port, dbname, user, password,
                        max_retries=5, retry_delay=2):
    """
    Establish a connection to RDS with retry logic suitable for
    Multi-AZ failover scenarios. On failover, DNS propagation
    may take 5-30 seconds; this loop handles the gap.
    """
    attempt = 0
    last_error = None

    while attempt < max_retries:
        try:
            conn = psycopg2.connect(
                host=host,
                port=port,
                dbname=dbname,
                user=user,
                password=password,
                connect_timeout=5,
                # options=-c statement_timeout=30000 for 30s query timeout
                options="-c statement_timeout=30000"
            )
            conn.set_session(autocommit=False)
            logger.info(f"Connected to RDS at {host} on attempt {attempt + 1}")
            return conn

        except OperationalError as e:
            last_error = e
            attempt += 1
            logger.warning(
                f"RDS connection attempt {attempt} failed: {e}. "
                f"Retrying in {retry_delay}s..."
            )
            time.sleep(retry_delay)
            retry_delay = min(retry_delay * 2, 30)  # exponential backoff, cap at 30s

    raise RuntimeError(
        f"Could not connect to RDS after {max_retries} attempts. "
        f"Last error: {last_error}"
    )


def execute_with_reconnect(host, port, dbname, user, password, query, params=None):
    """
    Execute a query and handle connection failures that can occur
    mid-session during failover events.
    """
    conn = get_rds_connection(host, port, dbname, user, password)
    try:
        with conn.cursor() as cur:
            cur.execute(query, params)
            conn.commit()
            return cur.fetchall() if cur.description else None
    except OperationalError as e:
        logger.error(f"Query failed, connection likely lost during failover: {e}")
        conn.close()
        # Re-raise so the caller can decide whether to retry at the business logic level
        raise
    finally:
        if not conn.closed:
            conn.close()

Azure SQL Business Critical tier uses an Always On Availability Group under the hood, with a hidden read-only secondary exposed via the ApplicationIntent=ReadOnly connection string parameter. Failover SLA for Business Critical is 99.99%. For General Purpose tier, Azure uses remote storage (Premium SSD) and failover involves attaching that storage to a new compute node — typically 20 to 30 seconds but potentially longer under high storage load.

For Cloud SQL, high availability within a region uses a standby instance with synchronous replication. Cross-region read replicas can be promoted manually but do not participate in automatic failover. For multi-region automatic failover in Google Cloud, the answer is AlloyDB or Cloud Spanner, not Cloud SQL.

Backup, Restore, and the Point-in-Time Recovery Model

On a self-managed PostgreSQL instance, your backup strategy involved pg_basebackup for full backups, WAL archiving to a shared storage target, and periodic validation through pg_restore test runs. You controlled the retention schedule, the storage location, and the restore procedure.

In RDS, automated backups are configured with a retention period (1 to 35 days) and a backup window. RDS takes daily snapshots and continuously archives transaction logs to S3 — storage you never see but that AWS uses to serve Point-in-Time Recovery (PITR). Restoring to a point in time always creates a new DB instance; you cannot overwrite the existing one. This is one of the most operationally significant differences from self-managed recovery. If you need to restore to 30 minutes ago, you restore to a new endpoint and redirect traffic, or you use the restored instance to extract specific data with pg_dump and load it into the production instance.

# Initiate a point-in-time restore on RDS to a new instance
aws rds restore-db-instance-to-point-in-time \
  --source-db-instance-identifier prod-pg-primary \
  --target-db-instance-identifier prod-pg-pitr-restore-20240115 \
  --restore-time 2024-01-15T14:30:00Z \
  --db-instance-class db.r6g.2xlarge \
  --db-subnet-group-name prod-subnet-group \
  --vpc-security-group-ids sg-0abc123def456789 \
  --no-publicly-accessible \
  --region us-east-1

# Monitor the restore status
aws rds describe-db-instances \
  --db-instance-identifier prod-pg-pitr-restore-20240115 \
  --query 'DBInstances[0].{Status:DBInstanceStatus,Endpoint:Endpoint.Address}' \
  --region us-east-1

The storage costs for automated backups in RDS are worth understanding before you enable 35-day retention on large instances. AWS provides backup storage equal to the total provisioned storage of your RDS instances at no additional charge. Storage beyond that is billed at $0.095 per GB-month in us-east-1. For a 10 TB RDS instance with 35-day retention and high write volume, backup storage costs can exceed the compute costs of the instance itself. Audit your backup retention policies as deliberately as you would on-prem.

Azure SQL automated backups use a similar model: full weekly, differential daily, transaction log every 5 to 12 minutes. Retention for Azure SQL Database is 1 to 35 days for PITR, with long-term retention (LTR) available for up to 10 years. PITR restores, like RDS, create a new database — they do not overwrite the source.

One genuine improvement in managed services over self-managed setups for most teams: backup validation is handled for you. AWS verifies that automated snapshots are restorable. On-prem shops frequently discover corrupted backup chains during actual recovery events. That said, this does not replace the need to periodically test restore procedures yourself — automated backup verification confirms the backup file is intact, not that your application functions correctly on the restored data.

Terraform Provisioning: Encoding the Decisions You Cannot Delegate

Infrastructure as Code is not optional in a production cloud database environment. The decisions you make at provisioning time — instance class, storage type, Multi-AZ enablement, encryption, parameter group, deletion protection — are significantly harder to change after the fact than on-prem settings. Terraform locks these decisions into version-controlled code and makes them reproducible.

# Terraform configuration for a production-grade RDS PostgreSQL instance
# This encodes the decisions that matter most at provisioning time.

terraform {
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }
}

# KMS key for RDS encryption at rest
resource "aws_kms_key" "rds_key" {
  description             = "RDS encryption key for production PostgreSQL"
  deletion_window_in_days = 30
  enable_key_rotation     = true

  tags = {
    Environment = "production"
    ManagedBy   = "terraform"
  }
}

# Custom parameter group for PostgreSQL 15
resource "aws_db_parameter_group" "pg15_prod" {
  name   = "prod-pg15-custom"
  family = "postgres15"
  description = "Production PostgreSQL 15 parameters"

  parameter {
    name         = "work_mem"
    value        = "65536"
    apply_method = "immediate"
  }

  parameter {
    name         = "checkpoint_completion_target"
    value        = "0.9"
    apply_method = "immediate"
  }

  parameter {
    name         = "log_min_duration_statement"
    value        = "1000"
    apply_method = "immediate"
  }

  parameter {
    name         = "log_connections"
    value        = "1"
    apply_method = "immediate"
  }

  parameter {
    name         = "shared_preload_libraries"
    value        = "pg_stat_statements,auto_explain"
    apply_method = "pending-reboot"
  }
}

# RDS subnet group
resource "aws_db_subnet_group" "prod" {
  name       = "prod-rds-subnet-group"
  subnet_ids = var.private_subnet_ids
  description = "Private subnets for RDS — no public route"

  tags = {
    Environment = "production"
  }
}

# Production RDS PostgreSQL instance
resource "aws_db_instance" "prod_pg" {
  identifier        = "prod-pg-primary"
  engine            = "postgres"
  engine_version    = "15.4"
  instance_class    = "db.r6g.2xlarge"  # 8 vCPU, 64 GB RAM

  # Storage: gp3 gives 3000 IOPS baseline, upgradeable independently of storage size
  allocated_storage     = 500
  max_allocated_storage = 2000  # autoscaling ceiling in GB
  storage_type          = "gp3"
  iops                  = 12000  # provisioned IOPS for gp3
  storage_throughput    = 500    # MB/s throughput for gp3
  storage_encrypted     = true
  kms_key_id            = aws_kms_key.rds_key.arn

  db_name  = "appdb"
  username = "dbadmin"
  # In production: use aws_secretsmanager_secret and reference here
  password = var.db_master_password

  # HA and networking
  multi_az               = true
  db_subnet_group_name   = aws_db_subnet_group.prod.name
  vpc_security_group_ids = [var.db_security_group_id]
  publicly_accessible    = false

  # Parameter group
  parameter_group_name = aws_db_parameter_group.pg15_prod.name

  # Backup configuration
  backup_retention_period   = 14
  backup_window             = "03:00-04:00"  # UTC
  maintenance_window        = "sun:04:30-sun:05:30"
  copy_tags_to_snapshot     = true
  delete_automated_backups  = false

  # Protection and logging
  deletion_protection      = true
  skip_final_snapshot      = false
  final_snapshot_identifier = "prod-pg-primary-final-snapshot"

  enabled_cloudwatch_logs_exports = [
    "postgresql",
    "upgrade"
  ]

  # Performance Insights for query-level visibility
  performance_insights_enabled          = true
  performance_insights_retention_period = 7  # days; 731 for long-term (additional cost)

  tags = {
    Environment = "production"
    Owner       = "database-team"
    ManagedBy   = "terraform"
  }
}

Two provisioning decisions in this template deserve attention for DBAs coming from on-prem.

First, storage_type = "gp3". On RDS, gp3 replaced gp2 as the default general-purpose storage type. gp2 tied IOPS to storage size (3 IOPS per GB, bursting to 3000 for volumes under 1 TB). gp3 provides 3000 baseline IOPS and 125 MB/s throughput regardless of volume size, with independent scaling up to 16,000 IOPS and 1000 MB/s. For any database workload where you are not running io1 or io2 (the provisioned IOPS types), gp3 is almost always the right choice and is typically 20% cheaper than equivalent gp2 configurations.

Second, max_allocated_storage. RDS Storage Autoscaling can expand your volume automatically when free space drops below 10% of allocated storage (or 5 GB, whichever is larger). Setting a ceiling with max_allocated_storage prevents runaway storage growth from a data incident — a bulk insert gone wrong, a logging misconfiguration — from producing a surprise billing event.

Key Takeaways

  • → Managed services eliminate OS, storage, and replication management, but replace those responsibilities with cloud-native configuration objects — parameter groups, database flags, security groups, IAM policies — that require the same deliberate attention as the files they replace.
  • → The `rds_superuser`, `cloudsqlsuperuser`, and Azure SQL `db_owner` roles are not equivalent to OS-level superuser access. Any automation, extension, or process that depends on superuser filesystem or binary access requires architectural redesign before it can run on a managed service.
  • → PITR restores on RDS, Azure SQL, and Cloud SQL always create a new database instance, not an in-place overwrite. Recovery procedures must account for the time to provision a new instance, redirect connections, and validate data — typically 15 to 45 minutes beyond the recovery point itself.
  • → Aurora's separated storage architecture makes it faster to fail over than standard RDS Multi-AZ (typically under 30 seconds versus 60 to 120 seconds) but introduces a different cost model and different read-scaling behavior through reader endpoints — understanding the storage layer is prerequisite to sizing Aurora correctly.
  • → Infrastructure as Code is not a convenience in cloud database environments — it is the mechanism through which provisioning-time decisions (encryption keys, storage type, Multi-AZ, deletion protection) become auditable, reproducible, and reviewable before a change reaches production.
