Chapter 33: Zero-Trust Security

The traditional security model for databases assumed that anything inside the network perimeter was trustworthy. Once a user or application cleared the firewall, they were given broad access to the database layer with minimal additional scrutiny. That model is dead. Modern infrastructure — cloud-hosted databases, containerized workloads, remote developers, third-party integrations, and microservices — has dissolved the concept of a trusted perimeter entirely. Zero-trust security replaces perimeter trust with a principle that has become the foundation of serious production security: *never trust, always verify*. This chapter walks through what zero-trust means specifically for PostgreSQL and SQL Server environments, how to implement it at every layer of the database stack, and how to build systems that are genuinely difficult to compromise even when individual components are.

---

The Zero-Trust Mindset for Database Environments

Zero-trust is not a product you buy or a feature you enable. It is an architectural philosophy that assumes breach is inevitable and builds controls accordingly. For database administrators, this translates into a set of working principles that govern how access is granted, authenticated, monitored, and revoked.

The first principle is *explicit verification*. Every connection to a database — regardless of where it originates — must be authenticated and authorized using the strongest mechanism available. A service account connecting from an internal Kubernetes pod is not implicitly trustworthy simply because it is inside the cluster. It must prove identity, and that identity must be checked against access policies every time, not cached indefinitely.

The second principle is *least privilege*. Users and applications receive only the permissions they need to perform their specific function, and nothing more. A reporting service that reads from three tables has no business holding INSERT or DELETE privileges anywhere. An application user has no business seeing the database schema, the pg_stat_activity view, or the system catalog tables beyond what it explicitly needs.

The third principle is *assume breach*. Security controls should not be designed with the goal of preventing all unauthorized access — they must also detect and contain unauthorized access when it occurs. This means comprehensive audit logging, anomaly detection, and clear response procedures for when something goes wrong.

These principles sound intuitive, but they require deliberate engineering to implement. Most database environments in production today are over-permissioned, under-audited, and built on the implicit trust that the network provides a meaningful boundary. Rebuilding on zero-trust foundations means revisiting authentication, authorization, network controls, secrets management, and monitoring — often simultaneously.

---

Authentication: Proving Identity at Every Layer

Authentication is the gateway to everything else. Before any privilege question is asked, the database must know — with certainty — who is asking. Both PostgreSQL and SQL Server offer multiple authentication mechanisms, and the choice matters enormously in a zero-trust context.

PostgreSQL Authentication

PostgreSQL's authentication is controlled through the pg_hba.conf file, which defines rules for how each connection is authenticated based on connection type, database, user, and source address. In a zero-trust environment, the configuration should lean heavily toward certificate-based authentication (via cert method) or SCRAM-SHA-256 rather than plaintext passwords (password) or the insecure md5 method.

A properly locked-down pg_hba.conf for a production environment looks like this:

# Reject all by default
# local connections for the postgres superuser only, using peer
local   all             postgres                                peer

# Application accounts via SSL with certificate validation
hostssl appdb           appuser         10.0.10.0/24            scram-sha-256

# Reject everything else
host    all             all             0.0.0.0/0               reject

The hostssl method requires that the connection use TLS — without it, the connection is rejected before a password is even asked for. Combined with SCRAM-SHA-256, this means passwords are never sent in plaintext and are verified using a challenge-response mechanism that prevents replay attacks.

For service accounts in automated systems, certificate authentication eliminates passwords entirely. The database validates the client certificate's CN (Common Name) against the username being claimed:

hostssl appdb           etl_service     10.0.20.0/24            cert clientcert=verify-full

SQL Server Authentication

SQL Server supports Windows Authentication (Kerberos/NTLM under the hood) and SQL Server Authentication (username and password). In a zero-trust context, Windows Authentication integrated with Active Directory is strongly preferred for human accounts because it uses Kerberos tickets, supports multi-factor authentication flows, and integrates with group policy for password complexity and rotation.

For service accounts in cloud or hybrid environments, Azure Active Directory authentication extends this model to cloud-native workloads. SQL authentication should be disabled entirely where the architecture allows it.

You can verify which authentication mode is active:

PostgreSQL:

-- Check pg_hba.conf rules in effect (PostgreSQL 10+)
SELECT type, database, user_name, address, auth_method
FROM pg_hba_file_rules
ORDER BY line_number;

SQL Server:

-- Check server authentication mode (1 = Windows only, 2 = Mixed)
SELECT SERVERPROPERTY('IsIntegratedSecurityOnly') AS windows_auth_only,
       name,
       create_date,
       modify_date,
       is_disabled
FROM sys.server_principals
WHERE type_desc IN ('SQL_LOGIN', 'WINDOWS_LOGIN', 'EXTERNAL_LOGIN')
ORDER BY type_desc, name;

Multi-factor authentication (MFA) deserves specific mention here. For human DBA accounts connecting to production databases, MFA is non-negotiable in a zero-trust model. PostgreSQL does not implement MFA natively, but it can be layered in through PAM (Pluggable Authentication Modules) on Linux, which allows integration with RADIUS-based MFA systems, Duo Security, or TOTP mechanisms. SQL Server integrates with Azure AD Conditional Access policies that enforce MFA before a token is issued, which means MFA enforcement happens at the identity provider layer rather than at the database layer — the right place for it.

Authorization: Least Privilege Implemented Precisely

Authentication tells the database who is connecting. Authorization tells the database what they can do. In most production environments, authorization is where zero-trust breaks down — not because the tools are lacking, but because privilege accumulates over time without systematic review.

Role Hierarchies and Ownership

Both PostgreSQL and SQL Server implement role-based access control (RBAC). The key discipline is to build a role hierarchy that reflects job functions, assign permissions to roles rather than individual users, and assign users to roles — never grant privileges directly to user accounts.

In PostgreSQL, roles and users are unified concepts. A well-structured role model for an application database looks like this:

PostgreSQL:

-- Create functional roles
CREATE ROLE app_readonly NOLOGIN;
CREATE ROLE app_readwrite NOLOGIN;
CREATE ROLE app_admin NOLOGIN;

-- Grant schema usage
GRANT USAGE ON SCHEMA public TO app_readonly, app_readwrite, app_admin;

-- Read-only: SELECT only on all current tables
GRANT SELECT ON ALL TABLES IN SCHEMA public TO app_readonly;

-- Read-write: SELECT, INSERT, UPDATE, DELETE
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_readwrite;
GRANT USAGE, SELECT ON ALL SEQUENCES IN SCHEMA public TO app_readwrite;

-- Ensure future tables inherit the same grants
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT ON TABLES TO app_readonly;

ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_readwrite;

-- Assign the application login to the appropriate role
GRANT app_readwrite TO appuser;

SQL Server:

-- Create database roles
CREATE ROLE db_app_readonly;
CREATE ROLE db_app_readwrite;
CREATE ROLE db_app_admin;

-- Grant schema-level permissions
GRANT SELECT ON SCHEMA::dbo TO db_app_readonly;
GRANT SELECT, INSERT, UPDATE, DELETE ON SCHEMA::dbo TO db_app_readwrite;

-- Add users to roles
ALTER ROLE db_app_readwrite ADD MEMBER appuser;

-- Verify effective permissions for a user
EXECUTE AS USER = 'appuser';
SELECT * FROM fn_my_permissions(NULL, 'DATABASE') ORDER BY permission_name;
REVERT;

Column-Level and Row-Level Security

Least privilege does not stop at the table boundary. In many systems, different users need to see different columns or different rows within the same table. Both databases provide mechanisms for this.

PostgreSQL's Row-Level Security (RLS) allows you to define policies that filter rows based on the current user context. This is particularly powerful for multi-tenant SaaS applications where a single table holds data for multiple customers:

PostgreSQL:

-- Enable RLS on the orders table
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;

-- Policy: users can only see their own tenant's orders
CREATE POLICY tenant_isolation ON orders
  USING (tenant_id = current_setting('app.current_tenant_id')::integer);

-- The application sets the tenant context on each connection
-- SET LOCAL app.current_tenant_id = '42';

-- Verify RLS is active
SELECT tablename, rowsecurity, forcerowsecurity
FROM pg_tables
WHERE tablename = 'orders';

SQL Server:

-- SQL Server Row-Level Security
CREATE SCHEMA Security;
GO

-- Inline table-valued function as the RLS predicate
CREATE FUNCTION Security.fn_tenant_predicate(@tenant_id INT)
RETURNS TABLE
WITH SCHEMABINDING
AS
RETURN SELECT 1 AS result
WHERE @tenant_id = CAST(SESSION_CONTEXT(N'TenantId') AS INT)
   OR IS_MEMBER('db_owner') = 1;
GO

-- Bind the filter policy to the orders table
CREATE SECURITY POLICY TenantFilter
ADD FILTER PREDICATE Security.fn_tenant_predicate(tenant_id) ON dbo.orders
WITH (STATE = ON);
GO

-- Application sets context before querying
-- EXEC sp_set_session_context N'TenantId', 42;

Privilege Auditing

Privileges accumulate silently. A developer is given temporary access to diagnose a production issue and the access is never revoked. An application role picks up permissions from a shared schema that expanded over time. A service account created three years ago still has DBA rights that made sense in development but were never trimmed for production. Regular privilege audits are a core operational practice in zero-trust environments.

PostgreSQL:

-- Identify superusers and roles with elevated privileges
SELECT rolname,
       rolsuper,
       rolcreaterole,
       rolcreatedb,
       rolbypassrls,
       rolconnlimit
FROM pg_roles
WHERE rolsuper = true
   OR rolcreaterole = true
   OR rolbypassrls = true
ORDER BY rolname;

SQL Server:

-- Identify server-level elevated principals
SELECT sp.name AS principal_name,
       sp.type_desc,
       spm.permission_name,
       spm.state_desc
FROM sys.server_permissions spm
JOIN sys.server_principals sp ON spm.grantee_principal_id = sp.principal_id
WHERE spm.permission_name IN ('CONTROL SERVER', 'ALTER ANY LOGIN', 
                               'VIEW SERVER STATE', 'SHUTDOWN')
  AND sp.type_desc NOT IN ('SERVER_ROLE')
ORDER BY sp.name;

Network Controls: Encryption and Segmentation

Zero-trust architecture treats the network as hostile. Data in transit between applications and the database must be encrypted, even on internal networks. Lateral movement — where an attacker who has compromised one service can freely reach other services on the same network — must be constrained.

TLS Everywhere

PostgreSQL enforces TLS through ssl = on in postgresql.conf and the use of hostssl entries in pg_hba.conf. Setting ssl_min_protocol_version = 'TLSv1.2' in postgresql.conf ensures that outdated protocol versions like TLS 1.0 and TLS 1.1 are rejected. TLS 1.3 is recommended where client library support allows.

The SSL certificate configuration requires a server certificate, private key, and root CA certificate. For mutual TLS (mTLS), clients also present a certificate, and the database validates it against the CA. mTLS is the strongest option for service-to-database authentication because it eliminates passwords from the authentication flow entirely.

For SQL Server, TLS is configured through SQL Server Configuration Manager, where a server certificate is bound to the SQL Server instance. The Force Encryption setting in SQL Server Configuration Manager ensures that clients must use encrypted connections or be rejected. SQL Server 2022 and Azure SQL Database support TLS 1.3.

Network Segmentation

At the infrastructure level, databases should not be directly reachable from the internet or from general application subnets. The principle is defense in depth: even if an application server is compromised, the attacker should not be able to make arbitrary connections to the database because network controls prevent it.

In practice, this means:

  • → Database servers live in a dedicated private subnet with no public IP addresses.
  • → Security groups or firewall rules allow inbound traffic only on the database port (5432 for PostgreSQL, 1433 for SQL Server) from specific application subnets or instance IDs.
  • → Egress from the database subnet is tightly restricted — databases should not be initiating outbound connections to arbitrary destinations.
  • → VPC endpoints or Private Link are used in cloud environments to ensure that traffic between application and database layers never transits the public internet, even if both are cloud-hosted.

    • Bastions or jump hosts for DBA access follow the same pattern. Direct SSH or RDP access to database servers should not be permitted from developer laptops. Instead, DBAs authenticate through a PAM (Privileged Access Management) system — CyberArk, HashiCorp Boundary, or AWS Systems Manager Session Manager — that brokers the session, records it, and enforces MFA before granting access.

    Secrets Management: Eliminating Static Credentials

    Static credentials — passwords hardcoded in configuration files, environment variables, or source code — are one of the most common and most exploitable weaknesses in database security. Zero-trust environments eliminate static credentials by integrating secrets management systems that provide dynamic, short-lived credentials that rotate automatically.

    Vault-Based Dynamic Credentials

    HashiCorp Vault's database secrets engine is the reference implementation for this pattern. Vault connects to PostgreSQL or SQL Server using a privileged account, and when an application requests credentials, Vault creates a new database user with the appropriate permissions, issues the credentials to the application with a TTL (time to live) of minutes or hours, and automatically drops the user when the TTL expires.

    The application never stores a password. It authenticates to Vault using its own identity (a Kubernetes service account token, an AWS IAM role, or a TLS certificate), receives short-lived credentials, uses them to connect to the database, and the credentials expire before an attacker who intercepts them could reuse them meaningfully.

    On the PostgreSQL side, Vault's management role needs permission to create and drop roles:

    PostgreSQL:

    -- Create a Vault management role
CREATE ROLE vault_manager LOGIN PASSWORD 'STRONG_MANAGED_PASSWORD';

-- Grant the ability to create roles and manage them
GRANT CREATEROLE TO vault_manager;

-- Vault will use a template like this to create dynamic credentials:
-- CREATE ROLE "v-app-XYZ123" LOGIN PASSWORD '...' 
--   VALID UNTIL '2024-01-01 12:30:00';
-- GRANT app_readwrite TO "v-app-XYZ123";

-- Verify active dynamic credentials (identify Vault-managed accounts)
SELECT rolname, rolvaliduntil, rolcanlogin
FROM pg_roles
WHERE rolname LIKE 'v-%'
ORDER BY rolvaliduntil;

    SQL Server:

    -- Create the Vault management login at the server level
CREATE LOGIN vault_manager WITH PASSWORD = 'StrongManagedPassword!';

-- Grant permissions to create logins (requires securityadmin role or explicit grants)
ALTER SERVER ROLE securityadmin ADD MEMBER vault_manager;

-- Check for short-lived dynamic accounts (Vault naming convention)
SELECT name,
       create_date,
       modify_date,
       is_disabled,
       LOGINPROPERTY(name, 'PasswordLastSetTime') AS pw_set_time
FROM sys.sql_logins
WHERE name LIKE 'v-%'
ORDER BY create_date DESC;

    Kubernetes and Cloud-Native Patterns

    In Kubernetes environments, the External Secrets Operator can sync secrets from AWS Secrets Manager, Azure Key Vault, or GCP Secret Manager into Kubernetes Secrets, while the secrets management system handles rotation. The database password that a pod uses is never baked into an image or a ConfigMap — it is fetched at startup from the secrets store and refreshed on a schedule.

    AWS IAM authentication for RDS PostgreSQL and Aurora takes this further by eliminating passwords entirely for IAM-aware applications. The application generates a short-lived authentication token using its IAM role's credentials and presents that token as the database password. The token expires after 15 minutes and is cryptographically tied to the IAM role — an attacker who intercepts it has a narrow window and cannot use it without the corresponding IAM role.

    Password rotation for traditional accounts should be automated through the secrets management system, not performed manually. Manual rotation processes are error-prone and inconsistent. When a rotation is needed — after a suspected compromise, after an employee departure, on a regular schedule — the secrets manager updates the credential in the database and propagates the new value to all consuming applications without human intervention.

    Audit Logging and Anomaly Detection

    Authentication, authorization, and network controls prevent unauthorized access. Audit logging detects it when those controls fail. In a zero-trust environment, audit logging is not optional and not retroactive — it is a live system that generates structured data consumed by detection tools.

    What to Log

    The minimum useful set of audit events for a database in a zero-trust architecture:

  • → All authentication attempts (success and failure), with source IP, username, and timestamp
  • → All privilege escalation events (GRANT, REVOKE, ALTER ROLE)
  • → All DDL statements (CREATE, ALTER, DROP) on production objects
  • → All access to sensitive tables (identified through a data classification process)
  • → All bulk data exports (large SELECT queries returning many rows, COPY operations)
  • → All connection and disconnection events for privileged accounts
  • → Failed authorization attempts (queries rejected due to insufficient privilege)

    • PostgreSQL Audit Logging

    PostgreSQL's built-in logging parameters capture many of these events, but pgaudit — an open-source extension — provides structured, role-aware audit logging that is designed specifically for compliance and security use cases.

    -- Install pgaudit (after loading the shared library in postgresql.conf)
-- shared_preload_libraries = 'pgaudit'

-- Configure audit logging at the session level
SET pgaudit.log = 'ddl, role, read, write';

-- Or configure globally in postgresql.conf:
-- pgaudit.log = 'ddl, role'
-- pgaudit.log_relation = on
-- pgaudit.log_parameter = on

-- Check current pgaudit configuration
SHOW pgaudit.log;
SHOW pgaudit.log_relation;

    PostgreSQL's standard logging parameters, independent of pgaudit, should also be configured:

    log_connections = on
log_disconnections = on
log_failed_connections = on
log_statement = 'ddl'         -- at minimum; 'all' for high-security environments
log_duration = on
log_min_duration_statement = 1000  -- log slow queries (over 1 second)
log_line_prefix = '%t [%p]: [%l-1] user=%u,db=%d,app=%a,client=%h '

    SQL Server Audit Logging

    SQL Server provides native audit capabilities through SQL Server Audit, which is configured through the SQL Server Audit object at the server level and Audit Specifications at both the server and database level.

    SQL Server:

    -- Create a server audit writing to a file
CREATE SERVER AUDIT ZeroTrustAudit
TO FILE (
    FILEPATH = 'D:\SQLAudit\',
    MAXSIZE = 100 MB,
    MAX_ROLLOVER_FILES = 10,
    RESERVE_DISK_SPACE = OFF
)
WITH (QUEUE_DELAY = 1000, ON_FAILURE = CONTINUE);
GO

ALTER SERVER AUDIT ZeroTrustAudit WITH (STATE = ON);
GO

-- Create a database audit specification for sensitive operations
CREATE DATABASE AUDIT SPECIFICATION AppDB_SensitiveAccess
FOR SERVER AUDIT ZeroTrustAudit
ADD (SELECT, INSERT, UPDATE, DELETE ON dbo.customers BY PUBLIC),
ADD (SCHEMA_OBJECT_ACCESS_GROUP),
ADD (DATABASE_ROLE_MEMBER_CHANGE_GROUP)
WITH (STATE = ON);
GO

-- Query the audit log
SELECT event_time,
       action_id,
       succeeded,
       server_principal_name,
       database_name,
       object_name,
       statement,
       client_ip
FROM sys.fn_get_audit_file('D:\SQLAudit\ZeroTrustAudit_*.sqlaudit', DEFAULT, DEFAULT)
WHERE event_time > DATEADD(HOUR, -24, GETUTCDATE())
ORDER BY event_time DESC;

    Behavioral Anomaly Detection

    Raw audit logs are necessary but not sufficient. A log file that nobody reads does not catch attacks in progress. The real operational value comes from shipping audit logs to a SIEM (Security Information and Event Management) platform — Splunk, Elastic Security, Microsoft Sentinel, or similar — and defining detection rules that alert on anomalous patterns.

    Patterns worth alerting on in a database security context:

  • → A service account that normally issues only SELECT queries suddenly issues a DROP TABLE or an UPDATE to a user credentials table
  • → A login that normally connects from a single IP address connecting from a new country or a Tor exit node
  • → A spike in failed login attempts against a specific account (credential stuffing or brute force)
  • → A large SELECT returning millions of rows from a table that normally serves paginated queries of tens of rows
  • → A connection at 3 AM from a developer account that normally operates during business hours
  • → A new superuser or sysadmin account appearing in the database without a corresponding change ticket

    • These detections require a baseline — a known-good behavioral profile built from historical log data — and threshold-based or machine-learning-based rules that identify deviations. Many cloud platforms (AWS GuardDuty for RDS, Microsoft Defender for SQL) provide managed anomaly detection that processes database activity logs automatically and generates findings without requiring custom rule development.

    ---

    Key Takeaways

  • → **Zero-trust replaces perimeter trust with continuous verification.** Every connection to the database must authenticate using strong mechanisms — SCRAM-SHA-256 or certificate-based authentication for PostgreSQL, Kerberos or Azure AD with MFA for SQL Server — regardless of where the connection originates.
  • → **Least privilege must be actively maintained, not just initially configured.** Role hierarchies should map to job functions, row-level and column-level security should restrict data access within tables, and privilege audits should run regularly to identify and remediate privilege creep before it becomes a liability.
  • → **The network is not a trust boundary.** TLS must be enforced for all database connections, mutual TLS is the strongest option for service accounts, and network segmentation should prevent databases from being directly reachable from anything other than explicitly authorized application tiers.
  • → **Static credentials are a fundamental vulnerability.** Dynamic secrets management through HashiCorp Vault, AWS IAM authentication, or equivalent systems eliminates long-lived passwords from the attack surface. Credentials that expire in minutes cannot be effectively exploited even if intercepted.
  • → **Audit logging generates the evidence; detection rules generate the alerts.** Structured audit logging through pgaudit or SQL Server Audit, shipped to a SIEM with behavioral detection rules, is what converts the zero-trust philosophy into operational security — the ability to detect, investigate, and respond to anomalies before they become breaches.
