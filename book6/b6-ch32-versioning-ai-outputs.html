Chapter 32: Versioning AI Outputs: Storing Model Generations

Every AI application eventually faces the same uncomfortable moment: something changed, and nobody can explain what. The model was swapped, a prompt was tweaked, a temperature setting drifted, and now outputs that users loved yesterday feel different today. Without a systematic approach to storing and versioning model generations, debugging these regressions is archaeology — sifting through logs, guessing at configurations, and hoping someone left a comment somewhere.

This chapter builds a complete versioning system for AI-generated outputs using PostgreSQL as the authoritative store. You will design schemas that capture not just the text a model produced, but the full generative context: model identifiers, prompt versions, sampling parameters, token usage, latency, embedding representations of outputs, and lineage chains that connect a generation to its ancestors. You will implement audit trails, diff capabilities, rollback logic, and comparison queries that let you answer questions like "how did GPT-4o's outputs change after our prompt refactor?" or "which model version produced the highest user-rated responses on this topic cluster?" PostgreSQL's JSONB, pgvector, row-level security, and logical replication make it uniquely suited for this work — not as a bolt-on logging system, but as the operational core of a generation lifecycle.

---

The Case for Storing Generations (Not Just Logs)

Most teams start by writing model outputs to application logs or a flat events table. This works until you need to answer a question that requires joining generations with prompts, comparing outputs across model versions, or clustering similar responses. Log files can not do joins. A flat events table collapses context you will later wish you had preserved.

The distinction between *logging* and *storing generations* is architectural. Logging answers "did this happen?" Storing generations answers "what was produced, under what conditions, and how does it compare to everything else we have produced?" The second question requires schema.

PostgreSQL earns its place here through several capabilities that vector-only stores and document databases do not provide:

Relational integrity between generations and their inputs. A generation is the product of a prompt, a model, a parameter set, and optionally a conversation thread. Foreign keys enforce that you cannot store a generation that references a prompt version that does not exist. This sounds obvious until you have a production system where prompt templates are deployed independently of application code, and a misconfigured deployment introduces orphan generations.

JSONB for heterogeneous generation metadata. Different model families return different metadata shapes. OpenAI returns usage, finish_reason, and system_fingerprint. Anthropic returns stop_reason, stop_sequence, and model-specific identifiers. Rather than normalizing every API response shape into a rigid schema, you store the raw API response in JSONB alongside normalized columns for the fields you query most. You get structure where you need it and flexibility where schemas diverge.

pgvector for semantic comparison. Embedding each generation's output text lets you query "find all past generations semantically similar to this new output" — which is how you detect when a model starts drifting, when two supposedly different prompts produce convergent outputs, or when a generation is suspiciously similar to training data.

Window functions and CTEs for lineage queries. When a user iterates on a generated document through multiple rounds of model refinement, the generation chain forms a tree. PostgreSQL's recursive CTEs can traverse that tree efficiently. No comparable feature exists in Redis or a dedicated vector store.

Schema Design: The Generation Store

The schema centers on four tables: prompt_versions, model_configs, generation_runs, and generation_outputs. A fifth table, generation_embeddings, keeps vector representations separated from text storage for indexing efficiency.

-- Enable required extensions
CREATE EXTENSION IF NOT EXISTS pgvector;
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS pg_trgm;

-- ─────────────────────────────────────────────
-- Prompt versioning: track every template change
-- ─────────────────────────────────────────────
CREATE TABLE prompt_versions (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    prompt_key      TEXT NOT NULL,           -- logical name, e.g. 'summarize_document'
    version         INTEGER NOT NULL,
    template        TEXT NOT NULL,
    variables       JSONB NOT NULL DEFAULT '[]',  -- declared variable names
    system_prompt   TEXT,
    author          TEXT NOT NULL,
    commit_hash     TEXT,                    -- git SHA if deployed via CI
    deployed_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deprecated_at   TIMESTAMPTZ,
    metadata        JSONB NOT NULL DEFAULT '{}',
    UNIQUE (prompt_key, version)
);

CREATE INDEX idx_prompt_versions_key_version
    ON prompt_versions (prompt_key, version DESC);

CREATE INDEX idx_prompt_versions_deployed
    ON prompt_versions (deployed_at DESC)
    WHERE deprecated_at IS NULL;

-- ─────────────────────────────────────────────
-- Model configs: every parameter set is versioned
-- ─────────────────────────────────────────────
CREATE TABLE model_configs (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    config_name     TEXT NOT NULL,
    provider        TEXT NOT NULL,           -- 'openai', 'anthropic', 'cohere', etc.
    model_id        TEXT NOT NULL,           -- 'gpt-4o', 'claude-3-5-sonnet-20241022'
    model_version   TEXT,                    -- provider's internal version string
    parameters      JSONB NOT NULL,          -- temperature, top_p, max_tokens, etc.
    is_active       BOOLEAN NOT NULL DEFAULT TRUE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deprecated_at   TIMESTAMPTZ,
    notes           TEXT,
    UNIQUE (config_name)
);

CREATE INDEX idx_model_configs_active
    ON model_configs (provider, model_id)
    WHERE is_active = TRUE;

-- ─────────────────────────────────────────────
-- Generation runs: one row per API call batch
-- A run groups one or more outputs from the same
-- invocation context (e.g., sampling N=3)
-- ─────────────────────────────────────────────
CREATE TABLE generation_runs (
    id                  UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    run_key             TEXT,                -- caller-assigned idempotency key
    prompt_version_id   UUID NOT NULL REFERENCES prompt_versions(id),
    model_config_id     UUID NOT NULL REFERENCES model_configs(id),
    rendered_prompt     TEXT NOT NULL,       -- the actual prompt after variable substitution
    rendered_system     TEXT,
    input_variables     JSONB NOT NULL DEFAULT '{}',
    conversation_id     UUID,               -- FK to conversation table if part of a thread
    parent_run_id       UUID REFERENCES generation_runs(id),  -- for iterative refinement
    requested_by        TEXT NOT NULL,       -- user_id or service_id
    session_id          TEXT,
    started_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    completed_at        TIMESTAMPTZ,
    status              TEXT NOT NULL DEFAULT 'pending'
                            CHECK (status IN ('pending','running','completed','failed','cancelled')),
    error_message       TEXT,
    client_metadata     JSONB NOT NULL DEFAULT '{}',
    UNIQUE (run_key) WHERE run_key IS NOT NULL
);

CREATE INDEX idx_generation_runs_prompt
    ON generation_runs (prompt_version_id, started_at DESC);

CREATE INDEX idx_generation_runs_model
    ON generation_runs (model_config_id, started_at DESC);

CREATE INDEX idx_generation_runs_conversation
    ON generation_runs (conversation_id)
    WHERE conversation_id IS NOT NULL;

CREATE INDEX idx_generation_runs_parent
    ON generation_runs (parent_run_id)
    WHERE parent_run_id IS NOT NULL;

CREATE INDEX idx_generation_runs_user
    ON generation_runs (requested_by, started_at DESC);

-- ─────────────────────────────────────────────
-- Generation outputs: one row per candidate output
-- A run with n=3 produces 3 output rows
-- ─────────────────────────────────────────────
CREATE TABLE generation_outputs (
    id                  UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    run_id              UUID NOT NULL REFERENCES generation_runs(id) ON DELETE CASCADE,
    sequence_index      INTEGER NOT NULL DEFAULT 0,  -- position within run (for n>1)
    output_text         TEXT NOT NULL,
    output_tokens       INTEGER,
    input_tokens        INTEGER,
    total_tokens        INTEGER,
    finish_reason       TEXT,               -- 'stop', 'length', 'content_filter', etc.
    latency_ms          INTEGER,
    raw_response        JSONB NOT NULL,     -- complete API response object
    selected            BOOLEAN,            -- was this output chosen by the application?
    user_rating         SMALLINT CHECK (user_rating BETWEEN 1 AND 5),
    user_feedback       TEXT,
    flagged             BOOLEAN NOT NULL DEFAULT FALSE,
    flagged_reason      TEXT,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (run_id, sequence_index)
);

CREATE INDEX idx_generation_outputs_run
    ON generation_outputs (run_id, sequence_index);

CREATE INDEX idx_generation_outputs_selected
    ON generation_outputs (run_id)
    WHERE selected = TRUE;

CREATE INDEX idx_generation_outputs_rated
    ON generation_outputs (user_rating, created_at DESC)
    WHERE user_rating IS NOT NULL;

-- Full-text search on output content
CREATE INDEX idx_generation_outputs_fts
    ON generation_outputs
    USING gin(to_tsvector('english', output_text));

-- ─────────────────────────────────────────────
-- Embeddings: separated for index efficiency
-- Store embedding of output_text for semantic search
-- ─────────────────────────────────────────────
CREATE TABLE generation_embeddings (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    output_id       UUID NOT NULL UNIQUE REFERENCES generation_outputs(id) ON DELETE CASCADE,
    embedding       VECTOR(1536),           -- text-embedding-3-small dimensions
    embedding_model TEXT NOT NULL DEFAULT 'text-embedding-3-small',
    embedded_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_generation_embeddings_vector
    ON generation_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- ─────────────────────────────────────────────
-- Audit log: immutable record of all mutations
-- Written via trigger, never updated
-- ─────────────────────────────────────────────
CREATE TABLE generation_audit_log (
    id              BIGSERIAL PRIMARY KEY,
    table_name      TEXT NOT NULL,
    row_id          UUID NOT NULL,
    operation       TEXT NOT NULL CHECK (operation IN ('INSERT', 'UPDATE', 'DELETE')),
    changed_by      TEXT,
    changed_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    old_data        JSONB,
    new_data        JSONB,
    delta           JSONB    -- computed diff for UPDATE operations
);

CREATE INDEX idx_audit_log_row
    ON generation_audit_log (table_name, row_id, changed_at DESC);

CREATE INDEX idx_audit_log_time
    ON generation_audit_log (changed_at DESC);

-- Audit trigger function
CREATE OR REPLACE FUNCTION fn_generation_audit()
RETURNS TRIGGER AS $$
DECLARE
    v_delta JSONB := '{}';
    v_key   TEXT;
BEGIN
    IF TG_OP = 'UPDATE' THEN
        -- Compute delta: only changed keys
        FOR v_key IN SELECT key FROM jsonb_each(to_jsonb(OLD)) LOOP
            IF (to_jsonb(OLD) -> v_key) IS DISTINCT FROM (to_jsonb(NEW) -> v_key) THEN
                v_delta := v_delta || jsonb_build_object(
                    v_key,
                    jsonb_build_object('from', to_jsonb(OLD) -> v_key,
                                       'to',   to_jsonb(NEW) -> v_key)
                );
            END IF;
        END LOOP;

        INSERT INTO generation_audit_log
            (table_name, row_id, operation, changed_by, old_data, new_data, delta)
        VALUES
            (TG_TABLE_NAME, OLD.id, 'UPDATE',
             current_setting('app.current_user', TRUE),
             to_jsonb(OLD), to_jsonb(NEW), v_delta);
        RETURN NEW;

    ELSIF TG_OP = 'DELETE' THEN
        INSERT INTO generation_audit_log
            (table_name, row_id, operation, changed_by, old_data)
        VALUES
            (TG_TABLE_NAME, OLD.id, 'DELETE',
             current_setting('app.current_user', TRUE),
             to_jsonb(OLD));
        RETURN OLD;

    ELSIF TG_OP = 'INSERT' THEN
        INSERT INTO generation_audit_log
            (table_name, row_id, operation, changed_by, new_data)
        VALUES
            (TG_TABLE_NAME, NEW.id, 'INSERT',
             current_setting('app.current_user', TRUE),
             to_jsonb(NEW));
        RETURN NEW;
    END IF;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- Attach audit triggers
CREATE TRIGGER trg_audit_prompt_versions
    AFTER INSERT OR UPDATE OR DELETE ON prompt_versions
    FOR EACH ROW EXECUTE FUNCTION fn_generation_audit();

CREATE TRIGGER trg_audit_generation_outputs
    AFTER INSERT OR UPDATE OR DELETE ON generation_outputs
    FOR EACH ROW EXECUTE FUNCTION fn_generation_audit();

This schema gives you full traceability from a specific output all the way back to the exact prompt template version and model configuration that produced it. The separation of generation_runs (the invocation context) from generation_outputs (the candidate texts) is deliberate — it mirrors how APIs like OpenAI handle n > 1 sampling and lets you store multiple candidates from a single call without duplicating the prompt or metadata.
