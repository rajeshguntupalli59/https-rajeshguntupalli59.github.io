Chapter 32: Graph and Vector: Combining Relationship and Semantic Search

Semantic search answers the question *what does this mean?* Graph traversal answers the question *how are these things connected?* Neither question alone is sufficient for many real-world AI applications. A knowledge graph without semantic understanding forces exact string matches and brittle schema design. A vector index without relationship awareness loses the structural context that determines whether two semantically similar documents are actually related, authored by the same person, part of the same regulatory framework, or connected through a chain of citations. This chapter explores how to combine graph and vector capabilities into a unified retrieval layer — covering the architectural patterns, the specific tools that support native graph-vector hybrid search, and the practical tradeoffs that determine when you need both versus when one is sufficient.

---

Why Graph Structure and Vector Similarity Serve Different Retrieval Needs

Start with a concrete example: a pharmaceutical knowledge base containing drug compounds, clinical trials, regulatory approvals, adverse event reports, and the researchers who authored each document. Pure vector search can find documents semantically similar to "cardiovascular side effects in elderly patients." But it cannot tell you that two apparently similar adverse event reports refer to different formulations of the same drug, or that the most semantically relevant paper was authored by a researcher whose institution's data has been retracted. That structural context lives in the graph.

Graph databases model entities and relationships as first-class citizens. In property graph terms, nodes hold attributes and edges hold typed, directional relationships with their own properties. Neo4j's Cypher, Amazon Neptune's Gremlin and SPARQL, and Weaviate's cross-references all express this structure. The query planner traverses the graph using index-free adjacency — following pointers directly from one node to its neighbors rather than performing joins across tables. This gives graph databases their characteristic performance on multi-hop traversal queries: O(depth × branching_factor) rather than O(n log n) for a join-based equivalent.

Vector databases answer a different kind of query. Given a query embedding, find the k nearest vectors in high-dimensional space. The performance envelope is defined by approximate nearest neighbor (ANN) algorithms — HNSW, IVF, ScaNN — that trade recall for speed. A well-tuned HNSW index on a 1536-dimensional embedding space with 10 million vectors returns 10 nearest neighbors in under 10ms at 99% recall. No graph traversal is involved; the structure being searched is a proximity graph of embeddings, not an entity-relationship graph of domain objects.

The architectural question is where these two search modes intersect. Three patterns emerge in production:

Pre-filter by graph, rank by vector. Traverse the graph to identify a candidate set of entity IDs, then perform vector similarity search over that restricted set. This uses graph traversal to answer structural questions ("all drugs approved by the FDA in the last three years") and vector search to rank within that set.

Post-filter by graph, retrieve by vector. Retrieve top-k semantically similar documents, then apply graph traversal to expand, filter, or re-rank based on structural relationships. A recommendation system might retrieve 50 semantically similar items, then use the user's purchase graph to remove already-purchased items and boost items purchased by structurally similar users.

Graph-guided vector traversal. Use graph relationships to guide which regions of the vector space to explore. This is the approach taken by graph-aware RAG systems that follow citation or entity linkage graphs to gather context before embedding retrieval.

Native Graph-Vector Databases: Weaviate, Neo4j Vector, and Amazon Neptune Analytics

Several databases now offer native support for both paradigms within a single query engine, eliminating the need for application-level orchestration between separate systems.

Weaviate was designed from the start around the idea that objects have both vector representations and references to other objects. Its schema system defines cross-references between classes, and its GraphQL query layer lets you traverse those references while filtering by vector similarity in the same request.

# Python
import weaviate
import weaviate.classes as wvc

client = weaviate.connect_to_local()

# Schema: Drug nodes reference ClinicalTrial nodes
# This query retrieves trials semantically similar to a query
# that are linked to a specific drug compound
query_text = "cardiovascular adverse events in elderly patients"

response = (
    client.collections.get("ClinicalTrial")
    .query.near_text(
        query=query_text,
        limit=10,
        return_metadata=wvc.query.MetadataQuery(distance=True),
        filters=wvc.query.Filter.by_ref("hasDrug").by_property("name").equal("Atorvastatin"),
    )
)

for obj in response.objects:
    print(f"Trial ID: {obj.properties['trial_id']}")
    print(f"Title: {obj.properties['title']}")
    print(f"Distance: {obj.metadata.distance:.4f}")
    print()

client.close()

The Filter.by_ref() call translates to a graph traversal: follow the hasDrug reference edge and filter by the target node's property. Weaviate executes this as a pre-filter before the vector search, meaning the ANN index is queried only over the subset of trials linked to Atorvastatin. At scale this matters: if 10% of your 10M trial documents are linked to the target drug, you are running ANN over 1M vectors rather than 10M, which is roughly a 10x improvement in query latency with no change in recall.

Neo4j added native vector indexing in version 5.11, allowing a CALL db.index.vector.queryNodes() procedure alongside standard Cypher graph traversal. This enables a query that first does vector retrieval and then traverses the graph from the results.

# Python - Neo4j combined graph + vector query
from neo4j import GraphDatabase
from openai import OpenAI

openai_client = OpenAI()
driver = GraphDatabase.driver("bolt://localhost:7687", auth=("neo4j", "password"))

def get_embedding(text: str) -> list[float]:
    response = openai_client.embeddings.create(
        input=text,
        model="text-embedding-3-small"
    )
    return response.data[0].embedding

def graph_vector_search(query: str, hop_depth: int = 2, top_k: int = 5):
    embedding = get_embedding(query)

    cypher = """
    // Step 1: Vector similarity search to find seed nodes
    CALL db.index.vector.queryNodes('document_embeddings', $top_k, $embedding)
    YIELD node AS doc, score AS similarity_score

    // Step 2: Traverse the graph from each seed node
    // Follow AUTHORED_BY and CITES relationships up to hop_depth
    CALL apoc.path.subgraphNodes(doc, {
        relationshipFilter: 'AUTHORED_BY>|CITES>',
        maxLevel: $hop_depth
    })
    YIELD node AS related

    // Step 3: Collect and return with structural context
    WITH doc, similarity_score,
         collect(DISTINCT related) AS graph_neighbors
    RETURN
        doc.title AS title,
        doc.document_id AS doc_id,
        similarity_score,
        [n IN graph_neighbors WHERE n:Author | n.name] AS authors,
        [n IN graph_neighbors WHERE n:Document | n.title] AS cited_docs,
        size(graph_neighbors) AS graph_connectivity
    ORDER BY similarity_score DESC
    """

    with driver.session() as session:
        result = session.run(
            cypher,
            embedding=embedding,
            top_k=top_k,
            hop_depth=hop_depth
        )
        return [record.data() for record in result]

results = graph_vector_search(
    "machine learning approaches for drug interaction prediction",
    hop_depth=2,
    top_k=5
)

for r in results:
    print(f"Title: {r['title']}")
    print(f"Similarity: {r['similarity_score']:.4f}")
    print(f"Authors: {', '.join(r['authors'])}")
    print(f"Cites: {len(r['cited_docs'])} documents")
    print(f"Graph Connectivity: {r['graph_connectivity']} related nodes")
    print()

driver.close()

The graph_connectivity metric here is not just informational. Documents with high graph connectivity in a citation network are generally more influential and more likely to represent canonical sources rather than derivative work. This is a structural signal that pure vector similarity cannot capture.

Amazon Neptune Analytics (not Neptune Database) introduced vector similarity search in 2024, enabling openCypher queries that combine graph traversal with approximate nearest neighbor search over stored embeddings. The design pattern follows the same logic: use Neptune's property graph for structural filtering, Neptune Analytics' vector index for semantic ranking.

pgvector Meets Recursive CTEs: Graph-Vector Hybrid in PostgreSQL

If your stack is already on PostgreSQL, you may not need a dedicated graph database for moderate-complexity relationship modeling. PostgreSQL's recursive CTEs provide graph traversal over adjacency tables, and pgvector handles the vector similarity layer. The combination is not as expressive as Cypher or Gremlin for deep traversal, but for depth-limited graph queries it is entirely production-viable and keeps your stack consolidated.

-- PostgreSQL / pgvector
-- Schema: documents with embeddings, plus a relationship graph

CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE documents (
    doc_id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title       TEXT NOT NULL,
    content     TEXT,
    embedding   vector(1536),
    created_at  TIMESTAMPTZ DEFAULT now()
);

CREATE TABLE doc_relationships (
    source_id   UUID REFERENCES documents(doc_id),
    target_id   UUID REFERENCES documents(doc_id),
    rel_type    TEXT NOT NULL,  -- 'CITES', 'AUTHORED_BY_SAME', 'PART_OF_SERIES'
    weight      FLOAT DEFAULT 1.0,
    PRIMARY KEY (source_id, target_id, rel_type)
);

CREATE INDEX ON documents USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

CREATE INDEX ON doc_relationships (source_id);
CREATE INDEX ON doc_relationships (target_id);

-- Graph-vector hybrid query:
-- 1. Find top-10 semantically similar documents (vector layer)
-- 2. Traverse 2 hops of the citation graph from those seeds (graph layer)
-- 3. Re-rank the expanded set by a combined score

WITH query_embedding AS (
    -- In production, this comes from your embedding service
    -- Here represented as a placeholder cast
    SELECT '[0.1, 0.2, ...]'::vector(1536) AS emb
),

-- Step 1: Vector similarity search - seed nodes
vector_seeds AS (
    SELECT
        d.doc_id,
        d.title,
        d.embedding,
        1 - (d.embedding <=> q.emb) AS cosine_similarity,
        1 AS hop_distance
    FROM documents d, query_embedding q
    ORDER BY d.embedding <=> q.emb
    LIMIT 10
),

-- Step 2: Graph traversal from seed nodes (up to 2 hops)
graph_expansion AS (
    -- Anchor: the seed nodes
    SELECT doc_id, doc_id AS origin_id, 0 AS hops
    FROM vector_seeds

    UNION ALL

    -- Recursive: follow citation relationships
    SELECT
        dr.target_id AS doc_id,
        ge.origin_id,
        ge.hops + 1
    FROM graph_expansion ge
    JOIN doc_relationships dr
        ON dr.source_id = ge.doc_id
        AND dr.rel_type = 'CITES'
    WHERE ge.hops < 2
),

-- Deduplicate expanded graph nodes
expanded_candidates AS (
    SELECT DISTINCT doc_id, MIN(hops) AS min_hops
    FROM graph_expansion
    GROUP BY doc_id
),

-- Step 3: Compute combined score for the expanded candidate set
final_ranking AS (
    SELECT
        d.doc_id,
        d.title,
        1 - (d.embedding <=> q.emb) AS cosine_similarity,
        ec.min_hops,
        -- Graph proximity penalty: penalize nodes found only through traversal
        -- Seeds (hop=0) get full score; each hop reduces score by 15%
        (1 - (d.embedding <=> q.emb)) * POWER(0.85, ec.min_hops) AS combined_score,
        CASE WHEN vs.doc_id IS NOT NULL THEN true ELSE false END AS is_seed_node
    FROM expanded_candidates ec
    JOIN documents d ON d.doc_id = ec.doc_id
    JOIN query_embedding q ON true
    LEFT JOIN vector_seeds vs ON vs.doc_id = ec.doc_id
)

SELECT
    doc_id,
    title,
    ROUND(cosine_similarity::numeric, 4) AS semantic_score,
    min_hops AS graph_hops,
    ROUND(combined_score::numeric, 4) AS combined_score,
    is_seed_node
FROM final_ranking
ORDER BY combined_score DESC
LIMIT 20;

The hop penalty (0.85 per hop) is a tunable parameter. In citation networks where relevance decays sharply with distance, values between 0.75 and 0.85 work well. In social graphs where second-degree connections are highly relevant, you might use 0.95 or no penalty at all. Benchmark this against your human-labeled relevance set before deploying.

Performance characteristics to expect: with 1M documents and an HNSW index, the vector seed retrieval runs in ~5ms. The recursive CTE traversal over 10 seeds at depth 2 with average branching factor 5 will generate ~250 candidate nodes, and the secondary embedding lookups add ~1ms. Total query time is typically 10–30ms for this pattern at this scale — well within interactive response time budgets.

Graph-Aware RAG: Using Relationship Context in LLM Pipelines

Retrieval-Augmented Generation pipelines typically operate in a flat retrieval model: embed the query, retrieve top-k chunks, stuff them into the context window. This loses structural information. Graph-aware RAG incorporates relationship traversal into the retrieval step so the context passed to the LLM includes not just semantically similar content but contextually connected content.

Microsoft Research published GraphRAG in 2024, which builds a community detection graph over extracted entities and uses that graph to enable global queries (summarizing broad themes across a corpus) that flat RAG cannot handle. The architecture is more complex than simple hybrid search but the principle is the same: use graph structure to decide what to retrieve.

A more immediately deployable pattern is entity-linked RAG: extract named entities from the query, look up those entities in a knowledge graph to gather structured context, then combine that structured context with vector-retrieved chunks.

# Python
# Entity-linked Graph-aware RAG with Neo4j + pgvector + OpenAI

from neo4j import GraphDatabase
from openai import OpenAI
import psycopg2
import json
from typing import Optional

openai_client = OpenAI()
neo4j_driver = GraphDatabase.driver("bolt://localhost:7687", auth=("neo4j", "password"))
pg_conn = psycopg2.connect("postgresql://user:pass@localhost/knowledge_db")

def extract_entities_from_query(query: str) -> list[str]:
    """Use LLM to extract named entities from the query."""
    response = openai_client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {
                "role": "system",
                "content": "Extract named entities (people, organizations, drugs, diseases, locations) "
                           "from the query. Return a JSON array of entity names only."
            },
            {"role": "user", "content": query}
        ],
        response_format={"type": "json_object"}
    )
    result = json.loads(response.choices[0].message.content)
    return result.get("entities", [])

def get_graph_context(entities: list[str]) -> dict:
    """Traverse knowledge graph for entity context."""
    if not entities:
        return {}

    cypher = """
    UNWIND $entity_names AS name
    MATCH (e:Entity)
    WHERE e.name =~ ('(?i)' + name)

    // Get direct relationships (1 hop)
    OPTIONAL MATCH (e)-[r1]->(neighbor1:Entity)
    OPTIONAL MATCH (e)<-[r2]-(neighbor2:Entity)

    RETURN
        e.name AS entity_name,
        e.type AS entity_type,
        e.description AS entity_description,
        collect(DISTINCT {
            name: neighbor1.name,
            type: neighbor1.type,
            relationship: type(r1),
            direction: 'outgoing'
        }) AS outgoing_relations,
        collect(DISTINCT {
            name: neighbor2.name,
            type: neighbor2.type,
            relationship: type(r2),
            direction: 'incoming'
        }) AS incoming_relations
    """

    with neo4j_driver.session() as session:
        result = session.run(cypher, entity_names=entities)
        graph_context = {}
        for record in result:
            entity = record["entity_name"]
            graph_context[entity] = {
                "type": record["entity_type"],
                "description": record["entity_description"],
                "relations": record["outgoing_relations"] + record["incoming_relations"]
            }
        return graph_context

def vector_search_with_graph_filter(
    query: str,
    entity_ids: Optional[list[str]] = None,
    top_k: int = 5
) -> list[dict]:
    """Vector similarity search in pgvector, optionally filtered by entity graph."""
    embedding = openai_client.embeddings.create(
        input=query,
        model="text-embedding-3-small"
    ).data[0].embedding

    embedding_str = "[" + ",".join(str(x) for x in embedding) + "]"

    with pg_conn.cursor() as cur:
        if entity_ids:
            cur.execute("""
                SELECT
                    c.chunk_id,
                    c.content,
                    c.source_doc,
                    c.section_title,
                    1 - (c.embedding <=> %s::vector) AS similarity
                FROM document_chunks c
                JOIN chunk_entities ce ON ce.chunk_id = c.chunk_id
                WHERE ce.entity_id = ANY(%s)
                ORDER BY c.embedding <=> %s::vector
                LIMIT %s
            """, (embedding_str, entity_ids, embedding_str, top_k))
        else:
            cur.execute("""
                SELECT
                    chunk_id,
                    content,
                    source_doc,
                    section_title,
                    1 - (embedding <=> %s::vector) AS similarity
                FROM document_chunks
                ORDER BY embedding <=> %s::vector
                LIMIT %s
            """, (embedding_str, embedding_str, top_k))

        rows = cur.fetchall()
        return [
            {
                "chunk_id": r[0],
                "content": r[1],
                "source_doc": r[2],
                "section_title": r[3],
                "similarity": float(r[4])
            }
            for r in rows
        ]

def graph_aware_rag(query: str) -> str:
    """Full graph-aware RAG pipeline."""

    # Step 1: Extract entities from query
    entities = extract_entities_from_query(query)
    print(f"Extracted entities: {entities}")

    # Step 2: Get structured graph context for those entities
    graph_context = get_graph_context(entities)
    print(f"Graph context retrieved for {len(graph_context)} entities")

    # Step 3: Vector search (filtered if we have entity context)
    # In production, translate entity names to IDs via a lookup table
    retrieved_chunks = vector_search_with_graph_filter(query, top_k=5)
    print(f"Retrieved {len(retrieved_chunks)} chunks via vector search")

    # Step 4: Construct enriched context for LLM
    graph_context_text = ""
    for entity, ctx in graph_context.items():
        graph_context_text += f"\n[Entity: {entity} ({ctx['type']})]\n"
        if ctx["description"]:
            graph_context_text += f"Description: {ctx['description']}\n"
        if ctx["relations"]:
            relations_summary = [
                f"  - {rel['relationship']} -> {rel['name']}"
                for rel in ctx["relations"][:5]  # Limit to top 5 relations
            ]
            graph_context_text += "Relationships:\n" + "\n".join(relations_summary) + "\n"

    chunks_text = "\n\n".join([
        f"[Source: {c['source_doc']} | Section: {c['section_title']} | Similarity: {c['similarity']:.3f}]\n{c['content']}"
        for c in retrieved_chunks
    ])

    # Step 5: LLM generation with structured + semantic context
    prompt = f"""Answer the following question using the provided context.

KNOWLEDGE GRAPH CONTEXT (Structured relationships):
{graph_context_text if graph_context_text else "No graph context available."}

DOCUMENT CONTEXT (Semantically similar passages):
{chunks_text}

QUESTION: {query}

Provide a comprehensive answer grounded in the provided context."""

    response = openai_client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": "You are a precise research assistant. Ground your answers in the provided context."},
            {"role": "user", "content": prompt}
        ],
        temperature=0.1
    )

    return response.choices[0].message.content

# Example usage
answer = graph_aware_rag(
    "What are the known drug interactions between statins and blood thinners in elderly patients?"
)
print(answer)

The separation of graph context (structured, relational facts) from vector context (semantically similar text passages) in the prompt is intentional. LLMs attend differently to structured relational facts than to free-text passages, and keeping them visually distinct in the prompt tends to improve attribution accuracy in the generated answer.

Choosing Between Graph-Vector Integration Strategies

The decision tree for graph-vector architecture is driven by three factors: relationship depth, data volume, and team operational capacity.

Relationship depth is the primary driver. If your graph queries require more than three hops to answer meaningful questions, PostgreSQL recursive CTEs become slow and operationally awkward. Neo4j, Amazon Neptune, or ArangoDB handle this naturally. At one to two hops, PostgreSQL with a well-indexed adjacency table performs comparably to a dedicated graph database and eliminates operational complexity.

Data volume determines whether you can afford native hybrid query engines or need to separate the systems. Weaviate performs well at 50M+ vectors with cross-references, but its graph capabilities are limited to direct object references — it is not a property graph database. Neo4j's vector index, as of version 5.x, handles tens of millions of vectors with HNSW but is not optimized for the billion-vector scale that Pinecone or Milvus serve. If you need billion-scale vectors with complex multi-hop graph traversal, you are building a hybrid architecture with two separate systems communicating at the application layer.

Team operational capacity is the factor that gets underweighted in architecture decisions. Running two production database systems — a graph database and a vector database — means two monitoring setups, two backup procedures, two capacity planning models, and two teams who need to understand both. Unless the retrieval quality improvement from the native graph-vector approach is significant and measurable, the simpler architecture wins.

A practical decision matrix:

| Scenario | Recommended Architecture |

|---|---|

| <10M vectors, shallow relationships (≤2 hops) | PostgreSQL + pgvector + recursive CTEs |

| <50M vectors, moderate relationship complexity | Weaviate with cross-references |

| Any scale, deep graph traversal (>3 hops) | Neo4j 5.x or Amazon Neptune Analytics |

| >500M vectors + complex graphs |
