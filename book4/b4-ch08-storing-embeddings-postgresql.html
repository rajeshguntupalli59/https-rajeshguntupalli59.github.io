Chapter 8: Creating and Storing Embeddings in PostgreSQL

Embeddings are the bridge between unstructured text and the mathematical operations that make semantic search possible. If you've spent years designing schemas for structured data, the shift to storing 1,536-dimensional floating-point vectors alongside your rows can feel disorienting—but the underlying database engineering is familiar territory. This chapter walks through the complete workflow: generating embeddings from text using the OpenAI API and open-source alternatives, storing them efficiently in PostgreSQL using the pgvector extension, and structuring your schema for the query patterns you'll actually run in production. By the end, you'll have a working embedding pipeline that integrates with existing PostgreSQL infrastructure and handles the edge cases that break naive implementations.

---

8.1 How Embeddings Fit Into a PostgreSQL Schema

Before writing any code, it helps to understand what you're actually storing. An embedding is a fixed-length array of 32-bit floats. The OpenAI text-embedding-3-small model produces 1,536 floats per input string. A single embedding consumes 1,536 × 4 bytes = 6,144 bytes, or roughly 6 KB. At 1 million rows, that's approximately 6 GB of raw vector data before indexes.

From a PostgreSQL internals perspective, pgvector introduces a new column type called vector(n) where n is the number of dimensions. This type stores the float array in a binary format with a header, and it participates in standard PostgreSQL mechanics: TOAST for oversized tuples, MVCC, WAL, logical replication, and point-in-time recovery all work as expected. The extension adds three distance operators that map directly to similarity functions:

| Operator | Function | Use Case |

|---|---|---|

| <-> | L2 (Euclidean) distance | When absolute magnitude matters |

| <#> | Negative inner product | Max inner product search, dot product similarity |

| <=> | Cosine distance | Normalized text embeddings (most common) |

For text embeddings from OpenAI, Cohere, or sentence-transformers, cosine distance (<=>) is almost always correct. These models output normalized or near-normalized vectors, and cosine distance measures the angle between them regardless of magnitude. L2 distance works better for embeddings where magnitude carries meaning, such as certain vision or audio models.

A minimal schema that stores document embeddings alongside their source text looks like this:

-- PostgreSQL / pgvector

-- Enable the extension (requires superuser or pg_extension_owner membership)
CREATE EXTENSION IF NOT EXISTS vector;

-- Core documents table
CREATE TABLE documents (
    id          BIGSERIAL PRIMARY KEY,
    source_url  TEXT,
    title       TEXT NOT NULL,
    body        TEXT NOT NULL,
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    updated_at  TIMESTAMPTZ DEFAULT NOW()
);

-- Chunks table — documents are split before embedding
-- Typical chunk size: 256–512 tokens for retrieval tasks
CREATE TABLE document_chunks (
    id              BIGSERIAL PRIMARY KEY,
    document_id     BIGINT NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    chunk_index     INT NOT NULL,           -- position within parent document
    chunk_text      TEXT NOT NULL,          -- the actual text that was embedded
    token_count     INT,                    -- tokens used for this chunk
    embedding       vector(1536),           -- OpenAI text-embedding-3-small dimensions
    model_name      TEXT NOT NULL DEFAULT 'text-embedding-3-small',
    embedded_at     TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE (document_id, chunk_index)
);

-- Standard B-tree indexes for filtered queries
CREATE INDEX ON document_chunks (document_id);
CREATE INDEX ON document_chunks (model_name);
CREATE INDEX ON document_chunks (embedded_at);

-- Partial index: only index rows where embedding is populated
-- This avoids wasting index space on unprocessed chunks
CREATE INDEX ON document_chunks (id) WHERE embedding IS NULL;

The separation between documents and document_chunks is not optional for production systems. Embedding models have token limits—OpenAI's embedding models cap at 8,191 tokens. More importantly, shorter chunks produce more precise retrieval results. A chunk of 256–512 tokens typically outperforms a full 2,000-token document on recall benchmarks because the embedding captures a specific concept rather than a diffuse mixture of topics. The chunk_index column preserves ordering so you can reconstruct context windows around matching chunks during retrieval.

8.2 Generating Embeddings with the OpenAI API

The OpenAI embeddings endpoint is the most common starting point. The API is stateless and synchronous: you send text, you receive a vector. The key operational concerns are batching, error handling, token limits, and cost.

OpenAI's text-embedding-3-small costs $0.02 per million tokens as of mid-2024, compared to $0.13 per million tokens for text-embedding-3-large. For most retrieval applications, text-embedding-3-small at 1,536 dimensions outperforms the older text-embedding-ada-002 at the same cost point, making it the default choice for new projects.

The following Python script handles the full embedding generation workflow: chunking a document, calling the API in batches, and inserting results into PostgreSQL via psycopg2. Batching up to 2,048 inputs per API call is supported by OpenAI; staying at 100–500 per batch is a practical compromise between throughput and memory pressure.

# Python
import os
import time
import psycopg2
import psycopg2.extras
from openai import OpenAI
from typing import Generator

client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

DB_DSN = os.environ.get(
    "DATABASE_URL",
    "postgresql://postgres:password@localhost:5432/vectordb"
)

EMBEDDING_MODEL = "text-embedding-3-small"
EMBEDDING_DIMENSIONS = 1536
BATCH_SIZE = 100          # inputs per API call
MAX_TOKENS_PER_CHUNK = 500  # approximate; 1 token ≈ 4 characters


def chunk_text(text: str, max_chars: int = 2000) -> Generator[str, None, None]:
    """
    Simple paragraph-aware chunker.
    Production systems should use tiktoken for precise token counting.
    """
    paragraphs = text.split("\n\n")
    current_chunk = []
    current_len = 0

    for para in paragraphs:
        para = para.strip()
        if not para:
            continue
        if current_len + len(para) > max_chars and current_chunk:
            yield "\n\n".join(current_chunk)
            current_chunk = [para]
            current_len = len(para)
        else:
            current_chunk.append(para)
            current_len += len(para)

    if current_chunk:
        yield "\n\n".join(current_chunk)


def get_embeddings(texts: list[str], retries: int = 3) -> list[list[float]]:
    """
    Call OpenAI embeddings API with retry logic for rate limits.
    Returns a list of embedding vectors in the same order as input texts.
    """
    for attempt in range(retries):
        try:
            response = client.embeddings.create(
                model=EMBEDDING_MODEL,
                input=texts,
            )
            # The API guarantees same-order response as input
            return [item.embedding for item in response.data]
        except Exception as e:
            if "rate_limit" in str(e).lower() and attempt < retries - 1:
                wait = 2 ** attempt  # exponential backoff: 1s, 2s, 4s
                print(f"Rate limited. Waiting {wait}s before retry {attempt + 1}")
                time.sleep(wait)
            else:
                raise


def embed_document(document_id: int, title: str, body: str) -> int:
    """
    Chunk and embed a document, storing results in document_chunks.
    Returns the number of chunks created.
    """
    chunks = list(chunk_text(body))
    if not chunks:
        return 0

    conn = psycopg2.connect(DB_DSN)
    inserted = 0

    try:
        # Process in batches to respect API limits
        for batch_start in range(0, len(chunks), BATCH_SIZE):
            batch_texts = chunks[batch_start : batch_start + BATCH_SIZE]
            embeddings = get_embeddings(batch_texts)

            rows = []
            for i, (text, embedding) in enumerate(zip(batch_texts, embeddings)):
                chunk_index = batch_start + i
                token_estimate = len(text) // 4  # rough estimate
                rows.append({
                    "document_id": document_id,
                    "chunk_index": chunk_index,
                    "chunk_text": text,
                    "token_count": token_estimate,
                    "embedding": embedding,
                    "model_name": EMBEDDING_MODEL,
                })

            with conn.cursor() as cur:
                psycopg2.extras.execute_values(
                    cur,
                    """
                    INSERT INTO document_chunks
                        (document_id, chunk_index, chunk_text, token_count, embedding, model_name)
                    VALUES %s
                    ON CONFLICT (document_id, chunk_index) DO UPDATE
                        SET chunk_text  = EXCLUDED.chunk_text,
                            token_count = EXCLUDED.token_count,
                            embedding   = EXCLUDED.embedding,
                            model_name  = EXCLUDED.model_name,
                            embedded_at = NOW()
                    """,
                    [
                        (
                            r["document_id"],
                            r["chunk_index"],
                            r["chunk_text"],
                            r["token_count"],
                            r["embedding"],
                            r["model_name"],
                        )
                        for r in rows
                    ],
                    template="(%s, %s, %s, %s, %s::vector, %s)",
                    page_size=BATCH_SIZE,
                )
            conn.commit()
            inserted += len(rows)

    finally:
        conn.close()

    return inserted


if __name__ == "__main__":
    # Example: insert a document and embed it
    conn = psycopg2.connect(DB_DSN)
    with conn.cursor() as cur:
        cur.execute(
            "INSERT INTO documents (title, body) VALUES (%s, %s) RETURNING id",
            ("PostgreSQL Architecture", "PostgreSQL uses a process-per-connection model...\n\n"
             "The shared buffer pool is the primary caching mechanism...\n\n"
             "Write-ahead logging ensures durability across crashes..."),
        )
        doc_id = cur.fetchone()[0]
    conn.commit()
    conn.close()

    count = embed_document(doc_id, "PostgreSQL Architecture", "sample body text")
    print(f"Created {count} embedded chunks for document {doc_id}")

Three implementation details here are worth explaining. First, the ON CONFLICT ... DO UPDATE clause handles re-embedding gracefully. When you upgrade models or fix chunking logic, you re-run the pipeline without managing deletes. Second, the explicit ::vector cast in the execute_values template is required—psycopg2 doesn't know how to serialize Python lists as the pgvector type natively. Third, token estimation using character count divided by 4 is intentionally rough; production systems should use the tiktoken library for precise counts.

8.3 Using Open-Source Embedding Models

The OpenAI API introduces a network call, a cost, and a dependency on an external service. For on-premises deployments, regulated industries, or high-throughput scenarios where per-token costs accumulate quickly, running an embedding model locally is the correct tradeoff.

The sentence-transformers library provides access to hundreds of pretrained models that run on CPU or GPU. The all-MiniLM-L6-v2 model produces 384-dimensional embeddings and runs at roughly 14,000 sentences per second on an 8-core CPU. The BAAI/bge-large-en-v1.5 model at 1,024 dimensions scores near OpenAI's ada-002 on the MTEB benchmark while running entirely locally. For most RAG applications, one of these two models is the right starting point.

# Python
import os
import numpy as np
import psycopg2
import psycopg2.extras
from sentence_transformers import SentenceTransformer

# Model selection:
# all-MiniLM-L6-v2 → 384 dims, fast, good for general retrieval
# BAAI/bge-large-en-v1.5 → 1024 dims, higher quality, 2x slower
# intfloat/e5-large-v2 → 1024 dims, strong on asymmetric retrieval tasks
MODEL_NAME = "BAAI/bge-large-en-v1.5"
EMBEDDING_DIMENSIONS = 1024

# Load once at module level — model loading takes 2–5 seconds
model = SentenceTransformer(MODEL_NAME)

DB_DSN = os.environ.get(
    "DATABASE_URL",
    "postgresql://postgres:password@localhost:5432/vectordb"
)


def embed_texts_local(texts: list[str], batch_size: int = 64) -> np.ndarray:
    """
    Generate embeddings using a local sentence-transformers model.
    Returns a numpy array of shape (len(texts), EMBEDDING_DIMENSIONS).

    BGE models expect a query prefix for retrieval tasks:
    - Documents: embed as-is
    - Queries: prefix with "Represent this sentence: "
    """
    embeddings = model.encode(
        texts,
        batch_size=batch_size,
        normalize_embeddings=True,   # ensures cosine similarity = dot product
        show_progress_bar=len(texts) > 100,
        convert_to_numpy=True,
    )
    return embeddings


def embed_query(query_text: str) -> list[float]:
    """
    Embed a user query. BGE models use a different prefix for queries vs. documents.
    This asymmetric encoding improves retrieval quality on BGE models.
    """
    prefixed = f"Represent this sentence for searching relevant passages: {query_text}"
    embedding = model.encode([prefixed], normalize_embeddings=True)
    return embedding[0].tolist()


def bulk_embed_pending_chunks(batch_size: int = 500) -> int:
    """
    Find chunks without embeddings and fill them using the local model.
    Designed to run as a background job or one-time migration.
    """
    conn = psycopg2.connect(DB_DSN)
    total_processed = 0

    try:
        while True:
            with conn.cursor() as cur:
                # Use the partial index we created on (id) WHERE embedding IS NULL
                cur.execute(
                    """
                    SELECT id, chunk_text
                    FROM document_chunks
                    WHERE embedding IS NULL
                    ORDER BY id
                    LIMIT %s
                    """,
                    (batch_size,),
                )
                rows = cur.fetchall()

            if not rows:
                break

            ids = [r[0] for r in rows]
            texts = [r[1] for r in rows]

            embeddings = embed_texts_local(texts)

            update_data = [
                (embedding.tolist(), model_id)
                for embedding, model_id in zip(embeddings, ids)
            ]

            with conn.cursor() as cur:
                psycopg2.extras.execute_values(
                    cur,
                    """
                    UPDATE document_chunks AS dc
                    SET embedding  = data.embedding::vector,
                        model_name = %s,
                        embedded_at = NOW()
                    FROM (VALUES %s) AS data(embedding, id)
                    WHERE dc.id = data.id::bigint
                    """,
                    update_data,
                    template="(%s, %s)",
                )

            conn.commit()
            total_processed += len(rows)
            print(f"Embedded {total_processed} chunks so far...")

    finally:
        conn.close()

    return total_processed


if __name__ == "__main__":
    # Smoke test: embed a single query and print its first 5 dimensions
    vec = embed_query("How does PostgreSQL handle concurrent writes?")
    print(f"Query embedding: {len(vec)} dimensions, first 5: {vec[:5]}")

    # Run the backfill job
    count = bulk_embed_pending_chunks()
    print(f"Total chunks embedded: {count}")

One tradeoff to understand: sentence-transformers models are session-local. On a multi-process application server, each worker loads the model separately, consuming 500 MB–2 GB of RAM per worker depending on the model. The standard mitigation is to move embedding generation out of the web application entirely—into a dedicated embedding service or a background queue that workers dispatch to asynchronously.

The dimension mismatch between models is a schema problem that must be handled explicitly. If you embed documents with all-MiniLM-L6-v2 (384 dimensions) and later migrate to BAAI/bge-large-en-v1.5 (1,024 dimensions), the column type vector(384) will reject new inserts. The model_name column in the schema above makes this tractable: you can run both models simultaneously during a migration window by using separate columns or a second table, verify retrieval quality, then cut over.

8.4 Building the Vector Index

Storing vectors without an index is viable only up to approximately 100,000 rows. Beyond that threshold, every similarity query becomes a full sequential scan—PostgreSQL reads every row, computes the distance, and sorts. At 1 million rows and 1,536 dimensions, a sequential scan typically takes 3–8 seconds depending on hardware. With an HNSW index, the same query returns in 5–50 milliseconds.

pgvector supports two index types: IVFFlat (inverted file with flat quantization) and HNSW (Hierarchical Navigable Small World graphs). The performance characteristics follow directly from their algorithms:

  • → **IVFFlat** partitions the vector space into `lists` (Voronoi cells) during index build. At query time, it searches only the nearest `probes` cells. Build is fast, memory footprint is low, but recall is lower than HNSW at equivalent speed. Requires data to be pre-loaded before building (`CREATE INDEX AFTER` populating, not before).
  • → **HNSW** builds a multi-layer proximity graph. Query traversal is faster and recall is higher, but the index takes longer to build and consumes more memory (approximately 6–10 bytes per vector per edge, plus the vectors themselves). New rows are inserted directly into the graph, making it better for append-heavy workloads.

    • For most production deployments, HNSW is the right default. The following SQL creates and tunes it:

    -- PostgreSQL / pgvector

-- HNSW index using cosine distance
-- m: number of connections per layer (default 16, range 4–64)
--    Higher m → better recall, larger index, slower build
-- ef_construction: size of the dynamic candidate list during build (default 64)
--    Higher ef_construction → better recall, much slower build
--    Recommended range: 64–200 for production

CREATE INDEX CONCURRENTLY idx_chunks_embedding_hnsw
ON document_chunks
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- For filtered queries (e.g., "find similar chunks within document X"),
-- you CANNOT rely on the HNSW index alone — PostgreSQL may choose a seq scan
-- when the WHERE clause is selective. Partial indexes help:

-- Index only chunks from a specific model version
CREATE INDEX CONCURRENTLY idx_chunks_embedding_v2
ON document_chunks
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64)
WHERE model_name = 'text-embedding-3-small';

-- Control query-time recall vs. speed tradeoff
-- ef_search=40 is the default; higher values improve recall at cost of latency
-- Set this per-session for queries that require higher accuracy
SET hnsw.ef_search = 100;

-- Verify the index is being used
EXPLAIN (ANALYZE, BUFFERS)
SELECT
    dc.id,
    dc.chunk_text,
    dc.embedding <=> '[0.1, 0.2, ...]'::vector AS distance
FROM document_chunks dc
WHERE dc.model_name = 'text-embedding-3-small'
ORDER BY distance
LIMIT 10;

-- Check index size and usage statistics
SELECT
    indexname,
    pg_size_pretty(pg_relation_size(indexname::regclass)) AS index_size,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch
FROM pg_stat_user_indexes
WHERE tablename = 'document_chunks';

    One operational gotcha: CREATE INDEX CONCURRENTLY on an HNSW index with 1 million rows and ef_construction = 64 takes 20–90 minutes depending on hardware and dimensionality. Plan for this in maintenance windows. The index build is CPU-bound; parallelism is controlled by max_parallel_maintenance_workers. Setting this to 4–8 for the index build session reduces wall-clock time significantly:

    -- PostgreSQL / pgvector

-- Increase parallel workers for this session before running the index build
SET max_parallel_maintenance_workers = 4;

-- For IVFFlat, the rule of thumb for lists is sqrt(row_count)
-- At 1M rows: lists = 1000
-- probes at query time: typically lists / 10 = 100 (higher → better recall, slower)
CREATE INDEX CONCURRENTLY idx_chunks_embedding_ivfflat
ON document_chunks
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 1000);

-- Set probes at query time
SET ivfflat.probes = 100;

    The recall-latency tradeoff is a parameter you tune against your data, not a fixed property of the extension. For RAG applications where top-10 recall at 95% is the target, HNSW with m=16, ef_construction=64, and ef_search=80 is a reasonable starting configuration. Run the MTEB retrieval benchmarks against a sample of your own data to validate.
