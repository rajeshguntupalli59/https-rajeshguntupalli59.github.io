Chapter 27: Evaluating RAG Quality: Metrics That Matter

Building a RAG pipeline is the easy part. Knowing whether it actually works—and proving that it works well enough to ship to production—requires a measurement framework that most teams bolt on too late or skip entirely. This chapter covers the metrics that separate production-grade RAG systems from prototypes that performed well in demos: how retrieval quality, generation faithfulness, and end-to-end answer accuracy are measured, the tools that automate those measurements, and the database instrumentation patterns that let you track quality over time rather than just at launch.

---

The RAG Quality Problem: Why "It Feels Right" Doesn't Scale

When a DBA tunes a query, the feedback loop is immediate and objective: execution time, rows examined, index usage. RAG pipelines don't offer the same clarity. A system can return a confident, grammatically perfect answer that is factually wrong, subtly incomplete, or grounded in a chunk retrieved for the wrong reasons. Without explicit metrics, teams make decisions based on anecdote—a few test questions, stakeholder impressions, a vague sense that the model "seems smarter now."

The measurement challenge has three layers, and they fail independently:

Retrieval quality: Did the vector search surface the right chunks? A retrieval system can score highly on semantic similarity while missing the specific paragraph that contains the answer. Conversely, it can retrieve highly relevant text that the LLM then ignores or misinterprets.

Generation faithfulness: Did the LLM's answer stay grounded in the retrieved context? Models hallucinate even when given correct source material. They may blend retrieved facts with parametric knowledge in ways that introduce errors.

Answer correctness: Does the final output match the ground truth? This is the metric users care about, but it's the hardest to compute at scale because it typically requires a reference answer or human judgment.

Each layer needs its own measurement strategy. Optimizing only for end-to-end correctness without decomposing the pipeline hides the root cause of failures—a problem database engineers recognize immediately as analogous to treating slow query symptoms without examining the execution plan.

---

Retrieval Metrics: Measuring What the Vector Search Returns

Retrieval quality is the most database-familiar part of RAG evaluation. The vocabulary maps directly from information retrieval: precision, recall, and ranking quality.

Precision@K measures the fraction of retrieved chunks that are relevant, at cutoff K. If you retrieve 5 chunks and 3 are genuinely relevant, Precision@5 = 0.60. Recall@K measures the fraction of all relevant chunks in the corpus that appear in the top K results. These two metrics trade off against each other. Increasing chunk overlap and retrieval K improves recall but degrades precision, which increases LLM context length and raises both latency and cost.

Mean Reciprocal Rank (MRR) evaluates where the first relevant result lands. If the first relevant chunk is at position 2, the reciprocal rank is 0.50. MRR averaged across queries tells you how consistently the best answer surfaces near the top.

Normalized Discounted Cumulative Gain (NDCG) handles graded relevance—useful when chunks aren't simply relevant or not, but have degrees of relevance (the exact answer paragraph versus a related paragraph versus tangential context).

Computing these requires a labeled evaluation set: queries paired with the chunk IDs (or document IDs) that constitute a correct answer. Building this dataset is the work most teams skip. A practical minimum is 200–300 labeled question-chunk pairs, sampled from your actual query distribution.

# Python
# Retrieval evaluation with a labeled dataset
# Uses pgvector for retrieval and a simple evaluation harness

import psycopg2
import numpy as np
from openai import OpenAI
from dataclasses import dataclass
from typing import List, Dict

client = OpenAI()

@dataclass
class RetrievalEvalSample:
    query: str
    relevant_chunk_ids: List[int]  # Ground truth chunk IDs

def embed_query(text: str) -> List[float]:
    response = client.embeddings.create(
        model="text-embedding-3-small",
        input=text
    )
    return response.data[0].embedding

def retrieve_chunks(
    conn,
    query_embedding: List[float],
    k: int = 5
) -> List[int]:
    """Returns ordered list of chunk IDs from pgvector search."""
    embedding_str = "[" + ",".join(map(str, query_embedding)) + "]"
    with conn.cursor() as cur:
        cur.execute("""
            SELECT chunk_id
            FROM document_chunks
            ORDER BY embedding <=> %s::vector
            LIMIT %s
        """, (embedding_str, k))
        return [row[0] for row in cur.fetchall()]

def precision_at_k(
    retrieved: List[int],
    relevant: List[int],
    k: int
) -> float:
    retrieved_k = retrieved[:k]
    hits = len(set(retrieved_k) & set(relevant))
    return hits / k if k > 0 else 0.0

def recall_at_k(
    retrieved: List[int],
    relevant: List[int],
    k: int
) -> float:
    retrieved_k = retrieved[:k]
    hits = len(set(retrieved_k) & set(relevant))
    return hits / len(relevant) if relevant else 0.0

def reciprocal_rank(
    retrieved: List[int],
    relevant: List[int]
) -> float:
    for rank, chunk_id in enumerate(retrieved, start=1):
        if chunk_id in relevant:
            return 1.0 / rank
    return 0.0

def evaluate_retrieval(
    conn,
    eval_samples: List[RetrievalEvalSample],
    k: int = 5
) -> Dict[str, float]:
    precisions, recalls, rrs = [], [], []

    for sample in eval_samples:
        embedding = embed_query(sample.query)
        retrieved = retrieve_chunks(conn, embedding, k=k)

        precisions.append(
            precision_at_k(retrieved, sample.relevant_chunk_ids, k)
        )
        recalls.append(
            recall_at_k(retrieved, sample.relevant_chunk_ids, k)
        )
        rrs.append(
            reciprocal_rank(retrieved, sample.relevant_chunk_ids)
        )

    return {
        f"precision@{k}": float(np.mean(precisions)),
        f"recall@{k}": float(np.mean(recalls)),
        "mrr": float(np.mean(rrs)),
        "num_samples": len(eval_samples)
    }

# Example usage
conn = psycopg2.connect("postgresql://user:pass@localhost:5432/ragdb")

eval_samples = [
    RetrievalEvalSample(
        query="What is the maximum connection pool size for pgBouncer?",
        relevant_chunk_ids=[1042, 1043]
    ),
    RetrievalEvalSample(
        query="How does WAL archiving work in PostgreSQL?",
        relevant_chunk_ids=[287, 288, 295]
    ),
]

results = evaluate_retrieval(conn, eval_samples, k=5)
print(results)
# Example output:
# {'precision@5': 0.52, 'recall@5': 0.81, 'mrr': 0.74, 'num_samples': 2}

What numbers to target: For domain-specific enterprise RAG, Precision@5 above 0.55 and Recall@5 above 0.75 are reasonable starting thresholds before the retrieval layer is considered production-ready. MRR above 0.70 means the best chunk is usually in the top two positions, which matters because most LLMs weight earlier context more heavily.

When retrieval scores are low, the diagnostic path mirrors index tuning: check embedding model alignment with your query distribution, examine chunk size and overlap, verify that the index type (HNSW vs IVFFlat) matches your ef_search/nprobe settings, and consider hybrid search (vector + BM25) for queries with strong keyword signals.

Generation Metrics: Faithfulness, Relevance, and Hallucination Detection

Once you've confirmed the retrieval layer is surfacing good chunks, the next question is what the LLM does with them. Generation quality breaks into three independent concerns:

Faithfulness (also called groundedness): Every factual claim in the answer should be supported by the retrieved context. A faithfulness score of 1.0 means all claims trace back to the provided chunks. Below 0.80 in production is a warning sign for hallucination risk.

Answer relevance: Does the answer actually address the question? A model can produce a faithful, well-grounded response that is technically responsive but misses what the user asked. This is measured separately from faithfulness.

Context relevance: Of the chunks passed to the LLM, what fraction were actually needed to produce the answer? High context relevance means retrieval is efficient. Low scores indicate noise in the retrieved context that increases token cost without improving output.

RAGAS (Retrieval Augmented Generation Assessment) is the most widely adopted framework for computing these metrics programmatically. It uses an LLM as a judge—a common pattern where one model evaluates the output of another—which introduces its own reliability concerns but scales well compared to human evaluation.

# Python
# RAGAS evaluation pipeline for a RAG system built with LangChain + pgvector
# pip install ragas langchain langchain-openai psycopg2-binary

from ragas import evaluate
from ragas.metrics import (
    faithfulness,
    answer_relevancy,
    context_recall,
    context_precision,
)
from ragas.metrics.critique import harmfulness
from datasets import Dataset
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_postgres import PGVector
from langchain.chains import RetrievalQA
import os

# Initialize the vector store (pgvector via LangChain)
CONNECTION_STRING = os.environ["PG_CONNECTION_STRING"]
COLLECTION_NAME = "docs_collection"

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = PGVector(
    embeddings=embeddings,
    collection_name=COLLECTION_NAME,
    connection=CONNECTION_STRING,
)
retriever = vectorstore.as_retriever(search_kwargs={"k": 5})

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)
qa_chain = RetrievalQA.from_chain_type(
    llm=llm,
    retriever=retriever,
    return_source_documents=True
)

# Evaluation questions with ground truth answers
test_questions = [
    {
        "question": "What are the ACID properties in PostgreSQL?",
        "ground_truth": (
            "ACID stands for Atomicity, Consistency, Isolation, and Durability. "
            "PostgreSQL enforces all four properties through its MVCC implementation "
            "and write-ahead logging."
        )
    },
    {
        "question": "How does connection pooling improve database performance?",
        "ground_truth": (
            "Connection pooling reuses existing database connections instead of "
            "creating new ones for each client request, reducing overhead from "
            "connection establishment and reducing memory usage on the database server."
        )
    },
]

# Run the RAG pipeline and collect inputs for RAGAS
questions, answers, contexts, ground_truths = [], [], [], []

for item in test_questions:
    result = qa_chain.invoke({"query": item["question"]})
    questions.append(item["question"])
    answers.append(result["result"])
    contexts.append([
        doc.page_content for doc in result["source_documents"]
    ])
    ground_truths.append(item["ground_truth"])

# Build the RAGAS dataset
eval_dataset = Dataset.from_dict({
    "question": questions,
    "answer": answers,
    "contexts": contexts,
    "ground_truth": ground_truths,
})

# Run RAGAS evaluation
results = evaluate(
    eval_dataset,
    metrics=[
        faithfulness,
        answer_relevancy,
        context_recall,
        context_precision,
    ],
    llm=ChatOpenAI(model="gpt-4o-mini"),
    embeddings=embeddings,
)

print(results)
# Example output:
# {'faithfulness': 0.87, 'answer_relevancy': 0.91,
#  'context_recall': 0.78, 'context_precision': 0.65}

df = results.to_pandas()
df.to_csv("ragas_eval_results.csv", index=False)

Interpreting RAGAS scores in production terms: Faithfulness below 0.75 indicates the LLM is introducing information not present in the retrieved chunks—reduce temperature, tighten the system prompt, or increase retrieval precision. Context precision below 0.60 means roughly 40% of retrieved chunks aren't contributing to the answer—a retrieval efficiency problem that inflates latency and cost without quality benefit. Answer relevancy below 0.80 usually points to a prompt engineering issue or query transformation gap.

The LLM-as-judge pattern has a known limitation: judge models share biases with the models being evaluated. For high-stakes applications, supplement RAGAS with human evaluation on a stratified sample (typically 5–10% of the full evaluation set). The stratification should include edge cases: short queries, multi-hop questions, queries with no answer in the corpus, and queries that span multiple documents.

End-to-End Correctness: Answer Quality at Scale

Faithfulness and relevance are process metrics—they measure how well the pipeline executes its steps. Answer correctness is the outcome metric: did the system produce an answer that is factually accurate relative to a known reference?

Exact match is the strictest form: the answer string matches the reference exactly. Useful for structured outputs (dates, product codes, numeric values) but too brittle for natural language responses.

Semantic similarity uses embedding cosine similarity between the generated answer and the reference answer. Scores above 0.85 typically indicate strong alignment. This is fast to compute at scale but can be fooled by paraphrased wrong answers.

LLM-based scoring submits the question, reference answer, and generated answer to a judge model with a structured rubric. This is the most accurate automated method but adds latency and cost. Frameworks like DeepEval, TruLens, and Athina AI provide pre-built judge prompts and scoring rubrics.

# Python
# Answer correctness evaluation using DeepEval
# pip install deepeval

from deepeval import evaluate
from deepeval.metrics import (
    AnswerRelevancyMetric,
    FaithfulnessMetric,
    ContextualRecallMetric,
    ContextualPrecisionMetric,
    HallucinationMetric,
)
from deepeval.test_case import LLMTestCase
import os

os.environ["OPENAI_API_KEY"] = os.environ["OPENAI_API_KEY"]

# A single test case mirrors the structure of a RAGAS sample
# but DeepEval gives finer control over scoring thresholds

test_case = LLMTestCase(
    input="What isolation levels does PostgreSQL support?",
    actual_output=(
        "PostgreSQL supports four isolation levels: "
        "Read Uncommitted, Read Committed, Repeatable Read, and Serializable. "
        "In practice, Read Uncommitted behaves like Read Committed in PostgreSQL."
    ),
    expected_output=(
        "PostgreSQL supports Read Committed, Repeatable Read, and Serializable. "
        "Read Uncommitted is accepted but treated as Read Committed."
    ),
    retrieval_context=[
        "PostgreSQL implements four transaction isolation levels as defined "
        "by the SQL standard: Read Uncommitted, Read Committed, Repeatable Read, "
        "and Serializable. PostgreSQL's Read Uncommitted mode behaves like "
        "Read Committed.",
        "The default isolation level in PostgreSQL is Read Committed.",
    ]
)

# Define metrics with explicit thresholds
metrics = [
    AnswerRelevancyMetric(threshold=0.80, model="gpt-4o-mini"),
    FaithfulnessMetric(threshold=0.80, model="gpt-4o-mini"),
    HallucinationMetric(threshold=0.20, model="gpt-4o-mini"),
    ContextualRecallMetric(threshold=0.75, model="gpt-4o-mini"),
    ContextualPrecisionMetric(threshold=0.60, model="gpt-4o-mini"),
]

# Evaluate and print per-metric verdicts
for metric in metrics:
    metric.measure(test_case)
    print(
        f"{metric.__class__.__name__}: "
        f"score={metric.score:.3f}, "
        f"passed={metric.is_successful()}, "
        f"reason={metric.reason}"
    )

Connecting evaluation to CI/CD: Evaluation suites should run on every pipeline change, just as query regression tests run on schema changes. The evaluation dataset is a versioned artifact in your repository. A PR that introduces a new embedding model or changes chunk size should trigger an evaluation run, and the results should gate deployment. This is the same discipline as maintaining a query performance baseline in a DBA workflow—the tooling is different, but the principle is identical.

The cost of automated LLM-based evaluation is real: evaluating 500 test cases with GPT-4o-mini as judge costs roughly $0.15–0.40 per full run depending on context length. This is low enough to run on every deploy for most teams, but you should budget for it explicitly when setting up the pipeline.

Storing and Trending Evaluation Results in PostgreSQL

Metrics measured once tell you where you stand today. Metrics tracked over time tell you whether changes are helping or degrading quality—the same distinction between a point-in-time EXPLAIN and a continuous slow query log. Storing evaluation results in a relational database enables trend analysis, regression detection, and correlation with system changes.

-- PostgreSQL / pgvector
-- Schema for tracking RAG evaluation results over time

CREATE TABLE rag_pipeline_versions (
    version_id        SERIAL PRIMARY KEY,
    version_tag       TEXT NOT NULL UNIQUE,         -- e.g. 'v2.4.1'
    embedding_model   TEXT NOT NULL,
    llm_model         TEXT NOT NULL,
    chunk_size        INTEGER,
    chunk_overlap     INTEGER,
    retrieval_k       INTEGER,
    index_type        TEXT,                          -- 'hnsw' or 'ivfflat'
    deployed_at       TIMESTAMPTZ DEFAULT now(),
    notes             TEXT
);

CREATE TABLE rag_eval_runs (
    run_id            SERIAL PRIMARY KEY,
    version_id        INTEGER REFERENCES rag_pipeline_versions(version_id),
    eval_dataset_tag  TEXT NOT NULL,               -- versioned eval set name
    run_at            TIMESTAMPTZ DEFAULT now(),
    num_samples       INTEGER NOT NULL,
    -- Retrieval metrics
    precision_at_5    NUMERIC(5,4),
    recall_at_5       NUMERIC(5,4),
    mrr               NUMERIC(5,4),
    -- Generation metrics (RAGAS)
    faithfulness      NUMERIC(5,4),
    answer_relevancy  NUMERIC(5,4),
    context_precision NUMERIC(5,4),
    context_recall    NUMERIC(5,4),
    -- Correctness
    hallucination_rate NUMERIC(5,4),
    -- Operational
    avg_latency_ms    NUMERIC(8,2),
    p99_latency_ms    NUMERIC(8,2),
    avg_tokens_per_query INTEGER,
    eval_cost_usd     NUMERIC(8,4)
);

CREATE TABLE rag_eval_sample_results (
    sample_id         SERIAL PRIMARY KEY,
    run_id            INTEGER REFERENCES rag_eval_runs(run_id),
    question          TEXT NOT NULL,
    retrieved_chunk_ids INTEGER[],
    generated_answer  TEXT,
    faithfulness      NUMERIC(5,4),
    answer_relevancy  NUMERIC(5,4),
    hallucinated      BOOLEAN,
    latency_ms        INTEGER
);

-- Index for trend queries
CREATE INDEX idx_eval_runs_version ON rag_eval_runs(version_id, run_at);
CREATE INDEX idx_sample_results_run ON rag_eval_sample_results(run_id);

-- Query: metric trend across pipeline versions
SELECT
    v.version_tag,
    v.embedding_model,
    v.chunk_size,
    r.run_at,
    r.faithfulness,
    r.answer_relevancy,
    r.precision_at_5,
    r.recall_at_5,
    r.mrr,
    r.avg_latency_ms
FROM rag_eval_runs r
JOIN rag_pipeline_versions v ON r.version_id = v.version_id
WHERE r.eval_dataset_tag = 'postgres-docs-v1'
ORDER BY r.run_at DESC
LIMIT 20;

-- Query: find regressions — versions where faithfulness dropped > 0.05
WITH version_scores AS (
    SELECT
        v.version_tag,
        r.run_at,
        r.faithfulness,
        LAG(r.faithfulness) OVER (
            ORDER BY r.run_at
        ) AS prev_faithfulness
    FROM rag_eval_runs r
    JOIN rag_pipeline_versions v ON r.version_id = v.version_id
    WHERE r.eval_dataset_tag = 'postgres-docs-v1'
)
SELECT
    version_tag,
    run_at,
    faithfulness,
    prev_faithfulness,
    faithfulness - prev_faithfulness AS delta
FROM version_scores
WHERE faithfulness - prev_faithfulness < -0.05
ORDER BY run_at DESC;

-- Query: worst-performing samples in the latest run
SELECT
    s.question,
    s.faithfulness,
    s.answer_relevancy,
    s.hallucinated,
    s.latency_ms
FROM rag_eval_sample_results s
JOIN rag_eval_runs r ON s.run_id = r.run_id
WHERE r.run_id = (
    SELECT run_id FROM rag_eval_runs
    ORDER BY run_at DESC
    LIMIT 1
)
ORDER BY s.faithfulness ASC
LIMIT 20;

This schema treats evaluation runs as first-class records—versioned, queryable, and correlated with pipeline configuration. The rag_pipeline_versions table is the equivalent of a change log: every modification to embedding model, chunk size, or retrieval parameters creates a new record, making it possible to bisect regressions precisely.

The rag_eval_sample_results table enables failure analysis at the sample level. Sorted by lowest faithfulness, it surfaces exactly which questions the current pipeline handles poorly—the same way sorting a slow query log by total execution time surfaces the highest-impact optimization targets.

Production Monitoring: Online Metrics and Feedback Loops

Offline evaluation on a labeled dataset tells you how the pipeline performs on known questions. Production traffic contains questions you've never seen, spanning query types, user populations, and document states that evolve over time. Online monitoring closes that gap.

Implicit feedback signals are the lowest-friction source of production quality data. Thumbs up/down ratings, answer regeneration requests, session abandonment after a RAG response, and follow-up clarification questions all carry quality signal. These signals are noisy individually but aggregate into reliable trend indicators.

Shadow scoring runs the evaluation metrics asynchronously on production traffic. For each query, the system logs the question, retrieved chunks, and generated answer to a queue. A background worker submits samples to the RAGAS or DeepEval scorer and writes results to the evaluation schema. This adds zero latency to the user path while providing continuous quality data.
