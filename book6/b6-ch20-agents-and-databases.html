Chapter 20: How AI Agents Use Databases as Tools

AI agents don't just query databases — they use them as instruments for reasoning, memory, state management, and coordination. Where a traditional application executes a fixed query on behalf of a user, an agent decides at runtime which database operations to perform, in what order, and with what parameters. PostgreSQL, with its rich feature set spanning structured queries, vector similarity, full-text search, JSON document storage, and procedural logic, is unusually well-suited to serve as the operational backbone for agent systems. This chapter covers how to design PostgreSQL schemas and Python applications so that agents can treat your database as a first-class tool — reading context, writing state, searching knowledge, and coordinating with other systems.

---

What It Means for an Agent to Use a Database

Classical applications treat databases as passive storage: the code decides what to do, then persists or retrieves data. Agents invert this relationship. The agent is given a toolkit of available operations and decides — at inference time — which tools to invoke, with what arguments, and how to chain them together to complete a goal.

A database tool from an agent's perspective is a callable function with a defined schema: it takes structured inputs, performs a well-bounded operation, and returns structured outputs the agent can reason about. The agent doesn't know your internal schema; it knows the tool's interface. PostgreSQL's capabilities map naturally onto agent tools:

| Agent capability | PostgreSQL feature |

|---|---|

| Long-term memory | Persistent tables with structured recall |

| Semantic search | pgvector + cosine_distance |

| Fact lookup | Indexed relational tables |

| State tracking | JSONB columns with partial updates |

| Event signaling | pg_notify / LISTEN |

| Audit trail | Append-only log tables |

| Multi-agent coordination | Row-level locking, status columns |

The architecture of an agent using PostgreSQL as tooling has three layers:

1. Tool definitions — Python functions registered with the LLM framework as callable tools

2. Tool implementations — the actual database operations, query logic, and result formatting

3. PostgreSQL schema — the tables, indexes, functions, and constraints that make the operations safe and fast

Everything flows through this stack. The LLM sees tool names and JSON schemas; your Python code translates those into SQL; PostgreSQL executes and returns results that get fed back into the model context window.

Designing the Agent State Schema

Before writing any agent logic, the database schema needs to support agent workflows. An agent must be able to track its own execution context: what task is it working on, what has it already tried, what did it learn, and what decisions did it make? This is distinct from application user data — it's metadata about agent runs themselves.

-- Core agent infrastructure schema
-- Run this migration to set up agent state tracking

CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS vector;

-- Agent runs: one row per invocation of an agent
CREATE TABLE agent_runs (
    run_id          UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    agent_type      TEXT NOT NULL,               -- 'research', 'sql_gen', 'support', etc.
    user_id         UUID REFERENCES users(id),
    session_id      UUID NOT NULL,
    status          TEXT NOT NULL DEFAULT 'running'
                    CHECK (status IN ('running', 'paused', 'completed', 'failed', 'cancelled')),
    goal            TEXT NOT NULL,               -- the original task description
    context         JSONB NOT NULL DEFAULT '{}', -- arbitrary agent-specific context
    tool_budget     INTEGER NOT NULL DEFAULT 20, -- max tool calls allowed
    tools_used      INTEGER NOT NULL DEFAULT 0,
    started_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    completed_at    TIMESTAMPTZ,
    final_answer    TEXT,
    error_message   TEXT,
    model_used      TEXT NOT NULL DEFAULT 'claude-opus-4-5',
    total_tokens    INTEGER,
    metadata        JSONB NOT NULL DEFAULT '{}'
);

CREATE INDEX idx_agent_runs_session ON agent_runs (session_id);
CREATE INDEX idx_agent_runs_status ON agent_runs (status) WHERE status = 'running';
CREATE INDEX idx_agent_runs_user ON agent_runs (user_id, started_at DESC);

-- Individual tool calls made during an agent run
CREATE TABLE agent_tool_calls (
    call_id         UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    run_id          UUID NOT NULL REFERENCES agent_runs(run_id) ON DELETE CASCADE,
    tool_name       TEXT NOT NULL,
    sequence_num    INTEGER NOT NULL,            -- order within the run
    input_args      JSONB NOT NULL,
    raw_result      JSONB,
    error           TEXT,
    duration_ms     INTEGER,
    called_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (run_id, sequence_num)
);

CREATE INDEX idx_tool_calls_run ON agent_tool_calls (run_id, sequence_num);
CREATE INDEX idx_tool_calls_name ON agent_tool_calls (tool_name, called_at DESC);

-- Agent memory: facts the agent discovered and wants to retain across runs
CREATE TABLE agent_memory (
    memory_id       UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    agent_type      TEXT NOT NULL,
    user_id         UUID REFERENCES users(id),
    memory_type     TEXT NOT NULL               -- 'fact', 'preference', 'failed_approach', 'entity'
                    CHECK (memory_type IN ('fact', 'preference', 'failed_approach', 'entity', 'summary')),
    content         TEXT NOT NULL,
    content_vec     VECTOR(1536),               -- embedding for semantic retrieval
    source_run_id   UUID REFERENCES agent_runs(run_id),
    confidence      FLOAT NOT NULL DEFAULT 1.0 CHECK (confidence BETWEEN 0 AND 1),
    access_count    INTEGER NOT NULL DEFAULT 0,
    last_accessed   TIMESTAMPTZ,
    expires_at      TIMESTAMPTZ,               -- NULL = permanent
    metadata        JSONB NOT NULL DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_agent_memory_user_type ON agent_memory (user_id, agent_type, memory_type);
CREATE INDEX idx_agent_memory_vec ON agent_memory
    USING hnsw (content_vec vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
CREATE INDEX idx_agent_memory_expiry ON agent_memory (expires_at)
    WHERE expires_at IS NOT NULL;

-- Shared agent workspace: structured data agents produce or consume
CREATE TABLE agent_artifacts (
    artifact_id     UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    run_id          UUID REFERENCES agent_runs(run_id) ON DELETE CASCADE,
    artifact_type   TEXT NOT NULL,              -- 'sql_query', 'report', 'data_extract', 'plan'
    name            TEXT NOT NULL,
    content         TEXT NOT NULL,
    content_json    JSONB,                      -- structured version when available
    embedding       VECTOR(1536),
    is_final        BOOLEAN NOT NULL DEFAULT FALSE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_artifacts_run ON agent_artifacts (run_id, artifact_type);

This schema gives you end-to-end observability over agent behavior — what did it try, in what order, for how long, and what did it conclude? The agent_memory table with its vector index allows agents to retrieve relevant past knowledge using semantic search rather than exact key lookups.

The tool_budget column is worth highlighting: uncontrolled agents can spiral into loops. Tracking tools_used and enforcing a hard budget at the database level (via a trigger, shown below) prevents runaway executions from consuming resources.

-- Trigger to enforce tool budget and auto-update run stats
CREATE OR REPLACE FUNCTION enforce_agent_tool_budget()
RETURNS TRIGGER AS $$
DECLARE
    v_budget        INTEGER;
    v_used          INTEGER;
    v_status        TEXT;
BEGIN
    SELECT tool_budget, tools_used, status
    INTO v_budget, v_used, v_status
    FROM agent_runs
    WHERE run_id = NEW.run_id
    FOR UPDATE;

    IF v_status != 'running' THEN
        RAISE EXCEPTION 'Cannot add tool call to run % with status %',
            NEW.run_id, v_status;
    END IF;

    IF v_used >= v_budget THEN
        UPDATE agent_runs
        SET status = 'failed',
            error_message = 'Tool budget exhausted after ' || v_used || ' calls',
            completed_at = NOW()
        WHERE run_id = NEW.run_id;

        RAISE EXCEPTION 'Agent run % exceeded tool budget of %',
            NEW.run_id, v_budget;
    END IF;

    -- Increment counter atomically
    UPDATE agent_runs
    SET tools_used = tools_used + 1
    WHERE run_id = NEW.run_id;

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_enforce_tool_budget
    BEFORE INSERT ON agent_tool_calls
    FOR EACH ROW
    EXECUTE FUNCTION enforce_agent_tool_budget();

Implementing PostgreSQL Tools for Agents

With the state schema in place, you can build the tool functions that agents actually call. These are Python functions wrapped with the tool-calling interface of your LLM SDK. Each tool function should do exactly one coherent thing, return clean structured data, and log its execution to agent_tool_calls.
