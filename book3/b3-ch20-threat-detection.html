Chapter 20: Autonomous Threat Detection

Security in a self-managing database platform cannot depend on a human analyst reviewing logs at the end of a shift. By the time an intrusion is confirmed, a privilege escalation has propagated, or an exfiltration query has run to completion, the damage is already bounded only by how long the attack went undetected. This chapter builds the architecture for a database system that detects, classifies, and responds to threats autonomously — closing the gap between signal and action from hours to seconds. The patterns here cover behavioral baselining, anomaly scoring, automated containment, and integration with broader security pipelines, all grounded in what PostgreSQL and SQL Server actually expose at the engine level.

---

20.1 The Threat Surface at the Engine Layer: What Autonomous Systems Must Monitor

Most security tooling lives above the database — WAF rules, network ACLs, SIEM ingestion of application logs. These are necessary but insufficient. An autonomous threat detection system must instrument the database engine itself because the most dangerous attacks succeed after the perimeter has already been crossed: stolen credentials, compromised application accounts, malicious insiders, or injected queries that bypass all upstream filtering.

The threat surface that matters for autonomous detection divides into four observable planes:

Authentication and session plane. Who is connecting, from where, at what time, using which authentication method. Both engines expose this. PostgreSQL records this in pg_stat_activity and the server log via log_connections, log_disconnections. SQL Server surfaces it through Extended Events and the sys.dm_exec_sessions DMV. An autonomous system tracks not just raw connection counts but behavioral fingerprints: which application users connect from which hosts, at what hour distribution, with what typical session duration.

Query execution plane. What SQL is running. This is where SQL injection, data exfiltration, and reconnaissance manifest. PostgreSQL pg_stat_statements captures normalized query text with execution statistics. SQL Server Query Store does the same. The autonomous system must detect structural anomalies — queries with unusual UNION counts, bulk SELECT * against sensitive tables, absence of WHERE clauses on large tables, and EXECUTE/xp_cmdshell patterns.

Privilege and schema mutation plane. DDL changes, role grants, ownership transfers. These are low-frequency operations in production that almost never happen outside of planned change windows. Any CREATE USER, GRANT, ALTER TABLE, DROP operation occurring outside of a sanctioned window is a high-confidence threat signal.

Data volume plane. Row counts and byte volumes returned by queries. A support engineer querying a customer table and returning 50 rows is normal. The same query returning 2 million rows is not. Neither PostgreSQL nor SQL Server exposes per-query row return counts in a single DMV, so this plane requires instrumentation at the connection pooler or proxy layer — pgBouncer statement logging, SQL Server Audit, or a eBPF-based collector at the network layer.

The autonomous threat detection system must monitor all four planes simultaneously, correlate signals across them, and produce a threat score that triggers graduated responses. A single anomalous signal is noise. The same session showing unusual connection time, a novel query pattern, and a high row-return volume simultaneously is an incident.

20.2 Behavioral Baselining: Building the Normal Model Autonomously

Detection requires a model of normal. Static rule systems — blacklists of bad query patterns, IP allowlists — fail against sophisticated attackers who operate within the envelope of legitimate access. Behavioral baselining builds a statistical model of normal operation that evolves continuously as the workload changes.

The baseline must be per-identity, not aggregate. The behavior of app_user is not the behavior of reporting_user. The autonomous system maintains a profile per database principal covering:

  • → Typical connection hour distribution (histogram by hour-of-day)
  • → Source host/IP distribution
  • → Average and 99th-percentile query execution time
  • → Query template frequency (normalized `pg_stat_statements` queryid)
  • → Tables accessed per session
  • → Average rows examined and returned

    • PostgreSQL:

    -- Create baseline table for per-user query behavior
CREATE TABLE security.query_baseline (
    usename          TEXT NOT NULL,
    query_id         BIGINT NOT NULL,
    normalized_query TEXT,
    calls_per_hour   NUMERIC,
    avg_rows         NUMERIC,
    stddev_rows      NUMERIC,
    avg_exec_ms      NUMERIC,
    stddev_exec_ms   NUMERIC,
    first_seen       TIMESTAMPTZ,
    last_updated     TIMESTAMPTZ DEFAULT now(),
    PRIMARY KEY (usename, query_id)
);

-- Populate/refresh baseline from pg_stat_statements (run on schedule)
INSERT INTO security.query_baseline (
    usename, query_id, normalized_query,
    calls_per_hour, avg_rows, stddev_rows,
    avg_exec_ms, stddev_exec_ms, first_seen, last_updated
)
SELECT
    r.rolname                          AS usename,
    s.queryid,
    s.query                            AS normalized_query,
    s.calls / NULLIF(
        EXTRACT(EPOCH FROM (now() - stats_reset)) / 3600, 0
    )                                  AS calls_per_hour,
    s.rows / NULLIF(s.calls, 0)        AS avg_rows,
    -- stddev approximated from total_exec_time variance proxy
    sqrt(s.total_exec_time / NULLIF(s.calls, 0)
         - power(s.mean_exec_time, 2)) AS stddev_exec_ms,
    s.mean_exec_time                   AS avg_exec_ms,
    NULL                               AS stddev_exec_ms_placeholder,
    now()                              AS first_seen,
    now()                              AS last_updated
FROM pg_stat_statements s
JOIN pg_roles r ON r.oid = s.userid
ON CONFLICT (usename, query_id) DO UPDATE
    SET calls_per_hour  = EXCLUDED.calls_per_hour,
        avg_rows        = EXCLUDED.avg_rows,
        avg_exec_ms     = EXCLUDED.avg_exec_ms,
        last_updated    = now();

    SQL Server:

    -- Baseline table for per-login query behavior
CREATE TABLE SecurityBaseline.QueryProfile (
    LoginName       NVARCHAR(128) NOT NULL,
    QueryHash       BINARY(8)     NOT NULL,
    QueryText       NVARCHAR(MAX),
    AvgRowsReturned FLOAT,
    StdDevRows      FLOAT,
    AvgDurationMs   FLOAT,
    StdDevDurationMs FLOAT,
    CallsPerHour    FLOAT,
    FirstSeen       DATETIME2,
    LastUpdated     DATETIME2 DEFAULT SYSUTCDATETIME(),
    CONSTRAINT PK_QueryProfile PRIMARY KEY (LoginName, QueryHash)
);

-- Refresh from Query Store
MERGE SecurityBaseline.QueryProfile AS tgt
USING (
    SELECT
        es.login_name,
        qs.query_hash,
        qt.query_sql_text,
        AVG(CAST(rs.avg_rowcount AS FLOAT))        AS avg_rows,
        STDEV(CAST(rs.avg_rowcount AS FLOAT))      AS stddev_rows,
        AVG(rs.avg_duration / 1000.0)              AS avg_duration_ms,
        STDEV(rs.avg_duration / 1000.0)            AS stddev_duration_ms,
        COUNT(*) * 1.0 /
            NULLIF(DATEDIFF(HOUR,
                MIN(rs.last_execution_time),
                MAX(rs.last_execution_time)), 0)   AS calls_per_hour
    FROM sys.query_store_query          qs
    JOIN sys.query_store_query_text     qt ON qt.query_text_id = qs.query_text_id
    JOIN sys.query_store_plan           qp ON qp.query_id      = qs.query_id
    JOIN sys.query_store_runtime_stats  rs ON rs.plan_id       = qp.plan_id
    -- join to current sessions is best-effort; use audit log for persistent mapping
    CROSS JOIN sys.dm_exec_sessions     es
    WHERE es.is_user_process = 1
    GROUP BY es.login_name, qs.query_hash, qt.query_sql_text
) AS src
ON tgt.LoginName = src.login_name
   AND tgt.QueryHash = src.query_hash
WHEN MATCHED THEN UPDATE SET
    AvgRowsReturned  = src.avg_rows,
    StdDevRows       = src.stddev_rows,
    AvgDurationMs    = src.avg_duration_ms,
    StdDevDurationMs = src.stddev_duration_ms,
    CallsPerHour     = src.calls_per_hour,
    LastUpdated      = SYSUTCDATETIME()
WHEN NOT MATCHED THEN INSERT (
    LoginName, QueryHash, QueryText, AvgRowsReturned,
    StdDevRows, AvgDurationMs, StdDevDurationMs,
    CallsPerHour, FirstSeen, LastUpdated
) VALUES (
    src.login_name, src.query_hash, src.query_sql_text,
    src.avg_rows, src.stddev_rows,
    src.avg_duration_ms, src.stddev_duration_ms,
    src.calls_per_hour, SYSUTCDATETIME(), SYSUTCDATETIME()
);

    The baseline refresh runs on a scheduled job — every 6 hours during stable periods, paused during incident response so the attack behavior does not contaminate the model. The autonomous system tracks first_seen to exclude newly observed query templates from scoring until they have accumulated at least 7 days of data; new templates are flagged as *novel* but not scored as threats until a minimum observation window passes.

    Connection-hour histograms are stored separately as JSON arrays — one cell per hour of the week (168 cells) — with exponential decay weighting so that recent behavior has higher influence than patterns from 90 days ago. The decay parameter (α = 0.05 per day) means the effective half-life is about 14 days, which tracks seasonal workload changes without being so volatile that a week of unusual traffic permanently distorts the model.

    20.3 Real-Time Anomaly Scoring at the Session Level

    With a behavioral baseline in place, the scoring engine evaluates every active session continuously. Scoring runs as a Python process that polls both engines every 30 seconds, computes Z-scores for observed session behavior against the baseline, and produces a composite threat score per session.

    The scoring formula weights four dimensions:

  • → `Z_time`: How far outside the normal hour distribution this connection falls
  • → `Z_rows`: Standard deviations above mean row return for the query being executed
  • → `Z_query_novelty`: Binary flag (0/1) — is this query template absent from the baseline?
  • → `Z_volume`: Aggregate data scanned in this session vs. the user's historical session average
    • import psycopg2
import pyodbc
import json
import math
import time
from dataclasses import dataclass, field
from datetime import datetime, timezone
from typing import Dict, Optional
import logging

logging.basicConfig(level=logging.INFO, format="%(asctime)s %(levelname)s %(message)s")
logger = logging.getLogger("threat_scorer")

THREAT_THRESHOLD_WARN  = 2.5   # Z-score composite: raise alert
THREAT_THRESHOLD_ACT   = 4.0   # Z-score composite: autonomous action
DECAY_ALPHA            = 0.05  # per-day exponential decay for baseline

@dataclass
class SessionProfile:
    session_id: str
    username: str
    client_addr: str
    query_id: Optional[int]
    rows_returned: int
    exec_ms: float
    hour_of_week: int           # 0–167

@dataclass
class ThreatScore:
    session_id: str
    username: str
    composite_score: float
    components: Dict[str, float]
    triggered_at: datetime = field(default_factory=lambda: datetime.now(timezone.utc))

def z_score(value: float, mean: float, stddev: float) -> float:
    """Return Z-score clamped to [0, 10]; negative deviations are not threats."""
    if stddev is None or stddev < 1e-9:
        return 0.0 if abs(value - mean) < 1e-9 else 5.0  # no variance → novel
    return max(0.0, (value - mean) / stddev)

def hour_distribution_score(hour: int, histogram: list[float]) -> float:
    """
    Score how anomalous this hour is relative to the user's historical distribution.
    Returns 0 (normal) to 10 (never seen at this hour).
    """
    if not histogram or len(histogram) != 168:
        return 2.0   # no data: mildly suspicious
    freq = histogram[hour]
    total = sum(histogram)
    if total < 1e-9:
        return 2.0
    expected = freq / total
    if expected < 0.001:   # less than 0.1% of sessions at this hour
        return 8.0
    if expected < 0.01:
        return 4.0
    return 0.0

def score_session(session: SessionProfile, baseline_row: Optional[dict]) -> ThreatScore:
    components = {}

    if baseline_row is None:
        # Novel user or query — no baseline to compare against
        components["novelty"]  = 6.0
        components["rows"]     = 0.0
        components["exec_ms"]  = 0.0
        components["time"]     = 0.0
    else:
        histogram = json.loads(baseline_row["hour_histogram"] or "[]")
        components["time"]    = hour_distribution_score(session.hour_of_week, histogram)
        components["rows"]    = z_score(
            session.rows_returned,
            baseline_row["avg_rows"]    or 0,
            baseline_row["stddev_rows"] or 0
        )
        components["exec_ms"] = z_score(
            session.exec_ms,
            baseline_row["avg_exec_ms"]    or 0,
            baseline_row["stddev_exec_ms"] or 0
        )
        components["novelty"] = 0.0

    # Weighted composite
    composite = (
        components.get("rows",    0) * 0.40 +
        components.get("time",    0) * 0.25 +
        components.get("exec_ms", 0) * 0.20 +
        components.get("novelty", 0) * 0.15
    )

    return ThreatScore(
        session_id=session.session_id,
        username=session.username,
        composite_score=round(composite, 3),
        components=components
    )

def poll_and_score_postgres(conn_pg, baseline_cache: dict) -> list[ThreatScore]:
    scores = []
    now_dow  = datetime.now(timezone.utc).weekday()    # 0=Mon
    now_hour = datetime.now(timezone.utc).hour
    hour_of_week = now_dow * 24 + now_hour

    with conn_pg.cursor() as cur:
        cur.execute("""
            SELECT
                pid::TEXT           AS session_id,
                usename             AS username,
                client_addr::TEXT   AS client_addr,
                queryid             AS query_id,
                0                   AS rows_returned,   -- enriched separately
                EXTRACT(EPOCH FROM (clock_timestamp() - query_start)) * 1000
                                    AS exec_ms
            FROM pg_stat_activity a
            LEFT JOIN pg_stat_statements s
                   ON s.userid = a.usesysid
                  AND s.query  = a.query
            WHERE state = 'active'
              AND usename NOT IN ('postgres', 'replication')
        """)
        rows = cur.fetchall()
        cols = [d[0] for d in cur.description]

    for row in rows:
        r = dict(zip(cols, row))
        session = SessionProfile(
            session_id   = r["session_id"],
            username     = r["username"],
            client_addr  = r["client_addr"] or "unknown",
            query_id     = r["query_id"],
            rows_returned= r["rows_returned"],
            exec_ms      = r["exec_ms"] or 0.0,
            hour_of_week = hour_of_week
        )
        baseline = baseline_cache.get((r["username"], r["query_id"]))
        scores.append(score_session(session, baseline))

    return scores

def dispatch_threat_response(score: ThreatScore, severity: str):
    logger.warning(
        "THREAT %s | session=%s user=%s score=%.2f components=%s",
        severity, score.session_id, score.username,
        score.composite_score, score.components
    )
    # Downstream: push to Prometheus Alertmanager, PagerDuty, SIEM
    # For ACT-level threats: trigger terminate_session() in 20.4

if __name__ == "__main__":
    pg_conn = psycopg2.connect(dsn="host=localhost dbname=prod user=monitor")
    baseline_cache: dict = {}    # populated from security.query_baseline on startup

    while True:
        scores = poll_and_score_postgres(pg_conn, baseline_cache)
        for s in scores:
            if s.composite_score >= THREAT_THRESHOLD_ACT:
                dispatch_threat_response(s, "ACT")
            elif s.composite_score >= THREAT_THRESHOLD_WARN:
                dispatch_threat_response(s, "WARN")
        time.sleep(30)

    The scoring loop runs every 30 seconds per engine. For SQL Server the same logic applies but pulls from sys.dm_exec_requests and sys.dm_exec_sessions rather than pg_stat_activity. The composite score crosses two thresholds: WARN (2.5) triggers alerting to the SIEM and on-call channel; ACT (4.0) triggers autonomous containment actions described in the next section.

    Threshold calibration matters. In the first 30 days after deployment on a new cluster, the system operates in *shadow mode* — it scores and logs but does not act. The false positive rate is measured as alerts that an operator manually classifies as non-threats. The ACT threshold is tuned upward until the false positive rate drops below 1 per week on steady-state workloads, then autonomous action is enabled.

    20.4 Autonomous Containment: From Detection to Response Without Human Intervention

    Detection without response is just expensive logging. The autonomous containment layer closes the loop: when a threat score crosses the ACT threshold, the system takes a graduated sequence of actions proportional to confidence and risk. The escalation ladder has four levels:

    Level 1 — Rate-limit the session. For scores between 4.0 and 5.5, the system does not terminate the session but reduces its resource access. In PostgreSQL, this means dynamically setting a low statement_timeout and work_mem for the offending role. In SQL Server, it means placing the login in a Resource Governor pool capped at 10% CPU and a low memory grant ratio.

    Level 2 — Terminate the active query. For scores between 5.5 and 7.0, the specific query is killed. The session remains active; the system monitors for recurrence. Two Level-2 terminations within a 5-minute window auto-escalate to Level 3.

    Level 3 — Terminate the session. For scores above 7.0, or after Level-2 escalation, the session is forcibly terminated and the user's ability to establish new connections is suspended for a configurable quarantine window (default: 15 minutes).

    Level 4 — Revoke and isolate. For the highest-confidence threats — novel query + off-hours connection + bulk data access simultaneously — the system revokes the role's login privilege entirely and creates a forensic snapshot of what was accessed.

    PostgreSQL:

    -- Level 1: Rate-limit suspicious session dynamically
-- Executed by the monitoring user with pg_signal_backend privilege
SELECT pg_cancel_backend(pid)              -- cancel current query
FROM pg_stat_activity
WHERE pid = :target_pid;

-- Apply restrictive settings to the role
ALTER ROLE suspicious_user
    SET statement_timeout    = '5s'
    SET work_mem             = '4MB'
    SET idle_in_transaction_session_timeout = '30s';

-- Level 3: Terminate session and block new connections
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE usename = :target_user
  AND pid <> pg_backend_pid();

-- Revoke login (Level 4)
REVOKE CONNECT ON DATABASE prod FROM suspicious_user;

-- Forensic snapshot: record what tables were accessed in this session
INSERT INTO security.access_forensics (
    incident_id, username, query_id,
    query_text, tables_accessed, captured_at
)
SELECT
    :incident_id,
    s.usename,
    st.queryid,
    st.query,
    -- pg_stat_statements doesn't expose table-level detail; supplement with audit log
    NULL,
    now()
FROM pg_stat_activity   s
JOIN pg_stat_statements st ON st.userid = s.usesysid
WHERE s.usename = :target_user;

    SQL Server:

    -- Level 1: Move login to restricted Resource Governor pool
-- (assumes pool 'ThreatContainment' is pre-configured with max CPU 10%)
ALTER RESOURCE GOVERNOR RECONFIGURE;  -- classifier function re-evaluated on next connection

-- Level 2: Kill specific request
KILL :session_id;   -- session_id from sys.dm_exec_sessions

-- Level 3: Disable the login entirely
ALTER LOGIN [suspicious_user] DISABLE;

-- Terminate all existing sessions for this login
DECLARE @sql NVARCHAR(512);
SELECT @sql = STRING_AGG('KILL ' + CAST(session_id AS VARCHAR), '; ')
FROM sys.dm_exec_sessions
WHERE login_name = 'suspicious_user'
  AND is_user_process = 1;

IF @sql IS NOT NULL
    EXEC sp_executesql @sql;

-- Level 4 forensic snapshot using Query Store
INSERT INTO SecurityForensics.IncidentQueryLog (
    IncidentId, LoginName, QueryHash, QueryText,
    ExecutionCount, LastExecutionTime, CapturedAt
)
SELECT
    @IncidentId,
    'suspicious_user',
    qs.query_hash,
    qt.query_sql_text,
    SUM(rs.count_executions),
    MAX(rs.last_execution_time),
    SYSUTCDATETIME()
FROM sys.query_store_query         qs
JOIN sys.query_store_query_text    qt ON qt.query_text_id  = qs.query_text_id
JOIN sys.query_store_plan          qp ON qp.query_id       = qs.query_id
JOIN sys.query_store_runtime_stats rs ON rs.plan_id        = qp.plan_id
WHERE qs.last_execution_time > DATEADD(HOUR, -1, SYSUTCDATETIME())
  -- filter to this login via audit correlation in production
GROUP BY qs.query_hash, qt.query_sql_text;

    The autonomous containment logic integrates with the scoring engine through a simple state machine. Each session has a containment state (CLEAR, MONITORING, RATE_LIMITED, TERMINATED, QUARANTINED). State transitions are append-only to an audit table; no state is silently overwritten. This creates an immutable incident timeline that satisfies SOC-2 and ISO 27001 audit requirements — the autonomous system's every action is traceable.
