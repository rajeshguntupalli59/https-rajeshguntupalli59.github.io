Chapter 16: Autonomous Scaling in Cloud Databases

Scaling decisions made by humans are slow, reactive, and constrained by organizational bandwidth. A senior DBA wakes up to an on-call alert, interprets a spike, provisions additional capacity, and reruns workloads — by which time the damage is done. Autonomous scaling flips this model entirely: the system observes its own resource consumption, predicts demand trajectories, and executes capacity changes without human approval. This chapter covers the architecture and implementation of autonomous vertical and horizontal scaling for both PostgreSQL and SQL Server in cloud environments, including the signal pipelines that drive decisions, the execution layers that implement them, and the guardrails that prevent runaway automation from causing worse problems than it solves.

---

The Signal Pipeline: From Metric Collection to Scaling Decisions

Autonomous scaling begins not with the scaling action itself but with the quality of the signals feeding the decision engine. A system that scales on raw CPU percentage will thrash. A system that scales on composite signals — query latency percentiles, buffer hit ratios, replication lag, active connection depth, and queue depth simultaneously — makes durable decisions.

The canonical metric stack for autonomous scaling uses Prometheus for collection, a time-series anomaly model (ARIMA, Prophet, or a lightweight LSTM) for prediction, and a rule engine that translates predictions into scaling intents.

For PostgreSQL, the baseline signals come from pg_stat_activity, pg_stat_bgwriter, pg_stat_replication, and the pg_buffercache extension. For SQL Server, equivalent depth comes from Dynamic Management Views: sys.dm_exec_requests, sys.dm_os_wait_stats, sys.dm_os_ring_buffers, and sys.dm_db_resource_stats on Azure SQL.

PostgreSQL:

-- Composite health signal query for the scaling decision engine
SELECT
    (SELECT count(*) FROM pg_stat_activity
     WHERE state = 'active' AND wait_event_type IS NOT NULL) AS blocked_queries,
    (SELECT count(*) FROM pg_stat_activity
     WHERE state = 'active') AS active_queries,
    (SELECT round(100.0 * sum(heap_blks_hit) /
        NULLIF(sum(heap_blks_hit) + sum(heap_blks_read), 0), 2)
     FROM pg_statio_user_tables) AS buffer_hit_ratio,
    (SELECT EXTRACT(EPOCH FROM (now() - pg_last_xact_replay_timestamp()))
     FROM pg_stat_replication LIMIT 1) AS replica_lag_seconds,
    (SELECT max(EXTRACT(EPOCH FROM (now() - query_start)))
     FROM pg_stat_activity
     WHERE state = 'active' AND query NOT ILIKE '%pg_stat%') AS max_query_duration_sec;

SQL Server:

-- Composite health signal query for the scaling decision engine
SELECT
    (SELECT COUNT(*) FROM sys.dm_exec_requests
     WHERE blocking_session_id <> 0) AS blocked_queries,
    (SELECT COUNT(*) FROM sys.dm_exec_requests
     WHERE status = 'running') AS active_queries,
    (SELECT
        CAST(100.0 * SUM(CASE WHEN physical_reads = 0 THEN 1 ELSE 0 END)
            / NULLIF(COUNT(*), 0) AS DECIMAL(5,2))
     FROM sys.dm_exec_requests
     WHERE status = 'running') AS logical_read_ratio,
    (SELECT TOP 1 DATEDIFF(SECOND, last_commit_time,
        secondary_last_commit_time)
     FROM sys.dm_hadr_database_replica_states
     WHERE is_local = 0) AS replica_lag_seconds,
    (SELECT MAX(DATEDIFF(SECOND, start_time, GETDATE()))
     FROM sys.dm_exec_requests
     WHERE status IN ('running','suspended')) AS max_query_duration_sec;

These two queries form the heartbeat of the scaling signal pipeline. A Python collector runs them on a 15-second cadence, pushes results to Prometheus via the prometheus_client library, and feeds a sliding window evaluator that determines whether observed values represent a transient spike or a sustained load pattern requiring a scaling response.

The distinction between a spike and a trend is the most important judgment the scaling engine must make. Spikes warrant connection pool adjustments or query prioritization changes. Sustained trends warrant compute scaling. The window evaluator applies a simple rule: if the composite score (a weighted average of the normalized metric deviations) exceeds 0.75 for three consecutive five-minute evaluation windows, a scaling intent is raised. A single window breach does not trigger scaling; it resets a cooldown timer and increases monitoring frequency.

Vertical Scaling Automation: Compute Resize Without Downtime

Vertical scaling — adding vCPUs or memory to an existing instance — is the first lever most cloud database architectures pull because it requires no application topology changes. The challenge is executing it with zero or near-zero downtime while maintaining ACID guarantees throughout the resize window.

On Azure SQL Database and Azure Database for PostgreSQL Flexible Server, compute tier changes are API-driven and can be automated via Terraform or direct REST calls. On AWS RDS, instance class modifications support the apply-immediately flag for minor version changes but require a maintenance window for major compute class changes unless using RDS Multi-AZ, where the standby is upgraded first and a failover is executed transparently.

The autonomous scaling controller must understand these platform-specific behaviors. A platform-agnostic abstraction layer is dangerous here; the controller needs to know whether the target platform supports online resize before issuing the command.

# Autonomous vertical scaling controller
# Handles both Azure Database for PostgreSQL and AWS RDS

import boto3
import requests
import json
import time
import logging
from dataclasses import dataclass
from enum import Enum

logger = logging.getLogger("autonomous_scaler")

class Platform(Enum):
    AWS_RDS = "aws_rds"
    AZURE_PG_FLEXIBLE = "azure_pg_flexible"
    AZURE_SQL = "azure_sql"

@dataclass
class ScalingIntent:
    platform: Platform
    resource_id: str
    current_sku: str
    target_sku: str
    trigger_metric: str
    trigger_value: float
    threshold: float
    cooldown_minutes: int = 30

class VerticalScalingController:
    def __init__(self, azure_token: str, aws_region: str):
        self.azure_token = azure_token
        self.rds_client = boto3.client('rds', region_name=aws_region)
        self.last_scale_time = {}

    def _is_in_cooldown(self, resource_id: str, cooldown_minutes: int) -> bool:
        if resource_id not in self.last_scale_time:
            return False
        elapsed = (time.time() - self.last_scale_time[resource_id]) / 60
        return elapsed < cooldown_minutes

    def execute_scaling(self, intent: ScalingIntent) -> dict:
        if self._is_in_cooldown(intent.resource_id, intent.cooldown_minutes):
            logger.info(f"Resource {intent.resource_id} in cooldown. Skipping.")
            return {"status": "skipped", "reason": "cooldown"}

        logger.warning(
            f"Scaling {intent.resource_id}: {intent.current_sku} -> {intent.target_sku} "
            f"[trigger: {intent.trigger_metric}={intent.trigger_value:.2f}, "
            f"threshold={intent.threshold}]"
        )

        if intent.platform == Platform.AWS_RDS:
            return self._scale_rds(intent)
        elif intent.platform in (Platform.AZURE_PG_FLEXIBLE, Platform.AZURE_SQL):
            return self._scale_azure(intent)

    def _scale_rds(self, intent: ScalingIntent) -> dict:
        try:
            response = self.rds_client.modify_db_instance(
                DBInstanceIdentifier=intent.resource_id,
                DBInstanceClass=intent.target_sku,
                ApplyImmediately=True  # safe for Multi-AZ; failover handles HA
            )
            self.last_scale_time[intent.resource_id] = time.time()
            return {
                "status": "initiated",
                "db_instance_status": response['DBInstance']['DBInstanceStatus']
            }
        except Exception as e:
            logger.error(f"RDS scaling failed: {e}")
            return {"status": "failed", "error": str(e)}

    def _scale_azure(self, intent: ScalingIntent) -> dict:
        # Azure Database for PostgreSQL Flexible Server SKU update
        parts = intent.resource_id.split("/")
        subscription_id = parts[2]
        resource_group = parts[4]
        server_name = parts[-1]

        url = (
            f"https://management.azure.com/subscriptions/{subscription_id}"
            f"/resourceGroups/{resource_group}"
            f"/providers/Microsoft.DBforPostgreSQL/flexibleServers/{server_name}"
            f"?api-version=2022-12-01"
        )
        payload = {"sku": {"name": intent.target_sku, "tier": "GeneralPurpose"}}
        headers = {
            "Authorization": f"Bearer {self.azure_token}",
            "Content-Type": "application/json"
        }
        response = requests.patch(url, headers=headers, data=json.dumps(payload))
        if response.status_code in (200, 202):
            self.last_scale_time[intent.resource_id] = time.time()
            return {"status": "initiated", "http_status": response.status_code}
        return {"status": "failed", "http_status": response.status_code,
                "body": response.text}

The cooldown mechanism is non-negotiable. Without it, a scaling controller can enter a feedback loop: scale up triggers a brief latency increase during the resize, which looks like continued stress, which triggers another scale-up, until the instance reaches the maximum available SKU. Thirty minutes is a sensible default; production systems may tune this based on the platform's typical resize completion time plus a safety buffer.

For SQL Server on RDS or Azure SQL Managed Instance, vertical scaling follows identical API patterns but operates on different SKU nomenclature. Azure SQL Elastic Pools support dynamic DTU or vCore allocation within a pool without instance restarts — making pool-level scaling the preferred lever when multiple databases share infrastructure.

Horizontal Scaling: Read Replica Automation and Sharding Orchestration

Vertical scaling has a ceiling. When the workload outgrows the largest available instance class, or when write latency targets require distributing read traffic, horizontal scaling becomes the operative architecture. Autonomous horizontal scaling for PostgreSQL and SQL Server takes two distinct forms: replica fleet management and, for write scalability, shard orchestration.

Autonomous Read Replica Management

The replica fleet should grow and shrink based on read queue depth and replica lag. When replica lag exceeds a defined threshold — say, 5 seconds — adding another replica reduces the per-replica replication fan-out, indirectly reducing lag on existing replicas while absorbing additional read traffic. When replica lag returns to acceptable levels and the replica connection utilization drops below 30% for two consecutive evaluation windows, the controller decommissions the youngest replica.

On Kubernetes, this is expressed through a custom operator pattern. The operator watches a DatabaseScalingPolicy custom resource and controls the replica count of a PostgresCluster (via CrunchyData PGO or Zalando postgres-operator).

For pgBouncer, horizontal scaling of the connection proxy layer accompanies every replica addition. Each new replica gets its own pgBouncer sidecar in transaction pooling mode, and the application-layer load balancer (HAProxy or Envoy) is updated to include the new replica endpoint with read-only routing tags.

PostgreSQL:

-- Replication health query used by the replica autoscaler
-- Run against each standby's local pg_stat_replication view via dblink
SELECT
    application_name,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    ROUND(
        (EXTRACT(EPOCH FROM write_lag) +
         EXTRACT(EPOCH FROM flush_lag) +
         EXTRACT(EPOCH FROM replay_lag)) / 3.0, 3
    ) AS avg_lag_seconds,
    sync_state
FROM pg_stat_replication
ORDER BY avg_lag_seconds DESC NULLS LAST;

SQL Server:

-- Replica health for Availability Group autoscaler
SELECT
    ar.replica_server_name,
    drs.database_name,
    drs.synchronization_state_desc,
    drs.synchronization_health_desc,
    drs.log_send_queue_size AS log_send_queue_kb,
    drs.redo_queue_size AS redo_queue_kb,
    drs.log_send_rate AS log_send_rate_kb_sec,
    drs.redo_rate AS redo_rate_kb_sec,
    CASE
        WHEN drs.log_send_rate > 0
        THEN CAST(drs.log_send_queue_size * 1.0
                  / drs.log_send_rate AS DECIMAL(10,2))
        ELSE NULL
    END AS estimated_lag_seconds
FROM sys.dm_hadr_database_replica_states drs
JOIN sys.availability_replicas ar
    ON drs.replica_id = ar.replica_id
WHERE drs.is_local = 0
ORDER BY estimated_lag_seconds DESC NULLS LAST;

Write Scalability and Shard Orchestration

For write-bound workloads, Citus (now part of Azure Cosmos DB for PostgreSQL) provides shard-level horizontal scaling for PostgreSQL. The autonomous layer here orchestrates shard rebalancing in response to per-shard row count skew and per-shard query latency divergence. When a shard's query p99 exceeds 2× the fleet median for two evaluation periods, the rebalancer moves that shard to a less-loaded worker node — transparently to the application.

# Autonomous shard rebalancer for Citus / Azure Cosmos DB for PostgreSQL
# Triggers rebalancing when shard p99 latency diverges beyond threshold

import psycopg2
import logging

logger = logging.getLogger("shard_rebalancer")

COORDINATOR_DSN = "host=coordinator port=5432 dbname=app user=citus_admin"
P99_DIVERGENCE_THRESHOLD = 2.0  # target shard p99 must not exceed 2x fleet median
MIN_SHARD_SIZE_REBALANCE_MB = 500  # avoid rebalancing tiny shards

def get_shard_latency_profile(conn):
    with conn.cursor() as cur:
        cur.execute("""
            SELECT
                s.shardid,
                s.logicalrelid::text AS table_name,
                sp.nodename,
                sp.nodeport,
                pg_size_pretty(citus_shard_sizes.size) AS shard_size,
                citus_shard_sizes.size AS size_bytes
            FROM pg_dist_shard s
            JOIN pg_dist_shard_placement sp ON s.shardid = sp.shardid
            JOIN citus_shard_sizes() AS citus_shard_sizes(shardid, size)
                ON s.shardid = citus_shard_sizes.shardid
            WHERE sp.shardstate = 1
            ORDER BY size_bytes DESC
        """)
        return cur.fetchall()

def trigger_rebalance_if_needed(conn, shard_profile):
    sizes = [row[5] for row in shard_profile]
    if not sizes:
        return

    median_size = sorted(sizes)[len(sizes) // 2]
    oversized_shards = [
        row for row in shard_profile
        if row[5] > median_size * P99_DIVERGENCE_THRESHOLD
        and row[5] > MIN_SHARD_SIZE_REBALANCE_MB * 1024 * 1024
    ]

    if oversized_shards:
        logger.warning(
            f"Triggering shard rebalance: {len(oversized_shards)} oversized shards "
            f"detected (median={median_size / 1024 / 1024:.1f}MB)"
        )
        with conn.cursor() as cur:
            # Citus rebalancer runs asynchronously
            cur.execute("SELECT rebalance_table_shards(rebalance_strategy := 'by_shard_count');")
        conn.commit()
        logger.info("Shard rebalance initiated successfully.")
    else:
        logger.debug("Shard distribution within acceptable bounds. No action required.")

if __name__ == "__main__":
    conn = psycopg2.connect(COORDINATOR_DSN)
    profile = get_shard_latency_profile(conn)
    trigger_rebalance_if_needed(conn, profile)
    conn.close()

SQL Server's horizontal scaling story differs architecturally. Availability Groups with readable secondaries are the standard mechanism, but for write scale, Distributed Availability Groups and Stretch Database (deprecated in SQL Server 2022) have been largely superseded by application-layer sharding with elastic pools on Azure SQL. The autonomous controller for SQL Server targets pool elasticity — dynamically moving databases between pool service tiers based on DTU consumption patterns observed over rolling 24-hour windows.

Kubernetes-Native Scaling: Operators, HPA, and KEDA Integration

When PostgreSQL or SQL Server (via containers) runs on Kubernetes, the scaling surface expands to include Kubernetes-native primitives: Horizontal Pod Autoscaler (HPA), Vertical Pod Autoscaler (VPA), and KEDA (Kubernetes Event-Driven Autoscaling). The challenge is that databases are stateful workloads — naive HPA configurations that treat database pods like stateless microservices will corrupt data. Autonomous scaling on Kubernetes requires an operator-aware approach that understands the difference between scaling the connection proxy layer (always safe to scale horizontally) and scaling the database server layer (requires careful leader election and persistent volume management).

The recommended architecture separates three independently scalable layers:

1. Connection Proxy (pgBouncer / ProxySQL) — scaled horizontally via standard HPA on CPU and connection queue depth metrics from Prometheus. Safe to scale aggressively.

2. Read Replicas — scaled via the Postgres Operator's replica count field, gated by the replication health checks described in the previous section.

3. Primary Compute — scaled vertically via VPA (for PostgreSQL pods) or via cloud API calls for managed services. Never scaled horizontally without sharding.

KEDA enables event-driven scaling triggers that go beyond the metrics available natively to Kubernetes. A KEDA ScaledObject targeting the pgBouncer deployment can use a Prometheus query as its scaling trigger — for example, scaling pgBouncer replicas when the active connection count as a fraction of max_connections exceeds 0.70.

# KEDA ScaledObject for pgBouncer — scales proxy tier on connection pressure
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: pgbouncer-connection-scaler
  namespace: database
spec:
  scaleTargetRef:
    name: pgbouncer-deployment
  pollingInterval: 15
  cooldownPeriod: 120
  minReplicaCount: 2
  maxReplicaCount: 12
  triggers:
    - type: prometheus
      metadata:
        serverAddress: http://prometheus.monitoring.svc.cluster.local:9090
        metricName: pgbouncer_active_connections_ratio
        query: |
          sum(pgbouncer_pools_cl_active) /
          sum(pgbouncer_config_max_client_conn)
        threshold: "0.70"
    - type: prometheus
      metadata:
        serverAddress: http://prometheus.monitoring.svc.cluster.local:9090
        metricName: pgbouncer_wait_time_p95
        query: |
          histogram_quantile(0.95,
            rate(pgbouncer_pool_wait_seconds_bucket[2m]))
        threshold: "0.5"

For the Zalando postgres-operator, replica scaling is expressed as a patch to the Postgresql custom resource. The autonomous controller generates and applies this patch programmatically:

# Kubernetes operator patch to scale PostgreSQL read replicas
# Uses the Zalando postgres-operator CRD

from kubernetes import client, config
import json
import logging

logger = logging.getLogger("k8s_pg_scaler")

def scale_postgres_replicas(
    namespace: str,
    cluster_name: str,
    target_replica_count: int,
    current_replica_count: int
):
    config.load_incluster_config()  # use in-cluster ServiceAccount
    custom_api = client.CustomObjectsApi()

    patch_body = {
        "spec": {
            "numberOfInstances": target_replica_count + 1  # +1 for primary
        }
    }

    logger.info(
        f"Patching {cluster_name}: replicas {current_replica_count} "
        f"-> {target_replica_count} "
        f"(total instances: {target_replica_count + 1})"
    )

    try:
        custom_api.patch_namespaced_custom_object(
            group="acid.zalan.do",
            version="v1",
            namespace=namespace,
            plural="postgresqls",
            name=cluster_name,
            body=patch_body,
            content_type="application/merge-patch+json"
        )
        logger.info(f"Replica scale operation accepted by operator for {cluster_name}")
    except client.ApiException as e:
        logger.error(f"Kubernetes API error during replica scaling: {e.status} {e.reason}")
        raise

The controller invokes scale_postgres_replicas only after passing the replication health gate (lag under threshold, no pending base backup, WAL archiver not in error state). These preconditions are checked synchronously before the patch is applied; if any fail, the scale-up is deferred by one evaluation cycle with an alert raised so the operator can investigate if the gate remains blocked beyond the defined timeout.

Scale-Down Safety: Draining, Guardrails, and Reversibility

Scale-up is operationally simple compared to scale-down. Scaling down a database tier risks connection drops, in-flight transaction loss, and replication topology instability. Autonomous scale-down requires a disciplined drain-before-terminate sequence and a set of hard guardrails that prevent the controller from scaling below the minimum viable topology for the current workload.

The drain sequence for a PostgreSQL replica before decommissioning:

1. Remove the replica endpoint from the pgBouncer databases.ini and signal pgBouncer to reload. New connections stop routing to this replica.

2. Wait for the replica's active connection count (from pg_stat_activity) to reach zero or the drain timeout (120 seconds default) to expire.

3. If the drain timeout expires with active connections remaining, force-disconnect using pg_terminate_backend() for idle connections only. Running queries are allowed to complete.

4. Verify replication lag on remaining replicas has not increased beyond the acceptable threshold before proceeding.

5. Issue the Kubernetes delete or cloud API decommission.
