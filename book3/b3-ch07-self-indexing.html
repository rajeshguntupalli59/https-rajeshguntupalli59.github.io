Chapter 7: Self-Indexing and Schema Adaptation

Schema and index management represent two of the most labor-intensive responsibilities in traditional database operations — and two of the highest-leverage targets for autonomous systems. A DBA who manually reviews missing-index DMVs or watches pg_stat_user_indexes for bloat is performing work that a well-designed autonomous layer can execute continuously, with finer granularity and faster response time than any human rotation schedule allows. This chapter builds the full architecture: how autonomous systems observe workload patterns, infer optimal index structures, apply changes safely under live traffic, and adapt the physical schema as access patterns evolve — without waiting for a human to file a ticket.

---

7.1 Workload Observation: Building the Signal Pipeline for Index Intelligence

Autonomous index management lives or dies on the quality of its input signal. Before any decision logic can fire, the system must continuously capture what queries are running, how often, at what cost, and which predicates drive the most scan overhead. This is not a once-a-day snapshot job — it is a streaming observation pipeline.

On PostgreSQL, pg_stat_statements is the foundation. It records normalized query text, execution count, total and mean planning and execution time, shared block reads and hits, and row estimates vs. actuals. However, raw pg_stat_statements is insufficient on its own. For index inference, you need predicate-level visibility — which columns appear in WHERE, JOIN ON, ORDER BY, and GROUP BY clauses — and that requires either query log parsing or the auto_explain extension with log_analyze and log_buffers enabled.

PostgreSQL:

-- Enable pg_stat_statements and auto_explain for signal collection
-- In postgresql.conf:
-- shared_preload_libraries = 'pg_stat_statements,auto_explain'
-- auto_explain.log_min_duration = 500   -- capture plans for queries > 500ms
-- auto_explain.log_analyze = on
-- auto_explain.log_buffers = on
-- auto_explain.log_format = json

-- Query the statement store for high-cost sequential scan candidates
SELECT
    queryid,
    query,
    calls,
    mean_exec_time,
    total_exec_time,
    rows,
    shared_blks_read,
    shared_blks_hit,
    round(
        100.0 * shared_blks_read /
        NULLIF(shared_blks_read + shared_blks_hit, 0), 2
    ) AS cache_miss_pct
FROM pg_stat_statements
WHERE
    mean_exec_time > 100          -- queries averaging over 100ms
    AND calls > 50                 -- called frequently enough to matter
    AND shared_blks_read > 1000   -- reading significant data from disk
ORDER BY total_exec_time DESC
LIMIT 50;

SQL Server:

-- Query the plan cache DMVs for sequential scan and high-cost query signals
SELECT TOP 50
    qs.total_elapsed_time / qs.execution_count        AS avg_elapsed_us,
    qs.total_logical_reads / qs.execution_count       AS avg_logical_reads,
    qs.execution_count,
    qs.total_elapsed_time,
    SUBSTRING(qt.text, (qs.statement_start_offset/2)+1,
        ((CASE qs.statement_end_offset
            WHEN -1 THEN DATALENGTH(qt.text)
            ELSE qs.statement_end_offset END
          - qs.statement_start_offset)/2)+1)          AS statement_text,
    qp.query_plan
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle)      AS qt
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle)   AS qp
WHERE
    qs.execution_count > 50
    AND qs.total_logical_reads / qs.execution_count > 1000
ORDER BY avg_logical_reads DESC;

On SQL Server, the Missing Index DMVs provide a pre-computed signal layer that should be consumed directly. sys.dm_db_missing_index_details, sys.dm_db_missing_index_groups, and sys.dm_db_missing_index_group_stats together give you column-level index recommendations weighted by estimated impact. The critical discipline is interpreting these signals correctly rather than naively applying everything the engine suggests — the DMVs do not account for write amplification, duplicate near-miss indexes, or the cost of maintaining a large index catalog on write-heavy tables.

The observation pipeline feeds a structured fact store — a dedicated schema, typically in a monitoring database or a time-series store like TimescaleDB on the PostgreSQL side, that retains:

  • → Query fingerprint (normalized hash)
  • → Predicate columns and their selectivity estimates
  • → Execution frequency per hour and per day
  • → Estimated and actual row counts (to catch plan regressions)
  • → Current plan type (index scan, bitmap scan, seq scan, nested loop, hash join)
  • → Block I/O cost

    • This fact store is the working memory of your autonomous index advisor. Without retention and trend analysis, the system cannot distinguish a transient spike from a structural workload shift — and those two cases demand completely different responses.

    7.2 Index Recommendation Logic: From Raw Signal to Actionable Proposals

    A signal pipeline produces observations. A recommendation engine converts observations into specific, prioritized index proposals with confidence scores and risk assessments. This is where most autonomous index systems either add real value or generate noise.

    The recommendation logic must enforce several hard constraints before surfacing any proposal:

    1. Duplicate suppression: If an index on (customer_id, created_at) already exists, do not propose (customer_id) as a separate index unless cardinality analysis justifies it.

    2. Write-cost awareness: On a table with an insert rate above a threshold (say, 5,000 rows/second), each additional index adds measurable write overhead. The scoring function must penalize proposals on high-write tables.

    3. Partial index preference: If the workload filter consistently includes WHERE status = 'active' and active rows represent less than 15% of the table, a partial index almost always outperforms a full-column index. The recommendation engine should detect this pattern automatically.

    4. Covering index consolidation: Multiple proposals touching overlapping column sets on the same table should be consolidated into a single covering index candidate before surfacing to the action layer.

    The scoring function used in production at several platform teams looks roughly like this:

    import psycopg2
import hashlib
from dataclasses import dataclass, field
from typing import Optional

@dataclass
class IndexProposal:
    table_name: str
    columns: list[str]
    include_columns: list[str] = field(default_factory=list)
    partial_predicate: Optional[str] = None
    confidence_score: float = 0.0
    estimated_benefit_ms: float = 0.0
    estimated_write_overhead_pct: float = 0.0
    risk_tier: str = "low"   # low | medium | high

def score_index_proposal(
    proposal: IndexProposal,
    avg_query_time_ms: float,
    daily_executions: int,
    table_write_rate_per_sec: float,
    existing_index_count: int
) -> float:
    """
    Scoring model: benefit weighted against write cost and catalog overhead.
    Scores above 0.70 are auto-apply candidates.
    Scores 0.40–0.70 require async human review before apply.
    Scores below 0.40 are suppressed.
    """
    # Estimated time saved per day (normalized to 0-1 range, cap at 10min)
    time_saved_daily = (avg_query_time_ms * 0.65 * daily_executions) / 1000
    benefit_score = min(time_saved_daily / 600_000, 1.0)

    # Write overhead penalty: each index on a hot-write table is expensive
    write_penalty = min(table_write_rate_per_sec / 10_000, 0.5)

    # Catalog bloat penalty: diminishing returns after 10 indexes
    catalog_penalty = min(max(existing_index_count - 10, 0) * 0.03, 0.30)

    # Partial index bonus
    partial_bonus = 0.15 if proposal.partial_predicate else 0.0

    raw_score = benefit_score - write_penalty - catalog_penalty + partial_bonus
    proposal.confidence_score = round(max(0.0, min(raw_score, 1.0)), 4)

    if proposal.confidence_score >= 0.70:
        proposal.risk_tier = "low"
    elif proposal.confidence_score >= 0.40:
        proposal.risk_tier = "medium"
    else:
        proposal.risk_tier = "high"

    return proposal.confidence_score

    For SQL Server environments, the native Hypothetical Index API (sys.dm_db_missing_index_details combined with the Database Engine Tuning Advisor's XML output interface) can be programmatically invoked to pre-validate proposed indexes against the query optimizer before applying them. This pre-validation step — running the optimizer against a hypothetical index without creating the real structure — is one of SQL Server's most underused autonomous features.

    SQL Server:

    -- Use hypothetical indexes to pre-validate a proposal before creation
-- The optimizer will consider these during EXPLAIN without touching storage

DBCC FREEPROCCACHE;  -- clear plan cache to force fresh optimization

-- Create a statistics object to simulate the index presence
CREATE STATISTICS stat_orders_customer_created
ON dbo.orders (customer_id, created_at)
WITH FULLSCAN;

-- Now run EXPLAIN to observe if optimizer would use the proposed structure
-- Compare logical reads before/after statistics creation
SET STATISTICS IO ON;
SELECT order_id, order_total
FROM dbo.orders
WHERE customer_id = 10042
  AND created_at >= '2024-01-01'
ORDER BY created_at DESC;
SET STATISTICS IO OFF;

    The recommendation engine should also integrate with the LLM layer introduced in Book 2's AIOps patterns. When a proposal's confidence score falls in the 0.40–0.70 range, routing the query workload context, current table statistics, and the proposal details to an OpenAI or Claude API call can generate a human-readable risk summary and a recommendation to approve or defer — without requiring a DBA to perform the analysis manually. The autonomous system isn't guessing blindly; it's delegating ambiguous cases to a reasoning layer that produces auditable output.

    7.3 Zero-Downtime Index Application and the Concurrent Build Safety Net

    Generating a good index recommendation is straightforward relative to applying it safely under production load. Index creation on large tables is a blocking operation in its default form — and on a system processing thousands of transactions per second, even a one-second lock on a critical table triggers cascading timeouts. The autonomous apply layer must enforce non-blocking build patterns as an absolute invariant.

    PostgreSQL's CREATE INDEX CONCURRENTLY is the mechanism. It performs two full sequential scans of the table with only brief ShareUpdateExclusiveLock acquisitions, allowing reads and writes to continue throughout the build. The cost is a longer build time and the possibility of a failed build (which leaves an invalid index behind that must be cleaned up). The autonomous apply layer handles both.

    PostgreSQL:

    -- Autonomous index apply with concurrent build and validity check
-- This pattern should be wrapped in the orchestration layer,
-- not run interactively

-- Step 1: Create concurrently (non-blocking)
CREATE INDEX CONCURRENTLY IF NOT EXISTS
    idx_orders_customer_created_active
ON public.orders (customer_id, created_at DESC)
WHERE status = 'active';

-- Step 2: Immediately verify the index is valid (not left in broken state)
SELECT
    indexname,
    indexdef,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_indexes
JOIN pg_class ON pg_class.relname = pg_indexes.indexname
WHERE
    tablename = 'orders'
    AND indexname = 'idx_orders_customer_created_active';

-- Step 3: Check pg_index for validity — invalid indexes must be dropped
SELECT
    i.relname              AS index_name,
    ix.indisvalid          AS is_valid,
    ix.indisready          AS is_ready
FROM pg_index ix
JOIN pg_class i ON i.oid = ix.indexrelid
JOIN pg_class t ON t.oid = ix.indrelid
WHERE
    t.relname = 'orders'
    AND i.relname = 'idx_orders_customer_created_active';

    SQL Server:

    -- Non-blocking index creation with ONLINE = ON
-- Requires Enterprise or Developer edition; validate edition before attempting

CREATE INDEX idx_orders_customer_created
ON dbo.orders (customer_id, created_at DESC)
INCLUDE (order_total, status)
WHERE status = 'active'
WITH (
    ONLINE = ON,              -- non-blocking, allows concurrent DML
    SORT_IN_TEMPDB = ON,      -- reduces contention on the primary filegroup
    DATA_COMPRESSION = ROW,   -- optional: compress the index pages
    FILLFACTOR = 85,          -- leave 15% free for future inserts
    MAXDOP = 4                -- cap parallelism to limit system impact
);

    The autonomous apply layer in a production system adds several safety controls around these primitives:

    Maintenance window gating: Even with concurrent/online builds, large index operations consume I/O bandwidth. The orchestration layer checks a configurable maintenance schedule before starting any build — or consults a real-time load signal from Prometheus to determine if the current CPU and I/O utilization permits a background operation.

    Rollback via drop: If a build completes but query performance does not improve (measured by comparing the rolling average execution time of affected queries before and after), the system drops the index automatically. This is the index equivalent of a feature flag rollback — and it requires retaining the pre-build query timing baseline in the fact store.

    Locking circuit breaker: The autonomous apply script must monitor pg_stat_activity (PostgreSQL) or sys.dm_exec_requests (SQL Server) before and during the build for blocked sessions accumulating against the target table. If the blocker count exceeds a threshold — typically three or more sessions blocked for more than five seconds — the build is aborted and rescheduled.

    7.4 Autonomous Index Retirement: Detecting and Removing the Dead Weight

    Every team builds indexes they later forget about. The autonomous system that only adds indexes without retiring unused ones eventually produces the opposite of its intended effect: a swollen index catalog that adds write overhead on every INSERT, UPDATE, and DELETE with no corresponding read benefit. Index retirement is not a nice-to-have — it is a necessary counterbalance to the recommendation engine.

    Unused index detection relies on usage statistics reset cycles. PostgreSQL accumulates index scan counts in pg_stat_user_indexes but resets them on server restart. SQL Server's sys.dm_db_index_usage_stats resets on service restart. For reliable detection, usage data must be persisted externally across restarts — another function of the observation fact store.

    PostgreSQL:

    -- Identify unused indexes with size and write overhead context
-- Cross-reference with external retention store to confirm zero usage
-- across multiple observation periods, not just since last reset

SELECT
    schemaname,
    tablename,
    indexname,
    pg_size_pretty(pg_relation_size(indexrelid))        AS index_size,
    idx_scan                                             AS scan_count,
    idx_tup_read,
    idx_tup_fetch,
    n_tup_ins + n_tup_upd + n_tup_del                  AS write_ops_on_table
FROM pg_stat_user_indexes
JOIN pg_stat_user_tables USING (schemaname, relname)
WHERE
    idx_scan = 0
    AND pg_relation_size(indexrelid) > 10 * 1024 * 1024  -- larger than 10MB
    AND schemaname NOT IN ('pg_catalog', 'information_schema')
ORDER BY pg_relation_size(indexrelid) DESC;

    SQL Server:

    -- Identify unused indexes with write cost visibility
-- Note: usage stats are per-instance, reset on SQL Server restart
SELECT
    OBJECT_SCHEMA_NAME(i.object_id)          AS schema_name,
    OBJECT_NAME(i.object_id)                 AS table_name,
    i.name                                   AS index_name,
    i.type_desc,
    ISNULL(s.user_seeks, 0)                  AS user_seeks,
    ISNULL(s.user_scans, 0)                  AS user_scans,
    ISNULL(s.user_lookups, 0)                AS user_lookups,
    ISNULL(s.user_updates, 0)                AS user_updates,
    p.rows                                   AS table_rows,
    (SUM(a.total_pages) * 8) / 1024          AS index_size_mb
FROM sys.indexes i
LEFT JOIN sys.dm_db_index_usage_stats s
    ON s.object_id = i.object_id
    AND s.index_id = i.index_id
    AND s.database_id = DB_ID()
JOIN sys.partitions p
    ON p.object_id = i.object_id
    AND p.index_id = i.index_id
JOIN sys.allocation_units a
    ON a.container_id = p.partition_id
WHERE
    i.type_desc <> 'HEAP'
    AND ISNULL(s.user_seeks, 0) = 0
    AND ISNULL(s.user_scans, 0) = 0
    AND ISNULL(s.user_lookups, 0) = 0
    AND ISNULL(s.user_updates, 0) > 100     -- being maintained but never read
    AND i.is_primary_key = 0
    AND i.is_unique_constraint = 0          -- never retire PKs or unique constraints
GROUP BY
    i.object_id, i.name, i.type_desc, s.user_seeks, s.user_scans,
    s.user_lookups, s.user_updates, p.rows
ORDER BY index_size_mb DESC;

    The retirement pipeline applies graduated action rather than immediate deletion. The standard state machine is:

    1. Observation: Index has zero reads for 14+ consecutive days across at least three stat-reset cycles (confirmed via the external fact store).

    2. Disable (SQL Server only): ALTER INDEX idx_name ON table DISABLE — a reversible operation that drops the index pages but retains the metadata. If workload changes cause a performance regression within 72 hours, re-enable without a full rebuild.

    3. Invisible Index Testing (PostgreSQL): Use the pg_visibility extension or manually set the index as not usable by the planner to simulate removal: UPDATE pg_index SET indisvalid = false WHERE indexrelid = 'idx_name'::regclass. Monitor for 24 hours. This is a superuser operation and should only be used in controlled staging scenarios — in production, proceed directly to DROP after the observation window.

    4. Drop: Issue the DROP with concurrent semantics on PostgreSQL, or a standard DROP after confirming no blocking sessions on SQL Server.

    5. Post-drop monitoring: Retain a suppression record in the fact store so the recommendation engine does not immediately propose recreating the same index without evidence of workload change.

    7.5 Schema Adaptation Pipelines: Aligning Physical Structure to Evolving Access Patterns

    Self-indexing is one layer of schema adaptation. The deeper layer — modifying column definitions, partition boundaries, table structures, and data types in response to workload evolution — is where autonomous systems move from reactive optimization to proactive design. This is still an emerging practice, but the building blocks are well-defined.

    Partition boundary adaptation is the most mature use case. On a time-series table partitioned by month, if the system detects that 80% of queries land on the last two partitions while earlier partitions are accessed only by archival processes, the schema adaptation layer should:

  • → Verify the partition access distribution using execution plan analysis
  • → Compress or archive cold partitions automatically (PostgreSQL tablespace reassignment; SQL Server partition switching to a compressed filegroup)
  • → Pre-create future partitions on a rolling schedule without DBA intervention

    • PostgreSQL:

    -- Autonomous partition maintenance: pre-create next month's partition
-- and migrate data from the cold partition to an archive tablespace

-- Pre-create the next partition (run from a scheduled autonomous job)
DO $$
DECLARE
    next_month_start DATE := DATE_TRUNC('month', CURRENT_DATE + INTERVAL '1 month');
    next_month_end   DATE := DATE_TRUNC('month', CURRENT_DATE + INTERVAL '2 months');
    partition_name   TEXT := 'events_' || TO_CHAR(next_month_start, 'YYYY_MM');
BEGIN
    IF NOT EXISTS (
        SELECT 1 FROM pg_class
        WHERE relname = partition_name
    ) THEN
        EXECUTE FORMAT(
            'CREATE TABLE %I PARTITION OF events
             FOR VALUES FROM (%L) TO (%L)
             TABLESPACE pg_default',
            partition_name,
            next_month_start,
            next_month_end
        );
        RAISE NOTICE 'Created partition: %', partition_name;
    ELSE
        RAISE NOTICE 'Partition already exists: %', partition_name;
    END IF;
END $$;

    Column-level type evolution is riskier but actionable. When the observation layer detects consistent implicit casts in query plans — for example, a WHERE user_id = $1 where user_id is VARCHAR(36) storing UUIDs and $1 is always bound as a proper UUID — the system can flag a type migration proposal. The actual migration follows the expand/contract pattern:

    1. Add user_id_new UUID column with a default and a trigger that dual-writes.

    2. Backfill in small batches using an autonomous migration runner.

    3. Update the application's index bindings to reference the new column.

    4. Swap column names (or update application queries) and drop the old column.

    This pattern is implemented in tools like pgroll (PostgreSQL) and integrated into CI/CD pipelines, but the autonomous layer can initiate the sequence when it detects a persistent type mismatch penalty in execution plans — without waiting for a developer to file a migration ticket.

    The schema adaptation pipeline also governs statistics freshness. Stale statistics produce incorrect cardinality estimates, which produce bad plans, which look to the observation layer like unindexed queries — generating false positive index proposals. The autonomous system avoids this by monitoring the n_mod_since_analyze counter on PostgreSQL and the statistics update date via sys.stats on SQL Server, triggering ANALYZE or UPDATE STATISTICS when modification thresholds are breached rather than relying on autovacuum's schedule alone.

    7.6 Governance, Auditability, and the Human Override Layer

    Autonomous systems that act without any governance trail are operationally dangerous regardless of how well the decision logic is tuned. At the ADE level, every index created, modified, or dropped by the autonomous layer must be logged with full context: the signal that triggered the proposal, the scoring rationale, the time of application, the pre- and post-apply performance delta, and the identity of the executing service account. This is not optional overhead — it is the mechanism that lets teams trust the system and the evidence that justifies expanding its operating envelope over time.

    The governance layer also defines the authority tiers that determine when the system acts versus when it surfaces a recommendation for human approval:

    | Action | Condition for Auto-Apply | Escalation Path |

    |---|---|---|

    | Create non-unique index | Score ≥ 0.70, table not in freeze list, maintenance window active | Slack/webhook alert with approve/deny button |

    | Drop unused index | Zero reads for 21+ days, not PK or unique constraint | Ticket created in Jira/Linear with 48h approval window |

    | Modify partition boundary | No active transactions spanning boundary | Requires manual approval regardless of score |

    | Column type migration | Expand/contract phase 1 only (additive) | Phase 3 (removal) always requires human sign-off |
