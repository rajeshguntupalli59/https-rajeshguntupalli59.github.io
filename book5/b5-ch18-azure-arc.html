Chapter 18: Azure Arc-Enabled SQL Server: Hybrid Cloud

Azure Arc-Enabled SQL Server extends Azure's management plane to SQL Server instances running anywhere — on-premises data centers, colocation facilities, edge locations, or competing cloud providers. For DBAs managing hybrid environments, this changes the operational calculus: instead of maintaining separate toolchains for on-prem and cloud databases, you get unified monitoring, policy enforcement, security scanning, and licensing management through a single Azure portal and API surface. This chapter covers how Arc-Enabled SQL Server works architecturally, how to onboard existing SQL Server instances, how to configure Azure services like Microsoft Defender and Azure Monitor against non-Azure hosts, and how to use Arc as a bridge during phased cloud migrations rather than a permanent destination.

---

18.1 Understanding Azure Arc Architecture for SQL Server

Before Arc, hybrid database management meant running parallel monitoring stacks — perhaps SCOM or Datadog for on-premises SQL Server alongside Azure Monitor for Azure SQL. Security assessments ran separately. Licensing compliance was tracked in spreadsheets. Arc collapses this by projecting non-Azure resources into Azure Resource Manager (ARM) as first-class objects.

The architecture rests on two components: the Connected Machine Agent (the Arc agent installed on the host OS) and the Arc SQL Server extension. The Connected Machine agent establishes an outbound HTTPS connection to Azure's Arc service endpoints on port 443. No inbound firewall rules are required. The agent authenticates using a service principal or managed identity, receives its configuration from ARM, and reports telemetry back to Azure. The SQL Server extension layer sits on top, communicating with the local SQL Server instance via a Windows service (SqlServerExtension) or systemd service on Linux.

On-Premises Host
├── SQL Server 2019 (or 2022, 2017, 2016)
├── Arc Connected Machine Agent
│   └── Heartbeat → Azure Arc Service Endpoints (TCP 443)
└── Arc SQL Server Extension
    ├── Reads SQL metadata, license info, configuration
    ├── Sends to Azure Monitor Log Analytics
    └── Applies policies from Azure Policy

From ARM's perspective, your on-premises SQL Server appears under Microsoft.AzureArcData/sqlServerInstances as a resource type. You can tag it, apply RBAC, include it in Azure Policy assignments, and view it in the Azure portal alongside Azure SQL Managed Instances and Azure SQL Databases. This ARM projection is what makes the rest of Arc's capabilities possible — every feature (Defender, Monitor, Best Practices Assessment) hooks into ARM resource identity.

What Arc does not do: It does not move your data. It does not replicate your databases to Azure. It does not change how SQL Server operates internally. The Arc agent has read-only access to SQL Server metadata; it cannot execute DML or DDL. DBAs coming from environments where any agent was treated as a security risk should understand this boundary clearly.

Network requirements differ from what DBAs typically manage. Arc agents must reach several Azure endpoints:

| Endpoint | Purpose |

|---|---|

| management.azure.com | ARM operations |

| login.microsoftonline.com | Azure AD authentication |

| *.his.arc.azure.com | Hybrid Identity Service |

| *.guestconfiguration.azure.com | Policy guest configuration |

| dc.services.visualstudio.com | Telemetry |

In air-gapped environments, Arc supports proxy server configuration — the agent can route all outbound traffic through an authenticated HTTP proxy. Some highly secure environments route Arc traffic through Azure Private Link by deploying a Private Link scope for Azure Monitor and Arc endpoints, though this requires significant network engineering.

Supported SQL Server versions span from SQL Server 2012 through 2022. Extended Security Updates (ESUs) through Arc are a significant licensing benefit for shops still running 2012 or 2014: Microsoft provides ESU patches free of charge for Arc-connected instances (subject to Software Assurance or Azure subscription conditions). For DBAs managing legacy infrastructure that cannot be migrated immediately, this alone often justifies the Arc onboarding effort.

18.2 Onboarding SQL Server Instances to Arc

Onboarding has three paths: single-machine (interactive), at-scale via script, and policy-driven. Interactive onboarding suits labs and initial testing. At-scale is what production environments use.

Single-machine onboarding using Azure CLI:

# Prerequisites: Azure CLI with arcdata extension
az extension add --name arcdata
az extension add --name connectedmachine

# Create a resource group for Arc resources
az group create \
  --name rg-arc-sqlserver-prod \
  --location eastus2

# Create a service principal for Arc onboarding
# Store credentials securely - this SP will be used by agents
az ad sp create-for-rbac \
  --name "sp-arc-onboarding" \
  --role "Azure Connected Machine Onboarding" \
  --scopes "/subscriptions/<subscription-id>/resourceGroups/rg-arc-sqlserver-prod"

# Output:
# {
#   "appId": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
#   "displayName": "sp-arc-onboarding",
#   "password": "xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
#   "tenant": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
# }

# Download the onboarding script from Azure portal or generate via CLI
az connectedmachine connect \
  --resource-group rg-arc-sqlserver-prod \
  --location eastus2 \
  --subscription <subscription-id> \
  --tenant-id <tenant-id> \
  --service-principal-id <appId> \
  --service-principal-secret <password>
# Run this command ON the target SQL Server host

For at-scale onboarding across dozens or hundreds of servers, the recommended approach is a PowerShell script deployed via configuration management (Ansible, Chef, SCCM, or GPO):

# Deploy-ArcAgent.ps1
# Run this on each SQL Server host via your configuration management tool

param(
    [Parameter(Mandatory=$true)]
    [string]$TenantId,
    
    [Parameter(Mandatory=$true)]
    [string]$SubscriptionId,
    
    [Parameter(Mandatory=$true)]
    [string]$ResourceGroup,
    
    [Parameter(Mandatory=$true)]
    [string]$Location,
    
    [Parameter(Mandatory=$true)]
    [string]$ServicePrincipalId,
    
    [Parameter(Mandatory=$true)]
    [string]$ServicePrincipalSecret,
    
    [string]$Environment = "Production",
    [string]$CostCenter = "DBA-Ops"
)

# Download the Arc agent installer
$AgentInstallerUrl = "https://aka.ms/AzureConnectedMachineAgent"
$InstallerPath = "$env:TEMP\AzureConnectedMachineAgent.msi"

Write-Host "Downloading Arc Connected Machine Agent..."
Invoke-WebRequest -Uri $AgentInstallerUrl -OutFile $InstallerPath

# Install silently
Write-Host "Installing Arc Connected Machine Agent..."
msiexec.exe /i $InstallerPath /l*v "$env:TEMP\arc_install.log" /qn

# Wait for service to start
Start-Sleep -Seconds 30

# Connect to Azure Arc
$ArcConnectArgs = @(
    "connect",
    "--tenant-id", $TenantId,
    "--subscription-id", $SubscriptionId,
    "--resource-group", $ResourceGroup,
    "--location", $Location,
    "--service-principal-id", $ServicePrincipalId,
    "--service-principal-secret", $ServicePrincipalSecret,
    "--tags", "Environment=$Environment,CostCenter=$CostCenter,OnboardedBy=AutoScript"
)

& "C:\Program Files\AzureConnectedMachineAgent\azcmagent.exe" @ArcConnectArgs

if ($LASTEXITCODE -eq 0) {
    Write-Host "Arc agent connected successfully for $env:COMPUTERNAME"
    
    # Install SQL Server Arc extension
    # This happens automatically if the agent detects SQL Server,
    # but we can trigger it explicitly
    $ExtensionArgs = @(
        "extensions", "install",
        "--name", "WindowsAgent.SqlServer",
        "--publisher", "Microsoft.AzureData",
        "--version", "latest"
    )
    & "C:\Program Files\AzureConnectedMachineAgent\azcmagent.exe" @ExtensionArgs
} else {
    Write-Error "Arc agent connection failed for $env:COMPUTERNAME with exit code $LASTEXITCODE"
    exit 1
}

After onboarding, verify the Arc SQL Server resource exists:

# Verify Arc-connected SQL Server instances
az sql server-arc list \
  --resource-group rg-arc-sqlserver-prod \
  --output table

# Get details of a specific instance
az sql server-arc show \
  --name SQLPROD01 \
  --resource-group rg-arc-sqlserver-prod

# List all databases discovered on the Arc SQL Server
az sql db-arc list \
  --server-name SQLPROD01 \
  --resource-group rg-arc-sqlserver-prod \
  --output table

License type configuration is one of the most operationally important settings. Arc tracks whether each instance uses License Included, Azure Hybrid Benefit, or Pay-As-You-Go licensing. Setting this incorrectly leads to billing surprises or compliance gaps:

# Set Azure Hybrid Benefit license type for an Arc SQL instance
# Use this if you have existing SQL Server licenses with Software Assurance
az sql server-arc update \
  --name SQLPROD01 \
  --resource-group rg-arc-sqlserver-prod \
  --license-type LicenseIncluded

# License type options:
# LicenseIncluded  - You're paying for the license through Azure (ESU scenario)
# PAYG             - Pay-as-you-go (useful for dev/test)
# Paid             - Azure Hybrid Benefit (bring your own SA license)

Gotcha: SQL Server instances with multiple named instances on the same host appear as separate Arc SQL Server resources, but they share the same Connected Machine Agent. If the agent goes offline, all instances become unreachable in Arc simultaneously. Monitor agent health separately from SQL Server health.

Gotcha: Linux-hosted SQL Server (SQL Server on RHEL, Ubuntu, SLES) is fully supported by Arc, but the extension service name differs (mssql-server-arc via systemd). Automation scripts that assume Windows service names will fail silently on Linux hosts.

18.3 Azure Monitor and Log Analytics Integration

Once Arc is connected, Azure Monitor becomes the unified observability layer for both your on-premises SQL Server and Azure SQL services. The key component is the Log Analytics agent (or its newer replacement, Azure Monitor Agent / AMA), which ships SQL Server performance counters and event log data to a Log Analytics workspace.

Configure Log Analytics data collection:

# Create a Log Analytics workspace
az monitor log-analytics workspace create \
  --resource-group rg-arc-sqlserver-prod \
  --workspace-name law-sqlserver-hybrid \
  --location eastus2 \
  --sku PerGB2018 \
  --retention-time 90

# Get workspace ID and key for agent configuration
WORKSPACE_ID=$(az monitor log-analytics workspace show \
  --resource-group rg-arc-sqlserver-prod \
  --workspace-name law-sqlserver-hybrid \
  --query customerId -o tsv)

WORKSPACE_KEY=$(az monitor log-analytics workspace get-shared-keys \
  --resource-group rg-arc-sqlserver-prod \
  --workspace-name law-sqlserver-hybrid \
  --query primarySharedKey -o tsv)

echo "Workspace ID: $WORKSPACE_ID"
echo "Workspace Key: $WORKSPACE_KEY"

# Enable monitoring extension on Arc SQL Server
az connectedmachine extension create \
  --machine-name SQLPROD01 \
  --resource-group rg-arc-sqlserver-prod \
  --name MMA \
  --publisher Microsoft.EnterpriseCloud.Monitoring \
  --type MicrosoftMonitoringAgent \
  --settings "{\"workspaceId\": \"$WORKSPACE_ID\"}" \
  --protected-settings "{\"workspaceKey\": \"$WORKSPACE_KEY\"}"

Once data flows into Log Analytics, you write KQL queries against it exactly as you would for Azure SQL — the workspace abstraction normalizes the data regardless of origin:

// KQL: Find longest-running queries on Arc-connected SQL instances
// Run this in Log Analytics workspace
AzureDiagnostics
| where ResourceProvider == "MICROSOFT.AZUREARCDATA"
| where Category == "SQLQueryStoreRuntime"
| where TimeGenerated > ago(1h)
| extend ExecutionDurationMs = todouble(max_elapsed_time_d) / 1000
| where ExecutionDurationMs > 5000
| project 
    TimeGenerated,
    Resource,
    database_name_s,
    query_id_d,
    ExecutionDurationMs,
    total_worker_time_d,
    execution_count_d
| order by ExecutionDurationMs desc
| take 50

// KQL: Wait stats comparison across hybrid fleet
Perf
| where ObjectName == "SQLServer:Wait Statistics"
| where CounterName in ("Lock waits", "Log write waits", "Network IO waits")
| where TimeGenerated > ago(24h)
| summarize AvgWait = avg(CounterValue) by Computer, CounterName, bin(TimeGenerated, 1h)
| render timechart

// KQL: Identify Arc instances with high CPU over 80%
Perf
| where ObjectName == "Processor Information"
| where CounterName == "% Processor Time"
| where TimeGenerated > ago(1h)
| summarize AvgCPU = avg(CounterValue) by Computer, bin(TimeGenerated, 5m)
| where AvgCPU > 80
| join kind=inner (
    AzureDiagnostics
    | where ResourceProvider == "MICROSOFT.AZUREARCDATA"
    | distinct Computer = Resource
) on Computer
| project TimeGenerated, Computer, AvgCPU
| order by AvgCPU desc

Best Practices Assessment is an Arc-specific capability that runs scheduled SQL Server configuration assessments and reports findings to Log Analytics. It checks roughly 400 rules covering security configuration, performance settings, availability configuration, and deprecated features. Unlike ad-hoc assessment tools, this runs on a schedule (weekly by default, configurable to daily) and trends results over time.

# Enable Best Practices Assessment on Arc SQL Server
az sql server-arc assessment create \
  --name SQLPROD01 \
  --resource-group rg-arc-sqlserver-prod \
  --workspace-id $WORKSPACE_ID \
  --schedule "0 2 * * 0"  # Weekly at 2 AM Sunday (cron format)

# View assessment results via CLI
az sql server-arc assessment show \
  --name SQLPROD01 \
  --resource-group rg-arc-sqlserver-prod

# Assessment findings appear in Log Analytics under SqlAssessment_CL table

Query assessment findings programmatically:

// KQL: Critical and High severity assessment findings
SqlAssessment_CL
| where TimeGenerated > ago(7d)
| where Severity_s in ("Critical", "High")
| where IsRuleInWarningState_b == true
| project
    TimeGenerated,
    Computer,
    RuleName_s,
    Severity_s,
    Category_s,
    Description_s,
    Recommendation_s
| order by Severity_s asc, TimeGenerated desc

Alert rules for Arc SQL Server instances use the same Azure Monitor alert framework as Azure SQL. The difference is that metric sources reference microsoft.azurearcdata/sqlserverinstances rather than microsoft.sql/servers. Some metrics available for Azure SQL (like DTU consumption) have no Arc equivalent since Arc instances aren't DTU-based.

# Create an alert rule for high CPU on Arc SQL instances
az monitor metrics alert create \
  --name "ArcSQL-HighCPU-Alert" \
  --resource-group rg-arc-sqlserver-prod \
  --scopes "/subscriptions/<sub-id>/resourceGroups/rg-arc-sqlserver-prod/providers/Microsoft.AzureArcData/sqlServerInstances/SQLPROD01" \
  --condition "avg Processor/PercentProcessorTime > 85" \
  --window-size 5m \
  --evaluation-frequency 1m \
  --action-group "/subscriptions/<sub-id>/resourceGroups/rg-arc-sqlserver-prod/actionGroups/dba-oncall" \
  --description "CPU exceeded 85% on Arc-connected SQL Server SQLPROD01" \
  --severity 2

18.4 Microsoft Defender for SQL via Arc

Microsoft Defender for SQL provides vulnerability assessment and advanced threat protection for SQL Server instances. Without Arc, Defender for SQL was limited to Azure SQL services. Arc extends it to on-premises and multicloud SQL Server instances under the same policy and billing framework.

Enable Defender for SQL on Arc instances:

# Enable Defender for SQL at subscription level
# This applies to all Arc SQL instances in the subscription
az security pricing create \
  --name SqlServerVirtualMachines \
  --tier Standard

# Verify Defender is active on specific Arc instance
az security sql-vulnerability-assessment list \
  --server-name SQLPROD01 \
  --resource-group rg-arc-sqlserver-prod

# Run an on-demand vulnerability assessment
az security sql-vulnerability-assessment scan \
  --server-name SQLPROD01 \
  --resource-group rg-arc-sqlserver-prod \
  --database-name HRDB

Defender for SQL performs two distinct functions. Vulnerability Assessment scans SQL Server configuration against a database of known misconfigurations — things like sa account enabled, CLR enabled without justification, xp_cmdshell exposed, databases with TRUSTWORTHY ON, weak password policies, and over-privileged logins. It produces a baseline on first run and flags deviations from that baseline on subsequent scans.

Advanced Threat Protection analyzes SQL Server audit logs for behavioral anomalies: SQL injection patterns, unusual data export volumes, logins from unexpected locations, and brute-force login patterns. This requires SQL Server Audit to be configured and shipping logs to a storage account or Log Analytics workspace.

-- Configure SQL Server Audit for Defender ATP integration
-- Run on the on-premises Arc-connected SQL Server instance

-- Create server audit pointing to Log Analytics
-- For Arc instances, use FILE audit and let Arc extension ship logs
USE master;
GO

CREATE SERVER AUDIT [ArcDefenderAudit]
TO FILE (
    FILEPATH = 'D:\SQLAudit\',
    MAXSIZE = 100 MB,
    MAX_ROLLOVER_FILES = 50,
    RESERVE_DISK_SPACE = OFF
)
WITH (
    QUEUE_DELAY = 1000,
    ON_FAILURE = CONTINUE
);
GO

ALTER SERVER AUDIT [ArcDefenderAudit] WITH (STATE = ON);
GO

-- Create audit specification for Defender-relevant events
CREATE SERVER AUDIT SPECIFICATION [ArcDefenderAuditSpec]
FOR SERVER AUDIT [ArcDefenderAudit]
ADD (FAILED_LOGIN_GROUP),
ADD (SUCCESSFUL_LOGIN_GROUP),
ADD (LOGOUT_GROUP),
ADD (DATABASE_PERMISSION_CHANGE_GROUP),
ADD (SERVER_PERMISSION_CHANGE_GROUP),
ADD (DATABASE_ROLE_MEMBER_CHANGE_GROUP),
ADD (SERVER_ROLE_MEMBER_CHANGE_GROUP),
ADD (USER_CHANGE_PASSWORD_GROUP),
ADD (SCHEMA_OBJECT_ACCESS_GROUP)
WITH (STATE = ON);
GO

-- Verify audit is active
SELECT 
    a.name AS audit_name,
    a.status_desc,
    a.type_desc,
    s.name AS spec_name,
    s.is_state_enabled
FROM sys.server_audits a
JOIN sys.server_audit_specifications s 
    ON a.audit_guid = s.audit_guid
WHERE a.name = 'ArcDefenderAudit';

Defender for SQL alerts surface in Azure Security Center (now Microsoft Defender for Cloud) under the same dashboard as Azure SQL alerts. A DBA managing 200 on-premises instances previously had no centralized security alert surface for SQL injection attempts. With Arc, a brute-force attack against an on-premises SQL Server in a Chicago data center triggers the same Defender for Cloud alert workflow as an attack against an Azure SQL Database in East US.

Gotcha: Defender for SQL charges per instance per month (approximately $15/vCore/month as of 2024, check current pricing). For shops with large SQL Server estates, enabling Defender at subscription scope without an instance count audit first produces unexpected billing. Run the count before enabling:

# Count Arc SQL instances before enabling Defender
az sql server-arc list \
  --subscription <subscription-id> \
  --query "length(@)" \
  --output tsv

# List instances with vCore count to estimate Defender cost
az sql server-arc list \
  --subscription <subscription-id> \
  --query "[].{Name:name, Location:location, vCores:properties.vCores, Edition:properties.edition}" \
  --output table

18.5 Azure Policy for Hybrid SQL Server Governance

Azure Policy becomes your configuration enforcement mechanism across the hybrid fleet. Where you previously used SQL Server Policy-Based Management (PBM) for on-premises enforcement and separate tooling for Azure SQL, Arc unifies both under Azure Policy.

Built-in Arc SQL Server policies cover:

  • → SQL Server should have Azure Defender enabled
  • → SQL Server should have transparent data encryption enabled
  • → SQL Server should have the latest cumulative update installed
  • → SQL Server should not use the sa account
  • → Arc-enabled SQL Server instances should have Azure AD authentication configured

    • You assign these policies to management groups, subscriptions, or resource groups. Instances that drift from policy show as non-compliant in the Defender for Cloud compliance dashboard.

    # Assign built-in policy: Defender for SQL should be enabled on Arc instances
POLICY_DEFINITION_ID=$(az policy definition list \
  --query "[?displayName=='Microsoft Defender for SQL should be enabled for unprotected SQL Servers'].id" \
  --output tsv)

az policy assignment create \
  --name "require-defender-arcsql" \
  --display-name "Require Defender for Arc SQL Servers" \
  --policy $POLICY_DEFINITION_ID \
  --scope "/subscriptions/<subscription-id>/resourceGroups/rg-arc-sqlserver-prod" \
  --enforcement-mode Default

# Create custom policy: Ensure minimum SQL Server version
cat > custom-policy-sqlversion.json << 'EOF'
{
  "mode": "All",
  "policyRule": {
    "if": {
      "allOf": [
        {
          "field": "type",
          "equals": "Microsoft.AzureArcData/sqlServerInstances"
        },
        {
          "field": "Microsoft.AzureArcData/sqlServerInstances/version",
          "notIn": ["SQL Server 2019", "SQL Server 2022"]
        }
      ]
    },
    "then": {
      "effect": "audit"
    }
  },
  "parameters": {}
}
EOF

az policy definition create \
  --name "require-sql-2019-or-later" \
  --display-name "Arc SQL Server must be 2019 or later" \
  --description "Audits Arc-connected SQL Servers running versions prior to SQL Server 2019" \
  --rules custom-policy-sqlversion.json \
  --mode All

# Assign the custom policy
az policy assignment create \
  --name "enforce-sql-version" \
  --policy "require-sql-2019-or-later" \
  --scope "/subscriptions/<subscription-id>"
