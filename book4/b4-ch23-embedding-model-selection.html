Chapter 23: Embedding Model Selection: OpenAI, Cohere, Sentence Transformers

The embedding model you choose is one of the highest-leverage architectural decisions in a vector database system—roughly equivalent to choosing a data type for a primary key column. Get it wrong and you face cascading problems: mismatched dimensions between your stored vectors and your query vectors, poor retrieval quality on domain-specific content, unnecessary API costs at query time, or latency profiles that make your application unusable. This chapter walks through the three major embedding ecosystems—OpenAI, Cohere, and Sentence Transformers—examining their output dimensions, quality characteristics, cost structures, and operational tradeoffs. You will leave with a framework for matching model to workload, and production-ready code for each provider integrated against pgvector, Pinecone, and Weaviate.

---

What an Embedding Model Actually Does (and Why It Matters to a DBA)

An embedding model is a function: it takes text as input and returns a fixed-length vector of floating-point numbers. That vector is a compressed semantic fingerprint—documents with similar meaning produce vectors that are geometrically close in high-dimensional space. When you run a vector similarity search, you are asking: "which stored vectors are nearest to my query vector?"

This framing immediately surfaces the DBA's core concern: schema compatibility. Every embedding model produces a specific vector dimension. OpenAI's text-embedding-3-small outputs 1536 dimensions by default. Cohere's embed-english-v3.0 outputs 1024. A sentence-transformers/all-MiniLM-L6-v2 model outputs 384. If you store embeddings generated by one model and query with a different one—even a different version of the same model—the similarity scores become meaningless. This is not a soft warning; it is the vector database equivalent of a type mismatch on a join key.

Think of model selection like choosing between TEXT, VARCHAR(255), and BYTEA. All three store string-like data, but the right choice depends on access patterns, storage constraints, and how downstream consumers will process the data. Similarly, embedding model choice must account for:

| Concern | DBA Analogy |

|---|---|

| Vector dimension | Column data type / storage width |

| Model versioning | Schema migration discipline |

| Hosted vs. local | Managed service vs. self-hosted database |

| Batch vs. real-time | Bulk insert vs. OLTP insert pattern |

| Domain specificity | Partial index on a specialized query pattern |

One practical implication DBAs often miss: you must re-embed your entire corpus whenever you change models. There is no ALTER TABLE equivalent that converts 1536-dimension vectors to 1024-dimension vectors. Plan for this from day one. In production systems with millions of rows, a model migration is a multi-hour or multi-day batch operation requiring careful coordination between the old and new vector columns during the cutover window.

OpenAI Embeddings: Production Workhorse

OpenAI offers two current-generation embedding models that have become the de facto baseline for most commercial RAG applications:

  • → **`text-embedding-3-small`**: 1536 dimensions, ~$0.02 per million tokens
  • → **`text-embedding-3-large`**: 3072 dimensions, ~$0.13 per million tokens

    • Both support dimension reduction via the dimensions parameter—you can request fewer dimensions than the maximum, and the model truncates the vector in a way that preserves most semantic quality. This is a meaningful operational feature: you can store 256-dimension vectors from text-embedding-3-large and still outperform text-embedding-3-small at full dimensionality on many benchmarks, while cutting storage by 83%.

    The older text-embedding-ada-002 (1536 dimensions) still appears in legacy deployments but should not be chosen for new projects. It performs worse than both text-embedding-3-* models on MTEB benchmarks and costs the same as text-embedding-3-small.

    Rate limits and batching behavior matter significantly at scale. OpenAI processes embedding requests in batches of up to 2048 inputs per API call. Each input is limited to 8192 tokens. For a DBA loading a 10-million-row document table, you need to batch your inserts carefully to avoid hitting per-minute token limits and to maximize throughput.

    # Python
import openai
import psycopg2
import numpy as np
from typing import List

client = openai.OpenAI()  # Uses OPENAI_API_KEY from environment

def embed_batch_openai(texts: List[str], model: str = "text-embedding-3-small", dimensions: int = 1536) -> List[List[float]]:
    """
    Embed a batch of texts using OpenAI. Returns list of float vectors.
    Max batch size: 2048 inputs. Max tokens per input: 8192.
    """
    response = client.embeddings.create(
        model=model,
        input=texts,
        dimensions=dimensions  # Supported on text-embedding-3-* only
    )
    # Response items are ordered to match input order
    return [item.embedding for item in response.data]


def load_documents_to_pgvector(
    documents: List[dict],  # Each dict: {"id": int, "content": str}
    conn_string: str,
    model: str = "text-embedding-3-small",
    dimensions: int = 1536,
    batch_size: int = 500  # Well under the 2048 API limit; controls DB insert size
):
    """
    Batch-embed documents and store in PostgreSQL with pgvector.
    Assumes table: documents(id BIGINT, content TEXT, embedding VECTOR(1536))
    """
    conn = psycopg2.connect(conn_string)
    cur = conn.cursor()

    for i in range(0, len(documents), batch_size):
        batch = documents[i : i + batch_size]
        texts = [doc["content"] for doc in batch]
        ids = [doc["id"] for doc in batch]

        embeddings = embed_batch_openai(texts, model=model, dimensions=dimensions)

        records = [
            (ids[j], texts[j], embeddings[j])
            for j in range(len(batch))
        ]

        cur.executemany(
            "INSERT INTO documents (id, content, embedding) VALUES (%s, %s, %s::vector) "
            "ON CONFLICT (id) DO UPDATE SET content = EXCLUDED.content, embedding = EXCLUDED.embedding",
            records
        )
        conn.commit()
        print(f"Loaded rows {i} to {i + len(batch) - 1}")

    cur.close()
    conn.close()


# Similarity search using the same model at query time
def search_documents(query: str, conn_string: str, top_k: int = 5, model: str = "text-embedding-3-small"):
    query_embedding = embed_batch_openai([query], model=model)[0]
    
    conn = psycopg2.connect(conn_string)
    cur = conn.cursor()
    cur.execute(
        """
        SELECT id, content, 1 - (embedding <=> %s::vector) AS cosine_similarity
        FROM documents
        ORDER BY embedding <=> %s::vector
        LIMIT %s
        """,
        (query_embedding, query_embedding, top_k)
    )
    results = cur.fetchall()
    cur.close()
    conn.close()
    return results

    The pgvector schema for this setup:

    -- PostgreSQL / pgvector
-- Create table with matching dimension
CREATE TABLE documents (
    id          BIGINT PRIMARY KEY,
    content     TEXT NOT NULL,
    embedding   VECTOR(1536),           -- Must match model output dimension
    model_name  TEXT DEFAULT 'text-embedding-3-small',  -- Track which model generated this
    embedded_at TIMESTAMPTZ DEFAULT NOW()
);

-- HNSW index for ANN search (preferred over IVFFlat for most workloads)
CREATE INDEX idx_documents_embedding_hnsw
    ON documents
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- If you reduce dimensions to 256, alter and rebuild:
-- ALTER TABLE documents ALTER COLUMN embedding TYPE VECTOR(256);
-- REINDEX INDEX idx_documents_embedding_hnsw;

    The key operational discipline with OpenAI embeddings: treat the model identifier as part of your schema definition. Store model_name and dimensions alongside each vector row. When OpenAI releases text-embedding-4-*, you will want to identify exactly which rows were generated by which model version before deciding whether to migrate.

    Cohere Embeddings: Retrieval-Optimized with Input Type Awareness

    Cohere's embedding models introduce a concept that OpenAI's API does not expose directly: input type separation. Cohere lets you specify whether each embedding request is a search_document or a search_query. Under the hood, the model applies different transformations for indexing versus querying, which meaningfully improves retrieval precision—particularly for asymmetric retrieval tasks where the query is short (a user question) but the documents are long (technical paragraphs).

    Current production models:

  • → **`embed-english-v3.0`**: 1024 dimensions, English-only, strongest English retrieval
  • → **`embed-multilingual-v3.0`**: 1024 dimensions, 100+ languages, competitive multilingual quality
  • → **`embed-english-light-v3.0`**: 384 dimensions, English-only, optimized for low-latency

    • Cohere also supports binary and int8 quantization natively—you can request compressed embeddings directly from the API, reducing storage by 32x (binary) or 4x (int8) at the cost of some precision. For DBAs managing multi-terabyte vector stores, this is a first-class feature rather than an afterthought.

    # Python
import cohere
import pinecone
from pinecone import Pinecone, ServerlessSpec
import time

co = cohere.Client("YOUR_COHERE_API_KEY")
pc = Pinecone(api_key="YOUR_PINECONE_API_KEY")

# Initialize Pinecone index — dimension must match Cohere embed-english-v3.0 output
INDEX_NAME = "cohere-docs-1024"

if INDEX_NAME not in [i.name for i in pc.list_indexes()]:
    pc.create_index(
        name=INDEX_NAME,
        dimension=1024,                       # Matches embed-english-v3.0
        metric="cosine",
        spec=ServerlessSpec(cloud="aws", region="us-east-1")
    )
    # Wait for index to be ready
    while not pc.describe_index(INDEX_NAME).status["ready"]:
        time.sleep(1)

index = pc.Index(INDEX_NAME)


def embed_documents_cohere(texts: List[str]) -> List[List[float]]:
    """
    Embed documents for indexing. Use input_type='search_document'.
    Cohere batch limit: 96 texts per call. Max 512 tokens per text.
    """
    response = co.embed(
        texts=texts,
        model="embed-english-v3.0",
        input_type="search_document",    # Critical: use 'search_document' for corpus
        embedding_types=["float"]        # Options: float, int8, uint8, binary, ubinary
    )
    return response.embeddings.float_


def embed_query_cohere(query: str) -> List[float]:
    """
    Embed a search query. Use input_type='search_query'.
    This is a different transformation than search_document.
    """
    response = co.embed(
        texts=[query],
        model="embed-english-v3.0",
        input_type="search_query",       # Critical: must differ from document embedding
        embedding_types=["float"]
    )
    return response.embeddings.float_[0]


def upsert_to_pinecone(documents: List[dict], batch_size: int = 96):
    """
    documents: [{"id": "doc_001", "content": "...", "metadata": {...}}]
    """
    for i in range(0, len(documents), batch_size):
        batch = documents[i : i + batch_size]
        texts = [doc["content"] for doc in batch]
        embeddings = embed_documents_cohere(texts)

        vectors = [
            {
                "id": batch[j]["id"],
                "values": embeddings[j],
                "metadata": {
                    "content": batch[j]["content"],
                    **batch[j].get("metadata", {})
                }
            }
            for j in range(len(batch))
        ]
        index.upsert(vectors=vectors)
        print(f"Upserted {len(vectors)} vectors to Pinecone")


def search_pinecone(query: str, top_k: int = 5, filter_dict: dict = None):
    query_embedding = embed_query_cohere(query)
    results = index.query(
        vector=query_embedding,
        top_k=top_k,
        include_metadata=True,
        filter=filter_dict  # Pinecone metadata filter, e.g. {"department": "engineering"}
    )
    return results.matches

    The input_type distinction is easy to forget and expensive to debug. If you embed all your documents with search_document during batch loading and then embed queries with search_document instead of search_query at runtime, retrieval quality degrades measurably—typically 5–15% drop in recall@10 on technical content—and the root cause is invisible from SQL query plans or vector search metrics. Build a wrapper layer that enforces input type by function signature, as shown above.

    Binary quantization example for high-scale storage:

    # Python
def embed_documents_cohere_binary(texts: List[str]) -> List[List[int]]:
    """
    Binary embeddings reduce 1024-dimension float32 (4KB per vector) 
    to 128-byte binary vectors — 32x storage reduction.
    Recall degradation: typically 5–10% vs float32 at equivalent index quality.
    """
    response = co.embed(
        texts=texts,
        model="embed-english-v3.0",
        input_type="search_document",
        embedding_types=["ubinary"]    # Unsigned binary: each dim becomes 0 or 1
    )
    return response.embeddings.ubinary_  # Returns list of packed byte arrays

    Cohere's pricing model (per-token rather than per-call) and multilingual support make it the strongest commercial choice for non-English corpora. If your application serves users in multiple languages, embed-multilingual-v3.0 avoids the need to maintain separate indexes per language.

    Sentence Transformers: Local Embeddings for Privacy and Cost Control

    Sentence Transformers is a Python library built on HuggingFace Transformers that gives you access to hundreds of pre-trained embedding models you can run entirely on your own infrastructure. No API calls, no per-token billing, no data leaving your network. For DBAs managing regulated data—healthcare, financial, government—this is often not optional.

    The quality range is wide. The smallest usable model (all-MiniLM-L6-v2, 384 dims) runs on CPU at roughly 2,000 sentences/second on a modern 16-core server. The strongest general-purpose model (all-mpnet-base-v2, 768 dims) runs at about 400 sentences/second on the same hardware. With a GPU (A10G or better), throughput scales 20–50x depending on batch size.

    Popular models and their characteristics:

    | Model | Dimensions | Speed (CPU) | MTEB Score | Best Use |

    |---|---|---|---|---|

    | all-MiniLM-L6-v2 | 384 | ~2000/s | 56.3 | High-throughput, latency-sensitive |

    | all-mpnet-base-v2 | 768 | ~400/s | 57.8 | General purpose RAG |

    | multi-qa-mpnet-base-dot-v1 | 768 | ~400/s | 57.6 | Q&A retrieval tasks |

    | BAAI/bge-large-en-v1.5 | 1024 | ~150/s | 63.9 | High-quality English retrieval |

    | intfloat/e5-large-v2 | 1024 | ~150/s | 62.2 | Instruction-following retrieval |

    BAAI/bge-large-en-v1.5 is currently one of the strongest open-source English models, competitive with commercial providers on many MTEB tasks. If you have GPU infrastructure, it should be your default open-source choice.

    # Python
from sentence_transformers import SentenceTransformer
import weaviate
from weaviate.classes.init import Auth
from weaviate.classes.config import Configure, Property, DataType, VectorDistances
import numpy as np

# Load model once — keep in memory across requests
# Device auto-detects CUDA if available, falls back to CPU
model = SentenceTransformer("BAAI/bge-large-en-v1.5", device="cuda")  

# BGE models use instruction prefix for queries (not documents)
# This is a model-specific convention — read the model card
QUERY_PREFIX = "Represent this sentence for searching relevant passages: "

def embed_documents_local(texts: List[str], batch_size: int = 64) -> np.ndarray:
    """
    Embed documents for indexing. No prefix needed for BGE document embeddings.
    Returns numpy array of shape (len(texts), 1024).
    """
    embeddings = model.encode(
        texts,
        batch_size=batch_size,
        normalize_embeddings=True,  # Required for cosine similarity via dot product
        show_progress_bar=True
    )
    return embeddings  # numpy float32 array


def embed_query_local(query: str) -> np.ndarray:
    """
    BGE query embedding requires instruction prefix.
    Other models (MiniLM, mpnet) do NOT use prefixes.
    """
    prefixed_query = QUERY_PREFIX + query
    embedding = model.encode(
        [prefixed_query],
        normalize_embeddings=True
    )
    return embedding[0]


# Connect to Weaviate and create a collection
weaviate_client = weaviate.connect_to_weaviate_cloud(
    cluster_url="YOUR_WEAVIATE_CLOUD_URL",
    auth_credentials=Auth.api_key("YOUR_WEAVIATE_API_KEY")
)

COLLECTION_NAME = "TechDocuments"

# Create collection — using "none" vectorizer since we supply our own vectors
if not weaviate_client.collections.exists(COLLECTION_NAME):
    weaviate_client.collections.create(
        name=COLLECTION_NAME,
        vectorizer_config=Configure.Vectorizer.none(),   # We provide vectors manually
        vector_index_config=Configure.VectorIndex.hnsw(
            distance_metric=VectorDistances.COSINE,
            ef_construction=128,
            max_connections=16
        ),
        properties=[
            Property(name="doc_id", data_type=DataType.TEXT),
            Property(name="content", data_type=DataType.TEXT),
            Property(name="source", data_type=DataType.TEXT),
        ]
    )

collection = weaviate_client.collections.get(COLLECTION_NAME)


def load_documents_to_weaviate(documents: List[dict], batch_size: int = 64):
    """
    documents: [{"doc_id": "...", "content": "...", "source": "..."}]
    """
    texts = [doc["content"] for doc in documents]
    embeddings = embed_documents_local(texts, batch_size=batch_size)

    with collection.batch.fixed_size(batch_size=200) as batch:
        for i, doc in enumerate(documents):
            batch.add_object(
                properties={
                    "doc_id": doc["doc_id"],
                    "content": doc["content"],
                    "source": doc.get("source", "")
                },
                vector=embeddings[i].tolist()
            )
    
    failed = collection.batch.failed_objects
    if failed:
        print(f"Warning: {len(failed)} objects failed to insert")


def search_weaviate(query: str, top_k: int = 5):
    from weaviate.classes.query import MetadataQuery
    
    query_vector = embed_query_local(query).tolist()
    results = collection.query.near_vector(
        near_vector=query_vector,
        limit=top_k,
        return_metadata=MetadataQuery(distance=True, score=True),
        return_properties=["doc_id", "content", "source"]
    )
    return results.objects

    Model loading strategy matters for production API servers. Loading a 1.3GB model (bge-large) takes 3–8 seconds and consumes significant RAM. Load it once at application startup—not per request—and share it across worker threads with appropriate locking if your framework requires it. For Python web frameworks (FastAPI, Flask), use a module-level singleton.

    # Python
# Singleton pattern for embedding model in a FastAPI service
from fastapi import FastAPI
from contextlib import asynccontextmanager
import threading

_model = None
_model_lock = threading.Lock()

@asynccontextmanager
async def lifespan(app: FastAPI):
    global _model
    _model = SentenceTransformer("BAAI/bge-large-en-v1.5", device="cuda")
    _model.max_seq_length = 512  # Clip long inputs to prevent OOM
    print(f"Model loaded. Max sequence length: {_model.max_seq_length}")
    yield
    # Cleanup on shutdown (optional for GPU memory release)
    del _model

app = FastAPI(lifespan=lifespan)

@app.post("/embed")
def embed_text(payload: dict):
    query = payload["text"]
    with _model_lock:
        vector = _model.encode([query], normalize_embeddings=True)[0]
    return {"embedding": vector.tolist(), "dimension": len(vector)}

    Comparing Models: Quality, Cost, and Latency at Production Scale

    No single embedding model wins across all dimensions. The right selection depends on your specific retrieval task, throughput requirements, data sensitivity, and budget. Here is how the comparison maps out across the factors that matter in production:

    Retrieval quality (MTEB Benchmark, English)

    BAAI/bge-large-en-v1.5 and OpenAI text-embedding-3-large are roughly equivalent at the top of the open leaderboard, both scoring in the 62–64 range on MTEB average. text-embedding-3-small and Cohere embed-english-v3.0 cluster around 61–62. MiniLM sits at 56–57. For most production RAG use cases, the difference between top-tier models is 3–5% in recall@10—meaningful but not the dominant quality driver. Chunking strategy and retrieval reranking typically have larger effects.

    Latency

    API-hosted models (OpenAI, Cohere) have network round-trip latency of 80–300ms for single requests, regardless of model quality.
