Why self-hosted

Your query plans stay on your server

Most SaaS database tools require you to upload connection strings, query plans, or schema structure to an external service. For regulated industries, security-conscious teams, or air-gapped environments — that's simply not an option. Every tool here runs completely on your infrastructure. Nothing leaves your network.

No subscription required to run your own database tools

The SaaS model works well for many things. For database diagnostics tools — which need to connect to your production systems — a one-time license makes more sense. You install it, you own it, it keeps running even if the vendor disappears. No seats, no per-query pricing, no surprise invoices.

Tools that connect to live data show you what's real

Static reports go stale. Manually exported CSV files miss what happened five minutes ago. Every tool here connects directly to your live database and reads from the catalog, DMVs, and pg_stat_statements — so you always see the current state of your system, not a snapshot from last Tuesday.

Things learned the hard way

01

An index on the right columns in the wrong order does nothing. EXPLAIN will still show an Index Scan. The planner may still pick it. But your sequential page reads triple and your query gets slower, not faster. Composite index column order is the most common indexing mistake I see.

02

VACUUM is not optional. In PostgreSQL, table bloat from dead tuples is silent until it isn't. A table that should be 2GB sitting at 18GB because autovacuum couldn't keep up — that's a performance cliff, not a gradual slide. Monitor it before it matters.

03

The query plan changes when the data changes. A query that ran in 12ms for six months starts taking 4 seconds. Nothing in the code changed. But the table grew from 50k to 5M rows and the planner switched strategies. Statistics staleness is a real production issue, not an edge case.

04

Locking problems look like slow queries. If a query is consistently slow only during business hours and fast at night, look at lock waits before looking at the query plan. The query is fine. Something is holding a lock it shouldn't.

05

Documentation written against a snapshot is wrong the day after migration. Schema docs need to come from a live connection, not an export. The only schema documentation worth trusting is the one generated from the catalog at the moment you read it.

Common questions

Do the tools work in air-gapped environments?
Yes. The tools run entirely on your own server. The only outbound connection they make is to your database — no telemetry, no license phone-home after activation, no analytics. Suitable for regulated industries, government environments, and air-gapped infrastructure.
What technical knowledge do I need to install the tools?
Basic comfort with a terminal is enough. Each tool ships with a setup script that handles the environment, dependencies, and configuration. You provide the database credentials; the script does the rest. Python 3.10+ and Node.js are the only prerequisites.
Are the books really free? What's the catch?
No catch. The books are free to read online, chapter by chapter. The philosophy is simple: if you find the content useful, you're more likely to try the tools. That's the business model — give away the knowledge, build trust, and let the tools speak for themselves.
Do you offer support for the tools?
Yes. Email support is included with every license. If something doesn't work in your environment, get in touch and we'll figure it out. The connection guide covers the most common setup issues, firewall configurations, and permission requirements for each database engine.
I use MySQL — which tools support it?
DB Hub supports all three engines: PostgreSQL, SQL Server, and MySQL. QueryOptimizer and DB Documenter currently support PostgreSQL and SQL Server. MySQL support is being added to the standalone tools. If MySQL is your primary engine, DB Hub is the right choice today.
