Chapter 24: Caching Strategies

Database performance is rarely just about writing better queries or adding the right indexes. At a certain scale, the bottleneck shifts from execution logic to how efficiently the system avoids doing work it has already done. Caching sits at the center of that idea. This chapter covers how PostgreSQL and SQL Server approach caching internally, how you as a DBA can observe and influence those mechanisms, and how application-layer and query-result caching strategies complement what the database engine provides. By the end of this chapter, you will have a clear picture of how data moves from disk to buffer to application, where the opportunities for optimization lie, and what can go wrong when caching is mismanaged.

---

How Databases Use Memory to Avoid Disk I/O

Every modern database engine operates on a fundamental principle: disk access is expensive, and memory access is cheap. When a query needs a page of data — whether a heap page, index page, or internal metadata — the engine first checks whether that page already lives in memory. If it does, the access is served immediately. If it does not, the engine reads it from disk, stores a copy in its in-memory buffer pool, and then serves the query. This buffer pool is the beating heart of database caching.

In PostgreSQL, this memory region is called shared buffers. Every backend process that executes a query shares access to this pool. When PostgreSQL reads a page from disk, it writes it into shared buffers, and subsequent requests for the same page find it there. The default size for shared buffers is 128 MB, a conservative value that almost every production deployment should raise significantly. A common starting point for a dedicated database server is 25% of total system RAM, though workload characteristics ultimately determine the right value.

In SQL Server, the equivalent structure is the Buffer Pool, managed by the Buffer Manager component. SQL Server is more aggressive by default — it claims as much RAM as the operating system allows (up to max server memory) and does not readily return it. SQL Server's buffer pool handles not just data pages but also index pages, execution plan caches, and connection memory. The two engines differ meaningfully in how they manage memory pressure and eviction, but the underlying goal is the same: keep frequently needed pages in memory.

Both engines use variations of the LRU (Least Recently Used) algorithm to decide which pages to evict when the buffer pool fills up. PostgreSQL uses a clock-sweep algorithm, a practical approximation of LRU that avoids the overhead of maintaining a strict ordered list. SQL Server uses a more sophisticated version that also considers page reference counts and system memory pressure signals. In both systems, hot pages — those accessed frequently — tend to stay in memory, while cold pages get pushed out as new data enters.

Understanding this behavior matters operationally. When you restart a database server, the buffer pool is cold — empty. Queries that were fast before the restart will suddenly be slow because every page access now requires a disk read. This is called a cold cache, and it explains why performance benchmarks should never be taken immediately after a server restart, and why production deployments use warm-up scripts or delayed traffic ramp-ups after maintenance windows.

Observing Buffer Cache Hit Rates

Before you can optimize caching, you need to measure it. Both PostgreSQL and SQL Server expose cache hit rate information through system views, and this metric tells you how often your workload is being served from memory versus disk.

In PostgreSQL, the pg_statio_user_tables view tracks buffer hits and disk reads per table. A buffer hit means the requested block was found in shared buffers. A block read means the engine had to go to disk (or the OS file system cache, discussed shortly).

PostgreSQL:

-- Cache hit ratio per table
SELECT
    schemaname,
    relname AS table_name,
    heap_blks_read AS disk_reads,
    heap_blks_hit  AS buffer_hits,
    ROUND(
        heap_blks_hit::numeric /
        NULLIF(heap_blks_hit + heap_blks_read, 0) * 100,
        2
    ) AS hit_ratio_pct
FROM pg_statio_user_tables
WHERE heap_blks_hit + heap_blks_read > 0
ORDER BY disk_reads DESC;

SQL Server:

-- Buffer cache hit ratio from performance counters
SELECT
    object_name,
    counter_name,
    cntr_value AS hit_ratio
FROM sys.dm_os_performance_counters
WHERE counter_name IN (
    'Buffer cache hit ratio',
    'Buffer cache hit ratio base'
)
AND object_name LIKE '%Buffer Manager%';

A healthy OLTP system typically shows buffer cache hit ratios above 95%, and most well-configured production databases run at 98–99%. If you are seeing ratios significantly below 95%, the first question is whether shared buffers (or SQL Server's max server memory) is sized appropriately for the working set of data your queries access regularly.

You can also look at which specific tables are generating the most disk reads, which helps prioritize tuning efforts. In PostgreSQL, tables with a persistently low hit ratio may benefit from either a larger shared_buffers value or investigation into whether large sequential scans are polluting the buffer pool with data that is only needed once.

PostgreSQL:

-- Identify tables with poor cache hit ratios
SELECT
    relname AS table_name,
    heap_blks_read,
    heap_blks_hit,
    ROUND(
        100.0 * heap_blks_hit /
        NULLIF(heap_blks_hit + heap_blks_read, 0),
        1
    ) AS hit_pct
FROM pg_statio_user_tables
WHERE heap_blks_read > 1000
ORDER BY hit_pct ASC
LIMIT 20;

SQL Server:

-- Pages in buffer pool per database
SELECT
    DB_NAME(database_id) AS database_name,
    COUNT(*) AS pages_in_buffer,
    COUNT(*) * 8 / 1024 AS size_mb
FROM sys.dm_os_buffer_descriptors
WHERE database_id > 4  -- exclude system databases
GROUP BY database_id
ORDER BY pages_in_buffer DESC;

In SQL Server, examining sys.dm_os_buffer_descriptors lets you see exactly how much of the buffer pool each database is consuming, which is valuable on shared instances. You can extend that query to the object level by joining through allocation units to find which tables are consuming the most buffer space — a table that holds a large fraction of the buffer pool but serves infrequent queries may be a candidate for moving to a separate filegroup or partitioning strategy.

The Operating System Page Cache and Its Role

There is a layer of caching that many DBAs overlook: the operating system's own page cache. When PostgreSQL reads a page from disk, the operating system reads it through the file system into its own kernel page cache before PostgreSQL copies it into shared buffers. This means a page may actually be cached in two places: the OS page cache and PostgreSQL's shared buffers.

This double-buffering is one reason why PostgreSQL's effective_cache_size parameter exists. This parameter does not allocate any memory — it is purely a hint to the query planner indicating how much memory is available for caching across both shared buffers and the OS page cache combined. Setting it accurately helps the planner make better decisions about whether index scans (which benefit from cached pages) or sequential scans are more appropriate for a given query.

SQL Server takes a different approach on Windows through its Large Pages and Lock Pages in Memory features. When configured, SQL Server can lock its buffer pool pages in physical RAM, preventing the operating system from paging them out under memory pressure. This eliminates OS-level eviction but requires specific Windows privileges and is typically enabled only on dedicated database servers where memory contention with other processes is not a concern.

PostgreSQL does not use direct I/O by default — it goes through the OS page cache. This is deliberate; PostgreSQL benefits from the OS cache as a second tier. SQL Server on Windows can bypass the OS cache for some operations using FILE_FLAG_NO_BUFFERING, but the default behavior also goes through the Windows cache manager. On Linux, SQL Server behaves similarly to PostgreSQL in relying on the OS page cache.

The practical implication is that tuning a database server means thinking about total memory holistically. On a 64 GB PostgreSQL server, setting shared_buffers to 16 GB and letting the OS use another 20–30 GB as a page cache is a reasonable configuration. The OS cache acts as an overflow layer — data that does not fit in shared buffers but is still needed frequently often gets served from the OS cache rather than actual disk, which is still significantly faster than a true cold disk read.

Query Plan Caching and Prepared Statements

Parsing, analyzing, and planning a query takes CPU time. For a simple lookup query that executes thousands of times per second, re-planning the query on every execution is wasteful. Both PostgreSQL and SQL Server maintain a plan cache — an in-memory store of compiled query plans — that allows previously seen queries to skip the planning phase entirely.

In SQL Server, the Procedure Cache (also called the Plan Cache) stores execution plans for ad-hoc queries, stored procedures, and prepared statements. SQL Server uses parameterization to recognize when two queries differ only in their literal values and can share a plan. You can inspect the plan cache through sys.dm_exec_cached_plans and related DMVs.

SQL Server:

-- Top cached plans by execution count and CPU cost
SELECT TOP 20
    qs.execution_count,
    qs.total_worker_time / 1000 AS total_cpu_ms,
    qs.total_worker_time / qs.execution_count / 1000 AS avg_cpu_ms,
    SUBSTRING(st.text, 1, 200) AS query_text,
    qp.query_plan
FROM sys.dm_exec_query_stats AS qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) AS st
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) AS qp
ORDER BY qs.execution_count DESC;

PostgreSQL:

-- Cached plan usage via pg_prepared_statements (session-level)
SELECT
    name,
    statement,
    prepare_time,
    calls,
    total_exec_time,
    mean_exec_time
FROM pg_prepared_statements
ORDER BY calls DESC;

PostgreSQL handles plan caching differently from SQL Server. In PostgreSQL, query plan caching happens primarily through prepared statements. When a client prepares a statement using PREPARE or a protocol-level prepare (as most connection poolers and ORMs do), PostgreSQL initially uses a generic plan after enough executions convince it that the generic plan is efficient. For the first five executions of a prepared statement, PostgreSQL creates a custom plan for each execution using the actual parameter values. After five executions, it evaluates whether a generic cached plan would be cheaper on average.

This behavior — called plan stabilization — exists because the optimal plan for WHERE customer_id = 1 might differ from WHERE customer_id = 99999 depending on data distribution. A generic plan must work reasonably well for all parameter values. The plan_cache_mode parameter (introduced in PostgreSQL 12) allows DBAs to force either always using custom plans (force_custom_plan) or always using generic cached plans (force_generic_plan), which is sometimes useful when dealing with parameters that have highly skewed distributions.

SQL Server's equivalent challenge is parameter sniffing. When SQL Server first compiles a plan for a stored procedure, it uses the actual parameter values at that moment to estimate cardinality, and the resulting plan is cached. If those initial values were unusual — either very selective or very broad — the cached plan may perform poorly for typical parameter values. This is one of the most common performance problems in SQL Server, and solutions range from OPTION (RECOMPILE) on the query, to OPTIMIZE FOR hints, to breaking procedures into smaller pieces. From PostgreSQL 16 onward, query identifier tracking through pg_stat_statements makes it easier to identify plan instability patterns without needing to instrument the application layer.

Application-Layer Caching Strategies

The database buffer pool and plan cache are internal, automatic mechanisms. But at scale, even a perfectly tuned database will struggle if the application sends the same expensive queries repeatedly for data that does not change frequently. This is where application-layer caching strategies enter the picture, and DBAs need to understand them — not necessarily to implement them, but to design the database tier so that caching at higher layers works effectively.

The most common pattern is a read-through cache using a system like Redis or Memcached. The application checks the cache first; on a miss, it queries the database, stores the result in the cache with a TTL (time-to-live), and returns the result. Subsequent requests for the same data hit the cache and never reach the database. From the DBA's perspective, this pattern creates a tiered system where the database handles cold reads, initial population, and writes, while the cache handles the bulk of read traffic.

This has an important implication for schema design. Data that benefits most from application-level caching tends to be read-heavy, relatively stable, and natural to serialize as a whole unit — a product catalog row, a user profile, a configuration setting. Data that changes frequently, or that requires joining many tables to assemble, is harder to cache correctly because cache invalidation becomes complex.

Write-through caching ensures that every write goes to both the cache and the database simultaneously, keeping the cache consistent. Write-behind caching (also called write-back) buffers writes in the cache and flushes them to the database asynchronously. Write-behind can significantly reduce database write throughput pressure, but it introduces risk: if the cache layer fails before flushing, those writes are lost. As a DBA, you should be skeptical of write-behind designs for anything that requires durability, and you should ensure that any such design has explicit failure handling and monitoring.

One pattern that databases themselves support, and that serves as an alternative to external caches for certain workloads, is materialized views. A materialized view pre-computes the result of a complex query and stores it as a physical table. Queries that would otherwise join several large tables and aggregate millions of rows can instead read from the materialized view in milliseconds.

PostgreSQL:

-- Create a materialized view for a complex aggregation
CREATE MATERIALIZED VIEW monthly_sales_summary AS
SELECT
    DATE_TRUNC('month', order_date) AS month,
    product_category,
    SUM(order_total) AS total_revenue,
    COUNT(*) AS order_count,
    AVG(order_total) AS avg_order_value
FROM orders
JOIN products USING (product_id)
GROUP BY 1, 2
WITH DATA;

-- Refresh it on a schedule
REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_sales_summary;

SQL Server:

-- Indexed view (SQL Server's equivalent of a materialized view)
CREATE VIEW monthly_sales_summary
WITH SCHEMABINDING
AS
SELECT
    DATEFROMPARTS(YEAR(o.order_date), MONTH(o.order_date), 1) AS month,
    p.product_category,
    SUM(o.order_total) AS total_revenue,
    COUNT_BIG(*) AS order_count
FROM dbo.orders AS o
JOIN dbo.products AS p ON o.product_id = p.product_id
GROUP BY
    DATEFROMPARTS(YEAR(o.order_date), MONTH(o.order_date), 1),
    p.product_category;

-- Create a unique clustered index to materialize it
CREATE UNIQUE CLUSTERED INDEX IX_monthly_sales_summary
ON monthly_sales_summary (month, product_category);

PostgreSQL's REFRESH MATERIALIZED VIEW CONCURRENTLY is notable because it allows the view to remain queryable during the refresh — locks are minimal, and readers see the previous version of the data until the new version is ready. The CONCURRENTLY option requires a unique index on the materialized view. SQL Server's indexed views are automatically maintained as the underlying data changes, which is both convenient and a source of overhead on write-heavy tables. The DBA must weigh the write overhead against the query performance gain.

Cache Eviction, Pollution, and Connection Pooling Interactions

Understanding that caching exists is one thing; understanding when it works against you is another. Three specific problems — cache eviction under pressure, cache pollution from large scans, and the interaction between connection poolers and plan caches — represent the production edge cases that experienced DBAs encounter and that are rarely documented clearly.

Cache eviction under memory pressure occurs when the buffer pool fills and the eviction algorithm must push out pages. In PostgreSQL, if a large ETL job or reporting query performs sequential scans over many tables, it will load millions of pages into shared buffers, potentially evicting hot pages that the OLTP workload depends on. PostgreSQL partially mitigates this through a mechanism called ring buffers for sequential scans: large sequential scans use a small, fixed ring of buffer slots rather than populating the full shared buffer pool. This is automatic for scans over tables larger than a certain threshold (roughly 256 KB by default, controlled by effective_io_concurrency and internal heuristics). The result is that sequential scans of large tables do not destroy the buffer pool, though they can still contribute to pressure.

SQL Server provides the Resource Governor feature to limit memory grants to specific workloads. By classifying connections into resource pools, you can ensure that a reporting query cannot consume buffer pool space at the expense of OLTP queries. This is one of the most underused features in SQL Server environments where mixed workloads run on the same instance.

Cache pollution from analytics workloads is a real problem that many teams address by routing analytical queries to a read replica. Both PostgreSQL streaming replication and SQL Server Always On Availability Groups support read-intent replicas. Directing reporting workloads to the replica not only keeps the primary's buffer pool clean but also offloads CPU and I/O from the primary entirely.

Connection poolers and plan cache invalidation create a subtle problem. Tools like PgBouncer (for PostgreSQL) in session-mode maintain prepared statements across connection reuse, which is efficient. In transaction-mode pooling — where a client gets a backend connection only for the duration of a transaction — prepared statements cannot persist across transactions, because the client might get a different backend each time. This forces applications to use protocol-level prepared statements carefully, or to fall back to non-prepared queries, giving up plan cache benefits. When configuring a connection pooler, the DBA must understand whether the application relies on server-side prepared statements and choose the pooling mode accordingly.

In SQL Server, connection pooling is typically handled by the client driver (ADO.NET, JDBC), and the plan cache is server-side and query-keyed rather than connection-keyed. Switching pooling modes is less disruptive to plan caching. However, frequent DBCC FREEPROCCACHE calls — sometimes used as a troubleshooting step — clear the entire plan cache, forcing recompilation of everything. This is rarely the right production action; targeted eviction using the plan handle is always preferable.

PostgreSQL:

-- Check for plans that may be causing problems (high planning time)
SELECT
    query,
    calls,
    ROUND(total_plan_time::numeric / calls, 2) AS avg_plan_ms,
    ROUND(total_exec_time::numeric / calls, 2) AS avg_exec_ms
FROM pg_stat_statements
WHERE calls > 100
  AND total_plan_time / calls > total_exec_time / calls
ORDER BY avg_plan_ms DESC
LIMIT 10;

SQL Server:

-- Find plans with high recompile counts
SELECT
    qs.sql_handle,
    qs.plan_generation_num,
    qs.execution_count,
    qs.total_worker_time / 1000 AS total_cpu_ms,
    SUBSTRING(st.text, 1, 300) AS query_text
FROM sys.dm_exec_query_stats AS qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) AS st
WHERE qs.plan_generation_num > 1
ORDER BY qs.plan_generation_num DESC;

A query with a high plan_generation_num in SQL Server is being recompiled repeatedly — typically because of statistics changes, schema changes, or explicit RECOMPILE hints. Each recompile consumes CPU and represents time where the query is not being executed. If recompiles are frequent and the query is hot, investigate whether the triggering cause (often a statistics update on a large table) can be controlled through manual statistics management or table variable instead of temp table usage.

---

Key Takeaways

  • → **The buffer pool is the primary cache** in both PostgreSQL (shared buffers) and SQL Server (Buffer Pool). Sizing it appropriately for the working data set is the single most impactful caching configuration decision a DBA makes.
  • → **Cache hit rate is a key health metric.** Values below 95% warrant investigation into buffer pool sizing, workload patterns, and whether large analytical scans are evicting hot OLTP pages. Both engines expose this data through system views and DMVs.
  • → **Plan caching reduces planning overhead**, but each engine has failure modes — parameter sniffing in SQL Server and plan instability with skewed parameters in PostgreSQL. Understanding how to diagnose and address these without blunt tools like cache-clearing is an important production skill.
  • → **Application-layer caching through materialized views, Redis, or similar systems** reduces database read pressure for stable data, but requires careful invalidation strategies. Materialized views — whether PostgreSQL's refreshable variant or SQL Server's auto-maintained indexed views — are a powerful database-native option that does not require external infrastructure.
  • → **Cache pollution and memory pressure from mixed workloads** are common in production environments that run OLTP and analytics on the same instance. Read replicas, resource pools, and understanding each engine's sequential scan buffering behavior are the tools for keeping caches effective under these conditions.
