Chapter 32: VPC, Private Endpoints, and Network Security

Network security for cloud databases is where experienced on-prem DBAs often feel the most disoriented. On-premises, you controlled the physical network, configured firewall rules on the host, and trusted that your database server sat inside a known perimeter. In the cloud, that perimeter dissolves and reforms as a collection of software-defined constructs — VPCs, subnets, security groups, private endpoints, and DNS resolution chains — that you must understand completely before putting production data anywhere near them. This chapter covers how AWS, Azure, and Google Cloud each implement database network isolation, how to connect applications to managed databases without exposing them to the public internet, and how to build defense-in-depth network security that will survive a security audit. The patterns here apply equally to RDS, Aurora, Azure SQL, Cloud SQL, and AlloyDB.

---

32.1 Virtual Networks: The Foundation of Cloud Database Isolation

Before touching a single database configuration, you need a clear mental model of how virtual networks work across the three clouds. The terminology differs but the concepts are nearly identical.

Mapping Your On-Prem Concepts

On-premises, you had VLANs, subnets, ACLs, and firewall rules implemented in hardware. In the cloud:

| On-Premises Concept | AWS | Azure | GCP |

|---|---|---|---|

| Data center network | VPC | Virtual Network (VNet) | VPC |

| VLAN / network segment | Subnet | Subnet | Subnet |

| Firewall rules (host-based) | Security Group | NSG (Network Security Group) | Firewall Rules |

| Network-level ACL | Network ACL | NSG on subnet | VPC Firewall Rules |

| Private RFC 1918 space | Private subnets | Private subnets | Private subnets |

| DNS server | Route 53 Private Hosted Zone | Azure Private DNS Zone | Cloud DNS private zone |

The one thing that consistently surprises on-prem DBAs: security groups in AWS are stateful (return traffic is automatically allowed), while Network ACLs are stateless. For database connectivity, you typically work with security groups, not NACLs directly.

Subnet Architecture for Database Workloads

The standard production layout places your database instances in private subnets — subnets with no route to an internet gateway. Your application tier sits in separate subnets, and all traffic flows through security group rules, not public IP addresses.

For RDS and Aurora, AWS requires subnets in at least two Availability Zones to create a DB Subnet Group. This is not optional — it is a hard requirement even for single-AZ deployments:

# Create a DB subnet group spanning two AZs — required before any RDS deployment
aws rds create-db-subnet-group \
  --db-subnet-group-name prod-db-subnet-group \
  --db-subnet-group-description "Production database subnets" \
  --subnet-ids subnet-0a1b2c3d4e5f67890 subnet-0f9e8d7c6b5a43210 \
  --tags Key=Environment,Value=production Key=Team,Value=database

# Verify the subnet group is in the correct VPC and AZs
aws rds describe-db-subnet-groups \
  --db-subnet-group-name prod-db-subnet-group \
  --query 'DBSubnetGroups[0].{VpcId:VpcId,Subnets:Subnets[*].{AZ:SubnetAvailabilityZone.Name,Id:SubnetIdentifier}}'

The equivalent in Terraform, which is how most production environments should be managing this:

# Terraform: VPC layout for production RDS deployment
resource "aws_vpc" "prod" {
  cidr_block           = "10.0.0.0/16"
  enable_dns_hostnames = true
  enable_dns_support   = true

  tags = {
    Name        = "prod-vpc"
    Environment = "production"
  }
}

resource "aws_subnet" "database_az1" {
  vpc_id            = aws_vpc.prod.id
  cidr_block        = "10.0.32.0/24"
  availability_zone = "us-east-1a"

  tags = {
    Name = "prod-db-subnet-az1"
    Tier = "database"
  }
}

resource "aws_subnet" "database_az2" {
  vpc_id            = aws_vpc.prod.id
  cidr_block        = "10.0.33.0/24"
  availability_zone = "us-east-1b"

  tags = {
    Name = "prod-db-subnet-az2"
    Tier = "database"
  }
}

resource "aws_db_subnet_group" "prod" {
  name       = "prod-db-subnet-group"
  subnet_ids = [aws_subnet.database_az1.id, aws_subnet.database_az2.id]

  tags = {
    Name = "Production DB Subnet Group"
  }
}

# Security group: database instances
# Only allows inbound from the application tier security group
resource "aws_security_group" "database" {
  name        = "prod-database-sg"
  description = "Security group for RDS/Aurora instances"
  vpc_id      = aws_vpc.prod.id

  ingress {
    description     = "PostgreSQL from application tier"
    from_port       = 5432
    to_port         = 5432
    protocol        = "tcp"
    security_groups = [aws_security_group.application.id]
  }

  egress {
    description = "All outbound — needed for IAM auth token validation"
    from_port   = 0
    to_port     = 0
    protocol    = "-1"
    cidr_blocks = ["0.0.0.0/0"]
  }

  tags = {
    Name = "prod-database-sg"
  }
}

What You Cannot Control in Managed Services

On self-managed PostgreSQL, you had pg_hba.conf for host-based authentication, could bind to specific interfaces, and could install host-level firewalls. With RDS:

  • → You cannot access the operating system or configure pg_hba.conf directly
  • → You cannot install host-based firewall rules
  • → You cannot bind to a specific IP address — RDS assigns its own private IP within your subnet
  • → The RDS endpoint DNS name resolves to different IPs during failover — never hardcode IPs in connection strings

    • This last point is where on-prem DBAs get burned. Always use the CNAME endpoint, never the underlying IP address.

    32.2 Private Endpoints and Private Link: Eliminating Public Exposure

    The default configuration for many cloud services — including Azure SQL — allows public connectivity. Turning it off and routing all traffic through private endpoints is non-negotiable for production database workloads.

    AWS PrivateLink and RDS

    For RDS and Aurora, the database instance itself sits in your private subnet and never has a public IP if you configure it correctly (PubliclyAccessible: false). But connectivity from other AWS services — Lambda, ECS tasks, other VPCs — requires either VPC peering, AWS Transit Gateway, or VPC endpoints for the AWS APIs your database tooling uses.

    For Aurora Serverless v2 and RDS Proxy, you also need VPC endpoints for Secrets Manager if you are rotating credentials:

    # Verify RDS instance is NOT publicly accessible — this should be false in production
aws rds describe-db-instances \
  --db-instance-identifier prod-postgres-01 \
  --query 'DBInstances[0].{PubliclyAccessible:PubliclyAccessible,Endpoint:Endpoint,VpcId:DBSubnetGroup.VpcId}'

# Create VPC endpoint for Secrets Manager — required for RDS Proxy credential rotation
# without routing through the public internet
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-0a1b2c3d4e5f67890 \
  --service-name com.amazonaws.us-east-1.secretsmanager \
  --vpc-endpoint-type Interface \
  --subnet-ids subnet-0a1b2c3d4e5f67890 subnet-0f9e8d7c6b5a43210 \
  --security-group-ids sg-0a1b2c3d4e5f67890 \
  --private-dns-enabled \
  --tag-specifications 'ResourceType=vpc-endpoint,Tags=[{Key=Name,Value=secretsmanager-endpoint}]'

    Azure SQL Private Endpoint Configuration

    Azure SQL has both a public endpoint (disabled by requirement in secure environments) and supports Private Link, which creates a private IP in your VNet that maps to the Azure SQL managed instance or logical server. The Azure CLI workflow:

    # Step 1: Disable public access on Azure SQL Server
az sql server update \
  --name prod-sql-server \
  --resource-group prod-rg \
  --enable-public-network false

# Step 2: Create private endpoint in your VNet
az network private-endpoint create \
  --name prod-sql-private-endpoint \
  --resource-group prod-rg \
  --vnet-name prod-vnet \
  --subnet prod-db-subnet \
  --private-connection-resource-id \
    /subscriptions/{subscription-id}/resourceGroups/prod-rg/providers/Microsoft.Sql/servers/prod-sql-server \
  --group-id sqlServer \
  --connection-name prod-sql-private-link

# Step 3: Create private DNS zone for Azure SQL
az network private-dns zone create \
  --resource-group prod-rg \
  --name "privatelink.database.windows.net"

# Step 4: Link DNS zone to your VNet
az network private-dns link vnet create \
  --resource-group prod-rg \
  --zone-name "privatelink.database.windows.net" \
  --name prod-vnet-dns-link \
  --virtual-network prod-vnet \
  --registration-enabled false

# Step 5: Create DNS record for the private endpoint
az network private-endpoint dns-zone-group create \
  --resource-group prod-rg \
  --endpoint-name prod-sql-private-endpoint \
  --name sql-dns-zone-group \
  --private-dns-zone "privatelink.database.windows.net" \
  --zone-name sql

    After completing this sequence, your application connects to prod-sql-server.database.windows.net and the private DNS zone resolves it to the private IP address in your VNet — not the public Azure SQL endpoint. Verify this from within the VNet:

    # From a VM in the VNet — should resolve to a 10.x.x.x private IP
nslookup prod-sql-server.database.windows.net
# Expected: Non-authoritative answer showing a 10.x.x.x address
# If it shows a public Azure IP, your DNS zone or link is misconfigured

    Google Cloud Private Service Connect for Cloud SQL

    Google Cloud SQL uses Private Service Connect (PSC) or older VPC peering. PSC is the current recommended approach and creates a forwarding rule with a private IP in your VPC:

    # Enable Private Service Connect on Cloud SQL instance
gcloud sql instances patch prod-postgres-instance \
  --project=my-project \
  --enable-private-service-connect \
  --allowed-psc-projects=consumer-project-id

# Get the PSC service attachment URI from the instance
gcloud sql instances describe prod-postgres-instance \
  --project=my-project \
  --format='value(settings.ipConfiguration.pscConfig.serviceAttachmentLink)'

# Create a forwarding rule pointing to the PSC attachment
gcloud compute forwarding-rules create prod-cloudsql-psc \
  --region=us-central1 \
  --network=prod-vpc \
  --address=prod-cloudsql-private-ip \
  --target-service-attachment=projects/PRODUCER_PROJECT/regions/us-central1/serviceAttachments/ATTACHMENT_NAME \
  --project=my-project

# Create DNS entry so applications use a consistent hostname
gcloud dns record-sets create prod-postgres.internal.example.com. \
  --rrdatas=10.0.32.15 \
  --type=A \
  --ttl=300 \
  --zone=internal-zone \
  --project=my-project

    The Common Gotcha: DNS Resolution

    Private endpoints are useless if your application's DNS resolver bypasses the private zone. This happens most frequently when:

    1. Applications run in a different VPC without VPC peering and shared DNS resolution

    2. Containers use their own DNS servers instead of the VPC resolver

    3. On-premises systems connecting over VPN or Direct Connect use corporate DNS that does not forward to the cloud private zones

    Test DNS resolution from every network segment that needs database access, not just from one representative host.

    32.3 Security Groups, Firewall Rules, and the Principle of Least Privilege

    Think of security groups as the evolved equivalent of iptables rules — but managed, stateful, and attached to resources rather than hosts. For databases, the configuration philosophy is the same as on-prem: explicit allow, implicit deny, minimum necessary access.

    Security Group Design for Multi-Tier Applications

    The correct pattern chains security groups by reference, not by CIDR range. This means the database security group allows traffic from the application security group, not from 10.0.0.0/8. When auto-scaling adds application instances, they inherit the application security group automatically and immediately get database access — no rule changes required.

    # Audit existing RDS security group rules — look for 0.0.0.0/0 which should never appear
aws ec2 describe-security-groups \
  --group-ids sg-0a1b2c3d4e5f67890 \
  --query 'SecurityGroups[0].IpPermissions[*].{
    Port:FromPort,
    Protocol:IpProtocol,
    Sources:IpRanges[*].CidrIp,
    SGSources:UserIdGroupPairs[*].GroupId
  }' \
  --output table

# Add RDS Proxy security group as an allowed source to the database SG
# This is required when you introduce RDS Proxy in front of your instances
aws ec2 authorize-security-group-ingress \
  --group-id sg-database-sg-id \
  --protocol tcp \
  --port 5432 \
  --source-group sg-rdsproxy-sg-id

# Verify: RDS Proxy needs outbound to the database, database needs inbound from proxy
# Application needs outbound to proxy, proxy needs inbound from application
# These are four distinct security group rules, not two

    Azure SQL Firewall Rules and Service Endpoints

    Azure SQL has two distinct layers: server-level firewall rules (IP-based, older model) and VNet service endpoints or Private Endpoints (the correct production model). Service endpoints are a middle-ground — they restrict access to specific VNet subnets but traffic still traverses the Azure backbone routing to the public endpoint. Private Endpoints are stronger: traffic stays entirely within your VNet.

    # Remove overly permissive firewall rules from Azure SQL
# First, list all existing rules
az sql server firewall-rule list \
  --server prod-sql-server \
  --resource-group prod-rg \
  --output table

# Delete any rule that allows 0.0.0.0 to 255.255.255.255 ("Allow Azure Services")
# This rule allows ALL Azure services, including other tenants' services
az sql server firewall-rule delete \
  --server prod-sql-server \
  --resource-group prod-rg \
  --name "AllowAllWindowsAzureIps"

# Add VNet rule for your application subnet (service endpoint approach)
az sql server vnet-rule create \
  --server prod-sql-server \
  --resource-group prod-rg \
  --name allow-app-subnet \
  --vnet-name prod-vnet \
  --subnet app-subnet

    Note that "Allow Azure Services" firewall rule in Azure SQL is frequently misunderstood. It does not mean "allow only my services" — it means allow any traffic originating from any Azure datacenter IP, including other customers' Azure services. Remove it.

    GCP Firewall Rules for Cloud SQL

    Cloud SQL with Private Service Connect or VPC peering does not use GCP firewall rules directly for database port access — the peering or PSC handles routing. However, you still need firewall rules to control which compute resources can reach the database:

    # GCP Firewall rule: allow PostgreSQL from application instances
# Tagged approach — apply 'db-client' tag to instances that need database access
gcloud compute firewall-rules create allow-postgres-from-app \
  --project=my-project \
  --network=prod-vpc \
  --direction=INGRESS \
  --priority=1000 \
  --action=ALLOW \
  --rules=tcp:5432 \
  --source-tags=db-client \
  --target-tags=db-server \
  --description="Allow PostgreSQL from tagged application instances"

# For Cloud SQL specifically, also allow the Cloud SQL proxy if used
gcloud compute firewall-rules create allow-cloudsql-proxy \
  --project=my-project \
  --network=prod-vpc \
  --direction=EGRESS \
  --priority=1000 \
  --action=ALLOW \
  --rules=tcp:3307 \
  --destination-ranges=35.191.0.0/16,130.211.0.0/22 \
  --description="Allow outbound to Cloud SQL proxy API"

    32.4 Cross-VPC Connectivity: Peering, Transit Gateway, and PrivateLink

    Production environments rarely have a single VPC. You will have dedicated VPCs for different applications, different environments (dev, staging, prod), or different business units. Getting databases to talk across these boundaries without exposing them to the internet requires deliberate architecture.

    AWS VPC Peering vs. Transit Gateway

    VPC peering connects two VPCs directly. It does not support transitive routing — if VPC A peers with VPC B, and VPC B peers with VPC C, VPC A cannot reach VPC C through VPC B. For database access patterns where multiple application VPCs need to reach a central database VPC, this forces a hub-and-spoke design with Transit Gateway:

    # Terraform: Transit Gateway setup for central database VPC access
resource "aws_ec2_transit_gateway" "main" {
  description                     = "Central TGW for database connectivity"
  default_route_table_association = "disable"
  default_route_table_propagation = "disable"
  dns_support                     = "enable"
  vpn_ecmp_support                = "enable"

  tags = {
    Name = "prod-tgw"
  }
}

# Attach the database VPC to the Transit Gateway
resource "aws_ec2_transit_gateway_vpc_attachment" "database_vpc" {
  transit_gateway_id = aws_ec2_transit_gateway.main.id
  vpc_id             = aws_vpc.database.id
  subnet_ids         = [aws_subnet.database_az1.id, aws_subnet.database_az2.id]

  dns_support    = "enable"
  ipv6_support   = "disable"

  tags = {
    Name = "db-vpc-attachment"
  }
}

# Attach an application VPC
resource "aws_ec2_transit_gateway_vpc_attachment" "app_vpc" {
  transit_gateway_id = aws_ec2_transit_gateway.main.id
  vpc_id             = aws_vpc.application.id
  subnet_ids         = [aws_subnet.app_az1.id, aws_subnet.app_az2.id]

  tags = {
    Name = "app-vpc-attachment"
  }
}

# Route table: application VPC -> database VPC only
# Database traffic is 10.0.0.0/16, restrict to only that CIDR
resource "aws_ec2_transit_gateway_route_table" "app_to_db" {
  transit_gateway_id = aws_ec2_transit_gateway.main.id

  tags = {
    Name = "app-to-db-only"
  }
}

resource "aws_ec2_transit_gateway_route" "app_to_db" {
  destination_cidr_block         = "10.0.0.0/16"  # Database VPC CIDR
  transit_gateway_attachment_id  = aws_ec2_transit_gateway_vpc_attachment.database_vpc.id
  transit_gateway_route_table_id = aws_ec2_transit_gateway_route_table.app_to_db.id
}

    Connecting On-Premises to Cloud Databases

    This is the most common architecture during migration phases: on-premises applications need to reach cloud databases, or cloud databases need to reach on-premises systems for DMS replication. The options in order of preference:

    1. AWS Direct Connect / Azure ExpressRoute / GCP Dedicated Interconnect — dedicated physical connection, predictable latency (typically 1–5ms for co-located facilities), no internet exposure. Required for any data volume above a few hundred MB/day and for compliance environments

    2. Site-to-Site VPN — encrypted tunnel over internet, variable latency (20–100ms typical), acceptable for lower-volume migration workloads

    3. Public internet with SSL/TLS — last resort, acceptable only for development

    For Direct Connect with RDS, you still need a Virtual Private Gateway attached to your VPC and route propagation configured. The database endpoint resolves to a private IP that routes over the Direct Connect circuit:

    # Verify that RDS endpoint resolves to a private IP from on-premises
# (run this from an on-premises host after Direct Connect + BGP route propagation)
dig +short prod-postgres-01.cluster-xyz.us-east-1.rds.amazonaws.com
# Should return: 10.x.x.x (private IP in your VPC subnet)
# If returns: 52.x.x.x or 34.x.x.x (public AWS IP), your routing is misconfigured

# Check route propagation from Virtual Private Gateway
aws ec2 describe-route-tables \
  --filters Name=vpc-id,Values=vpc-0a1b2c3d4e5f67890 \
  --query 'RouteTables[*].Routes[?GatewayId!=`local`].{Destination:DestinationCidrBlock,Target:GatewayId,State:State}'

    Azure VNet Peering for SQL Managed Instance

    Azure SQL Managed Instance (not the logical server) requires its own dedicated subnet within a VNet and has strict requirements: the subnet must be delegated to Microsoft.Sql/managedInstances, must have a route table attached, and must have specific NSG rules. Peering from application VNets to the Managed Instance VNet requires the peering to allow gateway transit:

    # Create VNet peering from application VNet to SQL MI VNet
az network vnet peering create \
  --name app-to-sqlmi-peering \
  --resource-group prod-rg \
  --vnet-name app-vnet \
  --remote-vnet /subscriptions/{sub-id}/resourceGroups/prod-rg/virtualNetworks/sqlmi-vnet \
  --allow-vnet-access true \
  --allow-forwarded-traffic true

# Create the reverse peering (required — Azure peering is not automatically bidirectional)
az network vnet peering create \
  --name sqlmi-to-app-peering \
  --resource-group prod-rg \
  --vnet-name sqlmi-vnet \
  --remote-vnet /subscriptions/{sub-id}/resourceGroups/prod-rg/virtualNetworks/app-vnet \
  --allow-vnet-access true \
  --allow-forwarded-traffic true

# Verify peering state — both sides must show "Connected" not "Initiated"
az network vnet peering show \
  --name app-to-sqlmi-peering \
  --resource-group prod-rg \
  --vnet-name app-vnet \
  --query peeringState
