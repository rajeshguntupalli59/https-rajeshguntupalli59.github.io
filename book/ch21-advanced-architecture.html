Chapter 21: Advanced Architecture Patterns

Architecture is where database administration meets systems engineering. You can tune queries all day and optimize indexes until they're perfect, but if the underlying architecture is wrong, you're building on sand. This chapter covers the high-level structural patterns that determine how databases are deployed, scaled, and made resilient — from the foundational decision of how data gets distributed across nodes, to the operational complexities of running multi-region deployments that survive datacenter failures. These patterns show up in every serious production environment, and understanding them deeply changes how you think about every other decision you make as a DBA.

---

How Data Gets Distributed: Sharding, Partitioning, and Replication

Before any advanced architecture can be understood, you need a precise mental model of the three fundamental ways data gets spread across infrastructure. These concepts are often conflated in conversation, but confusing them in practice leads to catastrophic design mistakes.

Partitioning splits a single logical table into multiple physical segments, usually on the same server or cluster. The application sees one table; the database engine routes reads and writes to the correct partition behind the scenes. Partitioning primarily solves performance problems — queries that touch only recent data don't scan years of history, and maintenance operations like archival and vacuuming can target individual partitions. Both PostgreSQL and SQL Server have mature declarative partitioning support.

Replication creates copies of data across multiple servers. It solves availability and read scalability. The primary accepts writes; replicas follow behind and serve reads. Replication doesn't reduce the data volume on any single node — every replica holds the full dataset. It buys you redundancy and read throughput, not horizontal write scaling.

Sharding distributes different rows of a table to different servers based on a shard key. A user table sharded by user_id might put users 1–1,000,000 on shard A and users 1,000,001–2,000,000 on shard B. Each shard holds a fraction of the total data. Sharding is the only pattern that scales write throughput horizontally, because writes go to different machines. It also dramatically complicates queries that span shard boundaries, cross-shard joins, and distributed transactions.

Understanding which problem you actually have determines which tool you reach for. A 500GB table with slow queries from full scans probably needs partitioning. A reporting workload overwhelming the primary needs a read replica. A write-heavy application that has outgrown any single server's I/O capacity needs sharding.

PostgreSQL's native table partitioning uses the CREATE TABLE ... PARTITION BY syntax introduced in version 10 and matured significantly in versions 11–14. SQL Server's equivalent is the partition function and partition scheme model, which maps ranges to filegroups.

PostgreSQL:

-- Create a range-partitioned table for time-series events
CREATE TABLE events (
    event_id    BIGSERIAL,
    occurred_at TIMESTAMPTZ NOT NULL,
    user_id     BIGINT,
    event_type  TEXT,
    payload     JSONB
) PARTITION BY RANGE (occurred_at);

-- Create monthly partitions
CREATE TABLE events_2024_01
    PARTITION OF events
    FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE events_2024_02
    PARTITION OF events
    FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

-- Default partition catches anything outside defined ranges
CREATE TABLE events_default
    PARTITION OF events DEFAULT;

-- Verify partition pruning is working
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM events
WHERE occurred_at BETWEEN '2024-01-15' AND '2024-01-20';

SQL Server:

-- Create a partition function mapping date ranges to partitions
CREATE PARTITION FUNCTION pf_events_monthly (DATETIME2)
AS RANGE RIGHT FOR VALUES (
    '2024-01-01',
    '2024-02-01',
    '2024-03-01'
);

-- Create a partition scheme mapping partitions to filegroups
CREATE PARTITION SCHEME ps_events_monthly
AS PARTITION pf_events_monthly
ALL TO ([PRIMARY]);  -- In production, map to separate filegroups

-- Create the partitioned table
CREATE TABLE events (
    event_id    BIGINT IDENTITY(1,1),
    occurred_at DATETIME2       NOT NULL,
    user_id     BIGINT,
    event_type  NVARCHAR(100),
    payload     NVARCHAR(MAX)
) ON ps_events_monthly(occurred_at);

-- Check partition row counts
SELECT
    p.partition_number,
    p.rows,
    rv.value AS boundary_value
FROM sys.partitions p
JOIN sys.tables t ON p.object_id = t.object_id
LEFT JOIN sys.partition_range_values rv
    ON rv.function_id = (
        SELECT ps.function_id
        FROM sys.partition_schemes ps
        JOIN sys.indexes i ON i.data_space_id = ps.data_space_id
        WHERE i.object_id = t.object_id AND i.index_id IN (0,1)
    )
    AND rv.boundary_id = p.partition_number - 1
WHERE t.name = 'events';

The key architectural insight here is that partitioning and sharding exist at different layers of abstraction. Partitioning is a storage optimization within a single logical database. Sharding requires application-layer awareness or a routing proxy that knows which physical node to contact. Many teams reach for sharding prematurely when partitioning would have solved their problem with far less operational complexity.

CQRS and Read/Write Splitting in Practice

Command Query Responsibility Segregation — CQRS — is an application architecture pattern with deep database implications. The core idea is that the data model optimized for writing is often poorly suited for reading, and vice versa. Writes need normalized data to avoid anomalies. Reads, especially for reporting and analytics, need denormalized, pre-aggregated, or differently indexed data to return fast results.

At the database level, CQRS typically manifests as two distinct data stores or schemas: a write-optimized transactional store and a read-optimized query store. In practice, DBAs implement this spectrum several ways:

Physical read replicas are the simplest form. The application routes all writes to the primary and all reads to one or more replicas. PostgreSQL's streaming replication and SQL Server's Always On Availability Groups both support this pattern. The tradeoff is replication lag — a replica might be a few milliseconds to several seconds behind the primary, so reads that require the absolute latest data must still go to the primary.

Materialized views represent a middle path. You write to normalized tables, and a materialized view or indexed view pre-computes the denormalized form for reads. This keeps the read and write paths in the same database instance while giving each path an optimized physical structure.

Separate OLAP stores complete the spectrum. The write store is a normalized OLTP database. An ETL or CDC pipeline continuously feeds a separate analytical store — often a columnar data warehouse — that's optimized for aggregation-heavy queries. The OLAP store might be minutes or hours behind, which is acceptable for reporting but not for operational lookups.

Deciding where on this spectrum to operate requires understanding your read/write ratio and your read latency tolerance. A system with 95% reads, a few seconds of acceptable lag, and complex aggregate queries is a strong candidate for a replica plus materialized views. A system where every read must reflect the latest write must stay on the primary.

PostgreSQL:

-- Create a materialized view for dashboard reads
-- The write path stays on the normalized orders and order_items tables
CREATE MATERIALIZED VIEW mv_daily_sales_summary AS
SELECT
    DATE_TRUNC('day', o.created_at)::DATE  AS sale_date,
    p.category,
    COUNT(DISTINCT o.order_id)              AS order_count,
    SUM(oi.quantity)                        AS units_sold,
    SUM(oi.quantity * oi.unit_price)        AS gross_revenue,
    SUM(oi.quantity * oi.unit_price * 0.85) AS estimated_margin
FROM orders o
JOIN order_items oi ON oi.order_id = o.order_id
JOIN products p ON p.product_id = oi.product_id
WHERE o.status = 'completed'
GROUP BY 1, 2
WITH DATA;

-- Index the materialized view for common query patterns
CREATE INDEX idx_mv_daily_sales_date
    ON mv_daily_sales_summary (sale_date DESC);

CREATE INDEX idx_mv_daily_sales_category
    ON mv_daily_sales_summary (category, sale_date DESC);

-- Refresh strategy: use CONCURRENTLY during business hours
-- so reads aren't blocked
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_daily_sales_summary;

SQL Server:

-- SQL Server uses indexed views (schema-bound) for similar functionality
-- Note: SQL Server indexed views have stricter requirements than pg materialized views
CREATE VIEW vw_daily_sales_summary
WITH SCHEMABINDING
AS
SELECT
    CAST(o.created_at AS DATE)    AS sale_date,
    p.category,
    COUNT_BIG(DISTINCT o.order_id) AS order_count,
    SUM(oi.quantity)               AS units_sold,
    SUM(oi.quantity * oi.unit_price) AS gross_revenue
FROM dbo.orders o
JOIN dbo.order_items oi ON oi.order_id = o.order_id
JOIN dbo.products p ON p.product_id = oi.product_id
WHERE o.status = 'completed'
GROUP BY CAST(o.created_at AS DATE), p.category;
GO

-- Create clustered index to materialize the view on disk
CREATE UNIQUE CLUSTERED INDEX cidx_daily_sales_summary
    ON vw_daily_sales_summary (sale_date, category);

-- Create additional non-clustered index
CREATE NONCLUSTERED INDEX idx_daily_sales_category
    ON vw_daily_sales_summary (category, sale_date DESC)
    INCLUDE (order_count, units_sold, gross_revenue);

One operational pitfall worth addressing explicitly: when you introduce read replicas, your application developers need to think carefully about read-after-write consistency. A user creates an order and immediately navigates to their order history. If that read goes to a replica that hasn't caught up yet, the order they just placed won't appear — which looks like a bug to the user even though the data is technically correct. The solution is to route reads to the primary for a brief window after any write, or to accept that certain reads are always primary-bound.

Event Sourcing, CDC, and Keeping Systems in Sync

Traditional systems store current state — the users table holds the current email address, the accounts table holds the current balance. Event sourcing stores the sequence of events that led to that state instead. The balance isn't stored; it's computed by replaying deposits and withdrawals. This pattern has profound database architecture implications.

In an event-sourced system, the database becomes an append-only event log. Updates and deletes, in the traditional sense, largely disappear. Instead, you append new events. The "current state" is a projection — a view computed from the event stream and often cached in a separate read model for query performance. The database DBA role shifts from managing mutable state to managing a growing immutable log and the projections built on top of it.

Even if your application doesn't fully adopt event sourcing, Change Data Capture (CDC) brings many of the same benefits at the infrastructure level. CDC reads the database transaction log and publishes a stream of change events — inserts, updates, deletes — to downstream consumers. Those consumers might update a search index, populate a data warehouse, invalidate a cache, or trigger business workflows.

PostgreSQL implements CDC through its logical replication system. Logical replication slots expose the write-ahead log (WAL) as a decoded stream of row-level changes. Tools like Debezium consume this stream and publish to Kafka, from which any number of consumers can read independently at their own pace.

SQL Server's equivalent is SQL Server Change Data Capture, a built-in feature that captures changes into shadow tables within the same database. Downstream systems query these capture tables to read changes since a given LSN (Log Sequence Number).

PostgreSQL:

-- Check current replication slots (used by CDC tools like Debezium)
SELECT
    slot_name,
    plugin,
    slot_type,
    active,
    restart_lsn,
    confirmed_flush_lsn,
    pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) * 8192 / 8192
    ) AS approximate_lag
FROM pg_replication_slots;

-- Create a logical replication slot for a CDC consumer
-- 'pgoutput' is the built-in logical decoder in PostgreSQL 10+
SELECT pg_create_logical_replication_slot(
    'debezium_orders_slot',
    'pgoutput'
);

-- Monitor WAL accumulation caused by unconsumed slots
-- (this is critical: inactive slots cause WAL to accumulate indefinitely)
SELECT
    slot_name,
    active,
    pg_size_pretty(
        pg_wal_lsn_diff(
            pg_current_wal_lsn(),
            restart_lsn
        ) * current_setting('block_size')::BIGINT / 8192
    ) AS wal_retained
FROM pg_replication_slots
ORDER BY restart_lsn;

SQL Server:

-- Enable CDC on the database
EXEC sys.sp_cdc_enable_db;

-- Enable CDC on a specific table
EXEC sys.sp_cdc_enable_table
    @source_schema = N'dbo',
    @source_name   = N'orders',
    @role_name     = NULL,          -- NULL means no role restriction
    @supports_net_changes = 1;      -- Enable net changes queries

-- Query changes in a time range
DECLARE @from_lsn BINARY(10) = sys.fn_cdc_map_time_to_lsn(
    'smallest greater than or equal',
    DATEADD(HOUR, -1, GETUTCDATE())
);
DECLARE @to_lsn BINARY(10) = sys.fn_cdc_get_max_lsn();

SELECT
    __$operation,       -- 1=delete, 2=insert, 3=before-update, 4=after-update
    __$commit_time,
    order_id,
    customer_id,
    status,
    total_amount
FROM cdc.fn_cdc_get_all_changes_dbo_orders(
    @from_lsn,
    @to_lsn,
    N'all with merge'
)
ORDER BY __$seqval;

The operational concern with CDC that catches teams off guard is backpressure management. A CDC consumer that falls behind creates a situation where the source database must retain ever-growing amounts of transaction log or CDC data. In PostgreSQL, an inactive logical replication slot will cause WAL to accumulate until the disk fills up — a silent, creeping disaster. Monitoring slot lag is non-negotiable in any CDC deployment. In SQL Server, the CDC cleanup jobs handle retention automatically, but you need to verify those jobs are running and that their retention window matches your consumers' recovery time expectations.

Multi-Region and High Availability Architecture

Running a single-datacenter database is operationally simple compared to running across regions, but single-datacenter deployments carry existential risk: a datacenter failure takes down your entire system. Multi-region architectures trade operational complexity for resilience and sometimes latency improvements for globally distributed users.

The foundational challenge in multi-region database architecture is the CAP theorem's practical implications. During a network partition between regions — which is the failure mode you're designing for — you cannot simultaneously guarantee consistency (every read sees the most recent write) and availability (every request receives a response). You must choose which to sacrifice.

Active-passive setups have one primary region accepting writes and one or more secondary regions running replicas. In a failure, you promote a secondary to primary, accepting some data loss (the replication lag at the time of failure) and some downtime (the time to detect the failure and complete promotion). This is the most common pattern for OLTP databases because it preserves strong consistency during normal operations. PostgreSQL's streaming replication and SQL Server's Always On Availability Groups both implement this pattern well.

Active-active setups allow writes to multiple regions simultaneously. This eliminates the promotion step during failover but introduces the hardest problem in distributed systems: conflict resolution. If a user updates their email in region A and region B simultaneously, which write wins? Multi-master replication systems handle this through last-write-wins policies, vector clocks, or application-defined merge logic — none of which is simple. PostgreSQL doesn't natively support multi-master replication; tools like BDR (Bi-Directional Replication) from EDB provide it as an extension. SQL Server has no native multi-master either, though some architectures use distributed availability groups with conflict-avoidance through shard-per-region.

The practical middle ground most production systems land on is active-passive with fast failover and read-local routing. Writes always go to the primary region. Reads are served from the nearest replica. Replication is synchronous (or semi-synchronous) to at least one replica to bound data loss in a failover. This balances consistency, availability, and operational complexity reasonably well.

A topic that becomes urgent at multi-region scale is connection management. Connections to a database carry state — open transactions, session settings, prepared statements. When a failover occurs, those connections become invalid. Applications must handle connection pool invalidation and reconnection gracefully. PgBouncer in front of PostgreSQL and SQL Server's transparent network name (via Windows Server Failover Clustering or the AG listener) both abstract the connection endpoint so applications don't need to know which server is currently primary.

Monitoring in multi-region setups requires watching replication lag from multiple vantage points. A replica that's 5 seconds behind the primary is fine under normal load. A replica that's 5 minutes behind because network conditions degraded undetected is a problem you want to know about before a failover forces you to discover it.

PostgreSQL:

-- On the primary: view current replication state for all standbys
SELECT
    application_name,
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    EXTRACT(EPOCH FROM write_lag)  AS write_lag_seconds,
    EXTRACT(EPOCH FROM flush_lag)  AS flush_lag_seconds,
    EXTRACT(EPOCH FROM replay_lag) AS replay_lag_seconds,
    sync_state
FROM pg_stat_replication
ORDER BY application_name;

-- On a standby: check how far behind the primary it is
SELECT
    CASE
        WHEN pg_is_in_recovery() THEN
            EXTRACT(EPOCH FROM (NOW() - pg_last_xact_replay_timestamp()))
        ELSE 0
    END AS replication_lag_seconds,
    pg_last_wal_receive_lsn()  AS received_lsn,
    pg_last_wal_replay_lsn()   AS replayed_lsn,
    pg_is_in_recovery()        AS is_standby;

SQL Server:

-- Monitor Always On AG replica synchronization health
SELECT
    ag.name                             AS availability_group,
    ar.replica_server_name,
    ar.availability_mode_desc,
    ar.failover_mode_desc,
    ars.role_desc,
    ars.synchronization_health_desc,
    ars.connected_state_desc,
    ars.last_connect_error_description,
    adbrs.synchronization_state_desc,
    adbrs.log_send_queue_size          AS log_send_queue_kb,
    adbrs.redo_queue_size              AS redo_queue_kb,
    adbrs.log_send_rate                AS log_send_rate_kb_sec,
    adbrs.redo_rate                    AS redo_rate_kb_sec,
    adbrs.estimated_data_loss_seconds,
    adbrs.estimated_recovery_time_seconds
FROM sys.availability_groups ag
JOIN sys.availability_replicas ar
    ON ar.group_id = ag.group_id
JOIN sys.dm_hadr_availability_replica_states ars
    ON ars.replica_id = ar.replica_id
JOIN sys.dm_hadr_database_replica_states adbrs
    ON adbrs.replica_id = ar.replica_id
ORDER BY ag.name, ar.replica_server_name;

Recovery time objective (RTO) and recovery point objective (RPO) are the two numbers that should drive every high availability decision. RTO is how long the system can be down. RPO is how much data loss is acceptable. If RPO is zero, synchronous replication to at least one replica is mandatory — you confirm a commit only after at least one replica has acknowledged it. If RTO is under 30 seconds, automated failover must be configured and tested. Every architectural decision flows from these two numbers. If you don't have explicit SLAs for RTO and RPO from your business stakeholders, getting them is the most valuable thing you can do before designing your HA topology.

Connection Pooling Architecture and the Problem It Solves

Every database connection consumes real resources. PostgreSQL forks a backend process per connection; a connection that's idle still holds memory, file descriptors, and a slot in the shared memory structures. SQL Server's thread model is more efficient, but connections still consume memory and worker threads. At scale, the number of application instances multiplied by the number of threads per instance quickly exceeds what a database can handle directly.

A connection pool sits between the application and the database. The application thinks it has a dedicated connection; the pool maintains a smaller set of actual database connections and multiplexes application requests across them. This decouples the number of application threads (which can be in the thousands across a fleet of servers) from the number of database connections (which ideally stays in the dozens to low hundreds).

PgBouncer is the standard external connection pooler for PostgreSQL. It operates in three modes:

  • → **Session mode**: One server connection per client session, held until the client disconnects. This barely improves anything and is rarely the right choice.
  • → **Transaction mode**: A server connection is assigned to a client only for the duration of a transaction. Between transactions, the server connection returns to the pool and can be reused by other clients. This is the mode that provides real multiplexing benefits.
  • → **Statement mode**: A server connection is held only for the duration of a single statement. This maximizes reuse but breaks any application that uses multi-statement transactions.

    • Transaction mode is almost always the right choice. The catch is that it's incompatible with certain PostgreSQL features: SET commands, advisory locks, LISTEN/NOTIFY, and named prepared statements don't behave correctly because the server connection changes between transactions. Applications must be designed with this in mind.

    SQL Server doesn't need an external pooler the same way PostgreSQL does because SQL Server's connection model is more efficient and its client libraries (ADO.NET, JDBC, ODBC) implement connection pooling in-process. However, at very high scale or in environments with hundreds of microservices, external poolers or PgBouncer-style middleware become relevant for SQL Server too.

    The architectural question that matters most is where to deploy the connection pooler. Deploying PgBouncer on the same host as the database eliminates network overhead between pooler and database but creates a single point of failure. Deploying it on a separate set of hosts (typically 2–3 pooler nodes behind a load balancer) is more resilient but adds a network hop. In Kubernetes environments, a PgBouncer sidecar per application pod is increasingly common, but this defeats the pooling purpose for the database — you want the pooler close to the database, not distributed across application pods.

    Sizing the pool correctly requires understanding your workload. The rule of thumb for PostgreSQL — rooted in research and widely tested in production — is that the optimal number of active database connections is approximately 2 × num_cores + num_disks. Beyond that number, connections queue behind each other and throughput actually decreases due to lock contention and context switching. A 32-core server with fast NVMe storage might perform optimally at 70 active connections, even if it could technically accept 500. PgBouncer's max_client_conn handles the overflow by queuing client connections, which is far cheaper than having 500 active backend processes.
