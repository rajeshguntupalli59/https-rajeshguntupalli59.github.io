Chapter 16: Active Geo-Replication and Failover Groups

Geographic replication in cloud databases is the production answer to a question on-prem DBAs have wrestled with for decades: how do you keep a copy of your data synchronized across data centers without building your own log shipping infrastructure, managing your own replication slots, or manually promoting standby servers at 2 AM? This chapter covers how AWS Aurora Global Database, Azure SQL Active Geo-Replication and Failover Groups, and Google Cloud Spanner multi-region configurations solve this problem — and where each approach has sharp edges you need to understand before you commit to them in production. We cover configuration, latency expectations, failover mechanics, RPO/RTO characteristics, and the automation patterns that separate a well-architected geo-redundant deployment from one that fails its first real disaster recovery test.

---

16.1 The Geography Problem: What You're Actually Solving

Before touching any configuration, get the requirements straight. Geo-replication serves three distinct purposes that are often conflated:

Disaster recovery — a full regional outage requires a replica in another region to take over. RPO of seconds to low minutes is achievable. RTO depends heavily on whether failover is automatic or manual.

Read scaling — routing read traffic to replicas closer to users in other regions reduces latency without touching the primary. A user in Tokyo reading from a US-East primary adds 150–200ms of round-trip latency that disappears when queries route to an Asia-Pacific replica.

Compliance and data residency — some regulated industries require data to exist within specific geographic boundaries, but still need disaster recovery within that boundary. These are not the same as global replication and sometimes conflict with it.

On-prem, you would have built this with SQL Server's Always On availability groups across stretched clusters, or PostgreSQL streaming replication with custom failover scripts on top of Pacemaker/Corosync. The managed cloud equivalent is less flexible in some ways — you cannot, for instance, tune the replication protocol itself — but it removes an enormous amount of operational surface area.

Replication Lag Is Inherent and Physical

The speed of light is not a cloud vendor problem. At the physical limit, New York to London is roughly 70ms round-trip. Synchronous replication across that distance would add 70ms minimum to every committed write. Every cloud vendor therefore makes the same decision: cross-region replication is asynchronous. This means RPO is not zero. Measure it. Know your actual lag before a disaster occurs.

Typical lag numbers in production:

| Scenario | Expected Replication Lag |

|---|---|

| Aurora Global, same continent | 1–3 seconds typical, <5s P99 |

| Aurora Global, US to EU | 3–8 seconds typical, <10s P99 |

| Azure SQL Active Geo-Replication, paired regions | 2–5 seconds typical |

| Azure SQL Active Geo-Replication, unpaired regions | 5–15 seconds depending on write volume |

| Cloud Spanner multi-region | Sub-second within a continent configuration |

| Spanner global multi-region | 5–10 seconds external consistency latency |

These numbers change dramatically under write load. A database doing 50,000 writes per second with 8KB average row size is generating roughly 400MB/s of WAL. Moving that across a WAN link takes real bandwidth, and replication lag expands if the link cannot sustain throughput. Size your workload against these numbers, not just the idle case.

16.2 Aurora Global Database: Architecture and Configuration

Aurora Global Database uses a storage-level replication mechanism rather than logical replication. The primary region writes to Aurora's distributed storage, and those writes are replicated asynchronously at the storage layer to secondary regions. This is architecturally different from PostgreSQL streaming replication, where the primary WAL sender streams WAL records to a standby's WAL receiver. Aurora Global bypasses the Aurora writer instance in secondary regions entirely — secondary regions have their own Aurora cluster, but that cluster's storage receives writes directly from the primary's storage nodes.

The practical implications:

  • → Secondary clusters are read-only at the SQL layer
  • → You cannot use logical replication slots on a secondary cluster
  • → Secondary clusters can have their own read replicas within the secondary region
  • → Each secondary region introduces one hop of latency: primary writes to storage, storage replicates to secondary region storage, secondary cluster reads from its regional storage

    • Setting Up Aurora Global Database with AWS CLI

    First, create the primary Aurora cluster if you do not already have one:

    # Create the primary Aurora cluster in us-east-1
aws rds create-db-cluster \
  --db-cluster-identifier prod-global-primary \
  --engine aurora-postgresql \
  --engine-version 15.4 \
  --master-username pgadmin \
  --master-user-password "${DB_PASSWORD}" \
  --db-subnet-group-name prod-subnet-group \
  --vpc-security-group-ids sg-0abc123def456789 \
  --storage-encrypted \
  --kms-key-id arn:aws:kms:us-east-1:123456789012:key/mrk-abc123 \
  --region us-east-1

# Create the writer instance
aws rds create-db-instance \
  --db-instance-identifier prod-global-primary-writer \
  --db-cluster-identifier prod-global-primary \
  --db-instance-class db.r7g.4xlarge \
  --engine aurora-postgresql \
  --region us-east-1

# Create the Global Database wrapper (wraps the existing cluster)
aws rds create-global-cluster \
  --global-cluster-identifier prod-aurora-global \
  --source-db-cluster-identifier arn:aws:rds:us-east-1:123456789012:cluster:prod-global-primary \
  --region us-east-1

    Adding a secondary region requires the secondary region to have its own VPC, subnet group, and KMS key. If you are using a customer-managed KMS key (which you should be in production), you need a multi-region key or separate regional keys. A common mistake is using a single-region KMS key and then being surprised when the secondary cluster creation fails.

    # Add a secondary cluster in eu-west-1
aws rds create-db-cluster \
  --db-cluster-identifier prod-global-secondary-eu \
  --engine aurora-postgresql \
  --engine-version 15.4 \
  --global-cluster-identifier prod-aurora-global \
  --db-subnet-group-name prod-subnet-group-eu \
  --vpc-security-group-ids sg-0def456abc123789 \
  --storage-encrypted \
  --kms-key-id arn:aws:kms:eu-west-1:123456789012:key/mrk-def456 \
  --region eu-west-1

# Add a reader instance in the secondary cluster
aws rds create-db-instance \
  --db-instance-identifier prod-global-secondary-eu-reader \
  --db-cluster-identifier prod-global-secondary-eu \
  --db-instance-class db.r7g.2xlarge \
  --engine aurora-postgresql \
  --region eu-west-1

    Monitoring Replication Lag with CloudWatch

    Aurora Global exposes the AuroraGlobalDBReplicationLag metric in CloudWatch. Set an alarm on this. A reasonable alert threshold for most OLTP workloads is 5 seconds; a page-worthy threshold is 30 seconds.

    # Create a CloudWatch alarm for Global DB replication lag
aws cloudwatch put-metric-alarm \
  --alarm-name aurora-global-replication-lag-warning \
  --alarm-description "Aurora Global replication lag exceeds 5 seconds" \
  --metric-name AuroraGlobalDBReplicationLag \
  --namespace AWS/RDS \
  --statistic Maximum \
  --period 60 \
  --evaluation-periods 3 \
  --threshold 5000 \
  --comparison-operator GreaterThanThreshold \
  --dimensions Name=DBClusterIdentifier,Value=prod-global-secondary-eu \
  --alarm-actions arn:aws:sns:eu-west-1:123456789012:dba-alerts \
  --treat-missing-data notBreaching \
  --region eu-west-1

    Failover Behavior and What to Expect

    Aurora Global failover is not automatic by default. There is no built-in equivalent to an Always On availability group's automatic failover with a failure detection timeout. You must either trigger it manually or build automation around it.

    When you initiate a managed failover (the preferred path that minimizes data loss), Aurora promotes the secondary cluster in the target region and demotes the primary. The process takes 1–2 minutes and involves:

    1. Stopping writes on the primary

    2. Waiting for the secondary to apply all pending replication entries

    3. Promoting the secondary to read-write

    4. Updating the Global Database configuration to reflect the new primary

    If you use an unplanned failover (the primary region is completely unavailable), you detach the secondary cluster from the Global Database and promote it independently. In this scenario, the data in-flight that had not yet replicated is lost — that is your actual RPO.

    Your application connection string must change after failover. Aurora does not automatically redirect the cluster endpoint. Applications pointing at the old primary region's cluster endpoint will fail until you update DNS or connection strings. This is a common operational gap. Solve it with Route53 health checks and a CNAME that points to whichever region is primary, or use a global accelerator pattern.

    16.3 Azure SQL Active Geo-Replication and Failover Groups

    Azure SQL offers two related but distinct features. Active Geo-Replication is the lower-level primitive: it creates readable secondary databases in up to four other regions, with asynchronous replication at the database level. Failover Groups are the higher-level abstraction: they manage one or more databases together, provide a stable listener endpoint that does not change after failover, and support automatic failover through a health-check based policy.

    If you are building a production application, use Failover Groups. Active Geo-Replication alone requires you to update connection strings after failover. Failover Groups give you a fixed endpoint — .database.windows.net for the primary and .secondary.database.windows.net for the secondary — that routes to whichever server is currently primary.

    Creating a Failover Group with Azure CLI

    # Variables
PRIMARY_SERVER="prod-sql-eastus"
SECONDARY_SERVER="prod-sql-westeu"
RESOURCE_GROUP="prod-rg"
FAILOVER_GROUP="prod-fog-01"
DATABASE_NAME="applicationdb"

# Create the secondary server in West Europe
az sql server create \
  --name $SECONDARY_SERVER \
  --resource-group $RESOURCE_GROUP \
  --location "westeurope" \
  --admin-user sqladmin \
  --admin-password "${SQL_ADMIN_PASSWORD}"

# Configure firewall rules and private endpoints on secondary
# (match whatever you have on primary)
az sql server firewall-rule create \
  --resource-group $RESOURCE_GROUP \
  --server $SECONDARY_SERVER \
  --name AllowAzureServices \
  --start-ip-address 0.0.0.0 \
  --end-ip-address 0.0.0.0

# Create the failover group
# This initiates geo-replication of all specified databases
az sql failover-group create \
  --name $FAILOVER_GROUP \
  --partner-server $SECONDARY_SERVER \
  --resource-group $RESOURCE_GROUP \
  --server $PRIMARY_SERVER \
  --add-db $DATABASE_NAME \
  --failover-policy Automatic \
  --grace-period 1

# The grace-period is in hours — 1 hour before automatic failover triggers
# Setting to 0 means immediate failover, which risks unnecessary failovers
# during transient connectivity issues between regions

    The --grace-period parameter deserves attention. Azure uses a health-check mechanism that monitors connectivity between the primary and secondary. If the primary is unreachable for longer than the grace period, automatic failover triggers. Setting this to 0 — immediate failover — risks data loss and unnecessary failovers if there are transient network issues between Azure regions that do not represent actual primary failure. For most production workloads, 1 hour is the recommended minimum, and many organizations use 2–4 hours depending on their RPO tolerance.

    Monitoring Geo-Replication Health

    Azure SQL exposes geo-replication state through sys.dm_geo_replication_link_states in the primary database:

    -- Run on the PRIMARY database
-- Check replication health for all geo-replicas
SELECT
    link_guid,
    partner_server,
    partner_database,
    last_replication,
    replication_lag_sec,
    replication_state_desc,
    role_desc,
    secondary_allow_connections_desc
FROM sys.dm_geo_replication_link_states
ORDER BY replication_lag_sec DESC;

-- Practical alert query: lag over 30 seconds is worth investigating
-- lag over 300 seconds (5 minutes) warrants a page
SELECT
    partner_server,
    partner_database,
    replication_lag_sec,
    last_replication,
    CASE
        WHEN replication_lag_sec > 300 THEN 'CRITICAL'
        WHEN replication_lag_sec > 30  THEN 'WARNING'
        ELSE 'OK'
    END AS lag_status
FROM sys.dm_geo_replication_link_states
WHERE role_desc = 'PRIMARY';

    On the secondary, you can also query sys.dm_database_replica_states to see how far behind the secondary is in terms of log send queue and redo queue — the same DMVs you know from Always On, repurposed for geo-replication context.

    Failover Group Listener and Application Connection Patterns

    The Failover Group listener endpoint is where your application connection string points. After any failover — planned or unplanned — your application continues to connect to prod-fog-01.database.windows.net without any changes. Azure's DNS resolves this to whichever server is currently primary.

    # Initiate a planned failover (controlled, no data loss)
# Run this for maintenance or DR drills
az sql failover-group set-primary \
  --name $FAILOVER_GROUP \
  --resource-group $RESOURCE_GROUP \
  --server $SECONDARY_SERVER

# Check failover group status after failover
az sql failover-group show \
  --name $FAILOVER_GROUP \
  --resource-group $RESOURCE_GROUP \
  --server $SECONDARY_SERVER \
  --query "{primaryServer:replicationRole, location:location, databases:databases}"

    One constraint that catches DBAs migrating from Always On: Failover Groups in Azure SQL Database (the fully managed PaaS tier) are limited to two servers — one primary and one secondary. You cannot have two readable secondaries in two different regions plus your primary in a single Failover Group. If you need multiple secondary regions, you use Active Geo-Replication directly and accept that you manage connection string routing yourself. SQL Managed Instance supports up to four regions in a Failover Group, which gives more flexibility.

    Hyperscale Geo-Replication Differences

    If you are using Azure SQL Hyperscale — the tier that decouples compute from storage and supports up to 100TB databases — geo-replication works differently. Hyperscale uses Named Replicas and Geo-Replicas as distinct concepts. A Geo-Replica in Hyperscale is seeded from the distributed page server layer rather than through log shipping, which means the initial seeding is extremely fast even for multi-terabyte databases — often measured in minutes rather than the hours traditional log shipping takes to initialize a large secondary.

    The tradeoff: Hyperscale Geo-Replicas have higher replication lag than standard geo-replication at high write throughput. If your Hyperscale workload generates more than roughly 100MB/s of log, expect lag to grow during peak write periods.

    16.4 Google Cloud Spanner and Cloud SQL: Multi-Region Strategies

    Google Cloud offers two fundamentally different architectures for geo-redundancy, and choosing between them depends on whether you need strong consistency or are willing to tolerate eventual consistency for the benefit of scale.

    Cloud Spanner Multi-Region Configurations

    Spanner is the only globally distributed, strongly consistent, relational database available as a managed service. This is not marketing — it is an architectural statement with real implications. Spanner uses the TrueTime API, which relies on atomic clocks and GPS receivers in Google's data centers to provide bounded clock uncertainty and enable external consistency across regions.

    When you configure a Spanner instance with a multi-region configuration, writes are synchronously committed to a quorum of regions before returning success to the application. This means you get zero RPO under region failure — but you pay for it with higher write latency.

    Spanner's regional configurations use naming conventions that encode the topology:

  • → `nam4` — dual-region, US East and US Central, optimized for low-latency writes in the Eastern US
  • → `eur3` — dual-region, Netherlands and Belgium
  • → `nam-eur-asia1` — multi-region spanning North America, Europe, and Asia (five regions total)
  • → `nam6` — six North American regions
    • # Create a Spanner instance with multi-region configuration
gcloud spanner instances create prod-spanner-global \
  --config=nam-eur-asia1 \
  --description="Production global instance" \
  --nodes=3 \
  --project=prod-project-id

# Multi-region configs require at least 3 nodes per "region group"
# nam-eur-asia1 has 5 read-write regions + 2 witness regions
# Minimum nodes for production: 3 per region group = expensive
# Cost: roughly $0.90/node/hour * 3 nodes * 7 region groups in nam-eur-asia1

# Check instance replication details
gcloud spanner instances describe prod-spanner-global \
  --project=prod-project-id \
  --format="yaml(config,displayName,nodeCount,state)"

# Create a database in the multi-region instance
gcloud spanner databases create applicationdb \
  --instance=prod-spanner-global \
  --project=prod-project-id

    The write latency penalty for global multi-region Spanner is real. For nam-eur-asia1, a write commit from a US East application needs to reach quorum across regions. Expect 50–100ms commit latency under normal operation versus 5–15ms for a single-region Spanner instance. Applications built for sub-10ms write latency will need to reconsider whether global Spanner is the right tool.

    Cloud SQL for PostgreSQL: Multi-Region Read Replicas

    Cloud SQL does not offer Spanner's synchronous multi-region consistency model. Instead, it offers cross-region read replicas backed by asynchronous replication, similar to Aurora Global. The replication mechanism is standard PostgreSQL streaming replication — the same process you know from on-prem setups, managed by Google's control plane.

    # Create a Cloud SQL primary instance
gcloud sql instances create prod-postgres-primary \
  --database-version=POSTGRES_15 \
  --tier=db-custom-16-65536 \
  --region=us-central1 \
  --storage-type=SSD \
  --storage-size=500GB \
  --storage-auto-increase \
  --backup-start-time=04:00 \
  --enable-bin-log \
  --availability-type=REGIONAL \
  --project=prod-project-id

# Create a cross-region read replica in europe-west1
gcloud sql instances create prod-postgres-replica-eu \
  --master-instance-name=prod-postgres-primary \
  --region=europe-west1 \
  --tier=db-custom-8-32768 \
  --database-version=POSTGRES_15 \
  --project=prod-project-id

# Check replication lag using Cloud SQL's built-in monitoring
gcloud sql instances describe prod-postgres-replica-eu \
  --project=prod-project-id \
  --format="yaml(replicaConfiguration,replicationLagSeconds)"

    Cloud SQL cross-region replicas cannot be automatically promoted on regional failure. Promotion is a manual or scripted action. Unlike Aurora Global, there is no managed failover path that handles orchestration for you. You must:

    1. Promote the replica to a standalone instance (which breaks replication)

    2. Update your application's connection string to point to the promoted instance

    3. Once the primary region recovers, recreate the replica relationship from the newly promoted instance back toward the old primary, or rebuild from scratch

    # Promote a cross-region replica after a regional outage
# This is a one-way operation — there is no "unpromote"
gcloud sql instances promote-replica prod-postgres-replica-eu \
  --project=prod-project-id

# After promotion, the instance becomes read-write standalone
# Verify it is now accepting writes
gcloud sql instances describe prod-postgres-replica-eu \
  --project=prod-project-id \
  --format="yaml(instanceType,region,state)"
# instanceType should now be "CLOUD_SQL_INSTANCE" not "READ_REPLICA_INSTANCE"

    AlloyDB Cross-Region Considerations

    AlloyDB, Google's PostgreSQL-compatible high-performance managed database, as of 2024 supports cross-region replication through AlloyDB Omni (the self-managed distribution) and is rolling out managed cross-region secondary clusters. For production deployments requiring geo-redundancy on AlloyDB, verify the current regional availability of cross-region secondary clusters against the GCP documentation, as this feature has been in preview with limited region pairs. The architecture follows a similar storage-layer replication model to Aurora Global rather than WAL-stream replication.
