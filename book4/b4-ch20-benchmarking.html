Chapter 20: Benchmarking Vector Databases: Throughput, Latency, Recall

Choosing a vector database without benchmarking it is the equivalent of selecting a relational engine based on marketing collateral alone — you will eventually regret it. The vector database ecosystem is young, vendor claims are aggressive, and the gap between synthetic benchmark numbers and production behavior is wide enough to cause real incidents at scale. This chapter builds a rigorous framework for measuring what actually matters in vector search systems: query throughput, tail latency, and recall accuracy. You will learn how these three metrics interact (improving one often degrades another), how to construct benchmarks that reflect your actual workload rather than a vendor's best-case scenario, and how to interpret results from the most common tools in the ecosystem — pgvector, Pinecone, Weaviate, Qdrant, and Milvus. By the end, you will have runnable harnesses, concrete thresholds to target, and a vocabulary for communicating vector database performance the same way you already communicate IOPS, latency percentiles, and cache hit ratios.

---

The Three Metrics That Define Vector Search Performance

Every vector database benchmark ultimately reduces to three numbers: throughput, latency, and recall. Understanding what each measures — and where the bodies are buried — is the prerequisite to designing any useful test.

Throughput is queries per second (QPS). It measures how many similarity searches the system can complete in a unit of time under sustained load. Like any throughput metric, it is meaningless without specifying concurrency. A system that delivers 500 QPS at 1 concurrent client and 480 QPS at 50 concurrent clients is a fundamentally different beast from one that delivers 500 QPS at 1 client and 12,000 QPS at 50 clients. Always measure throughput under the concurrency your application actually generates.

Latency is the time to complete a single query. In production systems, what matters is not mean latency but tail latency — specifically p95 and p99. A vector database with a 5ms mean and a 2,000ms p99 will make your application feel broken during load spikes. Most vector workloads are in the critical path of user-facing requests, which makes p99 the governing metric.

Recall is the fraction of true nearest neighbors that a query actually returns. It is the metric unique to approximate nearest neighbor (ANN) systems, and it has no direct analogue in traditional databases. If you ask for the 10 most similar vectors to a query, and the system returns 10 results but only 7 of them are genuinely in the true top-10, your recall@10 is 0.7. Recall is what separates ANN search from exact k-NN search — and it is what makes benchmarking vector databases harder than benchmarking relational systems.

The fundamental tension is this: recall, throughput, and latency form a triangle where you can optimize any two by sacrificing the third. Tightening recall requires searching more of the index graph, which increases latency and reduces throughput. Reducing latency requires searching less, which damages recall. This is not an implementation deficiency — it is the mathematical reality of ANN algorithms, and any honest benchmark must measure all three simultaneously.

The recall-latency curve is the most important diagnostic tool in vector database performance analysis. It plots recall (x-axis) against query latency (y-axis) at fixed concurrency, with each point representing a different index parameter setting. The database with the curve closest to the lower-right corner of the plot — high recall at low latency — wins that workload.

Practical thresholds for production systems:

| Metric | Acceptable | Good | Excellent |

|---|---|---|---|

| Recall@10 | > 0.90 | > 0.95 | > 0.99 |

| p99 latency (single node) | < 100ms | < 20ms | < 5ms |

| QPS (per node, 768-dim) | > 500 | > 2,000 | > 10,000 |

These numbers are for 768-dimensional embeddings (a common size from models like text-embedding-3-small and all-MiniLM-L6-v2) on a dataset of 1–10 million vectors. Dimensionality, dataset size, and filter complexity all shift these thresholds dramatically.

Constructing a Realistic Benchmark Dataset

Vendor benchmarks almost always use ANN-benchmarks datasets — SIFT1M, GIST1M, or GloVe — because they are public, reproducible, and make their systems look good. Your workload is probably none of those. Building a benchmark that predicts production behavior requires matching four properties of your real dataset: dimensionality, distribution, dataset size, and filter selectivity.

Dimensionality directly determines index build time, memory consumption, and query latency. OpenAI's text-embedding-3-large produces 3072-dimensional vectors. Cohere's embed-english-v3.0 produces 1024-dimensional vectors. Most open-source models produce 384–768 dimensions. Do not benchmark at 128 dimensions if you are going to deploy at 1536.

Distribution matters more than most engineers expect. ANN index structures (HNSW, IVF, ScaNN) exploit clustering in the vector space to skip regions during search. Randomly generated vectors are uniformly distributed in high-dimensional space and do not cluster — they are the worst case for ANN indices and the best case for demonstrating that an index provides no benefit. Real embedding vectors cluster by semantic topic. If your documents are all from a single domain (medical literature, legal contracts, software documentation), your vectors will cluster differently than a general-purpose corpus, and your recall numbers will differ from published benchmarks.

Filtered search is the single biggest gap between published benchmarks and production behavior. Most real applications combine vector similarity with metadata filters: "find the 10 most similar documents to this query, but only from this tenant's data, written after 2023-01-01, tagged as 'approved'." Pre-filtering (apply metadata filter first, then search remaining vectors) destroys recall when the filter is selective. Post-filtering (search vectors first, then apply filter) destroys recall when the filter eliminates most results. How a database handles this tradeoff — Qdrant's filterable HNSW, Weaviate's inverted-index integration, Pinecone's metadata filtering — is one of its most important production characteristics.

The following script generates a synthetic benchmark dataset that mirrors real embedding distributions by using actual sentence-transformers embeddings on sampled text:

# Python
# benchmark_dataset_generator.py
# Generates a realistic benchmark dataset using real embedding models
# Requires: sentence-transformers, numpy, h5py, datasets

import numpy as np
import h5py
import time
from sentence_transformers import SentenceTransformer
from datasets import load_dataset

def generate_benchmark_dataset(
    n_corpus: int = 100_000,
    n_queries: int = 1_000,
    model_name: str = "sentence-transformers/all-MiniLM-L6-v2",
    output_path: str = "benchmark_dataset.h5",
    batch_size: int = 512
):
    """
    Generates an HDF5 benchmark dataset in the ANN-benchmarks format.
    Uses real text embeddings for realistic vector distributions.
    
    HDF5 schema:
      /train   (n_corpus, dim)   float32  - corpus vectors
      /test    (n_queries, dim)  float32  - query vectors
      /neighbors (n_queries, 100) int32   - ground truth (computed exactly)
      /distances (n_queries, 100) float32 - ground truth distances
    """
    print(f"Loading model: {model_name}")
    model = SentenceTransformer(model_name)
    dim = model.get_sentence_embedding_dimension()
    print(f"Embedding dimension: {dim}")

    # Load a real text corpus for realistic distributions
    print("Loading corpus text from Wikipedia sample...")
    dataset = load_dataset(
        "wikipedia", "20220301.en",
        split="train",
        streaming=True,
        trust_remote_code=True
    )
    
    texts = []
    for item in dataset:
        # Extract first 256 characters of each article as a passage
        texts.append(item["text"][:256])
        if len(texts) >= n_corpus + n_queries:
            break
    
    corpus_texts = texts[:n_corpus]
    query_texts = texts[n_corpus:n_corpus + n_queries]
    
    print(f"Encoding {n_corpus} corpus documents...")
    t0 = time.time()
    corpus_vectors = model.encode(
        corpus_texts,
        batch_size=batch_size,
        show_progress_bar=True,
        normalize_embeddings=True  # Cosine similarity -> dot product
    ).astype(np.float32)
    encode_time = time.time() - t0
    print(f"Corpus encoding: {encode_time:.1f}s ({n_corpus/encode_time:.0f} docs/sec)")
    
    print(f"Encoding {n_queries} query vectors...")
    query_vectors = model.encode(
        query_texts,
        batch_size=batch_size,
        show_progress_bar=True,
        normalize_embeddings=True
    ).astype(np.float32)
    
    # Compute exact ground truth using brute-force cosine similarity
    # For normalized vectors, cosine similarity = dot product
    print("Computing exact ground truth (brute-force)...")
    k_gt = 100  # Store top-100 ground truth for evaluating recall@1 through @100
    
    neighbors = np.zeros((n_queries, k_gt), dtype=np.int32)
    distances = np.zeros((n_queries, k_gt), dtype=np.float32)
    
    # Process in batches to avoid OOM on large datasets
    query_batch_size = 100
    for i in range(0, n_queries, query_batch_size):
        batch = query_vectors[i:i + query_batch_size]
        # Matrix multiply: (batch_size, dim) x (dim, n_corpus) -> (batch_size, n_corpus)
        sims = np.dot(batch, corpus_vectors.T)
        # Get top-k indices (descending similarity)
        top_k_idx = np.argpartition(sims, -k_gt, axis=1)[:, -k_gt:]
        for j, row_idx in enumerate(range(i, min(i + query_batch_size, n_queries))):
            top_k_sorted = top_k_idx[j][np.argsort(sims[j, top_k_idx[j]])[::-1]]
            neighbors[row_idx] = top_k_sorted
            distances[row_idx] = sims[j, top_k_sorted]
        
        if i % 1000 == 0:
            print(f"  Ground truth: {i}/{n_queries} queries")
    
    # Write HDF5 compatible with ann-benchmarks tooling
    print(f"Writing dataset to {output_path}...")
    with h5py.File(output_path, "w") as f:
        f.create_dataset("train", data=corpus_vectors, compression="gzip")
        f.create_dataset("test", data=query_vectors, compression="gzip")
        f.create_dataset("neighbors", data=neighbors, compression="gzip")
        f.create_dataset("distances", data=distances, compression="gzip")
        f.attrs["dimension"] = dim
        f.attrs["n_corpus"] = n_corpus
        f.attrs["n_queries"] = n_queries
        f.attrs["model"] = model_name
        f.attrs["metric"] = "cosine"
    
    print(f"Dataset written: {corpus_vectors.nbytes / 1e6:.1f} MB corpus, "
          f"{query_vectors.nbytes / 1e6:.1f} MB queries")
    return output_path


if __name__ == "__main__":
    generate_benchmark_dataset(
        n_corpus=100_000,
        n_queries=1_000,
        model_name="sentence-transformers/all-MiniLM-L6-v2",
        output_path="minilm_100k_benchmark.h5"
    )

This produces an HDF5 file in the same format as ANN-benchmarks, so you can drop it into standard tooling or write your own evaluation loop. The critical design choice here is using real embeddings on real text: your benchmark dataset should come from the same distribution as your production data, or your recall numbers will be wrong by 5–15 percentage points.

Building a Multi-Database Benchmark Harness

With a dataset in hand, the next step is a consistent measurement harness that runs the same queries against multiple databases and captures the metrics that matter. The harness must be fair: each database should be given its best-case index configuration, not a default one. Default configurations almost always favor index build time over query performance, which is the opposite of production priorities.
