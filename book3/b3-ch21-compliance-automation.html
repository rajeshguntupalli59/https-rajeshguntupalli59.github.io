Chapter 21: Policy-Driven Compliance Automation

Compliance frameworks — SOC 2, HIPAA, PCI-DSS, GDPR, ISO 27001 — share a common engineering problem: humans cannot reliably enforce policy at the speed and scale that modern database platforms demand. An autonomous compliance system doesn't wait for an audit cycle to discover that a staging database received production PII, that a service account accumulated excessive privileges over six months, or that an unencrypted backup landed in the wrong storage tier. It detects the drift, evaluates it against a machine-readable policy graph, and corrects or escalates the deviation without a ticket being opened. This chapter walks through how to build that system end-to-end — from encoding compliance rules as enforceable policy objects, to building the detection layer, to wiring autonomous remediation actions that leave a complete, auditor-readable record of every decision made.

---

From Compliance Checklists to Machine-Readable Policy Objects

The foundational shift in autonomous compliance is treating regulatory requirements not as PDF documents reviewed quarterly, but as versioned, structured objects that the platform can evaluate programmatically at any time. A policy object answers three questions: what condition must be true, what evidence proves it is true, and what action restores compliance when it is not.

The schema for a policy object should be stable enough to version in Git and expressive enough to generate both detection queries and remediation procedures from the same source.

# Policy object schema (Python dataclass, stored as YAML in Git)
from dataclasses import dataclass, field
from typing import List, Optional
from enum import Enum

class Severity(Enum):
    CRITICAL = "critical"
    HIGH     = "high"
    MEDIUM   = "medium"
    LOW      = "low"

class RemediationMode(Enum):
    AUTO     = "auto"        # system acts immediately
    APPROVE  = "approve"     # system proposes, human approves
    ALERT    = "alert"       # system only notifies

@dataclass
class CompliancePolicy:
    policy_id:          str
    framework:          str          # "PCI-DSS", "HIPAA", "SOC2", "GDPR"
    control_ref:        str          # e.g. "PCI-DSS 8.6.1"
    description:        str
    target_scope:       List[str]    # ["production", "staging-pii"]
    detection_query:    str          # SQL or PromQL
    detection_platform: str          # "postgresql", "sqlserver", "prometheus"
    severity:           Severity
    remediation_mode:   RemediationMode
    remediation_action: str          # function name in remediation registry
    evidence_query:     str          # query that produces audit evidence
    sla_minutes:        int          # time to remediate before escalation
    owner:              str
    tags:               List[str] = field(default_factory=list)

A YAML representation of a real policy under PCI-DSS 8.6.1 (system account password management) might look like this:

policy_id: pci-8-6-1-pg-default-passwords
framework: PCI-DSS
control_ref: "PCI-DSS 8.6.1"
description: "No database role may retain a default or empty password in production"
target_scope: ["production"]
detection_query: |
  SELECT rolname FROM pg_authid
  WHERE rolcanlogin = true
  AND (rolpassword IS NULL OR rolpassword = 'md5' || md5(''))
detection_platform: postgresql
severity: critical
remediation_mode: auto
remediation_action: rotate_default_password
evidence_query: |
  SELECT rolname, rolpassword IS NOT NULL as has_password,
         now() as checked_at
  FROM pg_authid WHERE rolcanlogin = true
sla_minutes: 15
owner: platform-security@company.com
tags: ["pci", "authentication", "credentials"]

This structure means every compliance check is a first-class, testable artifact. It can be linted for completeness, loaded into a policy engine at startup, and every evaluation is traceable back to a specific version in your policy repository. When the auditor asks "how do you enforce PCI-DSS 8.6.1?", the answer is a Git commit hash.

The policy registry itself should live in a table structure that supports efficient querying by framework, scope, and severity — the compliance loop pulls active policies from this registry on each evaluation cycle.

PostgreSQL:

-- Policy registry table
CREATE TABLE compliance_policies (
    policy_id           TEXT PRIMARY KEY,
    framework           TEXT NOT NULL,
    control_ref         TEXT NOT NULL,
    description         TEXT NOT NULL,
    target_scope        TEXT[] NOT NULL,
    detection_query     TEXT NOT NULL,
    detection_platform  TEXT NOT NULL CHECK (detection_platform IN ('postgresql','sqlserver','prometheus')),
    severity            TEXT NOT NULL CHECK (severity IN ('critical','high','medium','low')),
    remediation_mode    TEXT NOT NULL CHECK (remediation_mode IN ('auto','approve','alert')),
    remediation_action  TEXT NOT NULL,
    evidence_query      TEXT NOT NULL,
    sla_minutes         INT NOT NULL DEFAULT 60,
    owner               TEXT NOT NULL,
    tags                TEXT[],
    is_active           BOOLEAN NOT NULL DEFAULT TRUE,
    version             INT NOT NULL DEFAULT 1,
    last_modified       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    modified_by         TEXT NOT NULL DEFAULT current_user
);

CREATE INDEX idx_compliance_policies_framework
    ON compliance_policies (framework)
    WHERE is_active = TRUE;

CREATE INDEX idx_compliance_policies_scope
    ON compliance_policies USING GIN (target_scope)
    WHERE is_active = TRUE;

-- Findings table — every detected violation is a row
CREATE TABLE compliance_findings (
    finding_id          BIGSERIAL PRIMARY KEY,
    policy_id           TEXT NOT NULL REFERENCES compliance_policies (policy_id),
    detected_at         TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    target_database     TEXT NOT NULL,
    target_host         TEXT NOT NULL,
    violation_detail    JSONB NOT NULL,
    severity            TEXT NOT NULL,
    status              TEXT NOT NULL DEFAULT 'open'
                        CHECK (status IN ('open','remediating','resolved','suppressed','escalated')),
    remediation_started TIMESTAMPTZ,
    remediation_result  JSONB,
    resolved_at         TIMESTAMPTZ,
    sla_deadline        TIMESTAMPTZ GENERATED ALWAYS AS
                        (detected_at + (sla_minutes || ' minutes')::INTERVAL) STORED,
    sla_minutes         INT NOT NULL
);

CREATE INDEX idx_findings_status ON compliance_findings (status, detected_at DESC);
CREATE INDEX idx_findings_sla    ON compliance_findings (sla_deadline) WHERE status = 'open';

                        REFERENCES compliance_policies (policy_id),

    target_database     NVARCHAR(200)  NOT NULL,
    target_host         NVARCHAR(200)  NOT NULL,
    violation_detail    NVARCHAR(MAX)  NOT NULL,  -- JSON
    severity            NVARCHAR(20)   NOT NULL,
    status              NVARCHAR(30)   NOT NULL DEFAULT 'open'
                        CHECK (status IN ('open','remediating','resolved',
                                          'suppressed','escalated')),
    remediation_started DATETIMEOFFSET NULL,
    remediation_result  NVARCHAR(MAX)  NULL,      -- JSON
    resolved_at         DATETIMEOFFSET NULL,
    sla_deadline        AS (DATEADD(MINUTE, sla_minutes, detected_at)) PERSISTED,
    sla_minutes         INT            NOT NULL

CREATE INDEX IX_findings_status ON compliance_findings (status, detected_at DESC);
CREATE INDEX IX_findings_sla    ON compliance_findings (sla_deadline)
    WHERE status = 'open';

The separation of detection_query from evidence_query is intentional: the detection query is optimized for fast violation identification (it may return only violating rows), while the evidence query produces the full snapshot that goes into the audit report. Running the evidence query only at finding time and at resolution time keeps audit storage manageable without sacrificing completeness.

The Compliance Detection Loop: Continuous Evaluation Architecture

A compliance system that runs once per day is not autonomous — it is a scheduled report. An autonomous system evaluates policies on a cadence proportional to severity: critical policies scan every one to five minutes, high-severity policies every fifteen, medium and low every hour. The detection loop is the engine that makes this happen.

The loop architecture has four components: the policy scheduler (determines which policies are due for evaluation), the query executor (runs detection queries against each registered database), the finding processor (compares results against open findings to detect new violations and resolutions), and the action dispatcher (routes new findings to the remediation engine or approval workflow based on remediation mode).

# compliance_loop.py — core detection loop
import asyncio
import asyncpg
import logging
from datetime import datetime, timezone
from typing import Dict, List, Any
from policy_registry import PolicyRegistry
from finding_processor import FindingProcessor
from action_dispatcher import ActionDispatcher

logger = logging.getLogger("compliance_loop")

CADENCE_MAP = {
    "critical": 60,      # seconds
    "high":     900,
    "medium":   3600,
    "low":      14400,
}

class ComplianceLoop:
    def __init__(self, registry: PolicyRegistry,
                 processor: FindingProcessor,
                 dispatcher: ActionDispatcher,
                 target_databases: List[Dict]):
        self.registry   = registry
        self.processor  = processor
        self.dispatcher = dispatcher
        self.targets    = target_databases
        self._last_run: Dict[str, datetime] = {}

    async def run_forever(self):
        logger.info("Compliance detection loop started")
        while True:
            policies = await self.registry.get_active_policies()
            now = datetime.now(timezone.utc)

            tasks = []
            for policy in policies:
                last = self._last_run.get(policy.policy_id)
                cadence = CADENCE_MAP[policy.severity]
                if last is None or (now - last).total_seconds() >= cadence:
                    for target in self.targets:
                        if self._scope_matches(policy.target_scope, target["env"]):
                            tasks.append(
                                self._evaluate_policy(policy, target, now)
                            )
                            self._last_run[policy.policy_id] = now

            if tasks:
                results = await asyncio.gather(*tasks, return_exceptions=True)
                for r in results:
                    if isinstance(r, Exception):
                        logger.error("Evaluation error: %s", r)

            await asyncio.sleep(10)  # tight loop; cadence enforcement is internal

    async def _evaluate_policy(self, policy, target: Dict, eval_time: datetime):
        try:
            violations = await self._run_detection_query(policy, target)
            await self.processor.process(
                policy=policy,
                target=target,
                violations=violations,
                eval_time=eval_time
            )
        except Exception as e:
            logger.error("Policy %s failed on %s: %s",
                         policy.policy_id, target["host"], e)
            raise

    async def _run_detection_query(self, policy, target: Dict) -> List[Any]:
        if policy.detection_platform == "postgresql":
            conn = await asyncpg.connect(
                host=target["host"], port=target.get("port", 5432),
                database=target["database"],
                user=target["user"], password=target["password"]
            )
            try:
                rows = await conn.fetch(policy.detection_query)
                return [dict(r) for r in rows]
            finally:
                await conn.close()
        elif policy.detection_platform == "sqlserver":
            # pyodbc path, called synchronously in executor
            return await asyncio.get_event_loop().run_in_executor(
                None, self._run_sqlserver_query,
                policy.detection_query, target
            )
        return []

    @staticmethod
    def _scope_matches(scope_list: List[str], env: str) -> bool:
        return env in scope_list or "all" in scope_list

The finding processor is the component responsible for idempotency — it must not create duplicate findings for a violation that is already open, and it must automatically resolve findings when the detection query no longer returns the violating rows.

PostgreSQL:

-- Finding processor upsert logic — run inside the Python processor
-- New finding: insert if no open finding exists for this policy+target+detail
INSERT INTO compliance_findings
    (policy_id, target_database, target_host,
     violation_detail, severity, status, sla_minutes)
SELECT
    $1::TEXT,                      -- policy_id
    $2::TEXT,                      -- target_database
    $3::TEXT,                      -- target_host
    $4::JSONB,                     -- violation_detail (fingerprint key inside)
    $5::TEXT,                      -- severity
    'open',
    $6::INT                        -- sla_minutes
WHERE NOT EXISTS (
    SELECT 1 FROM compliance_findings
    WHERE policy_id       = $1
      AND target_database = $2
      AND target_host     = $3
      AND violation_detail->>'fingerprint' = $4::JSONB->>'fingerprint'
      AND status IN ('open', 'remediating')
);

-- Auto-resolve findings whose violations are no longer detected
UPDATE compliance_findings
SET    status      = 'resolved',
       resolved_at = NOW(),
       remediation_result = jsonb_build_object(
           'resolution', 'auto_resolved',
           'note', 'Violation no longer detected by detection query'
       )
WHERE  policy_id       = $1
  AND  target_database = $2
  AND  target_host     = $3
  AND  status IN ('open', 'remediating')
  AND  violation_detail->>'fingerprint' NOT IN (
      SELECT jsonb_array_elements_text($7::JSONB)  -- current violation fingerprints
  );

SQL Server:

-- Finding processor upsert — SQL Server equivalent
-- Insert new finding if not already open
INSERT INTO compliance_findings
    (policy_id, target_database, target_host,
     violation_detail, severity, status, sla_minutes)
SELECT
    @policy_id, @target_db, @target_host,
    @violation_detail, @severity, 'open', @sla_minutes
WHERE NOT EXISTS (
    SELECT 1 FROM compliance_findings
    WHERE policy_id       = @policy_id
      AND target_database = @target_db
      AND target_host     = @target_host
      AND JSON_VALUE(violation_detail, '$.fingerprint')
            = JSON_VALUE(@violation_detail, '$.fingerprint')
      AND status IN ('open', 'remediating')
);

-- Auto-resolve cleared findings
UPDATE compliance_findings
SET    status      = 'resolved',
       resolved_at = SYSDATETIMEOFFSET(),
       remediation_result = JSON_MODIFY('{}', '$.resolution', 'auto_resolved')
WHERE  policy_id       = @policy_id
  AND  target_database = @target_db
  AND  target_host     = @target_host
  AND  status IN ('open', 'remediating')
  AND  JSON_VALUE(violation_detail, '$.fingerprint')
         NOT IN (SELECT value FROM OPENJSON(@current_fingerprints));

The fingerprint approach deserves emphasis. Each violation row produced by a detection query must be reduced to a stable string key (typically a hash of the identifying fields — a username, a table name, a setting name) before comparison. Without fingerprinting, every evaluation cycle would generate spurious new findings for violations already in flight, and auto-resolution logic would fire incorrectly. The fingerprint is computed in the finding processor before writing to the database, not inside the SQL.

SLA enforcement runs as a separate, lightweight background coroutine that wakes every sixty seconds and promotes any open finding past its sla_deadline to escalated, then fires the escalation webhook:

async def sla_watchdog(pool, escalation_webhook: str):
    """Promote overdue open findings to escalated and notify."""
    import aiohttp, json
    async with pool.acquire() as conn:
        overdue = await conn.fetch("""
            UPDATE compliance_findings
            SET    status = 'escalated'
            WHERE  status = 'open'
              AND  sla_deadline < NOW()
            RETURNING finding_id, policy_id, target_host,
                      severity, violation_detail, detected_at, sla_deadline
        """)
    if overdue:
        payload = {
            "alert": "compliance_sla_breach",
            "count": len(overdue),
            "findings": [dict(r) for r in overdue]
        }
        async with aiohttp.ClientSession() as session:
            await session.post(escalation_webhook,
                               json=payload,
                               timeout=aiohttp.ClientTimeout(total=10))
        logger.warning("Escalated %d overdue compliance findings", len(overdue))

Autonomous Remediation: What Gets Auto-Fixed and What Doesn't

Not all compliance violations should be autonomously remediated. The remediation mode embedded in each policy object encodes the decision: auto means the system corrects the violation immediately and records what it did; approve means it opens an approval request and waits up to the SLA deadline before either escalating or self-approving based on risk tier; alert means it only notifies.

The remediation registry maps action names to Python callables. Each callable receives the finding, the target database connection parameters, and the full policy object. It must return a RemediationResult that includes success/failure, the SQL or commands executed, and enough context that an auditor could reconstruct the action from the result alone.

The set of actions that are safe to automate fully (auto mode) across production systems is narrower than intuition suggests. Good candidates for full automation:

  • → Revoking excessive privileges added to service accounts outside the change window
  • → Disabling login for accounts inactive beyond the policy threshold
  • → Forcing TLS requirement at the session level for connections detected without encryption
  • → Rotating credentials that match default or empty password patterns
  • → Re-enabling audit logging extensions that have been dropped or disabled
  • → Reclassifying data sensitivity labels on newly discovered columns containing PII patterns

    • Actions that warrant approve mode even when technically automatable:

  • → Dropping database objects (even temp tables containing unmasked PII)
  • → Changing authentication configuration at the cluster level
  • → Modifying role membership for roles used by application service accounts
  • → Encryption-at-rest changes that require restart or key rotation

    • The following shows a full remediation action for one of the highest-frequency PCI-DSS violations: an account with excessive login privileges that was provisioned outside the IAM workflow.

    PostgreSQL:

    -- Remediation: revoke LOGIN from unauthorized service account
-- Called after human approval (remediation_mode: approve)
DO $$
DECLARE
    v_role    TEXT := $ROLE_NAME$;
    v_reason  TEXT := 'Compliance auto-remediation: PCI-8.2.1 unauthorized login privilege';
BEGIN
    -- Capture current state for audit record before change
    INSERT INTO compliance_remediation_audit
        (action, role_name, old_state, new_state, executed_at, policy_id)
    SELECT
        'REVOKE_LOGIN',
        v_role,
        jsonb_build_object(
            'rolcanlogin', rolcanlogin,
            'rolvaliduntil', rolvaliduntil,
            'rolconfig', rolconfig
        ),
        jsonb_build_object('rolcanlogin', false),
        NOW(),
        $POLICY_ID$
    FROM pg_roles WHERE rolname = v_role;

    -- Execute the remediation
    EXECUTE format('ALTER ROLE %I NOLOGIN', v_role);

    -- Terminate any existing sessions immediately
    PERFORM pg_terminate_backend(pid)
    FROM pg_stat_activity
    WHERE usename = v_role AND pid <> pg_backend_pid();

    RAISE NOTICE 'Login revoked for role % and % sessions terminated',
        v_role,
        (SELECT count(*) FROM pg_stat_activity WHERE usename = v_role);
END;
$$;

    SQL Server:

    -- Remediation: disable unauthorized login — SQL Server equivalent
BEGIN TRANSACTION;

    -- Capture current state
    INSERT INTO compliance_remediation_audit
        (action, login_name, old_state, new_state, executed_at, policy_id)
    SELECT
        'DISABLE_LOGIN',
        @login_name,
        (SELECT is_disabled, create_date, modify_date
         FROM sys.server_principals
         WHERE name = @login_name
         FOR JSON PATH, WITHOUT_ARRAY_WRAPPER),
        '{"is_disabled": true}',
        SYSDATETIMEOFFSET(),
        @policy_id
    FROM sys.server_principals
    WHERE name = @login_name;

    -- Execute remediation
    DECLARE @sql NVARCHAR(500) =
        N'ALTER LOGIN ' + QUOTENAME(@login_name) + N' DISABLE';
    EXEC sp_executesql @sql;

    -- Kill active sessions for this login
    DECLARE @session_id INT;
    DECLARE session_cursor CURSOR FOR
        SELECT session_id FROM sys.dm_exec_sessions
        WHERE login_name = @login_name;

    OPEN session_cursor;
    FETCH NEXT FROM session_cursor INTO @session_id;
    WHILE @@FETCH_STATUS = 0
    BEGIN
        DECLARE @kill_sql NVARCHAR(50) = N'KILL ' + CAST(@session_id AS NVARCHAR);
        EXEC sp_executesql @kill_sql;
        FETCH NEXT FROM session_cursor INTO @session_id;
    END;
    CLOSE session_cursor;
    DEALLOCATE session_cursor;

COMMIT TRANSACTION;

    Notice that both remediation scripts write to a compliance_remediation_audit table *before* executing the change. This is non-negotiable for autonomous systems: the audit record must exist even if the remediation command partially fails. The finding's remediation_result column then receives the full result including the audit record ID, so any auditor can trace a finding through its detection, approval, remediation, and evidence chain as a linked set
