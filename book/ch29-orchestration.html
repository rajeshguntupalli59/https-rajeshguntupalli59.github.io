Chapter 29: Orchestration

Database orchestration is the practice of coordinating, scheduling, and managing the execution of database-related tasks across systems, environments, and time. For a DBA, this means moving beyond individual maintenance scripts and one-off jobs into a structured approach where backup pipelines, ETL workflows, failover sequences, schema migrations, and health checks are all managed as interdependent, observable processes. This chapter covers the core concepts of orchestration as they apply to PostgreSQL and SQL Server, from basic job scheduling through to multi-step pipeline coordination using modern tools — giving you the mental model and practical skills to build workflows that are reliable, auditable, and recoverable when things go wrong.

---

What Orchestration Actually Means for a DBA

The word "orchestration" gets borrowed from container infrastructure discussions so frequently that its original meaning for data teams sometimes gets lost. For a DBA, orchestration is fundamentally about answering three questions: *What needs to run? In what order? Under what conditions?*

A single cron job that runs a backup script at 2 AM is not orchestration. It is scheduling. Orchestration begins when you have dependencies between tasks — when a backup must complete before archiving begins, when a statistics refresh must happen after a bulk load, when a failover script must pause, verify replica lag, then proceed only if lag is below a threshold. Managing that kind of conditional, multi-step workflow in a way that is visible, retryable, and auditable is what orchestration means in practice.

For PostgreSQL and SQL Server shops, the motivation for orchestration comes from two directions. First, maintenance operations grow in complexity as databases scale. A single VACUUM ANALYZE on a small table is trivial. Coordinating VACUUM operations across hundreds of tables, prioritizing bloat, avoiding conflicts with long-running transactions, and logging outcomes is not. Second, data pipelines increasingly depend on the database as a participant rather than just a destination — triggering downstream processes after loads complete, signaling applications when schema changes take effect, or coordinating across multiple databases in a fleet.

The traditional DBA response to this complexity was to write shell scripts or stored procedures and schedule them with the native scheduler. That works until it doesn't — until a script silently fails, until a dependency is assumed rather than checked, until a pipeline runs for six hours before anyone realizes step two never started because step one exited with code zero but produced no output. Modern orchestration tools address these failure modes directly.

---

Native Scheduling: pgAgent, SQL Server Agent, and Their Limits

Both PostgreSQL and SQL Server ship with or have closely associated native scheduling tools, and every DBA should know them deeply even if they eventually move beyond them. Understanding what these tools do well tells you exactly what gaps you will need to fill.

SQL Server Agent is built into SQL Server (excluding Express Edition) and provides a mature job scheduling framework. A SQL Server Agent job consists of one or more steps, each of which can be a T-SQL batch, an SSIS package execution, a PowerShell script, or a command-line executable. Steps are linked by success and failure outcomes — step two can be configured to run on success, run on failure, or be skipped entirely. Alerts can be attached to jobs based on SQL Server error severity levels, and the job history is stored in msdb, queryable through the sysjobs, sysjobsteps, and sysjobhistory tables.

SQL Server:

-- Query recent job execution history with duration and outcome
SELECT
    j.name AS job_name,
    h.step_name,
    h.run_date,
    h.run_time,
    CASE h.run_status
        WHEN 0 THEN 'Failed'
        WHEN 1 THEN 'Succeeded'
        WHEN 2 THEN 'Retry'
        WHEN 3 THEN 'Cancelled'
        WHEN 4 THEN 'In Progress'
    END AS status,
    -- Convert HHMMSS integer to seconds
    (h.run_duration / 10000 * 3600)
        + ((h.run_duration % 10000) / 100 * 60)
        + (h.run_duration % 100) AS duration_seconds
FROM msdb.dbo.sysjobhistory h
JOIN msdb.dbo.sysjobs j ON h.job_id = j.job_id
WHERE h.run_date >= CONVERT(INT, CONVERT(VARCHAR, GETDATE()-7, 112))
ORDER BY h.run_date DESC, h.run_time DESC;

pgAgent is a companion tool for PostgreSQL, installed separately and typically managed through pgAdmin. It supports multi-step jobs with steps written in SQL or shell script, scheduled using cron-style expressions. Job history is stored in the pgagent schema within a nominated database. pgAgent is functional but less feature-rich than SQL Server Agent — there is no native conditional branching between steps, and alerting capabilities are minimal unless extended externally.

PostgreSQL:

-- Query pgAgent job run history (requires pgagent schema installed)
SELECT
    j.jobname,
    jl.jlgstart AS started_at,
    jl.jlgduration AS duration,
    CASE jl.jlgstatus
        WHEN 'r' THEN 'Running'
        WHEN 's' THEN 'Succeeded'
        WHEN 'f' THEN 'Failed'
        WHEN 'i' THEN 'Internal Error'
        ELSE jl.jlgstatus
    END AS status
FROM pgagent.pga_joblog jl
JOIN pgagent.pga_job j ON jl.jlgjobid = j.jobid
WHERE jl.jlgstart >= NOW() - INTERVAL '7 days'
ORDER BY jl.jlgstart DESC;

The limits of native schedulers become apparent in several situations. Both SQL Server Agent and pgAgent store their configuration in the same database they are maintaining — a problem when that database is unavailable. Multi-database or multi-host coordination is essentially manual. Retry logic is rudimentary: SQL Server Agent can retry a step a configurable number of times with a fixed delay, but there is no backoff, no dead-letter handling, and no way to inspect the exact condition under which a retry fired. Cross-environment workflows — development to staging to production promotion — cannot be expressed natively.

These are not criticisms of the tools for what they were designed to do. For a single SQL Server instance running nightly maintenance jobs, SQL Server Agent is completely appropriate. The signal to look beyond native schedulers is when a workflow involves multiple systems, conditional logic that cannot be expressed as step outcomes, or a requirement for operational visibility and alerting that a team can act on in real time.

Expressing Database Workflows as Directed Acyclic Graphs

The concept borrowed most productively from general-purpose orchestration tools is the directed acyclic graph, or DAG. A DAG is simply a way of expressing tasks and the dependencies between them without any cycles — meaning no task can depend on itself, even indirectly. For database workflows, this model is clarifying.

Consider a typical end-of-day processing workflow. Raw transactions are loaded into a staging table, then validated, then merged into the fact table, then the materialized view is refreshed, then a downstream reporting database is updated, then a summary email is sent. Each step depends on the previous one completing successfully. Expressed as a DAG, the shape of that workflow is visible — you can see the critical path, identify which steps could be parallelized, and reason about what partial completion looks like if something fails midway.

Tools like Apache Airflow, Prefect, and Dagster have made DAG-based orchestration accessible to data teams. All three can orchestrate SQL tasks directly — executing queries against PostgreSQL or SQL Server as part of a larger pipeline — and all three provide a web UI for monitoring pipeline runs, inspecting logs, and manually retrying failed tasks. The choice between them is largely a matter of operational preference and team familiarity, but from a DBA's perspective the relevant capabilities are: native database connection management, the ability to pass results between tasks, retry configuration with backoff, and clear failure alerting.

Even if your organization does not use a dedicated orchestration tool, thinking in DAG terms improves how you design and document workflows. Before writing a maintenance script, draw the dependency graph. Identify what each step requires to be true before it can start, and what it must produce for the next step. This discipline catches a common failure mode: implicit ordering that breaks when execution time changes. A VACUUM step that "always finishes before the backup starts" because they run at 1 AM and 2 AM respectively is not an orchestrated dependency — it is a timing assumption waiting to be violated on a night when VACUUM runs long.

Coordinating Maintenance Pipelines Across PostgreSQL and SQL Server

The most common orchestration work a DBA does is coordinating database maintenance — backup, VACUUM or update statistics, index rebuild or reorganize, and integrity checks — in a way that respects dependencies, respects maintenance windows, and generates auditable records. Here is how that coordination looks in practice for each platform.

For PostgreSQL, a production maintenance pipeline typically includes: identifying bloated tables programmatically, running VACUUM ANALYZE on high-priority targets, rebuilding indexes concurrently for the most bloated, and capturing before/after metrics. The programmatic identification step is itself a query that becomes part of the orchestration:

PostgreSQL:

-- Identify tables with significant dead tuple bloat, suitable for
-- feeding a VACUUM prioritization step in a pipeline
SELECT
    schemaname,
    relname AS table_name,
    n_dead_tup,
    n_live_tup,
    ROUND(100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 2)
        AS dead_tuple_pct,
    last_vacuum,
    last_autovacuum,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||relname))
        AS total_size
FROM pg_stat_user_tables
WHERE n_dead_tup > 10000
  AND (last_vacuum IS NULL OR last_vacuum < NOW() - INTERVAL '24 hours')
ORDER BY dead_tuple_pct DESC
LIMIT 20;

This query is the first task in a pipeline. An orchestration tool runs it, captures the result set, and passes the list of table names to subsequent VACUUM tasks — potentially running several in parallel with a configurable concurrency limit to avoid overwhelming the server. After each VACUUM, a verification task checks that n_dead_tup has decreased and logs the result. This is not possible with a static cron-scheduled script, because the set of tables to process is dynamic.

For SQL Server, the equivalent pipeline for index maintenance queries fragmentation data first:

SQL Server:

-- Identify indexes with significant fragmentation for a maintenance pipeline
SELECT
    DB_NAME() AS database_name,
    OBJECT_SCHEMA_NAME(ips.object_id) AS schema_name,
    OBJECT_NAME(ips.object_id) AS table_name,
    i.name AS index_name,
    ips.index_type_desc,
    ROUND(ips.avg_fragmentation_in_percent, 2) AS fragmentation_pct,
    ips.page_count,
    CASE
        WHEN ips.avg_fragmentation_in_percent >= 30 THEN 'REBUILD'
        WHEN ips.avg_fragmentation_in_percent >= 10 THEN 'REORGANIZE'
        ELSE 'SKIP'
    END AS recommended_action
FROM sys.dm_db_index_physical_stats(
    DB_ID(), NULL, NULL, NULL, 'LIMITED') ips
JOIN sys.indexes i
    ON ips.object_id = i.object_id
    AND ips.index_id = i.index_id
WHERE ips.page_count > 1000
  AND ips.avg_fragmentation_in_percent > 10
ORDER BY ips.avg_fragmentation_in_percent DESC;

The orchestration layer takes this result, fans out REBUILD and REORGANIZE tasks per index, tracks which completed successfully, and generates a maintenance summary. When a REBUILD fails — perhaps because a table was locked by a long-running transaction — the pipeline marks that task as failed, retries with a backoff, and alerts if retries are exhausted. Compare this to a monolithic T-SQL maintenance script that encounters the same lock: it either hangs, times out and aborts the entire run, or skips the problem silently depending on how error handling was written.

A practical architectural decision at this stage is whether database maintenance pipelines should be expressed inside the database (as stored procedures called by native schedulers) or outside it (as orchestration tool tasks calling the database). The answer depends on your operational environment. Pure SQL-based maintenance keeps the logic close to the engine and avoids external dependencies — a valid choice for stable, well-understood workloads. External orchestration gives you better observability, multi-system coordination, and decoupled retry logic — worth the additional complexity when pipelines cross database boundaries or need integration with broader data infrastructure.

Handling Failures, Retries, and Idempotency

The difference between a maintenance script and a production-grade pipeline is almost entirely about how failure is handled. An orchestration system that cannot gracefully handle partial failures is a system that will be manually repaired repeatedly under time pressure. Building failure handling in from the start is not defensive programming — it is the design.

The first principle is idempotency. An idempotent task can be run multiple times and produce the same result as running it once. For database tasks, this usually means using upsert patterns rather than blind inserts, checking preconditions before executing destructive operations, and designing cleanup tasks that are safe to re-run. When an orchestration system retries a failed task, it should be safe to do so without worrying about partial state from the previous attempt.

For SQL Server, the MERGE statement or conditional INSERT/UPDATE patterns provide idempotency in data loading steps. For PostgreSQL, INSERT ... ON CONFLICT DO UPDATE (upsert) handles the same scenario:

PostgreSQL:

-- Idempotent upsert into a pipeline audit log table
-- Safe to retry without creating duplicate records
INSERT INTO pipeline_run_log (
    pipeline_name,
    run_date,
    step_name,
    status,
    rows_affected,
    completed_at
)
VALUES (
    $1, CURRENT_DATE, $2, $3, $4, NOW()
)
ON CONFLICT (pipeline_name, run_date, step_name)
DO UPDATE SET
    status = EXCLUDED.status,
    rows_affected = EXCLUDED.rows_affected,
    completed_at = EXCLUDED.completed_at;

SQL Server:

-- Equivalent idempotent upsert for pipeline audit logging
MERGE pipeline_run_log AS target
USING (SELECT @pipeline_name, CAST(GETDATE() AS DATE),
              @step_name, @status, @rows_affected, GETDATE())
      AS source (pipeline_name, run_date, step_name,
                 status, rows_affected, completed_at)
ON  target.pipeline_name = source.pipeline_name
AND target.run_date       = source.run_date
AND target.step_name      = source.step_name
WHEN MATCHED THEN
    UPDATE SET status        = source.status,
               rows_affected = source.rows_affected,
               completed_at  = source.completed_at
WHEN NOT MATCHED THEN
    INSERT (pipeline_name, run_date, step_name,
            status, rows_affected, completed_at)
    VALUES (source.pipeline_name, source.run_date, source.step_name,
            source.status, source.rows_affected, source.completed_at);

The second principle is explicit retry configuration. Each step in a pipeline should have a defined retry count, retry interval, and backoff strategy. A task that calls an external API should retry quickly — perhaps three times with five-second intervals. A VACUUM that fails because of a conflicting lock should retry with exponential backoff — perhaps three times with intervals of 5 minutes, 15 minutes, and 45 minutes. Flat retry intervals cause thundering herd problems when multiple parallel tasks fail simultaneously and all retry at the same moment.

The third principle is clear failure escalation. When retries are exhausted, the pipeline should stop, mark the task as permanently failed, and alert the responsible team. "Stop and alert" is almost always the correct behavior. The alternative — skipping failed steps and continuing — can leave the system in a partially consistent state that is worse than a clean failure. There are exceptions: a summary report step that fails should not prevent an hourly data load from continuing. Modeling these exceptions explicitly in the DAG, as separate branches with different failure policies, makes the behavior intentional rather than accidental.

Orchestrating Schema Migrations as Part of Deployment Pipelines

Schema changes are operationally distinct from data changes, but they need the same orchestration discipline — arguably more, because a bad schema migration cannot be rolled back with a simple DELETE. Incorporating schema migrations into deployment pipelines with proper pre-checks, execution, and post-validation steps is one of the higher-value applications of orchestration for a DBA.

Tools like Flyway and Liquibase express schema migrations as versioned, ordered scripts. Each migration has a checksum that prevents modification after application, and the tool tracks which migrations have been applied against each target environment. The orchestration layer's job is to run the migration tool at the right point in the deployment sequence — after the application artifact is built but before new application instances start, with rollback capability if validation fails after migration.

The pre-migration step matters as much as the migration itself. Before applying a migration that adds a NOT NULL column to a large table, an orchestration pipeline should verify estimated table size, check current lock wait times, confirm replication lag is acceptable, and verify that a backup completed successfully within the last window. These checks are straightforward queries:

PostgreSQL:

-- Pre-migration checks: table size, active connections, replication lag
SELECT
    pg_size_pretty(pg_total_relation_size('public.orders')) AS table_size,
    (SELECT count(*) FROM pg_stat_activity
     WHERE state = 'active'
       AND wait_event_type = 'Lock') AS active_lock_waiters,
    (SELECT CASE
        WHEN pg_is_in_recovery() THEN
            EXTRACT(EPOCH FROM (NOW() - pg_last_xact_replay_timestamp()))
        ELSE 0
     END) AS replication_lag_seconds;

SQL Server:

-- Pre-migration checks: table size, blocking, AG replica lag
SELECT
    (8 * SUM(a.total_pages)) / 1024 AS table_size_mb,
    (SELECT COUNT(*)
     FROM sys.dm_exec_requests
     WHERE blocking_session_id <> 0) AS blocked_requests,
    (SELECT MAX(DATEDIFF(SECOND,
                         last_commit_time,
                         GETDATE()))
     FROM sys.dm_hadr_database_replica_states drs
     JOIN sys.availability_replicas ar
         ON drs.replica_id = ar.replica_id
     WHERE is_local = 0) AS max_replica_lag_seconds
FROM sys.tables t
JOIN sys.schemas s ON t.schema_id = s.schema_id
JOIN sys.partitions p ON t.object_id = p.object_id
JOIN sys.allocation_units a ON p.partition_id = a.container_id
WHERE s.name = 'dbo'
  AND t.name = 'orders'
  AND p.index_id IN (0,1);

If any pre-migration check fails a defined threshold — replica lag above 30 seconds, more than zero active lock waiters on the target table — the orchestration pipeline halts and alerts rather than proceeding. This is the kind of guardrail that prevents a migration from running during a replication catch-up following a network event, which is exactly when most migration incidents happen.

Post-migration validation is equally important. After a migration runs, the pipeline should verify that the intended structural change exists, that critical indexes are still valid, and that a basic health query on the affected table succeeds. Only when all post-migration checks pass does the pipeline signal the deployment system to proceed with application startup. This end-to-end orchestration — pre-check, migrate, post-validate, signal — transforms schema deployment from a manual, anxiety-producing operation into a repeatable, observable process.

---

Key Takeaways

  • → **Orchestration is about dependencies and conditions, not just scheduling.** The moment a workflow has tasks that depend on each other's outcomes, you have moved beyond cron jobs into orchestration territory, and the design should reflect that explicitly.
  • → **Native schedulers (SQL Server Agent, pgAgent) are appropriate for isolated, single-system jobs** but hit their limits quickly with multi-host coordination, conditional branching, and retry logic — knowing these limits tells you when to invest in a dedicated orchestration layer.
  • → **Idempotency is the foundation of reliable pipelines.** Every database task in an orchestrated workflow should be safe to run multiple times without producing inconsistent state, using upsert patterns, pre-condition checks, and audit logging that handles re-runs cleanly.
  • → **Dynamic maintenance pipelines — querying fragmentation or bloat first, then targeting only what needs work — outperform static scripts** because they adapt to the actual state of the database rather than applying fixed operations to all objects on a schedule.
  • → **Schema migrations belong inside deployment pipelines with pre-checks and post-validation steps.** Treating a migration as an isolated script run by hand is the operational antipattern that produces incidents; orchestrating it with explicit guardrails is what makes schema changes predictable and safe in production.
