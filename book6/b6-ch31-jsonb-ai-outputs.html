Chapter 31: JSONB for Flexible AI Output Storage

AI models produce output in shapes that evolve constantly. A function-calling response from GPT-4o looks nothing like a streaming delta chunk from Claude, and neither resembles the structured JSON you'll get from a fine-tuned extraction model next quarter. Rigid relational schemas crack under this pressure — you either over-provision columns that stay NULL most of the time, or you serialize everything into TEXT and lose queryability entirely. PostgreSQL's JSONB column type threads this needle cleanly: you get schema flexibility without sacrificing the ability to index, query, aggregate, and join against the data stored inside. This chapter shows how to design JSONB-backed storage for every major category of AI output — completions, tool calls, structured extractions, agent traces, and evaluation results — and how to keep that storage fast, consistent, and observable in production.

---

31.1 Why JSONB Is the Right Primitive for AI Output

The core tension in AI application storage is this: you need structure to query, but AI output structure is unstable. A model's response schema changes when you upgrade model versions, add tools, change prompts, or switch providers. JSONB lets you commit to a stable outer envelope — known columns like session_id, created_at, model, status — while keeping the inner payload flexible.

JSONB differs from JSON in one important way: PostgreSQL parses and stores it as a binary decomposition, not as raw text. This means:

  • → Duplicate keys are removed at write time (last-value-wins)
  • → Key order is not preserved
  • → Containment operators (`@>`, `<@`) work without full row scans when backed by GIN indexes
  • → Path extraction (`->`, `->>`, `#>`, `#>>`) returns typed values
  • → Expression indexes can be built over extracted paths

    • For AI output, the practical wins are substantial. You can write a raw LLM response as-is, then later query WHERE response_body->>'finish_reason' = 'tool_calls' or aggregate AVG((usage->>'total_tokens')::int) without a schema migration.

    The envelope pattern is the foundational design decision. Every AI output row should have a stable relational envelope that carries metadata the application always knows at write time, plus one or more JSONB columns that carry the variable payload:

    -- Core envelope pattern for all AI output storage
CREATE TABLE ai_outputs (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id      UUID NOT NULL REFERENCES sessions(id),
    request_id      UUID NOT NULL UNIQUE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    model           TEXT NOT NULL,
    provider        TEXT NOT NULL CHECK (provider IN ('openai', 'anthropic', 'google', 'local')),
    output_type     TEXT NOT NULL CHECK (output_type IN ('completion', 'tool_call', 'extraction', 'embedding', 'evaluation')),
    status          TEXT NOT NULL DEFAULT 'success' CHECK (status IN ('success', 'error', 'partial', 'filtered')),

    -- Stable cost/usage data — always present, always queried
    prompt_tokens   INT,
    completion_tokens INT,
    total_tokens    INT,
    latency_ms      INT,

    -- Variable payload — shape depends on output_type and model
    response_body   JSONB NOT NULL,
    metadata        JSONB NOT NULL DEFAULT '{}',

    -- For error cases
    error_detail    JSONB
);

-- GIN index for containment queries across response_body
CREATE INDEX idx_ai_outputs_response_gin
    ON ai_outputs USING GIN (response_body);

-- GIN index for metadata filtering (tags, pipeline IDs, experiment labels)
CREATE INDEX idx_ai_outputs_metadata_gin
    ON ai_outputs USING GIN (metadata);

-- B-tree indexes on the envelope columns used in WHERE clauses
CREATE INDEX idx_ai_outputs_session_created
    ON ai_outputs (session_id, created_at DESC);

CREATE INDEX idx_ai_outputs_model_type
    ON ai_outputs (model, output_type, created_at DESC);

-- Expression index for finish_reason — frequently filtered
CREATE INDEX idx_ai_outputs_finish_reason
    ON ai_outputs ((response_body->>'finish_reason'));

-- Expression index for tool name in tool_call outputs
CREATE INDEX idx_ai_outputs_tool_name
    ON ai_outputs ((response_body->'tool_calls'->0->'function'->>'name'))
    WHERE output_type = 'tool_call';

    This schema handles every output type from every provider without a single ALTER TABLE as your AI stack evolves.

    31.2 Storing LLM Completion and Tool Call Responses

    Different providers return completion payloads with different shapes. OpenAI wraps everything in a choices array; Anthropic returns a content array with typed blocks; local models via Ollama or vLLM return yet another shape. Rather than normalizing these at write time — which introduces lossy transformation and makes debugging harder — store the raw response and extract at query time.

    -- Completion storage with provider-native shapes preserved
CREATE TABLE completions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    request_id      UUID NOT NULL UNIQUE,
    session_id      UUID REFERENCES sessions(id),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),

    -- Envelope: always populated, always relational
    model           TEXT NOT NULL,
    provider        TEXT NOT NULL,
    prompt_tokens   INT GENERATED ALWAYS AS
                        (COALESCE(
                            (response_body->'usage'->>'prompt_tokens')::int,
                            (response_body->'usage'->>'input_tokens')::int
                        )) STORED,
    completion_tokens INT GENERATED ALWAYS AS
                        (COALESCE(
                            (response_body->'usage'->>'completion_tokens')::int,
                            (response_body->'usage'->>'output_tokens')::int
                        )) STORED,
    finish_reason   TEXT GENERATED ALWAYS AS
                        (COALESCE(
                            response_body->'choices'->0->>'finish_reason',
                            response_body->>'stop_reason'
                        )) STORED,

    -- Raw provider response — zero transformation
    response_body   JSONB NOT NULL,

    -- Extracted content for fast retrieval without path expressions
    content_text    TEXT GENERATED ALWAYS AS
                        (COALESCE(
                            response_body->'choices'->0->'message'->>'content',
                            response_body->'content'->0->>'text'
                        )) STORED,

    -- Tool calls extracted as normalized array
    has_tool_calls  BOOLEAN GENERATED ALWAYS AS
                        (
                            (response_body->'choices'->0->'message'->'tool_calls') IS NOT NULL
                            OR
                            (response_body->>'stop_reason' = 'tool_use')
                        ) STORED,

    -- Application metadata
    pipeline_id     TEXT,
    prompt_version  TEXT,
    metadata        JSONB NOT NULL DEFAULT '{}'
);

-- Generated column indexes are just normal B-tree indexes
CREATE INDEX idx_completions_finish_reason
    ON completions (finish_reason, created_at DESC);

CREATE INDEX idx_completions_has_tool_calls
    ON completions (has_tool_calls, created_at DESC)
    WHERE has_tool_calls = TRUE;

CREATE INDEX idx_completions_session
    ON completions (session_id, created_at DESC);

-- Full GIN on response_body for ad-hoc exploration
CREATE INDEX idx_completions_response_gin
    ON completions USING GIN (response_body);

-- View that normalizes both provider shapes into a common interface
CREATE OR REPLACE VIEW completions_normalized AS
SELECT
    id,
    request_id,
    session_id,
    created_at,
    model,
    provider,
    finish_reason,
    content_text,
    has_tool_calls,
    prompt_tokens,
    completion_tokens,

    -- OpenAI tool calls
    CASE WHEN provider = 'openai' THEN
        response_body->'choices'->0->'message'->'tool_calls'
    -- Anthropic tool use
    WHEN provider = 'anthropic' THEN
        (
            SELECT jsonb_agg(
                jsonb_build_object(
                    'id', block->>'id',
                    'type', 'function',
                    'function', jsonb_build_object(
                        'name', block->>'name',
                        'arguments', block->'input'
                    )
                )
            )
            FROM jsonb_array_elements(response_body->'content') AS block
            WHERE block->>'type' = 'tool_use'
        )
    END AS tool_calls,

    pipeline_id,
    prompt_version,
    metadata
FROM completions;

    Now write the actual responses from both providers using Python:

    # storage/completion_store.py
import uuid
import time
import json
import asyncpg
import anthropic
from openai import AsyncOpenAI
from typing import Optional

openai_client = AsyncOpenAI()
anthropic_client = anthropic.AsyncAnthropic()


async def store_openai_completion(
    pool: asyncpg.Pool,
    session_id: str,
    messages: list[dict],
    tools: Optional[list[dict]] = None,
    pipeline_id: Optional[str] = None,
    prompt_version: Optional[str] = None,
) -> dict:
    request_id = str(uuid.uuid4())
    start_ts = time.monotonic()

    kwargs = {
        "model": "gpt-4o",
        "messages": messages,
    }
    if tools:
        kwargs["tools"] = tools

    response = await openai_client.chat.completions.create(**kwargs)
    latency_ms = int((time.monotonic() - start_ts) * 1000)

    # Store the raw response dict — no transformation
    response_dict = response.model_dump()

    async with pool.acquire() as conn:
        row = await conn.fetchrow(
            """
            INSERT INTO completions (
                request_id, session_id, model, provider,
                response_body, pipeline_id, prompt_version, metadata
            ) VALUES ($1, $2, $3, $4, $5, $6, $7, $8)
            RETURNING id, prompt_tokens, completion_tokens, finish_reason, has_tool_calls
            """,
            request_id,
            uuid.UUID(session_id) if session_id else None,
            response.model,
            "openai",
            json.dumps(response_dict),
            pipeline_id,
            prompt_version,
            json.dumps({"latency_ms": latency_ms, "request_id": request_id}),
        )

    return {
        "id": str(row["id"]),
        "request_id": request_id,
        "content": response.choices[0].message.content,
        "tool_calls": response.choices[0].message.tool_calls,
        "finish_reason": row["finish_reason"],
        "tokens": {
            "prompt": row["prompt_tokens"],
            "completion": row["completion_tokens"],
        },
    }


async def store_anthropic_completion(
    pool: asyncpg.Pool,
    session_id: str,
    messages: list[dict],
    system: Optional[str] = None,
    tools: Optional[list[dict]] = None,
    pipeline_id: Optional[str] = None,
) -> dict:
    request_id = str(uuid.uuid4())
    start_ts = time.monotonic()

    kwargs = {
        "model": "claude-opus-4-5",
        "max_tokens": 4096,
        "messages": messages,
    }
    if system:
        kwargs["system"] = system
    if tools:
        kwargs["tools"] = tools

    response = await anthropic_client.messages.create(**kwargs)
    latency_ms = int((time.monotonic() - start_ts) * 1000)

    response_dict = response.model_dump()

    async with pool.acquire() as conn:
        row = await conn.fetchrow(
            """
            INSERT INTO completions (
                request_id, session_id, model, provider,
                response_body, pipeline_id, metadata
            ) VALUES ($1, $2, $3, $4, $5, $6, $7)
            RETURNING id, prompt_tokens, completion_tokens, finish_reason, has_tool_calls
            """,
            request_id,
            uuid.UUID(session_id) if session_id else None,
            response.model,
            "anthropic",
            json.dumps(response_dict),
            pipeline_id,
            json.dumps({"latency_ms": latency_ms}),
        )

    # Extract text content from Anthropic's content block array
    text_content = next(
        (block.text for block in response.content if block.type == "text"),
        None,
    )
    tool_use_blocks = [
        block for block in response.content if block.type == "tool_use"
    ]

    return {
        "id": str(row["id"]),
        "request_id": request_id,
        "content": text_content,
        "tool_calls": tool_use_blocks if tool_use_blocks else None,
        "finish_reason": row["finish_reason"],
        "tokens": {
            "prompt": row["prompt_tokens"],
            "completion": row["completion_tokens"],
        },
    }


async def get_tool_call_stats(
    pool: asyncpg.Pool,
    pipeline_id: str,
    since_hours: int = 24,
) -> list[dict]:
    """Aggregate tool usage from stored JSONB — no separate tool_calls table needed."""
    async with pool.acquire() as conn:
        rows = await conn.fetch(
            """
            WITH tool_calls_expanded AS (
                SELECT
                    pipeline_id,
                    jsonb_array_elements(
                        CASE provider
                            WHEN 'openai' THEN
                                response_body->'choices'->0->'message'->'tool_calls'
                            WHEN 'anthropic' THEN
                                (
                                    SELECT jsonb_agg(b)
                                    FROM jsonb_array_elements(response_body->'content') b
                                    WHERE b->>'type' = 'tool_use'
                                )
                        END
                    ) AS tool_call
                FROM completions
                WHERE pipeline_id = $1
                  AND has_tool_calls = TRUE
                  AND created_at > now() - ($2 || ' hours')::interval
            )
            SELECT
                COALESCE(
                    tool_call->'function'->>'name',
                    tool_call->>'name'
                ) AS tool_name,
                COUNT(*) AS call_count
            FROM tool_calls_expanded
            GROUP BY 1
            ORDER BY 2 DESC
            """,
            pipeline_id,
            str(since_hours),
        )
    return [dict(r) for r in rows]

    31.3 Structured Extraction and Schema Validation

    One of the highest-value patterns in AI applications is structured extraction: feeding unstructured text to a model and getting back a JSON object conforming to a known schema. Resume parsing, contract analysis, medical note structuring, product catalog enrichment — all follow this pattern. PostgreSQL can enforce schema constraints on the extracted output using CHECK constraints against JSONB, giving you a validation layer that lives in the database regardless of what application code produces the data.

    -- Structured extraction results with in-database schema validation
CREATE TABLE extraction_results (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_id       UUID NOT NULL,          -- FK to whatever was parsed
    source_type     TEXT NOT NULL,          -- 'document', 'email', 'transcript'
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    model           TEXT NOT NULL,
    extraction_schema TEXT NOT NULL,        -- 'resume_v2', 'contract_v1', etc.
    schema_version  INT NOT NULL DEFAULT 1,

    -- The extracted payload
    extracted       JSONB NOT NULL,

    -- Confidence scores if model provides them
    confidence      JSONB,

    -- Validation state
    validation_status TEXT NOT NULL DEFAULT 'pending'
        CHECK (validation_status IN ('pending', 'valid', 'invalid', 'manual_review')),
    validation_errors JSONB,

    -- Human correction tracking
    corrected       JSONB,
    corrected_by    UUID REFERENCES users(id),
    corrected_at    TIMESTAMPTZ,

    raw_response    JSONB NOT NULL,
    latency_ms      INT
);

-- Constraint: resume extractions must have at minimum name and email
ALTER TABLE extraction_results ADD CONSTRAINT chk_resume_required_fields
    CHECK (
        extraction_schema NOT LIKE 'resume%'
        OR (
            extracted ? 'name'
            AND extracted ? 'email'
        )
    );

-- Constraint: contract extractions must have parties and effective_date
ALTER TABLE extraction_results ADD CONSTRAINT chk_contract_required_fields
    CHECK (
        extraction_schema NOT LIKE 'contract%'
        OR (
            extracted ? 'parties'
            AND extracted ? 'effective_date'
            AND jsonb_typeof(extracted->'parties') = 'array'
        )
    );

-- GIN index for querying inside extracted fields
CREATE INDEX idx_extraction_extracted_gin
    ON extraction_results USING GIN (extracted);

-- Find all extractions where a specific company appears as a party
CREATE INDEX idx_extraction_schema_status
    ON extraction_results (extraction_schema, validation_status, created_at DESC);

-- Function to run basic JSON schema validation and record errors
CREATE OR REPLACE FUNCTION validate_extraction(
    p_id UUID,
    p_schema TEXT
) RETURNS VOID AS $$
DECLARE
    v_extracted JSONB;
    v_errors    JSONB := '[]'::JSONB;
    v_status    TEXT := 'valid';
BEGIN
    SELECT extracted INTO v_extracted
    FROM extraction_results WHERE id = p_id;

    IF p_schema LIKE 'resume%' THEN
        IF NOT (v_extracted ? 'name') THEN
            v_errors := v_errors || '["missing required field: name"]'::JSONB;
        END IF;
        IF NOT (v_extracted ? 'email') THEN
            v_errors := v_errors || '["missing required field: email"]'::JSONB;
        END IF;
        IF (v_extracted ? 'years_experience')
           AND NOT (jsonb_typeof(v_extracted->'years_experience') = 'number') THEN
            v_errors := v_errors || '["years_experience must be numeric"]'::JSONB;
        END IF;
    END IF;

    IF jsonb_array_length(v_errors) > 0 THEN
        v_status := 'invalid';
    END IF;

    UPDATE extraction_results
    SET validation_status = v_status,
        validation_errors = v_errors
    WHERE id = p_id;
END;
$$ LANGUAGE plpgsql;
