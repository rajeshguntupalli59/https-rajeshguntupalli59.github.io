Chapter 11: Predictive Failure Detection

Reactive alerting is the floor, not the ceiling. When a database crashes, a disk fills, or a replication lag spikes to the point of data loss, the system has already failed — the alert is an obituary. Autonomous systems operate in a fundamentally different mode: they maintain continuous predictive models of component health and trigger remediation before the failure state is reached. This chapter covers the architecture, signal collection, model design, and automated response systems required to build genuine predictive failure detection into PostgreSQL and SQL Server environments — not as a monitoring dashboard that a human reads, but as a closed-loop system that detects, decides, and acts without waiting for a page.

---

From Threshold Alerts to Failure Probability Scores

Traditional monitoring assigns static thresholds: disk usage above 85% triggers a warning, above 95% triggers a critical. This model fails in production in two directions simultaneously. It fires too late — 95% disk is often already a crisis — and it fires too often on noise, causing alert fatigue that makes engineers ignore the signal. The ADE replaces static thresholds with failure probability scores derived from trend analysis, rate-of-change detection, and historical failure correlation.

The conceptual shift is from *state* to *trajectory*. A disk at 60% full that is growing at 4GB per hour is more dangerous than a disk at 80% that has been stable for three months. A probability score encodes both current state and projected state at some future horizon (typically T+1h, T+6h, T+24h).

The minimum viable predictive signal set for a database node includes:

  • → **Storage**: current utilization, fill rate (bytes/hour), projected exhaustion time
  • → **Memory**: buffer pool hit ratio trend, free memory rate-of-change, OOM kill history
  • → **Replication**: lag in bytes and seconds, lag acceleration, sender/receiver heartbeat delta
  • → **Connection pressure**: active vs. max connections ratio, wait event accumulation rate
  • → **I/O**: read/write latency percentiles (p50, p95, p99), IOPS saturation coefficient
  • → **Lock contention**: lock wait time trends, deadlock frequency (rolling 1h, 24h)
  • → **Query execution**: plan regression score (new execution time / baseline execution time)

    • Each signal is not evaluated in isolation but fed into a composite health score. The most effective early production implementation is a weighted linear model with hand-tuned weights derived from your historical incident data. More sophisticated deployments use gradient boosting models (XGBoost, LightGBM) trained on labeled failure events. Both approaches require the same foundation: a reliable, high-resolution signal pipeline.

    A signal pipeline at ADE scale looks like this: database-side metrics → Prometheus exporters (postgres_exporter, sql_exporter) → Prometheus TSDB → feature extraction layer → scoring model → decision engine → action executor. Every component except the database itself is replaceable; what cannot change is the requirement for sub-minute metric resolution and durable storage of at least 90 days of history for model training.

    Building the Failure Signal Pipeline for PostgreSQL and SQL Server

    Before prediction, you need instrumentation. The postgres_exporter alone is insufficient for predictive work — it exposes point-in-time snapshots. You need derived metrics: rates, deltas, moving averages, and percentiles computed over time windows. Prometheus recording rules are the right place to compute these at collection time rather than at query time.

    PostgreSQL — base signal queries (run by postgres_exporter custom queries):

    -- Replication lag signal: bytes behind + rate of change
SELECT
    application_name,
    pg_wal_lsn_diff(pg_current_wal_lsn(), sent_lsn)  AS sent_lag_bytes,
    pg_wal_lsn_diff(pg_current_wal_lsn(), replay_lsn) AS replay_lag_bytes,
    EXTRACT(EPOCH FROM (now() - write_lag))            AS write_lag_seconds,
    EXTRACT(EPOCH FROM (now() - replay_lag))           AS replay_lag_seconds,
    state,
    sync_state
FROM pg_stat_replication;

-- Buffer pool pressure signal
SELECT
    datname,
    blks_hit,
    blks_read,
    CASE WHEN (blks_hit + blks_read) = 0 THEN 1.0
         ELSE blks_hit::float / (blks_hit + blks_read)
    END AS buffer_hit_ratio,
    -- Low ratio is a leading indicator of memory pressure or bloat
    tup_returned,
    tup_fetched,
    deadlocks,
    temp_files,
    temp_bytes
FROM pg_stat_database
WHERE datname NOT IN ('template0', 'template1', 'postgres');

    SQL Server — equivalent signal surface:

    -- Replication/AG lag and synchronization health
SELECT
    ag.name                          AS ag_name,
    ar.replica_server_name,
    drs.database_name,
    drs.synchronization_state_desc,
    drs.synchronization_health_desc,
    drs.log_send_queue_size          AS log_send_queue_kb,
    drs.log_send_rate                AS log_send_rate_kbps,
    drs.redo_queue_size              AS redo_queue_kb,
    drs.redo_rate                    AS redo_rate_kbps,
    -- Derived: seconds of data at risk
    CASE WHEN drs.redo_rate > 0
         THEN drs.redo_queue_size / drs.redo_rate
         ELSE NULL
    END                              AS estimated_lag_seconds,
    drs.last_redone_time,
    drs.last_commit_time
FROM sys.dm_hadr_database_replica_states  drs
JOIN sys.availability_replicas             ar  ON ar.replica_id   = drs.replica_id
JOIN sys.availability_groups               ag  ON ag.group_id     = ar.group_id
WHERE drs.is_local = 0;

-- Buffer pool pressure and memory grant signal
SELECT
    counter_name,
    cntr_value
FROM sys.dm_os_performance_counters
WHERE object_name  LIKE '%Buffer Manager%'
   OR object_name  LIKE '%Memory Manager%'
   OR counter_name IN (
       'Buffer cache hit ratio',
       'Page life expectancy',
       'Memory Grants Pending',
       'Target Server Memory (KB)',
       'Total Server Memory (KB)'
   );

    The Prometheus recording rules that derive rate and trend signals belong in a dedicated predictive_rules.yml file, not mixed with alerting rules:

    # /etc/prometheus/rules/predictive_rules.yml
groups:
  - name: predictive_signals
    interval: 60s
    rules:
      # Disk fill rate: bytes per second, averaged over 30 minutes
      - record: db:disk_fill_rate_bytes_per_second:avg30m
        expr: >
          rate(node_filesystem_size_bytes{mountpoint="/var/lib/postgresql"}[30m])
          - rate(node_filesystem_free_bytes{mountpoint="/var/lib/postgresql"}[30m])

      # Projected seconds until disk exhaustion
      - record: db:disk_exhaustion_seconds:predicted
        expr: >
          node_filesystem_free_bytes{mountpoint="/var/lib/postgresql"}
          / clamp_min(db:disk_fill_rate_bytes_per_second:avg30m, 0.001)

      # Replication lag acceleration: is lag growing faster than last hour?
      - record: pg:replication_lag_acceleration:ratio
        expr: >
          rate(pg_replication_lag[10m]) / clamp_min(rate(pg_replication_lag[1h]), 0.001)

      # PLE trend for SQL Server nodes (via sql_exporter)
      - record: sqlserver:ple_trend:delta15m
        expr: >
          sqlserver_performance{counter="Page life expectancy"}
          - sqlserver_performance{counter="Page life expectancy"} offset 15m

    With these signals flowing, the prediction layer has the raw material it needs. The critical design principle: derive at collection time, not at alert evaluation time. Recording rules precompute trends continuously; alerting rules and prediction models consume them. This separates signal quality from alert latency.

    Failure Pattern Classification and Model Architecture

    Historical failure events are the training data. Every production incident — disk exhaustion, OOM kill, AG failover, replication disconnect, deadlock storm — leaves a signature in the metric stream for the 30–120 minutes preceding failure. The predictive model's job is to recognize that signature early enough to act.

    The practical model taxonomy for database failure prediction:

    Time-to-failure regression: Given current signal values and their recent trends, predict the number of minutes until a failure condition. Used for disk exhaustion, connection pool saturation, tablespace fill. Straightforward to implement with linear regression on rate signals.

    Anomaly-based classification: Build a baseline of "normal" metric distributions during healthy periods. Use statistical distance (Z-score, Mahalanobis distance) or isolation forest to score how far current behavior deviates from normal. Used for query plan regressions, lock pattern changes, unusual wait event distributions.

    Event sequence matching: Learn which sequences of events precede failures. "Checkpoint warnings followed by connection spike followed by lock escalation" precedes a specific class of write storm failure. Implemented with sequence pattern matching or lightweight LSTM on sliding windows.

    For most production deployments starting on this path, a Python-based scoring service using scikit-learn or XGBoost is the right balance of capability and operational simplicity:

    # failure_predictor.py — runs as a sidecar service, consumes Prometheus metrics
import time
import logging
import requests
import numpy as np
import joblib
from dataclasses import dataclass
from typing import Optional
from prometheus_client import Gauge, start_http_server

PROMETHEUS_URL = "http://prometheus:9090"
MODEL_PATH     = "/models/db_failure_model.pkl"
SCORE_INTERVAL = 60  # seconds

failure_score   = Gauge("db_failure_probability", "Predicted failure probability [0-1]",
                        ["instance", "failure_class"])
time_to_failure = Gauge("db_time_to_failure_minutes", "Predicted minutes to failure",
                        ["instance", "failure_class"])

@dataclass
class InstanceFeatures:
    instance:               str
    disk_fill_rate:         float
    disk_exhaustion_secs:   float
    replication_lag_secs:   float
    lag_acceleration:       float
    buffer_hit_ratio:       float
    active_connections_pct: float
    deadlocks_per_hour:     float
    ple_trend:              float  # SQL Server: Page Life Expectancy delta

def query_prometheus(promql: str) -> Optional[float]:
    try:
        r = requests.get(f"{PROMETHEUS_URL}/api/v1/query",
                         params={"query": promql}, timeout=5)
        r.raise_for_status()
        result = r.json()["data"]["result"]
        return float(result[0]["value"][1]) if result else None
    except Exception as e:
        logging.warning(f"Prometheus query failed: {promql[:60]} — {e}")
        return None

def collect_features(instance: str) -> Optional[InstanceFeatures]:
    label = f'instance="{instance}"'
    disk_rate     = query_prometheus(f'db:disk_fill_rate_bytes_per_second:avg30m{{{label}}}')
    disk_exhaust  = query_prometheus(f'db:disk_exhaustion_seconds:predicted{{{label}}}')
    rep_lag       = query_prometheus(f'pg_replication_lag{{{label}}}')
    lag_accel     = query_prometheus(f'pg:replication_lag_acceleration:ratio{{{label}}}')
    hit_ratio     = query_prometheus(f'pg_stat_database_blks_hit_ratio{{{label}}}')
    conn_pct      = query_prometheus(
        f'pg_stat_activity_count{{{label}}} / pg_settings_max_connections{{{label}}}')
    deadlocks     = query_prometheus(
        f'rate(pg_stat_database_deadlocks{{{label}}}[1h]) * 3600')
    ple_trend     = query_prometheus(f'sqlserver:ple_trend:delta15m{{{label}}}')

    # Require core signals; impute optional ones
    if any(v is None for v in [disk_rate, disk_exhaust, conn_pct]):
        return None

    return InstanceFeatures(
        instance               = instance,
        disk_fill_rate         = disk_rate       or 0.0,
        disk_exhaustion_secs   = disk_exhaust    or 86400.0,
        replication_lag_secs   = rep_lag         or 0.0,
        lag_acceleration       = lag_accel       or 1.0,
        buffer_hit_ratio       = hit_ratio       or 0.99,
        active_connections_pct = conn_pct        or 0.0,
        deadlocks_per_hour     = deadlocks       or 0.0,
        ple_trend              = ple_trend       or 0.0,
    )

def score_instance(model, features: InstanceFeatures) -> dict:
    X = np.array([[
        features.disk_fill_rate,
        features.disk_exhaustion_secs,
        features.replication_lag_secs,
        features.lag_acceleration,
        features.buffer_hit_ratio,
        features.active_connections_pct,
        features.deadlocks_per_hour,
        features.ple_trend,
    ]])
    proba  = model.predict_proba(X)[0][1]     # P(failure within horizon)
    ttf    = model.predict(X)[0]              # regression head: minutes
    return {"probability": float(proba), "time_to_failure_minutes": float(ttf)}

def run_prediction_loop(instances: list[str]):
    model = joblib.load(MODEL_PATH)
    start_http_server(9200)          # expose metrics for Prometheus scrape
    logging.info("Failure predictor started, exposing metrics on :9200")

    while True:
        for instance in instances:
            features = collect_features(instance)
            if features is None:
                logging.warning(f"Incomplete features for {instance}, skipping cycle")
                continue
            scores = score_instance(model, features)
            failure_score.labels(instance=instance,
                                  failure_class="composite").set(scores["probability"])
            time_to_failure.labels(instance=instance,
                                   failure_class="composite").set(
                                       scores["time_to_failure_minutes"])
            logging.info(f"{instance}: P(failure)={scores['probability']:.3f}  "
                         f"TTF={scores['time_to_failure_minutes']:.1f}m")
        time.sleep(SCORE_INTERVAL)

if __name__ == "__main__":
    import sys
    logging.basicConfig(level=logging.INFO,
                        format="%(asctime)s %(levelname)s %(message)s")
    instances = sys.argv[1:]  # pass instance labels as CLI args
    run_prediction_loop(instances)

    This service runs as a Kubernetes sidecar or a systemd unit alongside each database cluster. The scores it emits are first-class Prometheus metrics — they can trigger Alertmanager rules, feed Grafana dashboards, and most importantly, feed the autonomous action layer.

    Autonomous Remediation Triggers and Decision Logic

    A failure probability score is only useful if it connects to action. The action architecture must be explicit about three parameters for every failure class: the trigger threshold, the confidence window (how many consecutive scoring cycles must exceed the threshold before action fires), and the action class (notify, mitigate, escalate, failover).

    | Failure Class | Trigger Threshold | Confidence Window | Action |

    |---|---|---|---|

    | Disk exhaustion < 2h | P > 0.85 | 3 consecutive cycles | Automated cleanup + alert |

    | Replication lag acceleration | P > 0.70 | 5 cycles | Throttle writes + investigate |

    | Connection pool saturation | P > 0.75 | 3 cycles | pgBouncer pool resize |

    | Memory pressure (PLE collapse) | P > 0.80 | 2 cycles | Query kill + cache flush |

    | AG sync health degraded | P > 0.90 | 1 cycle | Pre-position failover candidate |

    The confidence window prevents action storms from model noise. A single anomalous scoring cycle should not trigger a failover. The combination of threshold and window is your system's risk tolerance encoded as code.

    The Alertmanager configuration that bridges prediction scores to the action layer:

    # alertmanager.yml — predictive failure routing
route:
  receiver: 'null'
  routes:
    - match:
        alertname: PredictedDiskExhaustion
      receiver: autonomous_remediation
      group_wait: 0s
      repeat_interval: 10m

    - match:
        alertname: PredictedReplicationFailure
      receiver: autonomous_remediation
      group_wait: 0s
      repeat_interval: 5m

    - match:
        alertname: PredictedAGFailover
      receiver: dba_escalation
      continue: true   # also fire autonomous prep action
    - match:
        alertname: PredictedAGFailover
      receiver: autonomous_remediation

receivers:
  - name: autonomous_remediation
    webhook_configs:
      - url: 'http://action-executor:8080/remediate'
        send_resolved: true
        max_alerts: 1

  - name: dba_escalation
    pagerduty_configs:
      - service_key: '<PD_KEY>'
        severity: warning
        description: '{{ .CommonAnnotations.summary }}'

    The Prometheus alert rules that consume the prediction metrics:

    PostgreSQL & SQL Server (Prometheus rules — engine-agnostic):

    # /etc/prometheus/rules/predictive_alerts.yml
groups:
  - name: predictive_failure_alerts
    rules:
      - alert: PredictedDiskExhaustion
        expr: >
          db:disk_exhaustion_seconds:predicted < 7200
          AND db:disk_fill_rate_bytes_per_second:avg30m > 0
        for: 3m
        labels:
          severity: critical
          action_class: autonomous
        annotations:
          summary: "Disk exhaustion predicted within 2h on {{ $labels.instance }}"
          runbook: "https://runbooks.internal/db/disk-exhaustion"
          ttf_minutes: "{{ $value | humanizeDuration }}"

      - alert: PredictedReplicationFailure
        expr: >
          db_failure_probability{failure_class="composite"} > 0.75
          AND pg:replication_lag_acceleration:ratio > 2.0
        for: 5m
        labels:
          severity: critical
          action_class: autonomous
        annotations:
          summary: "Replication failure predicted on {{ $labels.instance }}"

      - alert: PredictedAGFailover
        expr: >
          db_failure_probability{failure_class="composite"} > 0.90
          AND on(instance) sqlserver_hadr_replica_health == 0
        for: 1m
        labels:
          severity: critical
          action_class: pre_position_failover
        annotations:
          summary: "AG failover predicted, pre-positioning secondary on {{ $labels.instance }}"

    When the action executor webhook fires, it dispatches to the appropriate remediation handler. For disk pressure, this means querying for reclaimable space (dead tuples, old WAL segments, temporary files) and executing the cleanup autonomously. For replication lag under acceleration, it means identifying the heavy write workload and either throttling it or routing new write connections to a secondary. For an AG in pre-failure state, it means ensuring the target secondary is caught up, its TCP keepalive settings are tuned for fast failover, and the monitoring system is primed for the transition — so that when the failover fires (either autonomously or by operator), it completes in seconds rather than minutes.

    Integrating LLM-Based Diagnosis into the Prediction Loop

    The quantitative prediction layer handles well-characterized failure modes. But production databases fail in novel ways — a new query pattern that stress-tests a previously safe code path, a schema migration that creates a lock chain topology the model has never seen, a cloud provider degrading a specific IOPS tier in a way that does not match historical disk failure signatures. For these cases, integrating an LLM reasoning layer into the prediction loop adds qualitative diagnostic depth.

    The pattern is not to replace the quantitative model with an LLM. It is to escalate to LLM reasoning when the quantitative model fires with moderate confidence (P between 0.50 and 0.75) and the signals are ambiguous. At this confidence range, the human DBA would normally investigate manually. The autonomous system instead queries an LLM with the full anomalous signal context and asks it to reason about likely failure class, estimated time horizon, and recommended action.
