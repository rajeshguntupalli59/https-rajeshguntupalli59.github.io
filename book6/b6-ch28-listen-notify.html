Chapter 28: PostgreSQL LISTEN/NOTIFY for Event-Driven AI

PostgreSQL's LISTEN/NOTIFY mechanism has existed since version 6.4, quietly doing useful work for developers who needed lightweight pub/sub without reaching for a separate message broker. In AI-native applications, it turns out to be remarkably well-suited for a problem that keeps coming up: how do you trigger AI processing in reaction to database events without adding operational complexity, introducing consistency gaps between your data store and your queue, or polling tables in a loop? This chapter covers how to build event-driven AI pipelines using LISTEN/NOTIFY as the signal layer — connecting PostgreSQL triggers to Python consumers that invoke LLMs, update embeddings, route agent tasks, and fan out notifications to downstream systems. You'll see how this integrates with FastAPI, Celery, Redis, and the Anthropic and OpenAI SDKs, and how to operate these pipelines safely in production.

---

28.1 Why LISTEN/NOTIFY Belongs in Your AI Stack

Before building anything, it's worth understanding what LISTEN/NOTIFY actually is and what it isn't.

What it is: A lightweight, in-process pub/sub system built into PostgreSQL. A session issues NOTIFY channel_name, 'payload' and every session currently listening on that channel receives the payload asynchronously. Notifications are transactional — a NOTIFY issued inside a transaction isn't delivered until that transaction commits. If the transaction rolls back, the notification is never sent.

What it isn't: A durable message queue. If no listener is connected when a NOTIFY fires, the notification is lost. There is no persistence, no replay, no dead-letter queue. The payload is limited to 8,000 bytes.

For AI applications, this combination — transactional delivery, zero persistence, low latency — makes LISTEN/NOTIFY ideal as a *signal layer* rather than a *delivery layer*. The pattern is:

1. Data changes in PostgreSQL (new document, updated record, user message)

2. A trigger fires NOTIFY with a lightweight payload (usually just a row ID)

3. A Python consumer receives the notification and fetches the full record

4. The consumer invokes an LLM or embedding model

5. Results are written back to PostgreSQL

The notification is the trigger. The database is the source of truth. The consumer is stateless. If the consumer crashes, you need a recovery path (we'll cover that), but the data itself is safe.

Why not just use Celery with a broker? You can, and for heavy workloads you should — and we'll show how they compose. But adding Redis or RabbitMQ as a required dependency means you now have infrastructure to manage, a potential consistency gap (the row was inserted but was the task enqueued?), and operational overhead. For many AI features — embedding new content as it's inserted, routing agent events, invalidating LLM caches — LISTEN/NOTIFY is sufficient and dramatically simpler. The transactional guarantee is the key: you cannot accidentally queue a task for a row that doesn't exist yet.

28.2 Schema Design for Event-Driven AI Pipelines

Good schema design here means structuring your tables so that triggers can extract meaningful signals and consumers can efficiently fetch what they need.

Now the trigger infrastructure. The key design decision: triggers write to ai_events *and* issue NOTIFY. The event log gives you durability and recovery; the notification gives you low-latency delivery.

-- Generic notification function — used by multiple triggers
CREATE OR REPLACE FUNCTION notify_ai_event()
RETURNS TRIGGER AS $$
DECLARE
    v_channel   TEXT;
    v_event_type TEXT;
    v_entity_id  UUID;
    v_tenant_id  UUID;
    v_payload    JSONB;
    v_event_id   BIGINT;
    v_notify_payload TEXT;
BEGIN
    -- Determine event type
    v_event_type := TG_OP;  -- INSERT, UPDATE, DELETE

    -- Extract entity info based on table
    IF TG_TABLE_NAME = 'documents' THEN
        v_channel   := 'ai_documents';
        v_entity_id := NEW.id;
        v_tenant_id := NEW.tenant_id;
        v_payload   := jsonb_build_object(
            'operation', TG_OP,
            'id', NEW.id,
            'tenant_id', NEW.tenant_id,
            'status', NEW.status,
            'has_body', length(NEW.body) > 0
        );

        -- Only notify on meaningful state changes
        IF TG_OP = 'UPDATE' AND OLD.body = NEW.body AND OLD.status = NEW.status THEN
            RETURN NEW;
        END IF;

    ELSIF TG_TABLE_NAME = 'messages' THEN
        v_channel   := 'ai_messages';
        v_entity_id := NEW.id;
        v_tenant_id := NEW.tenant_id;
        v_payload   := jsonb_build_object(
            'operation', TG_OP,
            'id', NEW.id,
            'conversation_id', NEW.conversation_id,
            'tenant_id', NEW.tenant_id,
            'role', NEW.role
        );

        -- Only process user messages for LLM response generation
        IF NEW.role != 'user' THEN
            RETURN NEW;
        END IF;

    ELSIF TG_TABLE_NAME = 'agent_tasks' THEN
        v_channel   := 'ai_agent_tasks';
        v_entity_id := NEW.id;
        v_tenant_id := NEW.tenant_id;
        v_payload   := jsonb_build_object(
            'operation', TG_OP,
            'id', NEW.id,
            'agent_id', NEW.agent_id,
            'task_type', NEW.task_type,
            'status', NEW.status,
            'priority', NEW.priority
        );

        IF TG_OP = 'UPDATE' AND NEW.status != 'queued' THEN
            RETURN NEW;
        END IF;
    END IF;

    -- Write durable event record first
    INSERT INTO ai_events (
        event_type, entity_type, entity_id, tenant_id,
        payload, channel
    ) VALUES (
        v_event_type, TG_TABLE_NAME, v_entity_id, v_tenant_id,
        v_payload, v_channel
    ) RETURNING id INTO v_event_id;

    -- Notify payload is minimal — just enough to fetch the record
    v_notify_payload := jsonb_build_object(
        'event_id', v_event_id,
        'entity_id', v_entity_id,
        'event_type', v_event_type,
        'table', TG_TABLE_NAME
    )::text;

    -- Enforce 8000 byte limit
    IF length(v_notify_payload) > 8000 THEN
        v_notify_payload := jsonb_build_object(
            'event_id', v_event_id,
            'entity_id', v_entity_id
        )::text;
    END IF;

    PERFORM pg_notify(v_channel, v_notify_payload);

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Attach to tables
CREATE TRIGGER trg_documents_ai_notify
    AFTER INSERT OR UPDATE ON documents
    FOR EACH ROW EXECUTE FUNCTION notify_ai_event();

CREATE TRIGGER trg_messages_ai_notify
    AFTER INSERT ON messages
    FOR EACH ROW EXECUTE FUNCTION notify_ai_event();

CREATE TRIGGER trg_agent_tasks_ai_notify
    AFTER INSERT OR UPDATE ON agent_tasks
    FOR EACH ROW EXECUTE FUNCTION notify_ai_event();

-- Recovery function: find events that were notified but never processed
CREATE OR REPLACE FUNCTION get_unprocessed_events(
    p_channel TEXT,
    p_older_than_seconds INT DEFAULT 30,
    p_limit INT DEFAULT 100
)
RETURNS TABLE (
    event_id    BIGINT,
    entity_id   UUID,
    entity_type TEXT,
    payload     JSONB,
    notified_at TIMESTAMPTZ,
    retry_count INT
) AS $$
BEGIN
    RETURN QUERY
    SELECT
        e.id,
        e.entity_id,
        e.entity_type,
        e.payload,
        e.notified_at,
        e.retry_count
    FROM ai_events e
    WHERE e.channel = p_channel
      AND e.status IN ('notified', 'failed')
      AND e.notified_at < now() - (p_older_than_seconds || ' seconds')::interval
      AND e.retry_count < 5
    ORDER BY e.notified_at ASC
    LIMIT p_limit;
END;
$$ LANGUAGE plpgsql;

28.3 Python Consumers with asyncpg

The Python side uses asyncpg for async LISTEN/NOTIFY support. asyncio event loop, connection pooling, and the AI SDK calls all compose cleanly.

# listener.py — core LISTEN/NOTIFY consumer for AI events
import asyncio
import asyncpg
import json
import logging
import signal
import time
import uuid
from contextlib import asynccontextmanager
from dataclasses import dataclass, field
from typing import Any, Callable, Coroutine

import anthropic
import openai
from openai import AsyncOpenAI

logger = logging.getLogger(__name__)

DATABASE_URL = "postgresql://app_user:secret@localhost:5432/aiapp"
PROCESSOR_ID = f"worker-{uuid.uuid4().hex[:8]}"


@dataclass
class NotificationEvent:
    channel: str
    payload: dict
    event_id: int
    entity_id: str
    entity_type: str
    event_type: str


class AIEventConsumer:
    """
    Listens on PostgreSQL NOTIFY channels and dispatches to AI processors.
    Handles reconnection, recovery, and graceful shutdown.
    """

    def __init__(self, dsn: str, channels: list[str]):
        self.dsn = dsn
        self.channels = channels
        self.conn: asyncpg.Connection | None = None
        self.pool: asyncpg.Pool | None = None
        self.handlers: dict[str, Callable] = {}
        self.running = False
        self.shutdown_event = asyncio.Event()
        self._notification_queue: asyncio.Queue = asyncio.Queue(maxsize=1000)

    def register_handler(self, channel: str, handler: Callable) -> None:
        self.handlers[channel] = handler

    async def start(self) -> None:
        self.running = True
        self.pool = await asyncpg.create_pool(
            self.dsn,
            min_size=2,
            max_size=10,
            command_timeout=60,
        )

        # Start listener connection (separate from pool — LISTEN requires dedicated connection)
        await self._connect_listener()

        # Start worker tasks
        tasks = [
            asyncio.create_task(self._listen_loop()),
            asyncio.create_task(self._dispatch_loop()),
            asyncio.create_task(self._recovery_loop()),
        ]

        logger.info(f"Consumer {PROCESSOR_ID} started, listening on: {self.channels}")

        await asyncio.gather(*tasks)

    async def _connect_listener(self) -> None:
        """Establish dedicated connection for LISTEN commands."""
        self.conn = await asyncpg.connect(self.dsn)

        def _on_notification(conn, pid, channel, payload):
            """Called by asyncpg from the connection's receive loop."""
            try:
                parsed = json.loads(payload)
                event = NotificationEvent(
                    channel=channel,
                    payload=parsed,
                    event_id=parsed.get("event_id", 0),
                    entity_id=parsed.get("entity_id", ""),
                    entity_type=parsed.get("table", ""),
                    event_type=parsed.get("event_type", ""),
                )
                # Non-blocking put — drop if queue is full (backpressure)
                try:
                    self._notification_queue.put_nowait(event)
                except asyncio.QueueFull:
                    logger.warning(
                        f"Notification queue full, dropping event {event.event_id} "
                        f"on channel {channel}. Recovery loop will catch it."
                    )
            except json.JSONDecodeError as e:
                logger.error(f"Failed to parse notification payload: {payload!r}: {e}")

        self.conn.add_listener(None, _on_notification)  # None = all channels

        for channel in self.channels:
            await self.conn.execute(f"LISTEN {channel}")
            logger.debug(f"Listening on channel: {channel}")

    async def _listen_loop(self) -> None:
        """Keep listener connection alive with reconnection logic."""
        while self.running:
            try:
                if self.conn is None or self.conn.is_closed():
                    logger.warning("Listener connection lost, reconnecting...")
                    await asyncio.sleep(2)
                    await self._connect_listener()

                # Keep the connection alive — asyncpg delivers notifications
                # through the connection's internal receive loop; we just need
                # to hold the event loop open
                await asyncio.sleep(1)

            except asyncpg.PostgresConnectionStatusError:
                logger.exception("Connection error in listen loop")
                self.conn = None
                await asyncio.sleep(5)
            except asyncio.CancelledError:
                break
            except Exception:
                logger.exception("Unexpected error in listen loop")
                await asyncio.sleep(5)

    async def _dispatch_loop(self) -> None:
        """Pull from notification queue and invoke handlers."""
        while self.running:
            try:
                event = await asyncio.wait_for(
                    self._notification_queue.get(),
                    timeout=1.0,
                )
                handler = self.handlers.get(event.channel)
                if handler is None:
                    logger.warning(f"No handler for channel: {event.channel}")
                    continue

                # Claim the event in the database
                claimed = await self._claim_event(event.event_id)
                if not claimed:
                    logger.debug(f"Event {event.event_id} already claimed by another worker")
                    continue

                # Dispatch without blocking the queue
                asyncio.create_task(
                    self._run_handler(handler, event)
                )

            except asyncio.TimeoutError:
                continue
            except asyncio.CancelledError:
                break
            except Exception:
                logger.exception("Error in dispatch loop")

    async def _claim_event(self, event_id: int) -> bool:
        """
        Atomically claim an event for this processor.
        Returns True if successfully claimed, False if already taken.
        """
        async with self.pool.acquire() as conn:
            result = await conn.fetchval(
                """
                UPDATE ai_events
                SET status = 'processing',
                    processor_id = $1
                WHERE id = $2
                  AND status = 'notified'
                RETURNING id
                """,
                PROCESSOR_ID,
                event_id,
            )
            return result is not None

    async def _run_handler(self, handler: Callable, event: NotificationEvent) -> None:
        """Execute handler and record result."""
        start = time.monotonic()
        try:
            result = await handler(event, self.pool)
            elapsed = time.monotonic() - start

            async with self.pool.acquire() as conn:
                await conn.execute(
                    """
                    UPDATE ai_events
                    SET status = 'completed',
                        processed_at = now(),
                        result = $1
                    WHERE id = $2
                    """,
                    json.dumps({"elapsed_ms": int(elapsed * 1000), "result": result}),
                    event.event_id,
                )
        except Exception as exc:
            logger.exception(f"Handler failed for event {event.event_id}")
            async with self.pool.acquire() as conn:
                await conn.execute(
                    """
                    UPDATE ai_events
                    SET status = 'failed',
                        error = $1,
                        retry_count = retry_count + 1
                    WHERE id = $2
                    """,
                    str(exc),
                    event.event_id,
                )

    async def _recovery_loop(self) -> None:
        """
        Periodically scan for unprocessed events — handles the case where
        notifications were dropped (queue full, consumer was down, etc.)
        """
        while self.running:
            try:
                await asyncio.sleep(30)  # Check every 30 seconds

                async with self.pool.acquire() as conn:
                    for channel in self.channels:
                        rows = await conn.fetch(
                            "SELECT * FROM get_unprocessed_events($1, 60, 50)",
                            channel,
                        )
                        if rows:
                            logger.info(
                                f"Recovery: found {len(rows)} unprocessed events on {channel}"
                            )
                        for row in rows:
                            event = NotificationEvent(
                                channel=channel,
                                payload=dict(row["payload"]),
                                event_id=row["event_id"],
                                entity_id=str(row["entity_id"]),
                                entity_type=row["entity_type"],
                                event_type=row["payload"].get("operation", ""),
                            )
                            await self._notification_queue.put(event)

            except asyncio.CancelledError:
                break
            except Exception:
                logger.exception("Error in recovery loop")

    async def stop(self) -> None:
        self.running = False
        if self.conn:
            await self.conn.close()
        if self.pool:
            await self.pool.close()
        logger.info(f"Consumer {PROCESSOR_ID} stopped")
