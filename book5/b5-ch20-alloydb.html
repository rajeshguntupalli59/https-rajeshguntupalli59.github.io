Chapter 20: AlloyDB: PostgreSQL-Compatible with ML Built In

Google's AlloyDB for PostgreSQL is not simply a managed PostgreSQL service with a different logo. It is a ground-up reimagining of the PostgreSQL storage engine — wire-compatible with PostgreSQL 14 and 15, but architecturally closer to Aurora in its separation of compute and storage, and unlike any other managed PostgreSQL offering in its native integration of machine learning inference directly at the query layer. For production DBAs who have spent years tuning autovacuum, managing WAL pressure, and hand-crafting connection pools, AlloyDB represents a shift in what the database engine itself takes responsibility for. This chapter covers how AlloyDB's columnar engine, memory-optimized storage, and ML inference extensions work in practice, how to provision and connect clusters, how to migrate existing PostgreSQL workloads, and where AlloyDB genuinely outperforms Cloud SQL for PostgreSQL and where it falls short in ways that will surprise you.

---

20.1 Architecture: What AlloyDB Actually Is Under the Hood

If you are coming from self-managed PostgreSQL, you understand the storage hierarchy: the buffer pool in shared memory, heap files on disk, WAL on a separate spindle if you are careful, and the constant negotiation between autovacuum and your OLTP workload. AlloyDB dismantles this hierarchy and reassembles it differently.

Disaggregated Storage

AlloyDB separates the primary instance compute from a distributed storage layer, similar to Aurora's approach. The storage layer — Google calls it the *Log Store* — receives and durably replicates WAL records across six replicas in three zones before acknowledging writes. The primary instance never writes heap files directly to local disk. Instead, background storage nodes apply WAL to materialized heap pages asynchronously. This means your primary is doing dramatically less I/O work than a traditional PostgreSQL primary, and crash recovery does not require replaying hours of WAL — the storage layer maintains consistent page state independently.

What this means for you operationally:

  • → **Checkpoint pressure disappears.** In self-managed PostgreSQL, tuning `checkpoint_completion_target`, `max_wal_size`, and `checkpoint_timeout` is a ritual. In AlloyDB, checkpoints still exist in the PostgreSQL code path, but their I/O impact on the primary is negligible because no dirty pages are being flushed to local disk in the traditional sense.
  • → **Read replicas are fast to provision.** A read replica in AlloyDB attaches to the same storage layer — it does not stream WAL and rebuild a base backup. A new replica is queryable in seconds to minutes, not hours.
  • → **Replication lag behaves differently.** Because replicas read from the shared storage layer rather than applying streamed WAL, replication lag is reported differently than `pg_stat_replication.replay_lag`. AlloyDB exposes this through its own metrics, and you need to retrain your intuition about what "lag" means.

    • The Ultra-Fast Shared Memory Cache

    AlloyDB introduces a tiered memory architecture: beyond the standard PostgreSQL shared_buffers, it maintains a second layer called the *ultra-fast cache* built on local NVMe SSD attached to each instance. This cache is 2× persistent — it survives instance restarts because it is backed by NVMe rather than DRAM. When a page is evicted from shared_buffers, it goes to this NVMe cache before it goes back to the distributed storage layer. The practical implication: your read-heavy workloads that overflow shared_buffers do not pay remote storage latency on every cache miss. They pay NVMe latency, which is roughly 100–200 microseconds rather than 1–4 milliseconds for remote storage.

    In sizing terms, shared_buffers in AlloyDB is set automatically based on instance size and is not a parameter you directly tune in the same way. On an instance with 64 GB RAM, AlloyDB allocates roughly 40–45 GB to shared_buffers, with the NVMe cache providing another layer below it.

    The Columnar Engine

    AlloyDB includes an optional columnar engine — a PostgreSQL foreign data wrapper-style extension that maintains a compressed, columnar representation of tables or table partitions in memory. When a query contains analytical predicates (range scans, aggregations, GROUP BY) that the query planner identifies as benefiting from columnar access, it routes those operations through the columnar engine transparently. You write standard SQL; the planner decides.

    This is meaningful because it means you can run mixed OLTP and analytical workloads on the same AlloyDB cluster without physically separating your data into a separate OLAP store. The columnar engine is not a replacement for BigQuery or a data warehouse at scale — but for the DBA who has been maintaining a separate reporting replica just to run month-end aggregates without killing the OLTP primary, it is a serious operational simplification.

    Enabling the columnar engine on a table:

    -- Enable the AlloyDB columnar engine extension
-- This is typically already installed in AlloyDB clusters
SHOW alloydb_enable_columnar_engine;

-- Enable columnar engine for a specific table
ALTER TABLE orders SET (columnar_engine = 'on');

-- You can also enable it for specific columns or let AlloyDB auto-select
-- Check what the auto-columnarization has selected
SELECT *
FROM google_columnar_engine_recommended_tables()
ORDER BY estimated_memory_mb DESC;

-- View current columnar engine population status
SELECT table_name,
       status,
       rows_count,
       size_in_bytes
FROM google_columnar_engine_tables;

    The google_columnar_engine_recommended_tables() function is one of AlloyDB's genuinely useful operational functions — it analyzes your query workload and recommends which tables will benefit most from columnarization, with estimated memory requirements. Treat this output as advisory; validate with EXPLAIN (ANALYZE, BUFFERS) before and after.

    20.2 Provisioning AlloyDB Clusters and Instances

    AlloyDB uses a two-tier model: a *cluster* and *instances within that cluster*. The cluster defines the regional boundary, the VPC network, and the storage configuration. Instances are the PostgreSQL compute endpoints — a primary instance and optionally one or more read pool instances.

    Provisioning with gcloud

    # Set your project and region
gcloud config set project your-gcp-project-id
gcloud config set compute/region us-central1

# Create an AlloyDB cluster
# This provisions the storage layer — no compute yet
gcloud alloydb clusters create prod-alloydb-cluster \
  --region=us-central1 \
  --network=projects/your-gcp-project-id/global/networks/prod-vpc \
  --password=your-initial-password \
  --database-version=POSTGRES_15

# Create the primary instance
# machine-type options: db-n1-standard-2, db-n1-highmem-8, db-n1-highmem-32, etc.
gcloud alloydb instances create prod-primary \
  --cluster=prod-alloydb-cluster \
  --region=us-central1 \
  --instance-type=PRIMARY \
  --cpu-count=8 \
  --no-assign-inbound-public-ip

# Create a read pool instance (for read scaling and analytics)
# Read pools in AlloyDB are groups of nodes — min 2 for HA
gcloud alloydb instances create prod-read-pool \
  --cluster=prod-alloydb-cluster \
  --region=us-central1 \
  --instance-type=READ_POOL \
  --cpu-count=8 \
  --read-pool-node-count=2 \
  --no-assign-inbound-public-ip

# Verify both instances are READY
gcloud alloydb instances list \
  --cluster=prod-alloydb-cluster \
  --region=us-central1

    Terraform is the better choice for production clusters where you need reproducible infrastructure. The AlloyDB Terraform provider reached GA in 2024 and covers the full feature set:

  description = "Primary instance private IP — use this for read/write connections"

  description = "Read pool private IP — use this for analytics and read-heavy workloads"

    Connection Topology

    AlloyDB does not expose a CNAME-based endpoint the way Aurora does. Connection is always by private IP within your VPC. There is no AlloyDB equivalent of RDS Proxy built into the service — you bring your own connection pooler, and PgBouncer running on a GCE instance or Cloud Run is the standard pattern. If you are coming from Aurora, this is a gap you will feel immediately. The AlloyDB Auth Proxy handles IAM authentication and SSL termination and is the recommended client-side component:

    # Download and run the AlloyDB Auth Proxy
# This is the equivalent of Cloud SQL Auth Proxy — same pattern, different binary

wget https://storage.googleapis.com/alloydb-auth-proxy/v1.x.x/alloydb-auth-proxy.linux.amd64 \
  -O alloydb-auth-proxy
chmod +x alloydb-auth-proxy

# Run the proxy — it listens on localhost:5432 and handles IAM auth
./alloydb-auth-proxy \
  "projects/your-project/locations/us-central1/clusters/prod-alloydb-cluster/instances/prod-primary" \
  --port=5432 \
  --credentials-file=/path/to/service-account.json

# Application connection string then becomes:
# host=127.0.0.1 port=5432 dbname=mydb user=iam_service_account sslmode=disable
# SSL is handled by the proxy — disable in the client connection string

    The AlloyDB Auth Proxy terminates SSL and handles token refresh automatically, which means your application connection strings do not embed database passwords. Instead, the service account identity determines access. This is the same pattern you used with Cloud SQL Auth Proxy if you have worked with Cloud SQL before.

    20.3 IAM Authentication and Access Control

    In self-managed PostgreSQL, pg_hba.conf is your access control boundary. You define which hosts can connect, which users, and which authentication method — md5, scram-sha-256, cert, ldap. In AlloyDB, the network boundary is your VPC (enforced at the infrastructure level), and authentication has two modes: native PostgreSQL password authentication and IAM-based authentication.

    IAM authentication in AlloyDB works similarly to Cloud SQL's IAM auth. A Google identity — a service account or a user account — is mapped to a PostgreSQL role. The IAM-authenticated connection uses a short-lived OAuth2 token as the password, which the AlloyDB Auth Proxy or the alloydb_pg_iam_authn extension handles.

    Setting Up IAM Authentication

    # Grant IAM user or service account the AlloyDB Client role
# This is the network-level permission — required before database-level grants
gcloud projects add-iam-policy-binding your-gcp-project-id \
  --member="serviceAccount:app-service-account@your-project.iam.gserviceaccount.com" \
  --role="roles/alloydb.client"

# For IAM database authentication specifically:
gcloud projects add-iam-policy-binding your-gcp-project-id \
  --member="serviceAccount:app-service-account@your-project.iam.gserviceaccount.com" \
  --role="roles/alloydb.databaseUser"
    -- Inside the AlloyDB instance, create the IAM-mapped PostgreSQL role
-- The username format for service accounts: strip @project.iam.gserviceaccount.com
-- For user accounts: the full email address

-- Create the PostgreSQL role corresponding to the IAM service account
CREATE USER "app-service-account@your-project.iam" WITH LOGIN;

-- Grant appropriate database permissions
GRANT CONNECT ON DATABASE production TO "app-service-account@your-project.iam";
GRANT USAGE ON SCHEMA public TO "app-service-account@your-project.iam";
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public
  TO "app-service-account@your-project.iam";

-- For future tables:
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES
  TO "app-service-account@your-project.iam";

-- Verify the role exists and has the right attributes
SELECT rolname, rolcanlogin, rolsuper, rolcreatedb
FROM pg_roles
WHERE rolname LIKE '%iam%';

    What You Cannot Do That You Could in Self-Managed PostgreSQL

    The pg_hba.conf comparison is instructive here. In self-managed PostgreSQL, you can define rules like:

    # Allow cert auth from specific CIDR, reject everything else
hostssl all all 10.0.0.0/8 cert
host    all all all        reject

    In AlloyDB, your network-level controls are VPC firewall rules and Private Service Connect configuration — you cannot write arbitrary pg_hba.conf entries. AlloyDB manages its own pg_hba.conf internally. You can enable or disable IAM authentication at the cluster level, and you can require SSL (which is on by default), but fine-grained auth method selection per CIDR range or per database is not exposed. This is a real constraint for organizations that use client certificate authentication today.

    Connection from Python Using IAM Token

    import google.auth
import google.auth.transport.requests
import psycopg2
from google.oauth2 import service_account

def get_alloydb_connection_iam(
    host: str,
    database: str,
    iam_user: str,
    service_account_file: str
) -> psycopg2.extensions.connection:
    """
    Connect to AlloyDB using IAM authentication.
    The AlloyDB Auth Proxy must be running, or use direct IP with SSL.
    For proxy connections, host is 127.0.0.1 and no SSL needed at client level.
    """
    credentials = service_account.Credentials.from_service_account_file(
        service_account_file,
        scopes=["https://www.googleapis.com/auth/cloud-platform"]
    )

    # Refresh to get a current token
    auth_request = google.auth.transport.requests.Request()
    credentials.refresh(auth_request)

    # The OAuth2 token is used as the password
    token = credentials.token

    conn = psycopg2.connect(
        host=host,
        port=5432,
        database=database,
        user=iam_user,  # e.g., "app-service-account@your-project.iam"
        password=token,
        sslmode="disable"  # SSL handled by Auth Proxy
    )

    return conn

def run_query_with_retry(host: str, database: str, query: str) -> list:
    """
    Production pattern: IAM tokens expire in 1 hour.
    Use a connection factory that refreshes tokens on reconnect.
    """
    import time

    max_retries = 3
    for attempt in range(max_retries):
        try:
            conn = get_alloydb_connection_iam(
                host=host,
                database=database,
                iam_user="app-service-account@your-project.iam",
                service_account_file="/etc/secrets/sa-key.json"
            )
            with conn.cursor() as cur:
                cur.execute(query)
                results = cur.fetchall()
            conn.close()
            return results

        except psycopg2.OperationalError as e:
            if attempt < max_retries - 1:
                time.sleep(2 ** attempt)  # Exponential backoff
                continue
            raise

    return []

    The token expiry behavior is something that catches people coming from password-based PostgreSQL connections. A long-lived connection established with an IAM token does not get disconnected when the token expires — but any new connection attempt after the token expires will fail. PgBouncer in transaction pooling mode mitigates this because connections are returned to the pool and reestablished regularly, but you need to wire the token refresh into your connection factory, not just your startup code.

    20.4 ML Inference with `google_ml_integration`

    This is where AlloyDB genuinely has no equivalent in Aurora, Azure SQL, or Cloud SQL for PostgreSQL. AlloyDB includes a built-in extension called google_ml_integration that allows you to call Vertex AI hosted models directly from SQL, with the model call executing as a PostgreSQL function. The ML inference runs server-side, in the database tier, without your application code needing to orchestrate a round trip to Vertex AI.

    The practical use cases for DBAs are:

  • → **Text embedding generation** at insert time, stored as `vector` columns (AlloyDB natively supports `pgvector`)
  • → **Classification and scoring** within a query, avoiding application-side ML overhead
  • → **Semantic search** using vector similarity within standard SQL queries

    • Enabling the ML Integration Extension

    -- Enable the extension (requires superuser or alloydbsuperuser role)
CREATE EXTENSION IF NOT EXISTS google_ml_integration CASCADE;

-- Verify installation
SELECT extname, extversion
FROM pg_extension
WHERE extname IN ('google_ml_integration', 'vector');

-- Grant usage to application roles
GRANT EXECUTE ON FUNCTION embedding(text, text) TO app_role;
GRANT EXECUTE ON FUNCTION ml_predict_row(text, jsonb) TO app_role;
    # Before calling Vertex AI from AlloyDB, the AlloyDB service account
# needs permission to invoke the Vertex AI API
# Get your AlloyDB service account
gcloud alloydb clusters describe prod-alloydb-cluster \
  --region=us-central1 \
  --format="value(serviceAccountEmailAddress)"

# Grant the Vertex AI User role to the AlloyDB service account
gcloud projects add-iam-policy-binding your-gcp-project-id \
  --member="serviceAccount:alloydb-sa@your-project.iam.gserviceaccount.com" \
  --role="roles/aiplatform.user"

    Generating Embeddings in SQL

    -- Create a table with a vector column for semantic search
CREATE TABLE product_catalog (
    product_id    SERIAL PRIMARY KEY,
    product_name  TEXT NOT NULL,
    description   TEXT,
    category      TEXT,
    price         NUMERIC(10,2),
    -- 768-dimension vector for text-embedding-004 model
    description_embedding VECTOR(768)
);

-- Create an IVFFlat index for approximate nearest neighbor search
-- lists = sqrt(row_count) is a common starting point
CREATE INDEX idx_product_embedding
ON product_catalog
USING ivfflat (description_embedding vector_cosine_ops)
WITH (lists = 100);

-- Generate embeddings using Vertex AI text-embedding-004
-- The embedding() function calls Vertex AI and returns a VECTOR
UPDATE product_catalog
SET description_embedding = embedding(
    'textembedding-gecko@003',   -- Vertex AI model ID
    description                  -- The text to embed
)
WHERE description_embedding IS NULL
  AND description IS NOT NULL;

-- Semantic search: find products similar to a natural language query
-- This entire operation happens inside PostgreSQL — no app-side ML call
SELECT
    product_id,
    product_name,
    description,
    price,
    1 - (description_embedding <=> embedding(
        'textembedding-gecko@003',
        'lightweight running shoes for trail conditions'
    )) AS cosine_similarity
FROM product_catalog
WHERE category = 'footwear'
ORDER BY description_embedding <=> embedding(
    'textembedding-gecko@003',
    'lightweight running shoes for trail conditions'
)
LIMIT 10;

-- Verify the index is being used (set_limit controls IVFFlat probe count)
SET ivfflat.probes = 10;
EXPLAIN (ANALYZE, BUFFERS)
SELECT product_id, product_name,
       description_embedding <=> embedding(
           'textembedding-gecko@003',
           'trail running footwear'
       ) AS distance
FROM product_catalog
ORDER BY description_embedding <=> embedding(
    'textembedding-gecko@003',
    'trail running footwear'
)
LIMIT 5;
