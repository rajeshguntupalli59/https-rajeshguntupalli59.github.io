Chapter 12: Parameter Groups, Option Groups, and Tuning on RDS

If you've spent years editing postgresql.conf or SQL Server's sp_configure to squeeze performance out of a physical server, RDS parameter groups will feel simultaneously familiar and frustrating. The concept is the same — a named collection of engine configuration values applied to one or more instances — but the execution lives entirely in AWS's control plane, enforced through API calls rather than file edits, and constrained by what AWS chooses to expose. This chapter walks through how parameter groups and option groups work across RDS MySQL, PostgreSQL, and SQL Server, how to build a production tuning workflow around them, and where the managed service boundary forces you to rethink habits you've built over a decade.

---

12.1 How Parameter Groups Work: The RDS Configuration Model

On a self-managed PostgreSQL instance, you tune the database by editing postgresql.conf, reloading or restarting the server, and confirming changes with SHOW work_mem or pg_file_settings. You have full access to every knob the engine exposes. On RDS, AWS inserts an abstraction layer between you and those knobs.

A parameter group in RDS is a named container of engine parameters. Every RDS instance is associated with exactly one parameter group. When you create an instance without specifying one, AWS assigns the default parameter group for that engine version — and the default group is read-only. You cannot modify it. This is the first thing that trips up experienced DBAs: the moment you need to change any parameter, you must create a custom parameter group, modify it, and associate it with the instance.

Parameter groups come in two types:

  • → **DB Parameter Groups** — applied to a single RDS instance
  • → **DB Cluster Parameter Groups** — applied to an Aurora cluster, affecting all instances in that cluster

    • For Aurora specifically, you manage two separate parameter groups simultaneously: one at the cluster level (for shared settings like binlog_format in MySQL or rds.force_ssl in PostgreSQL) and one at the instance level (for memory settings like work_mem or innodb_buffer_pool_size). This split catches Aurora newcomers off guard when a parameter change doesn't take effect because they edited the wrong group.

    Parameters fall into two apply-types:

  • → **Static** — require a database restart to take effect
  • → **Dynamic** — can be applied immediately without a restart

    • In PostgreSQL terms, static parameters correspond to those that require a full server restart in postgresql.conf (like max_connections or shared_buffers). Dynamic parameters correspond to those that reload with pg_reload_conf(). The distinction matters enormously in production: changing max_connections on an RDS PostgreSQL instance requires a reboot, which means a brief outage on Single-AZ instances or a failover on Multi-AZ.

    AWS exposes roughly 200–350 parameters per engine, depending on version. Many parameters that exist in the upstream engine are locked or absent. You cannot set fsync = off on RDS PostgreSQL regardless of what you put in a parameter group — RDS overrides it. You cannot configure huge_pages at the OS level. You get the knobs AWS decides you should have.

    You can see which parameters are modifiable for a given engine version with the AWS CLI:

    # List all modifiable parameters for a PostgreSQL 16 parameter group
aws rds describe-engine-default-parameters \
  --db-parameter-group-family postgres16 \
  --query "EngineDefaults.Parameters[?IsModifiable==\`true\`].{Name:ParameterName,Type:ApplyType,Default:ParameterValue}" \
  --output table \
  --region us-east-1

    This command returns the full parameter catalog for that engine family with current defaults, apply types, and allowed values. Run this before you start tuning — it tells you exactly what you're working with.

    The parameter group family name ties a group to a specific engine and major version. A group created for postgres15 cannot be applied to a postgres16 instance. When you upgrade a major version, you must either create a new parameter group for the target version or verify that your existing customizations are compatible with the new family. AWS does not migrate your tuning automatically.

    12.2 Creating and Managing Parameter Groups

    The production workflow for parameter groups follows a pattern: create, customize, test, associate. You should never modify a parameter group that is already associated with a running production instance without understanding the apply-type of every parameter you're changing.

    Here is the complete workflow for creating a custom PostgreSQL 16 parameter group using the AWS CLI:

    # Create the custom parameter group
aws rds create-db-parameter-group \
  --db-parameter-group-name prod-postgres16-oltp \
  --db-parameter-group-family postgres16 \
  --description "Production OLTP tuning for PostgreSQL 16" \
  --region us-east-1

# Apply a batch of parameters
aws rds modify-db-parameter-group \
  --db-parameter-group-name prod-postgres16-oltp \
  --parameters \
    "ParameterName=work_mem,ParameterValue=65536,ApplyMethod=immediate" \
    "ParameterName=maintenance_work_mem,ParameterValue=2097152,ApplyMethod=immediate" \
    "ParameterName=checkpoint_completion_target,ParameterValue=0.9,ApplyMethod=immediate" \
    "ParameterName=effective_cache_size,ParameterValue=12582912,ApplyMethod=immediate" \
    "ParameterName=random_page_cost,ParameterValue=1.1,ApplyMethod=immediate" \
    "ParameterName=log_min_duration_statement,ParameterValue=1000,ApplyMethod=immediate" \
    "ParameterName=shared_preload_libraries,ParameterValue='pg_stat_statements,auto_explain',ApplyMethod=pending-reboot" \
  --region us-east-1

# Associate the parameter group with an existing instance
aws rds modify-db-instance \
  --db-instance-identifier prod-postgres-primary \
  --db-parameter-group-name prod-postgres16-oltp \
  --apply-immediately \
  --region us-east-1

# Verify the parameter group is in sync (not pending-reboot)
aws rds describe-db-instances \
  --db-instance-identifier prod-postgres-primary \
  --query "DBInstances[0].DBParameterGroups" \
  --region us-east-1

    That last command returns either in-sync or pending-reboot. If you see pending-reboot, the static parameters you changed will not be active until the next restart. In production, this often means the instance is running with a mix of old static values and new dynamic values — a state that can last for weeks if nobody schedules the reboot.

    The equivalent in Terraform, which is where production parameter group management belongs for teams running infrastructure as code:

    resource "aws_db_parameter_group" "prod_postgres16_oltp" {
  name        = "prod-postgres16-oltp"
  family      = "postgres16"
  description = "Production OLTP tuning for PostgreSQL 16"

  parameter {
    name         = "work_mem"
    value        = "65536"
    apply_method = "immediate"
  }

  parameter {
    name         = "maintenance_work_mem"
    value        = "2097152"
    apply_method = "immediate"
  }

  parameter {
    name         = "checkpoint_completion_target"
    value        = "0.9"
    apply_method = "immediate"
  }

  parameter {
    name         = "effective_cache_size"
    value        = "12582912"
    apply_method = "immediate"
  }

  parameter {
    name         = "random_page_cost"
    value        = "1.1"
    apply_method = "immediate"
  }

  parameter {
    name         = "log_min_duration_statement"
    value        = "1000"
    apply_method = "immediate"
  }

  parameter {
    name         = "shared_preload_libraries"
    value        = "pg_stat_statements,auto_explain"
    apply_method = "pending-reboot"
  }

  tags = {
    Environment = "production"
    Engine      = "postgres16"
    Workload    = "oltp"
  }

  lifecycle {
    create_before_destroy = true
  }
}

resource "aws_db_instance" "prod_postgres_primary" {
  identifier             = "prod-postgres-primary"
  engine                 = "postgres"
  engine_version         = "16.2"
  instance_class         = "db.r7g.4xlarge"
  parameter_group_name   = aws_db_parameter_group.prod_postgres16_oltp.name

  # Additional configuration omitted for brevity
}

    The lifecycle { create_before_destroy = true } block matters here. When Terraform needs to replace a parameter group (because you changed its family or name), it will create the new group before destroying the old one, avoiding a gap where the instance references a nonexistent group.

    One operational discipline that saves significant pain: treat parameter groups as immutable artifacts. Instead of modifying prod-postgres16-oltp in place, create prod-postgres16-oltp-v2, validate it on a staging instance, then cut production over. In-place modification of a live parameter group makes rollback much harder, and AWS has no native parameter group version history. If you modify a group and the change causes problems, you need to know what the previous values were — which means you need your own version control.

    12.3 Option Groups: SQL Server and MySQL Extensions

    PostgreSQL on RDS handles engine extensions through parameter groups (specifically shared_preload_libraries) and the CREATE EXTENSION SQL command. MySQL and SQL Server on RDS use an additional construct: option groups.

    An option group is a collection of engine-specific features that go beyond configuration parameters. For SQL Server, this includes features like Transparent Data Encryption (TDE), SQL Server Audit, and SQL Server Agent. For MySQL, it includes features like MariaDB Audit Plugin, MySQL Memcached, and S3 integration.

    The distinction between parameters and options reflects the architectural difference in these engines. A MySQL parameter like innodb_buffer_pool_size is a configuration value passed to the engine at startup. The MySQL Memcached plugin, by contrast, is a daemon process that runs alongside MySQL and requires its own configuration, network ports, and lifecycle management — it's an option, not a parameter.

    For SQL Server DBAs, option groups are how you enable SQL Server Agent, which is critical because many SQL Server jobs, maintenance plans, and SSIS integrations depend on Agent. Without enabling it through the option group, your RDS SQL Server instance has no Agent — and you will not find a way around this constraint.

    # Create an option group for SQL Server 2019 Enterprise
aws rds create-option-group \
  --option-group-name prod-sqlserver-ee-2019 \
  --engine-name sqlserver-ee \
  --major-engine-version "15.00" \
  --option-group-description "SQL Server 2019 EE production options" \
  --region us-east-1

# Add SQL Server Agent and SQL Server Audit
aws rds add-option-to-option-group \
  --option-group-name prod-sqlserver-ee-2019 \
  --options \
    '{"OptionName":"SQLSERVER_BACKUP_RESTORE","OptionSettings":[{"Name":"IAM_ROLE_ARN","Value":"arn:aws:iam::123456789012:role/rds-backup-role"}]}' \
    '{"OptionName":"SQLSERVER_AUDIT","OptionSettings":[{"Name":"IAM_ROLE_ARN","Value":"arn:aws:iam::123456789012:role/rds-audit-role"},{"Name":"S3_BUCKET_ARN","Value":"arn:aws:s3:::audit-logs-prod"},{"Name":"AUDIT_POLICY_FILE_URL","Value":"https://s3.amazonaws.com/audit-policy-bucket/policy.xml"}]}' \
  --apply-immediately \
  --region us-east-1

# Verify options are active
aws rds describe-option-groups \
  --option-group-name prod-sqlserver-ee-2019 \
  --query "OptionGroupsList[0].Options[*].{Option:OptionName,Status:OptionSettings}" \
  --region us-east-1

    The SQLSERVER_BACKUP_RESTORE option is what enables native backup/restore to and from S3 — the equivalent of your on-premises backup jobs writing to network shares. Without this option, you cannot restore a .bak file from S3 into RDS SQL Server. This is one of the most common blockers during SQL Server migrations to RDS, because DBAs attempt to restore their database and get The RESTORE option is not supported on this platform until they enable the option.

    SQL Server TDE on RDS deserves special mention. You can enable TDE via option group, but the certificate management model is different from on-premises. RDS manages the TDE certificate internally, and you cannot export it. If you're migrating a TDE-encrypted database from on-premises to RDS, you must decrypt the database first, or use AWS DMS to migrate the data without the encryption artifact. This is a hard constraint, not a workaround-able limitation.

    For MySQL, audit logging through the MariaDB Audit Plugin is configured as an option:

    # Add MariaDB Audit Plugin to MySQL 8.0 option group
aws rds add-option-to-option-group \
  --option-group-name prod-mysql80-options \
  --options '[{
    "OptionName": "MARIADB_AUDIT_PLUGIN",
    "OptionSettings": [
      {"Name": "SERVER_AUDIT_EVENTS", "Value": "CONNECT,QUERY_DDL,QUERY_DML"},
      {"Name": "SERVER_AUDIT_EXCL_USERS", "Value": "rdsadmin"},
      {"Name": "SERVER_AUDIT_QUERY_LOG_LIMIT", "Value": "1024"}
    ]
  }]' \
  --apply-immediately \
  --region us-east-1

    One operational constraint with option groups: certain options are persistent. A persistent option cannot be removed from an option group once added if there is a database in a state that depends on it. TDE is the most common example — once you enable TDE on a SQL Server instance, you cannot remove the option without first decrypting all affected databases. Plan your option group contents carefully before attaching them to production instances.

    12.4 Production Tuning Patterns for RDS PostgreSQL

    RDS PostgreSQL tuning starts from the same first principles as self-managed PostgreSQL but requires adjustments for the managed environment. You don't know the exact hardware specs of the underlying host — you know the instance class, and AWS publishes the vCPU count and RAM for each class, which is enough to derive reasonable starting values.

    For a db.r7g.4xlarge (16 vCPU, 128 GB RAM), a production OLTP baseline looks like this:

    | Parameter | Value | Reasoning |

    |---|---|---|

    | shared_buffers | Not directly settable on RDS | RDS auto-sets based on instance RAM |

    | work_mem | 64 MB (65536 kB) | Conservative for OLTP with high concurrency |

    | maintenance_work_mem | 2 GB | Speeds up VACUUM and index builds |

    | effective_cache_size | 96 GB | 75% of RAM, signals to planner what's cacheable |

    | checkpoint_completion_target | 0.9 | Spreads checkpoint I/O |

    | random_page_cost | 1.1 | For gp3/io1 SSD storage on RDS |

    | max_connections | 500 | Static; requires reboot; use RDS Proxy if > 500 |

    | log_min_duration_statement | 1000 ms | Log slow queries; adjust down for tighter SLA |

    | auto_vacuum_vacuum_cost_delay | 2 ms | Aggressive autovacuum for write-heavy workloads |

    A critical difference from self-managed PostgreSQL: shared_buffers is managed by RDS. AWS sets it to 25% of instance RAM by default and does not expose it as a directly settable parameter — the parameter exists in the group, but AWS overrides it based on instance class at startup. You will find shared_buffers listed in the parameter group with a formula like {DBInstanceClassMemory/32768} — that formula evaluates at instance startup, not when you modify the parameter group. This surprises DBAs who try to set shared_buffers = 32GB and then query SHOW shared_buffers and find a different value.

    For OLAP workloads on the same instance class, you'd push work_mem to 256 MB or higher and set max_parallel_workers_per_gather to 4, trading concurrency headroom for single-query throughput. On RDS, parallel query is available and functions as it does on self-managed PostgreSQL — but you cannot control OS scheduler settings or NUMA topology that affect parallel worker performance on large instances.

    A Python script using boto3 to audit parameter group drift across your fleet — identifying instances where custom parameters have diverged from your standard baseline — is one of the most valuable operational tools you can build:

    import boto3
import json

def audit_parameter_groups(region: str, baseline_group: str) -> dict:
    """
    Compare all custom parameter groups against a baseline group.
    Returns instances where parameter values differ from baseline.
    """
    rds = boto3.client('rds', region_name=region)
    
    # Load baseline parameters
    baseline_params = {}
    paginator = rds.get_paginator('describe_db_parameters')
    for page in paginator.paginate(DBParameterGroupName=baseline_group):
        for param in page['Parameters']:
            if param.get('ParameterValue') and param.get('IsModifiable'):
                baseline_params[param['ParameterName']] = param['ParameterValue']
    
    drift_report = {}
    
    # Get all DB instances
    instance_paginator = rds.get_paginator('describe_db_instances')
    for page in instance_paginator.paginate():
        for instance in page['DBInstances']:
            instance_id = instance['DBInstanceIdentifier']
            engine = instance['Engine']
            
            # Only check PostgreSQL instances
            if not engine.startswith('postgres'):
                continue
            
            param_groups = instance.get('DBParameterGroups', [])
            for pg in param_groups:
                pg_name = pg['DBParameterGroupName']
                pg_status = pg['ParameterApplyStatus']
                
                # Skip default groups and instances using baseline
                if pg_name.startswith('default.') or pg_name == baseline_group:
                    continue
                
                # Compare parameters
                instance_params = {}
                try:
                    for param_page in paginator.paginate(DBParameterGroupName=pg_name):
                        for param in param_page['Parameters']:
                            if param.get('ParameterValue') and param.get('IsModifiable'):
                                instance_params[param['ParameterName']] = param['ParameterValue']
                except rds.exceptions.DBParameterGroupNotFound:
                    continue
                
                diffs = {}
                for param_name, baseline_value in baseline_params.items():
                    instance_value = instance_params.get(param_name)
                    if instance_value and instance_value != baseline_value:
                        diffs[param_name] = {
                            'baseline': baseline_value,
                            'actual': instance_value
                        }
                
                if diffs or pg_status == 'pending-reboot':
                    drift_report[instance_id] = {
                        'parameter_group': pg_name,
                        'apply_status': pg_status,
                        'parameter_diffs': diffs
                    }
    
    return drift_report

if __name__ == '__main__':
    report = audit_parameter_groups(
        region='us-east-1',
        baseline_group='prod-postgres16-oltp'
    )
    print(json.dumps(report, indent=2))

    Run this script as a daily CloudWatch Events job and publish the results to an SNS topic. Any instance with pending-reboot status means you have parameters that aren't actually in effect — a common source of mysterious performance differences between instances that should be identically tuned.

    12.5 Tuning Limits: What RDS Won't Let You Do

    Understanding the hard boundaries of RDS tuning is as important as knowing what you can configure. These limits are not bugs or oversights — they are deliberate constraints of the managed service model. Knowing them up front prevents the frustration of spending hours chasing a tuning approach that will never work on RDS.

    OS-level tuning is unavailable. On self-managed PostgreSQL, you might set vm.overcommit_memory = 2, tune vm.dirty_ratio, configure huge_pages, or pin PostgreSQL to specific NUMA nodes. None of this is available on RDS. AWS handles the underlying OS configuration. Some instance classes benefit from huge pages, and AWS does enable them on instances with sufficient RAM (r-class instances above a certain size), but you cannot control this or verify it directly.

    pg_hba.conf is replaced by security groups and IAM. The familiar host-based authentication file is not editable on RDS. Network access control is handled by VPC security groups, and authentication beyond password-based methods is handled through IAM database authentication. If your on-premises PostgreSQL security model relied on pg_hba.conf to restrict specific users to specific databases from specific IP ranges, you'll reconstruct that model using a combination of VPC security groups (for network-layer restriction), database-level GRANT statements (for authorization), and RDS IAM auth (for eliminating static passwords).

    superuser access is restricted. RDS creates a master user that has broad privileges but is not a true PostgreSQL superuser. The rds_superuser role gives elevated access — you can create extensions, manage other users, and access most system functions — but you cannot access the PostgreSQL superuser privilege that allows operations like pg_read_file() on arbitrary paths, modifying pg_authid directly, or using COPY ... TO/FROM PROGRAM. Some extensions require true superuser installation and cannot be installed on RDS at all. Before planning an extension-heavy migration, verify each extension against the [list of supported extensions](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/CHAP_PostgreSQL.html#PostgreSQL.Concepts.General.Extensions) for your target RDS engine version.

    Write-ahead log access is limited. Logical replication slots work on RDS PostgreSQL, but you cannot inspect or manage physical WAL files directly. Tools like pg_waldump, custom WAL archiving scripts that pull from the WAL stream, and certain CDC implementations that expect direct WAL file access won't function as they would on self-managed instances. Use RDS's native logical replication and the pgoutput or wal2json plugins instead.

    Parameter group changes don't propagate to read replicas automatically. A read replica in RDS has its own parameter group association. When you tune the primary, you must also update the replica's parameter group separately. This is particularly easy to miss with max_connections — your primary handles 500 connections, your replica is still at the default 100, and connection attempts to the replica during a read-heavy event start failing. Set both explicitly, and consider whether replica parameter groups should be identical to or different from primary groups (replicas often benefit from higher work_mem values for analytical queries that run against them).
