Chapter 25: Re-ranking Results for Accuracy

Vector search is fast, but it is not always right. When you query a vector index for the top-10 nearest neighbors, you are asking a mathematical distance function to approximate semantic relevance — and that approximation has real limits. A document retrieved at rank 3 may be far more relevant to the user's intent than the document at rank 1, but the embedding space collapses enough meaning that the raw cosine scores cannot distinguish them. Re-ranking is the step that fixes this: a second-stage model reads the actual text of the query and each candidate document, computes a deeper cross-attention relevance score, and reorders the results before they are shown to the user or passed to an LLM. This chapter covers how re-ranking works, why it belongs in every production RAG pipeline, and how to implement it with Cohere, sentence-transformers, and LLM-based scoring — with full attention to latency, cost, and accuracy tradeoffs.

---

25.1 Why Vector Search Alone Is Not Enough

Vector embeddings are trained to map semantically similar text to nearby points in a high-dimensional space. That works well globally, but at retrieval time you are asking about local neighborhoods — specifically, whether the top-20 documents returned for a given query are ordered by *relevance to that exact query*, not just proximity in embedding space.

Three structural weaknesses appear repeatedly in production.

Embedding compression loss. A 1,536-dimensional embedding must encode all semantic content of a 512-token document into a fixed-size vector. Information is lost. Two documents with different specific claims can land at nearly identical distances from a query vector because their general topics are the same. A question about "PostgreSQL index bloat mitigation" may return documents about "PostgreSQL index performance" ranked higher than a document that actually explains REINDEX CONCURRENTLY — because "index performance" is a broader, more common phrase in training data.

Asymmetric query–document length mismatch. Queries are short. Documents are long. The embedding of a short query occupies a different statistical region than the embedding of a long, detailed document, even when the document directly answers the query. Bi-encoder models (the architecture behind most embedding APIs) encode the query and document independently, then compare the resulting vectors. This is efficient but structurally limited.

Approximate nearest neighbor (ANN) index error. HNSW and IVF indexes trade recall for speed. A well-tuned HNSW index with ef_search=100 typically achieves 95–98% recall against exact k-NN. That means 2–5 out of every 100 results are wrong even before relevance ordering enters the picture. Re-ranking does not fix ANN recall loss directly — that requires tuning ef_search or using hybrid search — but it does improve the ordering of whatever candidates are returned.

The practical solution is a two-stage retrieval pattern: retrieve a larger candidate set (top-50 to top-200) using fast ANN search, then re-rank that set with a slower but more accurate model, and return the top-k of that re-ranked set to the application. This keeps latency acceptable because the re-ranker only processes tens of documents, not millions.

25.2 Cross-Encoder Architecture and Why It Works Better

The fundamental architectural difference between bi-encoders (used for embedding generation) and cross-encoders (used for re-ranking) explains the accuracy gap.

A bi-encoder encodes query and document separately:

query  → encoder → q_vector
document → encoder → d_vector
score = cosine_similarity(q_vector, d_vector)

A cross-encoder processes the query and document jointly as a single input:

[query, document] → encoder → relevance_score

Because the cross-encoder sees both texts together, its attention mechanism can directly compare tokens across query and document. It can identify that the query phrase "index bloat mitigation" matches the document phrase "reclaiming space after index page splits" through multi-layer cross-attention, even though those phrases share no vocabulary. Bi-encoders cannot do this — the vectors are already frozen before comparison.

The tradeoff is latency and throughput. A bi-encoder query requires one inference pass for the query (which can be cached) plus a lookup against pre-computed document vectors. A cross-encoder requires one inference pass per (query, document) pair. At 50 candidates, that is 50 inference passes. This is why cross-encoders are never used as primary retrieval mechanisms against full indexes.

#### Sentence-Transformers Cross-Encoder

The sentence-transformers library provides a clean interface to cross-encoder models. The model family cross-encoder/ms-marco-MiniLM-L-6-v2 is a common choice for English re-ranking — it balances speed and accuracy, running roughly 200–400 pairs per second on a modern CPU.

# Python
from sentence_transformers import CrossEncoder
import numpy as np

# Load a cross-encoder trained on MS MARCO passage ranking
model = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2", max_length=512)

query = "How do I reduce index bloat in PostgreSQL?"

# Candidate documents from ANN retrieval (top-20 from pgvector)
candidates = [
    {
        "id": "doc_001",
        "text": "PostgreSQL VACUUM reclaims storage occupied by dead tuples. Running VACUUM ANALYZE regularly keeps table statistics current and prevents table bloat.",
    },
    {
        "id": "doc_002",
        "text": "Index bloat occurs when B-tree indexes accumulate dead pages after heavy UPDATE and DELETE workloads. REINDEX CONCURRENTLY rebuilds the index without taking an exclusive lock, making it safe for production use.",
    },
    {
        "id": "doc_003",
        "text": "The pg_stat_user_indexes view shows index scans. If an index has zero scans over a long period, it may be unused and safe to drop.",
    },
    {
        "id": "doc_004",
        "text": "autovacuum_vacuum_scale_factor controls how much table bloat triggers autovacuum. Lowering this value from the default 0.2 to 0.05 for high-write tables reduces both table and index bloat over time.",
    },
    {
        "id": "doc_005",
        "text": "Index performance in PostgreSQL depends on statistics. Running ANALYZE after bulk loads ensures the query planner uses indexes correctly.",
    },
]

# Build input pairs for the cross-encoder
pairs = [(query, doc["text"]) for doc in candidates]

# Score all pairs in one batched call
scores = model.predict(pairs, batch_size=16, show_progress_bar=False)

# Attach scores and sort descending
for doc, score in zip(candidates, scores):
    doc["rerank_score"] = float(score)

reranked = sorted(candidates, key=lambda x: x["rerank_score"], reverse=True)

print("Re-ranked results:")
for rank, doc in enumerate(reranked, 1):
    print(f"  Rank {rank:02d} | Score {doc['rerank_score']:+.4f} | {doc['id']}: {doc['text'][:80]}...")

Expected output shows doc_002 (about REINDEX CONCURRENTLY) rising to rank 1, and doc_004 (about autovacuum_vacuum_scale_factor) to rank 2 — both directly addressing bloat mitigation, while doc_001 (general VACUUM) and doc_005 (index performance, not bloat) fall.

25.3 Re-ranking with Cohere and Production API Integration

For teams using managed embedding services, Cohere's Rerank API provides cross-encoder quality re-ranking without model hosting. It accepts a query and a list of document strings, returns relevance scores and re-ordered indices, and handles tokenization and batching internally.

Cohere Rerank v3 supports up to 10,000 document tokens per call across all documents, processes up to 1,000 documents in a single request, and returns results in under 200ms for typical 50-document payloads.

# Python
import cohere
import os
from typing import Any

co = cohere.Client(api_key=os.environ["COHERE_API_KEY"])

def rerank_with_cohere(
    query: str,
    documents: list[dict[str, Any]],
    text_field: str = "text",
    top_n: int = 5,
    model: str = "rerank-english-v3.0",
) -> list[dict[str, Any]]:
    """
    Re-rank a list of candidate documents using Cohere Rerank.

    Args:
        query: The user query string.
        documents: List of dicts, each containing at least `text_field`.
        text_field: Key in each dict that holds the document text.
        top_n: How many top results to return after re-ranking.
        model: Cohere rerank model identifier.

    Returns:
        Top-n documents sorted by rerank relevance score, descending.
    """
    texts = [doc[text_field] for doc in documents]

    response = co.rerank(
        query=query,
        documents=texts,
        top_n=top_n,
        model=model,
        return_documents=True,
    )

    reranked = []
    for result in response.results:
        original_doc = documents[result.index].copy()
        original_doc["rerank_score"] = result.relevance_score
        original_doc["original_rank"] = result.index
        reranked.append(original_doc)

    return reranked


# --- Integration with Pinecone vector search ---
from pinecone import Pinecone
from openai import OpenAI

pc = Pinecone(api_key=os.environ["PINECONE_API_KEY"])
index = pc.Index("knowledge-base")
openai_client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

def retrieve_and_rerank(query: str, top_k_retrieve: int = 50, top_k_final: int = 5):
    # Stage 1: embed the query
    embed_response = openai_client.embeddings.create(
        input=query,
        model="text-embedding-3-small",
    )
    query_vector = embed_response.data[0].embedding

    # Stage 2: ANN retrieval — fetch a large candidate set
    pinecone_response = index.query(
        vector=query_vector,
        top_k=top_k_retrieve,
        include_metadata=True,
    )

    candidates = [
        {
            "id": match["id"],
            "text": match["metadata"].get("text", ""),
            "source": match["metadata"].get("source", ""),
            "vector_score": match["score"],
        }
        for match in pinecone_response["matches"]
        if match["metadata"].get("text")  # skip empty metadata
    ]

    if not candidates:
        return []

    # Stage 3: cross-encoder re-ranking via Cohere
    reranked = rerank_with_cohere(
        query=query,
        documents=candidates,
        text_field="text",
        top_n=top_k_final,
    )

    return reranked


results = retrieve_and_rerank(
    query="What autovacuum settings reduce index bloat on high-write tables?",
    top_k_retrieve=50,
    top_k_final=5,
)

for r in results:
    print(
        f"ID: {r['id']} | Rerank: {r['rerank_score']:.4f} | "
        f"VectorScore: {r['vector_score']:.4f} | {r['text'][:100]}"
    )

One important operational note: vector_score and rerank_score are not comparable. Cosine similarity from Pinecone runs roughly 0.7–0.95 for retrieved documents. Cohere relevance scores run 0.0–1.0 but are calibrated differently — a score of 0.3 from Cohere may represent a highly relevant document depending on the corpus. Always sort by rerank_score, never blend the two scores numerically without calibration.

25.4 Implementing Re-ranking in pgvector Pipelines

When your vector store is PostgreSQL with pgvector, the re-ranking step happens in the application layer after the SQL retrieval. The database returns candidate rows with their distance values; Python applies the cross-encoder and returns the re-ordered set.

A common pattern stores document chunks with their embeddings and full text in the same table, allowing the retrieval query to return everything needed for re-ranking in a single round trip.

-- PostgreSQL / pgvector
-- Schema for a knowledge base with full text available for re-ranking

CREATE TABLE knowledge_chunks (
    id          BIGSERIAL PRIMARY KEY,
    document_id UUID NOT NULL,
    chunk_index INT NOT NULL,
    chunk_text  TEXT NOT NULL,          -- full text, needed by re-ranker
    embedding   VECTOR(1536),
    source_url  TEXT,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX ON knowledge_chunks USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- Retrieve top-50 candidates with full text for re-ranking
-- The application layer will reduce this to top-5 after re-ranking
PREPARE retrieve_candidates (VECTOR(1536), INT) AS
    SELECT
        id,
        document_id,
        chunk_index,
        chunk_text,
        source_url,
        1 - (embedding <=> $1) AS cosine_similarity
    FROM knowledge_chunks
    ORDER BY embedding <=> $1
    LIMIT $2;

-- Execute: retrieve 50 candidates for a given query embedding
EXECUTE retrieve_candidates('[...query vector...]', 50);
# Python
import psycopg2
import psycopg2.extras
from sentence_transformers import CrossEncoder
from openai import OpenAI
import os

openai_client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])
cross_encoder = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2", max_length=512)

def get_query_embedding(query: str) -> list[float]:
    response = openai_client.embeddings.create(
        input=query,
        model="text-embedding-3-small",
    )
    return response.data[0].embedding


def pgvector_retrieve_and_rerank(
    query: str,
    conn_string: str,
    top_k_retrieve: int = 50,
    top_k_final: int = 5,
) -> list[dict]:

    embedding = get_query_embedding(query)

    # Format vector as PostgreSQL array literal
    vector_literal = "[" + ",".join(f"{v:.8f}" for v in embedding) + "]"

    with psycopg2.connect(conn_string) as conn:
        with conn.cursor(cursor_factory=psycopg2.extras.RealDictCursor) as cur:
            cur.execute(
                """
                SELECT
                    id,
                    document_id::TEXT,
                    chunk_index,
                    chunk_text,
                    source_url,
                    1 - (embedding <=> %s::vector) AS cosine_similarity
                FROM knowledge_chunks
                ORDER BY embedding <=> %s::vector
                LIMIT %s
                """,
                (vector_literal, vector_literal, top_k_retrieve),
            )
            candidates = [dict(row) for row in cur.fetchall()]

    if not candidates:
        return []

    # Cross-encoder scoring
    pairs = [(query, doc["chunk_text"]) for doc in candidates]
    scores = cross_encoder.predict(pairs, batch_size=32, show_progress_bar=False)

    for doc, score in zip(candidates, scores):
        doc["rerank_score"] = float(score)

    reranked = sorted(candidates, key=lambda x: x["rerank_score"], reverse=True)
    return reranked[:top_k_final]


# Usage
results = pgvector_retrieve_and_rerank(
    query="How does REINDEX CONCURRENTLY differ from REINDEX in terms of locking?",
    conn_string=os.environ["DATABASE_URL"],
    top_k_retrieve=50,
    top_k_final=5,
)

for rank, doc in enumerate(results, 1):
    print(
        f"Rank {rank} | Rerank: {doc['rerank_score']:+.3f} | "
        f"Vector: {doc['cosine_similarity']:.3f} | "
        f"Chunk {doc['chunk_index']} | {doc['chunk_text'][:120]}"
    )

For latency budgets: on a typical m5.large EC2 instance, retrieving 50 rows from pgvector takes 5–15ms. The ms-marco-MiniLM-L-6-v2 cross-encoder scores 50 pairs on CPU in approximately 120–200ms. Total latency for the two-stage retrieval stays under 300ms, well within a 500ms SLA for interactive search. If CPU latency is unacceptable, Cohere's API often beats local CPU inference: 50 documents through Cohere Rerank typically completes in 80–150ms round-trip from a co-located cloud instance.

25.5 LLM-Based Re-ranking and Reciprocal Rank Fusion

Cross-encoder re-ranking is the dominant pattern, but two additional techniques appear in production pipelines and are worth understanding.

#### LLM-Based Relevance Scoring

For domains with specialized vocabulary where general cross-encoder models underperform — legal documents, medical literature, internal engineering wikis — an LLM can score relevance directly. This is slower and more expensive than a cross-encoder but leverages the LLM's domain knowledge.

# Python
import asyncio
import json
from openai import AsyncOpenAI

async_client = AsyncOpenAI(api_key=os.environ["OPENAI_API_KEY"])

RELEVANCE_PROMPT = """You are a relevance scoring assistant. Given a query and a document passage, output a JSON object with a single key "score" containing an integer from 0 to 10, where:
- 0-2: The passage is not relevant to the query.
- 3-5: The passage is tangentially related but does not answer the query.
- 6-8: The passage is relevant and partially answers the query.
- 9-10: The passage directly and completely answers the query.

Output ONLY valid JSON. No explanation.

Query: {query}
Passage: {passage}"""


async def score_document_relevance(
    query: str, passage: str, model: str = "gpt-4o-mini"
) -> float:
    prompt = RELEVANCE_PROMPT.format(query=query, passage=passage[:1000])
    try:
        response = await async_client.chat.completions.create(
            model=model,
            messages=[{"role": "user", "content": prompt}],
            temperature=0.0,
            max_tokens=20,
            response_format={"type": "json_object"},
        )
        result = json.loads(response.choices[0].message.content)
        return float(result.get("score", 0)) / 10.0
    except (json.JSONDecodeError, KeyError, ValueError):
        return 0.0


async def llm_rerank(
    query: str, candidates: list[dict], text_field: str = "text", top_n: int = 5
) -> list[dict]:
    tasks = [
        score_document_relevance(query, doc[text_field]) for doc in candidates
    ]
    scores = await asyncio.gather(*tasks)

    for doc, score in zip(candidates, scores):
        doc["llm_rerank_score"] = score

    reranked = sorted(candidates, key=lambda x: x["llm_rerank_score"], reverse=True)
    return reranked[:top_n]


# Run async re-ranking
import asyncio

candidates = [...]  # from ANN retrieval
reranked = asyncio.run(llm_rerank(query="...", candidates=candidates))

LLM re-ranking with gpt-4o-mini costs approximately $0.0002 per document scored and adds 50–150ms per concurrent batch of 10 documents. It is justifiable for high-value queries in professional applications; it is not justifiable for general-purpose search with thousands of queries per minute.

#### Reciprocal Rank Fusion

Reciprocal Rank Fusion (RRF) combines multiple ranked lists — for example, a vector search result list and a keyword BM25 result list — into a single merged ranking without requiring score normalization. It is not a re-ranking model but a re-ranking *algorithm*, and it pairs naturally with hybrid search (covered in Chapter 22).

The formula for RRF score of document d given ranked lists R₁, R₂, ..., Rₖ is:

RRF(d) = Σ 1 / (k + rank_i(d))

where k is a constant (typically 60) and rank_i(d) is the rank of document d in list i.

# Python
from collections import defaultdict


def reciprocal_rank_fusion(
    ranked_lists: list[list[str]],
    k: int = 60,
) -> list[tuple[str, float]]:
    """
    Merge multiple ranked lists of document IDs using Reciprocal Rank Fusion.

    Args:
        ranked_lists: Each inner list is a ranked list of document IDs,
                      index 0 = rank 1 (highest), index -1 = lowest rank.
        k: RRF constant. Higher values reduce the impact of top ranks.
           k=60 is the standard value from the original Cormack et al. 2009 paper.

    Returns:
        Sorted list of (document_id, rrf_score) tuples, descending by score.
    """
    scores: dict[str, float] = defaultdict(float)

    for ranked_list in ranked_lists:
        for rank_zero_based, doc_id in enumerate(ranked_list):
            rank_one_based = rank_zero_based + 1
            scores[doc_id] += 1.0 / (k + rank_one_based)

    return sorted(scores.items(), key=lambda x: x[1], reverse=True)


# Example: combine vector search results with BM25 keyword search results
vector_result_ids = ["doc_002", "doc_005", "doc_001", "doc_004", "doc_003"]
bm25_result_ids   = ["doc_004", "doc_002", "doc_003", "doc_001", "doc_005"]

fused = reciprocal_rank_fusion([vector_result_ids, bm25_result_ids])

print("RRF-merged ranking:")
for rank, (doc_id, score) in enumerate(fused[:5], 1):
    print(f"  Rank {rank}: {doc_id} | RRF score: {score:.5f}")

RRF is computationally free — it runs in microseconds. Its value comes from combining signal sources, not from deeper relevance modeling. Use it when you have hybrid search returning two lists; use a cross-encoder when you need actual semantic re-ranking.
