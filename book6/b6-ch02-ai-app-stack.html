Chapter 2 of 40 AI-Native Apps on PostgreSQL

Chapter 2: The AI Application Stack: LLM + Vector + Relational Together

Modern AI applications are not single-layer systems. A chatbot that just calls an LLM and returns text will fail in production — it has no memory, no grounding in your actual data, no access controls, and no audit trail. The applications that work — the ones users trust and businesses depend on — layer three capabilities on top of each other: language model reasoning, semantic vector search, and relational data management. This chapter shows you how those three layers fit together architecturally, how PostgreSQL sits at the center of all of them, and how to build the concrete schemas and service patterns that make the whole stack function as a coherent system.

---

The Three-Layer Architecture

Before writing a single line of code, you need a clear mental model of what each layer does and why you cannot skip any of them.

The language model layer handles reasoning, generation, and language understanding. The LLM reads context and produces text — answers, summaries, structured output, tool call decisions. What the LLM cannot do is remember anything between calls, query your database, or know what happened in your system an hour ago. Every capability beyond raw language generation must be injected as context.

The vector layer handles semantic retrieval. When a user asks a question in natural language, keyword search fails because the question uses different words than the documents. Vector embeddings convert text into high-dimensional numerical representations where semantic similarity becomes geometric proximity. You retrieve relevant chunks not by keyword but by meaning. pgvector brings this capability directly into PostgreSQL as a native index type.

The relational layer handles everything else — and everything else turns out to be most of your application. User identity and authentication. Conversation history and session state. Document metadata and access permissions. Audit logs and compliance records. Structured product catalogs, customer records, orders, and configurations. The relationships between all of these entities. The transactional guarantees that keep them consistent.

The insight that makes PostgreSQL uniquely powerful here is that these three layers do not have to live in three separate systems. pgvector makes PostgreSQL a vector store. JSONB makes it a document store. The relational engine makes it a transactional system. You get all three — with joins, transactions, row-level security, and a thirty-year-old query planner — in a single database.

Here is the architectural reality compared to the "polyglot" alternative:

Polyglot Stack:                         PostgreSQL Stack:
─────────────────                       ────────────────────────────────
LLM API (OpenAI/Anthropic)              LLM API (OpenAI/Anthropic)
     │                                       │
Pinecone (vectors)   ←── sync? ──→     PostgreSQL
Postgres (relational)                    ├── pgvector (semantic search)
Redis (sessions)                         ├── JSONB (document storage)
Elasticsearch (search)                   ├── relational tables (everything)
     │                                   ├── pg_notify (events)
4 systems to operate,                    └── row-level security (access)
4 consistency models,
4 failure domains                        1 system. ACID everywhere.

This is not religious preference. It is operational economics. Every additional system multiplies your failure surface, your operational burden, and your consistency complexity. For most AI applications — including high-scale production ones — PostgreSQL as the unified data layer is the right architecture.

Schema Design: The Foundation Tables

The schema for an AI application has a recognizable shape regardless of what the application does. You will always need tables for users, conversations, messages, documents, embeddings, and audit events. Getting these right at the start saves you from painful migrations later.

-- Enable required extensions
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS pg_trgm;  -- for hybrid search

-- ─────────────────────────────────────────────────────────
-- Core user and tenant management
-- ─────────────────────────────────────────────────────────

CREATE TABLE tenants (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    name            TEXT NOT NULL,
    slug            TEXT NOT NULL UNIQUE,
    plan            TEXT NOT NULL DEFAULT 'free' CHECK (plan IN ('free', 'pro', 'enterprise')),
    settings        JSONB NOT NULL DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE users (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    tenant_id       UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    email           TEXT NOT NULL,
    display_name    TEXT,
    role            TEXT NOT NULL DEFAULT 'member' CHECK (role IN ('admin', 'member', 'viewer')),
    metadata        JSONB NOT NULL DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (tenant_id, email)
);

-- ─────────────────────────────────────────────────────────
-- Conversation and message history
-- ─────────────────────────────────────────────────────────

CREATE TABLE conversations (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    tenant_id       UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    title           TEXT,
    model           TEXT NOT NULL DEFAULT 'claude-3-5-sonnet-20241022',
    system_prompt   TEXT,
    token_count     INTEGER NOT NULL DEFAULT 0,
    message_count   INTEGER NOT NULL DEFAULT 0,
    status          TEXT NOT NULL DEFAULT 'active' CHECK (status IN ('active', 'archived', 'deleted')),
    metadata        JSONB NOT NULL DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_message_at TIMESTAMPTZ
);

CREATE INDEX idx_conversations_user_id ON conversations(user_id, created_at DESC);
CREATE INDEX idx_conversations_tenant_id ON conversations(tenant_id, status);

CREATE TABLE messages (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    conversation_id UUID NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
    role            TEXT NOT NULL CHECK (role IN ('system', 'user', 'assistant', 'tool')),
    content         TEXT NOT NULL,
    content_tokens  INTEGER,
    tool_calls      JSONB,          -- structured tool call data from LLM
    tool_call_id    TEXT,           -- for tool result messages
    model           TEXT,           -- which model generated this message
    finish_reason   TEXT,           -- 'stop', 'length', 'tool_use', etc.
    latency_ms      INTEGER,        -- how long the LLM took
    cost_usd        NUMERIC(10, 6), -- computed token cost
    metadata        JSONB NOT NULL DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_messages_conversation_id ON messages(conversation_id, created_at ASC);
CREATE INDEX idx_messages_created_at ON messages(created_at DESC);

-- ─────────────────────────────────────────────────────────
-- Document storage and embedding tables
-- ─────────────────────────────────────────────────────────

CREATE TABLE document_collections (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    tenant_id       UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    name            TEXT NOT NULL,
    description     TEXT,
    embedding_model TEXT NOT NULL DEFAULT 'text-embedding-3-small',
    embedding_dim   INTEGER NOT NULL DEFAULT 1536,
    chunk_strategy  JSONB NOT NULL DEFAULT '{"size": 512, "overlap": 64}',
    document_count  INTEGER NOT NULL DEFAULT 0,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (tenant_id, name)
);

CREATE TABLE documents (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    collection_id   UUID NOT NULL REFERENCES document_collections(id) ON DELETE CASCADE,
    tenant_id       UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    title           TEXT NOT NULL,
    source_uri      TEXT,           -- s3://, https://, file://, etc.
    source_type     TEXT NOT NULL DEFAULT 'text',  -- 'pdf', 'markdown', 'html', 'sql'
    content_hash    TEXT NOT NULL,  -- SHA-256 for deduplication
    raw_content     TEXT,           -- original text (may be large)
    metadata        JSONB NOT NULL DEFAULT '{}',  -- author, date, tags, etc.
    status          TEXT NOT NULL DEFAULT 'pending'
                        CHECK (status IN ('pending', 'processing', 'indexed', 'failed')),
    error_message   TEXT,
    chunk_count     INTEGER NOT NULL DEFAULT 0,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    indexed_at      TIMESTAMPTZ,
    UNIQUE (collection_id, content_hash)
);

CREATE INDEX idx_documents_collection_status ON documents(collection_id, status);
CREATE INDEX idx_documents_tenant_id ON documents(tenant_id);
CREATE INDEX idx_documents_metadata ON documents USING GIN(metadata);

CREATE TABLE document_chunks (
    id              UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    document_id     UUID NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    collection_id   UUID NOT NULL REFERENCES document_collections(id) ON DELETE CASCADE,
    tenant_id       UUID NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    chunk_index     INTEGER NOT NULL,  -- position within document
    content         TEXT NOT NULL,
    content_tokens  INTEGER,
    embedding       vector(1536),      -- pgvector column
    embedding_model TEXT NOT NULL DEFAULT 'text-embedding-3-small',
    metadata        JSONB NOT NULL DEFAULT '{}',  -- inherited + chunk-specific
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (document_id, chunk_index)
);

-- HNSW index for fast approximate nearest neighbor search
CREATE INDEX idx_chunks_embedding ON document_chunks
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- Full-text search index for hybrid retrieval
CREATE INDEX idx_chunks_content_fts ON document_chunks
    USING GIN(to_tsvector('english', content));

-- Tenant-scoped retrieval index
CREATE INDEX idx_chunks_tenant_collection ON document_chunks(tenant_id, collection_id);

-- ─────────────────────────────────────────────────────────
-- Audit and observability
-- ─────────────────────────────────────────────────────────

CREATE TABLE ai_audit_log (
    id              BIGSERIAL PRIMARY KEY,
    tenant_id       UUID REFERENCES tenants(id),
    user_id         UUID REFERENCES users(id),
    conversation_id UUID REFERENCES conversations(id),
    event_type      TEXT NOT NULL,  -- 'message', 'retrieval', 'tool_call', 'error'
    model           TEXT,
    input_tokens    INTEGER,
    output_tokens   INTEGER,
    cost_usd        NUMERIC(10, 6),
    latency_ms      INTEGER,
    retrieval_count INTEGER,        -- how many chunks were retrieved
    metadata        JSONB NOT NULL DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
) PARTITION BY RANGE (created_at);

-- Monthly partitions for audit log
CREATE TABLE ai_audit_log_2025_01 PARTITION OF ai_audit_log
    FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');
CREATE TABLE ai_audit_log_2025_02 PARTITION OF ai_audit_log
    FOR VALUES FROM ('2025-02-01') TO ('2025-03-01');
-- Automate partition creation with pg_partman in production

-- ─────────────────────────────────────────────────────────
-- Row-level security for multi-tenant isolation
-- ─────────────────────────────────────────────────────────

ALTER TABLE conversations ENABLE ROW LEVEL SECURITY;
ALTER TABLE messages ENABLE ROW LEVEL SECURITY;
ALTER TABLE documents ENABLE ROW LEVEL SECURITY;
ALTER TABLE document_chunks ENABLE ROW LEVEL SECURITY;

-- Application sets this at the start of each session
CREATE POLICY tenant_isolation ON conversations
    USING (tenant_id = current_setting('app.tenant_id')::UUID);

CREATE POLICY tenant_isolation ON messages
    USING (conversation_id IN (
        SELECT id FROM conversations
        WHERE tenant_id = current_setting('app.tenant_id')::UUID
    ));

CREATE POLICY tenant_isolation ON document_chunks
    USING (tenant_id = current_setting('app.tenant_id')::UUID);

Notice what this schema does beyond just storing text. The messages table records cost and latency per LLM call — you need this for billing and SLA monitoring. The document_chunks table carries both a vector embedding and a full-text search index — you need both for production retrieval quality. The audit log partitions by month — you need this when the table hits hundreds of millions of rows. Row-level security enforces tenant isolation at the database level — no amount of application bugs can leak one tenant's data to another.

Implementation: The RAG Retrieval Pipeline

Retrieval-augmented generation is the workhorse pattern for grounding LLM responses in real data. The implementation has four steps: embed the query, retrieve relevant chunks, assemble a context window, and call the LLM. PostgreSQL is the active participant in steps two and three.
