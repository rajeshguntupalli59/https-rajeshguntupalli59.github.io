Chapter 40: Building a Production Vector DB Monitoring Dashboard

Monitoring a vector database in production is not the same as monitoring a relational database. The familiar metrics—query throughput, lock waits, buffer hit ratio, replication lag—still matter when the underlying engine is PostgreSQL or a similar system, but they tell you nothing about the health of your embedding pipeline, the quality of your search results, or whether your index structures are serving queries efficiently under load. This chapter brings together everything from earlier in the book and shows you how to build a dashboard that covers all three layers: infrastructure metrics, vector-specific operational metrics, and application-layer quality signals. By the end, you will have runnable code for a Prometheus/Grafana stack that monitors pgvector, Pinecone, and Weaviate simultaneously, a Python-based quality probe that runs continuously against your RAG pipeline, and a pattern for surfacing drift and recall degradation before your users notice.

---

The Three-Layer Monitoring Model

Most teams start with infrastructure monitoring because it is familiar: CPU, memory, disk I/O, and connection counts. That baseline is necessary but insufficient for vector workloads. You need to think in three distinct layers, each requiring different instrumentation.

Layer 1 — Infrastructure and Engine Metrics

These are the metrics your existing tools already collect. For pgvector on PostgreSQL, that means pg_stat_statements, autovacuum activity, shared_buffers utilization, and WAL generation rate. For Pinecone, it means pod CPU and memory exposed through the Pinecone console API. For Weaviate, it means the built-in /v1/nodes endpoint and Prometheus scrape target.

Layer 2 — Vector Index Operational Metrics

These metrics are specific to the vector engine and have no direct SQL equivalent. Index build status, HNSW graph construction rate, quantization compression ratio, segment count, and—most importantly—approximate nearest neighbor (ANN) query latency at the 50th, 95th, and 99th percentiles. Think of the HNSW index as an analogy to a B-tree: just as you would monitor B-tree bloat and split frequency, you monitor HNSW graph connectivity and segment merger rates.

Layer 3 — Embedding and Search Quality Metrics

This layer has no equivalent in traditional database monitoring. It answers questions like: Are the embeddings being stored still aligned with the model that generated the query embeddings? Is the top-k recall degrading as new data is inserted without re-indexing? Are cosine similarity scores drifting toward lower values, indicating that the vector space is becoming less discriminative? These signals require synthetic probes—queries you run against the database with known expected results—and they need to run on a schedule, not just when a user complains.

The architectural principle is that your dashboard should surface a failure at Layer 3 before it becomes visible at Layer 2, and a Layer 2 failure before it cascades to Layer 1. In practice, embedding drift shows up weeks before your HNSW index becomes so fragmented that query latency spikes.

Instrumenting pgvector with Prometheus and Custom Metrics

PostgreSQL does not expose vector-specific metrics natively. You need to instrument them yourself, either through a custom postgres_exporter configuration or by running a metrics collection script that queries system views and writes to a Prometheus pushgateway.

The following script collects the most important pgvector operational metrics and pushes them to a Prometheus pushgateway on a 60-second interval. It covers index status, approximate row counts per collection, and query latency histograms derived from pg_stat_statements.

# Python
# pgvector_metrics_collector.py
# Requires: psycopg2, prometheus_client, schedule

import time
import schedule
import psycopg2
import psycopg2.extras
from prometheus_client import (
    CollectorRegistry,
    Gauge,
    Histogram,
    push_to_gateway,
)

PUSHGATEWAY_URL = "http://localhost:9091"
PG_DSN = "host=localhost dbname=vectordb user=monitor password=monitor_pass"

registry = CollectorRegistry()

# Gauge: number of live rows per vector table
vector_table_row_count = Gauge(
    "pgvector_table_live_rows",
    "Estimated live row count for vector tables",
    ["table_name", "index_name"],
    registry=registry,
)

# Gauge: index size in bytes
vector_index_size_bytes = Gauge(
    "pgvector_index_size_bytes",
    "On-disk size of vector indexes",
    ["index_name", "table_name"],
    registry=registry,
)

# Gauge: HNSW index build progress (0=not building, 1=building)
vector_index_building = Gauge(
    "pgvector_index_building",
    "Whether an HNSW index is currently being built (1=yes)",
    ["index_name"],
    registry=registry,
)

# Histogram: ANN query latency from pg_stat_statements (p50, p95, p99)
vector_query_latency_ms = Gauge(
    "pgvector_ann_query_mean_latency_ms",
    "Mean latency in ms for ANN queries (from pg_stat_statements)",
    ["query_pattern"],
    registry=registry,
)

vector_query_calls_total = Gauge(
    "pgvector_ann_query_calls_total",
    "Total call count for ANN query patterns",
    ["query_pattern"],
    registry=registry,
)


def collect_metrics():
    conn = psycopg2.connect(PG_DSN)
    cur = conn.cursor(cursor_factory=psycopg2.extras.DictCursor)

    # Collect live row counts and index sizes for vector tables
    # Identify vector tables by looking for vector-typed columns
    cur.execute("""
        SELECT
            c.relname                          AS table_name,
            i.relname                          AS index_name,
            pg_relation_size(i.oid)            AS index_size_bytes,
            pg_stat_get_live_tuples(c.oid)     AS live_rows,
            am.amname                          AS index_method
        FROM
            pg_class c
            JOIN pg_index ix ON c.oid = ix.indrelid
            JOIN pg_class i  ON i.oid = ix.indexrelid
            JOIN pg_am am    ON i.relam = am.oid
            JOIN pg_attribute a ON a.attrelid = c.oid
                AND a.attnum = ANY(ix.indkey)
            JOIN pg_type t  ON a.atttypid = t.oid
        WHERE
            t.typname = 'vector'
            AND c.relkind = 'r'
            AND am.amname IN ('hnsw', 'ivfflat')
        ORDER BY
            c.relname, i.relname;
    """)
    rows = cur.fetchall()
    for row in rows:
        vector_table_row_count.labels(
            table_name=row["table_name"],
            index_name=row["index_name"],
        ).set(row["live_rows"])
        vector_index_size_bytes.labels(
            index_name=row["index_name"],
            table_name=row["table_name"],
        ).set(row["index_size_bytes"])

    # Detect in-progress index builds
    cur.execute("""
        SELECT
            relname AS index_name,
            phase,
            blocks_done,
            blocks_total
        FROM
            pg_stat_progress_create_index
        WHERE
            command = 'CREATE INDEX CONCURRENTLY'
            OR command = 'CREATE INDEX';
    """)
    building = cur.fetchall()
    # Reset all to 0 first, then set active ones
    for row in building:
        vector_index_building.labels(index_name=row["index_name"]).set(1)

    # ANN query latency from pg_stat_statements
    # Filter for queries that contain '<->' (L2), '<=>' (cosine), or '<#>' (inner product)
    cur.execute("""
        SELECT
            LEFT(query, 80)                             AS query_pattern,
            calls                                       AS total_calls,
            mean_exec_time                              AS mean_ms
        FROM
            pg_stat_statements
        WHERE
            query LIKE '%<->%'
            OR query LIKE '%<=>%'
            OR query LIKE '%<#>%'
        ORDER BY
            total_exec_time DESC
        LIMIT 20;
    """)
    query_rows = cur.fetchall()
    for qrow in query_rows:
        label = qrow["query_pattern"].strip().replace("\n", " ")[:60]
        vector_query_latency_ms.labels(query_pattern=label).set(qrow["mean_ms"])
        vector_query_calls_total.labels(query_pattern=label).set(qrow["total_calls"])

    cur.close()
    conn.close()

    push_to_gateway(
        PUSHGATEWAY_URL,
        job="pgvector_metrics",
        registry=registry,
    )
    print(f"[{time.strftime('%H:%M:%S')}] Metrics pushed to Prometheus gateway")


schedule.every(60).seconds.do(collect_metrics)

if __name__ == "__main__":
    collect_metrics()  # Run immediately on start
    while True:
        schedule.run_pending()
        time.sleep(1)

This script intentionally uses pg_stat_statements rather than wrapping every query with timing code. That means it captures the full query load from all connections, not just the ones your application explicitly instruments. The mean_exec_time from pg_stat_statements is a rolling mean since the last statistics reset; pair it with the calls counter and you can compute a rate-weighted average in Grafana.

For the Grafana dashboard, create a panel with this PromQL expression to show the top-5 slowest ANN query patterns:

# shell
# Prometheus query for Grafana panel: Top 5 slowest ANN query patterns
# Panel type: Table
# PromQL:
topk(5, pgvector_ann_query_mean_latency_ms)

# Panel type: Time series for index size growth
# PromQL:
rate(pgvector_index_size_bytes[5m]) * 300
# Multiplied by 300 seconds (5-minute interval) to get bytes added per scrape window

# Alert rule: ANN query latency exceeds 200ms mean
# In Prometheus alerting rules YAML:
# alert: VectorQueryLatencyHigh
# expr: pgvector_ann_query_mean_latency_ms > 200
# for: 5m
# labels:
#   severity: warning
# annotations:
#   summary: "ANN query mean latency > 200ms for {{ $labels.query_pattern }}"

Monitoring Pinecone and Weaviate via Their Native APIs

Pinecone and Weaviate expose operational metrics through REST APIs, but they each have different granularity and different concepts of what "healthy" means.

Pinecone Monitoring

Pinecone's Python client exposes describe_index_stats(), which gives you namespace-level vector counts and fullness. The fullness metric is the single most important operational signal for a pod-based Pinecone deployment: when it exceeds 0.90, query latency climbs steeply and inserts may fail.

# Python
# pinecone_metrics_collector.py
# Requires: pinecone-client>=3.0, prometheus_client

import os
import time
import schedule
from pinecone import Pinecone
from prometheus_client import (
    CollectorRegistry,
    Gauge,
    push_to_gateway,
)

PUSHGATEWAY_URL = "http://localhost:9091"
PINECONE_API_KEY = os.environ["PINECONE_API_KEY"]

pc = Pinecone(api_key=PINECONE_API_KEY)

registry = CollectorRegistry()

pinecone_vector_count = Gauge(
    "pinecone_namespace_vector_count",
    "Number of vectors in a Pinecone namespace",
    ["index_name", "namespace"],
    registry=registry,
)

pinecone_index_fullness = Gauge(
    "pinecone_index_fullness_ratio",
    "Index fullness ratio (0.0 to 1.0). Alert at > 0.85",
    ["index_name"],
    registry=registry,
)

pinecone_index_dimension = Gauge(
    "pinecone_index_dimension",
    "Vector dimension for a Pinecone index",
    ["index_name", "metric"],
    registry=registry,
)

pinecone_index_ready = Gauge(
    "pinecone_index_ready",
    "1 if the index state is Ready, 0 otherwise",
    ["index_name"],
    registry=registry,
)


def collect_pinecone_metrics():
    indexes = pc.list_indexes()

    for index_info in indexes:
        index_name = index_info.name
        idx = pc.Index(index_name)

        # Index status
        desc = pc.describe_index(index_name)
        is_ready = 1 if desc.status.ready else 0
        pinecone_index_ready.labels(index_name=index_name).set(is_ready)
        pinecone_index_dimension.labels(
            index_name=index_name,
            metric=desc.metric,
        ).set(desc.dimension)

        # Stats per namespace
        stats = idx.describe_index_stats()
        pinecone_index_fullness.labels(index_name=index_name).set(
            stats.index_fullness
        )

        for namespace, ns_stats in stats.namespaces.items():
            pinecone_vector_count.labels(
                index_name=index_name,
                namespace=namespace if namespace else "__default__",
            ).set(ns_stats.vector_count)

    push_to_gateway(
        PUSHGATEWAY_URL,
        job="pinecone_metrics",
        registry=registry,
    )
    print(f"[{time.strftime('%H:%M:%S')}] Pinecone metrics pushed")


schedule.every(30).seconds.do(collect_pinecone_metrics)

if __name__ == "__main__":
    collect_pinecone_metrics()
    while True:
        schedule.run_pending()
        time.sleep(1)

Weaviate Monitoring

Weaviate ships with a built-in Prometheus endpoint at http://:2112/metrics when you set PROMETHEUS_MONITORING_ENABLED=true in the environment. The native metrics cover object count, index segment count, vector dimension, and batch import throughput. Beyond those, you want to query the /v1/nodes REST endpoint for cluster health.

# Python
# weaviate_health_probe.py
# Requires: weaviate-client>=4.0, requests, prometheus_client

import os
import time
import requests
import schedule
import weaviate
from weaviate.classes.init import Auth
from prometheus_client import (
    CollectorRegistry,
    Gauge,
    push_to_gateway,
)

PUSHGATEWAY_URL = "http://localhost:9091"
WEAVIATE_URL = os.environ.get("WEAVIATE_URL", "http://localhost:8080")
WEAVIATE_API_KEY = os.environ.get("WEAVIATE_API_KEY", "")

registry = CollectorRegistry()

weaviate_node_status = Gauge(
    "weaviate_node_healthy",
    "1 if the Weaviate node is healthy",
    ["node_name"],
    registry=registry,
)

weaviate_object_count = Gauge(
    "weaviate_collection_object_count",
    "Number of objects in a Weaviate collection",
    ["collection_name", "node_name"],
    registry=registry,
)

weaviate_shard_count = Gauge(
    "weaviate_collection_shard_count",
    "Number of shards for a Weaviate collection",
    ["collection_name"],
    registry=registry,
)

weaviate_vector_index_queue_length = Gauge(
    "weaviate_async_index_queue_length",
    "Number of objects waiting to be indexed asynchronously",
    ["collection_name", "shard_name"],
    registry=registry,
)


def collect_weaviate_metrics():
    # Use REST directly for /v1/nodes — gives richer cluster info than client
    headers = {}
    if WEAVIATE_API_KEY:
        headers["Authorization"] = f"Bearer {WEAVIATE_API_KEY}"

    try:
        resp = requests.get(
            f"{WEAVIATE_URL}/v1/nodes",
            headers=headers,
            params={"output": "verbose"},
            timeout=10,
        )
        resp.raise_for_status()
        nodes_data = resp.json()
    except Exception as e:
        print(f"Failed to reach Weaviate nodes endpoint: {e}")
        return

    collection_shards: dict[str, int] = {}
    collection_objects: dict[tuple, int] = {}

    for node in nodes_data.get("nodes", []):
        node_name = node.get("name", "unknown")
        node_status = node.get("status", "UNHEALTHY")
        weaviate_node_status.labels(node_name=node_name).set(
            1 if node_status == "HEALTHY" else 0
        )

        for shard in node.get("shards", []) or []:
            coll_name = shard.get("class", "unknown")
            shard_name = shard.get("name", "unknown")
            obj_count = shard.get("objectCount", 0)
            queue_len = shard.get("vectorIndexingQueue", 0)

            collection_shards[coll_name] = (
                collection_shards.get(coll_name, 0) + 1
            )
            key = (coll_name, node_name)
            collection_objects[key] = collection_objects.get(key, 0) + obj_count

            weaviate_vector_index_queue_length.labels(
                collection_name=coll_name,
                shard_name=shard_name,
            ).set(queue_len)

    for (coll_name, node_name), count in collection_objects.items():
        weaviate_object_count.labels(
            collection_name=coll_name,
            node_name=node_name,
        ).set(count)

    for coll_name, shard_count in collection_shards.items():
        weaviate_shard_count.labels(collection_name=coll_name).set(shard_count)

    push_to_gateway(
        PUSHGATEWAY_URL,
        job="weaviate_metrics",
        registry=registry,
    )
    print(f"[{time.strftime('%H:%M:%S')}] Weaviate metrics pushed")


schedule.every(30).seconds.do(collect_weaviate_metrics)

if __name__ == "__main__":
    collect_weaviate_metrics()
    while True:
        schedule.run_pending()
        time.sleep(1)

The weaviate_async_index_queue_length metric is particularly important when using Weaviate's async indexing mode. A queue that grows monotonically without draining indicates that your import rate exceeds the indexing throughput—the vector equivalent of a replication lag that never catches up. Set an alert at queue depth > 50,000 for most workloads, and > 10,000 for latency-sensitive applications.

Embedding Quality and Recall Probes

Infrastructure metrics tell you whether your database is running. Quality probes tell you whether it is working. This distinction matters enormously for RAG applications because a system can show green on every infrastructure dashboard while silently returning wrong results.

The right way to implement quality probes is to maintain a golden dataset: a small set of queries (typically 50–200) for which you know the correct top-k results. These are generated once, stored in a separate table or file, and then replayed continuously. The probe computes recall@k—the fraction of expected results that appear in the actual results—and reports it as a metric.
