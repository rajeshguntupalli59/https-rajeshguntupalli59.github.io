Chapter 15: Intelligent Workload Classification

Autonomous database systems can only act intelligently when they understand *what* they're running. Query tuning, resource allocation, connection routing, and scheduling decisions all depend on knowing whether a given workload is OLTP, analytical, batch, ETL, or something in between — and that classification must happen continuously, at runtime, without human annotation. This chapter builds the full stack of intelligent workload classification: from raw signal extraction through feature engineering, model-based classification, and the downstream control loops that consume those labels to make autonomous decisions. The architecture described here runs in production today, using PostgreSQL's statistics infrastructure and SQL Server's Query Store as primary data sources, augmented by custom instrumentation and ML inference layers that operate with sub-second latency for hot decisions and background model refresh cycles for drift correction.

---

15.1 Signal Extraction: Reading Workload DNA from Native Instrumentation

Classification begins with signals. Both PostgreSQL and SQL Server expose rich runtime telemetry, but the signals that matter for workload classification are not the obvious ones like CPU or memory — they are behavioral fingerprints embedded in execution statistics, wait profiles, and access patterns.

The core features for workload classification fall into four categories:

  • → **Execution profile**: rows examined vs. rows returned ratio, execution frequency, plan stability
  • → **Resource signature**: buffer hits vs. disk reads, temp space usage, parallelism degree
  • → **Wait topology**: the distribution of wait types encountered during execution
  • → **Temporal pattern**: time-of-day clustering, burst coefficient, inter-arrival time variance

    • PostgreSQL exposes most of this through pg_stat_statements, pg_stat_user_tables, and pg_stat_activity. The challenge is that pg_stat_statements normalizes queries and loses some instance-level variance. Augmenting it with auto_explain for sampling and a custom background worker that snapshots pg_stat_activity at 1-second intervals gives you the full picture.

    PostgreSQL:

    -- Extract raw workload signals from pg_stat_statements
-- This query builds the feature vector for classification
SELECT
    queryid,
    query,
    calls,
    total_exec_time / NULLIF(calls, 0)                          AS avg_exec_ms,
    stddev_exec_time / NULLIF(total_exec_time / NULLIF(calls,0), 0) AS exec_time_cv,
    rows / NULLIF(calls, 0)                                     AS avg_rows_returned,
    shared_blks_hit / NULLIF(shared_blks_hit + shared_blks_read, 0) AS buffer_hit_ratio,
    shared_blks_read / NULLIF(calls, 0)                         AS avg_physical_reads,
    temp_blks_written / NULLIF(calls, 0)                        AS avg_temp_writes,
    local_blks_hit + local_blks_read                            AS total_local_blks,
    wal_bytes / NULLIF(calls, 0)                                AS avg_wal_bytes,
    -- Frequency signal: calls per hour over observation window
    calls / EXTRACT(EPOCH FROM (now() - stats_since)) * 3600   AS calls_per_hour
FROM pg_stat_statements
WHERE calls > 10  -- minimum sample size for stable features
  AND query NOT LIKE '/* internal */%'
ORDER BY total_exec_time DESC;

    SQL Server:

    -- Extract equivalent workload signals from Query Store
-- Maps to the same feature vector schema used for classification
SELECT
    qsq.query_id,
    qsqt.query_sql_text,
    SUM(qsrs.count_executions)                                          AS total_calls,
    AVG(qsrs.avg_duration / 1000.0)                                     AS avg_exec_ms,
    STDEV(qsrs.avg_duration) / NULLIF(AVG(qsrs.avg_duration), 0)       AS exec_time_cv,
    AVG(qsrs.avg_rowcount)                                              AS avg_rows_returned,
    AVG(qsrs.avg_logical_io_reads)                                      AS avg_logical_reads,
    AVG(qsrs.avg_physical_io_reads)                                     AS avg_physical_reads,
    AVG(qsrs.avg_tempdb_space_used)                                     AS avg_temp_writes,
    AVG(qsrs.avg_log_bytes_used)                                        AS avg_log_bytes,
    SUM(qsrs.count_executions) /
        NULLIF(DATEDIFF(HOUR,
            MIN(qsrs.first_execution_time),
            MAX(qsrs.last_execution_time)), 0)                          AS calls_per_hour,
    AVG(qsrs.avg_dop)                                                   AS avg_parallelism
FROM sys.query_store_query             qsq
JOIN sys.query_store_query_text        qsqt ON qsq.query_text_id = qsqt.query_text_id
JOIN sys.query_store_plan              qspp ON qsq.query_id       = qspp.query_id
JOIN sys.query_store_runtime_stats     qsrs ON qspp.plan_id       = qsrs.plan_id
WHERE qsrs.count_executions > 10
GROUP BY qsq.query_id, qsqt.query_sql_text
ORDER BY SUM(qsrs.avg_duration * qsrs.count_executions) DESC;

    Wait profiles require a separate collection path. In PostgreSQL, pg_stat_activity.wait_event and pg_stat_activity.wait_event_type give per-session instantaneous wait data. A 1-second polling collector running as a background process builds a wait histogram per query pattern over time. In SQL Server, sys.dm_os_wait_stats and per-query waits from sys.dm_exec_query_stats joined with sys.dm_exec_query_plan provide richer per-plan wait attribution.

    The wait topology feature is then encoded as a distribution vector: what fraction of total wait time falls into Lock, LWLock/Buffer, IO, CPU, Network, and Idle categories. OLTP workloads show high Lock and short CPU spikes. Analytical workloads show sustained CPU with sequential IO. Batch ETL workloads show high IO and Log write waits with low Lock contention.

    15.2 Feature Engineering for Runtime Classification

    Raw telemetry numbers are not directly usable by a classifier. The feature engineering layer transforms them into scale-invariant, semantically meaningful representations that generalize across databases of different sizes and query volumes.

    The derived features that carry the most discriminative signal are:

    | Feature | Formula | Interpretation |

    |---|---|---|

    | scan_dominance | seq_scan_rows / total_rows_examined | High = analytical |

    | write_amplification | wal_bytes / (rows_modified + 1) | High = batch/ETL |

    | result_compression | rows_returned / rows_examined | Low = aggregation-heavy |

    | temporal_burst_coef | calls_p99 / calls_median over 5-min windows | High = bursty OLTP |

    | plan_instability | distinct_plans / total_calls | High = parameter sensitivity |

    | parallelism_affinity | avg_dop / max_dop | High = analytical intent |

    The following Python feature engineering pipeline runs as a scheduled collector — pulling from both databases, computing derived features, and publishing to a Redis time-series store that feeds the classification inference layer.

    import psycopg2
import pyodbc
import redis
import json
import time
from dataclasses import dataclass, asdict
from typing import Optional
import numpy as np

@dataclass
class WorkloadFeatureVector:
    query_id: str
    database_type: str  # 'postgresql' | 'sqlserver'
    avg_exec_ms: float
    exec_time_cv: float           # coefficient of variation
    avg_rows_returned: float
    buffer_hit_ratio: float
    avg_physical_reads: float
    avg_temp_writes: float
    avg_wal_bytes: float
    calls_per_hour: float
    avg_parallelism: float
    # Derived features
    result_compression: float     # rows_returned / rows_examined
    temporal_burst_coef: float
    plan_instability: float
    write_amplification: float

def compute_derived_features(raw: dict, burst_history: list[float]) -> dict:
    """
    Compute scale-invariant derived features from raw telemetry.
    burst_history: list of calls_per_hour values from last 12 x 5-min windows
    """
    rows_examined = raw['avg_rows_returned'] / max(raw['buffer_hit_ratio'], 0.001)

    result_compression = raw['avg_rows_returned'] / max(rows_examined, 1)

    # Burst coefficient: ratio of 95th percentile to median over recent windows
    if len(burst_history) >= 4:
        p95 = float(np.percentile(burst_history, 95))
        median = float(np.median(burst_history))
        burst_coef = p95 / max(median, 0.1)
    else:
        burst_coef = 1.0

    write_amplification = raw['avg_wal_bytes'] / max(raw['avg_rows_returned'], 1)

    return {
        'result_compression': result_compression,
        'temporal_burst_coef': burst_coef,
        'write_amplification': write_amplification,
    }

class FeatureCollector:
    def __init__(self, pg_dsn: str, mssql_dsn: str, redis_url: str):
        self.pg_conn = psycopg2.connect(pg_dsn)
        self.mssql_conn = pyodbc.connect(mssql_dsn)
        self.redis = redis.from_url(redis_url)
        self.burst_cache: dict[str, list[float]] = {}

    def collect_and_publish(self):
        pg_features = self._collect_postgresql()
        ms_features = self._collect_sqlserver()

        for fv in pg_features + ms_features:
            key = f"wl:features:{fv.database_type}:{fv.query_id}"
            self.redis.setex(key, 3600, json.dumps(asdict(fv)))
            # Append to time-series for burst calculation
            ts_key = f"wl:timeseries:{fv.database_type}:{fv.query_id}"
            self.redis.lpush(ts_key, fv.calls_per_hour)
            self.redis.ltrim(ts_key, 0, 11)  # keep 12 windows = 60 minutes

    def _collect_postgresql(self) -> list[WorkloadFeatureVector]:
        # ... execute the pg_stat_statements query and build feature vectors
        # burst history retrieved from Redis time-series
        pass  # implementation follows same pattern as SQL Server below

    def _collect_sqlserver(self) -> list[WorkloadFeatureVector]:
        cur = self.mssql_conn.cursor()
        cur.execute("""
            SELECT
                CAST(qsq.query_id AS VARCHAR(20)) AS query_id,
                AVG(qsrs.avg_duration / 1000.0)   AS avg_exec_ms,
                STDEV(qsrs.avg_duration) / NULLIF(AVG(qsrs.avg_duration),0) AS exec_time_cv,
                AVG(qsrs.avg_rowcount)             AS avg_rows,
                AVG(qsrs.avg_logical_io_reads)     AS avg_logical,
                AVG(qsrs.avg_physical_io_reads)    AS avg_physical,
                AVG(qsrs.avg_tempdb_space_used)    AS avg_temp,
                AVG(qsrs.avg_log_bytes_used)       AS avg_log,
                SUM(qsrs.count_executions) /
                    NULLIF(DATEDIFF(HOUR,
                        MIN(qsrs.first_execution_time),
                        GETUTCDATE()), 0)           AS calls_per_hour,
                AVG(qsrs.avg_dop)                  AS avg_dop,
                COUNT(DISTINCT qspp.plan_id) * 1.0 /
                    NULLIF(SUM(qsrs.count_executions), 0) AS plan_instability
            FROM sys.query_store_query         qsq
            JOIN sys.query_store_plan          qspp ON qsq.query_id   = qspp.query_id
            JOIN sys.query_store_runtime_stats qsrs ON qspp.plan_id   = qsrs.plan_id
            WHERE qsrs.count_executions > 10
              AND qsrs.last_execution_time > DATEADD(MINUTE, -60, GETUTCDATE())
            GROUP BY qsq.query_id
        """)
        results = []
        for row in cur.fetchall():
            burst_key = f"wl:timeseries:sqlserver:{row.query_id}"
            history = [float(v) for v in self.redis.lrange(burst_key, 0, 11)]
            derived = compute_derived_features({
                'avg_rows_returned': row.avg_rows or 0,
                'buffer_hit_ratio': (row.avg_logical - (row.avg_physical or 0)) /
                                    max(row.avg_logical or 1, 1),
                'avg_wal_bytes': row.avg_log or 0,
            }, history)
            fv = WorkloadFeatureVector(
                query_id=row.query_id,
                database_type='sqlserver',
                avg_exec_ms=row.avg_exec_ms or 0,
                exec_time_cv=row.exec_time_cv or 0,
                avg_rows_returned=row.avg_rows or 0,
                buffer_hit_ratio=(row.avg_logical - (row.avg_physical or 0)) /
                                  max(row.avg_logical or 1, 1),
                avg_physical_reads=row.avg_physical or 0,
                avg_temp_writes=row.avg_temp or 0,
                avg_wal_bytes=row.avg_log or 0,
                calls_per_hour=row.calls_per_hour or 0,
                avg_parallelism=row.avg_dop or 1,
                plan_instability=row.plan_instability or 0,
                **derived
            )
            results.append(fv)
        return results

    15.3 The Classification Model: Architecture and Label Taxonomy

    The classification model sits at the core of the autonomous control loop. It receives feature vectors and produces workload labels that drive downstream decisions. The label taxonomy needs to be precise enough to guide action but stable enough to avoid thrashing.

    A five-class taxonomy works well across all production scenarios:

  • → **OLTP_POINT**: Short-duration, high-frequency, index-bound, single-digit row access. Classic transactional. Action: prioritize in connection pools, protect from query plan regression.
  • → **OLTP_MODERATE**: Slightly wider queries — small range scans, multi-table joins, under 100ms. Action: standard SLA tier.
  • → **ANALYTICAL**: Long-running, high-row-count aggregations, sequential scans, parallel execution. Action: route to read replica, assign lower CPU priority, allow longer lock waits.
  • → **BATCH_ETL**: High write amplification, sequential access, predictable schedule, large temp usage. Action: schedule during maintenance window, pre-warm buffers, expand autovacuum budget post-run.
  • → **MAINTENANCE**: VACUUM, ANALYZE, index builds, statistics updates, DBCC operations. Action: track separately, do not apply SLA enforcement, use as calibration signal.

    • The classifier itself is a gradient-boosted tree model (XGBoost or LightGBM) trained on labeled historical data from pg_stat_statements and Query Store. The labeling phase is the only place human judgment enters: a DBA manually labels 500–1000 representative query patterns once, and the model generalizes from there. Semi-supervised bootstrapping can extend coverage to new query patterns by treating high-confidence predictions as pseudo-labels for retraining.

    In production, the classifier runs as a FastAPI microservice with a 5ms SLA for batch classification of up to 200 feature vectors per request. The model is loaded into memory and refreshed weekly via a background retraining pipeline triggered by concept drift detection.

    import lightgbm as lgb
import numpy as np
import joblib
from fastapi import FastAPI
from pydantic import BaseModel
from typing import List
from enum import Enum

class WorkloadClass(str, Enum):
    OLTP_POINT    = "OLTP_POINT"
    OLTP_MODERATE = "OLTP_MODERATE"
    ANALYTICAL    = "ANALYTICAL"
    BATCH_ETL     = "BATCH_ETL"
    MAINTENANCE   = "MAINTENANCE"

FEATURE_COLUMNS = [
    'avg_exec_ms', 'exec_time_cv', 'avg_rows_returned',
    'buffer_hit_ratio', 'avg_physical_reads', 'avg_temp_writes',
    'avg_wal_bytes', 'calls_per_hour', 'avg_parallelism',
    'result_compression', 'temporal_burst_coef',
    'plan_instability', 'write_amplification'
]

LABEL_MAP = {0: WorkloadClass.OLTP_POINT,
             1: WorkloadClass.OLTP_MODERATE,
             2: WorkloadClass.ANALYTICAL,
             3: WorkloadClass.BATCH_ETL,
             4: WorkloadClass.MAINTENANCE}

class ClassificationRequest(BaseModel):
    query_id: str
    features: dict

class ClassificationResponse(BaseModel):
    query_id: str
    workload_class: WorkloadClass
    confidence: float
    probabilities: dict

app = FastAPI()
model: lgb.Booster = joblib.load("/models/workload_classifier_v3.pkl")

@app.post("/classify", response_model=List[ClassificationResponse])
def classify_batch(requests: List[ClassificationRequest]):
    if not requests:
        return []

    feature_matrix = np.array([
        [r.features.get(col, 0.0) for col in FEATURE_COLUMNS]
        for r in requests
    ])

    # LightGBM returns probability matrix (n_samples x n_classes)
    proba_matrix = model.predict(feature_matrix)

    results = []
    for req, probas in zip(requests, proba_matrix):
        predicted_idx = int(np.argmax(probas))
        predicted_class = LABEL_MAP[predicted_idx]
        confidence = float(probas[predicted_idx])

        # Refuse to classify if confidence is below threshold —
        # route to human review queue instead
        if confidence < 0.65:
            predicted_class = WorkloadClass.OLTP_MODERATE  # safe default
            confidence = 0.0  # signals uncertainty to downstream consumers

        results.append(ClassificationResponse(
            query_id=req.query_id,
            workload_class=predicted_class,
            confidence=confidence,
            probabilities={LABEL_MAP[i].value: float(p)
                           for i, p in enumerate(probas)}
        ))
    return results

    The confidence threshold of 0.65 is not arbitrary. Below this level, empirical testing shows that misclassifications cause more harm (wrong resource allocation, incorrect routing) than the safe default of OLTP_MODERATE. Queries below threshold are published to a review queue consumed weekly by the DBA for manual labeling — feeding the retraining loop.

    15.4 Real-Time Classification Routing and Resource Policy Enforcement

    Classification labels are only valuable when they drive action at runtime. The downstream control loop connects workload labels to three enforcement mechanisms: connection pool routing, resource group assignment, and query timeout policies.

    Connection Pool Routing via pgBouncer

    pgBouncer does not natively support dynamic pool assignment, but its configuration can be rewritten and reloaded without dropping connections via RELOAD. An orchestration agent monitors the classification service and, when it detects that a query pattern has been reclassified (e.g., a formerly OLTP query has migrated to ANALYTICAL due to data growth), it rewrites the pgBouncer pool assignment and issues a reload.

    PostgreSQL:

    -- PostgreSQL: Resource group assignment using pg_cgroup integration
-- or using custom session parameters that the application layer respects.
-- This procedure is called by the classification agent post-classification.

CREATE OR REPLACE PROCEDURE assign_workload_resource_group(
    p_query_id      BIGINT,
    p_workload_class TEXT
)
LANGUAGE plpgsql AS $$
DECLARE
    v_target_pool TEXT;
    v_statement_timeout TEXT;
    v_work_mem TEXT;
BEGIN
    -- Map classification to resource parameters
    CASE p_workload_class
        WHEN 'OLTP_POINT' THEN
            v_target_pool       := 'pool_oltp_primary';
            v_statement_timeout := '5000';    -- 5 seconds
            v_work_mem          := '4MB';
        WHEN 'OLTP_MODERATE' THEN
            v_target_pool       := 'pool_oltp_primary';
            v_statement_timeout := '30000';   -- 30 seconds
            v_work_mem          := '16MB';
        WHEN 'ANALYTICAL' THEN
            v_target_pool       := 'pool_replica_read';
            v_statement_timeout := '300000';  -- 5 minutes
            v_work_mem          := '256MB';
        WHEN 'BATCH_ETL' THEN
            v_target_pool       := 'pool_batch';
            v_statement_timeout := '3600000'; -- 1 hour
            v_work_mem          := '512MB';
        WHEN 'MAINTENANCE' THEN
            v_target_pool       := 'pool_maintenance';
            v_statement_timeout := '0';       -- no limit
            v_work_mem          := '1GB';
        ELSE
            v_target_pool       := 'pool_oltp_primary';
            v_statement_timeout := '30000';
            v_work_mem          := '16MB';
    END CASE;

    -- Record the mapping for the orchestration agent to pick up
    INSERT INTO workload_routing_map
        (query_id, workload_class, target_pool,
         statement_timeout_ms, work_mem, updated_at)
    VALUES
        (p_query_id, p_workload_class, v_target_pool,
         v_statement_timeout::INT, v_work_mem, now())
    ON CONFLICT (query_id) DO UPDATE SET
        workload_class     = EXCLUDED.workload_class,
        target_pool        = EXCLUDED.target_pool,
        statement_timeout_ms = EXCLUDED.statement_timeout_ms,
        work_mem           = EXCLUDED.work_mem,
        updated_at         = EXCLUDED.updated_at;
END;
$$;
