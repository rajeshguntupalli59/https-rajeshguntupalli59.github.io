Chapter 19: Testing and Validation

Testing and validation are the disciplines that separate a reactive DBA from a proactive one. Whether you're deploying schema changes, tuning queries, upgrading a major version, or migrating data between systems, every action you take on a production database carries risk — and the only way to manage that risk is to test deliberately, validate thoroughly, and build confidence before you commit. This chapter walks through the full spectrum of database testing: from verifying that a schema migration didn't break anything, to stress-testing a new index strategy under load, to confirming that a data migration preserved every row exactly as intended. The goal is not to test for testing's sake, but to build the habits and tooling that let you move fast without breaking things.

---

Why Database Testing Is Different from Application Testing

Most developers understand unit testing and integration testing for application code. Database testing shares some of the same vocabulary but operates under fundamentally different constraints. A database isn't stateless — every test you run against it potentially changes data, which means tests can interfere with each other, leave residue behind, or produce results that only make sense in a specific sequence. A test that passes once may fail the second time because the data it inserted the first time is still there.

The other major difference is that database changes are often irreversible in practice. Dropping a column, truncating a table, or migrating a partition scheme can be done in seconds but may take hours to undo — or may not be undoable at all if your backup window has passed. This asymmetry means the cost of a failed deployment in a database context is much higher than deploying a bad application build, where you roll back in thirty seconds. A DBA who ships untested changes isn't being bold; they're being reckless.

There are four broad categories of database testing that a DBA should be fluent in:

Schema validation ensures that DDL changes — new tables, modified columns, added constraints, dropped indexes — were applied correctly and didn't produce unexpected side effects. Data validation confirms that rows were migrated, transformed, or loaded without loss or corruption. Performance testing measures whether a change improved or degraded query execution. Regression testing verifies that existing queries still behave correctly after a change.

These categories overlap. A new index might satisfy a schema validation check but silently break a query plan that was previously efficient because the planner now chooses it in a context where a sequential scan would have been better. Real-world database testing requires thinking across all four categories simultaneously.

Building a Test Environment That Mirrors Production

No test result is meaningful unless the environment it was produced in resembles production closely enough to matter. This is the most underrated principle in database testing, and violating it is the source of most "it worked in staging" disasters.

The relevant dimensions of similarity are: data volume, data distribution, schema version, configuration settings, and hardware (or resource allocation). You don't necessarily need an exact clone of production — but you need to understand how each dimension of difference might invalidate your test results. A staging database with 10,000 rows cannot tell you how a query will perform against 80 million rows. A staging instance with shared_buffers set to 256MB cannot predict cache hit ratios on a production server with 32GB allocated.

For PostgreSQL, pg_dump and pg_restore are the standard tools for creating a schema-level copy. For a data subset, you can dump specific tables with row filtering. For SQL Server, database backups with restore, or database snapshots, are the typical approach.

PostgreSQL:

-- Create a schema-only copy of the production database for test environment setup
-- Run this from the OS shell, not inside psql
-- pg_dump -h prod-server -U postgres -s -F c -f schema_only.dump mydb

-- Then restore into a test database
-- pg_restore -h test-server -U postgres -d mydb_test schema_only.dump

-- Verify the object count matches production
SELECT
    table_schema,
    COUNT(*) AS table_count
FROM information_schema.tables
WHERE table_type = 'BASE TABLE'
  AND table_schema NOT IN ('pg_catalog', 'information_schema')
GROUP BY table_schema
ORDER BY table_schema;

SQL Server:

-- Restore a production backup to a test server with a new database name
RESTORE DATABASE mydb_test
FROM DISK = 'D:\Backups\mydb_prod.bak'
WITH MOVE 'mydb_data' TO 'D:\TestData\mydb_test.mdf',
     MOVE 'mydb_log'  TO 'D:\TestLogs\mydb_test.ldf',
     NORECOVERY,
     REPLACE,
     STATS = 10;
RESTORE DATABASE mydb_test WITH RECOVERY;

-- Confirm object count matches your expectation
SELECT
    s.name AS schema_name,
    COUNT(*) AS table_count
FROM sys.tables t
JOIN sys.schemas s ON t.schema_id = s.schema_id
GROUP BY s.name
ORDER BY s.name;

One pattern that scales well in organizations with strict data governance requirements is data masking on the test copy. Personally identifiable information (PII) — names, email addresses, credit card numbers — can be obfuscated using deterministic functions that preserve data distribution without exposing real values. PostgreSQL's pgcrypto extension and SQL Server's built-in HASHBYTES or dynamic data masking features both support this. The key requirement is that masked data should retain realistic cardinality and length distributions so that query plans generated during testing remain representative of production behavior.

Configuration parity matters too. If your production PostgreSQL server runs with work_mem = 64MB and enable_hashjoin = on, your test environment should match those settings when you're evaluating query plans. A hash join that's chosen in production but blocked by lower work_mem in staging will make your test results meaningless.

Schema Change Validation

Schema migrations are among the most common — and most dangerous — database operations DBAs perform. A migration script that looks correct can silently truncate data (adding a NOT NULL constraint to a column that has nulls), cause lock contention (adding a column with a default value in older PostgreSQL versions), or break application queries (renaming a column that wasn't fully searched in the codebase).

The validation process for a schema change should cover three phases: pre-migration checks, post-migration verification, and rollback readiness.

Pre-migration checks answer questions like: Are there any rows that would violate the new constraint? Are there any dependent objects that will be affected by this change? Is there enough disk space for a table rewrite? Will this operation lock the table?

PostgreSQL:

-- Before adding a NOT NULL constraint, check for nulls
SELECT COUNT(*) AS null_count
FROM orders
WHERE customer_id IS NULL;

-- Before adding a unique constraint, identify duplicates
SELECT customer_id, COUNT(*) AS duplicate_count
FROM orders
GROUP BY customer_id
HAVING COUNT(*) > 1
ORDER BY duplicate_count DESC;

-- Check dependent views before dropping or altering a column
SELECT
    dependent_ns.nspname AS dependent_schema,
    dependent_view.relname AS dependent_view,
    source_ns.nspname AS source_schema,
    source_table.relname AS source_table,
    pg_attribute.attname AS column_name
FROM pg_depend
JOIN pg_rewrite ON pg_depend.objid = pg_rewrite.oid
JOIN pg_class AS dependent_view ON pg_rewrite.ev_class = dependent_view.oid
JOIN pg_class AS source_table ON pg_depend.refobjid = source_table.oid
JOIN pg_attribute ON pg_depend.refobjid = pg_attribute.attrelid
    AND pg_depend.refobjsubid = pg_attribute.attnum
JOIN pg_namespace dependent_ns ON dependent_ns.oid = dependent_view.relnamespace
JOIN pg_namespace source_ns ON source_ns.oid = source_table.relnamespace
WHERE source_table.relname = 'orders'
  AND pg_attribute.attname = 'customer_id'
  AND source_ns.nspname = 'public';

SQL Server:

-- Before adding a NOT NULL constraint, check for nulls
SELECT COUNT(*) AS null_count
FROM dbo.orders
WHERE customer_id IS NULL;

-- Before adding a unique constraint, identify duplicates
SELECT customer_id, COUNT(*) AS duplicate_count
FROM dbo.orders
GROUP BY customer_id
HAVING COUNT(*) > 1
ORDER BY duplicate_count DESC;

-- Check dependent objects before altering a column
SELECT
    referencing_schema_name,
    referencing_entity_name,
    referencing_class_desc
FROM sys.dm_sql_referencing_entities('dbo.orders', 'OBJECT')
ORDER BY referencing_entity_name;

Post-migration verification is the step that most teams skip because they're in a hurry after the migration completes. This is a mistake. A post-migration checklist should confirm that all expected objects exist, constraints are active and valid, indexes were rebuilt if necessary, and the row count is what you expected.

In PostgreSQL, adding a foreign key constraint with large tables can create an index entry in pg_constraint with convalidated = false if you used NOT VALID during creation — meaning the constraint was added but hasn't been checked against existing rows yet. You need to explicitly run VALIDATE CONSTRAINT afterward, and you should verify this in your post-migration checks.

PostgreSQL:

-- Post-migration: confirm all constraints on a table are valid
SELECT
    conname AS constraint_name,
    contype AS constraint_type,
    convalidated AS is_validated
FROM pg_constraint
JOIN pg_class ON pg_class.oid = pg_constraint.conrelid
WHERE pg_class.relname = 'orders'
ORDER BY contype, conname;

SQL Server:

-- Post-migration: check for untrusted (not-validated) constraints
SELECT
    t.name AS table_name,
    c.name AS constraint_name,
    c.type_desc,
    c.is_not_trusted
FROM sys.check_constraints c
JOIN sys.tables t ON c.parent_object_id = t.object_id
WHERE c.is_not_trusted = 1
UNION ALL
SELECT
    t.name,
    fk.name,
    'FOREIGN_KEY_CONSTRAINT',
    fk.is_not_trusted
FROM sys.foreign_keys fk
JOIN sys.tables t ON fk.parent_object_id = t.object_id
WHERE fk.is_not_trusted = 1
ORDER BY table_name, constraint_name;

Rollback readiness means you have a clear, tested path back to the pre-migration state before you apply the change. This might be a backup taken immediately before the migration, a DDL script that reverses the change, or a transactional migration script wrapped in a BEGIN/ROLLBACK block that you tested manually first to confirm it compiles and executes cleanly. The worst time to write a rollback script is after a migration has gone wrong.

Data Migration Validation and Row-Level Reconciliation

Moving data between tables, databases, or systems is one of the highest-risk operations in database management. Silent data loss — where the migration completes without an error but rows or values went missing — is far more dangerous than a migration that fails loudly. A failed migration is a known problem. Silent data loss might not surface until weeks later when a customer reports a discrepancy.

The foundation of data migration validation is reconciliation: comparing source and target at multiple levels of granularity to confirm that the data matches. The hierarchy of reconciliation checks, from coarse to fine, is: total row count, aggregate value comparison, checksum or hash comparison, and finally row-by-row differential queries.

Start with row counts. This catches bulk failures — a truncated load, a filter that accidentally excluded rows, a partition that wasn't included in the migration script. Then move to aggregate checks: sum of key financial columns, count of distinct values in high-cardinality columns, min and max of date ranges. A migration that has the right row count but wrong aggregates almost always indicates a data transformation bug.

PostgreSQL:

-- Row count reconciliation across source and target tables
SELECT 'source' AS location, COUNT(*) AS row_count FROM source_schema.orders
UNION ALL
SELECT 'target', COUNT(*) FROM target_schema.orders;

-- Aggregate reconciliation for financial integrity
SELECT
    'source' AS location,
    SUM(order_total)    AS total_revenue,
    COUNT(DISTINCT customer_id) AS unique_customers,
    MIN(order_date)     AS earliest_order,
    MAX(order_date)     AS latest_order
FROM source_schema.orders
UNION ALL
SELECT
    'target',
    SUM(order_total),
    COUNT(DISTINCT customer_id),
    MIN(order_date),
    MAX(order_date)
FROM target_schema.orders;

SQL Server:

-- Row count reconciliation
SELECT 'source' AS location, COUNT(*) AS row_count FROM source_schema.dbo.orders
UNION ALL
SELECT 'target', COUNT(*) FROM target_schema.dbo.orders;

-- Aggregate reconciliation
SELECT
    'source' AS location,
    SUM(order_total)    AS total_revenue,
    COUNT(DISTINCT customer_id) AS unique_customers,
    MIN(order_date)     AS earliest_order,
    MAX(order_date)     AS latest_order
FROM source_schema.dbo.orders
UNION ALL
SELECT
    'target',
    SUM(order_total),
    COUNT(DISTINCT customer_id),
    MIN(order_date),
    MAX(order_date)
FROM target_schema.dbo.orders;

When aggregate checks pass but you still suspect inconsistencies, hash-based row comparison is the next step. By hashing all columns of a row into a single checksum value and comparing checksums between source and target, you can identify rows where the data exists in both places but doesn't match exactly. This technique catches subtle issues like encoding differences, trailing whitespace, timezone normalization problems, and type conversion errors that change a value without obviously breaking the migration.

For row-level differential queries, use an EXCEPT or MINUS based approach (PostgreSQL supports EXCEPT; SQL Server supports EXCEPT as well). This approach finds rows that exist in the source but not in the target, and vice versa. On large tables, this can be expensive — run it on a sampled subset or partition it by date range if the full comparison isn't feasible within your maintenance window.

One production pattern worth highlighting is the dual-write validation technique used during zero-downtime migrations. While migrating data from an old schema to a new one, the application writes to both simultaneously. After a stabilization period, you run reconciliation queries to confirm the two tables agree on every row touched during the overlap period. Once validated, you cut over reads to the new table and stop writes to the old one. The validation queries become part of the migration runbook, not an afterthought.

Query Regression Testing and Execution Plan Validation

Schema changes and index changes can invalidate query execution plans in ways that are completely invisible to the application. A query that ran in 50 milliseconds before an index was dropped might now run in 8 seconds. The application doesn't know — it's still issuing the same SQL. If nobody checks, this kind of regression can degrade production for days before anyone investigates.

Query regression testing means capturing the expected behavior of your critical queries before a change and comparing it against observed behavior after the change. The two dimensions you care about most are execution plan shape (which join type, which index, which scan method was chosen) and execution time.

Before any significant schema or index change, capture the query plans and timing for your top 20–30 queries by frequency and resource consumption. You can identify these from pg_stat_statements in PostgreSQL or from the Query Store in SQL Server. Run each of them explicitly in a controlled test environment before and after the change.

PostgreSQL:

-- Capture a query's execution plan before a change
-- Use EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) for full detail
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT
    o.order_id,
    o.customer_id,
    o.order_total,
    c.email
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id
WHERE o.order_date >= '2024-01-01'
  AND o.status = 'PENDING';

SQL Server:

-- Capture a query's execution plan before a change
-- Use SET STATISTICS IO and TIME for runtime metrics
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

SELECT
    o.order_id,
    o.customer_id,
    o.order_total,
    c.email
FROM dbo.orders o
JOIN dbo.customers c ON o.customer_id = c.customer_id
WHERE o.order_date >= '2024-01-01'
  AND o.status = 'PENDING';

SET STATISTICS IO OFF;
SET STATISTICS TIME OFF;
-- Also capture the actual execution plan in SSMS via Ctrl+M
-- Or query sys.dm_exec_query_stats for historical plans

For PostgreSQL, auto_explain is a valuable extension for capturing plans of slow queries automatically during a test run without manually running each one. Load it as a shared library, set auto_explain.log_min_duration = 100 (in milliseconds), and run your workload. Plans for all queries exceeding the threshold are written to the PostgreSQL log.

SQL Server's Query Store serves a similar purpose with much more tooling around it. After enabling Query Store on your test database, running your workload, and then applying the schema change, you can compare before-and-after plans directly in the Query Store views or in SQL Server Management Studio's built-in Query Store UI. If a plan regression is detected, Query Store even allows you to force the old plan while you investigate.

One class of regression that catches DBAs off guard is plan instability after statistics updates. When you load a large dataset, run ANALYZE (PostgreSQL) or UPDATE STATISTICS (SQL Server), and then test queries, the planner now has better information and may choose different plans than before the statistics update. Whether those new plans are better or worse depends on the query and the data. Always update statistics in your test environment before benchmarking, and always update statistics in production after a large data load — but recognize that it can change behavior.

Load Testing and Stress Testing Under Realistic Concurrency

A query that runs beautifully in isolation — one session, no competing load — may behave entirely differently when fifty application threads are hitting the database simultaneously. Lock contention, buffer pool competition, connection overhead, and autovacuum interference are all concurrency effects that only appear under load. Testing in isolation is necessary but not sufficient; you need to test under conditions that approximate real production concurrency.

Load testing a database has two goals: confirming that performance targets are met at expected workload levels (load testing proper), and discovering the failure mode — what happens when the system is pushed beyond its capacity (stress testing). Both are useful. Knowing your system handles 500 transactions per second comfortably is good. Knowing it degrades gracefully at 800 and starts rejecting connections at 1,200 is essential for capacity planning.

For PostgreSQL, pgbench is the built-in load testing tool. It ships with a standard TPC-B-like workload, but its real value is that it accepts custom SQL scripts so you can drive it with your actual application queries. A typical pre-deployment validation run uses a custom script containing the top five write paths in the application, run at 50 concurrent clients for ten minutes, and measures transactions per second and average latency.

For SQL Server, tools like HammerDB, Distributed Replay, and the Query Store's workload simulation capabilities are commonly used. For teams with existing workloads captured from production, SQL Server's Distributed Replay can replay a real workload trace against a test instance, which is much more representative than synthetic benchmarks.

The metrics you should capture during a load test are: transactions per second, average and P99 query latency, lock wait time, buffer cache hit ratio, and connection pool utilization. A change that improves average latency but increases P99 from 200ms to 2 seconds has created a worse user experience, even though the average looks better.

One specific thing to watch in PostgreSQL during load tests is autovacuum activity. Autovacuum is triggered by write load, so a load test that drives a high INSERT/UPDATE rate will also trigger autovacuum on the affected tables. If autovacuum can't keep up, table bloat increases and query plans may degrade. Monitor pg_stat_user_tables for n_dead_tup growing during your load test — a sign that autovacuum is falling behind.

In SQL Server, watch for PAGELATCH and LCK_M waits in sys.dm_os_wait_stats during load tests. A sudden increase in lock waits that doesn't appear in single-session testing is a signal of a concurrency problem in your query design or transaction scope — something that would have been invisible without load testing.

---

Key Takeaways

  • → **Test environments must mirror production in meaningful dimensions** — data volume, configuration settings, and schema version — or your test results will mislead you. A staging environment with 0.1% of production data cannot predict production query plans or performance.
  • → **Schema migration validation is a three-phase process**: pre-migration checks to catch constraint violations and dependencies before they cause errors, post-migration verification to confirm all objects are in the expected state, and rollback readiness to ensure you have a tested path back if something goes wrong.
  • → **Data migration validation requires multiple reconciliation layers** — row counts catch bulk failures, aggregate checks catch transformation errors, and hash or differential comparisons catch subtle value-level discrepancies. Silent data loss is more dangerous than a migration that fails loudly.
  • → **Query regression testing must happen before and after every significant schema or index change.** Capture execution plans and timing from `pg_stat_statements` or SQL Server Query Store for your critical queries, and compare them explicitly rather than assuming the planner will make the same choices.
  • → **Load testing reveals a class of problems that single-session testing cannot** — lock contention, autovacuum lag, P99 latency spikes, and connection pool exhaustion are all concurrency effects. Validate performance under realistic concurrency levels before deploying changes to production systems.
