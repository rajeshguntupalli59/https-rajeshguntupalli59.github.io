Chapter 17: API Interactions

Modern applications rarely talk to a database directly through a query tool. Instead, they interact through layers of code — REST APIs, GraphQL endpoints, microservices, and background workers — all of which ultimately translate user actions into SQL. As a DBA, understanding how these API layers communicate with your database is no longer optional. The queries that arrive at your server carry context from the calling application: how connections are managed, how parameters are passed, how transactions are scoped, and how errors flow back to the client. This chapter covers what DBAs need to know about API-to-database interactions, from connection patterns and parameterized queries to bulk operations, pagination, and the failure modes that only reveal themselves under production load.

---

How Applications Connect: Connection Pooling and the DBA's Role

Every API call that needs data starts with a connection. In a naive implementation, the application opens a new database connection for each request and closes it when done. This works fine for a single developer testing locally and falls apart completely when a thousand users hit the service simultaneously. Establishing a PostgreSQL or SQL Server connection involves TCP handshaking, authentication, session initialization, and memory allocation on the server — often 5–50ms of overhead that adds up fast.

Connection poolers sit between the application and the database, maintaining a pool of pre-established connections that can be handed to incoming requests. PgBouncer is the dominant solution in the PostgreSQL ecosystem. SQL Server applications typically rely on ADO.NET connection pooling built into the .NET runtime, or external proxies like ProxySQL in heterogeneous stacks.

From the DBA's perspective, the first thing to know is how many connections your server is actually handling. In PostgreSQL, pg_stat_activity shows every backend process and what it is doing. In SQL Server, sys.dm_exec_sessions provides equivalent visibility. Monitoring these views regularly reveals whether a connection pool is sized correctly, whether connections are leaking, or whether a runaway service is spawning connections without releasing them.

PostgreSQL:

-- See all active connections, their state, and wait events
SELECT
    pid,
    usename,
    application_name,
    client_addr,
    state,
    wait_event_type,
    wait_event,
    now() - state_change AS time_in_state,
    left(query, 80) AS current_query
FROM pg_stat_activity
WHERE backend_type = 'client backend'
ORDER BY time_in_state DESC NULLS LAST;

SQL Server:

-- See all active sessions, their status, and what they're waiting on
SELECT
    s.session_id,
    s.login_name,
    s.program_name,
    s.client_interface_name,
    s.status,
    s.last_request_start_time,
    wt.wait_type,
    wt.wait_duration_ms,
    LEFT(qt.text, 80) AS current_query
FROM sys.dm_exec_sessions s
LEFT JOIN sys.dm_exec_requests r ON s.session_id = r.session_id
LEFT JOIN sys.dm_os_waiting_tasks wt ON r.task_address = wt.task_address
OUTER APPLY sys.dm_exec_sql_text(r.sql_handle) qt
WHERE s.is_user_process = 1
ORDER BY s.last_request_start_time DESC;

Understanding the pool mode matters too. PgBouncer offers three modes: session pooling (a connection is held for the full client session), transaction pooling (the server connection is returned to the pool after each transaction), and statement pooling (after each statement). Transaction pooling is the most efficient for stateless API services but has restrictions — prepared statements behave differently, and SET commands that modify session-level configuration can leak state between requests unless the pooler resets the session. DBAs who deploy PgBouncer in transaction mode and then receive complaints about inconsistent behavior from developers need to check whether the application is relying on session-level settings that aren't surviving across requests.

One thing worth tracking is the maximum connections setting. In PostgreSQL, each connection spawns a separate OS process that consumes shared memory and file descriptors. Setting max_connections to 1000 doesn't mean 1000 concurrent connections are free — each one costs real RAM. A rough rule of thumb is 5–10MB per connection in a moderately active system, meaning 500 connections can cost 2.5–5GB just in process overhead before a single query runs. PgBouncer's value is that it lets you maintain a small number of actual server connections while serving hundreds or thousands of application threads.

SQL Server uses a thread-based model rather than a process-per-connection model, making it more tolerant of high connection counts, but connection sprawl still causes contention on internal memory structures and the thread scheduler. Keeping connection counts disciplined matters regardless of platform.

Parameterized Queries: Safety, Performance, and What DBAs Actually See

The most important habit any API developer can have is using parameterized queries. As a DBA, you care about parameterization for three separate reasons: security, plan reuse, and query diagnostics.

The security argument is well known — string interpolation allows SQL injection. The performance argument is less appreciated but equally important. When a query arrives at PostgreSQL or SQL Server with a hardcoded literal, the query planner treats each unique literal as a potentially unique query and may parse and plan it separately. When queries are parameterized, the planner can recognize that the structure is the same and reuse a cached execution plan, saving planning overhead on every subsequent execution.

In PostgreSQL, the server uses the pg_stat_statements extension to track query performance. Parameterized queries from applications using server-side prepared statements appear with their parameter placeholders intact, making it easy to identify slow query patterns. Ad-hoc queries with literal values get normalized by pg_stat_statements anyway — it replaces literals with $1, $2, etc. — but server-prepared statements go further by actually caching the plan on the connection.

In SQL Server, the equivalent mechanism is plan caching in the procedure cache. Parameterized queries submitted through sp_executesql are the T-SQL equivalent of prepared statements, and they allow the server to match the query to an existing cached plan by its hash.

PostgreSQL:

-- Check pg_stat_statements for most time-consuming parameterized queries
-- (requires pg_stat_statements extension in postgresql.conf)
SELECT
    left(query, 100)                          AS query_pattern,
    calls,
    round(total_exec_time::numeric, 2)        AS total_ms,
    round(mean_exec_time::numeric, 2)         AS avg_ms,
    round(stddev_exec_time::numeric, 2)       AS stddev_ms,
    rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 20;

SQL Server:

-- Check plan cache for most expensive queries by total CPU
SELECT TOP 20
    qs.execution_count,
    qs.total_worker_time / 1000             AS total_cpu_ms,
    qs.total_worker_time / qs.execution_count / 1000 AS avg_cpu_ms,
    qs.total_elapsed_time / 1000            AS total_elapsed_ms,
    LEFT(qt.text, 100)                      AS query_pattern,
    qp.query_plan
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) qt
CROSS APPLY sys.dm_exec_query_plan(qs.plan_handle) qp
ORDER BY qs.total_worker_time DESC;

One subtle issue DBAs encounter with parameterized queries is parameter sniffing in SQL Server. The first time a parameterized query executes, the optimizer generates a plan based on the actual parameter values provided. If that first execution happens to use a highly selective value, the cached plan will assume a small result set and use an index seek. When the same query later runs with a parameter that matches millions of rows, it still uses the seek plan, which is now catastrophically inefficient. This is one of the more common causes of intermittent performance problems in production SQL Server environments.

The PostgreSQL equivalent is the generic plan problem. For extended query protocol prepared statements, PostgreSQL initially generates a custom plan (using the actual parameter values) for the first five executions, then switches to a generic plan if the estimated cost is comparable. If the generic plan is substantially worse for some parameter values, you can influence this behavior with the plan_cache_mode setting (available from PostgreSQL 12).

APIs built on ORMs (Object-Relational Mappers like Hibernate, Entity Framework, or SQLAlchemy) typically parameterize queries automatically, which is good for security but creates its own diagnostics challenges. The SQL that arrives at the server is often unrecognizable from the ORM code that generated it — aliased tables, redundant joins, and nested subqueries are common. When a developer tells you "this endpoint is slow," your first step is capturing the actual SQL hitting the database, not the ORM method call.

Transaction Scope and API Design: Where Things Go Wrong

Transactions in API contexts fail in two distinct directions: they are either too long or too short. Both cause production problems, and as a DBA you are usually the one who discovers the evidence after something has gone wrong.

Long transactions occur when an API endpoint opens a transaction, does some work, and then makes a network call (to another service, to an external payment provider, to a file system) while the transaction is still open. If the network call takes 3 seconds and the API handles 200 concurrent requests, you can have 200 transactions open simultaneously, each holding locks on rows they have already touched. In PostgreSQL, long-running transactions also block autovacuum from reclaiming dead tuples, which leads to table bloat and eventual transaction ID wraparound — one of the more severe operational hazards in PostgreSQL.

You can identify long-running transactions and their oldest open transaction age directly from the catalog:

PostgreSQL:

-- Find long-running transactions and what they're waiting on
SELECT
    pid,
    usename,
    application_name,
    now() - xact_start                  AS transaction_age,
    now() - query_start                 AS query_age,
    state,
    wait_event_type,
    wait_event,
    left(query, 120)                    AS last_query
FROM pg_stat_activity
WHERE xact_start IS NOT NULL
  AND now() - xact_start > interval '30 seconds'
ORDER BY transaction_age DESC;

SQL Server:

-- Find long-running transactions and associated blocking
SELECT
    at.transaction_id,
    at.name                             AS transaction_name,
    at.transaction_begin_time,
    DATEDIFF(SECOND, at.transaction_begin_time, GETDATE()) AS age_seconds,
    s.session_id,
    s.login_name,
    s.program_name,
    LEFT(qt.text, 120)                  AS last_query
FROM sys.dm_tran_active_transactions at
JOIN sys.dm_tran_session_transactions st ON at.transaction_id = st.transaction_id
JOIN sys.dm_exec_sessions s ON st.session_id = s.session_id
LEFT JOIN sys.dm_exec_requests r ON s.session_id = r.session_id
OUTER APPLY sys.dm_exec_sql_text(r.sql_handle) qt
WHERE DATEDIFF(SECOND, at.transaction_begin_time, GETDATE()) > 30
ORDER BY age_seconds DESC;

Short transactions (or rather, the lack of explicit transactions in multi-step operations) are equally problematic. An API that performs a debit from one account and a credit to another as two separate, autocommitted statements risks ending up in a state where the debit succeeds and the credit fails — a real data integrity problem. ORMs with lazy loading or deferred constraint checking can also execute statements in an order that surprises developers if they have not thought carefully about transaction boundaries.

The correct design principle is that a transaction should encompass exactly the database operations that must succeed or fail together — nothing more, nothing less. In practice, this means API handlers should open a transaction, complete all database work, and commit before doing anything that might block or fail externally. This is standard advice that gets violated constantly under deadline pressure, and the DBA is usually the one producing the evidence of what happened.

Savepoints are a useful but underused feature in API-driven transaction design. They allow an application to mark a point within a transaction to which it can roll back partially, without aborting the entire transaction. PostgreSQL supports savepoints natively. SQL Server calls them partial rollbacks using SAVE TRANSACTION.

One nuance specific to connection pools operating in transaction mode (PgBouncer being the most common example): if an application begins a transaction and then crashes or loses its connection before committing, the pooler will roll back the transaction when it reclaims the connection — but this behavior depends on pool configuration, and in some setups orphaned transactions can hold locks until the idle connection timeout fires. Building retry logic and monitoring for blocked queries are both essential defenses.

Bulk Operations and Efficient Data Loading via APIs

A common pattern in API-driven systems is the batch endpoint: receive a payload of 100, 1000, or 10,000 records from an upstream service and persist them in the database. How you implement the persistence layer for these operations has an enormous effect on throughput, lock contention, and recoverability.

The worst approach is a loop: iterate over the records in application code and execute one INSERT per row. This results in one network round-trip per row, one transaction per row if autocommit is on, and minimal throughput. A service loading 10,000 records one at a time with a 1ms round-trip time will spend at least 10 seconds just on network overhead.

The better approach is multi-row INSERT, which batches multiple rows into a single statement. Most drivers support this transparently when you use their batch insert APIs. Both PostgreSQL and SQL Server handle multi-row inserts efficiently.

For very large volumes, PostgreSQL's COPY command is the fastest ingestion mechanism available. It bypasses much of the per-row overhead of individual INSERT statements and is optimized for streaming large datasets. On the SQL Server side, BULK INSERT or the SqlBulkCopy API in .NET provides equivalent bulk load capability.

PostgreSQL:

-- Multi-row INSERT with RETURNING for API response building
INSERT INTO orders (customer_id, product_id, quantity, created_at)
VALUES
    ($1, $2, $3, NOW()),
    ($4, $5, $6, NOW()),
    ($7, $8, $9, NOW())
RETURNING order_id, created_at;

-- For very large loads, COPY from a CSV or program pipe is fastest:
-- COPY orders (customer_id, product_id, quantity, created_at)
-- FROM STDIN WITH (FORMAT csv, HEADER false);

SQL Server:

-- Multi-row INSERT with OUTPUT clause for API response building
INSERT INTO orders (customer_id, product_id, quantity, created_at)
OUTPUT INSERTED.order_id, INSERTED.created_at
VALUES
    (@cust1, @prod1, @qty1, GETDATE()),
    (@cust2, @prod2, @qty2, GETDATE()),
    (@cust3, @prod3, @qty3, GETDATE());

-- For large bulk loads, use SqlBulkCopy from .NET or:
-- BULK INSERT orders
-- FROM 'C:\data\orders.csv'
-- WITH (FORMAT = 'CSV', FIRSTROW = 2);

An important consideration for bulk INSERT operations in production is what happens when some rows violate constraints. A batch of 1,000 records where row 743 has a duplicate key will, by default, fail the entire statement and insert nothing. Applications need a strategy for handling partial failures.

PostgreSQL offers INSERT ... ON CONFLICT DO NOTHING or ON CONFLICT DO UPDATE (the UPSERT pattern), which handles duplicate key scenarios gracefully without aborting the entire batch. SQL Server provides MERGE for similar upsert semantics, though MERGE has well-documented edge cases around race conditions and should be used carefully in high-concurrency environments. A safer SQL Server pattern for many upsert scenarios is a targeted UPDATE followed by an INSERT WHERE NOT EXISTS.

Index maintenance during bulk loads deserves attention. If you are loading data into a table with many indexes, each inserted row triggers index updates. For very large loads into a table where you control the process (a data warehouse refresh, for instance), dropping non-essential indexes before the load and rebuilding them afterward is often faster than loading with indexes in place. For OLTP tables where you cannot drop indexes, loading in smaller batches during off-peak hours and monitoring lock contention is a more realistic approach.

Pagination, Cursor-Based Reads, and Result Set Design

APIs that expose query results to clients almost always need pagination. Returning all 500,000 rows from an orders table in a single API response is not a design option. The two dominant approaches — offset-based pagination and cursor-based pagination — have very different performance profiles, and the choice matters as table sizes grow.

Offset-based pagination uses LIMIT and OFFSET (PostgreSQL) or OFFSET FETCH (SQL Server). It is simple to implement and easy to understand: the client requests page 1, page 2, page 3. The problem is that OFFSET N is not free. The database must scan and discard the first N rows to reach the page you want. Page 1 is fast. Page 1000 of a 1,000-row page size requires the database to process 999,000 rows before returning anything. For many API use cases — mobile feeds, admin dashboards, reports — users rarely navigate past the first few pages, and offset pagination is perfectly adequate. For deep pagination scenarios, it breaks down.

Cursor-based pagination uses a stable column value (typically a primary key or a timestamp + ID combination) as an anchor point. Instead of "give me rows 10,000 through 10,025," the query says "give me the 25 rows where the ID is greater than the last ID I saw." This is consistently fast regardless of how deep into the dataset you are, because the database uses an index seek rather than a scan-then-skip.

PostgreSQL:

-- Offset-based pagination (simple, but degrades at high offsets)
SELECT order_id, customer_id, total_amount, created_at
FROM orders
WHERE status = $1
ORDER BY created_at DESC, order_id DESC
LIMIT $2
OFFSET $3;

-- Cursor-based pagination (keyset pagination — stays fast at any depth)
SELECT order_id, customer_id, total_amount, created_at
FROM orders
WHERE status = $1
  AND (created_at, order_id) < ($2, $3)   -- ($2=last_created_at, $3=last_order_id)
ORDER BY created_at DESC, order_id DESC
LIMIT $4;

SQL Server:

-- Offset-based pagination using OFFSET-FETCH
SELECT order_id, customer_id, total_amount, created_at
FROM orders
WHERE status = @status
ORDER BY created_at DESC, order_id DESC
OFFSET @offset ROWS
FETCH NEXT @page_size ROWS ONLY;

-- Cursor-based (keyset) pagination
SELECT TOP (@page_size)
    order_id, customer_id, total_amount, created_at
FROM orders
WHERE status = @status
  AND (created_at < @last_created_at
       OR (created_at = @last_created_at AND order_id < @last_order_id))
ORDER BY created_at DESC, order_id DESC;

The keyset approach has a meaningful limitation: you cannot jump to an arbitrary page number. The client must walk forward sequentially, using the last-seen values as its bookmark. This is exactly the right behavior for infinite scroll UIs and API consumers processing a full dataset in order, but it does not suit UIs that show numbered page links.

From the DBA's perspective, there is another pagination-related problem that surfaces in production: the count query. Many APIs return a total record count alongside each page so the UI can display "Page 3 of 47." This requires a COUNT(*) over the full filtered dataset on every page request. For large tables with complex WHERE clauses, this count query can be more expensive than the page query itself. Partial solutions include caching the count for short periods, using pg_class.reltuples or SQL Server's sys.partitions for approximate counts, or simply omitting exact counts and using "has more results" indicators instead.

A related issue is sort stability. If rows can be inserted or deleted between page requests, an ORDER BY created_at DESC may produce different row orderings on subsequent pages. Rows can be skipped or appear twice if a new row is inserted at the boundary between pages. Adding the primary key as a secondary sort column (as shown in the examples above) makes the sort deterministic and ensures stable pagination even under concurrent writes.

Error Handling, Retries, and the API-Database Contract Under Pressure

Production API systems fail in ways that development environments never simulate. Network partitions, connection pool exhaustion, deadlocks, serialization failures, and query timeouts all arrive at runtime, and how the application handles them determines whether end users see a graceful degradation or a cascade failure.

From the database side, errors come with codes that carry meaning. PostgreSQL uses SQLSTATE codes — a five-character standard inherited from ANSI SQL. SQL Server uses error numbers and severity levels. API developers often treat all database errors as equivalent "something went wrong" conditions and return a 500 to the client. As a DBA, you want developers to distinguish between categories of errors because the correct response differs.

A serialization failure (SQLSTATE 40001 in PostgreSQL, which occurs when using REPEATABLE READ or SERIALIZABLE isolation levels and the transaction conflicts with a concurrent one) is a signal to retry the transaction, not to fail it permanently. The database is telling the application: "Your transaction would have produced an incorrect result due to concurrent activity, so I aborted it. Try again." Applications that catch this error and retry with exponential backoff will succeed. Applications that propagate it as an error will confuse users.

A deadlock (SQLSTATE 40P01 in PostgreSQL, error 1205 in SQL Server) is similar — it is a recoverable condition that indicates two transactions were waiting on each other's locks. The database kills one of them and the application should retry. Frequent deadlocks indicate that either the order of lock acquisition is inconsistent across transactions (a design problem) or that contention on hot rows is high (a schema or indexing problem).

A connection pool exhaustion error is a different category entirely. If PgBouncer or the ADO.NET pool cannot hand a connection to the caller, the request should be rejected quickly rather than queued indefinitely, because queuing without a timeout will cause the application to accumulate threads or goroutines waiting for connections, eventually exhausting other resources. The correct response is a 503 Service Unavailable with a Retry-After header, not a 500.

Statement timeouts are an important tool for protecting the database from runaway queries originating from API calls. PostgreSQL's statement_timeout can be set at the session, role, or database level. When a query exceeds the timeout, it is terminated with a clear error. SQL Server uses QUERY_GOVERNOR_COST_LIMIT or the lock_timeout and query_timeout settings at the connection level, and application drivers (JDBC, ADO.NET) typically expose a CommandTimeout property.

Setting appropriate timeouts is a balancing act. Too tight, and legitimate slow queries start failing. Too loose, and a slow query from a misbehaving API endpoint monopolizes connections and degrades the whole system. A common production approach is to set a conservative default (say, 30 seconds at the role level for API service accounts) and override it upward for specific long-running operations like report generation, rather than setting a single permissive timeout for everything.

For teams adopting microservices, the concept of circuit breakers in the API layer has a direct relationship to database health. A circuit breaker monitors the error rate from the database and, when it exceeds a threshold, stops sending requests for a cooldown period. This prevents a struggling database — one that is under lock contention or I/O pressure — from being hammered with retry storms that make recovery slower. This is application-level behavior, but DBAs who understand it can collaborate more effectively with developers when diagnosing cascading failures.

Finally, observability: the most useful thing an API service can do from the DBA's perspective is propagate a correlation ID from the HTTP request into the database session. PostgreSQL's application_name connection parameter and SQL Server's sp_set_session_context can both carry arbitrary string metadata that appears in the server's activity views. When an alert fires at 2 AM and you need to know which API endpoint is generating the slow queries, a correlation ID in the session context is the difference between a five-minute diagnosis and a two-hour investigation.
