Chapter 4 of 40 AI-Native Apps on PostgreSQL

Chapter 4: Designing Schemas for AI Workloads

The database schema is where an AI application either holds together or falls apart under pressure. Most AI application tutorials skip past schema design entirely, jumping straight to API calls and vector searches, leaving developers to discover the hard way that a flat embeddings table and a chat_history JSON blob won't survive a real production deployment. This chapter builds the foundational schemas that underpin every AI-native application pattern covered in this book — conversation memory, document ingestion, embedding storage, agent state tracking, and audit logging. You'll see how PostgreSQL's combination of relational integrity, JSONB flexibility, vector support via pgvector, row-level security, and change notification mechanisms gives you a database layer that purpose-built vector stores simply cannot match.

---

The Architecture of an AI-Native Schema

Before writing a single CREATE TABLE statement, it pays to understand what makes AI workloads different from traditional OLTP schemas.

A conventional application schema is primarily transactional: users place orders, orders contain line items, payments reference orders. The relationships are relatively stable, the data types are well-understood, and queries are largely predictable. An AI application introduces several complications that stress-test schema assumptions:

Variable-length unstructured content. LLM inputs and outputs are long strings. Documents can be megabytes of raw text. You need to store them efficiently without killing your page cache.

High-dimensional vectors. Embedding models produce float arrays of 768 to 3072 dimensions. These vectors need to live close to the metadata they describe so that hybrid search — combining vector similarity with relational filters — doesn't require expensive cross-table joins at query time.

Temporal conversation state. Unlike a stateless web request, an AI conversation has memory. You need to efficiently retrieve the last N turns of a conversation, apply token-budget constraints, and support branching or rollback of conversation threads.

Schema evolution at runtime. LLM applications often discover new metadata requirements after deployment — you realize mid-sprint that you need to track the chunk_overlap parameter used during ingestion, or the temperature setting used for a specific response. JSONB columns let you evolve metadata without migrations, while structured columns enforce the invariants you actually need to query.

Audit and observability requirements. In production, you need to know exactly what prompt was sent, what model responded, how many tokens were consumed, and what latency was experienced. This is both an operational requirement (for cost tracking) and a product requirement (for debugging hallucinations).

The schemas in this chapter address all of these concerns. They are designed to work together as a unified system, not as isolated tables.

Core Table Foundations: Users, Sessions, and Conversations

Every AI application that involves users needs a clean foundation that ties identity to conversation state. This sounds obvious, but getting it wrong early means painful migrations later.

The conversation schema needs to support:

  • → Multiple users, each with multiple conversation threads
  • → Threads that can be paused and resumed across sessions
  • → Message-level metadata (model used, latency, token counts, error states)
  • → The ability to reconstruct exactly what the model was sent (the full prompt, not just the user message)
    • -- Core AI application schema foundations
-- Run this in order; dependencies matter

CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS pg_trgm;  -- for text search on conversation content

-- Users: keep this lean; your app likely has richer user data elsewhere
CREATE TABLE ai_users (
    user_id         UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    external_id     TEXT UNIQUE NOT NULL,  -- your app's user ID
    display_name    TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    preferences     JSONB NOT NULL DEFAULT '{}'::jsonb,
    -- e.g. {"default_model": "claude-3-5-sonnet", "language": "en", "tone": "concise"}
    is_active       BOOLEAN NOT NULL DEFAULT TRUE
);

-- Conversation threads: one thread = one persistent context window
CREATE TABLE conversations (
    conversation_id  UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    user_id          UUID NOT NULL REFERENCES ai_users(user_id) ON DELETE CASCADE,
    title            TEXT,  -- auto-generated or user-provided
    model_id         TEXT NOT NULL,  -- e.g. 'claude-3-5-sonnet-20241022'
    system_prompt    TEXT,  -- the system prompt for this thread
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    archived_at      TIMESTAMPTZ,
    metadata         JSONB NOT NULL DEFAULT '{}'::jsonb,
    -- e.g. {"tags": ["support"], "source": "web", "rag_collection": "docs_v2"}
    token_count      INTEGER NOT NULL DEFAULT 0,  -- running total for budget tracking
    message_count    INTEGER NOT NULL DEFAULT 0
);

-- Messages: the core of conversation memory
CREATE TABLE messages (
    message_id       UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    conversation_id  UUID NOT NULL REFERENCES conversations(conversation_id) ON DELETE CASCADE,
    role             TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system', 'tool')),
    content          TEXT NOT NULL,
    -- The raw content as sent/received; never modified after insert
    
    -- Prompt tracking: what was actually sent to the model
    prompt_tokens    INTEGER,
    completion_tokens INTEGER,
    total_tokens     INTEGER,
    
    -- Model and timing
    model_id         TEXT,  -- may differ from conversation default if switched mid-thread
    latency_ms       INTEGER,  -- round-trip time to the LLM API
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    -- For tool/function calls (agents)
    tool_call_id     TEXT,  -- OpenAI/Anthropic tool call identifier
    tool_name        TEXT,
    tool_input       JSONB,   -- structured input to the tool
    tool_output      JSONB,   -- structured output from the tool
    
    -- Error tracking
    error_code       TEXT,
    error_message    TEXT,
    is_complete      BOOLEAN NOT NULL DEFAULT TRUE,  -- FALSE for streaming incomplete
    
    metadata         JSONB NOT NULL DEFAULT '{}'::jsonb,
    
    -- Ordering: use created_at + sequence for deterministic ordering
    sequence_number  INTEGER NOT NULL,
    UNIQUE (conversation_id, sequence_number)
);

-- Indexes for common AI query patterns
CREATE INDEX idx_messages_conversation_created 
    ON messages (conversation_id, created_at DESC);

CREATE INDEX idx_messages_conversation_sequence 
    ON messages (conversation_id, sequence_number);

CREATE INDEX idx_conversations_user_updated 
    ON conversations (user_id, updated_at DESC);

-- Partial index for active conversations only
CREATE INDEX idx_conversations_active 
    ON conversations (user_id, updated_at DESC) 
    WHERE archived_at IS NULL;

-- Trigger to update conversations.updated_at and counters on new message
CREATE OR REPLACE FUNCTION update_conversation_on_message()
RETURNS TRIGGER AS $$
BEGIN
    UPDATE conversations
    SET 
        updated_at    = NOW(),
        message_count = message_count + 1,
        token_count   = token_count + COALESCE(NEW.total_tokens, 0)
    WHERE conversation_id = NEW.conversation_id;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_message_insert
    AFTER INSERT ON messages
    FOR EACH ROW
    EXECUTE FUNCTION update_conversation_on_message();

    With this foundation, fetching the last N messages for context injection becomes a single indexed query:

    -- Retrieve last 20 messages for context window construction
-- Used directly before each LLM API call
SELECT 
    role,
    content,
    tool_call_id,
    tool_name,
    tool_input,
    tool_output,
    sequence_number
FROM messages
WHERE 
    conversation_id = $1
    AND is_complete = TRUE
    AND error_code IS NULL
ORDER BY sequence_number DESC
LIMIT 20;

    Document Storage and the Embedding Pipeline

    RAG applications live and die by their document ingestion pipeline. The schema needs to track the full lifecycle of a document: from raw ingest, through chunking, to embedding, to retrieval. This matters because you will need to re-embed documents when you upgrade embedding models, debug why a particular chunk was or wasn't retrieved, and audit what content was available at a given point in time.

    -- Document and embedding schema for RAG applications

-- Collections: logical groupings of documents (like namespaces)
CREATE TABLE document_collections (
    collection_id    UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    name             TEXT UNIQUE NOT NULL,
    description      TEXT,
    embedding_model  TEXT NOT NULL,  -- e.g. 'text-embedding-3-large'
    embedding_dim    INTEGER NOT NULL,  -- e.g. 1536, 3072, 768
    chunking_strategy JSONB NOT NULL DEFAULT '{
        "method": "recursive_character",
        "chunk_size": 1000,
        "chunk_overlap": 200
    }'::jsonb,
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    metadata         JSONB NOT NULL DEFAULT '{}'::jsonb
);

-- Source documents: the raw ingested files
CREATE TABLE documents (
    document_id      UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    collection_id    UUID NOT NULL REFERENCES document_collections(collection_id),
    
    -- Source identification
    source_uri       TEXT NOT NULL,  -- file path, S3 URI, URL, etc.
    source_type      TEXT NOT NULL CHECK (source_type IN (
                         'pdf', 'markdown', 'html', 'text', 'docx', 
                         'confluence', 'notion', 'github', 'database'
                     )),
    title            TEXT,
    
    -- Content storage: store raw content for re-processing
    raw_content      TEXT,          -- original text (may be NULL for binary sources)
    content_hash     TEXT NOT NULL, -- SHA-256 of raw content for dedup
    
    -- Processing state machine
    status           TEXT NOT NULL DEFAULT 'pending' 
                         CHECK (status IN (
                             'pending', 'processing', 'embedded', 
                             'failed', 'outdated'
                         )),
    
    -- Ingestion tracking
    ingested_at      TIMESTAMPTZ,
    embedded_at      TIMESTAMPTZ,
    failed_at        TIMESTAMPTZ,
    error_message    TEXT,
    
    -- Version tracking: when source changes, we re-embed
    version          INTEGER NOT NULL DEFAULT 1,
    previous_version_id UUID REFERENCES documents(document_id),
    
    -- Source metadata: author, created_date, page_count, etc.
    source_metadata  JSONB NOT NULL DEFAULT '{}'::jsonb,
    
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    UNIQUE (collection_id, content_hash)  -- prevent duplicate ingestion
);

-- Chunks: the atomic retrieval units
-- Each document is split into overlapping chunks before embedding
CREATE TABLE document_chunks (
    chunk_id         UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    document_id      UUID NOT NULL REFERENCES documents(document_id) ON DELETE CASCADE,
    collection_id    UUID NOT NULL REFERENCES document_collections(collection_id),
    
    -- Content
    content          TEXT NOT NULL,
    content_tokens   INTEGER,  -- token count for budget-aware retrieval
    
    -- Position within document
    chunk_index      INTEGER NOT NULL,  -- 0-based position in document
    start_char       INTEGER,           -- character offset in raw_content
    end_char         INTEGER,
    page_number      INTEGER,           -- for PDFs
    
    -- The embedding vector — lives right next to the content it represents
    embedding        vector(1536),      -- dimension matches collection's embedding_model
    embedding_model  TEXT,              -- the model that produced this embedding
    embedded_at      TIMESTAMPTZ,
    
    -- Rich metadata for hybrid search filtering
    -- Store anything you might want to filter on at query time
    metadata         JSONB NOT NULL DEFAULT '{}'::jsonb,
    -- e.g. {"section": "Installation", "headings": ["Setup", "Prerequisites"],
    --        "language": "en", "has_code": true, "doc_type": "technical"}
    
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    UNIQUE (document_id, chunk_index)
);

-- Vector similarity index — HNSW for production (better query performance than IVFFlat)
CREATE INDEX idx_chunks_embedding_hnsw
    ON document_chunks 
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- Composite index for hybrid search: filter by collection, then vector search
CREATE INDEX idx_chunks_collection_embedding
    ON document_chunks (collection_id)
    INCLUDE (content, metadata, embedding);

-- GIN index for JSONB metadata filtering
CREATE INDEX idx_chunks_metadata
    ON document_chunks USING gin (metadata jsonb_path_ops);

-- Full-text search index for BM25-style keyword matching
CREATE INDEX idx_chunks_content_fts
    ON document_chunks 
    USING gin (to_tsvector('english', content));

-- Retrieval log: track every vector search for debugging and relevance tuning
CREATE TABLE retrieval_log (
    retrieval_id     UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    conversation_id  UUID REFERENCES conversations(conversation_id),
    message_id       UUID REFERENCES messages(message_id),
    collection_id    UUID NOT NULL REFERENCES document_collections(collection_id),
    
    query_text       TEXT NOT NULL,
    query_embedding  vector(1536),
    
    -- The chunks that were returned and their scores
    results          JSONB NOT NULL,
    -- [{"chunk_id": "...", "score": 0.87, "rank": 1}, ...]
    
    -- Search parameters used
    top_k            INTEGER NOT NULL,
    similarity_threshold FLOAT,
    metadata_filter  JSONB,
    
    latency_ms       INTEGER,
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

    The retrieval_log table is worth emphasizing. Production RAG systems require constant tuning. Without logging what the retriever actually returned for each query, you're debugging hallucinations and missed retrievals with no evidence. This table is how you build an offline evaluation dataset and improve your retrieval pipeline over time.
