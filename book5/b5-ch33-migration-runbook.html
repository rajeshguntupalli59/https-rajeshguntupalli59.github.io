Chapter 33: Database Migration to Cloud: Practical Runbook

Migration is where cloud theory meets operational reality. You have spent months planning your cloud architecture, selecting instance types, designing VPC topology, and writing Terraform modules. Now someone hands you a production Oracle 19c database, a legacy SQL Server 2012 instance, or a PostgreSQL 14 cluster running on bare metal, and says: get it to cloud without losing data and without a multi-hour outage. This chapter is the runbook for that moment. It covers the full migration lifecycle from assessment through cutover — using AWS DMS, the Schema Conversion Tool, Azure Database Migration Service, and GCP Database Migration Service — with the specific commands, gotchas, and production patterns that experienced DBAs need when the pressure is on.

---

33.1 Migration Assessment: Knowing What You're Moving Before You Move It

The single most common migration failure mode is discovering surprises during execution rather than during planning. A migration that looks like a simple database lift-and-shift often hides Oracle DBMS packages, SQL Server Agent jobs, linked server dependencies, or stored procedures that reference system tables unavailable in managed services. Your assessment phase exists to eliminate those surprises.

Inventory What Managed Services Cannot Replicate

Before touching a migration tool, build an explicit incompatibility list. Managed RDS, Azure SQL, and Cloud SQL impose hard restrictions that self-managed databases do not:

  • → RDS PostgreSQL: no `pg_hba.conf` direct edit, no superuser, no OS-level extensions not on the approved list, no `wal_level = logical` on read replicas by default
  • → Azure SQL (PaaS): no SQL Agent with full capability (use Elastic Jobs instead), no linked servers to on-prem without Azure Arc, no `xp_cmdshell`, no CLR assemblies without explicit opt-in
  • → Cloud SQL PostgreSQL: no superuser, no `pg_basebackup` direct access, limited extension list per engine version
  • → Aurora PostgreSQL: `rds_superuser` role replaces `postgres` superuser; `aurora_replica_status()` replaces direct `pg_stat_replication` in some contexts

    • AWS provides the Schema Conversion Tool (SCT) for automated incompatibility scanning. Run it even for homogeneous migrations — PostgreSQL to Aurora PostgreSQL — because it catches version-specific function differences.

    # Install AWS SCT CLI wrapper and run an assessment report
# SCT itself is a GUI tool, but you can script it via its project files
# First, install SCT from: https://docs.aws.amazon.com/SchemaConversionTool/

# Generate an assessment report using SCT command line
java -jar aws-schema-conversion-tool.jar \
  --source-engine postgresql \
  --source-host onprem-pg.internal \
  --source-port 5432 \
  --source-user migration_user \
  --source-password "${SOURCE_PW}" \
  --source-database prod_db \
  --target-engine aurora-postgresql \
  --report-file /reports/assessment-$(date +%Y%m%d).pdf \
  --action GenerateAssessmentReport

# For SQL Server to Aurora PostgreSQL migrations, run conversion
java -jar aws-schema-conversion-tool.jar \
  --source-engine sqlserver \
  --source-host onprem-sql.internal \
  --source-port 1433 \
  --source-user sct_user \
  --source-password "${SOURCE_PW}" \
  --source-database AdventureWorks \
  --target-engine aurora-postgresql \
  --report-file /reports/sqlserver-pg-assessment.pdf \
  --action GenerateAssessmentReport

    Gather Baseline Metrics Before Migration

    You cannot validate a successful migration without knowing what normal looks like on the source. Capture these metrics at least two weeks before cutover:

    | Metric | Why It Matters |

    |---|---|

    | Peak TPS and QPS | Determines target instance class |

    | 95th percentile query latency | Post-migration baseline for regression detection |

    | Replication lag (if HA) | Sets your DMS replication lag acceptance threshold |

    | Database size + growth rate per week | Determines migration window length and storage provisioning |

    | Active connections at peak | Informs RDS Proxy or Azure connection pool sizing |

    | Long-running transactions (> 60 seconds) | DMS and logical replication will struggle with these |

    -- Run on source PostgreSQL to gather migration sizing data
-- Capture database sizes
SELECT
    datname AS database_name,
    pg_size_pretty(pg_database_size(datname)) AS size,
    pg_database_size(datname) AS size_bytes
FROM pg_database
WHERE datistemplate = FALSE
ORDER BY size_bytes DESC;

-- Identify tables that will cause DMS problems: no primary key
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname || '.' || tablename)) AS total_size
FROM pg_tables t
WHERE NOT EXISTS (
    SELECT 1 FROM pg_constraint c
    JOIN pg_class cl ON c.conrelid = cl.oid
    JOIN pg_namespace n ON cl.relnamespace = n.oid
    WHERE c.contype = 'p'
      AND n.nspname = t.schemaname
      AND cl.relname = t.tablename
)
AND schemaname NOT IN ('pg_catalog', 'information_schema')
ORDER BY pg_total_relation_size(schemaname || '.' || tablename) DESC;

-- Find objects with dependencies that may not migrate cleanly
SELECT
    dependent_ns.nspname AS dependent_schema,
    dependent_view.relname AS dependent_view,
    source_ns.nspname AS source_schema,
    source_table.relname AS source_table
FROM pg_depend
JOIN pg_rewrite ON pg_depend.objid = pg_rewrite.oid
JOIN pg_class AS dependent_view ON pg_rewrite.ev_class = dependent_view.oid
JOIN pg_class AS source_table ON pg_depend.refobjid = source_table.oid
JOIN pg_namespace dependent_ns ON dependent_ns.oid = dependent_view.relnamespace
JOIN pg_namespace source_ns ON source_ns.oid = source_table.relnamespace
WHERE source_table.relkind = 'r'
AND dependent_view.relname != source_table.relname
ORDER BY 1, 2;

    Migration Strategy Selection

    Match your strategy to your downtime budget:

    | Strategy | Downtime | Complexity | Use Case |

    |---|---|---|---|

    | Dump and restore | Hours to days | Low | Small databases < 50 GB, scheduled maintenance |

    | AWS DMS full load + CDC | Minutes | Medium | Databases up to several TB with minimal downtime requirement |

    | Logical replication (pg_logical / pglogical) | Minutes | Medium-High | PostgreSQL to PostgreSQL with full control |

    | Physical snapshot + DMS CDC | Minutes | High | Large databases > 500 GB where full DMS load is too slow |

    | AWS DataSync + DMS | Minutes | High | Very large databases with NFS/SFTP accessible backups |

    For most production migrations, the physical snapshot seeding approach is optimal at scale: restore a backup to the target, then use DMS or logical replication for CDC to catch up the delta, then cut over.

    33.2 AWS DMS: Full Load and Change Data Capture

    AWS Database Migration Service handles the replication engine so you do not have to manage Debezium, pglogical, or custom trigger-based CDC infrastructure. The trade-off is that DMS runs on managed replication instances that you cannot SSH into, and its behavior with edge cases — LOB columns, tables without primary keys, DDL changes mid-migration — requires explicit configuration.

    Provision the DMS Infrastructure with Terraform

    # terraform/dms-migration.tf
# Replication instance — this is the compute that runs DMS tasks
resource "aws_dms_replication_instance" "migration" {
  replication_instance_id     = "prod-migration-ri"
  replication_instance_class  = "dms.r6i.2xlarge"  # 8 vCPU, 64 GB — suitable for most production loads
  allocated_storage           = 500                  # GB — DMS uses this for LOB caching and logs
  engine_version              = "3.5.2"
  multi_az                    = false               # Single AZ is fine during migration; this is not HA production
  publicly_accessible         = false
  vpc_security_group_ids      = [aws_security_group.dms_sg.id]
  replication_subnet_group_id = aws_dms_replication_subnet_group.migration.id

  tags = {
    Environment = "migration"
    Project     = "db-to-cloud"
  }
}

# Source endpoint — on-premises PostgreSQL
resource "aws_dms_endpoint" "source_pg" {
  endpoint_id   = "onprem-postgres-source"
  endpoint_type = "source"
  engine_name   = "postgres"

  server_name   = "10.0.1.50"  # Internal IP of source DB, reachable via Direct Connect or VPN
  port          = 5432
  database_name = "prod_db"
  username      = "dms_replication_user"
  password      = var.source_db_password  # Store in AWS Secrets Manager, reference here

  extra_connection_attributes = join(";", [
    "pluginName=pglogical",         # Use pglogical plugin for CDC
    "heartbeatEnable=true",
    "heartbeatFrequency=5",
    "heartbeatSchema=dms_heartbeat",  # Schema must exist on source
    "captureDDLs=false",              # Disable DDL capture — apply schema changes manually
    "executeTimeout=120"
  ])

  ssl_mode = "require"
}

# Target endpoint — Aurora PostgreSQL cluster
resource "aws_dms_endpoint" "target_aurora" {
  endpoint_id   = "aurora-pg-target"
  endpoint_type = "target"
  engine_name   = "aurora-postgresql"

  server_name   = aws_rds_cluster.aurora_prod.endpoint
  port          = 5432
  database_name = "prod_db"
  username      = "dms_target_user"
  password      = var.target_db_password

  extra_connection_attributes = join(";", [
    "executeTimeout=300",
    "maxFileSize=512000",   # 512 MB file size for bulk load
    "parallelLoadThreads=4"
  ])
}

# Replication task — full load then CDC
resource "aws_dms_replication_task" "full_load_cdc" {
  replication_task_id       = "prod-full-load-cdc"
  migration_type            = "full-load-and-cdc"
  replication_instance_arn  = aws_dms_replication_instance.migration.replication_instance_arn
  source_endpoint_arn       = aws_dms_endpoint.source_pg.endpoint_arn
  target_endpoint_arn       = aws_dms_endpoint.target_aurora.endpoint_arn

  table_mappings = jsonencode({
    rules = [
      {
        rule-type = "selection"
        rule-id   = "1"
        rule-name = "include-all-tables"
        object-locator = {
          schema-name = "public"
          table-name  = "%"
        }
        rule-action = "include"
      },
      {
        # Exclude audit/log tables that are too large and not needed on target
        rule-type = "selection"
        rule-id   = "2"
        rule-name = "exclude-audit-tables"
        object-locator = {
          schema-name = "public"
          table-name  = "audit_%"
        }
        rule-action = "exclude"
      }
    ]
  })

  replication_task_settings = jsonencode({
    TargetMetadata = {
      TargetSchema              = ""
      SupportLobs               = true
      FullLobMode               = false   # Limited LOB mode is faster; full LOB mode for large BLOBs
      LobChunkSize              = 64      # KB
      LimitedSizeLobMode        = true
      LobMaxSize                = 32768   # 32 MB max LOB size in limited mode
    }
    FullLoadSettings = {
      TargetTablePrepMode       = "DROP_AND_CREATE"  # Safe for initial load; change to DO_NOTHING for reseeding
      CreatePkAfterFullLoad     = true               # Apply PKs after load for performance
      StopTaskCachedChangesApplied = false
      StopTaskCachedChangesNotApplied = false
      MaxFullLoadSubTasks       = 8                  # Parallel table loads
      TransactionConsistencyTimeout = 600
      CommitRate                = 50000              # Rows per commit in full load
    }
    Logging = {
      EnableLogging = true
      LogComponents = [
        { Id = "SOURCE_UNLOAD", Severity = "LOGGER_SEVERITY_DEFAULT" },
        { Id = "TARGET_LOAD",   Severity = "LOGGER_SEVERITY_DEFAULT" },
        { Id = "TASK_MANAGER",  Severity = "LOGGER_SEVERITY_DEFAULT" }
      ]
    }
    ControlTablesSettings = {
      historyTimeslotInMinutes  = 5
      StatusTableEnabled        = true
      SuspendedTablesTableEnabled = true
    }
  })
}

    Prepare the Source PostgreSQL for DMS Logical Replication

    DMS requires logical replication to be enabled on the source and a replication user with appropriate privileges. On self-managed PostgreSQL, you control this directly. On RDS PostgreSQL source (if migrating between RDS instances), the parameter group controls it.

    -- On source PostgreSQL (self-managed): enable logical replication
-- Edit postgresql.conf or run as superuser:
ALTER SYSTEM SET wal_level = 'logical';
ALTER SYSTEM SET max_replication_slots = 10;
ALTER SYSTEM SET max_wal_senders = 10;
-- Requires a restart:
-- SELECT pg_reload_conf(); -- Only for parameters that don't require restart

-- Create the DMS replication user with minimum required privileges
CREATE USER dms_replication_user WITH
    REPLICATION
    LOGIN
    PASSWORD 'strong_password_here';

-- Grant access to all tables DMS needs to replicate
GRANT CONNECT ON DATABASE prod_db TO dms_replication_user;
GRANT USAGE ON SCHEMA public TO dms_replication_user;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO dms_replication_user;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
    GRANT SELECT ON TABLES TO dms_replication_user;

-- Create heartbeat schema for DMS health checks
CREATE SCHEMA IF NOT EXISTS dms_heartbeat;
GRANT ALL ON SCHEMA dms_heartbeat TO dms_replication_user;

-- If using pglogical plugin (recommended over test_decoding for production):
CREATE EXTENSION IF NOT EXISTS pglogical;

-- Verify logical replication is active
SELECT slot_name, plugin, slot_type, active, restart_lsn
FROM pg_replication_slots;

-- Monitor DMS replication lag during migration
SELECT
    slot_name,
    pg_size_pretty(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)) AS replication_lag_size,
    active,
    active_pid
FROM pg_replication_slots
WHERE slot_type = 'logical';

    Critical DMS Gotchas in Production

    DMS has a set of well-documented limitations that will surprise you the first time they surface at 2 AM:

    *Tables without primary keys* fall back to full-table scans for CDC, which is catastrophically slow on large tables. Run the assessment query from section 33.1 to identify these before starting. Either add surrogate primary keys to the source before migration or exclude these tables from CDC and migrate them separately at cutover.

    *LOB column handling* in limited LOB mode will silently truncate data exceeding your LobMaxSize setting. Set it conservatively high and verify with a row count and checksum comparison after full load.

    *DDL changes during CDC* — DMS does not replicate DDL by default for PostgreSQL sources. If your source application runs ALTER TABLE during the migration window, you must apply those DDL changes manually to the target and then resume replication. This is one strong argument for freeze windows on schema changes during migration.

    *DMS does not migrate sequences*. After full load completes, manually synchronize sequence values:

    -- Run on source after full load completes, before cutover
-- Generate sequence synchronization statements
SELECT
    'SELECT setval(''' || schemaname || '.' || sequencename ||
    ''', COALESCE((SELECT MAX(' || quote_ident(a.attname) ||
    ') FROM ' || schemaname || '.' || tablename || '), 1), true);' AS sync_command
FROM pg_sequences s
JOIN pg_class c ON c.relname = s.sequencename
JOIN pg_depend d ON d.objid = c.oid
JOIN pg_class ct ON d.refobjid = ct.oid
JOIN pg_attribute a ON a.attrelid = ct.oid AND a.attnum = d.refobjsubid
JOIN pg_tables t ON t.tablename = ct.relname AND t.schemaname = s.schemaname
WHERE s.schemaname = 'public';

-- Execute each of those statements on the TARGET Aurora instance
-- This is typically a 5-10 minute manual step during cutover

    33.3 Azure Database Migration Service and SQL Server Migration

    When moving SQL Server workloads to Azure SQL — either Azure SQL Database, Azure SQL Managed Instance, or SQL Server on Azure VM — the Azure Database Migration Service (DMS) handles the heavy lifting. The key architectural decision you make before starting is whether your target is Azure SQL Database (PaaS, fully managed, most restrictive), Azure SQL Managed Instance (PaaS, near-full SQL Server compatibility, fewer restrictions), or SQL Server on Azure VM (IaaS, full control, your responsibility for patching and HA).

    For most enterprise SQL Server migrations, Azure SQL Managed Instance is the right target: it supports SQL Agent, cross-database queries, linked servers, CLR, Service Broker, and the full SQL Server surface area — things Azure SQL Database does not. The trade-off is cost; Managed Instance is more expensive.

    Assess Compatibility Before Running DMS

    The Data Migration Assistant (DMA) is your pre-migration assessment tool. Run it against every source SQL Server database:

    # Azure DMA is a Windows GUI tool, but you can automate assessments via its command line
# Install: https://www.microsoft.com/en-us/download/details.aspx?id=53595

# Run DMA assessment (from Windows command prompt or PowerShell)
"C:\Program Files\Microsoft Data Migration Assistant\DmaCmd.exe" `
  /AssessmentName="ProdDBAssessment" `
  /AssessmentSourcePlatform=SqlServer `
  /AssessmentTargetPlatform=AzureSqlDatabaseManagedInstance `
  /AssessmentDatabases="Server=onprem-sql.internal;Initial Catalog=ProdDB;User Id=dma_user;Password=$env:DMA_PASSWORD" `
  /AssessmentEvaluateCompatibilityIssues=True `
  /AssessmentEvaluateFeatureParity=True `
  /AssessmentOverwriteResult=True `
  /AssessmentResultJson="C:\Reports\assessment-result.json"

# Parse results to get action items
python parse_dma_results.py --input C:\Reports\assessment-result.json
    # parse_dma_results.py — Extract blocking issues from DMA output
import json
import sys
from pathlib import Path

def parse_dma_report(report_path: str) -> None:
    with open(report_path) as f:
        report = json.load(f)

    blocking_issues = []
    warning_issues = []

    for server_report in report.get("serverReports", []):
        for db_report in server_report.get("databaseReports", []):
            db_name = db_report.get("databaseName")

            for category in db_report.get("items", []):
                for rule_result in category.get("ruleResults", []):
                    severity = rule_result.get("severity", "")
                    title = rule_result.get("title", "")
                    impact = rule_result.get("impact", "")
                    affected_objects = [
                        obj.get("name") for obj in rule_result.get("affectedObjects", [])
                    ]

                    entry = {
                        "database": db_name,
                        "title": title,
                        "impact": impact,
                        "affected_objects": affected_objects[:5],  # Top 5
                        "total_affected": len(affected_objects)
                    }

                    if severity == "Breaking":
                        blocking_issues.append(entry)
                    elif severity == "Warning":
                        warning_issues.append(entry)

    print(f"\n{'='*60}")
    print(f"BLOCKING ISSUES ({len(blocking_issues)}) — Must fix before migration:")
    print(f"{'='*60}")
    for issue in blocking_issues:
        print(f"\n[{issue['database']}] {issue['title']}")
        print(f"  Impact: {issue['impact']}")
        print(f"  Affected objects ({issue['total_affected']}): {', '.join(issue['affected_objects'])}")

    print(f"\n{'='*60}")
    print(f"WARNINGS ({len(warning_issues)}) — Review post-migration:")
    print(f"{'='*60}")
    for issue in warning_issues[:10]:  # Show top 10 warnings
        print(f"\n[{issue['database']}] {issue['title']}")

if __name__ == "__main__":
    parse_dma_report(sys.argv[1])
