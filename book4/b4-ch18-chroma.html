Chapter 18: Chroma: Local-First Vector DB for Development

Chroma occupies a unique position in the vector database landscape: it is the tool developers reach for first, the one that runs in a Python process with zero infrastructure, and the one that disappears from the conversation the moment a team decides to move to production. That reputation is both earned and slightly unfair. Chroma is a genuinely capable vector store that handles millions of embeddings, supports persistent disk-backed storage, and can run as a standalone server with a client/server split that mirrors how teams actually work. What it lacks is the operational maturity of Pinecone or Weaviate — but for local development, rapid prototyping, and small-to-medium deployments where you control the machine, it is often the most productive choice. This chapter covers Chroma's architecture, its embedding and collection model, persistent versus in-memory operation, querying patterns that transfer directly to production systems, and the specific scenarios where Chroma remains the right answer even after your application ships.

---

18.1 Chroma's Architecture: An Embedded Database Perspective

Database professionals will recognize Chroma's design immediately once you map it to familiar concepts. Chroma is, at its core, an embedded vector database in the same lineage as SQLite — it runs inside your process, writes to a local directory, and requires no separate daemon. This is its defining characteristic and the source of both its appeal and its limitations.

The storage model

Chroma uses a layered storage architecture. The vector index is backed by HNSW (Hierarchical Navigable Small World), the same algorithm used internally by pgvector and Weaviate. Metadata and document text are stored in DuckDB in versions prior to 0.4, and in Chroma's own embedded SQLite-based system in the 0.4+ architecture. This separation — vectors in an HNSW structure, metadata in a columnar or relational store — mirrors what you see in every production vector database. Chroma just makes the layers visible in a single local directory.

The chroma directory after initialization looks like this:

# shell
ls -la ./chroma_data/
# chroma.sqlite3          <- metadata, collections, embeddings catalog
# <uuid>/                 <- one directory per collection
#   data_level0.bin       <- HNSW graph (bottom layer, all nodes)
#   header.bin            <- index parameters
#   length.bin            <- node count
#   link_lists.bin        <- HNSW upper-layer links

If you have managed PostgreSQL databases, think of chroma.sqlite3 as the system catalog and the HNSW binary files as a specialized index tablespace. The collection UUID directory is analogous to a table's data files — the collection is the primary organizing unit in Chroma, equivalent to a table with a mandatory vector column and optional metadata columns.

Collections as tables

A Chroma collection holds:

  • → **IDs** — string primary keys (you supply these, no auto-increment)
  • → **embeddings** — float arrays of fixed dimension (dimensionality is inferred from the first insert)
  • → **documents** — optional raw text (the source that was embedded)
  • → **metadatas** — optional key/value dictionaries (analogous to non-indexed columns)

    • The dimension constraint works like a typed column: once you insert a 1536-dimensional embedding, every subsequent insert must match. Mixing models is a schema violation Chroma enforces at insert time.

    Client modes

    Chroma supports three operational modes that map to a deployment progression:

    | Mode | Use Case | Analogy |

    |------|----------|---------|

    | Ephemeral (in-memory) | Unit tests, notebooks | :memory: SQLite |

    | Persistent (embedded) | Local dev, single-process apps | File-based SQLite |

    | Client/Server (HTTP) | Multi-process, shared dev environment | Separate DB server |

    Most tutorials show only the ephemeral mode. Production-adjacent usage requires understanding the persistent and client/server modes, which this chapter covers in detail.

    18.2 Getting Started: Collections, Embeddings, and Basic Queries

    Install Chroma with its default dependencies, which include the HNSW library and sentence-transformers for local embedding generation:

    # shell
pip install chromadb sentence-transformers openai
# For the server mode:
pip install "chromadb[server]"

    Ephemeral client (testing and notebooks)

    # Python
import chromadb

# In-memory client — resets on process exit
client = chromadb.EphemeralClient()

# Create a collection with cosine similarity (default is L2)
collection = client.create_collection(
    name="product_docs",
    metadata={"hnsw:space": "cosine"}
)

# Add documents — Chroma uses its default embedding function if you don't specify one
# Default is all-MiniLM-L6-v2 via sentence-transformers (384 dimensions)
collection.add(
    documents=[
        "PostgreSQL supports JSONB for semi-structured data",
        "pgvector extends PostgreSQL with vector similarity search",
        "Chroma is an open-source embedding database",
        "HNSW indexing provides approximate nearest neighbor search",
        "RAG pipelines combine retrieval with language model generation",
    ],
    ids=["doc_001", "doc_002", "doc_003", "doc_004", "doc_005"],
    metadatas=[
        {"source": "pg_docs", "category": "database"},
        {"source": "pg_docs", "category": "vector"},
        {"source": "chroma_docs", "category": "vector"},
        {"source": "research", "category": "algorithms"},
        {"source": "ml_docs", "category": "architecture"},
    ]
)

# Query — Chroma embeds the query text automatically using the same function
results = collection.query(
    query_texts=["how does vector similarity search work?"],
    n_results=3,
    include=["documents", "metadatas", "distances"]
)

for i, (doc, meta, dist) in enumerate(zip(
    results["documents"][0],
    results["metadatas"][0],
    results["distances"][0]
)):
    print(f"Rank {i+1} | Distance: {dist:.4f} | Source: {meta['source']}")
    print(f"  {doc}\n")

    The include parameter controls what Chroma returns — omit "embeddings" unless you need the raw vectors, since returning full embedding arrays for large result sets is expensive.

    Persistent client with metadata filtering

    # Python
import chromadb
from chromadb.utils import embedding_functions

# Persistent client — survives process restart
client = chromadb.PersistentClient(path="./chroma_data")

# Use OpenAI embeddings instead of the default local model
openai_ef = embedding_functions.OpenAIEmbeddingFunction(
    api_key="sk-...",          # or read from env: os.environ["OPENAI_API_KEY"]
    model_name="text-embedding-3-small"  # 1536 dimensions
)

# get_or_create is idempotent — safe to call on restart
collection = client.get_or_create_collection(
    name="engineering_kb",
    embedding_function=openai_ef,
    metadata={"hnsw:space": "cosine"}
)

# Batch insert — Chroma handles the embedding calls internally
documents = [
    "Connection pooling reduces database overhead by reusing connections",
    "Read replicas distribute query load across multiple database instances",
    "Partitioning splits large tables into smaller, manageable segments",
    "Vacuuming reclaims storage from deleted rows in PostgreSQL",
    "WAL (Write-Ahead Logging) ensures durability and enables replication",
]
ids = [f"kb_{i:04d}" for i in range(len(documents))]
metadatas = [
    {"topic": "performance", "db": "general"},
    {"topic": "scaling", "db": "general"},
    {"topic": "storage", "db": "general"},
    {"topic": "maintenance", "db": "postgresql"},
    {"topic": "reliability", "db": "postgresql"},
]

collection.add(documents=documents, ids=ids, metadatas=metadatas)

# Query with metadata filter — equivalent to WHERE clause
results = collection.query(
    query_texts=["how to improve query performance?"],
    n_results=3,
    where={"db": "postgresql"},          # metadata filter
    include=["documents", "metadatas", "distances"]
)

print(f"Found {len(results['ids'][0])} results filtered to PostgreSQL topics:")
for doc, meta, dist in zip(
    results["documents"][0],
    results["metadatas"][0],
    results["distances"][0]
):
    print(f"  [{meta['topic']}] dist={dist:.4f}: {doc[:60]}...")

    The where clause supports operators that map directly to SQL predicates:

    # Python
# Single condition
where={"category": "vector"}

# AND condition
where={"$and": [{"category": "vector"}, {"source": "pg_docs"}]}

# OR condition
where={"$or": [{"db": "postgresql"}, {"db": "general"}]}

# Comparison operators
where={"score": {"$gte": 0.8}}          # score >= 0.8
where={"year": {"$in": [2023, 2024]}}   # year IN (2023, 2024)
where={"status": {"$ne": "archived"}}   # status != 'archived'

    For DBA readers: these operators compile down to SQLite predicates against the metadata store before the HNSW search executes. Pre-filtering on metadata is more efficient than post-filtering large result sets, particularly when the metadata column has high cardinality.

    18.3 Embedding Functions: Wiring Chroma to Your Model Layer

    Chroma's embedding function abstraction is one of its more thoughtful design decisions. Rather than requiring you to compute embeddings externally and pass float arrays, Chroma lets you attach an embedding function to a collection and then work exclusively with text. The function is called transparently during add() and query().

    Built-in embedding functions

    # Python
from chromadb.utils import embedding_functions

# 1. Default: all-MiniLM-L6-v2 (384-dim, runs locally, no API key needed)
default_ef = embedding_functions.DefaultEmbeddingFunction()

# 2. OpenAI — text-embedding-3-small (1536-dim) or text-embedding-3-large (3072-dim)
openai_ef = embedding_functions.OpenAIEmbeddingFunction(
    api_key="sk-...",
    model_name="text-embedding-3-small"
)

# 3. Cohere — good multilingual support
cohere_ef = embedding_functions.CohereEmbeddingFunction(
    api_key="...",
    model_name="embed-english-v3.0"
)

# 4. Sentence Transformers — local, no API costs, many model choices
st_ef = embedding_functions.SentenceTransformerEmbeddingFunction(
    model_name="BAAI/bge-large-en-v1.5"  # 1024-dim, strong retrieval performance
)

# 5. HuggingFace Inference API — cloud-hosted open models
hf_ef = embedding_functions.HuggingFaceEmbeddingFunction(
    api_key="hf_...",
    model_name="sentence-transformers/all-mpnet-base-v2"
)

    Custom embedding function for Azure OpenAI or AWS Bedrock

    When your organization's models are hosted on enterprise AI platforms, you write a custom embedding function by subclassing EmbeddingFunction:

    # Python
import numpy as np
from chromadb import EmbeddingFunction, Documents, Embeddings
import boto3
import json

class BedrockEmbeddingFunction(EmbeddingFunction):
    """Chroma embedding function backed by Amazon Bedrock Titan Embeddings."""

    def __init__(self, region_name: str = "us-east-1"):
        self.client = boto3.client("bedrock-runtime", region_name=region_name)
        self.model_id = "amazon.titan-embed-text-v1"

    def __call__(self, input: Documents) -> Embeddings:
        embeddings = []
        for text in input:
            body = json.dumps({"inputText": text})
            response = self.client.invoke_model(
                modelId=self.model_id,
                body=body,
                contentType="application/json",
                accept="application/json"
            )
            result = json.loads(response["body"].read())
            embeddings.append(result["embedding"])
        return embeddings


class AzureOpenAIEmbeddingFunction(EmbeddingFunction):
    """Chroma embedding function backed by Azure OpenAI."""

    def __init__(
        self,
        azure_endpoint: str,
        api_key: str,
        deployment_name: str,
        api_version: str = "2024-02-01"
    ):
        from openai import AzureOpenAI
        self.client = AzureOpenAI(
            azure_endpoint=azure_endpoint,
            api_key=api_key,
            api_version=api_version
        )
        self.deployment_name = deployment_name

    def __call__(self, input: Documents) -> Embeddings:
        # Azure OpenAI supports batch embedding — send all texts at once
        response = self.client.embeddings.create(
            input=input,
            model=self.deployment_name
        )
        # Sort by index to preserve order
        sorted_data = sorted(response.data, key=lambda x: x.index)
        return [item.embedding for item in sorted_data]


# Usage
bedrock_ef = BedrockEmbeddingFunction(region_name="us-west-2")
azure_ef = AzureOpenAIEmbeddingFunction(
    azure_endpoint="https://your-resource.openai.azure.com",
    api_key="...",
    deployment_name="text-embedding-3-small"
)

client = chromadb.PersistentClient(path="./chroma_data")
collection = client.get_or_create_collection(
    name="enterprise_docs",
    embedding_function=azure_ef,
    metadata={"hnsw:space": "cosine"}
)

    A dimension mismatch is a hard schema error. If you create a collection with text-embedding-3-small (1536 dimensions) and later try to query it with a custom function that returns 768-dimensional vectors, Chroma raises a ValueError at query time. There is no silent dimension reduction. Treat the embedding function attached to a collection as a schema constraint — document it alongside your collection metadata.

    18.4 Client/Server Mode and LangChain Integration

    Running Chroma as a server unlocks multi-process access, which is the minimum viable architecture for any application with a web server, background workers, and a database — they cannot all share an embedded client safely. The server exposes a FastAPI HTTP interface, and the Python client speaks to it over HTTP.

    Starting the Chroma server

    # shell
# Install with server dependencies
pip install "chromadb[server]"

# Start server on default port 8000
chroma run --path ./chroma_data --host 0.0.0.0 --port 8000

# Or with Docker — useful for CI and team shared environments
docker run -d \
  --name chroma \
  -p 8000:8000 \
  -v $(pwd)/chroma_data:/chroma/chroma \
  -e IS_PERSISTENT=TRUE \
  chromadb/chroma:latest

# Verify the server is running
curl http://localhost:8000/api/v1/heartbeat
# {"nanosecond heartbeat": 1234567890}

curl http://localhost:8000/api/v1/collections
# Returns list of existing collections as JSON

    Connecting from Python (client/server mode)

    # Python
import chromadb

# HTTP client — connects to running Chroma server
client = chromadb.HttpClient(
    host="localhost",
    port=8000,
    # Optional: add auth headers if you've configured Chroma with token auth
    # settings=chromadb.Settings(chroma_client_auth_provider="token",
    #                            chroma_client_auth_credentials="secret-token")
)

# Everything else is identical to the embedded client API
collection = client.get_or_create_collection(
    name="shared_collection",
    metadata={"hnsw:space": "cosine"}
)

# Test round-trip
collection.add(
    documents=["Server mode enables multi-process access to Chroma"],
    ids=["server_test_001"]
)

count = collection.count()
print(f"Collection contains {count} documents")

results = collection.query(
    query_texts=["multi-process database access"],
    n_results=1
)
print(results["documents"][0])

    LangChain integration

    LangChain treats Chroma as a VectorStore, which means you can swap it for Pinecone or Weaviate by changing two lines. This is exactly the abstraction you want during development — build against Chroma, promote to a production vector database without rewriting retrieval logic.

    # Python
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings
from langchain_core.documents import Document
import chromadb

# Option 1: Let LangChain manage the Chroma client (simple case)
embedding_function = OpenAIEmbeddings(model="text-embedding-3-small")

vectorstore = Chroma(
    collection_name="langchain_demo",
    embedding_function=embedding_function,
    persist_directory="./chroma_langchain_data"
)

# Add LangChain Document objects (they carry page_content and metadata)
docs = [
    Document(
        page_content="Indexes in PostgreSQL speed up query execution by reducing full table scans",
        metadata={"source": "pg_manual", "chapter": "11", "topic": "indexing"}
    ),
    Document(
        page_content="B-tree indexes are the default index type in PostgreSQL",
        metadata={"source": "pg_manual", "chapter": "11", "topic": "indexing"}
    ),
    Document(
        page_content="GIN indexes are optimal for full-text search and JSONB queries",
        metadata={"source": "pg_manual", "chapter": "11", "topic": "indexing"}
    ),
    Document(
        page_content="Partial indexes include only rows matching a WHERE condition",
        metadata={"source": "pg_manual", "chapter": "11", "topic": "indexing"}
    ),
]

vectorstore.add_documents(docs)

# Similarity search — returns Documents ranked by relevance
query = "what index type should I use for text search?"
similar_docs = vectorstore.similarity_search(query, k=2)
for doc in similar_docs:
    print(f"[{doc.metadata['topic']}] {doc.page_content}")

# Similarity search with scores
docs_with_scores = vectorstore.similarity_search_with_score(query, k=2)
for doc, score in docs_with_scores:
    print(f"Score: {score:.4f} | {doc.page_content[:60]}...")

# Option 2: Connect LangChain to a running Chroma server
http_client = chromadb.HttpClient(host="localhost", port=8000)

server_vectorstore = Chroma(
    client=http_client,
    collection_name="langchain_server_collection",
    embedding_function=embedding_function
)

# Use as a retriever in a RAG chain
retriever = server_vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 4, "filter": {"topic": "indexing"}}
)

# The retriever works with any LangChain chain
retrieved = retriever.invoke("how do partial indexes work?")
for doc in retrieved:
    print(doc.page_content)

    The as_retriever() call is the bridge to the full LangChain ecosystem — you can pass this retriever directly to RetrievalQA, ConversationalRetrievalChain, or LangGraph nodes without modification.

    18.5 HNSW Tuning, Scaling Limits, and When to Graduate

    Chroma exposes HNSW parameters through collection metadata. Understanding these parameters gives you control over the precision/performance tradeoff that every vector index presents — the same tradeoff exists in pgvector's ivfflat and Weaviate's HNSW configuration.

    HNSW parameters exposed by Chroma

    # Python
import chromadb

client = chromadb.PersistentClient(path="./chroma_data")

# Create collection with explicit HNSW tuning
collection = client.get_or_create_collection(
    name="tuned_collection",
    metadata={
        "hnsw:space": "cosine",        # Distance metric: cosine, l2, ip (inner product)
        "hnsw:construction_ef": 200,   # Candidates examined during index build (default: 100)
        "hnsw:search_ef": 100,         # Candidates examined during search (default: 10)
        "hnsw:M": 32,                  # Connections per node in graph (default: 16)
        "hnsw:num_threads": 4,         # Parallel threads for index operations
        "hnsw:resize_factor": 1.2,     # Growth factor when index needs to expand
        "hnsw:batch_size": 100,        # Vectors per batch during construction
        "hnsw:sync_threshold": 1000,   # Flush to disk after N mutations
    }
)

    Parameter guidance from database operations experience:

    | Parameter | Default | When to Increase | Memory Impact |

    |-----------|---------|-----------------|---------------|

    | hnsw:M | 16 | Recall < 0.90, dataset > 100K | ~2x at M=32 |

    | hnsw:construction_ef | 100 | Recall < 0.95 at query time | Index build time only |

    | hnsw:search_ef | 10 | Recall < 0.90 in production | Per-query CPU |

    | hnsw:num_threads | 1 | Batch ingestion too slow | None (CPU only) |

    The key insight: hnsw:construction_ef affects index quality permanently. hnsw:search_ef can be tuned per-query without rebuilding the index. If you index with construction_ef=100 and find recall is poor, you must rebuild — there is no online reindex in Chroma.
