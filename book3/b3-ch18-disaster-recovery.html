Chapter 18: Autonomous Disaster Recovery

Disaster recovery has always been the discipline where human judgment mattered most — the 3 a.m. call, the war-room decision, the tense minutes choosing between data loss and extended downtime. Autonomous systems change that contract entirely. When your DR pipeline self-monitors replication health, self-diagnoses failure mode, self-selects a recovery path, and self-executes a tested runbook without waiting for a human to wake up, your RTO shrinks from hours to seconds and your RPO approaches zero. This chapter covers the architecture and implementation of fully autonomous DR systems for PostgreSQL and SQL Server — systems that observe failure signals at the infrastructure layer, reason about failure type and blast radius, execute tiered recovery actions with circuit breakers, and validate recovery integrity before re-admitting traffic, all without manual intervention.

---

Failure Taxonomy and Autonomous Signal Acquisition

Before an autonomous system can recover from a disaster, it must correctly classify what it is recovering from. Treating a network partition the same as a storage corruption or an application-level connection flood produces wrong recovery actions and can worsen the situation. A taxonomy-first architecture forces the system to gate all recovery paths through a classification layer before execution.

The five failure classes that autonomous DR must handle independently are:

1. Primary node loss — the database process or host becomes unreachable

2. Replication lag breach — standby is alive but diverging beyond RPO threshold

3. Split-brain ambiguity — multiple nodes believe they are primary

4. Data corruption — block-level, page-level, or logical corruption on disk or in transit

5. Region/AZ failure — an entire failure domain becomes unavailable

Signal acquisition for each class requires distinct sensor types. For node loss, TCP health checks and PostgreSQL pg_isready / SQL Server @@SERVERNAME reachability form the first layer. For replication lag, you instrument the replication stream directly.

PostgreSQL:

-- Replication lag sensor query executed from monitoring host
SELECT
    application_name,
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    (sent_lsn - replay_lsn) AS replication_lag_bytes,
    EXTRACT(EPOCH FROM (now() - reply_time)) AS heartbeat_age_seconds
FROM pg_stat_replication
WHERE state = 'streaming';

SQL Server:

-- Replication lag sensor for Always On AG
SELECT
    ag.name AS ag_name,
    ar.replica_server_name,
    drs.database_name,
    drs.synchronization_state_desc,
    drs.synchronization_health_desc,
    drs.log_send_queue_size,
    drs.redo_queue_size,
    drs.last_commit_time,
    DATEDIFF(SECOND, drs.last_commit_time, GETUTCDATE()) AS lag_seconds
FROM sys.dm_hadr_database_replica_states drs
JOIN sys.availability_replicas ar
    ON drs.replica_id = ar.replica_id
JOIN sys.availability_groups ag
    ON ar.group_id = ag.group_id
WHERE drs.is_local = 0;

These queries run on a 5-second scrape interval from a Prometheus exporter. When replication_lag_bytes exceeds your RPO threshold (converted to bytes using average WAL generation rate) or heartbeat_age_seconds exceeds 15 seconds, the signal is emitted to the classification engine.

The classification engine is a Python process that aggregates signals across layers — network, OS, database process, replication, and application — before assigning a failure class. It deliberately waits 10–30 seconds before acting, using a voting window to distinguish transient blips from real events:

import time
import collections
from dataclasses import dataclass, field
from enum import Enum
from typing import Optional

class FailureClass(Enum):
    NODE_LOSS = "node_loss"
    REPLICATION_LAG = "replication_lag"
    SPLIT_BRAIN = "split_brain"
    DATA_CORRUPTION = "data_corruption"
    AZ_FAILURE = "az_failure"
    UNKNOWN = "unknown"

@dataclass
class FailureSignal:
    source: str
    signal_type: str
    severity: float          # 0.0 to 1.0
    timestamp: float = field(default_factory=time.time)

class FailureClassifier:
    VOTE_WINDOW_SECONDS = 20
    CONFIDENCE_THRESHOLD = 0.70

    def __init__(self):
        self.signal_buffer: list[FailureSignal] = []

    def ingest(self, signal: FailureSignal):
        self.signal_buffer.append(signal)
        self._evict_stale()

    def classify(self) -> tuple[FailureClass, float]:
        if not self.signal_buffer:
            return FailureClass.UNKNOWN, 0.0

        votes: dict[FailureClass, float] = collections.defaultdict(float)

        for sig in self.signal_buffer:
            if sig.signal_type == "tcp_unreachable" and sig.severity > 0.8:
                votes[FailureClass.NODE_LOSS] += sig.severity
            elif sig.signal_type == "replication_lag_bytes" and sig.severity > 0.6:
                votes[FailureClass.REPLICATION_LAG] += sig.severity
            elif sig.signal_type == "dual_primary_detected":
                votes[FailureClass.SPLIT_BRAIN] += sig.severity * 2  # escalate weight
            elif sig.signal_type == "checksum_failure":
                votes[FailureClass.DATA_CORRUPTION] += sig.severity
            elif sig.signal_type == "az_health_degraded" and sig.severity > 0.9:
                votes[FailureClass.AZ_FAILURE] += sig.severity

        if not votes:
            return FailureClass.UNKNOWN, 0.0

        top_class = max(votes, key=votes.__getitem__)
        total = sum(votes.values())
        confidence = votes[top_class] / total if total > 0 else 0.0

        if confidence >= self.CONFIDENCE_THRESHOLD:
            return top_class, confidence
        return FailureClass.UNKNOWN, confidence

    def _evict_stale(self):
        cutoff = time.time() - self.VOTE_WINDOW_SECONDS
        self.signal_buffer = [s for s in self.signal_buffer if s.timestamp >= cutoff]

The classifier only fires a recovery action when confidence exceeds 70% within the vote window. This is not a soft threshold — it is a hard gate. Below 70%, the system escalates to a human via PagerDuty while continuing to collect signals. Above 70%, the recovery pipeline fires autonomously.

Autonomous Failover Architecture: Patroni, pg_auto_failover, and SQL Server Always On

Once failure is classified and confidence is sufficient, the autonomous system hands off to the failover execution layer. For PostgreSQL, two production-grade tools represent distinct architectural philosophies: Patroni (consensus-based, etcd/Consul/ZooKeeper-backed) and pg_auto_failover (coordinator-based, simpler operationally). For SQL Server, Always On Availability Groups with the Windows Server Failover Cluster or a Pacemaker equivalent on Linux provide the failover substrate.

#### Patroni: Consensus-Driven Autonomous Failover

Patroni uses a distributed configuration store (etcd by default) as its consensus layer. The leader key in etcd has a TTL; when the primary fails to renew it, a candidate replica wins the election and promotes itself. For fully autonomous operation, the key configuration parameters are:

# patroni.yml — production autonomous configuration
name: pg-primary-az1

restapi:
  listen: 0.0.0.0:8008
  connect_address: 10.0.1.10:8008

etcd3:
  hosts: 10.0.0.1:2379,10.0.0.2:2379,10.0.0.3:2379

bootstrap:
  dcs:
    ttl: 30
    loop_wait: 10
    retry_timeout: 30
    maximum_lag_on_failover: 1048576      # 1 MB — RPO gate
    master_start_timeout: 300
    synchronous_mode: true                # enforce sync replication
    synchronous_mode_strict: false        # allow async fallback if sync replica lost
    postgresql:
      use_pg_rewind: true
      use_slots: true
      parameters:
        wal_level: replica
        hot_standby: "on"
        max_wal_senders: 10
        max_replication_slots: 10
        wal_log_hints: "on"

postgresql:
  listen: 0.0.0.0:5432
  connect_address: 10.0.1.10:5432
  data_dir: /data/postgresql/14/main
  pgpass: /tmp/pgpass0
  authentication:
    replication:
      username: replicator
      password: "${REPLICATION_PASSWORD}"
    superuser:
      username: postgres
      password: "${POSTGRES_PASSWORD}"

tags:
  nofailover: false
  noloadbalance: false
  clonefrom: false
  nosync: false

The maximum_lag_on_failover: 1048576 parameter is your RPO gate. If a candidate replica is more than 1 MB behind the last known primary LSN, Patroni will not promote it — preventing an autonomous failover that would lose unacceptable data. Combined with synchronous_mode: true, Patroni ensures at least one replica is always synchronous, making the 1 MB threshold meaningful primarily for asynchronous replicas in multi-replica topologies.

#### pg_auto_failover: Coordinator-Based Simplicity

For teams running smaller topologies (2–3 nodes), pg_auto_failover provides a monitor-based approach with less operational overhead than Patroni's etcd dependency:

# Initialize the monitor node (runs on a dedicated host)
pg_autoctl create monitor \
  --pgdata /data/pg_auto_failover/monitor \
  --pgport 5500 \
  --hostname monitor.internal.example.com \
  --auth trust \
  --ssl-self-signed

# Register the primary
pg_autoctl create postgres \
  --pgdata /data/postgresql/main \
  --pgport 5432 \
  --hostname primary.internal.example.com \
  --name pg-primary \
  --monitor postgres://autoctl_node@monitor.internal.example.com:5500/pg_auto_failover \
  --pg-hba-lan

# Register the standby
pg_autoctl create postgres \
  --pgdata /data/postgresql/main \
  --pgport 5432 \
  --hostname standby.internal.example.com \
  --name pg-standby \
  --monitor postgres://autoctl_node@monitor.internal.example.com:5500/pg_auto_failover \
  --pg-hba-lan

The monitor runs a finite state machine. You can query its current state directly:

PostgreSQL:

-- Query pg_auto_failover monitor state
SELECT
    nodename,
    nodehost,
    nodeport,
    goalstate,
    reportedstate,
    reportedpgisrunning,
    reportedrepstate,
    reporttime,
    health,
    EXTRACT(EPOCH FROM (now() - reporttime)) AS seconds_since_report
FROM pgautofailover.node
ORDER BY nodeid;

When seconds_since_report exceeds the health-check-period (default 5 seconds) for the primary node and the standby is within replication_quorum, the monitor transitions the standby's goalstate to primary autonomously. No human action required.

#### SQL Server Always On with Automatic Failover

SQL Server AG automatic failover requires WSFC quorum on Windows or Pacemaker on Linux. The key autonomous configuration is the FAILOVER_MODE = AUTOMATIC clause combined with health check thresholds:

SQL Server:

-- Create AG with autonomous failover configuration
CREATE AVAILABILITY GROUP [AG_Production]
WITH (
    AUTOMATED_BACKUP_PREFERENCE = SECONDARY,
    HEALTH_CHECK_TIMEOUT = 30000,          -- 30 seconds before declaring unhealthy
    DB_FAILOVER = ON,                       -- fail over on database-level health issues
    DTC_SUPPORT = NONE,
    REQUIRED_SYNCHRONIZED_SECONDARIES_TO_COMMIT = 1  -- synchronous commit gate
)
FOR DATABASE [AppDB], [ReportDB]
REPLICA ON
    N'sql-primary' WITH (
        ENDPOINT_URL = N'TCP://sql-primary.internal:5022',
        AVAILABILITY_MODE = SYNCHRONOUS_COMMIT,
        FAILOVER_MODE = AUTOMATIC,
        SEEDING_MODE = AUTOMATIC,
        SECONDARY_ROLE (ALLOW_CONNECTIONS = READ_ONLY)
    ),
    N'sql-secondary' WITH (
        ENDPOINT_URL = N'TCP://sql-secondary.internal:5022',
        AVAILABILITY_MODE = SYNCHRONOUS_COMMIT,
        FAILOVER_MODE = AUTOMATIC,
        SEEDING_MODE = AUTOMATIC,
        SECONDARY_ROLE (ALLOW_CONNECTIONS = READ_ONLY)
    );

-- Set flexible failover policy (SQL Server 2019+)
ALTER AVAILABILITY GROUP [AG_Production]
    MODIFY REPLICA ON N'sql-primary'
    WITH (FAILURE_CONDITION_LEVEL = 3);
-- Level 3: fail on critical SQL Server errors, not just process crashes

FAILURE_CONDITION_LEVEL = 3 is the recommended production setting. Level 1 triggers on server-down only; Level 3 adds SQL Server internal critical errors and orphaned spinlocks. Level 5 (any qualified failure condition) produces too many false positives in multi-tenant environments.

Connection string design matters enormously for autonomous failover. Applications must use the AG listener, not the instance name, and must set MultiSubnetFailover=True and ConnectRetryCount=3 to survive the ~15–30 second WSFC transition window without returning errors to the application layer.

RPO-Aware Autonomous Recovery Decision Engine

Automatic failover tools (Patroni, AG automatic failover) handle the happy path — clean primary failure, healthy synchronous replica, quorum intact. Production disasters rarely follow this path. Your autonomous system needs a decision engine that reasons about recovery options when conditions are degraded: the synchronous replica is also down, the lag breach is severe, the DCS is unreachable, or you face a true split-brain.

The recovery decision engine is a state machine that maps (failure class, current topology state) → recovery action. The topology state is pulled from the cluster's own metadata store:

import psycopg2
import pyodbc
import subprocess
import logging
from enum import Enum
from typing import NamedTuple

logger = logging.getLogger("autonomous_dr")

class RecoveryAction(Enum):
    PROMOTE_SYNC_REPLICA   = "promote_sync_replica"
    PROMOTE_ASYNC_REPLICA  = "promote_async_replica_with_data_loss_ack"
    RESTORE_FROM_BACKUP    = "restore_from_backup"
    WAIT_FOR_PRIMARY       = "wait_for_primary_recovery"
    ESCALATE_TO_HUMAN      = "escalate_to_human"
    FENCE_OLD_PRIMARY      = "fence_old_primary"

class TopologyState(NamedTuple):
    sync_replica_available: bool
    sync_replica_lag_bytes: int
    async_replica_available: bool
    async_replica_lag_bytes: int
    dcs_quorum_healthy: bool
    old_primary_network_reachable: bool
    rpo_bytes: int    # configured RPO converted to bytes

class RecoveryDecisionEngine:

    def decide(self, failure_class, topo: TopologyState) -> RecoveryAction:
        # Split-brain: fence before anything else
        if failure_class == "split_brain":
            if topo.old_primary_network_reachable:
                return RecoveryAction.FENCE_OLD_PRIMARY
            return RecoveryAction.ESCALATE_TO_HUMAN

        # Primary lost, DCS unhealthy — can't safely elect
        if not topo.dcs_quorum_healthy:
            logger.warning("DCS quorum unhealthy — cannot safely elect new primary")
            return RecoveryAction.ESCALATE_TO_HUMAN

        # Primary lost, sync replica available and within RPO
        if (failure_class == "node_loss"
                and topo.sync_replica_available
                and topo.sync_replica_lag_bytes <= topo.rpo_bytes):
            return RecoveryAction.PROMOTE_SYNC_REPLICA

        # Primary lost, sync replica down or lagging, async within RPO
        if (failure_class == "node_loss"
                and not topo.sync_replica_available
                and topo.async_replica_available
                and topo.async_replica_lag_bytes <= topo.rpo_bytes):
            logger.warning(
                "Promoting async replica — lag=%d bytes within RPO=%d bytes",
                topo.async_replica_lag_bytes, topo.rpo_bytes
            )
            return RecoveryAction.PROMOTE_ASYNC_REPLICA

        # Primary lost, all replicas beyond RPO
        if failure_class == "node_loss" and not topo.sync_replica_available:
            logger.error("All replicas exceed RPO — data loss promotion requires human sign-off")
            return RecoveryAction.ESCALATE_TO_HUMAN

        # Replication lag breach on running primary — don't failover, investigate
        if failure_class == "replication_lag":
            return RecoveryAction.WAIT_FOR_PRIMARY  # triggers lag remediation pipeline

        # Data corruption detected
        if failure_class == "data_corruption":
            return RecoveryAction.RESTORE_FROM_BACKUP

        return RecoveryAction.ESCALATE_TO_HUMAN

Notice the deliberate conservatism: the engine escalates to a human whenever data loss would exceed the RPO and a human sign-off was not pre-authorized. This reflects a core ADE principle — autonomous systems must have explicit permission to cause data loss, not implicit permission by default. In practice, you can configure a data_loss_authorized: true flag per environment (acceptable in a dev/staging cluster, never in production without an SLA change).

The fencing action deserves special attention. Promoting a replica while the original primary is still reachable and accepting writes produces split-brain. The fence step must shoot the old primary in the head (STONITH — Shoot The Other Node In The Head) before completing promotion. On bare metal or VM infrastructure this is a power fence via IPMI/BMC. On cloud infrastructure it is an API call:

import boto3

def fence_ec2_instance(instance_id: str, region: str) -> bool:
    """
    Hard fence an EC2 instance by stopping it via AWS API.
    Returns True if fence succeeded, False if API call failed.
    The caller must verify instance state before proceeding with promotion.
    """
    ec2 = boto3.client("ec2", region_name=region)
    try:
        response = ec2.stop_instances(
            InstanceIds=[instance_id],
            Force=True   # equivalent to power cut — no graceful shutdown
        )
        state = response["StoppingInstances"][0]["CurrentState"]["Name"]
        logger.info("Fence initiated for %s — current state: %s", instance_id, state)

        # Poll for stopped state — max 60 seconds
        waiter = ec2.get_waiter("instance_stopped")
        waiter.wait(
            InstanceIds=[instance_id],
            WaiterConfig={"Delay": 5, "MaxAttempts": 12}
        )
        logger.info("Fence confirmed for %s", instance_id)
        return True
    except Exception as exc:
        logger.error("Fence failed for %s: %s", instance_id, exc)
        return False

If the fence fails, promotion must not proceed. A failed fence is an escalation event.

Automated Backup Validation and Point-in-Time Recovery Pipelines

Autonomous DR is only as strong as the backups it can restore from. Most organizations discover their backups are corrupted or incomplete during the actual disaster, not before. The autonomous pattern inverts this: restores are executed continuously in a shadow environment, and any failure in shadow restoration is treated as a current-severity alert.

The continuous restore pipeline operates on a schedule (typically every 4–8 hours for production databases) and validates three properties: restore completeness, data integrity, and RPO alignment.
