Chapter 39: Maintenance Playbooks

Databases don't maintain themselves. Behind every stable, high-performing production system is a DBA who has thought carefully about what needs to happen regularly, under what conditions, and in what order. This chapter brings together the recurring tasks that keep PostgreSQL and SQL Server healthy—vacuuming and statistics gathering, index maintenance, integrity checks, log management, and scheduled job coordination—and organizes them into actionable playbooks you can adapt for your own environment. Rather than treating these tasks in isolation, we'll look at how they interact, how to sequence them intelligently, and how to handle the edge cases that surface only after systems have been running for months or years.

---

Understanding the Maintenance Landscape

Before writing a single maintenance script, it's worth stepping back and asking what "maintenance" actually means for a database system. At the highest level, maintenance is the set of activities that prevent gradual degradation. Databases accumulate dead rows, statistics drift away from reality, indexes fragment, transaction ID counters advance toward dangerous limits, and log files grow without bound. None of these problems are dramatic on day one—they're slow burns that become crises six months later when nobody has been watching.

PostgreSQL and SQL Server approach the maintenance problem differently, and those differences shape the playbooks you'll build. PostgreSQL uses a multi-version concurrency control (MVCC) model that leaves dead row versions in place until VACUUM reclaims them. Without regular vacuuming, tables bloat, sequential scans slow down, and in the worst case, transaction ID wraparound halts the entire cluster. SQL Server uses a row-versioning or locking model depending on isolation level, and its equivalent maintenance concerns center on page fragmentation inside the B-tree structures it calls clustered and nonclustered indexes, as well as statistics staleness that causes the query optimizer to make poor plan choices.

Both platforms have auto-maintenance mechanisms built in—PostgreSQL's autovacuum and SQL Server's auto-update statistics—but neither should be treated as a complete solution on its own. Autovacuum is designed for steady-state workloads. It will fall behind on tables that receive large batch loads or suffer from high update rates. SQL Server's automatic statistics updates fire based on a row-modification threshold that doesn't scale linearly with table size; a billion-row table needs a change to twenty percent of rows before an automatic update triggers, which almost never happens. Manual maintenance jobs fill in the gaps that automation leaves.

The maintenance landscape also includes tasks that no automatic mechanism handles at all: validating physical data integrity with DBCC CHECKDB or pg_filechecksum-based monitoring, archiving or rotating transaction logs, purging old audit data, refreshing materialized views, and validating foreign key constraints that may have been temporarily suspended during bulk loads. A mature maintenance playbook covers all of these, with clear ownership, documented schedules, and alerting when something fails to run or runs longer than expected.

Building the Vacuum and Statistics Playbook for PostgreSQL

In PostgreSQL, VACUUM is not optional—it is a fundamental part of how the storage engine functions. VACUUM reclaims space occupied by dead tuples, updates the visibility map so that index-only scans can work correctly, and advances the freeze horizon to prevent transaction ID wraparound. ANALYZE collects column statistics that the planner uses to estimate row counts and choose join strategies. In most shops, AUTOVACUUM handles both of these continuously in the background, but the DBA needs to understand its behavior well enough to know when to intervene manually.

The first sign that autovacuum is falling behind is usually bloat. A table that should occupy 2 GB on disk starts using 8 GB. Queries that were fast start doing far more I/O than expected. You can observe this by comparing the live tuple count to the dead tuple count in pg_stat_user_tables.

PostgreSQL:

-- Identify tables with significant dead tuple accumulation
SELECT
    schemaname,
    relname AS table_name,
    n_live_tup,
    n_dead_tup,
    ROUND(100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS dead_pct,
    last_vacuum,
    last_autovacuum,
    last_analyze,
    last_autoanalyze
FROM pg_stat_user_tables
WHERE n_dead_tup > 10000
ORDER BY dead_pct DESC;

When you find a table with a dead tuple percentage above ten or fifteen percent, autovacuum should have already addressed it—if it hasn't, the first question is whether autovacuum is running at all, and the second is whether the table has storage parameters that override the default thresholds. Tables with a very high row velocity sometimes need custom autovacuum settings applied directly to them.

For the manual vacuum playbook, the key distinction is between VACUUM and VACUUM FULL. Regular VACUUM reclaims dead space for reuse by future inserts but doesn't return space to the operating system—the file size stays the same. VACUUM FULL rewrites the entire table, compacting it to minimum size, but requires an ACCESS EXCLUSIVE lock, which blocks all reads and writes for the duration. On a large table, that could be hours. In practice, VACUUM FULL should be reserved for tables where bloat is so severe that it's causing real performance problems and you have a maintenance window to accommodate the lock.

A safer alternative to VACUUM FULL for reducing table size without extended locking is the pg_repack extension, which rebuilds tables in the background using triggers to capture concurrent changes, swapping the old and new versions at the end with only a brief lock.

Transaction ID wraparound deserves its own attention in any PostgreSQL maintenance playbook. Every row in PostgreSQL is stamped with the transaction ID (XID) that inserted it. XIDs are 32-bit unsigned integers, meaning they wrap around after about 4 billion transactions. PostgreSQL tracks how far the oldest unfrozen XID is from the wraparound limit, and if the gap falls below the autovacuum_freeze_max_age threshold (default 200 million transactions), it will force an aggressive VACUUM on affected tables. If the gap falls below vacuum_freeze_min_age, PostgreSQL will begin freezing rows during regular vacuums. The DBA's job is to make sure this never becomes an emergency.

PostgreSQL:

-- Monitor transaction ID wraparound risk by database
SELECT
    datname,
    age(datfrozenxid) AS xid_age,
    2147483647 - age(datfrozenxid) AS xids_remaining,
    ROUND(100.0 * age(datfrozenxid) / 2147483647, 2) AS pct_toward_wraparound
FROM pg_database
ORDER BY xid_age DESC;

A pct_toward_wraparound above fifteen percent should be a monitored alert. Above thirty percent, you should be scheduling manual VACUUM FREEZE runs during off-peak hours on the oldest tables in each affected database.

For statistics, the maintenance playbook should include a scheduled ANALYZE run after any significant batch load or ETL job, because autovacuum's ANALYZE pass may not fire quickly enough for the query planner to produce good plans for the next batch of work. A simple approach is to ANALYZE the tables that were modified as part of the batch job script itself, immediately after the data load completes.

Index Maintenance Playbooks for Both Platforms

Index maintenance serves two purposes: removing fragmentation that forces the engine to read more pages than necessary, and identifying indexes that are either missing or have stopped being used so resources aren't wasted maintaining them. The mechanics differ significantly between PostgreSQL and SQL Server, so it's worth treating each platform separately before looking at the combined strategy.

In SQL Server, index fragmentation is measured as the percentage of pages in the index that are out of logical order relative to their physical location on disk. This matters most for range scans, where the engine benefits from physically sequential reads. SQL Server provides two remediation operations: ALTER INDEX ... REORGANIZE, which defragments in place with minimal locking and can be interrupted and restarted, and ALTER INDEX ... REBUILD, which reconstructs the index from scratch. REBUILD is more thorough and also updates statistics as a side effect, but it takes an exclusive lock on the index unless you specify ONLINE = ON (available in Enterprise Edition).

The conventional guidance is to use REORGANIZE for fragmentation between fifteen and thirty percent, and REBUILD for fragmentation above thirty percent, though these thresholds should be treated as starting points rather than rules. For very small indexes, fragmentation numbers are statistically meaningless—rebuilding a 10-page index every week wastes more I/O than the fragmentation would ever cost.

SQL Server:

-- Identify index fragmentation across a database
SELECT
    OBJECT_SCHEMA_NAME(ips.object_id) AS schema_name,
    OBJECT_NAME(ips.object_id) AS table_name,
    i.name AS index_name,
    ips.index_type_desc,
    ROUND(ips.avg_fragmentation_in_percent, 2) AS fragmentation_pct,
    ips.page_count,
    ips.record_count
FROM sys.dm_db_index_physical_stats(
    DB_ID(), NULL, NULL, NULL, 'LIMITED') AS ips
JOIN sys.indexes i
    ON ips.object_id = i.object_id
    AND ips.index_id = i.index_id
WHERE ips.avg_fragmentation_in_percent > 10
  AND ips.page_count > 100
ORDER BY ips.avg_fragmentation_in_percent DESC;

In PostgreSQL, the fragmentation story is different. PostgreSQL's B-tree indexes can accumulate dead index entries that point to dead heap tuples, which inflate the index without providing any value. Regular VACUUM reclaims these dead index entries as part of its normal operation. There is no equivalent to SQL Server's REORGANIZE—the main tool for reclaiming index space is REINDEX, which rebuilds the index from scratch and requires an ACCESS EXCLUSIVE lock unless you use the CONCURRENTLY option, which avoids blocking but takes longer and cannot run inside a transaction block.

In practice, most PostgreSQL indexes don't need manual reindexing if autovacuum is running properly. The cases where reindexing helps are heavily updated indexes where many deleted entries accumulate faster than vacuum can reclaim them, or indexes that have grown significantly through one-time bulk inserts and now have structural imbalance.

PostgreSQL:

-- Find bloated indexes using pg_stat_user_indexes and size information
SELECT
    schemaname,
    tablename,
    indexname,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
    idx_scan AS scans_since_reset,
    idx_tup_read,
    idx_tup_fetch
FROM pg_stat_user_indexes
WHERE idx_scan < 50
  AND pg_relation_size(indexrelid) > 10 * 1024 * 1024  -- larger than 10 MB
ORDER BY pg_relation_size(indexrelid) DESC;

Identifying unused indexes deserves its own place in the maintenance playbook. Both platforms maintain statistics about how frequently each index is accessed, but there's an important caveat: SQL Server's index usage stats reset when the service restarts, and PostgreSQL's reset when pg_stat_reset() is called or the server restarts. Before removing an index based on usage stats, verify that the collection window is long enough to cover all workload patterns—including month-end reports, quarterly batch jobs, and infrequent operational queries that might run only a few times a year. A safe practice is to mark candidate indexes for removal, communicate the intent to application owners, and observe for at least one full business cycle before dropping.

The index maintenance playbook should also include a step to check for duplicate indexes—multiple indexes on the same columns in the same order, or indexes that are strict prefixes of other indexes and therefore redundant for most query patterns. On both platforms, duplicate indexes increase write overhead and consume storage without providing additional query acceleration.

Database Integrity Checks and Corruption Prevention

The most expensive database problem is one that goes undetected for months—a storage-level corruption that slowly propagates into backups, making recovery impossible. Integrity checks are the insurance policy against this scenario. Running them consistently and acting on failures immediately is not optional in any production environment.

SQL Server provides DBCC CHECKDB as its primary integrity validation tool. It verifies the structural integrity of all objects in the database: allocation structures, B-tree consistency, row data against index data, and cross-page linkages. A full CHECKDB on a large database can take hours and consumes significant I/O bandwidth, which is why many shops run it during off-peak maintenance windows or use the PHYSICAL_ONLY option during frequent checks and reserve full logical consistency checks for weekly runs.

SQL Server:

-- Run a full integrity check with informational output
DBCC CHECKDB (N'YourDatabaseName')
    WITH NO_INFOMSGS, ALL_ERRORMSGS;

-- Faster physical-only check suitable for nightly runs on large databases
DBCC CHECKDB (N'YourDatabaseName')
    WITH PHYSICAL_ONLY, NO_INFOMSGS;

When CHECKDB reports errors, the severity determines the response. Allocation errors and B-tree consistency failures often indicate hardware or storage problems and may require restoring from backup. Never run CHECKDB repair options (REPAIR_ALLOW_DATA_LOSS) without exhausting restore options first—the repair commands sacrifice data to restore structural consistency, which is sometimes necessary but should never be the first response.

PostgreSQL doesn't have a built-in equivalent to CHECKDB, which is one of the more significant operational gaps between the two platforms. The closest built-in tool is VACUUM VERBOSE, which reports any page-level errors it encounters while scanning tables, but it doesn't actively validate B-tree index consistency. The amcheck extension, available in PostgreSQL 10 and later, provides functions to check B-tree index structural integrity without any locking.

PostgreSQL:

-- Use amcheck to verify B-tree index consistency (requires amcheck extension)
CREATE EXTENSION IF NOT EXISTS amcheck;

-- Check a specific index for structural corruption
SELECT bt_index_check('your_schema.your_index_name'::regclass);

-- Check with heap verification (more thorough, checks index-to-heap cross-references)
SELECT bt_index_parent_check(
    'your_schema.your_index_name'::regclass,
    heapalloc => true
);

For table-level data verification in PostgreSQL, the pg_filechecksum utility introduced in PostgreSQL 12 (when data checksums are enabled at cluster initialization with initdb --data-checksums) detects block-level corruption by verifying stored checksums against computed checksums when pages are read. Data checksums should be enabled on all new PostgreSQL clusters—the performance overhead is typically one to two percent, and the early detection capability is worth far more than that.

The integrity check playbook should document expected runtimes for each check so that you have a baseline. When a check starts taking significantly longer than its baseline, it can indicate either growing database size (expected and acceptable) or an underlying storage performance issue (needs investigation). Capturing check runtimes in a logging table alongside the completion status gives you a historical record that makes anomaly detection straightforward.

A complete integrity playbook also includes verifying backup integrity. A CHECKDB that passes on the live database is necessary but not sufficient—you also need to know that your backups are restorable. For SQL Server, this means periodically restoring a backup to a test server and running CHECKDB against the restored copy. For PostgreSQL, it means testing pg_restore or base backup recovery on a non-production server at regular intervals. The backup that has never been tested is not a backup—it's a hope.

Scheduling, Logging, and Operational Hygiene

A maintenance task that runs unreliably is only marginally better than one that doesn't run at all. The operational infrastructure around your maintenance jobs—how they're scheduled, how their outcomes are recorded, how failures are surfaced—is as important as the maintenance tasks themselves.

Both platforms provide native scheduling mechanisms. SQL Server Agent is the standard scheduler for SQL Server environments, with job steps, schedules, alerts, and notifications built in. PostgreSQL has no built-in scheduler; the common choices are pg_cron (a PostgreSQL extension that provides cron-syntax scheduling inside the database), external system cron, or dedicated job scheduling tools like pgAgent. In containerized or cloud environments, Kubernetes CronJobs or cloud-native schedulers often take over this role.

Regardless of what technology handles scheduling, every maintenance job should follow the same operational pattern:

1. Record the start time and the job name to a maintenance log table.

2. Execute the maintenance work.

3. Record the completion time, row count affected (where applicable), and success or failure status.

4. If the job fails, write the error detail and raise an alert.

This pattern makes the maintenance log table a first-class operational artifact. When someone asks why a query is suddenly slow, you can check whether the most recent statistics refresh ran on schedule. When a developer asks why index fragmentation is high on a specific table, you can check whether the index maintenance job ran successfully and whether that table has unusually high write volume.

SQL Server:

-- Create a maintenance log table and record job execution
CREATE TABLE dba_maintenance.maintenance_log (
    log_id          INT IDENTITY(1,1) PRIMARY KEY,
    job_name        NVARCHAR(200) NOT NULL,
    target_object   NVARCHAR(500) NULL,
    started_at      DATETIME2 NOT NULL DEFAULT SYSDATETIME(),
    completed_at    DATETIME2 NULL,
    rows_affected   BIGINT NULL,
    status          NVARCHAR(20) NOT NULL DEFAULT 'RUNNING',
    error_message   NVARCHAR(MAX) NULL
);

-- At the start of a maintenance step:
DECLARE @log_id INT;
INSERT INTO dba_maintenance.maintenance_log (job_name, target_object)
VALUES ('IndexRebuild', 'Sales.Orders');
SET @log_id = SCOPE_IDENTITY();

-- At completion:
UPDATE dba_maintenance.maintenance_log
SET completed_at  = SYSDATETIME(),
    status        = 'SUCCESS',
    rows_affected = @@ROWCOUNT
WHERE log_id = @log_id;

PostgreSQL:

-- Equivalent maintenance log table in PostgreSQL
CREATE TABLE dba_maintenance.maintenance_log (
    log_id          BIGSERIAL PRIMARY KEY,
    job_name        TEXT NOT NULL,
    target_object   TEXT,
    started_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    completed_at    TIMESTAMPTZ,
    rows_affected   BIGINT,
    status          TEXT NOT NULL DEFAULT 'RUNNING',
    error_message   TEXT
);

-- Record the start of a job and capture the ID for later update
WITH ins AS (
    INSERT INTO dba_maintenance.maintenance_log (job_name, target_object)
    VALUES ('vacuum_analyze', 'public.orders')
    RETURNING log_id
)
SELECT log_id INTO TEMP TABLE _current_job FROM ins;

-- On completion, update the log record
UPDATE dba_maintenance.maintenance_log
SET completed_at = now(),
    status       = 'SUCCESS'
WHERE log_id = (SELECT log_id FROM _current_job);

Maintenance windows deserve explicit planning rather than implicit assumptions. The question "when can we run this?" has different answers for different tasks. A VACUUM ANALYZE on a non-critical table might run during business hours with minimal impact. A VACUUM FULL that holds an exclusive lock for forty minutes needs a maintenance window agreement with application owners. CHECKDB on a 2 TB database will saturate I/O for several hours and should run during the deepest off-peak period available.

Documenting maintenance windows—not just the schedules, but the rationale and the stakeholders who agreed to them—prevents the frustrating situation where a maintenance job that's been running successfully for two years suddenly becomes "causing problems" because a new application team didn't know it existed. The maintenance playbook should be a document that other teams can read, not just a collection of SQL Agent jobs.

Finally, maintenance jobs need to be protected against overlap. If a nightly index rebuild is still running when the next night's job starts, you'll end up with concurrent rebuilds competing for resources and potentially deadlocking. Both SQL Server Agent and pg_cron have mechanisms for detecting running jobs and preventing duplicate execution, but verifying that these protections are in place—rather than assuming they are—is part of good operational hygiene.

Tuning Maintenance for Large-Scale and High-Availability Environments

At a certain scale, the maintenance playbooks that work for a 500 GB single-server deployment start to create their own problems. A CHECKDB that takes forty minutes on a 200 GB database takes four hours on a 2 TB database and fourteen hours on a 10 TB database. An index rebuild job that cycles through fifty tables in twenty minutes needs to cycle through five hundred tables without blowing past the maintenance window. And in high-availability environments—Always On Availability Groups, Patroni clusters, streaming replicas—maintenance activities on the primary can have unexpected effects on secondaries.

For SQL Server at scale, the standard approach to index maintenance is to move from blanket rebuild/reorganize jobs to smart, adaptive scripts that query fragmentation levels at runtime and only perform work above meaningful thresholds. The Ola Hallengren maintenance solution is the industry-standard implementation of this approach, handling index fragmentation, statistics updates, and integrity checks with configurable thresholds, parallelism controls, and detailed logging. Rather than rebuilding every index nightly, it rebuilds only those that need it, which dramatically reduces the total work and makes the maintenance window predictable regardless of how many tables the database contains.

For PostgreSQL at scale, the equivalent strategy is tuning autovacuum aggressiveness per table rather than running manual VACUUM jobs across the board. Tables with high transaction rates need lower autovacuum thresholds so that vacuum fires more frequently with smaller accumulated dead tuples. Tables with very low write rates can have higher thresholds so autovacuum doesn't spend cycles on them unnecessarily.

-- PostgreSQL: Set aggressive autovacuum for a high-write table
ALTER TABLE public.events
SET (
    autovacuum_vacuum_scale_factor = 0.01,    -- vacuum when 1% of rows are dead
    autovacuum_analyze_scale_factor = 0.005,  -- analyze when 0.5% of rows changed
    autovacuum_vacuum_cost_delay = 2           -- reduce IO throttling for this table
);

In SQL Server Always On Availability Groups, index rebuilds on the primary generate redo log that must be applied on secondary replicas. For a large index rebuild, this can cause the secondary to lag behind the primary, which matters if read workloads are routed to the secondary. ONLINE index rebuilds, while they avoid blocking the primary, still generate substantial log volume. Strategies to manage this include performing rebuilds in smaller batches (one partition at a time on partitioned tables), scheduling them during windows when secondary read workloads are low, and monitoring secondary redo queue during maintenance to detect and pause if lag grows too large.

PostgreSQL streaming replicas are similarly affected by heavy write operations during maintenance. A VACUUM FULL or REINDEX CONCURRENTLY generates WAL that must be shipped and replayed on standbys. In some configurations, particularly with synchronous replication, this can stall the primary if standbys can't keep up. Monitoring replication lag before and during maintenance operations is good practice, with documented thresholds at which the DBA should pause or throttle maintenance work.

For integrity checks at scale, SQL Server Enterprise Edition's CHECKDB supports parallel execution across multiple CPUs, and the WITH PHYSICAL_ONLY option dramatically reduces runtime by skipping logical consistency checks. A practical approach for very large databases is to run physical checks nightly and full logical checks weekly, potentially using a secondary replica as the target for CHECKDB to avoid impacting the primary (SQL Server allows running CHECKDB against a database snapshot, which can be taken from a secondary in some configurations).

A final consideration for large environments is maintenance of the maintenance system itself. Log tables accumulate rows. Statistics on the maintenance log table itself can go stale. Jobs that ran once while a database was 100 GB may need their timeout parameters revisited now that the database is 2 TB. Building a quarterly review of maintenance configurations into the DBA calendar—reviewing runtimes, adjusting thresholds, removing jobs for objects that no longer exist, and adding jobs for new high-volume tables—is what separates a maintenance system that drifts toward failure from one that continues to serve the database well over time.
