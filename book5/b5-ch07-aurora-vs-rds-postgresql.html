Chapter 7: Aurora PostgreSQL vs RDS PostgreSQL: When to Choose Which

Aurora PostgreSQL and RDS PostgreSQL both run PostgreSQL, both live inside your VPC, and both appear in the same AWS console. That surface-level similarity trips up engineers who assume they're interchangeable and then build the wrong architecture for their workload. They are not the same product. They have different storage engines, different replication topologies, different failover mechanics, different cost profiles, and different ceilings. This chapter gives you the framework to make the right call before you provision — not after you've already migrated 3 TB of data and discovered that Aurora's distributed storage doesn't behave the way you expected under write-heavy OLTP load.

---

The Architectural Divide: What's Actually Different Under the Hood

If you've run PostgreSQL on bare metal or on VMware, your mental model looks like this: a primary server writes to local disk using the PostgreSQL WAL mechanism, standby servers replay that WAL, and storage is something you manage at the OS layer — LVM, ZFS, EBS volumes, SAN LUNs. Both Aurora and RDS start by dismantling that model, but they do it in different ways.

RDS PostgreSQL is the more conservative option. AWS provisions an EC2 instance, attaches EBS storage to it, and runs stock PostgreSQL on top. The PostgreSQL version you get is a genuine community PostgreSQL build, with minor AWS patches for authentication and monitoring. Multi-AZ standby is implemented using synchronous physical replication, identical to what you'd configure manually with synchronous_commit = on and a hot standby. When the primary fails, AWS promotes the standby — typically within 20–60 seconds depending on instance size and database load. The storage is EBS gp3 or io2 volumes. You manage storage capacity. You choose your IOPS tier. You pay for provisioned storage whether you use it or not.

Aurora PostgreSQL replaces this entire I/O path. Aurora separates compute from storage entirely. Your PostgreSQL process doesn't write to local EBS; it writes log records across Aurora's distributed storage layer — six copies across three Availability Zones, maintained by a purpose-built storage service that handles its own quorum writes. The PostgreSQL engine on the primary only needs to confirm four of six storage nodes acknowledged the write. The standby instances don't replay WAL from the primary at all — they share the same storage layer and simply maintain their buffer caches. This is why Aurora can promote a read replica to primary in under 30 seconds without any data copy or log replay.

The practical consequences of this architecture:

  • → **Write amplification**: Aurora writes six copies of every log record at the storage layer. For write-heavy workloads with small transactions, this storage overhead is real. On large instances pushing tens of thousands of writes per second, you'll see higher per-write latency than equivalent RDS PostgreSQL on io2.
  • → **Read scaling**: Aurora read replicas share storage, so they're always in sync to within single-digit milliseconds. RDS read replicas replicate asynchronously over network WAL streaming and routinely lag 5–30 seconds under heavy write load.
  • → **Storage billing**: Aurora bills for storage consumed, not provisioned, starting at ~$0.10/GB-month. It grows automatically to 128 TB. RDS charges for provisioned storage whether you use it or not, and you must manage size manually or configure storage autoscaling.

    • Understanding which of these trade-offs matters for your workload is the entire decision.

    ---

    When RDS PostgreSQL Is the Right Answer

    RDS PostgreSQL wins in a specific class of workload: write-intensive, cost-sensitive, or requiring strict community PostgreSQL compatibility.

    Write-heavy OLTP with high transaction rates. Aurora's distributed write path introduces latency that flat EBS does not. In benchmarks on db.r6g.2xlarge instances (8 vCPU, 64 GB RAM) pushing 50,000 transactions per second with sub-millisecond writes, RDS PostgreSQL on io2 with 32,000 IOPS consistently beats Aurora PostgreSQL on commit latency by 20–40%. If your workload is write-bound — logging pipelines, high-frequency inserts, heavy UPDATE/DELETE churn — RDS PostgreSQL on io2 is frequently the better choice.

    Strict PostgreSQL version parity requirements. Aurora PostgreSQL lags behind community PostgreSQL releases by 6–18 months. At the time of writing, Aurora PostgreSQL 16.x is available, but specific minor versions and extensions you rely on may not be. RDS PostgreSQL tracks community releases faster. If your application requires a specific extension version, a specific PostgreSQL minor version, or a recently released PostgreSQL feature, check Aurora's compatibility matrix before assuming it's available. Extensions like pg_partman, TimescaleDB, pgvector, and PostGIS have different version constraints on Aurora versus RDS.

    Single-tenant or small databases with predictable storage. For databases under 100 GB with well-understood growth, Aurora's storage pricing provides no advantage, and the additional per-request storage cost can exceed the RDS provisioned storage cost. A 50 GB database on db.t4g.medium running RDS PostgreSQL with gp3 storage is meaningfully cheaper per month than the same database on Aurora.

    Cost modeling matters here. A useful back-of-envelope: RDS PostgreSQL on db.r6g.xlarge (4 vCPU, 32 GB) with 500 GB gp3 storage runs approximately $350–400/month in us-east-1. An equivalent Aurora PostgreSQL configuration on the same instance class, consuming 500 GB of Aurora storage, runs approximately $420–460/month. The Aurora I/O request charges — $0.20 per million requests — add significantly to that number under high read/write workloads and can double the storage cost for I/O-intensive workloads unless you use Aurora I/O Optimized pricing (which eliminates per-request charges at a higher storage rate of ~$0.225/GB-month).

    ---

    When Aurora PostgreSQL Is the Right Answer

    Aurora earns its higher base cost in several specific scenarios where its architecture genuinely changes what's operationally possible.

    Multi-reader workloads requiring near-zero replication lag. If you're running reporting, analytics, or read-heavy application tiers against read replicas, Aurora's shared storage architecture gives you replicas that are current to within 10–20 milliseconds. This isn't tunable in RDS — WAL-based replication lag is a function of network throughput and write rate, and under heavy write load it climbs. Applications that route reads to replicas and expect fresh data need Aurora. RDS replicas are appropriate only when staleness of seconds or tens of seconds is acceptable.

    High-availability requirements with sub-30-second failover. RDS Multi-AZ failover involves a DNS cutover after the standby is promoted and confirmed healthy. Under typical conditions this takes 20–60 seconds, but during high-load or large buffer flush situations it can extend to 90–120 seconds. Aurora's failover to a replica — because no storage transfer occurs — completes in under 30 seconds, often under 15. For applications with aggressive availability SLAs (99.99% uptime targets translate to under 52 minutes of annual downtime), Aurora's failover behavior is often the deciding factor.

    Aurora Serverless v2 for variable workloads. Aurora Serverless v2 scales compute capacity in increments of 0.5 Aurora Capacity Units (ACUs), where 1 ACU is approximately 2 GB of RAM and proportional CPU. It scales up within seconds and scales down during idle periods. For development environments, batch workloads with spiky demand, or SaaS applications with unpredictable per-tenant load, Serverless v2 eliminates the need to overprovision. RDS has no equivalent — you provision a fixed instance class and it runs at that size regardless of load.

    Large storage with automatic growth. Aurora storage grows automatically in 10 GB increments, up to 128 TB, without downtime or manual intervention. If your database routinely grows unpredictably — large time-series tables, append-heavy audit logs, media metadata at scale — Aurora's storage management eliminates a class of operational work that RDS requires (monitoring storage thresholds, triggering resize events, managing replication lag during storage scaling).

    Aurora Global Database for cross-region disaster recovery. Aurora Global Database replicates across up to five regions with typical replication lag under one second. You get a cross-region standby that can be promoted in under a minute. The equivalent RDS architecture requires a cross-region read replica, which has higher replication lag, longer promotion time, and more complex DNS management. If cross-region DR with near-zero RPO is a hard requirement, Aurora Global Database is the only RDS-family option that meets it.

    Provisioning and Configuration: Hands-On Comparison

    Here's how each looks at provisioning time using Terraform and the AWS CLI. Understanding the configuration surface tells you exactly where the operational differences live.

    Provisioning RDS PostgreSQL with Multi-AZ:

    # terraform/rds_postgresql.tf
resource "aws_db_instance" "prod_postgres" {
  identifier        = "prod-postgres-primary"
  engine            = "postgres"
  engine_version    = "16.3"
  instance_class    = "db.r6g.2xlarge"
  allocated_storage = 500
  storage_type      = "gp3"
  iops              = 12000
  storage_throughput = 500

  db_name  = "appdb"
  username = "dbadmin"
  password = var.db_password

  multi_az               = true
  db_subnet_group_name   = aws_db_subnet_group.private.name
  vpc_security_group_ids = [aws_security_group.rds_sg.id]

  # Parameter group with production tunings
  parameter_group_name = aws_db_parameter_group.postgres16_prod.name

  backup_retention_period   = 14
  backup_window             = "03:00-04:00"
  maintenance_window        = "sun:04:00-sun:05:00"
  deletion_protection       = true
  performance_insights_enabled = true
  performance_insights_retention_period = 731  # 2 years

  enabled_cloudwatch_logs_exports = ["postgresql", "upgrade"]

  tags = {
    Environment = "production"
    Team        = "platform-db"
  }
}

resource "aws_db_parameter_group" "postgres16_prod" {
  name   = "postgres16-prod"
  family = "postgres16"

  parameter {
    name  = "shared_buffers"
    value = "{DBInstanceClassMemory/4}"
  }

  parameter {
    name  = "max_connections"
    value = "500"
  }

  parameter {
    name  = "work_mem"
    value = "65536"  # 64 MB
  }

  parameter {
    name  = "checkpoint_completion_target"
    value = "0.9"
  }

  parameter {
    name  = "log_min_duration_statement"
    value = "1000"  # Log queries over 1 second
  }

  parameter {
    name  = "log_autovacuum_min_duration"
    value = "0"  # Log all autovacuum activity
  }
}

    Provisioning Aurora PostgreSQL with a read replica:

    # terraform/aurora_postgresql.tf
resource "aws_rds_cluster" "aurora_prod" {
  cluster_identifier      = "aurora-prod-cluster"
  engine                  = "aurora-postgresql"
  engine_version          = "16.2"
  database_name           = "appdb"
  master_username         = "dbadmin"
  master_password         = var.db_password

  db_subnet_group_name            = aws_db_subnet_group.private.name
  vpc_security_group_ids          = [aws_security_group.aurora_sg.id]
  db_cluster_parameter_group_name = aws_rds_cluster_parameter_group.aurora16_prod.name

  backup_retention_period = 14
  preferred_backup_window = "03:00-04:00"
  deletion_protection     = true

  # Aurora I/O Optimized eliminates per-request charges
  # Enable this when I/O costs exceed 25% of compute costs
  storage_type = "aurora-iopt1"

  enabled_cloudwatch_logs_exports = ["postgresql"]

  tags = {
    Environment = "production"
    Team        = "platform-db"
  }
}

resource "aws_rds_cluster_instance" "aurora_writer" {
  identifier         = "aurora-prod-writer"
  cluster_identifier = aws_rds_cluster.aurora_prod.id
  instance_class     = "db.r6g.2xlarge"
  engine             = aws_rds_cluster.aurora_prod.engine
  engine_version     = aws_rds_cluster.aurora_prod.engine_version

  performance_insights_enabled          = true
  performance_insights_retention_period = 731
  monitoring_interval                   = 60
  monitoring_role_arn                   = aws_iam_role.rds_enhanced_monitoring.arn

  db_parameter_group_name = aws_db_parameter_group.aurora16_instance_prod.name
}

resource "aws_rds_cluster_instance" "aurora_readers" {
  count              = 2
  identifier         = "aurora-prod-reader-${count.index}"
  cluster_identifier = aws_rds_cluster.aurora_prod.id
  instance_class     = "db.r6g.xlarge"  # Readers can be smaller
  engine             = aws_rds_cluster.aurora_prod.engine
  engine_version     = aws_rds_cluster.aurora_prod.engine_version

  # Readers in different AZs for HA
  availability_zone = data.aws_availability_zones.available.names[count.index]

  performance_insights_enabled = true
  db_parameter_group_name      = aws_db_parameter_group.aurora16_instance_prod.name
}

# Aurora has two-level parameter groups: cluster-level and instance-level
resource "aws_rds_cluster_parameter_group" "aurora16_prod" {
  name   = "aurora16-prod-cluster"
  family = "aurora-postgresql16"

  parameter {
    name  = "shared_preload_libraries"
    value = "pg_stat_statements,auto_explain"
  }

  parameter {
    name  = "log_min_duration_statement"
    value = "1000"
  }
}

resource "aws_db_parameter_group" "aurora16_instance_prod" {
  name   = "aurora16-prod-instance"
  family = "aurora-postgresql16"

  parameter {
    name  = "max_connections"
    value = "1000"
  }

  parameter {
    name  = "work_mem"
    value = "65536"
  }
}

    Note the structural difference: Aurora has cluster-level parameter groups (applied to the cluster) and instance-level parameter groups (applied to each instance). RDS has only instance-level parameter groups. This matters because certain parameters — shared_preload_libraries, replication settings, and Aurora-specific parameters — must go into the cluster parameter group. Putting them in the instance parameter group has no effect, which is a common mistake when migrating RDS parameter group configurations to Aurora.

    Failover, Replication, and Connection Management

    Both services handle failover via DNS endpoint updates. Your application connects to the cluster endpoint (for Aurora) or the instance endpoint (for RDS), and AWS updates the DNS record to point at the new primary after failover. This means your application TTL settings matter — if your application caches DNS for 300 seconds, it will not discover the new primary for up to 5 minutes regardless of how fast the failover itself occurs.

    For RDS PostgreSQL, the recommended mitigation is RDS Proxy. For Aurora, RDS Proxy is also available and adds connection pooling, but Aurora's own cluster endpoint DNS is designed with a 5-second TTL, which reduces connection failover time significantly without requiring Proxy. Both benefit from Proxy when you have large numbers of application connections (hundreds of Lambda functions, microservices with short-lived connections, or connection counts that approach max_connections).

    Setting up RDS Proxy for Aurora PostgreSQL:

    # Create IAM role for RDS Proxy to access Secrets Manager
aws iam create-role \
  --role-name rds-proxy-role \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"Service": "rds.amazonaws.com"},
      "Action": "sts:AssumeRole"
    }]
  }'

# Store credentials in Secrets Manager (don't use parameter groups for passwords)
aws secretsmanager create-secret \
  --name prod/aurora/dbadmin \
  --secret-string '{"username":"dbadmin","password":"your-password-here"}'

# Create the RDS Proxy
aws rds create-db-proxy \
  --db-proxy-name aurora-prod-proxy \
  --engine-family POSTGRESQL \
  --auth '[{
    "AuthScheme": "SECRETS",
    "SecretArn": "arn:aws:secretsmanager:us-east-1:123456789:secret:prod/aurora/dbadmin",
    "IAMAuth": "REQUIRED"
  }]' \
  --role-arn arn:aws:iam::123456789:role/rds-proxy-role \
  --vpc-subnet-ids subnet-aaa subnet-bbb subnet-ccc \
  --vpc-security-group-ids sg-proxy-sg \
  --require-tls

# Register the proxy target — point it at the Aurora cluster
aws rds register-db-proxy-targets \
  --db-proxy-name aurora-prod-proxy \
  --db-cluster-identifiers aurora-prod-cluster

    With RDS Proxy in front of Aurora, you get connection multiplexing — 5,000 application connections can map to a much smaller pool of actual database connections. Proxy is not free: it charges approximately $0.015 per vCPU-hour of the underlying database instance class, so for a db.r6g.2xlarge (8 vCPU) it adds roughly $86/month.

    Testing replication lag on Aurora read replicas:

    -- Run on Aurora read replica to check replication currency
-- Aurora uses aurora_replica_status(), not standard pg_stat_replication
SELECT server_id,
       session_id,
       aws_query_id,
       durable_lsn,
       highest_lsn_rcvd,
       feedback_epoch,
       feedback_xmin,
       replica_lag_in_msec,   -- This is what you care about
       log_stream_speed_in_kib_per_second,
       log_buffer_sequence_number,
       log_buffer_count
FROM aurora_replica_status();

-- For RDS PostgreSQL read replicas, check from the primary:
SELECT client_addr,
       state,
       sent_lsn,
       write_lsn,
       flush_lsn,
       replay_lsn,
       write_lag,
       flush_lag,
       replay_lag
FROM pg_stat_replication;

-- Set up a replication slot health check for RDS read replicas
-- Check for dangerous slot lag that blocks WAL cleanup:
SELECT slot_name,
       active,
       pg_size_pretty(
         pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn)
       ) AS replication_lag_size
FROM pg_replication_slots
WHERE slot_type = 'physical'
ORDER BY pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) DESC;

    One important operational difference: RDS PostgreSQL read replicas have replication slots on the primary. If a replica becomes unreachable and you don't drop its slot, the primary will retain WAL indefinitely, eventually filling your storage. This is a well-known gotcha in self-managed PostgreSQL, and it's equally present in RDS. Aurora does not have this problem — because replicas share storage with the primary, there are no WAL slots to manage. This alone removes a significant class of production incidents.

    Monitoring, Limits, and Production Gotchas

    The monitoring surfaces are different between the two services, and knowing which metrics actually matter in production prevents misdiagnosis.

    CloudWatch metrics that behave differently on Aurora vs RDS:

    | Metric | RDS PostgreSQL | Aurora PostgreSQL |

    |---|---|---|

    | FreeStorageSpace | Critical — approaches zero as disk fills | Not applicable — Aurora auto-extends |

    | WriteIOPS | Direct EBS write IOPS | Aurora storage layer I/O (different billing unit) |

    | ReplicaLag | Seconds of WAL lag | Milliseconds of replica lag |

    | AuroraVolumeBytesLeftTotal | Not present | Remaining capacity before 128 TB limit |

    | CommitLatency | P99 commit time | P99 commit time (higher baseline on Aurora) |

    Limits you will hit and need to plan for:

    *RDS PostgreSQL limits:*

  • → Maximum storage: 64 TB (io2) or 16 TB (gp3)
  • → Maximum IOPS: 256,000 (io2 Block Express on supported instance types)
  • → Maximum read replicas: 5 per instance
  • → Cross-region replicas: supported, with WAL streaming lag

    • *Aurora PostgreSQL limits:*

  • → Maximum cluster storage: 128 TB
  • → Maximum instances per cluster: 15 (1 writer + 14 readers)
  • → Maximum Aurora Global Database regions: 5 secondary
  • → Aurora Serverless v2 maximum: 128 ACUs per instance (approximately 256 GB RAM)
  • → Not all PostgreSQL extensions are available — check the [Aurora PostgreSQL extensions list](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/AuroraPostgreSQL.Extensions.html) before migrating

    • Production gotchas specific to Aurora PostgreSQL:

    1. Temp table storage uses local instance storage, not Aurora distributed storage. Large sort operations, hash joins, and queries that spill to disk will use instance-local temp space. If your workload involves large analytical queries, the instance class must have sufficient local NVMe or you must tune work_mem aggressively enough to keep operations in memory. This catches teams who sized their Aurora instance purely on storage, not RAM.

    2. Vacuum behavior on Aurora is different. Because Aurora uses a distributed storage model, the vacuum process interacts with storage nodes rather than local disk. In practice this means autovacuum runs normally, but manual VACUUM FULL is even more impactful on Aurora — it rewrites the entire table, which causes significant write amplification across all six storage copies. Avoid VACUUM FULL on Aurora. Use pg_repack if you need to reclaim space and reorganize tables.

    3. Aurora does not support all replication-related PostgreSQL parameters. Parameters like wal_level, archive_mode, archive_command, and restore_command are managed by Aurora and cannot be modified. If you're using logical replication to feed a data warehouse or a CDC pipeline (Debezium, AWS DMS), Aurora supports logical replication, but the setup is different from what you'd do with postgresql.conf directly.

    # Enable logical replication on Aurora via cluster parameter group
# (Cannot be done in postgresql.conf — Aurora manages the file)
aws rds modify-db-cluster-parameter-group \
  --db-cluster-parameter-group-name aurora16-prod-cluster \
  --parameters \
    ParameterName=rds.logical_replication,\
    ParameterValue=1,\
    ApplyMethod=pending-reboot

# Verify the change after reboot
aws rds describe-db-cluster-parameters \
  --db-cluster-parameter-group-name aurora16-prod-cluster \
  --query 'Parameters[?ParameterName==`rds.logical_replication`]'

# Then create a logical replication slot as you normally would
# (This is standard PostgreSQL from this point)

    4. Writer instance failure during a write storm. During the few seconds of Aurora failover, the cluster endpoint DNS updates but existing connections receive connection errors. Applications must implement retry logic with exponential backoff. AWS recommends catching OperationalError with error code 08006 (connection_failure) or 57P01 (admin_shutdown) and retrying.
