Chapter 39: Testing AI Features: Regression Testing with DB Fixtures

AI features break in ways that traditional software tests rarely anticipate. A RAG pipeline that returns accurate answers today may silently degrade when your embedding model changes, your vector index gets rebuilt with different parameters, or your prompt template is tweaked by a well-meaning developer. An LLM-powered chatbot that passes its unit tests might still return factually wrong answers in production because nobody tested the retrieval step against realistic data. The gap between "the code runs" and "the AI feature works correctly" is where most AI application bugs live.

This chapter builds a complete regression testing framework for AI features using PostgreSQL as the testing backbone. You will create database fixtures that represent realistic retrieval scenarios, implement golden-dataset testing for RAG pipelines, build deterministic evaluation harnesses for LLM responses, and automate the entire test suite so that schema migrations, model upgrades, and prompt changes are validated before they reach production. PostgreSQL's transactional semantics, JSONB flexibility, and pgvector integration make it uniquely suited to this — you get reproducible state, queryable test results, and the ability to diff evaluation runs the same way you diff code.

39.1 Why AI Features Need Different Testing Strategies

Traditional software testing assumes determinism: given the same input, the code produces the same output. AI features break this assumption in two ways. First, LLM responses are stochastic — the same prompt can produce different outputs across calls. Second, RAG pipeline quality depends on data state — the same question produces different answers when your document corpus changes. Standard unit tests cannot catch either class of failure.

The testing strategy that works for AI features has four layers:

Layer 1 — Fixture-based retrieval testing: Given a fixed corpus of documents loaded into a known database state, does the retrieval step return the expected documents in the expected rank order?

Layer 2 — Golden dataset evaluation: Against a fixed set of (question, expected answer) pairs, does the full RAG pipeline produce responses that score above a threshold on semantic similarity or factual accuracy?

Layer 3 — Regression comparison: Does the current pipeline produce responses that are meaningfully different from the last approved baseline? If so, flag it for human review before deployment.

Layer 4 — Contract testing for agents: When an AI agent calls a database tool, does the tool return data in the schema the agent expects, and does the agent's response reflect that data accurately?

PostgreSQL is the right place to anchor all four layers. Fixtures live in the database. Test results are stored as JSONB records. Comparisons use SQL window functions. The entire history of evaluation runs is queryable.

Here is the schema that underpins the entire framework:

-- Core testing schema for AI feature regression
CREATE SCHEMA IF NOT EXISTS ai_testing;

-- Fixture sets: named snapshots of document corpus state
CREATE TABLE ai_testing.fixture_sets (
    fixture_id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name                TEXT NOT NULL UNIQUE,
    description         TEXT,
    created_at          TIMESTAMPTZ DEFAULT now(),
    created_by          TEXT DEFAULT current_user,
    is_active           BOOLEAN DEFAULT true,
    metadata            JSONB DEFAULT '{}'
);

-- Individual fixture documents with pre-computed embeddings
CREATE TABLE ai_testing.fixture_documents (
    doc_id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    fixture_id          UUID NOT NULL REFERENCES ai_testing.fixture_sets(fixture_id) ON DELETE CASCADE,
    source_document_id  TEXT NOT NULL,
    content             TEXT NOT NULL,
    embedding           vector(1536),
    chunk_index         INTEGER DEFAULT 0,
    metadata            JSONB DEFAULT '{}',
    created_at          TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX ON ai_testing.fixture_documents USING ivfflat (embedding vector_cosine_ops)
    WITH (lists = 10);
CREATE INDEX ON ai_testing.fixture_documents (fixture_id);

-- Golden dataset: question/expected_answer pairs tied to a fixture set
CREATE TABLE ai_testing.golden_dataset (
    golden_id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    fixture_id          UUID NOT NULL REFERENCES ai_testing.fixture_sets(fixture_id),
    question            TEXT NOT NULL,
    expected_answer     TEXT NOT NULL,
    expected_doc_ids    TEXT[] DEFAULT '{}',   -- documents that MUST appear in retrieval
    min_similarity      FLOAT DEFAULT 0.85,    -- minimum cosine sim for acceptable answer
    tags                TEXT[] DEFAULT '{}',
    created_at          TIMESTAMPTZ DEFAULT now(),
    is_active           BOOLEAN DEFAULT true
);

-- Evaluation runs: one record per test suite execution
CREATE TABLE ai_testing.evaluation_runs (
    run_id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    fixture_id          UUID NOT NULL REFERENCES ai_testing.fixture_sets(fixture_id),
    run_at              TIMESTAMPTZ DEFAULT now(),
    triggered_by        TEXT,                  -- 'ci', 'manual', 'pre-deploy'
    model_name          TEXT NOT NULL,
    embedding_model     TEXT NOT NULL,
    pipeline_version    TEXT,
    git_commit          TEXT,
    overall_score       FLOAT,                 -- aggregate score for this run
    passed              BOOLEAN,
    summary             JSONB DEFAULT '{}'
);

-- Per-question evaluation results
CREATE TABLE ai_testing.evaluation_results (
    result_id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    run_id              UUID NOT NULL REFERENCES ai_testing.evaluation_runs(run_id) ON DELETE CASCADE,
    golden_id           UUID NOT NULL REFERENCES ai_testing.golden_dataset(golden_id),
    actual_answer       TEXT,
    retrieved_doc_ids   TEXT[],
    retrieval_hit_rate  FLOAT,                 -- % of expected_doc_ids that appeared
    semantic_similarity FLOAT,                 -- cosine sim between expected and actual answer embeddings
    answer_embedding    vector(1536),
    latency_ms          INTEGER,
    passed              BOOLEAN,
    failure_reason      TEXT,
    raw_context         TEXT,
    metadata            JSONB DEFAULT '{}'
);

CREATE INDEX ON ai_testing.evaluation_results (run_id);
CREATE INDEX ON ai_testing.evaluation_results (golden_id);

-- Regression baseline: the approved "good" run to compare against
CREATE TABLE ai_testing.regression_baselines (
    baseline_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    fixture_id          UUID NOT NULL REFERENCES ai_testing.fixture_sets(fixture_id),
    run_id              UUID NOT NULL REFERENCES ai_testing.evaluation_runs(run_id),
    approved_by         TEXT,
    approved_at         TIMESTAMPTZ DEFAULT now(),
    notes               TEXT,
    is_current          BOOLEAN DEFAULT true
);

-- Ensure only one current baseline per fixture
CREATE UNIQUE INDEX ON ai_testing.regression_baselines (fixture_id)
    WHERE is_current = true;

39.2 Building Database Fixtures for AI Pipelines

A fixture is a deterministic, versioned snapshot of your document corpus with pre-computed embeddings. Loading a fixture before each test run ensures that retrieval behavior is attributable to pipeline changes, not data changes.

The fixture loader pulls documents from your production corpus, chunks them, embeds them using the same model your pipeline uses, and stores them in the ai_testing schema. Critically, it wraps the entire operation in a transaction so the fixture is either fully loaded or not loaded at all.

# fixture_loader.py — builds and loads AI test fixtures into PostgreSQL
import asyncio
import hashlib
import json
import uuid
from dataclasses import dataclass, field
from typing import Optional

import asyncpg
import numpy as np
from openai import AsyncOpenAI

EMBEDDING_MODEL = "text-embedding-3-small"
EMBEDDING_DIM = 1536

@dataclass
class FixtureDocument:
    source_document_id: str
    content: str
    chunk_index: int = 0
    metadata: dict = field(default_factory=dict)

@dataclass
class GoldenExample:
    question: str
    expected_answer: str
    expected_doc_ids: list[str] = field(default_factory=list)
    min_similarity: float = 0.85
    tags: list[str] = field(default_factory=list)

class FixtureLoader:
    def __init__(self, dsn: str, openai_api_key: str):
        self.dsn = dsn
        self.client = AsyncOpenAI(api_key=openai_api_key)

    async def embed_batch(self, texts: list[str]) -> list[list[float]]:
        """Embed a batch of texts, respecting token limits."""
        response = await self.client.embeddings.create(
            model=EMBEDDING_MODEL,
            input=texts
        )
        return [item.embedding for item in response.data]

    async def create_fixture_set(
        self,
        conn: asyncpg.Connection,
        name: str,
        description: str,
        documents: list[FixtureDocument],
        golden_examples: list[GoldenExample],
        metadata: dict = None
    ) -> str:
        """Load a complete fixture set within a single transaction."""
        fixture_id = str(uuid.uuid4())

        async with conn.transaction():
            # 1. Create the fixture set record
            await conn.execute("""
                INSERT INTO ai_testing.fixture_sets
                    (fixture_id, name, description, metadata)
                VALUES ($1, $2, $3, $4)
                ON CONFLICT (name) DO UPDATE
                    SET description = EXCLUDED.description,
                        metadata = EXCLUDED.metadata,
                        created_at = now()
                RETURNING fixture_id
            """, fixture_id, name, description, json.dumps(metadata or {}))

            # 2. Delete existing documents for this fixture (idempotent reload)
            await conn.execute("""
                DELETE FROM ai_testing.fixture_documents
                WHERE fixture_id = (
                    SELECT fixture_id FROM ai_testing.fixture_sets WHERE name = $1
                )
            """, name)

            # Re-fetch the actual fixture_id in case of ON CONFLICT update
            row = await conn.fetchrow(
                "SELECT fixture_id FROM ai_testing.fixture_sets WHERE name = $1", name
            )
            fixture_id = str(row["fixture_id"])

            # 3. Embed all documents in batches of 50
            print(f"Embedding {len(documents)} documents...")
            batch_size = 50
            all_embeddings = []
            for i in range(0, len(documents), batch_size):
                batch = documents[i:i + batch_size]
                texts = [doc.content for doc in batch]
                embeddings = await self.embed_batch(texts)
                all_embeddings.extend(embeddings)

            # 4. Insert fixture documents with embeddings
            doc_records = [
                (
                    str(uuid.uuid4()),
                    fixture_id,
                    doc.source_document_id,
                    doc.content,
                    json.dumps(doc.metadata),
                    doc.chunk_index,
                    all_embeddings[i]
                )
                for i, doc in enumerate(documents)
            ]

            await conn.executemany("""
                INSERT INTO ai_testing.fixture_documents
                    (doc_id, fixture_id, source_document_id, content,
                     metadata, chunk_index, embedding)
                VALUES ($1, $2, $3, $4, $5, $6, $7)
            """, [
                (r[0], r[1], r[2], r[3], r[4], r[5],
                 f"[{','.join(str(x) for x in r[6])}]")
                for r in doc_records
            ])

            # 5. Delete and reload golden examples
            await conn.execute("""
                DELETE FROM ai_testing.golden_dataset WHERE fixture_id = $1
            """, fixture_id)

            for ex in golden_examples:
                await conn.execute("""
                    INSERT INTO ai_testing.golden_dataset
                        (fixture_id, question, expected_answer,
                         expected_doc_ids, min_similarity, tags)
                    VALUES ($1, $2, $3, $4, $5, $6)
                """,
                    fixture_id,
                    ex.question,
                    ex.expected_answer,
                    ex.expected_doc_ids,
                    ex.min_similarity,
                    ex.tags
                )

        print(f"Fixture '{name}' loaded: {len(documents)} docs, "
              f"{len(golden_examples)} golden examples")
        return fixture_id


# Define a realistic fixture for a product documentation RAG system
PRODUCT_DOCS_FIXTURE = [
    FixtureDocument(
        source_document_id="doc-001",
        content="""PostgreSQL connection pooling with PgBouncer reduces connection overhead
        for high-traffic applications. PgBouncer supports three pooling modes: session,
        transaction, and statement. Transaction mode is recommended for most web applications
        because it allows the connection pool to be shared across multiple concurrent requests.
        Configure max_client_conn to 1000 and default_pool_size to 25 for a typical workload.""",
        chunk_index=0,
        metadata={"category": "infrastructure", "product": "pgbouncer"}
    ),
    FixtureDocument(
        source_document_id="doc-002",
        content="""Row-level security in PostgreSQL enables fine-grained access control at
        the row level without modifying application queries. Enable RLS with ALTER TABLE
        orders ENABLE ROW LEVEL SECURITY. Create policies using CREATE POLICY that define
        USING clauses for SELECT and WITH CHECK clauses for INSERT/UPDATE. Always test
        policies with SET ROLE to verify they work from the application user's perspective.""",
        chunk_index=0,
        metadata={"category": "security", "product": "postgresql"}
    ),
    FixtureDocument(
        source_document_id="doc-003",
        content="""pgvector enables semantic search in PostgreSQL by storing embedding vectors
        alongside relational data. Create embedding columns with the vector type: embedding
        vector(1536). Build HNSW indexes for high-recall similarity search:
        CREATE INDEX ON documents USING hnsw (embedding vector_cosine_ops). HNSW indexes
        support concurrent inserts and provide better recall than IVFFlat at the cost of
        higher memory usage during index construction.""",
        chunk_index=0,
        metadata={"category": "vector-search", "product": "pgvector"}
    ),
]

GOLDEN_EXAMPLES = [
    GoldenExample(
        question="How should I configure PgBouncer for a web application?",
        expected_answer=("Use transaction mode pooling with max_client_conn=1000 and "

        expected_doc_ids=["doc-001"],
        min_similarity=0.80,
        tags=["infrastructure", "pooling"]
    ),
    GoldenExample(
        question="How do I enable row-level security on a table?",
        expected_answer=("Run ALTER TABLE orders ENABLE ROW LEVEL SECURITY and create "
                        "policies with USING and WITH CHECK clauses."),
        expected_doc_ids=["doc-002"],
        min_similarity=0.82,
        tags=["security", "rls"]
    ),
    GoldenExample(
        question="What index type should I use for high-recall vector search?",
        expected_answer=("Use HNSW indexes with vector_cosine_ops for high-recall "
                        "semantic search. HNSW provides better recall than IVFFlat."),
        expected_doc_ids=["doc-003"],
        min_similarity=0.80,
        tags=["vector-search", "indexing"]
    ),
]


async def main():
    import os
    conn = await asyncpg.connect(os.environ["DATABASE_URL"])
    loader = FixtureLoader(
        dsn=os.environ["DATABASE_URL"],
        openai_api_key=os.environ["OPENAI_API_KEY"]
    )
    fixture_id = await loader.create_fixture_set(
        conn=conn,
        name="product-docs-v1",
        description="Product documentation RAG fixture for regression testing",
        documents=PRODUCT_DOCS_FIXTURE,
        golden_examples=GOLDEN_EXAMPLES,
        metadata={"version": "1.0", "domain": "product-docs"}
    )
    print(f"Created fixture: {fixture_id}")
    await conn.close()

if __name__ == "__main__":
    asyncio.run(main())

Fixture loading is intentionally idempotent — running it twice produces the same state. This lets your CI pipeline call python fixture_loader.py before every test run without accumulating stale data.

39.3 Evaluating RAG Pipeline Quality Against Fixtures

With fixtures loaded, the evaluation harness runs the actual RAG pipeline against each golden example and scores the results. The scoring uses two metrics: retrieval hit rate (did the right documents appear in the top-k results?) and semantic similarity (does the generated answer mean the same thing as the expected answer?).

The pipeline under test runs exactly as it would in production — same embedding model, same vector similarity query, same LLM prompt. The only difference is that it queries ai_testing.fixture_documents instead of the production documents table.
