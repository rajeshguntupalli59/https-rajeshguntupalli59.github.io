Chapter 16: Metadata Filtering with SQL and Vector Search Together

Most vector search tutorials show you a single embedding column, a distance operator, and a LIMIT 10. That works for demos. In production AI applications, you almost never want the top-10 most semantically similar documents from your entire corpus — you want the top-10 most semantically similar documents *that belong to this tenant, in this language, published after this date, with this approval status, and not already shown to this user*. The moment you add those constraints, you are doing metadata filtering combined with vector search, and how you implement that combination determines whether your RAG system is fast, accurate, and correct — or slow, leaky, and wrong.

This chapter covers the full stack: schema design that makes filtering efficient, the different architectural patterns for combining SQL predicates with ANN search, how pgvector's HNSW index interacts with WHERE clauses, query patterns for common RAG filtering scenarios, and the Python layer that ties it all together into a production-ready retrieval pipeline.

---

Why Metadata Filtering Is Not an Afterthought

Vector similarity search answers a deceptively simple question: which embeddings in this set are closest to my query embedding? The problem is that "this set" — the entire embedding table — is almost never the right scope for a production query.

Consider a multi-tenant knowledge base serving 500 companies. Each company has uploaded their own documents. Without metadata filtering, a similarity search for "how do I reset my password?" might return results from a competitor's documentation. This is not a minor inconvenience — it is a data isolation failure. Row-level security and tenant-scoped queries are standard database thinking, and they apply equally to vector search.

Beyond multi-tenancy, filtering requirements in real RAG systems include:

  • → **Temporal filters** — only retrieve documents updated within the last 90 days, because older content may be stale
  • → **Status filters** — only retrieve `published` or `approved` documents, not drafts
  • → **Language filters** — retrieve documents matching the detected user language
  • → **Category/taxonomy filters** — pre-narrow the corpus by topic before semantic ranking
  • → **Permission filters** — only show content the authenticated user has access to
  • → **Deduplication** — exclude document IDs already retrieved in previous turns

    • Each of these is a standard SQL predicate. PostgreSQL is extraordinarily good at evaluating them. The engineering challenge is combining them efficiently with the approximate nearest neighbor (ANN) search that pgvector provides.

    The fundamental tension is this: HNSW and IVFFlat indexes are built to traverse the vector space efficiently, but they work best when they can consider a large candidate set. If your SQL WHERE clause eliminates 99% of rows before the index traversal, the ANN index may not help much — or may even be bypassed in favor of a sequential scan. Understanding this tradeoff lets you design schemas and queries that give you both correctness and speed.

    Schema Design for Filterable Vector Search

    The schema is where the filtering strategy lives. A common mistake is storing metadata as a single JSONB blob and expecting ad-hoc filtering to be fast. Some metadata should be JSONB; most filtering attributes should be typed columns with indexes.

    Here is a realistic schema for a multi-tenant document knowledge base:

    -- Extensions
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS pg_trgm;  -- for text search on titles

-- Tenants
CREATE TABLE tenants (
    tenant_id     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name          TEXT NOT NULL,
    plan          TEXT NOT NULL DEFAULT 'free',
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Document sources (S3 buckets, web crawls, uploads, etc.)
CREATE TABLE document_sources (
    source_id     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id     UUID NOT NULL REFERENCES tenants(tenant_id),
    name          TEXT NOT NULL,
    source_type   TEXT NOT NULL,  -- 'upload', 'web_crawl', 'api_sync'
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Documents (parent records — a PDF, a web page, a Confluence article)
CREATE TABLE documents (
    document_id   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id     UUID NOT NULL REFERENCES tenants(tenant_id),
    source_id     UUID REFERENCES document_sources(source_id),
    title         TEXT NOT NULL,
    url           TEXT,
    language      TEXT NOT NULL DEFAULT 'en',
    status        TEXT NOT NULL DEFAULT 'draft',
        -- CHECK constraint ensures valid statuses
        CONSTRAINT documents_status_check
            CHECK (status IN ('draft', 'processing', 'published', 'archived')),
    category      TEXT,
    tags          TEXT[] NOT NULL DEFAULT '{}',
    author_id     UUID,
    word_count    INTEGER,
    -- Timestamps matter for recency filtering
    content_updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    -- Arbitrary metadata that doesn't need indexing
    extra_metadata      JSONB NOT NULL DEFAULT '{}'
);

-- Indexes for common filter patterns
CREATE INDEX idx_documents_tenant_status
    ON documents (tenant_id, status);

CREATE INDEX idx_documents_tenant_language
    ON documents (tenant_id, language);

CREATE INDEX idx_documents_tenant_category
    ON documents (tenant_id, category)
    WHERE category IS NOT NULL;

CREATE INDEX idx_documents_content_updated_at
    ON documents (tenant_id, content_updated_at DESC)
    WHERE status = 'published';

CREATE INDEX idx_documents_tags
    ON documents USING GIN (tags);

-- Chunks (the actual units of retrieval — documents split into overlapping chunks)
CREATE TABLE document_chunks (
    chunk_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id     UUID NOT NULL REFERENCES documents(document_id) ON DELETE CASCADE,
    tenant_id       UUID NOT NULL,  -- denormalized for faster filtering
    chunk_index     INTEGER NOT NULL,
    chunk_text      TEXT NOT NULL,
    token_count     INTEGER,
    -- Denormalized filter columns from parent document
    -- This is intentional: avoids JOINs during hot-path retrieval
    language        TEXT NOT NULL DEFAULT 'en',
    status          TEXT NOT NULL DEFAULT 'draft',
    category        TEXT,
    tags            TEXT[] NOT NULL DEFAULT '{}',
    content_updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    -- The embedding
    embedding       vector(1536),  -- OpenAI text-embedding-3-small dimensions
    -- Bookkeeping
    embedding_model TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT fk_chunk_tenant
        FOREIGN KEY (tenant_id) REFERENCES tenants(tenant_id)
);

-- Partial index: only index published chunks
-- This is the most important optimization for metadata-filtered vector search
CREATE INDEX idx_chunks_embedding_published
    ON document_chunks
    USING hnsw (embedding vector_cosine_ops)
    WHERE status = 'published';

-- Composite b-tree indexes for metadata filtering
CREATE INDEX idx_chunks_tenant_status
    ON document_chunks (tenant_id, status);

CREATE INDEX idx_chunks_tenant_language_status
    ON document_chunks (tenant_id, language, status);

CREATE INDEX idx_chunks_tenant_category
    ON document_chunks (tenant_id, category, status)
    WHERE category IS NOT NULL;

CREATE INDEX idx_chunks_tags
    ON document_chunks USING GIN (tags)
    WHERE status = 'published';

-- User retrieval history (for deduplication across conversation turns)
CREATE TABLE retrieval_history (
    id              BIGSERIAL PRIMARY KEY,
    session_id      UUID NOT NULL,
    user_id         UUID NOT NULL,
    tenant_id       UUID NOT NULL,
    chunk_id        UUID NOT NULL REFERENCES document_chunks(chunk_id),
    retrieved_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_retrieval_history_session
    ON retrieval_history (session_id, retrieved_at DESC);

    The key design decisions here:

    Denormalized filter columns on chunks — tenant_id, language, status, category, and tags are duplicated from the parent document onto every chunk. This avoids a JOIN on every retrieval query. The JOIN cost matters when you are running hundreds of queries per second.

    Partial HNSW index on status = 'published' — The HNSW index only covers rows where status = 'published'. This means the index is smaller (faster to build and traverse), and queries that filter on status = 'published' can use the index efficiently. Draft documents never pollute semantic search results.

    Typed columns for high-cardinality filters — tenant_id, language, category, and status are typed columns, not JSONB keys. These can be covered by B-tree composite indexes that PostgreSQL's planner knows how to use.

    The Three Patterns for Combining Filters and Vector Search

    There are three fundamental architectural patterns for combining SQL predicates with vector similarity search. PostgreSQL supports all three, and the right choice depends on your data distribution and query selectivity.

    Pattern 1: Pre-filter then search (SQL first)

    Apply the SQL WHERE clause to narrow the candidate set, then do vector search on the result.

    -- Pre-filter pattern: PostgreSQL evaluates the WHERE clause,
-- then applies vector distance ordering within the filtered set
SELECT
    dc.chunk_id,
    dc.document_id,
    dc.chunk_text,
    dc.category,
    dc.content_updated_at,
    1 - (dc.embedding <=> $1::vector) AS cosine_similarity
FROM document_chunks dc
WHERE
    dc.tenant_id    = $2
    AND dc.status   = 'published'
    AND dc.language = $3
    AND dc.content_updated_at >= NOW() - INTERVAL '90 days'
ORDER BY dc.embedding <=> $1::vector
LIMIT 10;

    This pattern works well when filters are highly selective — if tenant_id + status + language reduces the candidate set to a few thousand rows, PostgreSQL will scan those rows and compute distances sequentially. With a small filtered set, sequential distance computation is fast enough that the HNSW index may not be needed.

    Pattern 2: Search then post-filter (vector first)

    Retrieve a large pool of nearest neighbors from the full vector space, then apply SQL filters to the pool.

    -- Post-filter pattern: retrieve k*multiplier candidates, then filter
-- Use a CTE to make the intent explicit
WITH vector_candidates AS (
    SELECT
        chunk_id,
        document_id,
        chunk_text,
        tenant_id,
        status,
        language,
        category,
        content_updated_at,
        1 - (embedding <=> $1::vector) AS cosine_similarity
    FROM document_chunks
    ORDER BY embedding <=> $1::vector
    LIMIT 200  -- retrieve 20x more than we need, then filter
),
filtered AS (
    SELECT *
    FROM vector_candidates
    WHERE
        tenant_id    = $2
        AND status   = 'published'
        AND language = $3
        AND content_updated_at >= NOW() - INTERVAL '90 days'
)
SELECT
    chunk_id,
    document_id,
    chunk_text,
    category,
    content_updated_at,
    cosine_similarity
FROM filtered
ORDER BY cosine_similarity DESC
LIMIT 10;

    This pattern works well when filters have low selectivity (most documents match) or when you cannot afford a sequential scan of the filtered set (the filtered set is still very large). The risk is that your filters might eliminate most of the 200 candidates, leaving you with fewer than 10 results. You need to tune the over-fetch multiplier.

    Pattern 3: Partitioned HNSW indexes (partition-per-tenant)

    For large multi-tenant deployments, partition the table by tenant_id and build separate HNSW indexes per partition. Each tenant's vector search is isolated to their partition.

    -- Create a partitioned chunks table
CREATE TABLE document_chunks_partitioned (
    chunk_id        UUID NOT NULL DEFAULT gen_random_uuid(),
    document_id     UUID NOT NULL,
    tenant_id       UUID NOT NULL,
    chunk_index     INTEGER NOT NULL,
    chunk_text      TEXT NOT NULL,
    language        TEXT NOT NULL DEFAULT 'en',
    status          TEXT NOT NULL DEFAULT 'draft',
    category        TEXT,
    tags            TEXT[] NOT NULL DEFAULT '{}',
    content_updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    embedding       vector(1536),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (tenant_id, chunk_id)
) PARTITION BY HASH (tenant_id);

-- Create 16 partitions
DO $$
BEGIN
    FOR i IN 0..15 LOOP
        EXECUTE format(
            'CREATE TABLE document_chunks_p%s
             PARTITION OF document_chunks_partitioned
             FOR VALUES WITH (modulus 16, remainder %s)',
            i, i
        );
    END LOOP;
END;
$$;

-- Create HNSW indexes on each partition
DO $$
BEGIN
    FOR i IN 0..15 LOOP
        EXECUTE format(
            'CREATE INDEX idx_chunks_p%s_embedding
             ON document_chunks_p%s
             USING hnsw (embedding vector_cosine_ops)
             WITH (m = 16, ef_construction = 64)
             WHERE status = ''published''',
            i, i
        );
    END LOOP;
END;
$$;

    When you query with WHERE tenant_id = $1, PostgreSQL's partition pruning eliminates all partitions not matching the hash of that tenant ID. The HNSW index on the remaining partition is smaller and faster to traverse.
