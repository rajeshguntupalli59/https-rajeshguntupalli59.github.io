Chapter 38: GDPR and AI: Right to Be Forgotten with Embeddings

When a user submits a GDPR deletion request — "erase everything about me" — your PostgreSQL DELETE statements are the easy part. The hard part is the embedding. That user's words, preferences, and personal details may have been vectorized and stored in pgvector tables, woven into RAG indices, cached in Redis, queued in Celery jobs, and potentially absorbed into model fine-tuning datasets. A row deletion satisfies the letter of the law; actually removing personal information from an AI system's memory requires a coordinated strategy across the entire stack.

This chapter covers how to design PostgreSQL-backed AI systems that can honor GDPR's Right to Be Forgotten without destroying your AI features. You will build deletion pipelines that scrub user data from embedding tables, invalidate caches, regenerate affected indices, audit every step, and prove compliance — all while keeping the rest of the system operational.

---

38.1 What "Forgotten" Actually Means in an AI System

GDPR Article 17 requires erasure of personal data. In a traditional application this means deleting database rows. In an AI-native application, personal data can exist in at least six places simultaneously:

Structured rows — user profiles, conversation history, preferences, metadata.

Embeddings — vector representations of user-authored content. The embedding itself is a mathematical transformation of text that contained personal information. Courts and regulators have not fully settled whether embeddings constitute personal data, but the conservative and defensible position is that embeddings derived from personal content must also be deleted.

RAG document chunks — if a user uploaded documents or their conversation turns were indexed for retrieval, those chunks live in your document store alongside their vectors.

Caches — Redis may hold pre-computed embeddings or LLM responses that referenced the user.

Async job queues — Celery jobs may still be processing documents that contain user data.

Audit and log tables — these often need to be retained for compliance, but must be scrubbed of PII while preserving the audit trail.

The architecture you need is one where every piece of user-derived AI content is traceable back to a user_id. PostgreSQL's relational model is your ally here. A pure vector database with no relational structure makes this nearly impossible. With PostgreSQL and pgvector, every embedding row can carry a user_id foreign key, and a single deletion event can cascade correctly through the schema.

Start with the foundational schema that makes GDPR compliance tractable:

-- Core user registry with deletion tracking
CREATE TABLE users (
    user_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email           TEXT NOT NULL UNIQUE,
    display_name    TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deletion_requested_at TIMESTAMPTZ,
    deletion_completed_at TIMESTAMPTZ,
    deletion_job_id UUID,
    is_active       BOOLEAN NOT NULL DEFAULT TRUE
);

-- Every piece of user-authored content is tracked here
-- This is the master registry for AI-derived artifacts
CREATE TABLE user_content (
    content_id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(user_id) ON DELETE RESTRICT,
    content_type    TEXT NOT NULL, -- 'message', 'document', 'profile_bio', 'preference'
    source_text     TEXT,          -- original text, nulled out after deletion
    content_hash    TEXT,          -- SHA-256 for deduplication, kept after deletion
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at      TIMESTAMPTZ,
    metadata        JSONB NOT NULL DEFAULT '{}'
);

CREATE INDEX idx_user_content_user_id ON user_content(user_id);
CREATE INDEX idx_user_content_type    ON user_content(content_type);
CREATE INDEX idx_user_content_deleted ON user_content(deleted_at) WHERE deleted_at IS NULL;

-- Embeddings table — every vector is owned by a user
CREATE TABLE content_embeddings (
    embedding_id    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    content_id      UUID NOT NULL REFERENCES user_content(content_id) ON DELETE CASCADE,
    user_id         UUID NOT NULL REFERENCES users(user_id) ON DELETE RESTRICT,
    model_name      TEXT NOT NULL,          -- 'text-embedding-3-small', 'embed-english-v3.0'
    embedding_dim   INTEGER NOT NULL,
    embedding        vector(1536),          -- pgvector column
    chunk_index     INTEGER NOT NULL DEFAULT 0,
    chunk_text      TEXT,                   -- the actual chunk text, nulled on deletion
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at      TIMESTAMPTZ,
    metadata        JSONB NOT NULL DEFAULT '{}'
);

CREATE INDEX idx_embeddings_user_id    ON content_embeddings(user_id);
CREATE INDEX idx_embeddings_content_id ON content_embeddings(content_id);
CREATE INDEX idx_embeddings_deleted    ON content_embeddings(deleted_at) WHERE deleted_at IS NULL;
CREATE INDEX idx_embeddings_model      ON content_embeddings(model_name);

-- HNSW index only on active embeddings
CREATE INDEX idx_embeddings_vector ON content_embeddings 
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64)
    WHERE deleted_at IS NULL;

-- Conversation history — must be fully scrubbed
CREATE TABLE conversations (
    conversation_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(user_id) ON DELETE RESTRICT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at      TIMESTAMPTZ,
    title           TEXT,
    metadata        JSONB NOT NULL DEFAULT '{}'
);

CREATE TABLE conversation_turns (
    turn_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    conversation_id UUID NOT NULL REFERENCES conversations(conversation_id) ON DELETE CASCADE,
    user_id         UUID NOT NULL REFERENCES users(user_id) ON DELETE RESTRICT,
    role            TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system')),
    content         TEXT,          -- nulled after deletion
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at      TIMESTAMPTZ,
    token_count     INTEGER,
    metadata        JSONB NOT NULL DEFAULT '{}'
);

CREATE INDEX idx_turns_user_id ON conversation_turns(user_id);
CREATE INDEX idx_turns_conv_id ON conversation_turns(conversation_id);

-- Deletion audit log — this is kept forever for compliance
-- but contains NO personal data
CREATE TABLE deletion_audit_log (
    audit_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    deletion_job_id UUID NOT NULL,
    user_id_hash    TEXT NOT NULL,  -- SHA-256 of user_id, not the user_id itself
    event_type      TEXT NOT NULL,  -- 'deletion_requested', 'step_completed', 'deletion_verified'
    step_name       TEXT,
    records_affected INTEGER,
    executed_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    executor        TEXT NOT NULL,  -- 'system', 'celery_worker', 'manual_dba'
    notes           TEXT,
    checksum        TEXT            -- SHA-256 of row content for tamper detection
);

-- Row-level security so users can only see their own data
ALTER TABLE user_content        ENABLE ROW LEVEL SECURITY;
ALTER TABLE content_embeddings  ENABLE ROW LEVEL SECURITY;
ALTER TABLE conversations       ENABLE ROW LEVEL SECURITY;
ALTER TABLE conversation_turns  ENABLE ROW LEVEL SECURITY;

CREATE POLICY user_content_isolation ON user_content
    USING (user_id = current_setting('app.current_user_id', TRUE)::UUID);

CREATE POLICY embeddings_isolation ON content_embeddings
    USING (user_id = current_setting('app.current_user_id', TRUE)::UUID);

CREATE POLICY conversations_isolation ON conversations
    USING (user_id = current_setting('app.current_user_id', TRUE)::UUID);

CREATE POLICY turns_isolation ON conversation_turns
    USING (user_id = current_setting('app.current_user_id', TRUE)::UUID);

38.2 The Deletion Job Pipeline

A GDPR deletion is not an instantaneous operation in an AI system. It is a workflow. The user submits a request; the system has 30 days to comply under GDPR (though best practice is faster). You need a tracked, idempotent, audited pipeline that handles partial failures gracefully.

The pipeline stages are: request intake → scope identification → content deletion → embedding deletion → cache invalidation → queue drain → verification → notification.

PostgreSQL drives the coordination via a deletion jobs table and pg_notify for real-time handoff to async workers:

-- Deletion job coordination table
CREATE TABLE deletion_jobs (
    job_id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(user_id),
    requested_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deadline_at     TIMESTAMPTZ NOT NULL DEFAULT (NOW() + INTERVAL '30 days'),
    status          TEXT NOT NULL DEFAULT 'pending'
                    CHECK (status IN ('pending', 'running', 'verifying', 'completed', 'failed')),
    current_step    TEXT,
    steps_completed JSONB NOT NULL DEFAULT '[]',
    error_details   TEXT,
    started_at      TIMESTAMPTZ,
    completed_at    TIMESTAMPTZ,
    verified_at     TIMESTAMPTZ,
    notified_at     TIMESTAMPTZ,
    retry_count     INTEGER NOT NULL DEFAULT 0,
    metadata        JSONB NOT NULL DEFAULT '{}'
);

CREATE INDEX idx_deletion_jobs_user    ON deletion_jobs(user_id);
CREATE INDEX idx_deletion_jobs_status  ON deletion_jobs(status) WHERE status != 'completed';
CREATE INDEX idx_deletion_jobs_deadline ON deletion_jobs(deadline_at) WHERE status != 'completed';

-- Function to initiate a deletion and fire notification
CREATE OR REPLACE FUNCTION initiate_deletion_request(p_user_id UUID)
RETURNS UUID
LANGUAGE plpgsql
AS $$
DECLARE
    v_job_id UUID;
    v_user_hash TEXT;
BEGIN
    -- Mark user as pending deletion (stops new data creation)
    UPDATE users
    SET deletion_requested_at = NOW(),
        is_active = FALSE
    WHERE user_id = p_user_id
      AND deletion_requested_at IS NULL;

    IF NOT FOUND THEN
        RAISE EXCEPTION 'User % not found or deletion already in progress', p_user_id;
    END IF;

    -- Create deletion job
    INSERT INTO deletion_jobs (user_id)
    VALUES (p_user_id)
    RETURNING job_id INTO v_job_id;

    -- Link job to user
    UPDATE users SET deletion_job_id = v_job_id WHERE user_id = p_user_id;

    -- Write audit entry with hashed user_id (no PII in audit log)
    v_user_hash := encode(digest(p_user_id::TEXT, 'sha256'), 'hex');

    INSERT INTO deletion_audit_log (
        deletion_job_id, user_id_hash, event_type, step_name,
        executor, notes
    ) VALUES (
        v_job_id, v_user_hash, 'deletion_requested', 'intake',
        'system', 'Deletion job created and user marked inactive'
    );

    -- Notify async workers via pg_notify
    PERFORM pg_notify(
        'deletion_requests',
        json_build_object(
            'job_id', v_job_id,
            'user_id', p_user_id,
            'requested_at', NOW()
        )::TEXT
    );

    RETURN v_job_id;
END;
$$;

-- Function to get the full scope of what must be deleted
CREATE OR REPLACE FUNCTION get_deletion_scope(p_user_id UUID)
RETURNS TABLE (
    artifact_type   TEXT,
    count           BIGINT,
    oldest_record   TIMESTAMPTZ,
    newest_record   TIMESTAMPTZ
)
LANGUAGE sql
AS $$
    SELECT 'user_content'::TEXT, COUNT(*), MIN(created_at), MAX(created_at)
    FROM user_content WHERE user_id = p_user_id AND deleted_at IS NULL
    UNION ALL
    SELECT 'content_embeddings', COUNT(*), MIN(created_at), MAX(created_at)
    FROM content_embeddings WHERE user_id = p_user_id AND deleted_at IS NULL
    UNION ALL
    SELECT 'conversation_turns', COUNT(*), MIN(created_at), MAX(created_at)
    FROM conversation_turns WHERE user_id = p_user_id AND deleted_at IS NULL
    UNION ALL
    SELECT 'conversations', COUNT(*), MIN(created_at), MAX(created_at)
    FROM conversations WHERE user_id = p_user_id AND deleted_at IS NULL;
$$;

-- Atomic step executor with audit logging
CREATE OR REPLACE FUNCTION execute_deletion_step(
    p_job_id    UUID,
    p_step_name TEXT,
    p_user_id   UUID
)
RETURNS INTEGER
LANGUAGE plpgsql
AS $$
DECLARE
    v_affected   INTEGER := 0;
    v_user_hash  TEXT;
BEGIN
    v_user_hash := encode(digest(p_user_id::TEXT, 'sha256'), 'hex');

    UPDATE deletion_jobs
    SET current_step = p_step_name,
        status = 'running',
        started_at = COALESCE(started_at, NOW())
    WHERE job_id = p_job_id;

    IF p_step_name = 'scrub_conversation_turns' THEN
        UPDATE conversation_turns
        SET content = NULL,
            deleted_at = NOW(),
            metadata = jsonb_set(metadata, '{gdpr_deleted}', 'true')
        WHERE user_id = p_user_id AND deleted_at IS NULL;
        GET DIAGNOSTICS v_affected = ROW_COUNT;

    ELSIF p_step_name = 'scrub_content_source_text' THEN
        UPDATE user_content
        SET source_text = NULL,
            deleted_at = NOW(),
            metadata = jsonb_set(metadata, '{gdpr_deleted}', 'true')
        WHERE user_id = p_user_id AND deleted_at IS NULL;
        GET DIAGNOSTICS v_affected = ROW_COUNT;

    ELSIF p_step_name = 'delete_embeddings' THEN
        -- Hard delete embeddings — nulling a vector field is not sufficient
        -- because the vector IS the personal data
        DELETE FROM content_embeddings
        WHERE user_id = p_user_id;
        GET DIAGNOSTICS v_affected = ROW_COUNT;

    ELSIF p_step_name = 'scrub_user_profile' THEN
        UPDATE users
        SET email = 'deleted-' || encode(digest(email, 'sha256'), 'hex') || '@deleted.invalid',
            display_name = NULL,
            deletion_completed_at = NOW()
        WHERE user_id = p_user_id;
        GET DIAGNOSTICS v_affected = ROW_COUNT;
    END IF;

    -- Record completed step on the job
    UPDATE deletion_jobs
    SET steps_completed = steps_completed || jsonb_build_object(
        'step', p_step_name,
        'completed_at', NOW(),
        'records_affected', v_affected
    )
    WHERE job_id = p_job_id;

    -- Audit log entry
    INSERT INTO deletion_audit_log (
        deletion_job_id, user_id_hash, event_type,
        step_name, records_affected, executor
    ) VALUES (
        p_job_id, v_user_hash, 'step_completed',
        p_step_name, v_affected, 'celery_worker'
    );

    RETURN v_affected;
END;
$$;

The Celery worker listens to pg_notify events, orchestrates each deletion step, handles cache invalidation in Redis, and re-indexes any shared knowledge base content that happened to co-mingle with user data.
