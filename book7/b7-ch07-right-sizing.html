Chapter 7: Right-Sizing Instances: The Methodology

Organizations running cloud databases consistently overpay by 30–40% on compute alone. The root cause is almost always the same: instances were sized for peak load at launch, workloads evolved, and nobody went back to revisit the original choices. A single db.r6g.4xlarge on AWS RDS running at 12% average CPU costs roughly $1,847/month on-demand. Downsizing to db.r6g.xlarge with a one-year Reserved Instance brings that to $412/month — a saving of $1,435 per month, per instance, without touching a line of application code. This chapter gives you the complete methodology: how to collect the right metrics, how to analyze them without being fooled by outliers, how to build the business case, and how to automate the entire process so right-sizing becomes a continuous operational discipline rather than a quarterly fire drill.

---

7.1 Why Instances Stay Oversized (And Why That's Not Stupidity)

Before reaching for the resize button, it is worth understanding why oversizing is so persistent. The forces that created it are rational, even if the outcome is wasteful.

The provisioning bias. When a database is first deployed, the team responsible for it has no production workload data. They extrapolate from development benchmarks, add a safety margin, round up to the next instance tier, and ship. This is not negligence — it is risk management under uncertainty. The problem is that the uncertainty resolves after go-live, but the safety margin never gets reclaimed.

The fear of the resize operation. On AWS RDS, resizing a Multi-AZ instance causes a failover. On Azure SQL, changing a service tier can trigger minutes of reduced connectivity. These are real concerns, especially for teams that have been burned by a botched maintenance window. The organizational response is to defer the operation indefinitely.

Ownership fragmentation. The DBA knows the CPU numbers. The finance team sees the invoice. The application team owns the deployment pipeline. Nobody owns the intersection of all three. FinOps as a discipline exists precisely to close this gap.

The right-sizing window. Practical experience shows that the best time to right-size is 8–12 weeks after a workload goes live. By then, you have enough data to identify true peak patterns but the team is still close enough to the original architecture to make confident changes. If that window passes, the next best opportunity is a quarterly review cycle anchored to billing data.

7.2 Collecting the Right Metrics

Right-sizing decisions made on CPU alone are dangerous. A database can run at 8% CPU while holding 95% of its RAM in buffer pool — shrink the instance and you blow the cache, triggering a cascade of disk I/O that destroys query latency. Collect all four dimensions before drawing any conclusions.

The four dimensions:

| Dimension | What to Measure | Safe Threshold for Downsizing |

|---|---|---|

| CPU | Average and P95 utilization | P95 < 40%, average < 20% |

| Memory | Buffer pool hit ratio, freeable memory | Freeable memory consistently > 20% of total |

| I/O | Read/write IOPS, throughput MB/s | Sustained utilization < 50% of provisioned |

| Connections | Active vs. max connections | Peak active < 40% of max\_connections |

Collecting from AWS CloudWatch

The following script pulls 30 days of RDS metrics for a given instance and outputs a summary. Schedule this with EventBridge to run on the first of every month.

#!/bin/bash
# collect_rds_metrics.sh
# Usage: ./collect_rds_metrics.sh <db-instance-identifier> <region>
# Schedule: AWS EventBridge, 1st of each month, 06:00 UTC

DB_INSTANCE="${1:-mydb-prod}"
REGION="${2:-us-east-1}"
END_DATE=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
START_DATE=$(date -u -d "30 days ago" +"%Y-%m-%dT%H:%M:%SZ")

echo "=== RDS Right-Sizing Report: $DB_INSTANCE ==="
echo "Period: $START_DATE to $END_DATE"
echo ""

# CPU Utilization - Average and P95
echo "--- CPU Utilization ---"
aws cloudwatch get-metric-statistics \
  --region "$REGION" \
  --namespace AWS/RDS \
  --metric-name CPUUtilization \
  --dimensions Name=DBInstanceIdentifier,Value="$DB_INSTANCE" \
  --start-time "$START_DATE" \
  --end-time "$END_DATE" \
  --period 86400 \
  --statistics Average Maximum \
  --output table

# Freeable Memory (bytes)
echo ""
echo "--- Freeable Memory (GB) ---"
aws cloudwatch get-metric-statistics \
  --region "$REGION" \
  --namespace AWS/RDS \
  --metric-name FreeableMemory \
  --dimensions Name=DBInstanceIdentifier,Value="$DB_INSTANCE" \
  --start-time "$START_DATE" \
  --end-time "$END_DATE" \
  --period 86400 \
  --statistics Average Minimum \
  --query 'Datapoints[*].{Date:Timestamp,AvgGB:to_number(Average)/1073741824,MinGB:to_number(Minimum)/1073741824}' \
  --output table

# Database Connections
echo ""
echo "--- Database Connections ---"
aws cloudwatch get-metric-statistics \
  --region "$REGION" \
  --namespace AWS/RDS \
  --metric-name DatabaseConnections \
  --dimensions Name=DBInstanceIdentifier,Value="$DB_INSTANCE" \
  --start-time "$START_DATE" \
  --end-time "$END_DATE" \
  --period 86400 \
  --statistics Average Maximum \
  --output table

# Read/Write IOPS
echo ""
echo "--- Read IOPS (avg/day) ---"
aws cloudwatch get-metric-statistics \
  --region "$REGION" \
  --namespace AWS/RDS \
  --metric-name ReadIOPS \
  --dimensions Name=DBInstanceIdentifier,Value="$DB_INSTANCE" \
  --start-time "$START_DATE" \
  --end-time "$END_DATE" \
  --period 86400 \
  --statistics Average Maximum \
  --output table

echo ""
echo "--- Write IOPS (avg/day) ---"
aws cloudwatch get-metric-statistics \
  --region "$REGION" \
  --namespace AWS/RDS \
  --metric-name WriteIOPS \
  --dimensions Name=DBInstanceIdentifier,Value="$DB_INSTANCE" \
  --start-time "$START_DATE" \
  --end-time "$END_DATE" \
  --period 86400 \
  --statistics Average Maximum \
  --output table

Querying workload behavior from inside PostgreSQL

CloudWatch tells you about resource consumption. pg_stat_statements tells you *why*. These two views must be read together. A database showing 15% CPU might have one query consuming 14% of it — downsize the instance and that query becomes a latency problem.

-- PostgreSQL: identify top resource consumers over the past 24 hours
-- Run this on the replica to avoid overhead on the primary
-- Requires pg_stat_statements extension (enabled by default on RDS, Aurora, Cloud SQL)

SELECT
    round(total_exec_time::numeric / 1000, 2)          AS total_exec_sec,
    calls,
    round(mean_exec_time::numeric, 2)                   AS mean_ms,
    round(stddev_exec_time::numeric, 2)                 AS stddev_ms,
    round((100 * total_exec_time /
           sum(total_exec_time) OVER ())::numeric, 2)   AS pct_of_total,
    rows,
    round(rows::numeric / nullif(calls, 0), 1)          AS rows_per_call,
    left(query, 120)                                    AS query_snippet
FROM
    pg_stat_statements
WHERE
    total_exec_time > 1000          -- at least 1 second cumulative
    AND calls > 10                  -- not a one-off
ORDER BY
    total_exec_time DESC
LIMIT 20;

-- Buffer cache hit ratio — if this drops below 95%, do NOT downsize memory
SELECT
    sum(heap_blks_hit)  AS heap_hits,
    sum(heap_blks_read) AS heap_reads,
    round(
        100.0 * sum(heap_blks_hit)
        / nullif(sum(heap_blks_hit) + sum(heap_blks_read), 0),
        2
    ) AS buffer_hit_ratio_pct
FROM
    pg_statio_user_tables;

-- Active connection distribution by state and wait event
-- A high count of "ClientRead" is a connection pooling problem, not a compute problem
SELECT
    state,
    wait_event_type,
    wait_event,
    count(*) AS connection_count
FROM
    pg_stat_activity
WHERE
    backend_type = 'client backend'
GROUP BY
    state, wait_event_type, wait_event
ORDER BY
    connection_count DESC;

Equivalent query for SQL Server on RDS or Azure SQL

-- SQL Server: top queries by CPU consumption (last execution plan cache)
-- Run on RDS SQL Server or Azure SQL Database
SELECT TOP 20
    qs.total_worker_time / 1000000.0               AS total_cpu_sec,
    qs.execution_count,
    qs.total_worker_time / qs.execution_count / 1000.0
                                                    AS avg_cpu_ms,
    qs.total_elapsed_time / qs.execution_count / 1000.0
                                                    AS avg_elapsed_ms,
    qs.total_logical_reads / qs.execution_count     AS avg_logical_reads,
    SUBSTRING(
        qt.text,
        (qs.statement_start_offset / 2) + 1,
        ((CASE qs.statement_end_offset
            WHEN -1 THEN DATALENGTH(qt.text)
            ELSE qs.statement_end_offset
          END - qs.statement_start_offset) / 2) + 1
    )                                               AS query_text
FROM
    sys.dm_exec_query_stats AS qs
    CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) AS qt
ORDER BY
    qs.total_worker_time DESC;

-- Memory pressure: page life expectancy
-- PLE < 300 seconds is a warning; < 100 seconds means the buffer pool is too small
SELECT
    object_name,
    counter_name,
    cntr_value AS page_life_expectancy_seconds
FROM
    sys.dm_os_performance_counters
WHERE
    counter_name = 'Page life expectancy'
    AND object_name LIKE '%Buffer Manager%';

7.3 Reading the Data Without Being Fooled

Raw metric averages mislead. A database that runs at 5% CPU for 23 hours and 95% CPU for one hour shows an average of 8.75% — and that one-hour burst is your month-end payroll batch or your nightly ETL. Right-size to remove that headroom and you break payroll.

The three-tier analysis framework

Tier 1 — Baseline: What does the instance look like during normal business hours on a typical weekday? Use the 50th percentile (median) across a 30-day window. This is the floor for sizing.

Tier 2 — Sustained peak: What is the P95 over the same window? This is the target utilization ceiling. If P95 CPU is 35%, the instance should be sized so that 35% CPU represents roughly 70% of its actual capacity, leaving a 2× headroom buffer.

Tier 3 — Burst events: What is the P99 or absolute maximum? These are the events that would cause customer-facing impact if the instance were undersized. Identify whether they are periodic and schedulable (ETL jobs, month-end reports) or unpredictable (traffic spikes from marketing campaigns). Periodic bursts can be handled with scheduled scaling or by moving those jobs to a read replica. Unpredictable bursts require true headroom in the primary.

The headroom formula

Required vCPUs = ceil(P95_CPU_utilization_pct / 0.70 × current_vCPUs)
Required RAM   = ceil(working_set_GB / 0.75)

Where working_set_GB is estimated as: total_database_size_GB × active_data_ratio. The active data ratio is typically 0.15–0.30 for OLTP workloads (only a fraction of tables are hot at any given time). If your buffer cache hit ratio is above 99%, your working set already fits in RAM. If it is between 95% and 99%, you are cache-constrained and should not reduce memory even if the instance appears to have freeable memory — freeable memory in RDS often reflects OS-level cache that PostgreSQL or MySQL will use immediately under load.

Instance family selection

AWS RDS offers multiple instance families, and the price difference between them is substantial:

| Family | vCPU:RAM Ratio | Best For | Example (4 vCPU) | On-Demand/mo |

|---|---|---|---|---|

| db.t4g | 1:2 (burstable) | Dev/test, low-traffic | db.t4g.xlarge | $119 |

| db.m7g | 1:4 (general) | Balanced OLTP | db.m7g.xlarge | $291 |

| db.r7g | 1:8 (memory) | Large buffer pools | db.r7g.xlarge | $450 |

| db.x2g | 1:16 (memory) | In-memory analytics | db.x2g.xlarge | $1,125 |

Moving a workload from db.r7g.2xlarge ($900/mo) to db.m7g.2xlarge ($582/mo) saves $318/month if the workload is CPU-bound rather than memory-bound. Over a three-year reserved term, that is $11,448 saved on a single instance.
