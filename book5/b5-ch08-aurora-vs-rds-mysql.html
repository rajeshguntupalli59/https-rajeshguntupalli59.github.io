Chapter 8: Aurora MySQL vs RDS MySQL

Aurora MySQL and RDS MySQL are both managed MySQL offerings on AWS, but they are architecturally different products that share a query interface. If you've spent years tuning InnoDB buffer pools and wrestling with MySQL replication, the differences matter enormously in production. One is a drop-in replacement for a self-managed MySQL installation with a managed control plane layered on top. The other is a purpose-built distributed storage engine disguised as MySQL. This chapter walks through how Aurora MySQL diverges from RDS MySQL at the storage layer, replication model, failover behavior, and cost structure — and how to choose between them for real production workloads.

---

8.1 Architectural Differences: Storage Layer and Replication Model

The single most important thing to understand about Aurora MySQL is that it is not running the InnoDB storage engine against local EBS volumes. Aurora separates compute from storage entirely. The Aurora storage subsystem is a distributed, fault-tolerant, self-healing storage layer that spans six storage nodes across three Availability Zones. When your writer instance commits a transaction, Aurora writes redo log records to quorum across those six nodes — four of six must acknowledge before the commit returns. The actual page materialization happens asynchronously in the storage layer. Aurora does not ship full page writes from compute to storage. It ships log records.

For a DBA who has tuned innodb_flush_log_at_trx_commit and worried about the fsync pipeline, this is a meaningful shift. The durability guarantee is enforced in the storage tier, not at the instance level. A single Aurora writer can sustain significantly higher write throughput than an equivalently sized RDS MySQL instance because it is not performing double-write buffer operations to local storage. The double-write buffer exists to protect against torn page writes — a problem Aurora's storage architecture eliminates by design.

RDS MySQL, by contrast, runs MySQL — with its full InnoDB engine including double-write buffer, undo tablespaces, and local EBS storage — with AWS managing provisioning, patching, automated backups, and Multi-AZ failover. Multi-AZ RDS MySQL uses synchronous block-level replication to a standby in a second AZ. That standby is not readable. It is purely a failover target. Failover on Multi-AZ RDS MySQL typically takes 60 to 120 seconds, involves a DNS flip, and requires applications to reconnect.

Aurora MySQL failover works differently. The reader instances are not replicas in the traditional sense — they share the same distributed storage volume as the writer. A reader can be promoted to writer in 30 seconds or less in typical conditions because it already has access to the same underlying storage. There is no replication lag between the writer and readers with respect to durable data; readers do see a brief lag in their buffer cache view of in-flight transactions, but they are reading from the same storage pages. Aurora advertises sub-30-second failover and typically achieves it.

Aurora also supports up to 15 read replicas. RDS MySQL supports up to 5 read replicas, and those replicas use standard MySQL binlog replication — which means they can lag, especially under heavy write load or large transactions. A single large ALTER TABLE on RDS MySQL can cause read replicas to fall minutes or hours behind. Aurora readers don't have that problem because they are not replaying binlog; they are reading pages that have already been materialized in the shared storage layer.

The trade-off: Aurora MySQL costs more per instance-hour than an equivalently sized RDS MySQL instance, and Aurora's storage pricing model is consumption-based ($0.10 per GB-month for storage, $0.20 per million I/O requests at time of writing — verify current pricing in your region). For read-heavy, low-write workloads, RDS MySQL with read replicas may be cheaper. For write-intensive workloads where replication lag on reads is unacceptable, Aurora wins.

8.2 Provisioning Aurora MySQL vs RDS MySQL with Terraform and AWS CLI

Provisioning an Aurora MySQL cluster differs structurally from provisioning an RDS MySQL instance. Aurora has two objects: the cluster and the instances within it. You provision the cluster — which owns the storage volume, the endpoints, and the parameter groups — and then you provision one or more DB instances into that cluster.

# Create an Aurora MySQL 8.0 cluster
aws rds create-db-cluster \
  --db-cluster-identifier prod-aurora-mysql \
  --engine aurora-mysql \
  --engine-version 8.0.mysql_aurora.3.04.0 \
  --master-username admin \
  --master-user-password "YourSecurePassword123!" \
  --db-subnet-group-name prod-subnet-group \
  --vpc-security-group-ids sg-0abc123def456 \
  --backup-retention-period 7 \
  --preferred-backup-window "02:00-03:00" \
  --preferred-maintenance-window "sun:04:00-sun:05:00" \
  --storage-encrypted \
  --kms-key-id arn:aws:kms:us-east-1:123456789012:key/your-kms-key-id \
  --enable-cloudwatch-logs-exports '["audit","error","general","slowquery"]' \
  --deletion-protection

# Add the writer instance to the cluster
aws rds create-db-instance \
  --db-instance-identifier prod-aurora-mysql-writer \
  --db-cluster-identifier prod-aurora-mysql \
  --db-instance-class db.r6g.2xlarge \
  --engine aurora-mysql

# Add a reader instance
aws rds create-db-instance \
  --db-instance-identifier prod-aurora-mysql-reader-1 \
  --db-cluster-identifier prod-aurora-mysql \
  --db-instance-class db.r6g.xlarge \
  --engine aurora-mysql

The equivalent in Terraform makes the cluster-versus-instance separation explicit:

resource "aws_rds_cluster" "aurora_mysql" {
  cluster_identifier      = "prod-aurora-mysql"
  engine                  = "aurora-mysql"

  database_name           = "appdb"
  master_username         = "admin"
  master_password         = var.db_master_password
  db_subnet_group_name    = aws_db_subnet_group.prod.name
  vpc_security_group_ids  = [aws_security_group.aurora.id]
  
  backup_retention_period = 7
  preferred_backup_window = "02:00-03:00"
  storage_encrypted       = true
  kms_key_id              = aws_kms_key.rds.arn
  deletion_protection     = true
  skip_final_snapshot     = false
  final_snapshot_identifier = "prod-aurora-mysql-final"
  
  enabled_cloudwatch_logs_exports = ["audit", "error", "general", "slowquery"]
  
  db_cluster_parameter_group_name = aws_rds_cluster_parameter_group.aurora_mysql8.name
}

resource "aws_rds_cluster_instance" "writer" {
  identifier         = "prod-aurora-mysql-writer"
  cluster_identifier = aws_rds_cluster.aurora_mysql.id
  instance_class     = "db.r6g.2xlarge"
  engine             = aws_rds_cluster.aurora_mysql.engine
  engine_version     = aws_rds_cluster.aurora_mysql.engine_version
  
  db_parameter_group_name = aws_db_parameter_group.aurora_mysql8_instance.name
  
  performance_insights_enabled          = true
  performance_insights_retention_period = 7
  monitoring_interval                   = 60
  monitoring_role_arn                   = aws_iam_role.rds_enhanced_monitoring.arn
}

resource "aws_rds_cluster_instance" "readers" {
  count              = 2
  identifier         = "prod-aurora-mysql-reader-${count.index + 1}"
  cluster_identifier = aws_rds_cluster.aurora_mysql.id
  instance_class     = "db.r6g.xlarge"
  engine             = aws_rds_cluster.aurora_mysql.engine
  engine_version     = aws_rds_cluster.aurora_mysql.engine_version
  
  db_parameter_group_name = aws_db_parameter_group.aurora_mysql8_instance.name
}

# Aurora has TWO levels of parameter groups: cluster-level and instance-level
resource "aws_rds_cluster_parameter_group" "aurora_mysql8" {
  name   = "prod-aurora-mysql8-cluster-params"
  family = "aurora-mysql8.0"

  parameter {
    name  = "character_set_server"
    value = "utf8mb4"
  }
  parameter {
    name  = "time_zone"
    value = "UTC"
  }
  parameter {
    name  = "slow_query_log"
    value = "1"
  }
  parameter {
    name  = "long_query_time"
    value = "1"
  }
}

resource "aws_db_parameter_group" "aurora_mysql8_instance" {
  name   = "prod-aurora-mysql8-instance-params"
  family = "aurora-mysql8.0"

  parameter {
    name  = "innodb_print_all_deadlocks"
    value = "1"
  }
}

Notice that Aurora uses two tiers of parameter groups: the cluster parameter group applies to all instances in the cluster and controls settings like character sets and binlog format. The instance parameter group applies to individual instances and controls settings like buffer pool behavior. In RDS MySQL, there is only one parameter group per instance. This is one of the first configuration gotchas that trips up DBAs coming from RDS MySQL.

For RDS MySQL, provisioning is simpler because there is no cluster abstraction:

# Create an RDS MySQL 8.0 Multi-AZ instance
aws rds create-db-instance \
  --db-instance-identifier prod-rds-mysql \
  --db-instance-class db.m6g.2xlarge \
  --engine mysql \
  --engine-version 8.0.35 \
  --master-username admin \
  --master-user-password "YourSecurePassword123!" \
  --allocated-storage 500 \
  --storage-type gp3 \
  --iops 12000 \
  --storage-encrypted \
  --multi-az \
  --db-subnet-group-name prod-subnet-group \
  --vpc-security-group-ids sg-0abc123def456 \
  --backup-retention-period 7 \
  --preferred-backup-window "02:00-03:00" \
  --preferred-maintenance-window "sun:04:00-sun:05:00" \
  --enable-performance-insights \
  --performance-insights-retention-period 7 \
  --monitoring-interval 60 \
  --monitoring-role-arn arn:aws:iam::123456789012:role/rds-monitoring-role \
  --deletion-protection \
  --no-publicly-accessible

With RDS MySQL you specify storage size explicitly — 500 GB in this case, provisioned with gp3 at 12,000 IOPS. With Aurora MySQL you never specify storage size; the storage grows automatically in 10 GB increments up to 128 TB. If your workload balloons unexpectedly, Aurora handles it. With RDS MySQL, you need to monitor free storage and either resize proactively or enable storage autoscaling.

8.3 Parameter Groups: What You Can and Cannot Tune

On-prem, you owned the MySQL configuration file entirely. On managed RDS or Aurora, AWS restricts a substantial number of parameters. Understanding which parameters are tunable, which are fixed, and which behave differently than on-prem is essential before production deployment.

The following query, run against a running Aurora MySQL instance, shows you which parameters are customer-modifiable:

-- Connect to your Aurora instance and check parameter apply status
-- This is better done via AWS CLI against the parameter group

-- But once connected, verify current effective values
SHOW VARIABLES LIKE 'innodb_buffer_pool_size';
SHOW VARIABLES LIKE 'innodb_io_capacity';
SHOW VARIABLES LIKE 'max_connections';
SHOW VARIABLES LIKE 'binlog_format';
SHOW VARIABLES LIKE 'transaction_isolation';

-- Aurora-specific variables worth checking
SHOW VARIABLES LIKE 'aurora%';
SHOW VARIABLES LIKE 'aurora_parallel_query';
SHOW VARIABLES LIKE 'aurora_read_replica_read_consistency';
# Check which parameters in your cluster parameter group are user-modifiable
aws rds describe-engine-default-cluster-parameters \
  --db-parameter-group-family aurora-mysql8.0 \
  --query 'EngineDefaults.Parameters[?IsModifiable==`true`].[ParameterName,ParameterValue,ApplyType]' \
  --output table | head -60

# Check a specific parameter's current value and constraints
aws rds describe-db-cluster-parameters \
  --db-cluster-parameter-group-name prod-aurora-mysql8-cluster-params \
  --query 'Parameters[?ParameterName==`innodb_io_capacity`]'

Key parameters and their cloud behavior:

innodb_buffer_pool_size: On RDS and Aurora, this is typically set automatically based on the instance class. On Aurora, the default is 75% of instance memory, which is generally correct. You can override it, but AWS recommends against it. On a db.r6g.2xlarge with 64 GB RAM, the buffer pool defaults to roughly 48 GB.

max_connections: Aurora sets this automatically based on instance size. On a db.r6g.xlarge (32 GB RAM), the default is approximately 2,000 connections. You can increase it, but doing so is usually a signal that you need RDS Proxy rather than a larger max_connections value. More on this in Section 8.4.

binlog_format: On Aurora MySQL, binlog is not required for replication within the cluster — Aurora uses its own log-based replication internally. However, if you're using Aurora as a source for DMS, or replicating to an external MySQL target, binlog must be enabled explicitly. Set binlog_format=ROW at the cluster parameter group level and enable log_bin if needed.

innodb_flush_log_at_trx_commit: This parameter exists on Aurora but has reduced significance. Aurora does not perform traditional InnoDB log flushing to local storage; the durability guarantee is enforced at the storage layer. The parameter is modifiable, but changing it from the default of 1 does not provide the traditional trade-off between durability and performance it would on self-managed MySQL.

Parameters you cannot change on managed services:

  • → `datadir` — fixed by AWS
  • → `innodb_data_file_path` — not applicable, Aurora manages tablespace layout
  • → `plugin_dir` — you cannot install arbitrary MySQL plugins
  • → `general_log` to a file location of your choosing — logs go to CloudWatch, not local filesystem
  • → `secure_file_priv` — `LOAD DATA INFILE` from arbitrary paths is restricted; use S3 integration instead

    • Aurora MySQL supports a native S3 integration for LOAD DATA operations:

    -- Load data from S3 into Aurora MySQL (requires aws_default_s3_role IAM role attached to cluster)
LOAD DATA FROM S3 's3://your-bucket/data/customers.csv'
INTO TABLE customers
FIELDS TERMINATED BY ','
ENCLOSED BY '"'
LINES TERMINATED BY '\n'
IGNORE 1 ROWS;

-- Export query results to S3
SELECT customer_id, email, created_at
INTO OUTFILE S3 's3://your-bucket/exports/customers_export'
FIELDS TERMINATED BY ','
LINES TERMINATED BY '\n'
FROM customers
WHERE created_at >= '2024-01-01';

    This replaces the on-prem pattern of mounting a network share or copying files locally before LOAD DATA INFILE. Secure it properly: the IAM role attached to the Aurora cluster needs s3:GetObject and s3:PutObject on the target bucket, with s3:ListBucket on the bucket itself.

    8.4 Connection Management: RDS Proxy and Aurora

    One of the most persistent operational pain points when migrating from self-managed MySQL to RDS is connection management. Self-managed MySQL installations behind ProxySQL or HAProxy gave you control over connection pooling, load balancing, query routing, and failover handling. AWS provides RDS Proxy as its managed connection pooler, and it integrates with both RDS MySQL and Aurora MySQL.

    RDS Proxy maintains a pool of persistent connections to the database and presents a proxy endpoint to applications. Applications connect to the proxy; the proxy multiplexes those connections against a smaller pool of real database connections. This is significant for Lambda functions, containerized applications, or any workload that opens and closes database connections frequently. Without a connection pooler, a spike in Lambda concurrency can open thousands of connections to Aurora simultaneously, exhausting max_connections and causing Too many connections errors.

    # Create an RDS Proxy for Aurora MySQL
aws rds create-db-proxy \
  --db-proxy-name prod-aurora-proxy \
  --engine-family MYSQL \
  --auth '[{
    "AuthScheme": "SECRETS",
    "SecretArn": "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/aurora/admin-AbCdEf",
    "IAMAuth": "REQUIRED"
  }]' \
  --role-arn arn:aws:iam::123456789012:role/rds-proxy-role \
  --vpc-subnet-ids subnet-0abc123 subnet-0def456 subnet-0ghi789 \
  --vpc-security-group-ids sg-0abc123def456 \
  --require-tls

# Register the Aurora cluster with the proxy
aws rds register-db-proxy-targets \
  --db-proxy-name prod-aurora-proxy \
  --db-cluster-identifiers prod-aurora-mysql

# Create a read-only proxy endpoint pointing to readers
aws rds create-db-proxy-endpoint \
  --db-proxy-name prod-aurora-proxy \
  --db-proxy-endpoint-name prod-aurora-proxy-readonly \
  --vpc-subnet-ids subnet-0abc123 subnet-0def456 subnet-0ghi789 \
  --vpc-security-group-ids sg-0abc123def456 \
  --target-role READ_ONLY

    RDS Proxy uses AWS Secrets Manager to store and rotate credentials. If you set IAMAuth: REQUIRED, applications authenticate to the proxy using IAM authentication rather than username/password. The proxy then uses the stored secret to authenticate to the actual database. This is the cloud equivalent of what you used to do with pg_hba.conf certificate authentication or LDAP integration — centralize credential management, enforce consistent authentication policies, and enable rotation without application restarts.

    import boto3
import pymysql
import os

def get_db_connection_with_iam():
    """
    Connect to RDS Proxy using IAM authentication.
    The proxy endpoint accepts an IAM auth token as the password.
    """
    region = 'us-east-1'
    proxy_host = 'prod-aurora-proxy.proxy-abc123.us-east-1.rds.amazonaws.com'
    db_user = 'app_user'
    db_name = 'appdb'
    port = 3306

    # Generate an IAM authentication token
    client = boto3.client('rds', region_name=region)
    token = client.generate_db_auth_token(
        DBHostname=proxy_host,
        Port=port,
        DBUsername=db_user,
        Region=region
    )

    # The token is used as the password; it expires after 15 minutes
    connection = pymysql.connect(
        host=proxy_host,
        user=db_user,
        password=token,
        database=db_name,
        port=port,
        ssl_ca='/path/to/us-east-1-bundle.pem',  # Download from AWS
        ssl_verify_cert=True,
        connect_timeout=5,
        read_timeout=30,
        write_timeout=30
    )
    
    return connection

def execute_with_retry(query, params=None, max_retries=3):
    """
    Production pattern: handle proxy reconnection on failover.
    RDS Proxy maintains connections across Aurora failover,
    but applications should still handle transient errors.
    """
    import time
    
    for attempt in range(max_retries):
        conn = None
        try:
            conn = get_db_connection_with_iam()
            with conn.cursor() as cursor:
                cursor.execute(query, params)
                conn.commit()
                return cursor.fetchall()
        except pymysql.err.OperationalError as e:
            error_code = e.args[0]
            # 2003: Can't connect, 2006: Server gone away
            if error_code in (2003, 2006) and attempt < max_retries - 1:
                wait = 2 ** attempt  # exponential backoff
                time.sleep(wait)
                continue
            raise
        finally:
            if conn:
                conn.close()

    RDS Proxy has important gotchas to understand before production deployment:

    Pinning: RDS Proxy multiplexes connections by reusing them between requests. When a client uses session-level state that the proxy cannot safely track — user variables (SET @var = value), temporary tables, or LOCK TABLES — the proxy "pins" that client to a specific backend connection and cannot share it. Excessive pinning defeats the purpose of connection pooling. Audit your application code for session-level state before enabling RDS Proxy. The CloudWatch metric DatabaseConnectionsCurrentlySessionPinned will tell you if this is a problem.

    Transaction pinning: Any open transaction pins a connection until commit or rollback. Long-running transactions in a pooled environment are as harmful here as they are with any connection pooler.

    Max connections to RDS Proxy: The proxy itself has a connection limit based on its size configuration. The default proxy scales automatically, but if you're routing very high connection counts through it, verify the proxy's MaxConnectionsPercent setting, which controls how much of the database's max_connections the proxy is allowed to use.

    8.5 Failover, Backups, and Aurora-Specific Features

    Failover behavior

    Aurora's failover is tier-based. You assign a failover priority from 0 to 15 to each reader instance using the PromotionTier parameter. In the event the writer fails, Aurora promotes the reader with the lowest tier number. If two readers share the same tier, Aurora promotes the one with the largest instance class. Set your highest-capacity reader to tier 0 to make failover behavior predictable.

    # Set failover priority on Aurora reader instances
aws rds modify-db-instance \
  --db-instance-identifier prod-aurora-mysql-reader-1 \
  --promotion-tier 0

aws rds modify-db-instance \
  --db-instance-identifier prod-aurora-mysql-reader-2 \
  --promotion-tier 1

# Trigger a manual failover to test promotion
aws rds failover-db-cluster \
  --db-cluster-identifier prod-aurora-mysql \
  --target-db-instance-identifier prod-aurora-mysql-reader-1

    Applications connecting to the Aurora cluster endpoint automatically route to whichever instance is the current writer. After failover, the cluster endpoint resolves to the new writer within approximately 30 seconds. Applications must handle the brief reconnection window — typically by catching connection errors and retrying with exponential backoff, as shown in the Python example above. Using RDS Proxy further reduces the impact because the proxy absorbs the failover and queues incoming requests while promotion completes.

    Backups and point-in-time recovery

    Aurora continuously backs up to S3. The backup window you configure is for the full snapshot, but redo logs are shipped to S3 continuously, enabling point-in-time recovery (PITR) to within 5 minutes of any point in the last 35 days (with maximum backup retention).
