Chapter 38: System Audits

A system audit is not just a compliance checkbox — it is one of the most powerful diagnostic and security tools a DBA has. Whether you are responding to a security incident, proving regulatory compliance, troubleshooting an unexpected data change, or simply establishing a health baseline for your environment, auditing gives you a traceable record of what happened, who did it, and when. This chapter covers how auditing works in both PostgreSQL and SQL Server, from the fundamental concepts of what you should be capturing to the architecture decisions that determine whether your audit system helps or hurts your production workload. By the end, you will know how to design, implement, and maintain an audit strategy that holds up under scrutiny — from internal reviews to regulatory bodies.

---

What System Audits Actually Capture

Before writing a single line of audit configuration, you need to understand what a database audit is actually recording. At its core, an audit is a structured log of events — actions taken against a database system — attributed to an identity, stamped with a time, and associated with an outcome. The events worth capturing fall into a few natural categories.

Authentication events record who logged in, when, from where, and whether the attempt succeeded or failed. A spike in failed logins against a service account is a meaningful signal. Without auditing, you would never see it.

Authorization and privilege changes record when someone was granted or revoked a role, when a schema permission changed, or when a superuser-level operation was performed. These events are often the most sensitive in a regulated environment because they represent structural changes to who is allowed to do what.

Data Definition Language (DDL) events record structural changes to objects — CREATE, ALTER, DROP, TRUNCATE. A production table being dropped at 2 AM on a Sunday is exactly the kind of event that should be immediately visible in an audit trail.

Data Manipulation Language (DML) events record changes to the data itself — INSERT, UPDATE, DELETE. Full DML auditing is expensive and generates enormous volume, so most organizations apply it selectively, targeting specific sensitive tables rather than the entire database.

Query and access events record what was read, which is particularly relevant for compliance frameworks like HIPAA, PCI-DSS, and GDPR where simply reading a record — viewing a patient's medication list, accessing a cardholder's stored number — is a recordable event.

Understanding these categories shapes your entire audit design. You rarely need to audit everything. You need to audit the right things — and that decision must be intentional.

---

PostgreSQL Audit Architecture

PostgreSQL does not ship with a built-in, GUI-driven audit module, but it provides robust primitives from which a complete audit system can be built. The two main mechanisms are the built-in logging system and the pgaudit extension.

PostgreSQL's native logging is controlled through postgresql.conf parameters. The log_connections and log_disconnections parameters record session lifecycle events. The log_statement parameter accepts values of none, ddl, mod, or all, providing coarse-grained control over which SQL statements are written to the server log. The log_min_duration_statement parameter is primarily a performance tool, but it creates an implicit audit trail of slow queries. These native logs land in flat files and are useful but unstructured — they are difficult to query systematically, and they do not cleanly separate audit information from operational noise.

pgaudit is the production-grade solution. It is an open-source extension maintained by the pgaudit project and supported on PostgreSQL versions back through 9.5. It hooks into the executor and produces clean, structured audit log records that are interleaved with the PostgreSQL log but clearly identifiable by their AUDIT: prefix. pgaudit operates at two levels: session-level auditing, which applies to the entire connection, and object-level auditing, which applies to specific tables, views, or functions.

Installing and enabling pgaudit requires adding it to shared_preload_libraries, installing the extension, and then configuring which statement classes to capture:

# postgresql.conf
shared_preload_libraries = 'pgaudit'
pgaudit.log = 'ddl, role, connection'
pgaudit.log_catalog = off
pgaudit.log_relation = on
pgaudit.log_parameter = on

The pgaudit.log parameter accepts a comma-separated list of statement classes: read, write, function, role, ddl, misc, and all. Setting log_catalog = off prevents audit noise from internal catalog lookups, which PostgreSQL generates aggressively during normal operation.

Once pgaudit is active, a DDL event produces a log entry like this:

AUDIT: SESSION,1,1,DDL,CREATE TABLE,,,CREATE TABLE employees (id serial, name text);,<none>

Each field is positional: audit type (SESSION or OBJECT), statement number, substatement number, statement class, command, object type, object name, statement text, and parameter list. This structure makes log parsing tractable.

For object-level auditing — tracking DML against specific sensitive tables — you create a dedicated audit role and grant it to the objects you want to monitor:

PostgreSQL:

-- Create the audit role and apply object-level auditing
CREATE ROLE auditor;
GRANT SELECT, INSERT, UPDATE, DELETE ON TABLE patients TO auditor;

-- Enable object auditing for the current session
SET pgaudit.role = 'auditor';

With this configuration, any access to the patients table — regardless of which user performed it — will generate an OBJECT-type audit record. This is more targeted and far less voluminous than session-level all auditing.

SQL Server Audit Architecture

SQL Server provides a native, enterprise-grade auditing framework built directly into the engine since SQL Server 2008. The SQL Server Audit feature is distinct from the older C2 audit mode and from SQL Trace/Extended Events — it is purpose-built for compliance and produces structured, tamper-evident audit records.

The architecture has three layers. A Server Audit defines where audit records go — a file on disk, the Windows Application Event Log, or the Windows Security Event Log. A Server Audit Specification captures server-level events such as logins, logouts, role membership changes, and server permission changes. A Database Audit Specification captures database-level events such as schema changes, DML against specific tables, and permission modifications within a database.

This layered model means you can have one audit destination collecting events from multiple specifications — both server-level and database-level — simultaneously.

SQL Server:

-- Step 1: Create the Server Audit (destination)
CREATE SERVER AUDIT [ProdAudit]
TO FILE
(
    FILEPATH = 'D:\SQLAudit\',
    MAXSIZE = 500 MB,
    MAX_ROLLOVER_FILES = 20,
    RESERVE_DISK_SPACE = OFF
)
WITH
(
    QUEUE_DELAY = 1000,
    ON_FAILURE = CONTINUE
);

ALTER SERVER AUDIT [ProdAudit] WITH (STATE = ON);

The QUEUE_DELAY parameter (in milliseconds) controls how quickly audit records are flushed. A value of 0 means synchronous write — the transaction cannot commit until the audit record is written. This is the most reliable setting but adds latency. A value of 1000 allows asynchronous buffering, which is appropriate for high-throughput environments where the audit destination can keep up. The ON_FAILURE clause determines behavior when the audit target becomes unavailable — CONTINUE keeps the server running, SHUTDOWN stops it, and FAIL_OPERATION rolls back only the audited operations.

With the Server Audit in place, you attach specifications:

SQL Server:

-- Step 2: Create a Database Audit Specification
USE HospitalDB;
GO

CREATE DATABASE AUDIT SPECIFICATION [HospitalDB_DataAudit]
FOR SERVER AUDIT [ProdAudit]
ADD (SELECT, INSERT, UPDATE, DELETE ON dbo.patients BY public),
ADD (SCHEMA_OBJECT_CHANGE_GROUP),
ADD (DATABASE_ROLE_MEMBER_CHANGE_GROUP)
WITH (STATE = ON);

The SCHEMA_OBJECT_CHANGE_GROUP and DATABASE_ROLE_MEMBER_CHANGE_GROUP are audit action groups — predefined collections of related events. SQL Server provides over 60 audit action groups covering nearly every auditable event category. Using groups rather than individual actions is the recommended pattern because groups are more maintainable and are automatically updated when Microsoft adds related sub-events.

Reading and Querying Audit Data

An audit trail is only useful if you can interrogate it. The mechanics of reading audit data differ significantly between the two platforms.

In PostgreSQL with pgaudit, audit records live in the PostgreSQL log files. On most Linux deployments using systemd, you can capture logs with journalctl, or you can configure PostgreSQL to write to CSV format using log_destination = 'csvlog' and logging_collector = on. CSV logging makes the audit records machine-readable without a parsing script:

PostgreSQL:

-- Create a foreign table to query the CSV log directly (PostgreSQL 14+)
-- Using file_fdw extension
CREATE EXTENSION IF NOT EXISTS file_fdw;

CREATE SERVER pglog FOREIGN DATA WRAPPER file_fdw;

CREATE FOREIGN TABLE postgres_log
(
    log_time             timestamp with time zone,
    user_name            text,
    database_name        text,
    process_id           integer,
    connection_from      text,
    session_id           text,
    session_line_num     bigint,
    command_tag          text,
    session_start_time   timestamp with time zone,
    virtual_transaction_id text,
    transaction_id       bigint,
    error_severity       text,
    sql_state_code       text,
    message              text,
    detail               text,
    hint                 text,
    internal_query       text,
    internal_query_pos   integer,
    context              text,
    query                text,
    query_pos            integer,
    location             text,
    application_name     text
)
SERVER pglog
OPTIONS (filename '/var/log/postgresql/postgresql-2024-01-15_0000.csv', format 'csv');

-- Query for audit events specifically
SELECT log_time, user_name, database_name, message
FROM postgres_log
WHERE message LIKE 'AUDIT:%'
ORDER BY log_time DESC
LIMIT 100;

For SQL Server, audit records written to file are read using the sys.fn_get_audit_file function, which presents them as a relational result set:

SQL Server:

-- Read audit records from the audit file path
SELECT
    event_time,
    server_principal_name,
    database_name,
    object_name,
    statement,
    action_id,
    succeeded,
    client_ip,
    application_name,
    session_id
FROM sys.fn_get_audit_file('D:\SQLAudit\ProdAudit_*.sqlaudit', DEFAULT, DEFAULT)
WHERE database_name = 'HospitalDB'
  AND event_time >= DATEADD(day, -7, GETUTCDATE())
ORDER BY event_time DESC;

The wildcard in the file path allows the function to read across rollover files, which is essential in production where a single query window might span multiple audit files. The action_id column is a four-character code (for example, SL for SELECT, IN for INSERT) — you can cross-reference these against sys.dm_audit_actions to get human-readable action names.

Both platforms benefit from moving audit data into a centralized repository. Many organizations ship PostgreSQL CSV logs to Elasticsearch or Splunk via Filebeat or Logstash. SQL Server shops frequently forward audit records to Azure Monitor or a centralized SQL table using scheduled jobs. In either case, the design principle is the same: audit data should not live only on the database server it was generated from, because that server is also the thing that might be compromised.

Auditing in High-Volume Environments

The tension between comprehensive auditing and system performance is real. Full DML auditing on a table that receives 50,000 inserts per second will produce audit records faster than most systems can write and store them. Production audit design must account for this.

In PostgreSQL, the performance cost of pgaudit is primarily I/O — writing more data to the log pipeline. You can reduce this pressure by being selective about which tables and operations require object-level auditing, and by tuning log_rotation_size and log_rotation_age to keep log files manageable. You can also configure pgaudit.log_parameter carefully — logging query parameters captures the actual values (essential for DML auditing) but adds significant log volume compared to capturing only statement text.

Another technique in PostgreSQL is using trigger-based auditing for high-value DML rather than relying on pgaudit's session-level write class. Triggers execute within the transaction, can capture old and new row values, and write directly to an audit table with whatever structure you define:

PostgreSQL:

-- Trigger-based audit table for targeted DML auditing
CREATE TABLE audit_log (
    audit_id     bigserial PRIMARY KEY,
    table_name   text NOT NULL,
    operation    text NOT NULL,
    changed_by   text NOT NULL DEFAULT current_user,
    changed_at   timestamptz NOT NULL DEFAULT now(),
    old_data     jsonb,
    new_data     jsonb
);

CREATE OR REPLACE FUNCTION audit_trigger_func()
RETURNS trigger AS $$
BEGIN
    IF TG_OP = 'DELETE' THEN
        INSERT INTO audit_log (table_name, operation, old_data)
        VALUES (TG_TABLE_NAME, TG_OP, to_jsonb(OLD));
        RETURN OLD;
    ELSIF TG_OP = 'UPDATE' THEN
        INSERT INTO audit_log (table_name, operation, old_data, new_data)
        VALUES (TG_TABLE_NAME, TG_OP, to_jsonb(OLD), to_jsonb(NEW));
        RETURN NEW;
    ELSIF TG_OP = 'INSERT' THEN
        INSERT INTO audit_log (table_name, operation, new_data)
        VALUES (TG_TABLE_NAME, TG_OP, to_jsonb(NEW));
        RETURN NEW;
    END IF;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

-- Apply the trigger to the sensitive table
CREATE TRIGGER patients_audit
AFTER INSERT OR UPDATE OR DELETE ON patients
FOR EACH ROW EXECUTE FUNCTION audit_trigger_func();

The SECURITY DEFINER clause means the function executes with the permissions of its owner (typically a privileged DBA account), not the session user. This prevents a user from bypassing the audit by revoking INSERT on the audit_log table from themselves.

In SQL Server, the QUEUE_DELAY parameter you saw earlier is the primary performance knob for the native audit. For very high-throughput environments, the file-based audit target is generally preferable to the Windows Event Log because it offers better write performance and supports larger rollover configurations. When the audit file target cannot keep up, SQL Server also supports using Extended Events sessions as an alternative or complement to native audit — Extended Events can write asynchronously to a ring buffer in memory or to an .xel file, and they add virtually no overhead when the buffer is not full.

A point of failure that DBAs sometimes miss: audit file paths must have sufficient disk space with a maintenance plan. SQL Server's MAX_ROLLOVER_FILES setting caps how many files are retained, which means old audit data is deleted automatically. If compliance requires retaining audit records for two years, the rollover policy and the archival process must be coordinated. The moment rollover deletes a file that a compliance officer needs, you have a problem.

Protecting Audit Integrity

An audit trail is only trustworthy if it cannot be silently modified. This is more nuanced than it sounds. Most DBAs intuitively protect the audit destination from ordinary users, but a determined attacker with superuser or sysadmin access can often modify or delete audit records if the destination is on the same system as the database.

In PostgreSQL, protect audit integrity through several layers. First, ensure the PostgreSQL server process does not own the log directory with write-modify-delete access for any other OS user. Second, restrict who can reconfigure pgaudit settings — only superusers can set pgaudit.log, so managing superuser access tightly is directly related to audit integrity. Third, for trigger-based audit tables, deny TRUNCATE, DELETE, and DROP on the audit table to all application roles:

PostgreSQL:

-- Revoke destructive permissions on the audit table from application roles
REVOKE DELETE, TRUNCATE ON TABLE audit_log FROM app_role;
REVOKE ALL ON TABLE audit_log FROM PUBLIC;

-- Grant only INSERT to the audit function's owner
GRANT INSERT ON TABLE audit_log TO audit_owner;

-- Grant SELECT to the auditor role for review purposes
GRANT SELECT ON TABLE audit_log TO auditor_read_role;

For trigger-based audit tables, also consider partitioning by month and creating new partitions at the beginning of each period. Once a partition is past its period, it can be set to read-only (by revoking write permissions) and archived off-system before being detached.

In SQL Server, the native audit file format is binary and tamper-evident — SQL Server can detect if an audit file has been modified externally. However, a member of the sysadmin role can stop and restart the audit, potentially creating a gap in coverage. To detect this, monitor for AUDIT_CHANGE_GROUP events in the Server Audit Specification — this group fires whenever an audit configuration is modified, started, or stopped. Alerting on this event closes the loop on audit tampering at the configuration level.

SQL Server:

-- Add audit configuration change detection to the Server Audit Specification
ALTER SERVER AUDIT SPECIFICATION [ServerLevelAudit]
    ADD (AUDIT_CHANGE_GROUP)
WITH (STATE = OFF);  -- Must be off to modify, then re-enable

ALTER SERVER AUDIT SPECIFICATION [ServerLevelAudit] WITH (STATE = ON);

-- Query for any audit configuration changes in the last 30 days
SELECT
    event_time,
    server_principal_name,
    action_id,
    statement
FROM sys.fn_get_audit_file('D:\SQLAudit\*.sqlaudit', DEFAULT, DEFAULT)
WHERE action_id = 'AUSC'  -- AUSC = Audit Session Change
ORDER BY event_time DESC;

Beyond the database itself, audit data should be replicated to a write-once or append-only destination outside the database server's control as quickly as operationally feasible. Cloud storage with object lock (S3 Object Lock, Azure Immutable Blob Storage) is an increasingly common choice. These services allow you to set a retention lock that even the storage account owner cannot break during the lock period — exactly the property regulators expect from an audit trail.

---

Key Takeaways

  • → Effective auditing requires intentional scope — capturing authentication events, privilege changes, DDL, and targeted DML on sensitive tables covers most regulatory requirements without burying your system in log volume.
  • → PostgreSQL auditing is built on `pgaudit` for structured session and object-level logging, optionally supplemented by trigger-based audit tables that capture old and new row values directly to a queryable schema.
  • → SQL Server's native audit architecture — Server Audit, Server Audit Specification, and Database Audit Specification — provides a layered, enterprise-grade system where audit action groups simplify configuration and the binary file format provides tamper evidence.
  • → High-volume environments must balance audit completeness against I/O cost; asynchronous write modes, selective object-level scoping, and off-server archival are the primary tools for managing that trade-off.
  • → Audit integrity depends not just on capturing events but on protecting the audit destination from modification — deny destructive operations on audit tables, monitor for audit configuration changes, and archive records to write-once storage outside the database server's control.
