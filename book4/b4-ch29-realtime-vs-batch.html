Chapter 29: Real-Time vs Batch Embedding Generation

Embedding generation sits at the boundary between your application's request path and your vector database's indexing pipeline. Every vector in your store started as a raw document, image, or structured record that passed through a model to become a float array — and the decision about *when* that transformation happens shapes your latency profile, cost structure, operational complexity, and freshness guarantees in ways that compound at scale. This chapter maps out the two fundamental strategies — real-time and batch — examines where each breaks down, and shows the hybrid architectures that production systems almost always end up building.

---

29.1 Understanding the Embedding Generation Pipeline

Before choosing a strategy, you need a clear model of what the pipeline actually does. Embedding generation is a compute-intensive inference step: a neural network reads tokens, runs a forward pass, and returns a fixed-size dense vector. That vector is then written to your vector store and becomes searchable.

Think of it like a computed column in a relational database. You have a source table of raw content and a derived representation stored separately for efficient retrieval. The question isn't whether to compute it — you must — it's whether you compute it synchronously during write operations or asynchronously in a scheduled or streaming process.

The pipeline stages:

Raw Content → Chunking/Preprocessing → Model Inference → Vector Storage → Index Build

Each stage introduces latency and can fail independently. In real-time mode, all of this sits in your request path. In batch mode, only the final query lookup is synchronous.

Key parameters that drive strategy selection:

| Parameter | Favors Real-Time | Favors Batch |

|---|---|---|

| Content ingestion rate | Low to moderate | High volume |

| Latency tolerance | < 500ms end-to-end | Minutes to hours acceptable |

| Content update frequency | Continuous | Periodic |

| Embedding model size | API-hosted, fast | Large local models |

| Cost sensitivity | Bursty, pay-per-call | Predictable volume pricing |

| Content at query time | Dynamic, user-generated | Stable corpus |

The embedding model itself is the most constrained resource. OpenAI's text-embedding-3-small processes roughly 1M tokens per minute on the API tier, costing $0.02 per 1M tokens. A local sentence-transformers/all-MiniLM-L6-v2 running on CPU processes approximately 500 sentences per second on a modern machine — fast enough for many real-time use cases, but it becomes a bottleneck as soon as content volume spikes.

---

29.2 Real-Time Embedding Generation

Real-time embedding generation means computing vectors synchronously as new content arrives — either during document ingestion or at query time for incoming search strings. The user or upstream system waits for the embedding to complete before the request continues.

When this approach works well:

  • → A user submits a search query and you need the query embedding immediately
  • → A chat application ingests user messages that must be retrievable within seconds
  • → An API receives webhook events that feed a RAG pipeline with sub-minute freshness requirements

    • When it breaks:

  • → Document ingestion volume spikes (model API rate limits become throttling events)
  • → Embedding latency becomes the dominant term in your p95 response time
  • → Upstream content sources send batches disguised as streams (100 documents at once)

    • The following example shows a FastAPI endpoint that ingests documents and generates embeddings synchronously using the OpenAI API and pgvector for storage:

    # Python
import asyncio
import time
from typing import Optional

import asyncpg
from fastapi import FastAPI, HTTPException
from openai import AsyncOpenAI
from pydantic import BaseModel

app = FastAPI()
openai_client = AsyncOpenAI()

# Connection pool — reuse across requests
pool: Optional[asyncpg.Pool] = None


async def get_pool() -> asyncpg.Pool:
    global pool
    if pool is None:
        pool = await asyncpg.create_pool(
            dsn="postgresql://app_user:secret@localhost:5432/vectordb",
            min_size=5,
            max_size=20,
        )
    return pool


class DocumentIngest(BaseModel):
    document_id: str
    content: str
    metadata: dict = {}


class SearchQuery(BaseModel):
    query: str
    top_k: int = 10


@app.post("/documents")
async def ingest_document(doc: DocumentIngest):
    """Real-time ingestion: embed and store in a single synchronous operation."""
    start = time.monotonic()

    # Step 1: Generate embedding — this is the latency bottleneck
    try:
        response = await openai_client.embeddings.create(
            model="text-embedding-3-small",
            input=doc.content,
            encoding_format="float",
        )
    except Exception as e:
        raise HTTPException(status_code=502, detail=f"Embedding API error: {e}")

    embedding = response.data[0].embedding
    embed_latency_ms = (time.monotonic() - start) * 1000

    # Step 2: Write to pgvector
    db = await get_pool()
    try:
        await db.execute(
            """
            INSERT INTO documents (document_id, content, embedding, metadata, embedded_at)
            VALUES ($1, $2, $3::vector, $4, NOW())
            ON CONFLICT (document_id) DO UPDATE
              SET content = EXCLUDED.content,
                  embedding = EXCLUDED.embedding,
                  embedded_at = NOW()
            """,
            doc.document_id,
            doc.content,
            str(embedding),
            doc.metadata,
        )
    except Exception as e:
        raise HTTPException(status_code=500, detail=f"DB write error: {e}")

    total_latency_ms = (time.monotonic() - start) * 1000

    return {
        "document_id": doc.document_id,
        "embed_latency_ms": round(embed_latency_ms, 2),
        "total_latency_ms": round(total_latency_ms, 2),
        "dimensions": len(embedding),
    }


@app.post("/search")
async def search(query: SearchQuery):
    """Query embedding is always real-time — there is no alternative."""
    response = await openai_client.embeddings.create(
        model="text-embedding-3-small",
        input=query.query,
        encoding_format="float",
    )
    query_vector = response.data[0].embedding

    db = await get_pool()
    rows = await db.fetch(
        """
        SELECT document_id, content, metadata,
               1 - (embedding <=> $1::vector) AS cosine_similarity
        FROM documents
        ORDER BY embedding <=> $1::vector
        LIMIT $2
        """,
        str(query_vector),
        query.top_k,
    )

    return [dict(r) for r in rows]

    The critical observation here: query embedding is *always* real-time. You cannot precompute embeddings for arbitrary user queries. The only decision is whether document embeddings are also real-time or handled separately.

    Concurrency and rate limit management become the primary operational concerns. A real-time pipeline that handles 50 concurrent document writes will fire 50 simultaneous API calls. OpenAI's text-embedding-3-small has a default rate limit of 3,000 RPM (requests per minute) on paid tiers — roughly 50 per second. You need a semaphore or queue to stay within bounds:

    # Python
import asyncio
from openai import AsyncOpenAI

client = AsyncOpenAI()

# Limit concurrent embedding calls to avoid rate limit errors
EMBED_SEMAPHORE = asyncio.Semaphore(20)


async def safe_embed(text: str, model: str = "text-embedding-3-small") -> list[float]:
    """Rate-limited embedding call with exponential backoff."""
    async with EMBED_SEMAPHORE:
        for attempt in range(4):
            try:
                response = await client.embeddings.create(
                    model=model,
                    input=text,
                    encoding_format="float",
                )
                return response.data[0].embedding
            except Exception as e:
                if "rate_limit" in str(e).lower() and attempt < 3:
                    wait = (2**attempt) + 0.1
                    await asyncio.sleep(wait)
                else:
                    raise


async def embed_batch_concurrent(texts: list[str], max_concurrency: int = 20) -> list[list[float]]:
    """Embed a list of texts with bounded concurrency."""
    tasks = [safe_embed(t) for t in texts]
    return await asyncio.gather(*tasks)

    29.3 Batch Embedding Generation

    Batch processing treats embedding generation as an offline transformation job — a scheduled or event-triggered pipeline that processes accumulated content and writes vectors to the store without blocking application writes. The source system records raw content immediately; embedding happens asynchronously.

    This maps directly to the ETL/ELT model that DBAs know well. The source table stores canonical data, and a separate process materializes the derived representation. You can think of it as a materialized view that refreshes on a schedule, except the "computation" is a neural network inference rather than a SQL aggregation.

    Architecture pattern:

    Source Table (raw) → Embedding Worker → Vector Store
     ↓ writes immediately               ↑ available after job completes
     ↓ marks as embedding_status='pending'

    The embedding_status column is the coordination mechanism. Writers insert content and mark it pending; the embedding worker polls or subscribes for pending records, processes them in optimized batches, and marks them complete.

    -- PostgreSQL / pgvector

-- Source table with embedding lifecycle tracking
CREATE TABLE documents (
    id             BIGSERIAL PRIMARY KEY,
    document_id    TEXT UNIQUE NOT NULL,
    content        TEXT NOT NULL,
    content_hash   TEXT GENERATED ALWAYS AS (md5(content)) STORED,
    embedding      vector(1536),
    embed_model    TEXT,
    embedding_status TEXT NOT NULL DEFAULT 'pending'
                   CHECK (embedding_status IN ('pending', 'processing', 'complete', 'failed')),
    retry_count    INT NOT NULL DEFAULT 0,
    embedded_at    TIMESTAMPTZ,
    created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Index for the worker's polling query — only pending/failed rows
CREATE INDEX idx_documents_embed_queue
    ON documents (created_at)
    WHERE embedding_status IN ('pending', 'failed') AND retry_count < 3;

-- Claim a batch atomically using SKIP LOCKED — prevents double-processing
-- across multiple worker instances
CREATE OR REPLACE FUNCTION claim_embedding_batch(batch_size INT)
RETURNS SETOF documents AS $$
    UPDATE documents
    SET embedding_status = 'processing',
        updated_at = NOW()
    WHERE id IN (
        SELECT id FROM documents
        WHERE embedding_status IN ('pending', 'failed')
          AND retry_count < 3
        ORDER BY created_at
        LIMIT batch_size
        FOR UPDATE SKIP LOCKED
    )
    RETURNING *;
$$ LANGUAGE sql;

    The FOR UPDATE SKIP LOCKED pattern is the same technique used in job queue implementations like Sidekiq's PostgreSQL backend. Multiple embedding workers can run in parallel without coordination — each atomically claims its batch, processes it, and writes results back.

    # Python
import time
from datetime import datetime
from typing import Optional

import psycopg2
import psycopg2.extras
from sentence_transformers import SentenceTransformer

# Using a local model for batch processing — no API costs, full throughput control
model = SentenceTransformer("sentence-transformers/all-MiniLM-L6-v2")
MODEL_NAME = "all-MiniLM-L6-v2"
BATCH_SIZE = 256  # Optimal for CPU inference; increase to 512+ with GPU


def get_connection():
    return psycopg2.connect(
        dsn="postgresql://app_user:secret@localhost:5432/vectordb"
    )


def process_batch(conn) -> int:
    """Claim and process one batch of pending documents. Returns count processed."""
    with conn.cursor(cursor_factory=psycopg2.extras.RealDictCursor) as cur:
        # Atomically claim a batch
        cur.execute("SELECT * FROM claim_embedding_batch(%s)", (BATCH_SIZE,))
        rows = cur.fetchall()
        conn.commit()

    if not rows:
        return 0

    ids = [r["id"] for r in rows]
    texts = [r["content"] for r in rows]

    start = time.monotonic()
    # sentence-transformers handles internal batching efficiently
    embeddings = model.encode(
        texts,
        batch_size=64,
        show_progress_bar=False,
        normalize_embeddings=True,  # Cosine similarity becomes dot product
    )
    inference_ms = (time.monotonic() - start) * 1000

    # Bulk update — much faster than row-by-row
    with conn.cursor() as cur:
        update_data = [
            (
                embeddings[i].tolist(),
                MODEL_NAME,
                "complete",
                datetime.utcnow(),
                ids[i],
            )
            for i in range(len(ids))
        ]
        psycopg2.extras.execute_batch(
            cur,
            """
            UPDATE documents
            SET embedding = %s::vector,
                embed_model = %s,
                embedding_status = %s,
                embedded_at = %s,
                updated_at = NOW()
            WHERE id = %s
            """,
            update_data,
            page_size=100,
        )
        conn.commit()

    throughput = len(ids) / (inference_ms / 1000)
    print(
        f"Processed {len(ids)} docs in {inference_ms:.0f}ms "
        f"({throughput:.0f} docs/sec)"
    )
    return len(ids)


def handle_failures(conn, failed_ids: list[int]):
    """Mark failed rows for retry or permanent failure."""
    with conn.cursor() as cur:
        cur.execute(
            """
            UPDATE documents
            SET embedding_status = CASE
                WHEN retry_count >= 2 THEN 'failed'
                ELSE 'pending'
            END,
            retry_count = retry_count + 1,
            updated_at = NOW()
            WHERE id = ANY(%s)
            """,
            (failed_ids,),
        )
        conn.commit()


def run_worker(idle_sleep_sec: float = 5.0):
    """Continuous worker loop."""
    conn = get_connection()
    print(f"Embedding worker started. Model: {MODEL_NAME}, Batch size: {BATCH_SIZE}")
    try:
        while True:
            processed = process_batch(conn)
            if processed == 0:
                time.sleep(idle_sleep_sec)
    finally:
        conn.close()


if __name__ == "__main__":
    run_worker()

    Throughput benchmarks with local models are worth establishing early. On a single CPU worker with all-MiniLM-L6-v2:

  • → Batch size 64: ~400 docs/sec
  • → Batch size 256: ~600 docs/sec
  • → Batch size 512: ~650 docs/sec (diminishing returns)

    • On a single A10G GPU:

  • → Batch size 256: ~4,500 docs/sec
  • → Batch size 1024: ~8,000 docs/sec

    • For comparison, OpenAI's batch embedding API (asynchronous, 50% cheaper than synchronous) processes up to 50,000 requests per batch with a 24-hour completion window.

    29.4 Hybrid Architectures: Combining Both Strategies

    Production systems rarely use pure real-time or pure batch. The practical pattern is a tiered architecture where query embeddings are always real-time, freshly ingested high-priority content uses real-time embedding with graceful degradation, and bulk content uses batch processing.

    A common pattern that works well: write the raw document to storage immediately (making it visible to the application), generate the embedding asynchronously, and return a pending state to callers that need to know embedding status. This is analogous to an eventually consistent write — your database accepts the write, your derived index catches up.

    The following example shows this pattern using Pinecone as the vector store and a message queue (Redis Streams) for coordination:

    # Python
import json
import time
import uuid
from enum import Enum
from typing import Optional

import redis
from openai import OpenAI
from pinecone import Pinecone, ServerlessSpec

# Configuration
REDIS_URL = "redis://localhost:6379"
PINECONE_API_KEY = "your-pinecone-key"
PINECONE_INDEX = "documents-prod"
EMBED_MODEL = "text-embedding-3-small"
EMBED_DIMS = 1536
STREAM_KEY = "embed:queue"
CONSUMER_GROUP = "embed-workers"

# Clients
redis_client = redis.from_url(REDIS_URL, decode_responses=True)
openai_client = OpenAI()
pc = Pinecone(api_key=PINECONE_API_KEY)

# Ensure index exists
if PINECONE_INDEX not in [i.name for i in pc.list_indexes()]:
    pc.create_index(
        name=PINECONE_INDEX,
        dimension=EMBED_DIMS,
        metric="cosine",
        spec=ServerlessSpec(cloud="aws", region="us-east-1"),
    )

index = pc.Index(PINECONE_INDEX)


class EmbedPriority(str, Enum):
    HIGH = "high"   # Real-time path: SLA < 2 seconds
    NORMAL = "normal"  # Async path: acceptable delay up to 30 seconds
    LOW = "low"        # Batch path: delay up to hours acceptable


def ingest_document(
    document_id: str,
    content: str,
    metadata: dict,
    priority: EmbedPriority = EmbedPriority.NORMAL,
) -> dict:
    """
    Write document to the queue. HIGH priority documents are embedded inline.
    NORMAL/LOW are deferred to the async worker.
    """
    if priority == EmbedPriority.HIGH:
        # Synchronous path — caller waits for embedding
        embedding = _embed_text(content)
        _write_to_pinecone(document_id, embedding, content, metadata)
        return {"document_id": document_id, "status": "indexed", "priority": priority}
    else:
        # Async path — enqueue for worker
        message = {
            "document_id": document_id,
            "content": content,
            "metadata": json.dumps(metadata),
            "priority": priority,
            "enqueued_at": time.time(),
        }
        redis_client.xadd(STREAM_KEY, message, maxlen=100_000)
        return {"document_id": document_id, "status": "pending", "priority": priority}


def _embed_text(text: str) -> list[float]:
    response = openai_client.embeddings.create(
        model=EMBED_MODEL,
        input=text,
        encoding_format="float",
    )
    return response.data[0].embedding


def _write_to_pinecone(
    document_id: str,
    embedding: list[float],
    content: str,
    metadata: dict,
):
    index.upsert(
        vectors=[{
            "id": document_id,
            "values": embedding,
            "metadata": {**metadata, "content_preview": content[:500]},
        }],
        namespace="documents",
    )


def run_async_worker(batch_size: int = 50, idle_sleep: float = 1.0):
    """
    Redis Streams consumer that processes the embedding queue.
    Multiple instances can run concurrently — each reads from the same group.
    """
    worker_id = f"worker-{uuid.uuid4().hex[:8]}"

    # Create consumer group if it doesn't exist
    try:
        redis_client.xgroup_create(STREAM_KEY, CONSUMER_GROUP, id="0", mkstream=True)
    except redis.exceptions.ResponseError:
        pass  # Group already exists

    print(f"Worker {worker_id} started, consuming from {STREAM_KEY}")

    while True:
        # Read up to batch_size messages, blocking for 1 second if queue is empty
        messages = redis_client.xreadgroup(
            groupname=CONSUMER_GROUP,
            consumername=worker_id,
            streams={STREAM_KEY: ">"},
            count=batch_size,
            block=int(idle_sleep * 1000),
        )

        if not messages:
            continue

        stream_messages = messages[0][1]  # (stream_key, [(msg_id, fields), ...])
        msg_ids = [m[0] for m in stream_messages]
        payloads = [m[1] for m in stream_messages]

        # Sort by priority — HIGH items get embedded first within the batch
        payloads_with_ids = sorted(
            zip(msg_ids, payloads),
            key=lambda x: 0 if x[1]["priority"] == EmbedPriority.HIGH else 1,
        )

        texts = [p["content"] for _, p in payloads_with_ids]

        # Batch embed — single API call for the whole batch
        try:
            response = openai_client.embeddings.create(
                model=EMBED_MODEL,
                input=texts,
                encoding_format="float",
            )
            embeddings = [d.embedding for d in response.data]
        except Exception as e:
            print(f"Embedding batch failed: {e}")
            # Messages remain in PEL (pending entries list) for reprocessing
            time.sleep(5)
            continue

        # Upsert to Pinecone in a single batch call
        vectors = []
        for i, (msg_id, payload) in enumerate(payloads_with_ids):
            vectors.append({
                "id": payload["document_id"],
                "values": embeddings[i],
                "metadata": {
                    **json.loads(payload["metadata"]),
                    "content_preview": payload["content"][:500],
                    "embedded_at": time.time(),
                },
            })

        index.upsert(vectors=vectors, namespace="documents")

        # Acknowledge successful processing
        redis_client.xack(STREAM_KEY, CONSUMER_GROUP, *msg_ids)
        print(f"Worker {worker_id}: processed {len(vectors)} documents")

    This architecture gives you exactly three knobs to tune: the concurrency of async workers, the batch size per worker iteration, and the priority threshold for synchronous embedding. Most teams start with HIGH priority for user-generated content that needs immediate search availability and NORMAL/LOW for system-generated or bulk content.

    29.5 Cost Optimization and Model Selection

    Embedding generation costs break down differently depending on your strategy. Real-time API calls are billed per token with no volume discount at standard tiers; batch jobs can qualify for discounted batch pricing or allow you to substitute cheaper local models where latency isn't constrained.

    Cost comparison for 100M tokens per month:

    | Model | Mode | Cost | Latency (p50) |

    |---|---|---|---|

    | OpenAI text-embedding-3-small | Synchronous API | $2.00 | ~80ms |
