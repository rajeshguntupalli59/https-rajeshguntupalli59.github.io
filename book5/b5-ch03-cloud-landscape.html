Chapter 3: Cloud Database Landscape: AWS, Azure, GCP Side-by-Side

Every cloud provider sells you the same promise: managed databases that eliminate the undifferentiated heavy lifting of patching, backups, and hardware provisioning. The reality is that AWS, Azure, and GCP have taken fundamentally different architectural philosophies to reach that promise, and those differences shape every decision you make — from instance sizing to failover behavior to how you debug a slow query at 2 a.m. This chapter builds the mental map you need before writing a single line of Terraform or clicking a single console button. You will see how each platform organizes its database services, where their managed offerings genuinely replace on-prem work, and where the seams show.

---

3.1 The Service Taxonomy: What Each Cloud Actually Offers

Before comparing specifics, you need to understand the organizational logic of each platform. They do not map cleanly onto each other, and conflating them causes real architectural mistakes.

AWS Database Services

AWS built its database portfolio as a collection of purpose-built, largely independent services. For relational workloads, the primary offerings are:

  • → **Amazon RDS**: The foundational managed relational service. Supports MySQL, PostgreSQL, MariaDB, Oracle, and SQL Server. You pick an engine, pick an instance class, and AWS handles patching, backups, and failover. RDS is the closest thing to "take your on-prem database and move it to a managed host."
  • → **Amazon Aurora**: AWS's rewritten storage layer, available in MySQL-compatible and PostgreSQL-compatible variants. Aurora separates compute from storage in a way standard RDS does not. The storage layer automatically grows in 10 GB increments up to 128 TB, replicates across three Availability Zones at the storage level, and allows up to 15 read replicas without the replication lag penalty you see on standard RDS. Aurora is not just "faster RDS" — it is a different internal architecture.
  • → **Aurora Serverless v2**: Compute that scales in Aurora Capacity Units (ACUs) from 0.5 ACU to 128 ACUs, with sub-second scaling response. One ACU is approximately 2 GB of memory with proportional CPU. Useful for variable workloads, but the cold-start behavior when scaling from near-zero can surprise you in production.
  • → **RDS Proxy**: A fully managed connection pooler that sits in front of RDS or Aurora. It maintains a warm pool of connections to the database and presents a single endpoint to your application. Relevant when you have Lambda functions or containers generating thousands of short-lived connections.

    • Azure Database Services

    Azure organizes its relational offerings around the concept of deployment models — how much of the infrastructure stack you control:

  • → **Azure SQL Database**: A fully managed PaaS service built on SQL Server. No OS access, no SQL Server Agent (limited version available), but 99.99% SLA and automatic tuning available. The purchasing model adds complexity: you choose between DTU-based pricing (a bundled compute/IO unit AWS has no equivalent for) and vCore-based pricing (which maps more cleanly to familiar instance sizing).
  • → **Azure SQL Managed Instance**: The "nearly full SQL Server" option. You get SQL Agent, cross-database queries, linked servers, and CLR — features that disappear in Azure SQL Database. The tradeoff is a longer provisioning time (up to four hours for some operations) and a higher cost floor.
  • → **Azure Database for PostgreSQL / MySQL**: Managed open-source database services, analogous to RDS PostgreSQL/MySQL. The Flexible Server deployment model is the current recommended path; the older Single Server model is being retired.
  • → **Azure SQL Hyperscale**: A distributed storage architecture for Azure SQL Database that allows databases up to 100 TB. The page server architecture separates compute from storage and enables near-instant database restores regardless of database size — a restore of a 10 TB database takes roughly the same time as restoring a 10 GB database.
  • → **Elastic Pools**: A resource-sharing model where multiple databases share a pool of DTUs or vCores. This is Azure's answer to database consolidation — comparable to running multiple schemas on one PostgreSQL instance, but with more granular resource governance.

    • Google Cloud Database Services

    GCP's philosophy leans toward horizontal scale and global distribution more heavily than the other two:

  • → **Cloud SQL**: The managed MySQL, PostgreSQL, and SQL Server service. Functionally similar to RDS — it handles patching, backups, and HA failover. Storage scales automatically up to 64 TB. Cloud SQL HA uses a standby instance in a different zone with synchronous replication, promoted automatically on failure.
  • → **AlloyDB for PostgreSQL**: Google's reengineered PostgreSQL-compatible service, analogous to how Aurora relates to RDS. AlloyDB separates log processing from storage, uses a distributed storage layer with columnar caching, and claims 4x faster transactional throughput than standard Cloud SQL PostgreSQL. AlloyDB also integrates the Vertex AI embedding generation directly within the database, which becomes relevant in Book 4 of this series.
  • → **Cloud Spanner**: A globally distributed, horizontally scalable relational database with external consistency (a stronger guarantee than serializable isolation). Spanner is in a category of its own — it uses TrueTime to order transactions across geographic regions. The SQL dialect is ANSI-compatible but not identical to PostgreSQL or SQL Server. Spanner is not a drop-in replacement for anything; it requires application-level rethinking.
  • → **BigQuery**: Analytical, serverless, columnar. Not covered in this book except where it intersects with operational database exports.
  • → **Firestore / Bigtable**: NoSQL services outside the scope of this book.

    • The practical implication of this taxonomy: when a product manager says "we're going multi-cloud," you are not mapping AWS RDS to Azure SQL Database to Cloud SQL as equivalent services. Each has architectural behaviors that will affect your SLA, your cost model, and your operational runbooks differently.

    3.2 Compute and Storage Architecture: What Changes Under the Hood

    As an on-prem DBA, you have a mental model of "database server": CPU, RAM, disks, and a file system you can inspect with ls. That model breaks in specific ways on each cloud platform. Understanding what breaks — and what remains — determines how you approach performance tuning and capacity planning.

    AWS RDS: EBS-Based Single Instance

    Standard RDS runs a single database process on an EC2-backed instance. Storage is Amazon EBS (Elastic Block Store). The key operational parameters:

  • → **gp3 storage**: General purpose SSD, baseline 3,000 IOPS up to 16,000 IOPS, throughput up to 1,000 MiB/s. You configure IOPS and throughput independently. For most OLTP workloads under 16,000 IOPS, gp3 is the right choice and costs less than provisioned IOPS.
  • → **io1/io2 storage**: Provisioned IOPS SSD, up to 256,000 IOPS for io2 Block Express. You pay per IOPS-month provisioned, so overprovisioning is expensive. Rule of thumb: if your peak IOPS demand exceeds 16,000 sustainably, move to io2.
  • → **Storage autoscaling**: You can enable autoscaling with a maximum threshold. RDS will expand storage automatically but will never shrink it. A common gotcha: a runaway job fills the disk, autoscaling kicks in, and you end up paying for 10 TB of storage you no longer need because you cannot scale down.
  • → **Instance classes**: RDS uses the `db.` prefix on EC2 instance families. The `db.r7g` family (Graviton3, memory-optimized) is the current sweet spot for most PostgreSQL workloads — a `db.r7g.4xlarge` gives you 16 vCPUs and 128 GB RAM. SQL Server workloads requiring Windows licensing have fewer Graviton options and typically run on `db.m7i` or `db.r7i`.

    • Aurora: Shared Distributed Storage

    Aurora's storage layer is the architectural departure that matters most. The Aurora storage is a distributed volume that spans six storage nodes across three AZs. Your Aurora cluster's "writer" instance does not write to local disk — it writes log records to the distributed storage layer, which handles redo application independently of the compute layer. This means:

  • → Failover to a reader promotes without storage copy, typically completing in under 30 seconds (often under 10 seconds with Aurora Global Database's managed planned failover).
  • → Read replicas share the same storage volume. There is no binary log replication lag in the traditional sense — replicas apply the cache invalidation messages from the storage layer. Replica lag on a lightly loaded Aurora cluster is typically 10–20 milliseconds, not the 100ms–several seconds you may have seen on RDS read replicas under write pressure.
  • → You cannot SSH to the storage nodes. You cannot run `iostat` on the underlying storage. You lose the low-level observability you had on-prem and gain instead Aurora-specific CloudWatch metrics like `AuroraReplicaLag` and `VolumeWriteIOPs`.

    • Azure SQL: DTUs vs. vCores

    The DTU (Database Transaction Unit) purchasing model is Azure's attempt to simplify capacity selection by bundling CPU, memory, and IO into a single unit. The problem is that DTUs are opaque — a DTU is defined relative to performance on a specific Microsoft benchmark, not to hardware specs. You cannot predict whether a workload that runs on 8 CPU cores and 64 GB RAM on-prem will fit in 500 DTUs without testing.

    The vCore model is more familiar. An 8-vCore General Purpose Azure SQL Database gives you 8 vCores and 40.8 GB memory. The Business Critical tier adds local SSD storage and an in-memory read replica included in the cost. For new deployments, prefer vCore unless the workload is small and predictable enough that DTU bundles save money.

    Azure SQL Hyperscale's page server architecture is the most aggressive storage separation on the market. Page servers store the database pages, log service writes the transaction log, and the compute replica only caches the pages it needs. A 50 TB Hyperscale database restores in minutes because restore means redirecting pointers to a new snapshot, not copying bytes.

    GCP Cloud SQL: Zonal with Optional HA

    Cloud SQL instances run on GCP Compute Engine VMs with Persistent Disk (PD) storage. A standard Cloud SQL PostgreSQL instance is single-zone; if that zone fails, your database is unavailable until it recovers. Enabling High Availability provisions a standby instance in a different zone with synchronous replication through a regional Persistent Disk. Failover is automatic but takes 60–120 seconds in practice — significantly slower than Aurora's sub-30-second failover.

    Storage on Cloud SQL scales automatically up to 64 TB for PostgreSQL, but like RDS, it does not scale down. The --storage-auto-increase-limit flag lets you cap automatic growth.

    AlloyDB uses a disaggregated storage design similar to Aurora but adds a columnar storage layer that caches frequently accessed data in a columnar format on the compute node. The practical effect: analytical queries that would require a separate OLAP system in traditional deployments can run acceptably on AlloyDB without a full scan of row-based storage.

    3.3 Networking: VPCs, Private Link, and the End of pg_hba.conf

    On-prem, you controlled database access through a combination of network topology (firewall rules, VLANs) and database-level access control (pg_hba.conf for PostgreSQL, logins/users for SQL Server). Cloud managed services extend and complicate this with additional abstraction layers. Getting networking wrong in cloud databases does not just cause connection failures — it creates security vulnerabilities and unexpected egress costs.

    AWS VPC and RDS

    Every RDS and Aurora instance lives inside a VPC subnet. You define which subnets the instance can use via a DB Subnet Group, which must span at least two AZs. Security Groups control inbound access — they are stateful firewall rules at the instance level.

    The critical operational point: RDS instances in private subnets cannot be reached from the internet by default. To connect from your workstation, you need either a bastion host, AWS Systems Manager Session Manager tunneling, or a VPN/Direct Connect. This is the right posture for production, but it surprises engineers used to opening a pgAdmin connection directly to a server.

    # Create a DB Subnet Group spanning three AZs
aws rds create-db-subnet-group \
  --db-subnet-group-name prod-postgres-subnets \
  --db-subnet-group-description "Production PostgreSQL subnets across 3 AZs" \
  --subnet-ids subnet-0a1b2c3d4e5f00001 subnet-0a1b2c3d4e5f00002 subnet-0a1b2c3d4e5f00003 \
  --region us-east-1

# Create Security Group allowing PostgreSQL access only from application subnet
aws ec2 create-security-group \
  --group-name rds-postgres-sg \
  --description "RDS PostgreSQL access from app tier" \
  --vpc-id vpc-0a1b2c3d4e5f00001

aws ec2 authorize-security-group-ingress \
  --group-id sg-0a1b2c3d4e5f12345 \
  --protocol tcp \
  --port 5432 \
  --source-group sg-0a1b2c3d4e5f67890  # app tier security group

    RDS Proxy and Connection Management

    If you have ever tuned max_connections in PostgreSQL and dealt with the "too many connections" error under load, RDS Proxy addresses exactly this problem for managed services. Lambda functions are the primary driver — each Lambda invocation can open its own database connection, and a spike to 1,000 concurrent invocations means 1,000 simultaneous connections, which will collapse a db.r6g.large (400 connection limit before performance degrades).

    RDS Proxy maintains a persistent connection pool to the database and multiplexes application connections against it. The proxy endpoint replaces your database endpoint in the application connection string. The proxy also integrates with AWS Secrets Manager and IAM for authentication, so your application never handles a plaintext database password.

    # Create RDS Proxy for Aurora PostgreSQL cluster
aws rds create-db-proxy \
  --db-proxy-name prod-aurora-proxy \
  --engine-family POSTGRESQL \
  --auth '[{
    "AuthScheme": "SECRETS",
    "SecretArn": "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/aurora/pgcreds",
    "IAMAuth": "REQUIRED"
  }]' \
  --role-arn arn:aws:iam::123456789012:role/rds-proxy-role \
  --vpc-subnet-ids subnet-0a1b2c3d4e5f00001 subnet-0a1b2c3d4e5f00002 \
  --vpc-security-group-ids sg-0a1b2c3d4e5f12345 \
  --require-tls

    Azure Private Link and Service Endpoints

    Azure SQL Database has a public endpoint by default (in older deployments). For production, you want a Private Endpoint — an Azure Private Link resource that assigns a private IP from your VNet to the Azure SQL logical server. Once a private endpoint exists, you can deny all public access via the firewall.

    The Azure SQL firewall is conceptually similar to pg_hba.conf in that it operates at the server level before the database authentication step. But unlike pg_hba.conf where you write IP ranges directly into a config file, the Azure SQL firewall rules are API-managed objects attached to the logical server.

    # Create Azure SQL Private Endpoint
az network private-endpoint create \
  --name prod-sql-private-endpoint \
  --resource-group prod-rg \
  --vnet-name prod-vnet \
  --subnet prod-db-subnet \
  --private-connection-resource-id \
    "/subscriptions/12345678-abcd-1234-abcd-123456789012/resourceGroups/prod-rg/providers/Microsoft.Sql/servers/prod-sql-server" \
  --group-id sqlServer \
  --connection-name prod-sql-pe-connection

# Disable public network access after private endpoint is confirmed active
az sql server update \
  --name prod-sql-server \
  --resource-group prod-rg \
  --enable-public-network false

    GCP VPC and Cloud SQL Authorized Networks

    Cloud SQL can use two connectivity models: Authorized Networks (an IP allowlist — the simplest but least secure approach, similar to pg_hba.conf host-based entries) and Private Service Access (a VPC peering-based model that gives Cloud SQL a private IP in your VPC).

    The recommended production approach uses Private Service Access combined with the Cloud SQL Auth Proxy, a sidecar process that handles IAM authentication and TLS tunneling without requiring direct IP allowlisting. The proxy runs alongside your application and presents a local socket that your application connects to as if it were a local PostgreSQL instance.

    # Assign private IP to Cloud SQL instance
gcloud sql instances patch prod-postgres \
  --network=projects/my-project/global/networks/prod-vpc \
  --no-assign-ip \
  --project my-project

# Run Cloud SQL Auth Proxy (typically as a sidecar in Kubernetes)
# Download: https://cloud.google.com/sql/docs/postgres/sql-proxy
./cloud-sql-proxy \
  --address 127.0.0.1 \
  --port 5432 \
  my-project:us-central1:prod-postgres &

    The Cloud SQL Auth Proxy is analogous to AWS RDS Proxy in that it handles authentication abstraction, but it does not provide connection pooling. For pooling with Cloud SQL, you run PgBouncer or pgpool-II separately — either as a sidecar or as a dedicated VM.

    3.4 Authentication and Access Control: IAM Is the New pg_hba.conf

    The phrase "IAM is the new pg_hba.conf" captures the conceptual shift without being entirely accurate. IAM controls which entities can perform management-plane operations (create instances, modify parameter groups, take snapshots). Database-level authentication — which users can log in to the database itself — remains a separate concern on every cloud platform, but IAM is increasingly integrated into that layer too.

    AWS IAM Database Authentication

    RDS and Aurora support IAM database authentication for MySQL and PostgreSQL. Instead of password-based login, an IAM entity (user or role) generates a short-lived authentication token using the AWS SDK. The token is valid for 15 minutes. On the database side, the user is created with the rds_iam authentication plugin enabled. The workflow:

    1. Application assumes an IAM role.

    2. Application calls generate_db_auth_token to get a 15-minute token.

    3. Application connects using the token as the password.

    4. RDS validates the token against IAM.

    import boto3
import psycopg2

def get_iam_auth_token(hostname, port, username, region):
    client = boto3.client('rds', region_name=region)
    token = client.generate_db_auth_token(
        DBHostname=hostname,
        Port=port,
        DBUsername=username,
        Region=region
    )
    return token

def connect_rds_iam():
    hostname = "prod-aurora.cluster-xyz.us-east-1.rds.amazonaws.com"
    port = 5432
    username = "app_service_user"
    region = "us-east-1"
    database = "appdb"
    
    token = get_iam_auth_token(hostname, port, username, region)
    
    # SSL is required for IAM auth
    conn = psycopg2.connect(
        host=hostname,
        port=port,
        database=database,
        user=username,
        password=token,
        sslmode="require"
    )
    return conn

    The database user still needs to exist as a PostgreSQL role — IAM auth does not bypass PostgreSQL's internal user system. The integration point is the authentication method: the password field carries the IAM token, and RDS validates it before PostgreSQL processes the login.

    Azure SQL: Azure Active Directory Integration

    Azure SQL supports both SQL authentication (username/password stored in the database) and Azure Active Directory authentication. AAD authentication is the recommended path for production because it enables conditional access policies, MFA for human users, and managed identity authentication for applications.

    An Azure Managed Identity — a service principal automatically managed by Azure for a VM, App Service, or container — can authenticate to Azure SQL without any credentials in the application configuration. This is functionally equivalent to the IAM role-based RDS authentication above:

    # Enable AAD admin for Azure SQL Server
az sql server ad-admin create \
  --server prod-sql-server \
  --resource-group prod-rg \
  --display-name "DBA-Team-AAD-Group" \
  --object-id "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"

# Create contained database user for a Managed Identity
# Run this SQL as the AAD admin inside the target database
    -- Create database user for App Service Managed Identity
-- Run as AAD admin inside the target database
CREATE USER [prod-app-service] FROM EXTERNAL PROVIDER;
ALTER ROLE db_datareader ADD MEMBER [prod-app-service];
ALTER ROLE db_datawriter ADD MEMBER [prod-app-service];

-- Verify AAD users
SELECT name, type_desc, authentication_type_desc
FROM sys.database_principals
WHERE type IN ('E', 'X');  -- E = external user, X = external group

    GCP Cloud IAM and Cloud SQL

    Cloud SQL integrates with GCP IAM through a feature called Cloud SQL IAM database authentication for PostgreSQL. A Cloud IAM user or service account is mapped to a PostgreSQL role using a login trigger. The user authenticates with an OAuth2 access token rather than a password. The role name in the database matches the email address of the IAM identity (truncated to 63 characters, which can cause collisions you need to test for in advance).

    The conceptual alignment: in pg_hba.conf, you would write an ident or peer line to map OS users to database roles. Cloud SQL IAM auth does the same thing, replacing OS identity with GCP IAM identity.

    Always Encrypted (Azure SQL)

    Azure SQL offers Always Encrypted — a client-side encryption mechanism where encryption and decryption happen in the application layer, never in the database engine. The database server stores ciphertext and cannot read the plaintext values, even with administrative access. This is a fundamentally different threat model than TDE (Transparent Data Encryption), which only encrypts at-rest storage and is transparent to anyone with database access.

    Always Encrypted requires application-level changes: the connection string must include the column encryption setting, and the client driver handles the cryptographic operations using keys stored in Azure Key Vault, Windows Certificate Store, or other supported keystores. The operational tradeoff: you cannot filter on encrypted columns with inequality operators (a search for salaries between $50,000 and $100,000 on an encrypted column requires decrypting all rows client-side). Always Encrypted with Randomized Encryption supports no server-side filtering at all; Deterministic Encryption supports equality comparisons only.

    3.5 High Availability and Disaster Recovery: Comparing the Failover Mechanics

    HA is where managed services diverge most sharply from on-prem setups, and where the "managed" abstraction creates the most operational surprises. You need to understand what each platform does automatically, what you still configure, and what the failure scenarios look like.

    AWS RDS Multi-AZ

    Standard RDS Multi-AZ maintains a synchronous standby replica in a second AZ. The standby is not readable — it is purely a failover target. Failover happens automatically when RDS detects instance failure, AZ outage, or network partitioning. The DNS endpoint for the instance is updated to point to the promoted standby. Expected failover time: 60–120 seconds. Applications that cache DNS or use long connection timeouts may not recover within this window without explicit reconnect logic.

    RDS Multi-AZ DB Cluster (a newer feature, not to be confused with Aurora) provides two readable standby instances in separate AZs with a stated failover time under 35 seconds. It uses semi-synchronous replication — different from Aurora's storage-level replication. Choose this over standard Multi-AZ when you need the read capacity of the standbys and can tolerate the additional cost.

    Aurora Global Database
