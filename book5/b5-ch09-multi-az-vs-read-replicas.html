Chapter 9: Multi-AZ vs Read Replicas: Architecture Decisions

Availability and read scalability are two distinct problems that cloud databases solve with two distinct mechanisms, and confusing them in production is one of the most expensive architectural mistakes a DBA can make. Multi-AZ deployments protect you from failures — infrastructure failures, AZ outages, host-level corruption — and do almost nothing for your read throughput. Read replicas scale your reads and do almost nothing for your write availability. Every experienced on-prem DBA has built some version of both concepts using streaming replication, log shipping, or Always On Availability Groups, but the managed cloud implementations have specific behaviors, limits, and gotchas that don't map cleanly to what you've done before. This chapter walks through how each mechanism works under the hood across AWS RDS, Aurora, Azure SQL, and Google Cloud SQL, how to configure them for production workloads, and how to make the architectural decisions that distinguish a well-designed cloud database tier from one that fails during the incident it was supposed to survive.

---

9.1 Multi-AZ: What It Is, What It Isn't, and How It Actually Works

When you enable Multi-AZ on an RDS instance, AWS provisions a synchronous standby replica in a different Availability Zone and manages the failover for you. The standby receives every write commit before the primary acknowledges it to your application — this is synchronous replication, not asynchronous. The standby is completely invisible to your application. It doesn't serve reads. It doesn't appear in your connection string. It exists solely to be promoted when the primary fails.

This is conceptually close to synchronous streaming replication in PostgreSQL or a synchronous commit Always On replica in SQL Server, except that AWS manages every operational aspect: the standby instance is provisioned, maintained, patched, and failed over without your involvement. The failover target is a DNS CNAME record — your RDS endpoint — which AWS flips to point to the promoted standby when a failure is detected. This DNS update typically takes 60 to 120 seconds in practice, though AWS's stated target is under 60 seconds for most failure types.

The gotcha most DBAs hit immediately: your application must respect DNS TTL values, or failover will silently fail. If your JDBC connection pool, pg connection pool, or application ORM is caching the resolved IP of your RDS endpoint beyond the TTL (which RDS sets to 5 seconds), your connections will continue pointing at the old primary address after failover. The promoted standby has a new IP. Your application sits there timing out while wondering why the database disappeared. The fix is to ensure your connection library honors TTL. For Java applications, set networkaddress.cache.ttl=0 or use RDS Proxy, which maintains persistent connections through failover transparently. For Python applications using psycopg2, verify that your connection pool isn't holding resolved IPs indefinitely.

Aurora Multi-AZ works differently from RDS Multi-AZ, and this distinction matters. Aurora doesn't replicate the database to a standby instance — it writes to a distributed storage layer across six storage nodes spread across three AZs by default. The "replica" in Aurora Multi-AZ is a reader instance that shares this storage layer rather than receiving replicated writes over the network. Failover in Aurora is therefore faster, typically 30 seconds or less, because the promoted replica doesn't need to replay a redo log to catch up — it already has access to the same storage. The tradeoff is that Aurora's storage pricing model means you're paying for six copies of your storage whether you have one reader instance or five.

For Azure SQL Database, high availability is handled differently depending on the service tier. In the Business Critical tier, Azure SQL maintains an Always On Availability Group with four replicas internally — one primary and three secondaries, one of which is readable. You don't configure this. You don't see it in your portal. It happens automatically as part of what you're paying for in the Business Critical pricing tier. In the General Purpose tier, Azure uses remote storage with a stateless compute layer; failover involves restarting the compute layer against the same storage, which is slower (60 to 120 seconds) but dramatically cheaper. Knowing which tier you're in and what its failover behavior looks like is foundational to your SLA conversations.

Google Cloud SQL High Availability provisions a standby instance in a different zone within the same region. Writes are synchronously replicated to this standby. Failover is DNS-based, similar to RDS, with typical observed times of 60 to 90 seconds. Cloud SQL also has a quirk: during maintenance windows, the instance restarts on the standby node, and your application will see a brief disconnection. This is expected behavior, not a failure.

9.2 Read Replicas: Asynchronous Replication and the Lag Problem

Read replicas exist to take SELECT load off your primary. They do this through asynchronous replication, which means there is always some lag between what the primary has committed and what the replica has visible. The size of that lag depends on write volume, replica instance size, network conditions, and whether the replica is in the same region or a different one. In a well-tuned configuration with a modestly loaded primary, replica lag sits below one second. In a write-heavy system, or when the replica is under-provisioned, lag can grow to seconds, minutes, or more.

This asynchronous nature is not a bug — it's the fundamental tradeoff that makes read replicas useful. Synchronous replication (Multi-AZ) adds latency to every write commit because the primary must wait for the standby acknowledgment before returning to the application. If your standby is in a different AZ connected via a 1ms link, you're adding approximately 1ms to every write commit. Across a cross-region link of 50ms or more, synchronous replication would be operationally untenable. Asynchronous replication lets the replica fall behind so the primary can proceed.

The architectural decision this forces on you: you cannot route all reads to replicas unless you can tolerate reading stale data. Sessions that have just committed a write and immediately query that data will read their own write from the primary, not the replica. Sessions that serve time-sensitive reporting — inventory counts, financial balances, order status — cannot use a replica with multiple seconds of lag without surfacing incorrect data to users. Sessions that serve analytics, historical reports, cached summaries, and background jobs are perfect replica candidates.

On RDS for PostgreSQL, replica lag is visible through the standard PostgreSQL view:

-- Run this on the replica to check lag
SELECT
    now() - pg_last_xact_replay_timestamp() AS replication_lag,
    pg_is_in_recovery() AS is_replica,
    pg_last_wal_receive_lsn() AS received_lsn,
    pg_last_wal_replay_lsn() AS replayed_lsn,
    pg_wal_lsn_diff(
        pg_last_wal_receive_lsn(),
        pg_last_wal_replay_lsn()
    ) AS lag_bytes;

In CloudWatch, RDS exposes ReplicaLag as a metric measured in seconds. Setting a CloudWatch alarm when ReplicaLag exceeds your application's tolerance threshold — typically somewhere between 5 and 30 seconds depending on the workload — gives you early warning before replica lag becomes a user-visible problem.

Aurora handles read replicas differently from standard RDS in a way that directly affects lag. Because all Aurora instances in a cluster share the same distributed storage layer, Aurora read replicas don't receive replicated WAL over the network. Instead, they apply log records generated by the storage layer. This results in typical replica lag of 10 to 20 milliseconds, which is an order of magnitude lower than what you'll see with RDS read replicas under load. For applications where read-your-writes consistency matters and you're considering routing certain reads to replicas, Aurora's storage-level replication architecture makes that decision less risky.

Azure SQL Hyperscale introduces a different read scaling mechanism. Hyperscale read replicas are named "named replicas" or "high availability (HA) replicas" depending on their purpose. Named replicas in Hyperscale have their own endpoint, their own independent vCore allocation, and serve reads from the same page server layer. Lag in Hyperscale reads is measured in single-digit milliseconds for most workloads because replicas read from the shared page server tier rather than receiving replicated transaction logs. The practical effect is that Hyperscale read scale-out behaves more like Aurora than like traditional log-shipping replication.

Google AlloyDB provides a similar storage-sharing architecture. AlloyDB read pools — horizontally scalable groups of reader nodes — share the same distributed storage as the primary and advertise a single read pool endpoint. Adding a node to the read pool takes minutes rather than the hours required to provision and sync a traditional read replica. For workloads that need elastic read scaling during business hours, AlloyDB's read pool architecture is significantly more operationally convenient than managing individual Cloud SQL read replicas.

9.3 Configuring Multi-AZ and Read Replicas Across Providers

The configuration mechanics differ enough across providers that it's worth walking through each one with actual commands rather than summarizing abstractly.

AWS RDS: Enabling Multi-AZ and Creating a Read Replica

You can enable Multi-AZ on an existing RDS instance with minimal downtime. AWS will provision the standby in the background, synchronize it, and then perform a brief failover to establish the synchronous replication pair. This background sync can take minutes to hours depending on database size.

# Enable Multi-AZ on an existing RDS instance
aws rds modify-db-instance \
    --db-instance-identifier prod-postgres-primary \
    --multi-az \
    --apply-immediately \
    --region us-east-1

# Create a read replica in the same region
aws rds create-db-instance-read-replica \
    --db-instance-identifier prod-postgres-replica-1 \
    --source-db-instance-identifier prod-postgres-primary \
    --db-instance-class db.r6g.2xlarge \
    --availability-zone us-east-1b \
    --publicly-accessible false \
    --region us-east-1

# Create a cross-region read replica (for DR read access or geo-distribution)
aws rds create-db-instance-read-replica \
    --db-instance-identifier prod-postgres-replica-us-west \
    --source-db-instance-identifier prod-postgres-primary \
    --db-instance-class db.r6g.2xlarge \
    --region us-west-2 \
    --source-region us-east-1

# Monitor replica lag via CloudWatch
aws cloudwatch get-metric-statistics \
    --namespace AWS/RDS \
    --metric-name ReplicaLag \
    --dimensions Name=DBInstanceIdentifier,Value=prod-postgres-replica-1 \
    --start-time $(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%S) \
    --end-time $(date -u +%Y-%m-%dT%H:%M:%S) \
    --period 60 \
    --statistics Average \
    --region us-east-1

Aurora: Cluster Configuration with Terraform

Aurora exposes Multi-AZ through the cluster layer rather than individual instances. The following Terraform block creates an Aurora PostgreSQL cluster with one writer and two readers, where one reader acts as the failover target and the other serves read traffic:

resource "aws_rds_cluster" "production" {
  cluster_identifier      = "prod-aurora-cluster"
  engine                  = "aurora-postgresql"
  engine_version          = "15.4"
  database_name           = "appdb"
  master_username         = "dbadmin"
  manage_master_user_password = true

  db_subnet_group_name    = aws_db_subnet_group.production.name
  vpc_security_group_ids  = [aws_security_group.rds.id]

  backup_retention_period = 7
  preferred_backup_window = "03:00-04:00"

  storage_encrypted       = true
  kms_key_id              = aws_kms_key.rds.arn

  enabled_cloudwatch_logs_exports = ["postgresql"]

  deletion_protection     = true
  skip_final_snapshot     = false
  final_snapshot_identifier = "prod-aurora-final-snapshot"
}

resource "aws_rds_cluster_instance" "writer" {
  identifier         = "prod-aurora-writer"
  cluster_identifier = aws_rds_cluster.production.id
  instance_class     = "db.r6g.4xlarge"
  engine             = aws_rds_cluster.production.engine
  engine_version     = aws_rds_cluster.production.engine_version

  # Promotion tier 0 means this instance is preferred for failover promotion
  promotion_tier     = 0
  availability_zone  = "us-east-1a"
}

resource "aws_rds_cluster_instance" "reader_failover" {
  identifier         = "prod-aurora-reader-failover"
  cluster_identifier = aws_rds_cluster.production.id
  instance_class     = "db.r6g.4xlarge"
  engine             = aws_rds_cluster.production.engine
  engine_version     = aws_rds_cluster.production.engine_version

  # Same instance class as writer ensures promotion doesn't introduce a
  # performance cliff when this reader becomes the new writer
  promotion_tier     = 1
  availability_zone  = "us-east-1b"
}

resource "aws_rds_cluster_instance" "reader_read_traffic" {
  identifier         = "prod-aurora-reader-reads"
  cluster_identifier = aws_rds_cluster.production.id
  instance_class     = "db.r6g.2xlarge"
  engine             = aws_rds_cluster.production.engine
  engine_version     = aws_rds_cluster.production.engine_version

  # Lower promotion tier — this instance serves reads but won't be promoted
  # to primary unless all higher-priority readers are unavailable
  promotion_tier     = 15
  availability_zone  = "us-east-1c"
}

The promotion_tier setting on Aurora cluster instances is something many DBAs overlook. If you have a large reader instance serving analytics workloads and a smaller reader instance designated for failover, you must set the analytics reader to a higher promotion tier number (lower priority). Otherwise Aurora may promote your undersized analytics reader to primary during a failover, and you've degraded your write path.

Azure SQL: Geo-Replication and Readable Secondaries

Azure SQL Database's equivalent of read replicas is Active Geo-Replication, which creates readable secondaries in other regions. Unlike RDS read replicas which are independent instances, Azure geo-secondary databases must be in an Elastic Pool or have their own service tier provisioned, and they're intended primarily for disaster recovery rather than everyday read offloading.

# Create an Azure SQL Database with Business Critical tier (includes HA replicas)
az sql db create \
    --resource-group prod-rg \
    --server prod-sql-server \
    --name appdb \
    --service-objective BC_Gen5_8 \
    --backup-storage-redundancy Zone \
    --zone-redundant true

# Add a geo-secondary for DR (readable secondary in another region)
az sql db replica create \
    --resource-group prod-rg \
    --server prod-sql-server \
    --name appdb \
    --partner-resource-group prod-dr-rg \
    --partner-server prod-sql-server-dr \
    --service-objective BC_Gen5_4 \
    --secondary-type Geo

# Check replication status and lag
az sql db replica list-links \
    --resource-group prod-rg \
    --server prod-sql-server \
    --name appdb \
    --output table

# Query replication lag from within Azure SQL (run on secondary)
# Azure SQL exposes this through sys.dm_database_replica_states
-- Check geo-replication lag on Azure SQL secondary
-- Run this against the secondary database
SELECT
    r.partner_server,
    r.role_desc,
    r.replication_state_desc,
    r.secondary_lag_seconds,
    r.last_commit_time,
    DATEDIFF(SECOND, r.last_commit_time, GETUTCDATE()) AS lag_seconds_computed
FROM sys.dm_geo_replication_link_status r;

-- On Business Critical tier, check internal HA replica state
SELECT
    replica_id,
    role_desc,
    operational_state_desc,
    recovery_health_desc,
    synchronization_health_desc,
    last_commit_time
FROM sys.dm_hadr_availability_replica_states
WHERE is_local = 0;

9.4 Failover Behavior: What Actually Happens During the Event

Understanding what happens during a failover is more valuable than understanding how to configure one. Production incidents reveal the gaps in architectural assumptions, and the best time to close those gaps is before the incident, not during it.

When RDS Multi-AZ failover is triggered — by a host failure, storage failure, AZ outage, or a manual failover you initiate for testing — the sequence of events is:

1. RDS detects the primary instance is unreachable (detection typically takes 20 to 40 seconds for host failures, faster for storage failures which are detected at the volume level).

2. The standby is promoted to primary.

3. The RDS endpoint CNAME record is updated to point to the new primary's IP.

4. DNS propagation occurs — clients respecting the 5-second TTL will resolve the new IP within 5 to 10 seconds of the CNAME update.

5. New connections succeed; existing connections must be re-established.

The gap that trips production applications: existing database connections are dropped during failover. Connection pools that don't handle OperationalError exceptions and reconnect will sit with broken connections until their health check cycles. A well-configured connection pool — whether that's HikariCP in Java, pgbouncer, or RDS Proxy — will test connections before handing them to the application and replace broken connections automatically.

RDS Proxy addresses the failover connection problem more cleanly than any connection pool tuning. RDS Proxy maintains a connection pool between itself and the RDS instance, and it presents a stable endpoint to applications. During failover, RDS Proxy handles the reconnection to the promoted standby internally. Applications see their connections paused briefly (the queries in flight will be retried automatically in some configurations, or will receive an error and need application-level retry) but don't need to re-resolve DNS. For serverless or bursty workloads that already benefit from connection pooling, RDS Proxy is the correct insertion point.

import boto3
import psycopg2
import time
from botocore.exceptions import ClientError

def initiate_and_monitor_failover(db_instance_id: str, region: str):
    """
    Initiate a manual Multi-AZ failover and monitor until completion.
    Use this in maintenance testing, not production incidents.
    """
    rds_client = boto3.client('rds', region_name=region)
    cw_client = boto3.client('cloudwatch', region_name=region)

    print(f"Initiating failover for {db_instance_id}")
    start_time = time.time()

    try:
        rds_client.reboot_db_instance(
            DBInstanceIdentifier=db_instance_id,
            ForceFailover=True
        )
    except ClientError as e:
        print(f"Failover initiation failed: {e}")
        raise

    # Poll until the instance is available again
    waiter = rds_client.get_waiter('db_instance_available')
    waiter.config.delay = 10
    waiter.config.max_attempts = 30

    print("Waiting for instance to become available after failover...")
    waiter.wait(DBInstanceIdentifiers=[db_instance_id])

    elapsed = time.time() - start_time
    print(f"Failover completed in {elapsed:.1f} seconds")

    # Verify we can connect to the new primary
    instance_info = rds_client.describe_db_instances(
        DBInstanceIdentifier=db_instance_id
    )['DBInstances'][0]

    endpoint = instance_info['Endpoint']['Address']
    new_az = instance_info['AvailabilityZone']
    print(f"New primary endpoint: {endpoint}")
    print(f"New primary AZ: {new_az}")

    return elapsed, endpoint


def test_connection_with_retry(host: str, port: int, dbname: str,
                                user: str, password: str,
                                max_retries: int = 10,
                                retry_interval: float = 5.0):
    """
    Test database connectivity with retry logic appropriate for post-failover
    reconnection. This pattern belongs in your application's connection layer.
    """
    for attempt in range(max_retries):
        try:
            conn = psycopg2.connect(
                host=host,
                port=port,
                dbname=dbname,
                user=user,
                password=password,
                connect_timeout=5,
                options="-c statement_timeout=10000"
            )
            # Confirm we're on the primary
            with conn.cursor() as cur:
                cur.execute("SELECT pg_is_in_recovery(), inet_server_addr()")
                is_replica, server_addr = cur.fetchone()
                if is_replica:
                    print(f"Warning: Connected to replica at {server_addr}")
                else:
                    print(f"Connected to primary at {server_addr}")
            conn.close()
            return True
        except psycopg2.OperationalError as e:
            print(f"Attempt {attempt + 1}/{max_retries} failed: {e}")
            if attempt < max_retries - 1:
                time.sleep(retry_interval)

    return False

For Aurora, the failover process deserves a different mental model. Aurora maintains a cluster endpoint (the writer endpoint) and a reader endpoint that automatically load-balances across all healthy reader instances. When the writer fails, Aurora promotes one of the readers based on promotion tier order. Because readers share the storage layer, there is no redo log replay phase — promotion is essentially a metadata operation. The practical result is that Aurora writer failover typically completes in 30 seconds or less, while standard RDS Multi-AZ typically takes 60 to 120 seconds.

Azure SQL Business Critical failover is fast — typically under 30 seconds — because the Always On AG secondaries are already synchronized and one of them is always designated as the failover target. Azure SQL failover is transparent to applications using the server FQDN as the connection endpoint; the internal AG mechanics are not exposed. The main operational concern with Azure SQL failover is connection string validation: Azure SQL requires that your connection string targets the server name, not a specific IP, and that your firewall rules or private endpoint configuration is correct on both sides.
