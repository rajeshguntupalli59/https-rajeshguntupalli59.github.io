Chapter 30: Handling Embedding Model Version Changes

Embedding models are not static infrastructure. OpenAI retires text-embedding-ada-002 in favor of text-embedding-3-small and text-embedding-3-large. Cohere ships embed-v3 with a different vector dimension than embed-v2. A sentence-transformers model you fine-tuned six months ago gets replaced by a better checkpoint. Every time the model changes, every vector in your database becomes potentially incompatible with every new query vector — and the search results degrade silently unless you have a plan. This chapter covers what actually breaks during an embedding model transition, how to detect degradation before your users do, and the concrete migration strategies that let you move from one model to another without taking your search system offline.

---

Why Model Changes Break Vector Search

The core problem is geometric. Every embedding model defines its own latent space — a coordinate system where semantically similar concepts cluster together. When you switch models, you are not changing the data; you are changing the entire coordinate system. A document vector produced by text-embedding-ada-002 (1536 dimensions) and a query vector produced by text-embedding-3-small (also 1536 dimensions, but a completely different latent space) are meaningless when compared with cosine similarity. The numbers look plausible. The system returns results. But those results are essentially random relative to what the user actually asked.

This failure mode is the worst kind in database systems: silent and gradual. There is no error thrown. There is no query timeout. The cosine distance computation succeeds perfectly — it just computes the distance between two points in incompatible coordinate systems. If you have monitoring on query latency but not on result quality, this class of failure will go undetected until users start complaining or A/B test metrics collapse.

From a database perspective, think of it like changing your collation rules on a text column after the data is already indexed. The index physically exists and returns results, but the sort order is now wrong relative to the application's expectations. The difference is that collation failures at least sometimes produce obviously garbled output. Embedding model mismatches produce results that look completely reasonable.

The specific failure scenarios are:

Dimension mismatch: The new model produces vectors of a different length than the old model. pgvector will throw a hard error if you attempt to insert a 1024-dimensional vector into a vector(1536) column. Pinecone will reject the upsert with an HTTP 400. This is the *easy* failure to catch.

Latent space drift: Same vector dimension, completely different coordinate space. text-embedding-ada-002 and text-embedding-3-small both produce 1536-dimensional vectors, but they are not comparable. This is the silent failure that requires active quality monitoring to detect.

Normalization differences: Some models produce unit-normalized vectors by default; others do not. If your index uses dot product similarity (which many ANN indexes optimize for), switching from a normalized to an unnormalized model changes the effective similarity function without changing your index configuration.

Tokenization boundary shifts: A new model version might chunk text differently at the tokenizer level, meaning that the same 512-token input produces a vector that reflects subtly different semantic content. This is the hardest to detect because the degradation is partial rather than total.

Detecting Degradation Before and After Migration

Before you migrate a single vector, establish a baseline you can measure against. Detection requires two things: a ground-truth evaluation set and a continuous monitoring pipeline.

Building an evaluation set

A ground truth evaluation set is a collection of queries where you already know which documents should rank in the top K results. For a customer support system, this might be 200 sample support tickets paired with the knowledge base articles that human agents actually used to resolve them. For a code search system, it might be 100 natural language queries paired with the functions they should retrieve.

The metric to track is Recall@K — specifically, what percentage of the known-relevant documents appear in the top K results. For most production systems, Recall@10 between 0.75 and 0.90 indicates healthy retrieval. A drop below 0.60 is a signal that something has gone wrong with your retrieval layer.

def embed_queries_openai(queries: list[str], model: str) -> np.ndarray:
    response = client.embeddings.create(input=queries, model=model)
    return np.array([r.embedding for r in response.data])

def embed_queries_st(queries: list[str], model_name: str) -> np.ndarray:
    model = SentenceTransformer(model_name)
    return model.encode(queries, normalize_embeddings=True)

def recall_at_k(
    query_embeddings: np.ndarray,
    corpus_embeddings: np.ndarray,
    doc_id_to_index: dict,
    eval_set: list,
    k: int = 10
) -> float:
    """Compute Recall@K across the evaluation set."""
    index_to_doc_id = {v: k for k, v in doc_id_to_index.items()}
    sims = cosine_similarity(query_embeddings, corpus_embeddings)
    
    recall_scores = []
    for i, (query, relevant_ids) in enumerate(eval_set):
        top_k_indices = np.argsort(sims[i])[::-1][:k]
        top_k_doc_ids = {index_to_doc_id[idx] for idx in top_k_indices}
        relevant_set = set(relevant_ids)
        
        hits = len(relevant_set & top_k_doc_ids)
        recall_scores.append(hits / len(relevant_set))
    
    return float(np.mean(recall_scores))

def compare_models(eval_set: list, k: int = 10):
    queries = [q for q, _ in eval_set]
    
    models_to_test = [

            "embed_fn": lambda q: embed_queries_st(q, "BAAI/bge-large-en-v1.5"),
        },
    ]
    
    print(f"{'Model':<20} {'Recall@' + str(k):<12} {'Latency (ms)':<15}")
    print("-" * 48)

        import time
        start = time.perf_counter()
        query_embeddings = model_cfg["embed_fn"](queries)
        elapsed_ms = (time.perf_counter() - start) * 1000

                f"{model_cfg['label']:<20} {score:<12.4f} {elapsed_ms:<15.1f}"
            )

            print(
                f"{model_cfg['label']:<20} {'N/A (load corpus)':<12} {elapsed_ms:<15.1f}"
            )

if __name__ == "__main__":
    compare_models(EVAL_SET, k=10)

Continuous monitoring in production

Run the evaluation set against your live system on a scheduled basis — every hour during a migration, daily in steady state. Store the scores in a time-series table alongside deployment metadata:

-- PostgreSQL / pgvector
-- Schema for tracking embedding model quality over time

CREATE TABLE embedding_model_registry (
    model_id        SERIAL PRIMARY KEY,
    model_name      TEXT NOT NULL,           -- 'text-embedding-3-small'
    model_version   TEXT NOT NULL,           -- 'v1', 'v2', or commit hash
    vector_dim      INTEGER NOT NULL,
    provider        TEXT NOT NULL,           -- 'openai', 'cohere', 'local'
    deployed_at     TIMESTAMPTZ,
    deprecated_at   TIMESTAMPTZ,
    is_active       BOOLEAN NOT NULL DEFAULT FALSE,
    metadata        JSONB DEFAULT '{}'::jsonb,
    UNIQUE (model_name, model_version)
);

CREATE TABLE retrieval_quality_log (
    log_id          BIGSERIAL PRIMARY KEY,
    recorded_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    model_id        INTEGER REFERENCES embedding_model_registry(model_id),
    eval_set_name   TEXT NOT NULL,
    k               INTEGER NOT NULL,
    recall_at_k     NUMERIC(5,4) NOT NULL,
    ndcg_at_k       NUMERIC(5,4),
    eval_query_count INTEGER NOT NULL,
    notes           TEXT
);

CREATE INDEX idx_rql_recorded_at ON retrieval_quality_log (recorded_at DESC);
CREATE INDEX idx_rql_model_id    ON retrieval_quality_log (model_id, recorded_at DESC);

-- Query: detect recall drops over the last 24 hours
WITH recent AS (
    SELECT
        r.model_id,
        reg.model_name,
        r.recorded_at,
        r.recall_at_k,
        AVG(r.recall_at_k) OVER (
            PARTITION BY r.model_id
            ORDER BY r.recorded_at
            ROWS BETWEEN 23 PRECEDING AND CURRENT ROW
        ) AS rolling_24h_avg
    FROM retrieval_quality_log r
    JOIN embedding_model_registry reg USING (model_id)
    WHERE r.recorded_at > NOW() - INTERVAL '48 hours'
      AND r.eval_set_name = 'production_eval_v1'
      AND r.k = 10
)
SELECT
    model_name,
    recorded_at,
    recall_at_k,
    rolling_24h_avg,
    (recall_at_k - rolling_24h_avg) AS delta
FROM recent
WHERE ABS(recall_at_k - rolling_24h_avg) > 0.05  -- alert if >5 point swing
ORDER BY recorded_at DESC;

Migration Strategies: Zero-Downtime Approaches

There are three practical migration strategies. Which one you choose depends on your corpus size, your SLA requirements, and whether you have the infrastructure to run dual indexes simultaneously.

Strategy 1: Big Bang (Full Re-index)

Re-embed the entire corpus with the new model, load it into a new collection or table, then atomically switch traffic. Simple to reason about, but requires 2x storage during the transition and a cutover window where you must freeze writes to the old index while completing the final sync.

Suitable for corpora under ~5 million documents where the re-embedding job completes in under 4 hours and your application can tolerate brief read-only mode.

Strategy 2: Dual-Model Shadow Period

Run both the old model and the new model simultaneously. New documents are written to both indexes. Queries are served from the old index until you validate the new one. Then you drain the old index by re-embedding documents in batches during off-peak hours.

This is the pattern most production teams use because it decouples the "new model is ready" decision from the "all data is migrated" state.

# Python
# Dual-model write path using pgvector via psycopg2 and asyncpg
# Demonstrates shadow-writing to both old and new embedding columns

import asyncio
import asyncpg
from openai import AsyncOpenAI

client = AsyncOpenAI()

CREATE_SCHEMA = """
CREATE TABLE IF NOT EXISTS documents (
    id              TEXT PRIMARY KEY,
    content         TEXT NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    -- Old model column (ada-002, 1536 dims)
    embedding_ada002        vector(1536),
    embedding_ada002_at     TIMESTAMPTZ,
    
    -- New model column (3-small, 1536 dims — different latent space)
    embedding_3small        vector(1536),
    embedding_3small_at     TIMESTAMPTZ,
    
    -- Which model is authoritative for reads
    active_model    TEXT NOT NULL DEFAULT 'ada-002'
        CHECK (active_model IN ('ada-002', '3-small'))
);

-- Indexes for both models
CREATE INDEX IF NOT EXISTS idx_docs_ada002
    ON documents USING hnsw (embedding_ada002 vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

CREATE INDEX IF NOT EXISTS idx_docs_3small
    ON documents USING hnsw (embedding_3small vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
"""

async def embed_text(text: str, model: str) -> list[float]:
    response = await client.embeddings.create(input=text, model=model)
    return response.data[0].embedding

async def upsert_document_dual_write(
    conn: asyncpg.Connection,
    doc_id: str,
    content: str,
    shadow_write_new_model: bool = True
):
    """
    Write to old model always.
    Write to new model if shadow_write_new_model=True.
    Enables gradual validation before flipping active_model.
    """
    # Embed with old model (always)
    ada_embedding = await embed_text(content, "text-embedding-ada-002")
    
    if shadow_write_new_model:
        # Embed with new model simultaneously
        small_embedding = await embed_text(content, "text-embedding-3-small")
        
        await conn.execute(
            """
            INSERT INTO documents (
                id, content,
                embedding_ada002, embedding_ada002_at,
                embedding_3small, embedding_3small_at
            )
            VALUES ($1, $2, $3::vector, NOW(), $4::vector, NOW())
            ON CONFLICT (id) DO UPDATE SET
                content              = EXCLUDED.content,
                embedding_ada002     = EXCLUDED.embedding_ada002,
                embedding_ada002_at  = EXCLUDED.embedding_ada002_at,
                embedding_3small     = EXCLUDED.embedding_3small,
                embedding_3small_at  = EXCLUDED.embedding_3small_at
            """,
            doc_id, content,
            str(ada_embedding),
            str(small_embedding),
        )
    else:
        await conn.execute(
            """
            INSERT INTO documents (id, content, embedding_ada002, embedding_ada002_at)
            VALUES ($1, $2, $3::vector, NOW())
            ON CONFLICT (id) DO UPDATE SET
                content             = EXCLUDED.content,
                embedding_ada002    = EXCLUDED.embedding_ada002,
                embedding_ada002_at = EXCLUDED.embedding_ada002_at
            """,
            doc_id, content, str(ada_embedding),
        )

async def search_documents(
    conn: asyncpg.Connection,
    query: str,
    model_override: str | None = None,
    top_k: int = 10
) -> list[dict]:
    """
    Search using whichever model is currently active,
    or a specific model if overriding for A/B testing.
    """
    # Determine active model (read from a config row or feature flag)
    active_model = model_override or await get_active_model(conn)
    
    if active_model == "ada-002":
        embedding = await embed_text(query, "text-embedding-ada-002")
        col = "embedding_ada002"
    else:
        embedding = await embed_text(query, "text-embedding-3-small")
        col = "embedding_3small"
    
    rows = await conn.fetch(
        f"""
        SELECT
            id,
            content,
            active_model,
            1 - ({col} <=> $1::vector) AS cosine_similarity
        FROM documents
        WHERE {col} IS NOT NULL
        ORDER BY {col} <=> $1::vector
        LIMIT $2
        """,
        str(embedding), top_k
    )
    return [dict(r) for r in rows]

async def get_active_model(conn: asyncpg.Connection) -> str:
    # In practice, store this in a config table or read from a feature flag service
    row = await conn.fetchrow(
        "SELECT value FROM app_config WHERE key = 'active_embedding_model'"
    )
    return row["value"] if row else "ada-002"

async def flip_active_model(conn: asyncpg.Connection, new_model: str):
    """
    Atomic model flip — called only after backfill is complete
    and quality metrics have been validated.
    """
    # Verify backfill completeness before flipping
    gap = await conn.fetchval(
        f"""
        SELECT COUNT(*) FROM documents
        WHERE embedding_3small IS NULL
          AND active_model = 'ada-002'
        """
    )
    if gap > 0:
        raise RuntimeError(
            f"Cannot flip: {gap} documents still missing new embeddings"
        )
    
    await conn.execute(
        "UPDATE app_config SET value = $1 WHERE key = 'active_embedding_model'",
        new_model
    )
    # Also update all rows to reflect new authoritative model
    await conn.execute("UPDATE documents SET active_model = $1", new_model)
    print(f"Active model flipped to: {new_model}")
