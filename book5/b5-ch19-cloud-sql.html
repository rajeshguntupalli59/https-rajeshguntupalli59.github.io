Chapter 19: Cloud SQL: PostgreSQL and SQL Server on GCP

Google Cloud SQL is the workhorse managed database service on GCP — not the most glamorous product in Google's portfolio, but the one that most production teams actually run day-to-day. This chapter covers two of its three supported engines: PostgreSQL and SQL Server. If you've spent years tuning postgresql.conf, managing WAL archiving, and nursing SQL Server AlwaysOn through network partitions, Cloud SQL will feel both familiar and occasionally frustrating. Google handles the infrastructure, but the tradeoffs are real: you gain automated patching, HA failover, and integrated backups while giving up superuser access, direct OS-level tuning, and certain PostgreSQL extensions you may have relied on. This chapter maps your existing DBA knowledge onto Cloud SQL's architecture, walks through production configuration for both engines, covers HA and replication patterns, and shows you how IAM authentication, Private Service Connect, and Cloud SQL Auth Proxy replace the access-control mechanisms you've used on-premises.

---

19.1 Cloud SQL Architecture: What GCP Manages and What You Still Own

Cloud SQL instances run on managed VMs inside Google's infrastructure. Unlike AWS RDS, which abstracts the underlying VM almost completely from day one, Cloud SQL has historically been more transparent — you can see machine types (shared-core, standard, high-memory), attached SSD storage, and even the regional zone placement directly in the console and API.

What Google manages:

  • → OS patching and kernel updates (maintenance windows apply)
  • → Storage encryption at rest (Google-managed or customer-managed CMEK)
  • → Automated backups and point-in-time recovery (PITR)
  • → High availability via synchronous regional replication
  • → Minor and major version upgrades (in-place for PostgreSQL, with caveats)
  • → The physical failover between primary and standby

    • What you still own:

  • → Database schema and query performance
  • → Flag configuration (the Cloud SQL equivalent of `postgresql.conf` and SQL Server sp_configure)
  • → User management and privilege grants
  • → Connection pooling (Cloud SQL does not include a built-in proxy equivalent to PgBouncer — the Auth Proxy handles TLS tunneling, not pooling)
  • → Logical replication setup for outbound replication to external consumers
  • → Extension management within the allowed extension list

    • The superuser question: On Cloud SQL for PostgreSQL, you never get a true PostgreSQL superuser. The default administrative user (postgres) is a highly privileged user, but certain system-level operations are blocked. You cannot run pg_read_file(), cannot directly manipulate pg_hba.conf, and cannot install arbitrary extensions from source. The allowed extension list is maintained by Google and currently includes roughly 90 extensions for PostgreSQL 14 and 15 — including pgvector, pg_stat_statements, postgis, pg_partman, and pglogical, but not plsh, multicorn, or any extension requiring loading of an external .so not included in the base image.

    For SQL Server, Cloud SQL runs SQL Server 2017, 2019, and 2022 across Enterprise, Standard, and Express editions. You get sysadmin-equivalent access via the sqlserver user, but you cannot execute xp_cmdshell (disabled and blocked at the API level), cannot directly manage SQL Server Agent jobs through T-SQL in the traditional sense (the agent is present but surfaced through GCP APIs), and cannot access the underlying file system.

    Instance sizing follows Google Compute Engine machine families:

    | Tier | vCPUs | RAM | Max Storage | Max Connections (PG) |

    |---|---|---|---|---|

    | db-f1-micro | 1 (shared) | 0.6 GB | 3 TB | ~25 |

    | db-n1-standard-4 | 4 | 15 GB | 64 TB | ~500 |

    | db-n1-highmem-8 | 8 | 52 GB | 64 TB | ~1000 |

    | db-custom-16-65536 | 16 | 64 GB | 64 TB | ~2000 |

    | db-custom-96-393216 | 96 | 384 GB | 64 TB | ~12000 |

    The connection limits shown are practical guideline values — Cloud SQL does not hard-cap connections at these numbers by default, but PostgreSQL's max_connections defaults to 100 for db-f1-micro and scales with memory. For anything above 500 connections, you should plan your connection pooling architecture before deployment, not after.

    19.2 Provisioning Cloud SQL Instances: PostgreSQL and SQL Server

    Provisioning with gcloud

    The fastest path to a production-ready Cloud SQL instance uses gcloud sql instances create. Every flag here maps to something you configured manually on-premises.

    # Create a Cloud SQL for PostgreSQL 15 instance
# -- equivalent to provisioning a new PostgreSQL server with WAL archiving,
#    streaming replication standby, and daily base backups preconfigured

gcloud sql instances create prod-pg-01 \
  --database-version=POSTGRES_15 \
  --tier=db-custom-8-32768 \
  --region=us-central1 \
  --availability-type=REGIONAL \
  --storage-type=SSD \
  --storage-size=500GB \
  --storage-auto-increase \
  --backup-start-time=02:00 \
  --enable-bin-log \
  --retained-backups-count=14 \
  --retained-transaction-log-days=7 \
  --maintenance-window-day=SUN \
  --maintenance-window-hour=4 \
  --deletion-protection \
  --no-assign-ip \
  --network=projects/my-project/global/networks/prod-vpc

# Create the default database and application user
gcloud sql databases create appdb --instance=prod-pg-01

gcloud sql users create appuser \
  --instance=prod-pg-01 \
  --password=CHANGE_ME_USE_SECRET_MANAGER

    A few things worth noting here:

  • → `--availability-type=REGIONAL` enables synchronous HA replication to a standby in a second zone within the same region. This is not cross-region. The standby is not readable — it exists only for failover. Cross-region reads require read replicas configured separately.
  • → `--enable-bin-log` on PostgreSQL enables WAL archiving for PITR. Without it, you only get daily backup restoration. For production, this flag is non-negotiable.
  • → `--no-assign-ip` with `--network` places the instance on a private IP in your VPC. Always do this. Public IP Cloud SQL instances exist, but deploying them in production means relying entirely on authorized networks IP allowlisting, which is a fragile access control model.

    • Cloud SQL for SQL Server:

    # Create Cloud SQL for SQL Server 2022 Standard Edition
gcloud sql instances create prod-mssql-01 \
  --database-version=SQLSERVER_2022_STANDARD \
  --tier=db-custom-8-32768 \
  --region=us-east1 \
  --availability-type=REGIONAL \
  --storage-type=SSD \
  --storage-size=1TB \
  --storage-auto-increase \
  --backup-start-time=03:00 \
  --retained-backups-count=14 \
  --maintenance-window-day=SAT \
  --maintenance-window-hour=2 \
  --deletion-protection \
  --no-assign-ip \
  --network=projects/my-project/global/networks/prod-vpc \
  --root-password=CHANGE_ME_USE_SECRET_MANAGER

# Create a database
gcloud sql databases create SalesDB --instance=prod-mssql-01

# Create a SQL login
gcloud sql users create sqladmin \
  --instance=prod-mssql-01 \
  --password=CHANGE_ME_USE_SECRET_MANAGER

    Provisioning with Terraform is the right pattern for production environments where you need reproducible, version-controlled infrastructure:

    # terraform/modules/cloud_sql_postgres/main.tf

resource "google_sql_database_instance" "prod_pg" {
  name             = "prod-pg-01"
  database_version = "POSTGRES_15"
  region           = "us-central1"
  deletion_protection = true

  settings {
    tier              = "db-custom-8-32768"
    availability_type = "REGIONAL"
    disk_type         = "PD_SSD"
    disk_size         = 500
    disk_autoresize   = true

    backup_configuration {
      enabled                        = true
      start_time                     = "02:00"
      point_in_time_recovery_enabled = true
      transaction_log_retention_days = 7
      backup_retention_settings {
        retained_backups = 14
        retention_unit   = "COUNT"
      }
    }

    maintenance_window {
      day          = 7   # Sunday
      hour         = 4
      update_track = "stable"
    }

    ip_configuration {
      ipv4_enabled    = false
      private_network = "projects/${var.project_id}/global/networks/prod-vpc"
    }

    database_flags {
      name  = "max_connections"
      value = "500"
    }

    database_flags {
      name  = "log_min_duration_statement"
      value = "1000"  # Log queries slower than 1 second
    }

    database_flags {
      name  = "pg_stat_statements.track"
      value = "all"
    }

    insights_config {
      query_insights_enabled  = true
      query_string_length     = 4096
      record_application_tags = true
      record_client_address   = true
    }
  }
}

resource "google_sql_database" "appdb" {
  name     = "appdb"
  instance = google_sql_database_instance.prod_pg.name
}

resource "google_sql_user" "appuser" {
  name     = "appuser"
  instance = google_sql_database_instance.prod_pg.name
  password = var.db_password  # Sourced from Secret Manager via data source
}

    The insights_config block enables Query Insights, which is Cloud SQL's integrated slow query analysis tool — think of it as a managed pg_stat_statements dashboard built into the GCP console. It captures query plans, wait events, and client attribution. For PostgreSQL shops accustomed to pg_stat_statements + pgBadger, this is a useful first-line tool, though it doesn't replace dedicated APM for complex workloads.

    19.3 Database Flags: Configuring PostgreSQL and SQL Server Parameters

    What you previously controlled through postgresql.conf and pg_hba.conf (PostgreSQL) or sp_configure and SQL Server Configuration Manager (SQL Server) is surfaced in Cloud SQL as database flags. Most parameters are settable; some are fixed by Google and cannot be changed regardless of your flag configuration.

    Fixed parameters in Cloud SQL for PostgreSQL (you cannot override these):

  • → `wal_level` — fixed at `logical` (actually beneficial for logical replication)
  • → `archive_mode` — managed by Google for PITR
  • → `ssl` — always `on`
  • → `log_destination` — managed by Cloud Logging
  • → `shared_preload_libraries` — you can add to this list from approved extensions, but certain libraries are always present

    • Configurable flags (selected important ones):

    # Apply multiple performance flags to a PostgreSQL instance
# Note: Some flags require instance restart; Cloud SQL will warn you

gcloud sql instances patch prod-pg-01 \
  --database-flags=\
max_connections=500,\
shared_buffers=8192MB,\
work_mem=64MB,\
maintenance_work_mem=2GB,\
effective_cache_size=24576MB,\
checkpoint_completion_target=0.9,\
wal_buffers=64MB,\
random_page_cost=1.1,\
log_min_duration_statement=1000,\
log_checkpoints=on,\
log_connections=on,\
log_lock_waits=on,\
autovacuum_vacuum_scale_factor=0.02,\
autovacuum_analyze_scale_factor=0.01,\
pg_stat_statements.track=all,\
track_io_timing=on

    An important gotcha: Cloud SQL does not let you set shared_buffers directly by name on some tiers — the flag name is shared_buffers and the value must be expressed in kB or MB, not the shorthand notation PostgreSQL accepts directly. Always check the [Cloud SQL flags documentation](https://cloud.google.com/sql/docs/postgres/flags) for your specific database version because accepted value formats differ from what you'd type in postgresql.conf.

    For work_mem: be conservative. work_mem=64MB sounds modest but with 200 concurrent connections each potentially running a sort or hash join, your effective RAM usage for working memory alone is 200 × 64 MB × (number of sort operations per query) = easily 25+ GB on a busy OLTP instance. The number in postgresql.conf is per-sort-node, not per-connection — this is a footgun that causes OOM kills even on generously sized instances.

    SQL Server flags are surfaced similarly but map to sp_configure options:

    # Configure SQL Server instance flags
gcloud sql instances patch prod-mssql-01 \
  --database-flags=\
max\ degree\ of\ parallelism=4,\
cost\ threshold\ for\ parallelism=50,\
optimize\ for\ ad\ hoc\ workloads=on,\
remote\ query\ timeout\ \(s\)=600,\
fill\ factor\ \(\%\)=80

    What you cannot do with SQL Server on Cloud SQL:

  • → `xp_cmdshell` is permanently disabled. There is no workaround.
  • → Linked servers are supported in limited form (to other Cloud SQL instances and Cloud Spanner via a preview connector), but not to arbitrary external servers.
  • → SQL Server Integration Services (SSIS) is not available. For ETL, you're expected to use Dataflow or Cloud Data Fusion.
  • → SQL Server Reporting Services (SSRS) is not available.
  • → SQL Server Analysis Services (SSAS) is not available.
  • → Database mirroring is replaced by the HA standby (which is automatic and not configurable as a traditional mirroring endpoint).

    • If your current SQL Server environment depends heavily on SSIS packages, linked servers to Oracle, or SSRS-based reporting, Cloud SQL is not the right target without significant architectural changes.

    19.4 High Availability, Replication, and Failover

    Regional HA in Cloud SQL works through synchronous replication to a standby instance in a different zone within the same region. This is conceptually similar to PostgreSQL synchronous streaming replication (synchronous_commit = remote_apply) but implemented entirely by Google's infrastructure layer. You don't configure replication slots, recovery.conf, or standby postgresql.conf. When you set --availability-type=REGIONAL, the standby materializes automatically.

    Failover characteristics:

  • → **Automatic failover** triggers when the primary becomes unavailable, typically completing in 60–120 seconds. This is not near-instantaneous like some marketing materials imply.
  • → The standby is in a **different zone** but the **same region**. A full regional outage takes down both primary and standby.
  • → The standby is **not readable**. There is no hot-standby query capability on the HA standby. For read scaling, you need read replicas.
  • → After failover, the old primary becomes the standby. You can initiate a manual failover (called a "switchover" in Cloud SQL) to return the primary to the original zone.
    • # Initiate a manual failover (useful for DR testing)
gcloud sql instances failover prod-pg-01

# Check replication status
gcloud sql instances describe prod-pg-01 \
  --format="value(settings.availabilityType,failoverReplica)"

    Read replicas are separate Cloud SQL instances that receive asynchronous replication from the primary:

    # Create a read replica in a different region (cross-region replica)
gcloud sql instances create prod-pg-01-replica-us-east \
  --master-instance-name=prod-pg-01 \
  --region=us-east1 \
  --tier=db-custom-4-16384 \
  --availability-type=ZONAL \
  --no-assign-ip \
  --network=projects/my-project/global/networks/prod-vpc

# Promote a replica to a standalone instance (for migrations or DR activation)
gcloud sql instances promote-replica prod-pg-01-replica-us-east

    Cascade replicas (replicas of replicas) are supported in Cloud SQL for PostgreSQL — useful when you want to offload replication traffic from the primary by having multiple tiers of read replicas without every replica pulling directly from the primary.

    Replication lag monitoring should be part of your operational baseline. Cloud SQL surfaces replication lag as a Cloud Monitoring metric:

    # Query replication lag for all replicas using gcloud monitoring
# (More practical to set up an alerting policy via Terraform)
gcloud monitoring metrics list \
  --filter="metric.type=cloudsql.googleapis.com/database/replication/replica_lag"

    In practice, you want an alerting policy that fires when replica_lag exceeds 30 seconds for OLTP replicas used for read traffic, and perhaps 300 seconds for reporting replicas. Lag above 30 seconds on a read replica that's serving application queries means users are reading stale data that may materially affect application behavior.

    External replication — Cloud SQL supports acting as a replica of an external PostgreSQL primary (for live migrations from on-premises) and also supports serving as a primary to external PostgreSQL replicas via logical replication (using pglogical or native logical replication in PostgreSQL 10+). This architecture is commonly used for zero-downtime migrations:

    -- On your on-premises PostgreSQL primary:
-- Enable logical replication (wal_level must be logical)
-- Create a replication publication
CREATE PUBLICATION cloud_migration_pub
  FOR TABLE orders, customers, products, order_items;

-- On Cloud SQL (after setting up as external replica):
-- Cloud SQL handles the subscription setup via the import/replica mechanism,
-- but for pglogical you would configure it as follows after enabling the extension:

CREATE EXTENSION pglogical;

SELECT pglogical.create_node(
  node_name := 'cloud_sql_subscriber',
  dsn := 'host=127.0.0.1 port=5432 dbname=appdb'
);

SELECT pglogical.create_subscription(
  subscription_name := 'onprem_to_cloudsql',
  provider_dsn := 'host=10.0.1.50 port=5432 dbname=appdb 
                   user=replication_user password=REPL_PASS',
  replication_sets := ARRAY['default']
);

    19.5 IAM Authentication and Network Security

    On-premises PostgreSQL access control lives in two places: pg_hba.conf controls who can connect from where and via which authentication method, and pg_ident.conf handles OS user mapping. In Cloud SQL, this model is replaced — and improved in some ways — by IAM integration and Cloud SQL Auth Proxy.

    IAM database authentication lets you authenticate to PostgreSQL or SQL Server using a Google Cloud service account or user identity token instead of a database password. This is meaningful for production because it eliminates the credential rotation problem: IAM tokens are short-lived (1 hour), and service accounts can be centrally managed through IAM policies without touching the database.

    # Grant IAM authentication to a service account for PostgreSQL
# The user format is the service account email address

gcloud sql users create "myapp-sa@my-project.iam.gserviceaccount.com" \
  --instance=prod-pg-01 \
  --type=CLOUD_IAM_SERVICE_ACCOUNT

# Grant IAM authentication to a user account (for human access)
gcloud sql users create "dba@mycompany.com" \
  --instance=prod-pg-01 \
  --type=CLOUD_IAM_USER

# Then grant database-level permissions using standard PostgreSQL GRANT
# Connect as the postgres user first, then:
    -- Grant the IAM service account access to the application database
-- The IAM user appears in PostgreSQL as the email address (service account)
-- or as the user portion before @ for human accounts
GRANT CONNECT ON DATABASE appdb TO "myapp-sa@my-project.iam.gserviceaccount.com";
GRANT USAGE ON SCHEMA public TO "myapp-sa@my-project.iam.gserviceaccount.com";
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public 
  TO "myapp-sa@my-project.iam.gserviceaccount.com";

-- For new tables created in the future:
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES 
  TO "myapp-sa@my-project.iam.gserviceaccount.com";

    Cloud SQL Auth Proxy handles TLS tunnel establishment and IAM-based authentication. Think of it as the operational layer between your application and the database — it intercepts connections, authenticates to Google's APIs, and forwards traffic over an encrypted tunnel. It is not a connection pooler. Applications that open many short-lived connections still need PgBouncer or pgpool-II in front of the Auth Proxy.

    # Download and run Cloud SQL Auth Proxy (Linux/amd64)
curl -o cloud-sql-proxy \
  https://storage.googleapis.com/cloud-sql-connectors/cloud-sql-proxy/v2.7.0/cloud-sql-proxy.linux.amd64
chmod +x cloud-sql-proxy

# Start proxy for a single PostgreSQL instance
# Applications connect to localhost:5432
./cloud-sql-proxy \
  --address 127.0.0.1 \
  --port 5432 \
  my-project:us-central1:prod-pg-01

# For Kubernetes deployments, run Auth Proxy as a sidecar container
# The GKE workload identity binds the pod's service account to the GCP service account
# No credential files needed — identity is automatic

# For multiple instances (useful in multi-tenant architectures)
./cloud-sql-proxy \
  my-project:us-central1:prod-pg-01?port=5432 \
  my-project:us-central1:prod-pg-02?port=5433 \
  my-project:us-east1:prod-pg-01-replica-us-east?port=5434

    Private Service Connect is Google's newer private connectivity mechanism, replacing the older Private IP approach for certain architectures. Where Private IP places the Cloud SQL instance directly on your VPC subnet via a VPC peering arrangement, Private Service Connect creates a private endpoint in your VPC that routes to a Cloud SQL instance without requiring VPC peering or consuming IP ranges from your subnet. This matters at enterprise scale where VPC peering IP range exhaustion becomes a real operational constraint.

    # Create a Private Service Connect endpoint for Cloud SQL
# First, ensure the Cloud SQL instance has PSC enabled (set at creation time
# or via patch if the feature is available for the tier)

gcloud sql instances patch prod-pg-01 \
  --enable-private-service-connect \
  --allowed-psc-projects=my-project,consumer-project-1,consumer-project-2

# Create the Private Service Connect forwarding rule in the consumer VPC
gcloud compute forwarding-rules create cloud-sql-psc-endpoint \
  --region=us-central1 \
  --network=prod-vpc \
  --address=10.100.0.50 \
  --target-service-attachment=\
projects/my-project/regions/us-central1/serviceAttachments/prod-pg-01-psc

    Authorized networks (IP allowlisting) applies only to public IP instances. If you've correctly configured your instance with private IP only, authorized networks are irrelevant. For development instances with public IP, the mechanism is straightforward but operationally brittle for anything beyond a small static set of IPs.
