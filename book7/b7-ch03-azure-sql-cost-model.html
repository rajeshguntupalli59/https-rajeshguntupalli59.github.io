Chapter 3: Azure SQL Cost Model: DTU vs vCore vs Serverless

Organizations running workloads on Azure SQL Database routinely overspend by 40–60% because they inherited a DTU tier chosen during initial migration, never revisited it, and now pay for capacity that sits idle 70% of the day. This chapter breaks down all three Azure SQL purchasing models — DTU-based, vCore-based, and Serverless — explains exactly when each one saves or costs you money, and gives you the queries, CLI commands, and Python automation to identify mismatches and fix them on a schedule. A team running a 500 DTU Premium database that migrates to a General Purpose 4 vCore with a reserved instance and the Azure Hybrid Benefit can realistically reduce that single database's monthly bill from roughly $920 to $310 — a saving of $610/month without touching a line of application code.

---

3.1 Understanding the Three Purchasing Models

Before optimizing, you need a precise mental model of what you are actually buying. Azure SQL charges you differently depending on which of its three cost models you choose, and conflating them is the primary reason teams make poor sizing decisions.

#### DTU-Based Purchasing

DTU stands for Database Transaction Unit — a blended measure of CPU, memory, and I/O that Microsoft bundles into a single number. You cannot see or control the individual components. A 100 DTU Standard database costs roughly $15/month; a 500 DTU Premium database costs around $465/month. The tiers are:

| Tier | DTU Range | Max Storage | Approximate Monthly Cost |

|------|-----------|-------------|--------------------------|

| Basic | 5 | 2 GB | $5 |

| Standard (S0–S9) | 10–3,000 | 1 TB | $15–$450 |

| Premium (P1–P15) | 125–4,000 | 4 TB | $465–$14,000+ |

The DTU model's core problem is opacity. You cannot correlate DTU consumption to actual CPU or memory utilization. If a query is I/O bound and you're buying a CPU-heavy DTU tier, you're paying a bundled price for resources you don't consume. DTU databases also do not support the Azure Hybrid Benefit for SQL Server licenses, which alone eliminates 30% of the vCore price.

#### vCore-Based Purchasing

The vCore model lets you choose cores and memory independently (within hardware tiers), apply your own SQL Server licenses, and see actual CPU and memory metrics. There are three hardware families:

  • → **General Purpose (Fsv2 or Gen5)** — balanced compute and storage, good for most OLTP workloads.
  • → **Business Critical** — local SSD, built-in read replica, high memory-to-core ratio. Equivalent to a local Premium SSD RAID; suited for latency-sensitive OLTP.
  • → **Hyperscale** — distributed storage that scales to 100 TB, with rapid snapshot-based backups.

    • A General Purpose 4 vCore (Gen5) runs approximately $368/month pay-as-you-go. Apply the Azure Hybrid Benefit if you have existing SQL Server Enterprise licenses with Software Assurance, and the cost drops to roughly $186/month. Buy a 1-year reserved instance on top of that and you reach approximately $155/month. Buy a 3-year reserved instance and you reach approximately $115/month. The math on that last number versus a 500 DTU Premium ($465/month) tells the whole story for moderately sized OLTP databases.

    #### Serverless

    Serverless is a vCore-based sub-model where you define a minimum and maximum vCore count and pay only for the compute you use, billed per second. When the database sits idle for a configurable period (the auto-pause delay, minimum 1 hour), Azure pauses it entirely and you pay only for storage. When the next connection arrives, the database resumes — the cold-start latency typically runs 30–90 seconds.

    Serverless pricing for General Purpose: approximately $0.000145 per vCore-second. A database running at 2 vCores for 8 hours/day costs roughly $8.35/month in compute, versus the always-on General Purpose 2 vCore at approximately $184/month. That is a 95% compute saving for a development database used only during business hours, assuming it auto-pauses for the remaining 16 hours.

    The break-even point between Serverless and provisioned vCore is roughly 50% average utilization across a 730-hour month. Workloads above that threshold are cheaper on provisioned; workloads below it are cheaper on Serverless.

    3.2 Diagnosing Your Current Spend and Utilization

    The first step is understanding what you are actually paying for versus what your databases actually consume. Azure provides this data in two places: Azure Cost Management and the database's own DMVs.

    #### Pulling Cost Data with Azure CLI

    The following command exports the last 30 days of Azure SQL spend by resource, broken down to the database level:

    #!/bin/bash
# Pull Azure SQL cost breakdown for the last 30 days
# Requires: Azure CLI authenticated, correct subscription set

SUBSCRIPTION_ID="your-subscription-id"
RESOURCE_GROUP="your-rg"
START_DATE=$(date -d "30 days ago" '+%Y-%m-%d')
END_DATE=$(date '+%Y-%m-%d')

az consumption usage list \
  --start-date "$START_DATE" \
  --end-date "$END_DATE" \
  --query "[?contains(instanceName, 'sql')].{
    Resource:instanceName,
    Cost:pretaxCost,
    Currency:currency,
    UsageDate:usageStart
  }" \
  --output table \
  --subscription "$SUBSCRIPTION_ID"

# For more granular breakdown by SKU (DTU tier, vCore family)
az costmanagement query \
  --type Usage \
  --timeframe Custom \
  --time-period from="$START_DATE" to="$END_DATE" \
  --dataset-configuration columns='["SubscriptionId","ResourceGroup","ResourceId","MeterCategory","MeterSubCategory","MeterName","Quantity","Unit","PreTaxCost","UsageDate"]' \
  --dataset-filter '{
    "and": [
      {
        "dimensions": {
          "name": "MeterCategory",
          "operator": "In",
          "values": ["SQL Database"]
        }
      }
    ]
  }' \
  --scope "subscriptions/$SUBSCRIPTION_ID" \
  --output json | jq '.properties.rows[] | {
    resource: .[2],
    meter: .[4],
    sku: .[5],
    quantity: .[6],
    cost: .[8],
    date: .[9]
  }'

    #### Reading Utilization from the Database DMVs

    Once you know what you're paying, you need to know what you're using. The following query runs inside Azure SQL and returns the 15-day average DTU (or CPU) consumption alongside the tier you're paying for — giving you the utilization percentage that drives every sizing decision:

    -- Run this inside the Azure SQL database (not master)
-- Returns: 15-day average resource utilization vs. purchased tier
-- Works for both DTU and vCore databases

SELECT
    database_name                                               AS [Database],
    service_objective                                           AS [Current_SKU],
    elastic_pool_name                                           AS [Elastic_Pool],
    -- DTU utilization (NULL for vCore databases)
    AVG(avg_cpu_percent)                                        AS [Avg_CPU_Pct_15Day],
    MAX(avg_cpu_percent)                                        AS [Max_CPU_Pct_15Day],
    AVG(avg_data_io_percent)                                    AS [Avg_DataIO_Pct_15Day],
    MAX(avg_data_io_percent)                                    AS [Max_DataIO_Pct_15Day],
    AVG(avg_log_write_percent)                                  AS [Avg_LogWrite_Pct_15Day],
    MAX(avg_log_write_percent)                                  AS [Max_LogWrite_Pct_15Day],
    AVG(avg_memory_usage_percent)                               AS [Avg_Mem_Pct_15Day],
    MAX(avg_memory_usage_percent)                               AS [Max_Mem_Pct_15Day],
    -- Derived overall DTU utilization (highest of the four dimensions)
    AVG(GREATEST(
        avg_cpu_percent,
        avg_data_io_percent,
        avg_log_write_percent
    ))                                                          AS [Avg_Effective_DTU_Pct],
    MAX(GREATEST(
        avg_cpu_percent,
        avg_data_io_percent,
        avg_log_write_percent
    ))                                                          AS [Max_Effective_DTU_Pct],
    MIN(end_time)                                               AS [Oldest_Sample],
    MAX(end_time)                                               AS [Newest_Sample],
    COUNT(*)                                                    AS [Sample_Count]
FROM sys.dm_db_resource_stats
CROSS JOIN (
    SELECT
        d.name                  AS database_name,
        d.service_objective     AS service_objective,
        d.elastic_pool_name     AS elastic_pool_name
    FROM sys.database_service_objectives d
    WHERE d.database_id = DB_ID()
) meta
GROUP BY
    database_name,
    service_objective,
    elastic_pool_name;

    > Reading the output: If Avg_Effective_DTU_Pct is below 40% and Max_Effective_DTU_Pct is below 75%, you are a candidate for downsizing. If Avg_Effective_DTU_Pct is above 70% consistently, downsizing will cause performance degradation — address query efficiency first.

    For longer historical windows (up to 14 days) from the master database, use sys.resource_stats instead:

    -- Run this in the MASTER database on a logical server
-- 14-day utilization view across all databases on the server

SELECT
    database_name,
    sku,
    -- 5-minute granularity samples
    COUNT(*)                                        AS sample_count,
    AVG(avg_cpu_percent)                            AS avg_cpu_pct,
    PERCENTILE_CONT(0.95) WITHIN GROUP
        (ORDER BY avg_cpu_percent)                  AS p95_cpu_pct,
    AVG(avg_data_io_percent)                        AS avg_io_pct,
    PERCENTILE_CONT(0.95) WITHIN GROUP
        (ORDER BY avg_data_io_percent)              AS p95_io_pct,
    -- Cost signal: hours where utilization < 20% (idle capacity)
    SUM(CASE WHEN avg_cpu_percent < 20
             AND avg_data_io_percent < 20
             THEN 1 ELSE 0 END) * 5.0 / 60         AS idle_hours_approx,
    SUM(CASE WHEN avg_cpu_percent > 80
             THEN 1 ELSE 0 END) * 5.0 / 60         AS saturated_hours_approx
FROM sys.resource_stats
WHERE start_time >= DATEADD(DAY, -14, GETUTCDATE())
GROUP BY database_name, sku
ORDER BY avg_cpu_pct DESC;

    3.3 Choosing the Right Tier: Decision Framework

    Given utilization data, you can now apply a structured decision process. The following flowchart in prose form captures the logic:

    1. Is the database used fewer than 12 hours per day on average, and can it tolerate a 30–90 second cold-start? → Serverless. Set min_capacity = 0.5 vCores, max_capacity to your peak requirement, and set auto_pause_delay_minutes = 60 for dev/test or 120 for lightly used production databases.

    2. Is the database used more than 12 hours per day with predictable load? → Provisioned vCore General Purpose. Apply Hybrid Benefit if you own licenses. Buy reserved capacity.

    3. Does the workload require < 5 ms read latency, local SSD, or a free read replica for reporting? → Business Critical vCore. This tier is significantly more expensive (~3× General Purpose) but eliminates the need for a separate read replica or Premium storage.

    4. Is the database in a DTU tier and used inconsistently across a shared pool of multiple databases? → Elastic Pool (vCore or DTU-based), covered in Chapter 4. Do not optimize a single database in isolation if it shares load patterns with ten others.

    5. Does the database need to grow beyond 4 TB or requires fast, large-scale backup/restore? → Hyperscale. Cost is separate from the above tiers; storage is billed at $0.115/GB/month for Hyperscale versus $0.115/GB/month for General Purpose remote storage — similar per-GB pricing but Hyperscale adds compute cost for its page servers.

    #### Translating DTU to vCore (The Conversion Table)

    Microsoft's rough conversion guideline: 100 DTUs ≈ 1 vCore for Standard/General Purpose. For Premium/Business Critical, it's closer to 125 DTUs ≈ 1 vCore. However, these are approximations. The utilization percentages matter more than the raw numbers.

    | Current Tier | Approx. Monthly Cost | vCore Equivalent | vCore Pay-as-Go | With Hybrid Benefit | With 1-Yr RI + HB |

    |---|---|---|---|---|---|

    | S3 (100 DTU) | $150 | GP 1 vCore | $184 | $93 | $77 |

    | P1 (125 DTU) | $465 | GP 4 vCore | $368 | $186 | $155 |

    | P2 (250 DTU) | $930 | GP 8 vCore | $736 | $372 | $310 |

    | P4 (500 DTU) | $1,860 | GP 16 vCore | $1,471 | $743 | $619 |

    The P1→GP4 + HB + 1-Yr RI move illustrated here is where the $310/month number in the chapter opener comes from. The P4 → GP16 equivalent at 3-year RI with HB reaches approximately $490/month versus $1,860 — a $1,370/month saving on a single database.

    3.4 Activating Azure Hybrid Benefit and Reserved Instances

    The Azure Hybrid Benefit (AHB) and reserved instances are the two highest-leverage levers that require zero application change. Yet many organizations leave both on the table because no one on the database team owns the licensing conversation.

    #### Enabling Hybrid Benefit via Azure CLI

    #!/bin/bash
# Enable Azure Hybrid Benefit on an existing Azure SQL Database (vCore model)
# Prerequisites: SQL Server license with Software Assurance

RESOURCE_GROUP="prod-databases-rg"
SERVER_NAME="prod-sqlserver-eastus"
DATABASE_NAME="orders-db"

# Check current license type
az sql db show \
  --resource-group "$RESOURCE_GROUP" \
  --server "$SERVER_NAME" \
  --name "$DATABASE_NAME" \
  --query "{name:name, licenseType:licenseType, sku:sku}" \
  --output table

# Apply Hybrid Benefit (change licenseType from LicenseIncluded to BasePrice)
az sql db update \
  --resource-group "$RESOURCE_GROUP" \
  --server "$SERVER_NAME" \
  --name "$DATABASE_NAME" \
  --license-type BasePrice

# Verify the change
az sql db show \
  --resource-group "$RESOURCE_GROUP" \
  --server "$SERVER_NAME" \
  --name "$DATABASE_NAME" \
  --query "{name:name, licenseType:licenseType, edition:edition, capacity:currentServiceObjectiveName}" \
  --output table

# Apply to all databases on a server in bulk
echo "Applying Hybrid Benefit to all vCore databases on $SERVER_NAME..."
az sql db list \
  --resource-group "$RESOURCE_GROUP" \
  --server "$SERVER_NAME" \
  --query "[?licenseType=='LicenseIncluded'].name" \
  --output tsv | while read db_name; do
    echo "Updating: $db_name"
    az sql db update \
      --resource-group "$RESOURCE_GROUP" \
      --server "$SERVER_NAME" \
      --name "$db_name" \
      --license-type BasePrice
done

echo "Done. Verify in Azure Portal under Cost Management."

    #### Purchasing Reserved Instances

    Reserved instances for Azure SQL are purchased through the Azure Portal under Reservations, not through the SQL Database blade itself. You reserve at the vCore family and region level, not at the individual database level. A 4-vCore General Purpose East US reservation covers any General Purpose 4-vCore database in East US, regardless of the server name. This means you can replace databases, rename them, or move workloads between databases, and the reservation continues applying.

    Reservation calculation before purchasing:

    Before committing, verify utilization will remain stable. A 1-year commitment on compute that you later delete runs to waste. The break-even for a 1-year RI versus pay-as-you-go is approximately 5–6 months of continuous operation. For databases you know will run for 12+ months, the 1-year RI is always the right choice unless the team plans a migration to a different service within the year.
