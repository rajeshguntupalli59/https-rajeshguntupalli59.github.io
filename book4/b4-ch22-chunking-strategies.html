Chapter 22: Chunking Strategies: Size, Overlap, Semantic Splitting

Before a single embedding gets generated or a vector stored, every document passing through your RAG pipeline must be broken into pieces. That process—chunking—is one of the least glamorous and most consequential decisions you will make as a vector database engineer. Get it wrong and your retrieval system returns fragments that answer adjacent questions, embedding models that overflow their token limits, and LLM context windows stuffed with repeated text that buries the actual answer. This chapter covers the mechanics of chunking: fixed-size splits, overlap strategies, semantic and structural splitting, and the production patterns that determine which approach belongs in which pipeline.

---

22.1 Why Chunking Is a Database Design Problem

Database professionals recognize chunking immediately once you reframe it correctly: it is a normalization decision. You are decomposing a large, unstructured artifact into retrievable units whose granularity determines both storage efficiency and query precision. Too coarse and your "rows" carry too much noise; too fine and you lose context. Both failures show up at query time.

The analogy to indexing is tight. In a relational database, if you index too wide a column set, the index becomes bloated and less discriminating. If you index too narrow, queries return too many false positives that require expensive post-filtering. Embedding vectors work the same way. An embedding over 8,000 tokens captures broad thematic meaning but loses fine-grained retrieval precision. An embedding over 50 tokens is precise but brittle—small phrasing differences cause cosine distance to diverge even when semantic meaning is equivalent.

Embedding models impose a hard ceiling that makes this concrete. OpenAI's text-embedding-3-small and text-embedding-3-large support up to 8,191 tokens. Cohere's embed-english-v3.0 supports up to 512 tokens per input by default (extendable, but with documented precision tradeoffs). sentence-transformers/all-MiniLM-L6-v2, popular for on-premise workloads, caps at 256 tokens and silently truncates anything longer. Overflowing these limits does not throw an error in most pipelines—the model simply truncates the input, which means your stored vector represents a partial document without any warning in the logs.

Three variables govern every chunking decision:

  • → **Chunk size**: The maximum number of tokens (or characters) in a single chunk. This is your primary granularity control.
  • → **Overlap**: The number of tokens repeated between adjacent chunks. Overlap preserves context across boundaries at the cost of storage and some retrieval redundancy.
  • → **Splitting logic**: Where you place boundaries—at fixed intervals, at sentence or paragraph boundaries, or at semantically coherent transition points.

    • A fourth variable that DBAs naturally gravitate toward is metadata attachment: every chunk should carry enough context to reconstruct its provenance—source document ID, position in the document, section heading, and any filtering dimensions like document type or access tier. Treating chunks as rows with rich metadata columns, rather than as opaque text blobs, is what separates a retrievable knowledge base from a vector landfill.

    22.2 Fixed-Size Chunking: Mechanics and Limits

    Fixed-size chunking splits documents at a specified token or character count, optionally with overlap. It is the SUBSTR of the chunking world—simple, predictable, and wrong in ways you can anticipate.

    The implementation is straightforward. LangChain's RecursiveCharacterTextSplitter is the most widely used production implementation. It tries a hierarchy of separators—paragraph breaks, newlines, spaces, then character-by-character—to split at the target size while respecting natural boundaries where possible.

    # Python
from langchain.text_splitter import RecursiveCharacterTextSplitter
import tiktoken

def chunk_document(text: str, chunk_size: int = 512, chunk_overlap: int = 64) -> list[dict]:
    """
    Fixed-size chunking with token-accurate sizing using tiktoken.
    Returns chunks as dicts with text and positional metadata.
    """
    # Use cl100k_base (GPT-4 / text-embedding-3-* tokenizer)
    enc = tiktoken.get_encoding("cl100k_base")

    def token_length(s: str) -> int:
        return len(enc.encode(s))

    splitter = RecursiveCharacterTextSplitter(
        chunk_size=chunk_size,
        chunk_overlap=chunk_overlap,
        length_function=token_length,
        separators=["\n\n", "\n", ". ", "! ", "? ", " ", ""],
    )

    raw_chunks = splitter.split_text(text)

    chunks = []
    for i, chunk_text in enumerate(raw_chunks):
        chunks.append({
            "chunk_index": i,
            "text": chunk_text,
            "token_count": token_length(chunk_text),
            "char_count": len(chunk_text),
        })

    return chunks


# Example: chunking a technical document
sample_text = """
PostgreSQL's MVCC implementation maintains multiple versions of each row
to support concurrent transactions without locking. When a transaction
modifies a row, the old version remains visible to concurrent readers
that started before the modification.

The visibility rules are governed by transaction IDs (XIDs). Each row
version carries xmin and xmax values representing the creating and
deleting transaction respectively. The snapshot isolation level reads
the most recent committed version visible at transaction start time.

Autovacuum reclaims dead tuples—row versions no longer visible to any
active transaction—by marking their storage space as reusable. Without
regular vacuuming, table bloat accumulates and query plans degrade as
the planner's statistics drift from physical reality.
"""

chunks = chunk_document(sample_text, chunk_size=128, chunk_overlap=16)
for chunk in chunks:
    print(f"Chunk {chunk['chunk_index']}: {chunk['token_count']} tokens")
    print(f"  Text: {chunk['text'][:80]}...")
    print()

    The length_function parameter is the detail most tutorials omit. Without it, LangChain defaults to character count. Characters and tokens are not the same unit. A 512-character chunk might be 90 tokens (dense prose) or 200 tokens (code with many single-character tokens). Using tiktoken with the correct encoding for your embedding model gives you accurate sizing and prevents silent truncation.

    Overlap mechanics: An overlap of 64 tokens on 512-token chunks means 12.5% of each chunk is repeated in the next. This costs roughly 12.5% more storage and 12.5% more embedding API calls. The benefit is that sentences or concepts split across a chunk boundary appear in full in at least one chunk. A common production setting is 10–20% overlap. Beyond 25%, you start seeing retrieval results that are near-duplicates of each other, which frustrates reranking and LLM context packing.

    Fixed-size chunking degrades in three predictable situations:

    1. Structured documents: A 512-token chunk that starts mid-table and ends mid-code-block produces embeddings that mix multiple data types, making cosine similarity less meaningful.

    2. Documents with natural section boundaries: A 4,000-token technical specification split mechanically will produce chunks that contain the end of one section and the beginning of another. The embedding represents neither topic cleanly.

    3. Short documents: If most of your documents are under 200 tokens, chunking adds overhead without benefit. Store them as single vectors.

    The performance characteristics are well-understood: for a corpus of one million documents averaging 2,000 tokens, 512-token chunks with 64-token overlap produce approximately 4.3 million chunks. At 1,536 dimensions (OpenAI's default), that is roughly 25 GB of raw vector data before indexing overhead. These numbers matter when sizing your pgvector table or your Pinecone pod configuration.

    22.3 Overlap Design: What Gets Repeated and Why

    Overlap deserves its own treatment because most engineers implement it mechanically without thinking through what the repeated content actually does to retrieval.

    Consider a document about database transaction isolation levels. A chunk boundary falls in the middle of the paragraph explaining phantom reads. The first chunk ends with "...preventing dirty reads and non-repeatable reads. However, at this isolation level" and the second chunk begins with "...phantom reads can still occur when a transaction re-executes a range query." Without overlap, neither chunk contains a complete explanation of phantom reads. With 64-token overlap, both chunks carry the boundary sentence, and whichever chunk scores higher for the query "what are phantom reads" will contain enough context to answer it.

    This logic leads to a counter-intuitive insight: overlap primarily helps with retrieval precision, not recall. You are not finding more relevant chunks; you are ensuring the chunks you do find are self-contained enough to be useful.

    # Python
# Demonstrating overlap behavior and its effect on boundary content

def analyze_chunk_boundaries(chunks: list[dict], overlap_tokens: int) -> None:
    """
    Shows what content is duplicated between adjacent chunks.
    Useful for verifying overlap is working as expected.
    """
    enc = tiktoken.get_encoding("cl100k_base")

    for i in range(len(chunks) - 1):
        current_tokens = enc.encode(chunks[i]["text"])
        next_tokens = enc.encode(chunks[i + 1]["text"])

        # Find the tail of the current chunk in the beginning of the next
        overlap_window = current_tokens[-overlap_tokens * 2:]  # search region
        overlap_found = 0

        for size in range(min(overlap_tokens * 2, len(next_tokens)), 0, -1):
            if overlap_window[-size:] == next_tokens[:size]:
                overlap_found = size
                break

        overlap_text = enc.decode(next_tokens[:overlap_found]) if overlap_found else ""
        print(f"Boundary {i}->{i+1}: {overlap_found} overlapping tokens")
        if overlap_text:
            print(f"  Repeated: '{overlap_text[:60]}...'")


import tiktoken
chunks = chunk_document(sample_text, chunk_size=128, chunk_overlap=16)
analyze_chunk_boundaries(chunks, overlap_tokens=16)

    Overlap and vector deduplication: When you store overlapping chunks, adjacent chunks will have higher cosine similarity to each other than to any other chunk in the corpus. This is expected. The problem arises during retrieval when a query returns both chunk N and chunk N+1 as top-K results, and the LLM receives essentially the same content twice. A production pipeline should include a deduplication pass during retrieval. The canonical approach is Maximum Marginal Relevance (MMR): after retrieving the top-20 candidates, re-rank to maximize diversity so that the top-5 passed to the LLM cover distinct semantic ground.

    Sliding window vs. stepped overlap: The standard implementation steps forward by chunk_size - overlap tokens and re-embeds. An alternative is a sliding window that moves one sentence at a time and generates a chunk for each window position. This is expensive (a 1,000-sentence document generates ~1,000 embeddings) but produces very high retrieval density. Few production systems use pure sliding windows; the more practical approach is sentence-aware chunking described in the next section.

    22.4 Structural and Semantic Splitting

    Fixed-size splitting ignores document structure. Structural splitting respects it. Semantic splitting goes further and finds topical boundaries in the content itself.

    Structural splitting uses document markers to define chunk boundaries: Markdown headings, HTML tags, PDF section headers, code block delimiters. For technical documentation, this is almost always superior to fixed-size splitting because your users' questions map to document sections, not to arbitrary 512-token windows.

    LlamaIndex provides a MarkdownNodeParser that splits on heading hierarchy. For HTML, HTMLNodeParser extracts text from tagged sections. For code repositories, CodeSplitter uses the tree-sitter library to split at function or class boundaries—a significant advantage over splitting a Python file mid-function.

    # Python
from llama_index.core.node_parser import (
    MarkdownNodeParser,
    SentenceSplitter,
    SemanticSplitterNodeParser,
)
from llama_index.core import Document
from llama_index.embeddings.openai import OpenAIEmbedding
import os

# --- Structural splitting: Markdown ---
markdown_text = """
# PostgreSQL Performance Tuning

## Connection Pooling

PgBouncer operates in three modes: session, transaction, and statement pooling.
Transaction pooling provides the best concurrency for OLTP workloads because
the server connection is held only during an active transaction, not for the
entire client session lifetime.

Connection limits should be set conservatively. A PostgreSQL instance with
`max_connections = 200` and 16 CPU cores will contend heavily beyond
~4x core count active connections under CPU-bound workloads.

## Index Strategy

Partial indexes reduce index size by only including rows matching a predicate.
A partial index on `orders(customer_id) WHERE status = 'pending'` is far
smaller than a full index on `customer_id` and serves the most performance-
sensitive queries—those looking up active work—with lower I/O cost.

### B-Tree vs. BRIN

B-Tree indexes support equality and range queries with O(log n) lookup.
BRIN (Block Range INdex) is appropriate for naturally ordered data—timestamps,
sequential IDs—where physical row order correlates with column values.
A BRIN index on a 500GB timestamp column might be 50KB versus a 10GB B-Tree.
"""

markdown_doc = Document(text=markdown_text)
markdown_parser = MarkdownNodeParser()
markdown_nodes = markdown_parser.get_nodes_from_documents([markdown_doc])

print("=== Structural (Markdown) Chunks ===")
for node in markdown_nodes:
    print(f"Node ID: {node.node_id[:8]}")
    print(f"  Metadata: {node.metadata}")
    print(f"  Text ({len(node.text)} chars): {node.text[:100]}...")
    print()

# --- Semantic splitting ---
# SemanticSplitterNodeParser embeds sentences and splits where
# cosine distance between adjacent sentence embeddings exceeds a threshold.
# This finds topic transitions without relying on document markup.

embed_model = OpenAIEmbedding(
    model="text-embedding-3-small",
    api_key=os.environ["OPENAI_API_KEY"],
)

semantic_splitter = SemanticSplitterNodeParser(
    buffer_size=1,              # sentences to combine before comparing
    breakpoint_percentile_threshold=95,  # split at top 5% of distance jumps
    embed_model=embed_model,
)

# A document where topics shift without heading markers
prose_text = """
Database replication ensures that multiple servers hold copies of the same data.
In streaming replication, the primary sends WAL records to standbys continuously.
The standby applies these records to stay current with the primary's state.
Replication lag is the delay between a change on the primary and its appearance
on the standby, measured in bytes of WAL or elapsed time.

Machine learning model training requires large volumes of labeled data.
The training loop iterates over batches, computes loss against ground truth,
and updates model weights via backpropagation. Gradient descent variants like
Adam and AdaGrad adjust the learning rate per parameter to accelerate convergence.
Overfitting occurs when a model learns training distribution noise rather than
generalizable patterns, detected by divergence between training and validation loss.

Query optimization in PostgreSQL uses a cost-based planner that estimates
the computational expense of candidate execution plans. The planner relies on
table statistics—row counts, column cardinality, value distributions—collected
by ANALYZE. Stale statistics cause the planner to choose suboptimal join orders
or index strategies, which is why autovacuum schedules ANALYZE alongside VACUUM.
"""

prose_doc = Document(text=prose_text)
semantic_nodes = semantic_splitter.get_nodes_from_documents([prose_doc])

print("=== Semantic Chunks ===")
for i, node in enumerate(semantic_nodes):
    print(f"Semantic chunk {i}: {len(node.text)} chars")
    print(f"  First 80 chars: {node.text[:80]}...")
    print()

    The SemanticSplitterNodeParser embeds each sentence individually, then computes the cosine distance between adjacent sentence embeddings. Where distance spikes—at the 95th percentile in the example above—it places a chunk boundary. The prose example above has three clearly distinct topics: replication, ML training, and query optimization. Semantic splitting will find those three boundaries without relying on any markup. Fixed-size splitting at 256 tokens would likely put all three topics in a single chunk or split mid-paragraph within a topic.

    The cost is API calls. For a 10,000-sentence corpus, semantic splitting requires 10,000 embedding calls just to determine chunk boundaries—before embedding the final chunks. This is acceptable for offline batch processing but prohibitive for real-time ingestion pipelines. Two practical mitigations: run semantic splitting as a preprocessing step and cache boundaries, or use a fast local model (all-MiniLM-L6-v2) for boundary detection and a higher-quality model for final chunk embeddings.

    Hybrid approaches: The production pattern that works best for most technical corpora combines structural and semantic splitting. First, split on document structure (headings, sections). Then apply semantic splitting within each section if the section exceeds your target chunk size. This bounds the number of embedding calls for boundary detection to one call per sentence within oversized sections, rather than the entire corpus.

    22.5 Ingesting Chunks into pgvector and Pinecone

    The chunking strategy determines chunk count and size. The ingestion strategy determines throughput, cost, and metadata queryability. These two concerns must be designed together.

    The following example shows a complete ingestion pipeline: chunk a document, embed each chunk with OpenAI, and upsert into both pgvector (for hybrid search applications) and Pinecone (for pure vector workloads at scale).

    # Python
import os
import uuid
import psycopg2
import openai
from pinecone import Pinecone, ServerlessSpec
from langchain.text_splitter import RecursiveCharacterTextSplitter
import tiktoken
import json

# --- Configuration ---
OPENAI_API_KEY = os.environ["OPENAI_API_KEY"]
PINECONE_API_KEY = os.environ["PINECONE_API_KEY"]
PG_DSN = os.environ.get("PG_DSN", "postgresql://user:pass@localhost:5432/vectordb")
EMBEDDING_MODEL = "text-embedding-3-small"
EMBEDDING_DIM = 1536
CHUNK_SIZE = 512
CHUNK_OVERLAP = 64
BATCH_SIZE = 100  # embeddings per API call (OpenAI max: 2048)

# --- Clients ---
oai_client = openai.OpenAI(api_key=OPENAI_API_KEY)
pc = Pinecone(api_key=PINECONE_API_KEY)

# --- Tokenizer ---
enc = tiktoken.get_encoding("cl100k_base")
splitter = RecursiveCharacterTextSplitter(
    chunk_size=CHUNK_SIZE,
    chunk_overlap=CHUNK_OVERLAP,
    length_function=lambda s: len(enc.encode(s)),
    separators=["\n\n", "\n", ". ", " ", ""],
)

def embed_batch(texts: list[str]) -> list[list[float]]:
    """Embed a batch of texts. Handles API rate limits with simple retry."""
    import time
    for attempt in range(3):
        try:
            response = oai_client.embeddings.create(
                model=EMBEDDING_MODEL,
                input=texts,
                encoding_format="float",
            )
            return [r.embedding for r in response.data]
        except openai.RateLimitError:
            time.sleep(2 ** attempt)
    raise RuntimeError("Embedding API failed after 3 retries")

def ingest_document(
    doc_text: str,
    doc_id: str,
    metadata: dict,
    pg_conn,
    pinecone_index,
) -> int:
    """
    Chunk, embed, and store a document.
    Returns the number of chunks created.
    """
    raw_chunks = splitter.split_text(doc_text)
    if not raw_chunks:
        return 0

    # Build chunk records with metadata
    chunk_records = []
    for i, chunk_text in enumerate(raw_chunks):
        chunk_records.append({
            "id": str(uuid.uuid4()),
            "doc_id": doc_id,
            "chunk_index": i,
            "text": chunk_text,
            "token_count": len(enc.encode(chunk_text)),
            "metadata": {**metadata, "chunk_index": i, "doc_id": doc_id},
        })

    # Embed in batches
    all_embeddings = []
    for batch_start in range(0, len(chunk_records), BATCH_SIZE):
        batch = chunk_records[batch_start : batch_start + BATCH_SIZE]
        texts = [r["text"] for r in batch]
        embeddings = embed_batch(texts)
        all_embeddings.extend(embeddings)

    # Assign embeddings back to records
    for record, embedding in zip(chunk_records, all_embeddings):
        record["embedding"] = embedding

    # --- pgvector upsert ---
    with pg_conn.cursor() as cur:
        cur.executemany(
            """
            INSERT INTO document_chunks
                (id, doc_id, chunk_index, content, token_count, metadata, embedding)
            VALUES
                (%s, %s, %s, %s, %s, %s::jsonb, %s::vector)
            ON CONFLICT (id) DO UPDATE SET
                content = EXCLUDED.content,
                embedding = EXCLUDED.embedding,
                metadata = EXCLUDED.metadata
            """,
            [
                (
                    r["id"],
                    r["doc_id"],
                    r["chunk_index"],
                    r["text"],
                    r["token_count"],
                    json.dumps(r["metadata"]),
                    r["embedding"],
                )
                for r in chunk_records
            ],
        )
    pg_conn.commit()

    # --- Pinecone upsert ---
    pinecone_vectors = [
        {
            "id": r["id"],
            "values": r["embedding"],
            "metadata": {
                **r["metadata"],
                "text": r["text"][:1000],  # Pinecone metadata: 40KB limit per vector
            },
        }
        for r in chunk_records
    ]
    # Pinecone recommends batches of 100 vectors
    for batch_start in range(0, len(pinecone_vectors), 100):
        batch = pinecone_vectors[batch_start : batch_start + 100]
        pinecone_index.upsert(vectors=batch)

    return len(chunk_records)


# --- Schema (run once) ---
SCHEMA_SQL = """
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE IF NOT EXISTS document_chunks (
    id          UUID PRIMARY KEY,
    doc_id      TEXT NOT NULL,
    chunk_index INTEGER NOT NULL,
    content     TEXT NOT NULL,
    token_count INTEGER,
    metadata    JSONB DEFAULT '{}',
    embedding   vector(1536),
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_chunks_doc_id
    ON document_chunks(doc_id);

CREATE INDEX IF NOT EXISTS idx_chunks_metadata
    ON document_chunks USING GIN(metadata);

CREATE INDEX IF NOT EXISTS idx_chunks_embedding
    ON document_chunks USING hnsw(embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
"""
