Chapter 14: Document Ingestion: Chunking, Embedding, Storing

The pipeline that transforms raw documents into searchable, queryable vector knowledge is where most RAG systems either succeed or quietly fall apart. Getting a proof of concept working is straightforward — point a library at some PDFs, generate embeddings, and call it done. Running that same process reliably across millions of documents, with incremental updates, deduplication, failure recovery, and observable progress, is an entirely different problem. This chapter covers the complete ingestion pipeline: how to chunk documents intelligently so that embeddings carry meaningful semantic signal, how to store those chunks and their embeddings in PostgreSQL with a schema that supports real production operations, and how to orchestrate the process asynchronously so that your application stays responsive while large corpora process in the background. PostgreSQL is not just the destination here — its features actively enable a more robust pipeline than any vector-only store can offer.

---

14.1 Schema Design: Documents, Chunks, and Embeddings

Before writing a single line of Python, the schema deserves real thought. Many teams reach for a single flat table — one row per chunk, embedding included — and discover months later that they cannot answer basic operational questions: Which documents have been processed? Which chunks belong to a document that was updated last Tuesday? Which embeddings were generated with an older model that needs reprocessing?

A normalized schema separates concerns cleanly and gives you the relational power that makes PostgreSQL worth choosing over a dedicated vector store.

-- Enable required extensions
CREATE EXTENSION IF NOT EXISTS vector;
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE EXTENSION IF NOT EXISTS pgcrypto;

-- Source documents: one row per ingested file or URL
CREATE TABLE documents (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_uri      TEXT NOT NULL,
    source_type     TEXT NOT NULL CHECK (source_type IN ('pdf', 'html', 'markdown', 'txt', 'docx', 'confluence', 'notion')),
    title           TEXT,
    author          TEXT,
    language        TEXT DEFAULT 'en',
    content_hash    TEXT NOT NULL,           -- SHA-256 of raw content; detect changes
    byte_size       BIGINT,
    page_count      INT,
    metadata        JSONB DEFAULT '{}',      -- arbitrary source-specific fields
    ingestion_status TEXT NOT NULL DEFAULT 'pending'
                    CHECK (ingestion_status IN ('pending', 'chunking', 'embedding', 'complete', 'failed')),
    error_message   TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    last_ingested_at TIMESTAMPTZ,
    collection_id   UUID REFERENCES document_collections(id) ON DELETE SET NULL
);

CREATE UNIQUE INDEX idx_documents_source_uri ON documents(source_uri);
CREATE INDEX idx_documents_status ON documents(ingestion_status);
CREATE INDEX idx_documents_collection ON documents(collection_id);
CREATE INDEX idx_documents_metadata ON documents USING GIN(metadata);
CREATE INDEX idx_documents_content_hash ON documents(content_hash);

-- Collections: logical groupings of documents (a tenant's knowledge base, a project, etc.)
CREATE TABLE document_collections (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name        TEXT NOT NULL,
    description TEXT,
    owner_id    UUID,                        -- foreign key to your users table
    settings    JSONB DEFAULT '{}',          -- e.g., {"chunk_size": 512, "overlap": 64}
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Chunks: the actual text segments that get embedded
CREATE TABLE document_chunks (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id     UUID NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    chunk_index     INT NOT NULL,            -- ordinal position within document
    chunk_text      TEXT NOT NULL,
    token_count     INT,                     -- actual token count for the chunk
    char_count      INT GENERATED ALWAYS AS (char_length(chunk_text)) STORED,
    start_char      INT,                     -- character offset in original document
    end_char        INT,
    page_number     INT,                     -- for PDFs and paginated sources
    section_title   TEXT,                    -- heading or section this chunk falls under
    chunk_type      TEXT DEFAULT 'text'
                    CHECK (chunk_type IN ('text', 'table', 'code', 'header', 'caption')),
    metadata        JSONB DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (document_id, chunk_index)
);

CREATE INDEX idx_chunks_document ON document_chunks(document_id);
CREATE INDEX idx_chunks_page ON document_chunks(document_id, page_number);
CREATE INDEX idx_chunks_metadata ON document_chunks USING GIN(metadata);
-- Trigram index for keyword fallback search
CREATE INDEX idx_chunks_text_trgm ON document_chunks USING GIN(chunk_text gin_trgm_ops);

-- Embeddings: separate table so you can re-embed without losing chunks
-- Also supports multiple embedding models simultaneously
CREATE TABLE chunk_embeddings (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    chunk_id        UUID NOT NULL REFERENCES document_chunks(id) ON DELETE CASCADE,
    embedding_model TEXT NOT NULL,           -- e.g., 'text-embedding-3-small', 'text-embedding-3-large'
    embedding_dim   INT NOT NULL,
    embedding       vector(1536),            -- adjust per model; use CHECK or partitioning for multi-dim
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (chunk_id, embedding_model)
);

-- Partial indexes for each model improve query speed significantly
CREATE INDEX idx_embeddings_small ON chunk_embeddings USING hnsw (embedding vector_cosine_ops)
    WHERE embedding_model = 'text-embedding-3-small';

CREATE INDEX idx_embeddings_large ON chunk_embeddings USING hnsw (embedding vector_cosine_ops)
    WHERE embedding_model = 'text-embedding-3-large';

-- Ingestion job log: full audit trail
CREATE TABLE ingestion_jobs (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id     UUID REFERENCES documents(id) ON DELETE CASCADE,
    job_type        TEXT NOT NULL CHECK (job_type IN ('ingest', 'reprocess', 'reembed', 'delete')),
    status          TEXT NOT NULL DEFAULT 'queued'
                    CHECK (status IN ('queued', 'running', 'complete', 'failed', 'retrying')),
    celery_task_id  TEXT,
    chunks_created  INT DEFAULT 0,
    embeddings_created INT DEFAULT 0,
    started_at      TIMESTAMPTZ,
    completed_at    TIMESTAMPTZ,
    error_message   TEXT,
    retry_count     INT DEFAULT 0,
    worker_id       TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_jobs_document ON ingestion_jobs(document_id);
CREATE INDEX idx_jobs_status ON ingestion_jobs(status);
CREATE INDEX idx_jobs_celery ON ingestion_jobs(celery_task_id);

-- Trigger to keep updated_at current on documents
CREATE OR REPLACE FUNCTION set_updated_at()
RETURNS TRIGGER LANGUAGE plpgsql AS $$
BEGIN
    NEW.updated_at = now();
    RETURN NEW;
END;
$$;

CREATE TRIGGER trg_documents_updated_at
    BEFORE UPDATE ON documents
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();

The separation between document_chunks and chunk_embeddings deserves emphasis. When OpenAI releases a new embedding model, you can generate new embeddings for all existing chunks without re-parsing or re-chunking the source documents. When you need to compare retrieval quality between text-embedding-3-small and text-embedding-3-large, both sets of embeddings coexist and you can run A/B queries against them. The content_hash column on documents lets you skip re-ingestion for files that have not changed — a detail that pays dividends when your nightly sync touches ten thousand documents but only twenty have actually been modified.

14.2 Chunking Strategies That Preserve Semantic Meaning

Chunking is the step that most introductory RAG tutorials get wrong by treating it as a simple split_every_N_characters problem. The text you retrieve during query time needs to be semantically coherent — a chunk that contains half a sentence from one paragraph and the beginning of the next paragraph will embed into a muddled vector that retrieves poorly.

The right chunking strategy depends on document structure. Markdown files have clear heading boundaries. PDFs may have paragraph structure detectable through newlines and indentation. Code documentation often has function signatures and their docstrings as natural units. A production pipeline handles all of these.

# chunkers.py — semantic chunking with multiple strategies
import hashlib
import re
from dataclasses import dataclass, field
from typing import Generator
import tiktoken
from langchain.text_splitter import RecursiveCharacterTextSplitter, MarkdownHeaderTextSplitter
from llama_index.core.node_parser import SentenceSplitter, SemanticSplitterNodeParser
from llama_index.embeddings.openai import OpenAIEmbedding
import fitz  # PyMuPDF
from docx import Document as DocxDocument


@dataclass
class ChunkResult:
    text: str
    chunk_index: int
    token_count: int
    char_start: int
    char_end: int
    page_number: int | None = None
    section_title: str | None = None
    chunk_type: str = "text"
    metadata: dict = field(default_factory=dict)


def count_tokens(text: str, model: str = "text-embedding-3-small") -> int:
    """Accurate token counting using tiktoken."""
    # text-embedding-3-* uses cl100k_base
    enc = tiktoken.get_encoding("cl100k_base")
    return len(enc.encode(text))


class FixedSizeChunker:
    """
    Baseline chunker: fixed token windows with overlap.
    Good for homogeneous plain text with no meaningful structure.
    """

    def __init__(self, chunk_tokens: int = 512, overlap_tokens: int = 64):
        self.chunk_tokens = chunk_tokens
        self.overlap_tokens = overlap_tokens
        self.splitter = RecursiveCharacterTextSplitter.from_tiktoken_encoder(
            model_name="text-embedding-3-small",
            chunk_size=chunk_tokens,
            chunk_overlap=overlap_tokens,
            separators=["\n\n", "\n", ". ", " ", ""],
        )

    def chunk(self, text: str) -> Generator[ChunkResult, None, None]:
        splits = self.splitter.split_text(text)
        char_offset = 0
        for idx, split_text in enumerate(splits):
            # Locate the chunk position in the original text
            char_start = text.find(split_text, max(0, char_offset - 20))
            if char_start == -1:
                char_start = char_offset
            char_end = char_start + len(split_text)
            char_offset = char_end

            yield ChunkResult(
                text=split_text,
                chunk_index=idx,
                token_count=count_tokens(split_text),
                char_start=char_start,
                char_end=char_end,
            )


class MarkdownChunker:
    """
    Structure-aware chunking for Markdown.
    Splits on headers first, then further chunks large sections by size.
    Preserves the heading context in each chunk's metadata.
    """

    def __init__(self, max_chunk_tokens: int = 512, overlap_tokens: int = 64):
        self.max_chunk_tokens = max_chunk_tokens
        self.overlap_tokens = overlap_tokens
        self.header_splitter = MarkdownHeaderTextSplitter(
            headers_to_split_on=[
                ("#", "h1"),
                ("##", "h2"),
                ("###", "h3"),
                ("####", "h4"),
            ],
            strip_headers=False,
        )
        self.size_splitter = RecursiveCharacterTextSplitter.from_tiktoken_encoder(
            model_name="text-embedding-3-small",
            chunk_size=max_chunk_tokens,
            chunk_overlap=overlap_tokens,
        )

    def chunk(self, text: str) -> Generator[ChunkResult, None, None]:
        header_docs = self.header_splitter.split_text(text)
        global_idx = 0

        for doc in header_docs:
            section_heading = " > ".join(
                v for k, v in doc.metadata.items() if k in ("h1", "h2", "h3", "h4") and v
            ) or None

            section_text = doc.page_content.strip()
            if not section_text:
                continue

            # If the section fits within token budget, keep it whole
            if count_tokens(section_text) <= self.max_chunk_tokens:
                yield ChunkResult(
                    text=section_text,
                    chunk_index=global_idx,
                    token_count=count_tokens(section_text),
                    char_start=text.find(section_text),
                    char_end=text.find(section_text) + len(section_text),
                    section_title=section_heading,
                    metadata={"headers": doc.metadata},
                )
                global_idx += 1
            else:
                # Split large sections further
                sub_chunks = self.size_splitter.split_text(section_text)
                for sub_text in sub_chunks:
                    yield ChunkResult(
                        text=sub_text,
                        chunk_index=global_idx,
                        token_count=count_tokens(sub_text),
                        char_start=text.find(sub_text),
                        char_end=text.find(sub_text) + len(sub_text),
                        section_title=section_heading,
                        metadata={"headers": doc.metadata},
                    )
                    global_idx += 1


class PDFChunker:
    """
    PDF-aware chunker using PyMuPDF.
    Extracts text per page, detects paragraph boundaries via blank lines,
    then assembles chunks that respect page and paragraph structure.
    """

    def __init__(self, max_chunk_tokens: int = 512, overlap_tokens: int = 64):
        self.max_chunk_tokens = max_chunk_tokens
        self.overlap_tokens = overlap_tokens

    def chunk(self, pdf_bytes: bytes) -> Generator[ChunkResult, None, None]:
        doc = fitz.open(stream=pdf_bytes, filetype="pdf")
        global_idx = 0
        buffer_text = ""
        buffer_page = 1
        buffer_chars = 0

        for page_num in range(len(doc)):
            page = doc[page_num]
            page_text = page.get_text("text")

            # Split page into paragraphs on double newlines
            paragraphs = re.split(r"\n{2,}", page_text.strip())

            for para in paragraphs:
                para = para.strip()
                if not para or len(para) < 20:  # skip headers/noise lines
                    continue

                para_tokens = count_tokens(para)

                # Would adding this paragraph exceed our budget?
                if buffer_text and count_tokens(buffer_text + "\n\n" + para) > self.max_chunk_tokens:
                    # Emit current buffer as a chunk
                    yield ChunkResult(
                        text=buffer_text,
                        chunk_index=global_idx,
                        token_count=count_tokens(buffer_text),
                        char_start=buffer_chars,
                        char_end=buffer_chars + len(buffer_text),
                        page_number=buffer_page,
                    )
                    global_idx += 1
                    # Start new buffer with overlap: keep last sentence of previous chunk
                    sentences = buffer_text.split(". ")
                    overlap_text = ". ".join(sentences[-2:]) if len(sentences) > 1 else ""
                    buffer_text = (overlap_text + "\n\n" + para).strip() if overlap_text else para
                    buffer_chars += len(buffer_text)
                    buffer_page = page_num + 1
                else:
                    buffer_text = (buffer_text + "\n\n" + para).strip() if buffer_text else para
                    if buffer_page is None:
                        buffer_page = page_num + 1

        # Emit any remaining buffer
        if buffer_text:
            yield ChunkResult(
                text=buffer_text,
                chunk_index=global_idx,
                token_count=count_tokens(buffer_text),
                char_start=buffer_chars,
                char_end=buffer_chars + len(buffer_text),
                page_number=buffer_page,
            )


class SemanticChunker:
    """
    Embedding-based semantic chunking via LlamaIndex.
    Groups sentences that belong together semantically rather than by size.
    More expensive but produces higher quality chunks for dense technical content.
    """

    def __init__(self, openai_api_key: str, breakpoint_percentile: int = 95):
        embed_model = OpenAIEmbedding(
            model="text-embedding-3-small",
            api_key=openai_api_key,
        )
        self.splitter = SemanticSplitterNodeParser(
            buffer_size=1,
            breakpoint_percentile_threshold=breakpoint_percentile,
            embed_model=embed_model,
        )

    def chunk(self, text: str) -> Generator[ChunkResult, None, None]:
        from llama_index.core import Document as LIDocument
        doc = LIDocument(text=text)
        nodes = self.splitter.get_nodes_from_documents([doc])

        for idx, node in enumerate(nodes):
            chunk_text = node.get_content()
            yield ChunkResult(
                text=chunk_text,
                chunk_index=idx,
                token_count=count_tokens(chunk_text),
                char_start=node.start_char_idx or 0,
                char_end=node.end_char_idx or len(chunk_text),
            )


def select_chunker(source_type: str, settings: dict):
    """Factory that picks the right chunker based on document type and collection settings."""
    chunk_tokens = settings.get("chunk_size", 512)
    overlap_tokens = settings.get("overlap", 64)

    if source_type == "markdown":
        return MarkdownChunker(max_chunk_tokens=chunk_tokens, overlap_tokens=overlap_tokens)
    elif source_type == "pdf":
        return PDFChunker(max_chunk_tokens=chunk_tokens, overlap_tokens=overlap_tokens)
    else:
        return FixedSizeChunker(chunk_tokens=chunk_tokens, overlap_tokens=overlap_tokens)

The semantic chunker via LlamaIndex uses actual embedding similarity to find breakpoints — sentences whose embeddings diverge sharply from what came before represent natural topical boundaries. This approach works well for long-form technical documentation where paragraphs shift between related but distinct concepts. The cost is additional embedding API calls during ingestion, so reserve it for corpora where retrieval quality justifies the extra latency.

14.3 Generating and Storing Embeddings at Scale

With chunks in hand, the embedding step has two distinct challenges: rate limits from embedding APIs and the need to batch efficiently. Sending one API call per chunk will hit rate limits on any non-trivial corpus and waste significant time on network round trips. Batching chunks into groups of 100–500 tokens saturates throughput while staying within API constraints.
