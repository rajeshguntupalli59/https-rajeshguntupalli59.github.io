Chapter 24: Audit Logging Every AI Action in PostgreSQL

Every AI action your application takes — a prompt sent to Claude, a vector search against embeddings, a tool call made by an agent, a decision to retrieve or skip a document — is an event worth recording. Not for compliance theater, but because AI systems fail in subtle ways that only become visible in hindsight. A RAG pipeline that quietly started returning worse answers after an embedding model update. An agent that began making tool calls in the wrong order. A user who received a harmful response that nobody can now reconstruct. Without structured audit logs stored in PostgreSQL, you are flying blind.

This chapter builds a complete audit logging system for AI-native applications. You will design schemas that capture the full lifecycle of AI interactions, instrument Python application code to write audit records automatically, use PostgreSQL triggers and JSONB to make audit data queryable without schema migrations, and build production patterns for high-volume AI workloads where logging cannot be the bottleneck.

---

Why AI Audit Logging Is Different From Application Logging

Traditional application logging captures errors and system events. AI audit logging must capture *reasoning artifacts* — the inputs that drove a decision, the retrieved context that shaped a response, the token counts that drove costs, and the latency breakdown that explains why a response took seven seconds. These are not error logs. They are a permanent record of what your AI system knew, what it did, and what it said.

The differences matter for schema design:

Inputs are long and structured. A RAG prompt includes a user message, retrieved document chunks, conversation history, and a system prompt. Storing this as a plain text log line makes it unsearchable.

Outputs have metadata. LLM responses carry finish reasons, token usage, model versions, and confidence indicators. An agent tool call carries function names, arguments, and return values. These are first-class fields, not afterthoughts.

AI actions compose. A single user request may trigger an embedding lookup, three document retrievals, one LLM call, two tool calls, and a final synthesis call. These must be linked with a shared trace ID so you can reconstruct the full chain.

Failures are semantic, not just technical. The LLM returned HTTP 200 but hallucinated a citation. The vector search succeeded but returned semantically irrelevant results. Your audit system must capture enough information to detect these failures in analysis, not just log that the call completed.

PostgreSQL is uniquely suited for this. JSONB lets you store the variable-structure payload of any AI action while still indexing and querying specific fields. Row-level security lets you expose audit data to different stakeholders without separate databases. Foreign keys tie audit records to your user, session, and document tables. Window functions let you analyze token spend per user per day without exporting to a data warehouse.

---

Schema Design for Complete AI Audit Coverage

The core schema uses four tables: one for AI sessions, one for individual AI actions within sessions, one for retrieved context, and one for agent tool calls. Everything is linked by a session ID and a trace ID.

-- Core audit schema for AI-native applications
-- Run as a migration in your application database

CREATE SCHEMA IF NOT EXISTS ai_audit;

-- Sessions represent one logical user interaction (one chat turn, one agent run)
CREATE TABLE ai_audit.sessions (
    session_id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    external_session_id TEXT,                          -- your app's session/conversation ID
    user_id             TEXT NOT NULL,
    application_name    TEXT NOT NULL,                 -- 'chatbot', 'rag_search', 'agent_runner'
    environment         TEXT NOT NULL DEFAULT 'production',
    started_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    completed_at        TIMESTAMPTZ,
    total_input_tokens  INTEGER DEFAULT 0,
    total_output_tokens INTEGER DEFAULT 0,
    total_latency_ms    INTEGER,
    session_metadata    JSONB DEFAULT '{}',
    tags                TEXT[] DEFAULT '{}'
);

CREATE INDEX idx_sessions_user_id ON ai_audit.sessions (user_id, started_at DESC);
CREATE INDEX idx_sessions_app_name ON ai_audit.sessions (application_name, started_at DESC);
CREATE INDEX idx_sessions_metadata ON ai_audit.sessions USING GIN (session_metadata);

-- Individual AI actions: LLM calls, embedding calls, vector searches
CREATE TABLE ai_audit.actions (
    action_id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id          UUID NOT NULL REFERENCES ai_audit.sessions(session_id),
    parent_action_id    UUID REFERENCES ai_audit.actions(action_id),  -- for nested calls
    trace_id            TEXT NOT NULL,                 -- spans across services
    action_type         TEXT NOT NULL,                 -- 'llm_call', 'embedding', 'vector_search', 'rerank'
    action_sequence     INTEGER NOT NULL DEFAULT 0,   -- order within session
    
    -- Provider details
    provider            TEXT,                          -- 'anthropic', 'openai', 'cohere'
    model               TEXT,                          -- 'claude-3-5-sonnet-20241022'
    model_version       TEXT,
    
    -- Input capture
    prompt_text         TEXT,                          -- full prompt for LLM calls
    prompt_tokens       INTEGER,
    system_prompt_hash  TEXT,                          -- SHA256 of system prompt (don't store full)
    input_payload       JSONB DEFAULT '{}',            -- structured input for non-LLM calls
    
    -- Output capture
    response_text       TEXT,                          -- full response text
    completion_tokens   INTEGER,
    total_tokens        INTEGER,
    finish_reason       TEXT,                          -- 'end_turn', 'max_tokens', 'stop_sequence'
    output_payload      JSONB DEFAULT '{}',            -- structured output for non-LLM calls
    
    -- Performance
    started_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    completed_at        TIMESTAMPTZ,
    latency_ms          INTEGER GENERATED ALWAYS AS (
                            EXTRACT(EPOCH FROM (completed_at - started_at)) * 1000
                        ) STORED,
    
    -- Cost tracking (in USD, stored as numeric to avoid float precision issues)
    estimated_cost_usd  NUMERIC(10, 6),
    
    -- Status
    status              TEXT NOT NULL DEFAULT 'started', -- 'started', 'completed', 'failed', 'cancelled'
    error_code          TEXT,
    error_message       TEXT,
    
    -- Safety and quality signals
    safety_flags        TEXT[] DEFAULT '{}',
    quality_score       NUMERIC(3,2),                  -- 0.00 to 1.00 if available
    
    action_metadata     JSONB DEFAULT '{}'
);

CREATE INDEX idx_actions_session_id ON ai_audit.actions (session_id, action_sequence);
CREATE INDEX idx_actions_trace_id ON ai_audit.actions (trace_id);
CREATE INDEX idx_actions_type_status ON ai_audit.actions (action_type, status, started_at DESC);
CREATE INDEX idx_actions_model ON ai_audit.actions (provider, model, started_at DESC);
CREATE INDEX idx_actions_input_payload ON ai_audit.actions USING GIN (input_payload);
CREATE INDEX idx_actions_output_payload ON ai_audit.actions USING GIN (output_payload);

-- Retrieved context: documents/chunks fetched during RAG
CREATE TABLE ai_audit.retrieved_context (
    context_id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    action_id           UUID NOT NULL REFERENCES ai_audit.actions(action_id),
    session_id          UUID NOT NULL REFERENCES ai_audit.sessions(session_id),
    
    -- Source document reference
    document_id         TEXT,                          -- FK to your documents table (text for flexibility)
    chunk_id            TEXT,
    chunk_text          TEXT,                          -- actual text retrieved
    chunk_text_hash     TEXT,                          -- SHA256 for dedup analysis
    
    -- Search metadata
    search_query        TEXT,
    search_query_embedding_model TEXT,
    similarity_score    NUMERIC(6,5),                  -- cosine similarity, 0 to 1
    rank_position       INTEGER,                       -- position in result list
    was_used_in_prompt  BOOLEAN DEFAULT TRUE,          -- after reranking, was this included?
    
    -- Document metadata at retrieval time (snapshot)
    document_metadata   JSONB DEFAULT '{}',
    
    retrieved_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_context_action_id ON ai_audit.retrieved_context (action_id);
CREATE INDEX idx_context_session_id ON ai_audit.retrieved_context (session_id, retrieved_at DESC);
CREATE INDEX idx_context_document_id ON ai_audit.retrieved_context (document_id);
CREATE INDEX idx_context_similarity ON ai_audit.retrieved_context (similarity_score DESC);

-- Tool calls made by agents
CREATE TABLE ai_audit.tool_calls (
    tool_call_id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    action_id           UUID NOT NULL REFERENCES ai_audit.actions(action_id),
    session_id          UUID NOT NULL REFERENCES ai_audit.sessions(session_id),
    
    tool_name           TEXT NOT NULL,
    tool_call_sequence  INTEGER NOT NULL DEFAULT 0,
    
    -- Input and output
    tool_input          JSONB NOT NULL DEFAULT '{}',
    tool_output         JSONB DEFAULT '{}',
    tool_output_text    TEXT,
    
    -- Execution
    started_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    completed_at        TIMESTAMPTZ,
    latency_ms          INTEGER GENERATED ALWAYS AS (
                            EXTRACT(EPOCH FROM (completed_at - started_at)) * 1000
                        ) STORED,
    
    status              TEXT NOT NULL DEFAULT 'started',
    error_message       TEXT,
    
    -- Safety: was this tool call approved, auto-approved, or blocked?
    approval_status     TEXT DEFAULT 'auto_approved', -- 'auto_approved', 'human_approved', 'blocked'
    approved_by         TEXT,
    
    tool_metadata       JSONB DEFAULT '{}'
);

CREATE INDEX idx_tool_calls_action_id ON ai_audit.tool_calls (action_id);
CREATE INDEX idx_tool_calls_session_id ON ai_audit.tool_calls (session_id, started_at DESC);
CREATE INDEX idx_tool_calls_tool_name ON ai_audit.tool_calls (tool_name, started_at DESC);

-- Materialized view for daily cost and usage reporting
CREATE MATERIALIZED VIEW ai_audit.daily_usage_summary AS
SELECT
    date_trunc('day', a.started_at) AS usage_date,
    s.application_name,
    s.environment,
    a.provider,
    a.model,
    a.action_type,
    COUNT(*) AS action_count,
    COUNT(*) FILTER (WHERE a.status = 'completed') AS completed_count,
    COUNT(*) FILTER (WHERE a.status = 'failed') AS failed_count,
    SUM(a.total_tokens) AS total_tokens,
    SUM(a.estimated_cost_usd) AS total_cost_usd,
    AVG(a.latency_ms) AS avg_latency_ms,
    PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY a.latency_ms) AS p95_latency_ms,
    COUNT(DISTINCT s.user_id) AS unique_users
FROM ai_audit.actions a
JOIN ai_audit.sessions s ON a.session_id = s.session_id
GROUP BY 1, 2, 3, 4, 5, 6
WITH DATA;

CREATE UNIQUE INDEX ON ai_audit.daily_usage_summary (
    usage_date, application_name, environment, provider, model, action_type
);

The parent_action_id self-reference in actions is worth explaining. When an agent makes an LLM call that triggers three tool calls, each of which makes its own sub-calls, you get a tree. The parent_action_id lets you reconstruct that tree with a recursive CTE. The action_sequence gives you ordering within a flat view. Both are necessary.
