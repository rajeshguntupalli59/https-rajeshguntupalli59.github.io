Chapter 23: AlloyDB Omni: Running Google's Database On-Premises

AlloyDB Omni brings Google's columnar engine, adaptive query optimization, and PostgreSQL-compatible runtime to hardware you control — your on-premises servers, your colocation cage, or your edge deployment. For DBAs who have spent years tuning PostgreSQL instances but need the performance headroom of AlloyDB without migrating workloads into Google Cloud, Omni represents a genuinely different proposition: a managed-engine experience running in a container on your own infrastructure. This chapter covers deploying AlloyDB Omni from scratch, understanding what the columnar engine actually does under the hood, connecting Omni to Google Cloud services for observability and management, and building production patterns that keep the deployment maintainable over time. If you have worked with AlloyDB in Google Cloud (covered in Chapter 22), Omni will feel familiar in its SQL layer but distinctly different in its operational model.

---

23.1 What AlloyDB Omni Actually Is — and What It Is Not

Before deploying anything, you need a clear mental model of what Omni provides versus what you get from a self-compiled PostgreSQL binary or a full AlloyDB instance in Google Cloud.

The core architecture

AlloyDB Omni ships as a container image (gcr.io/alloydb-omni/alloydb-omni) that wraps a modified PostgreSQL 15 engine. The modifications include three components that differentiate it from vanilla Postgres:

1. The columnar engine — an in-memory columnar store that automatically populates based on query patterns. Google's documentation calls this the "columnar cache." It stores frequently accessed column data in a compressed, vectorized format alongside the standard row-based heap. You do not manage this explicitly; the engine decides what to populate based on workload.

2. The adaptive query optimizer — an overlay on top of the standard PostgreSQL planner that uses statistics and runtime feedback to choose between row and columnar execution paths. Unlike a traditional hint-based approach, this happens automatically per query.

3. The AlloyDB AI extensions — google_ml_integration and pgvector, pre-installed and configured for embedding generation calls that can reach out to Vertex AI endpoints if you configure a service account.

What Omni is not

Omni is not a fully managed service. You own the container runtime, the host OS, the storage subsystem, backup scheduling, and HA failover. Google does not patch it for you. You pull new container images when updates release, test them, and roll them out on your own schedule. This is the fundamental trade-off: you get the engine, not the platform.

Omni also does not replicate the AlloyDB storage layer from Google Cloud. In GCP, AlloyDB uses a distributed, disaggregated storage system with six-way replication across two regions. In Omni, you get standard PostgreSQL WAL-based storage on whatever block device you mount into the container. The columnar engine still works, but the durability model is your problem to design.

Comparing Omni to PostgreSQL

| Feature | Vanilla PostgreSQL 15 | AlloyDB Omni |

|---|---|---|

| Row storage | ✓ | ✓ |

| Columnar engine | ✗ | ✓ (automatic) |

| pgvector extension | Manual install | Pre-installed |

| google_ml_integration | ✗ | ✓ |

| GCP Observability integration | Manual | Built-in agent |

| License | PostgreSQL License | Google ToS |

| Patching | Your binary | Pull new container |

| Max column store memory | N/A | Configurable, default 1 GB |

The PostgreSQL ToS difference matters for production legal review. Omni is licensed under Google's terms, not the PostgreSQL License. Run that past your legal team before deploying in regulated environments.

23.2 Deploying AlloyDB Omni on Bare Metal and Virtual Machines

Omni runs on any container runtime that supports Docker or Podman. In practice, most production deployments use either Docker Engine on Linux or Kubernetes with the AlloyDB Omni Operator. This section covers both paths.

Prerequisites

Minimum host specifications for a production deployment:

  • → 8 vCPUs (16 recommended for OLAP workloads using the columnar engine)
  • → 32 GB RAM (the columnar engine uses a configurable slice of this)
  • → Local NVMe or SAN storage with at least 3,000 IOPS sustained
  • → Linux kernel 5.4 or newer (tested on RHEL 8/9, Ubuntu 20.04/22.04)
  • → Docker Engine 20.10+ or Podman 4.0+

    • Standalone container deployment

    # Pull the AlloyDB Omni image
# You need to authenticate with Google Container Registry first
gcloud auth configure-docker gcr.io

docker pull gcr.io/alloydb-omni/alloydb-omni:latest

# Create persistent storage directories on the host
# Use a dedicated mount point backed by your fastest storage
mkdir -p /alloydb/data /alloydb/logs /alloydb/tmp

# Deploy the container with production settings
# POSTGRES_PASSWORD sets the initial superuser password
# ALLOYDB_ENABLE_COLUMNAR_ENGINE enables the in-memory columnar cache
docker run -d \
  --name alloydb-omni \
  --restart unless-stopped \
  -e POSTGRES_PASSWORD='YourSecureP@ssword123' \
  -e ALLOYDB_ENABLE_COLUMNAR_ENGINE=true \
  -e ALLOYDB_COLUMNAR_MEMORY_SIZE_MB=8192 \
  -p 5432:5432 \
  -v /alloydb/data:/var/lib/postgresql/data \
  -v /alloydb/logs:/var/log/postgresql \
  --shm-size=2g \
  --cpus=16 \
  --memory=64g \
  gcr.io/alloydb-omni/alloydb-omni:latest

# Verify the container started cleanly
docker logs alloydb-omni | tail -30

# Connect and confirm AlloyDB extensions are present
docker exec -it alloydb-omni psql -U postgres -c "\dx"

    The --shm-size parameter is easy to forget and causes immediate performance problems. PostgreSQL uses shared memory for the buffer pool and inter-process communication. If you leave it at the Docker default of 64 MB, you will see errors in the PostgreSQL log about shared memory allocation failing when you try to set shared_buffers to a reasonable value.

    Configuring the columnar engine

    Once the container is running, you control the columnar engine through GUC parameters that AlloyDB Omni adds to the standard PostgreSQL parameter namespace:

    -- Connect as postgres superuser and check columnar engine status
SELECT name, setting, unit, short_desc
FROM pg_settings
WHERE name LIKE 'alloydb%'
ORDER BY name;

-- Key parameters to understand:
-- alloydb.enable_columnar_engine          (on/off)
-- alloydb.columnar_memory_size_mb         (default: 1024)
-- alloydb.auto_columnar_scan_threshold    (rows, default: 10000)
-- alloydb.columnar_engine_mode            (auto, manual)

-- Check what the columnar engine has populated
-- This view shows which relations have columnar data
SELECT
    schemaname,
    tablename,
    pg_size_pretty(columnar_size_bytes) AS columnar_size,
    columnar_row_count,
    last_refresh_time
FROM alloydb_columnar_info
ORDER BY columnar_size_bytes DESC;

-- Force a specific table into the columnar cache explicitly
-- Useful during initial load before query patterns establish themselves
ALTER TABLE order_line_items SET (columnar = on);

-- Check actual query execution to see if columnar path was chosen
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT
    product_id,
    SUM(quantity) AS total_qty,
    SUM(unit_price * quantity) AS revenue
FROM order_line_items
WHERE order_date >= '2024-01-01'
GROUP BY product_id;
-- Look for "Columnar Scan" nodes in the plan output
-- If you see "Seq Scan" instead, the table may not yet be in the columnar cache
-- or the row count is below the threshold

    The auto_columnar_scan_threshold parameter is the one most DBAs misconfigure. If a table has fewer rows than this threshold, the planner will not consider the columnar path even if the table is marked for columnar storage. Set it too high and small-but-wide analytical tables miss out. Set it too low and you waste columnar memory on tables too small to benefit. A starting value of 50,000 rows works well for most mixed OLTP/OLAP environments.

    Kubernetes deployment with the AlloyDB Omni Operator

    For environments running Kubernetes on-premises (OpenShift, Rancher, vanilla kubeadm clusters), Google provides the AlloyDB Omni Operator, which adds CRDs for DBCluster resources:

    # Install the AlloyDB Omni Operator using Helm
helm repo add alloydb-omni-operator \
  https://storage.googleapis.com/alloydb-omni-operator

helm repo update

helm install alloydb-omni-operator \
  alloydb-omni-operator/alloydb-omni-operator \
  --namespace alloydb-omni-system \
  --create-namespace \
  --set image.tag=1.2.0

# Verify the operator deployed successfully
kubectl get pods -n alloydb-omni-system
kubectl get crds | grep alloydb
    # production-dbcluster.yaml
# AlloyDB Omni DBCluster manifest for Kubernetes
apiVersion: alloydbomni.dbadmin.goog/v1
kind: DBCluster
metadata:
  name: prod-analytics-cluster
  namespace: database
spec:
  databaseVersion: "15.5"
  primarySpec:
    adminUser:
      passwordRef:
        name: alloydb-admin-secret
        key: password
    resources:
      cpu: "16"
      memory: 64Gi
      disks:
        - name: DataDisk
          size: 2Ti
          storageClass: fast-nvme
    features:
      enableColumnarEngine: true
      columnarEngineSizeGB: 16
    parameters:
      work_mem: "64MB"
      max_connections: "500"
      checkpoint_completion_target: "0.9"
      wal_level: "logical"
  allowExternalIncomingTraffic: false
    # Apply the manifest and watch the cluster come up
kubectl apply -f production-dbcluster.yaml
kubectl get dbcluster prod-analytics-cluster -n database -w

# Get the service endpoint once Ready
kubectl get svc -n database | grep prod-analytics

    23.3 Connecting AlloyDB Omni to Google Cloud Services

    Running Omni on-premises does not mean operating in isolation. Google designed Omni to optionally connect back to Google Cloud for observability, AI feature access, and fleet management through the AlloyDB Omni management API. This section covers those integrations and when to use them.

    Google Cloud Observability integration

    Omni can ship logs and metrics to Cloud Logging and Cloud Monitoring through the built-in observability agent. This is the fastest way to get production-grade monitoring without building a Prometheus stack from scratch.

    # Create a Google Cloud service account for the Omni observability agent
# Run this from a machine with gcloud access to your GCP project
gcloud iam service-accounts create alloydb-omni-agent \
  --project=your-project-id \
  --description="AlloyDB Omni observability and ML integration agent" \
  --display-name="AlloyDB Omni Agent"

# Grant the minimum required roles
gcloud projects add-iam-policy-binding your-project-id \
  --member="serviceAccount:alloydb-omni-agent@your-project-id.iam.gserviceaccount.com" \
  --role="roles/monitoring.metricWriter"

gcloud projects add-iam-policy-binding your-project-id \
  --member="serviceAccount:alloydb-omni-agent@your-project-id.iam.gserviceaccount.com" \
  --role="roles/logging.logWriter"

gcloud projects add-iam-policy-binding your-project-id \
  --member="serviceAccount:alloydb-omni-agent@your-project-id.iam.gserviceaccount.com" \
  --role="roles/aiplatform.user"

# Create and download a JSON key for the service account
# Store this securely — treat it like a root password
gcloud iam service-accounts keys create /etc/alloydb/gcp-credentials.json \
  --iam-account=alloydb-omni-agent@your-project-id.iam.gserviceaccount.com

chmod 600 /etc/alloydb/gcp-credentials.json
    # Redeploy the container with GCP credentials mounted
# The GOOGLE_APPLICATION_CREDENTIALS env var activates the observability agent
docker stop alloydb-omni && docker rm alloydb-omni

docker run -d \
  --name alloydb-omni \
  --restart unless-stopped \
  -e POSTGRES_PASSWORD='YourSecureP@ssword123' \
  -e ALLOYDB_ENABLE_COLUMNAR_ENGINE=true \
  -e ALLOYDB_COLUMNAR_MEMORY_SIZE_MB=8192 \
  -e GOOGLE_APPLICATION_CREDENTIALS=/etc/alloydb/gcp-credentials.json \
  -e GOOGLE_CLOUD_PROJECT=your-project-id \
  -e ALLOYDB_OBSERVABILITY_ENABLED=true \
  -p 5432:5432 \
  -v /alloydb/data:/var/lib/postgresql/data \
  -v /alloydb/logs:/var/log/postgresql \
  -v /etc/alloydb:/etc/alloydb:ro \
  --shm-size=2g \
  --cpus=16 \
  --memory=64g \
  gcr.io/alloydb-omni/alloydb-omni:latest

# After 2-3 minutes, verify metrics appear in Cloud Monitoring
gcloud monitoring metrics list \
  --filter="metric.type=starts_with(\"alloydb.googleapis.com\")" \
  --project=your-project-id | head -30

    Once connected, you can build Cloud Monitoring dashboards that show AlloyDB Omni metrics — active connections, cache hit ratios, columnar engine memory usage, transaction throughput — alongside metrics from your Cloud SQL and AlloyDB instances in GCP. This unified view is one of the practical reasons to wire Omni back to Google Cloud even when you prefer keeping data on-premises.

    Vertex AI integration for embedding generation

    The google_ml_integration extension in Omni can call Vertex AI to generate text embeddings from inside SQL queries. This is the same capability available in AlloyDB in GCP, now runnable from your on-premises database. The service account you created above needs the roles/aiplatform.user role for this to work.

    -- Enable the ML integration extension
CREATE EXTENSION IF NOT EXISTS google_ml_integration;

-- Configure the Vertex AI endpoint
-- Replace the region and model with your Vertex AI configuration
SELECT google_ml.create_model(
    model_id => 'text-embedding-005',
    model_provider => 'google',
    model_type => 'text_embedding',
    model_endpoint =>
        'https://us-central1-aiplatform.googleapis.com/v1/projects/your-project-id/locations/us-central1/publishers/google/models/text-embedding-005',
    model_qualified_name => 'publishers/google/models/text-embedding-005'
);

-- Test embedding generation
-- This call goes from your on-premises Omni instance to Vertex AI over HTTPS
SELECT google_ml.embedding(
    'text-embedding-005',
    'AlloyDB Omni columnar engine performance benchmark'
)::vector(768);

-- Practical use: generate embeddings for a product catalog
-- and store them for similarity search
ALTER TABLE product_catalog
ADD COLUMN IF NOT EXISTS description_embedding vector(768);

UPDATE product_catalog
SET description_embedding = google_ml.embedding(
    'text-embedding-005',
    product_description
)::vector(768)
WHERE description_embedding IS NULL
  AND product_description IS NOT NULL;

-- Create an IVFFlat index for approximate nearest neighbor search
CREATE INDEX idx_product_embeddings
ON product_catalog
USING ivfflat (description_embedding vector_cosine_ops)
WITH (lists = 100);

    The network path here matters for latency and for compliance. Every embedding call in a SQL query exits your on-premises network and reaches Google's API endpoints. If your data classification policy prohibits row data leaving the facility, passing product descriptions to Vertex AI may trigger a compliance review. Know what data you are embedding before you build this into production query patterns.

    23.4 Backup, HA, and the Operational Gap

    This is where the honest conversation about Omni needs to happen. In AlloyDB in GCP, Google handles backup, point-in-time recovery, HA failover, and storage durability automatically. In Omni, none of that is included. You are running a PostgreSQL-compatible engine in a container, and backup and HA are entirely your design responsibility.

    Backup strategy

    For Omni deployments, you have three practical backup approaches:

    1. pg_basebackup — same as vanilla PostgreSQL, works reliably

    2. WAL archiving to Cloud Storage — gives you PITR with Google Cloud as the archive destination

    3. Volume snapshots — snapshot the underlying block device at the host level

    WAL archiving to Cloud Storage is the approach that best preserves the Google Cloud integration story:

    # Create a Cloud Storage bucket for WAL archiving
gsutil mb -l us-central1 gs://your-company-alloydb-omni-wal

# Set lifecycle policy to expire old WAL segments after 30 days
cat > /tmp/wal-lifecycle.json << 'EOF'
{
  "lifecycle": {
    "rule": [
      {
        "action": {"type": "Delete"},
        "condition": {"age": 30}
      }
    ]
  }
}
EOF

gsutil lifecycle set /tmp/wal-lifecycle.json \
  gs://your-company-alloydb-omni-wal

# Inside the container, configure postgresql.conf for WAL archiving
# AlloyDB Omni respects all standard PostgreSQL archive parameters
docker exec -it alloydb-omni psql -U postgres << 'EOF'
ALTER SYSTEM SET wal_level = 'replica';
ALTER SYSTEM SET archive_mode = 'on';
ALTER SYSTEM SET archive_command =
  'gcloud storage cp %p gs://your-company-alloydb-omni-wal/%f';
ALTER SYSTEM SET archive_timeout = '300';
SELECT pg_reload_conf();
EOF

# Restart required for wal_level and archive_mode changes
docker restart alloydb-omni

# Take a base backup to Cloud Storage to anchor your PITR chain
docker exec alloydb-omni pg_basebackup \
  -U postgres \
  -D /tmp/basebackup \
  -Ft \
  -z \
  --checkpoint=fast \
  --wal-method=stream

# Upload the base backup
tar czf - /tmp/basebackup | \
  gcloud storage cp - \
    gs://your-company-alloydb-omni-wal/basebackups/$(date +%Y%m%d-%H%M%S).tar.gz

    High availability with Patroni

    The most production-proven HA approach for Omni is running Patroni to orchestrate multiple container instances. You already know how Patroni works with vanilla PostgreSQL; the configuration for Omni is identical at the Patroni level because Omni exposes the same replication API as PostgreSQL.

    # /etc/patroni/patroni.yml for AlloyDB Omni primary node
scope: alloydb-omni-cluster
namespace: /db/
name: omni-node-01

restapi:
  listen: 0.0.0.0:8008
  connect_address: 10.0.1.10:8008

etcd3:
  hosts:
    - 10.0.1.20:2379
    - 10.0.1.21:2379
    - 10.0.1.22:2379

bootstrap:
  dcs:
    ttl: 30
    loop_wait: 10
    retry_timeout: 10
    maximum_lag_on_failover: 1048576  # 1 MB — adjust for your RPO tolerance
  initdb:
    - encoding: UTF8
    - data-checksums

postgresql:
  listen: 0.0.0.0:5432
  connect_address: 10.0.1.10:5432
  data_dir: /alloydb/data
  bin_dir: /usr/lib/postgresql/15/bin
  parameters:
    max_connections: 500
    shared_buffers: 16GB
    work_mem: 64MB
    wal_level: replica
    max_wal_senders: 10
    max_replication_slots: 10
    hot_standby: on
    # AlloyDB Omni specific
    alloydb.enable_columnar_engine: on
    alloydb.columnar_memory_size_mb: 8192

tags:
  nofailover: false
  noloadbalance: false
  clonefrom: false
  nosync: false

    One thing to verify before committing to this architecture: the columnar engine state is not replicated. The columnar cache on the primary gets rebuilt from scratch on a standby that promotes to primary. The first 15–30 minutes after a failover will have degraded analytical query performance until the new primary repopulates the columnar cache from query traffic. This is acceptable in most cases but needs to be communicated to application teams as expected post-failover behavior.

    23.5 Performance Tuning the Columnar Engine

    The columnar engine in AlloyDB Omni is the feature that most DBAs want to exploit first, and the place where the most time gets wasted chasing configuration parameters that do not move the needle as much as query and schema design.

    What the columnar engine actually accelerates

    The columnar engine provides meaningful speedup for queries that:

  • → Scan a large number of rows but access only a subset of columns
  • → Perform aggregations (SUM, AVG, COUNT) over wide tables
  • → Filter on low-cardinality columns where column statistics enable fast skipping

    • It provides minimal or no benefit for:

  • → Point lookups by primary key (these use the row store via index)
  • → Heavy write workloads (columnar cache is read-optimized)
  • → Queries on tables with fewer than 50,000 rows (below the default threshold)
  • → Joins where both sides are small and fit in `work_mem`
