Chapter 5: When to Use pgvector vs a Dedicated Vector Database

The question practitioners ask most often when entering the vector database space is not how to build embeddings—it is where to store them. pgvector brings vector search into PostgreSQL, the database most teams already operate. Dedicated vector databases like Pinecone, Weaviate, Qdrant, and Milvus were purpose-built for embedding workloads from the ground up. Both camps have vocal advocates, and the honest answer is that both are correct under different conditions. This chapter gives you a framework for making that decision with precision—based on scale thresholds, operational posture, query patterns, and what your application actually needs at production load—not based on hype or vendor preference.

---

5.1 Understanding What You Are Actually Choosing Between

Before comparing tools, clarify what the choice involves architecturally. pgvector is a PostgreSQL extension. It adds a vector data type, two index types (IVFFlat and HNSW), and a family of distance operators to a relational engine that has been in production for over thirty years. The value proposition is transactional consistency, JOINs, ACID guarantees, and a single operational footprint. You already have monitoring, backups, replication, connection pooling, and team expertise around PostgreSQL. Adding pgvector means adding one extension, not one more managed service.

Dedicated vector databases are purpose-built systems where the entire storage engine, indexing layer, and query planner are optimized around one problem: finding approximate nearest neighbors at speed across large, high-dimensional vector collections. They make different engineering tradeoffs—often sacrificing ACID transactions, complex relational queries, and familiar tooling in exchange for throughput, recall at scale, and operational features like automatic shard rebalancing, real-time index updates without degradation, and native multi-tenancy patterns.

Think of it the way a DBA thinks about choosing between a relational database and a columnar analytics store. Both store data. Both answer queries. The choice depends on access patterns, scale, and what you are willing to give up on each side.

The relevant comparison axes are:

| Dimension | pgvector | Dedicated Vector DB |

|---|---|---|

| Vector index types | IVFFlat, HNSW | HNSW, DiskANN, IVF, quantization variants |

| Max practical vector count | ~5–50M (hardware-dependent) | Billions (Pinecone, Milvus) |

| ANN recall at scale | Good up to ~10M | Tunable at 100M+ |

| Transactional consistency | Full ACID | Eventual / limited |

| Hybrid queries (vector + relational) | Native SQL JOINs | Limited or workaround-based |

| Operational complexity | Low (already running PG) | Medium to high (new service) |

| Managed cloud options | Supabase, Neon, RDS, AlloyDB | Pinecone, Weaviate Cloud, Zilliz |

5.2 pgvector in Practice: Setup, Indexing, and Performance Baselines

Let us establish pgvector's operational baseline before making comparisons. The following setup reflects a realistic production schema for a document retrieval application storing OpenAI text-embedding-3-small embeddings (1536 dimensions).

-- PostgreSQL / pgvector
-- Install extension (requires PostgreSQL 14+ and pgvector 0.5+)
CREATE EXTENSION IF NOT EXISTS vector;

-- Document chunks table with embeddings
CREATE TABLE document_chunks (
    id          BIGSERIAL PRIMARY KEY,
    document_id BIGINT NOT NULL REFERENCES documents(id) ON DELETE CASCADE,
    chunk_index INTEGER NOT NULL,
    content     TEXT NOT NULL,
    metadata    JSONB,
    embedding   vector(1536),
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- Scalar filtering index for common metadata queries
CREATE INDEX idx_chunks_document_id ON document_chunks(document_id);
CREATE INDEX idx_chunks_metadata_source 
    ON document_chunks USING gin(metadata jsonb_path_ops);

-- HNSW index for approximate nearest neighbor search
-- m=16, ef_construction=64 is a reasonable starting point
-- Increase m to 32-64 for higher recall at cost of memory and build time
CREATE INDEX idx_chunks_embedding_hnsw 
    ON document_chunks 
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- At query time, set ef_search for recall/speed tradeoff
-- ef_search=40 is pgvector default; raise to 100-200 for higher recall
SET hnsw.ef_search = 100;

-- Semantic search query with pre-filter and JOIN
-- This is where pgvector shines: native SQL composition
SELECT 
    dc.id,
    dc.content,
    dc.metadata->>'source' AS source,
    d.title,
    1 - (dc.embedding <=> $1::vector) AS cosine_similarity
FROM document_chunks dc
JOIN documents d ON d.id = dc.document_id
WHERE 
    dc.metadata->>'tenant_id' = $2
    AND dc.created_at > NOW() - INTERVAL '90 days'
ORDER BY dc.embedding <=> $1::vector
LIMIT 10;

The <=> operator computes cosine distance. pgvector also supports <-> (L2/Euclidean) and <#> (negative inner product for dot product similarity). Choose based on how your embedding model was trained—OpenAI recommends cosine similarity for their models.

Now connect an embedding pipeline in Python:

# Python
import os
import psycopg2
from psycopg2.extras import execute_values
from openai import OpenAI

openai_client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

DB_DSN = os.environ["PG_DSN"]  
# e.g. "postgresql://user:pass@localhost:5432/mydb"

def get_embedding(text: str, model: str = "text-embedding-3-small") -> list[float]:
    response = openai_client.embeddings.create(input=text, model=model)
    return response.data[0].embedding

def upsert_chunks(chunks: list[dict], conn) -> None:
    """
    chunks: list of dicts with keys: document_id, chunk_index, content, metadata
    """
    embeddings = [
        get_embedding(chunk["content"]) for chunk in chunks
    ]
    rows = [
        (
            chunk["document_id"],
            chunk["chunk_index"],
            chunk["content"],
            psycopg2.extras.Json(chunk["metadata"]),
            embedding,
        )
        for chunk, embedding in zip(chunks, embeddings)
    ]
    with conn.cursor() as cur:
        execute_values(
            cur,
            """
            INSERT INTO document_chunks
                (document_id, chunk_index, content, metadata, embedding)
            VALUES %s
            ON CONFLICT (document_id, chunk_index) 
            DO UPDATE SET
                content   = EXCLUDED.content,
                metadata  = EXCLUDED.metadata,
                embedding = EXCLUDED.embedding
            """,
            rows,
            template="(%s, %s, %s, %s, %s::vector)",
        )
    conn.commit()

def semantic_search(
    query: str,
    tenant_id: str,
    top_k: int = 10,
    ef_search: int = 100,
) -> list[dict]:
    query_embedding = get_embedding(query)
    
    with psycopg2.connect(DB_DSN) as conn:
        with conn.cursor() as cur:
            cur.execute(f"SET hnsw.ef_search = {ef_search}")
            cur.execute(
                """
                SELECT 
                    dc.id,
                    dc.content,
                    dc.metadata->>'source' AS source,
                    d.title,
                    1 - (dc.embedding <=> %s::vector) AS similarity
                FROM document_chunks dc
                JOIN documents d ON d.id = dc.document_id
                WHERE dc.metadata->>'tenant_id' = %s
                ORDER BY dc.embedding <=> %s::vector
                LIMIT %s
                """,
                (query_embedding, tenant_id, query_embedding, top_k),
            )
            rows = cur.fetchall()
    
    return [
        {
            "id": row[0],
            "content": row[1],
            "source": row[2],
            "title": row[3],
            "similarity": float(row[4]),
        }
        for row in rows
    ]

Performance baselines you should expect from this setup:

  • → 1M vectors, 1536 dimensions, HNSW (m=16), `ef_search=100`: ~5–15ms per query on a 16-core machine with sufficient RAM to hold the index (approximately 6–8 GB for 1M × 1536-dim float32 vectors).
  • → At 5M vectors: query latency climbs to 30–80ms depending on hardware and index parameters, which is acceptable for many applications.
  • → At 20M vectors: HNSW index size exceeds 120 GB for 1536 dimensions. You need SSDs with high IOPS, careful `shared_buffers` and `effective_cache_size` tuning, and you will start hitting the ceiling of what pgvector can comfortably handle on a single node.

    • The memory pressure is the primary scaling constraint for pgvector. HNSW requires the graph structure to fit in memory for fast traversal. When it does not, performance degrades significantly as it falls back to disk reads.

    5.3 When pgvector Is the Right Answer

    pgvector wins decisively in several well-defined scenarios. Recognize these patterns in your own application before reaching for a new service.

    You already have relational data that must be joined with vector results. This is the most important pgvector advantage. If your application needs to filter by user permissions, tenant, subscription tier, date ranges, or any attribute stored in a relational schema, pgvector can perform pre-filter queries that combine B-tree index scans with ANN search in a single query plan. Dedicated vector databases have metadata filtering, but it is bolt-on—you express filters as key-value pairs against a limited metadata payload, not as SQL predicates across a normalized schema. Weaviate and Qdrant have improved their filtering significantly, but nothing matches the expressiveness of SQL when your data model is already relational.

    Your vector count is below 5 million. Below this threshold, pgvector with HNSW delivers sub-20ms queries reliably on commodity cloud instances. For most enterprise internal tools, document search, knowledge base applications, and recommendation features serving thousands to hundreds of thousands of users, 5M vectors is a ceiling you will not approach for years. There is no reason to add operational complexity for scale you do not have.

    You need transactional consistency between vector data and metadata. Consider an e-commerce catalog where a product's embedding and its price, inventory status, and description must be updated atomically. With pgvector, this is a single UPDATE in a transaction. With a dedicated vector database, you are managing two separate systems—your relational database for structured attributes and your vector database for embeddings—and you must handle the consistency window between them in application code. That consistency gap is a real operational burden.

    Your team runs PostgreSQL and nothing else. Operational expertise is a legitimate engineering resource. If your DBAs know PostgreSQL deeply—replication, failover, vacuuming, query plan analysis, connection pooling—adding pgvector costs almost nothing in operational overhead. Adding Pinecone or Weaviate adds a new observability surface, new failure modes, new cost centers, and new on-call burden. That cost is not zero.

    Managed services reduce pgvector's operational cost further. Supabase and Neon both offer pgvector as a first-class managed feature. Neon's serverless PostgreSQL with autoscaling is particularly useful for applications with variable embedding workloads.

    # shell
# Supabase CLI: create a new project with pgvector enabled by default
npx supabase init
npx supabase start

# Neon: provision a serverless PostgreSQL instance
# Install Neon CLI
npm install -g neonctl

# Create project
neonctl projects create --name "vector-app" --region-id "aws-us-east-2"

# Get connection string and connect
neonctl connection-string --project-id <project-id>

# pgvector is available on both Neon and Supabase without any manual installation
# Just run: CREATE EXTENSION vector;

    5.4 When Dedicated Vector Databases Win

    The case for dedicated vector databases is not about features checklist marketing. It is about specific engineering constraints that emerge at scale.

    Vector counts above 20–50 million. At this scale, a single pgvector node cannot hold the HNSW graph in memory. You can tune around this with IVFFlat (which is more memory-efficient but has lower recall) or accept higher latency from disk-based HNSW traversal, but you are fighting the architecture. Pinecone, Milvus, and Weaviate are built around distributed indexing. They shard vector collections across nodes, rebalance automatically as data grows, and maintain query latency SLAs at hundreds of millions of vectors. This is what they were built to do.

    High write throughput with real-time freshness requirements. pgvector's HNSW index is built and updated in-place, but index maintenance under heavy concurrent write load degrades query performance. If your application ingests tens of thousands of new vectors per minute while simultaneously serving ANN queries, you will see query latency spikes during index updates on pgvector. Qdrant and Weaviate use background indexing with HNSW segment merging that separates write and query paths more cleanly. Pinecone's managed index handles this transparently.

    Native multi-tenancy at scale. If you are building a SaaS product where each tenant has their own isolated vector namespace with potentially millions of vectors, managing this in pgvector requires row-level security, careful index design, and connection pooling per tenant—manageable but complex. Pinecone's namespaces and Weaviate's multi-tenancy features (tenant-isolated shards in v1.20+) handle this as a first-class concern.

    Sub-millisecond ANN queries at high QPS. Pinecone in particular is engineered for query latency at high throughput. If your application serves 1,000+ queries per second against a 50M vector collection with p99 latency requirements under 10ms, you are in territory where a managed, purpose-built system has a structural advantage. This is the regime of large-scale recommendation systems and real-time personalization engines, not typical enterprise document retrieval.

    Here is a production-grade Pinecone example demonstrating the operational pattern:

    # Python
import os
from pinecone import Pinecone, ServerlessSpec
from openai import OpenAI

pc = Pinecone(api_key=os.environ["PINECONE_API_KEY"])
openai_client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

INDEX_NAME = "document-chunks-prod"
DIMENSION = 1536  # text-embedding-3-small

# Create index (one-time setup)
# Serverless on AWS us-east-1 — no infrastructure to manage
if INDEX_NAME not in [idx.name for idx in pc.list_indexes()]:
    pc.create_index(
        name=INDEX_NAME,
        dimension=DIMENSION,
        metric="cosine",
        spec=ServerlessSpec(cloud="aws", region="us-east-1"),
    )

index = pc.Index(INDEX_NAME)

def upsert_to_pinecone(chunks: list[dict]) -> None:
    """
    chunks: list of dicts with id, content, metadata
    Pinecone expects: id (str), values (list[float]), metadata (dict)
    """
    texts = [chunk["content"] for chunk in chunks]
    
    # Batch embedding — OpenAI supports up to 2048 inputs per request
    response = openai_client.embeddings.create(
        input=texts,
        model="text-embedding-3-small",
    )
    embeddings = [item.embedding for item in response.data]
    
    vectors = [
        {
            "id": chunk["id"],
            "values": embedding,
            "metadata": {
                "content": chunk["content"],
                "source": chunk["metadata"].get("source", ""),
                "tenant_id": chunk["metadata"].get("tenant_id", ""),
                "created_at": chunk["metadata"].get("created_at", ""),
            },
        }
        for chunk, embedding in zip(chunks, embeddings)
    ]
    
    # Pinecone upsert — batch size 100 is recommended
    batch_size = 100
    for i in range(0, len(vectors), batch_size):
        index.upsert(
            vectors=vectors[i : i + batch_size],
            namespace=chunks[i]["metadata"].get("tenant_id", "default"),
        )

def search_pinecone(
    query: str,
    tenant_id: str,
    top_k: int = 10,
    source_filter: str | None = None,
) -> list[dict]:
    query_embedding = openai_client.embeddings.create(
        input=query,
        model="text-embedding-3-small",
    ).data[0].embedding
    
    # Metadata filter — Pinecone filter syntax is MongoDB-style
    filter_expr = {}
    if source_filter:
        filter_expr["source"] = {"$eq": source_filter}
    
    results = index.query(
        vector=query_embedding,
        top_k=top_k,
        namespace=tenant_id,       # tenant isolation via namespace
        filter=filter_expr if filter_expr else None,
        include_metadata=True,
    )
    
    return [
        {
            "id": match.id,
            "score": match.score,
            "content": match.metadata.get("content", ""),
            "source": match.metadata.get("source", ""),
        }
        for match in results.matches
    ]

    Pinecone's serverless tier eliminates cluster management entirely. You pay per query and storage unit, not for idle compute—a meaningful operational and cost advantage for workloads with bursty or unpredictable traffic patterns.

    5.5 Hybrid Approaches and Migration Paths

    The binary framing of "pgvector vs dedicated" obscures the most common production pattern: starting with pgvector and migrating when you cross a scale threshold. Design for this from the beginning.

    The migration trigger is usually one of three signals: HNSW index size exceeds available RAM on your largest affordable instance, query latency p95 exceeds your SLA at peak load, or write throughput causes measurable index maintenance latency spikes. Instrument these metrics from day one.

    The abstraction layer that makes migration tractable is a thin retrieval interface in your application code. If your RAG pipeline calls retriever.search(query, filters) rather than speaking directly to PostgreSQL, swapping the backend is a configuration change, not a refactor.

    LangChain and LlamaIndex both provide this abstraction:

    # Python
# LangChain vectorstore abstraction — swap backends without changing RAG logic
import os
from langchain_openai import OpenAIEmbeddings
from langchain_postgres import PGVector  # pgvector backend
from langchain_pinecone import PineconeVectorStore  # Pinecone backend
from langchain_weaviate import WeaviateVectorStore  # Weaviate backend

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# --- pgvector backend (development / early production) ---
PG_DSN = os.environ["PG_DSN"]

pgvector_store = PGVector(
    embeddings=embeddings,
    collection_name="document_chunks",
    connection=PG_DSN,
    use_jsonb=True,
)

# --- Pinecone backend (scale-out path) ---
from pinecone import Pinecone
pc = Pinecone(api_key=os.environ["PINECONE_API_KEY"])

pinecone_store = PineconeVectorStore(
    index=pc.Index("document-chunks-prod"),
    embedding=embeddings,
    text_key="content",
)

# --- Weaviate backend (alternative: self-hosted, richer filtering) ---
import weaviate
wv_client = weaviate.connect_to_local()  # or connect_to_weaviate_cloud()

weaviate_store = WeaviateVectorStore(
    client=wv_client,
    index_name="DocumentChunk",
    text_key="content",
    embedding=embeddings,
)

# Your RAG retriever is backend-agnostic
def build_retriever(backend: str = "pgvector"):
    store_map = {
        "pgvector": pgvector_store,
        "pinecone": pinecone_store,
        "weaviate": weaviate_store,
    }
    store = store_map[backend]
    return store.as_retriever(
        search_type="similarity",
        search_kwargs={"k": 10},
    )

# Usage — switch VECTOR_BACKEND env var to migrate
retriever = build_retriever(backend=os.environ.get("VECTOR_BACKEND", "pgvector"))

    For teams running Weaviate or Qdrant rather than Pinecone, note that Weaviate Cloud and Qdrant Cloud both offer managed hosting, while Qdrant also supports Docker and Kubernetes deployment for teams who prefer to operate their own infrastructure and want the flexibility of DiskANN-based indexing at scale.

    When migrating, do not do a hard cutover. Run both systems in parallel, shadow-write to the new backend, and validate recall quality against your evaluation dataset before shifting query traffic. This is the same dual-write pattern used in database migrations—the principles transfer directly.

    5.6 Making the Decision: A Practical Framework

    Reduce the decision to a checklist you can run through in thirty minutes for any new project or existing system approaching a scale threshold.

    Start with pgvector if all of the following are true:

  • → Your estimated vector count at 18-month growth is below 10 million
  • → Your team already operates PostgreSQL
  • → Your application logic requires JOINs or transactional consistency between vector and relational data
  • → Query latency requirements are relaxed enough to tolerate 10–50ms p95

    • Evaluate dedicated vector databases if any of the following are true:

  • → You are projecting above 20 million vectors within 12 months
  • → You need sub-5ms p99 at 500+ QPS
  • → Your write throughput exceeds 10,000 vectors per minute continuously
  • → Multi-tenancy isolation is a hard product requirement at large per-tenant scale
  • → Your vectors are the primary data, not a secondary index on relational data

    • Use Pinecone when you want zero infrastructure management and are comfortable with a fully managed SaaS cost model. Its serverless tier is the fastest path to production with no operational burden.

    Use Weaviate when you need richer hybrid search (BM25 + vector combined natively), a GraphQL query interface, or want self-hosted control over your data. Weaviate's hybrid search combining sparse and dense retrieval is production-ready and significantly better than pgvector's full-text search combination.

    Use Qdrant when you want Rust-based performance, on-premises control, and need advanced quantization (scalar, product, binary) to reduce index memory footprint at large scale.

    Use Milvus when you are operating at the very large end of the scale spectrum (hundreds of millions to billions of vectors) and have the engineering capacity to operate a distributed system with its full Kubernetes deployment model.

    The decision is not permanent. pgvector is a valid production choice for a system serving millions of users if the vectors stay within its sweet spot. The teams that get into trouble are the ones who either over-engineer toward a dedicated vector database before they have the scale to justify it, or who stay on pgvector past the point where its limitations are causing real user-facing degradation. Know your thresholds, instrument your metrics, and make the switch when the data tells you to—not when a vendor's benchmark slides tell you to.
