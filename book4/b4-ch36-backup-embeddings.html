Chapter 36: Backup Strategies for Embedding Data

Embedding data presents a deceptively familiar backup challenge. The vectors themselves are just floating-point arrays — and any experienced DBA knows how to back up a table of floats. What makes embedding backups genuinely different is the relationship between three distinct artifacts that must remain in sync: the raw source data, the embeddings derived from that data, and the model that produced those embeddings. Back up only two of the three and you have a restoration problem that won't reveal itself until you're already in a production incident. This chapter covers the full backup and recovery strategy for vector workloads — from pgvector snapshots to managed Pinecone exports, from model versioning to the operational patterns that keep RAG pipelines recoverable at scale.

---

The Three-Artifact Problem

In traditional database backups, data is self-contained. A row in a customer table means the same thing whether you restore it today or in three years. Embeddings are not self-contained — they are a function of source text and model weights, and that function changes when either input changes.

Consider a document chunk stored in pgvector:

-- PostgreSQL / pgvector
CREATE TABLE document_chunks (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_id   UUID NOT NULL REFERENCES documents(id),
    chunk_text  TEXT NOT NULL,
    embedding   VECTOR(1536),
    model_id    VARCHAR(100) NOT NULL,  -- e.g. 'text-embedding-3-small-v1'
    model_hash  VARCHAR(64),            -- SHA-256 of model release tag
    embedded_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT fk_source FOREIGN KEY (source_id)
        REFERENCES documents(id) ON DELETE CASCADE
);

-- Index for backup validation queries
CREATE INDEX idx_chunks_model ON document_chunks(model_id, embedded_at);

-- View: chunks without matching source data (backup integrity check)
CREATE VIEW orphaned_embeddings AS
SELECT dc.id, dc.source_id, dc.model_id, dc.embedded_at
FROM   document_chunks dc
LEFT   JOIN documents d ON d.id = dc.source_id
WHERE  d.id IS NULL;

The model_id and model_hash columns are the starting point for a coherent backup strategy. If you restore embeddings generated by text-embedding-3-small but your application is now calling text-embedding-3-large, every similarity search will return nonsense — not an error, not a crash, just silently wrong results. Backing up the vector data without capturing what produced it means the restore is incomplete by definition.

The three artifacts to track together:

| Artifact | What to Back Up | Where It Lives |

|---|---|---|

| Source data | Raw text, documents, metadata | Your primary RDBMS or object store |

| Embeddings | Vector arrays + model provenance | Vector database or pgvector table |

| Model reference | Model version, provider, API endpoint config | Model registry, config management |

The goal of an embedding backup strategy is to ensure that after any restore event, you can either (a) serve existing embeddings from a consistent snapshot where all three artifacts match, or (b) regenerate embeddings from source data using the same model version. Both paths need explicit support in your backup design.

---

Backup Strategies for pgvector

PostgreSQL with pgvector behaves like any other PostgreSQL workload for the purposes of backup tooling. The difference is in how large the data gets. A table with 10 million document chunks at 1536 dimensions (float4, 4 bytes per dimension) holds approximately 58 GB of vector data before indices. HNSW indices on that table add another 15–30 GB depending on m and ef_construction parameters. Plan storage accordingly.

Logical Backups with pg_dump

For moderate-scale deployments (under ~50 GB of vector data), pg_dump is still workable:

# shell
# Full logical backup — include custom format for parallel restore capability
pg_dump \
  --format=custom \
  --compress=9 \
  --jobs=4 \
  --table=document_chunks \
  --table=documents \
  --file=/backup/rag_vectors_$(date +%Y%m%d_%H%M%S).dump \
  rag_production_db

# Verify backup integrity before storing offsite
pg_restore --list /backup/rag_vectors_20240615_020000.dump | head -30

# Restore to a staging environment for validation
pg_restore \
  --format=custom \
  --jobs=4 \
  --dbname=rag_staging \
  /backup/rag_vectors_20240615_020000.dump

Note that pg_dump does not back up HNSW or IVFFlat indices — it captures the CREATE INDEX DDL, which means the index is rebuilt from scratch on restore. For a 10-million-row table, that rebuild takes 20–45 minutes depending on hardware. Budget this into your RTO. If rebuilding the index is unacceptable, use physical backup methods.

Physical Backups with pgBackRest

For large vector tables where index rebuild time breaks your RTO, pgBackRest with incremental backups is the right tool:

# shell
# pgbackrest.conf excerpt for vector workload
# /etc/pgbackrest/pgbackrest.conf

[global]
repo1-path=/mnt/backup/pgbackrest
repo1-retention-full=4          # 4 full backups (weekly cycle = 1 month)
repo1-retention-diff=14         # 14 differential backups
repo1-cipher-type=aes-256-cbc
repo1-cipher-pass=<VAULT_SECRET>
compress-type=lz4               # lz4 is faster than gzip for large float arrays
compress-level=3
process-max=4

[rag-production]
pg1-path=/var/lib/postgresql/16/main
pg1-host=db-primary.internal

# Full backup — run weekly
pgbackrest --stanza=rag-production backup --type=full

# Differential backup — run daily
pgbackrest --stanza=rag-production backup --type=diff

# Restore to point-in-time (critical for embedding consistency)
pgbackrest --stanza=rag-production restore \
  --target="2024-06-15 03:00:00" \
  --target-action=promote

The PITR capability is particularly valuable for embedding data. If your embedding pipeline had a bug that corrupted embeddings between 14:00 and 16:00 yesterday, you can restore to 13:59 and have a clean state — HNSW index intact, no rebuild required.

Incremental Strategy for Append-Heavy Workloads

Most RAG pipelines are append-heavy: new documents get embedded and inserted, old embeddings rarely change. This makes partial exports viable as a complement to full backups:

-- PostgreSQL / pgvector
-- Export only embeddings created in the last 24 hours
COPY (
    SELECT
        dc.id,
        dc.source_id,
        dc.chunk_text,
        dc.embedding::TEXT,   -- cast vector to text for portable export
        dc.model_id,
        dc.model_hash,
        dc.embedded_at,
        d.title,
        d.source_url
    FROM   document_chunks dc
    JOIN   documents d ON d.id = dc.source_id
    WHERE  dc.embedded_at >= NOW() - INTERVAL '24 hours'
    ORDER BY dc.embedded_at
)
TO '/backup/incremental/chunks_20240615.csv'
WITH (FORMAT CSV, HEADER, DELIMITER ',');

-- Track export completion for audit trail
INSERT INTO backup_audit (export_type, row_count, export_start, export_end, file_path)
SELECT
    'incremental_daily',
    COUNT(*),
    MIN(embedded_at),
    MAX(embedded_at),
    '/backup/incremental/chunks_20240615.csv'
FROM document_chunks
WHERE embedded_at >= NOW() - INTERVAL '24 hours';

Exporting from Managed Vector Databases

Managed services like Pinecone, Weaviate Cloud, and Qdrant Cloud do not expose file-level backup interfaces. Backup means export — pulling vectors out via API and storing them yourself. This is the primary gap in managed vector database offerings as of 2024, and every production deployment needs an explicit strategy to address it.

Pinecone: Batch Fetch Export

Pinecone does not offer a native export API as of this writing. The operational pattern is to maintain a shadow copy in your source database and reconstruct from there, but if you need the actual vectors:

# Python
import os
import json
import time
from pinecone import Pinecone
from pathlib import Path

pc = Pinecone(api_key=os.environ["PINECONE_API_KEY"])
index = pc.Index("rag-production")

def export_pinecone_namespace(
    index,
    namespace: str,
    output_dir: str,
    batch_size: int = 1000
) -> dict:
    """
    Export all vectors from a Pinecone namespace.
    Uses list() to paginate IDs, then fetch() in batches.
    Returns export statistics.
    """
    Path(output_dir).mkdir(parents=True, exist_ok=True)
    output_file = f"{output_dir}/pinecone_{namespace}_{int(time.time())}.jsonl"

    stats = index.describe_index_stats()
    namespace_stats = stats.namespaces.get(namespace, {})
    total_vectors = namespace_stats.get("vector_count", 0)

    print(f"Exporting {total_vectors} vectors from namespace '{namespace}'")

    exported = 0
    pages_written = 0

    with open(output_file, "w") as f:
        # list() paginates through all vector IDs
        for ids_page in index.list(namespace=namespace, limit=batch_size):
            if not ids_page:
                break

            # Fetch actual vectors + metadata
            result = index.fetch(ids=ids_page, namespace=namespace)

            for vector_id, vector_data in result.vectors.items():
                record = {
                    "id": vector_id,
                    "values": vector_data.values,
                    "metadata": vector_data.metadata or {},
                    "namespace": namespace
                }
                f.write(json.dumps(record) + "\n")
                exported += 1

            pages_written += 1
            if pages_written % 10 == 0:
                print(f"  Progress: {exported}/{total_vectors} vectors exported")

    return {
        "namespace": namespace,
        "total_exported": exported,
        "output_file": output_file,
        "timestamp": time.time()
    }

# Export all namespaces
stats = index.describe_index_stats()
for namespace in stats.namespaces.keys():
    result = export_pinecone_namespace(
        index=index,
        namespace=namespace,
        output_dir="/backup/pinecone",
        batch_size=1000
    )
    print(f"Exported: {result}")

# Save index configuration alongside data
index_config = {
    "name": "rag-production",
    "dimension": stats.dimension,
    "metric": "cosine",
    "namespaces": {
        ns: data.vector_count
        for ns, data in stats.namespaces.items()
    },
    "export_timestamp": time.time()
}

with open("/backup/pinecone/index_config.json", "w") as f:
    json.dump(index_config, f, indent=2)

This export produces JSONL files that can be reimported to a new Pinecone index or cross-loaded into Weaviate or Qdrant if you need to migrate. The key requirement: store this alongside your model configuration so you know exactly which embedding model produced these vectors.

Weaviate: Native Backup API

Weaviate has a first-class backup API that supports S3, GCS, and Azure Blob Storage backends. This is significantly more operationally clean than the Pinecone fetch-loop approach:

# Python
import weaviate
import time
import os

client = weaviate.connect_to_wcs(
    cluster_url=os.environ["WEAVIATE_URL"],
    auth_credentials=weaviate.auth.AuthApiKey(os.environ["WEAVIATE_API_KEY"])
)

def create_weaviate_backup(
    client,
    backup_id: str,
    classes: list[str] | None = None,  # None = all classes
    backend: str = "s3",
    bucket: str = "my-weaviate-backups",
    path: str = "backups"
) -> dict:
    """
    Trigger a Weaviate backup and wait for completion.
    Returns backup status details.
    """
    backup_config = {
        "id": backup_id,
        "include": classes  # omit to back up all classes
    }

    print(f"Starting Weaviate backup: {backup_id}")

    result = client.backup.create(
        backup_id=backup_id,
        backend=backend,
        include_classes=classes,
        wait_for_completion=False  # async — we'll poll
    )

    print(f"Backup initiated. Status: {result.status}")

    # Poll until completion
    while True:
        status = client.backup.get_create_status(
            backup_id=backup_id,
            backend=backend
        )

        if status.status == "SUCCESS":
            print(f"Backup completed: {backup_id}")
            return {
                "backup_id": backup_id,
                "status": "SUCCESS",
                "backend": backend,
                "path": f"s3://{bucket}/{path}/{backup_id}"
            }
        elif status.status == "FAILED":
            raise RuntimeError(f"Backup failed: {status.error}")
        else:
            print(f"  Status: {status.status} — waiting 10s...")
            time.sleep(10)

# Create a timestamped backup
backup_id = f"rag-backup-{time.strftime('%Y%m%d-%H%M%S')}"

backup_result = create_weaviate_backup(
    client=client,
    backup_id=backup_id,
    classes=["DocumentChunk", "Document"],  # specific classes only
    backend="s3",
    bucket="my-weaviate-backups"
)

print(f"Backup stored: {backup_result}")

# Restore from backup (e.g., disaster recovery scenario)
def restore_weaviate_backup(client, backup_id: str, backend: str = "s3"):
    result = client.backup.restore(
        backup_id=backup_id,
        backend=backend,
        wait_for_completion=True
    )
    print(f"Restore status: {result.status}")
    return result

client.close()

Weaviate backups are consistent snapshots — they do not require downtime and they include schema definitions, object properties, and vector indices together. This is the model all vector databases should eventually follow, and it simplifies the three-artifact problem considerably because schema (which includes the model configuration) is part of the backup payload.

Model Versioning as a Backup Dependency

Vectors are only interpretable in the context of the model that created them. This makes model version management a first-class backup concern, not an afterthought. DBAs entering the AI infrastructure space often underestimate this dependency because traditional database backups have no equivalent.

The failure mode looks like this: you restore your vector database from a backup taken six months ago. The backup is clean. All vectors are present. You rebuild the HNSW index. Queries return results. But your application team upgraded from text-embedding-ada-002 to text-embedding-3-small four months ago, and the two models produce vectors in incompatible embedding spaces. The cosine similarities you're computing between the restored old vectors and new query vectors are geometrically meaningless.

Model Registry Pattern

# Python
import json
import hashlib
import os
from datetime import datetime, timezone
from pathlib import Path

class EmbeddingModelRegistry:
    """
    Tracks model versions used to produce embeddings.
    Persisted as JSON alongside vector backups.
    In production, back this up to S3/GCS with versioning enabled.
    """

    def __init__(self, registry_path: str = "/etc/rag/model_registry.json"):
        self.registry_path = Path(registry_path)
        self.registry_path.parent.mkdir(parents=True, exist_ok=True)
        self._load()

    def _load(self):
        if self.registry_path.exists():
            with open(self.registry_path) as f:
                self.data = json.load(f)
        else:
            self.data = {"models": {}, "current": None}

    def _save(self):
        with open(self.registry_path, "w") as f:
            json.dump(self.data, f, indent=2)

    def register_model(
        self,
        model_id: str,
        provider: str,
        dimensions: int,
        endpoint: str,
        api_version: str | None = None,
        notes: str = ""
    ) -> str:
        """
        Register a model version. Returns a stable model_hash
        that can be stored alongside each embedding row.
        """
        fingerprint = f"{provider}:{model_id}:{dimensions}:{api_version or ''}"
        model_hash = hashlib.sha256(fingerprint.encode()).hexdigest()[:16]

        self.data["models"][model_hash] = {
            "model_id": model_id,
            "provider": provider,
            "dimensions": dimensions,
            "endpoint": endpoint,
            "api_version": api_version,
            "registered_at": datetime.now(timezone.utc).isoformat(),
            "notes": notes,
            "hash": model_hash
        }

        self._save()
        return model_hash

    def set_current(self, model_hash: str):
        assert model_hash in self.data["models"], f"Unknown model hash: {model_hash}"
        self.data["current"] = model_hash
        self._save()

    def get_current(self) -> dict:
        assert self.data["current"], "No current model set"
        return self.data["models"][self.data["current"]]

    def get_model(self, model_hash: str) -> dict:
        return self.data["models"].get(model_hash)

    def compatibility_check(
        self,
        backup_model_hash: str,
        current_model_hash: str
    ) -> dict:
        """
        Check whether backup vectors are compatible with current model.
        Returns a dict with compatibility status and recommended action.
        """
        backup_model = self.get_model(backup_model_hash)
        current_model = self.get_model(current_model_hash)

        if backup_model_hash == current_model_hash:
            return {"compatible": True, "action": "restore_directly"}

        if backup_model["dimensions"] != current_model["dimensions"]:
            return {
                "compatible": False,
                "action": "re_embed_all",
                "reason": f"Dimension mismatch: {backup_model['dimensions']} vs {current_model['dimensions']}"
            }

        if backup_model["model_id"] != current_model["model_id"]:
            return {
                "compatible": False,
                "action": "re_embed_all",
                "reason": f"Different model: {backup_model['model_id']} vs {current_model['model_id']}"
            }

        return {
            "compatible": False,
            "action": "evaluate",
            "reason": "Same model ID but different versions — test similarity quality before deciding"
        }


# Usage
registry = EmbeddingModelRegistry()

# Register models in use
hash_ada = registry.register_model(
    model_id="text-embedding-ada-002",
    provider="openai",
    dimensions=1536,
    endpoint="https://api.openai.com/v1/embeddings",
    notes="Legacy model, used before 2024-03"
)

hash_v3 = registry.register_model(
    model_id="text-embedding-3-small",
    provider="openai",
    dimensions=1536,
    endpoint="https://api.openai.com/v1/embeddings",
    api_version="2024-02-01",
    notes="Current production model"
)

registry.set_current(hash_v3)

# Before restore: check compatibility
check = registry.compatibility_check(hash_ada, hash_v3)
print(check)
# {'compatible': False, 'action': 're_embed_all', 'reason': 'Different model: ...'}

Back up the model registry JSON to the same location as your vector exports and treat it as part of the backup artifact set. If the registry says a restore requires re_embed_all, your recovery procedure needs to include re-embedding time in the RTO calculation — which at 10 million chunks and API rate limits can be measured in hours, not minutes.

Recovery Procedures and RTO Planning

Recovery from a backup event in a vector workload has three distinct phases, each with its own time budget:

1. Data restore — Moving backup files to the target system and importing them

2. Index rebuild — Reconstructing the vector index (HNSW, IVFFlat, etc.)

3. Re-embedding — If model compatibility check fails, regenerating all vectors from source

Understanding the time cost of each phase is the foundation of a realistic RTO commitment.

Phase Timing Reference (approximate, hardware-dependent)

| Operation | Scale | Typical Duration |

|---|---|---|

| pgBackRest physical restore | 100 GB | 15–30 min |

| pg_dump logical restore | 100 GB | 45–90 min |

| HNSW index rebuild (10M vectors, 1536-dim) | — | 20–45 min |

| Re-embedding via OpenAI API (10M chunks) | rate-limited | 4–12 hours |

| Re-embedding via local sentence-transformers | GPU available | 2–4 hours |

| Weaviate backup restore from S3 | 50 GB | 10–20 min |

| Pinecone upsert re-import | 10M vectors | 3–6 hours |

The re-embedding phase is the wildcard. If your recovery plan includes it, you need either pre-approved API rate limit increases with your embedding provider, or local model hosting capability. Planning for model-version-locked recovery avoids this phase entirely.
