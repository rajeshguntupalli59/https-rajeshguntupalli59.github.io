Chapter 3: Distance Metrics: Cosine, Euclidean, Dot Product

Every vector search query asks the same fundamental question: how similar is this vector to that one? The answer depends entirely on how you measure similarity — and that choice is not arbitrary. The three dominant distance metrics in production vector systems — cosine similarity, Euclidean distance, and dot product — each make different mathematical assumptions about what "closeness" means, and each is optimized for different embedding types, storage architectures, and query patterns. Getting this wrong means your semantic search returns irrelevant results, your recommendation engine surfaces stale content, and your RAG pipeline retrieves the wrong context — all while your vector index silently operates on numbers that look reasonable but mean nothing useful. This chapter gives you a precise, production-grounded understanding of all three metrics: the math behind them, how they behave with real embeddings, and how to configure them correctly in pgvector, Pinecone, Weaviate, Qdrant, and other systems.

---

3.1 What Distance Metrics Actually Measure (And Why It's Not Just Geometry)

Before writing a single line of SQL or configuration, you need to understand what these metrics are computing and why the choice matters at the embedding layer.

An embedding is a dense numerical vector — typically 768 to 3072 dimensions for modern language models — that encodes semantic meaning as a position in high-dimensional space. The embedding model learns to position semantically similar inputs near each other and dissimilar inputs far apart. "Near" and "far" are defined by the distance metric used during training. This is the point most database engineers miss: the metric you use at query time must match or be compatible with the metric the model was optimized for during training. Using the wrong metric against an embedding trained with a different objective function degrades search quality measurably.

Think of it in database terms. An embedding is like a computed column that encodes a complex function of the source data. The distance metric is the comparison operator used in the WHERE clause. Just as you would not use a string inequality comparison on a numeric column and expect meaningful results, you should not use Euclidean distance on embeddings trained with cosine objectives and expect meaningful ranking.

The three metrics differ in what they ignore and what they emphasize:

  • → **Cosine similarity** measures the angle between two vectors, ignoring their magnitudes. Two vectors pointing in the same direction score 1.0 regardless of how long they are.
  • → **Euclidean distance** (L2) measures the straight-line distance between two vector endpoints in space. It is sensitive to both direction and magnitude.
  • → **Dot product** (inner product) measures both angle and magnitude together. It rewards vectors that are both directionally aligned and large.

    • This means cosine similarity treats a document with 10 mentions of "database" identically to a document with 100 mentions, as long as the relative proportions of all terms are the same. Euclidean distance would treat those documents differently. Dot product would actively prefer the document with more signal — which is exactly the behavior you want in some recommendation scenarios.

    Here is a concrete Python example showing all three computed on the same pair of vectors:

    # Python example — computing all three distance metrics from scratch
import numpy as np

def cosine_similarity(a: np.ndarray, b: np.ndarray) -> float:
    """
    Measures angle between vectors. Range: -1.0 to 1.0.
    1.0 = identical direction, 0.0 = orthogonal, -1.0 = opposite.
    """
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

def euclidean_distance(a: np.ndarray, b: np.ndarray) -> float:
    """
    Measures straight-line distance. Range: 0.0 to infinity.
    Lower = more similar.
    """
    return np.linalg.norm(a - b)

def dot_product(a: np.ndarray, b: np.ndarray) -> float:
    """
    Measures both direction and magnitude. Range: -infinity to infinity.
    Higher = more similar (for normalized vectors, equals cosine similarity).
    """
    return np.dot(a, b)

# Two 4-dimensional vectors for illustration
# In production these would be 768–3072 dimensional embeddings
vec_a = np.array([0.2, 0.5, 0.1, 0.8])
vec_b = np.array([0.4, 1.0, 0.2, 1.6])  # vec_b = 2 * vec_a — same direction, 2x magnitude

print(f"Cosine similarity:   {cosine_similarity(vec_a, vec_b):.6f}")   # 1.000000 — identical angle
print(f"Euclidean distance:  {euclidean_distance(vec_a, vec_b):.6f}")  # 1.044031 — sees the difference
print(f"Dot product:         {dot_product(vec_a, vec_b):.6f}")         # 0.980000 — vec_b amplifies the signal

# Now normalize both to unit length and compare again
vec_a_norm = vec_a / np.linalg.norm(vec_a)
vec_b_norm = vec_b / np.linalg.norm(vec_b)

print("\n--- After L2 normalization ---")
print(f"Cosine similarity:   {cosine_similarity(vec_a_norm, vec_b_norm):.6f}")  # 1.000000
print(f"Euclidean distance:  {euclidean_distance(vec_a_norm, vec_b_norm):.6f}") # 0.000000 — identical after norm
print(f"Dot product:         {dot_product(vec_a_norm, vec_b_norm):.6f}")        # 1.000000 — equals cosine

    This last block is the key mathematical insight you need to internalize: when all vectors are L2-normalized to unit length, cosine similarity and dot product are mathematically equivalent, and the Euclidean distance between them is a monotonic transformation of the cosine distance. Many embedding models — including OpenAI's text-embedding-3-small and all-MiniLM-L6-v2 — output normalized or near-normalized vectors. In those cases, all three metrics produce the same ranking order, and the choice between them becomes a question of computational efficiency rather than mathematical correctness.

    3.2 Cosine Similarity in Practice: When Angle Is All That Matters

    Cosine similarity is the dominant metric for text embeddings and the default choice in most RAG pipelines. Its strength is scale invariance: the length of the vector does not affect the similarity score, only the direction. This property naturally handles documents of different lengths, embedding outputs from different batch sizes, and representations that might vary in norm due to quantization or compression.

    The cosine distance (1 - cosine similarity) is used when you need a proper distance metric that satisfies the triangle inequality, which is required by some index structures. Most vector databases accept configuration for both.

    In pgvector, cosine distance is expressed with the <=> operator:

    -- PostgreSQL / pgvector
-- Create a table for document embeddings (e.g., from OpenAI text-embedding-3-small, 1536 dims)
CREATE TABLE documents (
    id          BIGSERIAL PRIMARY KEY,
    content     TEXT NOT NULL,
    source_url  TEXT,
    embedding   VECTOR(1536)
);

-- Create an HNSW index optimized for cosine distance
-- HNSW is preferred over IVFFlat for cosine when dataset < 5M rows
CREATE INDEX ON documents USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- Semantic search: find the 5 most similar documents to a query embedding
-- The :query_embedding parameter is your 1536-dimensional query vector
SELECT
    id,
    content,
    source_url,
    1 - (embedding <=> :query_embedding) AS cosine_similarity
FROM documents
ORDER BY embedding <=> :query_embedding
LIMIT 5;

-- Note: pgvector uses cosine DISTANCE (<=>), not similarity
-- cosine_distance = 1 - cosine_similarity
-- Range of cosine distance: 0.0 (identical) to 2.0 (opposite)
-- For normalized vectors it ranges 0.0 to 2.0, practical range is usually 0.0 to 1.0

    The same operation in Pinecone uses cosine as the index-level setting — you cannot change the metric after index creation:

    # Python example — creating a Pinecone index with cosine metric and querying it
from pinecone import Pinecone, ServerlessSpec
import openai

pc = Pinecone(api_key="YOUR_PINECONE_API_KEY")

# Metric is set at index creation and cannot be changed afterward
# This is the key operational difference from a traditional database — plan ahead
pc.create_index(
    name="documents-cosine",
    dimension=1536,                      # matches OpenAI text-embedding-3-small
    metric="cosine",                     # options: cosine, euclidean, dotproduct
    spec=ServerlessSpec(cloud="aws", region="us-east-1")
)

index = pc.Index("documents-cosine")

# Generate a query embedding using OpenAI
client = openai.OpenAI(api_key="YOUR_OPENAI_API_KEY")

query_text = "how do I optimize PostgreSQL for write-heavy workloads?"
response = client.embeddings.create(
    input=query_text,
    model="text-embedding-3-small"
)
query_embedding = response.data[0].embedding  # list of 1536 floats

# Query the index — Pinecone returns cosine similarity scores (higher = more similar)
# Note: Pinecone inverts the cosine distance so 1.0 = identical, 0.0 = unrelated
results = index.query(
    vector=query_embedding,
    top_k=5,
    include_metadata=True
)

for match in results.matches:
    print(f"ID: {match.id:30s}  Score: {match.score:.4f}  Title: {match.metadata.get('title', 'N/A')}")

    One production pattern worth establishing early: normalize your embeddings before storing them if your embedding model does not guarantee unit-length output. The sentence-transformers library normalizes by default when you use encode(normalize_embeddings=True). OpenAI's text-embedding-3 models output vectors that are already close to unit norm but not exactly 1.0 due to floating-point precision. For strict cosine implementations, explicit normalization costs little at write time and eliminates a class of subtle ranking errors.

    3.3 Euclidean Distance: When Magnitude Carries Meaning

    Euclidean distance (L2 distance) is the default metric for many image embedding models, multimodal systems, and any domain where the magnitude of an embedding encodes meaningful information. In text applications it is less common, but in computer vision, audio, and scientific data contexts it often outperforms cosine.

    The intuition: if your embedding model is trained to place embeddings in a space where both direction and length encode distinct properties — for example, where longer vectors represent more "intense" or "prominent" concepts — then cosine similarity throws away half the information. Euclidean distance preserves it.

    In pgvector, Euclidean distance uses the <-> operator:

    -- PostgreSQL / pgvector
-- Image similarity search using Euclidean distance
-- e.g., embeddings from CLIP (512 dims) or ResNet features (2048 dims)
CREATE TABLE product_images (
    id           BIGSERIAL PRIMARY KEY,
    product_id   UUID NOT NULL,
    image_url    TEXT NOT NULL,
    embedding    VECTOR(512)             -- CLIP ViT-B/32 output dimension
);

-- HNSW with L2 operator class
CREATE INDEX ON product_images USING hnsw (embedding vector_l2_ops)
    WITH (m = 16, ef_construction = 64);

-- Find visually similar products to a given query image
-- Lower L2 distance = more similar
SELECT
    p.id,
    p.product_id,
    p.image_url,
    p.embedding <-> :query_image_embedding AS l2_distance
FROM product_images p
ORDER BY p.embedding <-> :query_image_embedding
LIMIT 10;

-- IVFFlat alternative for larger datasets (> 1M rows)
-- Requires ANALYZE after load; less accurate but faster at scale
CREATE INDEX ON product_images USING ivfflat (embedding vector_l2_ops)
    WITH (lists = 100);
-- Rule of thumb: lists = sqrt(total_rows) for balanced speed/recall
-- After loading data: ANALYZE product_images;

    The critical operational difference between IVFFlat and HNSW becomes concrete with Euclidean distance. IVFFlat clusters the vector space using k-means centroids, and L2 distance is the natural clustering objective. HNSW builds a hierarchical graph and works well with any metric. For datasets under roughly 1 million vectors on PostgreSQL, HNSW with vector_l2_ops gives better recall with less tuning. Above that scale — or on machines with limited RAM for the HNSW graph — IVFFlat is more memory-efficient.

    Here is a comparison that shows the recall and latency tradeoffs you should test against your actual data before committing to an index type:

    # Python example — benchmarking IVFFlat vs HNSW recall for L2 distance using pgvector
import psycopg2
import numpy as np
import time

conn = psycopg2.connect("host=localhost dbname=vectordb user=postgres password=secret")
cur = conn.cursor()

def run_ann_query(query_vector: list, index_type: str, k: int = 10) -> tuple[list, float]:
    """Run an ANN query and return (result_ids, latency_ms)."""
    vec_str = "[" + ",".join(str(v) for v in query_vector) + "]"
    
    # Set ef_search for HNSW (higher = better recall, slower)
    if index_type == "hnsw":
        cur.execute("SET hnsw.ef_search = 100")
    elif index_type == "ivfflat":
        cur.execute("SET ivfflat.probes = 10")  # probes = lists/10 is a common starting point
    
    start = time.perf_counter()
    cur.execute(
        f"SELECT id FROM product_images ORDER BY embedding <-> %s::vector LIMIT %s",
        (vec_str, k)
    )
    elapsed_ms = (time.perf_counter() - start) * 1000
    result_ids = [row[0] for row in cur.fetchall()]
    return result_ids, elapsed_ms

# Load 100 random query vectors for benchmarking
np.random.seed(42)
query_vectors = [np.random.rand(512).tolist() for _ in range(100)]

# Run benchmark
latencies = []
for qvec in query_vectors:
    _, latency = run_ann_query(qvec, index_type="hnsw", k=10)
    latencies.append(latency)

print(f"HNSW L2 — p50: {np.percentile(latencies, 50):.1f}ms  "
      f"p95: {np.percentile(latencies, 95):.1f}ms  "
      f"p99: {np.percentile(latencies, 99):.1f}ms")

cur.close()
conn.close()

    In Weaviate, Euclidean distance is configured at the collection level under vectorizer settings and requires the l2-squared distance metric parameter. Weaviate calls it l2-squared because it omits the square root for computational efficiency — the square root is a monotonic function, so it does not change the ranking order. You will see this optimization in several production vector databases: they compute L2-squared internally but report the API documentation in terms of standard L2.

    3.4 Dot Product: Maximum Throughput for Recommendation and Ranking Systems

    The dot product (also called inner product) is the most computationally efficient of the three metrics because it avoids the normalization step required by cosine similarity. On modern hardware with SIMD instructions, this matters at scale: the throughput difference between dot product and cosine can be 15–30% when operating on millions of vectors per second.

    Beyond performance, dot product is the correct metric for specific problem categories:

    1. Recommendation systems with popularity signals: If your embedding model is trained with a two-tower architecture (common in large-scale recommendation at companies like Google, Meta, and Spotify), the item embedding magnitudes are deliberately trained to encode relevance intensity. Using cosine would strip that signal away.

    2. Dense retrieval models trained with in-batch negatives: Models like DPR (Dense Passage Retrieval) and BGE-M3 with specific training configurations expect inner product similarity at inference time to match their training objective.

    3. Maximum Inner Product Search (MIPS) problems: A well-studied class of problems in recommendation, where the goal is to find the item with the highest dot product against a query embedding.

    In pgvector, dot product similarity uses the <#> operator. Note the sign convention: pgvector returns the negative inner product because all of its operators follow the "lower is more similar" convention:

    -- PostgreSQL / pgvector
-- Recommendation system using dot product (inner product)
-- Assumes item embeddings are NOT normalized — magnitude encodes popularity/quality signal
CREATE TABLE item_embeddings (
    item_id      BIGINT PRIMARY KEY,
    item_type    TEXT NOT NULL,          -- 'article', 'product', 'video'
    embedding    VECTOR(256)             -- two-tower model, 256-dimensional item tower
);

-- HNSW with inner product operator class
CREATE INDEX ON item_embeddings USING hnsw (embedding vector_ip_ops)
    WITH (m = 32, ef_construction = 128);
-- Higher m (32 vs default 16) improves recall for dot product
-- because the HNSW graph needs more connections to navigate non-Euclidean topology

-- Get top-10 recommendations for a user embedding
-- <#> returns NEGATIVE dot product — ORDER BY ascending to get highest dot product first
SELECT
    item_id,
    item_type,
    (embedding <#> :user_embedding) * -1 AS dot_product_score
FROM item_embeddings
WHERE item_type = 'article'
ORDER BY embedding <#> :user_embedding   -- ascending negative = descending positive
LIMIT 10;

    The sign inversion is a common source of bugs when engineers first switch from cosine to dot product in pgvector. Cosine distance is inherently non-negative (0.0 to 2.0), so "ORDER BY embedding <=> query ASC" feels natural. With <#>, which returns a negative value, you must either remember to sort ascending (which retrieves the most negative negated dot product, i.e., the highest actual dot product) or multiply by -1 in a subquery. Both approaches work; consistency in your codebase matters more than which pattern you choose.

    In Qdrant, dot product is a first-class metric with clean semantics — no sign inversion:

    # Python example — Qdrant collection with dot product metric for a recommendation system
from qdrant_client import QdrantClient
from qdrant_client.models import (
    Distance, VectorParams, PointStruct, Filter, FieldCondition, MatchValue
)
import numpy as np

client = QdrantClient(host="localhost", port=6333)

# Create collection with dot product distance
# Qdrant uses "Dot" — returns higher scores for more similar vectors (no sign inversion)
client.create_collection(
    collection_name="item_recommendations",
    vectors_config=VectorParams(
        size=256,
        distance=Distance.DOT       # also available: COSINE, EUCLID, MANHATTAN
    )
)

# Upsert item embeddings (simulating a two-tower model output)
np.random.seed(42)
items = [
    PointStruct(
        id=i,
        vector=np.random.randn(256).tolist(),   # NOT normalized — magnitude matters
        payload={
            "title": f"Article {i}",
            "category": "technology" if i % 2 == 0 else "science",
            "engagement_score": float(np.random.uniform(0.1, 5.0))
        }
    )
    for i in range(1, 10001)   # 10,000 items
]
client.upsert(collection_name="item_recommendations", points=items)

# Query: find top items for a user embedding, filtered by category
user_embedding = np.random.randn(256).tolist()

results = client.search(
    collection_name="item_recommendations",
    query_vector=user_embedding,
    query_filter=Filter(
        must=[FieldCondition(key="category", match=MatchValue(value="technology"))]
    ),
    limit=10,
    with_payload=True
)

for result in results:
    print(f"ID: {result.id:5d}  Score: {result.score:.4f}  "
          f"Title: {result.payload['title']:12s}  "
          f"Engagement: {result.payload['engagement_score']:.2f}")

    One warning about dot product at scale: HNSW index quality degrades more noticeably with dot product than with cosine when vectors have highly variable magnitudes. The HNSW graph is built using the dot product as the proximity function during construction, and if your training data has very large magnitude variance, the graph can develop poor connectivity in regions of the embedding space. Monitor your recall metrics in production and consider normalizing embeddings if recall drops below acceptable thresholds.

    3.5 Choosing the Right Metric for Your Embedding Model and Use Case

    The theoretical distinctions between metrics only matter if they translate into measurable differences in your system. Here is a framework for making the decision systematically rather than guessing.

    Step 1: Check the embedding model documentation. This is non-negotiable. Every major embedding model specifies its recommended similarity function:

    | Embedding Model | Recommended Metric | Normalized Output |

    |---|---|---|

    | OpenAI text-embedding-3-small/large | cosine | Near-unit (use cosine or dot product) |

    | OpenAI text-embedding-ada-002 | cosine | Near-unit |

    | Cohere embed-english-v3.0 | cosine | Yes (unit norm) |

    | sentence-transformers/all-MiniLM-L6-v2 | cosine | Yes (with normalize=True) |

    | CLIP (OpenAI) | cosine | Yes |

    | Google textembedding-gecko | dot product | No — magnitude is meaningful |

    | BGE-M3 (BAAI) | cosine or dot product (depends on mode) | Depends on usage |

    | DPR (Facebook) | dot product | No |

    Step 2: Test whether normalization changes your results. If an embedding model outputs nearly-normalized vectors, cosine and dot product will produce nearly identical rankings. You can verify this quickly:
