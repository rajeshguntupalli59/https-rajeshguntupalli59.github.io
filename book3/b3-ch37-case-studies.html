Chapter 37: Real-World Autonomous DBA Case Studies

The gap between autonomous database theory and production reality is where most platform engineering teams lose momentum. Architectural patterns that look elegant in whiteboards collapse against the friction of legacy schemas, bursty traffic, compliance requirements, and on-call engineers who override automation at the first sign of unfamiliarity. This chapter closes that gap by examining five real-world autonomous DBA deployments — drawn from high-scale SaaS platforms, financial institutions, healthcare data pipelines, and multi-cloud e-commerce systems — each built on PostgreSQL, SQL Server, or both. Every case study follows the same structure: the operational problem, the autonomous architecture chosen, the specific tooling and decision logic wired together, and the lessons that only surface in production. These are not success stories scrubbed clean for a conference talk. They include rollbacks, missed thresholds, and architectural pivots that led to better designs.

---

Case Study 1: Autonomous Failover and Connection Rerouting at a High-Frequency Trading Firm

A mid-sized proprietary trading firm running PostgreSQL 15 on bare metal needed sub-30-second failover with zero manual intervention. Their previous setup used Patroni with a manual promotion script triggered by on-call engineers — a process that averaged 4.5 minutes from detection to connection rerouting due to human latency and shift handoff ambiguity. The requirement from the trading desk: automated failover with automatic client rerouting in under 25 seconds, with a hard constraint that no transaction could be silently lost.

The architecture replaced manual triggers with a fully autonomous stack. Patroni remained the cluster manager, backed by etcd for distributed consensus. The critical addition was a custom Python daemon called trade-sentinel that consumed Patroni's REST API, pg_stat_replication, and a business-level heartbeat table written by the trading application. HAProxy sat in front of all database connections, with its backend configuration dynamically rewritten by trade-sentinel whenever Patroni reported a leader change.

The decision logic was explicit and layered:

1. Detection: If the Patroni leader endpoint returns HTTP 503 for three consecutive 2-second polls, or if the heartbeat table (trading.heartbeat) has not been updated within 8 seconds, enter failover evaluation mode.

2. Validation: Confirm the standby's replication lag is under 512KB and that etcd quorum is healthy. If either condition fails, escalate to PagerDuty and hold without promoting.

3. Promotion: Issue patronictl switchover via REST with --force, capture the new leader hostname, and rewrite HAProxy's backend stanza.

4. Connection rerouting: Reload HAProxy without dropping existing long-lived connections using haproxy -sf $(cat /var/run/haproxy.pid).

5. Verification: Run a synthetic write/read probe transaction against the new primary. If it fails within 10 seconds, re-escalate to human.

PostgreSQL:

-- Heartbeat monitoring table used by trade-sentinel
CREATE TABLE trading.heartbeat (
    node_id        TEXT PRIMARY KEY,
    last_written   TIMESTAMPTZ NOT NULL DEFAULT now(),
    sequence_num   BIGINT NOT NULL DEFAULT 0
);

-- Application writes every 5 seconds; sentinel reads this view
CREATE VIEW trading.heartbeat_lag AS
SELECT
    node_id,
    now() - last_written        AS lag_interval,
    EXTRACT(EPOCH FROM (now() - last_written)) AS lag_seconds
FROM trading.heartbeat;

-- Sentinel alert query: fetch nodes with lag > 8 seconds
SELECT node_id, lag_seconds
FROM trading.heartbeat_lag
WHERE lag_seconds > 8;

The trade-sentinel daemon managed promotion and HAProxy rewrite atomically:

import requests
import subprocess
import time
import logging
from datetime import datetime, timezone

PATRONI_API = "http://localhost:8008"
HAPROXY_CFG = "/etc/haproxy/haproxy.cfg"
HAPROXY_PID = "/var/run/haproxy.pid"
HEARTBEAT_THRESHOLD_SECONDS = 8
POLL_INTERVAL = 2
FAILURE_THRESHOLD = 3

logger = logging.getLogger("trade-sentinel")

def get_patroni_leader() -> str | None:
    try:
        resp = requests.get(f"{PATRONI_API}/master", timeout=3)
        if resp.status_code == 200:
            return resp.json().get("host")
    except requests.RequestException:
        return None
    return None

def trigger_patroni_switchover(candidate: str) -> bool:
    payload = {"leader": candidate, "force": True}
    try:
        resp = requests.post(
            f"{PATRONI_API}/switchover",
            json=payload,
            timeout=15
        )
        return resp.status_code == 200
    except requests.RequestException as e:
        logger.error(f"Switchover API call failed: {e}")
        return False

def rewrite_haproxy_backend(new_primary_host: str) -> None:
    with open(HAPROXY_CFG, "r") as f:
        cfg = f.read()
    # Replace the primary backend server line
    import re
    cfg = re.sub(
        r"(server primary\s+)\S+(:5432)",
        rf"\g<1>{new_primary_host}\g<2>",
        cfg
    )
    with open(HAPROXY_CFG, "w") as f:
        f.write(cfg)
    with open(HAPROXY_PID, "r") as f:
        old_pid = f.read().strip()
    subprocess.run(
        ["haproxy", "-f", HAPROXY_CFG, "-sf", old_pid],
        check=True
    )
    logger.info(f"HAProxy reloaded with new primary: {new_primary_host}")

def run_sentinel():
    consecutive_failures = 0
    while True:
        leader = get_patroni_leader()
        if leader is None:
            consecutive_failures += 1
            logger.warning(
                f"Patroni leader unreachable — failure count: {consecutive_failures}"
            )
        else:
            consecutive_failures = 0

        if consecutive_failures >= FAILURE_THRESHOLD:
            logger.critical("Threshold exceeded — initiating autonomous failover")
            # In production: query etcd for best replica before choosing candidate
            success = trigger_patroni_switchover("replica-01")
            if success:
                time.sleep(5)
                new_leader = get_patroni_leader()
                if new_leader:
                    rewrite_haproxy_backend(new_leader)
                else:
                    logger.error("Failover succeeded but new leader undetectable — escalating")
            consecutive_failures = 0
        time.sleep(POLL_INTERVAL)

if __name__ == "__main__":
    logging.basicConfig(level=logging.INFO)
    run_sentinel()

The first production failover completed in 18.4 seconds. The second, triggered six weeks later by a kernel panic on the primary host, completed in 21.1 seconds — within specification despite higher system load. The lessons were two-fold: the heartbeat table proved far more reliable as a detection signal than Patroni's own API, which exhibited a 3–4 second delay in reporting leader loss during network partitions; and HAProxy's graceful reload was safe only because the trading application used persistent connections with application-level retry — without that, the reload would have dropped in-flight orders.

Case Study 2: Self-Healing Index Management for a SaaS Analytics Platform (SQL Server)

A B2B analytics SaaS company ran SQL Server 2022 on Azure VMs (Standard_D32s_v5 instances) serving 400+ tenant databases within a single instance via schema-per-tenant isolation. Index fragmentation was causing query degradation across tenants at different rates — a scheduled maintenance window applied the same REBUILD/REORGANIZE policy to all tenants, causing unpredictable latency spikes during business hours for tenants in different time zones. The solution: a fully autonomous index management loop that evaluated each tenant schema independently and acted without scheduling.

The autonomous index agent ran as a SQL Server Agent job on a 15-minute cycle. Its logic tree:

  • → If `avg_fragmentation_in_percent > 30` AND `page_count > 1000`: REBUILD ONLINE
  • → If `avg_fragmentation_in_percent > 10` AND `page_count > 500`: REORGANIZE
  • → If `page_count <= 500`: skip (small table overhead not worth it)
  • → If estimated rebuild duration (based on historical stats) would exceed 4 minutes: defer to off-peak window specific to tenant's primary geography

    • SQL Server:

    -- Autonomous index maintenance procedure per tenant schema
CREATE OR ALTER PROCEDURE dbo.usp_AutoIndexMaintenance
    @TenantSchema  NVARCHAR(128),
    @DryRun        BIT = 0
AS
BEGIN
    SET NOCOUNT ON;

    DECLARE @SQL           NVARCHAR(MAX);
    DECLARE @TableName     NVARCHAR(256);
    DECLARE @IndexName     NVARCHAR(256);
    DECLARE @Fragmentation FLOAT;
    DECLARE @PageCount     BIGINT;
    DECLARE @Action        NVARCHAR(50);

    DECLARE index_cursor CURSOR LOCAL FAST_FORWARD FOR
        SELECT
            QUOTENAME(s.name) + '.' + QUOTENAME(t.name)    AS full_table_name,
            QUOTENAME(i.name)                               AS index_name,
            ips.avg_fragmentation_in_percent,
            ips.page_count,
            CASE
                WHEN ips.avg_fragmentation_in_percent > 30
                     AND ips.page_count > 1000              THEN 'REBUILD'
                WHEN ips.avg_fragmentation_in_percent > 10
                     AND ips.page_count > 500               THEN 'REORGANIZE'
                ELSE 'SKIP'
            END AS recommended_action
        FROM sys.dm_db_index_physical_stats(
                 DB_ID(), NULL, NULL, NULL, 'LIMITED') ips
        JOIN sys.indexes i
            ON ips.object_id = i.object_id
            AND ips.index_id = i.index_id
        JOIN sys.tables t
            ON i.object_id = t.object_id
        JOIN sys.schemas s
            ON t.schema_id = s.schema_id
        WHERE s.name     = @TenantSchema
          AND i.name     IS NOT NULL
          AND ips.page_count > 500;

    OPEN index_cursor;
    FETCH NEXT FROM index_cursor
        INTO @TableName, @IndexName, @Fragmentation, @PageCount, @Action;

    WHILE @@FETCH_STATUS = 0
    BEGIN
        IF @Action = 'REBUILD'
        BEGIN
            SET @SQL = N'ALTER INDEX ' + @IndexName
                     + N' ON ' + @TableName
                     + N' REBUILD WITH (ONLINE = ON, MAXDOP = 4);';
        END
        ELSE IF @Action = 'REORGANIZE'
        BEGIN
            SET @SQL = N'ALTER INDEX ' + @IndexName
                     + N' ON ' + @TableName
                     + N' REORGANIZE;';
        END

        IF @Action <> 'SKIP'
        BEGIN
            -- Audit the decision before acting
            INSERT INTO dbo.autonomous_index_log
                (tenant_schema, table_name, index_name,
                 fragmentation_pct, page_count, action_taken,
                 executed_sql, log_time)
            VALUES
                (@TenantSchema, @TableName, @IndexName,
                 @Fragmentation, @PageCount, @Action,
                 @SQL, SYSUTCDATETIME());

            IF @DryRun = 0
                EXEC sp_executesql @SQL;
        END

        FETCH NEXT FROM index_cursor
            INTO @TableName, @IndexName, @Fragmentation, @PageCount, @Action;
    END

    CLOSE index_cursor;
    DEALLOCATE index_cursor;
END;
GO

-- Master orchestrator job iterates all tenant schemas
DECLARE @TenantSchema NVARCHAR(128);
DECLARE tenant_cursor CURSOR LOCAL FAST_FORWARD FOR
    SELECT schema_name
    FROM   dbo.tenant_registry
    WHERE  maintenance_enabled = 1
      AND  GETUTCDATE() BETWEEN maintenance_window_start
                             AND maintenance_window_end;

OPEN tenant_cursor;
FETCH NEXT FROM tenant_cursor INTO @TenantSchema;
WHILE @@FETCH_STATUS = 0
BEGIN
    EXEC dbo.usp_AutoIndexMaintenance
        @TenantSchema = @TenantSchema,
        @DryRun       = 0;
    FETCH NEXT FROM tenant_cursor INTO @TenantSchema;
END
CLOSE tenant_cursor;
DEALLOCATE tenant_cursor;

    The tenant_registry table stored each tenant's primary geography and the associated UTC maintenance window — a 4-hour block during their 2–6 AM local time. The autonomous agent queried maintenance_window_start and maintenance_window_end before acting, ensuring a tenant headquartered in Tokyo received index rebuilds during Tokyo's off-hours, not London's. This was the architectural insight that the previous global maintenance window lacked entirely.

    After 90 days in production, average query duration across tenants dropped 31% during peak hours, and index-related blocking incidents — previously averaging 4 per week — dropped to zero. The autonomous_index_log table became the team's primary evidence artifact in customer SLA reviews.

    Case Study 3: Autonomous Query Plan Stabilization in a Healthcare Data Warehouse

    A regional healthcare network operated a PostgreSQL 16 data warehouse — approximately 12TB across 60 tables — that fed clinical dashboards consumed by 300+ concurrent analysts. The platform team faced a recurring problem: after statistics updates (triggered by autovacuum), query plans for 8–12 critical reporting queries would regress. A plan that previously used an index scan would switch to a sequential scan on a 900M-row fact table, spiking query time from 4 seconds to 3.5 minutes. The on-call response was manual: a DBA would run EXPLAIN ANALYZE, identify the regression, and inject SET enable_seqscan = off at the session level or create a pg_hint_plan override. This manual process averaged 45 minutes of analyst impact.

    The autonomous plan stabilization system worked in three tiers:

    Tier 1 — Continuous baseline capture: A background process recorded query fingerprints and their execution plans using pg_stat_statements and auto_explain, storing normalized plan JSON in a monitoring schema.

    Tier 2 — Regression detection: Every 5 minutes, a Python process compared recent execution time (90th percentile over the last 15 minutes) against the rolling 7-day baseline for each fingerprint. If the ratio exceeded 2.5x, it flagged a potential regression.

    Tier 3 — Autonomous remediation: The system retrieved the last known good plan, extracted the join type and scan method differences, and automatically injected a pg_hint_plan comment into the affected query's scheduled invocation wrapper.

    PostgreSQL:

    -- Plan history capture: store normalized plan snapshots
CREATE TABLE monitoring.query_plan_history (
    fingerprint       BIGINT       NOT NULL,
    query_sample      TEXT         NOT NULL,
    plan_json         JSONB        NOT NULL,
    p90_exec_ms       NUMERIC(12,3) NOT NULL,
    captured_at       TIMESTAMPTZ  NOT NULL DEFAULT now(),
    is_baseline       BOOLEAN      NOT NULL DEFAULT FALSE,
    PRIMARY KEY (fingerprint, captured_at)
);

-- Regression detection view: compare recent p90 to 7-day baseline
CREATE OR REPLACE VIEW monitoring.plan_regression_candidates AS
WITH baseline AS (
    SELECT
        fingerprint,
        AVG(p90_exec_ms)  AS baseline_p90_ms
    FROM monitoring.query_plan_history
    WHERE is_baseline = TRUE
      AND captured_at >= now() - INTERVAL '7 days'
    GROUP BY fingerprint
),
recent AS (
    SELECT
        fingerprint,
        AVG(p90_exec_ms)  AS recent_p90_ms
    FROM monitoring.query_plan_history
    WHERE is_baseline = FALSE
      AND captured_at >= now() - INTERVAL '15 minutes'
    GROUP BY fingerprint
)
SELECT
    r.fingerprint,
    b.baseline_p90_ms,
    r.recent_p90_ms,
    r.recent_p90_ms / NULLIF(b.baseline_p90_ms, 0) AS regression_ratio
FROM recent r
JOIN baseline b USING (fingerprint)
WHERE r.recent_p90_ms / NULLIF(b.baseline_p90_ms, 0) > 2.5;

    The Python remediation agent retrieved the baseline plan, parsed the scan node types from the JSONB, and synthesized a pg_hint_plan directive:

    import psycopg2
import json
import logging
from typing import Optional

logger = logging.getLogger("plan-stabilizer")

def extract_hint_from_plan(plan_json: dict) -> Optional[str]:
    """
    Walk the plan tree and extract scan/join hints.
    Returns a pg_hint_plan directive string or None.
    """
    hints = []

    def walk(node: dict):
        node_type = node.get("Node Type", "")
        relation = node.get("Relation Name", "")

        if node_type == "Index Scan" and relation:
            idx = node.get("Index Name", "")
            if idx:
                hints.append(f"IndexScan({relation} {idx})")
        elif node_type == "Seq Scan" and relation:
            # Baseline used index — force it back
            pass
        elif "Hash Join" in node_type:
            inner = node.get("Plans", [{}])[1].get("Relation Name", "")
            outer = node.get("Plans", [{}])[0].get("Relation Name", "")
            if inner and outer:
                hints.append(f"HashJoin({outer} {inner})")

        for child in node.get("Plans", []):
            walk(child)

    walk(plan_json.get("Plan", {}))
    return "/*+ " + " ".join(hints) + " */" if hints else None

def apply_plan_stabilization(conn_str: str, fingerprint: int) -> None:
    conn = psycopg2.connect(conn_str)
    cur = conn.cursor()

    # Retrieve last known good baseline plan
    cur.execute("""
        SELECT plan_json, query_sample
        FROM monitoring.query_plan_history
        WHERE fingerprint = %s
          AND is_baseline = TRUE
        ORDER BY captured_at DESC
        LIMIT 1
    """, (fingerprint,))
    row = cur.fetchone()

    if not row:
        logger.warning(f"No baseline plan found for fingerprint {fingerprint}")
        return

    plan_json, query_sample = row
    hint = extract_hint_from_plan(plan_json)

    if hint:
        # Store hint in the plan override registry — applications
        # read this table to prepend hints at query construction time
        cur.execute("""
            INSERT INTO monitoring.plan_override_registry
                (fingerprint, hint_directive, applied_at, source)
            VALUES (%s, %s, now(), 'autonomous-plan-stabilizer')
            ON CONFLICT (fingerprint) DO UPDATE
                SET hint_directive = EXCLUDED.hint_directive,
                    applied_at     = EXCLUDED.applied_at
        """, (fingerprint, hint))
        conn.commit()
        logger.info(
            f"Plan hint applied for fingerprint {fingerprint}: {hint}"
        )
    else:
        logger.warning(
            f"Could not synthesize hint for fingerprint {fingerprint} — escalating"
        )

    cur.close()
    conn.close()

    The production deployment stabilized 11 of the 12 identified regression fingerprints autonomously. The twelfth required manual intervention — the regression was caused by a schema change (a new partial index that changed cardinality estimates) rather than statistics drift, and the baseline plan was genuinely suboptimal for the new schema. This distinction — regression caused by data drift versus regression caused by schema change — became a classification dimension added to the detection layer in the next iteration.

    Case Study 4: Autonomous Capacity Scaling in a Multi-Tenant E-Commerce Platform (PostgreSQL on Kubernetes)

    A direct-to-consumer e-commerce company ran PostgreSQL 16 via the CloudNativePG operator on a Kubernetes cluster (AWS EKS) spanning three availability zones. Their traffic profile was violently bursty: normal load was 800 TPS, but flash sales and influencer-driven traffic spikes routinely hit 12,000–15,000 TPS within 90 seconds of an event trigger. Vertical scaling was too slow (node provisioning took 7–12 minutes); the autonomous solution was horizontal read scaling through automated replica addition and pgBouncer pool expansion, combined with query routing logic that redirected read queries based on real-time lag monitoring.

    The scaling loop was implemented as a Kubernetes controller (using the kopf framework in Python) that watched a custom resource PostgreSQLScalePolicy and acted on Prometheus metrics fed through the Prometheus Adapter for Kubernetes:

  • → When `pg_stat_activity_count{state="active"} / max_connections > 0.75` for 60 seconds: scale pgBouncer pool size up by 20%, add one read replica via CloudNativePG's `instances` field patch.
  • → When replica lag on any standby exceeds 30MB: do not add more replicas; instead reduce connection pressure by throttling non-critical background jobs via a priority queue table.
  • → When `pg_stat_activity_count{state="active"} / max_connections < 0.35` for 300 seconds: scale down one replica if replica count > 2.

    • PostgreSQL:

    -- Connection pressure monitoring with replica lag
SELECT
    sa.count_active,
    sa.max_connections,
    ROUND(sa.count_active::NUMERIC / sa.max_connections * 100, 2) AS utilization_pct,
    sr.application_name,
    pg_size_pretty(sr.sent_lsn - sr.replay_lsn)  AS replication_lag_bytes,
    (sr.sent_lsn - sr.replay_lsn)                AS lag_bytes_raw
FROM (
    SELECT
        COUNT(*) FILTER (WHERE state = 'active')  AS count_active,
        setting::INT                               AS max_connections
    FROM pg_stat_activity
    CROSS JOIN pg_settings
    WHERE name = 'max_connections'
    GROUP BY setting
) sa
CROSS JOIN pg_stat_replication sr
ORDER BY lag_bytes_raw DESC;
