Chapter 32: Autonomous PostgreSQL Systems

PostgreSQL was never designed to manage itself — but the ecosystem that has grown around it over the past decade has quietly assembled every component necessary to build a fully autonomous system. Patroni handles HA decisions. pg_auto_failover handles topology changes. pgBouncer manages connection routing. Prometheus and Alertmanager observe and react. Kubernetes operators orchestrate lifecycle events. What most teams lack is not the tooling but the architecture that ties these components into a coherent decision-making system: one that observes its own state, reasons about degradation, and takes corrective action without a human in the loop. This chapter builds that architecture from the ground up, covering autonomous topology management, self-tuning memory and parallelism, automated vacuum and bloat remediation, connection intelligence, and the control-plane patterns that make human intervention the exception rather than the rule.

---

Autonomous Topology Management with Patroni and pg_auto_failover

The first requirement of any autonomous PostgreSQL system is that it knows, at all times, exactly what its topology looks like — and can reshape that topology without human involvement when conditions demand it. Two tools dominate this space: Patroni, which uses a distributed consensus store (etcd, Consul, or ZooKeeper) to manage leader election and replication state, and pg_auto_failover, which embeds topology logic directly into a monitor process that coordinates a simpler two- or three-node architecture. Each has its place.

Patroni is appropriate for clusters where topology complexity is high: multiple standbys, cascading replication, geographic distribution across availability zones, or integration with Kubernetes operators like the Zalando postgres-operator or CloudNativePG. pg_auto_failover is better suited for smaller, high-velocity deployments where simplicity and determinism matter more than flexibility.

The autonomous pattern requires not just failover capability but *policy-driven* failover — the system must decide when to fail over based on observable signals, not just a timeout. A naive timeout-based failover system will promote a standby that has fallen behind due to network partition, producing a split-brain scenario or data loss. An autonomous system checks replication lag, WAL streaming state, and synchronous commit configuration before making a promotion decision.

In a Patroni-managed cluster, the DCS (distributed configuration store) is the source of truth. The following illustrates how to query the current cluster state programmatically, which forms the basis for any autonomous control plane decision:

import requests
import json

PATRONI_API = "http://patroni-primary:8008"

def get_cluster_state():
    response = requests.get(f"{PATRONI_API}/cluster", timeout=5)
    state = response.json()
    members = state.get("members", [])
    topology = {}
    for member in members:
        topology[member["name"]] = {
            "role": member["role"],
            "state": member["state"],
            "lag": member.get("lag", 0),
            "timeline": member.get("timeline"),
            "host": member["host"],
        }
    return topology

def evaluate_failover_eligibility(topology):
    eligible = []
    for name, info in topology.items():
        if info["role"] == "replica" and info["state"] == "running":
            if info["lag"] is not None and info["lag"] < 10485760:  # 10 MB lag threshold
                eligible.append((name, info["lag"]))
    return sorted(eligible, key=lambda x: x[1])

topology = get_cluster_state()
candidates = evaluate_failover_eligibility(topology)
print(f"Failover candidates by lag: {candidates}")

This decision logic — promote the replica with the lowest lag that is below an acceptable threshold — replaces the human judgment call that most teams make manually during an incident. The autonomous system runs this evaluation continuously and can trigger a patronictl switchover or patronictl failover via the REST API when the primary health check fails.

For clusters using pg_auto_failover, the monitor node exposes its own state machine. The autonomous controller subscribes to state transitions and enforces policy constraints before allowing promotion to proceed:

PostgreSQL:

-- Query pg_auto_failover monitor state (run against the monitor node)
SELECT
    nodename,
    nodehost,
    nodeport,
    goalstate,
    reportedstate,
    reportedlsn,
    reportedrepstate,
    health,
    pg_size_pretty(reportedlsn - '0/0'::pg_lsn) AS lsn_position
FROM pgautofailover.node
ORDER BY formation_id, groupid, nodeid;

SQL Server:

-- Equivalent: query AG replica status for autonomous evaluation
SELECT
    ag.name AS ag_name,
    ar.replica_server_name,
    ars.role_desc,
    ars.operational_state_desc,
    ars.connected_state_desc,
    ars.synchronization_health_desc,
    ars.last_commit_time,
    drs.synchronization_state_desc,
    drs.log_send_queue_size,
    drs.redo_queue_size
FROM sys.availability_groups ag
JOIN sys.availability_replicas ar ON ag.group_id = ar.group_id
JOIN sys.dm_hadr_availability_replica_states ars ON ar.replica_id = ars.replica_id
LEFT JOIN sys.dm_hadr_database_replica_states drs ON ar.replica_id = drs.replica_id
ORDER BY ag.name, ars.role_desc;

A mature autonomous topology system also enforces a *failover budget* — a maximum number of automated failovers permitted within a rolling time window. This prevents cascading failures caused by a flapping primary from triggering repeated promotions, each of which carries its own risk. The control plane should record every topology change in an immutable audit log, decrement from the budget, and escalate to human on-call if the budget is exhausted. Autonomy does not mean recklessness; it means that routine, well-understood failures are resolved without human latency.

Self-Tuning Memory, Parallelism, and Configuration Drift Correction

PostgreSQL's performance is governed by a relatively small set of configuration parameters — shared_buffers, work_mem, effective_cache_size, max_parallel_workers_per_gather, wal_buffers, checkpoint_completion_target — but the optimal values for these parameters shift as workloads change. A database serving an OLTP workload at 09:00 may be processing large analytical aggregations by 23:00. A static postgresql.conf will be wrong for both.

Autonomous configuration management addresses this through *workload fingerprinting* and *parameter policy enforcement*. The system continuously profiles the running workload, classifies it against known archetypes, and adjusts parameters within defined safety bounds using ALTER SYSTEM — which writes to postgresql.auto.conf and survives restarts without touching the managed configuration baseline.

The workload fingerprinting logic queries pg_stat_statements and derives a workload signature:

PostgreSQL:

-- Classify current workload: ratio of sequential scans, index scans, sort operations
SELECT
    round(100.0 * sum(seq_scan) / nullif(sum(seq_scan) + sum(idx_scan), 0), 2) AS pct_seq_scan,
    round(100.0 * sum(idx_scan) / nullif(sum(seq_scan) + sum(idx_scan), 0), 2) AS pct_idx_scan,
    pg_size_pretty(sum(blks_read) * 8192) AS physical_reads,
    pg_size_pretty(sum(blks_hit) * 8192) AS buffer_hits,
    round(100.0 * sum(blks_hit) / nullif(sum(blks_hit) + sum(blks_read), 0), 2) AS buffer_hit_ratio
FROM pg_stat_user_tables;

SQL Server:

-- Equivalent workload classification from DMVs
SELECT
    SUM(user_seeks + user_scans + user_lookups) AS total_reads,
    SUM(user_scans) AS full_scans,
    SUM(user_seeks) AS index_seeks,
    ROUND(100.0 * SUM(user_scans) /
        NULLIF(SUM(user_seeks + user_scans + user_lookups), 0), 2) AS pct_full_scans,
    SUM(range_scan_count) AS range_scans,
    SUM(singleton_lookup_count) AS point_lookups
FROM sys.dm_db_index_usage_stats
WHERE database_id = DB_ID();

When pct_seq_scan exceeds 60 percent combined with a buffer hit ratio below 85 percent, the system classifies the workload as analytically dominated and triggers a parameter adjustment pass. The automation increases effective_cache_size to reflect available OS page cache, raises work_mem for sort-heavy operations, and increases max_parallel_workers_per_gather toward the defined ceiling.

The critical discipline here is *bounded autonomy*: the system may never set shared_buffers above 40 percent of total system RAM, work_mem beyond a per-session ceiling that would exhaust memory under maximum connection count, or max_parallel_workers beyond the number of available CPU cores minus two (reserved for maintenance tasks). These bounds are encoded as policy, not as hardcoded values, so they can be updated through the configuration management plane without touching the automation code.

Configuration drift — the condition where postgresql.conf, postgresql.auto.conf, and runtime SHOW values have diverged from the declared policy baseline — must be detected and remediated automatically. The following illustrates an autonomous drift detection pass:

import subprocess
import psycopg2

POLICY_BASELINE = {
    "shared_buffers": "4GB",
    "work_mem": "64MB",
    "checkpoint_completion_target": "0.9",
    "wal_compression": "on",
    "max_parallel_workers_per_gather": "4",
}

def check_config_drift(conn_str: str) -> dict:
    conn = psycopg2.connect(conn_str)
    cur = conn.cursor()
    drift = {}
    for param, expected in POLICY_BASELINE.items():
        cur.execute("SHOW %s" % param)
        actual = cur.fetchone()[0]
        if actual != expected:
            drift[param] = {"expected": expected, "actual": actual}
    cur.close()
    conn.close()
    return drift

def remediate_drift(conn_str: str, drift: dict):
    conn = psycopg2.connect(conn_str)
    cur = conn.cursor()
    for param, values in drift.items():
        print(f"Correcting {param}: {values['actual']} -> {values['expected']}")
        cur.execute(f"ALTER SYSTEM SET {param} = %s", (values["expected"],))
    conn.commit()
    cur.execute("SELECT pg_reload_conf()")
    conn.commit()
    cur.close()
    conn.close()

drift = check_config_drift("host=pg-primary dbname=postgres user=automation")
if drift:
    remediate_drift("host=pg-primary dbname=postgres user=automation", drift)

This pattern runs as a scheduled job — every 15 minutes in production environments — and reports any remediations to the audit log with before/after values and timestamps. Parameters that require a restart rather than a reload are flagged but not automatically applied; they are queued for execution during the next maintenance window, which the autonomous scheduler determines based on traffic patterns derived from connection pool metrics.

Autonomous Vacuum Management and Bloat Remediation

Autovacuum is PostgreSQL's built-in mechanism for reclaiming dead tuples and maintaining planner statistics, but its default configuration is conservative. Left unmanaged, high-write OLTP tables will accumulate dead tuple bloat that degrades query performance, inflates table sizes, and — in the most severe case involving transaction ID wraparound — causes the database to enter read-only mode and refuse new connections. An autonomous system cannot tolerate any of these outcomes.

The autonomous vacuum controller operates on two tiers: reactive (responding to tables that have already exceeded bloat or dead tuple thresholds) and proactive (adjusting autovacuum parameters per-table based on observed write velocity before problems develop). Most teams only implement the reactive tier. The proactive tier is what distinguishes a truly autonomous system.

The foundation is a continuous dead tuple monitor that queries pg_stat_user_tables and cross-references with actual table sizes to compute a dead tuple ratio and an estimated bloat percentage:

PostgreSQL:

-- Autonomous vacuum priority queue: tables needing immediate attention
SELECT
    schemaname,
    relname,
    n_live_tup,
    n_dead_tup,
    round(100.0 * n_dead_tup / nullif(n_live_tup + n_dead_tup, 0), 2) AS dead_pct,
    last_vacuum,
    last_autovacuum,
    last_analyze,
    pg_size_pretty(pg_total_relation_size(schemaname || '.' || relname)) AS total_size,
    CASE
        WHEN n_dead_tup > 10000000 THEN 'CRITICAL'
        WHEN n_dead_tup > 1000000 THEN 'WARNING'
        WHEN round(100.0 * n_dead_tup / nullif(n_live_tup + n_dead_tup, 0), 2) > 20 THEN 'WARNING'
        ELSE 'OK'
    END AS vacuum_status
FROM pg_stat_user_tables
WHERE n_live_tup + n_dead_tup > 10000
ORDER BY n_dead_tup DESC
LIMIT 25;

SQL Server:

-- SQL Server equivalent: index fragmentation as the analog to dead tuple bloat
SELECT
    OBJECT_SCHEMA_NAME(ips.object_id) AS schema_name,
    OBJECT_NAME(ips.object_id) AS table_name,
    i.name AS index_name,
    ips.index_type_desc,
    ROUND(ips.avg_fragmentation_in_percent, 2) AS fragmentation_pct,
    ips.page_count,
    ips.record_count,
    CASE
        WHEN ips.avg_fragmentation_in_percent > 30 THEN 'REBUILD'
        WHEN ips.avg_fragmentation_in_percent > 10 THEN 'REORGANIZE'
        ELSE 'OK'
    END AS recommended_action
FROM sys.dm_db_index_physical_stats(DB_ID(), NULL, NULL, NULL, 'LIMITED') ips
JOIN sys.indexes i ON ips.object_id = i.object_id AND ips.index_id = i.index_id
WHERE ips.page_count > 1000
ORDER BY ips.avg_fragmentation_in_percent DESC;

When a table enters CRITICAL status, the autonomous controller evaluates whether to issue a manual VACUUM (non-blocking, reclaims dead tuples), VACUUM ANALYZE (reclaim plus statistics refresh), or VACUUM FULL (full rewrite, takes an exclusive lock — almost never appropriate in a live system). For tables where bloat has reached a structural level — the actual physical size has diverged significantly from the expected live-data size — the system invokes pg_repack via a background job, which performs the equivalent of VACUUM FULL without taking an exclusive lock.

The proactive tier is more subtle. For each high-write table, the system sets storage parameters directly to override autovacuum defaults:

-- Autonomous per-table autovacuum tuning for a high-write events table
ALTER TABLE events SET (
    autovacuum_vacuum_scale_factor = 0.01,    -- vacuum at 1% dead tuples (vs default 20%)
    autovacuum_analyze_scale_factor = 0.005,  -- analyze at 0.5% changes
    autovacuum_vacuum_cost_delay = 2,         -- ms; more aggressive than default 20ms
    autovacuum_vacuum_insert_scale_factor = 0.05
);

The automation derives these values dynamically: for a table receiving more than 50,000 writes per hour, autovacuum_vacuum_scale_factor is set to 0.01; above 500,000 writes per hour, it is set to 0.001. The autovacuum_vacuum_cost_delay is inversely proportional to available I/O headroom as measured from the storage layer metrics. If the system is under I/O pressure, cost delay increases to reduce autovacuum's I/O footprint; when I/O is available, it decreases to allow faster cleanup.

Transaction ID wraparound — where a database approaches the 2-billion transaction ID limit and PostgreSQL begins emitting "database is not accepting commands" errors — is a failure that should never reach production in an autonomous system. The controller monitors age(datfrozenxid) against a tiered alert schedule: at 500 million transactions of age, it increases autovacuum aggressiveness; at 1 billion, it triggers a forced VACUUM FREEZE on the affected tables; at 1.5 billion, it pages the on-call team regardless of the failover budget because this condition requires human decision-making about maintenance windows.

Connection Intelligence: pgBouncer Autonomous Routing and Pool Sizing

Connection management is one of the most visible failure modes in PostgreSQL under load. Each PostgreSQL backend is an OS process consuming memory; at hundreds or thousands of direct connections, the overhead of context switching and memory pressure degrades performance for everyone. pgBouncer sits in front of PostgreSQL and multiplexes application connections onto a smaller pool of actual database connections — but a static pgBouncer configuration becomes a bottleneck as workloads evolve.

An autonomous connection management layer continuously monitors pool utilization, client wait times, and backend saturation, and adjusts pgBouncer configuration dynamically. pgBouncer exposes a management interface via a pseudo-database called pgbouncer that accepts control commands at runtime:

PostgreSQL:

-- Connect to pgBouncer admin interface: psql -p 6432 pgbouncer
-- Query current pool state
SHOW POOLS;
-- Output includes: database, user, cl_active, cl_waiting, sv_active, sv_idle, sv_used, maxwait

-- Increase pool size for a specific database dynamically
SET pool_size=50 for database=orders_db;

-- View current client connections and wait times
SHOW CLIENTS;

-- Check server connections to PostgreSQL backend
SHOW SERVERS;

The autonomous pool controller interprets these metrics and applies the following decision logic: if cl_waiting exceeds 5 percent of cl_active for more than 30 consecutive seconds, pool_size is incremented by 10 percent, subject to a maximum that is calculated from max_connections minus reserved connections for administrative access and autovacuum workers. If sv_idle exceeds 40 percent of pool_size for more than 5 minutes, the pool is shrunk by 10 percent to reclaim backend resources. This hysteresis — requiring a sustained condition before acting, and acting conservatively in both directions — prevents the pool from oscillating continuously.

Autonomous routing adds another dimension: the connection layer should direct read-only queries to standby replicas when replication lag is below an acceptable threshold, and fall back to the primary when standbys are lagging or unavailable. HAProxy or Nginx can perform this routing based on health check endpoints exposed by Patroni (/master and /replica), but a more intelligent implementation uses a connection proxy that is aware of the query intent and the current replication state:

import psycopg2
import requests

PATRONI_PRIMARY = "http://pg-node-1:8008"
PATRONI_REPLICA_1 = "http://pg-node-2:8008"

PRIMARY_DSN = "host=pg-node-1 port=5432 dbname=app user=app_user"
REPLICA_DSN = "host=pg-node-2 port=5432 dbname=app user=app_user options=-c default_transaction_read_only=on"
MAX_ACCEPTABLE_LAG_BYTES = 5 * 1024 * 1024  # 5 MB

def get_replica_lag_bytes(patroni_url: str) -> int:
    try:
        resp = requests.get(f"{patroni_url}/patroni", timeout=2)
        data = resp.json()
        return data.get("replication", [{}])[0].get("sent_lsn", 0)
    except Exception:
        return float("inf")

def get_connection(read_only: bool = False) -> psycopg2.extensions.connection:
    if read_only:
        lag = get_replica_lag_bytes(PATRONI_REPLICA_1)
        if lag < MAX_ACCEPTABLE_LAG_BYTES:
            try:
                return psycopg2.connect(REPLICA_DSN)
            except Exception:
                pass  # fall through to primary
    return psycopg2.connect(PRIMARY_DSN)

This routing logic is the foundation of a connection intelligence layer. In production, it runs inside a pgBouncer-adjacent sidecar or a lightweight connection routing service, and the lag threshold itself is tunable per application tier: analytics applications can tolerate 60 seconds of replica lag; payment processing applications must use the primary exclusively.

Autonomous Index Management: Creation, Elimination, and Consolidation

Index sprawl is one of the most common performance debt patterns in mature PostgreSQL installations. Tables accumulate indexes added reactively during incidents, never revisited afterward. Write performance degrades as every index must be maintained on INSERT and UPDATE. Query plans become unpredictable as the optimizer weighs an expanding set of options. An autonomous index management system approaches this problem from both ends: identifying missing indexes that would improve query performance, and identifying existing indexes that are unused, duplicated, or partially redundant.

The missing index signal comes primarily from pg_stat_statements combined with EXPLAIN plan analysis. When a query with high total_time shows sequential scans on large tables, the autonomous system generates a candidate index, validates it on a standby using EXPLAIN (not EXPLAIN ANALYZE — we never run the query again just to test), and scores it against the existing index set for redundancy:

PostgreSQL:

-- Identify high-cost sequential scans from pg_stat_statements
SELECT
    pss.userid::regrole AS username,
    pss.query,
    round(pss.total_exec_time::numeric, 2) AS total_exec_ms,
    pss.calls,
    round((pss.total_exec_time / pss.calls)::numeric, 2) AS avg_exec_ms,
    round(pss.rows / pss.calls::numeric, 2) AS avg_rows,
    pss.shared_blks_read,
    pss.shared_blks_hit,
    round(100.0 * pss.shared_blks_read /
        nullif(pss.shared_blks_read + pss.shared_blks_hit, 0), 2) AS physical_read_pct
FROM pg_stat_statements pss
WHERE pss.calls > 100
  AND pss.total_exec_time > 10000
  AND pss.shared_blks_read > pss.shared_blks_hit
ORDER BY pss.total_exec_time DESC
LIMIT 20;
