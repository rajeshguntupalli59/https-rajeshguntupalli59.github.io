Chapter 31: Multi-Modal Search: Text and Image Together

Most production search systems today handle one modality at a time — a text query searches text, an image query searches images. But users do not experience information in those silos. A retail customer uploads a photo of shoes and types "like these but in red." A medical researcher searches with an MRI scan and a clinical note simultaneously. A news archive needs to find articles where the image and headline tell contradictory stories. These compound queries require a database layer that understands both text and images as first-class embedding types and can combine them during retrieval. This chapter covers the embedding models that bridge the modality gap, the storage and indexing strategies for mixed-type vectors, the query architectures that fuse signals from both dimensions, and the production patterns that make multi-modal search scale reliably.

---

The Embedding Bridge: How Text and Images Share a Vector Space

The foundational insight behind multi-modal search is that text and images can be embedded into the *same* vector space — a shared coordinate system where semantic proximity means the same thing regardless of modality. A vector for the text "golden retriever playing fetch" and a vector for an actual photograph of that scene should land close together in this space. This is not obvious from first principles; it required learning a joint embedding model from hundreds of millions of image-caption pairs.

CLIP and Its Descendants

OpenAI's CLIP (Contrastive Language-Image Pre-training, 2021) established the practical template. CLIP trains a text encoder and an image encoder jointly using contrastive loss — during training, the model pulls matching image-text pairs together while pushing non-matching pairs apart. The result is two encoders that produce vectors in the same 512- or 768-dimensional space.

For production use, the relevant CLIP derivatives include:

| Model | Dimensions | Notes |

|---|---|---|

| openai/clip-vit-base-patch32 | 512 | Baseline; widely benchmarked |

| openai/clip-vit-large-patch14 | 768 | Better quality, higher latency |

| laion/CLIP-ViT-H-14-laion2B-s32B-b79K | 1024 | Trained on LAION-2B; strong zero-shot |

| sentence-transformers/clip-ViT-B-32 | 512 | HuggingFace wrapper; easier pipeline |

| Google's SigLIP | 1152 | Sigmoid loss variant; better recall at scale |

| OpenAI text-embedding-3-large | 3072 | Text only; not image-capable |

The practical implication for database architects: you are storing vectors from the *same* model family in the same table or collection. If you embed images with clip-vit-large-patch14, you must embed all query text with that same model's text encoder. Mixing encoders from different CLIP checkpoints produces meaningless distance comparisons — equivalent to storing lengths in meters in one column and feet in another and comparing them without conversion.

Dimensionality and Distance Metrics

CLIP vectors use cosine similarity as the natural distance metric because the encoders produce unit-normalized vectors by default. In pgvector terms, that means using the vector type with <=> (cosine distance) rather than <-> (L2). Pinecone, Weaviate, and Qdrant all support cosine similarity natively and it should be your default for CLIP-family embeddings.

One architectural decision that often surprises DBAs: you can store text embeddings and image embeddings in the *same* column when they share the same model and dimensionality. There is no type system distinction between "image vector" and "text vector" at the database level. The distinction is carried by your application logic and metadata columns. This is similar to storing both customer and product records in a single entities table — possible, but you need a type discriminator column to avoid joining apples with oranges in business queries.

Storage Architecture for Mixed-Modal Data

Before writing any queries, you need a schema that accommodates multiple modalities without creating a maintenance nightmare. There are three patterns worth knowing, each with different tradeoffs.

Pattern 1: Single Table, Shared Vector Column

All embeddings live in one table. A modality column distinguishes text from image. This works well when text and image documents are genuinely interchangeable from a business perspective — a product catalog where each item has both a description text and one or more product images.

-- PostgreSQL / pgvector
-- Single-table multi-modal schema for a product catalog

CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE product_embeddings (
    id              BIGSERIAL PRIMARY KEY,
    product_id      BIGINT NOT NULL REFERENCES products(id),
    modality        TEXT NOT NULL CHECK (modality IN ('text', 'image')),
    source_ref      TEXT NOT NULL,   -- field name or image URL/path
    embedding       VECTOR(512),     -- CLIP ViT-B/32 dimensions
    created_at      TIMESTAMPTZ DEFAULT NOW(),
    metadata        JSONB
);

-- Separate HNSW indexes per modality for query efficiency
CREATE INDEX idx_product_emb_text
    ON product_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WHERE modality = 'text';

CREATE INDEX idx_product_emb_image
    ON product_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WHERE modality = 'image';

-- Composite index for filtered product lookups
CREATE INDEX idx_product_emb_product_modality
    ON product_embeddings (product_id, modality);

The partial HNSW indexes on modality are a pgvector-specific optimization: when a query filters by modality before applying the nearest-neighbor scan, a partial index prevents the planner from scanning irrelevant rows. Without them, a text query would still traverse image vectors in the HNSW graph before filtering — wasting both I/O and compute.

Pattern 2: Separate Tables per Modality

More explicit and easier to maintain independently. Use this when text and image pipelines have very different update rates or when the embedding models differ in dimensionality.

-- PostgreSQL / pgvector
-- Separate tables for text and image embeddings

CREATE TABLE document_text_embeddings (
    id          BIGSERIAL PRIMARY KEY,
    doc_id      BIGINT NOT NULL,
    chunk_index INT NOT NULL DEFAULT 0,
    content     TEXT NOT NULL,
    embedding   VECTOR(1536),   -- OpenAI text-embedding-3-small
    metadata    JSONB,
    UNIQUE (doc_id, chunk_index)
);

CREATE TABLE document_image_embeddings (
    id          BIGSERIAL PRIMARY KEY,
    doc_id      BIGINT NOT NULL,
    image_index INT NOT NULL DEFAULT 0,
    image_path  TEXT NOT NULL,
    embedding   VECTOR(512),    -- CLIP ViT-B/32
    metadata    JSONB,
    UNIQUE (doc_id, image_index)
);

CREATE INDEX idx_text_emb_hnsw
    ON document_text_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

CREATE INDEX idx_image_emb_hnsw
    ON document_image_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

The tradeoff: two tables means two search queries per multi-modal lookup, and you need an application-layer fusion step. The benefit is that you can update the image embedding model without touching the text pipeline, and you can set different HNSW parameters (m, ef_construction) appropriate to each vector space's geometry.

Pattern 3: Weaviate Multi-Modal Collections

Weaviate has native multi-modal support via its multi2vec-clip vectorizer module, which handles both text and image fields in a single collection and produces a single fused vector. This shifts the fusion problem from your application layer to the database layer.

# Python
import weaviate
from weaviate.classes.config import Configure, Property, DataType
import base64
from pathlib import Path

client = weaviate.connect_to_local()  # or weaviate.connect_to_weaviate_cloud()

# Create a multi-modal collection
client.collections.create(
    name="ProductCatalog",
    vectorizer_config=Configure.Vectorizer.multi2vec_clip(
        image_fields=["product_image"],
        text_fields=["name", "description"],
        # Weight the contribution of each field to the combined vector
        weights={
            "product_image": [0.6],   # 60% image signal
            "name": [0.2],
            "description": [0.2],
        }
    ),
    properties=[
        Property(name="name", data_type=DataType.TEXT),
        Property(name="description", data_type=DataType.TEXT),
        Property(name="product_image", data_type=DataType.BLOB),  # base64 image
        Property(name="sku", data_type=DataType.TEXT),
        Property(name="price", data_type=DataType.NUMBER),
        Property(name="category", data_type=DataType.TEXT),
    ]
)

# Ingest a product with both text and image
def encode_image(image_path: str) -> str:
    with open(image_path, "rb") as f:
        return base64.b64encode(f.read()).decode("utf-8")

collection = client.collections.get("ProductCatalog")

collection.data.insert({
    "name": "Running Shoe X500",
    "description": "Lightweight mesh upper with cushioned midsole, ideal for road running",
    "product_image": encode_image("./images/shoe_x500.jpg"),
    "sku": "SHOE-X500-BLK-10",
    "price": 129.99,
    "category": "footwear"
})

client.close()

The Weaviate multi2vec-clip module uses field weights to combine sub-embeddings into one vector before indexing. This is architecturally different from the two-query-and-fuse pattern — you get a single HNSW index, lower query latency, and simpler application code. The cost is that you cannot independently update image weights without re-embedding the entire collection.

Ingestion Pipelines: Embedding at Scale

Getting data into a multi-modal store efficiently requires a pipeline that handles image preprocessing, batching, and error recovery. Raw images rarely arrive in a form suitable for CLIP: they need resizing, normalization, and format validation before the model sees them.

# Python
# Production-grade multi-modal ingestion pipeline
# Using sentence-transformers (CLIP wrapper), Qdrant, and async batching

import asyncio
import logging
from dataclasses import dataclass
from pathlib import Path
from typing import Optional
import numpy as np
from PIL import Image
from sentence_transformers import SentenceTransformer
from qdrant_client import QdrantClient, AsyncQdrantClient
from qdrant_client.models import (
    Distance, VectorParams, PointStruct,
    NamedVector, VectorsConfig
)

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

# CLIP ViT-B/32 via sentence-transformers wrapper
# Note: This model produces BOTH text and image vectors in the same 512-d space
MODEL_NAME = "clip-ViT-B-32"
VECTOR_DIM = 512
BATCH_SIZE = 64
MAX_IMAGE_SIZE = (336, 336)  # CLIP ViT-B/32 optimal input resolution

@dataclass
class ProductRecord:
    product_id: str
    name: str
    description: str
    image_path: Optional[str]
    category: str
    price: float

def load_and_preprocess_image(image_path: str) -> Optional[Image.Image]:
    """Load image with validation and normalization."""
    try:
        img = Image.open(image_path).convert("RGB")
        # Preserve aspect ratio while bounding to MAX_IMAGE_SIZE
        img.thumbnail(MAX_IMAGE_SIZE, Image.LANCZOS)
        return img
    except Exception as e:
        logger.warning(f"Failed to load image {image_path}: {e}")
        return None

def embed_batch(
    model: SentenceTransformer,
    records: list[ProductRecord]
) -> tuple[list[np.ndarray], list[np.ndarray]]:
    """
    Embed a batch of records.
    Returns (text_embeddings, image_embeddings).
    image_embeddings[i] is None if the image failed to load.
    """
    # Text: concatenate name + description for richer signal
    texts = [f"{r.name}. {r.description}" for r in records]
    text_embeddings = model.encode(
        texts,
        batch_size=BATCH_SIZE,
        normalize_embeddings=True,
        show_progress_bar=False
    )

    images = []
    image_mask = []  # Track which records have valid images
    for r in records:
        if r.image_path:
            img = load_and_preprocess_image(r.image_path)
            if img:
                images.append(img)
                image_mask.append(True)
            else:
                image_mask.append(False)
        else:
            image_mask.append(False)

    image_embeddings_raw = []
    if images:
        image_embeddings_raw = model.encode(
            images,
            batch_size=BATCH_SIZE,
            normalize_embeddings=True,
            show_progress_bar=False
        )

    # Reconstruct image_embeddings list aligned with records
    img_iter = iter(image_embeddings_raw)
    image_embeddings = []
    for has_image in image_mask:
        if has_image:
            image_embeddings.append(next(img_iter))
        else:
            image_embeddings.append(None)

    return text_embeddings, image_embeddings

def build_qdrant_points(
    records: list[ProductRecord],
    text_embeddings: list[np.ndarray],
    image_embeddings: list[Optional[np.ndarray]]
) -> list[PointStruct]:
    """
    Build Qdrant points using named vectors.
    Qdrant supports multiple named vector spaces per collection.
    """
    points = []
    for i, record in enumerate(records):
        vectors = {
            "text": text_embeddings[i].tolist()
        }
        if image_embeddings[i] is not None:
            vectors["image"] = image_embeddings[i].tolist()

        points.append(PointStruct(
            id=abs(hash(record.product_id)) % (2**63),  # Qdrant needs uint64
            vector=vectors,
            payload={
                "product_id": record.product_id,
                "name": record.name,
                "description": record.description,
                "category": record.category,
                "price": record.price,
                "has_image": image_embeddings[i] is not None
            }
        ))
    return points

async def setup_collection(client: AsyncQdrantClient, collection_name: str):
    """Create collection with named vector spaces for text and image."""
    collections = await client.get_collections()
    existing = [c.name for c in collections.collections]

    if collection_name not in existing:
        await client.create_collection(
            collection_name=collection_name,
            vectors_config={
                "text": VectorParams(size=VECTOR_DIM, distance=Distance.COSINE),
                "image": VectorParams(size=VECTOR_DIM, distance=Distance.COSINE),
            }
        )
        logger.info(f"Created collection: {collection_name}")
    else:
        logger.info(f"Collection {collection_name} already exists, skipping creation")

async def ingest_products(
    records: list[ProductRecord],
    collection_name: str = "products_multimodal",
    qdrant_url: str = "http://localhost:6333"
):
    model = SentenceTransformer(MODEL_NAME)
    client = AsyncQdrantClient(url=qdrant_url)

    await setup_collection(client, collection_name)

    total_ingested = 0
    for batch_start in range(0, len(records), BATCH_SIZE):
        batch = records[batch_start:batch_start + BATCH_SIZE]

        text_embs, image_embs = embed_batch(model, batch)
        points = build_qdrant_points(batch, text_embs, image_embs)

        await client.upsert(
            collection_name=collection_name,
            points=points,
            wait=True
        )
        total_ingested += len(points)
        logger.info(
            f"Ingested batch {batch_start}–{batch_start + len(batch)}, "
            f"total: {total_ingested}"
        )

    await client.close()
    logger.info(f"Ingestion complete. Total records: {total_ingested}")

A few production details worth noting in this pipeline:

Qdrant named vectors allow multiple vector spaces per document in a single collection. This is superior to storing separate collections per modality because a single upsert keeps both vectors atomic — you cannot have a text vector without its corresponding image vector falling out of sync.

Image failure handling: Real product images fail to load regularly — corrupted files, unsupported formats, network-mounted paths that go stale. The pipeline tracks has_image in the payload so queries can filter to image-capable records when needed.

Normalization at embed time: CLIP vectors should be L2-normalized before storage. normalize_embeddings=True in sentence-transformers handles this. Pre-normalizing means the database computes cosine similarity as a dot product — numerically cheaper and avoids re-normalization overhead on the query path.
