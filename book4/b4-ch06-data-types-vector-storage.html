Chapter 6: Data Types in Vector Storage: Text, Images, Audio, Code

Vector databases are, at their core, indifferent to the original form of your data. Whether you hand them a paragraph from a legal document, a spectrogram of a customer service call, a JPEG thumbnail, or a Python function, the storage layer sees only a sequence of floating-point numbers. But the path from raw data to those numbers — the embedding pipeline — varies dramatically by modality, and the choices you make there determine search quality, index behavior, and infrastructure cost more than almost anything else. This chapter walks through the four modalities that drive the majority of production vector workloads: text, images, audio, and code. For each, you will see how embeddings are generated, what dimensionality and distance metrics apply, and how to wire it into real storage systems with runnable examples.

---

6.1 Text Embeddings: The Dominant Workload

Text is where most teams start, and for good reason. The tooling is mature, the models are well-benchmarked, and the use cases — semantic search, RAG pipelines, document classification, duplicate detection — map directly onto problems DBAs already understand. If you have maintained a full-text search index in PostgreSQL or Elasticsearch, you already understand the tension text embeddings are meant to resolve: keyword search finds documents that share vocabulary, but not necessarily meaning. A query for "myocardial infarction" will miss documents that only use "heart attack." Embedding-based search collapses that gap.

How text embeddings work

A text embedding model takes a string and outputs a dense vector. The vector encodes semantic meaning in a high-dimensional space: documents about similar topics land near each other under cosine similarity or dot product distance. The dimensionality ranges widely — 384 dimensions for fast models like all-MiniLM-L6-v2, 1536 for OpenAI's text-embedding-3-small, 3072 for text-embedding-3-large. Higher dimension is not always better; it increases storage cost and slows index construction, and for many retrieval tasks a 768-dimensional model is indistinguishable in quality from a 3072-dimensional one.

Chunking is a database partitioning problem

Before you embed a document, you must decide how to cut it up. Embedding models have token limits: text-embedding-3-small supports 8,191 tokens, sentence-transformers models typically support 512 tokens. Feeding a 200-page PDF as a single string is not an option. Chunking strategies include fixed-size windows, sentence boundaries, paragraph boundaries, and semantic chunking. The right choice depends on your retrieval pattern — the same tradeoff you make when deciding between row-level and block-level access in a traditional DBMS.

A practical baseline: 512-token chunks with 50-token overlap. The overlap prevents semantic content from being split across chunk boundaries the way a page boundary in a B-tree splits a row.

# Python
# Text embedding pipeline using OpenAI + pgvector
# Requirements: openai>=1.0, psycopg2-binary, tiktoken

import openai
import psycopg2
import tiktoken
from typing import List, Tuple

client = openai.OpenAI()  # reads OPENAI_API_KEY from environment

def chunk_text(text: str, model: str = "text-embedding-3-small",
               chunk_tokens: int = 512, overlap_tokens: int = 50) -> List[str]:
    """Split text into overlapping token-bounded chunks."""
    enc = tiktoken.encoding_for_model(model)
    tokens = enc.encode(text)
    chunks = []
    start = 0
    while start < len(tokens):
        end = min(start + chunk_tokens, len(tokens))
        chunk_tokens_slice = tokens[start:end]
        chunks.append(enc.decode(chunk_tokens_slice))
        if end == len(tokens):
            break
        start += chunk_tokens - overlap_tokens
    return chunks

def embed_texts(texts: List[str], model: str = "text-embedding-3-small") -> List[List[float]]:
    """Batch embed up to 2048 strings per API call."""
    response = client.embeddings.create(input=texts, model=model)
    return [item.embedding for item in response.data]

def ingest_document(conn, doc_id: str, title: str, body: str) -> int:
    """Chunk, embed, and store a document. Returns number of chunks inserted."""
    chunks = chunk_text(body)
    embeddings = embed_texts(chunks)

    with conn.cursor() as cur:
        cur.executemany(
            """
            INSERT INTO document_chunks (doc_id, title, chunk_index, chunk_text, embedding)
            VALUES (%s, %s, %s, %s, %s::vector)
            """,
            [
                (doc_id, title, i, chunk, str(emb))
                for i, (chunk, emb) in enumerate(zip(chunks, embeddings))
            ]
        )
    conn.commit()
    return len(chunks)

# ── Schema (run once) ─────────────────────────────────────────────────────────
SCHEMA_SQL = """
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE IF NOT EXISTS document_chunks (
    id          BIGSERIAL PRIMARY KEY,
    doc_id      TEXT        NOT NULL,
    title       TEXT,
    chunk_index INTEGER     NOT NULL,
    chunk_text  TEXT        NOT NULL,
    embedding   vector(1536) NOT NULL,
    created_at  TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX IF NOT EXISTS document_chunks_embedding_idx
    ON document_chunks
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);
"""

# ── Query ─────────────────────────────────────────────────────────────────────
def semantic_search(conn, query: str, top_k: int = 5) -> List[Tuple]:
    query_embedding = embed_texts([query])[0]
    with conn.cursor() as cur:
        cur.execute(
            """
            SELECT doc_id, title, chunk_index, chunk_text,
                   1 - (embedding <=> %s::vector) AS cosine_similarity
            FROM   document_chunks
            ORDER  BY embedding <=> %s::vector
            LIMIT  %s
            """,
            (str(query_embedding), str(query_embedding), top_k)
        )
        return cur.fetchall()

The <=> operator is pgvector's cosine distance. One minus that value gives you cosine similarity, which is more intuitive when you need to filter by a quality threshold (e.g., WHERE 1 - (embedding <=> query) > 0.75).

Choosing a text embedding model

| Model | Dimensions | Max Tokens | Cost / 1M tokens | Notes |

|---|---|---|---|---|

| text-embedding-3-small | 1536 | 8,191 | $0.020 | Best price/quality for RAG |

| text-embedding-3-large | 3072 | 8,191 | $0.130 | Marginal gains, 6.5× cost |

| all-MiniLM-L6-v2 | 384 | 512 | Free (local) | Fast, low memory, good for high-volume |

| BAAI/bge-large-en-v1.5 | 1024 | 512 | Free (local) | MTEB top-10, strong retrieval |

| embed-english-v3.0 (Cohere) | 1024 | 512 | $0.100 | Strong multilingual variants |

For offline or high-volume workloads, BAAI/bge-large-en-v1.5 via sentence-transformers is the practical default. For OpenAI-native stacks, text-embedding-3-small covers most production cases without the cost penalty of the large variant.

6.2 Image Embeddings: Visual Similarity at Database Scale

Image search appears exotic, but the database framing is familiar: you are building an index over a feature space, and the features happen to come from a convolutional or transformer-based vision model rather than a B-tree key. The use cases span e-commerce (find visually similar products), content moderation (detect near-duplicate or policy-violating images), medical imaging (retrieve histology slides similar to a reference), and manufacturing (surface defect matching).

How image embeddings are generated

Vision models map an image to a dense vector by passing it through a neural network and extracting the activations at a specific layer — typically the penultimate layer before the classification head. CLIP (Contrastive Language-Image Pretraining) models are especially useful because they embed images and text into the same vector space, which allows cross-modal queries: you can retrieve images using a text string as the query.

Common models and their output dimensions:

  • → **OpenAI CLIP (ViT-B/32):** 512 dimensions
  • → **OpenAI CLIP (ViT-L/14):** 768 dimensions
  • → **Google's ViT-G via Vertex AI:** 1408 dimensions
  • → **ResNet-50 (ImageNet features):** 2048 dimensions (older, rarely used for search)

    • CLIP at ViT-L/14 is the standard starting point for production image search. It handles both similarity search within images and text-to-image retrieval, which reduces the number of specialized indexes you need to maintain.

    # Python
# Image embedding pipeline using CLIP + Weaviate
# Requirements: weaviate-client>=4.0, open-clip-torch, Pillow, torch

import weaviate
import open_clip
import torch
from PIL import Image
from pathlib import Path
import base64, io, json

# ── Load CLIP model (CPU or GPU depending on hardware) ──────────────────────
model, _, preprocess = open_clip.create_model_and_transforms(
    "ViT-L-14", pretrained="openai"
)
model.eval()
tokenizer = open_clip.get_tokenizer("ViT-L-14")

def embed_image(image_path: str) -> list[float]:
    """Return a 768-dim CLIP vector for an image file."""
    img = preprocess(Image.open(image_path).convert("RGB")).unsqueeze(0)
    with torch.no_grad():
        features = model.encode_image(img)
        features /= features.norm(dim=-1, keepdim=True)  # L2 normalize
    return features.squeeze().tolist()

def embed_text_for_image_search(text: str) -> list[float]:
    """Embed a text query into the same CLIP space as images."""
    tokens = tokenizer([text])
    with torch.no_grad():
        features = model.encode_text(tokens)
        features /= features.norm(dim=-1, keepdim=True)
    return features.squeeze().tolist()

# ── Weaviate schema ──────────────────────────────────────────────────────────
client = weaviate.connect_to_local()  # or weaviate.connect_to_weaviate_cloud(...)

from weaviate.classes.config import Configure, Property, DataType, VectorDistances

if not client.collections.exists("ProductImage"):
    client.collections.create(
        name="ProductImage",
        vectorizer_config=Configure.Vectorizer.none(),  # we supply our own vectors
        vector_index_config=Configure.VectorIndex.hnsw(
            distance_metric=VectorDistances.COSINE,
            ef_construction=128,
            max_connections=32,
        ),
        properties=[
            Property(name="product_id",   data_type=DataType.TEXT),
            Property(name="file_path",    data_type=DataType.TEXT),
            Property(name="description",  data_type=DataType.TEXT),
        ],
    )

# ── Ingest images ─────────────────────────────────────────────────────────────
def ingest_product_images(image_dir: str):
    collection = client.collections.get("ProductImage")
    with collection.batch.dynamic() as batch:
        for path in Path(image_dir).glob("*.jpg"):
            product_id = path.stem
            vector = embed_image(str(path))
            batch.add_object(
                properties={
                    "product_id":  product_id,
                    "file_path":   str(path),
                    "description": f"Product image {product_id}",
                },
                vector=vector,
            )

# ── Text-to-image search ──────────────────────────────────────────────────────
def search_images_by_text(query: str, top_k: int = 10):
    query_vector = embed_text_for_image_search(query)
    collection = client.collections.get("ProductImage")
    results = collection.query.near_vector(
        near_vector=query_vector,
        limit=top_k,
        return_metadata=["distance"],
    )
    for obj in results.objects:
        print(f"{obj.properties['product_id']} | "
              f"distance={obj.metadata.distance:.4f} | "
              f"path={obj.properties['file_path']}")

# Example: search_images_by_text("red running shoes with white sole")

    Storage considerations for images

    A 768-dimensional float32 vector consumes 3,072 bytes — about 3 KB per image. At one million images, that is 3 GB of vector data before indexing overhead. The HNSW index typically adds 1.5–2× the raw vector size, putting you at roughly 6–9 GB for the index alone. That fits comfortably in the memory of a cloud instance sized for this workload. The images themselves should not go into the vector database; store them in object storage (S3, GCS) and keep only the path or a content-addressed hash in the metadata fields.

    Preprocessing matters

    Unlike text, images require explicit preprocessing before embedding: resize to the model's expected input resolution (224×224 for ViT-B/32, 336×336 for ViT-L/14@336), normalize pixel values, and handle color mode. Feeding a grayscale medical image directly to a model expecting three-channel RGB input without conversion will produce silently incorrect embeddings — the query will return results, they just will not be meaningful. Always validate your preprocessing pipeline with a small labeled set before bulk ingestion.

    6.3 Audio Embeddings: Time-Series Signals as Vectors

    Audio is the least common modality in general-purpose vector workloads, but it is heavily used in specific verticals: call center analytics, music recommendation, speaker verification, podcast search, and environmental sound classification. The embedding pipeline is more involved than text or images because audio must be converted from a raw waveform to a fixed-length vector in a way that captures the relevant signal while discarding noise.

    Representations and models

    There are two main approaches to audio embedding:

    1. Spectrogram-based: Convert the waveform to a mel-spectrogram (a 2D time-frequency representation), then pass it through a CNN or transformer. Models: OpenL3, PANNs (Pretrained Audio Neural Networks), CLAP (Contrastive Language-Audio Pretraining).

    2. Waveform-based: Feed raw audio samples directly to a transformer. Models: wav2vec 2.0, Whisper encoder features, EnCodec.

    For semantic audio search — finding recordings similar in content or speaker — CLAP is the current standard because it supports cross-modal text-to-audio queries, the same pattern that makes CLIP useful for images.

    For speaker identification, models like speechbrain/spkrec-ecapa-voxceleb produce 192-dimensional speaker embeddings optimized for cosine similarity between the same speaker across recordings.

    # Python
# Audio embedding pipeline using CLAP + Qdrant
# Requirements: qdrant-client, msclap, librosa, numpy

import librosa
import numpy as np
from qdrant_client import QdrantClient
from qdrant_client.models import (
    Distance, VectorParams, PointStruct, SearchRequest
)
from msclap import CLAP  # Microsoft CLAP implementation

# ── Initialize CLAP and Qdrant ────────────────────────────────────────────────
clap_model = CLAP(version="2023", use_cuda=False)
qdrant = QdrantClient(url="http://localhost:6333")

COLLECTION_NAME = "audio_clips"
VECTOR_DIM = 512  # CLAP 2023 audio embedding dimension

# ── Create collection ─────────────────────────────────────────────────────────
qdrant.recreate_collection(
    collection_name=COLLECTION_NAME,
    vectors_config=VectorParams(size=VECTOR_DIM, distance=Distance.COSINE),
)

def embed_audio_file(file_path: str) -> np.ndarray:
    """
    Load an audio file and return a CLAP embedding.
    CLAP internally resamples to 44.1 kHz and expects mono or stereo.
    """
    embeddings = clap_model.get_audio_embeddings(
        [file_path], resample=True
    )
    vec = embeddings[0]
    # L2 normalize for cosine similarity
    return (vec / np.linalg.norm(vec)).tolist()

def embed_audio_text_query(text: str) -> list[float]:
    """Embed a text query into the CLAP audio space."""
    embeddings = clap_model.get_text_embeddings([text])
    vec = embeddings[0]
    return (vec / np.linalg.norm(vec)).tolist()

def ingest_audio_clips(clips: list[dict]):
    """
    clips: list of {"id": int, "file_path": str, "label": str, "duration_s": float}
    """
    points = []
    for clip in clips:
        vector = embed_audio_file(clip["file_path"])
        points.append(
            PointStruct(
                id=clip["id"],
                vector=vector,
                payload={
                    "file_path":  clip["file_path"],
                    "label":      clip["label"],
                    "duration_s": clip["duration_s"],
                },
            )
        )
    qdrant.upsert(collection_name=COLLECTION_NAME, points=points)

def search_audio_by_text(query: str, top_k: int = 5):
    """Find audio clips matching a natural language description."""
    query_vector = embed_audio_text_query(query)
    results = qdrant.search(
        collection_name=COLLECTION_NAME,
        query_vector=query_vector,
        limit=top_k,
        with_payload=True,
    )
    for hit in results:
        print(f"Score: {hit.score:.4f} | "
              f"Label: {hit.payload['label']} | "
              f"File: {hit.payload['file_path']}")

# Example: search_audio_by_text("dogs barking outdoors")
# Example: search_audio_by_text("angry customer complaining about billing")

    Segmentation and temporal alignment

    Long audio recordings — interviews, conference calls, podcasts — require segmentation before embedding, similar to text chunking. A common strategy is to use voice activity detection (VAD) to find speech segments, then embed each segment of 5–30 seconds independently. This gives finer retrieval granularity: instead of finding "the episode that discusses refund policies," you can surface the 20-second clip where refund policies are explicitly mentioned.

    The metadata schema for audio should carry temporal anchors: start_time_s and end_time_s for each embedded segment. These allow the retrieval result to play back or link to the exact moment in a recording, which is the audio equivalent of returning a document chunk rather than a full document.

    Sampling rate and codec normalization

    Audio from different sources arrives at different sample rates (8 kHz telephony, 44.1 kHz music, 16 kHz podcast) and codecs (MP3, AAC, FLAC, OGG). Most embedding models expect a specific input: wav2vec 2.0 requires 16 kHz mono PCM. Feeding 8 kHz telephone audio without upsampling, or stereo audio without downmixing, degrades embedding quality. Build a normalization step (librosa's resample, ffmpeg via subprocess) at the front of your ingestion pipeline and enforce it — treat it the same way you would enforce data type coercion in an ETL pipeline.

    6.4 Code Embeddings: Semantic Search Across Source Files

    Code is a modality that most vector database introductions skip, but it is one of the most commercially valuable. Developers search for how a function was used across a codebase, how a specific API was called, or what code handles a particular edge case. Traditional regex-based code search (grep, ripgrep) is syntactically exact; embedding-based search finds code that does the same thing expressed differently.

    How code embeddings differ from text embeddings

    Code has structure that plain text does not: function signatures, indentation scope, docstrings, variable names, and control flow. Models trained specifically on code — microsoft/codebert-base, Salesforce/codet5-base, OpenAI's text-embedding-3-small with code content, and voyage-code-2 from Voyage AI — outperform generic text models on code retrieval tasks because they understand that def add(a, b): return a + b and func sum(x int, y int) int { return x + y } are semantically equivalent even across languages.

    voyage-code-2 from Voyage AI currently leads benchmarks for code retrieval, producing 1536-dimensional embeddings. For teams already on the OpenAI stack, text-embedding-3-small performs well enough on code and avoids introducing a second embedding provider.

    Chunking strategies for code

    File-level embedding loses too much precision; function-level or class-level chunking is the practical standard. Parse the AST of each source file and extract functions, methods, and classes as individual embedding units. Include the docstring, signature, and body as the text passed to the embedding model. For Python, the ast module handles this natively; for multi-language codebases, tree-sitter provides a consistent AST interface across languages.
