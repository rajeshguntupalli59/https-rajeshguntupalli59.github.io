Chapter 4: Google Cloud SQL and AlloyDB Pricing Deep Dive

Organizations running production workloads on Google Cloud SQL and AlloyDB routinely overspend by 35–55% compared to their optimized baseline. A mid-sized engineering team running three Cloud SQL for PostgreSQL instances—a db-custom-8-32768 primary, a read replica, and a dev instance left running around the clock—can easily accumulate $4,200/month in avoidable charges before anyone notices. This chapter breaks down exactly how Google prices Cloud SQL and AlloyDB, where the hidden costs live, and how to systematically eliminate waste. By the end, a disciplined implementation of the techniques here can recover $1,500–$8,000/month for a typical team running five to fifteen database instances.

---

4.1 How Google Cloud SQL and AlloyDB Pricing Actually Works

Understanding the billing model is the first step to controlling it. Cloud SQL and AlloyDB look similar on the surface but have meaningfully different pricing structures that affect optimization strategy.

Cloud SQL Pricing Components

Cloud SQL bills across five independent dimensions:

1. Instance (vCPU + RAM) — billed per second, based on machine type

2. Storage — billed per GB/month, separate from compute

3. Networking — egress to other regions, internet, or cross-zone replicas

4. Backups — beyond the free tier (equal to instance storage size)

5. IP address — $0.01/hour for a public IPv4 address when the instance is running

For a db-custom-8-32768 instance (8 vCPU, 32 GB RAM) running PostgreSQL 15 in us-central1, the on-demand pricing breaks down as:

| Component | Unit Price | Example Usage | Monthly Cost |

|---|---|---|---|

| vCPU | $0.0413/vCPU/hour | 8 vCPU × 730h | $241.59 |

| RAM | $0.0070/GB/hour | 32 GB × 730h | $163.84 |

| SSD Storage | $0.170/GB/month | 500 GB | $85.00 |

| Backup storage | $0.08/GB/month | 200 GB over free | $16.00 |

| IPv4 address | $0.01/hour | 730h | $7.30 |

| Total | | | $513.73 |

Add a same-region read replica with identical specs and you're at $1,027.46/month before any data egress. That's a useful baseline to anchor your optimization work.

Cloud SQL Committed Use Discounts (CUDs)

Google offers 1-year and 3-year committed use discounts for Cloud SQL compute (not storage):

  • → **1-year CUD**: 25% discount on vCPU and RAM
  • → **3-year CUD**: 52% discount on vCPU and RAM

    • For the 8 vCPU, 32 GB instance above, a 1-year CUD reduces the compute portion from $405.43 to $304.07—a savings of $101.36/month or $1,216.32/year. A 3-year CUD drops it to $194.61/month, saving $210.82/month or $2,529.84/year.

    Storage, networking, backups, and IP addresses are excluded from CUDs and billed at on-demand rates regardless.

    AlloyDB Pricing Components

    AlloyDB uses a fundamentally different model that trips up teams migrating from Cloud SQL:

    1. Cluster compute — billed per vCPU/hour for the primary and each read pool instance

    2. Storage — a unified, automatically managed storage layer billed per GB/month across the cluster

    3. I/O requests — billed separately at $0.10 per million read/write operations (this is the most common surprise cost)

    4. Backups — $0.023/GB/month for automated backups beyond the database size

    The I/O charge is what catches teams off guard. A busy OLTP database performing 500 million I/O operations per day generates $1,500/month in I/O costs alone. AlloyDB's storage is intelligent—it caches aggressively and reduces physical I/O compared to Cloud SQL—but the billing unit is logical I/O at the application layer, not physical disk operations.

    AlloyDB compute pricing in us-central1:

  • → vCPU: $0.0722/vCPU/hour
  • → RAM: $0.00963/GB/hour

    • A 16-vCPU, 128 GB AlloyDB primary instance costs approximately $845/month in compute before storage or I/O. AlloyDB CUDs provide 25% (1-year) and 52% (3-year) discounts on compute only.

    The Right Tool for the Right Workload

    Before optimizing, confirm you're on the right product:

  • → **Cloud SQL** is cost-effective for standard OLTP workloads under ~5,000 QPS with moderate I/O
  • → **AlloyDB** pays for itself on analytical and mixed OLAP/OLTP workloads where its columnar engine eliminates full table scans, or for workloads requiring 4× faster reads than Cloud SQL
  • → Running AlloyDB for a simple 200 QPS web application backend will cost 2–3× more than Cloud SQL with no performance benefit

    4.2 Diagnosing Where Your Money Is Going

    Before making changes, establish a clear picture of current spend by instance, component, and time pattern. The GCP Cost Table in BigQuery is the most granular source of billing data available.

    Setting Up BigQuery Billing Export

    If you haven't already enabled billing export to BigQuery, do it now—you need this data for everything that follows.

    # Enable billing export to BigQuery
# First, create a dataset in BigQuery to receive billing data
bcloud bigquery datasets create billing_export \
  --location=US \
  --description="GCP billing export for FinOps analysis"

# Enable billing export via gcloud (requires billing account admin role)
gcloud billing accounts list

# Note your BILLING_ACCOUNT_ID from the output, then enable export
# This must be done via the console UI or Terraform — gcloud does not 
# support direct export configuration via CLI as of 2024
# Navigate to: Billing > Billing export > BigQuery export > Edit settings

# Verify the export is active by querying the dataset after 24 hours
bq query --use_legacy_sql=false \
  'SELECT COUNT(*) as row_count, MIN(usage_start_time) as earliest_data
   FROM `YOUR_PROJECT.billing_export.gcp_billing_export_v1_XXXXXX_XXXXXX_XXXXXX`
   LIMIT 1'

    Once billing data is flowing into BigQuery, the following queries give you a complete cost map.

    Query 1: Cloud SQL Cost Breakdown by Instance and Component

    -- Run this in BigQuery against your billing export table
-- Replace the table name with your actual export table
WITH cloud_sql_costs AS (
  SELECT
    -- Extract instance name from resource labels
    resource.labels['database_id'] AS instance_id,
    resource.labels['region']      AS region,
    sku.description                AS cost_component,
    
    -- Categorize cost types for easier filtering
    CASE
      WHEN sku.description LIKE '%vCPU%'    THEN 'Compute-vCPU'
      WHEN sku.description LIKE '%RAM%'     THEN 'Compute-RAM'
      WHEN sku.description LIKE '%Storage%' THEN 'Storage'
      WHEN sku.description LIKE '%Backup%'  THEN 'Backup'
      WHEN sku.description LIKE '%Network%' THEN 'Network'
      WHEN sku.description LIKE '%IP%'      THEN 'IP-Address'
      ELSE 'Other'
    END AS cost_category,
    
    SUM(cost)                            AS total_cost_usd,
    SUM(usage.amount)                    AS total_usage,
    usage.unit                           AS usage_unit,
    
    -- Month for trending
    FORMAT_DATE('%Y-%m', DATE(usage_start_time)) AS billing_month
    
  FROM `YOUR_PROJECT.billing_export.gcp_billing_export_v1_XXXXXX`
  WHERE
    service.description = 'Cloud SQL'
    AND DATE(usage_start_time) >= DATE_SUB(CURRENT_DATE(), INTERVAL 90 DAY)
    AND cost > 0
  GROUP BY 1, 2, 3, 4, 9, 10
),

-- Pivot to show monthly trend per instance
instance_summary AS (
  SELECT
    instance_id,
    region,
    cost_category,
    billing_month,
    ROUND(total_cost_usd, 2) AS monthly_cost_usd
  FROM cloud_sql_costs
)

SELECT
  instance_id,
  region,
  cost_category,
  SUM(CASE WHEN billing_month = FORMAT_DATE('%Y-%m', DATE_SUB(CURRENT_DATE(), INTERVAL 1 MONTH))
           THEN monthly_cost_usd ELSE 0 END) AS last_month_usd,
  SUM(CASE WHEN billing_month = FORMAT_DATE('%Y-%m', DATE_SUB(CURRENT_DATE(), INTERVAL 2 MONTH))
           THEN monthly_cost_usd ELSE 0 END) AS two_months_ago_usd,
  SUM(CASE WHEN billing_month = FORMAT_DATE('%Y-%m', DATE_SUB(CURRENT_DATE(), INTERVAL 3 MONTH))
           THEN monthly_cost_usd ELSE 0 END) AS three_months_ago_usd
FROM instance_summary
GROUP BY 1, 2, 3
ORDER BY last_month_usd DESC;

    Query 2: AlloyDB I/O Cost Analysis

    AlloyDB I/O costs require separate tracking because they scale with query volume, not instance size. This query identifies your I/O spend trend and flags instances where I/O exceeds 30% of total cluster cost—a threshold that indicates optimization opportunity.

    -- BigQuery query to analyze AlloyDB I/O costs
WITH alloydb_costs AS (
  SELECT
    resource.labels['cluster_id']    AS cluster_id,
    resource.labels['instance_id']   AS instance_id,
    resource.labels['region']        AS region,
    sku.description                  AS sku_description,
    
    CASE
      WHEN sku.description LIKE '%Read IO%'   THEN 'IO-Read'
      WHEN sku.description LIKE '%Write IO%'  THEN 'IO-Write'
      WHEN sku.description LIKE '%vCPU%'      THEN 'Compute-vCPU'
      WHEN sku.description LIKE '%Memory%'    THEN 'Compute-RAM'
      WHEN sku.description LIKE '%Storage%'   THEN 'Storage'
      WHEN sku.description LIKE '%Backup%'    THEN 'Backup'
      ELSE 'Other'
    END AS cost_category,
    
    FORMAT_DATE('%Y-%m', DATE(usage_start_time)) AS billing_month,
    SUM(cost)                                    AS total_cost_usd,
    SUM(usage.amount)                            AS total_usage_units
    
  FROM `YOUR_PROJECT.billing_export.gcp_billing_export_v1_XXXXXX`
  WHERE
    service.description = 'AlloyDB for PostgreSQL'
    AND DATE(usage_start_time) >= DATE_SUB(CURRENT_DATE(), INTERVAL 90 DAY)
  GROUP BY 1, 2, 3, 4, 5, 6
),

cluster_totals AS (
  SELECT
    cluster_id,
    billing_month,
    SUM(total_cost_usd) AS cluster_total_cost
  FROM alloydb_costs
  GROUP BY 1, 2
),

io_costs AS (
  SELECT
    cluster_id,
    billing_month,
    SUM(total_cost_usd) AS io_total_cost
  FROM alloydb_costs
  WHERE cost_category IN ('IO-Read', 'IO-Write')
  GROUP BY 1, 2
)

SELECT
  a.cluster_id,
  a.billing_month,
  ROUND(c.cluster_total_cost, 2)    AS cluster_total_usd,
  ROUND(i.io_total_cost, 2)         AS io_cost_usd,
  ROUND(
    SAFE_DIVIDE(i.io_total_cost, c.cluster_total_cost) * 100, 1
  )                                 AS io_pct_of_total,
  CASE
    WHEN SAFE_DIVIDE(i.io_total_cost, c.cluster_total_cost) > 0.30
    THEN 'REVIEW: High I/O ratio - check for missing indexes or full scans'
    ELSE 'OK'
  END                               AS recommendation
FROM alloydb_costs a
JOIN cluster_totals c USING (cluster_id, billing_month)
JOIN io_costs i USING (cluster_id, billing_month)
GROUP BY 1, 2, 3, 4, 5, 6
ORDER BY billing_month DESC, cluster_total_usd DESC;

    Identifying Idle and Underutilized Instances

    Cloud Monitoring provides the CPU and connection utilization data needed to flag idle instances. An instance consuming less than 5% average CPU over 30 days with fewer than 5 concurrent connections is a strong candidate for deletion or significant downsizing.

    #!/bin/bash
# Script: find_idle_cloud_sql_instances.sh
# Scans all Cloud SQL instances in a project for low CPU utilization
# Schedule this with Cloud Scheduler weekly

PROJECT_ID="${1:-$(gcloud config get-value project)}"
LOOKBACK_DAYS=30
CPU_THRESHOLD=5  # Flag instances below 5% average CPU
CONN_THRESHOLD=5 # Flag instances with fewer than 5 avg connections

echo "Analyzing Cloud SQL instances in project: $PROJECT_ID"
echo "Lookback period: $LOOKBACK_DAYS days"
echo "---"

# Get all Cloud SQL instances
INSTANCES=$(gcloud sql instances list \
  --project="$PROJECT_ID" \
  --format="value(name,region,databaseVersion,settings.tier)" \
  --filter="state=RUNNABLE")

echo "Instance,Region,Engine,Tier,Avg_CPU_Pct,Avg_Connections,Status"

while IFS=$'\t' read -r INSTANCE REGION ENGINE TIER; do
  # Query Cloud Monitoring for average CPU over lookback period
  AVG_CPU=$(gcloud monitoring read \
    "metric.type=\"cloudsql.googleapis.com/database/cpu/utilization\" \
    resource.labels.database_id=\"$PROJECT_ID:$INSTANCE\"" \
    --freshness="$((LOOKBACK_DAYS * 24))h" \
    --format="value(points[0].value.doubleValue)" \
    2>/dev/null | awk '{sum+=$1; count++} END {printf "%.1f", (sum/count)*100}')

  # Query average connections
  AVG_CONN=$(gcloud monitoring read \
    "metric.type=\"cloudsql.googleapis.com/database/network/connections\" \
    resource.labels.database_id=\"$PROJECT_ID:$INSTANCE\"" \
    --freshness="$((LOOKBACK_DAYS * 24))h" \
    --format="value(points[0].value.int64Value)" \
    2>/dev/null | awk '{sum+=$1; count++} END {printf "%.0f", sum/count}')

  # Default to 0 if metric fetch failed
  AVG_CPU=${AVG_CPU:-0}
  AVG_CONN=${AVG_CONN:-0}

  # Classify the instance
  if (( $(echo "$AVG_CPU < $CPU_THRESHOLD" | bc -l) )) && \
     (( AVG_CONN < CONN_THRESHOLD )); then
    STATUS="IDLE - CANDIDATE FOR DELETION OR STOP"
  elif (( $(echo "$AVG_CPU < 20" | bc -l) )); then
    STATUS="UNDERUTILIZED - CONSIDER DOWNSIZING"
  else
    STATUS="ACTIVE"
  fi

  echo "$INSTANCE,$REGION,$ENGINE,$TIER,${AVG_CPU}%,$AVG_CONN,$STATUS"

done <<< "$INSTANCES"

    4.3 Right-Sizing Cloud SQL Instances

    Right-sizing is consistently the largest single savings lever available—larger than CUDs, larger than storage optimization, and larger than any architectural change. The reason is straightforward: most teams provision for peak capacity that never materializes, then leave those instances running at 8–15% CPU for months.

    The Cloud SQL Machine Type Landscape

    Cloud SQL offers three tiers:

  • → **Shared core** (`db-f1-micro`, `db-g1-small`): Dev/test only, not suitable for production
  • → **Standard machine types** (`db-n1-standard-*`, `db-n2-standard-*`): Fixed vCPU/RAM ratios
  • → **Custom machine types** (`db-custom-N-M`): Choose any vCPU count and RAM independently

    • Custom machine types are almost always cheaper than standard types for real workloads because you pay only for what you configure. A db-custom-4-15360 (4 vCPU, 15 GB RAM) costs less than db-n1-standard-4 (4 vCPU, 15 GB RAM) in most regions because the custom type uses N2 generation pricing which is more cost-efficient per unit.

    The Memory-to-vCPU Ratio Decision

    The most common right-sizing mistake is provisioning too much RAM relative to vCPU count, or vice versa. Use this framework:

    | Workload Pattern | Recommended Ratio | Example |

    |---|---|---|

    | OLTP, high concurrency | 4–8 GB RAM per vCPU | db-custom-8-32768 |

    | Analytics, large datasets | 8–16 GB RAM per vCPU | db-custom-4-61440 |

    | Connection-heavy, small queries | 2–4 GB RAM per vCPU | db-custom-16-32768 |

    | Batch processing | 4 GB RAM per vCPU | db-custom-4-16384 |

    Querying Actual Resource Utilization from pg_stat

    Before recommending a right-size target, pull actual utilization data from the database itself:

    -- Run directly on your Cloud SQL PostgreSQL instance
-- This query gives a complete utilization picture for right-sizing decisions

WITH 
-- Memory pressure assessment
memory_stats AS (
  SELECT
    name,
    setting,
    unit,
    CASE name
      WHEN 'shared_buffers'      THEN setting::bigint * 8192
      WHEN 'work_mem'            THEN setting::bigint * 1024
      WHEN 'maintenance_work_mem' THEN setting::bigint * 1024
      WHEN 'effective_cache_size' THEN setting::bigint * 8192
    END AS bytes
  FROM pg_settings
  WHERE name IN (
    'shared_buffers', 'work_mem', 
    'maintenance_work_mem', 'effective_cache_size',
    'max_connections'
  )
),

-- Buffer cache hit ratio (lower than 95% suggests memory pressure)
cache_stats AS (
  SELECT
    sum(heap_blks_read)  AS heap_disk_reads,
    sum(heap_blks_hit)   AS heap_cache_hits,
    ROUND(
      100.0 * sum(heap_blks_hit) / 
      NULLIF(sum(heap_blks_hit) + sum(heap_blks_read), 0), 2
    )                    AS cache_hit_ratio_pct
  FROM pg_statio_user_tables
),

-- Active connections vs. configured max
connection_stats AS (
  SELECT
    count(*)                           AS total_connections,
    count(*) FILTER (WHERE state = 'active')   AS active_connections,
    count(*) FILTER (WHERE state = 'idle')     AS idle_connections,
    count(*) FILTER (WHERE state = 'idle in transaction') AS idle_in_txn,
    max(now() - query_start) FILTER (WHERE state = 'active') AS longest_active_query,
    (SELECT setting::int FROM pg_settings WHERE name = 'max_connections') AS max_connections_cfg
  FROM pg_stat_activity
  WHERE pid <> pg_backend_pid()
),

-- Top CPU-consuming queries (requires pg_stat_statements)
top_queries AS (
  SELECT
    LEFT(query, 80)          AS query_preview,
    calls,
    ROUND(total_exec_time::numeric / 1000, 2) AS total_exec_sec,
    ROUND(mean_exec_time::numeric, 2)         AS avg_exec_ms,
    ROUND(stddev_exec_time::numeric, 2)       AS stddev_exec_ms,
    rows
  FROM pg_stat_statements
  ORDER BY total_exec_time DESC
  LIMIT 5
)

-- Combined output
SELECT 'CACHE HIT RATIO'            AS metric_type,
       cache_hit_ratio_pct::text    AS value,
       CASE WHEN cache_hit_ratio_pct < 95 
            THEN 'WARNING: Consider increasing shared_buffers or instance RAM'
            ELSE 'OK: Cache hit ratio is healthy'
       END                          AS recommendation
FROM cache_stats

UNION ALL

SELECT 'CONNECTION UTILIZATION',
       ROUND(100.0 * total_connections / max_connections_cfg, 1)::text || '%',
       CASE 
         WHEN 100.0 * total_connections / max_connections_cfg > 80
         THEN 'WARNING: Consider pgBouncer or increasing max_connections'
         WHEN idle_in_txn > 10
         THEN 'WARNING: ' || idle_in_txn || ' idle-in-transaction connections - check for connection leaks'
         ELSE 'OK'
       END
FROM connection_stats

UNION ALL

SELECT 'IDLE CONNECTIONS',
       idle_connections::text,
       CASE WHEN idle_connections > 50
            THEN 'COST ALERT: ' || idle_connections || 
                 ' idle connections consuming RAM. Deploy pgBouncer to reduce instance RAM needs.'
            ELSE 'OK'
       END
FROM connection_stats;

    Calculating Right-Size Savings

    Here is the actual math for a common right-sizing scenario. Suppose Cloud Monitoring shows your db-custom-8-32768 instance averages 18% CPU and 12 GB RAM in use over 30 days. The right-size target is db-custom-4-16384:

    | Metric | Current | Right-Sized | Savings |

    |---|---|---|---|

    | vCPU | 8 × $0.0413 × 730h | 4 × $0.0413 × 730h | $120.79/mo |

    | RAM | 32 GB × $0.007 × 730h | 16 GB × $0.007 × 730h | $81.92/mo |

    | Monthly savings | $513.73 | $310.01 | $203.72/mo |

    | Annual savings | | | $2,444.64 |

    Apply a 1-year CUD to the right-sized instance and the monthly cost drops further to $232.51, for a total monthly savings of $281.22 versus the original on-demand oversized instance.
