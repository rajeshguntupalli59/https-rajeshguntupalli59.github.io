Chapter 21: MCP and PostgreSQL: Direct AI Access to Your Database

The Model Context Protocol (MCP) changes how AI systems interact with databases. Instead of building custom tool integrations for every LLM application, MCP gives you a standardized interface that lets Claude, GPT-4, and any compliant AI model discover and invoke database capabilities directly — querying tables, running analytics, inspecting schemas, and executing transactions through a protocol the model understands natively. For PostgreSQL specifically, this means your database becomes a first-class participant in the AI reasoning loop, not just a backend storage system that humans translate for models. This chapter walks through building a production-grade MCP server backed by PostgreSQL, designing the schemas that make database exploration safe and powerful for AI agents, and the operational patterns that keep direct AI database access from becoming a liability.

---

What MCP Actually Is — And Why It Changes PostgreSQL Integration

Before MCP, connecting an LLM to PostgreSQL meant writing custom tool definitions for every capability you wanted to expose. You defined functions in your application layer, described them in whatever format your LLM framework expected, handled authentication, sanitized inputs, and translated results back into something the model could reason about. Every framework did this differently — LangChain tools, OpenAI function calling, Anthropic's tool use — each with its own schema, each requiring its own integration code.

MCP standardizes this at the protocol level. An MCP server exposes resources (things models can read), tools (functions models can call), and prompts (templates for common workflows). Any MCP-compatible client — Claude Desktop, a custom agent, an IDE plugin — can discover these capabilities automatically and invoke them without application-specific glue code.

For PostgreSQL, this is significant because your database already organizes information in ways that map cleanly to MCP concepts:

  • → **Tables and views** become resources — the model can list them, inspect their schemas, and read their contents
  • → **Functions and procedures** become tools — the model can call them with typed arguments
  • → **Query templates and common analytics** become prompts — reusable patterns the model can apply

    • The protocol runs over stdio or HTTP with a defined JSON-RPC message format. You build a server that speaks this protocol, connect it to PostgreSQL, and any MCP client gains structured, authenticated access to your database.

    Here's what the architecture looks like in practice:

    ┌─────────────────┐     MCP Protocol      ┌──────────────────┐
│  AI Client      │ ◄──────────────────► │  MCP Server      │
│  (Claude, etc.) │   (JSON-RPC/stdio)    │  (Python/Node)   │

    The MCP server is where you enforce policy: which tables are visible, which operations are allowed, what query complexity is acceptable, and how results get formatted for model consumption.

    Schema Design for MCP-Accessible PostgreSQL

    A database designed for direct AI access needs different affordances than one designed purely for application code. Schema legibility, access boundaries, and metadata richness matter more because the model navigating your database has no prior knowledge of your domain — it reads what's there.

    Start with a clean separation between AI-accessible and application-internal schemas:

    -- Core schema separation
CREATE SCHEMA IF NOT EXISTS ai_readable;      -- Views and functions the MCP layer exposes
CREATE SCHEMA IF NOT EXISTS ai_tools;         -- Stored procedures MCP can invoke
CREATE SCHEMA IF NOT EXISTS app_internal;     -- Application tables, never directly exposed
CREATE SCHEMA IF NOT EXISTS mcp_audit;        -- All AI-generated queries and their results

-- Role for AI access — no superuser, no DDL
CREATE ROLE mcp_agent_role NOINHERIT;
GRANT USAGE ON SCHEMA ai_readable TO mcp_agent_role;
GRANT USAGE ON SCHEMA ai_tools TO mcp_agent_role;
GRANT SELECT ON ALL TABLES IN SCHEMA ai_readable TO mcp_agent_role;
GRANT EXECUTE ON ALL FUNCTIONS IN SCHEMA ai_tools TO mcp_agent_role;

-- Audit table for every AI interaction
CREATE TABLE mcp_audit.query_log (
    id              BIGSERIAL PRIMARY KEY,
    session_id      UUID NOT NULL DEFAULT gen_random_uuid(),
    agent_id        TEXT NOT NULL,
    tool_name       TEXT NOT NULL,
    input_params    JSONB NOT NULL DEFAULT '{}',
    sql_executed    TEXT,
    rows_returned   INTEGER,
    execution_ms    INTEGER,
    error_message   TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX ON mcp_audit.query_log (agent_id, created_at DESC);
CREATE INDEX ON mcp_audit.query_log (session_id);
CREATE INDEX ON mcp_audit.query_log USING GIN (input_params);

-- Example: a product catalog in app_internal
CREATE TABLE app_internal.products (
    id              BIGSERIAL PRIMARY KEY,
    sku             TEXT NOT NULL UNIQUE,
    name            TEXT NOT NULL,
    description     TEXT,
    price_cents     INTEGER NOT NULL,
    category_id     INTEGER REFERENCES app_internal.categories(id),
    inventory_count INTEGER NOT NULL DEFAULT 0,
    is_active       BOOLEAN NOT NULL DEFAULT TRUE,
    metadata        JSONB NOT NULL DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Safe view for AI consumption — no PII, clear column names
CREATE VIEW ai_readable.products AS
SELECT
    id,
    sku,
    name,
    description,
    ROUND(price_cents / 100.0, 2)   AS price_usd,
    c.name                           AS category,
    inventory_count,
    is_active,
    -- Computed fields that help models reason
    CASE
        WHEN inventory_count = 0 THEN 'out_of_stock'
        WHEN inventory_count < 10 THEN 'low_stock'
        ELSE 'in_stock'
    END                              AS stock_status,
    metadata,
    created_at
FROM app_internal.products p
JOIN app_internal.categories c ON c.id = p.category_id;

COMMENT ON VIEW ai_readable.products IS
    'Product catalog. Use this to look up product details, pricing, and availability. '
    'price_usd is the retail price in US dollars. stock_status summarizes inventory.';

COMMENT ON COLUMN ai_readable.products.metadata IS
    'Unstructured product attributes as JSON. Common keys: brand, dimensions, weight_kg, tags.';

-- Comments are readable by MCP tools — they guide model behavior
COMMENT ON COLUMN ai_readable.products.sku IS
    'Stock Keeping Unit — unique product identifier used in orders and shipments.';

    Comments on tables and columns are not cosmetic here — your MCP server will expose them as part of schema introspection, and models use them to decide which fields to query and how to interpret results.

    Now define the tools schema — stored procedures that perform bounded, auditable operations:

    -- Parameterized search function — prevents freeform SQL injection
CREATE OR REPLACE FUNCTION ai_tools.search_products(
    search_query    TEXT,
    category_filter TEXT DEFAULT NULL,
    max_price_usd   NUMERIC DEFAULT NULL,
    in_stock_only   BOOLEAN DEFAULT FALSE,
    result_limit    INTEGER DEFAULT 20
)
RETURNS TABLE (
    id              BIGINT,
    sku             TEXT,
    name            TEXT,
    price_usd       NUMERIC,
    category        TEXT,
    stock_status    TEXT,
    relevance_score REAL
)
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = app_internal, ai_readable, public
AS $$
BEGIN
    -- Enforce reasonable limits — models can request large result sets
    result_limit := LEAST(result_limit, 100);

    RETURN QUERY
    SELECT
        p.id,
        p.sku,
        p.name,
        ROUND(p.price_cents / 100.0, 2)::NUMERIC AS price_usd,
        c.name                                    AS category,
        CASE
            WHEN p.inventory_count = 0 THEN 'out_of_stock'
            WHEN p.inventory_count < 10 THEN 'low_stock'
            ELSE 'in_stock'
        END                                       AS stock_status,
        ts_rank(
            to_tsvector('english', p.name || ' ' || COALESCE(p.description, '')),
            plainto_tsquery('english', search_query)
        )                                         AS relevance_score
    FROM app_internal.products p
    JOIN app_internal.categories c ON c.id = p.category_id
    WHERE
        p.is_active = TRUE
        AND to_tsvector('english', p.name || ' ' || COALESCE(p.description, ''))
            @@ plainto_tsquery('english', search_query)
        AND (category_filter IS NULL OR c.name ILIKE category_filter)
        AND (max_price_usd IS NULL OR p.price_cents <= (max_price_usd * 100)::INTEGER)
        AND (NOT in_stock_only OR p.inventory_count > 0)
    ORDER BY relevance_score DESC
    LIMIT result_limit;
END;
$$;

COMMENT ON FUNCTION ai_tools.search_products IS
    'Search products by keyword. Returns up to 100 results ordered by relevance. '
    'Use category_filter to narrow by product category, max_price_usd to cap results '
    'by price, in_stock_only=true to exclude unavailable items.';

-- Analytics tool — canned aggregations the model can call
CREATE OR REPLACE FUNCTION ai_tools.category_sales_summary(
    start_date  DATE DEFAULT CURRENT_DATE - INTERVAL '30 days',
    end_date    DATE DEFAULT CURRENT_DATE
)
RETURNS TABLE (
    category        TEXT,
    order_count     BIGINT,
    total_revenue   NUMERIC,
    avg_order_value NUMERIC,
    top_product     TEXT
)
LANGUAGE sql
SECURITY DEFINER
SET search_path = app_internal, public
AS $$
    WITH sales AS (
        SELECT
            c.name                          AS category,
            COUNT(DISTINCT o.id)            AS order_count,
            SUM(oi.quantity * oi.unit_price_cents) / 100.0 AS total_revenue,
            p.name                          AS product_name,
            ROW_NUMBER() OVER (
                PARTITION BY c.name
                ORDER BY SUM(oi.quantity * oi.unit_price_cents) DESC
            )                               AS product_rank
        FROM app_internal.orders o
        JOIN app_internal.order_items oi ON oi.order_id = o.id
        JOIN app_internal.products p    ON p.id = oi.product_id
        JOIN app_internal.categories c  ON c.id = p.category_id
        WHERE o.created_at::DATE BETWEEN start_date AND end_date
          AND o.status NOT IN ('cancelled', 'refunded')
        GROUP BY c.name, p.name
    )
    SELECT
        category,
        SUM(order_count)::BIGINT        AS order_count,
        SUM(total_revenue)              AS total_revenue,
        AVG(total_revenue / NULLIF(order_count, 0)) AS avg_order_value,
        MAX(CASE WHEN product_rank = 1 THEN product_name END) AS top_product
    FROM sales
    GROUP BY category
    ORDER BY total_revenue DESC;
$$;

    Row-level security lets you scope what different agent identities see without managing separate database users for each agent:

ALTER TABLE mcp_audit.query_log ENABLE ROW LEVEL SECURITY;

CREATE POLICY agent_isolation ON mcp_audit.query_log
    USING (agent_id = current_setting('mcp.agent_id', TRUE));
