Chapter 8: Reserved Instances vs Savings Plans vs On-Demand: The Math

Organizations that run AWS RDS, Azure SQL, or Google Cloud SQL on pure On-Demand pricing are typically overpaying by 30–72% compared to what they could be spending with the right commitment model. That is not a rounding error — on a $40,000/month RDS bill, a well-executed Reserved Instance strategy returns between $12,000 and $28,800 back to the business every single month. This chapter works through the exact math behind Reserved Instances (RIs), Savings Plans, and On-Demand pricing across all three major clouds, shows you how to identify which workloads qualify for each model, and provides automation to keep your commitment coverage optimized as your fleet evolves.

---

8.1 Understanding the Pricing Tiers: What You Are Actually Buying

Before running any numbers, it helps to understand what distinguishes each pricing model at a mechanical level, because the differences matter enormously when you map them to database workloads.

On-Demand is the default. You pay a fixed hourly rate with no commitment. For an db.r6g.2xlarge running PostgreSQL on RDS in us-east-1, that rate is $0.48/hour as of mid-2024, which works out to $345.60/month or $4,147.20/year. On-Demand is appropriate for workloads that run fewer than roughly 40% of the hours in a month, or for instances you cannot predict will exist in 12 months.

Reserved Instances (RIs) on AWS require a 1-year or 3-year commitment to a specific instance family, region, database engine, and license type. In exchange for that commitment, you receive a discount applied hourly whether the instance runs or not. The same db.r6g.2xlarge under a 1-year No Upfront RI costs $0.302/hour — a 37% discount. A 3-year All Upfront RI drops the effective hourly rate to $0.192, a 60% discount. The tradeoff is inflexibility: an RI for MySQL cannot be applied to PostgreSQL, and standard RIs cannot be moved across regions.

Convertible RIs on AWS partially address that inflexibility. You can exchange a Convertible RI for a different instance type, engine, or configuration, but the discount is reduced — typically 31–54% rather than 37–60%.

Compute Savings Plans on AWS (relevant for RDS Serverless v2 and Aurora Serverless) apply to any compute usage across a family regardless of engine, region, or size, in exchange for a $/hour commitment. They do not apply to standard RDS provisioned instances — a point that confuses many teams mixing RDS and Aurora.

Azure Reserved VM Instances and Azure SQL Reserved Capacity operate on similar logic: 1-year or 3-year terms with discounts of 33–65% compared to pay-as-you-go. Azure Reserved Capacity applies at the vCore level and can be shared across subscriptions within a billing account.

Google Cloud Committed Use Discounts (CUDs) for Cloud SQL operate in 1-year or 3-year terms against vCPU and memory resources rather than specific instance types, giving them more flexibility than AWS standard RIs.

The table below summarizes effective discounts for a representative 8-vCPU, 64GB memory database instance across clouds:

| Cloud | On-Demand $/mo | 1yr RI/CUD $/mo | 3yr RI/CUD $/mo | 3yr Savings |

|---|---|---|---|---|

| AWS RDS r6g.2xlarge (PostgreSQL) | $345.60 | $217.44 | $138.24 | $2,491/yr |

| Azure SQL GP Gen5 8vCore | $368.40 | $246.82 | $171.72 | $2,360/yr |

| GCP Cloud SQL n2-highmem-8 | $331.20 | $231.84 | $165.60 | $1,990/yr |

These figures exclude storage, I/O, and data transfer — all of which carry their own pricing structures covered in Chapter 9. The numbers here are compute-only, which is typically the largest single line item on a database bill.

8.2 Diagnosing Your Current Coverage Gap

You cannot optimize what you have not measured. Most organizations discover they have a patchwork of RIs purchased at different times, some expired, some mismatched to current instance types, with large portions of the fleet still running On-Demand. The following queries and CLI commands expose exactly where that gap is.

Step 1: Pull your RDS On-Demand spend from AWS Cost Explorer.

# Retrieve last 90 days of RDS On-Demand costs by instance type
# Requires Cost Explorer enabled and appropriate IAM permissions
aws ce get-cost-and-usage \
  --time-period Start=2024-04-01,End=2024-07-01 \
  --granularity MONTHLY \
  --filter '{
    "And": [
      {
        "Dimensions": {
          "Key": "SERVICE",
          "Values": ["Amazon Relational Database Service"]
        }
      },
      {
        "Dimensions": {
          "Key": "PURCHASE_TYPE",
          "Values": ["On Demand"]
        }
      }
    ]
  }' \
  --group-by '[
    {"Type": "DIMENSION", "Key": "INSTANCE_TYPE"},
    {"Type": "DIMENSION", "Key": "USAGE_TYPE"}
  ]' \
  --metrics "UnblendedCost" "UsageQuantity" \
  --output json | \
  jq -r '
    .ResultsByTime[] |
    .TimePeriod.Start as $period |
    .Groups[] |
    [$period,
     .Keys[0],
     .Keys[1],
     (.Metrics.UnblendedCost.Amount | tonumber | . * 100 | round / 100),
     (.Metrics.UsageQuantity.Amount | tonumber | round)
    ] | @csv
  ' | \
  sort -t',' -k4 -nr | \
  head -30

This produces a ranked list of your most expensive On-Demand RDS instance types. The instances appearing at the top of that list — running hundreds of hours per month — are your RI candidates.

Step 2: Check existing RI coverage and utilization.

# List all active RDS Reserved Instances and their utilization
aws rds describe-reserved-db-instances \
  --query 'ReservedDBInstances[?State==`active`].[
    ReservedDBInstanceId,
    DBInstanceClass,
    MultiAZ,
    ProductDescription,
    State,
    StartTime,
    Duration,
    FixedPrice,
    UsagePrice,
    DBInstanceCount,
    RecurringCharges
  ]' \
  --output table

# Check RI utilization (how much of what you paid for is actually being used)
aws ce get-reservation-utilization \
  --time-period Start=2024-04-01,End=2024-07-01 \
  --granularity MONTHLY \
  --filter '{
    "Dimensions": {
      "Key": "SERVICE",
      "Values": ["Amazon Relational Database Service"]
    }
  }' \
  --output json | \
  jq -r '
    .UtilizationsByTime[] |
    [.TimePeriod.Start,
     (.Total.UtilizationPercentage),
     (.Total.PurchasedHours),
     (.Total.UsedHours),
     (.Total.UnusedHours),
     (.Total.RealizedSavings.Amount),
     (.Total.UnrealizedSavings.Amount)
    ] | @csv
  '

An RI utilization percentage below 80% means you purchased commitments that are not being consumed — you are paying for capacity that sits idle. Anything below 60% warrants immediate investigation; in many cases, the underlying instance was resized or terminated and nobody updated the RI portfolio.

Step 3: Run the same analysis for Azure SQL Reserved Capacity.

# Azure: List reserved capacity orders for SQL
az reservations reservation-order list \
  --output table \
  --query "[].{
    OrderId:name,
    DisplayName:properties.displayName,
    Term:properties.term,
    ExpiryDate:properties.expiryDate,
    ProvisioningState:properties.provisioningState
  }"

# Get utilization for a specific reservation order
az consumption reservations summaries list \
  --reservation-order-id <your-order-id> \
  --grain monthly \
  --output table \
  --query "[].{
    UsageDate:usageDate,
    AvgUtilizationPct:avgUtilizationPercentage,
    UsedHours:usedHours,
    ReservedHours:reservedHours,
    MinUtilPct:minUtilizationPercentage
  }"

Step 4: Query the GCP Billing export in BigQuery for CUD coverage.

Most GCP teams export billing data to BigQuery. If yours does, this query identifies Cloud SQL compute spend by purchase type:

-- GCP BigQuery: Cloud SQL spend by SKU type (On-Demand vs CUD)
-- Run this against your billing export dataset
SELECT
  DATE_TRUNC(usage_start_time, MONTH)                  AS billing_month,
  service.description                                   AS service,
  sku.description                                       AS sku_description,
  resource.name                                         AS resource_name,
  CASE
    WHEN LOWER(sku.description) LIKE '%commit%'         THEN 'Committed'
    WHEN LOWER(sku.description) LIKE '%preemptible%'    THEN 'Preemptible'
    ELSE 'On-Demand'
  END                                                   AS purchase_type,
  project.id                                            AS project_id,
  ROUND(SUM(cost), 2)                                   AS total_cost_usd,
  ROUND(SUM(usage.amount), 2)                           AS total_usage,
  usage.unit                                            AS usage_unit
FROM
  `your-project.your_dataset.gcp_billing_export_v1_XXXXXX`
WHERE
  service.description = 'Cloud SQL'
  AND DATE(usage_start_time) >= DATE_SUB(CURRENT_DATE(), INTERVAL 90 DAY)
  AND cost > 0
GROUP BY
  1, 2, 3, 4, 5, 6, 9
ORDER BY
  billing_month DESC,
  total_cost_usd DESC
LIMIT 100;

8.3 Running the Break-Even Math

The break-even analysis determines exactly how many hours per month an instance must run before an RI becomes cheaper than On-Demand. This is the single most important calculation in the chapter.

The formula is:

Break-even hours = RI_monthly_cost / (On-Demand_hourly - RI_hourly_effective)

For a db.r6g.2xlarge PostgreSQL in us-east-1:

  • → On-Demand: $0.48/hour
  • → 1-year No Upfront RI: $0.302/hour ($217.44/month)
  • → Break-even: $217.44 / ($0.48 - $0.302) = $217.44 / $0.178 = **1,221 hours/month**

    • A month has 720–744 hours. The RI costs $217.44 regardless of runtime. On-Demand at full utilization (744 hours) costs $357.12. The RI wins the moment the instance exists at all during the month — you do not need to reach 1,221 hours because the RI commitment is monthly, not per-hour.

    The real break-even question is: how many hours must the instance run in a month for On-Demand to cost less than the RI commitment? If the instance runs fewer hours than this threshold, On-Demand is cheaper:

    Hours where On-Demand < RI_monthly_commitment:
$0.48 * H < $217.44
H < 453 hours/month

    Any instance running more than 453 hours per month (roughly 61% of the month) is cheaper under a 1-year RI. Most production databases run 720+ hours. The math is decisive.
