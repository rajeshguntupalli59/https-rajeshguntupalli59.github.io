Chapter 33: Autonomous SQL Server Systems

SQL Server has long been positioned as an enterprise database that "just works" — but in autonomous systems engineering, "just works" is the floor, not the ceiling. This chapter builds the architecture layer that transforms SQL Server from a managed platform into a self-observing, self-healing, self-optimizing engine. The patterns here treat SQL Server not as a monolithic appliance but as a controllable system with programmable decision surfaces — query store metadata, Dynamic Management Views, Always On Availability Groups, and Azure SQL's autonomy APIs are all levers that can be pulled by code rather than hands. Where PostgreSQL extensions and Patroni give you composition-level autonomy, SQL Server gives you a deeply integrated observability substrate and a mature HA framework that, with the right automation layer on top, can reach equivalent self-management depth.

---

Treating SQL Server DMVs as a Real-Time Autonomy Sensor Network

Every autonomous system needs a sensor layer — a continuous stream of state signals that drives downstream decisions. SQL Server's Dynamic Management Views (DMVs) are the richest built-in sensor network of any commercial relational database. The challenge at the ADE level is not learning DMVs; it is building a collection pipeline that turns DMV snapshots into time-series signals with enough resolution and history to drive autonomous decision logic.

The core DMVs for autonomous operation form three functional clusters:

Execution health: sys.dm_exec_requests, sys.dm_exec_sessions, sys.dm_os_wait_stats, sys.dm_exec_query_stats

Resource pressure: sys.dm_os_ring_buffers, sys.dm_os_memory_clerks, sys.dm_io_virtual_file_stats, sys.dm_os_performance_counters

Plan and optimizer state: sys.dm_exec_cached_plans, sys.dm_exec_query_plan, sys.query_store_runtime_stats, sys.query_store_plan

The autonomous sensor layer collects these at differential intervals — wait stats at 15-second deltas, query store at 60-second intervals, memory clerks at 30-second intervals. Collecting point-in-time snapshots without computing deltas is one of the most common mistakes in SQL Server observability. sys.dm_os_wait_stats is cumulative since the last service restart; what the decision engine needs is the delta over the collection window.

SQL Server:

-- Autonomous wait stats delta collector
-- Run this via a SQL Agent job or external Python scheduler at 15-second intervals

IF OBJECT_ID('tempdb..#wait_baseline', 'U') IS NULL
BEGIN
    SELECT
        wait_type,
        waiting_tasks_count,
        wait_time_ms,
        signal_wait_time_ms,
        GETUTCDATE() AS snapshot_time
    INTO tempdb..#wait_baseline
    FROM sys.dm_os_wait_stats
    WHERE wait_type NOT IN (
        'SLEEP_TASK','BROKER_TO_FLUSH','BROKER_EVENTHANDLER',
        'CHECKPOINT_QUEUE','DBMIRROR_EVENTS_QUEUE','SQLTRACE_BUFFER_FLUSH',
        'CLR_AUTO_EVENT','DISPATCHER_QUEUE_SEMAPHORE','FT_IFTS_SCHEDULER_IDLE_WAIT',
        'HADR_FILESTREAM_IOMGR_IOCOMPLETION','HADR_WORK_QUEUE','ONDEMAND_TASK_QUEUE',
        'REQUEST_FOR_DEADLOCK_MONITOR','RESOURCE_QUEUE','SERVER_IDLE_CHECK',
        'SLEEP_DBSTARTUP','SLEEP_DCOMSTARTUP','SLEEP_MASTERDBREADY',
        'SLEEP_MASTERMDREADY','SLEEP_MASTERUPGRADED','SLEEP_MSDBSTARTUP',
        'SLEEP_SYSTEMTASK','SLEEP_TEMPDBSTARTUP','SNI_HTTP_ACCEPT',
        'SP_SERVER_DIAGNOSTICS_SLEEP','WAITFOR','XE_DISPATCHER_WAIT',
        'XE_TIMER_EVENT','BROKER_RECEIVE_WAITFOR','WAITFOR_TASKSHUTDOWN'
    );

END

-- Delta computation against prior baseline
SELECT
    w.wait_type,
    w.waiting_tasks_count - b.waiting_tasks_count   AS delta_tasks,
    w.wait_time_ms        - b.wait_time_ms           AS delta_wait_ms,
    w.signal_wait_time_ms - b.signal_wait_time_ms    AS delta_signal_ms,
    CASE
        WHEN (w.wait_time_ms - b.wait_time_ms) = 0 THEN 0
        ELSE CAST((w.signal_wait_time_ms - b.signal_wait_time_ms) AS FLOAT)
           / (w.wait_time_ms - b.wait_time_ms) * 100
    END AS cpu_pressure_pct,
    GETUTCDATE() AS collected_at
FROM sys.dm_os_wait_stats w
JOIN tempdb..#wait_baseline b ON w.wait_type = b.wait_type
WHERE w.wait_time_ms - b.wait_time_ms > 0
ORDER BY delta_wait_ms DESC;

-- Refresh baseline
TRUNCATE TABLE tempdb..#wait_baseline;
INSERT INTO tempdb..#wait_baseline
SELECT wait_type, waiting_tasks_count, wait_time_ms, signal_wait_time_ms, GETUTCDATE()
FROM sys.dm_os_wait_stats
WHERE wait_type NOT IN (/* same exclusion list */);

Once delta-aware DMV collection is running, the signals feed into a time-series store — typically Prometheus with a custom SQL Server exporter or a purpose-built Python collector writing to InfluxDB or TimescaleDB. The decision engine then operates against a sliding window of 5–15 minutes of signal history rather than instantaneous reads. A single spike in PAGEIOLATCH_EX waits means nothing; a sustained 10-minute elevation above a learned baseline triggers an I/O investigation workflow.

The autonomous baseline itself deserves careful engineering. Rather than static thresholds, the system learns workload-relative bounds using a rolling percentile window:

import pandas as pd
import numpy as np
from datetime import datetime, timedelta
import pyodbc

class SQLServerWaitBaseline:
    """
    Maintains rolling percentile baselines for SQL Server wait categories.
    Baselines are hour-of-week aware to account for batch vs. OLTP patterns.
    """

    WAIT_CATEGORIES = {
        'CPU':     ['SOS_SCHEDULER_YIELD', 'CXPACKET', 'THREADPOOL'],
        'IO':      ['PAGEIOLATCH_EX', 'PAGEIOLATCH_SH', 'WRITELOG', 'IO_COMPLETION'],
        'LOCK':    ['LCK_M_X', 'LCK_M_S', 'LCK_M_U', 'LCK_M_IX'],
        'MEMORY':  ['RESOURCE_SEMAPHORE', 'RESOURCE_SEMAPHORE_QUERY_COMPILE',
                    'CMEMTHREAD'],
        'NETWORK': ['ASYNC_NETWORK_IO', 'NET_WAITFOR_PACKET'],
    }

    def __init__(self, conn_str: str, lookback_days: int = 14):
        self.conn_str = conn_str
        self.lookback_days = lookback_days

    def fetch_historical_waits(self) -> pd.DataFrame:
        """Pull wait delta history from the collector's persistent store."""
        query = """
            SELECT
                category,
                hour_of_week,
                delta_wait_ms,
                collected_at
            FROM dba.wait_stat_history
            WHERE collected_at >= DATEADD(DAY, -?, GETUTCDATE())
        """
        with pyodbc.connect(self.conn_str) as conn:
            return pd.read_sql(query, conn, params=[self.lookback_days])

    def compute_baselines(self) -> dict:
        df = self.fetch_historical_waits()
        df['hour_of_week'] = pd.to_datetime(df['collected_at']).dt.dayofweek * 24 \
                           + pd.to_datetime(df['collected_at']).dt.hour
        baselines = {}
        for category in self.WAIT_CATEGORIES:
            cat_df = df[df['category'] == category]
            baselines[category] = cat_df.groupby('hour_of_week')['delta_wait_ms'].agg(
                p50=lambda x: np.percentile(x, 50),
                p95=lambda x: np.percentile(x, 95),
                p99=lambda x: np.percentile(x, 99),
            ).to_dict('index')
        return baselines

    def is_anomalous(self, category: str, current_ms: float,
                     hour_of_week: int, threshold_pct: float = 95.0) -> bool:
        baselines = self.compute_baselines()
        if category not in baselines or hour_of_week not in baselines[category]:
            return False
        p95 = baselines[category][hour_of_week]['p95']
        return current_ms > p95

The output of this sensor network is not alerts — alerts are the human path. The output is structured decision events that flow to the autonomy controller described in later subparts.

Query Store as the Autonomous Query Governor

Query Store is SQL Server's most underutilized autonomous control surface. Most teams use it reactively — they open SSMS, look at the "Top Resource Consuming Queries" report, and manually force a plan. That is precisely the workflow that autonomous systems are designed to replace. At the ADE level, Query Store becomes a continuously monitored decision substrate where the system detects plan regressions, validates alternative plans, and forces or unforces them without a human in the loop.

The autonomous query governance loop has four stages:

1. Regression detection — identify queries where the latest execution plan performs materially worse than the best known plan for the same query hash.

2. Impact scoring — rank regressions by total elapsed time impact, not just per-execution cost.

3. Validation — before forcing a plan, verify the "best" plan is still valid given current statistics.

4. Remediation — force the plan or escalate to an index-change workflow if no valid plan exists.

SQL Server:

-- Autonomous plan regression detector
-- Identifies queries where active plan is >2x worse than best known plan
-- weighted by execution frequency over the last 2 hours

WITH recent_stats AS (
    SELECT
        qsq.query_id,
        qsq.query_hash,
        qsp.plan_id,
        qsp.is_forced_plan,
        SUM(qsrs.count_executions)                    AS exec_count,
        AVG(qsrs.avg_duration)                        AS avg_duration_us,
        AVG(qsrs.avg_cpu_time)                        AS avg_cpu_us,
        AVG(qsrs.avg_logical_io_reads)                AS avg_logical_reads,
        SUM(qsrs.count_executions * qsrs.avg_duration) AS total_elapsed_us,
        MAX(qsrs.last_execution_time)                 AS last_exec
    FROM sys.query_store_query         qsq
    JOIN sys.query_store_plan          qsp  ON qsq.query_id  = qsp.query_id
    JOIN sys.query_store_runtime_stats qsrs ON qsp.plan_id   = qsrs.plan_id
    JOIN sys.query_store_runtime_stats_interval qsrsi
                                             ON qsrs.runtime_stats_interval_id
                                              = qsrsi.runtime_stats_interval_id
    WHERE qsrsi.start_time >= DATEADD(HOUR, -2, GETUTCDATE())
    GROUP BY qsq.query_id, qsq.query_hash, qsp.plan_id, qsp.is_forced_plan
),
best_plan AS (
    SELECT
        query_id,
        MIN(avg_duration_us) AS best_avg_duration_us,
        MIN(avg_logical_reads) AS best_avg_logical_reads
    FROM recent_stats
    GROUP BY query_id
),
active_plan AS (
    -- The plan with the most recent execution for each query
    SELECT rs.*
    FROM recent_stats rs
    WHERE rs.last_exec = (
        SELECT MAX(rs2.last_exec)
        FROM recent_stats rs2
        WHERE rs2.query_id = rs.query_id
    )
)
SELECT
    ap.query_id,
    ap.plan_id                              AS active_plan_id,
    ap.is_forced_plan,
    ap.avg_duration_us                      AS active_avg_duration_us,
    bp.best_avg_duration_us,
    CAST(ap.avg_duration_us AS FLOAT)
        / NULLIF(bp.best_avg_duration_us,0) AS regression_ratio,
    ap.exec_count,
    ap.total_elapsed_us / 1000.0            AS total_elapsed_ms,
    -- Find the best plan_id for forcing
    (SELECT TOP 1 rs3.plan_id
     FROM recent_stats rs3
     WHERE rs3.query_id = ap.query_id
       AND rs3.avg_duration_us = bp.best_avg_duration_us) AS best_plan_id,
    GETUTCDATE()                            AS detected_at
FROM active_plan ap
JOIN best_plan   bp ON ap.query_id = bp.query_id
WHERE CAST(ap.avg_duration_us AS FLOAT)
        / NULLIF(bp.best_avg_duration_us,0) > 2.0   -- regression threshold
  AND ap.exec_count > 10                             -- statistical significance
  AND bp.best_avg_duration_us > 1000                -- ignore sub-millisecond queries
  AND ap.is_forced_plan = 0                          -- not already managed
ORDER BY total_elapsed_ms DESC;

The Python controller that acts on this output handles the force/unforce decision with a guard rail: before forcing any plan, it verifies that statistics on referenced objects are not stale beyond a threshold. Forcing an old plan against severely outdated statistics produces a different kind of regression.

import pyodbc
import logging
from dataclasses import dataclass
from typing import Optional
from datetime import datetime

logger = logging.getLogger('autonomous_qg')

@dataclass
class PlanRegression:
    query_id: int
    active_plan_id: int
    best_plan_id: int
    regression_ratio: float
    total_elapsed_ms: float
    detected_at: datetime

class AutonomousQueryGovernor:
    """
    Detects Query Store plan regressions and autonomously forces the best
    known plan after validating statistics freshness.
    """

    STATS_STALENESS_HOURS = 24
    MAX_AUTO_FORCE_RATIO  = 10.0   # Don't auto-force if regression > 10x (likely different workload)
    MIN_ELAPSED_MS        = 5000   # Only act if total impact > 5 seconds in window

    def __init__(self, conn_str: str, dry_run: bool = False):
        self.conn_str = conn_str
        self.dry_run  = dry_run

    def _conn(self):
        return pyodbc.connect(self.conn_str, autocommit=True)

    def detect_regressions(self) -> list[PlanRegression]:
        regression_sql = open('regression_detection.sql').read()  # The query above
        with self._conn() as conn:
            cursor = conn.cursor()
            cursor.execute(regression_sql)
            return [
                PlanRegression(
                    query_id        = row.query_id,
                    active_plan_id  = row.active_plan_id,
                    best_plan_id    = row.best_plan_id,
                    regression_ratio= row.regression_ratio,
                    total_elapsed_ms= row.total_elapsed_ms,
                    detected_at     = row.detected_at
                )
                for row in cursor.fetchall()
            ]

    def validate_stats_freshness(self, query_id: int) -> bool:
        """
        Check that all tables referenced by the best plan have stats
        updated within the staleness threshold.
        """
        sql = """
            SELECT COUNT(*) AS stale_count
            FROM sys.query_store_query_text qt
            JOIN sys.query_store_query      q  ON qt.query_text_id = q.query_text_id
            -- parse referenced objects and check sys.stats / sys.stats_columns
            -- simplified: check overall DB stats freshness as a proxy
            CROSS APPLY (
                SELECT COUNT(*) AS stale
                FROM sys.stats s
                JOIN sys.objects o ON s.object_id = o.object_id
                WHERE o.type = 'U'
                  AND STATS_DATE(s.object_id, s.stats_id) < DATEADD(
                      HOUR, -?, GETUTCDATE())
                  AND s.auto_created = 1
            ) stale_check
            WHERE q.query_id = ?
              AND stale_check.stale > 0
        """
        with self._conn() as conn:
            cursor = conn.cursor()
            cursor.execute(sql, [self.STATS_STALENESS_HOURS, query_id])
            row = cursor.fetchone()
            return row.stale_count == 0

    def force_plan(self, query_id: int, plan_id: int) -> bool:
        sql = "EXEC sys.sp_query_store_force_plan @query_id=?, @plan_id=?"
        try:
            if not self.dry_run:
                with self._conn() as conn:
                    conn.cursor().execute(sql, [query_id, plan_id])
            logger.info(
                "PLAN_FORCED query_id=%d plan_id=%d dry_run=%s",
                query_id, plan_id, self.dry_run
            )
            return True
        except Exception as e:
            logger.error("PLAN_FORCE_FAILED query_id=%d error=%s", query_id, e)
            return False

    def run_cycle(self):
        regressions = self.detect_regressions()
        logger.info("CYCLE_START detected_regressions=%d", len(regressions))

        for reg in regressions:
            if reg.regression_ratio > self.MAX_AUTO_FORCE_RATIO:
                logger.warning(
                    "ESCALATE_HUMAN query_id=%d ratio=%.1f "
                    "(exceeds autonomous threshold)",
                    reg.query_id, reg.regression_ratio
                )
                continue

            if reg.total_elapsed_ms < self.MIN_ELAPSED_MS:
                logger.debug(
                    "SKIP_LOW_IMPACT query_id=%d elapsed_ms=%.0f",
                    reg.query_id, reg.total_elapsed_ms
                )
                continue

            if not self.validate_stats_freshness(reg.query_id):
                logger.warning(
                    "STATS_STALE query_id=%d — triggering stats update first",
                    reg.query_id
                )
                # Emit a stats update work item to the autonomy queue
                # rather than forcing the plan
                continue

            self.force_plan(reg.query_id, reg.best_plan_id)

The same loop runs in reverse: when a forced plan itself starts regressing (because data distribution shifted and the forced plan is no longer optimal), the system unforces it and lets the optimizer replan. This bidirectional governance is what separates autonomous query management from the one-directional manual interventions most teams perform.

Automatic plan correction — introduced in SQL Server 2017 as a Database Compatibility feature — provides a native version of this loop. Enabling it is a prerequisite, but it is not sufficient for full autonomy because its correction logic is binary (force or don't) and does not participate in a broader decision pipeline. The ADE builds on top of it, using sys.dm_db_tuning_recommendations as an additional signal source:

SQL Server:

-- Surface the native engine's own tuning recommendations
-- Use these as pre-validated signals in the autonomous decision pipeline

SELECT
    name,
    type,
    reason,
    score,
    JSON_VALUE(details, '$.implementationDetails.script') AS recommended_action,
    JSON_VALUE(details, '$.planForceDetails.queryId')     AS affected_query_id,
    JSON_VALUE(details, '$.planForceDetails.regressedPlanId')
                                                          AS regressed_plan_id,
    JSON_VALUE(details, '$.planForceDetails.forcedPlanId')
                                                          AS recommended_plan_id,
    valid_since,
    last_refresh
FROM sys.dm_db_tuning_recommendations
WHERE state = JSON_VALUE(state, '$.currentValue')  -- active recommendations only
  AND score > 30                                    -- confidence threshold
ORDER BY score DESC;

Always On Availability Groups: Autonomous Failover and Replica Management

Always On Availability Groups are SQL Server's primary HA mechanism, but the default configuration places humans at the center of decisions that should be automated: when to fail over, when to promote a secondary, when to redirect read workloads, and when to suspend data movement to a lagging replica. The autonomous architecture eliminates those human decision points.

The autonomy surface for AG management has three layers. The first is the Windows Server Failover Cluster (WSFC) or Pacemaker layer, which handles the low-level quorum and node fencing decisions. The second is the SQL Server AG health detection layer — sys.dm_hadr_availability_group_states, sys.dm_hadr_database_replica_states, sys.dm_hadr_availability_replica_states. The third is the application-level routing layer, which manages connection routing independently of what the AG itself reports.

For environments running Always On across Azure VMs or on-premises with Pacemaker on Linux, the autonomous controller monitors redo queue depth as a leading indicator of failover risk. A secondary with a growing redo queue that is not catching up indicates either an I/O-bound secondary or a network partition that will worsen before it improves.
