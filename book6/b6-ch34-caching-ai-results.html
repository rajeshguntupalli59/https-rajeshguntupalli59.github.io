Chapter 34: Caching AI Results in PostgreSQL

AI inference is expensive. A single GPT-4 call with a 2,000-token prompt costs real money, and at scale — thousands of users, millions of queries — those costs compound fast. Latency is equally punishing: users waiting 3–8 seconds for every response abandon features. This chapter builds a complete caching layer for AI results directly inside PostgreSQL, covering exact-match caching, semantic similarity caching with pgvector, cache invalidation strategies, and the operational patterns that keep your cache healthy in production. You'll see how PostgreSQL's combination of transactional safety, JSONB flexibility, and vector search makes it a uniquely capable caching backend — one that understands the meaning of queries, not just their text.

---

34.1 Why Cache AI Results — and Why PostgreSQL Is the Right Place

Every AI application has a distribution of queries that looks roughly the same: a fat tail of rare, unique queries and a surprisingly large bulk of semantically similar questions asked repeatedly in different words. "What are your business hours?" "When do you open?" "Are you open on Saturdays?" These are three different strings that map to the same answer. A naive string-match cache misses all of that. A vector similarity cache catches it.

PostgreSQL with pgvector is the right place for this cache because:

1. Your application data already lives there. Cache entries can reference users, sessions, documents, and products with foreign keys — giving you invalidation triggers when source data changes.

2. Transactional integrity. A cache write and a conversation history insert happen in one transaction. No partial states.

3. pgvector's <=> operator turns semantic matching into a standard SQL query with an index. No separate infrastructure.

4. Row-level security lets you scope caches per tenant without query-level filtering code.

5. pg_notify enables real-time cache invalidation across application servers.

The alternative — Redis + a separate vector store + your relational database — requires you to keep three systems in sync. PostgreSQL handles all three roles.

#### Core schema design

-- Cache schema: all AI caching infrastructure lives here
CREATE SCHEMA IF NOT EXISTS ai_cache;

-- Exact-match cache for deterministic lookups
CREATE TABLE ai_cache.exact_results (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    cache_key           TEXT NOT NULL,           -- SHA-256 of (model + normalized_prompt)
    tenant_id           UUID REFERENCES public.tenants(id) ON DELETE CASCADE,
    model               TEXT NOT NULL,           -- 'gpt-4o', 'claude-3-5-sonnet-20241022'
    prompt_hash         TEXT NOT NULL,           -- SHA-256 of the raw prompt
    normalized_prompt   TEXT NOT NULL,           -- lowercase, stripped prompt
    response_text       TEXT NOT NULL,
    response_metadata   JSONB NOT NULL DEFAULT '{}',  -- token counts, finish reason, etc.
    token_input         INTEGER,
    token_output        INTEGER,
    cost_usd            NUMERIC(10, 6),
    hit_count           INTEGER NOT NULL DEFAULT 0,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    expires_at          TIMESTAMPTZ,
    last_hit_at         TIMESTAMPTZ,
    invalidated_at      TIMESTAMPTZ,
    CONSTRAINT uq_exact_cache_key UNIQUE (cache_key)
);

-- Semantic similarity cache using pgvector
CREATE TABLE ai_cache.semantic_results (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id           UUID REFERENCES public.tenants(id) ON DELETE CASCADE,
    model               TEXT NOT NULL,
    embedding_model     TEXT NOT NULL DEFAULT 'text-embedding-3-small',
    prompt_text         TEXT NOT NULL,
    prompt_embedding    vector(1536) NOT NULL,
    response_text       TEXT NOT NULL,
    response_metadata   JSONB NOT NULL DEFAULT '{}',
    token_input         INTEGER,
    token_output        INTEGER,
    cost_usd            NUMERIC(10, 6),
    similarity_threshold NUMERIC(4, 3) NOT NULL DEFAULT 0.92,
    hit_count           INTEGER NOT NULL DEFAULT 0,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    expires_at          TIMESTAMPTZ,
    last_hit_at         TIMESTAMPTZ,
    invalidated_at      TIMESTAMPTZ,
    source_document_ids UUID[] DEFAULT '{}',     -- for invalidation when docs change
    tags                TEXT[] DEFAULT '{}'
);

-- HNSW index for fast approximate nearest-neighbor search
CREATE INDEX idx_semantic_cache_embedding
    ON ai_cache.semantic_results
    USING hnsw (prompt_embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- Partial index: only active (non-invalidated, non-expired) entries
CREATE INDEX idx_semantic_cache_active
    ON ai_cache.semantic_results (tenant_id, model, created_at)
    WHERE invalidated_at IS NULL AND (expires_at IS NULL OR expires_at > now());

CREATE INDEX idx_exact_cache_active
    ON ai_cache.exact_results (cache_key)
    WHERE invalidated_at IS NULL AND (expires_at IS NULL OR expires_at > now());

-- Cache analytics table — track hits and costs saved
CREATE TABLE ai_cache.access_log (
    id              BIGSERIAL PRIMARY KEY,
    cache_type      TEXT NOT NULL CHECK (cache_type IN ('exact', 'semantic', 'miss')),
    cache_entry_id  UUID,
    tenant_id       UUID,
    model           TEXT,
    similarity      NUMERIC(6, 4),      -- NULL for exact hits
    latency_ms      INTEGER,
    cost_saved_usd  NUMERIC(10, 6),
    accessed_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_access_log_tenant_time
    ON ai_cache.access_log (tenant_id, accessed_at DESC);

34.2 Exact-Match Caching: The First Line of Defense

Exact-match caching is the cheapest, fastest layer. Before calling any LLM, normalize the prompt and compute a cache key. If that key exists in the cache and hasn't expired, return it immediately.

Normalization matters more than it seems. "What are your hours?" and "What are your hours? " (trailing space) are the same question. Lowercase folding, whitespace normalization, and stripping common filler phrases dramatically increase your exact-hit rate.

-- Function: normalize a prompt for exact cache lookup
CREATE OR REPLACE FUNCTION ai_cache.normalize_prompt(raw_prompt TEXT)
RETURNS TEXT
LANGUAGE plpgsql IMMUTABLE STRICT
AS $$
DECLARE
    normalized TEXT;
BEGIN
    -- Lowercase
    normalized := lower(raw_prompt);
    -- Collapse whitespace
    normalized := regexp_replace(normalized, '\s+', ' ', 'g');
    -- Strip leading/trailing whitespace
    normalized := trim(normalized);
    -- Remove trailing punctuation variations (optional, tune per use case)
    normalized := regexp_replace(normalized, '[.!?]+$', '', 'g');
    RETURN normalized;
END;
$$;

-- Function: compute cache key from model + normalized prompt
CREATE OR REPLACE FUNCTION ai_cache.compute_cache_key(
    p_model TEXT,
    p_normalized_prompt TEXT,
    p_tenant_id UUID DEFAULT NULL
)
RETURNS TEXT
LANGUAGE sql IMMUTABLE STRICT
AS $$
    SELECT encode(
        sha256(
            (COALESCE(p_tenant_id::TEXT, '') || '|' || p_model || '|' || p_normalized_prompt)::bytea
        ),
        'hex'
    );
$$;

-- Function: look up exact cache hit and bump hit counter atomically
CREATE OR REPLACE FUNCTION ai_cache.lookup_exact(
    p_model         TEXT,
    p_prompt        TEXT,
    p_tenant_id     UUID DEFAULT NULL
)
RETURNS TABLE (
    cache_id        UUID,
    response_text   TEXT,
    response_meta   JSONB,
    hit_count       INTEGER
)
LANGUAGE plpgsql
AS $$
DECLARE
    v_normalized    TEXT;
    v_cache_key     TEXT;
BEGIN
    v_normalized := ai_cache.normalize_prompt(p_prompt);
    v_cache_key  := ai_cache.compute_cache_key(p_model, v_normalized, p_tenant_id);

    RETURN QUERY
    UPDATE ai_cache.exact_results
    SET
        hit_count   = ai_cache.exact_results.hit_count + 1,
        last_hit_at = now()
    WHERE
        cache_key       = v_cache_key
        AND invalidated_at IS NULL
        AND (expires_at IS NULL OR expires_at > now())
    RETURNING
        id,
        ai_cache.exact_results.response_text,
        response_metadata,
        ai_cache.exact_results.hit_count;
END;
$$;

-- Function: store a new exact cache entry
CREATE OR REPLACE FUNCTION ai_cache.store_exact(
    p_model             TEXT,
    p_prompt            TEXT,
    p_response_text     TEXT,
    p_response_metadata JSONB,
    p_tenant_id         UUID DEFAULT NULL,
    p_token_input       INTEGER DEFAULT NULL,
    p_token_output      INTEGER DEFAULT NULL,
    p_cost_usd          NUMERIC DEFAULT NULL,
    p_ttl_seconds       INTEGER DEFAULT NULL
)
RETURNS UUID
LANGUAGE plpgsql
AS $$
DECLARE
    v_normalized    TEXT;
    v_cache_key     TEXT;
    v_prompt_hash   TEXT;
    v_expires_at    TIMESTAMPTZ;
    v_id            UUID;
BEGIN
    v_normalized  := ai_cache.normalize_prompt(p_prompt);
    v_cache_key   := ai_cache.compute_cache_key(p_model, v_normalized, p_tenant_id);
    v_prompt_hash := encode(sha256(p_prompt::bytea), 'hex');
    v_expires_at  := CASE WHEN p_ttl_seconds IS NOT NULL
                          THEN now() + (p_ttl_seconds || ' seconds')::interval
                          ELSE NULL END;

    INSERT INTO ai_cache.exact_results (
        cache_key, tenant_id, model, prompt_hash, normalized_prompt,
        response_text, response_metadata, token_input, token_output,
        cost_usd, expires_at
    )
    VALUES (
        v_cache_key, p_tenant_id, p_model, v_prompt_hash, v_normalized,
        p_response_text, p_response_metadata, p_token_input, p_token_output,
        p_cost_usd, v_expires_at
    )
    ON CONFLICT (cache_key) DO UPDATE
        SET response_text     = EXCLUDED.response_text,
            response_metadata = EXCLUDED.response_metadata,
            token_input       = EXCLUDED.token_input,
            token_output      = EXCLUDED.token_output,
            cost_usd          = EXCLUDED.cost_usd,
            expires_at        = EXCLUDED.expires_at,
            invalidated_at    = NULL  -- un-invalidate on refresh
    RETURNING id INTO v_id;

    RETURN v_id;
END;
$$;

34.3 Semantic Similarity Caching with pgvector

Exact caching handles repeats. Semantic caching handles paraphrases. The idea is simple: embed the incoming prompt, then search the cache for stored prompts whose embeddings are within a cosine similarity threshold. If you find one, return its cached response.

The threshold is the critical tuning knob. Too low (0.80) and you'll return answers to semantically adjacent but actually different questions. Too high (0.99) and you'll miss obvious paraphrases. A starting point of 0.92 works well for factual Q&A. For creative or nuanced tasks, raise it to 0.95.

-- Function: semantic cache lookup
CREATE OR REPLACE FUNCTION ai_cache.lookup_semantic(
    p_model             TEXT,
    p_prompt_embedding  vector(1536),
    p_tenant_id         UUID DEFAULT NULL,
    p_similarity_threshold NUMERIC DEFAULT 0.92,
    p_limit             INTEGER DEFAULT 1
)
RETURNS TABLE (
    cache_id        UUID,
    response_text   TEXT,
    response_meta   JSONB,
    similarity      NUMERIC,
    hit_count       INTEGER
)
LANGUAGE plpgsql
AS $$
BEGIN
    RETURN QUERY
    WITH candidates AS (
        SELECT
            id,
            ai_cache.semantic_results.response_text,
            response_metadata,
            1 - (prompt_embedding <=> p_prompt_embedding) AS cosine_similarity,
            ai_cache.semantic_results.hit_count
        FROM ai_cache.semantic_results
        WHERE
            model           = p_model
            AND (tenant_id = p_tenant_id OR (tenant_id IS NULL AND p_tenant_id IS NULL))
            AND invalidated_at IS NULL
            AND (expires_at IS NULL OR expires_at > now())
        ORDER BY prompt_embedding <=> p_prompt_embedding
        LIMIT 10  -- retrieve top-10 candidates from HNSW, then filter
    )
    SELECT
        id,
        candidates.response_text,
        response_metadata,
        cosine_similarity::NUMERIC,
        candidates.hit_count
    FROM candidates
    WHERE cosine_similarity >= p_similarity_threshold
    ORDER BY cosine_similarity DESC
    LIMIT p_limit;

    -- Update hit count for the returned entry (fire and forget via a CTE)
    UPDATE ai_cache.semantic_results sr
    SET
        hit_count   = sr.hit_count + 1,
        last_hit_at = now()
    FROM (
        SELECT id FROM ai_cache.semantic_results
        WHERE
            model = p_model
            AND (tenant_id = p_tenant_id OR (tenant_id IS NULL AND p_tenant_id IS NULL))
            AND invalidated_at IS NULL
            AND (expires_at IS NULL OR expires_at > now())
        ORDER BY prompt_embedding <=> p_prompt_embedding
        LIMIT 1
    ) AS top
    WHERE sr.id = top.id
      AND (1 - (sr.prompt_embedding <=> p_prompt_embedding)) >= p_similarity_threshold;
END;
$$;

-- Function: store semantic cache entry
CREATE OR REPLACE FUNCTION ai_cache.store_semantic(
    p_model                 TEXT,
    p_embedding_model       TEXT,
    p_prompt_text           TEXT,
    p_prompt_embedding      vector(1536),
    p_response_text         TEXT,
    p_response_metadata     JSONB,
    p_tenant_id             UUID DEFAULT NULL,
    p_token_input           INTEGER DEFAULT NULL,
    p_token_output          INTEGER DEFAULT NULL,
    p_cost_usd              NUMERIC DEFAULT NULL,
    p_ttl_seconds           INTEGER DEFAULT NULL,
    p_source_document_ids   UUID[] DEFAULT '{}',
    p_tags                  TEXT[] DEFAULT '{}'
)
RETURNS UUID
LANGUAGE plpgsql
AS $$
DECLARE
    v_expires_at    TIMESTAMPTZ;
    v_id            UUID;
BEGIN
    v_expires_at := CASE WHEN p_ttl_seconds IS NOT NULL
                         THEN now() + (p_ttl_seconds || ' seconds')::interval
                         ELSE NULL END;

    INSERT INTO ai_cache.semantic_results (
        tenant_id, model, embedding_model, prompt_text, prompt_embedding,
        response_text, response_metadata, token_input, token_output,
        cost_usd, expires_at, source_document_ids, tags
    )
    VALUES (
        p_tenant_id, p_model, p_embedding_model, p_prompt_text, p_prompt_embedding,
        p_response_text, p_response_metadata, p_token_input, p_token_output,
        p_cost_usd, v_expires_at, p_source_document_ids, p_tags
    )
    RETURNING id INTO v_id;

    RETURN v_id;
END;
$$;
