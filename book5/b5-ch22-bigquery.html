Chapter 22: BigQuery for Analytics: When to Push Workloads Out

Google BigQuery sits in a different category from every other database discussed in this book. It is not a replacement for Cloud SQL or AlloyDB. It is not a competitor to Aurora or Azure SQL. BigQuery is a serverless, columnar analytical engine designed to scan terabytes in seconds and return aggregated results to dashboards, data science notebooks, and reporting pipelines — workloads that would bring any transactional database to its knees if you left them there long enough. This chapter explains the architectural reasons why certain query patterns belong in BigQuery rather than your OLTP engine, how to move data from RDS, Cloud SQL, and AlloyDB into BigQuery efficiently, how to configure BigQuery for production use including IAM, partitioning, and cost control, and how to build the operational habits that keep your analytics layer running without surprise bills or query timeouts at 2 AM.

---

22.1 Understanding Why OLTP Engines Break Under Analytics Load

Before you can decide what to push to BigQuery, you need to understand precisely why transactional databases fail at analytical workloads — not in a vague hand-waving sense, but mechanically.

PostgreSQL, SQL Server, and their managed equivalents on AWS and Azure store data in row-oriented heap files. Every row's columns are stored contiguously on disk. This design is optimal when you fetch a complete record: a customer order, a user profile, a payment transaction. The buffer pool fills with pages containing complete rows, and repeated transactional reads and writes work efficiently.

Analytics queries do the opposite. A query like SELECT region, SUM(revenue) FROM orders WHERE order_date >= '2023-01-01' GROUP BY region needs only two columns from a table that might have forty. On a row store, the database must read all forty columns from every qualifying row page into memory just to extract two of them. If your orders table is 800 GB, you are reading 800 GB to return a handful of rows. Buffer pool pressure spikes. I/O throughput maxes out. Other sessions wait for locks on shared buffer pages. Your OLTP application starts throwing connection timeouts.

RDS PostgreSQL on an db.r6g.4xlarge instance gives you 128 GB of RAM and roughly 4,750 MB/s of EBS throughput on io2 volumes. A full table scan of 800 GB at that throughput takes over two and a half minutes in pure I/O time, ignoring CPU for hashing and aggregation. Now put that query in a BI tool that twenty analysts hit simultaneously. You have created a resource contention crisis on the same database your application depends on for writes.

Read replicas help but do not solve this. A read replica on RDS or Aurora still uses a row-oriented storage engine. You have moved the scan off the primary, but you still saturate the replica's I/O and memory. Aurora's parallel query feature can push some aggregation down to the storage layer, but it targets specific single-table scan patterns, not complex multi-join analytical workloads.

BigQuery's columnar storage changes the math entirely. For that same two-column query on 800 GB of row data, BigQuery reads only the bytes belonging to region and revenue columns — roughly 2–5% of the total data volume depending on column widths and compression ratios. It distributes that read across thousands of storage shards in parallel. Slot-based compute (more on this shortly) applies CPU in parallel across the result shards. Wall-clock time for a query that takes 150 seconds on RDS might take 3–8 seconds in BigQuery.

The workloads that belong in BigQuery share a recognizable profile:

  • → Aggregations over hundreds of millions to billions of rows
  • → Historical trend analysis spanning months or years of data
  • → Ad hoc exploration queries from analysts who don't know in advance which columns they'll filter on
  • → JOIN operations between a large fact table and multiple dimension tables
  • → Machine learning feature engineering pipelines
  • → Any query where full or near-full table scans are unavoidable

    • The workloads that do *not* belong in BigQuery are equally important to recognize:

  • → Point lookups by primary key: `SELECT * FROM users WHERE user_id = 12345`
  • → Low-latency transactional writes
  • → Queries requiring sub-100ms response for user-facing applications
  • → Workloads requiring row-level locking or transactional consistency

    • BigQuery has a minimum query latency of roughly 400–800ms even for tiny result sets, due to job scheduling overhead. It is architecturally the wrong tool for anything a user is waiting on in real time.

    22.2 BigQuery Architecture: What a DBA Needs to Know

    You arrive at BigQuery with mental models built around instances, storage engines, buffer pools, and query plans. BigQuery discards most of those concepts. Building accurate new mental models before you write production SQL saves you from expensive mistakes.

    Storage and compute separation

    BigQuery stores data in Google's Capacitor columnar format across a distributed file system called Colossus. Colossus is independent of compute. When you run a query, BigQuery allocates compute units called *slots* — each slot is a combination of CPU and memory — dynamically from a shared pool. When the query completes, those slots return to the pool. You pay for storage separately from compute, and compute is measured in slot-hours, not instance uptime.

    This means there are no instances to size. There are no storage volumes to provision. There are no connection pools to manage. For a DBA coming from RDS or Cloud SQL, this feels disorienting. The operational handles you're used to pulling simply do not exist.

    Tables are not heap files

    BigQuery tables have no visible physical layout you can control the way you would with FILLFACTOR in PostgreSQL or filegroup placement in SQL Server. What you *can* control — and must, for cost and performance — are partitioning and clustering.

    *Partitioning* in BigQuery means dividing a table into segments by a date/timestamp column or an integer range, or using ingestion time. BigQuery physically separates partitions into distinct storage segments. When a query includes a filter on the partition column, BigQuery reads only qualifying partitions — called partition pruning. A table with three years of daily data has roughly 1,095 partitions. A query filtering to one month reads 30/1,095 of the data. Bytes billed drops by a factor of 36.

    *Clustering* goes one level deeper. Within each partition, BigQuery sorts data by up to four clustering columns and stores it in sorted blocks with block-level metadata. Queries filtering on clustering columns skip non-qualifying blocks. Clustering is the BigQuery equivalent of a clustered index scan range, but it is approximate, not exact — the storage layer reorganizes data asynchronously as you insert, so clustering benefits improve over time rather than being guaranteed immediately.

    Datasets, projects, and IAM

    The organizational hierarchy is: Google Cloud Project → Dataset → Table. A dataset is roughly analogous to a schema in PostgreSQL. IAM permissions apply at the project, dataset, or table level. There is no pg_hba.conf equivalent and no database-level user management. Access control is entirely IAM-based — roles/bigquery.dataViewer grants SELECT on a dataset, roles/bigquery.dataEditor adds INSERT/UPDATE/DELETE, roles/bigquery.admin grants full control.

    Pricing model — the number that surprises DBAs most

    BigQuery has two pricing models. On-demand pricing charges $6.25 per TB of bytes processed by queries (as of current GCP pricing — verify at time of implementation). Flat-rate pricing (now called BigQuery editions with capacity commitments) reserves a fixed number of slots for a flat monthly cost. For organizations running many concurrent heavy queries, flat-rate is significantly cheaper. For organizations running occasional analytical workloads, on-demand is fine.

    The operational implication: in on-demand pricing, a runaway query that scans a 10 TB unpartitioned table costs $62.50 per execution. Run by twenty analysts simultaneously, that is $1,250 in query fees in a few minutes. Partition pruning, clustering, and query governance are not optional. They are cost controls.

    22.3 Loading Data from RDS, Cloud SQL, and AlloyDB into BigQuery

    There are four primary patterns for getting data from your OLTP engines into BigQuery. The right choice depends on your latency requirements, data volume, and operational complexity tolerance.

    Pattern 1: Batch export via Cloud Storage

    The simplest and most reliable pattern for many shops: export data from your OLTP database on a schedule (hourly, every 15 minutes, or nightly), land it in Cloud Storage as Parquet or CSV, and load it into BigQuery via a load job or external table.

    From Cloud SQL for PostgreSQL, you can export directly to GCS:

    # Export a table from Cloud SQL to GCS as CSV
gcloud sql export csv my-production-instance \
  gs://my-analytics-bucket/exports/orders/orders_$(date +%Y%m%d_%H%M).csv \
  --database=production \
  --query="SELECT * FROM orders WHERE updated_at >= NOW() - INTERVAL '1 hour'" \
  --offload

# Load the exported file into BigQuery
bq load \
  --source_format=CSV \
  --skip_leading_rows=1 \
  --autodetect \
  --time_partitioning_field=order_date \
  --clustering_fields=region,customer_segment \
  analytics_dataset.orders \
  "gs://my-analytics-bucket/exports/orders/orders_$(date +%Y%m%d_%H%M).csv"

    The --offload flag on the Cloud SQL export is important — it routes the export through a read replica offload path rather than consuming CPU on the primary instance.

    For RDS PostgreSQL, there is no direct export-to-GCS command. You use pg_dump or COPY to a local file, upload to GCS, then load into BigQuery:

    # Export from RDS PostgreSQL using COPY to S3 first, then transfer to GCS
# Step 1: Export from RDS to S3 using aws_s3 extension
psql -h myinstance.us-east-1.rds.amazonaws.com -U admin -d production << 'EOF'
SELECT * FROM aws_s3.query_export_to_s3(
  'SELECT order_id, customer_id, region, revenue, order_date
   FROM orders
   WHERE order_date >= CURRENT_DATE - INTERVAL ''1 day''',
  aws_commons.create_s3_uri(
    'my-rds-export-bucket',
    'exports/orders/daily_export.csv',
    'us-east-1'
  ),
  options := 'format csv, header true'
);
EOF

# Step 2: Transfer from S3 to GCS using Storage Transfer Service
gcloud transfer jobs create \
  --source-agent-pool=s3-source \
  --source-creds-file=aws-creds.json \
  s3://my-rds-export-bucket/exports/orders/ \
  gs://my-analytics-bucket/exports/orders/

# Step 3: Load into BigQuery
bq load \
  --source_format=CSV \
  --skip_leading_rows=1 \
  --time_partitioning_field=order_date \
  analytics_dataset.orders \
  "gs://my-analytics-bucket/exports/orders/daily_export.csv"

    Pattern 2: Datastream for CDC-based replication

    Google Datastream is a managed change data capture service that reads from MySQL or PostgreSQL binlog/WAL and streams changes into BigQuery with typical latency of 30–120 seconds. For RDS PostgreSQL or Cloud SQL, this is the standard near-real-time path.

    # Create a Datastream connection profile for RDS PostgreSQL
gcloud datastream connection-profiles create rds-postgres-source \
  --location=us-central1 \
  --type=postgresql \
  --postgresql-hostname=myinstance.us-east-1.rds.amazonaws.com \
  --postgresql-port=5432 \
  --postgresql-username=datastream_user \
  --postgresql-password=REDACTED \
  --postgresql-database=production \
  --display-name="RDS Production PostgreSQL"

# Create the BigQuery destination profile
gcloud datastream connection-profiles create bigquery-dest \
  --location=us-central1 \
  --type=bigquery \
  --display-name="Analytics BigQuery"

# Create the stream
gcloud datastream streams create orders-cdc-stream \
  --location=us-central1 \
  --display-name="Orders CDC Stream" \
  --source=rds-postgres-source \
  --postgresql-source-config-include-objects=public.orders,public.customers \
  --destination=bigquery-dest \
  --bigquery-destination-config-dataset-id=analytics_dataset \
  --backfill-all

    Before creating the stream, your RDS instance needs logical replication enabled. Set the rds.logical_replication parameter to 1 in your RDS parameter group, which sets wal_level=logical. The Datastream user needs REPLICATION privilege and SELECT on the tables being replicated.

    One production gotcha with Datastream and BigQuery: Datastream writes CDC events into BigQuery as append-only rows with a _metadata_row_is_deleted column. The destination table accumulates all versions of each row. To get the current state of your data, you query with a deduplification pattern:

    -- Query current state of orders from CDC-replicated table
-- This pattern is essential when using Datastream into BigQuery
WITH ranked_orders AS (
  SELECT
    *,
    ROW_NUMBER() OVER (
      PARTITION BY order_id
      ORDER BY _metadata_timestamp DESC
    ) AS row_num
  FROM `my-project.analytics_dataset.orders`
  WHERE _metadata_row_is_deleted IS FALSE
    OR _metadata_row_is_deleted IS NULL
)
SELECT
  order_id,
  customer_id,
  region,
  revenue,
  order_date,
  status
FROM ranked_orders
WHERE row_num = 1;

    In practice, you materialize this into a separate orders_current table using a scheduled query that runs every 15–30 minutes, rather than running the deduplication logic on every analytical query.

    Pattern 3: AlloyDB Omni and federated queries

    AlloyDB supports BigQuery federated queries via the AlloyDB connector. This allows BigQuery to query AlloyDB tables directly without a separate ETL step. The latency is higher than native BigQuery storage (AlloyDB still does row-oriented reads), but it removes the data movement entirely for low-frequency analytical queries.

    Pattern 4: BigQuery Data Transfer Service for SaaS sources

    If your data originates in SaaS platforms — Salesforce, Google Ads, YouTube — BigQuery Data Transfer Service ingests it natively. This is outside the DBA's typical concern but worth noting when the analytics team asks why their CRM data isn't in the same place as the orders data.

    22.4 Configuring BigQuery for Production Use

    Partitioning and clustering strategy

    Apply partitioning and clustering at table creation time. Retrofitting them requires recreating the table.

    -- Create a production-ready partitioned and clustered orders table
CREATE TABLE `my-project.analytics_dataset.orders`
(
  order_id        INT64 NOT NULL,
  customer_id     INT64 NOT NULL,
  region          STRING NOT NULL,
  customer_segment STRING,
  revenue         NUMERIC(18, 2),
  order_date      DATE NOT NULL,
  status          STRING,
  created_at      TIMESTAMP,
  updated_at      TIMESTAMP
)
PARTITION BY order_date
CLUSTER BY region, customer_segment
OPTIONS (
  partition_expiration_days = 1825,  -- 5 years
  require_partition_filter = TRUE,   -- Reject queries that don't filter on order_date
  description = "Orders fact table - partitioned by order_date, clustered by region and segment"
);

    The require_partition_filter = TRUE option is one of the most important production controls available. It forces every query against this table to include a filter on order_date, preventing full table scans. An analyst who forgets a WHERE clause gets an error rather than a $62 surprise on the billing report. You will get pushback from analysts who want to run aggregate queries across all time. The answer is to create a separate summary table for those use cases, not to remove the partition filter from the fact table.

    IAM and access control

    # Grant dataset-level read access to the analytics team service account
bq add-iam-policy-binding \
  --member=serviceAccount:analytics-readers@my-project.iam.gserviceaccount.com \
  --role=roles/bigquery.dataViewer \
  my-project:analytics_dataset

# Grant job execution permission at project level (required to run queries)
gcloud projects add-iam-policy-binding my-project \
  --member=serviceAccount:analytics-readers@my-project.iam.gserviceaccount.com \
  --role=roles/bigquery.jobUser

# Grant Looker Studio service account access to specific tables only
bq add-iam-policy-binding \
  --member=serviceAccount:looker-studio@my-project.iam.gserviceaccount.com \
  --role=roles/bigquery.dataViewer \
  --table \
  my-project:analytics_dataset.orders_summary

    Note the separation between roles/bigquery.dataViewer (read data) and roles/bigquery.jobUser (execute queries). Both are required. A common mistake is granting dataViewer but forgetting jobUser, then spending an hour debugging why a service account with read access can't run queries.

    Cost controls with per-user query quotas

    # Set a per-user daily query quota of 1 TB via the BigQuery API
# This prevents any single analyst from burning through budget
cat > quota-policy.json << 'EOF'
{
  "name": "projects/my-project/locations/us/quotas/QUERY_USAGE_PER_USER_PER_DAY",
  "preferredValue": "1099511627776"
}
EOF

# Apply via gcloud (requires quota admin permissions)
gcloud alpha bigquery quota update \
  --project=my-project \
  --location=us \
  --quota-id=QUERY_USAGE_PER_USER_PER_DAY \
  --preferred-value=1099511627776

    The preferred value is in bytes. 1,099,511,627,776 bytes = 1 TB. Analysts hitting this limit receive an error rather than a bill. Set this on day one before handing BigQuery access to anyone.

    Reservations and slot commitments for predictable costs

    If your organization runs more than roughly 50 heavy analytical queries per day, on-demand pricing becomes expensive relative to BigQuery editions (slot reservations):

    # Purchase a 100-slot baseline commitment (1-year)
gcloud bigquery reservations capacity-commitments create \
  --location=us-central1 \
  --slot-count=100 \
  --plan=ANNUAL \
  --project=my-project

# Create a reservation
gcloud bigquery reservations create analytics-baseline \
  --location=us-central1 \
  --slot-count=100 \
  --project=my-project

# Assign the analytics project to this reservation
gcloud bigquery reservations assignments create \
  --reservation=analytics-baseline \
  --location=us-central1 \
  --assignee-type=PROJECT \
  --assignee-id=my-project \
  --job-type=QUERY \
  --project=my-project

    With 100 committed slots, all queries in the assigned project draw from this reservation rather than the on-demand pool. Costs are predictable. The downside: 100 slots can become a bottleneck if five analysts simultaneously run multi-TB scans. Monitor slot utilization in Cloud Monitoring and adjust commitments quarterly.

    22.5 Query Patterns, Optimization, and What Surprises Production DBAs

    Reading INFORMATION_SCHEMA for query performance

    BigQuery's equivalent of pg_stat_statements or SQL Server DMVs is the INFORMATION_SCHEMA.JOBS view. You query it with SQL like any other table:

    -- Find the top 20 most expensive queries in the past 7 days
-- This is your first stop when investigating cost overruns
SELECT
  user_email,
  job_id,
  creation_time,
  ROUND(total_bytes_processed / POW(1024, 4), 3) AS tb_processed,
  ROUND(total_bytes_processed / POW(1024, 4) * 6.25, 2) AS estimated_cost_usd,
  ROUND(total_slot_ms / 1000.0, 1) AS total_slot_seconds,
  SUBSTR(query, 1, 200) AS query_preview,
  statement_type,
  state,
  error_result
FROM `region-us`.INFORMATION_SCHEMA.JOBS
WHERE
  creation_time >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY)
  AND job_type = 'QUERY'
  AND state = 'DONE'
ORDER BY total_bytes_processed DESC
LIMIT 20;

    Run this query weekly. You will find analysts running full table scans immediately. The pattern is always the same: someone copied a SQL Server query from a reporting tool that works fine on a 10 GB SQL Server table, pointed it at a 2 TB BigQuery table, and the partition filter was missing.

    Understanding query execution plans

    BigQuery's EXPLAIN equivalent is the query execution graph in the Cloud Console, and the INFORMATION_SCHEMA.JOBS view's query_plan column (available as JSON). For automated analysis, the Python client is more practical:

    from google.cloud import bigquery
import json

client = bigquery.Client(project="my-project")

# Retrieve the execution plan for a completed job
job = client.get_job("my-project.us.bq_job_id_here", location="us")

if hasattr(job, 'query_plan'):
    for step in job.query_plan:
        print(f"Stage: {step.name}")
        print(f"  Slot ms: {step.slot_ms}")
        print(f"  Records read: {step.records_read}")
        print(f"  Records written: {step.records_written}")
        print(f"  Shuffle output bytes: {step.shuffle_output_bytes}")
        for substep in step.steps:
            print(f"    {substep.kind}: {', '.join(substep.substeps)}")
        print()

# Also check bytes billed vs bytes processed
# Bytes billed = max(bytes_processed, 10 MB) — BigQuery charges a 10 MB minimum
print(f"Bytes processed: {job.total_bytes_processed:,}")
print(f"Bytes billed: {job.total_bytes_billed:,}")
print(f"Estimated cost: ${job.total_bytes_billed / 1e12 * 6.25:.4f}")

    The shuffle output bytes metric reveals data skew. If one stage writes dramatically more bytes than others, your aggregation key has a skew problem — a small number of values (NULL, 'UNKNOWN', a single country code) dominate a GROUP BY, causing one slot to process far more data than others. The fix is the same as in Spark or Hive: filter skewed values separately, or use APPROX_COUNT_DISTINCT instead of COUNT(DISTINCT ...) for non-exact counts.

    Three query patterns that surprise DBAs from relational backgrounds

    First, COUNT(DISTINCT column) on a billion-row table is extremely expensive in BigQuery. It requires a global sort and deduplication. Use APPROX_COUNT_DISTINCT instead if the business requirement allows for a ~1% error margin. Cardinality estimates for dashboard widgets virtually always permit this.

    Second, SELECT * is not merely a bad practice in BigQuery — it is a direct cost multiplier. Because bytes processed equals dollars spent, selecting forty columns when you need two costs twenty times more. Enforce column selection in your analytics team's coding standards and review it during query approval for scheduled reports.

    Third, JOIN order matters differently than it does in PostgreSQL. BigQuery's query optimizer handles many cases automatically, but when joining a large fact table to multiple dimension tables, always put the largest table first (leftmost) in the FROM clause and join smaller dimension tables to it. BigQuery broadcasts small tables to all slots for hash joins; if BigQuery misidentifies the small table, the broadcast goes the wrong direction and performance collapses.
