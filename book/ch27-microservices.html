Chapter 27: Microservices

The shift to microservices architecture has fundamentally changed how databases are designed, deployed, and operated. Instead of one monolithic application talking to one large database, you now have dozens or hundreds of small services, each potentially owning its own data store, communicating over networks, and failing independently. For a DBA, this is not simply a developer concern — it reshapes everything from connection management and schema ownership to backup strategy and query observability. This chapter covers what microservices mean for database operations, how PostgreSQL and SQL Server fit into that ecosystem, and the production-level patterns that keep data consistent and performant when your architecture is deliberately distributed.

---

What Microservices Mean for the Database Layer

In a monolithic application, the database is a shared resource. All parts of the application connect to the same instance, share the same schema, and often rely on joins across many tables to fulfill a single request. Transactions are easy because everything lives in one place. This simplicity comes at a cost: the monolith scales as a unit, deploys as a unit, and fails as a unit.

Microservices decompose that monolith into independent services, each responsible for a bounded domain. An order service owns order data. A user service owns user profiles. An inventory service tracks stock levels. The guiding principle, often called *database per service*, says that each service should own its data store exclusively. No other service is allowed to reach directly into that database. If the inventory service needs customer information, it calls the user service through an API — it does not run a cross-database join.

This principle feels clean on paper but immediately creates tension for anyone who has spent years writing normalized schemas with foreign keys spanning tables. Those cross-table relationships now become cross-service calls. Referential integrity, which the database once enforced, must now be maintained at the application layer or through event-driven patterns. Transactions that once spanned three tables now span three services and three network calls.

For the DBA, the practical implications are immediate. You are no longer managing one large instance with dozens of application accounts. You are managing many smaller instances — or schemas within a shared instance — with strict ownership boundaries. You are thinking about connection counts per service, schema versioning that cannot break other teams, and observability that lets you trace a single user request across multiple database interactions.

PostgreSQL handles this model well because it is lightweight enough to run as a dedicated instance per service, either in containers or on smaller virtual machines. SQL Server historically was heavier and more expensive to run at this granularity, though SQL Server on Linux and Azure SQL Database have made the economics more reasonable. In practice, many organizations using SQL Server compromise with a *schema per service* approach inside a shared instance, rather than a fully separate database per service — a pattern worth understanding for both its convenience and its risks.

Connection Management at Scale

In a monolith, you might have fifty application threads connecting to one database. Connection pooling is useful but not urgent. In microservices, you might have thirty services each running ten pods, with each pod holding its own connection pool of five to ten connections. Suddenly the database is receiving hundreds or thousands of connection attempts from dozens of independent processes, and the cost of idle connections adds up fast.

PostgreSQL's connection overhead is significant. Each connection spawns a backend process, consuming memory — typically two to five megabytes per connection at idle, more under load. A PostgreSQL instance accepting five hundred direct connections from microservice pods can exhaust memory before it processes a single query. This is why PgBouncer, the connection pooler for PostgreSQL, becomes nearly mandatory in microservices environments.

PgBouncer sits between the service pods and the PostgreSQL instance. Services connect to PgBouncer, which maintains a smaller pool of actual database connections. In transaction pooling mode, a database connection is held only for the duration of a transaction, then returned to the pool. This allows hundreds of application connections to share a much smaller number of real database connections. The tradeoff is that session-level state — prepared statements, temporary tables, advisory locks — does not survive across transactions when the connection is shared, which catches teams off guard when they migrate from a direct-connection model.

SQL Server uses a thread-based model rather than process-per-connection, so the raw memory cost per connection is lower. But SQL Server connection pools in ADO.NET and other drivers are per-connection-string, and in a microservices environment where each service has its own connection string, pool fragmentation becomes a real concern. SQL Server also has a license-sensitive context around the number of cores and concurrent workloads, which means the DBA must work with the service teams to understand expected connection counts and query volumes before services go to production.

For both databases, the DBA's job in a microservices environment includes setting maximum connection limits per service account, monitoring active versus idle connections, and alerting when connection counts approach the configured maximum.

PostgreSQL:

-- Check current connection counts by application and state
SELECT
    application_name,
    state,
    COUNT(*) AS connection_count,
    MAX(EXTRACT(EPOCH FROM (now() - state_change))) AS max_idle_seconds
FROM pg_stat_activity
WHERE datname = 'orders_db'
GROUP BY application_name, state
ORDER BY connection_count DESC;

SQL Server:

-- Check current connection counts by program name and status
SELECT
    s.program_name,
    s.status,
    COUNT(*) AS connection_count,
    MAX(DATEDIFF(SECOND, s.last_request_end_time, GETDATE())) AS max_idle_seconds
FROM sys.dm_exec_sessions s
WHERE s.database_id = DB_ID('orders_db')
  AND s.is_user_process = 1
GROUP BY s.program_name, s.status
ORDER BY connection_count DESC;

Reviewing this output regularly — or building it into a monitoring dashboard — tells you which services are holding idle connections, which helps identify misconfigured pools or services that are not closing connections properly after deployment.

Schema Ownership and Versioning Without Breaking Other Services

In a monolith, a DBA might apply a migration once, coordinate with a single deployment pipeline, and verify one application. In microservices, the schema belongs to one service, but that schema might be read by other services during a transitional period, consumed by reporting pipelines, or referenced by event consumers that are not under the same team's control.

The most dangerous operation in this environment is a destructive schema change — dropping a column, renaming a table, changing a data type in a way that breaks existing queries. If a service deploys a migration that removes a column while a previous version of that service is still running in other pods, you get runtime errors in the pods that have not yet restarted. This is the classic problem of *zero-downtime deployments* running into schema changes.

The discipline that solves this is *expand/contract migration*. Instead of renaming a column in one step, you first add the new column alongside the old one (expand), deploy code that writes to both and reads from the new one, then remove the old column only after the old version is fully retired (contract). The expand phase is backward compatible. The contract phase happens independently, once you are certain nothing depends on the old column.

Both PostgreSQL and SQL Server support adding nullable columns to large tables without a full table lock in modern versions, which makes the expand phase operationally safe. However, adding a NOT NULL column without a default requires a full table rewrite in older PostgreSQL versions — in PostgreSQL 11 and later, adding a column with a non-null default no longer rewrites the table, which was a significant improvement for microservices teams.

SQL Server has similar behavior: adding a nullable column is metadata-only and instant, but adding a NOT NULL column without a default requires a table scan to validate or fill the values.

PostgreSQL:

-- Expand phase: add new column with a default, no table rewrite in PG 11+
ALTER TABLE orders ADD COLUMN customer_reference_id UUID DEFAULT NULL;

-- Create an index concurrently so the table stays available
CREATE INDEX CONCURRENTLY idx_orders_customer_reference_id
    ON orders (customer_reference_id)
    WHERE customer_reference_id IS NOT NULL;

SQL Server:

-- Expand phase: add new nullable column (metadata-only, instant)
ALTER TABLE orders ADD customer_reference_id UNIQUEIDENTIFIER NULL;

-- Create a non-blocking index (online)
CREATE INDEX idx_orders_customer_reference_id
    ON orders (customer_reference_id)
    WHERE customer_reference_id IS NOT NULL
    WITH (ONLINE = ON);

The CREATE INDEX CONCURRENTLY in PostgreSQL and WITH (ONLINE = ON) in SQL Server are both ways to build indexes without blocking reads and writes — essential when services are always running and a maintenance window is not an option.

Schema migration tools like Flyway and Liquibase are common in microservices pipelines. The DBA's role shifts from applying changes manually to reviewing migration scripts before they enter the CI/CD pipeline, enforcing that destructive changes follow the expand/contract pattern, and ensuring migrations are idempotent so they can be retried safely.

Data Consistency Across Services

When a single transaction could span a hundred lines of business logic in a monolith, the database's ACID guarantees kept things consistent. A payment either went through and the order was confirmed, or neither happened. In microservices, the payment service and the order service are separate. You cannot wrap a network call in a database transaction.

The theoretical foundation here is the CAP theorem and its practical successor, BASE (Basically Available, Soft state, Eventually consistent). Distributed systems must choose their consistency guarantees deliberately, and the DBA has to understand which data must be strongly consistent and which can tolerate a window of eventual consistency.

The *Saga pattern* is the most common approach to multi-service transactions. A saga is a sequence of local transactions, each within a single service's database, with compensating transactions defined for the failure case. If the order service creates an order and then the payment service fails to charge the customer, the saga orchestrator triggers a compensating transaction in the order service to cancel the order. No distributed lock is held between steps. Each step commits locally and immediately.

The *Transactional Outbox pattern* solves a specific and insidious problem: how do you guarantee that a database write and a message publication happen atomically? Suppose the order service inserts a row into the orders table and then publishes an order_created event to Kafka. If the application crashes between the insert and the publish, the order exists but the downstream services never hear about it. The outbox pattern solves this by inserting the event into an outbox table in the same local transaction as the business data write. A separate process (a relay or Debezium CDC connector) reads the outbox and publishes to the message broker. The event is only removed from the outbox once publication is confirmed.

PostgreSQL:

-- Transactional outbox: insert business data and event in one transaction
BEGIN;

INSERT INTO orders (id, customer_id, total_amount, status, created_at)
VALUES (gen_random_uuid(), $1, $2, 'pending', NOW());

INSERT INTO outbox_events (id, aggregate_type, aggregate_id, event_type, payload, created_at)
VALUES (
    gen_random_uuid(),
    'Order',
    $3,
    'order_created',
    $4::jsonb,
    NOW()
);

COMMIT;

SQL Server:

-- Transactional outbox: insert business data and event in one transaction
BEGIN TRANSACTION;

INSERT INTO orders (id, customer_id, total_amount, status, created_at)
VALUES (NEWID(), @customer_id, @total_amount, 'pending', GETUTCDATE());

INSERT INTO outbox_events (id, aggregate_type, aggregate_id, event_type, payload, created_at)
VALUES (
    NEWID(),
    'Order',
    @aggregate_id,
    'order_created',
    @payload,
    GETUTCDATE()
);

COMMIT TRANSACTION;

The outbox table needs to be polled or streamed efficiently. In PostgreSQL, logical replication and Debezium can stream outbox rows as change events without a polling loop, which reduces latency significantly. In SQL Server, SQL Server Change Data Capture (CDC) or the transactional replication publication mechanism can serve the same purpose.

One underappreciated concern for the DBA is outbox table growth. If the relay process falls behind or fails, the outbox fills up. If the relay runs ahead but the cleanup job is not working, the table grows indefinitely. Partitioning the outbox by creation date and dropping old partitions is a production-grade approach that keeps the table from becoming a maintenance liability.

Observability: Tracing Slow Queries Back to Services

In a monolith, a slow query is a slow query — you look at the slow query log, identify the statement, and tune it. In microservices, a slow query might be coming from any of a dozen services, triggered by any of dozens of code paths, under load patterns you can only observe through distributed tracing.

The first tool is application-level labeling. Both PostgreSQL and SQL Server support application-level connection properties that show up in monitoring views. Developers should set the application name on every connection to include the service name and ideally the service version. In PostgreSQL, this appears in pg_stat_activity.application_name. In SQL Server, it shows up in sys.dm_exec_sessions.program_name.

Beyond that, the standard practice in observability-mature teams is embedding trace context in SQL comments. When a service executes a query, it prepends a comment containing the distributed trace ID to the SQL text. Tools like sqlcommenter (supported by most ORMs) automate this. The comment ends up in slow query logs and in query text captured by the database's monitoring interfaces, letting you correlate a slow database query with the upstream HTTP request and service call graph that triggered it.

PostgreSQL:

-- Query with embedded trace context (automatically added by sqlcommenter or ORMs)
/* traceparent='00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01',
   service='order-service',version='1.4.2' */
SELECT o.id, o.total_amount, o.status
FROM orders o
WHERE o.customer_id = $1
  AND o.created_at > NOW() - INTERVAL '30 days'
ORDER BY o.created_at DESC;

SQL Server:

-- Query with embedded trace context
/* traceparent='00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01',
   service='order-service',version='1.4.2' */
SELECT o.id, o.total_amount, o.status
FROM orders o
WHERE o.customer_id = @customer_id
  AND o.created_at > DATEADD(DAY, -30, GETUTCDATE())
ORDER BY o.created_at DESC;

Auto-explain in PostgreSQL (via the auto_explain extension) can log the execution plan of any query that exceeds a configurable duration threshold, writing the output to the server log. Combined with trace context in comments, a DBA can look at a slow plan and know exactly which service version on which endpoint triggered it.

For SQL Server, Query Store is the equivalent workhorse. It captures query plans and runtime statistics persistently, enabling comparison of a query's plan before and after a deployment — which matters in microservices where deployments happen continuously. Configuring Query Store to use QUERY_CAPTURE_MODE = AUTO reduces noise by excluding infrequent, low-cost queries.

Distributed tracing platforms — Jaeger, Zipkin, Datadog APM, Azure Application Insights — aggregate spans across services and can show you that a particular API call spent 800 milliseconds waiting on a database query. The DBA's role in this ecosystem is to ensure the database side of that picture is interpretable: meaningful application names, query parameterization so similar queries are grouped together rather than flooding the log with literal-value variants, and alerting on query latency percentiles rather than just averages.

Running Databases in Containers and Kubernetes

Containerization and Kubernetes have become the default deployment target for microservices. Running a database in a container is operationally viable for development and test environments, and with care, for production workloads at certain scales. The DBA needs to understand both what is possible and where the sharp edges are.

The core concern with stateful containers is data persistence. A container is ephemeral by default — if it restarts, the filesystem is gone. Both PostgreSQL and SQL Server running in Kubernetes require a *Persistent Volume Claim* (PVC) backed by durable storage. The choice of storage class matters: network-attached storage (NAS or AWS EFS) introduces latency that heavily impacts write-intensive databases. Local NVMe-backed persistent volumes on the node give much better performance but come with the constraint that the pod must schedule on the specific node where the volume lives.

PostgreSQL is well-supported in Kubernetes through operators — CloudNativePG and Crunchy Data PGO are the most production-ready. These operators handle streaming replication setup, automated failover, connection pooling via PgBouncer sidecar, scheduled backups to object storage (S3, GCS), and rolling restarts that respect replication lag before proceeding. For a team running microservices on Kubernetes, a PostgreSQL operator removes the manual operational burden that would otherwise require deep DBA involvement in every cluster provisioning event.

SQL Server has the official Microsoft SQL Server Operator for Kubernetes (currently focused on SQL Server Big Data Clusters and Availability Groups on Kubernetes). The operational model is more constrained: SQL Server requires more memory headroom, the container images are larger, and features like Always On Availability Groups in Kubernetes require careful network configuration. Many organizations running SQL Server in a microservices context find that Azure SQL Database (PaaS) is more practical than self-managed SQL Server containers, because it offloads HA, backup, and patching while still accepting standard T-SQL connections.

Resource limits in Kubernetes are another area the DBA must influence. If a container's memory limit is set lower than PostgreSQL's shared_buffers plus working memory, the container will be OOM-killed under load. A DBA reviewing Kubernetes manifests for a database container should verify that shared_buffers is set to roughly 25% of the container's memory limit, that work_mem is not set so high that concurrent sort operations exhaust available memory, and that max_connections is configured in concert with the PgBouncer pool size.

PostgreSQL:

-- Verify current memory configuration in a containerized instance
SELECT
    name,
    setting,
    unit,
    context
FROM pg_settings
WHERE name IN (
    'shared_buffers',
    'work_mem',
    'max_connections',
    'effective_cache_size',
    'maintenance_work_mem'
)
ORDER BY name;

SQL Server:

-- Verify current memory configuration in a containerized SQL Server instance
SELECT
    name,
    value_in_use,
    description
FROM sys.configurations
WHERE name IN (
    'max server memory (MB)',
    'min server memory (MB)',
    'max degree of parallelism',
    'cost threshold for parallelism'
)
ORDER BY name;

Backup strategy in Kubernetes requires deliberate planning. The assumption that a DBA manually triggers pg_dump or SQL Server backups on a schedule does not hold when pods are ephemeral and the backup tooling needs to live alongside the data. Operators that schedule continuous WAL archiving to S3 (for PostgreSQL) or transaction log shipping to Azure Blob Storage (for SQL Server) give you point-in-time recovery without relying on a human being physically present to initiate the backup.

---

Key Takeaways

  • → The *database per service* principle gives microservices teams independent deployability but moves referential integrity and cross-domain queries out of the database and into the application and event layer — a tradeoff the DBA must actively manage.
  • → Connection pooling is not optional at microservices scale; PgBouncer for PostgreSQL and properly sized connection pools per SQL Server service account are the operational baseline for keeping connection counts sane under pod-based deployment.
  • → Schema changes in microservices must follow the *expand/contract* pattern to support zero-downtime deployments — add before you remove, and never assume a single deployment window will cover all running service instances simultaneously.
  • → The *Transactional Outbox pattern* is the reliable way to bridge a local database write with a downstream event publication without relying on distributed transactions that span service boundaries.
  • → Observability in a distributed system requires the DBA to invest in application-name labeling, trace context in SQL comments, and tools like PostgreSQL's `auto_explain` and SQL Server's Query Store to correlate slow queries back to the specific service, version, and request path that triggered them.
