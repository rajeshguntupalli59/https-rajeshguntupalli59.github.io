Chapter 4 of 40 The Cloud Database Engineer

Chapter 4: The Cloud DBA Role: What Changes, What Stays

If you have spent years tuning PostgreSQL shared_buffers, managing SQL Server AlwaysAvailability Groups, and writing runbooks for failover procedures at 2 AM, the shift to managed cloud databases can feel disorienting. AWS, Azure, and Google Cloud take over large portions of the operational work you once owned completely — OS patching, storage provisioning, replication wiring — while introducing entirely new categories of responsibility that did not exist in your on-premises world. This chapter maps that transition precisely. It identifies what the cloud genuinely removes from your plate, what it hands back to you in a different form, and what it adds as net-new work. Understanding this boundary is the foundation for everything else in this book.

---

4.1 The Managed Service Contract: What AWS, Azure, and GCP Actually Own

When you move from self-managed PostgreSQL to Amazon RDS, you are entering into an implicit contract. The cloud provider owns the hypervisor, the operating system, the storage subsystem, the replication mechanism, and the software patching lifecycle. You own the database configuration, the schema design, the query performance, the access control model, and the backup retention policy. The line between those two domains is firm, and understanding it prevents both false assumptions and wasted effort.

What the provider owns:

On RDS for PostgreSQL, you will never run apt-get upgrade postgresql, restart the OS, or access /var/lib/postgresql directly. AWS handles minor version patching on a schedule you influence but do not fully control unless you disable auto minor version upgrades explicitly. The storage layer is EBS-backed with automatic replication within an Availability Zone. Multi-AZ deployments add a synchronous standby in a second AZ, and failover is handled automatically — typically completing in 60–120 seconds for RDS, and 30 seconds or less for Aurora.

On Azure SQL Database, Microsoft manages the underlying SQL Server engine, the Always On availability group that provides HA, and the Intelligent Query Processing features that ship with each compatibility level upgrade. Google Cloud SQL similarly abstracts OS management and handles read replica replication lag monitoring internally.

What you still own:

The provider's ownership does not extend to logical database design, index strategy, connection pooling, application-layer retry logic, or query plans. A poorly written query that causes a full table scan on a 2 TB table will perform exactly as badly on RDS as it did on your on-premises server. Cloud-managed does not mean self-healing at the query level. Your expertise in execution plans, wait events, lock escalation, and statistics management transfers directly and remains essential.

The practical implication is this: you spend less time on infrastructure operations and more time on database architecture decisions, performance engineering, and cost optimization. That is a genuine shift in how a senior DBA allocates time, not a reduction in the need for DBA expertise.

4.2 Access Patterns and Permissions: From pg_hba.conf to IAM

On-premises PostgreSQL access control flows through two files: postgresql.conf controls what the engine does, and pg_hba.conf controls who can connect, from where, and with what authentication method. SQL Server uses logins, server roles, and Windows Authentication through Active Directory. In the cloud, these familiar mechanisms are extended — not replaced — by identity and access management layers that sit above the database engine.

IAM Authentication on RDS

With IAM database authentication enabled on RDS PostgreSQL, a user connects using a short-lived authentication token generated by the AWS IAM service rather than a static password. The token is valid for 15 minutes and is generated using the AWS credentials of the calling identity — an EC2 instance profile, a Lambda execution role, or a developer's IAM user. The database-side configuration still uses pg_hba.conf, but the method becomes md5 or scram-sha-256 is bypassed in favor of the IAM token validation path.

# Enable IAM authentication on an existing RDS PostgreSQL instance
aws rds modify-db-instance \
  --db-instance-identifier prod-postgres-01 \
  --enable-iam-database-authentication \
  --apply-immediately

# Generate an authentication token for a database user named "app_user"
aws rds generate-db-auth-token \
  --hostname prod-postgres-01.cxyz1234.us-east-1.rds.amazonaws.com \
  --port 5432 \
  --region us-east-1 \
  --username app_user

On the database side, the user must exist with the rds_iam role granted:

-- Create a database user that authenticates via IAM
CREATE USER app_user WITH LOGIN;
GRANT rds_iam TO app_user;

-- The user has no password — authentication is entirely token-based
-- Grant object-level privileges as you normally would
GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO app_user;

This pattern replaces the practice of embedding static database passwords in application configuration files. The calling application retrieves a fresh token before each connection attempt (or connection pool initialization), using the AWS SDK. The conceptual analog is Kerberos authentication with Active Directory — the application proves its identity to a central authority and receives a time-limited credential.

Azure SQL and Azure Active Directory

Azure SQL Database integrates with Azure Active Directory using contained database users. The model is familiar if you have used Windows Authentication with SQL Server, but the administration happens through the Azure portal, Azure CLI, or Entra ID (formerly Azure AD) rather than through local Windows group policy.

# Set an Azure AD admin for an Azure SQL logical server
az sql server ad-admin create \
  --resource-group prod-rg \
  --server prod-sql-server-01 \
  --display-name "DBA-Team-Admin" \
  --object-id "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"

Inside the database, Entra-backed contained users are created without passwords:

-- Create an Azure AD-backed user in Azure SQL Database
CREATE USER [dba-engineer@company.com] FROM EXTERNAL PROVIDER;
ALTER ROLE db_datareader ADD MEMBER [dba-engineer@company.com];

-- For a managed identity (e.g., an Azure App Service)
CREATE USER [my-app-service] FROM EXTERNAL PROVIDER;
GRANT SELECT, INSERT, UPDATE ON SCHEMA::dbo TO [my-app-service];

The Core Shift

What changes is not the concept of authentication and authorization — it is the locus of control. You now manage database access through the same IAM, RBAC, and identity systems that govern access to every other cloud resource. Permissions are auditable through CloudTrail (AWS), Azure Monitor Activity Log (Azure), and Cloud Audit Logs (GCP). Rotating credentials means updating IAM policies, not editing pg_hba.conf and running pg_reload_conf(). This is a genuine improvement in auditability and a reduction in credential sprawl, but it requires the DBA to become conversant in cloud IAM, which is a new skill domain.

4.3 Configuration Management: Parameter Groups, Server Parameters, and What You Cannot Touch

On self-managed PostgreSQL, you edit postgresql.conf directly and control every engine parameter from max_connections to wal_level to autovacuum_vacuum_cost_delay. On RDS, that file is owned by AWS. You configure the engine through Parameter Groups — a named collection of engine parameters that you apply to one or more DB instances.

RDS Parameter Groups

The default parameter group for any RDS engine version is managed by AWS and is not modifiable. You create a custom parameter group, copy defaults into it, modify what you need, and associate it with your instance. Changes to dynamic parameters take effect immediately; changes to static parameters require an instance reboot.

# Create a custom parameter group for RDS PostgreSQL 15
aws rds create-db-parameter-group \
  --db-parameter-group-name prod-pg15-custom \
  --db-parameter-group-family postgres15 \
  --description "Production PostgreSQL 15 tuning"

# Modify key parameters
aws rds modify-db-parameter-group \
  --db-parameter-group-name prod-pg15-custom \
  --parameters \
    "ParameterName=shared_buffers,ParameterValue={DBInstanceClassMemory/4},ApplyMethod=pending-reboot" \
    "ParameterName=work_mem,ParameterValue=65536,ApplyMethod=immediate" \
    "ParameterName=autovacuum_vacuum_scale_factor,ParameterValue=0.01,ApplyMethod=immediate" \
    "ParameterName=log_min_duration_statement,ParameterValue=1000,ApplyMethod=immediate"

# Associate with an instance
aws rds modify-db-instance \
  --db-instance-identifier prod-postgres-01 \
  --db-parameter-group-name prod-pg15-custom \
  --apply-immediately

Note the formula syntax for shared_buffers. RDS supports arithmetic expressions using {DBInstanceClassMemory} as a variable — this is how AWS recommends setting memory-relative parameters so the configuration remains valid if you resize the instance class. For a db.r6g.2xlarge with 64 GB RAM, {DBInstanceClassMemory/4} evaluates to 16 GB, which is 25% of RAM — the standard PostgreSQL recommendation.

What You Cannot Configure

This is where experienced DBAs encounter real friction. Several parameters are locked in RDS because they control behaviors that AWS manages on your behalf:

  • → `wal_level` is fixed at `logical` on RDS PostgreSQL — you cannot lower it to `replica`
  • → `archive_mode` is managed by AWS; you cannot disable it without understanding it breaks PITR
  • → `listen_addresses` is managed by AWS
  • → `ssl` is forced on — you cannot disable SSL on RDS connections
  • → `superuser` access does not exist on RDS; the `rds_superuser` role is a functional substitute but does not grant OS-level access
  • → Extensions must be whitelisted by AWS; you cannot install arbitrary shared libraries

    • The absence of a true superuser is a significant operational change. Tasks that previously required superuser — like terminating backend connections owned by other users, accessing pg_authid, or installing custom extensions — require either the rds_superuser role (which handles most cases) or a support ticket to AWS (which handles almost none of the rest gracefully). Plan your extension requirements before migration. If your application depends on an extension that is not in the RDS supported list, you may need AlloyDB on GCP, Aurora, or a self-managed EC2 deployment.

    Azure SQL Server-Level vs. Database-Level Parameters

    Azure SQL Database presents a different model. Many parameters that SQL Server DBAs manage at the server level — max server memory, MAXDOP, cost threshold for parallelism — are controlled through database-scoped configurations or are set by Microsoft at the service tier level.

    -- Azure SQL: set database-scoped configuration
ALTER DATABASE SCOPED CONFIGURATION SET MAXDOP = 4;
ALTER DATABASE SCOPED CONFIGURATION SET LEGACY_CARDINALITY_ESTIMATION = OFF;
ALTER DATABASE SCOPED CONFIGURATION SET QUERY_OPTIMIZER_HOTFIXES = ON;

-- Check current database-scoped configuration
SELECT name, value, value_for_secondary
FROM sys.database_scoped_configurations
ORDER BY name;

    Maximum memory is determined by the service tier and compute size — there is no sp_configure 'max server memory' you can run. On a General Purpose 8 vCore instance, SQL Server gets approximately 22 GB of buffer pool. On a Business Critical 8 vCore, it gets approximately 40 GB. These numbers are fixed for the tier and not tunable.

    4.4 High Availability and Failover: Understanding What the Cloud Gives You

    HA in on-premises environments is expensive to build and maintain. A PostgreSQL streaming replication setup with Patroni, HAProxy, and etcd requires significant operational investment. SQL Server AlwaysOn Availability Groups require Windows Server Failover Clustering, shared storage or synchronous replication, and careful listener configuration. In the cloud, HA is a checkbox during instance creation, and the complexity of maintaining it disappears — but understanding what you got is essential because the behavior is not identical to what you built yourself.

    RDS Multi-AZ: What It Is and Is Not

    RDS Multi-AZ deploys a synchronous standby in a second Availability Zone. Writes are committed to both the primary and standby before returning to the application. Failover is automatic and DNS-based — the endpoint DNS name resolves to the standby after promotion, typically within 60–120 seconds including DNS TTL propagation. The standby is not readable. It is a hot standby only for failover purposes.

    This surprises many DBAs coming from Patroni setups where the replica serves read traffic. On RDS Multi-AZ (non-Aurora), you pay for the standby instance in full and receive zero read scaling from it. Read scaling on RDS requires separate Read Replicas, which use asynchronous replication and carry replication lag that is your responsibility to monitor.

    Aurora: A Different Architecture

    Aurora changes this model substantially. Aurora separates compute from storage. All Aurora instances — primary and replicas — share the same distributed storage volume, which replicates data six ways across three Availability Zones. Aurora replicas apply redo log directly from the shared storage layer rather than receiving WAL over the network. This reduces replica lag dramatically — typically under 20 milliseconds in normal operation — and makes reads from Aurora replicas a reliable pattern for production read scaling.

    Aurora also has a faster failover model. Because replicas already have access to the same storage volume as the primary, a failover involves promoting a replica to writer status rather than waiting for WAL replay to complete. Aurora typically fails over in under 30 seconds for replicas in the same Region, and some configurations achieve under 15 seconds with cluster-aware drivers.

    # Create an Aurora PostgreSQL cluster with two replicas in different AZs
aws rds create-db-cluster \
  --db-cluster-identifier prod-aurora-cluster \
  --engine aurora-postgresql \
  --engine-version 15.4 \
  --master-username dbadmin \
  --master-user-password "$(aws secretsmanager get-secret-value \
      --secret-id prod/aurora/master-password \
      --query SecretString --output text)" \
  --db-subnet-group-name prod-subnet-group \
  --vpc-security-group-ids sg-0abc123def456 \
  --storage-encrypted \
  --kms-key-id alias/prod-rds-key \
  --backup-retention-period 14 \
  --preferred-backup-window "03:00-04:00" \
  --preferred-maintenance-window "sun:04:00-sun:05:00"

# Add a writer instance
aws rds create-db-instance \
  --db-cluster-identifier prod-aurora-cluster \
  --db-instance-identifier prod-aurora-writer \
  --db-instance-class db.r6g.2xlarge \
  --engine aurora-postgresql

# Add a reader instance in a different AZ
aws rds create-db-instance \
  --db-cluster-identifier prod-aurora-cluster \
  --db-instance-identifier prod-aurora-reader-1 \
  --db-instance-class db.r6g.xlarge \
  --engine aurora-postgresql \
  --availability-zone us-east-1b

    Azure SQL Business Critical vs. General Purpose

    Azure SQL HA behavior depends on the service tier. General Purpose uses remote storage (Azure Premium SSD) with a database availability SLA achieved through Azure infrastructure-level redundancy. Failover in the General Purpose tier can take 25–30 seconds as data files are remounted on a new compute node. Business Critical deploys an Always On availability group across three replicas using local SSD storage. One of those replicas is readable and accessible to your application — unlike RDS Multi-AZ standby, the Business Critical secondary serves read workloads without additional configuration.

    The read-scale secondary in Business Critical is accessed via a connection string that includes ApplicationIntent=ReadOnly:

    -- Connect string for Azure SQL Business Critical read-only replica
-- Server=prod-sql-server-01.database.windows.net;
-- Database=prod-db;
-- Authentication=Active Directory Default;
-- ApplicationIntent=ReadOnly;

-- Verify you are on the secondary
SELECT DATABASEPROPERTYEX(DB_NAME(), 'Updateability') AS replica_role;
-- Returns: READ_ONLY when connected to secondary
-- Returns: READ_WRITE when connected to primary

    The HA tiers, failover behaviors, and read-scale capabilities across cloud providers have direct cost implications. Choosing between RDS Multi-AZ, Aurora, Azure SQL General Purpose, and Business Critical is as much an architectural decision as a financial one, and it belongs in the DBA's scope, not just the infrastructure team's.

    4.5 Monitoring: CloudWatch, Azure Monitor, and Cloud Logging vs. What You Had Before

    On-premises DBAs typically own their full monitoring stack — Prometheus with pg_exporter, Datadog agents, custom scripts querying pg_stat_activity, pg_stat_bgwriter, and pg_locks. In the cloud, the provider delivers a baseline monitoring layer automatically, and the DBA's job shifts from building that layer to interpreting it, extending it, and setting appropriate thresholds.

    CloudWatch and Enhanced Monitoring on RDS

    Every RDS instance publishes a set of CloudWatch metrics automatically with no configuration required: CPUUtilization, DatabaseConnections, FreeableMemory, ReadIOPS, WriteIOPS, ReadLatency, WriteLatency, ReplicaLag. These metrics refresh at 60-second intervals by default.

    Enhanced Monitoring is an additional feature that installs an agent on the DB instance OS (which you cannot access directly) and publishes 50+ OS-level metrics at 1-second to 60-second granularity to CloudWatch Logs. These include per-process CPU, memory, disk I/O broken out by device, and file system metrics. Enable it for production instances — the cost is negligible compared to the diagnostic value during an incident.

    # Enable Enhanced Monitoring with 15-second granularity
aws rds modify-db-instance \
  --db-instance-identifier prod-postgres-01 \
  --monitoring-interval 15 \
  --monitoring-role-arn arn:aws:iam::123456789012:role/rds-monitoring-role \
  --apply-immediately

# Query CloudWatch for replica lag over the last hour
aws cloudwatch get-metric-statistics \
  --namespace AWS/RDS \
  --metric-name ReplicaLag \
  --dimensions Name=DBInstanceIdentifier,Value=prod-postgres-reader-1 \
  --start-time $(date -u -d '1 hour ago' +%Y-%m-%dT%H:%M:%SZ) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --period 60 \
  --statistics Maximum \
  --output table

    Performance Insights

    Performance Insights is the cloud analog to Oracle's Active Session History and a significant improvement over what most PostgreSQL on-premises deployments have out of the box. It records database load in terms of active sessions broken down by wait event, SQL statement, host, and user. The default retention is 7 days free; extending to 2 years costs approximately $0.02 per vCPU per hour.

    For a production workload investigation, Performance Insights replaces the manual practice of sampling pg_stat_activity every few seconds and correlating the results. The UI presents a visual timeline of database load against the max_vCPUs line — anything above that line indicates the database is CPU-bound or waiting on a bottleneck that requires investigation.

    # Retrieve Performance Insights data via boto3
import boto3
from datetime import datetime, timedelta, timezone

pi_client = boto3.client('pi', region_name='us-east-1')

response = pi_client.get_resource_metrics(
    ServiceType='RDS',
    Identifier='db-ABCDEFGHIJ1234567',  # DBI resource ID, not instance ID
    MetricQueries=[
        {
            'Metric': 'db.load.avg',
            'GroupBy': {
                'Group': 'db.wait_event',
                'Limit': 10
            }
        }
    ],
    StartTime=datetime.now(timezone.utc) - timedelta(hours=1),
    EndTime=datetime.now(timezone.utc),
    PeriodInSeconds=60
)

for result in response['MetricList']:
    print(f"Metric: {result['Key']}")
    for dp in result['DataPoints'][-5:]:  # last 5 data points
        print(f"  {dp['Timestamp']}: {dp.get('Value', 'N/A'):.4f}")

    Azure Monitor and Query Performance Insight

    Azure SQL Database surfaces equivalent functionality through Azure Monitor metrics (DTU percentage, CPU percentage, data I/O percentage, log write percentage, connection counts) and the Query Performance Insight blade, which aggregates data from Query Store — a feature that has been running automatically in Azure SQL since it was introduced in SQL Server 2016.

    Query Store is particularly valuable because it captures query plan changes over time. A query that performed well for three months and suddenly regressed after an auto-statistics update will show the plan change in Query Store with the before and after execution statistics.

    -- Azure SQL: find the top 10 queries by total CPU consumption in the last 24 hours
SELECT TOP 10
    q.query_id,
    qt.query_sql_text,
    SUM(rs.avg_cpu_time * rs.count_executions) AS total_cpu_ms,
    SUM(rs.count_executions) AS total_executions,
    SUM(rs.avg_duration * rs.count_executions) AS total_duration_ms
FROM sys.query_store_query_text qt
JOIN sys.query_store_query q ON qt.query_text_id = q.query_text_id
JOIN sys.query_store_plan p ON q.query_id = p.query_id
JOIN sys.query_store_runtime_stats rs ON p.plan_id = rs.plan_id
JOIN sys.query_store_runtime_stats_interval rsi 
    ON rs.runtime_stats_interval_id = rsi.runtime_stats_interval_id
WHERE rsi.start_time >= DATEADD(HOUR, -24, GETUTCDATE())
GROUP BY q.query_id, qt.query_sql_text
ORDER BY total_cpu_ms DESC;

    The monitoring shift is real: you spend less time building monitoring infrastructure and more time interpreting richer, already-collected data. The tradeoff is that cloud monitoring tools have gaps — they do not expose lock wait graphs as clearly as pg_locks queries do on self-managed PostgreSQL, and correlating application traces with database wait events across CloudWatch, X-Ray, and RDS Performance Insights requires more configuration than running a single monitoring agent that sees both layers.

    4.6 The New DBA Work: Cost Management, IaC, and Architecture Ownership

    The reduction in operational work does not leave a vacuum. Cloud DBAs acquire a meaningful set of responsibilities that did not exist in their on-premises roles: database infrastructure as code, cost engineering, and cross-service architecture design.

    Infrastructure as Code for Databases

    In on-premises environments, provisioning a PostgreSQL server involved coordinating with a server team, a storage team, and a network team, filling out a change request, and waiting. In the cloud, a DBA with appropriate IAM permissions can provision a multi-AZ Aurora cluster with encrypted storage, automated backups, and VPC-isolated networking in under 10 minutes using Terraform. The DBA who understands both the database requirements and the infrastructure configuration is in a stronger position than one who waits for an infrastructure team to provision resources.
