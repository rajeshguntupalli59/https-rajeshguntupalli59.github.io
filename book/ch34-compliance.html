Chapter 34: Compliance and Governance

Compliance and governance are not the most glamorous parts of database administration, but they are among the most consequential. A missed audit requirement or an improperly protected column of personally identifiable information can result in regulatory fines, legal liability, and reputational damage that no index tuning or query optimization can fix. This chapter covers what compliance and governance actually mean in the context of PostgreSQL and SQL Server, how major regulatory frameworks translate into database-level controls, and how to implement auditing, access control, data masking, and retention policies that satisfy both security teams and external auditors. By the end, you will have a working mental model for building a compliant database environment rather than scrambling to patch gaps when an audit arrives.

---

What Compliance Means for a Database Administrator

The word "compliance" covers a wide range of obligations depending on your industry and geography. HIPAA governs the handling of protected health information in the United States. PCI DSS applies to any system that stores, processes, or transmits payment card data. GDPR extends data protection obligations to any organization that handles the personal data of EU residents, regardless of where that organization is headquartered. SOC 2 is an auditing framework used primarily by technology companies to demonstrate controls around security, availability, and confidentiality. Each framework has its own vocabulary and emphasis, but from a DBA's perspective they share a common core set of requirements: know who has access to data, record what was done with it, protect sensitive values at rest and in transit, and be able to prove all of this to an auditor.

The DBA's role in compliance is not to become a lawyer or a compliance officer. It is to translate the requirements handed down from those roles into concrete database controls. When a compliance officer says "we need to know every time someone reads a Social Security Number," that maps to row-level auditing on a specific column. When legal says "GDPR requires us to delete all data about a user upon request," that maps to a data lifecycle management process with verifiable deletion logs.

Understanding the mapping between regulatory language and database features is what separates a DBA who is a genuine asset in a compliance program from one who waits to be told exactly what to click. The frameworks themselves do not mandate specific technology implementations. PCI DSS does not say "use column-level encryption in SQL Server." It says cardholder data must be protected, and it is your job to determine that column-level encryption satisfies that requirement in your environment.

One concept that runs through every compliance framework is the principle of least privilege. Users and applications should have access to exactly what they need to do their jobs, and nothing more. This principle has direct database implications: avoiding shared service accounts, granting SELECT instead of broad data access roles, and separating the DBA role from application roles. Compliance auditors will ask to see evidence that this principle is enforced, not just declared.

Auditing: Recording What Happens to Your Data

An audit trail is a chronological record of who accessed or modified data, when, and from where. In a compliance context, the absence of an audit trail is itself a finding. Auditors need to see evidence that you can detect unauthorized access, and detecting it requires having a record of all access.

PostgreSQL does not have a built-in comprehensive audit facility comparable to SQL Server's native options, but the pgaudit extension fills that gap cleanly. pgaudit integrates with PostgreSQL's standard logging infrastructure and allows you to log session-level activity (everything a session does) or object-level activity (specific operations on specific objects). For compliance purposes, object-level auditing is usually what you want, because session logging tends to produce enormous volumes of noise.

PostgreSQL:

-- Install pgaudit extension (requires it to be in shared_preload_libraries in postgresql.conf)
-- In postgresql.conf: shared_preload_libraries = 'pgaudit'

-- Enable object-level auditing after extension is created
CREATE EXTENSION IF NOT EXISTS pgaudit;

-- Configure pgaudit to log SELECT, INSERT, UPDATE, DELETE on sensitive tables
-- Set in postgresql.conf or via ALTER SYSTEM:
-- pgaudit.log = 'read, write'
-- pgaudit.log_relation = on

-- Grant audit role to track specific table access
-- pgaudit uses role-based object auditing:
CREATE ROLE audited_role;
GRANT SELECT ON patients TO audited_role;

-- Set the audit role
ALTER SYSTEM SET pgaudit.role = 'audited_role';
SELECT pg_reload_conf();

-- Any SELECT on patients by any user will now be logged
-- because audited_role has SELECT on it

SQL Server:

-- SQL Server has native auditing via SQL Server Audit
-- Create a server audit that writes to a file
CREATE SERVER AUDIT PatientDataAudit
TO FILE (
    FILEPATH = 'C:\SQLAudit\',
    MAXSIZE = 100 MB,
    MAX_ROLLOVER_FILES = 10,
    RESERVE_DISK_SPACE = OFF
)
WITH (
    QUEUE_DELAY = 1000,
    ON_FAILURE = CONTINUE
);

-- Enable the server audit
ALTER SERVER AUDIT PatientDataAudit WITH (STATE = ON);

-- Create a database audit specification targeting specific tables
USE HealthcareDB;
GO
CREATE DATABASE AUDIT SPECIFICATION AuditSensitiveAccess
FOR SERVER AUDIT PatientDataAudit
ADD (SELECT, INSERT, UPDATE, DELETE
     ON dbo.Patients BY public),
ADD (SELECT ON dbo.SSNData BY public)
WITH (STATE = ON);

With SQL Server Audit, you can also write to the Windows Security Event Log or the Application Event Log, which is useful when your organization aggregates logs into a SIEM like Splunk or Microsoft Sentinel. Writing audit data to files on the database server itself is convenient, but it creates a risk: a compromised database administrator could potentially alter or delete those files. For high-assurance environments, audit data should flow to a system that the DBA does not have write access to.

SQL Server also provides the sys.fn_get_audit_file function for querying audit logs programmatically, which lets you build dashboards or feed audit events into monitoring systems without exporting raw log files.

SQL Server:

-- Query audit log files to find all SELECT operations on Patients
SELECT
    event_time,
    server_principal_name,
    database_name,
    object_name,
    statement,
    succeeded,
    client_ip
FROM sys.fn_get_audit_file(
    'C:\SQLAudit\PatientDataAudit*.sqlaudit',
    DEFAULT,
    DEFAULT
)
WHERE object_name = 'Patients'
  AND action_id = 'SL'  -- SL = SELECT
ORDER BY event_time DESC;

Beyond extension-based or native auditing, both databases support trigger-based audit tables — a traditional approach where INSERT, UPDATE, and DELETE operations on a table fire a trigger that writes to an audit log table. Triggers are flexible and require no additional tooling, but they have real costs: every write operation incurs the overhead of the trigger execution, and if the audit table is in the same database, a transaction that rolls back will also roll back the audit entry, which means you can lose the record of an attempted change. Using autonomous transactions (available in PostgreSQL through dblink to a separate connection, or through a foreign data wrapper) can address the rollback problem at the cost of additional complexity.

For GDPR specifically, auditing needs to cover not just who modified data but also who read it, because access to personal data without a legitimate purpose is itself a potential violation. That means READ auditing, not just write auditing, which increases log volume substantially. When implementing read auditing for GDPR, be selective: focus on tables that actually contain personal data, log the user and timestamp, and ensure the audit logs themselves are protected from unauthorized access.

Access Control and Role-Based Security

Access control is the practical expression of the least privilege principle. Every user account, service account, and application connection that touches your database should have exactly the permissions needed and nothing more. This is not just good security hygiene; it is a documented requirement in PCI DSS (Requirement 7), HIPAA (§164.312(a)(1)), and GDPR (Article 25, data protection by design).

PostgreSQL's permission model is built around roles. Roles can be users (they can log in) or groups (they cannot log in directly but can be granted to other roles). This makes it straightforward to build a layered permission model where you define roles by job function and assign those roles to users.

PostgreSQL:

-- Create functional roles that represent job functions
CREATE ROLE app_readonly;
CREATE ROLE app_readwrite;
CREATE ROLE reporting_user;
CREATE ROLE pii_access;  -- special role for columns containing PII

-- Grant appropriate permissions to each role
GRANT CONNECT ON DATABASE healthcare TO app_readonly, app_readwrite, reporting_user;
GRANT USAGE ON SCHEMA public TO app_readonly, app_readwrite, reporting_user;

GRANT SELECT ON ALL TABLES IN SCHEMA public TO app_readonly;
GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO app_readwrite;
GRANT SELECT ON ALL TABLES IN SCHEMA reporting TO reporting_user;

-- Assign roles to actual user accounts
CREATE USER app_service_account WITH PASSWORD 'strong_password_here' NOINHERIT;
GRANT app_readwrite TO app_service_account;

CREATE USER analyst_jane WITH PASSWORD 'strong_password_here';
GRANT reporting_user TO analyst_jane;

-- Ensure future tables also get the right permissions
ALTER DEFAULT PRIVILEGES IN SCHEMA public
    GRANT SELECT ON TABLES TO app_readonly;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
    GRANT SELECT, INSERT, UPDATE ON TABLES TO app_readwrite;

SQL Server:

-- Create database roles by job function
USE HealthcareDB;
GO

CREATE ROLE AppReadOnly;
CREATE ROLE AppReadWrite;
CREATE ROLE ReportingUsers;
CREATE ROLE PIIAccess;

-- Assign permissions to roles
GRANT SELECT ON SCHEMA::dbo TO AppReadOnly;
GRANT SELECT, INSERT, UPDATE ON SCHEMA::dbo TO AppReadWrite;
GRANT SELECT ON SCHEMA::reporting TO ReportingUsers;

-- Create logins and users, then assign roles
CREATE LOGIN app_service WITH PASSWORD = 'StrongPassword123!';
CREATE USER app_service FOR LOGIN app_service;
ALTER ROLE AppReadWrite ADD MEMBER app_service;

CREATE LOGIN analyst_jane WITH PASSWORD = 'StrongPassword456!';
CREATE USER analyst_jane FOR LOGIN analyst_jane;
ALTER ROLE ReportingUsers ADD MEMBER analyst_jane;

-- Deny direct table access to force role-based access
DENY SELECT ON dbo.SSNData TO AppReadOnly;
GRANT SELECT ON dbo.SSNData TO PIIAccess;

One common compliance failure is the accumulation of permissions over time. An analyst who needed temporary access to a production table six months ago still has that access because no one revoked it. This is called privilege creep, and it is one of the most common findings in access control audits. The DBA's job is not just to grant access correctly but to have a process for reviewing and revoking it.

A quarterly access review process should query the database for all users and their effective permissions and compare that list to an authoritative source like an HR system or an access request ticketing system. Any account in the database that is not in the authoritative source should be flagged for review.

PostgreSQL:

-- Query effective permissions for all non-system users
SELECT
    r.rolname AS username,
    r.rolsuper AS is_superuser,
    r.rolcreatedb AS can_create_db,
    r.rolcreaterole AS can_create_roles,
    r.rolcanlogin AS can_login,
    array_to_string(
        ARRAY(
            SELECT b.rolname
            FROM pg_catalog.pg_auth_members m
            JOIN pg_catalog.pg_roles b ON m.roleid = b.oid
            WHERE m.member = r.oid
        ),
        ', '
    ) AS member_of_roles
FROM pg_catalog.pg_roles r
WHERE r.rolcanlogin = true
  AND r.rolname NOT LIKE 'pg_%'
ORDER BY r.rolname;

SQL Server:

-- Query all database users and their role memberships
SELECT
    dp.name AS username,
    dp.type_desc AS principal_type,
    dp.create_date,
    dp.modify_date,
    dp.default_schema_name,
    STRING_AGG(rp.name, ', ') WITHIN GROUP (ORDER BY rp.name) AS role_memberships
FROM sys.database_principals dp
LEFT JOIN sys.database_role_members drm ON dp.principal_id = drm.member_principal_id
LEFT JOIN sys.database_principals rp ON drm.role_principal_id = rp.principal_id
WHERE dp.type IN ('S', 'U', 'G')  -- SQL users, Windows users, Windows groups
  AND dp.name NOT IN ('dbo', 'guest', 'INFORMATION_SCHEMA', 'sys')
GROUP BY dp.name, dp.type_desc, dp.create_date, dp.modify_date, dp.default_schema_name
ORDER BY dp.name;

Service accounts deserve special attention. Application service accounts often end up with broader permissions than they need because it was easier during development to grant db_owner than to figure out the exact minimum permissions required. In a production compliance environment, this is unacceptable. Every service account should have its permissions documented, and those permissions should be the minimum necessary. If an application only needs to read from two tables and write to one, it should have exactly SELECT on those two tables and INSERT/UPDATE on the third, and nothing else.

Protecting Sensitive Data: Encryption, Masking, and Tokenization

Access control determines who can see data. Encryption and masking determine what they see when they do. These are complementary controls. A user with SELECT permission on a table containing credit card numbers should still see only masked values unless they have a specific additional authorization.

Encryption at rest protects data if physical media is stolen or a backup file is accessed without authorization. Both PostgreSQL and SQL Server support transparent data encryption (TDE) at the storage level. SQL Server's TDE is a native enterprise feature available in Standard and Enterprise editions. PostgreSQL does not have built-in TDE, but file system-level encryption (like Linux's LUKS or dm-crypt) achieves the same protection for the data files. Some PostgreSQL distributions, including EDB's Advanced Server and certain cloud offerings, do provide TDE at the cluster level.

Column-level encryption protects specific sensitive values even from database users who can access the table. In PostgreSQL, the pgcrypto extension provides encryption and decryption functions. In SQL Server, Always Encrypted shifts encryption key management to the client application so that even the DBA cannot read plaintext values.

PostgreSQL:

-- Using pgcrypto for column-level encryption
CREATE EXTENSION IF NOT EXISTS pgcrypto;

-- Create a table with an encrypted column
CREATE TABLE patient_ssn (
    patient_id    UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    patient_name  TEXT NOT NULL,
    ssn_encrypted BYTEA  -- store encrypted value as bytes
);

-- Insert with encryption (key should come from a secure vault, not hardcoded)
INSERT INTO patient_ssn (patient_name, ssn_encrypted)
VALUES (
    'Jane Smith',
    pgp_sym_encrypt('123-45-6789', current_setting('app.encryption_key'))
);

-- Read with decryption (only applications with the key can decrypt)
SELECT
    patient_id,
    patient_name,
    pgp_sym_decrypt(ssn_encrypted, current_setting('app.encryption_key')) AS ssn
FROM patient_ssn
WHERE patient_id = '...';

SQL Server:

-- SQL Server Always Encrypted (column master key stored outside SQL Server)
-- First, create the column master key and column encryption key in SSMS or via PowerShell
-- Then define the table with encrypted columns

CREATE TABLE dbo.PatientSSN (
    PatientID    UNIQUEIDENTIFIER DEFAULT NEWID() PRIMARY KEY,
    PatientName  NVARCHAR(200) NOT NULL,
    SSN          NVARCHAR(11)
        ENCRYPTED WITH (
            ENCRYPTION_TYPE = DETERMINISTIC,
            ALGORITHM = 'AEAD_AES_256_CBC_HMAC_SHA_256',
            COLUMN_ENCRYPTION_KEY = SSNEncryptionKey
        ) NOT NULL
);
-- With Always Encrypted, the SQL Server engine never sees plaintext SSN values
-- Encryption and decryption happen in the client driver
-- DBAs with full SQL Server access still cannot read SSN values

Data masking is a different layer: it presents substitute values to unauthorized users while showing real values to authorized ones. SQL Server has built-in Dynamic Data Masking (DDM), which is a policy-based feature that applies masking rules to columns. It is worth noting that DDM does not encrypt data — it only transforms the output seen by certain users. A user with sufficient permissions can bypass masking, so it should not be relied upon as the sole protection mechanism.

SQL Server:

-- Add dynamic data masking to existing columns
ALTER TABLE dbo.Customers
ALTER COLUMN CreditCardNumber
    ADD MASKED WITH (FUNCTION = 'partial(0,"XXXX-XXXX-XXXX-",4)');

ALTER TABLE dbo.Customers
ALTER COLUMN EmailAddress
    ADD MASKED WITH (FUNCTION = 'email()');

ALTER TABLE dbo.Customers
ALTER COLUMN PhoneNumber
    ADD MASKED WITH (FUNCTION = 'default()');

-- Grant UNMASK permission only to privileged roles
GRANT UNMASK ON dbo.Customers TO PIIAccess;

-- Verify masking is working for a low-privilege user
EXECUTE AS USER = 'analyst_jane';
SELECT CreditCardNumber, EmailAddress, PhoneNumber FROM dbo.Customers;
REVERT;

PostgreSQL does not have a built-in equivalent to DDM, but masking can be implemented through views that apply masking logic based on the current user or a session-level setting. The anon (PostgreSQL Anonymizer) extension provides a more complete masking framework with declarative masking rules similar to DDM.

Tokenization is a technique commonly used for PCI DSS compliance where the actual sensitive value (like a PANs — primary account number) is stored in a separate, highly secured token vault and replaced in the application database with a randomly generated token that has no mathematical relationship to the original value. From a database architecture perspective, this means your primary application database never holds actual cardholder data. The token is meaningless to an attacker who compromises the application database. Implementing tokenization typically involves an external tokenization service or vault (like HashiCorp Vault with a transit secrets engine), and the DBA's role is to ensure the application database stores only tokens and that the token vault's database is treated as a separate, higher-security tier.

Data Retention, Deletion, and the Right to Be Forgotten

Data governance is not only about protecting data while it exists — it is also about managing its lifecycle from creation through deletion. Regulatory frameworks have specific requirements on both ends: HIPAA requires that certain records be retained for at least six years; GDPR requires that personal data not be kept longer than necessary and that individuals have the right to request deletion. These requirements can pull in opposite directions, and the DBA's job is to implement a system that satisfies both.

Retention policies typically involve archiving data that is past its active use period but must still be kept for legal reasons, and permanently deleting data that has exceeded its required retention period. In PostgreSQL, declarative partitioning by date makes retention management cleaner: a partition covering a specific month or year can be dropped with a single DROP TABLE command on the partition, which is far faster and less disruptive than a DELETE statement against millions of rows.

PostgreSQL:

-- Create a partitioned audit log table with monthly partitions
CREATE TABLE audit_log (
    log_id      BIGSERIAL,
    event_time  TIMESTAMPTZ NOT NULL,
    user_name   TEXT,
    action      TEXT,
    table_name  TEXT,
    row_id      TEXT
) PARTITION BY RANGE (event_time);

-- Create partitions for recent months
CREATE TABLE audit_log_2024_11 PARTITION OF audit_log
    FOR VALUES FROM ('2024-11-01') TO ('2024-12-01');

CREATE TABLE audit_log_2024_12 PARTITION OF audit_log
    FOR VALUES FROM ('2024-12-01') TO ('2025-01-01');

-- When a partition exceeds retention period, drop it atomically
-- This is instantaneous compared to DELETE on millions of rows
DROP TABLE audit_log_2022_01;  -- removes all data from January 2022

SQL Server:

-- SQL Server partition scheme for time-based retention
-- First create a partition function
CREATE PARTITION FUNCTION AuditLogPF (DATETIME2)
AS RANGE RIGHT FOR VALUES (
    '2023-01-01', '2023-07-01',
    '2024-01-01', '2024-07-01',
    '2025-01-01'
);

-- Create partition scheme mapping to filegroups
CREATE PARTITION SCHEME AuditLogPS
AS PARTITION AuditLogPF
ALL TO ([PRIMARY]);  -- In production, use separate filegroups

-- Create partitioned table
CREATE TABLE dbo.AuditLog (
    LogID      BIGINT IDENTITY(1,1),
    EventTime  DATETIME2 NOT NULL,
    UserName   NVARCHAR(128),
    Action     NVARCHAR(50),
    TableName  NVARCHAR(128),
    RowID      NVARCHAR(200)
) ON AuditLogPS(EventTime);

-- To remove old partitions, use SWITCH to move data to a staging table
-- then truncate the staging table (avoids logging overhead of DELETE)
CREATE TABLE dbo.AuditLog_Archive (
    LogID      BIGINT,
    EventTime  DATETIME2 NOT NULL,
    UserName   NVARCHAR(128),
    Action     NVARCHAR(50),
    TableName  NVARCHAR(128),
    RowID      NVARCHAR(200)
) ON [PRIMARY];

ALTER TABLE dbo.AuditLog
SWITCH PARTITION 1 TO dbo.AuditLog_Archive;

TRUNCATE TABLE dbo.AuditLog_Archive;

The right to be forgotten under GDPR is technically called the right to erasure. When a user submits a deletion request, you must be able to delete or irreversibly anonymize all personal data about that individual across all tables, and you must be able to prove you did it. The complication is that "all tables" includes backups and archive stores, not just the live database. This is an area where many organizations struggle: their backup strategy was designed purely for recovery, with no thought given to targeted deletion of specific records from backup media.

There are two common approaches. The first is to not store actual personal data in backups — instead store cryptographic tokens, and when a deletion request comes in, delete the encryption key associated with that user's data. The backed-up data remains, but it is permanently unreadable without the key. The second approach is to document that backup data is excluded from right-to-erasure scope based on a legal basis (usually a "legitimate interest" or "legal obligation" basis), combined with a defined backup retention window so that the data will be purged automatically when backups age out. Either approach requires deliberate architectural decisions made in advance, not after a deletion request arrives.

A practical deletion process for GDPR starts with a subject access request catalog — a mapping of every table and column that contains personal data, indexed by how to find records for a specific individual. Without this catalog, responding to a deletion request becomes an ad-hoc investigation into your own schema. Building this catalog during schema design rather than after the fact is significantly easier.

Governance Infrastructure: Policies, Change Control, and Evidence

Technical controls alone do not constitute a compliant environment. Auditors reviewing a HIPAA or PCI DSS program want to see documented policies, change management processes, evidence of periodic reviews, and incident response procedures. The DBA's role extends into providing the evidence and supporting the processes that make governance real.

Change control for database objects is an area where development practices and compliance requirements overlap directly. Every change to a production database — schema changes, stored procedure modifications, permission grants, configuration changes — should go through a documented process with an approver, a tested rollback plan, and a record of who made the change and when. This is both good operational practice and a compliance requirement (PCI DSS Requirement 6, for example, requires change management for all system components).

Version-controlled database migrations using tools like Flyway, Liquibase, or for SQL Server specifically, Redgate SQL Change Automation, provide the documented change history that auditors want to see.
