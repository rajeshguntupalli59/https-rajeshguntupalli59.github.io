Chapter 9: Access Control

Access control is the mechanism by which a database system determines who can connect, what objects they can see, and what operations they are permitted to perform. It sounds simple in principle — give users what they need, deny them what they don't — but in practice it becomes one of the most operationally complex responsibilities a DBA carries. Misconfigured permissions are among the most common causes of both accidental data loss and deliberate data breaches. This chapter walks through the complete access control model in PostgreSQL and SQL Server: how authentication and authorization work, how roles and privileges are structured, how to lock down a production environment, and how to audit who has access to what. By the end, you will have the mental model and the SQL vocabulary to design and enforce a security posture that holds up under real scrutiny.

---

Authentication vs. Authorization: Two Different Problems

These two concepts are often lumped together under the word "security," but they are distinct and handled differently in both systems.

Authentication answers the question: *Is this person who they claim to be?* Before a user can interact with a database at all, the system needs to verify their identity. In PostgreSQL, authentication is controlled entirely outside of SQL, through the pg_hba.conf file (Host-Based Authentication). Each line in that file matches a combination of connection type, database name, username, client address, and authentication method. Common methods include md5, scram-sha-256, peer, ldap, and cert. The order of rules matters — PostgreSQL reads the file top to bottom and applies the first matching rule.

In SQL Server, authentication happens at two layers. The first is the login, which is a server-level principal that grants access to the SQL Server instance. Logins can be SQL Server logins (username and password stored in the instance) or Windows logins (authenticated by Active Directory). The second layer is the user, which is a database-level principal mapped to a login. A login can exist without a corresponding user in a database, and in that case the login can connect to the instance but cannot access that database.

Authorization answers the question: *What is this authenticated identity allowed to do?* This is where GRANT, REVOKE, DENY, roles, and schemas come into play — the entire subject of permissions management. Both systems separate authentication from authorization, but the design philosophy differs: PostgreSQL uses a unified role model that merges the concepts of users and groups, while SQL Server maintains a strict separation between logins (server-level) and users (database-level).

Understanding this distinction prevents a class of common mistakes. A DBA who revokes all privileges from a user but leaves their login intact has not actually removed access — the user can still connect. Conversely, dropping a PostgreSQL role that still owns objects or has active dependencies will fail entirely until those dependencies are resolved.

---

The Role Model in PostgreSQL and the Login/User Model in SQL Server

Both systems use the concept of a *role* as the primary unit of identity and permission, but the implementation differs enough to cause confusion when you work across both platforms.

PostgreSQL treats everything as a role. There is no separate object type for "user" versus "group." When you run CREATE USER, PostgreSQL internally runs CREATE ROLE ... WITH LOGIN — the LOGIN attribute is what distinguishes an interactive user from a group role. Roles can be members of other roles, creating a hierarchy that allows permissions to be inherited.

PostgreSQL:

-- Create a group role (no LOGIN = cannot connect directly)
CREATE ROLE reporting_team;

-- Create a user role (WITH LOGIN = can connect)
CREATE ROLE alice WITH LOGIN PASSWORD 'securepassword123';

-- Add alice to the reporting_team group
GRANT reporting_team TO alice;

-- Verify role membership
SELECT r.rolname AS role,
       m.rolname AS member
FROM pg_auth_members am
JOIN pg_roles r ON r.oid = am.roleid
JOIN pg_roles m ON m.oid = am.member
WHERE r.rolname = 'reporting_team';

SQL Server:

-- Create a login at the server level
CREATE LOGIN alice WITH PASSWORD = 'SecurePassword123!';

-- Create a corresponding user in a specific database
USE SalesDB;
CREATE USER alice FOR LOGIN alice;

-- Create a database role (equivalent to a group)
CREATE ROLE reporting_team;

-- Add the user to the role
ALTER ROLE reporting_team ADD MEMBER alice;

-- Verify role membership
SELECT dp.name AS role,
       mp.name AS member
FROM sys.database_role_members drm
JOIN sys.database_principals dp ON dp.principal_id = drm.role_principal_id
JOIN sys.database_principals mp ON mp.principal_id = drm.member_principal_id
WHERE dp.name = 'reporting_team';

In PostgreSQL, the rolsuper, rolcreaterole, rolcreatedb, rolreplication, and rolbypassrls attributes on a role control powerful system-level capabilities. These should be treated like production keys — only granted to accounts that genuinely require them, and ideally to service accounts with no interactive login rather than to personal accounts. The superuser attribute in particular bypasses every permission check in the system, including row-level security.

In SQL Server, the equivalent of a PostgreSQL superuser is membership in the sysadmin server role. Members of sysadmin can do anything to the instance. At the database level, the db_owner fixed database role grants complete control over a single database. These fixed roles — db_datareader, db_datawriter, db_owner, and others — are convenient but coarse. For anything beyond a development environment, custom roles with explicit grants are the right approach.

One SQL Server behavior that surprises many DBAs coming from PostgreSQL is the guest user. Every SQL Server database has an implicit guest account. If guest is enabled (it is by default in some older configurations), any authenticated login that doesn't have an explicit user mapping in a database can still connect to that database through the guest account and receive whatever permissions guest holds. Unless you have a documented reason to use it, guest should be disabled in every user database by revoking its CONNECT permission.

SQL Server:

-- Disable the guest user in a database (recommended in production)
USE SalesDB;
REVOKE CONNECT FROM guest;

-- Verify guest connection permission
SELECT name, type_desc, authentication_type_desc,
       HAS_PERMS_BY_NAME('SalesDB', 'DATABASE', 'CONNECT') AS can_connect
FROM sys.database_principals
WHERE name = 'guest';

Granting, Revoking, and the Principle of Least Privilege

The principle of least privilege states that every identity should have exactly the permissions needed to do its job — no more. This sounds obvious, but it requires discipline to maintain in production environments that evolve over time. Schemas change, new tables get added, and permissions granted in urgency during an incident often never get cleaned up.

In PostgreSQL, the privilege system operates at several levels: database, schema, table, column, sequence, function, and more. The two commands you use most are GRANT and REVOKE. PostgreSQL also has a concept of default privileges, which allows you to define what permissions should automatically be applied to new objects created in the future — this is something many DBAs overlook and then wonder why a new table isn't accessible.

PostgreSQL:

-- Grant read-only access to a schema
GRANT USAGE ON SCHEMA sales TO reporting_team;
GRANT SELECT ON ALL TABLES IN SCHEMA sales TO reporting_team;

-- Grant access to future tables created in the schema
ALTER DEFAULT PRIVILEGES IN SCHEMA sales
    GRANT SELECT ON TABLES TO reporting_team;

-- Grant write access on specific tables only
GRANT INSERT, UPDATE, DELETE ON sales.orders TO app_writer;

-- Revoke an overly broad grant
REVOKE ALL PRIVILEGES ON sales.customer_pii FROM app_writer;

SQL Server:

-- Grant read-only access to a schema
GRANT SELECT ON SCHEMA::sales TO reporting_team;

-- Grant write access on specific tables only
GRANT INSERT, UPDATE, DELETE ON sales.orders TO app_writer;

-- Deny access to a sensitive table (DENY overrides GRANT)
DENY SELECT ON sales.customer_pii TO app_writer;

-- Revoke a previously granted permission
REVOKE SELECT ON sales.orders FROM reporting_team;

One of the sharpest differences between the two systems here is SQL Server's DENY command. In SQL Server, DENY explicitly prohibits an action and takes precedence over any GRANT — even grants inherited through role membership. If alice is a member of reporting_team which has SELECT on sales.orders, but alice has been individually DENY-ed SELECT on that table, she cannot read it. In PostgreSQL, there is no equivalent DENY. The model is purely additive: permissions accumulate through direct grants and role memberships, and if you want to restrict access you revoke grants rather than denying them.

This difference has a practical consequence. In a PostgreSQL multi-tenant application where different users need access to different row subsets of the same table, row-level security (covered in the next section) is usually the right tool. In SQL Server, you might use DENY at the table level and then GRANT through carefully scoped views or procedures to achieve granular control.

Schema ownership matters in both systems but is particularly important in SQL Server. In SQL Server, a schema is both a namespace and a security boundary. The schema owner has implicit control over all objects within it. When an application user is also the schema owner, it often ends up with more privileges than intended — a consequence of convenience-driven setup. Best practice is to use a service account as the schema owner and grant explicit permissions to application and reporting roles separately.

In PostgreSQL, the public schema deserves special attention. By default, all users have the USAGE and CREATE privileges on the public schema, and all users have CONNECT privilege on all databases. If you create a table in public, every authenticated user can see it unless you explicitly lock it down. For PostgreSQL 14 and earlier, this means that at database setup time you should immediately run:

-- Revoke public schema defaults (run as superuser at cluster setup time)
REVOKE CREATE ON SCHEMA public FROM PUBLIC;
REVOKE ALL ON DATABASE mydb FROM PUBLIC;

PostgreSQL 15 changed this behavior — CREATE on the public schema is no longer granted to PUBLIC by default — but if you're running an older version or inheriting a cluster, always verify these settings.

Row-Level Security and Column-Level Permissions

Standard table-level permissions control whether a user can touch a table at all. But in many real applications, the question isn't whether a user can read *orders* — it's whether a user can read *their own* orders, or the orders assigned to their region. That's where row-level security (RLS) and column-level permissions come in.

PostgreSQL Row-Level Security is implemented through policies. You enable RLS on a table and then define one or more policies that determine which rows are visible or modifiable for a given role. The policies are applied transparently — the application doesn't need to add any WHERE clause; the database enforces the filter automatically.

PostgreSQL:

-- Enable RLS on the orders table
ALTER TABLE sales.orders ENABLE ROW LEVEL SECURITY;

-- Force RLS to apply even to the table owner
ALTER TABLE sales.orders FORCE ROW LEVEL SECURITY;

-- Policy: users can only see orders assigned to their username
CREATE POLICY user_sees_own_orders
    ON sales.orders
    FOR SELECT
    USING (assigned_rep = current_user);

-- Policy: managers can see everything (check pg_has_role for group membership)
CREATE POLICY manager_sees_all
    ON sales.orders
    FOR ALL
    USING (pg_has_role(current_user, 'sales_managers', 'MEMBER'));

A subtlety worth knowing: USING controls which rows are visible for reads (SELECT) and which rows can be modified (the WHERE condition in UPDATE/DELETE). WITH CHECK controls which rows can be inserted or updated *into* — it validates the row after the change, not before. If you create an RLS policy without a WITH CHECK clause on an UPDATE or INSERT policy, PostgreSQL defaults to using the USING expression for that check too. Most production policies should define both.

Superusers and roles with BYPASSRLS bypass all row-level security policies. If you're using a superuser as your application database connection (a common but unfortunate pattern), RLS provides no protection at all.

SQL Server Row-Level Security is implemented through inline table-valued functions and security policies that wrap them. The mechanism is more verbose but conceptually similar.

SQL Server:

-- Create the filter function
CREATE FUNCTION sales.fn_order_security_predicate
    (@assigned_rep NVARCHAR(128))
RETURNS TABLE
WITH SCHEMABINDING
AS
RETURN
    SELECT 1 AS result
    WHERE @assigned_rep = USER_NAME()
       OR IS_MEMBER('sales_managers') = 1;
GO

-- Create the security policy using the function
CREATE SECURITY POLICY sales.OrderAccessPolicy
    ADD FILTER PREDICATE sales.fn_order_security_predicate(assigned_rep)
    ON sales.orders
    WITH (STATE = ON);
GO

For column-level permissions, both systems let you restrict access to individual columns rather than entire tables. This is useful for sensitive fields like Social Security numbers, salaries, or PII that only certain roles should see.

PostgreSQL:

-- Grant SELECT only on specific columns
GRANT SELECT (order_id, order_date, status) ON sales.orders TO reporting_team;

-- The role cannot select salary or pii columns
-- Query to show column-level privileges
SELECT grantee, table_schema, table_name, column_name, privilege_type
FROM information_schema.column_privileges
WHERE table_schema = 'sales'
  AND table_name = 'orders';

SQL Server:

-- Grant column-level SELECT on non-sensitive columns
GRANT SELECT ON sales.orders (order_id, order_date, status) TO reporting_team;

-- Deny access to sensitive column specifically
DENY SELECT ON sales.orders (ssn, salary) TO reporting_team;

-- Verify column permissions
SELECT class_desc, major_id, minor_id, grantee_principal_id,
       permission_name, state_desc
FROM sys.database_permissions
WHERE class = 1 -- object-level
  AND permission_name = 'SELECT';

An important operational note: column-level grants interact with SELECT * in sometimes unintuitive ways. In SQL Server, if a user has SELECT on only specific columns and executes SELECT *, the query fails entirely rather than returning only the permitted columns. Applications should always specify columns explicitly — this is good practice regardless of security constraints — but it becomes a hard requirement when column-level permissions are in play.

Auditing Access: Who Has What, and What Happened

Permissions drift over time. A developer gets temporary access during an incident, the incident resolves, and the access never gets revoked. A role accumulates grants from three different DBAs over two years. An application user turns out to have db_owner because someone didn't want to figure out the exact rights needed. Auditing is the only systematic way to catch this drift before it becomes a problem.

Querying the current permission state is the first step. Both systems expose this through catalog views.

PostgreSQL:

-- Show all table-level privileges in a schema
SELECT grantee,
       table_schema,
       table_name,
       string_agg(privilege_type, ', ' ORDER BY privilege_type) AS privileges
FROM information_schema.role_table_grants
WHERE table_schema NOT IN ('pg_catalog', 'information_schema')
GROUP BY grantee, table_schema, table_name
ORDER BY table_schema, table_name, grantee;

SQL Server:

-- Show all effective permissions for all users in the current database
SELECT dp.name       AS principal,
       dp.type_desc  AS principal_type,
       o.name        AS object_name,
       o.type_desc   AS object_type,
       p.permission_name,
       p.state_desc
FROM sys.database_permissions p
JOIN sys.database_principals dp ON dp.principal_id = p.grantee_principal_id
LEFT JOIN sys.objects o         ON o.object_id = p.major_id
WHERE dp.type NOT IN ('R') -- exclude roles, show only users
ORDER BY dp.name, o.name;

Beyond static permission snapshots, you need event-level auditing — a record of who actually did what, and when.

PostgreSQL provides pgaudit, an extension that integrates with the standard logging infrastructure to produce structured audit logs at the session level or object level. At the object level, you define which operations on which objects should be logged. pgaudit outputs through the normal PostgreSQL log facility, so your existing log shipping and SIEM integration infrastructure applies.

SQL Server has a native SQL Server Audit feature built into the engine. You create an audit object pointing to a destination (a file, the Windows Security log, or the Application log), then create a server audit specification or database audit specification that defines which events to capture. The built-in DATABASE_OBJECT_ACCESS_GROUP, SCHEMA_OBJECT_ACCESS_GROUP, and DATABASE_PRINCIPAL_IMPERSONATION_GROUP audit action groups cover most production needs.

Beyond formal audit tooling, both systems maintain system views that reveal useful security-related state. In SQL Server, sys.server_principals and sys.database_principals give you the current principal inventory. sys.login_token and sys.user_token show the effective permissions and group memberships of the current session — invaluable when debugging why a user can or cannot perform an operation. In PostgreSQL, pg_roles, pg_auth_members, and the has_table_privilege(), has_schema_privilege(), and has_column_privilege() functions let you test permissions from within the session, which is far faster than hunting through catalog joins.

Periodic access reviews should be a scheduled operational activity, not something done only when there's a problem. A practical pattern is to export the permission snapshot query above on a regular cadence and diff it against the previous period. Any change — a new grant, a role membership addition, a privilege on a newly created table — surfaces immediately. This is light infrastructure that catches a large class of security drift before it becomes a compliance issue.

Two production scenarios that illustrate why this matters: First, a GRANT on ALL TABLES IN SCHEMA in PostgreSQL affects every table that existed at the time of the grant, but not tables created afterward — unless ALTER DEFAULT PRIVILEGES was also set. This means that as the schema grows, the reporting user silently loses access to new tables without anyone noticing, or (in the worse scenario) someone sets ALTER DEFAULT PRIVILEGES too broadly and new tables grant access to roles that shouldn't have it. Second, in SQL Server, when a database is restored from a backup to a different server, the SIDs of SQL logins may not match the SIDs of users inside the database. The user exists in the database but has no corresponding server login — an "orphaned user." Running sp_change_users_login or ALTER USER ... WITH LOGIN resolves it, but it's easy to miss, and the result is a database where no one can connect using SQL authentication.

Ownership Chains, Impersonation, and the Edges of the Permission Model

Most access control discussions cover the straightforward path: user requests access, permission is checked, access is granted or denied. Production systems have edges — places where the normal rules don't apply in the way you'd expect.

Ownership chaining in SQL Server is one such edge. When a stored procedure and the tables it references are owned by the same principal, SQL Server does not re-check permissions on the underlying tables when the procedure executes. The permission check applies only to the procedure itself. This is by design — it allows you to grant EXECUTE on a procedure to a user who has no direct access to the underlying tables, which is a clean and useful security pattern. But it has a trap: if the procedure and one of the referenced tables are owned by *different* principals, the chain breaks and SQL Server checks permissions on that table against the calling user. In a schema where object ownership is inconsistent, a procedure that worked fine for years suddenly starts failing for certain users after an ownership change on a table it references.

Impersonation in SQL Server (EXECUTE AS) lets a principal temporarily assume the identity of another for the scope of a session or a module. This is useful for testing permissions, for delegation scenarios, and for creating security contexts within stored procedures. EXECUTE AS USER switches to a database user, EXECUTE AS LOGIN switches to a server login. Both require the caller to have IMPERSONATE permission on the target principal.

SQL Server:

-- Temporarily execute as another user to test permissions
EXECUTE AS USER = 'alice';

SELECT HAS_PERMS_BY_NAME('sales.orders', 'OBJECT', 'SELECT') AS can_select;

-- Return to original context
REVERT;

-- Grant impersonate permission explicitly
GRANT IMPERSONATE ON USER::alice TO dba_support;

PostgreSQL:

-- PostgreSQL uses SET ROLE to switch active role within a session
-- The user must be a member of the target role
SET ROLE reporting_team;

-- Test what privileges the current role has
SELECT has_table_privilege('sales.orders', 'SELECT') AS can_select;

-- Return to original role
RESET ROLE;

-- Or use SET SESSION AUTHORIZATION (requires superuser)
SET SESSION AUTHORIZATION alice;
RESET SESSION AUTHORIZATION;

PostgreSQL's SECURITY DEFINER functions deserve attention here. When a function is defined with SECURITY DEFINER, it executes with the privileges of the function's owner, not the calling user. This is the PostgreSQL equivalent of ownership chaining and EXECUTE AS combined. It allows non-privileged users to perform controlled operations on objects they otherwise cannot access. The risk is that a SECURITY DEFINER function that takes unsanitized user input and builds dynamic SQL is a privilege escalation vector. Any function using this attribute should be written with the same care as security-critical code: input validation, parameterized queries, and a clear understanding of what the owner's privileges allow.

Cross-database access is another edge worth understanding. In PostgreSQL, a connection is tied to a single database — there is no built-in cross-database query mechanism without extensions like dblink or postgres_fdw. This is actually a security asset: compromising access to one database does not inherently expose another. In SQL Server, linked servers and three-part names (OtherDB.dbo.tablename) allow cross-database queries, which creates cross-database permission dependencies that need to be tracked. The TRUSTWORTHY database property in SQL Server is particularly worth auditing: if enabled, it allows code in that database to access server-level resources using the database owner's server-level credentials. TRUSTWORTHY is off by default and should stay off in production unless there is a specific, documented need.

SQL Server:

-- Audit databases with TRUSTWORTHY enabled (should be rare)
SELECT name,
       is_trustworthy_on,
       owner_sid,
       SUSER_SNAME(owner_sid) AS owner_name
FROM sys.databases
WHERE is_trustworthy_on = 1
  AND name NOT IN ('msdb'); -- msdb legitimately uses TRUSTWORTHY

-- Check for EXECUTE AS in stored procedures (review these carefully)
SELECT SCHEMA_NAME(o.schema_id) AS schema_name,
       o.name                   AS procedure_name,
       m.execute_as_principal_id,
       dp.name                  AS execute_as_principal
FROM sys.sql_modules m
JOIN sys.objects o ON o.object_id = m.object_id
LEFT JOIN sys.database_principals dp
    ON dp.principal_id = m.execute_as_principal_id
WHERE m.execute_as_principal_id IS NOT NULL
  AND o.type = 'P';

The theme across all of these edge cases is that the permission model has complexity that goes beyond simple grant/revoke. Ownership, impersonation, function security attributes, and cross-database trust are all mechanisms that can override the apparent permission state. Knowing they exist — and knowing where to look for them — is what separates a DBA who manages access control from one who merely administers it.

Key Takeaways

  • → **Authentication and authorization are separate concerns.** Authentication controls who can connect; authorization controls what they can do. Resolving a security issue often requires addressing both independently.
  • → **Design roles around job functions, not individuals.** Grant permissions to roles, add users to roles. This makes access reviews tractable and ensures that when people change jobs, permissions change with them rather than accumulating silently.
