Chapter 13: Azure SQL Database vs Managed Instance vs SQL Server on VM

When you move SQL Server workloads to Azure, the first decision you face is not which region to use or how large an instance to provision — it is which deployment model to choose. Azure offers three fundamentally different ways to run SQL Server: Azure SQL Database (a fully managed PaaS service), Azure SQL Managed Instance (a near-complete SQL Server engine in a managed container), and SQL Server on Azure Virtual Machines (IaaS, essentially your on-premises server in the cloud). Each model trades administrative control for operational convenience at a different point on the spectrum. Getting this choice wrong costs money, causes compatibility headaches, and in some cases requires a full migration to correct. This chapter maps each deployment model to the scenarios where it genuinely fits, shows you how to configure them correctly from day one, and identifies the production gotchas that no marketing documentation warns you about.

---

13.1 The Three Deployment Models: Mapping the Landscape

Before writing a single line of Terraform or clicking through the Azure portal, you need to understand what you are actually buying with each model.

Azure SQL Database is a database-as-a-service offering. You provision a database (or a pool of databases), and Azure manages the underlying instance entirely. There is no SQL Server Agent in the traditional sense. Cross-database queries require elastic queries. Linked servers do not exist. Features like SQL Server Audit work, but you configure them through Azure-native tooling rather than T-SQL. The engine is constantly patched by Microsoft on a schedule you influence but do not fully control. For greenfield applications that fit the PaaS model, this is powerful. For lift-and-shift migrations, it is frequently a poor fit without application rework.

Azure SQL Managed Instance is the middle path. It runs a near-complete SQL Server engine — SQL Server Agent, cross-database queries, linked servers, CLR assemblies, Service Broker, Database Mail — inside a managed VNet-injected service. You get instance-level features that Azure SQL Database strips away. Patching still happens on Microsoft's schedule (with a configurable maintenance window), but you control far more of the engine. The compatibility surface area covers roughly 99% of on-premises SQL Server 2019 features. Lift-and-shift migrations from on-premises SQL Server work with minimal code changes.

SQL Server on Azure VM is IaaS. You own the OS, you own the SQL Server installation, you patch everything yourself (though Azure provides automated patching assistance), and you have complete feature parity with on-premises SQL Server because it is on-premises SQL Server, running on Azure hardware. You pay for this control in operational overhead: OS patching, SQL Server patching, backup management, HA configuration, and storage optimization are all your responsibility.

The decision matrix looks like this in practice:

| Requirement | Azure SQL DB | Managed Instance | SQL Server on VM |

|---|---|---|---|

| Lift-and-shift from on-prem | Poor | Excellent | Excellent |

| Greenfield cloud-native app | Excellent | Good | Overkill |

| SQL Server Agent jobs | No (elastic jobs) | Yes | Yes |

| Cross-database queries | Elastic query only | Yes | Yes |

| Linked servers | No | Yes | Yes |

| CLR assemblies | No | Yes (safe only) | Yes |

| Service Broker | No | Yes | Yes |

| Full OS access | No | No | Yes |

| Windows auth / AD | Azure AD only | Full AD integration | Full AD integration |

| HADR (Always On AG) | Built-in geo-replication | Built-in auto-failover groups | Manual config |

| Maximum single DB size | 4 TB (General Purpose) | 8 TB | 64 TB (Premium SSD v2) |

One mental shift experienced DBAs need to make: Managed Instance is not a hosted SQL Server. It is a service that exposes SQL Server semantics. You cannot RDP into it. You cannot modify startup parameters freely. You cannot install third-party tools at the OS layer. If you need to run a third-party backup agent or an encryption tool that hooks into the OS, you need SQL Server on VM.

13.2 Provisioning Each Model: Infrastructure as Code Patterns

Manual portal provisioning is fine for experimentation. Production deployments require repeatable infrastructure. The following examples use Terraform, which handles all three deployment models through the AzureRM provider.

Provisioning Azure SQL Database with an Elastic Pool:

# terraform/azure_sql_database.tf

terraform {
  required_providers {
    azurerm = {
      source  = "hashicorp/azurerm"
      version = "~> 3.75"
    }
  }
}

provider "azurerm" {
  features {}
}

resource "azurerm_resource_group" "sql_rg" {
  name     = "rg-sql-production-eastus"
  location = "East US"
}

resource "azurerm_mssql_server" "logical_server" {
  name                         = "sql-prod-eastus-001"
  resource_group_name          = azurerm_resource_group.sql_rg.name
  location                     = azurerm_resource_group.sql_rg.location
  version                      = "12.0"
  administrator_login          = "sqladmin"
  administrator_login_password = var.sql_admin_password

  # Enforce Azure AD authentication in addition to SQL auth
  azuread_administrator {
    login_username = "DBA-Admin-Group"
    object_id      = var.aad_admin_group_object_id
    tenant_id      = var.tenant_id
  }

  # Prevent public internet access — enforce private endpoint
  public_network_access_enabled = false

  tags = {
    environment = "production"
    owner       = "dba-team"
  }
}

resource "azurerm_mssql_elasticpool" "app_pool" {
  name                = "epool-prod-app-tier"
  resource_group_name = azurerm_resource_group.sql_rg.name
  location            = azurerm_resource_group.sql_rg.location
  server_name         = azurerm_mssql_server.logical_server.name

  sku {
    name     = "GP_Gen5"
    tier     = "GeneralPurpose"
    family   = "Gen5"
    capacity = 16   # 16 vCores shared across all databases in the pool
  }

  per_database_settings {
    min_capacity = 0.25   # Minimum vCores per DB — prevents starvation
    max_capacity = 8      # Maximum vCores a single DB can consume
  }

  max_size_gb = 512

  # Zone redundancy adds ~$0.60/vCore/hour but is non-negotiable for production
  zone_redundant = true
}

resource "azurerm_mssql_database" "app_db" {
  name           = "AppDatabase"
  server_id      = azurerm_mssql_server.logical_server.id
  elastic_pool_id = azurerm_mssql_elasticpool.app_pool.id

  # collation must match your application — changing this post-creation
  # requires a full database rebuild
  collation = "SQL_Latin1_General_CP1_CI_AS"

  # geo-redundant backups add cost but are required for PITR across regions
  geo_backup_enabled = true

  short_term_retention_policy {
    retention_days           = 35
    backup_interval_in_hours = 12
  }

  long_term_retention_policy {
    weekly_retention  = "P4W"
    monthly_retention = "P12M"
    yearly_retention  = "P5Y"
    week_of_year      = 1
  }
}

Provisioning Azure SQL Managed Instance:

Managed Instance requires a dedicated subnet — this is non-negotiable and frequently misunderstood. The subnet must be delegated exclusively to Microsoft.Sql/managedInstances and cannot be shared with any other resource type.

# terraform/managed_instance.tf

resource "azurerm_virtual_network" "mi_vnet" {
  name                = "vnet-mi-production"
  resource_group_name = azurerm_resource_group.sql_rg.name
  location            = azurerm_resource_group.sql_rg.location
  address_space       = ["10.10.0.0/16"]
}

resource "azurerm_subnet" "mi_subnet" {
  name                 = "snet-managed-instance"
  resource_group_name  = azurerm_resource_group.sql_rg.name
  virtual_network_name = azurerm_virtual_network.mi_vnet.name

  # /27 gives you 27 IP addresses — enough for a 2-node MI cluster.
  # A General Purpose MI consumes ~5 IPs, Business Critical ~9 IPs.
  # Undersizing this subnet is a common and painful mistake.
  address_prefixes = ["10.10.1.0/27"]

  delegation {
    name = "managedInstanceDelegation"
    service_delegation {
      name = "Microsoft.Sql/managedInstances"
      actions = [
        "Microsoft.Network/virtualNetworks/subnets/join/action",
        "Microsoft.Network/virtualNetworks/subnets/prepareNetworkPolicies/action",
        "Microsoft.Network/virtualNetworks/subnets/unprepareNetworkPolicies/action"
      ]
    }
  }
}

resource "azurerm_mssql_managed_instance" "prod_mi" {
  name                = "mi-prod-eastus-001"
  resource_group_name = azurerm_resource_group.sql_rg.name
  location            = azurerm_resource_group.sql_rg.location

  administrator_login          = "sqladmin"
  administrator_login_password = var.sql_admin_password

  license_type       = "LicenseIncluded"   # Or "BasePrice" if you have SA
  sku_name           = "GP_Gen5"           # General Purpose, 5th generation
  vcores             = 8
  storage_size_in_gb = 256

  subnet_id = azurerm_subnet.mi_subnet.id

  # Maintenance window — choose a low-traffic window.
  # DayOfWeek 6 = Saturday, start at 02:00 UTC
  maintenance_configuration_name = "SQL_EastUS_MI_2"

  # Timezone cannot be changed post-creation
  timezone_id = "Eastern Standard Time"

  # Minimum TLS version enforcement
  minimum_tls_version = "1.2"

  tags = {
    environment = "production"
    tier        = "managed-instance"
  }
}

Important subnet sizing note: A Business Critical Managed Instance with 2 nodes consumes approximately 9 IPs during normal operation and up to 17 during failover. A /27 subnet (27 usable IPs) is the absolute minimum for a single instance. For multiple instances or upgrade scenarios, use /24. Resizing a subnet after Managed Instance deployment requires taking the instance offline — plan this correctly the first time.

Azure CLI for quick deployment validation:

# Azure CLI — validate MI readiness before Terraform apply

# Check subnet delegation is correct
az network vnet subnet show \
  --resource-group rg-sql-production-eastus \
  --vnet-name vnet-mi-production \
  --name snet-managed-instance \
  --query "delegations[].serviceName" \
  --output tsv

# Verify no conflicting NSG rules blocking MI management ports
# MI requires inbound 9000, 9003, 1438, 1440, 1452 from AzureCloud service tag
az network nsg rule list \
  --resource-group rg-sql-production-eastus \
  --nsg-name nsg-managed-instance \
  --output table

# Check the MI provisioning state — MI can take 4-6 hours to provision
az sql mi show \
  --resource-group rg-sql-production-eastus \
  --name mi-prod-eastus-001 \
  --query "{State:state, FQDN:fullyQualifiedDomainName, vCores:vCores}" \
  --output table

13.3 Networking: Private Endpoints, VNet Integration, and the Public Internet

One of the largest mental shifts for on-premises DBAs is networking. On-premises, you controlled network access through firewall rules, pg_hba.conf equivalents, SQL Server login restrictions, and physical network segmentation. In Azure, the equivalent mechanisms are Private Endpoints, VNet Service Endpoints, firewall rules on the logical server, and network security groups.

Azure SQL Database uses a logical server model. By default, the logical server has a public endpoint. Every Azure best practice guide will tell you to disable it and use Private Endpoints instead, and they are correct. A private endpoint creates a network interface in your VNet with a private IP address that maps to your logical server. Traffic never traverses the public internet.

# Azure CLI — create a private endpoint for Azure SQL Database

# Step 1: Disable public access on the logical server
az sql server update \
  --resource-group rg-sql-production-eastus \
  --name sql-prod-eastus-001 \
  --enable-public-network false

# Step 2: Create the private endpoint
az network private-endpoint create \
  --name pe-sql-prod-eastus \
  --resource-group rg-sql-production-eastus \
  --vnet-name vnet-application-eastus \
  --subnet snet-application-tier \
  --private-connection-resource-id $(az sql server show \
      --resource-group rg-sql-production-eastus \
      --name sql-prod-eastus-001 \
      --query id --output tsv) \
  --group-id sqlServer \
  --connection-name conn-sql-prod-eastus

# Step 3: Create a private DNS zone for name resolution
# Without this, your application cannot resolve the SQL server FQDN
az network private-dns zone create \
  --resource-group rg-sql-production-eastus \
  --name "privatelink.database.windows.net"

az network private-dns link vnet create \
  --resource-group rg-sql-production-eastus \
  --zone-name "privatelink.database.windows.net" \
  --name dns-link-vnet-app \
  --virtual-network vnet-application-eastus \
  --registration-enabled false

# Step 4: Create DNS zone group to auto-register the private endpoint IP
az network private-endpoint dns-zone-group create \
  --resource-group rg-sql-production-eastus \
  --endpoint-name pe-sql-prod-eastus \
  --name sql-dns-zone-group \
  --private-dns-zone "privatelink.database.windows.net" \
  --zone-name sql

# Verify resolution from within the VNet
# sql-prod-eastus-001.database.windows.net should now resolve to 10.x.x.x
nslookup sql-prod-eastus-001.database.windows.net

Managed Instance operates differently. Because it is VNet-injected, it only has a private endpoint by default — the instance lives inside your subnet. The challenge with MI networking is that it requires specific NSG rules for the management plane. Azure's MI management infrastructure must be able to reach the instance on ports 9000, 9003, 1438, 1440, and 1452. If you lock down your NSG incorrectly, the MI enters a failed state and becomes unreachable. The required NSG rule uses the SqlManagement service tag as the source.

# Verify MI management traffic is not blocked
# Run this before deploying MI — a blocked management port causes provisioning failure

az network nsg rule create \
  --resource-group rg-sql-production-eastus \
  --nsg-name nsg-managed-instance \
  --name AllowAzureManagement \
  --priority 100 \
  --source-address-prefixes SqlManagement \
  --destination-address-prefixes "*" \
  --destination-port-ranges 9000 9003 1438 1440 1452 \
  --access Allow \
  --protocol Tcp \
  --direction Inbound

# Verify connectivity to MI from application subnet
# MI uses port 1433 for TDS (same as SQL Server) and 3342 for public endpoint (if enabled)
az network watcher test-connectivity \
  --resource-group rg-sql-production-eastus \
  --source-resource /subscriptions/${SUBSCRIPTION_ID}/resourceGroups/rg-application/providers/Microsoft.Compute/virtualMachines/vm-app-001 \
  --dest-address mi-prod-eastus-001.a1b2c3d4e5f6.database.windows.net \
  --dest-port 1433

SQL Server on VM uses standard Azure VM networking. You control NSGs at the NIC and subnet level, and you configure SQL Server's firewall through SQL Server Configuration Manager — exactly as you would on-premises. The main gotcha is that Azure VM firewall rules and SQL Server's built-in firewall are independent. Both must allow traffic.

13.4 High Availability and Business Continuity

This section is where the three deployment models diverge most dramatically from on-premises patterns. On-premises, you built Always On Availability Groups manually: you configured Windows Server Failover Clustering, set up quorum, joined replicas, created the listener, and spent a weekend validating failover behavior. In Azure, HA is either built into the service layer (Azure SQL DB and MI) or your responsibility (SQL Server on VM).

Azure SQL Database HA uses a multi-replica architecture internally. General Purpose tier uses remote storage (Azure Premium Storage) with a single compute node — there is no synchronous replica, so failover involves attaching storage to a new compute node, which takes 20–30 seconds. Business Critical tier maintains 3–4 synchronous replicas using Always On AG technology under the covers. You cannot see the replicas directly, but failover is near-instantaneous (typically under 5 seconds). The Business Critical tier also exposes one read-only replica that you can redirect read workloads to without extra cost.

Auto-failover groups provide geo-redundancy across regions for both Azure SQL DB and Managed Instance:

# Azure CLI — configure auto-failover group for Azure SQL Database

# Create the secondary server in paired region
az sql server create \
  --name sql-prod-westus-001 \
  --resource-group rg-sql-production-westus \
  --location "West US" \
  --admin-user sqladmin \
  --admin-password "${SQL_ADMIN_PASSWORD}"

# Create the failover group
az sql failover-group create \
  --name fog-prod-eastus-westus \
  --server sql-prod-eastus-001 \
  --resource-group rg-sql-production-eastus \
  --partner-server sql-prod-westus-001 \
  --partner-resource-group rg-sql-production-westus \
  --failover-policy Automatic \
  --grace-period 1   # Hours before automatic failover triggers — 1 is minimum

# Add databases to the failover group
az sql failover-group update \
  --name fog-prod-eastus-westus \
  --server sql-prod-eastus-001 \
  --resource-group rg-sql-production-eastus \
  --add-db AppDatabase

# The failover group creates two listener endpoints:
# Read-write: fog-prod-eastus-westus.database.windows.net (points to primary)
# Read-only:  fog-prod-eastus-westus.secondary.database.windows.net

# Test failover — this promotes the secondary to primary
az sql failover-group set-primary \
  --name fog-prod-eastus-westus \
  --server sql-prod-westus-001 \
  --resource-group rg-sql-production-westus

RPO and RTO characteristics in production:

| Deployment | HA Tier | RTO (same region) | RTO (geo-failover) | RPO |

|---|---|---|---|---|

| Azure SQL DB - General Purpose | Built-in | 20–30 sec | ~30 min | Up to 1 hour |

| Azure SQL DB - Business Critical | Built-in | <5 sec | ~30 min | ~5 sec |

| Managed Instance - General Purpose | Built-in | ~20 sec | ~1 hour | ~1 hour |

| Managed Instance - Business Critical | Built-in | <5 sec | ~30 min | ~5 sec |

| SQL Server on VM (manual AG) | DIY | ~30 sec | DIY | Depends on sync mode |

SQL Server on Azure VM requires you to build Always On AG using Windows Server Failover Clustering. Azure provides an Internal Load Balancer for the AG listener (equivalent to the on-premises listener IP). The key difference from on-premises is that the ILB requires a probe port configured in the AG listener, and the floating IP behavior differs slightly from on-premises due to Azure's networking model.

-- SQL Server on Azure VM: AG listener with Azure ILB probe
-- The probe port (59999 here) must match the ILB health probe configuration

ALTER AVAILABILITY GROUP [AG_Production]
ADD LISTENER 'AGListener' (
    WITH IP (('10.10.2.100', '255.255.255.0')),
    PORT = 1433
);
GO

-- Configure ILB probe response — run on each replica
-- Azure ILB sends TCP probes to 59999; only the primary should respond
EXEC sp_procoption N'sp_ag_probe', 'startup', 'true';
GO

-- Verify AG health
SELECT
    ag.name                     AS availability_group,
    ar.replica_server_name,
    ars.role_desc,
    ars.synchronization_health_desc,
    ars.connected_state_desc,
    ars.last_redone_time,
    ars.log_send_queue_size      AS log_send_queue_kb,
    ars.redo_queue_size          AS redo_queue_kb
FROM sys.availability_groups ag
JOIN sys.availability_replicas ar
    ON ag.group_id = ar.group_id
JOIN sys.dm_hadr_availability_replica_states ars
    ON ar.replica_id = ars.replica_id
ORDER BY ag.name, ars.role_desc;

13.5 Security: Authentication, Encryption, and Compliance Patterns

On-premises SQL Server security centered on SQL Server logins, Windows Authentication through Active Directory, TDE for encryption at rest, and SSL/TLS for encryption in transit. Azure maps these concepts onto a cloud-native model that is simultaneously more automated and more complex to configure correctly.

Azure Active Directory Authentication replaces Windows Authentication in Azure SQL Database and Managed Instance. For applications, this means using Managed Identity — the application's Azure AD identity — rather than a SQL login with a stored password. This is the cloud equivalent of using a Kerberos service account, except the credential rotation is automatic.
