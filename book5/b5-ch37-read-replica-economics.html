Chapter 37: Read Replica Economics: When They Pay Off

Read replicas are one of the most frequently misunderstood tools in cloud database architecture. On paper, the pitch is simple: pay for a second instance and double your read throughput. In practice, the economics are rarely that clean. Replicas introduce replication lag, connection management complexity, application-level routing logic, and real dollar costs that can easily exceed the value they deliver—unless you understand exactly when and where they make sense. This chapter gives you the framework to evaluate replica deployments across AWS RDS, Aurora, Azure SQL, and Cloud SQL, with attention to the cost models, performance characteristics, and operational patterns that separate a well-designed read scaling strategy from an expensive misfire.

---

37.1 How Cloud Read Replicas Differ from On-Premises Streaming Replication

If you've run PostgreSQL streaming replication or SQL Server Always On Availability Groups on-premises, you already understand the fundamentals: WAL records or log blocks ship to a secondary, the secondary applies them, and reads can be offloaded to the standby. What changes in the cloud is everything surrounding that core mechanism—storage architecture, failover behavior, lag semantics, and, critically, how you pay for it.

RDS PostgreSQL and MySQL replicas use the same logical streaming replication you know from self-managed Postgres. The primary generates WAL, the replica applies it asynchronously by default. This means the replication lag you measured with pg_stat_replication on-premises is the same metric you'll watch in CloudWatch under ReplicaLag. The difference is that RDS manages the replication slot, the wal_keep_size configuration, and the standby's recovery.conf (now postgresql.auto.conf in Postgres 12+) for you. What you lose is the ability to tune recovery_min_apply_delay to create intentional lag standbys—that's not exposed in RDS Parameter Groups. You also cannot promote a replica without going through the RDS API, which matters for your runbooks.

Aurora replicas operate on an entirely different model. Because Aurora uses a shared distributed storage layer across the writer and all readers, Aurora replicas don't ship WAL at all. Instead, the writer writes to the shared storage volume, and replicas read from the same pages directly. The replication "lag" in Aurora is not WAL application lag—it's the time for cache invalidation metadata to propagate. Aurora calls this AuroraReplicaLag, and it's typically measured in single-digit milliseconds rather than the seconds or minutes you might see in RDS streaming replication under write pressure. This is a fundamental architectural difference with real implications: Aurora can support up to 15 read replicas per cluster with sub-10ms lag even during heavy write workloads, while RDS PostgreSQL becomes increasingly difficult to manage at more than 3–4 replicas before lag compounds.

Azure SQL doesn't use the term "read replica" in the same way. On General Purpose tier, readable secondaries are not available for geo-replicas except at the Business Critical tier. At Business Critical, you get an Always On AG cluster under the hood, and one of the secondary replicas is readable at no additional cost—this is a meaningful economic advantage that's often overlooked. Azure SQL Hyperscale changes the model again: Hyperscale supports up to 30 named compute replicas that read from a shared log service and page server architecture similar in concept to Aurora's shared storage. Lag in Hyperscale is governed by log apply rate, not page transfer.

Cloud SQL uses PostgreSQL logical or streaming replication under the hood depending on version, and Spanner uses an entirely different synchronous Paxos-based replication. AlloyDB, Google's Aurora-equivalent, uses a disaggregated storage model similar to Aurora where read pool instances share the log stream without applying WAL independently.

The on-premises DBA's first instinct when reaching for a replica is to solve a read throughput problem. That's valid. But before you provision, you need to understand three cost dimensions that don't exist on-premises: compute cost of the replica instance, cross-AZ data transfer cost, and the cost of the engineering effort to route reads correctly.

37.2 The Real Cost Model: What You're Actually Paying For

Cloud providers make replica pricing look simple in their calculators. It is not simple in production.

RDS Replica Cost Components

A Multi-AZ RDS PostgreSQL db.r6g.2xlarge runs approximately $0.48/hour on-demand in us-east-1. Adding a read replica at the same instance class adds another $0.48/hour—100% of the primary cost. But that's not the whole picture:

  • → **Storage**: Replicas in RDS have their own EBS volumes. Storage is billed separately per replica at the same GP3 or io2 rates as the primary. A 2TB primary means you're paying for 2TB of storage on every replica.
  • → **Data transfer**: RDS charges $0.01/GB for data transferred between Availability Zones. Replication traffic between a primary in us-east-1a and a replica in us-east-1b crosses an AZ boundary. At 10 GB/hour of write throughput—not unusual for a busy OLTP database—that's $2.40/day in transfer charges alone, roughly $72/month. This is frequently omitted from cost analysis.
  • → **Cross-region replicas**: If you create a cross-region read replica for disaster recovery or latency reduction, data transfer fees are billed at standard inter-region rates, which range from $0.02 to $0.09/GB depending on region pairing.
    • # Estimate your current replication data volume before provisioning replicas
# Check bytes sent to replicas from the primary

aws cloudwatch get-metric-statistics \
  --namespace AWS/RDS \
  --metric-name NetworkTransmitThroughput \
  --dimensions Name=DBInstanceIdentifier,Value=prod-postgres-primary \
  --start-time 2024-01-15T00:00:00Z \
  --end-time 2024-01-16T00:00:00Z \
  --period 3600 \
  --statistics Average \
  --region us-east-1

# ReplicaLag monitoring — keep this under 30 seconds for most OLAP workloads
# Alert if it exceeds your application's staleness tolerance
aws cloudwatch put-metric-alarm \
  --alarm-name "prod-replica-lag-warning" \
  --metric-name ReplicaLag \
  --namespace AWS/RDS \
  --statistic Maximum \
  --period 60 \
  --threshold 30 \
  --comparison-operator GreaterThanThreshold \
  --dimensions Name=DBInstanceIdentifier,Value=prod-postgres-replica-1 \
  --evaluation-periods 3 \
  --alarm-actions arn:aws:sns:us-east-1:123456789:db-alerts \
  --region us-east-1

    Aurora Replica Cost Components

    Aurora pricing separates storage from compute. The cluster storage is billed once, shared across writer and all readers—this is the key economic difference from RDS. You pay for:

  • → Compute: Each reader instance at its instance class rate (same as primary compute)
  • → Storage: Once, shared, billed at Aurora's per-GB-month rate (approximately $0.10/GB/month)
  • → I/O: $0.20 per million I/O requests (unless using Aurora I/O-Optimized pricing at a 25% compute premium, which eliminates per-I/O charges)
  • → No cross-AZ replication data transfer charges—Aurora's shared storage handles this internally

    • The practical result: at scale, Aurora replicas cost less per replica than RDS replicas because you eliminate the per-replica storage cost. A cluster with 4TB of data and five read replicas pays for 4TB of storage once rather than 6× (one primary plus five replicas) at RDS rates.

    Azure SQL Business Critical Readable Secondary

    This is where Azure's model creates a genuine economic advantage for mixed read-write workloads. At Business Critical tier, you're already paying for an Always On cluster. One secondary is exposed as readable at no additional compute charge. If your read traffic can tolerate potential AG lag during failover scenarios, this effectively gives you a read replica for free as part of your HA investment.

    # Check if your Azure SQL instance is Business Critical tier
az sql db show \
  --resource-group prod-rg \
  --server prod-sqlserver \
  --name prod-database \
  --query "{sku: sku.tier, readScale: readScale}" \
  --output table

# Enable read scale-out on an existing Business Critical database
az sql db update \
  --resource-group prod-rg \
  --server prod-sqlserver \
  --name prod-database \
  --read-scale Enabled

# Add ApplicationIntent=ReadOnly to the connection string
# Server=prod-sqlserver.database.windows.net;Database=prod-database;
# Authentication=ActiveDirectoryMsi;ApplicationIntent=ReadOnly

    The Hidden Cost: Application Complexity

    Every replica introduces routing logic. Read/write splitting requires your application to maintain two connection strings, or you deploy a proxy layer (RDS Proxy, ProxySQL, PgBouncer on EC2). RDS Proxy costs approximately $0.015 per vCPU-hour of the database instance it fronts. For a db.r6g.2xlarge (8 vCPUs), that's $0.12/hour or roughly $87/month just for the proxy. PgBouncer on a t3.medium runs $0.0416/hour—about $30/month—but you own the HA and patching. Factor this into your cost model before assuming replicas are a net win.

    37.3 When Replicas Actually Pay Off: Workload Patterns That Justify the Cost

    The honest answer is that read replicas pay off in a narrower set of scenarios than most architecture diagrams suggest. Here are the patterns where the economics are genuinely favorable.

    Pattern 1: Long-Running Analytical Queries Against an OLTP Database

    This is the canonical case. You have a production OLTP database handling transactional traffic, and the business runs nightly or near-real-time reports against the same database. The reports lock nothing (they're read-only) but consume enormous I/O and CPU, creating contention with OLTP queries for buffer pool space and CPU cycles. A read replica isolates this workload completely. The replica doesn't need to match the primary's instance size—if reports run in off-peak hours, you might run the replica at db.r6g.xlarge and the primary at db.r6g.4xlarge, reducing the per-replica cost.

    -- Query to identify read-heavy sessions that are candidates for replica offload
-- Run on RDS PostgreSQL primary before deciding on replica strategy
SELECT 
    pid,
    usename,
    application_name,
    state,
    wait_event_type,
    wait_event,
    EXTRACT(EPOCH FROM (now() - query_start)) AS query_duration_seconds,
    LEFT(query, 120) AS query_preview
FROM pg_stat_activity
WHERE state = 'active'
  AND query NOT LIKE 'autovacuum%'
  AND query NOT LIKE 'VACUUM%'
ORDER BY query_duration_seconds DESC
LIMIT 20;

-- Check read/write ratio on the primary — if reads are 70%+ of total queries,
-- replicas are worth modeling. Below 50%, evaluate query optimization first.
SELECT 
    sum(seq_scan + idx_scan) AS total_reads,
    sum(n_tup_ins + n_tup_upd + n_tup_del) AS total_writes,
    round(
        sum(seq_scan + idx_scan)::numeric / 
        NULLIF(sum(seq_scan + idx_scan + n_tup_ins + n_tup_upd + n_tup_del), 0) * 100, 
        2
    ) AS read_percentage
FROM pg_stat_user_tables;

    Pattern 2: Geographically Distributed Read Traffic

    If your users span continents and the majority of their operations are reads—product catalogs, content delivery, configuration data—cross-region replicas can reduce latency from 150–200ms (transatlantic round trip) to under 20ms by serving reads from a regional replica. The business case here is user experience, not throughput. Do the math: at $0.09/GB for US-East to EU-West data transfer, a database generating 100 GB/day of replication traffic to a European replica costs $270/month in transfer alone. That's justifiable if the alternative is building a separate regional database with its own write path and synchronization logic.

    Pattern 3: Read-Heavy Microservices with Read-Mostly Data

    Reference data services—product catalogs, geography tables, lookup tables—that are updated rarely but read thousands of times per second are strong replica candidates. The replication lag is irrelevant when data changes at most a few times per day. Here you can often use a smaller replica instance and cache aggressively from it.

    Pattern 4: Reporting and BI Tool Isolation

    Business intelligence tools like Tableau, Power BI, and Looker are notorious for generating unoptimized queries—full table scans, cross joins, high-memory sort operations. Pointing these tools at a replica protects the primary from analyst-generated chaos. This is a support and reliability argument as much as a performance argument. The cost is straightforward to justify: the alternative is hiring a DBA to babysit BI query impact on production.

    When Replicas Don't Pay Off

  • → **Write-heavy workloads**: If your write throughput is high, replicas will perpetually lag. A replica that's 30 seconds behind is worthless for any query that needs current data, and most OLTP application reads need current data.
  • → **Low read-to-write ratios**: If you're doing more writes than reads, you have a sharding or caching problem, not a read scaling problem.
  • → **Small datasets that fit in buffer pool**: If your working set fits in shared_buffers, a replica adds cost without adding performance. A `db.r6g.4xlarge` with 128 GB RAM can cache a 50 GB working set entirely. Adding a replica doesn't help a workload that never hits storage.
  • → **Short connection pooling exhaustion**: If you're running out of connections, you need PgBouncer or RDS Proxy, not a replica. Connection exhaustion is a connection management problem.

    37.4 Provisioning and Routing: Getting the Architecture Right

    Understanding the economics is half the work. Building the routing layer correctly is where replicas either function cleanly or become an operational liability.

    Creating Replicas: RDS and Aurora with Terraform

    # Terraform: RDS PostgreSQL read replica configuration
# Note: replica inherits VPC, subnet group, and parameter group from primary
# unless explicitly overridden

resource "aws_db_instance" "primary" {
  identifier              = "prod-postgres-primary"
  engine                  = "postgres"
  engine_version          = "15.4"
  instance_class          = "db.r6g.2xlarge"
  allocated_storage       = 500
  storage_type            = "gp3"
  storage_throughput      = 250  # MiB/s, gp3 allows tuning independently
  iops                    = 12000
  
  multi_az                = true
  db_subnet_group_name    = aws_db_subnet_group.prod.name
  vpc_security_group_ids  = [aws_security_group.rds.id]
  
  parameter_group_name    = aws_db_parameter_group.postgres15.name
  
  backup_retention_period = 14
  backup_window           = "03:00-04:00"
  maintenance_window      = "sun:04:00-sun:05:00"
  
  deletion_protection     = true
  skip_final_snapshot     = false
  final_snapshot_identifier = "prod-postgres-final-snapshot"

  tags = {
    Environment = "production"
    Role        = "primary"
  }
}

resource "aws_db_instance" "read_replica" {
  identifier          = "prod-postgres-replica-1"
  replicate_source_db = aws_db_instance.primary.identifier
  
  # Replica can use a different (smaller) instance class for cost optimization
  # Match primary for HA-style replicas, downsize for analytics-only replicas
  instance_class      = "db.r6g.xlarge"
  
  # Storage is provisioned independently for RDS replicas
  # Must be >= primary storage
  allocated_storage   = 500
  storage_type        = "gp3"
  
  # Place replica in a different AZ from primary for resilience
  availability_zone   = "us-east-1b"
  
  # Replicas should NOT be Multi-AZ themselves unless used as failover targets
  multi_az            = false
  
  # Replicas can have independent parameter groups
  # Useful for tuning work_mem and max_parallel_workers for analytics workloads
  parameter_group_name = aws_db_parameter_group.postgres15_replica.name
  
  # Auto minor version upgrade should match primary policy
  auto_minor_version_upgrade = true
  
  # Performance Insights at replica level — separate from primary
  performance_insights_enabled = true
  performance_insights_retention_period = 7
  
  vpc_security_group_ids = [aws_security_group.rds.id]
  
  tags = {
    Environment = "production"
    Role        = "read-replica"
    Workload    = "analytics"
  }
}

# Aurora read replica cluster configuration
resource "aws_rds_cluster" "aurora_primary" {
  cluster_identifier      = "prod-aurora-cluster"
  engine                  = "aurora-postgresql"
  engine_version          = "15.4"
  database_name           = "proddb"
  master_username         = "dbadmin"
  manage_master_user_password = true  # Secrets Manager integration
  
  db_subnet_group_name    = aws_db_subnet_group.prod.name
  vpc_security_group_ids  = [aws_security_group.aurora.id]
  
  backup_retention_period = 14
  preferred_backup_window = "03:00-04:00"
  
  # Aurora I/O-Optimized: eliminates per-I/O charges at 25% compute premium
  # Break-even at roughly 1 million I/O requests per month per vCPU
  # Uncomment if I/O costs exceed 25% of your compute bill
  # storage_type = "aurora-iopt1"
  
  deletion_protection     = true
  
  tags = {
    Environment = "production"
  }
}

resource "aws_rds_cluster_instance" "writer" {
  identifier         = "prod-aurora-writer"
  cluster_identifier = aws_rds_cluster.aurora_primary.id
  instance_class     = "db.r6g.2xlarge"
  engine             = aws_rds_cluster.aurora_primary.engine
  engine_version     = aws_rds_cluster.aurora_primary.engine_version
  
  performance_insights_enabled = true
  
  tags = {
    Role = "writer"
  }
}

resource "aws_rds_cluster_instance" "readers" {
  count              = 2
  identifier         = "prod-aurora-reader-${count.index + 1}"
  cluster_identifier = aws_rds_cluster.aurora_primary.id
  instance_class     = "db.r6g.xlarge"  # Readers can be smaller for cost
  engine             = aws_rds_cluster.aurora_primary.engine
  engine_version     = aws_rds_cluster.aurora_primary.engine_version
  
  # Distribute readers across AZs
  availability_zone  = element(["us-east-1b", "us-east-1c"], count.index)
  
  performance_insights_enabled = true
  
  tags = {
    Role = "reader"
  }
}

    Connection Routing with RDS Proxy

    The Aurora cluster endpoint handles read/write routing natively—the writer endpoint always points to the current writer, and the reader endpoint load-balances across all reader instances. For RDS PostgreSQL, you need to build this routing yourself or use RDS Proxy.
