Chapter 5: The Hidden Costs: Data Transfer, Snapshots, and Logs

Most engineering teams can recite their RDS instance cost to within a few dollars. Almost none can tell you what they spent on data transfer last month. That gap between what teams track and what actually appears on the bill is where database budgets quietly hemorrhage. Across AWS, Azure, and GCP, data egress fees, automated snapshot accumulation, and unmanaged log retention collectively account for 18–35% of total database spend for organizations running production workloads at scale — and the majority of that cost is entirely recoverable with targeted intervention. This chapter shows you exactly where to find these charges in your bill, how to calculate their true cost, and how to automate the controls that eliminate them. Organizations that complete the optimizations in this chapter typically recover $2,000 to $40,000 per month depending on data volumes, with the largest savings coming from cross-region replication patterns that were designed for resilience without any cost accounting applied to them.

---

5.1 Understanding Data Transfer Pricing: The Billing Model Nobody Reads

Data transfer pricing is intentionally complex. Cloud providers publish egress rates buried in footnotes, vary them by region pair, and charge differently depending on whether traffic crosses availability zones, regions, or the open internet. Understanding the layered model before you optimize is not optional — misreading it leads to architectural changes that save nothing.

AWS Data Transfer Tiers (2024 pricing, us-east-1 baseline)

| Traffic Type | Cost |

|---|---|

| Inbound (ingress) | $0.00 |

| Same-AZ, same-region (private IP) | $0.00 |

| Cross-AZ within same region | $0.02/GB each direction |

| Cross-region | $0.02–$0.09/GB depending on destination |

| Internet egress (first 10TB/month) | $0.09/GB |

| Internet egress (next 40TB/month) | $0.085/GB |

| RDS to EC2 in same AZ (public IP) | $0.02/GB |

The last row deserves a second read. An application server connecting to an RDS instance in the same availability zone over the instance's public DNS endpoint incurs cross-AZ charges even though both resources are physically in the same building. This single misconfiguration — using the public endpoint instead of the private endpoint — generates measurable egress charges on databases processing more than a few hundred gigabytes per month.

Azure Data Transfer Tiers (2024 pricing, East US baseline)

| Traffic Type | Cost |

|---|---|

| Inbound | $0.00 |

| Same-region, same VNet | $0.00 |

| Cross-region (within continent) | $0.02–$0.05/GB |

| Cross-continent | $0.05–$0.08/GB |

| Internet egress (first 5GB/month) | $0.00 |

| Internet egress (over 5GB) | $0.087/GB |

GCP Data Transfer Tiers (2024 pricing, us-central1 baseline)

| Traffic Type | Cost |

|---|---|

| Inbound | $0.00 |

| Same-zone | $0.00 |

| Cross-zone, same region | $0.01/GB |

| Cross-region (North America) | $0.01/GB |

| Cross-region (other) | $0.02–$0.08/GB |

| Internet egress | $0.085–$0.23/GB |

GCP's cross-zone pricing is lower than AWS — $0.01/GB vs $0.02/GB — making the cost of Multi-AZ equivalent patterns roughly half the AWS price. This is a relevant consideration when comparing total cost of ownership across providers for read-heavy workloads with multiple replicas.

Finding Your Current Transfer Spend

Before optimizing, establish a baseline. On AWS, Cost Explorer provides transfer cost breakdowns but requires correct filtering to isolate database-related transfer from compute transfer.

aws ce get-cost-and-usage \
  --time-period Start=$(date -d '30 days ago' '+%Y-%m-%d'),End=$(date '+%Y-%m-%d') \
  --granularity MONTHLY \
  --filter '{
    "And": [
      {
        "Dimensions": {
          "Key": "SERVICE",
          "Values": ["Amazon Relational Database Service"]
        }
      },
      {
        "Dimensions": {
          "Key": "USAGE_TYPE_GROUP",
          "Values": ["RDS: Data Transfer - Internet", "RDS: Data Transfer - Region to Region", "RDS: Data Transfer - AZ"]
        }
      }

  }' \
  --metrics "BlendedCost" "UsageQuantity" \
  --group-by Type=DIMENSION,Key=USAGE_TYPE \

# Azure CLI: Pull data transfer costs by resource group
az consumption usage list \
  --start-date $(date -d '30 days ago' '+%Y-%m-%d') \
  --end-date $(date '+%Y-%m-%d') \
  --query "[?contains(meterCategory,'Bandwidth')].{Name:instanceName, Cost:pretaxCost, Quantity:usageQuantity, Unit:unitOfMeasure}" \

# GCP: Query billing export in BigQuery for network egress on Cloud SQL
bq query --use_legacy_sql=false \
  "SELECT
    service.description AS service,
    sku.description AS sku,
    SUM(usage.amount) AS total_gb,
    SUM(cost) AS total_cost_usd
   FROM \`PROJECT_ID.billing_dataset.gcp_billing_export_v1_ACCOUNT_ID\`
   WHERE DATE(_PARTITIONTIME) >= DATE_SUB(CURRENT_DATE(), INTERVAL 30 DAY)
     AND service.description = 'Cloud SQL'
     AND LOWER(sku.description) LIKE '%network%'
   GROUP BY 1, 2
   ORDER BY total_cost_usd DESC"

Run these three commands before any other action in this chapter. The output establishes your current baseline and prevents optimizing costs that were never significant for your specific workload.

5.2 Cross-AZ and Cross-Region Traffic: Finding and Fixing the Expensive Flows

The most common source of unexpected data transfer costs is cross-AZ traffic generated by Multi-AZ RDS configurations combined with application servers spread across availability zones. In a standard production setup, you have three application servers distributed across three AZs for high availability, and an RDS Multi-AZ instance whose primary is in one of those AZs. Every application server not co-located with the RDS primary is paying $0.02/GB in each direction for every byte they exchange.

For a database processing 10TB of query results per month to application servers (a moderate OLTP workload), this costs:

  • → 10TB × $0.02/GB = 10,000GB × $0.02 = **$200/month cross-AZ egress**
  • → Plus return traffic: another **$200/month**
  • → Total cross-AZ penalty: **$400/month for a single database**

    • This compounds with read replicas. An RDS instance with three read replicas spread across AZs for distribution generates replication traffic costs on top of application traffic costs.

    Identifying Cross-AZ Traffic Sources

    On AWS, VPC Flow Logs capture the source and destination of all traffic including AZ origins. The following query, run against Flow Logs exported to CloudWatch Logs Insights, identifies the largest cross-AZ flows involving database ports.

    -- CloudWatch Logs Insights query (not SQL — uses Logs Insights syntax)
-- Run this in the CloudWatch console under Logs Insights against your VPC Flow Log group
-- Paste this directly into the query editor:
-- fields @timestamp, srcAddr, dstAddr, dstPort, bytes, azId
-- | filter dstPort in [5432, 3306, 1433, 5439, 27017]
-- | filter srcAddr not like /^10\.0\.1\./   -- replace with your primary AZ subnet
-- | stats sum(bytes) as total_bytes by srcAddr, dstAddr, dstPort
-- | sort total_bytes desc
-- | limit 20

-- For PostgreSQL: query pg_stat_activity to see connection origins
-- Run this on your RDS PostgreSQL instance to identify which client IPs are connecting
SELECT 
    client_addr,
    COUNT(*) AS connection_count,
    MAX(state) AS state,
    application_name,
    usename
FROM pg_stat_activity
WHERE client_addr IS NOT NULL
  AND state != 'idle'
GROUP BY client_addr, application_name, usename
ORDER BY connection_count DESC;

-- For MySQL: equivalent connection origin query
-- Run this on RDS MySQL to identify connection sources
-- SELECT 
--     HOST,
--     DB,
--     COUNT(*) as connection_count,
--     SUM(DATA_READ)/1024/1024 as data_read_mb,
--     SUM(DATA_WRITTEN)/1024/1024 as data_written_mb
-- FROM information_schema.PROCESSLIST
-- GROUP BY HOST, DB
-- ORDER BY connection_count DESC;

    Cross-reference the client_addr values against your AWS subnet map. Any connection coming from a subnet in a different AZ than your RDS primary is generating cross-AZ charges.

    Fixing Cross-AZ Traffic: Three Approaches

    *Option 1: Subnet-Aware Connection Routing*

    Configure your application's database connection pool to prefer the endpoint nearest to its own AZ. For applications using a service mesh or a proxy layer like ProxySQL or pgBouncer, configure topology-aware routing. This is the lowest-risk change.

    *Option 2: Read Replica Placement*

    If your workload is read-heavy, place one read replica in each AZ where application servers run. Configure read traffic to route to the local replica. For a three-AZ setup processing 5TB of reads per month per AZ, this eliminates cross-AZ read traffic entirely. The cost of an additional read replica (for example, an db.r6g.large at approximately $146/month) is offset immediately if cross-AZ read traffic exceeds 7,300GB/month per AZ.

    *Option 3: Private Endpoint Enforcement*

    Audit all connection strings and ensure every application is connecting to the private DNS endpoint, not the public endpoint. This single change eliminates the class of cross-AZ charges that arise from public-IP routing.

    #!/usr/bin/env python3
"""
cross_az_detector.py
Scans RDS instances and their connecting EC2 instances,
flags mismatched AZ combinations that generate cross-AZ traffic costs.
Run on a schedule (daily recommended) via Lambda or cron.
"""

import boto3
import json
from datetime import datetime, timedelta
from collections import defaultdict

def get_rds_primary_az(rds_client, db_instance_id):
    """Return the AZ of the primary RDS instance."""
    response = rds_client.describe_db_instances(DBInstanceIdentifier=db_instance_id)
    return response['DBInstances'][0]['AvailabilityZone']

def get_ec2_instances_by_az(ec2_client):
    """Return dict mapping AZ to list of instance IDs and private IPs."""
    az_map = defaultdict(list)
    paginator = ec2_client.get_paginator('describe_instances')
    for page in paginator.paginate(Filters=[{'Name': 'instance-state-name', 'Values': ['running']}]):
        for reservation in page['Reservations']:
            for instance in reservation['Instances']:
                az = instance['Placement']['AvailabilityZone']
                az_map[az].append({
                    'id': instance['InstanceId'],
                    'private_ip': instance.get('PrivateIpAddress', 'unknown'),
                    'tags': {t['Key']: t['Value'] for t in instance.get('Tags', [])}
                })
    return az_map

def estimate_cross_az_cost(cloudwatch_client, rds_instance_id, days=30):
    """
    Estimate data transfer cost by looking at NetworkReceiveThroughput
    and NetworkTransmitThroughput on the RDS instance.
    Returns estimated GB transferred.
    """
    end_time = datetime.utcnow()
    start_time = end_time - timedelta(days=days)
    
    total_gb = 0
    for metric in ['NetworkReceiveThroughput', 'NetworkTransmitThroughput']:
        response = cloudwatch_client.get_metric_statistics(
            Namespace='AWS/RDS',
            MetricName=metric,
            Dimensions=[{'Name': 'DBInstanceIdentifier', 'Value': rds_instance_id}],
            StartTime=start_time,
            EndTime=end_time,
            Period=86400,  # daily datapoints
            Statistics=['Average']
        )
        
        # Average bytes/sec * seconds_per_day * days = total bytes
        for dp in response['Datapoints']:
            bytes_per_day = dp['Average'] * 86400
            total_gb += bytes_per_day / (1024**3)
    
    return round(total_gb, 2)

def main():
    region = 'us-east-1'
    rds_client = boto3.client('rds', region_name=region)
    ec2_client = boto3.client('ec2', region_name=region)
    cw_client = boto3.client('cloudwatch', region_name=region)
    
    report = {
        'generated_at': datetime.utcnow().isoformat(),
        'region': region,
        'findings': []
    }
    
    # Get all RDS instances
    paginator = rds_client.get_paginator('describe_db_instances')
    ec2_by_az = get_ec2_instances_by_az(ec2_client)
    
    for page in paginator.paginate():
        for db in page['DBInstances']:
            db_id = db['DBInstanceIdentifier']
            db_az = db['AvailabilityZone']
            db_endpoint = db.get('Endpoint', {}).get('Address', '')
            is_multi_az = db.get('MultiAZ', False)
            
            # Count EC2 instances in other AZs
            cross_az_instances = []
            for az, instances in ec2_by_az.items():
                if az != db_az:
                    cross_az_instances.extend(instances)
            
            total_gb_30d = estimate_cross_az_cost(cw_client, db_id)
            # Assume 50% of traffic crosses AZ (conservative estimate)
            estimated_cross_az_gb = total_gb_30d * 0.5
            estimated_monthly_cost = estimated_cross_az_gb * 0.02 * 2  # both directions
            
            if estimated_monthly_cost > 10:  # Only report if > $10/month
                finding = {
                    'db_instance': db_id,
                    'db_az': db_az,
                    'is_multi_az': is_multi_az,
                    'total_network_gb_30d': total_gb_30d,
                    'estimated_cross_az_gb_30d': estimated_cross_az_gb,
                    'estimated_monthly_cost_usd': round(estimated_monthly_cost, 2),
                    'ec2_instances_in_other_azs': len(cross_az_instances),
                    'recommendation': (
                        'Place read replica in each application AZ, '
                        'or configure topology-aware routing. '
                        f'Estimated savings: ${round(estimated_monthly_cost * 0.7, 2)}/month'
                    )
                }
                report['findings'].append(finding)
    
    # Sort by cost impact
    report['findings'].sort(key=lambda x: x['estimated_monthly_cost_usd'], reverse=True)
    
    total_recoverable = sum(f['estimated_monthly_cost_usd'] for f in report['findings'])
    report['total_estimated_recoverable_usd_monthly'] = round(total_recoverable, 2)
    
    print(json.dumps(report, indent=2))
    
    # In production: push to S3, SNS, or Slack webhook
    # s3_client.put_object(Bucket='finops-reports', Key=f'cross-az/{datetime.utcnow().date()}.json', Body=json.dumps(report))

if __name__ == '__main__':
    main()

    5.3 Snapshot Economics: Retention Policies That Cost Fortunes

    Automated snapshots are not free storage. They feel free because most cloud providers offer a limited window at no charge — AWS gives you free snapshot storage equal to the size of your database — but that boundary erodes almost immediately in practice.

    AWS Snapshot Pricing

    AWS charges $0.095/GB-month for snapshots stored in gp2/gp3 format beyond the provisioned database storage size. A 500GB database with 35 automated snapshots (the result of setting a 35-day retention period for a compliance requirement) stores approximately:

  • → First 500GB: **free** (offset by provisioned storage payment)
  • → Remaining ~16,500GB (33 snapshots × 500GB): 16,500 × $0.095 = **$1,567.50/month**

    • The relationship between retention period and cost is linear. Moving from a 35-day to a 7-day retention window for development databases reduces snapshot storage by 80%.

    Azure SQL Backup Pricing

    Azure charges for backup storage beyond the provisioned database size in the LRS tier at $0.0375/GB-month for locally redundant storage and $0.115/GB-month for geo-redundant storage. Point-in-time restore relies on full, differential, and log backups — the log backup volume grows proportionally with write throughput, making high-write databases disproportionately expensive to back up.

    GCP Cloud SQL Backup Pricing

    GCP charges $0.08/GB-month for automated backups. Unlike AWS, there is no free tier equivalent; you pay for the full snapshot volume from the first byte.

    Finding Your Current Snapshot Footprint

    -- PostgreSQL: Query pg_stat_user_tables for table size trends
-- This helps estimate how fast your database is growing (and thus snapshot sizes)
SELECT
    schemaname,
    tablename,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename)) AS total_size,
    pg_size_pretty(pg_relation_size(schemaname||'.'||tablename)) AS table_size,
    pg_size_pretty(pg_total_relation_size(schemaname||'.'||tablename) 
                   - pg_relation_size(schemaname||'.'||tablename)) AS index_size,
    n_live_tup AS live_rows,
    n_dead_tup AS dead_rows,
    ROUND(100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 2) AS dead_pct
FROM pg_stat_user_tables
WHERE pg_total_relation_size(schemaname||'.'||tablename) > 1073741824  -- > 1GB
ORDER BY pg_total_relation_size(schemaname||'.'||tablename) DESC;

-- SQL Server (Azure SQL): Estimate backup size from database file sizes
SELECT
    DB_NAME(database_id) AS database_name,
    type_desc,
    SUM(size * 8.0 / 1024) AS size_mb,
    SUM(size * 8.0 / 1024 / 1024) AS size_gb,
    -- Monthly backup cost estimate (LRS pricing)
    SUM(size * 8.0 / 1024 / 1024) * 0.0375 AS estimated_monthly_cost_lrs_usd,
    SUM(size * 8.0 / 1024 / 1024) * 0.115 AS estimated_monthly_cost_grs_usd
FROM sys.master_files
GROUP BY database_id, type_desc
ORDER BY size_gb DESC;

    Snapshot Optimization Strategy by Environment

    Blanket retention policies applied uniformly across production, staging, and development databases are the most expensive snapshot mistake. A tiered approach based on environment and compliance requirement cuts snapshot spend by 40–70% in most organizations:

    | Environment | Recommended Retention | Rationale |

    |---|---|---|

    | Production (regulated) | 35 days automated + 12 monthly manual | Compliance baseline |

    | Production (standard) | 7 days automated + 4 weekly manual | Covers most restore scenarios |

    | Staging | 3 days | Rarely restored, easily recreated |

    | Development | 1 day or snapshot-on-demand only | No production data, no restore need |

    | Data warehouse | Weekly only | Full rebuilds cheaper than snapshots |
