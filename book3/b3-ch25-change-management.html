Chapter 25: AI-Driven Change Management

Autonomous systems that can observe and decide are only as valuable as their ability to act — and acting on a production database system means managing change. Change management has historically been the domain of human judgment, CAB meetings, deployment windows, and rollback plans drafted at midnight. At the ADE level, the goal is to replace that human-in-the-loop bottleneck with a system that classifies risk, selects strategy, executes with precision, and rolls back autonomously when signals deteriorate. This chapter covers the full architecture: how an autonomous change management system ingests proposed changes, scores their risk against live telemetry, selects an execution strategy, instruments the rollout, and either completes or self-heals — without paging anyone at 2 AM.

---

Risk Scoring Engines: Classifying Change Before Execution

Every autonomous change pipeline begins with a classification gate. Before a DDL statement, configuration push, or schema migration touches a production instance, it must pass through a risk scoring engine that combines static analysis of the change itself with dynamic analysis of the current system state.

Static analysis examines the change in isolation. A CREATE INDEX CONCURRENTLY on a 50-million-row table is categorically different from the same operation on a 50-thousand-row table. An ALTER TABLE ... ADD COLUMN DEFAULT in PostgreSQL 11+ is trivially safe; the same statement on PostgreSQL 10 rewrites the table. A SQL Server ALTER TABLE ... ALTER COLUMN on a column with dependent indexed views is a blocking operation regardless of row count. The static analyzer must know these rules and apply them without human instruction.

Dynamic analysis examines what the system is doing right now. A change that scores "medium risk" at 3 AM may score "critical risk" at 2 PM when the instance is processing 40,000 transactions per second with 85% buffer cache utilization. The scoring engine must pull live metrics at decision time, not at submission time.

PostgreSQL:

-- View to surface real-time load context for risk scoring decisions
CREATE OR REPLACE VIEW autonomous_change.live_risk_context AS
SELECT
    (SELECT count(*) FROM pg_stat_activity WHERE state = 'active' AND wait_event_type = 'Lock') AS active_lock_waiters,
    (SELECT count(*) FROM pg_stat_activity WHERE state = 'active') AS active_connections,
    (SELECT sum(blks_hit)::float / nullif(sum(blks_hit) + sum(blks_read), 0)
     FROM pg_stat_database) AS buffer_cache_hit_ratio,
    (SELECT max(age(backend_xmin))
     FROM pg_stat_activity
     WHERE backend_xmin IS NOT NULL) AS max_xmin_age,
    (SELECT extract(epoch FROM now() - min(xact_start))
     FROM pg_stat_activity
     WHERE xact_start IS NOT NULL AND state = 'active') AS oldest_txn_seconds,
    (SELECT count(*) FROM pg_replication_slots WHERE active = false) AS inactive_repl_slots,
    (SELECT max(pg_wal_lsn_diff(sent_lsn, replay_lsn))
     FROM pg_stat_replication) AS max_replica_lag_bytes;

The scoring engine combines static weights with dynamic multipliers. A Python-based scorer might represent this as:

import psycopg2
import json
from dataclasses import dataclass
from enum import Enum
from typing import Optional

class RiskTier(Enum):
    LOW = "low"
    MEDIUM = "medium"
    HIGH = "high"
    CRITICAL = "critical"

@dataclass
class ChangeProposal:
    change_id: str
    change_type: str          # ddl, config, data_migration, index
    target_table: str
    estimated_row_count: int
    statement: str
    requires_lock: bool
    lock_level: str           # row, page, table, exclusive

def score_change_risk(proposal: ChangeProposal, db_conn) -> dict:
    # Static base score
    static_score = 0
    
    static_rules = {
        ("ddl", "exclusive"): 40,
        ("ddl", "table"): 30,
        ("index", "table"): 20,
        ("index", "row"): 5,
        ("config", "none"): 10,
        ("data_migration", "row"): 15,
    }
    static_score += static_rules.get((proposal.change_type, proposal.lock_level), 10)
    
    # Row count multiplier
    if proposal.estimated_row_count > 100_000_000:
        static_score *= 2.5
    elif proposal.estimated_row_count > 10_000_000:
        static_score *= 1.8
    elif proposal.estimated_row_count > 1_000_000:
        static_score *= 1.3
    
    # Dynamic context multipliers
    with db_conn.cursor() as cur:
        cur.execute("SELECT * FROM autonomous_change.live_risk_context")
        row = cur.fetchone()
        cols = [desc[0] for desc in cur.description]
        context = dict(zip(cols, row))
    
    dynamic_multiplier = 1.0
    
    if context["active_lock_waiters"] > 5:
        dynamic_multiplier += 0.4
    if context["buffer_cache_hit_ratio"] and context["buffer_cache_hit_ratio"] < 0.90:
        dynamic_multiplier += 0.3
    if context["oldest_txn_seconds"] and context["oldest_txn_seconds"] > 30:
        dynamic_multiplier += 0.5
    if context["max_replica_lag_bytes"] and context["max_replica_lag_bytes"] > 50_000_000:
        dynamic_multiplier += 0.6
    if context["inactive_repl_slots"] and context["inactive_repl_slots"] > 0:
        dynamic_multiplier += 0.8  # Inactive slots mean WAL accumulation risk
    
    final_score = static_score * dynamic_multiplier
    
    if final_score < 20:
        tier = RiskTier.LOW
    elif final_score < 45:
        tier = RiskTier.MEDIUM
    elif final_score < 75:
        tier = RiskTier.HIGH
    else:
        tier = RiskTier.CRITICAL
    
    return {
        "change_id": proposal.change_id,
        "static_score": static_score,
        "dynamic_multiplier": dynamic_multiplier,
        "final_score": final_score,
        "risk_tier": tier.value,
        "context_snapshot": context,
        "autonomous_action": _select_autonomous_action(tier, context)
    }

def _select_autonomous_action(tier: RiskTier, context: dict) -> str:
    if tier == RiskTier.LOW:
        return "execute_immediately"
    elif tier == RiskTier.MEDIUM:
        return "execute_with_monitoring"
    elif tier == RiskTier.HIGH:
        return "defer_to_maintenance_window"
    else:  # CRITICAL
        if context.get("active_lock_waiters", 0) > 10:
            return "block_and_escalate"
        return "block_and_defer_72h"

This scoring system is the gatekeeper. Every change proposal — whether generated by a CI/CD pipeline, a human DBA via a migration tool, or by the autonomous system itself as part of an optimization cycle — passes through this gate before touching production.

Autonomous Migration Execution: Online Schema Changes Without Human Oversight

Once a change clears the risk gate at an acceptable tier, the execution layer takes over. The most dangerous class of database change is DDL on large tables, and the autonomous execution layer must have first-class support for online schema change strategies — not as a fallback, but as the default path.

For PostgreSQL, the primary tools are pg_repack for table rewrites and CREATE INDEX CONCURRENTLY for index operations. Both can run without holding an exclusive lock for the full duration, but both require careful orchestration. CREATE INDEX CONCURRENTLY can fail partway through, leaving an invalid index that must be detected and cleaned up. pg_repack requires the pgstattuple extension and sufficient disk space.

For SQL Server, online index operations (WITH (ONLINE = ON)) and online column operations (WITH (ONLINE = ON) in SQL Server 2017+) serve similar purposes. The SQL Server Agent or an external orchestrator manages the execution, with Always On Availability Group lag monitoring integrated into the execution loop.

The autonomous execution controller wraps these operations in an instrumented harness:

PostgreSQL:

-- Function to capture metrics snapshot at execution boundaries
CREATE OR REPLACE FUNCTION autonomous_change.capture_metrics_snapshot()
RETURNS JSONB
LANGUAGE plpgsql AS $$
DECLARE
    snap JSONB;
BEGIN
    SELECT jsonb_build_object(
        'timestamp', now(),
        'tps', (SELECT sum(xact_commit + xact_rollback) FROM pg_stat_database WHERE datname = current_database()),
        'active_connections', (SELECT count(*) FROM pg_stat_activity WHERE state = 'active'),
        'cache_hit_ratio', (SELECT round((sum(blks_hit)::numeric / nullif(sum(blks_hit) + sum(blks_read), 0)) * 100, 2) FROM pg_stat_database),
        'replication_lag_bytes', (SELECT max(pg_wal_lsn_diff(sent_lsn, replay_lsn)) FROM pg_stat_replication),
        'longest_query_ms', (SELECT round(max(extract(epoch FROM (now() - query_start))) * 1000) FROM pg_stat_activity WHERE state = 'active')
    ) INTO snap;
    RETURN snap;
END;
$$;

The execution harness runs the change inside a monitoring loop. Every 10 seconds during a long-running operation, it samples progress, checks replica lag, checks for lock accumulation, and compares against pre-execution baselines. If any metric degrades beyond a configured threshold, the operation is aborted and the abort reason is logged.

For batched data migrations — moving data in chunks, backfilling columns, or archiving records — the autonomous executor uses a self-throttling loop that adjusts batch size based on observed write latency:

import time
import psycopg2
from typing import Callable

def autonomous_batched_migration(
    conn_string: str,
    batch_query: str,          # Must accept %(batch_size)s and %(last_id)s
    count_query: str,
    max_lag_bytes: int = 100_000_000,
    target_batch_duration_ms: int = 500,
    initial_batch_size: int = 1000,
    max_batch_size: int = 50_000,
    min_batch_size: int = 100,
    execution_id: str = None
):
    """
    Self-throttling batched migration executor.
    Adjusts batch size dynamically based on actual execution duration
    and replica lag, ensuring the migration stays within safe thresholds.
    """
    conn = psycopg2.connect(conn_string)
    conn.autocommit = True
    
    batch_size = initial_batch_size
    last_id = 0
    total_rows = 0
    consecutive_lag_violations = 0
    
    with conn.cursor() as cur:
        cur.execute(count_query)
        total_estimated = cur.fetchone()[0]
    
    print(f"[{execution_id}] Starting migration. Estimated rows: {total_estimated}")
    
    while True:
        # Check replica lag before each batch
        with conn.cursor() as cur:
            cur.execute("""
                SELECT COALESCE(max(pg_wal_lsn_diff(sent_lsn, replay_lsn)), 0)
                FROM pg_stat_replication
            """)
            current_lag = cur.fetchone()[0]
        
        if current_lag > max_lag_bytes:
            consecutive_lag_violations += 1
            wait_seconds = min(2 ** consecutive_lag_violations, 60)
            print(f"[{execution_id}] Replica lag {current_lag:,} bytes exceeds threshold. "
                  f"Backing off {wait_seconds}s (violation #{consecutive_lag_violations})")
            time.sleep(wait_seconds)
            # Shrink batch size aggressively when lag builds
            batch_size = max(min_batch_size, batch_size // 2)
            continue
        else:
            consecutive_lag_violations = 0
        
        # Execute batch and measure duration
        batch_start = time.monotonic()
        with conn.cursor() as cur:
            cur.execute(batch_query, {"batch_size": batch_size, "last_id": last_id})
            rows_processed = cur.rowcount
        batch_duration_ms = (time.monotonic() - batch_start) * 1000
        
        if rows_processed == 0:
            print(f"[{execution_id}] Migration complete. Total rows processed: {total_rows}")
            break
        
        total_rows += rows_processed
        last_id += rows_processed  # Assumes sequential ID or adjust per schema
        
        # Self-tuning batch size based on duration feedback
        if batch_duration_ms > 0:
            adjustment_ratio = target_batch_duration_ms / batch_duration_ms
            new_batch_size = int(batch_size * adjustment_ratio)
            batch_size = max(min_batch_size, min(max_batch_size, new_batch_size))
        
        progress_pct = round(total_rows / total_estimated * 100, 1) if total_estimated > 0 else 0
        print(f"[{execution_id}] Batch complete: {rows_processed} rows in {batch_duration_ms:.0f}ms. "
              f"New batch size: {batch_size}. Progress: {progress_pct}%")
        
        # Brief yield between batches to reduce contention
        time.sleep(0.05)
    
    conn.close()

This pattern means that a 500-million-row backfill migration runs without any human watching it. If the replica falls behind, the system slows down. If the replica catches up and the database is underloaded, the system speeds up. The batch size converges on the fastest rate that keeps lag within bounds.

AI-Assisted Change Plan Generation: From Intent to Executable Strategy

Risk scoring and execution harnesses handle the mechanics. But in a fully autonomous system, the change plan itself should not always originate from a human writing SQL. The system needs the ability to translate high-level intents — "this query is slow because it lacks an appropriate index" or "this table is bloated by 60% and needs a repack" — into concrete, sequenced, risk-scored change plans.

This is where large language model integration moves from a developer convenience into production infrastructure. The autonomous optimizer identifies a problem, describes it in structured natural language, and queries an LLM API to generate a candidate change plan. That plan is then parsed, statically analyzed by the risk scorer, and submitted through the same execution pipeline as any human-authored migration.

The integration requires strict guardrails. The LLM output is never executed directly. Every generated statement is passed through a SQL parser, validated against schema, compared against a catalog of known-dangerous patterns, and scored. Only then does it enter the execution queue.

import openai
import sqlglot
import re
from typing import Optional

DANGEROUS_PATTERNS = [
    r'\bDROP\s+TABLE\b',
    r'\bTRUNCATE\b',
    r'\bDELETE\s+FROM\b(?!.*\bWHERE\b)',  # DELETE without WHERE
    r'\bUPDATE\b(?!.*\bWHERE\b)',          # UPDATE without WHERE
    r'\bDROP\s+DATABASE\b',
    r'\bDROP\s+SCHEMA\b',
]

def generate_change_plan_from_intent(
    intent: str,
    schema_context: str,
    db_platform: str,  # "postgresql" or "sqlserver"
    model: str = "gpt-4o"
) -> Optional[dict]:
    """
    Generates a structured change plan from a natural-language optimization intent.
    Output is a JSON object with statements, sequencing, and reversibility metadata.
    All output is validated before entering the execution pipeline.
    """
    
    system_prompt = f"""You are a database change plan generator for {db_platform}.
    
    Given an optimization intent and schema context, produce a JSON change plan with this exact structure:
    {{
      "intent_summary": "...",
      "statements": [
        {{
          "sequence": 1,
          "statement": "...",
          "change_type": "ddl|index|config|data_migration",
          "lock_level": "none|row|page|table|exclusive",
          "reversible": true|false,
          "reversal_statement": "..." or null,
          "online_safe": true|false,
          "estimated_duration_hint": "seconds|minutes|hours"
        }}
      ],
      "requires_maintenance_window": true|false,
      "dependencies": []
    }}
    
    Rules:
    - Prefer CONCURRENT or ONLINE operations wherever the platform supports them
    - Always include reversal_statement for DDL that has a clean inverse
    - Never generate DROP TABLE, TRUNCATE, or DELETE without WHERE
    - For {db_platform}, use correct syntax precisely
    - Output ONLY valid JSON, no explanation text"""
    
    response = openai.chat.completions.create(
        model=model,
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": f"Intent: {intent}\n\nSchema context:\n{schema_context}"}
        ],
        temperature=0.1,  # Low temperature for deterministic, precise SQL output
        response_format={"type": "json_object"}
    )
    
    raw_plan = response.choices[0].message.content
    
    try:
        import json
        plan = json.loads(raw_plan)
    except json.JSONDecodeError as e:
        print(f"LLM returned unparseable JSON: {e}")
        return None
    
    # Validate every statement before returning
    validated_statements = []
    for stmt_obj in plan.get("statements", []):
        statement = stmt_obj.get("statement", "")
        
        # Pattern-based safety check
        for pattern in DANGEROUS_PATTERNS:
            if re.search(pattern, statement, re.IGNORECASE):
                print(f"BLOCKED: Dangerous pattern detected in generated statement: {statement[:100]}")
                return None
        
        # Syntax validation via sqlglot
        try:
            dialect = "postgres" if db_platform == "postgresql" else "tsql"
            parsed = sqlglot.parse(statement, dialect=dialect)
            if not parsed:
                print(f"BLOCKED: Statement failed SQL parsing: {statement[:100]}")
                return None
        except sqlglot.errors.ParseError as e:
            print(f"BLOCKED: SQL parse error in generated statement: {e}")
            return None
        
        validated_statements.append(stmt_obj)
    
    plan["statements"] = validated_statements
    plan["generated_by"] = "llm"
    plan["validated"] = True
    
    return plan

The schema context passed to this function is not a raw dump of all DDL. It is a curated, minimal representation: the target table's column definitions, existing indexes, approximate row count, and the specific query or metric that triggered the optimization intent. Sending a full schema dump to an LLM API is both expensive and produces worse results than targeted context.

Rollback Orchestration: Autonomous Recovery When Deployments Go Wrong

A change management system without autonomous rollback is not autonomous — it is merely automated. The difference is consequential. Automation runs a script. Autonomy decides when the script has produced a bad outcome and reverses it without being told to.

Autonomous rollback requires three things: a pre-execution baseline, continuous signal monitoring during execution, and a reversibility map that knows how to undo every change type the system can apply.
