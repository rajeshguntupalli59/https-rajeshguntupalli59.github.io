Chapter 12: Multi-Tenant Vector Storage in PostgreSQL

Multi-tenancy is one of the oldest problems in database engineering, and it doesn't get simpler when you add vectors to the mix. If you're building a SaaS product where every customer gets their own AI-powered search, a platform where multiple teams share infrastructure for semantic retrieval, or an enterprise system where business units need data isolation, you need a coherent strategy for partitioning vector data. This chapter walks through the three canonical approaches to multi-tenant vector storage in PostgreSQL with pgvector — row-level isolation, schema-level isolation, and database-level isolation — and gives you the tools to choose between them based on your scale, compliance requirements, and operational constraints. Beyond the data model, you'll also work through index management across tenants, query routing, and the performance cliffs that catch engineers off guard in production.

---

12.1 The Three Models of Multi-Tenancy (and Where Each Breaks Down)

Before writing a single line of SQL, you need to understand the fundamental tradeoff: shared infrastructure reduces cost and operational overhead, but increases blast radius and isolation risk. Vector workloads amplify this tradeoff because index structures are not row-addressable — an HNSW index built over millions of rows cannot be surgically partitioned the way a B-tree on a primary key can.

Row-Level Isolation puts all tenants in a single table with a tenant_id column. It is the simplest model and maps directly to how most DBAs handle multi-tenancy in OLTP systems. The problem is that pgvector's approximate nearest neighbor (ANN) indexes — HNSW and IVFFlat — are built over the entire table. When you query with a WHERE tenant_id = $1 filter, the planner has two options: scan the index and post-filter, or skip the index and do a sequential scan. For a tenant with 1,000 rows in a table of 10 million, neither option is fast. The index scan produces thousands of candidates before the filter reduces them to the tenant's actual result set. The sequential scan is simply slow. This is the "tenant size disparity" problem, and it is the primary reason row-level isolation fails at scale.

Schema-Level Isolation gives each tenant their own schema within a single PostgreSQL database. Each schema contains its own embeddings table and its own HNSW index. This sidesteps the index contamination problem entirely — each tenant's index is built only over their own data. The cost is operational complexity: adding a tenant means executing DDL, managing schema-scoped connections, and ensuring your migration tooling handles hundreds or thousands of schemas without timing out. At roughly 500–1,000 schemas, operational overhead becomes a real concern; beyond 5,000 schemas, PostgreSQL's catalog can become a performance bottleneck during connection and query planning.

Database-Level Isolation gives each tenant a separate PostgreSQL database. This provides the strongest isolation — separate connection pools, separate WAL streams, separate backup targets — but multiplies your operational surface area by the number of tenants. It's appropriate for enterprise SaaS where each customer is a large organization with their own compliance requirements, not for consumer applications with thousands of small tenants.

The decision matrix looks like this in practice:

| Model | Tenants | Tenant Size | Isolation Requirement | Recommended |

|---|---|---|---|---|

| Row-level | < 100 | Roughly equal | Low | Prototyping, small SaaS |

| Schema-level | 100–5,000 | Variable | Medium | Most production SaaS |

| Database-level | < 500 | Large | High (SOC 2, HIPAA) | Enterprise, regulated |

Most production systems end up in the schema-level model. The rest of this chapter focuses there, with coverage of row-level patterns where they remain applicable.

12.2 Implementing Row-Level Vector Isolation

Row-level isolation is worth understanding thoroughly, even if you'll eventually outgrow it, because it's the right starting point for most systems and the performance problems are instructive.

The baseline schema for a row-level multi-tenant vector store looks like this:

-- PostgreSQL / pgvector
CREATE EXTENSION IF NOT EXISTS vector;

-- Core document store with tenant partitioning
CREATE TABLE tenant_embeddings (
    id          BIGSERIAL PRIMARY KEY,
    tenant_id   UUID NOT NULL,
    doc_id      TEXT NOT NULL,
    content     TEXT NOT NULL,
    embedding   vector(1536),        -- OpenAI text-embedding-3-small dimensions
    metadata    JSONB DEFAULT '{}',
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE (tenant_id, doc_id)
);

-- Composite index for filtering: tenant_id first, then vector search
-- Note: this is a regular B-tree index, not a vector index
CREATE INDEX idx_tenant_embeddings_tenant
    ON tenant_embeddings (tenant_id, created_at DESC);

-- HNSW index over the full table
-- m=16, ef_construction=64 are conservative defaults for mixed workloads
CREATE INDEX idx_tenant_embeddings_hnsw
    ON tenant_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- Row-level security for application-layer enforcement
ALTER TABLE tenant_embeddings ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation_policy ON tenant_embeddings
    USING (tenant_id = current_setting('app.current_tenant_id')::UUID);

The RLS policy requires the application to set app.current_tenant_id before executing queries. In Python using psycopg2:

# Python
import psycopg2
import numpy as np
from openai import OpenAI

openai_client = OpenAI()

def get_connection(dsn: str):
    conn = psycopg2.connect(dsn)
    conn.autocommit = False
    return conn

def set_tenant_context(conn, tenant_id: str):
    """Set the RLS context for the current connection."""
    with conn.cursor() as cur:
        cur.execute(
            "SELECT set_config('app.current_tenant_id', %s, FALSE)",
            (tenant_id,)
        )

def embed_text(text: str) -> list[float]:
    response = openai_client.embeddings.create(
        model="text-embedding-3-small",
        input=text
    )
    return response.data[0].embedding

def insert_document(conn, tenant_id: str, doc_id: str, content: str, metadata: dict):
    embedding = embed_text(content)
    embedding_str = "[" + ",".join(str(x) for x in embedding) + "]"

    with conn.cursor() as cur:
        set_tenant_context(conn, tenant_id)
        cur.execute("""
            INSERT INTO tenant_embeddings (tenant_id, doc_id, content, embedding, metadata)
            VALUES (%s, %s, %s, %s::vector, %s)
            ON CONFLICT (tenant_id, doc_id) DO UPDATE
                SET content   = EXCLUDED.content,
                    embedding = EXCLUDED.embedding,
                    metadata  = EXCLUDED.metadata
        """, (tenant_id, doc_id, content, embedding_str, psycopg2.extras.Json(metadata)))
    conn.commit()

def semantic_search(conn, tenant_id: str, query: str, top_k: int = 10):
    """
    ANN search scoped to a single tenant.
    The WHERE clause on tenant_id will post-filter HNSW results.
    For small tenants in large tables, set_limit may need adjustment.
    """
    query_embedding = embed_text(query)
    embedding_str = "[" + ",".join(str(x) for x in query_embedding) + "]"

    with conn.cursor() as cur:
        set_tenant_context(conn, tenant_id)
        # hnsw.ef_search controls recall vs speed tradeoff at query time
        cur.execute("SET hnsw.ef_search = 100")
        cur.execute("""
            SELECT
                doc_id,
                content,
                metadata,
                1 - (embedding <=> %s::vector) AS cosine_similarity
            FROM tenant_embeddings
            WHERE tenant_id = %s
            ORDER BY embedding <=> %s::vector
            LIMIT %s
        """, (embedding_str, tenant_id, embedding_str, top_k))
        return cur.fetchall()

One specific behavior to understand: when pgvector executes an HNSW search with a WHERE tenant_id = $1 predicate, it scans the HNSW graph and collects ef_search candidates, then applies the filter. If your tenant has 500 rows in a table of 2 million, the probability that any given HNSW candidate passes the filter is 0.025%. The engine may return fewer than top_k results because so few candidates pass. Increasing hnsw.ef_search to 400 or 800 improves recall but increases latency proportionally. This is the wall you hit with row-level isolation.

The mitigation, short of switching to schema-level isolation, is partial indexes. If tenant access patterns are known ahead of time, you can create a dedicated HNSW index per tenant:

-- PostgreSQL / pgvector
-- Partial HNSW index for a specific high-volume tenant
-- Only viable for tenants with > ~50,000 rows (index overhead is otherwise not justified)
CREATE INDEX idx_embeddings_tenant_acme
    ON tenant_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64)
    WHERE tenant_id = 'a1b2c3d4-e5f6-7890-abcd-ef1234567890';

-- Check that the planner uses the partial index
EXPLAIN (ANALYZE, BUFFERS)
SELECT doc_id, embedding <=> '[0.1, 0.2, ...]'::vector AS distance
FROM tenant_embeddings
WHERE tenant_id = 'a1b2c3d4-e5f6-7890-abcd-ef1234567890'
ORDER BY embedding <=> '[0.1, 0.2, ...]'::vector
LIMIT 10;

Partial indexes work well for your top 10–20 largest tenants. Beyond that, the schema-level model is a cleaner solution.

12.3 Schema-Per-Tenant Architecture

Schema-level isolation is where most production vector SaaS platforms land. Each tenant gets a dedicated schema, and within that schema you have full DDL freedom: indexes tuned to that tenant's vector count, retention policies, custom metadata columns.

The first engineering problem is provisioning automation. You need to create schemas, tables, and indexes reliably, idempotently, and fast enough that a new tenant can be onboarded within seconds.

# Python
import psycopg2
from psycopg2 import sql
import logging

logger = logging.getLogger(__name__)

TENANT_SCHEMA_DDL = """
    CREATE SCHEMA IF NOT EXISTS {schema};

    CREATE TABLE IF NOT EXISTS {schema}.embeddings (
        id          BIGSERIAL PRIMARY KEY,
        doc_id      TEXT NOT NULL UNIQUE,
        content     TEXT NOT NULL,
        embedding   vector(1536),
        metadata    JSONB DEFAULT '{{}}',
        created_at  TIMESTAMPTZ DEFAULT NOW()
    );

    -- IVFFlat is preferred at smaller scales (< 1M rows per tenant)
    -- because it has lower memory overhead than HNSW and builds faster.
    -- Switch to HNSW when tenant exceeds ~500k rows.
    CREATE INDEX IF NOT EXISTS embeddings_ivfflat_{tenant_slug}
        ON {schema}.embeddings
        USING ivfflat (embedding vector_cosine_ops)
        WITH (lists = 100);

    COMMENT ON SCHEMA {schema} IS 'Tenant: {tenant_id}';
"""

def sanitize_schema_name(tenant_id: str) -> str:
    """
    Convert a tenant UUID to a safe schema name.
    PostgreSQL identifiers must start with a letter or underscore.
    """
    safe = tenant_id.replace("-", "_")
    return f"t_{safe}"

def provision_tenant_schema(conn, tenant_id: str) -> str:
    """
    Create schema and supporting objects for a new tenant.
    Returns the schema name on success.
    Idempotent — safe to call multiple times.
    """
    schema_name = sanitize_schema_name(tenant_id)

    try:
        with conn.cursor() as cur:
            # Use psycopg2.sql for safe identifier interpolation
            # String formatting is acceptable here only for the DDL block
            # where we've already sanitized the identifier above.
            ddl = TENANT_SCHEMA_DDL.format(
                schema=schema_name,
                tenant_id=tenant_id,
                tenant_slug=schema_name
            )
            cur.execute(ddl)
        conn.commit()
        logger.info(f"Provisioned schema {schema_name} for tenant {tenant_id}")
        return schema_name
    except Exception as e:
        conn.rollback()
        logger.error(f"Failed to provision schema for tenant {tenant_id}: {e}")
        raise

def deprovision_tenant_schema(conn, tenant_id: str, dry_run: bool = True) -> str:
    """
    Remove all data for a tenant. Irreversible.
    Always run with dry_run=True first in production.
    """
    schema_name = sanitize_schema_name(tenant_id)
    drop_stmt = sql.SQL("DROP SCHEMA IF EXISTS {} CASCADE").format(
        sql.Identifier(schema_name)
    )

    if dry_run:
        logger.info(f"[DRY RUN] Would execute: {drop_stmt.as_string(conn)}")
        return schema_name

    with conn.cursor() as cur:
        cur.execute(drop_stmt)
    conn.commit()
    logger.info(f"Deprovisioned schema {schema_name}")
    return schema_name

With schemas provisioned, the query path needs to route each request to the correct schema. Connection pools complicate this: a connection checked out from PgBouncer has no inherent tenant context, so you must set search_path explicitly on every query or use session-scoped configuration.

# Python
from contextlib import contextmanager
import psycopg2.pool

# Transaction-mode pooling (PgBouncer) requires setting search_path per query.
# Session-mode pooling allows SET search_path once per connection checkout,
# but connections cannot be reused across tenants without resetting.

class TenantVectorStore:
    def __init__(self, pool: psycopg2.pool.ThreadedConnectionPool):
        self.pool = pool

    @contextmanager
    def tenant_connection(self, tenant_id: str):
        """
        Context manager that yields a connection scoped to the tenant schema.
        Resets search_path on release to prevent cross-tenant leakage.
        """
        schema_name = sanitize_schema_name(tenant_id)
        conn = self.pool.getconn()
        try:
            with conn.cursor() as cur:
                # Set search_path for this transaction block.
                # The 'public' fallback gives access to pgvector functions.
                cur.execute(
                    sql.SQL("SET LOCAL search_path TO {}, public").format(
                        sql.Identifier(schema_name)
                    )
                )
            yield conn
        finally:
            conn.reset()    # Clears SET LOCAL changes
            self.pool.putconn(conn)

    def search(self, tenant_id: str, query_embedding: list[float], top_k: int = 10):
        embedding_str = "[" + ",".join(str(x) for x in query_embedding) + "]"

        with self.tenant_connection(tenant_id) as conn:
            with conn.cursor() as cur:
                cur.execute("""
                    SELECT
                        doc_id,
                        content,
                        metadata,
                        1 - (embedding <=> %s::vector) AS score
                    FROM embeddings
                    ORDER BY embedding <=> %s::vector
                    LIMIT %s
                """, (embedding_str, embedding_str, top_k))
                return cur.fetchall()

    def upsert(self, tenant_id: str, doc_id: str, content: str,
               embedding: list[float], metadata: dict):
        embedding_str = "[" + ",".join(str(x) for x in embedding) + "]"

        with self.tenant_connection(tenant_id) as conn:
            with conn.cursor() as cur:
                cur.execute("""
                    INSERT INTO embeddings (doc_id, content, embedding, metadata)
                    VALUES (%s, %s, %s::vector, %s)
                    ON CONFLICT (doc_id) DO UPDATE
                        SET content   = EXCLUDED.content,
                            embedding = EXCLUDED.embedding,
                            metadata  = EXCLUDED.metadata
                """, (doc_id, content, embedding_str,
                      psycopg2.extras.Json(metadata)))
            conn.commit()

The SET LOCAL search_path approach is important: SET LOCAL applies only within the current transaction and is automatically rolled back when the transaction ends or the connection is reset. This prevents a schema from bleeding across connection pool boundaries.

12.4 Index Management at Tenant Scale

The most underestimated operational challenge in schema-per-tenant architectures is index lifecycle management. When you have 1,000 tenants, each with an IVFFlat or HNSW index, you have 1,000 separate index structures that need to be monitored, rebuilt when stale, and upgraded when you change embedding models.

When to switch index types: IVFFlat is appropriate up to approximately 500,000 vectors per tenant. The rule of thumb is lists = sqrt(row_count), so a 100,000-row tenant should have lists = ~316. Beyond 500,000 rows, HNSW provides better recall at comparable query latency because IVFFlat's probe-based search degrades when the query vector's nearest list doesn't align well with its actual neighbors.

Rebuilding indexes after bulk loads: Both IVFFlat and HNSW are static structures that don't automatically rebalance. After a bulk import that increases a tenant's row count by more than 20–30%, the index becomes stale and recall drops. You need a process to detect this and schedule rebuilds.

-- PostgreSQL / pgvector
-- Monitoring view: index staleness across all tenant schemas
-- Run this periodically to identify schemas needing index maintenance

CREATE OR REPLACE VIEW public.tenant_index_health AS
SELECT
    schemaname,
    tablename,
    n_live_tup                                    AS live_rows,
    n_dead_tup                                    AS dead_rows,
    last_vacuum,
    last_analyze,
    ROUND(n_dead_tup::NUMERIC / NULLIF(n_live_tup, 0) * 100, 2) AS dead_pct,
    pg_size_pretty(pg_total_relation_size(
        schemaname || '.' || tablename
    ))                                            AS total_size
FROM pg_stat_user_tables
WHERE tablename = 'embeddings'
  AND schemaname LIKE 't_%'    -- matches our tenant schema naming convention
ORDER BY live_rows DESC;

-- Find tenants whose index needs rebuilding:
-- Heuristic: more than 25% dead tuples, or live rows have grown > 2x since last analyze
SELECT
    schemaname,
    live_rows,
    dead_pct,
    last_analyze,
    CASE
        WHEN dead_pct > 25                        THEN 'REBUILD_URGENT'
        WHEN last_analyze < NOW() - INTERVAL '7 days'
             AND live_rows > 10000                THEN 'REBUILD_SCHEDULED'
        ELSE 'OK'
    END AS index_status
FROM public.tenant_index_health
WHERE dead_pct > 10
   OR last_analyze < NOW() - INTERVAL '7 days'
ORDER BY dead_pct DESC, live_rows DESC;

When a tenant needs an index rebuild, you want to do it without locking out queries. REINDEX CONCURRENTLY is your friend, but it requires PostgreSQL 12+ and cannot run inside a transaction block:

# shell
# Rebuild HNSW index concurrently for a specific tenant schema
# Run as the application database user with sufficient privileges
# This takes 2-10 minutes for a 500k-row tenant on typical cloud hardware

TENANT_SCHEMA="t_a1b2c3d4_e5f6_7890_abcd_ef1234567890"

psql "$DATABASE_URL" -c "
    REINDEX INDEX CONCURRENTLY
    ${TENANT_SCHEMA}.embeddings_ivfflat_${TENANT_SCHEMA};
"

# After reindex, update statistics so the planner has accurate cost estimates
psql "$DATABASE_URL" -c "
    ANALYZE ${TENANT_SCHEMA}.embeddings;
"

# Verify the index is healthy
psql "$DATABASE_URL" -c "
    SELECT
        indexname,
        pg_size_pretty(pg_relation_size(indexname::regclass)) AS index_size,
        idx_scan,
        idx_tup_read,
        idx_tup_fetch
    FROM pg_stat_user_indexes
    WHERE schemaname = '${TENANT_SCHEMA}'
      AND tablename = 'embeddings';
"
