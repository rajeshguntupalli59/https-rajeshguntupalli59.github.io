Chapter 6: Building Your First Database Cost Dashboard

Organizations that build a centralized database cost dashboard reduce their monthly cloud spend by an average of 23% within the first 90 days — not because the dashboard itself saves money, but because visibility forces accountability. Before a dashboard exists, cost anomalies hide for weeks inside undifferentiated billing line items. After one exists, a $14,000 spike in RDS storage costs gets caught on day two. This chapter walks through building a production-ready database cost dashboard from scratch, covering data collection across AWS, Azure, and GCP, metric normalization, automated alerting, and the specific queries that surface actionable cost intelligence. By the end, you will have a working system that your DBAs can monitor daily and your CTO can review weekly without needing a cloud billing certification to interpret the numbers.

---

6.1 Understanding What Your Database Bill Actually Contains

Most engineers look at a cloud database bill and see a single number. The actual bill is a composition of six to twelve distinct cost dimensions, each of which responds to different optimization levers. Before building any dashboard, you need a precise accounting of what you are measuring.

For AWS RDS, the billable dimensions are:

  • → **Instance hours** — the per-hour rate for your DB instance class, multiplied by uptime
  • → **Multi-AZ surcharge** — typically 2× the single-AZ instance rate
  • → **Storage** — per GB-month for gp2, gp3, io1, or Aurora storage
  • → **I/O requests** — charged separately on gp2 and Aurora (not gp3)
  • → **Backup storage** — free up to 100% of provisioned storage, then $0.095/GB-month
  • → **Data transfer** — egress charges when data leaves the AWS region or crosses AZ boundaries
  • → **Snapshot storage** — manual snapshots billed at the same rate as backup storage
  • → **Proxy hours** — RDS Proxy billed per vCPU of the associated instance

    • For Azure SQL, the dimensions are:

  • → **vCore or DTU hours** — General Purpose, Business Critical, or Hyperscale tier
  • → **Storage** — $0.115/GB-month for General Purpose, included up to 32 GB
  • → **Backup storage LRS/GRS** — $0.05–$0.10/GB-month depending on redundancy tier
  • → **Long-term retention** — separate pricing for blobs in Azure Backup vault
  • → **Elastic Pool overhead** — pool-level eDTUs or vCores shared across databases

    • For Google Cloud SQL and AlloyDB, the dimensions are:

  • → **Instance vCPU and memory** — billed per second with sustained use discounts applied automatically
  • → **SSD storage** — $0.17/GB-month for Cloud SQL, $0.29/GB-month for AlloyDB
  • → **HDD storage** — $0.09/GB-month for Cloud SQL (not available for AlloyDB)
  • → **Egress** — billed when data crosses regions or exits Google's network
  • → **Backup storage** — $0.08/GB-month for Cloud SQL automated backups

    • The math that surprises most teams: a db.r6g.2xlarge RDS instance in us-east-1 running 24/7 on-demand costs $0.48/hour × 730 hours = $350.40/month for the instance alone. Add Multi-AZ ($700.80), 1TB gp2 storage ($115), backup retention across 7 days (~$95), and a conservative data transfer estimate ($40), and a single "medium" database costs approximately $950/month before you have written a single query against it. A reserved instance purchase for that same class drops the hourly rate to roughly $0.286 for a 1-year no-upfront commitment — saving $140/month per instance. With ten such instances, that is $1,680/year in savings from one configuration change.

    Your dashboard must expose all of these dimensions independently. An aggregated "database cost" number is not actionable. A breakdown that shows backup storage growing 15% month-over-month while instance costs remain flat tells a DBA to look at retention policy configuration immediately.

    6.2 Collecting Cost Data: AWS, Azure, and GCP APIs

    A database cost dashboard is only as good as the data feeding it. This section covers the exact API calls, queries, and export configurations needed to pull granular cost data from each cloud provider into a centralized store — in practice, a PostgreSQL table works well for this, and the irony of using a database to track database costs is not lost on anyone who has done it.

    #### AWS Cost Explorer via CLI and API

    AWS Cost Explorer exposes a rich API that returns cost breakdowns by service, usage type, tag, and linked account. The following bash script pulls the last 30 days of RDS costs grouped by usage type and writes them to a local JSON file for ingestion.

    #!/bin/bash
# collect_aws_rds_costs.sh
# Pulls RDS cost breakdown from AWS Cost Explorer for the trailing 30 days
# Schedule: daily via cron or EventBridge Scheduler
# Requirements: AWS CLI v2, IAM permissions for ce:GetCostAndUsage

START_DATE=$(date -d "30 days ago" +%Y-%m-%d 2>/dev/null || date -v-30d +%Y-%m-%d)
END_DATE=$(date +%Y-%m-%d)
OUTPUT_FILE="/var/log/dbfinops/rds_costs_$(date +%Y%m%d).json"

mkdir -p /var/log/dbfinops

echo "Collecting RDS costs from $START_DATE to $END_DATE"

aws ce get-cost-and-usage \
  --time-period Start="$START_DATE",End="$END_DATE" \
  --granularity DAILY \
  --filter '{
    "Dimensions": {
      "Key": "SERVICE",
      "Values": ["Amazon Relational Database Service"],
      "MatchOptions": ["EQUALS"]
    }
  }' \
  --group-by '[
    {"Type": "DIMENSION", "Key": "USAGE_TYPE"},
    {"Type": "TAG", "Key": "Environment"},
    {"Type": "TAG", "Key": "Team"}
  ]' \
  --metrics "BlendedCost" "UnblendedCost" "UsageQuantity" \
  --output json > "$OUTPUT_FILE"

echo "Cost data written to $OUTPUT_FILE"

# Also collect Reserved Instance utilization
aws ce get-reservation-utilization \
  --time-period Start="$START_DATE",End="$END_DATE" \
  --granularity MONTHLY \
  --filter '{
    "Dimensions": {
      "Key": "SERVICE",
      "Values": ["Amazon Relational Database Service"],
      "MatchOptions": ["EQUALS"]
    }
  }' \
  --output json > "/var/log/dbfinops/ri_utilization_$(date +%Y%m%d).json"

echo "RI utilization data collected"

# Collect Savings Plans coverage
aws ce get-savings-plans-coverage \
  --time-period Start="$START_DATE",End="$END_DATE" \
  --granularity MONTHLY \
  --output json > "/var/log/dbfinops/sp_coverage_$(date +%Y%m%d).json"

echo "Collection complete. Files in /var/log/dbfinops/"

    #### Azure Cost Management Export

    Azure requires a slightly different approach. The Cost Management API is powerful but verbose. The recommended production pattern is to configure a scheduled export from Azure Cost Management to a Storage Account blob, then process the CSV from there.

    #!/bin/bash
# configure_azure_cost_export.sh
# Creates a recurring Azure Cost Management export for SQL database costs
# Requires: az CLI, Contributor role on subscription or resource group

SUBSCRIPTION_ID="your-subscription-id"
RESOURCE_GROUP="rg-finops-monitoring"
STORAGE_ACCOUNT="sadbfinops$(echo $RANDOM | md5sum | head -c 8)"
CONTAINER_NAME="cost-exports"
EXPORT_NAME="SQLDatabaseDailyCosts"
LOCATION="eastus"

echo "Creating storage account for cost exports..."
az storage account create \
  --name "$STORAGE_ACCOUNT" \
  --resource-group "$RESOURCE_GROUP" \
  --location "$LOCATION" \
  --sku Standard_LRS \
  --kind StorageV2

az storage container create \
  --name "$CONTAINER_NAME" \
  --account-name "$STORAGE_ACCOUNT"

STORAGE_ID=$(az storage account show \
  --name "$STORAGE_ACCOUNT" \
  --resource-group "$RESOURCE_GROUP" \
  --query id --output tsv)

echo "Creating daily cost export..."
az costmanagement export create \
  --name "$EXPORT_NAME" \
  --type ActualCost \
  --scope "subscriptions/$SUBSCRIPTION_ID" \
  --storage-account-id "$STORAGE_ID" \
  --storage-container "$CONTAINER_NAME" \
  --storage-directory "sql-costs" \
  --recurrence Daily \
  --recurrence-period \
    from="$(date +%Y-%m-%dT00:00:00Z)" \
    to="$(date -d '+1 year' +%Y-%m-%dT00:00:00Z 2>/dev/null || date -v+1y +%Y-%m-%dT00:00:00Z)" \
  --dataset-configuration columns="Date,SubscriptionId,ResourceId,ResourceType,ResourceGroup,MeterCategory,MeterSubCategory,Quantity,UnitPrice,Cost,Tags"

echo "Export configured. Data will appear in blob storage daily."
echo "Storage account: $STORAGE_ACCOUNT"
echo "Container: $CONTAINER_NAME"

# Trigger an immediate export run
az costmanagement export run \
  --name "$EXPORT_NAME" \
  --scope "subscriptions/$SUBSCRIPTION_ID"

echo "Immediate export triggered."

    #### GCP BigQuery Billing Export

    GCP's billing export to BigQuery is the most analysis-friendly of the three providers. Once enabled in the Console under Billing → Budget & Alerts → Billing Export, you can query cost data with standard SQL. The following query extracts Cloud SQL and AlloyDB costs by project and label, normalized to a per-day grain.

    -- GCP BigQuery: daily Cloud SQL and AlloyDB cost breakdown
-- Run in BigQuery against your billing export dataset
-- Replace 'your-project.billing_export.gcp_billing_export_v1_XXXXXX'
-- with your actual billing export table name

SELECT
  DATE(usage_start_time) AS cost_date,
  project.name AS project_name,
  service.description AS service_name,
  sku.description AS sku_description,
  labels.value AS environment_label,
  resource.name AS resource_name,
  SUM(usage.amount) AS usage_amount,
  usage.unit AS usage_unit,
  SUM(cost) AS total_cost,
  SUM(
    IFNULL(
      (SELECT SUM(c.amount)
       FROM UNNEST(credits) c
       WHERE c.type IN ('SUSTAINED_USE_DISCOUNT', 'COMMITTED_USE_DISCOUNT')),
      0
    )
  ) AS total_credits,
  SUM(cost) + SUM(
    IFNULL(
      (SELECT SUM(c.amount)
       FROM UNNEST(credits) c),
      0
    )
  ) AS net_cost
FROM
  `your-project.billing_export.gcp_billing_export_v1_XXXXXX`,
  UNNEST(labels) AS labels
WHERE
  service.description IN ('Cloud SQL', 'AlloyDB for PostgreSQL')
  AND labels.key = 'environment'
  AND DATE(usage_start_time) >= DATE_SUB(CURRENT_DATE(), INTERVAL 30 DAY)
GROUP BY
  1, 2, 3, 4, 5, 6, 8
ORDER BY
  cost_date DESC,
  total_cost DESC;

    6.3 Building the Cost Metrics Store

    Rather than building a separate analytics platform for database costs, the most pragmatic approach for a team that already manages PostgreSQL is to create a dedicated schema within an existing monitoring database. This keeps the tooling familiar and allows you to join cost data with performance metrics from pg_stat_statements, slow query logs, and CloudWatch exports.

    The following schema creates the tables needed to store normalized cost data across providers, along with the performance correlation tables that make the dashboard genuinely useful rather than decorative.

    -- Database Cost Metrics Schema
-- Apply to a dedicated monitoring PostgreSQL instance
-- or a separate schema in your existing monitoring DB

CREATE SCHEMA IF NOT EXISTS dbfinops;

-- Normalized cost fact table (all providers, all dimensions)
CREATE TABLE dbfinops.daily_costs (
    id                  BIGSERIAL PRIMARY KEY,
    cost_date           DATE NOT NULL,
    provider            TEXT NOT NULL CHECK (provider IN ('aws', 'azure', 'gcp')),
    account_id          TEXT NOT NULL,           -- AWS account, Azure subscription, GCP project
    region              TEXT NOT NULL,
    database_identifier TEXT NOT NULL,           -- instance/cluster/pool name
    engine              TEXT,                    -- postgres, mysql, sqlserver, etc.
    instance_class      TEXT,                    -- db.r6g.2xlarge, GP_Gen5_4, etc.
    environment         TEXT,                    -- from tag/label: prod, staging, dev
    team                TEXT,                    -- from tag/label: payments, analytics, etc.
    cost_dimension      TEXT NOT NULL,           -- instance, storage, io, backup, transfer
    usage_quantity      NUMERIC(18, 6),
    usage_unit          TEXT,
    unblended_cost      NUMERIC(18, 6) NOT NULL,
    blended_cost        NUMERIC(18, 6),
    credits_applied     NUMERIC(18, 6) DEFAULT 0,
    net_cost            NUMERIC(18, 6) GENERATED ALWAYS AS
                          (unblended_cost + COALESCE(credits_applied, 0)) STORED,
    currency            CHAR(3) DEFAULT 'USD',
    loaded_at           TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_daily_costs_date
    ON dbfinops.daily_costs (cost_date DESC);

CREATE INDEX idx_daily_costs_provider_db
    ON dbfinops.daily_costs (provider, database_identifier, cost_date DESC);

CREATE INDEX idx_daily_costs_environment_team
    ON dbfinops.daily_costs (environment, team, cost_date DESC);

-- Unique constraint to prevent duplicate loads
CREATE UNIQUE INDEX idx_daily_costs_unique
    ON dbfinops.daily_costs (
        cost_date, provider, account_id, database_identifier,
        cost_dimension, COALESCE(region, ''), COALESCE(environment, '')
    );

-- Month-to-date cost summary (materialized, refreshed daily)
CREATE MATERIALIZED VIEW dbfinops.mtd_cost_summary AS
SELECT
    date_trunc('month', cost_date)   AS month_start,
    provider,
    database_identifier,
    environment,
    team,
    instance_class,
    engine,
    SUM(unblended_cost)              AS mtd_cost,
    SUM(credits_applied)             AS mtd_credits,
    SUM(net_cost)                    AS mtd_net_cost,
    MAX(cost_date)                   AS last_updated
FROM
    dbfinops.daily_costs
WHERE
    cost_date >= date_trunc('month', CURRENT_DATE)
GROUP BY
    1, 2, 3, 4, 5, 6, 7;

CREATE INDEX idx_mtd_summary_provider
    ON dbfinops.mtd_cost_summary (provider, environment, month_start);

-- Cost anomaly detection table
CREATE TABLE dbfinops.cost_anomalies (
    id                  BIGSERIAL PRIMARY KEY,
    detected_at         TIMESTAMPTZ DEFAULT NOW(),
    cost_date           DATE NOT NULL,
    provider            TEXT NOT NULL,
    database_identifier TEXT NOT NULL,
    cost_dimension      TEXT NOT NULL,
    expected_cost       NUMERIC(18, 6),
    actual_cost         NUMERIC(18, 6),
    deviation_pct       NUMERIC(8, 2),
    severity            TEXT CHECK (severity IN ('low', 'medium', 'high', 'critical')),
    acknowledged        BOOLEAN DEFAULT FALSE,
    acknowledged_by     TEXT,
    notes               TEXT
);

-- Reserved instance and commitment tracking
CREATE TABLE dbfinops.commitments (
    id                  BIGSERIAL PRIMARY KEY,
    provider            TEXT NOT NULL,
    account_id          TEXT NOT NULL,
    commitment_type     TEXT NOT NULL,   -- reserved_instance, savings_plan, committed_use
    instance_class      TEXT,
    region              TEXT,
    term_months         INT,
    start_date          DATE NOT NULL,
    end_date            DATE NOT NULL,
    hourly_commitment   NUMERIC(18, 6),
    monthly_cost        NUMERIC(18, 6),
    utilization_pct     NUMERIC(6, 2),
    break_even_date     DATE,
    notes               TEXT,
    created_at          TIMESTAMPTZ DEFAULT NOW()
);

-- Cost-per-query correlation (joins with pg_stat_statements data)
CREATE TABLE dbfinops.query_cost_attribution (
    id                  BIGSERIAL PRIMARY KEY,
    sample_date         DATE NOT NULL,
    database_identifier TEXT NOT NULL,
    queryid             BIGINT,          -- from pg_stat_statements
    query_fingerprint   TEXT,
    calls               BIGINT,
    total_exec_ms       NUMERIC(18, 3),
    mean_exec_ms        NUMERIC(18, 3),
    rows_returned       BIGINT,
    shared_blks_hit     BIGINT,
    shared_blks_read    BIGINT,
    estimated_io_cost   NUMERIC(18, 6), -- blks_read × storage_cost_per_gb
    pct_of_instance_cpu NUMERIC(6, 2),
    attributed_cost_usd NUMERIC(18, 6)  -- (pct_cpu / 100) × daily_instance_cost
);

    With this schema in place, you have a foundation that supports both operational alerting and executive reporting. The query_cost_attribution table is particularly valuable for the conversations that no one wants to have but every engineering team needs: "This ETL job costs us approximately $380/month in compute attribution. Is that acceptable for what it produces?"
