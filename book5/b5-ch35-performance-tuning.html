Chapter 35: Query Performance Tuning on Managed Services

Query performance tuning on managed cloud databases is fundamentally the same discipline you've practiced for years — identify the slow query, understand the execution plan, fix the root cause — but the environment imposes constraints that change how you investigate, what levers you can pull, and where the bottlenecks actually live. You can't attach a profiler directly to the OS, you can't touch the kernel scheduler, and you can't install arbitrary extensions. What you get instead is a set of cloud-native observability tools, managed parameter systems, and platform-specific optimizations that, once mastered, give you most of what you need. This chapter covers the full tuning workflow on AWS RDS/Aurora, Azure SQL, and Google Cloud SQL/AlloyDB — from capturing slow queries to fixing execution plans to understanding where managed service limits actually bite you in production.

---

35.1 Capturing and Identifying Slow Queries

The first step in any tuning engagement is visibility: find the queries causing problems before you guess at solutions. On self-managed PostgreSQL, you enabled pg_stat_statements, tailed the slow query log, or ran EXPLAIN ANALYZE in a session. On managed services, the same data exists but reaches you through different channels.

AWS RDS and Aurora (PostgreSQL)

On RDS PostgreSQL, slow query logging is controlled through the parameter group. The critical parameters are log_min_duration_statement (in milliseconds), log_statement, and pg_stat_statements.track. You cannot edit the parameter group in the console and expect it to take effect instantly on production without a reboot for static parameters — but log_min_duration_statement is dynamic, so you can apply it without restart.

# Modify the RDS parameter group to enable slow query logging
aws rds modify-db-parameter-group \
  --db-parameter-group-name prod-postgres15-params \
  --parameters \
    "ParameterName=log_min_duration_statement,ParameterValue=1000,ApplyMethod=immediate" \
    "ParameterName=log_statement,ParameterValue=none,ApplyMethod=immediate" \
    "ParameterName=log_lock_waits,ParameterValue=1,ApplyMethod=immediate" \
    "ParameterName=log_temp_files,ParameterValue=10240,ApplyMethod=immediate"

# Verify the change took effect without a reboot
aws rds describe-db-parameters \
  --db-parameter-group-name prod-postgres15-params \
  --query "Parameters[?ParameterName=='log_min_duration_statement']"

Logs are streamed to CloudWatch Logs automatically if you have EnableCloudwatchLogsExports set. From there, CloudWatch Logs Insights lets you run structured queries against the raw log text:

# Query CloudWatch Logs Insights for the top slow queries in the last hour
aws logs start-query \
  --log-group-name /aws/rds/instance/prod-db-01/postgresql \
  --start-time $(date -d '1 hour ago' +%s) \
  --end-time $(date +%s) \
  --query-string '
    fields @timestamp, @message
    | filter @message like /duration:/
    | parse @message "duration: * ms  statement: *" as duration_ms, statement
    | stats avg(duration_ms) as avg_ms, max(duration_ms) as max_ms, count() as executions
        by statement
    | sort max_ms desc
    | limit 25
  '

This gives you query text, average duration, and execution count directly from CloudWatch — no SSH session required.

For RDS SQL Server, Performance Insights is the primary tool. It surfaces the Average Active Sessions (AAS) metric, broken down by wait type, SQL statement, host, and user. This is the managed equivalent of querying sys.dm_exec_query_stats combined with sys.dm_os_wait_stats. Enable it at instance creation or modify the instance to turn it on:

aws rds modify-db-instance \
  --db-instance-identifier prod-sqlserver-01 \
  --enable-performance-insights \
  --performance-insights-retention-period 731 \
  --apply-immediately

Performance Insights retains 7 days free; 731 days costs roughly $0.02 per vCPU per hour. For a db.r6g.4xlarge (16 vCPUs), extended retention costs around $77/month — budget for it on any production instance.

Azure SQL Database

Azure SQL surfaces query performance data through Query Store, which is enabled by default on Azure SQL Database (unlike on-premises SQL Server where you had to enable it manually). The Query Store captures execution plans, runtime statistics, and wait statistics automatically. Combine this with the Query Performance Insight blade in the Azure portal or query it directly:

-- Top 20 queries by total CPU time in the last 24 hours from Query Store
SELECT TOP 20
    q.query_id,
    qt.query_sql_text,
    rs.count_executions,
    rs.avg_duration / 1000.0 AS avg_duration_ms,
    rs.avg_cpu_time / 1000.0 AS avg_cpu_ms,
    rs.avg_logical_io_reads,
    rs.avg_rowcount,
    CAST(rs.avg_cpu_time * rs.count_executions / 1e6 AS DECIMAL(10,2)) AS total_cpu_seconds
FROM sys.query_store_query q
JOIN sys.query_store_query_text qt ON q.query_text_id = qt.query_text_id
JOIN sys.query_store_plan p ON q.query_id = p.query_id
JOIN sys.query_store_runtime_stats rs ON p.plan_id = rs.plan_id
JOIN sys.query_store_runtime_stats_interval rsi ON rs.runtime_stats_interval_id = rsi.runtime_stats_interval_id
WHERE rsi.start_time >= DATEADD(HOUR, -24, GETUTCDATE())
ORDER BY total_cpu_seconds DESC;

Google Cloud SQL and AlloyDB

Cloud SQL for PostgreSQL streams logs to Cloud Logging. Enable log_min_duration_statement through a database flag on the instance:

# Set slow query threshold to 500ms on Cloud SQL PostgreSQL
gcloud sql instances patch prod-cloudsql-pg \
  --database-flags \
    log_min_duration_statement=500,\
    log_lock_waits=on,\
    log_temp_files=10240,\
    track_io_timing=on

# Query slow queries from Cloud Logging using the logging API
gcloud logging read \
  'resource.type="cloudsql_database" AND
   textPayload=~"duration:" AND
   resource.labels.database_id="my-project:prod-cloudsql-pg"' \
  --limit=50 \
  --format="value(textPayload)" \
  --freshness=1h

AlloyDB adds the google_columnar_engine extension for OLAP acceleration and exposes query insights through the AlloyDB console UI and Cloud Monitoring. The pg_stat_statements view is available on both, and AlloyDB's Query Insights surfaces plan regressions automatically — a capability you would have built yourself with a custom baseline table in a self-managed environment.

35.2 Reading Execution Plans on Managed Services

Getting an execution plan on a managed service is mechanically simple, but interpreting the plan in the context of managed infrastructure requires calibrating your assumptions. The optimizer is the same PostgreSQL or SQL Server optimizer you know — but the cost model is tuned differently, statistics maintenance behaves differently, and some plan choices are shaped by the storage architecture.

PostgreSQL Plans on RDS and Aurora

EXPLAIN (ANALYZE, BUFFERS, FORMAT JSON) works exactly as it does on self-managed PostgreSQL. The difference on Aurora is in what the buffer counts mean: Aurora uses a distributed storage layer, so buffer hits that look like shared cache hits may actually be served from the Aurora buffer pool, while misses trigger Aurora storage I/O rather than OS page cache misses. The I/O cost model in the planner doesn't fully account for Aurora's storage characteristics, which sometimes causes the planner to prefer sequential scans when an index seek would be faster on spinning disk, because Aurora's storage is effectively SSD-backed with 6-way replication — the sequential scan penalty is lower than what the seq_page_cost parameter assumes.

-- Capture a plan with full detail and serialize it for storage
EXPLAIN (ANALYZE, BUFFERS, VERBOSE, FORMAT JSON)
SELECT
    o.order_id,
    o.customer_id,
    o.order_date,
    SUM(oi.quantity * oi.unit_price) AS order_total
FROM orders o
JOIN order_items oi ON o.order_id = oi.order_id
WHERE o.order_date >= '2024-01-01'
  AND o.status = 'COMPLETED'
GROUP BY o.order_id, o.customer_id, o.order_date
HAVING SUM(oi.quantity * oi.unit_price) > 500
ORDER BY order_total DESC
LIMIT 100;

When you see a sequential scan on a large table and suspect the planner is making the wrong choice, check the statistics before blaming the plan:

-- Check when statistics were last collected and how accurate they are
SELECT
    schemaname,
    tablename,
    n_live_tup,
    n_dead_tup,
    last_autoanalyze,
    last_analyze,
    last_autoanalyze - now() AS time_since_autoanalyze,
    n_mod_since_analyze
FROM pg_stat_user_tables
WHERE tablename IN ('orders', 'order_items')
ORDER BY n_mod_since_analyze DESC;

-- Check column statistics for a specific column
SELECT
    attname,
    n_distinct,
    correlation,
    most_common_vals,
    most_common_freqs
FROM pg_stats
WHERE tablename = 'orders'
  AND attname IN ('status', 'order_date');

On RDS, autovacuum runs under the managed service's control. You can tune its parameters through the parameter group, but you cannot run VACUUM manually with OS-level priority settings or kill the autovacuum workers through system commands. If statistics are stale on a large partitioned table during a bulk load window, you can run ANALYZE orders; manually as the database superuser — RDS grants you rds_superuser which has sufficient privileges for this.

Aurora-Specific Plan Considerations

Aurora PostgreSQL's parallel query feature (available on Aurora v3, which is PostgreSQL 15-compatible) can push query processing down to the distributed storage layer. When enabled, plans show Parallel Seq Scan nodes that indicate storage-level parallelism. This is structurally different from PostgreSQL's parallel workers, which run on the compute node. You can control whether a query uses Aurora parallel query by setting aurora_parallel_query:

-- Enable Aurora parallel query for a specific session and compare plans
SET aurora_parallel_query = ON;
EXPLAIN SELECT COUNT(*), AVG(total_amount) FROM orders WHERE order_date >= '2023-01-01';

SET aurora_parallel_query = OFF;
EXPLAIN SELECT COUNT(*), AVG(total_amount) FROM orders WHERE order_date >= '2023-01-01';

Parallel query shows the largest gains on full table scans of tables larger than 1 million rows where the result set fits in memory after aggregation. For OLTP queries hitting indexes, it adds overhead.

SQL Server Plans on Azure SQL

Azure SQL Database uses the same execution plan XML format as SQL Server on-premises. The key difference is that sys.dm_exec_query_plan and sys.dm_exec_query_stats are available, but sys.dm_os_ring_buffers and some internal DMVs are restricted. For plan capture and analysis, use Query Store rather than ad hoc DMV queries — Query Store persists plans across failovers, which DMV caches do not.

To force a specific plan (the equivalent of using a plan guide on-premises), Azure SQL supports Query Store plan forcing:

-- Force a specific plan for a regressed query using Query Store
-- First, identify the query_id and the plan_id of the good historical plan
SELECT
    q.query_id,
    p.plan_id,
    rs.avg_duration,
    rs.count_executions,
    p.query_plan
FROM sys.query_store_query q
JOIN sys.query_store_plan p ON q.query_id = p.query_id
JOIN sys.query_store_runtime_stats rs ON p.plan_id = rs.plan_id
WHERE q.query_id = 1547
ORDER BY rs.last_execution_time DESC;

-- Force the known-good plan
EXEC sys.sp_query_store_force_plan
    @query_id = 1547,
    @plan_id = 892;

-- Verify the forcing is active
SELECT
    q.query_id,
    p.plan_id,
    p.is_forced_plan,
    p.force_failure_count,
    p.last_force_failure_reason_desc
FROM sys.query_store_query q
JOIN sys.query_store_plan p ON q.query_id = p.query_id
WHERE q.query_id = 1547;

Plan forcing in Query Store survives database restarts and failovers on Azure SQL — this is a meaningful improvement over on-premises plan guides, which you had to manage separately.

35.3 Index Tuning in the Cloud: What Works and What Doesn't

Index tuning on managed services is conceptually identical to on-premises: add missing indexes, remove unused ones, choose the right index type for the access pattern. The cloud layer changes the operational mechanics — how you identify candidates, how you apply changes safely, and what the platform does automatically.

AWS RDS PostgreSQL: pg_stat_user_indexes and Missing Index Detection

PostgreSQL has no built-in missing index advisor. On self-managed systems, you'd combine pg_stat_statements with EXPLAIN output and pg_stat_user_indexes to find candidates. On RDS, you have the same data plus the option to use the RDS Performance Insights API to correlate wait events with specific queries:

-- Find indexes that are never used (candidates for removal)
-- Run this after a representative workload period (at least 7 days post-restart)
SELECT
    schemaname,
    tablename,
    indexname,
    idx_scan,
    idx_tup_read,
    idx_tup_fetch,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND schemaname NOT IN ('pg_catalog', 'information_schema')
ORDER BY pg_relation_size(indexrelid) DESC;

-- Find sequential scans on large tables that might benefit from an index
SELECT
    schemaname,
    relname AS tablename,
    seq_scan,
    seq_tup_read,
    idx_scan,
    n_live_tup,
    pg_size_pretty(pg_total_relation_size(relid)) AS total_size,
    ROUND(seq_scan::NUMERIC / NULLIF(seq_scan + idx_scan, 0) * 100, 2) AS seq_scan_pct
FROM pg_stat_user_tables
WHERE n_live_tup > 100000
  AND seq_scan > 100
ORDER BY seq_tup_read DESC
LIMIT 20;

A gotcha on RDS: pg_stat_user_indexes.idx_scan resets when the instance restarts. RDS Multi-AZ failovers cause a restart on the new primary. If you collected baseline data before a recent failover, your unused index candidates may be incorrect. Always check pg_stat_bgwriter.stats_reset to see when statistics were last reset:

SELECT stats_reset FROM pg_stat_bgwriter;

Creating Indexes Concurrently

CREATE INDEX CONCURRENTLY works on RDS and Cloud SQL exactly as it does on-premises — it builds the index without holding an AccessExclusiveLock on the table, so production reads and writes continue. The gotcha is that if the concurrent build fails (connection drop, out-of-disk-space), it leaves an invalid index behind that you must drop manually. On Aurora, an invalid index in pg_index.indisvalid = false will not be used by the planner but still slows down DML because the executor still attempts to update it.

-- Build a partial index concurrently for the active order pattern
CREATE INDEX CONCURRENTLY idx_orders_active_date
ON orders (customer_id, order_date DESC)
WHERE status IN ('PENDING', 'PROCESSING');

-- Verify the index built successfully
SELECT
    indexname,
    indisvalid,
    indisready
FROM pg_indexes
JOIN pg_index ON indexrelid = (
    SELECT oid FROM pg_class WHERE relname = 'idx_orders_active_date'
)
WHERE tablename = 'orders'
  AND indexname = 'idx_orders_active_date';

Azure SQL: Automatic Index Recommendations

Azure SQL Database includes an automatic tuning engine that recommends and — if you enable it — automatically creates and drops indexes based on Query Store data. This is the cloud equivalent of the Database Engine Tuning Advisor, but it runs continuously and learns from production workload patterns.

# Check current automatic tuning state via Azure CLI
az sql db show \
  --resource-group prod-rg \
  --server prod-sql-server \
  --name prod-database \
  --query "autoPauseDelay, automaticTuning"

# Enable automatic index creation recommendations (monitoring only, not auto-apply)
az sql db update \
  --resource-group prod-rg \
  --server prod-sql-server \
  --name prod-database \
  --auto-tuning-mode Auto

The recommendation: enable automatic tuning in Auto mode for index recommendations on read-heavy OLAP databases. For transactional OLTP databases where index creation causes brief write stalls and where any schema change needs a change control approval, leave it in recommendation-only mode and apply changes manually during maintenance windows.

AlloyDB: Columnar Engine for HTAP Workloads

AlloyDB's columnar engine maintains a column store in memory alongside the row store. When a query would benefit from columnar access, the planner routes it to the columnar engine automatically. You can check which tables have columnar data populated and force or hint columnar scans:

-- Check columnar engine population status
SELECT
    database_name,
    schema_name,
    table_name,
    column_count,
    row_count,
    memory_size_in_bytes / 1024 / 1024 AS memory_mb
FROM g_columnar_relations;

-- Verify a query is using the columnar engine
EXPLAIN SELECT
    product_category,
    SUM(revenue) AS total_revenue,
    AVG(margin_pct) AS avg_margin
FROM sales_facts
WHERE sale_date BETWEEN '2024-01-01' AND '2024-12-31'
GROUP BY product_category;
-- Look for "Custom Scan (columnar scan)" in the plan output

35.4 Wait Event Analysis and Resource Bottleneck Identification

Experienced DBAs know that execution time alone is incomplete — a query that runs in 10ms but waits 2 seconds for a lock isn't a query problem, it's a concurrency problem. Cloud managed services surface wait events through different mechanisms than on-premises but with equivalent depth.

RDS Performance Insights Wait Event Analysis

Performance Insights breaks down Average Active Sessions by wait event type. For PostgreSQL, the wait event categories map directly to pg_stat_activity.wait_event_type values: LWLock, Lock, BufferPin, IO, Client. For SQL Server, they map to SQL Server wait types: LCK_M_X, PAGEIOLATCH_SH, ASYNC_NETWORK_IO.

Use the Performance Insights API to pull wait data programmatically for trend analysis:

import boto3
from datetime import datetime, timedelta, timezone

pi_client = boto3.client('pi', region_name='us-east-1')

# Pull top wait events for the last 3 hours at 1-minute granularity
response = pi_client.get_resource_metrics(
    ServiceType='RDS',
    Identifier='db:prod-db-01',  # Performance Insights resource ID
    StartTime=datetime.now(timezone.utc) - timedelta(hours=3),
    EndTime=datetime.now(timezone.utc),
    PeriodInSeconds=60,
    MetricQueries=[
        {
            'Metric': 'db.load.avg',
            'GroupBy': {
                'Group': 'db.wait_event',
                'Dimensions': ['db.wait_event.type', 'db.wait_event.name'],
                'Limit': 10
            }
        }
    ]
)

for metric_list in response['MetricList']:
    key = metric_list['Key']
    wait_type = key.get('Dimensions', {}).get('db.wait_event.type', 'unknown')
    wait_name = key.get('Dimensions', {}).get('db.wait_event.name', 'unknown')
    
    # Calculate average load contribution for this wait event
    data_points = metric_list['DataPoints']
    if data_points:
        avg_load = sum(dp.get('Value', 0) for dp in data_points) / len(data_points)
        print(f"{wait_type}/{wait_name}: avg AAS contribution = {avg_load:.3f}")

On Aurora PostgreSQL, the most common problematic wait events in production:

  • → `IO:DataFileRead` — Buffer pool misses, often from missing indexes or OLAP queries on OLTP instances
  • → `Lock:relation` — DDL blocking DML or long-running transactions holding table locks
  • → `LWLock:buffer_content` — High-concurrency UPDATE hot spots (counter tables, queue tables)
  • → `Client:ClientRead` — Network latency between application and database; often a VPC placement issue or RDS Proxy connection overhead
