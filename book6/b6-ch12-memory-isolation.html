Chapter 12 of 40 AI-Native Apps on PostgreSQL

Chapter 12: Multi-User Memory Isolation

When you build a single-tenant chatbot, memory management is straightforward — one conversation history table, one embedding space, one user. The moment you add a second user, every assumption breaks. Their memories bleed into each other's context, their embeddings compete in the same vector space, their conversation histories mix in ways that produce hallucinations, privacy violations, and deeply confused AI responses. Multi-user memory isolation is the engineering discipline that prevents this, and PostgreSQL's row-level security, schema design, and vector capabilities make it the ideal foundation for getting it right.

This chapter covers the full architecture: how to model user identity in a way that AI pipelines can enforce, how to use PostgreSQL Row Level Security to make memory isolation automatic and tamper-resistant, how to structure embedding tables so semantic search never crosses user boundaries, how to handle shared knowledge alongside private memory, and how to build the FastAPI layer that connects authenticated users to their isolated AI context. These patterns apply whether you are building a customer support chatbot, a personal AI assistant, a multi-tenant SaaS product, or an enterprise agent platform.

12.1 The Memory Isolation Problem in Multi-User AI Systems

Consider what "memory" means in a production AI application. It is not a single thing. It is at least four distinct layers:

1. Conversation history — the raw message turns from prior sessions

2. Episodic summaries — compressed representations of what happened in prior conversations

3. User-specific embeddings — vectors derived from documents, notes, or context that belong to one user

4. Shared knowledge embeddings — vectors from a corpus that all users should be able to query (product documentation, company knowledge base)

Each layer has different isolation requirements. Conversation history is strictly private. Episodic summaries are strictly private. User-specific embeddings are strictly private. Shared knowledge embeddings are read-accessible to all authenticated users but write-controlled.

The naive implementation puts everything in the same tables with a user_id column and filters in application code. This works until it doesn't — a missing WHERE clause in a query, a developer testing in production with the wrong session, a bug in the ORM layer. Application-level filtering is defense in one layer. PostgreSQL Row Level Security is defense at the database layer, making data leakage impossible regardless of what the application layer does.

The other failure mode is vector search. If user A has uploaded private financial documents and user B runs a semantic similarity search, an unguarded pgvector query will happily return results from user A's embedding space alongside user B's. The ORDER BY cosine similarity does not know about tenancy. The database policy does.

Here is the foundational schema that everything in this chapter builds from:

-- Core user identity table — the anchor for all isolation policies
CREATE TABLE users (
    user_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    external_id     TEXT UNIQUE NOT NULL,           -- from your auth provider (Auth0, Clerk, etc.)
    tenant_id       UUID NOT NULL,                  -- for multi-tenant SaaS; same as user_id for personal apps
    email           TEXT,
    display_name    TEXT,
    memory_tier     TEXT NOT NULL DEFAULT 'standard'
                        CHECK (memory_tier IN ('standard', 'extended', 'unlimited')),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    is_active       BOOLEAN NOT NULL DEFAULT TRUE
);

CREATE INDEX idx_users_external_id ON users(external_id);
CREATE INDEX idx_users_tenant_id ON users(tenant_id);

-- Session tracking — connects JWT tokens to database identity
CREATE TABLE user_sessions (
    session_id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    tenant_id       UUID NOT NULL,
    started_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_active     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    expires_at      TIMESTAMPTZ NOT NULL,
    ip_address      INET,
    user_agent      TEXT,
    is_revoked      BOOLEAN NOT NULL DEFAULT FALSE
);

CREATE INDEX idx_sessions_user_id ON user_sessions(user_id);
CREATE INDEX idx_sessions_expires ON user_sessions(expires_at) WHERE NOT is_revoked;

-- The PostgreSQL role that all application connections use
-- This is NOT a superuser — it has no permissions by default
CREATE ROLE ai_app_user LOGIN PASSWORD 'changeme_in_production';

-- Grant connect on the database
GRANT CONNECT ON DATABASE ai_production TO ai_app_user;
GRANT USAGE ON SCHEMA public TO ai_app_user;

-- Grant table-level permissions — RLS will enforce row-level access
GRANT SELECT, INSERT, UPDATE ON users TO ai_app_user;
GRANT SELECT, INSERT, UPDATE ON user_sessions TO ai_app_user;

The tenant_id column deserves explanation. In a pure personal AI assistant, tenant_id = user_id. In a B2B SaaS product, multiple users share a tenant. Some data is isolated per user within a tenant; other data is shared across the tenant. The schema handles both by carrying both identifiers everywhere.

12.2 Conversation History Tables with Row Level Security

Conversation history is the most privacy-sensitive memory layer. A single leaked message from another user's conversation is a serious incident. Here is the complete schema and RLS configuration:

-- Conversations group related message turns
CREATE TABLE conversations (
    conversation_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    tenant_id       UUID NOT NULL,
    title           TEXT,
    model_id        TEXT NOT NULL DEFAULT 'claude-3-5-sonnet-20241022',
    system_prompt   TEXT,
    metadata        JSONB NOT NULL DEFAULT '{}',
    token_count     INTEGER NOT NULL DEFAULT 0,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    archived_at     TIMESTAMPTZ
);

CREATE INDEX idx_conversations_user_id ON conversations(user_id);
CREATE INDEX idx_conversations_tenant_id ON conversations(tenant_id);
CREATE INDEX idx_conversations_updated ON conversations(user_id, updated_at DESC);

-- Individual message turns within a conversation
CREATE TABLE messages (
    message_id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    conversation_id UUID NOT NULL REFERENCES conversations(conversation_id) ON DELETE CASCADE,
    user_id         UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    tenant_id       UUID NOT NULL,
    role            TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system', 'tool')),
    content         TEXT NOT NULL,
    content_tokens  INTEGER,
    tool_calls      JSONB,
    tool_results    JSONB,
    model_id        TEXT,
    finish_reason   TEXT,
    latency_ms      INTEGER,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_messages_conversation ON messages(conversation_id, created_at);
CREATE INDEX idx_messages_user_id ON messages(user_id);
-- Partial index for recent messages — most queries only need recent history
CREATE INDEX idx_messages_recent ON messages(conversation_id, created_at DESC)
    WHERE created_at > NOW() - INTERVAL '90 days';

-- Episodic memory summaries — compressed representations of past conversations
CREATE TABLE memory_summaries (
    summary_id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    tenant_id       UUID NOT NULL,
    conversation_id UUID REFERENCES conversations(conversation_id) ON DELETE SET NULL,
    summary_text    TEXT NOT NULL,
    key_facts       JSONB NOT NULL DEFAULT '[]',   -- extracted entities and facts
    importance      FLOAT NOT NULL DEFAULT 0.5,    -- 0.0 to 1.0, used for retrieval ranking
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    valid_until     TIMESTAMPTZ                    -- NULL = permanent memory
);

CREATE INDEX idx_summaries_user_id ON memory_summaries(user_id, importance DESC);

-- Enable RLS on all user-data tables
ALTER TABLE conversations ENABLE ROW LEVEL SECURITY;
ALTER TABLE messages ENABLE ROW LEVEL SECURITY;
ALTER TABLE memory_summaries ENABLE ROW LEVEL SECURITY;

-- The application sets this at connection time from the JWT
-- current_setting('app.current_user_id') returns the user_id
-- current_setting('app.current_tenant_id') returns the tenant_id

-- Conversations: users see only their own; tenant admins see the whole tenant
CREATE POLICY conversations_isolation ON conversations
    FOR ALL
    TO ai_app_user
    USING (
        user_id = current_setting('app.current_user_id')::UUID
        OR (
            tenant_id = current_setting('app.current_tenant_id')::UUID
            AND current_setting('app.is_tenant_admin')::BOOLEAN = TRUE
        )
    )
    WITH CHECK (
        user_id = current_setting('app.current_user_id')::UUID
        AND tenant_id = current_setting('app.current_tenant_id')::UUID
    );

-- Messages: inherit conversation access — user_id check is redundant but explicit
CREATE POLICY messages_isolation ON messages
    FOR ALL
    TO ai_app_user
    USING (
        user_id = current_setting('app.current_user_id')::UUID
        OR (
            tenant_id = current_setting('app.current_tenant_id')::UUID
            AND current_setting('app.is_tenant_admin')::BOOLEAN = TRUE
        )
    )
    WITH CHECK (
        user_id = current_setting('app.current_user_id')::UUID
        AND tenant_id = current_setting('app.current_tenant_id')::UUID
    );

-- Memory summaries: strictly per-user, no tenant admin override
CREATE POLICY summaries_isolation ON memory_summaries
    FOR ALL
    TO ai_app_user
    USING (user_id = current_setting('app.current_user_id')::UUID)
    WITH CHECK (
        user_id = current_setting('app.current_user_id')::UUID
        AND tenant_id = current_setting('app.current_tenant_id')::UUID
    );

-- Helper function to set the session context from application code
CREATE OR REPLACE FUNCTION set_user_context(
    p_user_id UUID,
    p_tenant_id UUID,
    p_is_tenant_admin BOOLEAN DEFAULT FALSE
) RETURNS VOID
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
BEGIN
    PERFORM set_config('app.current_user_id', p_user_id::TEXT, TRUE);
    PERFORM set_config('app.current_tenant_id', p_tenant_id::TEXT, TRUE);
    PERFORM set_config('app.is_tenant_admin', p_is_tenant_admin::TEXT, TRUE);
END;
$$;

-- Grant execute only to the app role
GRANT EXECUTE ON FUNCTION set_user_context TO ai_app_user;

The set_config(..., TRUE) call at the end sets the configuration as local to the current transaction. This is important: with PgBouncer in transaction pooling mode, each transaction gets a potentially different physical connection, so the session-local settings must be re-established per transaction. If you are using session pooling, you can use FALSE to persist across the session, but you must explicitly clear context on logout.

12.3 Isolated Vector Embeddings for Private User Memory

Semantic search across user documents is where multi-user isolation gets architecturally interesting. You need fast approximate nearest-neighbor search within a user's embedding space while keeping it completely separated from other users' spaces. pgvector handles this elegantly with partial indexes.

-- Enable pgvector
CREATE EXTENSION IF NOT EXISTS vector;

-- User document store — private documents uploaded by each user
CREATE TABLE user_documents (
    document_id     UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    tenant_id       UUID NOT NULL,
    filename        TEXT,
    content_type    TEXT NOT NULL DEFAULT 'text/plain',
    raw_content     TEXT,
    metadata        JSONB NOT NULL DEFAULT '{}',
    ingested_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    chunk_count     INTEGER NOT NULL DEFAULT 0
);

CREATE INDEX idx_user_docs_user_id ON user_documents(user_id);
ALTER TABLE user_documents ENABLE ROW LEVEL SECURITY;

CREATE POLICY user_docs_isolation ON user_documents
    FOR ALL TO ai_app_user
    USING (user_id = current_setting('app.current_user_id')::UUID)
    WITH CHECK (
        user_id = current_setting('app.current_user_id')::UUID
        AND tenant_id = current_setting('app.current_tenant_id')::UUID
    );

-- Embedding chunks — each document is split into chunks with vector embeddings
CREATE TABLE user_embeddings (
    embedding_id    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    document_id     UUID NOT NULL REFERENCES user_documents(document_id) ON DELETE CASCADE,
    user_id         UUID NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    tenant_id       UUID NOT NULL,
    chunk_index     INTEGER NOT NULL,
    chunk_text      TEXT NOT NULL,
    embedding       VECTOR(1536),          -- OpenAI text-embedding-3-small dimensions
    token_count     INTEGER,
    metadata        JSONB NOT NULL DEFAULT '{}',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Standard index for lookup by user + document
CREATE INDEX idx_user_embeddings_user ON user_embeddings(user_id, document_id);

-- Per-user HNSW indexes using partial indexes
-- This is the key pattern: each user gets efficient ANN search within their own space
-- For a small number of power users, you might create these dynamically
-- For large multi-tenant systems, use the partitioned approach below

-- Example: partial HNSW index for a specific high-volume user
-- In practice, create these for users above a document threshold
CREATE INDEX idx_embeddings_hnsw_user_example ON user_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64)
    WHERE user_id = '00000000-0000-0000-0000-000000000001'::UUID;

-- For the general case: a single HNSW index on user_id + embedding
-- pgvector will use the user_id filter to narrow candidates
-- This is less optimal than per-user indexes but works for most scales
CREATE INDEX idx_embeddings_hnsw_all ON user_embeddings
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

-- Shared knowledge base — accessible to all users in a tenant
-- No user_id here — these are tenant-level or global resources
CREATE TABLE shared_knowledge (
    knowledge_id    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    tenant_id       UUID,                  -- NULL = global (platform-wide)
    source_name     TEXT NOT NULL,
    chunk_text      TEXT NOT NULL,
    embedding       VECTOR(1536),
    metadata        JSONB NOT NULL DEFAULT '{}',
    published_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    expires_at      TIMESTAMPTZ
);

CREATE INDEX idx_shared_knowledge_hnsw ON shared_knowledge
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

CREATE INDEX idx_shared_knowledge_tenant ON shared_knowledge(tenant_id);

-- RLS for shared knowledge: all authenticated users can read; only admins can write
ALTER TABLE shared_knowledge ENABLE ROW LEVEL SECURITY;

CREATE POLICY shared_knowledge_read ON shared_knowledge
    FOR SELECT TO ai_app_user
    USING (
        tenant_id IS NULL  -- global knowledge
        OR tenant_id = current_setting('app.current_tenant_id')::UUID
    );

CREATE POLICY shared_knowledge_write ON shared_knowledge
    FOR INSERT TO ai_app_user
    WITH CHECK (
        current_setting('app.is_tenant_admin')::BOOLEAN = TRUE
        AND (tenant_id IS NULL OR tenant_id = current_setting('app.current_tenant_id')::UUID)
    );

-- Semantic search function that automatically respects user context
-- RLS handles the isolation; this function provides a clean API
CREATE OR REPLACE FUNCTION search_user_memory(
    query_embedding VECTOR(1536),
    p_limit INTEGER DEFAULT 5,
    p_threshold FLOAT DEFAULT 0.7
) RETURNS TABLE (
    source          TEXT,
    chunk_text      TEXT,
    similarity      FLOAT,
    metadata        JSONB
)
LANGUAGE sql
STABLE
AS $$
    -- Search private user embeddings
    SELECT
        'private'::TEXT as source,
        ue.chunk_text,
        1 - (ue.embedding <=> query_embedding) as similarity,
        ue.metadata
    FROM user_embeddings ue
    WHERE 1 - (ue.embedding <=> query_embedding) >= p_threshold
    -- RLS automatically filters to current user — no explicit WHERE needed
    UNION ALL
    -- Search shared knowledge (tenant + global)
    SELECT
        'shared'::TEXT as source,
        sk.chunk_text,
        1 - (sk.embedding <=> query_embedding) as similarity,
        sk.metadata
    FROM shared_knowledge sk
    WHERE 1 - (sk.embedding <=> query_embedding) >= p_threshold
    -- RLS automatically filters to current tenant
    ORDER BY similarity DESC
    LIMIT p_limit;
$$;

The search_user_memory function looks dangerously simple — no explicit user_id filter anywhere. That is intentional and correct. RLS policies on user_embeddings and shared_knowledge enforce the context set at the start of the transaction. The function just expresses the query shape; PostgreSQL's policy engine does the enforcement transparently.
