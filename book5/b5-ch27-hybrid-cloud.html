Chapter 27: Hybrid Cloud: On-Premises and Cloud Together

Most production environments do not migrate to the cloud in a single weekend. They live in the space between — some databases running on-premises in your data center, some running in RDS or Azure SQL or Cloud SQL, and the two worlds needing to communicate as if the network boundary does not exist. This chapter is about that in-between state: how to architect it correctly, how to keep data synchronized across boundaries, how to handle failover and latency, and how to run hybrid configurations in production without turning them into a maintenance nightmare. If you are three months into a cloud migration and still have seventeen on-premises SQL Server instances feeding data to Azure SQL, this chapter is written for your situation.

---

27.1 Network Architecture: Connecting Your Data Center to the Cloud

The foundation of any hybrid database architecture is the network. Before a single row of data moves between your on-premises PostgreSQL instance and RDS, before DMS can reach your source database, and before an application server in AWS can query your on-premises Oracle, the network path must be stable, low-latency, and secured. This is not optional plumbing — it is the structural prerequisite that determines whether your hybrid architecture works or falls apart under production load.

#### The Two Options: VPN vs. Direct Connect (and Their Azure/GCP Equivalents)

AWS gives you two primary options for connecting on-premises infrastructure to your VPC: AWS Site-to-Site VPN and AWS Direct Connect.

Site-to-Site VPN runs over the public internet, uses IPSec encryption, and is inexpensive to set up. Bandwidth is limited to approximately 1.25 Gbps per tunnel, latency is variable (typically 20–100ms depending on geography and ISP), and it is prone to jitter. For database replication where your logs are small and your RPO tolerance is measured in minutes, VPN is acceptable. For bulk migration of a 20 TB database or for real-time streaming replication where lag matters in seconds, VPN will create problems.

AWS Direct Connect provides a dedicated physical connection from your data center to an AWS Direct Connect location. It delivers 1 Gbps, 10 Gbps, or 100 Gbps connections with consistent latency (typically 2–10ms from most data centers to a nearby AWS region). You pay for port hours and data transfer, but you eliminate the jitter and bandwidth contention of internet-based VPN. For production hybrid database architectures where you are replicating continuously or running applications that span on-premises and cloud, Direct Connect is the correct choice.

Azure's equivalents are Azure VPN Gateway (VPN) and Azure ExpressRoute (dedicated circuit). GCP offers Cloud VPN and Cloud Interconnect (Dedicated or Partner). The architectural decisions are identical regardless of cloud provider — VPN for non-critical or low-bandwidth scenarios, dedicated connectivity for production.

#### Setting Up an AWS Site-to-Site VPN to Your Data Center

# Step 1: Create a Customer Gateway representing your on-premises VPN device
aws ec2 create-customer-gateway \
  --type ipsec.1 \
  --public-ip 203.0.113.12 \
  --bgp-asn 65000 \
  --region us-east-1 \
  --tag-specifications 'ResourceType=customer-gateway,Tags=[{Key=Name,Value=datacenter-primary}]'

# Step 2: Create a Virtual Private Gateway and attach it to your VPC
aws ec2 create-vpn-gateway \
  --type ipsec.1 \
  --amazon-side-asn 64512 \
  --region us-east-1

# Capture the VpnGatewayId from the output, then attach
aws ec2 attach-vpn-gateway \
  --vpn-gateway-id vgw-0abc123def456789a \
  --vpc-id vpc-0123456789abcdef0

# Step 3: Create the Site-to-Site VPN connection
aws ec2 create-vpn-connection \
  --type ipsec.1 \
  --customer-gateway-id cgw-0abc123def456789a \
  --vpn-gateway-id vgw-0abc123def456789a \
  --options '{"StaticRoutesOnly":false}' \
  --region us-east-1

# Step 4: Enable route propagation so VPC route tables learn on-premises CIDRs
aws ec2 enable-vgw-route-propagation \
  --route-table-id rtb-0123456789abcdef0 \
  --gateway-id vgw-0abc123def456789a

Once the VPN is established, your RDS instances inside the VPC can communicate with on-premises servers using private IP addresses. Your security groups control what traffic is allowed — treat your on-premises CIDR block (say, 10.0.0.0/8) as a trusted network source, but be precise about which ports you open. Port 5432 for PostgreSQL, 1433 for SQL Server, 3306 for MySQL — no broader rules.

#### Private Link for Specific Service Exposure

If you do not want to route your entire data center network into a VPC, AWS PrivateLink lets you expose a single service endpoint across a network boundary without full VPC peering or VPN. This is useful when an on-premises application needs to reach a specific RDS endpoint and nothing else.

Azure Private Link functions identically for Azure SQL: you create a private endpoint inside a VNet, and the on-premises network (connected via ExpressRoute or VPN Gateway) reaches the database through that private IP rather than the public Azure SQL endpoint.

# Create a PrivateLink endpoint for an RDS instance
# First, the VPC endpoint service must exist — RDS uses Interface Endpoints

aws ec2 create-vpc-endpoint \
  --vpc-endpoint-type Interface \
  --vpc-id vpc-0123456789abcdef0 \
  --service-name com.amazonaws.us-east-1.rds \
  --subnet-ids subnet-0abc123def456789a subnet-0abc123def456790b \
  --security-group-ids sg-0abc123def456789a \
  --private-dns-enabled \
  --region us-east-1

#### What Latency Actually Means for Your Databases

When your application is in AWS and your database is on-premises (or vice versa), every synchronous query crosses the network. A 20ms round-trip added to every SELECT statement compounds quickly when an ORM generates 40 queries per page load. The math is brutal: 40 queries × 20ms = 800ms of pure network time before any query execution.

Measure your actual latency before you commit to any hybrid query pattern. Run a simple ping and then a pgbench or sysbench test across the VPN or Direct Connect circuit before you design your application topology. If your latency is above 5ms, co-locate the application with its primary database. Use async replication and caching to bridge the gap rather than synchronous cross-boundary queries.

27.2 Data Synchronization Patterns Across Boundaries

With the network established, the next problem is data: how do you keep data synchronized between an on-premises database and its cloud counterpart? The answer depends entirely on whether you need bidirectional synchronization, unidirectional replication, or bulk periodic loads — and on how much lag you can tolerate.

#### AWS Database Migration Service for Ongoing Replication

AWS DMS is covered in detail in Chapter 25 in the context of migration. In hybrid architectures, DMS is also a production replication tool. You can configure a replication instance, a source endpoint pointing at your on-premises PostgreSQL or SQL Server, and a target endpoint pointing at RDS, and run DMS in continuous replication (CDC) mode indefinitely.

The on-premises PostgreSQL source requires logical replication enabled and a replication slot. For SQL Server, the source must have MS-CDC enabled on the database and each table you want to replicate.

# Create DMS replication instance in the VPC that has connectivity to on-premises
aws dms create-replication-instance \
  --replication-instance-identifier hybrid-repl-01 \
  --replication-instance-class dms.r6i.xlarge \
  --allocated-storage 200 \
  --vpc-security-group-ids sg-0abc123def456789a \
  --replication-subnet-group-identifier my-dms-subnet-group \
  --multi-az \
  --engine-version 3.5.2 \
  --publicly-accessible false

# Source endpoint: on-premises PostgreSQL at 10.10.5.20
aws dms create-endpoint \
  --endpoint-identifier onprem-pg-source \
  --endpoint-type source \
  --engine-name postgres \
  --username dms_replication_user \
  --password 'SecureP@ssword123' \
  --server-name 10.10.5.20 \
  --port 5432 \
  --database-name production_db \
  --extra-connection-attributes "captureDdls=true;pluginName=pglogical"

# Target endpoint: RDS PostgreSQL
aws dms create-endpoint \
  --endpoint-identifier rds-pg-target \
  --endpoint-type target \
  --engine-name postgres \
  --username dms_target_user \
  --password 'SecureP@ssword123' \
  --server-name mydb.c9akciq32.us-east-1.rds.amazonaws.com \
  --port 5432 \
  --database-name production_db

# Create and start the replication task in CDC mode
aws dms create-replication-task \
  --replication-task-identifier hybrid-cdc-task \
  --source-endpoint-arn arn:aws:dms:us-east-1:123456789012:endpoint:ABCDEF... \
  --target-endpoint-arn arn:aws:dms:us-east-1:123456789012:endpoint:GHIJKL... \
  --replication-instance-arn arn:aws:dms:us-east-1:123456789012:rep:MNOPQR... \
  --migration-type cdc \
  --table-mappings file://table-mappings.json \
  --replication-task-settings file://task-settings.json

Monitor your DMS task latency continuously. In hybrid configurations, replication lag above 60 seconds is a warning threshold — above 300 seconds, investigate immediately. DMS exposes CDCLatencySource and CDCLatencyTarget metrics in CloudWatch.

import boto3

cloudwatch = boto3.client('cloudwatch', region_name='us-east-1')

def get_dms_replication_lag(task_arn_suffix: str) -> dict:
    """
    Fetch CDC latency metrics for a DMS replication task.
    task_arn_suffix is the last segment of the replication task ARN.
    """
    response = cloudwatch.get_metric_statistics(
        Namespace='AWS/DMS',
        MetricName='CDCLatencyTarget',
        Dimensions=[
            {
                'Name': 'ReplicationInstanceIdentifier',
                'Value': 'hybrid-repl-01'
            },
            {
                'Name': 'ReplicationTaskIdentifier',
                'Value': 'hybrid-cdc-task'
            }
        ],
        StartTime='2024-03-01T00:00:00Z',
        EndTime='2024-03-01T01:00:00Z',
        Period=60,
        Statistics=['Average', 'Maximum']
    )
    
    datapoints = response['Datapoints']
    if not datapoints:
        return {'status': 'no_data', 'lag_seconds': None}
    
    latest = sorted(datapoints, key=lambda x: x['Timestamp'])[-1]
    lag = latest['Maximum']
    
    status = 'healthy'
    if lag > 60:
        status = 'warning'
    if lag > 300:
        status = 'critical'
    
    return {
        'status': status,
        'lag_seconds': lag,
        'timestamp': latest['Timestamp'].isoformat()
    }

result = get_dms_replication_lag('hybrid-cdc-task')
print(f"DMS replication lag: {result['lag_seconds']}s — Status: {result['status']}")

#### Native Replication as an Alternative to DMS

DMS is convenient but it adds a layer of abstraction with its own failure modes. For PostgreSQL-to-PostgreSQL hybrid replication, you can use native logical replication directly — your on-premises primary publishes, your RDS instance subscribes.

-- On the on-premises PostgreSQL primary (source):
-- Ensure wal_level = logical in postgresql.conf
-- Then create a publication for the tables you want to replicate

CREATE PUBLICATION hybrid_pub
FOR TABLE
    orders,
    order_items,
    customers,
    products
WITH (publish = 'insert, update, delete');

-- Grant replication privilege to the user RDS will connect as
CREATE USER rds_subscriber WITH REPLICATION LOGIN PASSWORD 'ReplicaP@ss456';
GRANT SELECT ON ALL TABLES IN SCHEMA public TO rds_subscriber;
-- On RDS PostgreSQL (target) — the subscriber side:
-- Tables must already exist with matching schema

CREATE SUBSCRIPTION hybrid_sub
CONNECTION 'host=10.10.5.20 port=5432 dbname=production_db user=rds_subscriber password=ReplicaP@ss456'
PUBLICATION hybrid_pub
WITH (
    copy_data = true,
    create_slot = true,
    enabled = true,
    connect = true
);

-- Verify subscription status
SELECT subname, subenabled, subslotname, subpublications
FROM pg_subscription;

-- Check replication slot on the source to monitor lag
-- Run this on the on-premises primary:
SELECT slot_name,
       confirmed_flush_lsn,
       pg_current_wal_lsn() - confirmed_flush_lsn AS lag_bytes,
       pg_size_pretty(pg_current_wal_lsn() - confirmed_flush_lsn) AS lag_pretty
FROM pg_replication_slots
WHERE slot_type = 'logical';

The important difference between native logical replication and DMS: native replication maintains a replication slot on the source. If the connection drops or the subscriber falls behind, the source accumulates WAL files to serve those changes when the slot reconnects. On-premises servers often have less disk than cloud instances — an unconsumed replication slot can fill your pg_wal directory and crash your primary. Set max_slot_wal_keep_size in PostgreSQL 13+ to cap this growth.

#### Azure Data Sync for SQL Server Hybrid

Azure SQL Data Sync is Microsoft's native solution for bidirectional synchronization between on-premises SQL Server and Azure SQL Database. Unlike DMS (which is unidirectional from source to target), Data Sync can handle bidirectional sync — useful when applications write to both the on-premises database and Azure SQL simultaneously.

The configuration involves a Hub database in Azure SQL, a Data Sync Agent installed on your on-premises server, and member databases at both locations. Sync happens on a configurable schedule (minimum 5 minutes) rather than in real time. This is not a replication solution for low-lag requirements — it is a synchronization solution for scenarios where eventual consistency over minutes is acceptable.

Know the limits: Azure Data Sync does not support TRUNCATE, does not replicate DDL changes, has a 500-column limit per table, and cannot handle tables without primary keys. These are hard limits, not configuration options.

27.3 Hybrid Failover and Disaster Recovery

The most valuable use case for hybrid architecture is disaster recovery: your on-premises data center serves as DR for your cloud environment, or cloud serves as DR for your on-premises production. Either direction requires careful design around failover triggering, DNS cutover, and data divergence.

#### Cloud as DR for On-Premises Production

In this pattern, your on-premises PostgreSQL or SQL Server is the primary. The cloud database (RDS, Azure SQL) is a continuously-updated replica. If your data center fails, you promote the cloud replica, update your DNS or application connection strings, and resume operations.

The technical requirements are straightforward. The architectural challenges are operational: who decides to trigger failover? What is the SLA that justifies activating it? How do you handle applications that are split between on-premises compute and the cloud replica?

For PostgreSQL, native streaming replication or logical replication keeps the RDS replica current. Monitor pg_stat_replication on the primary and alert when lag exceeds your RPO threshold.

-- Run on the on-premises primary to monitor replica lag toward RDS
SELECT
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    sent_lsn - replay_lsn AS total_lag_bytes,
    pg_size_pretty(sent_lsn - replay_lsn) AS lag_pretty,
    write_lag,
    flush_lag,
    replay_lag
FROM pg_stat_replication
WHERE application_name = 'rds_replica';

-- If lag_pretty exceeds your RPO threshold (e.g., 100MB ~ 60s at typical write rates),
-- alert your on-call DBA before it becomes a recovery gap problem.

When you need to promote the RDS replica after an on-premises failure, the process for RDS PostgreSQL involves breaking the replication and promoting the read replica:

# Promote the RDS Read Replica to a standalone instance
aws rds promote-read-replica \
  --db-instance-identifier rds-dr-replica \
  --backup-retention-period 7 \
  --preferred-backup-window "03:00-04:00" \
  --region us-east-1

# Monitor until the instance status becomes "available"
aws rds wait db-instance-available \
  --db-instance-identifier rds-dr-replica

# Update the CNAME record for your application database endpoint
# (Using Route 53 here — adapt to your DNS provider)
aws route53 change-resource-record-sets \
  --hosted-zone-id Z1D633PJN98FT9 \
  --change-batch '{
    "Changes": [{
      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "db.internal.example.com",
        "Type": "CNAME",
        "TTL": 60,
        "ResourceRecords": [{"Value": "rds-dr-replica.c9akciq32.us-east-1.rds.amazonaws.com"}]
      }
    }]
  }'

Set your DNS TTL to 60 seconds — not 300 or 3600. When you need to cut over, you want the change to propagate in one minute, not five.

#### On-Premises as DR for Cloud Production

This is less common but occurs when regulatory requirements mandate that a copy of data remain in a specific physical location, or when cloud costs make on-premises a more economical secondary tier.

For RDS PostgreSQL, you cannot configure standard streaming replication *from* RDS to an external server the way you would between two self-managed instances. RDS does not expose the primary's WAL stream for external replication subscription without specific configuration. You have two options:

1. Use RDS logical replication (rds.logical_replication = 1 in the parameter group) to publish from RDS to an on-premises PostgreSQL subscriber.

2. Use AWS DMS in reverse — RDS as the source, on-premises as the target.

# Enable logical replication on RDS PostgreSQL via Parameter Group
aws rds modify-db-parameter-group \
  --db-parameter-group-name production-pg15-params \
  --parameters 'ParameterName=rds.logical_replication,ParameterValue=1,ApplyMethod=pending-reboot'

# Reboot is required for this parameter
aws rds reboot-db-instance \
  --db-instance-identifier prod-pg-primary

# After reboot, on the RDS instance, create the publication:
# (Connect via psql first)
# CREATE PUBLICATION onprem_dr_pub FOR ALL TABLES;

# On the on-premises PostgreSQL DR server, create the subscription:
# CREATE SUBSCRIPTION onprem_dr_sub
# CONNECTION 'host=prod-pg-primary.c9akciq32.us-east-1.rds.amazonaws.com
#             port=5432 dbname=production_db
#             user=replication_user password=SecurePassword'
# PUBLICATION onprem_dr_pub;

27.4 Hybrid Identity and Access Management

Network connectivity handles the infrastructure layer. Identity handles the security layer. In your on-premises data center, you used pg_hba.conf, Active Directory service accounts, or LDAP for database authentication. In the cloud, IAM controls who and what can connect. Hybrid environments need a coherent identity strategy that spans both worlds.

#### AWS IAM Database Authentication Across Boundaries

RDS supports IAM authentication for PostgreSQL and MySQL. Instead of a username/password, the application generates a short-lived authentication token using its AWS IAM credentials. The token is valid for 15 minutes. This works perfectly for applications running inside AWS. For on-premises applications, it requires that the on-premises server has access to IAM credentials — either through an IAM user's access key (acceptable for service accounts, not ideal) or through AWS STS and cross-account role assumption.

The practical pattern: create an IAM user specifically for on-premises database access, assign it a policy that allows rds-db:connect for the specific RDS resource and database user, and rotate the access keys using AWS Secrets Manager rotation.

import boto3
import psycopg2

def connect_to_rds_from_onpremises(
    db_host: str,
    db_port: int,
    db_name: str,
    db_user: str,
    region: str,
    aws_access_key_id: str,
    aws_secret_access_key: str
) -> psycopg2.extensions.connection:
    """
    Connect to RDS PostgreSQL from an on-premises server using IAM authentication.
    The IAM user must have rds-db:connect permission on the db resource.
    """
    # Generate the auth token using STS credentials
    session = boto3.Session(
        aws_access_key_id=aws_access_key_id,
        aws_secret_access_key=aws_secret_access_key,
        region_name=region
    )
    
    rds_client = session.client('rds')
    
    # This generates a token valid for 15 minutes
    auth_token = rds_client.generate_db_auth_token(
        DBHostname=db_host,
        Port=db_port,
        DBUsername=db_user,
        Region=region
    )
    
    # Connect using the token as the password
    # SSL is required when using IAM auth
    conn = psycopg2.connect(
        host=db_host,
        port=db_port,
        database=db_name,
        user=db_user,
        password=auth_token,
        sslmode='verify-full',
        sslrootcert='/path/to/us-east-1-bundle.pem'
    )
    
    return conn

# Usage from on-premises server
connection = connect_to_rds_from_onpremises(
    db_host='prod-rds.c9akciq32.us-east-1.rds.amazonaws.com',
    db_port=5432,
    db_name='production',
    db_user='app_iam_user',
    region='us-east-1',
    aws_access_key_id='AKIAIOSFODNN7EXAMPLE',
    aws_secret_access_key='wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY'
)
