Chapter 17: Zero-Touch High Availability

High availability in autonomous systems is not a feature you configure once and monitor manually — it is a continuous, self-executing contract between your infrastructure and your uptime SLA. The moment a human must intervene to restore service, you have failed the ADE standard. This chapter covers the architecture and implementation of HA systems that detect failure, make promotion decisions, reroute traffic, reintegrate replicas, and validate their own health — all without a pager going off at 2 a.m. The patterns here apply across Patroni-managed PostgreSQL clusters, SQL Server Always On Availability Groups orchestrated by Kubernetes operators, and hybrid topologies that span on-premises and cloud. The goal is not just fast failover — it is *correct*, *reversible*, and *observable* failover that a system can execute and explain without human involvement.

---

17.1 The Anatomy of a Zero-Touch Failover Event

Before automating failover, you must model every failure mode precisely. Autonomous systems do not fail over on a hunch — they fail over when a constellation of signals crosses a decision threshold. The difference between autonomous HA and naive auto-failover is the difference between a surgeon and a reflex: one evaluates context, the other reacts to a single stimulus.

The canonical failure taxonomy for HA systems has three tiers:

Tier 1 — Node failure. The primary database process is unreachable. This is unambiguous at the OS level: no TCP response on the database port, no heartbeat from the HA agent, process not running.

Tier 2 — Network partition. The primary is alive but cannot be reached by replicas or the load balancer. This is the most dangerous case for autonomous systems — a false positive here causes a split-brain condition where two nodes believe they are primary.

Tier 3 — Performance degradation masquerading as failure. The primary is responding, but query latency is so high that applications are timing out. Connections are queuing. Replication lag is climbing. This is not a hard failure — it is a soft failure that an autonomous system must distinguish from a real outage.

A production autonomous HA system evaluates signals across all three tiers before acting. The decision matrix looks like this:

| Signal | Source | Threshold | Weight |

|---|---|---|---|

| TCP connectivity | HAProxy / pgBouncer health check | 3 consecutive failures | High |

| Replication lag | pg_stat_replication / AG DMVs | > 30 seconds | Medium |

| Replica quorum | Patroni DCS / WSFC | Majority unreachable | High |

| Transaction throughput | Prometheus pg_stat_database | Drop > 80% in 60s | Medium |

| OS-level process | Patroni watchdog / WSFC | Process absent | High |

An autonomous system acts only when the aggregate signal exceeds a confidence threshold — typically, a combination of at least two High-weight signals or one High plus two Medium signals. This prevents failover storms triggered by transient network blips.

PostgreSQL — querying replication state for decision input:

-- Run on replica; used by the HA agent to evaluate whether primary is healthy
SELECT
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    (sent_lsn - replay_lsn)           AS replication_lag_bytes,
    extract(epoch FROM (now() - reply_time)) AS seconds_since_heartbeat
FROM pg_stat_replication
WHERE application_name = 'replica1';

SQL Server — querying AG synchronization state for decision input:

-- Run on any replica; exposes per-replica health for the orchestration layer
SELECT
    ag.name                                  AS ag_name,
    ar.replica_server_name,
    ars.role_desc,
    ars.synchronization_health_desc,
    ars.connected_state_desc,
    ars.operational_state_desc,
    drs.synchronization_state_desc,
    drs.log_send_queue_size                  AS send_queue_kb,
    drs.redo_queue_size                      AS redo_queue_kb,
    drs.last_commit_time
FROM sys.availability_groups ag
JOIN sys.availability_replicas ar
    ON ag.group_id = ar.group_id
JOIN sys.dm_hadr_availability_replica_states ars
    ON ar.replica_id = ars.replica_id
JOIN sys.dm_hadr_database_replica_states drs
    ON ar.replica_id = drs.replica_id
ORDER BY ag.name, ar.replica_server_name;

These queries are not run interactively — they are ingested every 10 seconds by the HA orchestration layer, stored as time-series metrics, and evaluated against the decision matrix. The HA agent is a consumer of this data, not a human.

17.2 Patroni and the DCS Contract: Autonomous Promotion Architecture

Patroni is the de facto autonomous HA framework for PostgreSQL at scale. Its architecture is worth studying not just as a tool but as a design pattern: it externalizes consensus to a Distributed Configuration Store (DCS — etcd, Consul, or ZooKeeper), making the cluster's leadership state a shared, durable, observable fact rather than an assumption held by individual nodes.

The Patroni architecture for zero-touch HA rests on three contracts:

The Leadership Lease. The primary holds a TTL-bound key in the DCS. If the primary fails to renew the lease within the TTL window (typically 30 seconds), the key expires and replicas become eligible for promotion. No external signal is needed — the absence of a heartbeat is itself the trigger.

The Candidate Election. When the leadership key expires, replicas race to acquire it. The winning replica must have the highest pg_last_wal_receive_lsn() among candidates — Patroni enforces this. The system promotes the replica with the most current WAL position, minimizing data loss.

The Watchdog Contract. Patroni runs a kernel-level watchdog daemon on the primary. If the Patroni process itself hangs (but the PostgreSQL process is still running), the watchdog reboots the node before the DCS lease expires. This prevents a scenario where a frozen Patroni agent holds a lease while being unable to serve writes — a subtle split-brain vector.

The Patroni configuration for autonomous failover behavior is dense with tunable parameters that directly affect the decision logic:

# patroni.yml — production autonomous HA configuration
bootstrap:
  dcs:
    ttl: 30                        # Primary lease TTL in seconds
    loop_wait: 10                  # DCS polling interval
    retry_timeout: 10              # DCS operation timeout
    maximum_lag_on_failover: 1048576  # Max replica lag (bytes) to be eligible for promotion
    synchronous_mode: true         # Only synchronous replicas can be promoted
    synchronous_mode_strict: false # If true, blocks writes when no sync replica is available

postgresql:
  parameters:
    wal_level: replica
    max_wal_senders: 10
    wal_keep_size: 1GB
    hot_standby: on
    synchronous_commit: on

watchdog:
  mode: required                   # Fail hard if watchdog is unavailable
  device: /dev/watchdog
  safety_margin: 5

The maximum_lag_on_failover parameter is critical for autonomous operation. If a replica is more than 1 MB behind the primary's WAL at the time of failover, Patroni excludes it from election. This is a data-safety gate that prevents promoting a significantly stale replica — a decision that would otherwise require human judgment.

When synchronous_mode: true is set, Patroni dynamically manages synchronous_standby_names in PostgreSQL. If the designated synchronous replica falls behind or disconnects, Patroni re-evaluates the sync replica set and updates the parameter without human intervention. This is autonomous configuration management driven by replication health signals.

Python — Patroni REST API integration for autonomous monitoring:

import httpx
import time
import logging
from dataclasses import dataclass
from typing import Optional

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("patroni_monitor")

@dataclass
class PatroniClusterState:
    leader: str
    members: list
    failover_in_progress: bool
    sync_standby: Optional[str]

PATRONI_ENDPOINTS = [
    "http://pg-node1:8008",
    "http://pg-node2:8008",
    "http://pg-node3:8008",
]

def fetch_cluster_state() -> Optional[PatroniClusterState]:
    """
    Query each Patroni node until one responds.
    In a real split-brain scenario, nodes may disagree on leadership —
    this function returns the DCS-authoritative view, not node self-report.
    """
    for endpoint in PATRONI_ENDPOINTS:
        try:
            r = httpx.get(f"{endpoint}/cluster", timeout=5.0)
            if r.status_code == 200:
                data = r.json()
                leader = next(
                    (m["name"] for m in data["members"] if m["role"] == "leader"), None
                )
                sync = next(
                    (m["name"] for m in data["members"] if m["role"] == "sync_standby"),
                    None,
                )
                return PatroniClusterState(
                    leader=leader,
                    members=data["members"],
                    failover_in_progress=data.get("failover", False),
                    sync_standby=sync,
                )
        except (httpx.RequestError, KeyError) as e:
            logger.warning(f"Could not reach {endpoint}: {e}")
    return None

def evaluate_and_alert(state: PatroniClusterState):
    lagging = [
        m for m in state.members
        if m.get("lag", 0) > 1_048_576 and m["role"] != "leader"
    ]
    if lagging:
        logger.warning(
            f"Replicas exceeding failover lag threshold: "
            f"{[m['name'] for m in lagging]}"
        )
    if not state.sync_standby:
        logger.error(
            "No synchronous standby present — cluster is operating in async mode. "
            "RPO guarantee is degraded."
        )

if __name__ == "__main__":
    while True:
        state = fetch_cluster_state()
        if state:
            evaluate_and_alert(state)
        time.sleep(15)

This script is not a replacement for Patroni's own decision-making — it is a secondary observer that feeds into an alerting and audit trail, allowing the autonomous system to record *why* it made a failover decision in language that a post-incident review can parse.

17.3 SQL Server Always On and the Kubernetes Operator Model

SQL Server's Always On Availability Groups provide the same leadership-election semantics as Patroni, but with a different operational model. In Windows Server Failover Cluster (WSFC) deployments, quorum is managed by a cluster quorum resource — a concept structurally identical to Patroni's DCS lease. In containerized deployments (SQL Server on Kubernetes), the SQL Server Operator from the mssql-operator or community sql-server-k8s-operator replaces WSFC entirely.

The Kubernetes operator model is significant because it brings SQL Server HA into the same orchestration plane as the rest of the application stack. An operator is a custom Kubernetes controller that watches a Custom Resource Definition (CRD) representing the AG's desired state and continuously reconciles actual state to match it. When a primary pod dies, Kubernetes detects the pod failure, the operator observes the AG state change via the DMVs, elects a new primary from the healthy replicas, and updates the Service object that application connections point to — all within the reconciliation loop.

The AG CRD for an autonomous deployment looks conceptually like this in YAML:

apiVersion: sql.microsoft.com/v1
kind: SqlServer
metadata:
  name: sqlag-primary
spec:
  saPassword: "$(SA_PASSWORD)"
  edition: Developer
---
apiVersion: sql.microsoft.com/v1
kind: AvailabilityGroup
metadata:
  name: production-ag
spec:
  name: ProductionAG
  databases:
    - orders
    - inventory
  replicas:
    - server: sqlag-primary
      availabilityMode: SYNCHRONOUS_COMMIT
      failoverMode: AUTOMATIC
    - server: sqlag-replica1
      availabilityMode: SYNCHRONOUS_COMMIT
      failoverMode: AUTOMATIC
    - server: sqlag-replica2
      availabilityMode: ASYNCHRONOUS_COMMIT
      failoverMode: MANUAL
  listenerPort: 1433

The AUTOMATIC failover mode on synchronous replicas is what enables zero-touch behavior. When the operator detects that the primary replica is in a NOT_SYNCHRONIZING or DISCONNECTED state for longer than the configured health check threshold, it executes the equivalent of ALTER AVAILABILITY GROUP ... FORCE_FAILOVER_ALLOW_DATA_LOSS or a clean automatic failover depending on whether data loss is acceptable.

For the operator to make this decision autonomously, it must have clear policy guidance baked into the CRD:

SQL Server — verifying AG readiness for automatic failover:

-- Validates that both synchronous replicas are healthy and within failover policy
SELECT
    ag.name                             AS availability_group,
    ar.replica_server_name,
    ar.availability_mode_desc,
    ar.failover_mode_desc,
    ars.synchronization_health_desc,
    ars.is_local,
    ars.role_desc,
    CASE
        WHEN ar.failover_mode_desc = 'AUTOMATIC'
         AND ars.synchronization_health_desc = 'HEALTHY'
        THEN 'ELIGIBLE_FOR_AUTO_FAILOVER'
        ELSE 'NOT_ELIGIBLE'
    END                                 AS failover_eligibility
FROM sys.availability_groups ag
JOIN sys.availability_replicas ar
    ON ag.group_id = ar.group_id
JOIN sys.dm_hadr_availability_replica_states ars
    ON ar.replica_id = ars.replica_id
WHERE ag.name = 'ProductionAG';

A critical architectural consideration in zero-touch SQL Server HA is the handling of the AG listener after failover. The listener is a virtual network name (VNN) that clients connect to. In WSFC, the cluster updates DNS automatically. In Kubernetes, the operator updates the ClusterIP Service. In both cases, the connection rerouting is transparent to application code — but only if applications are using the AG listener or Service name, never the direct node IP. Autonomous HA systems enforce this at the infrastructure layer: direct-node connection strings are blocked by network policy, forcing all traffic through the abstraction layer that the operator controls.

17.4 Traffic Rerouting Without Human Intervention: pgBouncer, HAProxy, and Service Mesh Patterns

Promoting a replica to primary is half the problem. The other half is getting client connections to the new primary before application-level timeouts cascade into a service outage. In autonomous systems, connection rerouting must happen in under 5 seconds in most SLA frameworks — which rules out DNS TTL-based approaches for latency-sensitive workloads.

The standard architecture for zero-touch traffic rerouting in PostgreSQL uses a layered approach:

Application → pgBouncer (connection pooler) → HAProxy (TCP load balancer) → PostgreSQL nodes

HAProxy performs active health checks against each PostgreSQL node and routes write traffic only to the node responding 200 on the Patroni REST /primary endpoint:

# haproxy.cfg — relevant sections for autonomous primary routing
frontend postgres_write
    bind *:5432
    mode tcp
    default_backend postgres_primary

backend postgres_primary
    mode tcp
    option httpchk GET /primary
    http-check expect status 200
    default-server inter 2s fall 3 rise 2 on-marked-down shutdown-sessions
    server pg-node1 10.0.1.1:5432 check port 8008
    server pg-node2 10.0.1.2:5432 check port 8008
    server pg-node3 10.0.1.3:5432 check port 8008

backend postgres_replicas
    mode tcp
    balance roundrobin
    option httpchk GET /replica
    http-check expect status 200
    server pg-node1 10.0.1.1:5432 check port 8008
    server pg-node2 10.0.1.2:5432 check port 8008
    server pg-node3 10.0.1.3:5432 check port 8008

The key directive here is on-marked-down shutdown-sessions. When HAProxy's health check marks the current primary as down, it immediately tears down all existing connections to that node. This forces pgBouncer to re-establish connections, which then hit the backend pool and land on the new primary (which is now returning 200 on /primary). The total rerouting time is bounded by: HAProxy check interval (2s) × fall threshold (3) = 6 seconds worst case.

For workloads that cannot tolerate even 6 seconds of connection disruption, a service mesh approach using Envoy or Istio reduces this to sub-second. An Envoy health check against the Patroni endpoint runs at 500ms intervals, and active request draining ensures in-flight transactions are completed before the connection is terminated. The trade-off is operational complexity — service mesh HA configuration is significantly more involved than HAProxy.

pgBouncer sits in front of HAProxy in this architecture to manage connection multiplexing and to handle the reconnection storm that follows a failover. Without pgBouncer, hundreds of application connections simultaneously attempting to reconnect to the new primary can overwhelm it. pgBouncer's server_login_retry and query_wait_timeout parameters absorb the reconnection burst:

# pgbouncer.ini — parameters relevant to failover resilience
[pgbouncer]
pool_mode = transaction
server_login_retry = 0.5         ; Retry server connection every 500ms
query_wait_timeout = 15          ; Hold client connections up to 15s waiting for a server
server_connect_timeout = 5       ; Give up on a server connection after 5s
server_check_delay = 10          ; Verify server health every 10s
server_check_query = SELECT 1    ; Health check query

The query_wait_timeout = 15 setting is particularly important: it means that during the 6-second HAProxy switchover window, client queries are queued in pgBouncer rather than failing with connection errors. From the application's perspective, queries slow down for a few seconds rather than returning errors. This is the correct behavior for a zero-touch system — transient slowness is acceptable; connection errors are not.

PostgreSQL — verifying connection routing post-failover:

-- Run against the new primary after promotion
-- Confirms that application traffic has successfully rerouted
SELECT
    datname,
    usename,
    application_name,
    client_addr,
    state,
    count(*) AS connection_count,
    max(extract(epoch FROM (now() - state_change))) AS max_seconds_in_state
FROM pg_stat_activity
WHERE datname NOT IN ('postgres', 'template0', 'template1')
  AND state != 'idle'
GROUP BY datname, usename, application_name, client_addr, state
ORDER BY connection_count DESC;

This query is executed by the post-failover validation routine — not a human — to confirm that application connections have landed on the new primary and are in active rather than idle in transaction state, which would indicate stuck transactions from the failed primary.

17.5 Self-Healing Replica Reintegration After Failover

Failover is a one-way event unless the system also handles replica reintegration autonomously. After a primary fails and is promoted away, the former primary — once recovered — must rejoin the cluster as a replica. This process involves: detecting that the former primary is back online, identifying its divergence point from the new primary's WAL timeline, rebuilding the data directory if necessary, and streaming replication from the new primary. In Patroni, this is handled natively through the pg_rewind integration. In SQL Server, the AG automatically re-synchronizes a rejoined replica.

The Patroni pg_rewind flow works as follows: when the former primary comes back online, Patroni checks whether its WAL history diverges from the current primary's timeline. If the divergence is within the rewind window (controlled by wal_log_hints = on and sufficient WAL on the primary), Patroni executes pg_rewind to synchronize the data directory to the branch point and then restarts streaming replication. If pg_rewind cannot reconcile the divergence — because the primary has already checkpointed past the divergence point and WAL is no longer available — Patroni falls back to pg_basebackup to rebuild the replica from scratch.

Monitoring the reintegration process autonomously requires tracking WAL receiver state and replication lag from the moment the former primary reconnects:

PostgreSQL — tracking replica reintegration progress:

-- Run on the rejoining node during pg_rewind / basebackup recovery
SELECT
    status,
    receive_start_lsn,
    receive_start_tli,
    written_lsn,
    flushed_lsn,
    received_tli,
    last_msg_send_time,
    last_msg_receipt_time,
    latest_end_lsn,
    latest_end_time,
    extract(epoch FROM (now() - last_msg_receipt_time)) AS seconds_since_last_message,
    slot_name,
    sender_host
FROM pg_stat_wal_receiver;

The autonomous reintegration policy is: if seconds_since_last_message exceeds 60 seconds during reintegration, trigger a Patroni restart on the node and retry. If three restart attempts fail, quarantine the node — remove it from the HAProxy backend pool, flag it in the DCS, and generate a human-escalation alert. This is the one scenario where zero-touch fails gracefully into a managed-intervention state: the system has exhausted its automated recovery options and is explicitly communicating that human attention is required.

For SQL Server, replica reintegration after an AG failover is observable through the DMV layer:

SQL Server — monitoring replica reintegration and data movement:

-- Tracks the progress of a replica rejoining the AG after primary failure
SELECT
    db_name(drs.database_id)            AS database_name,
    ar.replica_server_name,
    drs.synchronization_state_desc,
    drs.is_suspended,
    drs.suspend_reason_desc,
    drs.log_send_queue_size             AS pending_send_kb,
    drs.redo_queue_size                 AS pending_redo_kb,
    drs.log_send_rate                   AS send_rate_kb_per_sec,
    drs.redo_rate                       AS redo_rate_kb_per_sec,
    CASE
        WHEN drs.redo_rate > 0
        THEN (drs.redo_queue_size / drs.redo_rate)
        ELSE NULL
    END                                 AS estimated_seconds_to_sync
FROM sys.dm_hadr_database_replica_states drs
JOIN sys.availability_replicas ar
    ON drs.replica_id = ar.replica_id
WHERE drs.is_local = 0
ORDER BY drs.log_send_queue_size DESC;

The estimated_seconds_to_sync calculated column is what the autonomous system uses to decide whether to mark the rejoining replica as pending (expected to finish within SLA) or degraded (reintegration is taking longer than expected, possibly indicating storage or network issues on the rejoining node). This metric feeds into the cluster health dashboard and the replica promotion eligibility gate.
