Chapter 39: Right-Sizing Cloud Database Instances

Cloud billing has a way of focusing the mind. On-premises, a server purchased three years ago costs the same whether it runs at 8% CPU or 95%. In the cloud, every vCPU-hour, every gigabyte of provisioned IOPS, and every GB of storage is metered continuously. Right-sizing — selecting the correct instance class, storage configuration, and compute tier for each workload — is therefore not an optimization exercise you do once during migration. It is an ongoing engineering discipline that directly affects both application performance and monthly spend. This chapter covers the mechanics of right-sizing across AWS RDS and Aurora, Azure SQL Database, and Google Cloud SQL and AlloyDB: how to read the signals that indicate a mismatch between workload and instance, how to resize with minimal disruption, and how to build the automation that keeps sizing decisions current as workloads evolve.

---

39.1 Understanding the Cloud Instance Taxonomy

Before you can right-size anything, you need a working mental model of how cloud providers organize their compute offerings. Every managed database service presents a hierarchy: a general-purpose tier, a memory-optimized tier, and — on some platforms — a burstable tier that has no on-premises equivalent at all.

AWS RDS instance families follow the EC2 naming convention. db.t3.medium and db.t4g.medium are burstable instances that accumulate CPU credits when idle and burn them during bursts. db.m7g and db.m6i are the general-purpose workhorses. db.r7g, db.r6i, and db.x2idn are memory-optimized, relevant when your buffer pool needs to hold more than 32 GB in RAM. Aurora uses the same families but the underlying hardware is shared; Aurora Serverless v2 adds a fractional ACU model that sits orthogonally to the instance class hierarchy.

Azure SQL Database abstracts this differently. In the General Purpose service tier, you choose vCores (2 through 128) and the memory scales with them at a fixed ratio (approximately 5.1 GB per vCore in Gen5). The Business Critical tier uses local SSD and adds a readable secondary replica at no extra compute cost. Hyperscale decouples storage from compute entirely and adds independent read replicas called named replicas. The DTU model — which you will encounter on older provisioned databases — bundles compute, memory, and I/O into a single opaque unit; avoid it for new workloads because it gives you no visibility into which resource is actually the constraint.

Google Cloud SQL uses machine types derived from Compute Engine: shared-core for development (db-f1-micro, db-g1-small), standard (db-n1-standard-*), and high-memory (db-n1-highmem-*). AlloyDB uses a separate tier system anchored to vCPUs, with columnar memory and a distributed cache that changes the RAM sizing calculus significantly compared to vanilla Cloud SQL.

The mapping that helps on-premises DBAs most: think of memory-to-vCPU ratio, not raw core count. A db.r6i.2xlarge gives you 8 vCPUs and 64 GB RAM (8:1). A db.m6i.2xlarge gives you 8 vCPUs and 32 GB RAM (4:1). If your workload is read-heavy and buffer-pool-bound, the r6i is the right family even if it costs more per vCore. If your workload is CPU-bound — complex reporting queries, high connection rates, stored procedure execution — the m6i is more efficient.

Burstable instances deserve special attention. db.t3.medium (2 vCPU, 4 GB RAM) looks attractive for small workloads, and it often is appropriate for development and small OLTP databases. The risk is CPU credit exhaustion. When credits run out, performance drops to the CPU baseline, which for db.t3.medium is 20% of a single vCPU. On-premises you never had a server that suddenly ran at one-fifth its rated speed for sustained periods because the accounting department decided the bill was too high. In the cloud, this happens silently if you do not monitor credit balance.

-- Query to estimate buffer pool hit ratio on PostgreSQL (RDS or Cloud SQL)
-- A ratio below 0.98 on a read-heavy OLTP workload suggests the instance
-- needs more RAM, not necessarily more CPU.
SELECT
    sum(heap_blks_hit)  AS heap_hit,
    sum(heap_blks_read) AS heap_read,
    round(
        sum(heap_blks_hit)::numeric /
        nullif(sum(heap_blks_hit) + sum(heap_blks_read), 0),
        4
    ) AS hit_ratio
FROM pg_statio_user_tables;

On a well-sized instance running a warm OLTP workload, this ratio should be 0.985 or higher. Values below 0.95 indicate the working set is larger than the buffer pool, and the correct response is a memory upgrade before anything else.

39.2 Identifying Sizing Mismatches from Cloud Metrics

Right-sizing begins with observation, and cloud monitoring platforms expose far more granular data than most on-premises monitoring setups. The challenge is knowing which metrics to read and what thresholds actually indicate a problem versus normal variation.

On AWS, CloudWatch provides the primary signal set for RDS. The metrics that matter most for sizing decisions are:

  • → `CPUUtilization` — obvious, but interpret it in the context of the instance class. 70% CPU on a `db.r6i.8xlarge` is different from 70% on a `db.t3.medium`.
  • → `FreeableMemory` — the absolute free RAM plus cache/buffers that can be reclaimed. If this drops below 10–15% of total instance RAM consistently during business hours, you are working the buffer pool hard.
  • → `ReadIOPS` / `WriteIOPS` — compare against your provisioned IOPS. If you are consistently at 90%+ of provisioned IOPS, you either need more IOPS or a faster storage type.
  • → `DatabaseConnections` — watch for this approaching the `max_connections` limit. On RDS PostgreSQL, `max_connections` defaults to a formula based on RAM (`LEAST({DBInstanceClassMemory/9531392}, 5000)`), which gives you roughly 873 connections on a `db.r6i.xlarge` (32 GB). Once you approach 80% of that number with real connections, connection pressure becomes a latency multiplier.
  • → `CPUCreditBalance` (burstable only) — if this trends toward zero during business hours, you have chronic CPU credit exhaustion, and the correct fix is moving to a non-burstable instance.
    • # Retrieve RDS CPUUtilization average over the last 7 days using AWS CLI
# Useful for periodic right-sizing reviews in a weekly cron or CI pipeline

aws cloudwatch get-metric-statistics \
  --namespace AWS/RDS \
  --metric-name CPUUtilization \
  --dimensions Name=DBInstanceIdentifier,Value=prod-postgres-01 \
  --start-time $(date -u -d '7 days ago' +%Y-%m-%dT%H:%M:%SZ) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --period 3600 \
  --statistics Average Maximum \
  --output table

# Also pull FreeableMemory to pair with CPU
aws cloudwatch get-metric-statistics \
  --namespace AWS/RDS \
  --metric-name FreeableMemory \
  --dimensions Name=DBInstanceIdentifier,Value=prod-postgres-01 \
  --start-time $(date -u -d '7 days ago' +%Y-%m-%dT%H:%M:%SZ) \
  --end-time $(date -u +%Y-%m-%dT%H:%M:%SZ) \
  --period 3600 \
  --statistics Minimum \
  --output table

    On Azure SQL, the equivalent layer is Azure Monitor with Query Performance Insight built on top of Query Store. The metrics that drive sizing decisions are:

  • → `cpu_percent` — percentage of DTUs or vCores consumed. Sustained values above 80% warrant investigation.
  • → `physical_data_read_percent` — I/O consumption relative to the tier limit.
  • → `log_write_percent` — transaction log throughput as a percentage of tier limit; this is the metric most commonly missed by DBAs moving from on-premises, where log throughput limits were rarely a concern.
  • → `workers_percent` — percentage of the maximum worker thread count consumed. If this approaches 100%, connections will queue even if CPU is idle.

    • On Google Cloud SQL, Cloud Monitoring exposes metrics under the cloudsql.googleapis.com namespace. Particularly useful for right-sizing:

  • → `database/cpu/utilization` — normalized 0–1.
  • → `database/memory/utilization` — includes buffer cache in the denominator.
  • → `database/disk/read_ops_count` and `database/disk/write_ops_count` — raw I/O operations, not normalized to a tier limit.

    • For AlloyDB, the columnar memory pool (the HTAP layer) has its own memory allocation separate from PostgreSQL's shared_buffers. You need to monitor both database/memory/components to understand whether columnar memory pressure is evicting PostgreSQL buffer pool pages.

    AWS Compute Optimizer is worth enabling even if you are skeptical of automated recommendations. It ingests 14 days of CloudWatch metrics and produces a recommendation per instance that categorizes the current state as Under-provisioned, Over-provisioned, or Optimized, along with the projected cost and performance impact of moving to the recommended instance. These recommendations are not always correct — they cannot know about once-monthly batch jobs that do not appear in the 14-day window — but they are a useful starting point that on-premises DBAs would have had to build manually.

    39.3 Vertical Scaling: Resizing the Instance

    When you have identified a mismatch — too little CPU, too little RAM, or wrong storage — the corrective action on a managed database is usually a vertical resize. This is simpler than on-premises (no hardware procurement, no OS reinstall) but it is not zero-impact. Understanding the downtime windows and how to minimize them is the operational skill.

    RDS instance class changes require a brief outage. AWS performs the change by stopping the instance, modifying it, and restarting it. The typical duration is 3–10 minutes, though the official documentation declines to commit to specific numbers because workload state (dirty buffer pool pages to flush, active transactions to drain) affects the actual time. If you apply the change immediately during a maintenance window, that window is the impact boundary. If you apply it via modify-db-instance with --apply-immediately, it happens at the next low-activity moment that AWS determines — which in practice means during your next call to modify-db-instance for any reason, which is easy to forget.

    # Resize an RDS PostgreSQL instance to db.r6i.2xlarge
# --no-apply-immediately means the change waits for the next maintenance window
# Remove that flag (or use --apply-immediately) to resize at the next opportunity

aws rds modify-db-instance \
  --db-instance-identifier prod-postgres-01 \
  --db-instance-class db.r6i.2xlarge \
  --no-apply-immediately \
  --region us-east-1

# Verify the pending modification
aws rds describe-db-instances \
  --db-instance-identifier prod-postgres-01 \
  --query 'DBInstances[0].PendingModifiedValues' \
  --output json

    For Aurora, the vertical resize story is better. In Aurora Serverless v2, scaling is continuous and can happen without a restart: the instance scales from the minimum ACU to the maximum ACU as load increases, in increments as small as 0.5 ACUs. One Aurora Capacity Unit is approximately 2 GB of RAM and a proportional share of CPU. A minimum of 0.5 ACUs with a maximum of 128 ACUs covers the range from a tiny dev database to a 256 GB RAM instance. The catch: Aurora Serverless v2 does not scale to zero (unlike v1), and the minimum of 0.5 ACUs means you are paying for that perpetually, roughly $43/month at US East pricing as of 2024.

    For provisioned Aurora instances, the resize still requires a brief restart, but if you have Aurora read replicas, you can resize the replicas first, promote one to primary via a failover, resize the old primary (now a replica), and keep application downtime to the 15–30 seconds of the failover operation rather than the full instance restart time.

    Azure SQL vertical scaling is the smoothest across all three platforms because it is a service operation rather than a VM resize. Changing vCores in the General Purpose tier takes roughly 5–10 seconds for in-memory state transfer; the connection is briefly interrupted. Changing service tiers (e.g., General Purpose to Business Critical) takes longer — 10–30 minutes — because the underlying storage layer changes.

    # Scale an Azure SQL Database from GP_Gen5_4 to GP_Gen5_8 (vCores)
az sql db update \
  --resource-group prod-rg \
  --server prod-sqlserver \
  --name prod-database \
  --service-objective GP_Gen5_8 \
  --no-wait

# Monitor the scaling operation
az sql db show \
  --resource-group prod-rg \
  --server prod-sqlserver \
  --name prod-database \
  --query '{status: status, serviceLevelObjective: currentServiceObjectiveName}' \
  --output json

    Google Cloud SQL instance tier changes require the instance to restart. For standard instances (non-HA), this is a brief outage. For HA instances (primary + standby), Cloud SQL performs a failover to the standby, resizes the old primary, then fails back — similar to the Aurora replica promotion pattern but managed transparently. The total time is typically 2–5 minutes.

    Terraform management of instance sizing deserves attention because many production environments manage infrastructure as code, and a size change that happens outside Terraform will create drift. The pattern is to encode the instance class in a variable and use a terraform plan + terraform apply workflow for all resizes:

    # terraform/rds/variables.tf
variable "db_instance_class" {
  description = "RDS instance class — change this variable to trigger a resize"
  type        = string
  default     = "db.r6i.xlarge"
}

variable "apply_immediately" {
  description = "Apply instance changes immediately vs next maintenance window"
  type        = bool
  default     = false
}

# terraform/rds/main.tf
resource "aws_db_instance" "prod_postgres" {
  identifier              = "prod-postgres-01"
  engine                  = "postgres"
  engine_version          = "15.4"
  instance_class          = var.db_instance_class
  allocated_storage       = 500
  max_allocated_storage   = 2000
  storage_type            = "gp3"
  iops                    = 6000
  apply_immediately       = var.apply_immediately
  maintenance_window      = "Sun:03:00-Sun:04:00"
  backup_window           = "01:00-02:00"
  backup_retention_period = 7

  # Prevent accidental deletion in production
  deletion_protection = true

  tags = {
    Environment    = "production"
    CostCenter     = "engineering"
    LastResizeDate = "2024-11"
  }
}

    Tagging instances with the last resize date is a low-tech but effective practice. When you pull cost reports and see a large instance that has not been reviewed in 18 months, the tag tells you whether the sizing decision was intentional or inherited.

    39.4 Storage Right-Sizing: IOPS, Throughput, and Autoscaling

    Storage is frequently the second-largest cost item on a cloud database bill, and it is the area where DBAs most commonly over-provision because the on-premises intuition (provision for peak, never run out) collides with cloud storage pricing that charges for provisioned capacity and IOPS regardless of usage.

    AWS RDS storage types:

  • → `gp2` — general-purpose SSD. IOPS scale with volume size: 3 IOPS per GB, minimum 100, maximum 16,000. If you provision 500 GB of gp2 storage, you get exactly 1,500 IOPS. This coupling of size to IOPS was the dominant reason teams over-provisioned storage.
  • → `gp3` — the 2022 successor to gp2. IOPS and throughput are independently configurable. The baseline is 3,000 IOPS and 125 MB/s included in the storage price. You can provision up to 16,000 IOPS and 1,000 MB/s separately. For most workloads, gp3 is cheaper than gp2 and gives you more control. Migrate existing gp2 instances to gp3 unless you have a specific reason not to.
  • → `io1` / `io2` — provisioned IOPS SSD. Use these when you need more than 16,000 IOPS or when sub-millisecond storage latency consistency is a hard requirement (financial systems, high-frequency OLTP). io2 Block Express supports up to 256,000 IOPS per volume at multi-attach.

    • The specific gotcha: RDS storage can only be increased, never decreased. If you provision 2 TB and decide 500 GB was sufficient, you are paying for 2 TB until you rebuild the instance. This is fundamentally different from on-premises SANs where you could reclaim LUNs. Provision conservatively and rely on Storage Autoscaling:

    # Enable storage autoscaling on an existing RDS instance
# max-allocated-storage sets the ceiling; autoscaling will grow storage
# up to that limit in 10% increments when free space drops below 10%
# or below 5 GB (whichever is larger)

aws rds modify-db-instance \
  --db-instance-identifier prod-postgres-01 \
  --max-allocated-storage 2000 \
  --apply-immediately

# Storage autoscaling does NOT require an instance restart —
# it is one of the few RDS modifications that is truly online

    Azure SQL storage in the General Purpose tier is provisioned in 1 GB increments from 1 GB to 4 TB, and can be increased online. Unlike RDS, Azure SQL does allow storage reduction on the General Purpose tier (with some restrictions). The Business Critical tier uses local SSD attached to the compute node, so storage limits are tied to the instance size rather than being independently configurable — a BC_Gen5_4 instance gets 1 TB of local SSD.

    Google Cloud SQL storage always starts at HDD or SSD. Use SSD for any production workload; HDD pricing is lower but the latency difference is significant. Cloud SQL supports storage autoresize by default (enabled via --database-flags or the console), and it increases storage in 25 GB increments when free space drops below the configured threshold. The equivalent caution applies: Cloud SQL storage cannot be decreased after it has been increased.

    IOPS sizing math for RDS gp3: Start with your observed peak IOPS from the source system. Add 20% headroom. Compare that to the 3,000 IOPS baseline included in gp3. If your peak is below 3,000 IOPS, gp3 at baseline is sufficient. If your peak is 8,000 IOPS, provision 10,000 IOPS on gp3. At $0.065/provisioned IOPS-month (US East), 10,000 IOPS costs $650/month — substantially less than the io1 equivalent that also includes higher baseline storage pricing.

    # Python script to calculate gp3 IOPS cost vs io1 for a given workload
# Useful for building a right-sizing cost model during migration planning

import boto3
from datetime import datetime, timedelta

def estimate_storage_cost(
    storage_gb: int,
    peak_iops: int,
    peak_throughput_mbps: int,
    region: str = "us-east-1"
) -> dict:
    """
    Compare gp3 vs io1 costs for a given storage/IOPS requirement.
    Pricing as of late 2024 for us-east-1. Update pricing constants
    for other regions or pull from AWS Price List API for accuracy.
    """

    # gp3 pricing
    GP3_STORAGE_PER_GB   = 0.115   # $/GB-month
    GP3_IOPS_INCLUDED    = 3000
    GP3_IOPS_EXTRA       = 0.065   # $/provisioned IOPS-month above 3000
    GP3_THROUGHPUT_INCL  = 125     # MB/s included
    GP3_THROUGHPUT_EXTRA = 0.040   # $/MB/s-month above 125

    # io1 pricing
    IO1_STORAGE_PER_GB   = 0.125   # $/GB-month
    IO1_IOPS_PER_MONTH   = 0.065   # $/provisioned IOPS-month

    # gp3 cost
    gp3_storage_cost = storage_gb * GP3_STORAGE_PER_GB
    gp3_extra_iops   = max(0, peak_iops - GP3_IOPS_INCLUDED)
    gp3_iops_cost    = gp3_extra_iops * GP3_IOPS_EXTRA
    gp3_extra_tput   = max(0, peak_throughput_mbps - GP3_THROUGHPUT_INCL)
    gp3_tput_cost    = gp3_extra_tput * GP3_THROUGHPUT_EXTRA
    gp3_total        = gp3_storage_cost + gp3_iops_cost + gp3_tput_cost

    # io1 cost
    io1_storage_cost = storage_gb * IO1_STORAGE_PER_GB
    io1_iops_cost    = peak_iops * IO1_IOPS_PER_MONTH
    io1_total        = io1_storage_cost + io1_iops_cost

    return {
        "gp3": {
            "storage": round(gp3_storage_cost, 2),
            "iops":    round(gp3_iops_cost, 2),
            "throughput": round(gp3_tput_cost, 2),
            "total":   round(gp3_total, 2),
        },
        "io1": {
            "storage": round(io1_storage_cost, 2),
            "iops":    round(io1_iops_cost, 2),
            "throughput": 0.0,
            "total":   round(io1_total, 2),
        },
        "recommendation": "gp3" if gp3_total < io1_total else "io1",
        "monthly_savings_with_gp3": round(io1_total - gp3_total, 2),
    }


if __name__ == "__main__":
    result = estimate_storage_cost(
        storage_gb=1000,
        peak_iops=8000,
        peak_throughput_mbps=300,
    )
    print(f"gp3 monthly total: ${result['gp3']['total']}")
    print(f"io1 monthly total: ${result['io1']['total']}")
    print(f"Recommendation: {result['recommendation']}")
    print(f"Monthly savings with gp3: ${result['monthly_savings_with_gp3']}")

    39.5 Horizontal Scaling Patterns and Read Replica Sizing

    Vertical scaling has a ceiling. Every cloud platform has a largest available instance class, and even before you reach it, the cost curve steepens sharply. When a single large instance cannot absorb the workload, horizontal scaling through read replicas is the standard answer for read-heavy workloads.

    The key principle that on-premises DBAs sometimes miss: read replicas in cloud managed databases are not just disaster recovery standbys. They are production traffic receivers.
