Chapter 16: Weaviate: Schema Design and Hybrid Search

Weaviate occupies a distinct position in the vector database landscape: it ships with built-in vectorization modules, a graph-inspired object model, and first-class hybrid search that combines dense vector retrieval with BM25 keyword scoring — all within a single system that requires no external embedding pipeline at insert time. For database engineers, the learning curve is less about grasping vector concepts and more about accepting a schema model that sits somewhere between a document store and a relational system, with a query language (GraphQL) that rewards understanding before it rewards productivity. This chapter walks through Weaviate's schema design philosophy, its class-and-property model, practical hybrid search configuration, multi-tenancy, and the operational patterns that make Weaviate viable at production scale.

---

16.1 Weaviate's Data Model: Classes, Properties, and the Object Graph

Weaviate organizes data into classes — the rough equivalent of tables in a relational database, or collections in MongoDB. Each class holds objects that carry scalar properties plus an optional vector. Unlike most vector databases where you bring your own vector, Weaviate can generate vectors automatically using a configured vectorizer module (OpenAI, Cohere, sentence-transformers via its text2vec-transformers module, and others). The vector becomes a first-class citizen of the object, not an add-on.

The class definition is the schema contract. Every class declaration specifies:

  • → The vectorizer module (or `none` if you supply vectors yourself)
  • → Per-property data types (`text`, `int`, `number`, `boolean`, `date`, `uuid`, cross-references)
  • → Indexing behavior per property (BM25 inverted index, filterable, sortable)
  • → Vector index configuration (HNSW parameters, quantization)
  • → Module-specific settings (which model, tokenization strategy)

    • The cross-reference system is Weaviate's most database-distinct feature. A property can reference objects in another class, creating a directed relationship that can be traversed in GraphQL queries. Think of it as a foreign key with graph traversal semantics — closer to a JOIN than a document embed, but resolved lazily at query time.

    Tokenization matters more than it looks. For BM25 indexing, each text property has a tokenization setting: word, lowercase, whitespace, or field. The choice affects how terms are indexed and matched. Product codes, URLs, and structured identifiers should use field (exact match) or whitespace. Natural language content should use word (default). Getting this wrong produces silent retrieval failures — your keyword search simply misses matches because the indexed tokens don't align with the query tokens.

    Here is a realistic schema definition for a document retrieval system using the Weaviate Python client v4:

    # Python
import weaviate
import weaviate.classes as wvc
from weaviate.classes.config import (
    Configure,
    Property,
    DataType,
    Tokenization,
    VectorDistances,
)

client = weaviate.connect_to_local()  # or connect_to_weaviate_cloud(...)

# Define a DocumentChunk class with OpenAI vectorizer
client.collections.create(
    name="DocumentChunk",
    description="Chunked document passages for RAG retrieval",
    vectorizer_config=Configure.Vectorizer.text2vec_openai(
        model="text-embedding-3-small",
        dimensions=1536,
    ),
    vector_index_config=Configure.VectorIndex.hnsw(
        distance_metric=VectorDistances.COSINE,
        ef_construction=128,
        max_connections=16,
        ef=64,                    # query-time ef, higher = more accurate, slower
        dynamic_ef_factor=8,      # auto-scales ef based on result limit
        dynamic_ef_min=100,
        dynamic_ef_max=500,
    ),
    properties=[
        Property(
            name="content",
            data_type=DataType.TEXT,
            description="The chunk text content",
            tokenization=Tokenization.WORD,          # BM25 indexed on words
            vectorize_property_name=False,           # don't include prop name in vector
            skip_vectorization=False,                # this prop drives the vector
        ),
        Property(
            name="source_document",
            data_type=DataType.TEXT,
            tokenization=Tokenization.FIELD,         # exact match on document path/ID
            skip_vectorization=True,
        ),
        Property(
            name="chunk_index",
            data_type=DataType.INT,
            skip_vectorization=True,
        ),
        Property(
            name="created_at",
            data_type=DataType.DATE,
            skip_vectorization=True,
        ),
        Property(
            name="metadata_tags",
            data_type=DataType.TEXT_ARRAY,
            tokenization=Tokenization.WORD,
            skip_vectorization=True,
        ),
    ],
    # Enable inverted index for BM25 hybrid search
    inverted_index_config=Configure.inverted_index(
        bm25_b=0.75,
        bm25_k1=1.2,
        index_null_values=False,
        index_property_length=False,
    ),
)

print("DocumentChunk class created successfully")
client.close()

    The BM25 parameters k1 and b are tunable. k1=1.2 (term frequency saturation) and b=0.75 (document length normalization) are the classic defaults from Elasticsearch and Lucene — reasonable starting points for general English text. For short chunks (128–256 tokens), reduce b toward 0.5 because length normalization matters less when documents are uniformly short.

    16.2 Schema Design Patterns for Production RAG Systems

    Schema design mistakes in Weaviate are more expensive to fix than in a traditional RDBMS because the vector index is built during ingestion. Rebuilding it from scratch on a dataset of tens of millions of objects is an hours-long operation. The following patterns reflect hard-won production decisions.

    Pattern 1: Separate classes for separate embedding models. Do not mix objects vectorized with text-embedding-3-small and objects vectorized with text-embedding-3-large in the same class. The vector space is different even though both are OpenAI models. Weaviate v1.24+ supports named vectors (multiple vectors per object), which solves this cleanly for cross-modal retrieval, but for standard single-model pipelines, one model per class is the rule.

    Pattern 2: Push filtering metadata into scalar properties, not cross-references. Cross-references are powerful but expensive to filter on at scale. If you need to filter chunks by tenant_id, document_type, or language, store those as scalar properties with index_filterable=True. Filterable scalar indexes in Weaviate use a roaring bitmap structure — sub-millisecond for high-cardinality filters. Cross-reference traversal requires a separate resolution step and doesn't compose well with pre-filtering.

    Pattern 3: Use skip_vectorization=True aggressively. By default, Weaviate concatenates all non-skipped text properties before sending them to the vectorizer. A source_document path like s3://my-bucket/reports/Q4-2023-financials.pdf pollutes the semantic vector. Mark every property that isn't semantic content as skip_vectorization=True.

    Pattern 4: Named vectors for multi-representation objects. Weaviate v1.24 introduced named vectors, enabling one object to carry multiple vector representations. This matters for hybrid retrieval strategies where you want to search by question embedding AND summary embedding simultaneously:

    # Python
import weaviate
import weaviate.classes as wvc
from weaviate.classes.config import Configure, Property, DataType, Tokenization

client = weaviate.connect_to_local()

# Multi-vector schema: one object, two named vector spaces
client.collections.create(
    name="ResearchPaper",
    properties=[
        Property(name="title",    data_type=DataType.TEXT, tokenization=Tokenization.WORD),
        Property(name="abstract", data_type=DataType.TEXT, tokenization=Tokenization.WORD),
        Property(name="full_text",data_type=DataType.TEXT, tokenization=Tokenization.WORD),
        Property(name="authors",  data_type=DataType.TEXT_ARRAY, skip_vectorization=True),
        Property(name="year",     data_type=DataType.INT,  skip_vectorization=True),
        Property(name="doi",      data_type=DataType.TEXT,
                 tokenization=Tokenization.FIELD, skip_vectorization=True),
    ],
    vectorizer_config=[
        # Named vector 1: abstract-level semantic search
        Configure.NamedVectors.text2vec_openai(
            name="abstract_vector",
            source_properties=["title", "abstract"],
            model="text-embedding-3-small",
        ),
        # Named vector 2: full document retrieval
        Configure.NamedVectors.text2vec_openai(
            name="fulltext_vector",
            source_properties=["full_text"],
            model="text-embedding-3-large",
        ),
    ],
)

client.close()

    At query time, you specify which named vector to search against. The overhead is roughly 2× storage and index memory, but the retrieval flexibility is worth it for heterogeneous content where title/abstract search and full-text search have materially different precision characteristics.

    Property index flags reference:

    | Flag | Type | Purpose | Default |

    |------|------|---------|---------|

    | index_filterable | bool | Roaring bitmap for pre-filtering | true |

    | index_searchable | bool | BM25 inverted index | true for text |

    | index_range_filters | bool | Numeric range index | false |

    | skip_vectorization | bool | Exclude from embedding input | false |

    Turn off index_searchable for properties you never BM25-search. It saves disk and speeds up writes. For an INT property like chunk_index you'll never keyword-search, both index_searchable and skip_vectorization should be true and true respectively.

    16.3 Hybrid Search: BM25 + Vector Fusion

    Hybrid search is the single most operationally significant feature Weaviate offers over simpler vector stores. Pure vector search fails on proper nouns, product codes, rare terminology, and out-of-vocabulary tokens that the embedding model hasn't seen at training time. Pure BM25 fails on semantic paraphrase and synonym matching. Hybrid search fuses both signals.

    Weaviate implements hybrid search via Reciprocal Rank Fusion (RRF) or a relative score fusion strategy. Both take the ranked lists from BM25 and vector ANN independently, then merge them. The alpha parameter controls the blend: alpha=0.0 is pure BM25, alpha=1.0 is pure vector, alpha=0.5 is an equal blend. The default RRF formula is:

    RRF_score(d) = sum_over_rankings( 1 / (k + rank(d)) )

    where k=60 is a smoothing constant. This is the same formula used in Elasticsearch's hybrid search and in academic IR literature. It is robust to outliers and scale differences between BM25 and cosine similarity scores.

    # Python
import weaviate
import weaviate.classes as wvc
from weaviate.classes.query import HybridFusion, MetadataQuery, Filter

client = weaviate.connect_to_local()

collection = client.collections.get("DocumentChunk")

# Basic hybrid search
response = collection.query.hybrid(
    query="transformer architecture self-attention mechanism",
    alpha=0.7,           # 70% weight toward vector, 30% BM25
    limit=10,
    return_metadata=MetadataQuery(score=True, explain_score=True),
)

for obj in response.objects:
    print(f"Score: {obj.metadata.score:.4f}")
    print(f"Explain: {obj.metadata.explain_score}")
    print(f"Content: {obj.properties['content'][:120]}")
    print("---")

# Hybrid search with pre-filtering
response_filtered = collection.query.hybrid(
    query="quarterly revenue forecast",
    alpha=0.5,
    limit=10,
    filters=Filter.by_property("source_document").contains_any(
        ["finance/Q4-2023.pdf", "finance/Q3-2023.pdf"]
    ),
    return_metadata=MetadataQuery(score=True),
)

# Hybrid search with named vectors (multi-vector schema)
research_collection = client.collections.get("ResearchPaper")

response_named = research_collection.query.hybrid(
    query="attention is all you need transformer",
    alpha=0.6,
    target_vector="abstract_vector",   # search in abstract embedding space
    limit=5,
    return_metadata=MetadataQuery(score=True, explain_score=True),
)

client.close()

    Alpha tuning is empirical, not theoretical. Run your benchmark query set against your actual data, vary alpha from 0.3 to 0.9 in 0.1 increments, and measure recall@10 or NDCG@10 against human-labeled relevance judgments. Published benchmarks across IR datasets suggest alpha between 0.6 and 0.75 is optimal for most English-language knowledge retrieval tasks. For technical queries with precise terminology (SQL error codes, API method names, CVE identifiers), shift toward 0.3–0.4 to give BM25 more weight. For conversational or natural language questions, 0.7–0.8 is typical.

    Score fusion strategy choice:

  • → `HybridFusion.RANKED` (RRF): rank-based, insensitive to raw score magnitudes. Recommended when BM25 and vector scores have very different distributions (they almost always do).
  • → `HybridFusion.RELATIVE_SCORE`: normalizes both score lists to [0,1] then linearly combines. More sensitive to score distribution; can produce better results when both signals are calibrated.

    • RRF is the safer default for production deployments where score distribution is unknown or shifts with data updates.

    16.4 Multi-Tenancy: One Schema, Many Isolated Datasets

    Multi-tenancy in Weaviate is a first-class schema feature, not an application-layer workaround. This matters for SaaS platforms and enterprise deployments where you need per-customer data isolation without operating separate Weaviate instances.

    Weaviate's multi-tenancy model assigns each tenant a dedicated shard. The implications are significant:

  • → **Vector index isolation**: each tenant's HNSW graph is independent. A query for Tenant A never scans Tenant B's index.
  • → **Lifecycle control**: tenants can be activated (hot), deactivated (warm, on disk), or offloaded (cold, off disk) independently. This is the key mechanism for cost-optimized multi-tenant deployments where not all customers are active simultaneously.
  • → **No cross-tenant contamination**: a malformed insert or a slow query on one tenant cannot degrade another tenant's shard.

    • The tradeoff is memory. Each active tenant's HNSW graph must be loaded into memory. At 10,000 objects per tenant with ef_construction=128 and 1536-dimension float32 vectors, expect roughly 70–90 MB per tenant in memory. At 1,000 active tenants that's 70–90 GB — plan your node sizing accordingly, or use tenant activation/deactivation to manage the active set.

    # Python
import weaviate
import weaviate.classes as wvc
from weaviate.classes.config import Configure, Property, DataType, Tokenization
from weaviate.classes.tenants import Tenant, TenantActivityStatus

client = weaviate.connect_to_local()

# Create a multi-tenant collection
client.collections.create(
    name="CustomerDocuments",
    multi_tenancy_config=Configure.multi_tenancy(
        enabled=True,
        auto_tenant_creation=False,   # explicit tenant management
    ),
    vectorizer_config=Configure.Vectorizer.text2vec_openai(
        model="text-embedding-3-small",
    ),
    properties=[
        Property(name="content",      data_type=DataType.TEXT, tokenization=Tokenization.WORD),
        Property(name="doc_type",     data_type=DataType.TEXT, tokenization=Tokenization.FIELD),
        Property(name="uploaded_by",  data_type=DataType.TEXT, skip_vectorization=True),
    ],
)

collection = client.collections.get("CustomerDocuments")

# Create tenants (one per customer)
collection.tenants.create([
    Tenant(name="customer_acme",    activity_status=TenantActivityStatus.ACTIVE),
    Tenant(name="customer_globex",  activity_status=TenantActivityStatus.ACTIVE),
    Tenant(name="customer_initech", activity_status=TenantActivityStatus.INACTIVE),  # warm
])

# Insert objects into a specific tenant
tenant_collection = collection.with_tenant("customer_acme")

with tenant_collection.batch.dynamic() as batch:
    batch.add_object(properties={
        "content":     "Q4 revenue exceeded targets by 12% driven by APAC expansion.",
        "doc_type":    "financial_report",
        "uploaded_by": "finance_team",
    })
    batch.add_object(properties={
        "content":     "New product roadmap approved by executive committee.",
        "doc_type":    "strategy_doc",
        "uploaded_by": "product_team",
    })

# Query within a tenant
response = tenant_collection.query.hybrid(
    query="revenue targets quarterly results",
    alpha=0.6,
    limit=5,
)

for obj in response.objects:
    print(obj.properties["content"])

# Deactivate an inactive tenant to free memory (tenant data stays on disk)
collection.tenants.update([
    Tenant(
        name="customer_globex",
        activity_status=TenantActivityStatus.INACTIVE,
    )
])

# Reactivate when needed
collection.tenants.update([
    Tenant(
        name="customer_globex",
        activity_status=TenantActivityStatus.ACTIVE,
    )
])

client.close()

    Tenant activation patterns in practice:

    | Scenario | Strategy |

    |----------|---------|

    | Enterprise SaaS, most tenants active | All ACTIVE, scale horizontally |

    | Long-tail SaaS, 20% tenants use 80% of queries | Active set management via usage metrics |

    | Archive/compliance, tenants rarely queried | OFFLOADED by default, activate on-demand |

    | Testing/staging environments | INACTIVE until test run starts |

    Auto-tenant creation (auto_tenant_creation=True) creates a tenant shard the first time an object is inserted for an unknown tenant key. It's convenient for development but dangerous in production — a client bug with an unexpected tenant ID can create thousands of orphaned shards.

    16.5 Batch Ingestion, HNSW Tuning, and Query Performance

    Weaviate's HNSW implementation is configurable at class definition time. The parameters are identical in semantics to those in FAISS and hnswlib because Weaviate's implementation derives from the same academic specification, but the defaults are conservative — intentionally so, because Weaviate serves latency-sensitive HTTP workloads, not offline batch retrieval.

    The three HNSW parameters you will tune:

  • → `ef_construction` (default: 128): graph quality during build. Higher = better recall, slower ingestion. Range 64–256 covers most use cases. Don't go below 64 — recall degrades sharply. Don't exceed 256 unless you have a recall requirement above 99.5%.
  • → `max_connections` (M, default: 64): neighbors per node. Higher = better recall on large datasets, higher memory. For datasets under 10M objects, M=16 is enough. For 100M+ objects, M=32–64.
  • → `ef` (default: 64 or dynamic): query-time beam width. The primary recall vs. latency knob. Set via `dynamic_ef_factor` to auto-scale based on `limit`.

    • Memory estimation for HNSW:

    Memory (bytes) ≈ num_objects × (dimensions × 4 + max_connections × 8 × 2)

    For 1M objects, 1536 dimensions, M=16:

    1,000,000 × (1536 × 4 + 16 × 8 × 2) = 1,000,000 × (6144 + 256) = ~6.4 GB

    This is the graph memory floor. Add scalar property storage and inverted index overhead on top.

    Batch ingestion with error handling:

    # Python
import weaviate
import weaviate.classes as wvc
from weaviate.classes.config import ConsistencyLevel
import json
from pathlib import Path
from tqdm import tqdm

client = weaviate.connect_to_local()
collection = client.collections.get("DocumentChunk")

def load_chunks(filepath: str) -> list[dict]:
    """Load pre-chunked documents from JSONL."""
    with open(filepath) as f:
        return [json.loads(line) for line in f]

chunks = load_chunks("chunks.jsonl")

# Dynamic batching: Weaviate auto-tunes batch size based on server capacity
failed_objects = []

with collection.batch.dynamic() as batch:
    for chunk in tqdm(chunks, desc="Ingesting chunks"):
        batch.add_object(
            properties={
                "content":         chunk["text"],
                "source_document": chunk["source"],
                "chunk_index":     chunk["index"],
                "metadata_tags":   chunk.get("tags", []),
            },
            # Optionally supply a deterministic UUID for idempotency
            uuid=wvc.util.generate_uuid5(
                f"{chunk['source']}_{chunk['index']}"
            ),
        )

# Check for failed objects after batch completes
if batch.number_errors > 0:
    print(f"Batch completed with {batch.number_errors} errors")
    for failed in collection.batch.failed_objects:
        print(f"Failed: {failed.message} | Object: {failed.object_.properties}")

print(f"Total objects in collection: {collection.aggregate.over_all().total_count}")
client.close()

    Key batch performance observations from production deployments:

  • → Dynamic batching (`batch.dynamic()`) outperforms fixed-size batching in most cases because Weaviate's server-side admission control adapts to memory pressure during HNSW graph construction.
  • → With external vectorization (OpenAI, Cohere), the embedding API call is the bottleneck — not Weaviate's insert throughput. Use async batching or parallel chunk pipelines with a semaphore to control concurrent embedding calls.
  • → With built-in vectorizers (`text2vec-transformers` running locally), GPU availability dominates throughput. A single T4 GPU handles roughly 2,000–4,000 tokens per second for `sentence-transformers/all-MiniLM-L6-v2`.
  • → Set `WEAVIATE_GRPC_PORT` and use gRPC transport for batch inserts in v1.23+. gRPC batch ingestion is 40–60% faster than the REST API for large payloads because it avoids JSON serialization overhead.

    • Query latency benchmarks (approximate, single-node Weaviate, 1M objects):
