Chapter 17: Azure SQL Security: TDE, Always Encrypted, Ledger

Azure SQL Database ships with encryption features that would have required significant infrastructure investment in an on-premises SQL Server environment. Transparent Data Encryption was always available on-prem, but managing certificates, key rotation, and backup encryption was a full operational concern. Always Encrypted required application-side key management that most teams deferred indefinitely. Azure SQL Ledger didn't exist at all. In the cloud, these three technologies form a layered security posture that satisfies everything from SOC 2 to PCI DSS to HIPAA — but only if you understand where each layer applies, what it actually protects, and what its operational costs are. This chapter walks through each feature from first principles to production deployment, with the gotchas that the official documentation skips.

---

17.1 Transparent Data Encryption: What It Does (and Doesn't) Protect

If you've managed SQL Server on-prem, you know TDE: it encrypts the data files, log files, and backup files at rest. Azure SQL Database has TDE enabled by default on every new database. That sentence creates a false sense of security that has burned more than a few DBAs during audit preparation.

What TDE protects: Data files on disk. An attacker who steals raw storage media or data files gets ciphertext. Backups are encrypted automatically. This satisfies the "data at rest" requirement in most compliance frameworks.

What TDE does not protect: Data in memory, data in transit, data from a compromised SQL login, or a DBA with elevated permissions. TDE encryption and decryption happens transparently at the storage layer. Once the database engine decrypts data for query execution, TDE offers no protection. A rogue DBA running SELECT * FROM dbo.CreditCards gets plaintext every time.

The default TDE configuration uses a service-managed key: Microsoft generates and manages the Database Encryption Key (DEK) and the certificate that protects it. For most applications this is acceptable. For regulated industries, you need Customer-Managed Keys (CMK) via Azure Key Vault — where you control key lifecycle, rotation, and revocation.

#### Enabling TDE with Customer-Managed Key via Azure CLI

# Step 1: Create or identify the Azure Key Vault
az keyvault create \
  --name "prod-sqlsecurity-kv" \
  --resource-group "rg-prod-eastus" \
  --location "eastus" \
  --sku premium \
  --enable-purge-protection true \
  --retention-days 90

# Step 2: Create the key used as the TDE protector
az keyvault key create \
  --vault-name "prod-sqlsecurity-kv" \
  --name "sql-tde-protector" \
  --kts RSA-HSM \
  --size 2048

# Step 3: Assign the SQL Server managed identity access to the key vault
# First get the SQL Server's managed identity principal ID
SQL_IDENTITY=$(az sql server show \
  --name "prod-sql-server" \
  --resource-group "rg-prod-eastus" \
  --query identity.principalId \
  --output tsv)

az keyvault set-policy \
  --name "prod-sqlsecurity-kv" \
  --object-id "$SQL_IDENTITY" \
  --key-permissions get wrapKey unwrapKey list

# Step 4: Set the TDE protector to the customer-managed key
KEY_ID=$(az keyvault key show \
  --vault-name "prod-sqlsecurity-kv" \
  --name "sql-tde-protector" \
  --query key.kid \
  --output tsv)

az sql server tde-key set \
  --resource-group "rg-prod-eastus" \
  --server "prod-sql-server" \
  --server-key-type AzureKeyVault \
  --kid "$KEY_ID"

# Step 5: Verify TDE is active on the database
az sql db tde show \
  --resource-group "rg-prod-eastus" \
  --server "prod-sql-server" \
  --database "prod-app-db"

The output of that last command will show "state": "Enabled" when TDE is active. During the initial encryption scan — which runs as a background process — you'll see "state": "Enabling". On a 2TB database, that scan can take 6–12 hours. Queries continue running during this window, but you'll see elevated I/O and some query latency increase, typically 5–15% on write-heavy workloads.

#### Key Rotation

With CMK-based TDE, key rotation is a two-phase process: add the new key version, then set it as the active protector. The old key stays in Key Vault until you explicitly confirm rotation is complete — this is intentional, because any backup encrypted under the old key still needs the old key for restore.

# Rotate the TDE protector key
az keyvault key rotate \
  --vault-name "prod-sqlsecurity-kv" \
  --name "sql-tde-protector"

# Get the new key version ID
NEW_KEY_ID=$(az keyvault key show \
  --vault-name "prod-sqlsecurity-kv" \
  --name "sql-tde-protector" \
  --query key.kid \
  --output tsv)

# Update the server TDE protector to use the new version
az sql server tde-key set \
  --resource-group "rg-prod-eastus" \
  --server "prod-sql-server" \
  --server-key-type AzureKeyVault \
  --kid "$NEW_KEY_ID"

Production gotcha: If the SQL Server managed identity loses access to Key Vault — an expired policy, a deleted vault, a misconfigured network rule — the server goes into an inaccessible state within minutes. You cannot query the database, cannot restore from backup, cannot do anything until Key Vault access is restored. Set up Azure Monitor alerts on Key Vault access failures for the SQL Server managed identity before you go live with CMK.

17.2 Always Encrypted: Column-Level Encryption the Right Way

Always Encrypted addresses the threat model that TDE ignores: the privileged insider or compromised DBA account. With Always Encrypted, sensitive column data is encrypted client-side before it ever reaches the SQL Server engine. The database engine sees ciphertext. It stores ciphertext. It returns ciphertext. The plaintext never exists on the server.

Think of it this way: TDE protects your data from someone who steals the storage hardware. Always Encrypted protects your data from someone who steals the database credentials or has DBA access. These are different attackers in different positions.

The architecture has three components:

  • → **Column Encryption Key (CEK):** Encrypts the actual column data. One CEK per column group.
  • → **Column Master Key (CMK):** Encrypts the CEKs. Stored in Azure Key Vault (or Windows Certificate Store for dev). The database stores only metadata pointing to the CMK location — never the key itself.
  • → **Client application:** Holds the Azure Key Vault credentials. The ADO.NET or JDBC driver handles encryption/decryption transparently when the connection string includes `Column Encryption Setting=enabled`.

    • #### Setting Up Always Encrypted with Azure Key Vault

    -- Step 1: Create the Column Master Key metadata in SQL Server
-- The actual key material lives in Azure Key Vault, not here
-- This T-SQL runs in SSMS or sqlcmd after configuring AKV access

CREATE COLUMN MASTER KEY [AKV_CMK_Production]
WITH (
    KEY_STORE_PROVIDER_NAME = 'AZURE_KEY_VAULT',
    KEY_PATH = 'https://prod-sqlsecurity-kv.vault.azure.net/keys/always-encrypted-cmk/abc123def456'
);
GO

-- Step 2: Create the Column Encryption Key
-- This operation contacts Key Vault to wrap the CEK with the CMK
-- Requires the executing account to have Key Vault Wrap permissions

CREATE COLUMN ENCRYPTION KEY [CEK_PII_Data]
WITH VALUES (
    COLUMN_MASTER_KEY = [AKV_CMK_Production],
    ALGORITHM = 'RSA_OAEP',
    ENCRYPTED_VALUE = 0x016E000001630075007200720065006E...  -- wrapped CEK bytes
);
GO

-- Step 3: Create a table with encrypted columns
-- DETERMINISTIC encryption: same plaintext always produces same ciphertext
-- Supports equality comparisons (WHERE SSN = @ssn)
-- RANDOMIZED encryption: same plaintext produces different ciphertext each time
-- More secure but supports NO predicates — SELECT only by returning the column

CREATE TABLE dbo.CustomerPII (
    CustomerID      INT             NOT NULL PRIMARY KEY,
    FirstName       NVARCHAR(100)   NOT NULL,  -- plaintext, used in UI
    LastName        NVARCHAR(100)   NOT NULL,  -- plaintext
    SSN             CHAR(11)        COLLATE Latin1_General_BIN2
                                    ENCRYPTED WITH (
                                        COLUMN_ENCRYPTION_KEY = CEK_PII_Data,
                                        ENCRYPTION_TYPE = DETERMINISTIC,
                                        ALGORITHM = 'AEAD_AES_256_CBC_HMAC_SHA_256'
                                    ) NOT NULL,
    CreditCardNumber NVARCHAR(19)   COLLATE Latin1_General_BIN2
                                    ENCRYPTED WITH (
                                        COLUMN_ENCRYPTION_KEY = CEK_PII_Data,
                                        ENCRYPTION_TYPE = RANDOMIZED,
                                        ALGORITHM = 'AEAD_AES_256_CBC_HMAC_SHA_256'
                                    ) NULL,
    DateOfBirth     DATE            ENCRYPTED WITH (
                                        COLUMN_ENCRYPTION_KEY = CEK_PII_Data,
                                        ENCRYPTION_TYPE = RANDOMIZED,
                                        ALGORITHM = 'AEAD_AES_256_CBC_HMAC_SHA_256'
                                    ) NULL
);
GO

    Notice the COLLATE Latin1_General_BIN2 requirement on the NVARCHAR columns. Always Encrypted requires BIN2 collation for deterministic encryption on string columns. If you're retrofitting Always Encrypted onto an existing table that uses Latin1_General_CI_AS, you'll need to rebuild those columns. That's a blocking operation on large tables.

    #### Application Integration

    # Python application connecting to Always Encrypted columns
# Requires: pip install pyodbc, azure-identity, azure-keyvault-keys

import pyodbc
from azure.identity import DefaultAzureCredential
from azure.keyvault.keys.crypto import CryptographyClient

# The connection string does the heavy lifting
# Column Encryption Setting=enabled tells the driver to handle crypto
connection_string = (
    "Driver={ODBC Driver 18 for SQL Server};"
    "Server=prod-sql-server.database.windows.net;"
    "Database=prod-app-db;"
    "Authentication=ActiveDirectoryManagedIdentity;"
    "Column Encryption Setting=enabled;"
    "KeyVaultClientId=<managed-identity-client-id>;"
    "KeyVaultClientSecret=notneeded;"  # Not needed with managed identity
    "Encrypt=yes;"
    "TrustServerCertificate=no;"
)

conn = pyodbc.connect(connection_string)
cursor = conn.cursor()

# INSERT: application sends ciphertext, SQL Server stores ciphertext
# The ODBC driver encrypts the SSN value before it leaves the client machine
cursor.execute(
    "INSERT INTO dbo.CustomerPII (CustomerID, FirstName, LastName, SSN) "
    "VALUES (?, ?, ?, ?)",
    (1001, 'Jane', 'Smith', '123-45-6789')  # SSN encrypted by driver
)

# SELECT with parameterized query on deterministic column: works
# The driver encrypts the parameter value, SQL Server compares ciphertext
cursor.execute(
    "SELECT CustomerID, FirstName, LastName, SSN FROM dbo.CustomerPII "
    "WHERE SSN = ?",
    ('123-45-6789',)
)

row = cursor.fetchone()
if row:
    # SSN comes back as plaintext — decrypted by the driver on the client
    print(f"Found customer: {row.FirstName} {row.LastName}, SSN: {row.SSN}")

conn.commit()
conn.close()

    #### The Operational Constraints You Must Know Before Deploying

    Always Encrypted has constraints that are not obvious from the documentation until you run into them in production:

    You cannot do arithmetic on encrypted columns. WHERE Salary > 50000 on an encrypted Salary column fails. RANDOMIZED encrypted columns support no predicates whatsoever. DETERMINISTIC columns support only equality predicates (=, IN).

    Joins on encrypted columns work only for DETERMINISTIC encryption. JOIN ON t1.SSN = t2.SSN functions correctly with DETERMINISTIC. It fails with RANDOMIZED because the same plaintext produces different ciphertexts.

    Stored procedures that accept encrypted column values must use parameterized execution. The stored procedure definition looks normal in T-SQL; the application must pass values as parameters, not as inline SQL literals.

    GROUP BY, ORDER BY, and index ranges on encrypted columns: GROUP BY works with DETERMINISTIC. ORDER BY on encrypted data returns results ordered by ciphertext, which is meaningless for the application. There's no way around this. If you need sorted results on an encrypted column, sort client-side after decryption.

    Column size expansion: Every encrypted value is larger than the plaintext. A CHAR(11) SSN value becomes a 65-byte binary blob. Size your columns and row sizes accordingly. Encryption adds approximately 54 bytes of overhead plus the encrypted value size, rounded up to the nearest AES block.

    17.3 Always Encrypted with Secure Enclaves

    The encryption predicate limitations in classic Always Encrypted (described above) are severe enough that many teams avoid the feature entirely. Azure SQL's Always Encrypted with Secure Enclaves removes most of them.

    An enclave is a protected memory region in the SQL Server process — implemented via VBS (Virtualization Based Security) enclaves in Azure SQL Database. Sensitive cryptographic operations happen inside the enclave, where even the database engine process cannot inspect the memory. The application sends a CEK into the enclave via an attestation-verified channel. Query processing on encrypted columns happens inside the enclave, so the engine never exposes plaintext to the wider process.

    With enclave-enabled Always Encrypted, you gain:

  • → Range queries on encrypted columns (`WHERE Age BETWEEN 30 AND 45`)
  • → Pattern matching (`LIKE '%smith%'` on RANDOMIZED encrypted columns with enclave-enabled CEKs)
  • → Index creation on encrypted columns
  • → In-place column encryption without data movement

    • The tradeoff is complexity: the application connection must include attestation configuration pointing to a Microsoft Azure Attestation endpoint, and the Key Vault CMK must be enclave-enabled.

    # Enable enclave support on the Azure SQL logical server
az sql server update \
  --name "prod-sql-server" \
  --resource-group "rg-prod-eastus" \
  --set properties.enabledSecureEnclaveType="VBS"

# Verify the enclave type
az sql server show \
  --name "prod-sql-server" \
  --resource-group "rg-prod-eastus" \
  --query properties.enabledSecureEnclaveType

    The attestation URL for VBS enclaves in Azure SQL follows the pattern https://shareduks.uks.attest.azure.net (region-specific). The connection string needs:

    Enclave Attestation Url=https://shareduks.uks.attest.azure.net/attest/AzureIntelSGX;
Column Encryption Setting=enabled;

    Enclave-enabled queries carry a performance cost: the attestation handshake adds ~5–10ms per new connection. For long-lived connection pools this is negligible. For high-frequency short-lived connections it's measurable. Test with your actual connection pool configuration before production rollout.

    17.4 Azure SQL Ledger: Tamper-Evident Tables for Compliance

    Azure SQL Ledger is the newest of the three technologies in this chapter, and it solves a different problem than encryption. Ledger doesn't hide your data — it makes your data tamper-evident. Any modification to ledger table data leaves a cryptographically verifiable trail that can prove, to an external auditor, that the data has not been altered since it was written.

    The use cases are specific: financial transaction records, audit logs, configuration history, contract terms, any dataset where "we can prove this hasn't been modified" is a legal or regulatory requirement.

    Ledger works by maintaining a SHA-256 hash tree (similar to a blockchain Merkle tree) over table rows. Each database transaction produces a block containing a hash of all changed rows. Each block includes the hash of the previous block. The result is a chain: modifying any historical row invalidates all subsequent hashes. An auditor can verify the entire chain in seconds.

    Azure SQL Ledger has two table types:

    Updatable Ledger Tables: Support INSERT, UPDATE, DELETE. Maintain a separate ledger_history table containing the full history of every change, with who made it and when. The current table shows current state. The history table is append-only and protected by the hash chain.

    Append-Only Ledger Tables: Support INSERT only. No UPDATE, no DELETE. Appropriate for immutable audit logs, transaction records, event streams.

    -- Create an updatable ledger table for account balances
-- The LEDGER = ON clause activates all ledger machinery
CREATE TABLE dbo.AccountBalance (
    AccountID       INT             NOT NULL,
    CustomerName    NVARCHAR(100)   NOT NULL,
    Balance         DECIMAL(18,2)   NOT NULL,
    LastModified    DATETIME2       NOT NULL DEFAULT SYSUTCDATETIME()
)
WITH (
    SYSTEM_VERSIONING = ON,
    LEDGER = ON
);
GO

-- Azure SQL automatically creates dbo.AccountBalance_Ledger
-- and the ledger history table with these system columns:
-- ledger_start_transaction_id  BIGINT
-- ledger_end_transaction_id    BIGINT  
-- ledger_start_sequence_number BIGINT
-- ledger_end_sequence_number   BIGINT
-- ledger_operation_type        TINYINT  (1=INSERT, 2=DELETE for update tracking)
-- ledger_operation_type_desc   NVARCHAR(128)

-- Create an append-only ledger table for payment events
CREATE TABLE dbo.PaymentAudit (
    PaymentID       UNIQUEIDENTIFIER NOT NULL DEFAULT NEWSEQUENTIALID(),
    AccountID       INT              NOT NULL,
    Amount          DECIMAL(18,2)    NOT NULL,
    PaymentTimestamp DATETIME2       NOT NULL DEFAULT SYSUTCDATETIME(),
    PaymentMethod   NVARCHAR(50)     NOT NULL,
    ProcessorRef    NVARCHAR(100)    NULL
)
WITH (
    LEDGER = ON (APPEND_ONLY = ON)
);
GO

-- Verify the ledger configuration
SELECT 
    name,
    is_ledger,
    ledger_type_desc,
    is_dropped_ledger_table
FROM sys.tables
WHERE name IN ('AccountBalance', 'PaymentAudit');
GO

    #### Verifying the Ledger

    Verification is the feature that makes auditors comfortable. The sp_verify_database_ledger procedure recomputes the entire hash chain and reports any tampering.

    -- Generate the database ledger digest (send to external immutable storage)
-- This digest is what you share with auditors as the "known good" state
EXECUTE sp_generate_database_ledger_digest;
GO
-- Returns JSON:
-- {
--   "database_name": "prod-app-db",
--   "block_id": 42,
--   "hash": "0x3A7F8B2C...",
--   "last_transaction_commit_time": "2024-11-15T14:22:11.123Z",
--   "digest_version": 1
-- }

-- Verify ledger integrity using a previously stored digest
-- Pass the digest JSON retrieved from your immutable storage (Azure Blob with WORM)
EXECUTE sp_verify_database_ledger
    N'[{"database_name":"prod-app-db","block_id":42,"hash":"0x3A7F8B2C...","last_transaction_commit_time":"2024-11-15T14:22:11.123Z","digest_version":1}]';
GO
-- Returns: The database ledger has been successfully verified.

-- View the full audit history for a specific account
-- This queries the automatically maintained history table
SELECT 
    h.AccountID,
    h.CustomerName,
    h.Balance,
    h.LastModified,
    h.ledger_operation_type_desc AS Operation,
    t.commit_time AS ChangeTime,
    t.principal_name AS ChangedBy
FROM dbo.AccountBalance_Ledger h
INNER JOIN sys.database_ledger_transactions t
    ON h.ledger_start_transaction_id = t.transaction_id
WHERE h.AccountID = 10042
ORDER BY h.ledger_start_sequence_number;
GO

    #### Digest Storage Strategy

    A ledger without an independently stored digest is less useful than it appears. If an attacker can modify both the ledger data and the digest stored in the same database, they can reconstruct a consistent but fraudulent chain. Digests must be stored in immutable external storage:

  • → **Azure Blob Storage with immutable storage policies (WORM):** The native Azure integration. Use `sp_generate_database_ledger_digest` on a schedule (hourly or after each significant batch) and write digests to a locked container.
  • → **Azure Confidential Ledger:** A purpose-built service using hardware-backed distributed ledger. Digests written here are independently verified by an infrastructure you don't control.
    • # Configure automatic digest storage to Azure Blob immutable storage
az sql db ledger-digest-uploads enable \
  --resource-group "rg-prod-eastus" \
  --server "prod-sql-server" \
  --database "prod-app-db" \
  --endpoint "https://prodauditdigests.blob.core.windows.net/ledger-digests"

# Verify digest upload status
az sql db ledger-digest-uploads show \
  --resource-group "rg-prod-eastus" \
  --server "prod-sql-server" \
  --database "prod-app-db"

    17.5 Layering the Security Stack: TDE + Always Encrypted + Ledger Together

    These three features protect against different attackers and different threat vectors. In a production system handling regulated data, you often need all three:

    | Feature | Protects Against | Doesn't Protect Against |

    |---|---|---|

    | TDE | Stolen storage media, backup files | Privileged DB users, in-memory access |

    | Always Encrypted | Privileged DBAs, compromised credentials | Tampered historical records |

    | Ledger | Tampering with historical data | Data exposure to privileged users |

    The combination creates defense in depth: TDE ensures backups are useless without your Key Vault key. Always Encrypted ensures DBAs see ciphertext on sensitive columns. Ledger ensures that even if someone does gain write access, any modification is detectable.
